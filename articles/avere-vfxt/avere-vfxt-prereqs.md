# <a name="prepare-to-create-the-avere-vfxt"></a>Подготовка к созданию Avere vFXT

В этой статье разъясняются предварительные задачи для создания кластера Avere vFXT.

## <a name="create-a-new-subscription"></a>Создание подписки

Начните с создания подписки Azure. Используйте отдельную подписку для каждого проекта Avere vFXT для упрощения отслеживания расходов и очистки, а также для обеспечения того, чтобы другие проекты не были затронуты квотами или регулированием ресурсов при подготовке кластерного рабочего процесса.

Чтобы создать подписку Azure на портале Azure:

1. Перейдите в колонку [Подписки](https://ms.portal.azure.com/#blade/Microsoft_Azure_Billing/SubscriptionsBlade).
1. Нажмите кнопку **+ Добавить** вверху.
1. Войдите при появлении соответствующего запроса.
1. Выберите предложение и выполните действия, чтобы создать подписку.

## <a name="configure-subscription-owner-permissions"></a>Настройка разрешений владельца подписки

Кластер vFXT должен создать пользователь с разрешениями владельца подписки. Создание кластера требует от владельца принять условия обслуживания программного обеспечения и разрешить изменения сетевых и складских ресурсов.

Есть некоторые обходные пути, позволяющие невладельцу создать кластер Avere vFXT для кластера Azure. Эти сценарии включают ограничение ресурсов и назначение создателю дополнительных ролей управления доступом на основе ролей (RBAC). Во всех этих случаях владелец подписки также должен [заранее принять условия программного обеспечения Avere vFXT.](#accept-software-terms)

| Сценарий | Ограничения | Роли доступа, необходимые для создания кластера Avere vFXT |
|----------|--------|-------|
| Администратор группы ресурсов создает vFXT | Виртуальная сеть, кластерный контроллер и кластерные узлы должны создаваться в группе ресурсов. | [Администратор пользовательского доступа](../role-based-access-control/built-in-roles.md#user-access-administrator) и роли [вкладчика,](../role-based-access-control/built-in-roles.md#contributor) оба притаких к целевой группе ресурсов. |
| Используйте существующую внешнюю виртуальную сеть | Контроллер кластера и кластерные узлы создаются в группе ресурсов vFXT, но используют существующую виртуальную сеть в другой группе ресурсов. | (1) [Администратор пользовательского доступа](../role-based-access-control/built-in-roles.md#user-access-administrator) и роли [вкладчика,](../role-based-access-control/built-in-roles.md#contributor) относящиеся к группе ресурсов vFXT; и (2) [Виртуальный вклад в машину](../role-based-access-control/built-in-roles.md#virtual-machine-contributor), [Администратор пользовательского доступа](../role-based-access-control/built-in-roles.md#user-access-administrator)и [роли Avere Contributor,](../role-based-access-control/built-in-roles.md#avere-contributor) относящиеся к группе ресурсов виртуальной сети. |
| Пользовательская роль для создателей кластеров | Нет ограничений на размещение ресурсов. Этот метод дает невладельцам значительные привилегии. | Владелец подписки создает пользовательскую роль RBAC, как это объясняется в [этой статье.](avere-vfxt-non-owner.md) |

## <a name="quota-for-the-vfxt-cluster"></a>Квота для кластера vFXT

Убедитесь, что у вас есть достаточная квота для следующих компонентов Azure. При необходимости [подайте запрос на увеличение квоты](https://docs.microsoft.com/azure/azure-supportability/resource-manager-core-quotas-request).

> [!NOTE]
> Виртуальные машины и компоненты SSD, перечисленные здесь, предназначены для самого кластера vFXT. Помните, что вам также нужна квота для VMs и SSD, которые вы будете использовать для вашей вычислительной фермы.
>
> Убедитесь, что квота включена для региона, где планируется запустить рабочий процесс.

|Компонент Azure|Quota|
|----------|-----------|
|Виртуальные машины|3 или более E32s_v3 (по одному на кластерный узор) |
|Хранилище SSD (цен. категория "Премиум")|200 ГБ пространства ОС плюс от 1 до 4 ТБ объема кэша на узел |
|Учетная запись хранения (необязательно) |Версия 2|
|Серверное хранилище данных (необязательно) |Один новый контейнер больших двоичных объектов в LRS |

## <a name="accept-software-terms"></a>Принятие условий программного обеспечения

> [!TIP]
> Пропустите этот шаг, если владелец подписки создаст кластер Avere vFXT.

Во время создания кластера необходимо принять условия предоставления услуг для программного обеспечения Avere vFXT. Если вы не являетесь владельцем подписки, он должен принять условия заранее.

Этот шаг достаточно выполнить один раз для каждой подписки.

Чтобы принять условия программного обеспечения заранее:

1. Откройте Cloud Shell на портале Azure или перейдите по адресу <https://shell.azure.com>. Войдите с помощью идентификатора своей подписки.

   ```azurecli
    az login
    az account set --subscription <subscription ID>
   ```

1. Выполните эту команду, чтобы принять условия предоставления услуг и включить программный доступ для Avere vFXT для образа программного обеспечения в Azure:

   ```azurecli
   az vm image accept-terms --urn microsoft-avere:vfxt:avere-vfxt-controller:latest
   ```

## <a name="create-a-storage-service-endpoint-in-your-virtual-network-if-needed"></a>Создание конечная точка службы хранения данных в виртуальной сети (при необходимости)

[Конечная точка службы](../virtual-network/virtual-network-service-endpoints-overview.md) сохраняет локальный трафик Azure Blob вместо того, чтобы вымотать его за пределы виртуальной сети. Рекомендуется для любого кластера Avere vFXT для кластера Azure, использующему Azure Blob для хранения данных.

При создании кластера создается новая виртуальная сеть, автоматически создается конечная точка. Если вы предоставляете существующую виртуальную сеть, она должна иметь конечную точку службы хранения данных Майкрософт, если вы хотите создать новый контейнер для хранения blob во время создания кластера.<!-- if there is no endpoint in that situation, the cluster creation will fail -->

> [!TIP]
>
>* Пропустите этот шаг, если вы создаете новую виртуальную сеть в рамках создания кластера.
>* Конечная точка не является обязательной, если вы не создаете хранилище Blob во время создания кластера. В этом случае конечный пункт службы можно создать позже, если решите использовать Azure Blob.

Создайте конечную точку службы хранения с портала Azure.

1. На портале откройте список виртуальных сетей.
1. Выберите виртуальную сеть для кластера.
1. Нажмите **на конечные точки службы** из левого меню.
1. В верхней части страницы щелкните **Добавить**.
1. Выберите ``Microsoft.Storage``услугу .
1. Выберите подсеть кластера.
1. В нижней части страницы щелкните **Добавить**.
