# <a name="pricing-model-for-azure-logic-apps"></a>Модель ценообразования для Azure Logic Apps

[Служба Azure Logic Apps](../logic-apps/logic-apps-overview.md) облегчает создание и запуск автоматизированной интеграции рабочих процессов, можно масштабировать в облаке. Этой статье описывается, как работает выставление счетов и цены для Azure Logic Apps. Точные сведения о ценах см. в разделе [Azure ценами на Logic Apps](https://azure.microsoft.com/pricing/details/logic-apps).

<a name="consumption-pricing"></a>

## <a name="consumption-pricing-model"></a>Модель тарификации по мере использования

Для новых приложений логики, выполняющихся в службе Azure Logic Apps public или «глобальные» вы платите только за использование. Эти приложения логики используют план и модель ценообразования на основе потребления. Каждый шаг в определении приложения логики является действием. Например следующие действия.

* Триггеры, которые являются специальные действия. Все приложения логики требуется триггера в качестве первого шага.
* «Встроенных» или собственного действия, такие как HTTP, вызовы функций Azure и управления API и т. д.
* Вызовы соединители, такие как Outlook 365, Dropbox и т. д.
* Этапы потока, такие как циклы, условные операторы, управления и т. д

Служба Azure Logic Apps счетчики все действия, выполняемые в приложении логики. Дополнительные сведения о том, как работает выставление счетов для [триггеры](#triggers) и [действия](#actions).

<a name="fixed-pricing"></a>

## <a name="fixed-pricing-model"></a>Модель фиксированного ценообразования

[ *Среда службы интеграции* (ISE)](../logic-apps/connect-virtual-network-vnet-isolated-environment-overview.md) предоставляет закрытый, изолированную и выделенную способ для создания и запуска приложения логики, которые могут обращаться к ресурсам виртуальной сети Azure. Для новых приложений логики, которые выполняются в интегрированную среду Сценариев, вы платите [фиксированный месячный тариф](https://azure.microsoft.com/pricing/details/logic-apps) для встроенных действий и триггеров, а также для стандартных соединителей.

В интегрированной среде Сценариев также включает в себя один бесплатный пакет Enterprise соединитель, который включает в себя столько *подключений* желаемого. Использование дополнительных корпоративных соединителей взимается на основе [Enterprise цена потребления](https://azure.microsoft.com/pricing/details/logic-apps). Только обычно доступные корпоративные соединители, оплачивается по цена потребления Enterprise. Корпоративные соединители общедоступной предварительной версии взимается по [скорость стандартный соединитель](https://azure.microsoft.com/pricing/details/logic-apps).

> [!NOTE]
> В интегрированную среду Сценариев, встроенных триггеров и действий отображения **Core** метки и запустить в одной интегрированной среде Сценариев, как приложения логики. "Стандартный" и корпоративные соединители, которые отображают **интегрированной среды Сценариев** метку, выполните в одной интегрированной среде Сценариев, как приложения логики. Соединители, которые не отображать метки интегрированной среды Сценариев запуска в глобальной службе Logic Apps.

Вашей интегрированной среды Сценариев базовый модуль имеет фиксированную емкости, поэтому если вам требуется более высокая пропускная способность, вы можете [добавить дополнительные единицы масштабирования](../logic-apps/connect-virtual-network-vnet-isolated-environment.md#add-capacity), либо во время создания либо уже после этого. Логика приложения, работающие в интегрированную среду Сценариев не подлежат затраты на хранение данных.

Точные сведения о ценах см. в разделе [Azure ценами на Logic Apps](https://azure.microsoft.com/pricing/details/logic-apps).

<a name="connectors"></a>

## <a name="connectors"></a>Соединители

Соединители Azure Logic Apps помочь ваш доступ к приложениям логики приложения, служб и систем в облаке или в локальной среде, предоставив [триггеры](#triggers), [действия](#actions), или оба. Соединители подразделяются на выпуск Standard или Enterprise. Общие сведения об этих соединителей, см. в разделе [соединители для Azure Logic Apps](../connectors/apis-list.md). Если нет готовых соединителей доступны для REST API, которые вы хотите использовать в приложениях логики, вы можете создать [настраиваемые соединители](https://docs.microsoft.com/connectors/custom-connectors), которые представляют собой просто оболочками вокруг те REST API. Настраиваемые соединители оплачиваются как стандартные соединители. Следующие разделы содержат дополнительные сведения о как выставление счетов для триггеров и действия работать.

<a name="triggers"></a>

## <a name="triggers"></a>Триггеры

Триггер — это особое действие, при котором создается экземпляр приложения логики, когда происходит определенное событие. Есть несколько моделей поведения триггеров, которые влияют на тарификацию приложения логики. Ниже приведены различные типы триггеров, которые существуют в Azure Logic Apps.

* **Опрашивающий триггер**: Такой триггер непрерывно проверяет конечную точку для сообщения, удовлетворяющие критериям для создания экземпляра приложения логики и запуска рабочего процесса. Даже если не создается экземпляр приложения логики, Logic Apps учитывает каждый опрашивающий запрос как одно выполнение. Интервал опроса для триггера можно указать в конструкторе приложений логики.

  [!INCLUDE [logic-apps-polling-trigger-non-standard-metering](../../includes/logic-apps-polling-trigger-non-standard-metering.md)]

* **Триггер веб-перехватчика**: Этот триггер ожидает от клиента для отправки запроса на указанную конечную точку. Каждый запрос, отправленный в конечную точку webhook, учитывается как выполненное действие. К триггерам веб-перехватчика относятся триггеры веб-перехватчика HTTP и триггеры запросов.

* **Триггер повторения**: Этот триггер создает экземпляр приложения логики, в зависимости от интервала повторения, который настроен в триггере. Например можно задать триггер повторения, который выполняется каждые три дня или на более сложное расписание.

<a name="actions"></a>

## <a name="actions"></a>Действия

Служба Azure Logic Apps счетчики «встроенных» действия, такие как HTTP, как собственные действия. Например встроенные действия включить HTTP-вызовы, вызовы из функции Azure или службе управления API и этапы потока, например условия, циклы, управления и операторы switch. Каждое действие имеет свои собственные тип действия. Например, действия, которые вызывают [соединители](https://docs.microsoft.com/connectors) имеют тип «ApiConnection». Эти соединители подразделяются на стандартный или корпоративные соединители, которые измеряются в зависимости от соответствующих им [цены](https://azure.microsoft.com/pricing/details/logic-apps). Корпоративные соединители в *предварительной версии* , тарифицируются как стандартные соединители.

Служба Azure Logic Apps счетчики всех успешных и неуспешных действий как выполнения. Тем не менее приложения логики не вести учет этих действий:

* которые были пропущены из-за несоблюдения условий;
* которые не были выполнены из-за того, что приложение логики непредвиденно завершилось.

Для действий, которые выполняются в циклах Azure Logic Apps подсчитывает каждое действие для каждой итерации цикла. Давайте для примера рассмотрим цикл "for each", который обрабатывает список. Logic Apps при учете действие в этом цикле умножает число элементов списка на число действий в цикле, а также добавляет действие запуска цикла. Таким образом, расчет список 10 элементов: (10 * 1) + 1 мы получаем 11 выполнений действий.

## <a name="disabled-logic-apps"></a>Отключенные приложения логики

Отключить логику, приложения не взимается плата, так как они не могут создавать новые экземпляры, хотя они включены.
Когда вы отключаете приложение логики, может потребоваться некоторое время до полной остановки всех его запущенных экземпляров.

## <a name="integration-accounts"></a>Учетные записи службы интеграции

Фиксированный модель ценообразования применяется к [учетные записи интеграции](logic-apps-enterprise-integration-create-integration-account.md) которых можно исследовать, разрабатывать и тестировать [B2B и EDI](logic-apps-enterprise-integration-b2b.md) и [обработки XML](logic-apps-enterprise-integration-xml.md) возможности в Azure Logic Apps на нет Дополнительные затраты.
В каждом регионе Azure, можно иметь одну учетную запись интеграции. Каждая учетная запись интеграции может хранить не более определенного [числа артефактов](../logic-apps/logic-apps-limits-and-config.md), например торговых партнеров, соглашений, карт, схем, сборок, сертификатов и конфигураций пакетов.

Azure Logic Apps предоставляет Free, Basic и Standard учетные записи интеграции. На уровнях Basic и Standard, поддерживаются соглашением с Logic Apps уровня обслуживания (SLA), уровень "бесплатный" не поддерживается соглашением об уровне ОБСЛУЖИВАНИЯ и имеет ограничения на пропускную способность и использование.

Чтобы выбрать учетную запись интеграции Free, Basic или Standard:

* **Бесплатный**. Для при необходимости повторите произвольного сценариев, а не в рабочей среде.

* **Базовый**. Если требуется только обработку сообщений или в качестве представителем малого бизнеса с отношения с торговым партнером, с более крупным юридическим лицом.

* **Стандартный**. Когда у вас есть более сложные связи B2B и большего числа сущностей, которые необходимо Управляйте.

Точные сведения о ценах см. в разделе [цены на Azure Logic Apps](https://azure.microsoft.com/pricing/details/logic-apps).

<a name="data-retention"></a>

## <a name="data-retention"></a>Хранение данных

За исключением приложений логики, выполните в среде службы интеграции (ISE), все входные и выходные данные, которые хранятся в журнале выполнения приложения логики будет выставлен на основе приложения логики [запуска срок](logic-apps-limits-and-config.md#run-duration-retention-limits). Логика приложения, работающие в интегрированную среду Сценариев не подлежат затраты на хранение данных. Точные сведения о ценах см. в разделе [цены на Azure Logic Apps](https://azure.microsoft.com/pricing/details/logic-apps).

Чтобы помочь вам отслеживать использование ресурсов хранения приложения логики, вы можете:

* Просмотрите число единиц хранилища в ГБ, приложение логики использует ежемесячно.
* Просмотр размеров для ввода и вывода определенного действия в журнале выполнения приложения логики.

<a name="storage-consumption"></a>

### <a name="view-logic-app-storage-consumption"></a>Просмотр приложения логики хранения данных об использовании

1. На портале Azure найдите и откройте нужное приложение логики.

1. В меню приложения логики в разделе **мониторинг**выберите **метрики**.

1. На панели справа в разделе **заголовок диаграммы**, из **метрика** выберите **выставления счетов об использовании для выполнений потребление хранилища**.

   Эта метрика дает число потребления единиц хранения в ГБ в месяц, которые являются выставление счетов.

<a name="input-output-sizes"></a>

### <a name="view-action-input-and-output-sizes"></a>Просмотреть действие ввода и вывода размеры

1. На портале Azure найдите и откройте нужное приложение логики.

1. В меню приложения логики выберите **Обзор**.

1. На панели справа в разделе **журнал запусков**, выберите запуск, который имеет входные и выходные данные для проверки.

1. В разделе **запуск приложения логики**, выберите **подробности о выполнении**.

1. В **приложение логики, сведения о выполнении** области таблицы действия, которая возвращает значение, состояние и длительность каждого действия, выберите действие, вы хотите просмотреть.

1. В **действие приложения логики** области, поиска, размеры входные и выходные данные этого действия отображаются соответственно в разделе **ссылка на входные данные** и **ссылка на выходные данные**.
