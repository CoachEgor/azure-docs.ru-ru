### <a name="change-run-duration-and-run-retention-in-storage"></a>Изменение продолжительности выполнения и удержания в хранилище

Чтобы изменить лимит по умолчанию для длительности выполнения и удержания в хранилище, выполните следующие действия. Чтобы увеличить максимальный лимит, [обратитесь в команду Logic Apps](mailto://logicappsemail@microsoft.com) за помощью в ваших требованиях.

> [!NOTE]
> Для логических приложений в мультитенантной Azure 90-дневный лимит по умолчанию совпадает с максимальным лимитом. Вы можете только уменьшить это значение.
> Для логических приложений в среде служб интеграции можно уменьшить или увеличить 90-дневный лимит по умолчанию.

1. Перейдите на [портал Azure](https://portal.azure.com). В поле поиска портала, найти и выбрать **логику приложений**.

1. Выберите, а затем откройте приложение логики в Logic App Designer.

1. В меню приложения логики выберите **параметры рабочего процесса.**

1. Под **опциями Runtime,** из **удержания истории Run в** списке дней, выберите **Custom.**

1. Перетащите ползунок, чтобы изменить количество дней, которые вы хотите.

1. Когда вы закончите работу, на панели **параметров рабочего процесса** выберите **Сохранить.**

<a name="disable-delete"></a>

## <a name="disabling-or-deleting-logic-apps"></a>Отключение или удаление приложений логики

При отключении приложения логики новые экземпляры запуска не создаются.
Все незавершенные и ожидающие запуски продолжаются до тех пор, пока они не закончатся, что может занять время.

При удалении приложения логики новые экземпляры запуска не создаются.
Всех выполняющиеся и ожидающие выполнения запуски будут отменены.
При наличии тысяч операций запуска отмена может занять значительное время.

<a name="configuration"></a>

## <a name="firewall-configuration-ip-addresses-and-service-tags"></a>Конфигурация брандмауэра: IP-адреса и теги служб

IP-адреса, используемые Azure Logic Apps для входящих и исходящих вызовов, зависят от региона, в котором существует приложение логики. *Все логические* приложения в одном регионе используют одни и те же диапазоны IP-адресов. Некоторые вызовы [Power Automate,](https://docs.microsoft.com/power-automate/getting-started) такие как **http** и **HTTP и OpenAPI** запросы, идут непосредственно через службу приложений Логика Azure и приходят с IP-адресов, которые перечислены здесь. Для получения дополнительной информации об IP-адресах, используемых Power Automate, [см.](https://docs.microsoft.com/flow/limits-and-config#ip-address-configuration)

> [!TIP]
> Чтобы снизить сложность при создании правил безопасности, можно дополнительно использовать [теги служб,](../virtual-network/service-tags-overview.md)а не указывать IP-адреса Logic Apps для каждого региона, описанные позднее в этом разделе. Эти теги работают в регионах, где доступна служба Logic Apps:
>
> * **LogicAppsManagement**: Представляет префиксы входящих IP-адресов для службы Logic Apps.
> * **LogicApps**: Представляет исходящие префиксы IP-адреса для службы Logic Apps.

* Для поддержки вызовов, которые ваши логические приложения непосредственно делают с [HTTP,](../connectors/connectors-native-http.md) [HTTP и Swagger](../connectors/connectors-native-http-swagger.md), и другие запросы HTTP, настройте брандмауэр со всеми [входящими](#inbound) *и* [исходящими](#outbound) IP-адресами, которые используются службой Logic Apps, на основе регионов, где существуют приложения для логики. Эти адреса отображаются под заголовками **Входящие** и **Исходящие** в этом разделе и сортируются по регионам.

* Для поддержки вызовов [соединителей, управляемых Майкрософт](../connectors/apis-list.md), настройте брандмауэр так, чтобы он включал *все*[исходящие](#outbound) IP-адреса, используемые этими соединителями, в зависимости от регионов, где существуют приложения логики. Эти адреса отображаются под заголовком **Исходящие** в этом разделе и сортируются по регионам.

* Для обеспечения связи для логических приложений, которые работают в среде интеграционных служб (ISE), убедитесь, что вы [открываете эти порты.](../logic-apps/connect-virtual-network-vnet-isolated-environment.md#network-ports-for-ise)

* Если у ваших логических приложений есть проблемы с доступом к учетным записям хранения Azure, которые используют [брандмауэры и правила брандмауэра,](../storage/common/storage-network-security.md)у вас есть [различные параметры для включения доступа.](../connectors/connectors-create-api-azureblobstorage.md#access-storage-accounts-behind-firewalls)

  Например, логические приложения не могут напрямую получить доступ к учетным записям хранилищ, которые используют правила брандмауэра и существуют в одном регионе. Однако, если вы разрешаете [исходящие IP-адреса для управляемых разъемов в вашем регионе,](../logic-apps/logic-apps-limits-and-config.md#outbound)ваши логические приложения могут получить доступ к учетным записям хранения, которые находятся в другом регионе, за исключением случаев использования разъемов хранения таблицaz или разъемов azure Queue Storage. Чтобы получить доступ к таблице хранения или в очереди хранения, вы можете использовать триггер и действия HTTP вместо. Для других опций см. [учетные записи хранения доступа за брандмауэрами.](../connectors/connectors-create-api-azureblobstorage.md#access-storage-accounts-behind-firewalls)

* Для пользовательских разъемов, [Azure government](../azure-government/documentation-government-overview.md)и [Azure China 21Vianet](https://docs.microsoft.com/azure/china/), фиксированные или зарезервированные IP-адреса недоступны.
