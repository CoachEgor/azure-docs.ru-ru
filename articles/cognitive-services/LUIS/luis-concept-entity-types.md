# <a name="entities-and-their-purpose-in-luis"></a>Организации и их назначение в LUIS

Основная цель сущностей заключается в предоставлении клиенту приложения предсказуемой извлечения данных. _Необязательная,_ второстепенная цель заключается в том, чтобы повысить прогноз намерения или других сущностей с дескрипторами.

Существует два типа сущностей:

* машинно-изученный - от контекста
* не-машинно-изученных - для точных текстовых совпадений, шаблон совпадений, или обнаружения заранее построенных объектов

Машино-изученные сущности обеспечивают самый широкий спектр вариантов извлечения данных. Немашинообразованные сущности работают на основе сопоставления текста и могут использоваться независимо или в качестве [ограничения](#design-entities-for-decomposition) для объекта, извлеченного из машин.

## <a name="entities-represent-data"></a>Сущности представляют данные

Сущности — это данные, которые вы хотите вытащить из высказывания, такие как имена, даты, названия продуктов или любая значительная группа слов. Фраза может содержать много сущностей или вообще ни одной. Клиенту приложению _могут_ понадобиться данные для выполнения своей задачи.

Сущности должны быть последовательно помечены во всех учебных высказываниях для каждого намерения в модели.

 Вы можете определить свои собственные сущности или использовать заранее построенные объекты, чтобы сэкономить время для общих понятий, таких как [datetimeV2](luis-reference-prebuilt-datetimev2.md), [ordinal,](luis-reference-prebuilt-ordinal.md) [email](luis-reference-prebuilt-email.md)и [номер телефона.](luis-reference-prebuilt-phonenumber.md)

|Фраза|Сущность|Данные|
|--|--|--|
|Купить 3 билета в Нью-Йорк|Предварительно созданная сущность number<br>Location.Destination|3<br>Нью-Йорк|
|Купить билет из Нью-Йорка в Лондон на 5 марта|Location.Origin<br>Location.Destination<br>Предварительно созданная сущность datetimeV2|Нью-Йорк<br>London<br>5 марта 2018 г.|

### <a name="entities-are-optional"></a>Сущности не являются обязательными

Тогда как намерения являются обязательными, использовать сущности необязательно. Не нужно создавать сущности для каждого понятия в приложении — они требуются только для тех, которые требуются для работы клиентского приложения.

Если в ваших высказываниях нет данных, необходимых клиенту, не нужно добавлять сущности. По мере разработки приложения и выявления новой потребности в данных можно добавить соответствующие сущности в свою модель LUIS позже.

## <a name="entity-compared-to-intent"></a>Сущность по сравнению с намерением

Сущность представляет концепцию данных внутри высказывания, которое вы хотите извлечь.

Высказывание может по желанию включать сущности. Для сравнения, предсказание намерения для высказывания _не требуется_ и представляет собой все высказывание. LUIS требует примера высказывания, содержащиеся в намерениях.

Рассмотрим следующие 4 высказывания:

|Фраза|Прогнозирование намерения|Объекты, извлеченные|Объяснение|
|--|--|--|--|
|Справка|help|-|Ничего не добывать.|
|Отправить что-то|sendSomething|-|Ничего не добывать. Модель не была обучена для извлечения `something` в этом контексте, и нет получателя либо.|
|Отправить Бобу подарок|sendSomething|`Bob`, `present`|Модель была обучена с [personName](luis-reference-prebuilt-person.md) заранее построенной `Bob`сущности, которая извлекла имя . Для извлечения `present`используется объект, изученный машиной.|
|Отправить Боб коробку конфет|sendSomething|`Bob`, `box of chocolates`|Два важных фрагмента `Bob` данных, и `box of chocolates`, были извлечены субъектами.|

## <a name="design-entities-for-decomposition"></a>Конструкционные объекты для разложения

Это хороший дизайн сущности, чтобы сделать ваш топ-уровня лица машино-изученных лица. Это позволяет изменять конструкцию сущности с течением времени и использовать **подкомпоненты** (детские сущности), по желанию с **ограничениями** и **дескрипторами,** разлагать сущность верхнего уровня на части, необходимые клиентским приложением.

Проектирование для разложения позволяет LUIS вернуть глубокую степень разрешения сущности в ваше клиентское приложение. Это позволяет клиенту приложение сосредоточиться на бизнес-правилах и оставить разрешение данных НА LUIS.

### <a name="machine-learned-entities-are-primary-data-collections"></a>Машино-изученные сущности являются основными сборами данных

[**Машино-изученные сущности**](tutorial-machine-learned-entity.md) являются единицей данных верхнего уровня. Субкомпоненты — это детские сущности машино-изученных сущностей.

Машинно-изученная сущность триггеров на основе контекста, извлеченного через учебные высказывания. **Ограничения** — это дополнительные правила, применяемые к машино-изученной сущности, которые еще больше ограничивают срабатывание на основе точного текста, сопрягающего определение немашино-изученной сущности, такой как [Список](reference-entity-list.md) или [Regex.](reference-entity-regular-expression.md) Например, `size` объект, изучаемый машиной, `sizeList` может иметь ограничение `size` сущности списка, которое `sizeList` ограничивает срабатывание сущности только при обнаружении значений, содержащихся в объекте.

[**Дескрипторы**](luis-concept-feature.md) – это функции, применяемые для повышения актуальности слов или фраз для прогнозирования. Они называются *дескрипторами,* поскольку они используются для *описания* намерения или сущности. Дескрипторы описывают отличительные черты или атрибуты данных, такие как важные слова или фразы, которые LUIS наблюдает и учится.

При создании функции списка фраз в приложении LUIS она включена по умолчанию по умолчанию и равномерно применяется во всех намерениях и сущностях. Однако, если применить список фраз в качестве дескриптора (функции) объекта (или *модели),* то его сфера применения уменьшается только к этой модели и больше не используется со всеми другими моделями. Использование списка фраз в качестве дескриптора модели помогает разложению, помогая с точностью для модели, к которую она применяется.

## <a name="extracting-contextually-related-data"></a>Извлечение контекстуально связанных данных

Высказывание может содержать два или более случаев сущности, где значение данных основано на контексте в высказывании. Примером может быть высказывание при бронировании рейса, в который есть два места, происхождение и пункт назначения.

`Book a flight from Seattle to Cairo`

Два примера `location` сущности должны быть извлечены. Клиент-приложение должно знать тип местоположения для каждого для того, чтобы завершить покупку билета.

Существует два метода извлечения данных, связанных с контекстом:

 * Сущность `location` является машино-изученной сущностью и `origin` использует `destination` два подкомпонентных сущности для захвата и (предпочтительного)
 * Сущность `location` использует две `origin` **роли** и`destination`

Несколько сущностей могут существовать в высказывании и могут быть извлечены без использования разложения или ролей, если контекст, в котором они используются, не имеет значения. Например, если высказывание включает список `I want to travel to Seattle, Cairo, and London.`местоположений, это список, в котором каждый элемент не имеет дополнительного значения.

### <a name="using-subcomponent-entities-of-a-machine-learned-entity-to-define-context"></a>Использование подкомпонентных сущностей машино-изученной сущности для определения контекста

Можно использовать [**объект, изученный машиной,**](tutorial-machine-learned-entity.md) для извлечения данных, описывающих действие бронирования рейса, а затем для разложения сущности верхнего уровня на отдельные части, необходимые клиентским приложением.

В этом `Book a flight from Seattle to Cairo`примере сущность верхнего `travelAction` уровня может `flight from Seattle to Cairo`быть и помечена для извлечения. Затем создаются два подкомпонентных сущности, называемые `origin` и, `destination` `geographyV2` оба с ограничением, применяемым заранее построенной сущностью. В тренировках высказывания, `origin` и `destination` обозначены соотвествующим.

### <a name="using-entity-role-to-define-context"></a>Использование роли сущности для определения контекста

Роль — это псевдоним для сущности, основанной на контексте в высказывании. Роли можно использовать с любым предварительно созданным или настраиваемым типом сущности. Они используются в примерах речевых фрагментов и шаблонов. В этом примере сущность нуждается в `location` двух ролях `origin` и `destination` и обе должны быть отмечены в примере высказываний.

Если LUIS `location` находит, но не может определить роль, объект местоположения по-прежнему возвращается. Клиентское приложение должно будет следить за вопросом, чтобы определить, какой тип местоположения имел в виду пользователь.


## <a name="if-you-need-more-than-the-maximum-number-of-entities"></a>Если требуется количество сущностей, превышающее максимальное значение

Если вам нужно больше, чем предел, обратитесь в службу поддержки. Для этого соберите подробные сведения о системе, откройте веб-сайт [LUIS](luis-reference-regions.md#luis-website), а затем выберите элемент **Поддержка**. Если ваша подписка Azure включает службы поддержки, обратитесь в службу [технической поддержки Azure](https://azure.microsoft.com/support/options/).

## <a name="entity-prediction-status"></a>Состояние прогнозирования сущности

Портал LUIS показывает, когда сущность, в примере высказывания, имеет другой прогноз сущности, чем объект, который вы выбрали. Эта различная оценка основана на текущей обученной модели.
