# <a name="speech-service-for-telephony-data"></a>Речевой сервис для телефонных данных

Данные телефонии, полученные со стационарных и мобильных телефонов, а также радиоустройств, обычно имеют довольно низкое качество и узкую полосу частот (около 8 кГц). Это создает дополнительные трудности при преобразовании речи в текст. Последние модели распознавания речи от службы speech преуспевают в расшифровке этих телефонных данных, даже в тех случаях, когда данные трудно понять человеку. Эти модели обучаются с большими объемами телефонных данных и имеют лучшую на рынке точность распознавания, даже в шумных условиях.

Функция преобразования речи в текст часто применяется для транскрибирования больших объемов данных телефонии, которые могут поступать из различных систем, включая системы интерактивного речевого ответа. Это может быть стерео- или монофонический звуковой сигнал в необработанном виде или с минимальной пост-обработкой. Используя службу речевой речи и модель Единой речи, бизнес может получить высококачественные транскрипции, все системы используются для захвата звука.

Данные телефонии позволяют лучше понять потребности клиентов, выявить новые возможности для продаж или оценить работу операторов центра обработки вызовов. После того, как данные будут транскрибированы, компания может использовать результаты для таких целей, как улучшение телеметрии, определение ключевых фраз или анализ настроений клиентов.

Описанные на этой странице технологии используются корпорацией Майкрософт в режиме реального времени в пакетном режиме для внутренних задач, в том числе для разных служб обработки вызовов.

Рассмотрим некоторые технологии и связанные с ней функции, которые предлагает служба speech.

> [!IMPORTANT]
> Модель Службы речевой unified обучается с различными данными и предлагает одномодельное решение для ряда сценариев от диктовки до аналитики телефонии.

## <a name="azure-technology-for-call-centers"></a>Технологии Azure для центров обработки вызовов

Помимо функционального аспекта функций службы speech, их основная цель - при применении к колл-центру - это улучшение обслуживания клиентов. Эту задачу можно разделить на три направления:

- После колл-аналитики, которая по существу пакетной обработки записей вызовов после вызова.
- Аналитика в реальном времени, которая обрабатывает аудиосигнал для извлечения различных идей, как вызов происходит (с чувством является видным случае использования).
- Голосовые помощники (боты), либо движущей диалог между клиентом и ботом в попытке решить проблему клиента без участия агента, или как применение искусственного интеллекта (ИИ) протоколы для оказания помощи агенту.

## <a name="speech-analytics-technology-components"></a>Компоненты технологий для аналитики речи

Независимо от того, является ли домен после вызова или в режиме реального времени, Azure предлагает набор зрелых и новых технологий для улучшения обслуживания клиентов.

### <a name="speech-to-text-stt"></a>Преобразование речи в текст

[Речевой текст](speech-to-text.md) является наиболее востребованной функцией в любом решении колл-центра. Поскольку многие процессы аналитики ниже по течению полагаются на транскрибированный текст, частота ошибок слов _(WER)_ имеет первостепенное значение. Основные проблемы при транскрибировании телефонных вызовов — это шум в центрах обработки вызовов (например, фоновая речь операторов), разнообразие языковых стандартов и диалектов, а также низкое качество телефонного сигнала. WER сильно коррелирует с тем, насколько хорошо акустические и языковые модели обучаются для данного языка, поэтому важна возможность настройки модели под свой язык. Новейшие единые модели версии 4.x позволяют добиться одновременно высокой точности транскрибирования и низкой задержки. Обученные с десятками тысяч часов акустических данных и миллиардами лексической информации, Unified модели являются наиболее точными моделями на рынке для транскрибировать данные колл-центра.

### <a name="sentiment"></a>Мнение

Оценка удовлетворенности клиентов центров обработки вызовов — одна из важнейших задач речевой аналитики. Наш [API пакетного транскрибирования](batch-transcription.md) позволяет выполнять анализ тональности каждого речевого фрагмента. Вы можете объединить набор значений, полученных в расшифровке, чтобы оценить тональность речи операторов и клиентов в ходе разговора.

### <a name="silence-non-talk"></a>Тишина (отсутствие разговора)

Как правило, при вызовах в службу обработки около 35 % занимает время без разговоров. Некоторые сценарии, для которых не-говорить происходит: агенты ищут до истории с клиентом, агенты, использующие инструменты, которые позволяют им получить доступ к рабочему столу клиента и выполнять функции, клиенты сидят на удержание в ожидании передачи, и так далее. Чрезвычайно важно оценить, когда тишина происходит в вызове, поскольку существует ряд важных чувствительности клиентов, которые возникают вокруг этих типов сценариев и где они происходят в вызове.

### <a name="translation"></a>Перевод

Некоторые компании экспериментируют с предоставлением переведенных стенограмм из звонков поддержки иностранного языка, чтобы менеджеры по доставке могли понять мировой опыт своих клиентов. Мы предлагаем непревзойденные средства для [перевода](translation.md). Мы можем переводить аудио-аудио или аудио-к-текст для большого количества языков.

### <a name="text-to-speech"></a>Преобразование текста в речь

[Преобразование текста в речь](text-to-speech.md) – еще один важный аспект реализации бота, который взаимодействует с клиентами. Типичный рабочий процесс выглядит так: клиент произносит фразы, его голос транскрибируется в текст, по нему выполняется анализ намерений, на основе распознанных намерений синтезируется ответ, а затем клиенту предоставляется нужный ресурс или синтезированный голосовой ответ. Конечно, все это должно происходить быстро - таким образом, низкая задержка является важным компонентом в успехе этих систем.

Наша сквозная задержка значительно низка для различных технологий, таких как [речь к тексту](speech-to-text.md), [LUIS](https://azure.microsoft.com/services/cognitive-services/language-understanding-intelligent-service/), [Bot Framework](https://dev.botframework.com/), Текст [к речи](text-to-speech.md).

Наши новые голоса неотличимы от человеческих. Вы можете использовать наши голоса, чтобы придать вашему боту свою уникальную индивидуальность.

### <a name="search"></a>Поиск

Еще одно важное применение аналитики — определение взаимодействий во время возникновения событий или действий. Обычно это делается с помощью одного из двух подходов; либо специальный поиск, в котором пользователь просто вводит фразу и отвечает система, либо более структурированный запрос, в котором аналитик может создать набор логических инструкций, которые идентифицируют сценарий в вызове, а затем каждый вызов может быть проиндексирован на этот набор запросов. Хорошим примером поиска является повсеместное заявление о соответствии "этот вызов должен быть записан для целей качества ... ". Многие компании хотят, чтобы убедиться, что их агенты предоставляют эту оговорку для клиентов, прежде чем вызов на самом деле записаны. Большинство аналитических систем имеют возможность тренда поведения, найденного запрос / поиск алгоритмов, и это сообщение о тенденциях, в конечном счете, одна из наиболее важных функций системы аналитики. Используя [каталог Cognitive Services](https://azure.microsoft.com/services/cognitive-services/directory/search/), вы можете значительно расширить свое комплексное решение функциями индексирования и поиска.

### <a name="key-phrase-extraction"></a>Извлечение ключевых фраз

Эта область является одним из наиболее сложных приложений аналитики и тот, который извлекает выгоду из применения ИИ и машинного обучения. Основной сценарий в этом случае заключается в выводе о намерениях клиента. Для чего этот клиент позвонил? Какая у него возникла проблема? Почему этот клиент недоволен? Наша [служба текстовой аналитики](https://azure.microsoft.com/services/cognitive-services/text-analytics/) предоставляет набор аналитики из коробки для быстрого обновления сквозного решения для извлечения этих важных ключевых слов или фраз.

Теперь мы более подробно рассмотрим операции пакетной обработки и конвейеры распознавания речи в реальном времени.

## <a name="batch-transcription-of-call-center-data"></a>Пакетное транскрибирование данных центра обработки вызовов

Для расшифровки объемного аудио мы [разработали API транскрипции пакета.](batch-transcription.md) API пакетного транскрибирования разработан для асинхронного транскрибирования больших объемов аудиоданных. Что касается расшифровки данных колл-центра, то наше решение основано на следующих столпах:

- **Точность** - С унифицированными моделями четвертого поколения мы предлагаем непревзойденное качество транскрипции.
- **Задержка** - Мы понимаем, что при выполнении объемных транскрипций, транскрипции необходимы быстро. Задания транскрибирования, инициированные через [API пакетного транскрибирования](batch-transcription.md), помещаются в очередь немедленно. При запуске выполнения задания транскрибирование выполняется быстрее, чем в реальном времени.
- **Безопасность** - Мы понимаем, что вызовы могут содержать конфиденциальные данные. Но не беспокойтесь — безопасность для нас является одним из главных приоритетов. Наша служба получила сертификаты ISO, SOC, HIPAA и PCI.

Колл-центры генерируют большие объемы аудиоданных на ежедневной основе. Если ваша организация хранит данные телефонии в центральном расположении, например службе хранилища Azure, вы можете использовать [API пакетного транскрибирования](batch-transcription.md) для асинхронной отправки запросов и получения расшифровок.

Стандартное решение использует следующие службы:

- Служба речевого сообщения используется для расшифровки речевого текста. Стандартная подписка (S0) для службы speech требуется для использования API транскрипции пакета. Бесплатная подписка (F0) не подходит.
- [Служба хранилища Azure](https://azure.microsoft.com/services/storage/) используется для хранения данных телефонии и расшифровок, полученных от API пакетного транскрибирования. В этой учетной записи хранения следует настроить уведомления, в частности для активации при добавлении новых файлов. Эти уведомления используются для активации процесса транскрибирования.
- [Функции Azure](https://docs.microsoft.com/azure/azure-functions/) используются, чтобы создать подписанный URL-адрес (SAS) для каждой записи и отправить запрос HTTP POST для запуска транскрибирования. Кроме того, с помощью Функций Azure создаются запросы на получение и удаление расшифровок через API пакетного транскрибирования.

Корпорация Майкрософт применяет перечисленные выше технологии для внутренних целей, в частности для пакетной обработки вызовов в службу обработки.

## <a name="real-time-transcription-for-call-center-data"></a>Транскрибирование данных центра обработки вызовов в реальном времени

Если ваш режим использования требует обработки в реальном времени, мы рекомендуем использовать [Пакет SDK "Речь"](speech-sdk.md). Примеры для каждого языка вы найдете на сайте [GitHub](https://github.com/Azure-Samples/cognitive-services-speech-sdk). Последние новости и обновления см. в [заметках о выпуске](releasenotes.md).

## <a name="a-word-on-ivrs"></a>Системы интерактивного речевого ответа

Служба речевой речи может быть легко интегрирована в любое решение, используя либо [Speech SDK,](speech-sdk.md) либо [REST API.](rest-apis.md) Но для транскрибирования вызовов в центр обработки могут потребоваться дополнительные технологии. Обычно система интерактивного речевого ответа должна быть подключена к Azure. Хотя мы не предлагаем такие компоненты, вот описание того, что связь с IVR влечет за собой.

Некоторые продукты IVR или телефонии (например, Genesys или AudioCodes) предлагают возможности интеграции, которые можно использовать для включения всистемы и исходящего аудио-прохода к службе Azure. В принципе, пользовательский сервис Azure может предоставить определенный интерфейс для определения сеансов телефонного звонка (например, Call Start или Call End) и предоставить API WebSocket для получения входящего звука, который используется с помощью службы speech. Исходящие ответы, например транскрибирование бесед или подключения к Bot Framework, можно синтезировать с помощью службы преобразования текста в речь от Майкрософт и отправить в систему интерактивного речевого ответа для воспроизведения.

Другим сценарием является прямая интеграция с Протоколом инициации сессии (SIP). Служба Azure подключается к серверу SIP, чтобы получить входящий и исходящий потоки, которые используются на этапах преобразования речи в текст и текста в речь. Подключиться к серверу SIP можно с помощью нескольких коммерческих программных предложений, например Ozeki SDK или [API конференц-связи и собраний Teams](/graph/api/resources/communications-api-overview) (сейчас предоставляется в бета-версии). Все они поддерживают этот сценарий для голосовых вызовов.
