# <a name="custom-commands-concepts-and-definitions"></a>Основные понятия и определения пользовательских команд

Эта статья служит ссылкой на основные понятия и определения для приложений с пользовательскими командами.

## <a name="commands-configuration"></a>Настройка команд
Команды — это основные стандартные блоки приложения пользовательских команд. Команда — это набор конфигураций, необходимых для выполнения определенной задачи, определенной пользователем.

### <a name="example-sentences"></a>Примеры предложений
Например, фразы продолжительностью — это примеры набора, которые пользователь может сказать для активации определенной команды. Обратите внимание, что необходимо указать только пример фразы продолжительностью, а не исчерпывающий список. 

### <a name="parameters"></a>Параметры
Параметры — это сведения, необходимые командам для выполнения задачи. В сложных сценариях параметры также можно использовать для определения условий, которые активируют настраиваемые действия.

### <a name="completion-rules"></a>Правила завершения
Ряд правил, которые будут выполнены после готовности команды к выполнению, т. е. когда удовлетворяются все условия правил.

### <a name="interaction-rules"></a>Правила взаимодействия
Дополнительные правила для решения более конкретных или сложных ситуаций. Можно добавить дополнительные проверки или настроить дополнительные функции, такие как подтверждения или исправление с одним шагом. Вы также можете создать собственные правила взаимодействия.

## <a name="parameters-configuration"></a>Конфигурация параметров

Параметры — это сведения, необходимые командам для выполнения задачи. В сложных сценариях параметры также можно использовать для определения условий, которые активируют настраиваемые действия.

### <a name="name"></a>Имя
Параметр определяется свойством Name. Необходимо всегда давать описательное имя для параметра. На параметр можно ссылаться в разных разделах, например при создании условий, речевых ответах или других действиях.
 
### <a name="isglobal"></a>IsGlobal
Флажок, указывающий, является ли область этого параметра общей для всех команд в приложении. Если параметр является глобальным, его значение потенциально может быть предоставлено из любой области команды, а после назначения значения можно ссылаться из любой команды. 

### <a name="required"></a>Обязательно
Флажок, указывающий, требуется ли значение этого параметра для выполнения или завершения команды. Необходимо настроить ответы для предоставления пользователю запроса на ввод значения, если параметр помечен как обязательный.

### <a name="type"></a>Тип
Пользовательские команды поддерживают следующие типы параметров.


* Дата и время
* Geography
* Number
* Строка

Все эти типы параметров поддерживают конфигурацию значений по умолчанию, которую можно настроить на портале.

### <a name="configuration"></a>Параметр Configuration
Конфигурация — это свойство параметра, определенное только для типа: String. Ниже приведены поддерживаемые значения.
* Отсутствуют
* Принимать полные входные данные. Если включить этот параметр, параметр принимает любые входные utterance. Это полезно, когда пользователю требуется параметр с полным utterance. Например, почтовые адреса.
* Принимать предопределенные входные значения из внешнего каталога: используется для настройки параметра, который может рассчитывать на широкий спектр значений. Например, каталог продаж. В этом случае каталог размещается во внешней конечной точке веб-узла и может быть настроен независимо.
* Принимать предопределенные входные значения из внутреннего каталога: используется для настройки параметра, который может принимать несколько значений. В этом случае значения должны быть настроены в речевом студии.


### <a name="validation"></a>Проверка
Проверки — это конструкции, применимые к определенным типам параметров, которые позволяют настроить ограничения для значения параметра. В настоящее время пользовательские команды поддерживают проверки следующих типов параметров:
* Дата и время
* Number

## <a name="rules-configuration"></a>Конфигурация правил
Правило в пользовательских командах определяется набором **условий**, при выполнении которого выполняется набор **действий**. Правило также позволяет настроить свое **состояние выполнения после** и **ожидания** для следующей очереди.

### <a name="types"></a>Типы
Пользовательские команды поддерживают следующие категории правил:
* Правила завершения

    Список правил, выполняемых при выполнении команды. Все правила, настроенные в этом разделе, для которых выполняются условия, будут выполнены.
    
* Правила взаимодействия

    Правила взаимодействия можно использовать для настройки дополнительных пользовательских проверок, подтверждений, одношагового исправления или для выполнения любой другой пользовательской логики диалогового окна. Эти правила оцениваются при каждой обработке и могут использоваться для активации правил завершения.

Различные действия, настроенные как часть правила, выполняются в том порядке, в котором они отображаются на портале разработки.
    
### <a name="conditions"></a>Условия
Набор условий, которые должны быть выполнены для выполнения правила. Условие правила может иметь следующие типы:
* Значение параметра равно: значение настроенного параметра равно указанному значению.
* Нет значения параметра: настроенные параметры не должны иметь никаких значений.
* Обязательные параметры: настроенный параметр имеет значение.
* Все обязательные параметры: все параметры, которые были помечены как обязательные, имеют значение.
* Обновленные параметры: одно или несколько значений параметров были обновлены в результате обработки текущих входных данных (utterance/Activity).
* Подтверждение выполнено успешно: было успешно подтверждено входное utterance/действие (да).
* Подтверждение отклонено: входные данные utterance/Activity были успешно подтверждены (нет).
* Предыдущая команда должна быть обновлена: это условие используется в экземплярах, если необходимо перехватить подтверждение с отрицанием и обновление. В фоновом режиме это настраивается, когда обработчик диалогов обнаруживает отрицательное подтверждение, где цель совпадает с предыдущей, и пользователь ответил на обновление.

### <a name="actions"></a>Действия
* Отправка речевого ответа: отправка речевого ответа обратно клиенту.
* Обновление значения параметра: Обновление значения параметра команды до указанного значения.
* Очистить значение параметра очистить значение параметра команды.
* Вызвать конечную веб-точку: вызов конечной веб-точки.
* Отправить действие клиенту: отправка настраиваемого действия клиенту.

### <a name="expectations"></a>Превосходящи
Ожидания используются для настройки подсказок для обработки следующего пользовательского ввода. Поддерживаются следующие типы:
* Ожидается подтверждение от пользователя: указывает, что приложение ожидает подтверждения (да/нет) для следующего ввода данных пользователем.
* Ожидается ввод параметров от пользователя: укажите один или несколько параметров команды, которые приложение ожидает от введенных пользователем данных.

### <a name="post-execution-state"></a>Состояние после выполнения
Состояние диалогового окна после обработки текущего входа (utterance/Activity). Он имеет следующие типы:
* Команда выполнена: выполните команду, и никакие дополнительные правила команды не будут обработаны.
* Правила завершения выполнения: выполнение всех допустимых правил завершения.
* Ожидание входных данных пользователя: ожидание ввода следующего пользователя.
