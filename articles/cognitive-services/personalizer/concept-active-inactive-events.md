# <a name="active-and-inactive-events"></a>Активные и неактивные события

**Активным** событием является любой звонок в ранг, где вы знаете, что собираетесь показать результат клиенту и определить награду баллов. Это поведение по умолчанию.

**Неактивное** событие — это вызов в ранг, где вы не уверены, увидит ли пользователь когда-либо рекомендуемое действие из-за бизнес-логики. Это позволяет отказаться от события, чтобы Personalizer не обучен с вознаграждением по умолчанию. Неактивные события не должны вызывать API Вознаграждения.

Важно, чтобы цикл обучения знал фактический тип события. Неактивное событие не будет иметь вызова Reward. Активное событие должно иметь вызов Reward, но если вызов API не выполнен, применяется оценка вознаграждения по умолчанию. Измените статус события с неактивного на активное, как только вы знаете, что это повлияет на пользовательский опыт.

## <a name="typical-active-events-scenario"></a>Типичный сценарий активных событий

Когда приложение вызывает API ранга, вы получаете действие, которое приложение должно отображать в поле **rewardActionId.**  С этого момента Personalizer ожидает наградный звонок с наградным баллом, который имеет тот же eventId. Оценка вознаграждения используется для обучения модели для будущих вызовов ранга. Если для eventId не получен вызов Reward, применяется вознаграждение по умолчанию. [Вознаграждения по умолчанию](how-to-settings.md#configure-rewards-for-the-feedback-loop) устанавливаются на ресурсе Personalizer на портале Azure.

## <a name="other-event-type-scenarios"></a>Другие сценарии типа событий

В некоторых сценариях приложению может потребоваться вызвать Rank, прежде чем он даже узнает, будет ли результат использован или отображен пользователю. Это может произойти в ситуациях, когда, например, рендеринг рекламируемого контента на странице перезаписывается маркетинговой кампанией. Если результат вызова Rank никогда не использовался и пользователь никогда его не видел, не отправляйте соответствующий вызов Reward.

Как правило, эти сценарии происходят, когда:

* Вы prerendering пользовательского интерфейса, что пользователь может или не может получить, чтобы увидеть.
* Приложение выполняет прогностический персонализацию, при которой вызовы ранга выполняются с небольшим контекстом в реальном времени, и приложение может или не может использовать вывод.

В этих случаях используйте Personalizer для вызова rank, запрашивая _неактивность_события. Personalizer не ожидает вознаграждения за это событие, и он не будет применяться по умолчанию вознаграждение.

Позже в вашей бизнес-логике, если приложение использует информацию из вызова rank, просто _активируйте_ событие. Как только мероприятие активируется, Personalizer ожидает награду за событие. Если в API Reward не делается явного вызова, Personalizer применяет вознаграждение по умолчанию.

## <a name="inactive-events"></a>Неактивные события

Чтобы отключить обучение к событию, позвоните в rank с помощью `learningEnabled = False`.

Для неактивного события обучение неявно активируется, если вы отправляете `activate` вознаграждение за eventId или звоните в API для этого eventId.
