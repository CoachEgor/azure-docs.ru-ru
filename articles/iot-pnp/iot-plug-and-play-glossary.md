# <a name="glossary-of-terms-for-iot-plug-and-play-preview"></a>Глоссарий терминов для IoT Plug и Play Preview

Определения общих терминов, используемых в статьях IoT Plug and Play.

## <a name="azure-certified-for-iot-portal"></a>Azure сертифицирован для портала IoT

Вы можете использовать сертифицированный для портала IoT веб-сайт [Azure](https://aka.ms/ACFI) для:

- Завершите [процесс сертификации](#device-certification) для вашего [устройства IoT Plug and Play.](#iot-plug-and-play-device)
- Найдите [модели возможностей устройств.](#device-capability-model)
- Опубликовать модель возможностей устройства в [хранилище общедоступной модели.](#public-model-repository)

## <a name="azure-cli"></a>Azure CLI

Azure CLI — это кросс-платформенный командный инструмент для управления ресурсами Azure. Расширение Azure IoT для Azure CLI является командным инструментом для взаимодействия и тестирования [устройств IoT Plug and Play.](#iot-plug-and-play-device) Расширение можно использовать для:

- Подключитесь к устройству IoT Plug and Play.
- Просмотр [телеметрии,](#telemetry) отправленной устройством.
- Работайте со [свойствами](#properties)устройств.
- [Команды](#commands)устройства вызова.
- Управление [репозиториями моделей,](#model-repository) [интерфейсами](#interface)и [моделями возможностей устройств.](#device-capability-model)

## <a name="azure-iot-central"></a>Azure IoT Central

Azure IoT Central — это полностью управляемое программное обеспечение как сервисное решение, которое упрощает подключение, мониторинг и управление [устройствами IoT Plug и Play.](#iot-plug-and-play-device) Модели [возможностей устройств](#device-capability-model) можно использовать для автоматической настройки центрального приложения IoT для мониторинга и управления устройствами.

## <a name="azure-iot-certification-service"></a>Сервис сертификации Azure IoT

Служба сертификации Azure IoT проводит набор сертификационных тестов при отправке [устройства IoT Plug and Play](#iot-plug-and-play-device) для сертификации через портал [Azure Certified for IoT.](#azure-certified-for-iot-portal) Прежде чем вы сможете добавить устройство в [каталог сертифицированных устройств IoT,](#certified-for-iot-device-catalog)устройство должно быть сертифицировано.

## <a name="azure-iot-tools-extension"></a>Расширение инструментов Azure IoT

Azure IoT Tools — это набор расширений в [коде Visual Studio,](#visual-studio-code) которые помогают взаимодействовать с Концентратором IoT и разрабатывать устройства IoT. Для разработки устройств IoT Plug и Play, это поможет вам:

- Автор [моделей возможностей устройств](#device-capability-model) и [интерфейсов.](#interface)
- Публикация для [моделирования репозиториев](#model-repository).
- Создание кода скелета для реализации приложения устройства.

## <a name="azure-iot-explorer-tool"></a>Инструмент исследователя Azure IoT

Исследователь Azure IoT — это графический инструмент, с которым можно взаимодействовать и тестировать [устройства IoT Plug and Play.](#iot-plug-and-play-device) После установки инструмента на локальной машине, вы можете использовать его для:

- Просмотр устройств, подключенных к [концентратору IoT.](#azure-iot-hub)
- Подключитесь к устройству IoT Plug and Play.
- Просмотр [телеметрии,](#telemetry) отправленной устройством.
- Работайте со [свойствами](#properties)устройств.
- [Команды](#commands)устройства вызова.

## <a name="azure-iot-hub"></a>Центр Интернета вещей Azure

Центр Интернета вещей — размещенная в облаке управляемая служба, которая действует в качестве центра сообщений для двусторонней связи между приложением Интернета вещей и устройствами, которыми оно управляет. [Устройства IoT Plug and Play](#iot-plug-and-play-device) могут подключаться к концентратору IoT. Решение IoT использует концентратор IoT для включения:

- Устройства для отправки телеметрии в облачное решение.
- Облачное решение для управления подключенными устройствами.

## <a name="azure-iot-device-sdk"></a>Пакет SDK для устройств Azure IoT

Есть устройства SDK для нескольких языков, которые можно использовать для создания IoT Plug и воспроизведения клиентских приложений устройства. Одно из требований [для сертификации устройства](#device-certification) заключается в том, что клиентский код устройства использует один из SDK-устройства Azure IoT.

## <a name="certified-for-iot-device-catalog"></a>Сертифицирован для каталога устройств IoT

Сертифицированный [каталог устройств IoT](https://catalog.azureiotsolutions.com/) перечисляет [устройства IoT Plug и Play,](#iot-plug-and-play-device) прошедшие [сертификационные](#device-certification) тесты устройства. Модели [возможностей устройств](#device-capability-model) для устройств IoT Plug и Play в каталоге и опубликованы в общедоступном репозитории моделей.

## <a name="commands"></a>Команды

Команды, определяемые в [интерфейсе,](#interface) представляют методы, которые могут быть выполнены на [цифровом twin.](#digital-twin) Например, команда для перезагрузки устройства.

## <a name="common-interface"></a>Общий интерфейс

Все [устройства IoT Plug и Play,](#iot-plug-and-play-device) как ожидается, реализуют некоторые общие [интерфейсы.](#interface) Например, информационный интерфейс устройства определяет аппаратную и операционную систему информацию об устройстве. [Сертификация устройств](#device-certification) требует от устройства реализации нескольких общих интерфейсов. Можно получить общие определения интерфейса из общедоступного репозитория моделей.

## <a name="company-model-repository"></a>Репозиторий модели компании

Организация может использовать [репозиторий модели](#model-repository) компании в качестве частного магазина для [моделей и](#device-capability-model) [интерфейсов](#interface)устройств.

## <a name="connection-string"></a>Строка подключения

Строка соединения инкапсулирует информацию, необходимую для подключения к конечной точке. Строка подключения обычно содержит адрес конечной точки и сведения о безопасности, но ее формат отличается в зависимости от службы. Существует два типа строки подключения, связанной со службой Центра Интернета вещей:

- Строки подключения устройств [позволяют IoT Plug и Play устройствам](#iot-plug-and-play-device) подключаться к конечным точкам устройства на концентраторе IoT. Клиентский код на устройстве использует строку соединения для установления безопасного соединения с концентратором IoT.
- Строки подключения Концентратора IoT позволяют готовым решениям и инструментам безопасно подключаться к конечным точкам обслуживания в концентраторе IoT. Эти решения и инструменты управляют концентратором IoT и подключенными к нему устройствами.
- Строки репозитория модели компании позволяют готовым решениям и инструментам безопасно подключаться к [репозиторию модели компании.](#company-model-repository) Эти решения и инструменты потребляют или управляют [моделями и](#device-capability-model) [интерфейсами](#interface) возможностей устройства в репозитории.

## <a name="device-capability-model"></a>Модель возможностей устройства

Модель возможностей устройства описывает [ioT Plug and Play устройство](#iot-plug-and-play-device) и определяет набор [интерфейсов, реализованных](#interface) устройством. Модель возможностей устройства обычно соответствует физическому устройству, продукту или SKU. Для определения модели возможностей устройства используется [язык определения див.](#digital-twin-definition-language)

## <a name="device-certification"></a>Сертификация устройств

Сертификация устройств — это процесс сертификации [устройства IoT Plug and Play,](#iot-plug-and-play-device) чтобы его можно было добавить в [сертифицированный каталог устройств IoT,](#certified-for-iot-device-catalog) [а также модель возможностей и](#device-capability-model) интерфейсов устройств, [добавленных](#interface) в [общедоступное репозиторий модели.](#public-model-repository)

## <a name="device-developer"></a>Разработчик устройств

Разработчик устройства использует [модель возможностей устройства,](#device-capability-model) [интерфейсы](#interface)и [устройство SDK Azure IoT](#azure-iot-device-sdk) для реализации кода для запуска на [устройстве IoT Plug and Play.](#iot-plug-and-play-device)

## <a name="device-modeling"></a>Моделирование устройств

[Разработчик устройства](#device-developer) использует язык [определения див-близнецов](#digital-twin-definition-language) для моделирования возможностей устройства [IoT Plug and Play.](#iot-plug-and-play-device) Модель может быть совместно йога с помощью репозитория модели. Разработчик устройства может генерировать код устройства скелета из модели. [Разработчик решений](#solution-developer) может настроить решение IoT из модели.

## <a name="device-provisioning-service"></a>Служба подготовки устройств

[Azure IoT Central](#azure-iot-central) использует службу обеспечения устройств для управления регистрацией и подключением всех устройств. Для получения дополнительной информации см. [подключение устройств в Azure IoT Central](../iot-central/core/concepts-get-connected.md). Вы также можете использовать службу обеспечения устройств для управления регистрацией устройств и подключением к вашему ioT-решению на основе IoT на основе IoT. Для получения дополнительной [Provisioning devices with Azure IoT Hub Device Provisioning Service](../iot-dps/about-iot-dps.md)информации см.

## <a name="device-registration"></a>Регистрация устройства

Прежде чем [ioT Plug и Play устройство](#iot-plug-and-play-device) может подключиться к решению IoT, он должен быть зарегистрирован с решением. [Azure IoT Central](#azure-iot-central) использует [службу обеспечения устройств](#device-provisioning-service) для управления регистрацией устройств. В пользовательском решении IoT вы можете зарегистрировать устройства с помощью концентратора IoT на портале Azure или программно.

## <a name="device-first"></a>Устройство-первых

[Azure IoT Central](#azure-iot-central) поддерживает сценарий регистрации и подключения устройства. В этом случае [устройство IoT Plug and Play](#iot-plug-and-play-device) может подключаться к приложению IoT Central без предварительной регистрации. Регистрация происходит автоматически, когда устройство впервые подключается к приложению.

## <a name="digital-twin"></a>Цифровой близнец

Цифровой близнец является моделью [ioT Plug и play устройства.](#iot-plug-and-play-device) Цифровой близнец моделируется с использованием [цифрового языка определения близнецов.](#digital-twin-definition-language) Для взаимодействия с цифровыми близнецами во время выполнения можно использовать [SDK-устройства Azure IoT.](#azure-iot-device-sdk) Например, можно установить значение свойства в цифровом близнеце на устройстве, и SDK передает это изменение вашему решению IoT в облаке.

## <a name="digital-twin-change-events"></a>Цифровые события изменения близнецов

Когда [устройство IoT Plug and Play](#iot-plug-and-play-device) подключено к [концентратору IoT,](#azure-iot-hub)концентратор может использовать возможность их передачи для отправки уведомлений о цифровых двойных изменениях. Например, при изменении значения [свойства](#properties) на устройстве Концентратор IoT может отправить уведомление в конечную точку, такую как очередь Service Bus.

## <a name="digital-twin-definition-language"></a>Цифровой язык определения близнецов

Язык для описания моделей и интерфейсов для [устройств IoT Plug и Play.](#iot-plug-and-play-device) Используйте [digital Twin Definition Language](https://aka.ms/DTDL) для описания возможностей [цифрового близнеца](#digital-twin) и позволяет платформе IoT и решениям IoT использовать семантику сущности.

## <a name="digital-twin-route"></a>Цифровой двойной маршрут

Маршрут, созданный в [концентраторе IoT](#azure-iot-hub) для доставки [цифровых событий изменения близнецов](#digital-twin-change-events) в и конечные точки, такие как очередь Service Bus.

## <a name="interface"></a>Интерфейс

Интерфейс описывает связанные с ними возможности, реализованные [устройством IoT Plug and Play](#iot-plug-and-play-device) или [цифровым близнецом.](#digital-twin) Вы можете повторно использовать интерфейсы в различных [моделях возможностей устройства.](#device-capability-model)

## <a name="iot-hub-query-language"></a>Язык запросов Центра Интернета вещей

Язык запросов Концентратора IoT используется для нескольких целей. Например, язык можно использовать для поиска [устройств, зарегистрированных](#device-registration) в концентраторе IoT, или для уточнения [поведения цифровой двойной дивной сети.](#digital-twin-route)

## <a name="iot-plug-and-play-device"></a>IoT подключить и воспроизводить устройство

IoT Plug and Play устройство, как правило, является мелкомасштабным, автономным вычислительным устройством, которое собирает данные или управляет другими устройствами, и которое запускает программное обеспечение или прошивку, которая реализует [модель возможностей устройства.](#device-capability-model)  Например, устройство IoT Plug and Play может быть устройством мониторинга окружающей среды или контроллером для системы орошения смарт-сельского хозяйства. Вы можете написать облачное IoT-решение для управления, управления и получения данных с устройств IoT Plug и Play. В [каталоге устройств Azure, сертифицированных для IoT,](#certified-for-iot-device-catalog) перечислены доступные устройства IoT Plug и Play. Каждое устройство IoT Plug and Play в каталоге было проверено и имеет [модель возможностей устройства.](#device-capability-model)

## <a name="microsoft-partner-center"></a>Центр партнеров Майкрософт

[Партнерский центр Майкрософт](https://docs.microsoft.com/partner-center/) — это место, где ваша организация управляет своими сквозными отношениями с корпорацией Майкрософт. Вам нужна учетная запись Microsoft Partner Center, прежде чем вы сможете сертифицировать устройство [IoT Plug and Play](#iot-plug-and-play-device) [на портале Azure Certified for IoT.](#azure-certified-for-iot-portal)

## <a name="model-discovery"></a>Обнаружение модели

Когда [устройство IoT Plug and Play](#iot-plug-and-play-device) подключается к решению IoT, решение может обнаружить возможности устройства, найдя [модель возможностей устройства.](#device-capability-model) Устройство может отправить свою модель возможностей в решение, или решение может найти модель возможностей устройства в [репозитории модели.](#model-repository)

## <a name="model-repository"></a>Репозиторий модели

Репозиторий модели хранит модели и [интерфейсы](#interface) [устройств.](#device-capability-model) Есть единое [репозиторий общедоступной модели.](#public-model-repository) Организации могут создавать свои собственные хранилища организационных моделей.

## <a name="model-repository-rest-api"></a>Репозиторий модели REST API

API для управления и взаимодействия с репозиториями моделей. Например, можно использовать API для [добавления моделей возможностей устройств](#device-capability-model) и поиска моделей возможностей.

## <a name="properties"></a>Свойства

Свойства являются полями данных, определенными в [интерфейсе,](#interface) которые представляют некоторое состояние цифрового близнеца. Вы можете объявить свойства как только для чтения или writable. Свойства только для чтения, такие как серийный номер, устанавливаются кодом, работающим на самом [устройстве IoT Plug and Play.](#iot-plug-and-play-device)  Развражаемые свойства, такие как порог тревоги, обычно устанавливаются из облачного решения IoT.

## <a name="public-model-repository"></a>Репозиторий общедоступных моделей

Существует единый публичный репозиторий моделей, который хранит [модели возможностей устройств](#device-capability-model) и [интерфейсы](#interface) для [сертифицированных устройств.](#device-certification) В общедоступном репозитории модели также хранятся общие определения [интерфейса.](#common-interface)

## <a name="registration-id"></a>Идентификатор регистрации

Идентификатор регистрации однозначно идентифицирует устройство в [службе обеспечения устройств.](#device-provisioning-service) Этот идентификатор не совпадает с идентификатором устройства, который является уникальным идентификатором для устройства в [концентраторе IoT.](#azure-iot-hub)

## <a name="scope-id"></a>Идентификатор области

Область идентификатора scope id однозначно идентифицирует экземпляр [службы обеспечения устройств.](#device-provisioning-service)

## <a name="shared-access-signature"></a>Подписанный URL-адрес

Подписанные URL-адреса представляют собой механизм аутентификации на базе алгоритма безопасного хэширования SHA-256 или универсального кода ресурса. Общая проверка подлинности подписи доступа имеет два компонента: общую политику доступа и общую подпись доступа (часто называемую маркером). [Устройство IoT Plug and Play](#iot-plug-and-play-device) использует общую подпись доступа для проверки подлинности с помощью [концентратора IoT.](#azure-iot-hub)

## <a name="solution-developer"></a>Разработчик решений

Разработчик решений создает задний конец решения. Разработчик решений обычно работает с ресурсами Azure, такими как [IoT-концентратор](#azure-iot-hub) и [репозитории моделей,](#model-repository)или работает с [IoT Central.](#azure-iot-central)

## <a name="telemetry"></a>Телеметрия

Телеметрические поля, определенные в [интерфейсе,](#interface) представляют измерения. Эти измерения, как правило, значения, такие как показания датчиков, которые отправляются [IoT Plug и Play устройства](#iot-plug-and-play-device) в качестве потока данных.

## <a name="visual-studio-code"></a>Визуальный код студии

Визуальный код Studio — это современный редактор кода, доступный для нескольких платформ. Расширения, например в пакете [инструментов Azure IoT Tools,](https://marketplace.visualstudio.com/items?itemName=vsciot-vscode.azure-iot-tools) позволяют настроить редактор для поддержки широкого спектра сценариев разработки.
