# <a name="virtual-network-peering"></a>Пиринг между виртуальными сетями

Виртуальная сеть пиринга позволяет легко подключить сети в [Azure Virtual Network.](virtual-networks-overview.md) Виртуальные сети отображаются как одноразовые для целей подключения. Трафик между виртуальными машинами использует основную инфраструктуру Microsoft. Как и трафик между виртуальными машинами в одной сети, трафик направляется только через *частную* сеть Корпорации Майкрософт.

Azure поддерживает следующие типы пиринга:

* Виртуальная сеть пиринга: Подключение виртуальных сетей в том же регионе Azure.
* Глобальная виртуальная сеть пиринга: Подключение виртуальных сетей в регионах Azure.

Преимущества пиринговой связи между виртуальными сетями (как локальной, так и глобальной):

* подключение между ресурсами в разных виртуальных сетях, характеризующееся низкой задержкой и высокой пропускной способностью;
* Возможность для ресурсов в одной виртуальной сети общаться с ресурсами в другой виртуальной сети.
* Возможность передачи данных между виртуальными сетями в подписках Azure, наниматели активных каталогов Azure, модели развертывания и в регионах Azure.
* Возможность наглядных виртуальных сетей, созданных с помощью менеджера ресурсов Azure.
* Возможность пристычовать виртуальную сеть, созданную с помощью ресурсного менеджера, в одну, созданную с помощью классической модели развертывания. Дополнительные сведения о моделях развертывания Azure см. в статье [Развертывание с помощью Azure Resource Manager и классическое развертывание: сведения о моделях развертывания и состоянии ресурсов](../azure-resource-manager/management/deployment-models.md?toc=%2fazure%2fvirtual-network%2ftoc.json).
* Отсутствие задержки при доступе к ресурсам в любой виртуальной сети при создании пиринга или после его создания.

Сетевой трафик между одноранговыми виртуальными сетями является закрытым. Трафик между виртуальными сетями хранится в магистральной сети Майкрософт. При обмене данными между виртуальными сетями не требуется общий доступ к Интернету, шлюзы или шифрование.

## <a name="connectivity"></a>Соединение

Для виртуальных сетей, внеедывающих внедорожные виртуальные сети, ресурсы в любой виртуальной сети могут напрямую связываться с ресурсами в заглянувой виртуальной сети.

Задержки в сети между виртуальными машинами в пиринговых виртуальных сетях такие же, как и в пределах одной виртуальной сети. Пропускная способность сети зависит от пропускной способности виртуальной машины, которая пропорциональна ее размеру. Дополнительные ограничения пропускной способности при пиринге не применяются.

Трафик между виртуальными машинами в пиринговых виртуальных сетях передается напрямую через магистральную инфраструктуру Майкрософт, а не через шлюз или Интернет.

Вы можете применять группы сетевой безопасности в виртуальной сети, чтобы заблокировать доступ к другим виртуальным сетям или подсетям.
При настройке виртуальной сети пиринга, либо открыть или закрыть правила группы безопасности сети между виртуальными сетями. Если вы открываете полное подключение между виртуальными сетями, вы можете применить группы сетевой безопасности для блокировки или отказа в определенном доступе. Полное подключение является опцией по умолчанию. Чтобы узнать больше о [Security groups](security-overview.md)группах сетевой безопасности, см.

## <a name="service-chaining"></a>Цепочка служб

Цепочка обслуживания позволяет направлять трафик из одной виртуальной сети в виртуальный прибор или шлюз в заглянующей сети через маршруты, определяемые пользователями.

Для обеспечения цепочки служб настройте маршруты, определяемые пользователем, которые указывают на виртуальные машины в виртуальных сетях в качестве *следующего* IP-адреса. Маршруты, определяемые пользователями, могут также указывать на виртуальные сетевые шлюзы для обеспечения цепочки служб.

Вы можете развернуть концентрированные сети, где виртуальная сеть концентратора размещает компоненты инфраструктуры, такие как сетевой виртуальный прибор или VPN шлюз. Все остальные виртуальные сети ("лучи") могут подключаться по пиринговой связи к центральной виртуальной сети. Трафик течет через сетевые виртуальные приборы или VPN шлюзы в концентраторе виртуальной сети.

В определяемой пользователем таблице маршрутизации можно указать для следующего прыжка IP-адрес виртуальной машины, входящей в пиринговую виртуальную сеть, или VPN-шлюз. Нельзя маршрутизацить между виртуальными сетями с определенным маршрутом, который определяет шлюз Azure ExpressRoute как следующий тип перехода. Дополнительные сведения об определяемых пользователем маршрутах см. в статье [Определяемые пользователем маршруты и IP-пересылка](virtual-networks-udr-overview.md#user-defined). Чтобы узнать, как создать концентратор и говорить топологии сети, [см.](/azure/architecture/reference-architectures/hybrid-networking/hub-spoke?toc=%2fazure%2fvirtual-network%2ftoc.json)

## <a name="gateways-and-on-premises-connectivity"></a>Использование шлюзов для локального подключения

Каждая виртуальная сеть, включая заглянул виртуальной сети, может иметь свой собственный шлюз. Виртуальная сеть может использовать свой шлюз для подключения к предварительной сети. Вы также можете настроить [виртуальные сетевые сетевые соединения](../vpn-gateway/vpn-gateway-vnet-vnet-rm-ps.md?toc=%2fazure%2fvirtual-network%2ftoc.json) с помощью шлюзов, даже для виртуальных сетей.

При настройке обоих вариантов виртуальной сетевой взаимосвязанности трафик между виртуальными сетями проходит через конфигурацию пиринга. Трафик использует магистраль Azure.

Вы также можете настроить шлюз в заглянувиртуальной виртуальной сети в качестве транзитной точки для предварительной сети. В этом случае виртуальная сеть, используюая удаленный шлюз, не может иметь свой собственный шлюз. Виртуальная сеть имеет только один шлюз. Шлюз является либо локальным, либо удаленным шлюзом в виртуальной сети, как показано на следующей диаграмме:

![Транзит в пиринговой виртуальной сети](./media/virtual-networks-peering-overview/local-or-remote-gateway-in-peered-virual-network.png)

Как виртуальная сеть пиринга и глобальной виртуальной сети пиринг поддержки шлюза транзита.

Поддерживается транзит шлюза между виртуальными сетями, созданными с помощью различных моделей развертывания. Шлюз должен находиться в виртуальной сети в модели Resource Manager. Чтобы узнать больше об использовании шлюза для транзита, [см. Накончать VPN шлюз для транзита в виртуальной сети пиринг](../vpn-gateway/vpn-gateway-peering-gateway-transit.md?toc=%2fazure%2fvirtual-network%2ftoc.json).

Когда вы просматриваете виртуальные сети, которые разделяют одно соединение Azure ExpressRoute, трафик между ними проходит через пиринговые отношения. Этот трафик использует основную сеть Azure. Вы по-прежнему можете использовать в каждой виртуальной сети локальные шлюзы для подключения к локальному каналу. В противном случае можно использовать общий шлюз и настроить транзит для подключения на месте.

## <a name="troubleshoot"></a>Устранение неполадок

Чтобы подтвердить, что виртуальные сети заглянул, вы можете проверить эффективные маршруты. Проверьте маршруты для сетевого интерфейса в любой подсети в виртуальной сети. Если пиринг между виртуальными сетями настроен, все подсети виртуальной сети будут иметь маршруты со следующим прыжком типа *Пиринг виртуальных сетей* для каждого адресного пространства в каждой пиринговой виртуальной сети. Для получения дополнительной информации [см.](diagnose-network-routing-problem.md)

Вы также можете устранить неполадки подключения к виртуальной машине в загляну виртуальной сети с помощью Azure Network Watcher. Проверка подключения позволяет увидеть, как трафик направляется из сетевого интерфейса исходной виртуальной машины в сетевой интерфейс виртуальной машины назначения. Для получения дополнительной информации см. [Соединения Troubleshoot с помощью портала Azure.](../network-watcher/network-watcher-connectivity-portal.md#check-connectivity-to-a-virtual-machine)

Вы также можете попробовать [Troubleshoot виртуальной сети пиринг вопросов.](virtual-network-troubleshoot-peering-issues.md)

## <a name="constraints-for-peered-virtual-networks"></a>Ограничения для заглянул виртуальных сетей<a name="requirements-and-constraints"></a>

Когда создается глобальный пиринг виртуальных сетей, действуют следующие ограничения:

* Ресурсы в одной виртуальной сети не могут общаться с фронтовым IP-адресом Basic Internal Load Balancer (ILB) в глобально заглянувой виртуальной сети.
* Некоторые службы, которые используют балансиватор основной нагрузки, не работают над глобальным использоваванием виртуальной сети. Для получения дополнительной информации см. [Каковы ограничения, связанные с глобальными VNet Peering и балансеров нагрузки?](virtual-networks-faq.md#what-are-the-constraints-related-to-global-vnet-peering-and-load-balancers).

Дополнительные сведения см. в разделе [Требования и ограничения](virtual-network-manage-peering.md#requirements-and-constraints). Чтобы узнать больше об поддерживаемом количестве [Networking limits](../azure-resource-manager/management/azure-subscription-service-limits.md?toc=%2fazure%2fvirtual-network%2ftoc.json#azure-resource-manager-virtual-networking-limits)пирингов, см.

## <a name="pricing"></a>Цены

Существует номинальная плата за вход и выход трафика, который использует виртуальное подключение к пирингу сети. Для получения дополнительной информации, см [Виртуальная сеть ценообразования](https://azure.microsoft.com/pricing/details/virtual-network).

Gateway Transit — это пиринговое свойство, позволяющее виртуальной сети использовать шлюз VPN/ExpressRoute в виртуальной сети. Транзит шлюза работает как для перекрестных помещений, так и для подключения от сети к сети. Трафик на шлюз (вход или выход) в заглянувую виртуальную сеть взимает плату за вглядывание виртуальной сети на спице VNet (или не-шлюз VNet). Для получения дополнительной информации, см [VPN Gateway ценообразования](https://azure.microsoft.com/pricing/details/vpn-gateway/) для VPN шлюз сборов и ExpressRoute Gateway цены для ExpressRoute шлюз сборов.

>[!NOTE]
> В предыдущей версии этого документа говорилось, что плата за пиринг виртуальной сети не будет применяться на спицкой VNet (или не шлюз VNet) с Gateway Transit. Теперь он отражает точные цены на страницу ценообразования.
