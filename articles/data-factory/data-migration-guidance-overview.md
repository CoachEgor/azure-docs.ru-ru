# <a name="use-azure-data-factory-to-migrate-data-from-your-data-lake-or-data-warehouse-to-azure"></a>Используйте фабрику данных Azure для переноса данных из озера данных или хранилища данных в Azure

Если вы хотите перенести хранилище данных или корпоративные данные (EDW) в Microsoft Azure, подумайте об использовании Azure Data Factory. Фабрика данных Azure хорошо подходит для следующих сценариев:

- Миграция рабочей нагрузки больших данных из службы простого хранения Amazon (Amazon S3) или в локте Hadoop Distributed File System (HDFS) в Azure
- Миграция EDW из Oracle Exadata, Netezza, Teradata или Amazon Redshift в Azure

Фабрика данных Azure может перемещать петабайты (PB) данных для миграции озер данных и десятки терабайт (ТБ) данных для миграции хранилища данных.

## <a name="why-azure-data-factory-can-be-used-for-data-migration"></a>Почему фабрика данных Azure может использоваться для миграции данных

- Фабрика данных Azure может легко увеличить объем вычислительной мощности для перемещения данных без сервера с высокой производительностью, устойчивостью и масштабируемостью. И вы платите только за то, что вы используете. Имейте в виду следующее. 
  - Фабрика данных Azure не имеет ограничений ни по объему данных, ни по количеству файлов.
  - Фабрика данных Azure может в полной мере использовать пропускную способность сети и хранилища для достижения максимального объема пропускной способности передачи данных в вашей среде.
  - Azure Data Factory использует метод оплаты по мере использования, так что вы платите только за время, которое вы фактически используете для выполнения миграции данных в Azure.  
- Фабрика данных Azure может выполнять как одноразовую историческую нагрузку, так и запланированные дополнительные нагрузки.
- Azure Data Factory использует время выполнения интеграции Azure (IR) для перемещения данных между общедоступными конечными точками данных и конечными точками склада. Он также может использовать самостоятельно размещенные ИК для перемещения данных для данных озера и складских конечных точек внутри Azure Virtual Network (VNet) или за брандмауэром.
- Фабрика данных Azure имеет корпоративную безопасность: вы можете использовать Windows Installer (MSI) или идентификацию службы для безопасной интеграции службы в службу или использовать Azure Key Vault для управления учетными данными.
- Azure Data Factory предоставляет возможность авторинга без кода и богатую встроенную панель мониторинга.  

## <a name="online-vs-offline-data-migration"></a>Миграция данных онлайн и оффлайн

Azure Data Factory является стандартным инструментом онлайн-миграции данных для передачи данных через сеть (Интернет, ER или VPN). В то время как при переносной миграции данных пользователи физически отправляют устройства передачи данных из организации в Центр обработки данных Azure.  

При выборе подхода между онлайн-и автономным подходом к миграции существует три ключевых фактора:  

- Размер данных, которые будут мигрировать
- пропускная способность сети;
- Окно миграции

Например, предположим, что вы планируете использовать фабрику данных Azure для завершения миграции данных в течение двух недель *(окно миграции).* Обратите внимание на линию розового/синего разреза в следующей таблице. Самая низкая розовая ячейка для любого данного столбца показывает сопряжение полосы пропускания данных/сети, миграционное окно которого ближе всего к, но менее чем за две недели. (Любой размер/пропускная способность сопряжения в синей ячейке имеет окно миграции онлайн более двух недель.) 

![онлайн по сравнению с оффлайн](media/data-migration-guidance-overview/online-offline.png) Эта таблица поможет вам определить, можно ли удовлетворить предполагаемое окно миграции через онлайн-миграцию (Azure Data Factory) на основе размера данных и доступной пропускной способности сети. Если окно онлайн-миграции превышает две недели, необходимо использовать офлайн-миграцию.

> [!NOTE]
> Используя миграцию в Интернете, вы можете достичь как исторической загрузки данных, так и дополнительных каналов сквозной с помощью одного инструмента.  Благодаря этому подходу ваши данные могут быть синхронизированы между существующим хранилищем и новым хранилищем в течение всего окна миграции. Это означает, что вы можете восстановить логику ETL в новом хранилище с обновленными данными.
