# <a name="plan-an-automatic-user-provisioning-deployment"></a>Планирование развертывания автоматической подготовки пользователей

Многие организации полагаются на программное обеспечение в качестве сервисных приложений (SaaS), таких как ServiceNow, Sscaler и Slack для повышения производительности конечных пользователей. Исторически ИТ-персонал полагался на методы ручного подготовки, такие как загрузка файлов CSV или использование пользовательских скриптов для безопасного управления идентификаторами пользователей в каждом приложении SaaS. Эти процессы подвержены ошибкам, небезопасным и трудно управляемым.

Автоматическое предоставление пользовательского каталога Azure Active Directory (Azure AD) упрощает этот процесс, надежно автоматизируя создание, техническое обслуживание и удаление идентификаторов пользователей в приложениях SaaS на основе бизнес-правил. Эта автоматизация позволяет эффективно масштабировать системы управления идентификацией как на облачных, так и на гибридных средах по мере расширения их зависимости от облачных решений.

[См. Автоматическое обеспечение и деирование приложений SaaS с помощью Active Directory Azure,](../app-provisioning/user-provisioning.md) чтобы лучше понять функциональность.

## <a name="learn"></a>Обучение

Подготовка пользователей создает основу для постоянного управления идентификацией и повышает качество бизнес-процессов, опирающихся на авторитетные данные идентификации.

### <a name="key-benefits"></a>Основные преимущества

Ключевыми преимуществами автоматического подготовки пользователей являются:

* **Повышение производительности.** Вы можете управлять идентификаторами пользователей в приложениях SaaS с помощью единого интерфейса управления подготовкой пользователей. Этот интерфейс имеет единый набор политик подготовки.

* **Управление риском**. Повысить безопасность можно, автоматизируя изменения на основе статуса сотрудника или членства в группе, определяющих роли и/или доступ.

* **Адрес соответствия и управления**. Azure AD поддерживает журналы аудита для каждого запроса на подготовку пользователя. Запросы выполняются как в исходных, так и в целевых системах. Это позволяет отслеживать, кто имеет доступ к приложениям с одного экрана.

* **Снижение затрат.** Автоматическая подготовка пользователей снижает затраты, избегая неэффективности и человеческих ошибок, связанных с ручной подготовкой. Это уменьшает потребность в специально разработанных решениях для подготовки пользователей, скриптов и журналов аудита.

### <a name="licensing"></a>Лицензирование

Azure AD обеспечивает интеграцию приложений самообслуживания с помощью шаблонов, предусмотренных в меню галереи приложений. Полный список лицензионных требований можно узнать на [странице лицензирования Azure AD.](https://azure.microsoft.com/pricing/details/active-directory/)

#### <a name="application-licensing"></a>Лицензирование приложений

Вам понадобятся соответствующие лицензии для приложения (ы), которые вы хотите автоматически предоставить. Обсудите с владельцами приложений, имеют ли пользователи, назначенные в приложение, соответствующие лицензии на их роли приложения. Если Azure AD управляет автоматической подготовкой на основе ролей, роли, назначенные в Azure AD, должны привестись к лицензиям приложений. Неправильные лицензии, принадлежащие в приложении, могут привести к ошибкам при подготовке/обновлении пользователя.

### <a name="terms"></a>Термины

В этой статье используются следующие термины:

* Операции CRUD - Действия, предпринятые на учетных записях пользователей: Создание, Чтение, Обновление, Удаление.

* Единый вход (SSO) - Возможность для пользователя войти один раз и получить доступ ко всем приложениям с поддержкой SSO. В контексте подготовки пользователей SSO является результатом того, что пользователи имеют единую учетную запись для доступа ко всем системам, которые используют автоматическую подготовку пользователей.

* Исходная система - Репозиторий пользователей, из которого запрещается Azure AD. Azure AD является исходной системой для большинства предварительно интегрированных разъемов подготовки. Однако для облачных приложений, таких как SAP, Workday и AWS, есть некоторые исключения. Например, см. [Представление о работе пользователя от рабочего дня до AD.](../saas-apps/workday-inbound-tutorial.md)

* Целевая система - репозиторий пользователей, к которым привит Azure AD. Целевая система обычно является приложением SaaS, таким как ServiceNow, Sscaler и Slack. Целевая система также может быть на территории, такой как AD.

* [Система управления кросс-доменом идентификации (SCIM)](https://aka.ms/scimoverview) - открытый стандарт, позволяющий автоматизировать подготовку пользователей. SCIM передает данные личных данных пользователей между поставщиками идентификационных данных, такими как Microsoft, и поставщиками услуг, такими как Salesforce или другими приложениями SaaS, которые требуют идентификационную информацию пользователя.

### <a name="solution-architectures"></a>Архитектуры решений

Сервис по предоставлению Azure AD предоставляет пользователям приложения SaaS и другие системы, подключившись к конечным точкам управления пользователями API, предоставляемым каждым поставщиком приложений. Эти конечные точки API управления пользователями позволяют Azure AD создавать, обновлять и удалять пользователей путем программирования.

#### <a name="automatic-user-provisioning-for-hybrid-enterprises"></a>Автоматическое обеспечение пользователей для гибридных предприятий

В этом примере пользователи и группы создаются в базе данных кадров, подключенных к каталогу. Служба предоставления AD Azure AD управляет автоматической подготовкой пользователей к целевым приложениям SaaS.

 ![подготовка пользователей](./media/plan-auto-user-provisioning/hybridprovisioning.png)

**Описание рабочего процесса:**

1. Пользователи/группы создаются в предварительном HR-приложении/системе, например SAP. 

1. **Агент Azure AD Connect** запускает запланированную синхронизацию идентификаторов (пользователей и групп) от локального AD до Azure AD.

1. **Служба предоставления AD Azure** начинает [начальный цикл](../app-provisioning/user-provisioning.md) с исходной системой и целевой системой. 

1. **Служба предоставления AD Azure AD** запрашивает исходную систему для любых пользователей и групп, измененных с момента начала цикла, и толкает изменения в [дополнительных циклах.](../app-provisioning/user-provisioning.md)

#### <a name="automatic-user-provisioning-for-cloud-only-enterprises"></a>Автоматическое обеспечение пользователей для облачных предприятий

В этом примере создание пользователей происходит в Azure AD, а служба предоставления AD Azure AD управляет автоматической подготовкой пользователей к целевым (SaaS) приложениям.

![Рисунок 2](./media/plan-auto-user-provisioning/cloudprovisioning.png)

**Описание рабочего процесса:**

1. Пользователи/группы создаются в Azure AD.

1. **Служба предоставления AD Azure** начинает [начальный цикл](../app-provisioning/user-provisioning.md) с исходной системой и целевой системой. 

1. **Служба предоставления AD Azure AD** запрашивает исходную систему для всех пользователей и групп, обновляемых с начала цикла, и выполняет любые [дополнительные циклы.](../app-provisioning/user-provisioning.md)

#### <a name="automatic-user-provisioning-for-cloud-hr-applications"></a>Автоматическая подготовка пользователей для облачных HR-приложений 

В этом примере пользователи и группы создаются в облачном HR-приложении, например Workday и SuccessFactors. Служба предоставления AD Azure AD и агент по подготовке Azure AD Connect вносят данные пользователей из клиента облачного HR-приложения в AD. После обновления учетных записей в AD они синхронизируются с Azure AD через Azure AD Connect, а адреса электронной почты и атрибуты имен пользователей могут быть записаны на арендатора облачного HR-приложения.

![Рисунок 2](./media/plan-auto-user-provisioning/workdayprovisioning.png)

1.  **HR-группа** выполняет транзакции в облаке HR-клиента.
2.  **Служба предоставления AD Azure** запускает запланированные циклы от нанимателя приложения для HR и определяет изменения, которые необходимо обрабатывать для синхронизации с AD.
3.  **Служба предоставления AD Azure** вызывает агента по подготовке Azure AD Connect с полезной нагрузкой запроса, содержащей создание/обновление/включение/отключение операций AD.
4.  **Агент по подготовке Azure AD Connect** использует учетную запись службы для управления данными учетной записи AD.
5.  **Azure AD Connect** запускает синхронизацию дельты, чтобы вытащить обновления в AD.
6.  **Обновления AD** синхронизируются с Azure AD. 
7.  **Служба предоставления услуг Azure AD записывает** атрибут электронной почты и имя пользователя из Azure AD в сторону арендатора облачных HR-приложений.

## <a name="plan-the-deployment-project"></a>Планирование проекта развертывания

Рассмотрите свои организационные потребности для определения стратегии развертывания пользовательской подготовки в среде.

### <a name="engage-the-right-stakeholders"></a>Привлечение правильных заинтересованных сторон

Когда технологические проекты терпят неудачу, это, как правило, из-за несовпадающих ожиданий в отношении воздействия, результатов и обязанностей. Чтобы избежать этих ловушек, [убедитесь, что вы привлекаете нужных заинтересованных сторон](https://aka.ms/deploymentplans) и что роли заинтересованных сторон в проекте хорошо понятны, документируя заинтересованные стороны и их вклад в проект и учетные данные.

### <a name="plan-communications"></a>Планирование информирования

Коммуникация имеет решающее значение для успеха любой новой службы. Проактивно общайтесь с пользователями о том, как изменится их опыт, когда он изменится и как получить поддержку, если у них возникнут проблемы.

### <a name="plan-a-pilot"></a>Планирование пилотного проекта

Мы рекомендуем, чтобы начальная конфигурация автоматической подготовки пользователей была в тестовой среде с небольшим набором пользователей, прежде чем масштабировать его для всех пользователей в производственной среде. Ознакомьтесь с [рекомендациями](../fundamentals/active-directory-deployment-plans.md#best-practices-for-a-pilot) по запуску пилотного проекта.

#### <a name="best-practices-for-a-pilot"></a>Лучшие практики для пилота  

Пилотный проект позволяет протестировать с небольшой группой перед развертыванием возможностей для всех. Убедитесь, что в рамках тестирования каждый случай использования в вашей организации тщательно проверяется.

В первой волне нацелитесь на ИТ,, удобство использования и других подходящих пользователей, которые могут протестировать и предоставить обратную связь. Используйте эту обратную связь для дальнейшей разработки сообщений и инструкций, которые вы отправляете своим пользователям, а также для того, чтобы дать представление о типах проблем, которые могут увидеть сотрудники службы поддержки.

Расширьте развертывание для более крупных групп пользователей за счет увеличения охвата целевой группы (ы). Это может быть сделано путем [динамического членства](../users-groups-roles/groups-dynamic-membership.md)в группе или путем ручного добавления пользователей в целевую группу (ы).

## <a name="plan-application-connections-and-administration"></a>Планирование подключения приложений и администрирования

Используйте портал Azure AD для просмотра и управления всеми приложениями, поддерживающими подготовку. Смотрите [Поиск приложений на портале](../app-provisioning/configure-automatic-user-provisioning-portal.md).

### <a name="determine-the-type-of-connector-to-use"></a>Определение типа разъема для использования

Фактические действия, необходимые для включения и настройки автоматической подготовки, зависят от приложения. Если приложение, которое вы хотите автоматически предоставить, перечислено в [галерее приложений Azure AD SaaS,](../saas-apps/tutorial-list.md)то вы должны выбрать [учебник по интеграции для](../saas-apps/tutorial-list.md) создания предварительно интегрированного разъема для предоставления пользовательского резервирования.

Если нет, выполните ниже:

1. [Создайте запрос](../develop/howto-app-gallery-listing.md) на предварительно интегрированный разъем для подготовки пользователей. Наша команда будет работать с вами и разработчиком приложений, чтобы навести ваше приложение на нашей платформе, если оно поддерживает SCIM.

1. Используйте общую поддержку пользовательского обеспечения [BYOA SCIM](../app-provisioning/use-scim-to-provision-users-and-groups.md) для приложения. Это требование для Azure AD предоставлять пользователям приложение без предварительно интегрированного разъема подготовки.

1. Если приложение может использовать разъем BYOA SCIM, то обратитесь к учебнику по [интеграции BYOA SCIM,](../app-provisioning/use-scim-to-provision-users-and-groups.md) чтобы настроить разъем BYOA SCIM для приложения.

Для получения дополнительной информации, [см. Какие приложения и системы я могу использовать с Azure AD автоматической подготовки пользователей?](../app-provisioning/user-provisioning.md)

### <a name="collect-information-to-authorize-application-access"></a>Сбор информации для авторизации доступа к приложениям

Настройка автоматического обеспечения пользователей является процессом для приложений. Для каждого приложения необходимо предоставить [учетные данные администратора](../app-provisioning/configure-automatic-user-provisioning-portal.md) для подключения к конечном точке управления пользователем целевой системы.

На рисунке ниже показана одна из версий необходимых учетных данных админ:

![Создание экрана для управления настройками подготовки учетных записей пользователя](./media/plan-auto-user-provisioning/userprovisioning-admincredentials.png)

В то время как некоторые приложения требуют имя пользователя и пароль для подключения, другие могут потребовать токен предъявителя.

## <a name="plan-user-and-group-provisioning"></a>Планирование подготовки пользователей и групп

Если включить подготовку пользователей для корпоративных приложений, [портал Azure](https://portal.azure.com/) контролирует значения атрибутов с помощью отображения атрибутов.

### <a name="determine-operations-for-each-saas-app"></a>Определение операций для каждого приложения SaaS

Каждое приложение может иметь уникальные атрибуты пользователя или группы, которые должны быть отображены с атрибутами в Azure AD. Приложение может иметь только подмножество операций CRUD.

Для каждой заявки задокументируйте следующую информацию:

* CRUD-операции подготовки, которые будут выполняться на объектах пользователя и группы для целевых систем. Например, каждый владелец приложения SaaS может захотеть не все возможные операции.

* Атрибуты, доступные в исходной системе

* Атрибуты, доступные в целевой системе

* Картирование атрибутов между системами.

### <a name="choose-which-users-and-groups-to-provision"></a>Выберите, какие пользователи и группы должны предоставлять

Перед внедрением автоматической подготовки пользователей необходимо определить пользователей и группы, которые должны быть подготовлены к приложению.

* Используйте [фильтры для определения](../app-provisioning/define-conditional-rules-for-provisioning-user-accounts.md) правил, основанных на атрибутах, которые определяют, какие пользователи подготовлены к приложению.

* Далее используйте [пользовательские и групповые назначения](../manage-apps/assign-user-or-group-access-portal.md) по мере необходимости для дополнительной фильтрации.

### <a name="define-user-and-group-attribute-mapping"></a>Определение отображения атрибутов пользователя и группы

Для автоматической подготовки пользователей необходимо определить атрибуты пользователя и группы, необходимые для приложения. Между объектами пользователя Azure AD и каждым пользовательским объектом приложения SaaS имеется предварительно настроенный набор атрибутов и [отображений атрибутов.](../app-provisioning/configure-automatic-user-provisioning-portal.md) Не все приложения SaaS позволяют группировать атрибуты.

Azure AD поддерживается прямым отображением атрибутов, предоставлением постоянных значений или [написанием выражений для отображений атрибутов.](../app-provisioning/functions-for-customizing-application-data.md) Эта гибкость позволяет точно контролировать то, что будет заселено в атрибуте целевой системы. Можно использовать [Microsoft Graph API](../app-provisioning/export-import-provisioning-configuration.md) и Graph Explorer для экспорта отображения атрибутов и схемы подготовки пользователей в файл JSON и импорта его обратно в Azure AD.

Для получения дополнительной информации в [каталоге Active Directory можно найти настройку пользовательских карт для приложений SaaS для приложений SaaS.](../app-provisioning/customize-application-attributes.md)

### <a name="special-considerations-for-user-provisioning"></a>Особые соображения для подготовки пользователей

Рассмотрим следующие вопросы, чтобы уменьшить проблемы после развертывания:

* Убедитесь, что атрибуты, используемые для картирования объектов пользователя/группы между исходными и целевыми приложениями, являются устойчивыми. Они не должны приводить к неправильному подготовке пользователей/групп при изменении атрибутов (например, пользователь перемещается в другую часть компании).

* Приложения могут иметь определенные ограничения и/или требования, которые должны быть выполнены для правильной работы пользователя. Например, Slack усечения значений для определенных атрибутов. Обратитесь к [автоматическому подготовке пользователей учебники,](../saas-apps/tutorial-list.md) характерные для каждого приложения.

* Подтвердите согласованность схем между исходными и целевыми системами. Общие проблемы включают атрибуты, такие как UPN или не соответствует почте. Например, UPN в Azure *john_smith@contoso.com* AD устанавливается как *jsmith@contoso.com*и в приложении, это . Для получения дополнительной [User and group schema reference](../app-provisioning/use-scim-to-provision-users-and-groups.md)информации см.

## <a name="plan-testing-and-security"></a>Планирование тестирования и безопасности

На каждом этапе развертывания убедитесь, что вы тестируете результаты, как и ожидалось, и аудит циклов подготовки.

### <a name="plan-testing"></a>Планирование тестирования

После настройки автоматической подготовки пользователя для приложения вы запустите тестовые случаи, чтобы проверить, что это решение соответствует требованиям организации.

| Сценарии| Ожидаемые результаты |
| - | - |
| Пользователь добавляется в группу, назначенную целевой системе | Объект пользователя подготовлен в целевой системе. <br>Пользователь может войти в целевую систему и выполнить желаемые действия. |
| Пользователь удаляется из группы, назначенной целевой системе | Объект пользователя дерезервируется в целевой системе.<br>Пользователь не может войти в целевую систему. |
| Информация о пользователях обновляется в Azure AD любым методом | Обновленные атрибуты пользователя отражаются в целевой системе после поэтапного цикла |
| Пользователь выходит за рамки | Объект пользователя отключен или удален. <br>Примечание: Это поведение переопределяется для [подготовки рабочего дня.](skip-out-of-scope-deletions.md) |

### <a name="plan-security"></a>Планирование безопасности

Обычно проверка безопасности требуется в рамках развертывания. Если требуется проверка безопасности, см. множество [информационных бюллетеней](https://www.microsoft.com/download/details.aspx?id=36391) Azure AD, которые предоставляют обзор идентификации как службы.

### <a name="plan-rollback"></a>Откат плана

Если автоматическая реализация подготовки пользователя не работает по желанию в рабочей среде, следующие шаги отката ниже могут помочь вам вернуться к предыдущему известному хорошему состоянию:

1. Просмотрите [сводный отчет](../app-provisioning/check-status-user-account-provisioning.md) и [журналы подготовки,](../app-provisioning/check-status-user-account-provisioning.md#provisioning-logs-preview) чтобы определить, какие неправильные операции произошли на затронутых пользователях и/или группах.

1. Используйте журналы аудита для определения последнего известного хорошего состояния пользователей и/или затронутых групп. Также просмотрите исходные системы (Azure AD или AD).

1. Работайте с владельцем приложения, чтобы обновить пользователей и/или группы, затронутые непосредственно в приложении, с использованием последних известных значений хорошего состояния.

## <a name="deploy-automatic-user-provisioning-service"></a>Развертывание службы автоматического обеспечения пользователей

Выберите шаги, которые соответствуют требованиям решения.

### <a name="prepare-for-the-initial-cycle"></a>Подготовка к начальному циклу

Когда служба обеспечения AD Azure AD работает впервые, начальный цикл между исходной системой и целевыми системами создает снимок всех объектов пользователя для каждой целевой системы.

При автоматической подготовке приложения начальный цикл может занять от 20 минут до нескольких часов. Продолжительность зависит от размера каталога Azure AD и количества пользователей в области подготовки. [Узнайте, как улучшить производительность подготовки.](../app-provisioning/application-provisioning-when-will-provisioning-finish.md)

Служба обеспечения сохраняет состояние обеих систем после первоначального цикла, повышая производительность последующих дополнительных циклов.

### <a name="configure-automatic-user-provisioning"></a>Настройка автоматической подготовки пользователей

Используйте [портал Azure](https://portal.azure.com/) для автоматического подготовки и де-подготовки приложений, поддерживающих его, и де-подготовки. Выполните последующие действия в [Как настроить автоматическую подготовку к приложению?](../app-provisioning/user-provisioning.md)

Для настройки службы подготовки пользователей Azure AD и управления ею можно также использовать [API Microsoft Graph](https://developer.microsoft.com/graph/docs/api-reference/beta/resources/synchronization-overview).

## <a name="manage-automatic-user-provisioning"></a>Управление автоматическим обеспечением пользователей

Теперь, когда вы развернуты, вам нужно управлять решением.

### <a name="monitor-user-provisioning-operation-health"></a>Мониторинг работоспособности работы пользователя

После успешного [начального цикла](../app-provisioning/user-provisioning.md)служба предоставления Azure AD будет работать поэтапное обновление на неопределенный срок, с интервалами, определенными для каждого приложения, до тех пор, пока не произойдет одно из следующих событий:

* Служба останавливается вручную, и новый начальный цикл запускается с помощью [портала Azure](https://portal.azure.com/)или с помощью соответствующей команды [Microsoft Graph API.](https://developer.microsoft.com/graph/docs/api-reference/beta/resources/synchronization-overview)

* Новый начальный цикл срабатывает из-за изменения отображения атрибутов или фильтров для скопирования.

* Процесс подготовки переходит в карантин из-за высокой частоты ошибок и остается в карантине более четырех недель, когда он будет автоматически отключен.

Для анализа этих событий и всех других действий, выполняемых службой обеспечения, обратитесь к [журналам подготовки](../reports-monitoring/concept-provisioning-logs.md?context=azure/active-directory/manage-apps/context/manage-apps-context)Azure AD.

Чтобы понять, сколько времени занимает циклы подготовки и следит за ходом выполнения задания, можно [проверить состояние подготовки пользователя.](../app-provisioning/application-provisioning-when-will-provisioning-finish-specific-user.md)

### <a name="gain-insights-from-reports"></a>Получить информацию из отчетов

Azure AD может предоставить [дополнительные сведения об](../app-provisioning/application-provisioning-when-will-provisioning-finish-specific-user.md) использовании пользователей вашей организации и оперативном состоянии с помощью журналов аудита и отчетов.

Админы должны проверить подготовку сводной отчета для мониторинга оперативного состояния работы подготовки. Все действия, выполняемые службой обеспечения, записываются в журналы аудита Azure AD. [См. Учебник: Отчетность об автоматическом подготовке учетной записи пользователя](../app-provisioning/check-status-user-account-provisioning.md).

Мы рекомендуем вам взять на себя право собственности и использовать эти отчеты на каденцию, которая отвечает требованиям вашей организации. Azure AD сохраняет большинство аудиторских данных в течение 30 дней.
