# <a name="enable-automatic-user-provisioning-for-your-multi-tenant-application"></a>Включить автоматическое оформление приложения для мультиарендаторов

Автоматическое обеспечение пользователей — это процесс автоматизации создания, обслуживания и удаления идентификаторов пользователей в целевых системах, таких как приложения для программного обеспечения как услуга.

## <a name="why-enable-automatic-user-provisioning"></a>Зачем включать автоматическое представление пользователей?

Приложения, требующие присутствия записи пользователя в приложении до вхотливки первого знака пользователя, требуют подготовки пользователя. Есть преимущества для вас, как поставщика услуг, и преимущества для ваших клиентов.

### <a name="benefits-to-you-as-the-service-provider"></a>Преимущества для вас как поставщика услуг

* Увеличьте безопасность приложения с помощью платформы идентификации Майкрософт.

* Сократите фактические и предполагаемые усилия клиента по принятию приложения.

* Сократите затраты на интеграцию с несколькими поставщиками идентификационных данных (IdPs) для автоматической подготовки пользователей с помощью системы управления кросс-доменами (SCIM) на основе.

* Сократите расходы на поддержку, предоставляя богатые журналы, чтобы помочь клиентам устранить проблемы с подготовкой пользователей.

* Увеличьте видимость приложения в [галерее приложений Azure AD.](https://azuremarketplace.microsoft.com/marketplace/apps)

* Получите приоритетное объявление на странице App Tutorials.

### <a name="benefits-to-your-customers"></a>Преимущества для ваших клиентов

* Увеличьте безопасность, автоматически удалив доступ к приложению для пользователей, которые меняют роли или покидают организацию в вашем приложении.

* Упростите управление пользователем для вашего приложения, избегая человеческих ошибок и повторяющихся работ, связанных с ручной подготовкой.

* Сократите расходы на хостинг и обслуживание разработанных на заказ решений по подготовке.

## <a name="choose-a-provisioning-method"></a>Выбор метода подготовки

Azure AD предоставляет несколько путей интеграции для автоматической подготовки пользователей для вашего приложения.

* [Служба предоставления aD Azure](../app-provisioning/user-provisioning.md) управляет подготовкой и деированием пользователей из Azure AD в ваше приложение (исходящий подготовка) и от приложения к Azure AD (входная подготовка). Услуга подключается к Системе управления кросс-доменом (SCIM) API-конечных точек управления пользователями, предоставляемых вашим приложением.

* При использовании [Microsoft Graph](https://docs.microsoft.com/graph/)приложение управляет входящими и исходящими средствами подготовки пользователей и групп из Azure AD в ваше приложение, задав запрос в API Microsoft Graph.

* Язык проверки безопасности Markup Language Just in Time (SAML JIT) может быть включен, если ваше приложение использует SAML для федерации. Он использует информацию о претензиях, отправляемую в токен SAML, для предоставления пользователям.

Чтобы определить, какой вариант интеграции использовать для приложения, обратитесь к таблице сравнения высокого уровня, а затем ознакомьтесь с более подробной информацией по каждому варианту.

## <a name="azure-ad-provisioning-service-scim"></a>Служба предоставления aD Azure (SCIM)

Службы предоставления AD Azure используют [SCIM,](https://aka.ms/SCIMOverview)отраслевой стандарт для подготовки, поддерживаемый многими поставщиками идентификационных данных (IdPs), а также приложениями (например, Slack, G Suite, Dropbox). Мы рекомендуем использовать службу предоставления Azure AD, если вы хотите поддерживать ИДП в дополнение к Azure AD, так как любой ИДП, совместимый с SCIM, может подключиться к вашей цели SCIM. Создавая простую /пользовательскую конечную точку, вы можете включить подготовку без необходимости поддерживать свой собственный движок синхронизации.

## <a name="microsoft-graph-for-provisioning"></a>График Microsoft для подготовки

При использовании Microsoft Graph для подготовки у вас есть доступ ко всем богатым пользовательским данным, доступным в Graph. В дополнение к информации о пользователях и группах, вы также можете получить дополнительную информацию, как роли пользователя, менеджер и прямые отчеты, принадлежащих и зарегистрированных устройств, а также сотни других частей данных, доступных в [Microsoft Graph](https://docs.microsoft.com/graph/api/overview?view=graph-rest-1.0). 

Более 15 миллионов организаций и 90% компаний из состояния 500 используют Azure AD, подписываясь на облачные сервисы Майкрософт, такие как Office 365, Microsoft Azure, Enterprise Mobility Suite или Microsoft 365. Можно использовать Microsoft Graph для интеграции приложения с административными рабочими процессами, такими как операторский онбор (и прекращение работы), обслуживание профиля и многое другое.

## <a name="using-saml-jit-for-provisioning"></a>Использование SAML JIT для подготовки

Если вы хотите предоставить пользователям только при первом входе в приложение, и вам не нужно автоматически дезавуировать пользователей, SAML JIT является опцией. Ваша заявка должна поддерживать SAML 2.0 в качестве протокола федерации для использования SAML JIT.

SAML JIT использует информацию о претензиях в токене SAML для создания и обновления пользовательской информации в приложении. Клиенты могут настроить эти необходимые требования в приложении Azure AD по мере необходимости. Иногда подготовка JIT должна быть включена со стороны приложения, чтобы клиент мог использовать эту функцию. SAML JIT полезен для создания и обновления пользователей, но он не может удалить или отключить пользователей в приложении.
