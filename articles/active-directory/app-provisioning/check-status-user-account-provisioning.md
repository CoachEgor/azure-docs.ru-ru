# <a name="tutorial-reporting-on-automatic-user-account-provisioning"></a>Руководство по отчетам об автоматической подготовке учетных записей пользователей

Активный каталог Azure (Azure AD) включает в себя [службу предоставления учетных записей пользователей,](user-provisioning.md) которая помогает автоматизировать предоставление учетных записей пользователей в приложениях SaaS и других системах с целью сквозного управления жизненным циклом идентификации. Azure AD поддерживает предварительно интегрированные разъемы для подготовки пользователей для всех приложений и систем с учебниками по подготовке пользователей [здесь.](https://docs.microsoft.com/azure/active-directory/saas-apps/tutorial-list)

В этой статье описывается, как проверить состояние заданий подготовки после их настройки и устранить неполадки при подготовке отдельных пользователей и групп.

## <a name="overview"></a>Обзор

Соединители подготовки устанавливаются и настраиваются с помощью [портала Azure](https://portal.azure.com). Для этого можно воспользоваться [документацией](../saas-apps/tutorial-list.md), предоставленной для поддерживаемого приложения. После настройки и запуска заданий подготовки можно создать отчеты по ним, воспользовавшись одним из двух способов.

* **Портал Azure** - Эта статья в первую очередь описывает получение информации об отчете с [портала Azure,](https://portal.azure.com)который предоставляет как сводный отчет, так и подробные журналы аудита для данного приложения.
* **API аудита**. Azure Active Directory также предоставляет API аудита, который обеспечивает программное получение подробных журналов аудита. Документация по работе с этим API приведена в [справочнике по API аудита Azure Active Directory](https://developer.microsoft.com/graph/docs/api-reference/beta/resources/directoryaudit). Хотя в данной статье не рассматривается использование этого API, в ней описываются типы событий подготовки, записываемых в журнал аудита.

### <a name="definitions"></a>Определения

В этой статье используются термины, приведенные ниже.

* **Исходная система**. Исходный репозиторий пользователей для синхронизации службы подготовки Azure AD. Azure Active Directory является исходной системой для большинства предварительно интегрированных соединителей для подготовки, однако имеются некоторые исключения (пример: входящая синхронизация Workday).
* **Целевая система**. Целевой репозиторий пользователей для синхронизации службы подготовки Azure AD. Обычно это приложение SaaS (примеры: Salesforce, ServiceNow, G Suite, Dropbox for Business), но в некоторых случаях может быть предприимчивая система, такая как Active Directory (например: синхронизация рабочего дня для активного каталога).

## <a name="getting-provisioning-reports-from-the-azure-portal"></a>Получение отчетов о подготовке с портала Azure

Чтобы получить информацию о отчете о подготовке данного приложения, начните с запуска [портала Azure](https://portal.azure.com) и **журналов подготовки приложений для активных корпоративных** **приложений** &gt; Azure active **Directory** &gt; Apps (предварительный просмотр) в разделе **Activity.** Вы также можете просмотреть приложение Enterprise, для которого настроена подготовка. Например, если подготавливаются пользователи в LinkedIn Elevate, то путь к сведениям о приложении следующий:

**Azure Active Directory > Корпоративные приложения > Все приложения > LinkedIn Elevate**

Отсюда вы можете получить доступ как к панели прогресса, так и к журналам подготовки, описанным ниже.

## <a name="provisioning-progress-bar"></a>Подготовка панели прогресса

[Панель прогресса подготовки](application-provisioning-when-will-provisioning-finish-specific-user.md#view-the-provisioning-progress-bar) отображается во вкладке **"Подготовка"** для данного приложения. Он расположен в разделе **Текущего состояния** под **настройками**и показывает состояние текущего начального или поэтапного цикла. В этом разделе также показано:

* Общее количество пользователей и групп, которые синхронизированы и в настоящее время находятся в области действия подготовки между исходной системой и целевой системой.
* Время последнего запуска синхронизации. Синхронизация обычно происходит каждые 20-40 минут, после [завершения первоначального цикла.](../app-provisioning/how-provisioning-works.md#provisioning-cycles-initial-and-incremental)
* Завершен или нет [первоначальный цикл.](../app-provisioning/how-provisioning-works.md#provisioning-cycles-initial-and-incremental)
* Сведения о том, помещен ли процесс подготовки в карантин, а также о причине состояния карантина (например, не удалось связаться с целевой системой из-за неправильных учетных данных администратора).

**Текущий статус** должен быть первым местом, которое должны искать админы, чтобы проверить оперативное состояние работоспособности задания.

## <a name="provisioning-logs-preview"></a>Создание журналов (предварительный просмотр)

Все действия, выполняемые службой обеспечения, записываются в [журналы подготовки](../reports-monitoring/concept-provisioning-logs.md?context=azure/active-directory/manage-apps/context/manage-apps-context)Azure AD. Вы можете получить доступ к журналам подготовки на портале Azure, выбрав **журналы (предварительный просмотр»** &gt; **приложений для активных** &gt; **каталогов Azure,** то есть в разделе **«Активность».** Вы можете искать данные подготовки на основе имени пользователя или идентификатора либо в исходной системе, либо в целевой системе. Для получения подробной информации [см.](../reports-monitoring/concept-provisioning-logs.md?context=azure/active-directory/manage-apps/context/manage-apps-context) В журналы записываются следующие типы событий действия.

## <a name="troubleshooting"></a>Устранение неполадок

Краткий отчет и журналы подготовки играют ключевую роль, помогая админинам устранять различные проблемы с подготовкой учетных записей пользователей.

Руководство по устранению неполадок при автоматической подготовке пользователей со сценариями доступно в разделе [Проблемы при настройке и подготовке пользователей для приложения](../app-provisioning/application-provisioning-config-problem.md).
