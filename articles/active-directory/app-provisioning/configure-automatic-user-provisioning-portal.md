# <a name="managing-user-account-provisioning-for-enterprise-apps-in-the-azure-portal"></a>Управление подготовкой учетных записей пользователей для корпоративных приложений на портале Azure

В этой статье описаны общие шаги по управлению автоматической подготовкой учетных записей пользователей и де-подготовкой приложений, поддерживающих его. *Подготовка учетной записи пользователя* — это создание, обновление или отключение записей для учетных записей пользователя в локальном хранилище профилей пользователей. Большинство облачных и SaaS-приложений хранят роль и разрешения пользователей в локальном магазине профиля пользователя, и наличие такой записи пользователя в локальном магазине пользователя *требуется* для единого включения и доступа к работе. Чтобы узнать больше об автоматическом подготовке учетной записи пользователя, см. [Автоматизировать пользовательские приложения и деирование приложений SaaS с помощью Active Directory Azure.](user-provisioning.md)

> [!IMPORTANT]
> Активный каталог Azure (Azure AD) имеет галерею, которая содержит тысячи предварительно интегрированных приложений, которые включены для автоматического подготовки с Azure AD. Вы должны начать с поиска учебника по настройке подготовки, конкретного для вашего приложения, в [списке учебников о том, как интегрировать приложения SaaS с Azure Active Directory.](https://azure.microsoft.com/documentation/articles/active-directory-saas-tutorial-list/) Скорее всего, вы найдете пошаговое руководство для настройки приложения и Azure AD для создания подключения к подготовке.

## <a name="finding-your-apps-in-the-portal"></a>Поиск приложений на портале

Используйте портал Azure Active Directory для просмотра и управления всеми приложениями, настроенными для одного входного в каталоге. Корпоративные приложения — это приложения, развертываемые и используемые в вашей организации. Выполните следующие действия для просмотра и управления корпоративными приложениями:

1. Откройте [портал Active Directory Azure](https://aad.portal.azure.com).
1. Выберите **приложения Enterprise** из левого стекла. Отображается список всех настроенных приложений, включая приложения, добавленные из галереи.
1. Выберите любое приложение для загрузки панели ресурсов, где можно просматривать отчеты и управлять настройками приложения.
1. Выберите **«Подготовка»** для управления настройками подготовки учетной записи пользователя для выбранного приложения.

   ![Создание экрана для управления настройками подготовки учетных записей пользователя](./media/configure-automatic-user-provisioning-portal/enterprise-apps-provisioning.png)

## <a name="provisioning-modes"></a>Режимы подготовки

Панель **подготовки** начинается с меню **режима,** которое показывает режимы подготовки, поддерживаемые для корпоративного приложения, и позволяет настроить их. Доступны следующие параметры.

* **Автоматическая** - Эта опция отображается, если Azure AD поддерживает автоматическое предоставление или де-предоставление учетных записей пользователей этому приложению. Выберите этот режим для отображения интерфейса, который помогает администраторам:

  * Настройка Azure AD для подключения к API управления пользователями приложения
  * Создание карт оценок и рабочих процессов, определяющих, как должны передаваться данные учетной записи пользователя между Azure AD и приложением
  * Управление службой подготовки Azure AD

* **Руководство** - Эта опция отображается, если Azure AD не поддерживает автоматическое предоставление учетных записей пользователей этому приложению. В этом случае записи учетных записей пользователей, хранящиеся в приложении, должны управляться с помощью внешнего процесса, основанного на управлении пользователем и возможностях подготовки, предоставляемых этим приложением (которые могут включать подготовку SAML Just-In-Time).

## <a name="configuring-automatic-user-account-provisioning"></a>Настройка автоматической подготовки учетных записей пользователей

Выберите **опцию Automatic** для указания параметров для учетных данных, отображения, запуска и остановки и синхронизации.

### <a name="admin-credentials"></a>Учетные данные администратора

Расширьте **учетные данные admin** для ввода учетных данных, необходимых для подключения AD-приложения к API управления пользователями. Требуемые входные данные зависят от приложения. Дополнительные сведения о типах учетных данных и требованиях для конкретных приложений см. в [руководстве по конфигурации для конкретного приложения](user-provisioning.md).

Выберите **тестовое соединение** для проверки учетных данных, припопытки Azure AD подключиться к приложению, работая над приложением, используя предоставленные учетные данные.

### <a name="mappings"></a>Сопоставления

Расширяйте **картографы** для просмотра и отсвачения атрибутов пользователя, которые текут между Azure AD и целевым приложением при подготовке или обновлении учетных записей пользователей.

Существует предварительно настроенный набор картотеки между объектами пользователей Azure AD и пользовательскими объектами каждого приложения SaaS. Некоторые приложения управляют другими типами объектов, такими как группы или контакты. Выберите отображение в таблице, чтобы открыть отображающий редактор справа, где вы можете просмотреть и настроить их.

![Отображается экран отображения атрибутов](./media/configure-automatic-user-provisioning-portal/enterprise-apps-provisioning-mapping.png)

Поддерживаемые настройки

* Включение и отключение сопоставлений для определенных объектов, например между объектом пользователя Azure AD и приложения SaaS.
* Изменение атрибутов, поступающих из объекта пользователя Azure AD к объекту пользователя приложения. Дополнительные сведения о сопоставлении атрибутов см. в разделе [Основные сведения о типах сопоставления атрибутов](customize-application-attributes.md#understanding-attribute-mapping-types).
* Фильтрация действий подготовки, которые Azure AD выполняет в целевом приложении. Вместо того, чтобы Azure AD полностью синхронизировать объекты, можно ограничить запуск действий.

  Например, только выберите **Update** и Azure AD, только обновляя существующие учетные записи пользователей в приложении, но не создают новые. Только выберите **Create** и Azure только создает новые учетные записи пользователей, но не обновляет существующие. Эта функция позволяет адм-имам создавать различные отображения для создания и обновления рабочих процессов.

* Добавление нового отображения атрибутов. Выберите **Добавить новую картографию** в нижней части **панели отображения атрибутов.** Заполните форму **Edit Attribute** и выберите **Ok,** чтобы добавить новое отображение в список.

### <a name="settings"></a>Параметры

Можно запустить и остановить службу предоставления Azure AD для выбранного приложения в области **настроек** экрана **«Подготовка».** Вы также можете очистить кэш подготовки и перезапустить службу.

Если подготовка включается для приложения впервые, то включите службу, переместив переключатель **Состояние подготовки** в положение **Вкл**. Это изменение приводит к тому, что служба предоставления AD Azure AD выполняет начальный цикл. Он считывает пользователей, назначенных в разделе **«Пользователи и группы»,** запрашивает для них целевое приложение, а затем запускает действия подготовки, определенные в разделе AD Mappings Azure **AD.** Во время этого процесса служба обеспечения хранит кэшированные данные о том, какими учетными записями пользователей он управляет, поэтому неуправляемые учетные записи внутри целевых приложений, которые никогда не были в области назначения, не затрагиваются операциями де-обслуживания. После первоначального цикла служба обеспечения автоматически синхронизирует пользовательские и групповые объекты с интервалом в сорок минут.

Измените **статус «Обеспечение»** для **приостановки** службы обеспечения. В этом состоянии Azure не создает, не обновляет и не удаляет объекты пользователя или группы в приложении. Измените состояние обратно в **On,** и служба забирает, где она остановилась.

**Четкое текущее состояние и синхронизация перезапуска** вызывает начальный цикл. Затем служба снова оценит всех пользователей в исходной системе и определит, находятся ли они в возможностях подготовки. Это может быть полезно, когда ваше приложение в настоящее время находится в карантине или вам нужно внести изменения в отображение атрибутов. Обратите внимание, что первоначальный цикл занимает больше времени, чем типичный дополнительный цикл из-за количества объектов, которые должны быть оценены. Вы можете узнать больше о производительности начальных и дополнительных циклов [здесь](application-provisioning-when-will-provisioning-finish-specific-user.md).
