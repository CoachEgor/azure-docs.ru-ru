# <a name="authentication-vs-authorization"></a>Сравнение проверки подлинности и авторизации

В этой статье описывается проверка подлинности и авторизация, а также рассматривается использование платформы Microsoft Identity для проверки подлинности и авторизации пользователей в веб-приложениях, веб-API или приложениях, вызывающих защищенные веб-API. Если вы видите термин, который вам не знаком, попробуйте наш [Глоссарий](developer-glossary.md) или [видеоролики Майкрософт по платформе идентификации](identity-videos.md) , которые охватывают основные понятия.

## <a name="authentication"></a>Аутентификация

**Проверка подлинности** — это процесс подтверждения того, что вы говорите. Аутентификация иногда сокращенно обозначается AuthN (Authentication). Платформа Microsoft Identity реализует протоколы [OpenID Connect Connect](https://openid.net/connect/) и [SAML 2,0](http://docs.oasis-open.org/security/saml/Post2.0/sstc-saml-tech-overview-2.0.html) для обработки проверки подлинности.

## <a name="authorization"></a>Авторизация

**Авторизация** — это действие предоставления проверяющей стороны разрешения на выполнение каких-либо действий. Он определяет, к каким данным разрешен доступ и что можно делать с этими данными. Авторизация иногда сокращенно обозначается AuthZ (Authorization). Платформа Microsoft Identity реализует протокол [OAuth 2,0](https://oauth.net/2/) для обработки авторизации.

## <a name="authentication-and-authorization-using-microsoft-identity-platform"></a>Проверка подлинности и авторизация с помощью платформы Microsoft Identity

Вместо создания приложений, каждый из которых хранит свои собственные сведения о имени пользователя и пароле, что влечет за собой высокий уровень административной нагрузки при необходимости добавления или удаления пользователей в нескольких приложениях, приложения могут делегировать ответственность за централизованный поставщик удостоверений.

Azure Active Directory (Azure AD) — это централизованный поставщик удостоверений в облаке. Делегирование проверки подлинности и авторизации в него позволяет выполнять такие сценарии, как политики условного доступа, требующие, чтобы пользователь найдет в определенном месте, использовать многофакторную проверку подлинности, а также разрешить пользователю входить один раз, а затем автоматически войти во все веб-приложения, использующие один централизованный каталог. Эта возможность называется **единым входом (SSO)**.

Платформа идентификации Майкрософт упрощает проверку подлинности и авторизацию для разработчиков приложений, предоставляя удостоверение в качестве службы с поддержкой стандартных отраслевых протоколов, таких как OAuth 2,0, OpenID Connect Connect и SAML 2,0, а также библиотек с открытым исходным кодом для различных платформ, помогающих быстро начать программировать. Она позволяет разработчикам создавать приложения, которые подписывают все удостоверения Майкрософт, получать маркеры для вызова [Microsoft Graph](https://developer.microsoft.com/graph/), других API-интерфейсов Майкрософт или интерфейсов API, созданных разработчиками. Дополнительные сведения см. в [статье развитие платформы идентификации Майкрософт](about-microsoft-identity-platform.md).

Ниже приведено краткое сравнение различных протоколов, используемых платформой идентификации Майкрософт.

* **OAuth и OpenID Connect Connect**: OAuth используется для авторизации, а OpenID Connect Connect (OIDC) используется для проверки подлинности. OpenID Connect Connect построена на основе OAuth 2,0, поэтому терминология и последовательность сходны между ними. Можно даже проверить подлинность пользователя (с помощью OpenID Connect Connect) и получить авторизацию для доступа к защищенному ресурсу, который владеет пользователем (с помощью OAuth 2,0) в одном запросе. Дополнительные сведения см. в разделе [протоколы OAuth 2,0 и OpenID Connect Connect](active-directory-v2-protocols.md) и [протокол OpenID Connect Connect](v2-protocols-oidc.md).
* **OAuth и SAML**: OAuth используется для авторизации, а SAML используется для проверки подлинности. Дополнительные сведения о том, как можно использовать два протокола для проверки подлинности пользователя (с помощью SAML) и получить авторизацию для доступа к защищенному ресурсу (с помощью OAuth 2,0), см. в [статье о платформе Microsoft Identity и oauth 2,0 SAML Flow assertion](v2-saml-bearer-assertion.md) .
* **OpenID Connect Connect и SAML**: оба OpenID Connect соединения и SAML используются для проверки подлинности пользователя и используются для включения единого входа. Проверка подлинности SAML обычно используется с поставщиками удостоверений, например службы федерации Active Directory (AD FS) (ADFS), федеративными в Azure AD и поэтому часто используется в корпоративных приложениях.
