# <a name="desktop-app-that-calls-web-apis---app-registration"></a>Классическое приложение, вызывающее веб-API — регистрация приложений

Эта статья содержит сведения о регистрации приложений для классического приложения.

## <a name="supported-accounts-types"></a>Поддерживаемые типы учетных записей

Типы учетных записей, поддерживаемые в классическом приложении, зависят от опыта работы. Из-за этой связи Поддерживаемые типы учетных записей зависят от потоков, которые вы хотите использовать.

### <a name="audience-for-interactive-token-acquisition"></a>Аудитория для интерактивного получения маркера

Если в классическом приложении используется интерактивная проверка подлинности, вы можете выполнять вход пользователей из любого [типа учетной записи](quickstart-register-app.md#register-a-new-application-using-the-azure-portal).

### <a name="audience-for-desktop-app-silent-flows"></a>Аудитория для потоковой передачи в классическом приложении

- Чтобы использовать встроенную проверку подлинности Windows или имя пользователя и пароль, приложение должно входить в систему пользователей в вашем клиенте (бизнес-разработчик) или в организациях Azure Active Directory (сценарий ISV). Эти потоки проверки подлинности не поддерживаются для личных учетных записей Майкрософт.
- Если вы хотите использовать поток кода устройства, вы еще не сможете войти в систему пользователей с помощью личных учетных записей Майкрософт.
- Если войти в систему пользователей с удостоверениями социальных сетей, B2C центр и политику, можно использовать только интерактивную проверку подлинности и аутентификацию с использованием имени пользователя.

## <a name="redirect-uris"></a>URI-коды перенаправления

Используемые в настольных приложениях идентификаторы URI перенаправления будут зависеть от последовательности, которую вы хотите использовать.

- Если вы используете **интерактивную проверку подлинности** или **поток кода устройства**, необходимо использовать `https://login.microsoftonline.com/common/oauth2/nativeclient`. Эту конфигурацию можно получить, щелкнув соответствующий URL-адрес в разделе **authentication (проверка подлинности** ) для вашего приложения.
  
  > [!IMPORTANT]
  > Сегодня MSAL.NET по умолчанию использует другой URI перенаправления в классических приложениях под`urn:ietf:wg:oauth:2.0:oob`управлением Windows (). В будущем мы хотим изменить это значение по умолчанию, поэтому мы рекомендуем использовать`https://login.microsoftonline.com/common/oauth2/nativeclient`

- Если вы создаете собственное приложение с машинным кодом на C или SWIFT для macOS, вам нужно зарегистрировать redirectUri на основе идентификатора пакета приложения в следующем формате: **msauth. < The. app. пучок. id >://АУС** (replace < Ваш App. пучок. ID > с идентификатором пакета приложения.
- Если приложение использует только встроенную проверку подлинности Windows или имя пользователя и пароль, вам не нужно регистрировать универсальный код ресурса (URI) перенаправления для приложения. Эти потоки выполняют циклическое обращение к конечной точке платформы Microsoft Identity Platform версии 2.0, и ваше приложение не будет вызываться обратно по какому-либо конкретному URI.
- Чтобы отличать поток кода устройства, встроенную проверку подлинности Windows и имя пользователя и пароль из конфиденциального потока приложений клиента, не имеющего URI перенаправления (поток учетных данных клиента, используемый в управляющих приложениях), необходимо выразить приложение является общедоступным клиентским приложением. Чтобы получить эту конфигурацию, перейдите к разделу **authentication (проверка подлинности** ) для своего приложения. Затем в подразделе **Дополнительные параметры** в поле **Тип клиента по умолчанию** выберите **Да** для вопроса **рассматривать приложение как общедоступный клиент**.

  ![Разрешить общедоступный клиент](media/scenarios/default-client-type.png)

## <a name="api-permissions"></a>Разрешения API

Классические приложения вызывают интерфейсы API для пользователя, выполнившего вход. Они должны запрашивать делегированные разрешения. Однако они не могут запрашивать разрешения приложения, которые обрабатываются только в [управляющих приложениях](scenario-daemon-overview.md).
