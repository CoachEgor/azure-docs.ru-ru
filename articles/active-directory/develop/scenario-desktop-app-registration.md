# <a name="desktop-app-that-calls-web-apis---app-registration"></a>Приложения на рабочем столе, вызовы веб-интерфейсы API — Регистрация приложения

Эта статья содержит specificities регистрации приложения для классического приложения.

## <a name="supported-accounts-types"></a>Типы поддерживаемых учетных записей

Типы учетных записей, поддерживаемые в классическом приложении зависит от качества, для подготовки, и поэтому на потоки вы хотите использовать.

### <a name="audience-for-interactive-token-acquisition"></a>Аудитория для интерактивного получения маркера

Если приложение рабочего стола использует интерактивной проверки подлинности, можно выполнить вход пользователей из любого [тип учетной записи](quickstart-register-app.md#register-a-new-application-using-the-azure-portal)

### <a name="audience-for-desktop-app-silent-flows"></a>Аудитория для автоматической потоков приложения на рабочем столе

- Если вы планируете использовать проверку подлинности встроенная Windows или имя пользователя и пароль, ваше приложение должно выполнять вход пользователей в собственном клиенте (разработчик LOB), или в Azure Active directory организации (сценарием ISV). Эти потоки проверки подлинности не поддерживаются для личных учетных записей Майкрософт
- Если вы хотите использовать поток кода для устройства, не удается войти в систему с помощью личной учетной записью Майкрософт еще
- Если вход пользователей с удостоверениями социальных сетей, передав центра B2C, политики, можно использовать только проверки подлинности интерактивных и имени пользователя и пароля.

## <a name="redirect-uris"></a>URI перенаправления

Еще раз URI для использования классического приложения для перенаправления будет зависеть от последовательности, которую вы хотите использовать.

- Если вы используете **интерактивной проверки подлинности** или **поток кода устройства**, вы захотите использовать `https://login.microsoftonline.com/common/oauth2/nativeclient`. Вы обеспечите этой конфигурации, щелкнув соответствующий URL-адрес в **проверки подлинности** раздел для вашего приложения
  
  > [!IMPORTANT]
  > Сегодня MSAL.NET использует другой URI перенаправления по умолчанию в настольных приложениях, выполняемых на Windows (`urn:ietf:wg:oauth:2.0:oob`). В будущем мы захотите изменить это значение по умолчанию, и поэтому мы рекомендуем использовать `https://login.microsoftonline.com/common/oauth2/nativeclient`

- Если приложение использует только Windows интегрированной проверки подлинности, имя пользователя и пароль, не требуется для регистрации URI перенаправления для приложения. В самом деле эти потоки сделать обращение к конечной точке версии 2.0 платформы Microsoft identity, и приложение не обратный вызов для любого конкретного кода URI. 
- Чтобы отличить потока кода устройства, встроенная проверка подлинности Windows и имя пользователя и пароль из потока конфиденциального клиента приложения, которые не обеспечивают перенаправления идентификаторы URI либо (поток учетных данных клиента используется в приложениях управляющей программы), вам потребуется express что приложения — это приложение общедоступным клиентом. Эта конфигурация достигается за счет будет **проверки подлинности** раздел для вашего приложения, а также в **Дополнительные параметры** подраздел, выберите **Да**, на вопрос **Считать приложение общедоступным клиентом** (в **тип клиента по умолчанию** абзаца)

  ![Разрешить общедоступного клиента](media/scenarios/default-client-type.png)

## <a name="api-permissions"></a>Разрешения API

Классические приложения вызовите API от имени выполнившего вход пользователя. Они должны запрашивать делегированные разрешения. Они не могут запрашивать разрешения приложения (который обрабатываются только в [приложений управляющей программы](scenario-daemon-overview.md))
