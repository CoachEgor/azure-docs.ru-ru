Узнайте все необходимое для создания управляющее приложение, которое вызывает веб-API.

## <a name="prerequisites"></a>Технические условия

[!INCLUDE [Pre-requisites](../../../includes/active-directory-develop-scenarios-prerequisites.md)]

## <a name="overview"></a>Обзор

Приложение может получить токен для вызова веб-API от имени самого (не от имени пользователя). Этот сценарий полезен для приложений управляющей программы. Она использует стандартные OAuth 2.0 [учетные данные клиента](v2-oauth2-client-creds-grant-flow.md) предоставить.

![Управляющая программа](./media/scenario-daemon-app/daemon-app.svg)

Ниже приведены некоторые примеры вариантов использования для приложений управляющей программы:

- Веб-приложения, которые используются для подготовки или управлять пользователями, или пакетного процесса в каталоге
- Такими настольными приложениями, (службы windows в Windows) или управляющие программы процессы на платформе Linux, которые выполняют задания, или служба операционной системы в фоновом режиме
- Веб-API, необходимые для работы с каталогами, не определенных пользователей

Есть еще один стандартный пример, где приложения неуправляющих использовать учетные данные клиента: даже в том случае, если они действовать от имени пользователей, необходимые для доступа к веб-API или ресурс с помощью учетных данных по техническим причинам. Например, доступ к секретам в хранилище ключей или к базе данных Azure SQL для кэша.

Приложения, получить маркер для их удостоверений:

- — Это конфиденциального клиента приложения. Такие приложения, учитывая, что при доступе к ресурсам независимо от пользователя, должны подтверждать свою личность. Они также довольно важные приложения, которые они должны быть утверждены администраторами клиента Azure Active Directory (Azure AD).
- Зарегистрирована секрет (пароль приложения или сертификат) с помощью Azure AD. Этот секрет переданное при вызове Azure AD для получения маркера.

## <a name="specifics"></a>Особенности

> [!IMPORTANT]
>
> - С управляющей программой невозможно взаимодействие с пользователем. Управляющей программе требуется свой собственный удостоверение. Приложение данного типа запрашивает маркер доступа с помощью своего удостоверения и предоставляет его идентификатор приложения, учетные данные (пароль или сертификат) и приложения идентификатор URI в Azure AD. После успешной проверки подлинности управляющая программа получает маркер доступа (и маркер обновления) из конечной платформы удостоверений Microsoft, который затем используется для вызова веб-API (и обновляется по мере необходимости).
> - Так как взаимодействие с пользователем не поддерживается, добавочное согласие будет невозможно. Все необходимые разрешения API должны быть настроены при регистрации приложения и код приложения просто запрашивает статически определенные разрешения. Это также означает, что управляющая программа приложения не поддерживают добавочное согласие.

Для разработчиков взаимодействие end-to-end для этого сценария имеет следующие аспекты:

- Использование приложений управляющей программы работают только в клиентах Azure AD. Он не имеет смысла для построения приложения управляющей программы, который пытается обработать личные учетные записи Майкрософт. Если вы разработчик бизнес-приложения (LOB), вы создадите приложение управляющей программы в вашем клиенте. Если вы являетесь независимым поставщиком по, может потребоваться создать управляющей программе несколькими клиентами. Она должна быть дано администратором каждого клиента
- Во время [Регистрация приложения](./scenario-daemon-app-registration.md), **URI ответа** не требуется. Вам нужно совместно использовать секреты или сертификаты с помощью Azure AD, а вам нужно запросить разрешения приложений и предоставления согласия администратора, чтобы использовать эти разрешения приложения.
- [Конфигурации приложения](./scenario-daemon-app-configuration.md) должен предоставить учетные данные клиента, общего доступа с помощью Azure AD во время регистрации приложения.
- [Область](scenario-daemon-acquire-token.md#scopes-to-request) используется, чтобы получить маркер потока должна быть статической области действия учетных данных клиента.
