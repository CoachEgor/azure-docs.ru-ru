# <a name="the-new-azure-portal-app-registration-experience"></a>Новый опыт регистрации приложений портала Azure

На портале Azure есть много улучшений в новом опыте [регистрации приложений.](https://go.microsoft.com/fwlink/?linkid=2083908) Если вы больше знакомы с опытом регистрации приложений (apps.dev.microsoft.com) для регистрации или управления конвергентными приложениями, именуемым старым опытом, это руководство по обучению поможет вам начать использовать новый опыт.

## <a name="whats-not-changing"></a>Что не меняется?

- Ваши приложения и связанные с ними конфигурации можно найти как-это в новом опыте. Вам не нужно регистрировать приложения снова, и пользователям ваших приложений не нужно будет снова входить в систему.

    > [!NOTE]
    > Для того, чтобы найти их на портале Azure, необходимо войти в учетную запись, которая использовалась для регистрации приложений. Мы рекомендуем вам проверить, что пользователь, зарегистрированный на портале Azure, соответствует пользователю, который был зарегистрирован на портале регистрации приложений, сравнивая адрес электронной почты из вашего профиля.
    >
    > В некоторых случаях, особенно при входе в систему с использованием личных учетных записей Майкрософт (например, Outlook, Live, Xbox и т.д.) с адресом электронной почты Azure AD, мы выяснили, что, когда вы заходите на портал Azure из старого опыта, вы попадаете в другую учетную запись с тем же электронной почте в вашем арендаторе Azure AD. Если вы все еще считаете, что ваши приложения отсутствуют, ввешите в систему и вопийте с правильной учетной записью.

- Приложения Live SDK, созданные с использованием личных учетных записей Майкрософт, пока не поддерживаются на портале Azure и будут оставаться в прежнем опыте в ближайшем будущем.

## <a name="key-changes"></a>Ключевые изменения

-   В старом опыте приложения по умолчанию регистрировались как конвергентные приложения, поддерживающие все организационные учетные записи (мультитенанты), а также личные учетные записи Майкрософт. Это не может быть изменено с помощью старого опыта, что затрудняет создание приложений, которые поддерживают только организационные учетные записи (либо мультитенанты или один арендатор).
    Новый опыт позволяет регистрировать приложения, поддерживающие все эти параметры. [Подробнее о типах приложений.](active-directory-v2-registration-portal.md)

-   В новом опыте, если ваша личная учетная запись Майкрософт также находится в ad-арендаторе Azure AD, вы увидите три вкладки - все приложения в арендаторе, принадлежащие приложениям в арендаторе, а также приложения мнёс. Таким образом, если вы считаете, что приложения, зарегистрированные в вашей личной учетной записи Майкрософт, отсутствуют, проверьте **приложения из вашей личной учетной записи.**

-   В новом опыте вы можете легко переключаться между арендаторами, перемещаясь по профилю и выбирая каталог коммутатора.

## <a name="list-of-applications"></a>Список приложений

-   В новом списке приложений показаны приложения, зарегистрированные через устаревший опыт регистрации приложений на портале Azure (приложения, которые подписываются только в учетных записях Azure AD), а также приложения, зарегистрированные на [портале регистрации приложений](https://apps.dev.microsoft.com/) (приложения, которые регистрируются как в Azure AD, так и в личных учетных записях Майкрософт).

-   Новый список приложений имеет два дополнительных столбца: **Созданный на** столбце и сертификаты & столбец **секретов,** который показывает состояние (текущий, истекающий в ближайшее время или истек) учетных данных, которые были зарегистрированы в приложении.

## <a name="new-app-registration"></a>Регистрация нового приложения

В старом опыте, чтобы зарегистрировать конвергентное приложение, вам требовалось только указано имя. Созданные приложения были зарегистрированы как конвергентные приложения, поддерживающие все организационные каталоги (мультитенанты), а также личные учетные записи Майкрософт.  Это не может быть изменено с помощью старого опыта, что затрудняет создание приложений, которые поддерживают только организационные учетные записи (либо мультитенанты или один арендатор). [Подробнее о типах поддерживаемых учетных записей](v2-supported-account-types.md)

В новом опыте необходимо узнать имя приложения и выбрать типы аккаунтов Поддержки. Вы можете дополнительно предоставить перенаправление URI.
Если вы предоставляете перенаправление URI, вам нужно указать, если это веб/общественный (родной / мобильный и настольный). Для получения дополнительной информации о том, как зарегистрировать приложение, используя новый опыт регистрации приложений, смотрите [этот quickstart](quickstart-register-app.md).

## <a name="app-management-page"></a>Страница управления приложениями

Старый опыт имел одну страницу управления приложениями для конвергентных приложений со следующими разделами: Свойства, Секреты приложений, Платформы, Владельцы, Разрешения Награфика Майкрософт, Профиль и Расширенные опции.

Новый опыт на портале Azure представляет эти функции на отдельные страницы. Здесь вы можете найти эквивалентную функциональность:

-   Свойства - Имя и идентификатор приложения на странице Обзор.

-   Секреты приложения на странице Сертификаты & секреты

-   Конфигурация платформ находится на странице аутентификации

-   Разрешения Microsoft Graph на странице разрешений API наряду с другими разрешениями

-   Профиль на странице Брендинга

-   Расширенный вариант - Поддержка Live SDK находится на странице аутентификации.

## <a name="application-secretscertificates--secrets"></a>Секреты применения/Сертификаты & секреты

В новом опыте **секреты приложения** были переименованы в **Сертификаты & секреты.** Кроме того, **открытые ключи** называютсертификат **сертификатами,** а **пароли** — **секретами клиента.** Мы решили не приносить эту функциональность вместе в новом опыте по соображениям безопасности, следовательно, вы больше не можете создавать новую пару ключей.

## <a name="platformsauthentication-reply-urlsredirect-uris"></a>Платформы/Аутентификация: Ответ URL-адреса/перенаправление URIs
В старом опыте в приложении был раздел Платформ для Web, native и Web API для настройки URL-адресов Redirect, URL-адресов logout и implicit flow.

В новом опыте URL-адреса Reply можно\'найти в разделе аутентификации приложения. Кроме того, они называются перенаправлением URIs, и формат перенаправления УРИ изменился. Они должны быть связаны с типом приложения (веб или публичный клиент - мобильный и настольный). [Дополнительные сведения](quickstart-configure-app-access-web-apis.md#add-redirect-uris-to-your-application)

Web-aPI настроены на странице Expose aPI.

> [!NOTE]
> Ознакомьтесь с новым интерфейсом для настройки проверки подлинности, который позволяет указать для приложения параметры по критериям целевой платформы и (или) устройства. [Дополнительные сведения](quickstart-configure-app-access-web-apis.md#configure-platform-settings-for-your-application)

## <a name="microsoft-graph-permissionsapi-permissions"></a>Разрешения Microsoft Graph/API-разрешения

-   При выборе API в старом опыте можно выбрать только из API Microsoft Graph. В новом опыте вы можете выбрать из многих AA-интерфейсов Microsoft, включая Microsoft Graph, AA и AA из вашей организации и Ваши AA, это представлено в трех вкладках: AIS Майкрософт, AA, AA, которые использует моя организация, или Мои AA. Панель поиска на AIS, которая использует поиск вкладок, через принципы службы в арендаторе.

    > [!NOTE]
    > Вы не увидите эту вкладку, если приложение не связано с арендатором. Для получения дополнительной информации о том, как запрашивать разрешения с помощью нового опыта, смотрите [этот quickstart](https://github.com/MicrosoftDocs/azure-docs/blob/master/articles/active-directory/develop/quickstart-configure-app-access-web-apis.md).

-   В старом опыте не было **кнопки разрешения гранта.** В новом опыте в разделе API разрешений aPI-приложения есть раздел согласия Гранта с кнопкой **согласия админа Гранта.** Только админ может дать согласие, и эта кнопка включена только для админантов. Когда админ выбирает кнопку **согласия админ-консультанта Гранта,** согласие админполучит все запрошенные разрешения.

## <a name="profile"></a>Профиль
В старом опыте у Profile был логотип, URL-адрес главной страницы, URL-адрес «Условия обслуживания» и конфигурация URL-адреса заявления о конфиденциальности. В новом опыте, они можно найти на странице брендинга.

## <a name="application-manifest"></a>Манифест приложения
В новом опыте страница Manifest позволяет отсеивать и обновлять атрибуты приложения. Дополнительные сведения см. в статье [Манифест приложения Azure Active Directory](reference-app-manifest.md).

## <a name="new-ui"></a>Новый uI
Существует новый ui ii для свойств, которые ранее можно было установить только с помощью редактора манифеста или API, или не существовало.

-   Неявный поток грантов (oauth2AllowImplicitFlow) можно найти на странице аутентификации. В отличие от старого опыта, вы можете включить маркеры доступа или маркеры ID, или оба.

-   Области, определенные этим API (oauth2Permissions) и авторизованные клиентские приложения (preAuthorizedApplications), могут быть настроены через страницу Expose aPI. Для получения дополнительной информации о том, как настроить приложение в веб-API и разоблачить разрешения / области, смотрите [этот quickstart](quickstart-configure-app-expose-web-apis.md).

-   Домен издателя (который отображается пользователям по [запросу согласия\'приложения)](application-consent-experience.md)можно найти на странице Branding blade. Для получения дополнительной информации о том, как настроить домен издателя, [см.](howto-configure-publisher-domain.md)

## <a name="limitations"></a>Ограничения

Новый опыт имеет следующие ограничения:

-   Новый опыт пока не поддерживает регистрацию приложений для клиентов Azure AD B2C.

-   Новый опыт пока не поддерживает приложения Live SDK, созданные с помощью личных учетных записей Майкрософт.

-   Изменение значения для поддерживаемых учетных записей не поддерживается в uI. Вам нужно использовать манифест приложения, если вы\'не переключитесь между одним арендатором Azure AD и мультитенантом.

   > [!NOTE]
   > Если вы являетесь личным пользователем учетной записи Майкрософт в ad-арендаторе Azure, а админ-арендатор ограничил доступ к порталу Azure, доступ может быть отказано. Однако, если вы пройдете через ярлык, введя регистрацию приложений в панели поиска или прикрепив его, вы сможете получить доступ к новому опыту.
