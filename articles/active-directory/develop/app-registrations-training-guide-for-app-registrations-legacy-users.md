# <a name="new-azure-portal-app-registration-training-guide"></a>Новое учебное руководство по регистрации приложений на портале Azure

Вы можете найти много улучшений в новом опыте [регистрации приложений](https://go.microsoft.com/fwlink/?linkid=2083908) на портале Azure. Если вы знакомы с опытом регистрации приложений (наследие) на портале Azure, используйте это руководство по обучению, чтобы начать использовать новый опыт.

В Active Directory Azure, новый опыт регистрации приложений, описанный здесь, как правило, доступен (GA). В Azure Active Directory B2C (Azure AD B2C) этот опыт находится в предварительном просмотре.

## <a name="key-changes"></a>Ключевые изменения

- Регистрация приложений не ограничивается тем, что является *ни веб-приложением/API,* ни *нативным* приложением. Вы можете использовать одну и ту же регистрацию приложений для всех этих приложений, зарегистрировав соответствующие URL-е УРИ.

- Устаревший опыт поддерживал приложения, которые ввелись, используя только организационные учетные записи (Azure AD). Приложения были зарегистрированы как одноквартирные. Приложения поддерживали только организационные учетные записи из каталога, в который было зарегистрировано приложение. Приложения могут быть изменены, чтобы быть мультитенантными и поддерживать все организационные учетные записи. Новый опыт позволяет регистрировать приложения, которые могут поддерживать как эти варианты, так и третий вариант: все организационные учетные записи, а также личные учетные записи Майкрософт.

- Устаревший опыт был доступен только при подписании на портале Azure с использованием организационной учетной записи. С новым опытом можно использовать личные учетные записи Майкрософт, которые не связаны с каталогом.

## <a name="list-of-applications"></a>Список приложений

В новом списке приложений отображаются приложения, зарегистрированные через устаревший опыт регистрации приложений на портале Azure. Эти приложения вписываются с помощью учетных записей Azure AD. В новом списке приложений также отображает приложения, зарегистрированные на портале регистрации приложений. Эти приложения вписываются с помощью Azure AD и личных учетных записей Майкрософт.

>[!NOTE]
>Портал регистрации заявлений был унизан.

В новом списке приложений нет **столбца типа приложения,** поскольку регистрация одного приложения может быть несколько типов. Список имеет два дополнительных столбца: **Создано** и **сертификаты & секреты**. **Сертификаты & секреты** показывают состояние учетных данных, которые были зарегистрированы в приложении. Статусы включают **текущий,** **истекающий в ближайшее время,** и **истек.**

## <a name="new-app-registration"></a>Регистрация нового приложения

В устаревшем опыте необходимо зарегистрировать приложение, которое необходимо предоставить: **Name,** **Тип приложения**и **Регистрация URL/Redirect URI.** Созданные приложения были только приложениями САУ с одним арендатором. Они поддерживали только организационные учетные записи из каталога, в котором было зарегистрировано приложение.

В новом опыте необходимо узнать **имя** приложения и выбрать **типы учетной записи Поддержки.** Вы можете дополнительно предоставить **перенаправление URI**. Если вы предоставляете перенаправление URI, вам нужно указать, является ли это веб/общественность (мобильный и настольный). Для получения дополнительной информации [см.](quickstart-register-app.md) Для Azure AD B2C [см.](../../active-directory-b2c/tutorial-register-applications.md)

## <a name="differences-between-the-application-registration-portal-and-app-registrations-page"></a>Различия между порталом регистрации приложений и страницей регистрации приложений

### <a name="the-legacy-properties-page"></a>Страница «Наследие свойства»

Наследие опыт был **Свойства** страницы. **Свойства** имели следующие поля:

- **Название**
- **Идентификатор объекта**
- **Идентификатор приложения**
- **URI кода приложения**
- **Логотип**
- **URL-адрес домашней страницы**
- **URL-адрес выхода.**
- **URL-адрес услуг**
- **URL-адрес заявления о конфиденциальности**
- **Тип приложения**
- **Мультитенант**

Новый опыт не имеет этой страницы. Здесь вы можете найти эквивалентную функциональность:

- **Имя,** **логотип,** **URL-адрес главной страницы,** **URL-адрес службы**и **URL-адрес заявления о конфиденциальности** теперь находятся на странице **брендинга** приложения.
- **Идентификатор объекта** и **идентификатор приложения (клиента)** находятся на странице **Обзора.**
- Функциональность, контролируемая **переключателем Multi-tenant** в устаревшем опыте, была заменена **типами поддерживаемых учетных записей** на странице **аутентификации.** Для получения дополнительной информации [см.](quickstart-modify-supported-accounts.md)
- **URL-адрес входа** теперь находится на странице **аутентификации.**
- **Тип приложения** больше не является действительным полем. Вместо этого перенаправьте УРИ, которые вы можете найти на странице **аутентификации,** определите, какие типы приложений поддерживаются.
- **App ID URI** теперь называется **Application ID URI,** и вы можете найти его на **Expose API.** В наследном опыте это свойство было `https://{tenantdomain}/{appID}`автоматически зарегистрировано в следующем формате: , например, `https://microsoft.onmicrosoft.com/492439af-3282-44c3-b297-45463339544b`. В новом опыте, он автоматически `api://{appID}`генерируется как , но он должен быть явно сохранены. В клиентах Azure AD B2C `https://{tenantdomain}/{appID}` формат по-прежнему используется.

### <a name="reply-urlsredirect-urls"></a>Ответ URL-адреса / перенаправить URls

В устаревшем опыте у приложения была страница **URL-адресов Reply.** В новом опыте URL-адреса ответов можно найти на странице **аутентификации** приложения. Теперь они называются **перенаправить URIs**.

Изменился формат перенаправления URIs. Они должны быть связаны с типом приложения, веб или общественности. По соображениям безопасности `http://` подстановочные знаки и *http://localhost*схемы не поддерживаются, за исключением .

### <a name="keyscertificates--secrets"></a>Ключи/сертификаты & секреты

В унаследованном опыте, app имел страницу **ключей.** В новом опыте, он был переименован в **Сертификаты & секреты**.

**Открытые ключи** теперь называют **сертификатами.** **Пароли** теперь **называются секретами клиента.**

### <a name="required-permissionsapi-permissions"></a>Требуемые разрешения/разрешения API

В устаревшем опыте у приложения была страница **требуемых разрешений.** В новом опыте он был переименован в **разрешения API.**

При выборе API в устаревшем опыте можно выбрать небольшой список API Майкрософт. Вы также можете искать через директора службы в арендатора. В новом опыте вы можете выбрать один из нескольких вкладок: **Microsoft AIS**, **AA, AAP, которые использует моя организация,** или **Мои AA.** Панель поиска на **AIS, которая использует поиск** вкладок, через принципы службы в арендаторе.

> [!NOTE]
> Вы не увидите эту вкладку, если приложение не связано с арендатором. Для получения дополнительной информации о [Quickstart: Configure a client application to access web APIs](quickstart-configure-app-access-web-apis.md)том, как запрашивать разрешения, см.

В устаревшем опыте была кнопка **разрешений гранта** в верхней части **страницы Запросированных разрешений.** В новом опыте на странице **согласия Гранта** есть кнопка согласия на получение согласия на получение **гранта в** разделе **Разрешений API** приложения. Есть также некоторые различия в способах функции кнопок.

В устаревшем опыте логика менялась в зависимости от подписанного пользователя и запрашиваемых разрешений. Логика была:

- Если запрашивались только разрешения, способные к согласию пользователя, а подписанный в пользователе не был админом, пользователь мог бы предоставить пользователю согласие на запрашиваемые разрешения.
- Если было запрошено хотя бы одно разрешение, требующее согласия админа, и подписанный в пользователе не был админом, пользователь получил ошибку при попытке дать согласие.
- Если подписанный в пользователе был админ, согласие админ было предоставлено для всех запрошенных разрешений.

В новом опыте только админ может дать согласие. Когда админ выбирает **согласие админа Гранта,** согласие админ-запроса предоставляется всем запрошенным разрешениям.

## <a name="deleting-an-app-registration"></a>Исключение регистрации приложения

В устаревшем опыте можно удалить только приложения с одним арендатором. Кнопка удаления была отключена для мультитенантных приложений. В новом опыте вы можете удалить приложения в любом состоянии, но вы должны подтвердить действие. Для получения дополнительной информации [см.](quickstart-remove-app.md)

## <a name="new-ui"></a>Новый uI

Новый опыт добавляет элементы управления интерфейсом для следующих свойств:

- Страница **аутентификации** имеет **неявный поток грантов** ().`oauth2AllowImplicitFlow` В отличие от устаревшего опыта, вы можете включить **токены Access** или **id маркеры**или и то, и другое.
- Страница **Expose aPI** содержит **области, определенные этим API** ( )`oauth2Permissions`и **авторизованные клиентские приложения** ().`preAuthorizedApplications` Для получения дополнительной информации о том, как настроить приложение в веб-API и разоблачить разрешения / области, [см.](quickstart-configure-app-expose-web-apis.md)
- Страница **брендинга** содержит **домен Издателя.** Домен издателя отображается пользователям по [запросу согласия приложения.](application-consent-experience.md) Для получения дополнительной информации [см.](howto-configure-publisher-domain.md)

## <a name="limitations"></a>Ограничения

Новый опыт имеет следующие ограничения:

- Формат секретов клиентов (пароли приложений) отличается от формата устаревшего опыта и может нарушить CLI.
- Изменение значения для поддерживаемых учетных записей не поддерживается в uI. Вам нужно использовать манифест приложения, если вы не переключаетесь между одним арендатором Azure AD и мультитенантом.
