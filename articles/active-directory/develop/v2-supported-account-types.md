# <a name="supported-account-types"></a>Поддерживаемые типы учетных записей

В этой статье объясняется, какие типы учетных записей (иногда именуемых аудиторий) поддерживаются в приложениях.

## <a name="supported-accounts-types-in-microsoft-identity-platform-applications"></a>Поддерживаемые типы учетных записей в приложениях платформы удостоверений Майкрософт

В общедоступном облаке Microsoft Azure в большинстве типов приложений можно войти в систему с любой аудиторией:

- При написании бизнес-приложения (LOB) можно организовать вход в систему для пользователей вашей организации. Такое приложение иногда называют приложением для **одного клиента**.
- Независимый разработчик ПО может написать приложение для входа в систему:

  - В любой организации. Такое приложение называется веб-приложением **для нескольких клиентов**. Иногда войти в систему можно с помощью рабочих или учебных учетных записей пользователей.
  - Используя рабочую, учебную либо личную учетную запись Майкрософт.
  - Используя личные учетные записи Майкрософт.
    > [!NOTE]
    > В настоящее время платформа удостоверений Microsoft поддерживает личные учетные записи Майкрософт только путем регистрации приложения для **рабочих, учебных или личных учетных записей Майкрософт**. В коде приложения возможности входа можно ограничить, указав при создании приложения центр Azure AD, например`https://login.microsoftonline.com/consumers`.

- При написании бизнес-приложения для потребителей также можно войти в систему с помощью учетных данных пользователей в социальных сетях, используя Azure AD B2C.

## <a name="certain-authentication-flows-dont-support-all-the-account-types"></a>Некоторые процедуры проверки подлинности не поддерживают все типы учетных записей

Некоторые типы учетных записей нельзя использовать с определенными процедурами проверками подлинности. Например, на настольных ПК, в UWP-приложениях или управляющих программах:

- Управляющие программы можно использовать только с организациями Azure Active Directory. Не стоит использовать управляющие программы для управления личными учетными записями Майкрософт (администратор никогда не даст на это разрешение).
- Можно использовать встроенную проверку подлинности Windows с помощью рабочих или учебных учетных записей (в вашей организации или любой другой организации). Действительно, встроенная проверка подлинности Windows совместима с учетными записями домена при подключении компьютеров к домену или Azure AD. Этот поток не нужен для личных учетных записей Майкрософт.
- [Предоставление пароля владельца ресурса](./v2-oauth-ropc.md) (имя пользователя / пароль) нельзя использовать с личными учетными записями Microsoft. На самом деле личные учетные записи Майкрософт требуют, чтобы пользователь соглашался предоставить доступ к личным ресурсам при каждом входе в систему. Вот почему эти действия несовместимы с неинтерактивными потоками.
- Поток кода устройства еще не совместим с личными учетными записями Майкрософт.

## <a name="supported-account-types-in-national-clouds"></a>Поддерживаемые типы учетных записей в национальных облаках

 С помощью приложений также можно войти в системы в [национальных облаках](authentication-national-cloud.md). Однако личные учетные записи Майкрософт в этих облаках не поддерживаются (определениями этих облаков). Вот почему поддерживаемые типы учетных записей для этих облаков настроены или для одной организации (для одного клиента) или многих организаций (приложения для нескольких клиентов).
