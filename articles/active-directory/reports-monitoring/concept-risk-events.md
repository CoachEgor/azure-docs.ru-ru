# <a name="azure-active-directory-risk-detections"></a>Обнаружение рисков активных каталогов Azure

Подавляющее большинство нарушений безопасности — это случаи, когда злоумышленники получают доступ к среде за счет кражи удостоверения пользователя. Обнаружение скомпрометированных удостоверений — непростая задача. Azure Active Directory использует адаптивные алгоритмы машинного обучения и эвристические методы, чтобы обнаруживать подозрительные действия, связанные с вашими учетными записями. Каждое обнаруженное подозрительное действие хранится в записи, называемой **обнаружением риска.**

Есть два места, где вы просматриваете сообщения об обнаружении рисков:

 - **Отчетность Azure AD** - Обнаружение рисков является частью отчетов безопасности Azure AD. Дополнительные сведения см. в [отчете системы безопасности о пользователях под угрозой](concept-user-at-risk.md) и [отчете системы безопасности о входах, представляющих риск](concept-risky-sign-ins.md).

 - **Защита идентификационных данных Azure AD** - Обнаружение рисков также является частью возможностей отчетности [Active Directory Protection.](../active-directory-identityprotection.md)

Кроме того, можно использовать API обнаружения [рисков защиты идентификационных данных](https://developer.microsoft.com/graph/docs/api-reference/beta/resources/identityriskevent) для получения программного доступа к обнаружению безопасности с помощью Microsoft Graph. Дополнительные сведения см. в статье [Начало работы с защитой идентификации Azure Active Directory и Microsoft Graph](../identity-protection/graph-get-started.md). 

В настоящее время Active Directory Azure обнаруживает шесть типов обнаружения рисков:

- [Пользователи с утерянными учетными данными](#leaked-credentials) 
- [Входы в систему с анонимных IP-адресов](#sign-ins-from-anonymous-ip-addresses) 
- [Невозможно переместиться в нетипичные расположения](#impossible-travel-to-atypical-locations) 
- [Входы в систему с зараженных устройств](#sign-ins-from-infected-devices) 
- [Входы с IP-адресов с подозрительными действиями](#sign-ins-from-ip-addresses-with-suspicious-activity) 
- [Входы в систему из необычного расположения](#sign-in-from-unfamiliar-locations) 

![Обнаружение рисков](./media/concept-risk-events/91.png)

> [!IMPORTANT]
> Иногда вы можете обнаружить риск без соответствующей записи входа в отчет о [входах.](concept-sign-ins.md) Это связано с тем, что защита удостоверений оценивает риски как для **интерактивных**, так и для **неинтерактивных** входов, тогда как в отчете о входе отображаются только интерактивные входы.

Понимание обнаружения обнаруженного риска связано с подпиской Azure AD. 

* С помощью **выпуска Azure AD Premium P2** вы можете получить самые подробные сведения обо всех основных обнаружениях. 
* С **изданием Azure AD Premium P1**расширенные обнаружения (например, незнакомые свойства входной системы) не подпадают под действие вашей лицензии и будут отображаться под именем **Входе с обнаруженным дополнительным риском.** Кроме того, скрыты поля уровня риска и детализации риска.

Хотя обнаружение обнаружений рисков уже представляет собой важный аспект защиты идентификационных данных, у вас также есть возможность либо вручную решить их, либо внедрить автоматизированные ответы путем настройки политик условного доступа. Дополнительные сведения см. в статье [Защита идентификации Azure Active Directory](../active-directory-identityprotection.md).

## <a name="risk-detection-types"></a>Типы обнаружения рисков

Свойство **типа обнаружения рисков** является идентификатором подозрительного действия, для создания записи обнаружения риска.

Постоянное внимание корпорации Майкрософт к развитию процесса обнаружения позволило достичь таких результатов:

- Улучшение точности обнаружения существующих рисков 
- Новые типы обнаружения рисков, которые будут добавлены в будущем

### <a name="leaked-credentials"></a>Утерянные учетные данные

Киберпреступники зачастую предоставляют общий доступ к украденным ими действительным учетным данным законных пользователей. Обычно они публикуют их в теневом Интернете или сайтах для киберпреступников, либо продают их на теневом рынке. Служба Майкрософт, занимающаяся отслеживанием утечек учетных данных, узнает такие пары имен пользователя и паролей, обращаясь к таким общедоступным и теневым веб-сайтам, а также сотрудничая со следующими сторонами:

- исследователи;
- правоохранительные органы;
- группы безопасности в корпорации Майкрософт;
- другие доверенные источники. 

Когда служба получает пару "имя пользователя/пароль", они проходят проверку на соответствие текущим действительным учетным данным пользователей AAD. При обнаружении совпадения это означает, что пароль пользователя был скомпрометирован, и создается **обнаружение риска утечки учетных данных.**

### <a name="sign-ins-from-anonymous-ip-addresses"></a>Попытки входа с анонимных IP-адресов

Этот тип обнаружения рисков идентифицирует пользователей, которые успешно зарегистрировались с IP-адреса, который был идентифицирован как анонимный IP-адрес прокси. Эти прокси-серверы используют пользователи, желающие скрыть IP-адреса своих устройств. Иногда их используют злоумышленники.

### <a name="impossible-travel-to-atypical-locations"></a>Невозможно переместиться в нетипичные расположения

Этот тип обнаружения рисков идентифицирует два всходного, происходящих из географически удаленных мест, где по крайней мере одно из мест также может быть нетипичным для пользователя, учитывая прошлое поведение. Помимо других факторов, этот алгоритм машинного обучения учитывает промежуток времени между двумя входами, и время перехода пользователя из одного расположения в другое. Это значит, что другой пользователь использует те же учетные данные.

Этот алгоритм игнорирует очевидные ложные срабатывания, такие как VPN и расположения, которые постоянно используют другие пользователи в организации, и позволяет получить результаты, соответствующие невозможным условиям перемещения. В системе предусмотрен первоначальный период обучения — 14 дней, в течение которых она изучает поведение нового пользователя при входе. 

### <a name="sign-in-from-unfamiliar-locations"></a>Попытки входа из неизвестных расположений

Этот тип обнаружения риска учитывает прошлые места регистрации (IP, Latitude / Longitude и ASN) для определения новых/ незнакомых местоположений. Система хранит сведения о предыдущих расположениях входа пользователя и учитывает их в качестве знакомых. Обнаружение риска срабатывает при входе в систему из места, которого еще нет в списке знакомых местоположений. В системе предусмотрен первоначальный период обучения — 30 дней, в течение которых она не помечает новые расположения в качестве незнакомых. Система также игнорирует вход с помощью знакомых устройств и из расположений, географически близких к знакомому. 

Служба защиты идентификации обнаруживает входы из незнакомых расположений также с помощью обычной проверки подлинности или устаревших протоколов. Так как эти протоколы не оснащены привычными современными функциями, такими как идентификатор клиента, данных телеметрии недостаточно для сокращения количества ложноположительных результатов. Чтобы уменьшить количество обнаруженных обнаруженных обнаружений рисков, следует перейти к современной аутентификации.   

> [!NOTE]
> Если имя пользователя и пароль входа не совпадают, вход не сработает, а обнаружение риска не происходит. Вход в систему из незнакомого обнаружения риска местоположения срабатывает только на успешном логинах.

### <a name="sign-ins-from-infected-devices"></a>Попытки входа с инфицированных устройств

Этот тип обнаружения риска идентифицирует входе с устройств, зараженных вредоносными программами, которые, как известно, активно общаются с сервером ботов. Определение этих событий осуществляется путем сопоставления IP-адресов устройства пользователя с IP-адресами, которые использовались для связи с таким сервером. 

### <a name="sign-ins-from-ip-addresses-with-suspicious-activity"></a>Попытки входа с IP-адресов с подозрительными действиями
Этот тип обнаружения рисков определяет IP-адреса, с которых в течение короткого периода времени было замечено большое количество неудачных попыток ввода в систему в нескольких учетных записях пользователей. Такой шаблон использования IP-адресов характерен для злоумышленников и является признаком того, что учетная запись скомпрометирована или ее могут скомпрометировать. Этот алгоритм машинного обучения игнорирует очевидные ложные срабатывания, например, если IP-адреса часто используются другими корпоративными пользователями.  В системе предусмотрен первоначальный период обучения — 14 дней, в течение которых она изучает поведение нового пользователя и нового клиента при входе.

## <a name="risk-level"></a>Уровень риска

Свойство уровня риска обнаружения риска является индикатором **(High,** **Medium,** или **Low**) для тяжести и уверенности в обнаружении риска. Его можно использовать для расстановки приоритета действий, которые необходимо предпринять. 

Тяжесть обнаружения риска представляет силу сигнала как предиктора компромисса идентичности. Вероятность указывает на возможность возникновения ложных срабатываний. 

Например, 

* **Высокая**: Высокая достоверность и высокая степень серьезности обнаружения риска. Эти события являются достоверным показателем того, что удостоверение пользователя скомпрометировано, и для каждой такой учетной записи следует безотлагательно принять меры по исправлению.

* **Средний**: Высокая степень тяжести, но более низкий риск доверия обнаружения, или наоборот. Эти события потенциально опасны, и для каждой затронутой учетной записи пользователя следует принять меры по исправлению.

* **Низкий**: Низкая достоверность и низкая степень обнаружения риска тяжести. Это событие может не требовать немедленного действия, но в сочетании с другими обнаружениями рисков может дать четкое представление о том, что личность скомпрометирована.

![Уровень риска](./media/concept-risk-events/01.png)

### <a name="leaked-credentials"></a>Утерянные учетные данные

Обнаружение риска утечки учетных данных классифицируется как **High,** поскольку они дают четкое представление о том, что имя пользователя и пароль доступны злоумышленнику.

### <a name="sign-ins-from-anonymous-ip-addresses"></a>Попытки входа с анонимных IP-адресов

Уровень риска для этого типа обнаружения риска является **средним,** поскольку анонимный IP-адрес не является убедительным признаком компрометации учетной записи. В таком случае рекомендуем сразу же связаться с пользователем, чтобы проверить, использовал ли он анонимный IP-адрес.


### <a name="impossible-travel-to-atypical-locations"></a>Невозможно переместиться в нетипичные расположения

Невозможность такого перехода — обычно достоверный показатель того, что в учетную запись выполнил вход злоумышленник. Однако могут возникать и ложные срабатывания, если пользователь во время поездки использовал новое устройство или сеть VPN, которой обычно не пользуются другие пользователи в организации. Другой источник ложных срабатываний — приложения, которые неправильно передают IP-адрес сервера в качестве клиентского IP-адреса, тогда вход может расцениваться как вход из центра обработки данных, где размещен сервер приложения (такие ситуации часто возникают с приложениями в центрах обработки данных Майкрософт, когда при входе адрес распознается как IP-адрес, принадлежащий корпорации Майкрософт). В результате этих ложных срабатываний уровень риска для этого обнаружения риска является **средним.**

> [!TIP]
> Можно уменьшить количество ложных срабатываний для этого типа обнаружения рисков путем настройки [названных местоположений.](../active-directory-named-locations.md) 

### <a name="sign-in-from-unfamiliar-locations"></a>Попытки входа из неизвестных расположений

Вход из неизвестного расположения может явно указывать на то, что злоумышленник использует украденное удостоверение. Ложные срабатывания могут возникать, если пользователь путешествует, а также если он использует новое устройство или новую сеть VPN. В результате этих ложных срабатываний уровень риска для этого типа события является **средним.**

### <a name="sign-ins-from-infected-devices"></a>Попытки входа с инфицированных устройств

Это обнаружение рисков идентифицирует IP-адреса, а не пользовательские устройства. Если несколько устройств находятся за одним IP-адресом, и только некоторые из них контролируются сетью ботов, вступившие в систему с других устройств, мой триггер этого события излишне, поэтому это обнаружение риска классифицируется как **Низкий**.  

Мы советуем связаться с пользователем и сканировать все его устройства. Может оказаться, что инфицировано личное устройство этого пользователя или какого-то другого пользователя, который использовал тот же IP-адрес. Обычно инфицированные устройства — это устройства, антивирусное программное обеспечение которых еще не выявило вредоносные программы или у пользователей которых есть вредные привычки использования, повлекшие заражение.

Дополнительную информацию о том, как обеспечить защиту от вредоносных программ, см. в [центре защиты от вредоносных программ](https://www.microsoft.com/en-us/security/portal/definitions/adl.aspx/).

### <a name="sign-ins-from-ip-addresses-with-suspicious-activity"></a>Попытки входа с IP-адресов с подозрительными действиями

Мы рекомендуем связаться с пользователем, чтобы проверить, действительно ли он использовал IP-адрес, помеченный как подозрительный. У этого типа событий уровень риска**Средний**, так как один IP-адрес может использоваться несколькими устройствами, а с подозрительными действиями может быть связано только одно из них.
