# <a name="configure-azure-ad-role-settings-in-privileged-identity-management"></a>Настройка параметров роли Azure AD в привилегированном управлении идентификацией

Администратор привилегированных ролей может настроить Privileged Identity Management (PIM) в организации Azure Active Directory (Azure AD), включая изменение интерфейса для пользователя, который активирует подходящее назначение ролей.

## <a name="determine-your-version-of-pim"></a>Определите свою версию PIM

Начиная с ноября 2019 года, часть ролей Azure AD в Привилегированном управлении идентификацией обновляется в новую версию, которая соответствует опыту для ролей ресурсов Azure. Это создает дополнительные функции, а также [изменения в существующем API.](azure-ad-roles-features.md#api-changes) В то время как новая версия в настоящее время развернута, какие процедуры, которые вы следуете в этой статье зависит от версии Привилегированного управления идентификацией у вас в настоящее время. Выполните действия в этом разделе, чтобы определить, какая версия привилегированного управления идентификацией у вас есть. После того, как вы знаете свою версию Privileged Identity Management, вы можете выбрать процедуры в этой статье, которые соответствуют этой версии.

1. Вопием на [портал Azure](https://portal.azure.com/) с пользователем, который занимается ролью [администратора привилегированных.](../users-groups-roles/directory-assign-admin-roles.md#privileged-role-administrator)
1. Открытое **управление привилегированным удостоверением личности Azure AD.** Если у вас есть баннер в верхней части страницы обзора, следуйте инструкциям в **новой версии** вкладке этой статьи. В противном случае следуйте инструкциям во вкладке **Предыдущая версия.**

  [![](media/pim-how-to-add-role-to-user/pim-new-version.png "Select Azure AD > Privileged Identity Management")](media/pim-how-to-add-role-to-user/pim-new-version.png#lightbox)

Выполните действия в этой статье, чтобы утвердить или отклонить запросы на роли Azure AD.

## <a name="assignment-duration"></a>Длительность назначений

При настройке параметров роли можно выбрать один из двух вариантов продолжительности назначения для каждого типа назначения (допустимые и активные). Эти параметры становятся максимальной продолжительностью по умолчанию, когда пользователю присваивается роль в Привилегированном управлении идентификацией.

Вы можете выбрать один из этих **подходящих** вариантов продолжительности назначения:

| | |
| --- | --- |
| **Разрешить постоянную допустимую роль** | Глобальные админы и привилегированные роли могут назначить постоянное назначение. |
| **Срок действия подходящего назначения после** | Глобальные админы и привилегированные роли могут потребовать, чтобы все подходящие назначения имели указанную дату начала и окончания. |

Вы можете выбрать один из следующих **активных** вариантов продолжительности назначения:

| | |
| --- | --- |
| **Разрешить постоянную активную роль** | Глобальные админы и привилегированные роли могут назначать постоянное активное назначение. |
| **Срок действия активного назначения после** | Глобальные админы и привилегированные редемины ролей могут требовать, чтобы все активные назначения имели указанную дату начала и окончания. |

> [!NOTE]
> Все назначения с указанной датой окончания могут быть возобновлены глобальными админами и привилегированными ролевами. Кроме того, пользователи могут инициировать запросы на самообслуживание для [продления или продления ролевых заданий.](pim-resource-roles-renew-extend.md)

## <a name="require-multi-factor-authentication"></a>Требование многофакторной идентификации

Управление привилегированными пользователями поддерживает принудительное применение Многофакторной идентификации Azure (MFA) для двух разных сценариев.

### <a name="require-multi-factor-authentication-on-active-assignment"></a>Требовать Многофакторную идентификацию для активного назначения

В некоторых случаях может потребоваться назначить пользователю роль на короткий срок (например, один день). В этом случае назначенным пользователям не нужно запрашивать активацию. В этом сценарии Privileged Identity Management не может обеспечить многофакторную аутентификацию, когда пользователь использует свое назначение ролей, поскольку он уже активен в роли с момента его назначения.

Чтобы убедиться, что администратор, выполняющий назначение, является тем, кем он является, можно обеспечить многофакторную аутентификацию при активном назначении, проверив **требуемую многофакторную аутентификацию в активном** поле назначения.

### <a name="require-multi-factor-authentication-on-activation"></a>Требовать для активации Многофакторную идентификацию

Вы можете потребовать от пользователей, имеющих право на роль, доказать, кто именно они используют многофакторную аутентификацию Azure, прежде чем они смогут активировать. Многофакторная аутентификация гарантирует, что пользователь является тем, за кого себя выручен, с разумной уверенностью. Этот параметр защитит критически важные ресурсы даже при компрометации учетной записи пользователя.

Чтобы потребовать многофакторной аутентификации перед активацией, проверьте **обязательную многофакторную аутентификацию на** поле активации во вкладке «Назначение» **настройки роли Edit.**

Дополнительные сведения см. в статье [о многофакторной проверке подлинности и управлении привилегированными пользователями](pim-how-to-require-mfa.md).

## <a name="activation-maximum-duration"></a>Максимальная длительность активации

С помощью ползунка **Максимальная длительность активации (в часах)** установите максимальное время в часах, когда роль остается активной до истечения срока ее действия. Это значение может составить от одного до 24 часов.

## <a name="require-justification"></a>Требование обоснования

Вы можете потребовать, чтобы пользователи введите бизнес-обоснование при активации. Чтобы требовать обоснование, установите флажки **Require justification on active assignment** (Требовать обоснование для активного назначения) или **Require justification on activation** (Требовать обоснование для активации).

## <a name="require-approval-to-activate"></a>Настройка требования утверждения для активации

При установке нескольких одобрителей утверждение завершается, как только один из них одобряет или отрицает. Вы не можете требовать одобрения по крайней мере от двух пользователей. Чтобы получить одобрение для активации роли, выполните следующие действия.

1. Установите флажок **Require approval to activate** (Требование утверждения для активации).

1. **Выберите утвердителей.**

    ![Выберите пользовательское или групповое стекло для выбора одобрителей](./media/pim-resource-roles-configure-role-settings/resources-role-settings-select-approvers.png)

1. Выберите хотя бы одного пользователя, а затем нажмите **Выберите**. Нужно выбрать по меньшей мере одно утверждающее лицо. Утверждающих по умолчанию нет.

    Выбранные вами элементы отобразятся в списке выбранных утверждающих лиц.

1. После того, как вы указали все настройки ролей, выберите **обновление,** чтобы сохранить изменения.

1. Выберите роль, настройки которой вы хотите настроить.

    ![Роли Azure AD - Роли настройки](./media/pim-how-to-change-default-settings/pim-directory-roles-settings-role.png)

    На странице параметров для каждой роли есть ряд параметров, которые можно настроить. Эти параметры распространяются только на пользователей, которым доступны **соответствующие****временные** назначения.

## <a name="activations"></a>Активации

С помощью ползунка **Активации** задайте максимальное время в часах, в течение которого роль остается активной. Возможное значение: от 1 до 72 часов.

## <a name="notifications"></a>Уведомления

Используйте переключатель режима **Уведомления**, чтобы указать, будут ли администраторы получать уведомления по электронной почте при активации ролей. Это уведомление может быть полезно для обнаружения несанкционированных или незаконных активаций.

Если режим **включен**, уведомления будут отправляться таким администраторам:

- администратор привилегированных ролей;
- администратор безопасности;
- Глобальный администратор.

Для получения дополнительной информации смотрите [уведомления по электронной почте в Privileged Identity Management](pim-email-notifications.md).

## <a name="incidentrequest-ticket"></a>Обращение с запросом или инцидентом

Используйте переключатель **билета Incident/Request,** чтобы требовать от соответствующих администраторов включать номер билета, когда они активируют свою роль. Эта практика может сделать аудит доступа к роли более эффективным.

## <a name="multi-factor-authentication"></a>Многофакторная идентификация

С помощью переключателя **Многофакторная идентификация** можно выбрать, следует ли требовать от пользователей перед активацией ролей проходить аутентификацию с использованием Многофакторной идентификации. Они должны только проверить свою личность один раз за сессию, а не каждый раз, когда они активируют роль. Есть две подсказки, которые рекомендуется учитывать при включении проверки MFA:

- Пользователи, у которых есть учетные записи Майкрософт для их адресов электронной почты (как правило, @outlook.comно не всегда), не могут зарегистрироваться для azure Multi-Factor Authentication. Если вы хотите назначить роли пользователям с учетными записями Майкрософт, следует либо сделать их постоянными админами, либо отключить многофакторную аутентификацию для этой роли.
- Вы не можете отключить многофакторную аутентификацию Azure для высокопривилегированных ролей для Azure AD и Office 365. Эта функция безопасности помогает защитить следующие роли:  
  
  - Администратор защиты информации Azure
  - Администратор выставления счетов
  - Администратор облачных приложений
  - администратор соответствия требованиям.
  - Администратор условного доступа
  - Динамика 365 Администратор
  - лицо, утверждающее доступ клиентов к LockBox;
  - создатели каталогов;
  - администратор Exchange;
  - Глобальный администратор.
  - Администратор Intune
  - Администратор Power BI
  - администратор привилегированных ролей;
  - администратор безопасности;
  - администратор SharePoint;
  - администратор Skype для бизнеса;
  - Администратор пользователей

Дополнительные сведения см. в статье [о многофакторной проверке подлинности и управлении привилегированными пользователями](pim-how-to-require-mfa.md).

## <a name="require-approval"></a>Требуется утверждение

Если вы хотите делегировать необходимое одобрение для активации роли, выполните следующие действия.

1. Установите для переключателя **Требуется утверждение** значение **Включено**. Область развернется, отобразив параметры для выбора утверждающих лиц.

    ![Роли Azure AD - Настройки - Требуется одобрение](./media/pim-how-to-change-default-settings/pim-directory-roles-settings-require-approval.png)

    Если вы не указали каких-либо одобрителей, администратор привилегированных ролей становится утверждением по умолчанию, а затем обязан утвердить все запросы активации для этой роли.

1. Чтобы добавить удобрятелей, нажмите **Выберите утверждение**.

    ![Роли Azure AD - Настройки - Требуется одобрение](./media/pim-how-to-change-default-settings/pim-directory-roles-settings-require-approval-select-approvers.png)

1. Выберите одного или нескольких одобрителей в дополнение к администратору привилегированных ролей, а затем нажмите **Выберите**. Мы рекомендуем добавить по крайней мере двух одобрителей. Даже если вы добавите себя в качестве одобрителя, вы не можете самостоятельно одобрить активацию роли. Выбранные вами элементы отобразятся в списке выбранных утверждающих лиц.

1. После того, как вы указали все настройки ролей, выберите **Сохранить,** чтобы сохранить изменения.
