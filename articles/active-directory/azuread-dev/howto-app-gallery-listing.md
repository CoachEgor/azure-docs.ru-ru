# <a name="list-your-application-in-the-azure-active-directory-application-gallery"></a>Добавление приложения в коллекцию приложений Azure Active Directory

[!INCLUDE [active-directory-azuread-dev](../../../includes/active-directory-azuread-dev.md)]

В этой статье показано, как уотедать приложение в галерее приложений Azure Active Directory (Azure AD), реализовать единый ввод (SSO) и управлять списком.

## <a name="what-is-the-azure-ad-application-gallery"></a>Что такое коллекция приложений Azure AD?

- Клиенты получают самый удобный способ единого входа.
- Конфигурация приложения будет простой и минимально необходимой.
- Ваше приложение можно будет найти в коллекции с помощью быстрого поиска.
- Эту интеграцию могут использовать все клиенты Azure AD категорий "Бесплатный", "Базовый" и "Премиум".
- Нашим общим клиентам предлагается пошаговое руководство по настройке.
- Клиенты, которые используют Систему управления кросс-доменом идентификации[(SCIM),](https://techcommunity.microsoft.com/t5/Identity-Standards-Blog/Provisioning-with-SCIM-getting-started/ba-p/880010)могут использовать подготовку к тому же приложению.

## <a name="prerequisites"></a>Предварительные требования

- Для федеративных приложений (Open ID и SAML/WS-Fed) приложение должно поддерживать модель программного обеспечения как услуги (SaaS) для попадания в галерею приложений Azure AD. Приложения корпоративной галереи должны поддерживать несколько конфигураций клиентов, а не какого-либо конкретного клиента.
- Для Open ID Connect приложение должно быть мультитенантным, а [платформа согласия Azure AD](../develop/consent-framework.md?toc=/azure/active-directory/azuread-dev/toc.json&bc=/azure/active-directory/azuread-dev/breadcrumb/toc.json) должна быть надлежащим образом реализована для приложения. Пользователь может отправить запрос на входе в общую конечную точку, чтобы любой клиент мог дать согласие на приложение. Доступом пользователей можно управлять на основе идентификатора клиента и имени участника-пользователя, которые передаются в маркере.
- Для SAML 2.0/WS-Fed ваша заявка должна иметь возможность сделать интеграцию SAML/WS-Fed SSO в режиме SP или IDP. Убедитесь, что эта возможность работает правильно, прежде чем отправить запрос.
- Для sSO-пароля убедитесь, что ваше приложение поддерживает аутентификацию формы, чтобы можно было сделать свод пароля, чтобы получить один знак для работы, как ожидалось.
- Вам нужна постоянная учетная запись для тестирования, по крайней мере два пользователя зарегистрированы.

**Как получить Azure AD для разработчиков?**

## <a name="submit-the-request-in-the-portal"></a>Отправка запроса на портале

После проверки того, что интеграция приложений работает с Azure AD, отправьте запрос на доступ на [портале Application Network.](https://microsoft.sharepoint.com/teams/apponboarding/Apps) Если у вас есть учетная запись Office 365, используйте ее для входа на этот портал. В противном случае используйте учетную запись Майкрософт, например Outlook или Hotmail, для входиный восподписания.

Если после входной регистрации появляется следующая страница, обратитесь в [группу интеграции Azure AD SSO.](<mailto:SaaSApplicationIntegrations@service.microsoft.com>) Предоставьте учетную запись электронной почты, которую вы хотите использовать для отправки запроса. Команда Azure AD добавит учетную запись на портале Microsoft Application Network.

![Сообщение запроса доступа на портале SharePoint](./media/howto-app-gallery-listing/errorimage.png)

После добавления учетной записи вы можете войти на портал Microsoft Application Network.

Если следующая страница появится после вступления в систему, предоставьте бизнес-обоснование для необходимости доступа в текстовое поле. Затем выберите **Доступ к запросу**.

  ![Коробка бизнес-оправдания на портале SharePoint](./media/howto-app-gallery-listing/accessrequest.png)

Наша команда рассмотрит указанные сведения и предоставит доступ соответствующим образом. После того, как ваш запрос будет одобрен, вы можете войти на портал и отправить запрос, выбрав плитку **отправки запроса (ISV)** на главной странице.

![Отправить запрос (ISV) плитка на главной странице](./media/howto-app-gallery-listing/homepage.png)

## <a name="issues-on-logging-into-portal"></a>Вопросы при входе на портал

Если вы видите эту ошибку при входе в систему, то вот подробно по этому вопросу, и это, как вы можете исправить это.

* Если ваш входин был заблокирован, как показано ниже:

  ![вопросы решения приложения в галерее](./media/howto-app-gallery-listing/blocked.png)

**Что происходит:**

Гостевой пользователь используется для арендатора дома, который также является Azure AD. Гостевой пользователь находится в группе высокого риска. Корпорация Майкрософт не позволяет пользователям с высоким риском получать доступ к своим ресурсам. Все пользователи с высоким риском (сотрудники или гости / поставщики) должны исправления / закрыть свой риск для доступа к ресурсам Майкрософт. Для гостевых пользователей этот пользовательский риск исходит от арендатора дома, а политика исходит от арендатора ресурса (в данном случае Microsoft).
 
**Безопасные решения:**

* МИД зарегистрированных гостевых пользователей исправления собственного риска пользователя. Это может быть сделано гостем пользователя выполнения защищенногоhttps://aka.ms/sspr) изменения пароля или сбросить (у своего дома арендатора (это необходимо МИД и SSPR на дома арендатора). Защищенное изменение пароля или сбросить необходимо инициировать в Azure AD, а не на prem.

* Гость пользователи имеют своих админов исправления их риска. В этом случае админ выполнит сбросить пароль (временное создание пароля). Это не нуждается в защите личных данных. Админ гостевого пользователя может зайти https://aka.ms/RiskyUsers и нажать на пароль «Перезагрузка».

* Гость пользователи имеют своих аминей близко / уволить их риск. Опять же, это не нуждается в защите личных данных. Админ может пойти https://aka.ms/RiskyUsers и нажать на "Dismiss риск пользователя". Тем не менее, админ должен сделать должной осмотрительности, чтобы убедиться, что это была ложноположительная оценка риска до закрытия пользовательского риска. В противном случае они подвергают риску свои ресурсы и ресурсы Microsoft, подавляя оценку риска без проведения расследования.

> [!NOTE]
> Если у вас возникли проблемы с доступом, обратитесь в [группу интеграции Azure AD SSO.](<mailto:SaaSApplicationIntegrations@service.microsoft.com>)

## <a name="implement-sso-by-using-the-federation-protocol"></a>Реализация SSO с помощью протокола федерации

Чтобы включить приложение в коллекцию Azure AD, сначала необходимо реализовать один из следующих протоколов федерации, поддерживаемых Azure AD. Необходимо также согласовать условия галереи приложений Azure AD. Ознакомьте условия галереи приложений Azure AD на [этом сайте.](https://azure.microsoft.com/support/legal/active-directory-app-gallery-terms/)

- **OpenID Connect:** Чтобы интегрировать приложение с Azure AD с помощью протокола Open ID Connect, следуйте [инструкциям разработчиков.](v1-authentication-scenarios.md)

    ![Перечисление приложения OpenID Connect в галерее](./media/howto-app-gallery-listing/openid.png)

    * Если вы хотите добавить приложение в список в галерее с помощью OpenID Connect, выберите **OpenID Connect & OAuth 2.0,** как показано на рисунке.
    * Если у вас возникли проблемы с доступом, обратитесь в [группу интеграции Azure AD SSO.](<mailto:SaaSApplicationIntegrations@service.microsoft.com>)

- **SAML 2.0** или **WS-Fed:** Если ваше приложение поддерживает SAML 2.0, вы можете интегрировать его непосредственно с арендатором Azure AD, следуя [инструкциям по добавлению пользовательского приложения.](../active-directory-saas-custom-apps.md)

  ![Листинг приложения SAML 2.0 или WS-Fed в галерее](./media/howto-app-gallery-listing/saml.png)

  * Если вы хотите добавить приложение в список в галерее с помощью **SAML 2.0** или **WS-Fed,** выберите **SAML 2.0/WS-Fed,** как показано на рисунке.

  * Если у вас возникли проблемы с доступом, обратитесь в [группу интеграции Azure AD SSO.](<mailto:SaaSApplicationIntegrations@service.microsoft.com>)

## <a name="implement-sso-by-using-the-password-sso"></a>Реализация SSO с помощью пароля SSO

Создайте веб-приложение с HTML-страницей входа, чтобы настроить [единый вход на основе пароля](../manage-apps/what-is-single-sign-on.md). Единый вход с помощью пароля (хранение пароля в хранилище) позволяет управлять доступом и паролями для веб-приложений, которые не поддерживают федерацию удостоверений. Это также полезно для сценариев, в которых нескольким пользователям необходимо совместно поделиться одной учетной записью, например, с учетными записями приложений в социальных сетях вашей организации.

![Перечисление приложения SSO с паролем в галерее](./media/howto-app-gallery-listing/passwordsso.png)

* Если вы хотите добавить приложение в список в галерее с помощью пароля SSO, выберите **пароль SSO,** как показано на рисунке.
* Если у вас возникли проблемы с доступом, обратитесь в [группу интеграции Azure AD SSO.](<mailto:SaaSApplicationIntegrations@service.microsoft.com>)

## <a name="update-or-remove-an-existing-listing"></a>Обновление или удаление существующего объявления

Чтобы обновить или удалить существующее приложение в галерее приложений Azure AD, сначала необходимо отправить запрос на [портале Application Network.](https://microsoft.sharepoint.com/teams/apponboarding/Apps) Если у вас есть учетная запись Office 365, используйте ее для входа на этот портал. В противном случае используйте учетную запись Майкрософт, например Outlook или Hotmail, для входиный восподписания.

    ![Перечисление приложения SAML в галерее](./media/howto-app-gallery-listing/updateorremove.png)

    * Чтобы удалить существующее приложение из галереи приложений Azure AD, выберите **Удалить список приложений из галереи.**
    * Если у вас возникли проблемы с доступом, обратитесь в [группу интеграции Azure AD SSO.](<mailto:SaaSApplicationIntegrations@service.microsoft.com>)
