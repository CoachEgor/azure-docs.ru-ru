# <a name="how-to-close-active-risk-detections"></a>Как Закрыть обнаружение активных рисков

При [обнаружении рисков](../reports-monitoring/concept-risk-events.md)Azure Active Directory обнаруживает индикаторы для потенциально скомпрометированных учетных записей пользователей. Как администратор, вы хотите получить все обнаружения рисков, чтобы затронутые пользователи перестали быть подвержены риску.

В этой статье приводятся общие сведения о дополнительных параметрах, которые необходимо выполнить для закрытия активных обнаружений рисков.

## <a name="options-to-close-risk-detections"></a>Параметры для закрытия обнаружений рисков 

Состояние обнаружения риска — **активное** или **закрытое**. Все активные обнаружения рисков участвуют в вычислении значения, именуемого уровнем риска пользователя. Уровень риска для пользователя является индикатором (низким, средним, высоким) вероятности того, что учетная запись была скомпрометирована. 

Чтобы закрыть обнаружение активных рисков, доступны следующие варианты.

- Требование сброса пароля с помощью политики риска для пользователя
- Сброс паролей вручную
- Отклонить все обнаружения рисков 
- Закрытие отдельных обнаружений рисков вручную

## <a name="require-password-reset-with-a-user-risk-policy"></a>Требование сброса пароля с помощью политики риска для пользователя

Настроив [политику условного доступа пользователя](howto-user-risk-policy.md), можно запрашивать изменение пароля, если определенный уровень риска пользователя был автоматически обнаружен. 

![Сбросить пароль](./media/howto-close-active-risk-events/13.png)

При сбросе пароля закрываются все активные события риска для соответствующего пользователя, и идентификатор снова считается безопасным. Использование политики риска пользователя — предпочтительный метод закрытия активных обнаружений рисков, поскольку этот метод автоматизирован. Взаимодействие между затронутым пользователем и службой поддержки или администратором не требуется.

Но использовать политику риска для пользователя не всегда возможно. Это касается, например, следующих случаев:

- пользователи, не зарегистрированные для многофакторной проверки подлинности (MFA);
- Пользователи с активными обнаруженными рисками, которые были удалены.
- Исследование, которое показывает, что проверенный пользователь выполнил обнаружение рисков.

## <a name="manual-password-reset"></a>Сброс паролей вручную

При необходимости сброса пароля с помощью политики риска для пользователя можно получить все обнаружения рисков для пользователя, который закрылся вручную сбросом пароля.

![Сбросить пароль](./media/howto-close-active-risk-events/04.png)

Соответствующее диалоговое окно предлагает два метода для сброса пароля:

![Сбросить пароль](./media/howto-close-active-risk-events/05.png)

**Создание временного пароля** — создав временный пароль, можно немедленно вернуть удостоверение в безопасное состояние. Этот способ требует взаимодействия с затронутыми пользователями, так как им нужно знать временный пароль. Можно, например, отправить новый временный пароль по альтернативному адресу электронной почты пользователя или его руководителя. Так как пароль временный, при следующем входе пользователю будет предложено изменить его.

**Требование от пользователя сбросить пароль** — требование сброса пароля от пользователей позволяет включить самостоятельное восстановление без обращения в службу технической поддержки или к администратору. Как и в случае с политикой риска для пользователя, этот метод применяется только к пользователям, которые зарегистрированы для многофакторной проверки подлинности (MFA). Для пользователей, которые еще не зарегистрированы для MFA, этот параметр недоступен.

## <a name="dismiss-all-risk-detections"></a>Отклонить все обнаружения рисков

Если сброс пароля не предусмотрен, можно также закрыть все обнаружения рисков. 

![Сбросить пароль](./media/howto-close-active-risk-events/03.png)

При нажатии кнопки **Dismiss all events**(Закрыть все события) закрываются все события и соответствующий пользователь больше не находится под угрозой. Но так как этот метод не оказывает влияния на существующий пароль, он не возвращает связанный идентификатор в безопасное состояние. Предпочтительным вариантом использования этого метода является удаленный пользователь с активными определениями рисков. 

## <a name="close-individual-risk-detections-manually"></a>Закрытие отдельных обнаружений рисков вручную

Можно закрыть отдельные определения рисков вручную. После закрытия обнаружений рисков вручную можно уменьшить уровень риска для пользователя. Как правило, обнаружение рисков закрывается вручную в ответ на соответствующее исследование. Например, при разговоре с пользователем обнаруживается, что активное обнаружение рисков больше не требуется. 
 
При закрытии обнаружений рисков вручную можно выполнить любое из следующих действий, чтобы изменить состояние обнаружения риска.

![Действия](./media/howto-close-active-risk-events/06.png)

- **Разрешить** . Если после изучения обнаружения рисков вы выполнили соответствующее действие исправления вне защиты идентификации и считаете, что обнаружение рисков должно быть закрыто, пометьте событие как разрешенное. Разрешенные события задают состояние обнаружения риска "закрыто", а обнаружение рисков больше не влияет на риск для пользователей.
- **Пометить как ложное положительное** — в некоторых случаях можно исследовать обнаружение рисков и обнаружить, что оно ошибочно отмечено как опасное. Можно уменьшить количество таких вхождений, пометив обнаружение рисков ложным. Это позволит усовершенствовать классификацию таких событий при использовании алгоритмов машинного обучения. Состояние событий с ложным срабатыванием изменяется на "Закрыто", и они больше не учитываются при оценке риска для пользователя.
- **Ignore** — если вы не предпримете какие-либо действия по исправлению, но хотите удалить обнаружение рисков из активного списка, можно отметить, что обнаружение риска будет игнорироваться, а состояние события — закрыто. Пропущенные события не влияют на риск для пользователя. Эту возможность следует использовать только при чрезвычайных обстоятельствах.
- Повторная **Активация** — обнаружение рисков, которые были закрыты вручную (выбрав "разрешить", "ложный положительный" или "пропустить"), можно повторно активировать, установив для состояния события значение "активно". Повторная активация обнаруженных рисков влияет на вычисление уровня риска для пользователя. Обнаружение рисков, закрытых путем исправления (например, безопасный сброс пароля), невозможно активировать повторно.
