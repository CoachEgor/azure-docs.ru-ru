# <a name="how-to-configure-risk-policies-in-azure-active-directory-identity-protection-refreshed"></a>Как Настройка политики риска пользователя в Защите идентификации Azure Active Directory (обновленной)


Azure AD обнаруживает события риска, которые являются индикаторами для потенциально скомпрометированных удостоверений. Вы можете определить автоматическое реагирование на результаты обнаружения с помощью настройки политики рисков.

- Благодаря политике риска для входа в систему вы можете настроить ответ на события риска в реальном времени, которые были обнаружены во время входа пользователя. 
- Благодаря политике риска пользователя вы можете настроить ответ на все риски активного пользователя, которые со временем были обнаружены для пользователя.  

> [!VIDEO https://www.youtube.com/embed/zEsbbik-BTE]


## <a name="what-is-the-sign-in-risk-policy"></a>Что такое политика риска входа?

Azure AD анализирует каждый вход пользователя в систему. Это делается для определения подозрительных действий, выполняемых при входе. Например, выполняется ли вход с использованием анонимного IP-адреса, или он инициируется из незнакомого расположения? В Azure AD подозрительные действия, которые может обнаружить система, также называются событиями риска. На основе событий риска, которые были обнаружены во время входа в систему, Azure AD рассчитывает определенное значение. Это значение представляет вероятность (низкий, средний, высокий уровень) того, что вход выполняется несанкционированным пользователем. Такая вероятность называется **уровнем риска входа в систему**.

Политика риска входа представляет собой автоматический ответ, который можно настроить для определенного уровня риска входа в систему. В качестве ответа можно заблокировать доступ к ресурсам или потребовать пройти многофакторную проверку подлинности (MFA) для получения доступа.

После успешного завершения привилегиями многофакторной проверки Подлинности, выдаваемых политикой риска входа пользователя он предоставляет отзыв к службе защиты идентификации, входа в исходящее от полномочного пользователя. Таким образом будут автоматически закрыты событий риска входа, который инициировал запрос на многофакторную Идентификацию и защиты идентификации будет препятствовать вносящая вклад в повышение риска пользователя это событие. При включении политики риска входа можно сократить noisiness в представлении рискованные входы в систему, позволяя пользователям самостоятельно устранять при запросе многофакторной проверки Подлинности и впоследствии автоматическое завершение связанных событиях входа.

## <a name="what-you-should-know-about-sign-in-risk-policies"></a>Необходимые сведения о политиках риска для входа

Вы можете настроить политику риска при входе так, чтобы требовалась многофакторная идентификация.

![Требовать многофакторную идентификацию](./media/howto-configure-risk-policies/16.png)

Но из соображений безопасности этот параметр действует только для пользователей, уже зарегистрированных для многофакторной идентификации. Защита идентификации блокирует пользователей с требованием многофакторной проверки подлинности, если они еще не зарегистрированы для такой проверки.

Чтобы потребовать многофакторную идентификацию для рискованных входов в систему, нужно сделать следующее:

1. Включить политику регистрации для многофакторной идентификации соответствующих пользователей.

2. Требовать от затронутых пользователей входить в безопасные сеансы, чтобы регистрироваться для Многофакторной идентификации.

Если вы выполните эти действия, для рискованных входов будет требоваться многофакторная идентификация.

Политика риска входа:

- применяется ко всему трафику в браузере и при каждом входе с использованием современных способов проверки подлинности;

- не применяется к приложениям, которые используют протоколы безопасности старых версий, отключая конечную точку WS-Trust в федеративном IdP, таком как ADFS.

## <a name="what-is-a-user-risk-policy"></a>Что такое политика риска пользователя?

Azure AD анализирует каждый вход пользователя в систему. Это делается для определения подозрительных действий, выполняемых при входе. В Azure AD подозрительные действия, которые может обнаружить система, также называются событиями риска. Хотя некоторые события риска могут быть обнаружены в реальном времени, некоторым событиям требуется больше времени. Например, для обнаружения невозможных перемещений в нетипичные расположения системе требуется предварительный период обучения в 14 дней, чтобы изучить обычное поведение пользователя. Есть несколько вариантов устранять обнаруженные события риска. Например можно устранить события риска отдельных вручную, или их разрешить с помощью риска входа или пользователя риск политики условного доступа можно получить.

Все события риска, обнаруженные для пользователя и не устраненные, называются активными. Активные события риска, связанные с тем или иным пользователем, — это риск пользователя. На основании риска пользователя Azure AD вычисляет вероятность (низкий, средний, высокий уровень) того, что пользователь был скомпрометирован. Эта вероятность называется уровнем риска пользователя.

![Риск для пользователя](./media/howto-configure-risk-policies/11031.png)

Политика риска пользователя представляет собой автоматический ответ, который можно настроить для определенного уровня риска пользователя. С помощью политики риска пользователя можно заблокировать доступ к ресурсам или потребовать изменить пароль для возвращения учетной записи пользователя в исходное состояние.

## <a name="what-you-should-know-about-user-risk-polices"></a>Необходимые сведения о политиках риска пользователя

В политике безопасности в отношении риска пользователей можно установить блокировку входа на основе уровня риска.

![Блокировка](./media/howto-configure-risk-policies/116.png)


Блокировка входа:

* предотвращает создание новых событий риска для каждого затронутого пользователя;
* позволяет администраторам вручную устранить события риска, которые распространяются на удостоверение пользователя, и восстанавливать безопасное состояние.


























## <a name="best-practices"></a>Рекомендации

Если выбрать пороговое значение уровня **Высокий** , уменьшится количество случаев активации политики и степень влияния на пользователей.  

Тем не менее это исключит из политики пользователей, помеченных для событий риска с уровнем **Низкий** и **Средний**. Есть вероятность, что в таком случае злоумышленникам ничего не будет мешать использовать скомпрометированные удостоверения.

При настройке политики:

- исключите пользователей, которые не используют или не могут использовать многофакторную проверку подлинности;

- исключите пользователей, для языкового стандарта которых нецелесообразно включать политику (например, при его использовании недоступна техническая поддержка);

- исключите пользователей, из-за которых, скорее всего, создается большое количество ложных срабатываний (разработчиков, аналитиков системы безопасности);

- используйте пороговое значение уровня **Высокий** при начальном развертывании политики, а также если требуется уменьшить количество запросов защиты для конечного пользователя.

- Используйте пороговое значение уровня **Низкий**, если нужно повысить уровень безопасности для организации. Если вы выберете пороговое значение уровня **Низкий** , при входе для пользователей будет отображаться больше запросов защиты, но уровень безопасности будет выше.

Большинству организаций мы рекомендуем настроить для правил пороговое значение уровня **Средний** , чтобы обеспечить оптимальное соотношение удобства использования и безопасности.
