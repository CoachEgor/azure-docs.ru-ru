# <a name="identity-protection-and-b2b-users"></a>Защита идентификации и пользователи B2B

В рамках совместной работы Azure AD B2B организации могут применять политики, основанные на риске, для пользователей B2B, используя identity Protection. Эти политики настраиваться двумя способами:

- Администраторы могут настроить встроенные политики, основанные на риске Identity Protection, которые применяются ко всем приложениям, включающим гостевых пользователей.
- Администраторы могут настроить свои политики условного доступа, используя риск вхвока в качестве условия, включающий в себя гостевых пользователей.

## <a name="how-is-risk-evaluated-for-b2b-collaboration-users"></a>Как оценивается риск для пользователей B2B-сотрудничества

Риск для пользователей B2B-сотрудничества оценивается в их домашнем каталоге. Риск в реальном времени для этих пользователей оценивается в каталоге ресурсов при попытке получить доступ к ресурсу.

## <a name="limitations-of-identity-protection-for-b2b-collaboration-users"></a>Ограничения защиты идентификации для пользователей совместной работы B2B

Существуют ограничения в реализации Identity Protection для пользователей совместной работы B2B в каталоге ресурсов из-за их идентичности, существующей в их домашнем каталоге. Основные ограничения таковы:

- Если гость пользователя запускает политику защиты пользователей Identity Protection, чтобы заставить сбросить пароль, **они будут заблокированы.** Этот блок связан с невозможностью сбросить пароли в каталоге ресурсов.
- **Гость пользователей не отображается в отчете о рискованных пользователях.** Эта потеря видимости связана с оценкой риска, происходящей в домашнем каталоге пользователя B2B.
- Администраторы **не могут уволить или исправления рискованного пользователя B2B-совместной работы** в каталоге ресурсов. Эта потеря функциональности связана с тем, что администраторы в каталоге ресурсов не имеют доступа к домашнему каталогу пользователя B2B.

### <a name="why-cant-i-remediate-risky-b2b-collaboration-users-in-my-directory"></a>Почему я не могу усвоить рискованных пользователей B2B-коллаборации в моем каталоге?

Оценка риска и исправление для пользователей B2B происходит в их домашнем каталоге. Из-за этого гостевые пользователи не отображаются в отчете о рискованных пользователях в каталоге ресурсов, а админы в каталоге ресурсов не могут заставить этих пользователей сбросить пароль.

### <a name="what-do-i-do-if-a-b2b-collaboration-user-was-blocked-due-to-a-risk-based-policy-in-my-organization"></a>Что делать, если пользователь B2B-коллаборациониста был заблокирован из-за политики, основанной на риске в моей организации?

Если рискованный пользователь B2B в вашем каталоге заблокирован вашей политикой, основанной на риске, пользователь должен будет исправить этот риск в своем домашнем каталоге. Пользователи могут усвоить свой риск, выполнив безопасный сбросить пароль в своем домашнем каталоге. Если в главном каталоге не включено сбросить пароля самообслуживания, им необходимо связаться с ИТ-персоналом своей организации, чтобы администратор вручную отклонял риск или сдал пароль.

### <a name="how-do-i-prevent-b2b-collaboration-users-from-being-impacted-by-risk-based-policies"></a>Как предотвратить влияние политики, основанных на риске, на пользователей Совместной работы B2B?

Исключение пользователей B2B из политики условного доступа к риску вашей организации предотвратит влияние или блокировку пользователей B2B в результате оценки риска. Чтобы исключить этих пользователей B2B, создайте группу в Azure AD, которая содержит всех гостевых пользователей вашей организации. Затем добавьте эту группу в качестве исключения для встроенного пользователя Identity Protection и политики риска ввески, а также любых политик условного доступа, которые используют риск ввоза в качестве условия.
