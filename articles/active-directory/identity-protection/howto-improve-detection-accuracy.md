# <a name="how-to-improve-the-detection-accuracy"></a>Практическое руководство. Повышение точности обнаружения 

Защита идентификации предоставляет механизмы для отзыва с Azure AD об обнаружении рисков в вашей среде. Чтобы оставить отзыв, вы можете подтвердить статус обнаруженного рискованного пользователя или событие входа. Корпорация Майкрософт использует этот отзыв для принятия мер в отношении текущих обнаружений рисков и повышения точности будущих обнаружений. 


## <a name="what-is-detection"></a>Что такое обнаружение?

Обнаружение — это процесс идентификации подозрительных действий в сочетании с учетными записями пользователей. Azure AD может обнаруживать подозрительные действия, которые называются [событиями риска](../reports-monitoring/concept-risk-events.md). Процесс обнаружения основан на адаптивных алгоритмах машинного обучения и эвристических методах для обнаружения событий рисков для пользователей.

Результаты обнаружения используются для определения риска для пользователей и входов в систему. 


## <a name="how-can-i-improve-the-detection-accuracy"></a>Как повысить точность обнаружения?

Так как обнаружение является автоматизированным процессом, вероятно, Azure AD выдал ложные положительные результаты. Вы можете повысить точность обнаружения, предоставив отзыв Azure AD о результатах обнаружения.

Существует три способа повысить точность обнаружения: подтвердить скомпрометированный вход в систему, подтвердить безопасный вход в систему и исключить риск для пользователя. Вы можете сделать это из следующих отчетов.

- **Отчет о входах, представляющих риск**: в таком отчете вы можете подтвердить, что вход выполнен безопасно или скомпрометирован.

- **Отчет о пользователях, выполняющих рискованные действия**: в отчете о пользователях, выполняющих рискованные действия, вы можете проигнорировать риск пользователя. 

Azure AD обработает ваш отзыв для повышения точности результатов обнаружения. Как правило, вы предоставляете отзыв в рамках расследования риска пользователя или входа в систему. Дополнительные сведения см. в статье [How To: Investigate risky users and sign-ins](howto-investigate-risky-users-signins.md) (Практическое руководство. Изучение действий пользователей и операций входа, которые могут представлять риск).


## <a name="confirm-compromised"></a>Подтверждение компрометации

Подтверждение события входа как скомпрометированного сигнала для Azure AD о том, что вход не был авторизован владельцем удостоверения. При выборе "подтверждение компрометации" Azure AD выполняет следующие действия:

- увеличивает риск пострадавшего пользователя до высокого;

- помогает оптимизировать машинное обучение, которое обнаруживает события риска;
 
- выполняет дополнительные меры для обеспечения дальнейшей защиты вашей организации.



Чтобы подтвердить скомпрометированный вход, выполните следующие действия.

- **Отчет о входах, представляющих риск**. Этот параметр позволяет подтвердить скомпрометированный вход для одного или нескольких событий входа в систему.

    ![Параметр "Закрыть уведомление о риске для пользователя"](./media/howto-improve-detection-accuracy/07.png)

- **Представление сведений об отчете о входах, представляющих риск**. Этот параметр позволяет подтвердить скомпрометированную учетную запись для выбранного события входа в систему в отчете о входах, представляющих риск. 

    ![Параметр "Закрыть уведомление о риске для пользователя"](./media/howto-improve-detection-accuracy/04.png)


 
## <a name="confirm-safe"></a>Подтверждение безопасности


Подтверждение события входа как безопасного сигнала для Azure AD о том, что вход **был** авторизован владельцем соответствующих идентификаторов. При выборе "подтверждение безопасности" Azure AD выполняет следующие действия:

- отменяет изменения вклада пользователя риска выбранных входов;

- закрывает базовые события риска;

- помогает оптимизировать машинное обучение, которое обнаруживает события риска;

- выполняет дополнительные меры для обеспечения дальнейшей защиты вашей организации.
 

Для подтверждения безопасного входа выполните следующее:

- **Отчет о входах, представляющих риск**. Этот параметр позволяет подтвердить безопасный вход для одного или нескольких событий входа в систему.

    ![Параметр "Закрыть уведомление о риске для пользователя"](./media/howto-improve-detection-accuracy/08.png)

- **Представление сведений об отчете о входах, представляющих риск**. Этот параметр позволяет подтвердить безопасный вход для выбранного события входа в систему в отчете о входах, представляющих риск. 

    ![Параметр "Закрыть уведомление о риске для пользователя"](./media/howto-improve-detection-accuracy/05.png)




## <a name="dismiss-user-risk"></a>Параметр "Закрыть уведомление о риске для пользователя"

Если вы уже предприняли действия по исправлению для пользователя, подверженного риску, или считаете, что они были ошибочно помечены как рискованные, вы можете закрыть риск пользователя. Закрытие риска пользователя возвращает его в безопасное состояние. Все предыдущие рискованные входы и события риска для выбранного пользователя будут отклонены.


Вы можете отклонить указанный пользовательский риск в:

- **Отчет о пользователях, выполняющих рискованные действия**. Этот параметр позволяет отклонить риск для одного или нескольких выбранных пользователей.

    ![Параметр "Закрыть уведомление о риске для пользователя"](./media/howto-improve-detection-accuracy/02.png)

- **Представление сведений**. Этот параметр позволяет отклонить риск для выбранного пользователя в отчете риска пользователя. 

    ![Параметр "Закрыть уведомление о риске для пользователя"](./media/howto-improve-detection-accuracy/01.png)


**Важная информация**

- Это действие невозможно отменить.

- Выполнение этого действия может занять несколько минут, поэтому вам не следует повторно отправлять запрос.

- Это действие можно выполнить только в том случае, если AD управляет учетными данными пользователя. 



## <a name="best-practices"></a>Рекомендации

Отклонение риска пользователя является одним из способов разблокировать его, если он был заблокирован политикой риска пользователя и он не может самостоятельно исправиться из-за отсутствия сброса пароля или активации MFA. В этой ситуации лучше всего убедиться, что пользователь регистрируется для сброса пароля и MFA, чтобы они могли самостоятельно устранять любые будущие события риска.
