# <a name="user-experiences-with-azure-ad-identity-protection"></a>Взаимодействие пользователей с защитой идентификационных данных Azure AD

С помощью защиты идентификации Azure Active Directory можно:

* Требовать от пользователей регистрации на Azure Multi-Фактор аутентификации (MFA)
* Автоматизация исправления рискованных всхырей и скомпрометированных пользователей

Все политики защиты идентификационных данных влияют на вывески в опыте пользователей. Разрешение пользователям регистрироваться и использовать такие инструменты, как Azure MFA и сбросить пароли самообслуживания, может уменьшить воздействие. Эти инструменты наряду с соответствующими вариантами политики дают пользователям возможность самовосстановления, когда они в ней нуждаются.

## <a name="multi-factor-authentication-registration"></a>Регистрация с многофакторной проверкой подлинности

Включение политики защиты идентификации, требующей многофакторной регистрации аутентификации и ориентированной на всех пользователей, гарантирует, что они имеют возможность использовать Azure MFA для самостоятельного исправления в будущем. Настройка этой политики дает пользователям 14-дневный период, когда они могут зарегистрироваться и в конце вынуждены зарегистрироваться. Опыт для пользователей описан ниже. Более подробную информацию можно найти в документации для конечных пользователей в статье, [Обзор для двухфакторной проверки и вашей работы или школьной учетной записи](../user-help/user-help-two-step-verification-overview.md).

### <a name="registration-interrupt"></a>Перерыв на регистрацию

1. При входе в любое приложение, интегрированное в Azure AD, пользователь получает уведомление о необходимости настройки учетной записи для многофакторной аутентификации. Эта политика также срабатывает в Windows 10 Out of Box Experience для новых пользователей с новым устройством.
   
    ![Требуются дополнительные сведения](./media/concept-identity-protection-user-experience/identity-protection-experience-more-info-mfa.png)

1. Завершите управляемые шаги, чтобы зарегистрироваться для azure Multi-Factor Authentication и завершить входе.

## <a name="risky-sign-in-remediation"></a>Рискованная восктаврация ввески

Когда администратор настраивает политику для рисков входин, пострадавшие пользователи уведомляются при попытке войти в систему и запустить уровень риска политики. 

### <a name="risky-sign-in-self-remediation"></a>Рискованный всхвай самовосстановления

1. Пользователя информируют о том, что при входе обнаружена подозрительная активность, например, вход выполнен из нового расположения, с помощью нового устройства или приложения.
   
    ![Что-то необычное подсказоке](./media/concept-identity-protection-user-experience/120.png)

1. Пользователь должен доказать свою личность, заполнив Azure MFA одним из ранее зарегистрированных методов. 

### <a name="risky-sign-in-administrator-unblock"></a>Рискованный администратор вхаски разблокируют

Администраторы могут блокировать пользователей при входе в систему в зависимости от уровня риска. Чтобы разблокировать его, конечным пользователям необходимо связаться со своим ИТ-персоналом, или они могут попробовать войти в систему из знакомого места или устройства. Самовосстановление путем выполнения многофакторной аутентификации в данном случае не является вариантом.

![Заблокировано политикой риска регистрации](./media/concept-identity-protection-user-experience/200.png)

ИТ-персонал может следовать инструкциям в разделе [Разблокировка пользователей,](howto-identity-protection-remediate-unblock.md#unblocking-based-on-sign-in-risk) чтобы позволить пользователям войти обратно.

## <a name="risky-user-remediation"></a>Рискованное исправление пользователей

Когда настроена политика риска пользователя, пользователи, которые отвечают вероятности компрометации уровня риска пользователя, должны пройти через поток восстановления компромисса пользователя, прежде чем они смогут войти в систему. 

### <a name="risky-user-self-remediation"></a>Рискованное самовосстановление пользователя

1. Пользователя информируют о риске для безопасности учетной записи из-за подозрительной активности или утери учетных данных.
   
    ![Исправление](./media/concept-identity-protection-user-experience/101.png)

1. Пользователь должен доказать свою личность, заполнив Azure MFA одним из ранее зарегистрированных методов. 
1. Наконец, пользователь вынужден изменить свой пароль с помощью сбросить пароль самообслуживания, так как кто-то другой, возможно, имел доступ к своей учетной записи.

## <a name="risky-sign-in-administrator-unblock"></a>Рискованный администратор вхаски разблокируют

Администраторы могут блокировать пользователей при входе в систему в зависимости от уровня риска. Чтобы разблокировать их, конечные пользователи должны связаться со своим ИТ-персоналом. Самовосстановление путем выполнения многофакторной проверки подлинности и сбросить пароля самообслуживания в данном случае не является вариантом.

![Заблокировано политикой риска пользователя](./media/concept-identity-protection-user-experience/104.png)

ИТ-персонал может следовать инструкциям в разделе [Разблокировка пользователей,](howto-identity-protection-remediate-unblock.md#unblocking-based-on-user-risk) чтобы позволить пользователям войти обратно.
