# <a name="migrate-to-cloud-authentication-by-using-staged-rollout-preview"></a>Переход к облачной аутентификации с помощью поэтапного развертывания (предварительный просмотр)

Используя поэтапный подход развертывания, можно перейти от федеративной аутентификации к облачной аутентификации. В этой статье обсуждается, как сделать переключатель. Однако перед началом поэтапного развертывания следует учитывать последствия, если одно или несколько из следующих условий верны:
    
-  В настоящее время вы используете натерритории Многофакторная аутентификация сервера. 
-  Вы используете смарт-карты для проверки подлинности. 
-  Ваш текущий сервер предлагает определенные функции, связанные только с федерацией.

Прежде чем попробовать эту функцию, мы предлагаем вам ознакомиться с нашим руководством по выбору правильного метода аутентификации. Для получения дополнительной информации см. таблицу "Сравнение методов" в [книге "Выбрать правильный метод проверки подлинности для решения гибридной идентификации Azure Active Directory".](https://docs.microsoft.com/azure/security/fundamentals/choose-ad-authn#comparing-methods)

Для просмотра функции просмотрите этот "Активный каталог Azure: Что такое поэтапное развертывание?" Видео:

## <a name="prerequisites"></a>Предварительные требования

-   У вас есть арендатор Active Directory (Azure AD) с федеративными доменами.

-   Вы решили перейти к одному из двух вариантов:
    - **Вариант А** - *синхронизация хэша паролей (синхронизация)* + *бесшовная одиночная регистрация (SSO)*
    - **Вариант B** - *сквозной аутентификации* + *бесшовных SSO*
    
    Несмотря на то, что *бесшовная SSO* не является обязательной, мы рекомендуем дать ему возможность реализовать бесшумный опыт вхождения для пользователей, которые управляют компьютерами, работающими с доменом, из корпоративной сети.

-   Вы настроили все соответствующие политики брендинга арендаторов и условного доступа, необходимые для пользователей, которые переходят на проверку подлинности облака.

-   Если вы планируете использовать Azure Multi-Factor Authentication, мы рекомендуем использовать [конвергентную регистрацию для сбросить паролем самообслуживания (SSPR) и multi-Factor Authentication,](../authentication/concept-registration-mfa-sspr-combined.md) чтобы пользователи зарегистрировали свои методы аутентификации один раз.

-   Чтобы использовать функцию поэтапного развертывания, необходимо быть глобальным администратором в ашего арендаторе.

-   Для включения *бесшовной SSO* в определенный лес Active Directory необходимо быть администратором домена.

## <a name="supported-scenarios"></a>Поддерживаемые сценарии

Следующие сценарии поддерживаются для поэтапного развертывания. Функция работает только для:

- Пользователи, которые подготовлены к Azure AD с помощью Azure AD Connect. Он не распространяется только на пользователей, облачную.

- Пользователь вхозывает трафик на браузерах и современных клиентов *аутентификации.* Приложения или облачные службы, используюющие устаревшую аутентификацию, будут возвращаться к потокам федеративной аутентификации. Примером может быть Exchange онлайн с современной аутентификацией, или Outlook 2010, которая не поддерживает современную аутентификацию.

## <a name="unsupported-scenarios"></a>Неподдерживаемые сценарии

Следующие сценарии не поддерживаются для поэтапного развертывания:

- Некоторые приложения отправляют параметр запроса "domain_hint" в Azure AD во время проверки подлинности. Эти потоки будут продолжаться, и пользователи, включенные для поэтапного развертывания, будут продолжать использовать федерацию для проверки подлинности.

<!-- -->

- Админы могут развертывать проверку подлинности облака с помощью групп безопасности. Чтобы избежать задержки синхронизации при использовании групп безопасности Active Directory, рекомендуется использовать группы облачной безопасности. Применяются следующие условия:

    - Вы можете использовать максимум 10 групп на объект. То есть, вы можете использовать 10 групп каждый для *синхронизации хэша паролей,* *сквозной аутентификации,* и *бесшовные SSO*.
    - Nested *группы не поддерживаются.* Эта область относится и к общедоступному предварительному просмотру.
    - Динамические группы *не поддерживаются* для поэтапного развертывания.
    - Контактные объекты внутри группы будут блокировать добавление группы.

- Вам все еще нужно сделать окончательный перетекизн от федеративной до облачной аутентификации с помощью Azure AD Connect или PowerShell. Поэтапное развертывание не переключает домены с федеративного на управляемые.

- При первом добавлении группы безопасности для поэтапного развертывания, вы ограничены 200 пользователями, чтобы избежать тайм-аута UX. После добавления группы можно добавить в нее больше пользователей непосредственно по мере необходимости.

Чтобы проверить состояние *синхронизации хэша паролей,* вы можете использовать диагностику PowerShell в [синхронизации хэша паролей Troubleshoot с синхронизацией Azure AD Connect.](tshoot-connect-password-hash-synchronization.md)

## <a name="pre-work-for-pass-through-authentication"></a>Предварительная работа для сквозной аутентификации

1. Определите сервер, который работает под управлением Windows Server 2012 R2 или позже, где вы хотите запустить сквозной агент *аутентификации.* 

   *Не* выбирайте сервер Azure AD Connect.Убедитесь, что сервер соединен доменом, может аутентифицировать выбранных пользователей с помощью Active Directory и может общаться с Azure AD на исходящих портах и URL-адресах. Для получения дополнительной информации, см. "Шаг 1: Проверить предпосылки" [Квикстарт: Azure AD бесшовные одного входного .](how-to-connect-sso-quick-start.md)

1. [Загрузите агент проверки подлинности Azure AD Connect](https://aka.ms/getauthagent)и установите его на сервере. 

1. Для [обеспечения высокой доступности](how-to-connect-sso-quick-start.md)установите дополнительные агенты аутентификации на других серверах.

1. Убедитесь, что вы настроили [настройки Smart Lockout](../authentication/howto-password-smart-lockout.md) соответствующим образом. Это помогает гарантировать, что учетные записи Active Directory ваших пользователей не будут заблокированы плохими субъектами.

Мы рекомендуем включить *бесшовную SSO* независимо от метода регистрации *(синхронизация хэша пароля* или *сквозная аутентификация),* выбранного для поэтапного развертывания. Для обеспечения *бесшовной SSO*следуйте инструкциям по предварительной работе в следующем разделе.

## <a name="pre-work-for-seamless-sso"></a>Предварительная работа для бесшовных SSO

Включите *бесшовные SSO* в лесах Active Directory с помощью PowerShell. Если у вас есть несколько Active Directory лес, включить его для каждого леса индивидуально.  *Бесшовная SSO* срабатывает только для пользователей, отобранных для поэтапного развертывания. Это не влияет на существующую настройку федерации.

Включите *бесшовные SSO,* делая следующее:

1. Вопием на сервер Azure AD Connect Server.

2. Перейдите в *папку\\%programfiles% Microsoft Azure Active Directory Connect.* 

3. Импортируйте *бесшовный* модуль SSO PowerShell, запустив следующую команду: 

   `Import-Module .\AzureADSSO.psd1`

4. Откройте сеанс PowerShell от имени администратора. В PowerShell `New-AzureADSSOAuthenticationContext`звоните . Эта команда открывает панель, где вы можете ввести глобальные учетные данные администратора вашего клиента.

5. Звоните `Get-AzureADSSOStatus | ConvertFrom-Json`. Эта команда отображает список лесов Active Directory (см. список "Домены"), на котором включена эта функция. По умолчанию он устанавливается на ложное значение на уровне арендатора.

   ![Пример выхода Windows PowerShell](./media/how-to-connect-staged-rollout/sr3.png)

6. Звоните `$creds = Get-Credential`. В запросе введите учетные данные администратора домена для предназначенного active Directory forest.

7. Вызовите процедуру `Enable-AzureADSSOForest -OnPremCredentials $creds`. Эта команда создает компьютерную учетную запись A'''READSSOACC из предварительного контроллера домена для леса Active Directory, который необходим для *бесшовного SSO.*

8. *Бесшовные SSO* требует, чтобы URL-адреса были в зоне интрасети. Для развертывания этих [URL-адресов](how-to-connect-sso-quick-start.md#step-3-roll-out-the-feature)с помощью групповых политик см.

9. Для полного пошагового, вы также можете скачать наши [планы развертывания](https://aka.ms/SeamlessSSODPDownload) для *бесшовных SSO*.

## <a name="enable-staged-rollout"></a>Включить поэтапное развертывание

Чтобы выкатить определенную функцию *(сквозная аутентификация,* *синхронизация хэша паролей*или *бесшовная SSO)* для избранного набора пользователей в группе, следуйте инструкциям в следующих разделах.

### <a name="enable-a-staged-rollout-of-a-specific-feature-on-your-tenant"></a>Включить поэтапное развертывание конкретной функции на арендаторе

Вы можете выкатить один из этих вариантов:

- **Вариант Пароль** - *хэш синхронизации* + *бесшовных SSO*
- **Вариант B** - *сквозной аутентификации* + *бесшовных SSO*
- **Не поддерживается** - *синхронизация* + хэша паролей*сквозной проверки подлинности* + *бесшовных SSO*

Выполните следующие действия.

1. Чтобы получить доступ к предварительному просмотру UX, вопием на [портал Azure AD](https://aka.ms/stagedrolloutux).

2. Выберите поэтапное развертывание Enable для управляемой ссылки **для входиного пользователя (Preview).**

   Например, если вы хотите включить *вариант A,* сдвиньте **синхронизацию паролей хэш** **и бесшовные элементы управления одним подвохом** к **On,** как показано на следующих изображениях.

   ![Страница подключения к рекламе Azure](./media/how-to-connect-staged-rollout/sr4.png)

   ![Страница "Включить функции поэтапного развертывания (Preview)" страница](./media/how-to-connect-staged-rollout/sr5.png)

3. Добавьте группы к функции, чтобы обеспечить *сквозную аутентификацию* и *бесшовную SSO.* Чтобы избежать тайм-аута UX, убедитесь, что группы безопасности изначально содержат не более 200 членов.

   ![Страница "Управление группами синхронизации хэш-ассам паролей (Предварительный просмотр)" страница](./media/how-to-connect-staged-rollout/sr6.png)

   >[!NOTE]
   >Участники в группе автоматически включены для поэтапного развертывания. Nested и динамические группы не поддерживаются для поэтапного развертывания.

## <a name="validation"></a>Проверка

Чтобы протестировать *синхронизацию хэша пароля* или *сквозную аутентификацию* (имя пользователя и пароль, сделайте следующее:

1. На экстрасети перейдите на [страницу приложений](https://myapps.microsoft.com) в закрытом сеансе браузера, а затем введите UserPrincipalName (UPN) учетной записи пользователя, выбранной для поэтапного развертывания.

   Пользователи, ставшие мишенью для поэтапного развертывания, не перенаправляются на вашу федеративную страницу входа. Вместо этого им предлагается войти на страницу регистрации арендатора Azure AD.

1. Убедитесь, что входе успешно отображается в [отчете о деятельности Azure AD,](../reports-monitoring/concept-sign-ins.md) фильтруя с помощью UserPrincipalName.

Для тестирования входин ы с *бесшовной SSO:*

1. В интрасети перейдите на [страницу приложений](https://myapps.microsoft.com) в закрытом сеансе браузера, а затем введите UserPrincipalName (UPN) учетной записи пользователя, выбранной для поэтапного развертывания.

   Пользователям, которые были направлены на поэтапное развертывание *бесшовных SSO* представлены с "Попытка подписать вас в ..." сообщение, прежде чем они молча подписали дюйма

1. Убедитесь, что входе успешно отображается в [отчете о деятельности Azure AD,](../reports-monitoring/concept-sign-ins.md) фильтруя с помощью UserPrincipalName.

   Чтобы отслеживать входы пользователей, которые по-прежнему происходят в службах Active Directory Federation Services (AD FS) для отдельных пользователей поэтапного развертывания, следуйте инструкциям при [устранении неполадок AD FS: События и журналирование.](https://docs.microsoft.com/windows-server/identity/ad-fs/troubleshooting/ad-fs-tshoot-logging#types-of-events) Проверьте документацию поставщика о том, как проверить это на сторонних поставщиков федерации.

## <a name="remove-a-user-from-staged-rollout"></a>Удалить пользователя из постановочного развертывания

Удаление пользователя из группы отстраняет от поэтапного развертывания для этого пользователя. Чтобы отключить функцию поэтапного развертывания, сдвиньте элемент управления обратно в **Off.**

## <a name="frequently-asked-questions"></a>Часто задаваемые вопросы

**В: Могу ли я использовать эту возможность в производстве?**

О: Да, вы можете использовать эту функцию в вашем производственном арендаторе, но мы рекомендуем вам сначала попробовать ее в тестируемом арендаторе.

**Вопрос: Может ли эта функция использоваться для поддержания постоянного «сосуществования», когда некоторые пользователи используют федеративное аутентификацию, а другие — проверку подлинности облака?**

О: Нет, эта функция предназначена для перехода от федеративной проверки в облако поэтапно, а затем, чтобы в конечном итоге перейти к облачной аутентификации. Мы не рекомендуем использовать постоянное смешанное состояние, поскольку такой подход может привести к неожиданным потокам аутентификации.

**В: Могу ли я использовать PowerShell для выполнения постановочного развертывания?**

Ответ. Да. Чтобы узнать, как использовать PowerShell для [Azure AD Preview](https://docs.microsoft.com/powershell/module/azuread/?view=azureadps-2.0-preview#staged_rollout)выполнения поэтапного развертывания, см.
