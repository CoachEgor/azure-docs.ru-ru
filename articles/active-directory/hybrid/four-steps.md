# <a name="four-steps-to-a-strong-identity-foundation-with-azure-active-directory"></a>Четыре шага к прочному фундаменту идентификации с Azure Active Directory

Управление доступом к приложениям и данным больше не может полагаться на традиционные стратегии границ сетевой безопасности, такие как сети периметра и брандмауэры из-за быстрого перемещения приложений в облако. Теперь организации должны доверять своему решению, удостоверяющим личность, чтобы контролировать, кто и что имеет доступ к приложениям и данным организации. Все больше организаций разрешают сотрудникам приносить свои собственные устройства для работы и использования своих устройств из любого места, где они могут подключиться к Интернету. Обеспечение соответствия и безопасности этих устройств стало важным фактором в решении идентификации, в котором организация выбирает для реализации. В современном цифровом рабочем месте [идентификация является основной моделью управления](https://www.microsoft.com/security/technology/identity-access-management?rtc=1) любой организацией, перемещаемым в облако.

При внедрении решения гибридной идентификации Azure Active Directory (Azure AD) организации получают доступ к функциям премиум-класса, которые разблокировании производительности за счет автоматизации, делегирования, самообслуживания и возможностей единого входного доступа. Это позволяет вашим сотрудникам получать доступ к ресурсам компании, где им необходимо выполнять свою работу, позволяя иТ-отделу управлять этим доступом, гарантируя, что нужные люди имеют правильный доступ к нужным ресурсам для обеспечения безопасной производительности.

Основываясь на наших знаниях, этот контрольный список рекомендаций поможет вам быстро развернуть рекомендуемые действия для создания *прочной* идентичности в вашей организации:

* Легко подключиться к приложениям
* Установить одну идентификацию для каждого пользователя автоматически
* Уполномочить пользователей безопасно
* Эксплуатация ваших идей

## <a name="step-1---connect-to-apps-easily"></a>Шаг 1 - Легко подключиться к приложениям

Подключив приложения к Azure AD, вы можете повысить производительность и безопасность конечных пользователей, включив один очный учет (SSO) и сделать подготовку пользователей. Управляя приложениями в одном месте, Azure AD, вы можете минимизировать административные накладные расходы и достичь единой точки контроля для политики безопасности и соответствия требованиям.

Этот раздел охватывает ваши возможности для управления доступом пользователей к приложениям, что позволяет обеспечить удаленный доступ к внутренним приложениям, а также преимущества переноса приложений в Azure AD.

### <a name="make-apps-available-to-your-users-seamlessly"></a>Сделайте приложения доступными для пользователей бесшовно

Azure AD позволяет администраторам [добавлять приложения](https://docs.microsoft.com/azure/active-directory/manage-apps/add-application-portal) в галерею корпоративных приложений на [портале Azure.](https://portal.azure.com/) Добавление приложений в галерею приложений Enterprise упрощает настройку приложений для использования Azure AD в качестве поставщика идентификационных данных. Он также позволяет управлять доступом пользователей к приложению с помощью политик условного доступа и настраивать одиночный вход (SSO) на приложения, чтобы пользователям не пришлось вводить свои пароли повторно и автоматически подписывались как на месте, так и на облачные приложения.

После добавления приложений в галерею Azure AD пользователи могут видеть приложения, назначенные им, и искать и запрашивать другие приложения по мере необходимости. Azure AD предоставляет [пользователям несколько способов](https://docs.microsoft.com/azure/active-directory/manage-apps/end-user-experiences) доступа к своим приложениям:

* Панель доступа/Мои приложения
* Средство запуска приложений Office 365
* Прямой вход в федеративные приложения
* Ссылки для прямого входа

Чтобы узнать больше о доступе пользователей к приложениям, **см.**

### <a name="migrate-apps-from-active-directory-federation-services-to-azure-ad"></a>Перенос приложений из службы Федерации активных каталогов в Azure AD

Миграция одной конфигурации от системы регистрации от Active Directory Federation Services (ADFS) до Azure AD обеспечивает дополнительные возможности безопасности, более последовательной управляемости и совместной работы. Для достижения оптимальных результатов мы рекомендуем перенести приложения из AD FS в Azure AD. Приведение аутентификации и авторизации приложения в Azure AD предоставляет вам следующие преимущества:

* Управление расходами
* Управление рисками
* Повышение производительности труда
* Решение проблемы соблюдения и управления

Чтобы узнать больше, смотрите белую бумагу [«Миграция приложений» в «Активный каталог Azure».](https://aka.ms/migrateapps/whitepaper)

### <a name="enable-secure-remote-access-to-apps"></a>Обеспечить безопасный удаленный доступ к приложениям

[Azure AD Application Proxy](https://docs.microsoft.com/azure/active-directory/manage-apps/what-is-application-proxy) предоставляет простое решение для организаций для публикации в облаке приложений для удаленных пользователей, которым необходим безопасный доступ к внутренним приложениям. После одного входного вступления в Azure AD пользователи могут получить доступ как к облачным, так и по корпоративным приложениям через внешние URL-адреса или внутренний портал приложений.

Прокси-приложение Azure AD Прокси предлагает следующие преимущества:

* Расширение Azure AD для использования с локальными ресурсами
  * Безопасность и защита в масштабе облака
  * Такие функции, как условный доступ и многофакторная аутентификация, которые легко позволяют включить
* Нет компонентов в сети периметра, таких как VPN и традиционные решения обратного прокси
* Отсутствие необходимости во входящих соединениях
* Единый входной (SSO) между устройствами, ресурсами и приложениями в облаке и на территории
* Уполномочивает конечных пользователей быть продуктивными в любое время и в любом месте

## <a name="step-2---establish-one-identity-for-every-user-automatically"></a>Шаг 2 - Установить одну личность для каждого пользователя автоматически

Объединение находящихся на основе предварительных и облачных каталогов в гибридном идентификаторном решении Azure AD позволит вам повторно использовать существующие инвестиции active Directory, создав существующие идентификаторы в облаке. Решение синхронизирует локта и тождества с Azure AD, в то время как ИТ-центр ыней active Directory работает с любыми существующими решениями управления в качестве основного источника истины для идентификационных данных. Гибридное решение Microsoft Azure AD для создания идентификаторов охватывает локальные и облачные возможности, создавая общую идентификацию пользователя для проверки подлинности и авторизации для всех ресурсов, независимо от их местоположения.

Интеграция ваших предпосылок с Azure AD делает пользователей более продуктивными и не позволяет пользователям использовать несколько учетных записей в приложениях и службах, предоставляя общую идентификацию для доступа как к облачным, так и к ресурсам. Использование нескольких учетных записей является болевой точкой для конечных пользователей и ИТ- С точки зрения конечного пользователя наличие нескольких учетных записей означает необходимость запоминать несколько паролей. Чтобы избежать этого, многие пользователи повторно используют один и тот же пароль для каждой учетной записи, что плохо с точки зрения безопасности. С точки зрения ИТ повторное использование часто приводит к большему сбросу паролей и затратм на справочную службу наряду с жалобами конечных пользователей.

Azure AD Connect — это инструмент, который используется для синхронизации личных данных с Azure AD, который затем может использоваться для доступа к облачным приложениям. После того, как идентификаторы находятся в Azure AD, они могут предоставлять приложенияS SaaS, такие как Salesforce или Concur.

В этом разделе мы перечисляем рекомендации по обеспечению высокой доступности, современной аутентификации для облака и уменьшению объема вашего присутствия на месте.

> [!NOTE]
> Если вы хотите узнать больше о Azure AD Connect, [см.](https://docs.microsoft.com/azure/active-directory/hybrid/how-to-connect-sync-whatis)

### <a name="set-up-a-staging-server-for-azure-ad-connect-and-keep-it-up-to-date"></a>Настройка промежуточного сервера для Azure AD Connect и поддержание его в актуальном состоянии

Azure AD Connect играет ключевую роль в процессе подготовки. Если Sync Server по какой-либо причине переходит в автономный режим, изменения в помещениях не будут обновляться в облаке и вызывают проблемы с доступом пользователей. Важно определить стратегию неудачи, которая позволяет администраторам быстро возобновить синхронизацию после того, как сервер синхронизации перейдет в автономный режим.

Для обеспечения высокой доступности в случае, если основной сервер Azure AD Connect перейдет в автономный режим, рекомендуется развернуть отдельный [промежуточный сервер](https://docs.microsoft.com/azure/active-directory/hybrid/how-to-connect-sync-staging-server) для Azure AD Connect. Развертывание сервера позволяет администратору «продвигать» промежуточный сервер к производству простым переключателем конфигурации. Настройка резервного сервера в промежуточном режиме также позволяет тестировать и развертывать новые изменения конфигурации и вводить новый сервер при выводе из эксплуатации старого.

> [!TIP]
> Подключение Azure AD обновляется на регулярной основе. Поэтому настоятельно рекомендуется сохранить ток промежуточного сервера, чтобы воспользоваться преимуществами улучшений производительности, исправлений ошибок и новых возможностей, которые предоставляет каждая новая версия.

### <a name="enable-cloud-authentication"></a>Включить проверку подлинности облака

Организациям с предместным active Directory следует расширить свой каталог до Azure AD с помощью Azure AD Connect и настроить соответствующий метод проверки подлинности. [Выбор правильного метода аутентификации](https://docs.microsoft.com/azure/security/fundamentals/choose-ad-authn) для организации является первым шагом в вашем путешествии приложений в облако. Это критический компонент, поскольку он контролирует доступ ко всем облачным данным и ресурсам.

Самый простой и рекомендуемый метод обеспечения проверки подлинности облаков для объектов предварительного каталога в Azure AD — это [синхронизация паролей Хэш](https://docs.microsoft.com/azure/active-directory/hybrid/how-to-connect-password-hash-synchronization) (PHS). Кроме того, некоторые организации могут рассмотреть вопрос о включении [пропуска через аутентификацию](https://docs.microsoft.com/azure/active-directory/hybrid/how-to-connect-pta-quick-start) (PTA).

Независимо от того, выбираете ли вы PHS или PTA, не забудьте включить [бесшовный единый вход,](https://docs.microsoft.com/azure/active-directory/hybrid/how-to-connect-sso) чтобы позволить пользователям получать доступ к облачным приложениям без постоянного ввода их имени пользователя и пароля в приложении при использовании устройств Windows 7 и 8 в корпоративной сети. Без единого ввоза пользователи должны помнить пароли и вписаться в каждое приложение. Аналогичным образом ИТ-персоналу необходимо создавать и обновлять учетные записи пользователей для каждого приложения, например Office 365, Box и Salesforce. Пользователи должны помнить свои пароли, а также тратить время, чтобы войти в каждое приложение. Предоставление стандартизированного единого механизма регистрации для всего предприятия имеет решающее значение для наилучшего пользовательского опыта, снижения риска, способности отчитываться и управления.

Для организаций, уже использующих AD FS или другого поставщика аутентификации на месте, переход на Azure AD в качестве поставщика идентификационных данных может снизить сложность и повысить доступность. Если у вас нет конкретных случаев использования для использования федерации, мы рекомендуем перейти от федеративной аутентификации либо PHS и Seamless SSO или PTA и Seamless SSO, чтобы насладиться преимуществами уменьшенного присутствия на территории и гибкости, с которой предлагает облако улучшилось взаимодействие с пользователями. Для получения дополнительной информации [см.](https://docs.microsoft.com/azure/active-directory/hybrid/plan-migrate-adfs-password-hash-sync)

### <a name="enable-automatic-deprovisioning-of-accounts"></a>Включить автоматическую дезавуирование счетов

Автоматическое обеспечение и деирование приложений является наилучшей стратегией для управления жизненным циклом идентификационных данных в нескольких системах. Azure AD поддерживает [автоматизированную подготовку и деирование](https://docs.microsoft.com/azure/active-directory/manage-apps/configure-automatic-user-provisioning-portal) учетных записей пользователей на различные популярные приложения SaaS, такие как ServiceNow и Salesforce, и другие, которые реализуют [протокол SCIM 2.0.](https://docs.microsoft.com/azure/active-directory/manage-apps/use-scim-to-provision-users-and-groups) В отличие от традиционных решений для подготовки, требующих пользовательского кода или ручной загрузки файлов CSV, служба обеспечения размещается в облаке и имеет предварительно интегрированные разъемы, которые можно настроить и управлять с помощью портала Azure. Ключевым преимуществом автоматической дезадинга является то, что она помогает обезопасить вашу организацию, мгновенно удаляя идентификаторы пользователей из ключевых приложений SaaS, когда они покидают организацию.

Дополнительные сведения об автоматической подготовке учетных записей пользователей см. в статье [Автоматическая подготовка пользователей и ее отзыв для приложений SaaS в Azure Active Directory](https://docs.microsoft.com/azure/active-directory/manage-apps/user-provisioning).

## <a name="step-3---empower-your-users-securely"></a>Шаг 3 - Уполномочить пользователей безопасно

В современном цифровом рабочем месте важно сбалансировать безопасность и производительность. Однако конечные пользователи часто отодвигают меры безопасности, которые замедляют их производительность и доступ к облачным приложениям. Чтобы помочь решить эту проблему, Azure AD предоставляет возможности самообслуживания, которые позволяют пользователям оставаться продуктивными при минимизации административных накладных расходов.

В этом разделе перечислены рекомендации по удалению трений в организации, расширяя возможности пользователей, сохраняя при этом бдительность.

### <a name="enable-self-service-password-reset-for-all-users"></a>Включить сбросить пароли самообслуживания для всех пользователей

[Сбросить паролем самообслуживания](https://docs.microsoft.com/azure/active-directory/authentication/quickstart-sspr) Azure (SSPR) предлагает ИТ-администраторам простое средство, позволяющее пользователям сбросить и разблокировать свои пароли или учетные записи без вмешательства администратора. Эта система предоставляет подробные отчеты, которые позволяют отслеживать доступ к ней, а также создает уведомления в случае несанкционированного использования или применения не по назначению.

По умолчанию Azure AD разблокирует учетные записи при выполнении сброса пароля. Однако при включании [интеграции](https://docs.microsoft.com/azure/active-directory/authentication/concept-sspr-howitworks#on-premises-integration)Azure AD Connect в предварительном помещении у вас также есть возможность разделить эти две операции, которые позволяют пользователям разблокировать свою учетную запись без необходимости сбросить пароль.

### <a name="ensure-all-users-are-registered-for-mfa-and-sspr"></a>Убедитесь, что все пользователи зарегистрированы для МИД и SSPR

Azure предоставляет отчеты, которые могут быть использованы вами и вашей организацией для обеспечения регистрации пользователей на МИД и SSPR. Пользователям, которые не зарегистрировались, может потребоваться обучение по процессу.

Отчет о [входе в](https://docs.microsoft.com/azure/active-directory/authentication/howto-mfa-reporting) систему МИД содержит информацию об использовании МИД и дает вам представление о том, как МИД работает в вашей организации. Наличие доступа к деятельности, всданным (и аудиту и обнаружению рисков) для Azure AD имеет решающее значение для устранения неполадок, анализа использования и проведения судебно-медицинских экспертиз.

Аналогичным образом, [отчет об управлении паролем самообслуживания](https://docs.microsoft.com/azure/active-directory/authentication/howto-sspr-reporting) может быть использован для определения того, кто зарегистрировался (или не зарегистрировался) для SSPR.

### <a name="self-service-app-management"></a>Управление приложениями самообслуживания

Прежде чем пользователи смогут самостоятельно открывать приложения из панели доступа, необходимо включить [доступ к приложениям самообслуживания](https://docs.microsoft.com/azure/active-directory/manage-apps/access-panel-manage-self-service-access) к любым приложениям, которые вы хотите, чтобы пользователи могли самостоятельно обнаруживать и запрашивать доступ к ним. Доступ к приложениям самообслуживания — это отличный способ позволить пользователям самостоятельно открывать приложения и дополнительно разрешить бизнес-группе утверждать доступ к этим приложениям. Можно разрешить бизнес-группе управлять учетными данными, назначенными этим пользователям для [одного пароля на приложениях](https://docs.microsoft.com/azure/active-directory/manage-apps/application-sign-in-problem-password-sso-gallery#configure-password-sso-for-an-azure-ad-gallery-app) прямо из их панелей доступа.

### <a name="self-service-group-management"></a>Самостоятельное управление группами

Назначение пользователей приложениям лучше всего отображаться при использовании групп, поскольку они обеспечивают большую гибкость и способность управлять в масштабе:

* На основе атрибутов с использованием динамического членства в группе
* Делегация владельцам приложений

Azure AD предоставляет возможность управлять доступом к ресурсам с помощью групп безопасности и групп Office 365. Управлять этими группами может владелец группы, который может одобрить или отклонить запросы на членство и делегировать контроль членства в группе. Известная как [управление группами самообслуживания,](https://docs.microsoft.com/azure/active-directory/users-groups-roles/groups-self-service-management)эта функция экономит время, позволяя владельцам групп, которым не назначена административная роль, создавать и управлять группами без необходимости полагаться на администраторов для обработки их запросов.

## <a name="step-4---operationalize-your-insights"></a>Шаг 4 - Оперативные ваши идеи

Аудит и регистрация событий, связанных с безопасностью, и связанных с ними оповещений являются важными компонентами эффективной стратегии, обеспечивающей продуктивность пользователей и безопасность организации. Журналы безопасности и отчеты могут помочь ответить на такие вопросы, как:

* Используете ли вы то, за что платите?
* Есть ли что-нибудь подозрительное или злонамеренное происходит в моем арендаторе?
* На кого повлияли инцидент с безопасностью?

Отчеты и журналы безопасности обеспечивают электронную запись подозрительных действий и помогают выявлять закономерности, которые могут свидетельствовать об успешной или неудачной попытке проникнуть в сеть извне, а также о внутренних атаках. Вы можете использовать аудит для мониторинга активности пользователей, документирования нормативных требований, проведения судебно-медицинского анализа и многое другое. Оповещения содержат уведомления о событиях безопасности.

### <a name="assign-least-privileged-admin-roles-for-operations"></a>Назначать наименее привилегированные роли админовдляй должности для операций

Как вы думаете о вашем подходе к операциям, Есть несколько уровней администрации для рассмотрения. Первый уровень возлагает бремя администрирования на вашего глобального администратора (ы). Всегда используя роль глобального администратора, может быть подходящим для небольших компаний. Но для более крупных организаций с персоналом службы поддержки и администраторами, ответственными за конкретные задачи, назначение роли глобального администратора может быть риском для безопасности, поскольку оно предоставляет этим людям возможность управлять задачами, которые находятся выше и вне что они должны быть способны делать.

В этом случае следует рассмотреть следующий уровень администрирования. Используя Azure AD, можно назначить конечных пользователей как «ограниченных администраторов», которые могут управлять задачами в менее привилегированных ролях. Например, можно назначить персоналу службы поддержки роль [читателя безопасности,](https://docs.microsoft.com/azure/active-directory/users-groups-roles/directory-assign-admin-roles#security-reader) чтобы предоставить им возможность управлять функциями, связанными с безопасностью, с доступом только для чтения. Или, возможно, имеет смысл назначить роль [администратора аутентификации](https://docs.microsoft.com/azure/active-directory/users-groups-roles/directory-assign-admin-roles#authentication-administrator) отдельным лицам, чтобы дать им возможность сбросить учетные данные без пароля или читать и настраивать службу Azure Health.

Чтобы узнать больше, смотрите [разрешения роли администратора в Active Directory Azure.](https://docs.microsoft.com/azure/active-directory/users-groups-roles/directory-assign-admin-roles)

### <a name="monitor-hybrid-components-azure-ad-connect-sync-ad-fs-using-azure-ad-connect-health"></a>Мониторинг гибридных компонентов (синхронизация Azure AD Connect, AD FS) с помощью Azure AD Connect Health

Azure AD Connect и AD FS являются важнейшими компонентами, которые потенциально могут привести к разрыву управления жизненным циклом и аутентификации и в конечном итоге привести к простоям. Поэтому необходимо развернуть Систему подключения Azure для мониторинга и отчетности этих компонентов.

Чтобы узнать больше, прочитайте [Monitor AD FS с помощью Azure AD Connect Health.](https://docs.microsoft.com/azure/active-directory/hybrid/how-to-connect-health-adfs)

### <a name="use-azure-monitor-to-collect-data-logs-for-analytics"></a>Используйте Azure Monitor для сбора журналов данных для аналитики

[Azure Monitor](https://docs.microsoft.com/azure/azure-monitor/overview) — это единый портал мониторинга для всех журналов Azure AD, который обеспечивает глубокое понимание, расширенную аналитику и интеллектуальное машинное обучение. С помощью Azure Monitor можно использовать метрики и журналы в портале и с помощью AA, чтобы получить более подробную информацию о состоянии и производительности ресурсов. Это позволяет одно стеклянное стекло опыт в портале, позволяя широкий спектр интеграции продуктов через AIS и варианты экспорта данных, которые поддерживают традиционные сторонние системы SIEM. Azure Monitor также дает вам возможность настроить правила оповещения, чтобы получать уведомления или принимать автоматизированные действия по проблемам, влияющим на ваши ресурсы.

### <a name="create-custom-dashboards-for-your-leadership-and-your-day-to-day"></a>Создавайте пользовательские панели мониторинга для вашего руководства и повседневного руководства

Организации, не распорядившиеся на решении SIEM, могут загрузить [пакет содержимого Power BI](https://docs.microsoft.com/azure/active-directory/reports-monitoring/howto-power-bi-content-pack) для Azure AD. Пакет содержимого Power BI содержит готовые отчеты, которые помогут вам понять, как пользователи принимают и используют функции Azure AD, что позволяет получить представление обо всех действиях в каталоге. Вы также можете создать свою собственную [панель мониторинга](https://docs.microsoft.com/power-bi/service-dashboards) и поделиться с руководством команды, чтобы сообщить о повседневных мероприятиях. Панели мониторинга являются отличным способом для мониторинга вашего бизнеса и увидеть все ваши наиболее важные метрики с первого взгляда. Визуализации на панели мониторинга могут поступать из одного или нескольких базовых наборов данных и из одного или нескольких базовых отчетов. Общие проблемы, такие как забытые пароли и блокировки учетных записей, смягчаются путем реализации сбоя пароля самообслуживания Azure, в то время как предоставление доступа к приложениям самообслуживания позволяет пользователям самостоятельно открывать и запрашивать доступ к приложениям, не полагаясь на ИТ-персонале.

Если вы не наблюдаете сокращения вызовов поддержки, мы рекомендуем вам проанализировать драйверы вызовов поддержки в попытке подтвердить, правильно ли настроен доступ к SSPR или приложению самообслуживания или если есть какие-либо другие новые проблемы, которые могут быть систематически Рассмотрены.
