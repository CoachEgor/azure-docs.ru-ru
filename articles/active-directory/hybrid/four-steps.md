# <a name="four-steps-to-a-strong-identity-foundation-with-azure-active-directory"></a>Четыре шага для строгой идентификации основу с Azure Active Directory

Управление доступом к приложениям и данным больше не могут полагаться на таких периметра сети и брандмауэры стратегий границ безопасности традиционной сети из-за быстрого перемещения приложений в облако. Теперь организации должны доверять решение удостоверения, чтобы контролировать, кто и что есть доступ к приложениям и данным организации. Все больше организаций позволяя сотрудникам приносить собственные устройства, необходимо использовать свои устройства из везде, где они могут подключаться к Интернету. Убедитесь, что эти устройства, соответствия и безопасность становится важным аспектом в вами организации принято решение реализовать решения для идентификации. В современном мире цифровых [Идентификация является плоскостью управления основной](https://www.microsoft.com/security/technology/identity-access-management?rtc=1) любой организации, переход в облако.

Внедрения гибридной идентификации Azure Active Directory (Azure AD), организации получают доступ к расширенным функциям, которые разблокировать производительность труда за счет автоматизации, делегирование, самообслуживания и единого возможности единого входа. Он позволяет сотрудникам доступ к ресурсам компании в любом месте выполнять свою работу, предоставляя ИТ-специалисты для управления такой доступ, гарантируя, что людей, которым права доступа к ресурсам для установления безопасной работы.

Исходя из результатов работы, этот контрольный список рекомендаций поможет вам быстро развернуть рекомендуемые действия по построению *строгого* identity foundation в вашей организации:

* Легко подключаться к приложениям
* Может автоматически устанавливать одно удостоверение для каждого пользователя
* Позвольте пользователям безопасно
* Ввод в эксплуатацию аналитических данных

## <a name="step-1---connect-to-apps-easily"></a>Шаг 1 - легко подключаться к приложениям

Подключив приложений с Azure AD, можно повысить эффективность работы конечных пользователей и безопасность путем включения единого входа (SSO) и выполнять подготовку пользователей. По управлению приложениями в одном месте, Azure AD, можно свести к минимуму административные издержки и обеспечить единую точку управления для политик безопасности и соответствия требованиям.

В этом разделе рассматриваются параметры для управления доступом пользователей к приложениям, предоставляя безопасный удаленный доступ к внутренние приложения, а также преимущества переносу приложений в Azure AD.

### <a name="make-apps-available-to-your-users-seamlessly"></a>Сделать приложения доступными для пользователей без проблем

Azure AD позволяет администраторам [добавить приложения](https://docs.microsoft.com/azure/active-directory/manage-apps/add-application-portal) в коллекции приложений предприятия в [портала Azure](https://portal.azure.com/). Добавление приложений в коллекции приложений Enterprise упрощает настройку приложений для использования Azure AD в качестве поставщика удостоверений. Он также позволяет управлять доступом пользователей к приложению с помощью политик условного доступа и настроить единый вход (SSO) для приложений, позволяя пользователям не нужно вводить пароль несколько раз и автоматически вошли в локальной и облачные приложения.

После добавления приложений в коллекции Azure AD, пользователи могут видеть приложения, которые им назначены и поиска, а также запросить другие приложения, при необходимости. Azure AD предоставляет [несколько методов](https://docs.microsoft.com/azure/active-directory/manage-apps/end-user-experiences) пользователям получить доступ к приложениям:

* Панель доступа / Мои приложения
* Средство запуска приложений Office 365
* Прямой вход в федеративные приложения
* Ссылки для прямого входа

Дополнительные сведения о доступе пользователей к приложениям, см. в разделе **шаг 3 — Предоставление пользователям** в этой статье.

### <a name="migrate-apps-from-active-directory-federation-services-to-azure-ad"></a>Перенос приложений из службы федерации Active Directory в Azure AD

Миграция единый вход конфигурация из службы федерации Active Directory (ADFS) в Azure AD позволяет дополнительные возможности безопасности, управляемости, более согласованным и совместной работы. Для получения оптимальных результатов рекомендуется переносить ваши приложения от AD FS в Azure AD. Перенос приложения проверки подлинности и авторизации в Azure AD обеспечивает следующие преимущества:

* Управление расходами
* Управление рисками
* Повышение производительности
* Обеспечение соответствия и управления

Дополнительные сведения см. в разделе [миграция Your приложений в Azure Active Directory](https://aka.ms/migrateapps/whitepaper) Технический документ.

### <a name="enable-secure-remote-access-to-apps"></a>Включить безопасный удаленный доступ к приложениям

[Прокси приложения Azure AD](https://docs.microsoft.com/azure/active-directory/manage-apps/what-is-application-proxy) обеспечивает простое решение для организаций, для публикации локальных приложений в облако для удаленных пользователей, которым требуется доступ к внутренним приложениям в безопасном режиме. После единого входа в Azure AD пользователям облачных и локальных приложений через URL-адреса внешнего или внутреннего приложения портала.

Прокси приложения Azure AD предоставляет следующие преимущества:

* Расширение Azure AD для использования с локальными ресурсами
  * Безопасность и защита в масштабе облака
  * Функции, такие как условный доступ и многофакторную проверку подлинности, которые легко включить
* Компоненты, не в сети периметра, такие как VPN и решения традиционных обратного прокси-сервера
* Отсутствие необходимости во входящих соединениях
* Единый вход (SSO) на устройствах, ресурсы и приложения в облаке и локальной
* Дает пользователям возможность продуктивно работать в любое время и в любом месте

## <a name="step-2---establish-one-identity-for-every-user-automatically"></a>Шаг 2 - может автоматически устанавливать одно удостоверение для каждого пользователя

Объединение локальных и облачных каталогов в гибридной идентификации Azure AD дает возможность повторно использовать имеющиеся инвестиции в локальной Active Directory, подготовив существующих идентификаторов в облаке. Решение синхронизирует локальных удостоверений с Azure AD, при сохраняет ИТ в локальной Active Directory, все существующие решения управления, выполнив как основной источник истины для удостоверения. Microsoft Azure AD гибридной идентификации диапазонов локальными и облачными возможностями, создание общую систему идентификации пользователей для проверки подлинности и авторизации ко всем ресурсам, независимо от их расположения.

Интеграция локальных каталогов с Azure AD позволяет повысить производительность и запрещает использовать несколько учетных записей в приложениях и службах по предоставлению единого удостоверения для доступа к облачных, так и к локальным ресурсам. Использование нескольких учетных записей — становится болевой точкой для конечных пользователей и ИТ одинаково. С точки зрения конечных пользователей наличие нескольких означает учетные записи, нужно запоминать несколько паролей. Чтобы избежать этого, многие пользователи повторно использовать один и тот же пароль для каждой учетной записи, что плохо с точки зрения безопасности. С точки зрения ИТ повторное использование часто приводит к более сброс пароля и затраты на техническую поддержку вместе с жалоб конечных пользователей.

Azure AD Connect — это средство, которое используется для синхронизации своих локальных удостоверений в Azure AD, который затем может использоваться для доступа к облачным приложениям. После определения удостоверений Azure AD, они могут подготавливать к приложениям SaaS, таких как Salesforce или Concur.

В этом разделе перечислены рекомендации для обеспечения высокой доступности, современная проверка подлинности для облака и снижая объем локальной.

> [!NOTE]
> Если вы хотите узнать больше об Azure AD Connect, см. в разделе [что такое синхронизация Azure AD Connect?](https://docs.microsoft.com/azure/active-directory/hybrid/how-to-connect-sync-whatis)

### <a name="set-up-a-staging-server-for-azure-ad-connect-and-keep-it-up-to-date"></a>Настройка промежуточного сервера для Azure AD Connect и поддерживать актуальность

Azure AD Connect играет ключевую роль в процессе подготовки. Если сервер синхронизации переходит в автономный режим для какой-либо причине, изменения в локальную не будут обновлены в облаке и вызвать проблемы с доступом для пользователей. Очень важно определить стратегию отработки отказа, которая позволяет администраторам быстро возобновить синхронизацию, после отключения сервера синхронизации.

Для обеспечения высокой доступности в событие основной Azure AD Connect сервер переходит в автономный режим, рекомендуется развернуть отдельный [промежуточного сервера](https://docs.microsoft.com/azure/active-directory/hybrid/how-to-connect-sync-staging-server) для Azure AD Connect. Развертывание сервера позволяет администраторам «повысить» на промежуточный сервер в рабочей среде с помощью простой конфигурационный параметр. Наличие на резервный сервер, настроенный в промежуточный режим также позволяет протестировать развертывание новых изменений конфигурации и создать новый сервер, если списание старого.

> [!TIP]
> Azure AD Connect обновляется на регулярной основе. Таким образом настоятельно рекомендуется не отключать сервер в промежуточной текущего, чтобы воспользоваться преимуществами повышение производительности, исправления ошибок и новые возможности, предоставляемые каждой новой версии.

### <a name="enable-cloud-authentication"></a>Включить проверку подлинности в облаке

Организации с локальной Active Directory необходимо расширить свой каталог Azure AD с помощью Azure AD Connect и Настройка метода проверки подлинности. [Выбор правильного метода аутентификации](https://docs.microsoft.com/azure/security/azure-ad-choose-authn) для вашей организации является первым шагом в путях взаимодействия переноса приложений в облако. Он является критическим компонентом, так как он позволяет управлять доступом ко всем данных в облаке и ресурсам.

Самый простой и рекомендуемый метод для включения облачной проверки подлинности для локальных объектов каталога в Azure AD является возможность [синхронизации хэшированных паролей](https://docs.microsoft.com/azure/active-directory/hybrid/how-to-connect-password-hash-synchronization) (PHS). Кроме того, в некоторых организациях может рассмотрите возможность включения [сквозной проверки подлинности](https://docs.microsoft.com/azure/active-directory/hybrid/how-to-connect-pta-quick-start) (PTA).

Независимо от выбранного PHS или PTA, не забудьте включить [простого единого входа](https://docs.microsoft.com/azure/active-directory/hybrid/how-to-connect-sso) позволяет пользователям получать доступ к облачным приложениям без постоянно ввода имени пользователя и пароля в приложении, при использовании Windows 7 и 8 устройств в вашей организации сеть. Без единого входа пользователи должны помнить пароли конкретного приложения и войдите в каждое приложение. Аналогичным образом ИТ-специалистам необходимо создавать и обновлять учетные записи пользователей для каждого приложения, такие как Office 365, Box и Salesforce. Пользователи должны запоминать свои пароли, а также тратить время на вход в каждое приложение. Предоставляя стандартизированные единый вход механизм для всего предприятия имеет решающее значение для удобства работы пользователей, сокращения риска, возможность передачи и управления.

Для организаций, которые уже используют AD FS или другой локальный поставщик проверки подлинности перемещение в Azure AD в качестве поставщика удостоверений можно уменьшить сложность и повысить уровень доступности. Если у вас нет определенного варианта использования федерации, мы рекомендуем перейти с федеративной аутентификации, PHS и простого единого входа PTA или простого единого входа, чтобы пользоваться преимуществами ограниченной локально занимаемое место и очень гибкий в настройке облака с Улучшенная взаимодействия с пользователями. Дополнительные сведения см. в разделе [Миграция с федерации на синхронизацию хэшей паролей для Azure Active Directory](https://docs.microsoft.com/azure/active-directory/hybrid/plan-migrate-adfs-password-hash-sync).

### <a name="enable-automatic-deprovisioning-of-accounts"></a>Включить автоматическое отзыв учетных записей

Включить автоматическую подготовку и ее отзыв для приложений — это лучшая стратегия для управления жизненным циклом удостоверений в нескольких системах. Azure AD поддерживает [автоматической основе политик, подготовки и отзыва](https://docs.microsoft.com/azure/active-directory/manage-apps/configure-automatic-user-provisioning-portal) учетных записей пользователей для различных популярных приложений SaaS такие как ServiceNow, Salesforce и другим пользователям, которые реализуют [SCIM 2.0 протокол](https://docs.microsoft.com/azure/active-directory/manage-apps/use-scim-to-provision-users-and-groups). В отличие от традиционных решений подготовки, которые требуют пользовательский код или ручной отправки CSV-файлов, служба подготовки размещается в облаке, и предлагает предварительно интегрированную соединители, которые могут настраиваться и управляться с помощью портала Azure. Ключевым преимуществом автоматического отзыва является возможность обезопасить свою организацию, мгновенно удаляя из ключевых приложений SaaS удостоверения пользователей, когда они покидают организацию.

Дополнительные сведения об автоматической подготовке учетных записей пользователей см. в статье [Автоматическая подготовка пользователей и ее отзыв для приложений SaaS в Azure Active Directory](https://docs.microsoft.com/azure/active-directory/manage-apps/user-provisioning).

## <a name="step-3---empower-your-users-securely"></a>Шаг 3 — Предоставление пользователям безопасно

В современном мире цифровых важно, чтобы сбалансировать безопасность с эффективностью. Тем не менее конечные пользователи часто откладывают переход на мер обеспечения безопасности, снизить их работы и доступа к облачным приложениям. Чтобы решить эту проблему, Azure AD предоставляет возможности самообслуживания, которые позволяют пользователям сохранить высокую производительность труда, одновременно снижая затраты на администрирование.

В этом разделе приводятся рекомендации по удалению трения из вашей организации, предоставляя пользователям, сохраняя при этом бдительность.

### <a name="enable-self-service-password-reset-for-all-users"></a>Включение самостоятельного сброса паролей для всех пользователей

Azure [самостоятельного сброса пароля](https://docs.microsoft.com/azure/active-directory/authentication/quickstart-sspr) (SSPR) предлагает простой означает, что для ИТ-администраторам, разрешить пользователям сбрасывать и разблокировать их пароли и учетные записи без вмешательства администратора. Эта система предоставляет подробные отчеты, которые позволяют отслеживать доступ к ней, а также создает уведомления в случае несанкционированного использования или применения не по назначению.

По умолчанию Azure AD разблокирует учетные записи при выполнении сброса пароля. Тем не менее, при включении Azure AD Connect [интеграцию локальных](https://docs.microsoft.com/azure/active-directory/authentication/concept-sspr-howitworks#on-premises-integration), у вас также есть возможность разделить эти две операции, которые позволяют пользователям разблокировать свою учетную запись без сброса пароля.

### <a name="ensure-all-users-are-registered-for-mfa-and-sspr"></a>Убедитесь, что все пользователи зарегистрированы для многофакторной проверки Подлинности и SSPR

Azure предоставляет отчеты, которые могут использоваться для вас и вашей организации, чтобы убедиться, что пользователи зарегистрированы для многофакторной проверки Подлинности и SSPR. Пользователей, которые еще не зарегистрированы может потребоваться донести информацию о процессе.

MFA [отчет о действиях входа](https://docs.microsoft.com/azure/active-directory/authentication/howto-mfa-reporting) включает в себя сведения об использовании многофакторной проверки Подлинности и предоставляет сведений о том, как работает многофакторная проверка Подлинности в вашей организации. Наличие доступа для входа в действие (и аудита и событий риска) для Azure AD имеет решающее значение для устранения неполадок, аналитика использования и расследований.

Аналогичным образом [отчетов самостоятельного управления паролями](https://docs.microsoft.com/azure/active-directory/authentication/howto-sspr-reporting) позволяют определить, кто имеет (или не) зарегистрированным для SSPR.

### <a name="self-service-app-management"></a>Управление приложениями самообслуживания

Прежде чем пользователи могут самостоятельно находить приложения панели доступа, необходимо включить [самостоятельный доступ к приложениям](https://docs.microsoft.com/azure/active-directory/manage-apps/access-panel-manage-self-service-access) для любых приложений, которые вы хотите разрешить пользователям самостоятельно находить и запрашивать доступ к. Доступ к приложению самообслуживания — это отличный способ разрешить пользователям самостоятельно находить приложения и при необходимости разрешить бизнес-группе утверждать доступ к этим приложениям. Вы можете разрешить бизнес-группе управлять учетными данными пользователей для [пароль единого входа в приложения](https://docs.microsoft.com/azure/active-directory/manage-apps/application-sign-in-problem-password-sso-gallery#configure-the-application-for-password-single-sign-on) прямо на панелях доступа.

### <a name="self-service-group-management"></a>Самостоятельное управление группами

Назначение пользователей для приложений лучше всего сопоставлено при использовании групп, так как они обеспечивают большую гибкость и возможность управления в масштабе:

* С помощью динамического членства на основе атрибутов
* Делегирование владельцам приложений

Azure AD предоставляет возможность управлять доступом к ресурсам с помощью групп безопасности и групп Office 365. Эти группы можно управлять владельцем группы, можно утверждать или отклонять запросы на членство и делегировать управление членством. Известный как [Самостоятельное управление группами](https://docs.microsoft.com/azure/active-directory/users-groups-roles/groups-self-service-management), эта функция экономит время, позволяет владельцам группы, которым не назначены роли администратора создавать группы и управлять ими без необходимости использования администраторов для обработки запросов.

## <a name="step-4---operationalize-your-insights"></a>Шаг 4 — ввод в эксплуатацию аналитических данных

Аудит и ведение журнала событий, связанных с безопасностью и связанные оповещения являются важными компонентами эффективной стратегии, чтобы оставаться продуктивными, пользователи и организации является безопасным. Журналы безопасности и отчеты помогут ответ на вопрос такие как:

* Используете ли вы, что вы платите?
* Есть что-то подозрительные или злонамеренного происходит в моем клиенте?
* Кто изменилась во время инцидента безопасности?

Отчеты и журналы безопасности обеспечивают электронную запись подозрительных действий и помогают выявлять закономерности, которые могут свидетельствовать об успешной или неудачной попытке проникнуть в сеть извне, а также о внутренних атаках. Аудит можно использовать для отслеживания действий пользователей, нормативных документов, выполните криминалистического анализа и многое другое. Оповещения содержат уведомления о событиях безопасности.

### <a name="assign-least-privileged-admin-roles-for-operations"></a>Назначение наименьшие права ролей администратора для операций

При выборе подхода к операции, существует несколько уровней администрирования, которые следует учитывать. Первый уровень возлагается администрирования на глобальные Администраторы. Всегда использовать роль глобального администратора, может подходить для небольших компаний. Но для более крупных организациях с сотрудников службы поддержки и администраторов, ответственных за определенные задачи назначения роли глобального администратора может представлять угрозу безопасности так как оно предоставляет пользователям возможность управлять задачами, которые являются больше и шире что они должны быть возможностями.

В этом случае следует следующий уровень администрирования. С помощью Azure AD, можно назначить пользователям как «ограниченные Администраторы», которые могут управлять задачи в роли с меньшими правами доступа. Например, можно назначить вашей помощи персоналу службы поддержки [читатель безопасности](https://docs.microsoft.com/azure/active-directory/users-groups-roles/directory-assign-admin-roles#security-reader) роли, чтобы предоставить им возможность управлять функций безопасности с доступом только для чтения. Или возможно имеет смысл назначить [проверки подлинности администратора](https://docs.microsoft.com/azure/active-directory/users-groups-roles/directory-assign-admin-roles#authentication-administrator) роли для отдельных пользователей, чтобы предоставить им возможность сбросить учетные данные без использования пароля или считать и настроить работоспособности служб Azure.

Дополнительные сведения см. в разделе [разрешения роли администратора в Azure Active Directory](https://docs.microsoft.com/azure/active-directory/users-groups-roles/directory-assign-admin-roles).

### <a name="monitor-hybrid-components-azure-ad-connect-sync-ad-fs-using-azure-ad-connect-health"></a>Мониторинг гибридных компонентов (Azure AD Connect sync, AD FS) с помощью Azure AD Connect Health

Azure AD Connect и AD FS являются критически важных компонентов, которые могут нарушить работу управление жизненным циклом и проверки подлинности и в конечном итоге привести к простоев. Таким образом следует развернуть Azure AD Connect Health для наблюдения и создания отчетов из этих компонентов.

Дополнительные сведения, перейдите в чтения [монитор AD FS, с помощью Azure AD Connect Health](https://docs.microsoft.com/azure/active-directory/hybrid/how-to-connect-health-adfs).

### <a name="use-azure-monitor-to-collect-data-logs-for-analytics"></a>Используйте Azure Monitor для сбора данных журналов для аналитики

[Azure Monitor](https://docs.microsoft.com/azure/azure-monitor/overview) — это единый портал наблюдения для всех журналов в Azure AD, который предоставляет ценную информацию, расширенной аналитики и интеллектуальных машинного обучения. С помощью Azure Monitor вы можете использовать метрики и журналы на портале, а также через интерфейс API, чтобы получить дополнительные сведения о состоянии и производительности ресурсов. Он включает одну область работы стекла на портале при включении широкий спектр интеграции продуктов с помощью API-интерфейсов и данные параметры экспорта, которые поддерживают традиционных систем SIEM сторонних. Кроме того, Azure Monitor предоставляет возможность настройки правил генерации оповещений для уведомления или выполнять автоматизированные действия для проблем, влияющих на ресурсы.

### <a name="create-custom-dashboards-for-your-leadership-and-your-day-to-day"></a>Создание настраиваемых панелей мониторинга для вашего руководства и дня в день

Можно загрузить организаций, у которых нет решения SIEM [пакет содержимого Power BI](https://docs.microsoft.com/azure/active-directory/reports-monitoring/howto-power-bi-content-pack) для Azure AD. Power BI, содержимого, что пакет содержит предварительно созданные отчеты помогут вам понять, как ваши пользователи внедряют и использовать функции Azure AD, что позволяет получить ценные сведения обо всех действиях, связанных в каталоге. Можно также создать свои собственные [пользовательской панели мониторинга](https://docs.microsoft.com/power-bi/service-dashboards) и совместное использование с коллегами лидерство, чтобы создать отчет для повседневной работы. Панели мониторинга — отличный способ отслеживать свой бизнес и просматривать все наиболее важные метрики с первого взгляда. Панель мониторинга объединяет локальных и облачных данных, предоставляя единое представление независимо от того, где находятся данные.

Распространенные проблемы такие, как забытых паролей и блокировки учетных записей предотвращаются путем реализации Azure самостоятельного сброса пароля, при включении самостоятельный доступ к приложениям позволяет пользователям самостоятельно находить и запрашивать доступ к приложениям, не полагаясь ИТ-специалистов.
