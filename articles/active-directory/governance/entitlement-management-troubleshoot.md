# <a name="troubleshoot-azure-ad-entitlement-management"></a>Устранение неполадок управления назначениями Azure AD

В этой статье описываются некоторые элементы, которые необходимо проверить, чтобы помочь в устранении неполадок управления обслуживанием Azure Active Directory (Azure AD).

## <a name="administration"></a>Администрирование

* Если при настройке управления назначением вы получаете сообщение об отказе в доступе и являетесь глобальным администратором, убедитесь, что в каталоге есть [лицензия Azure AD Premium P2 (или EMS)](entitlement-management-overview.md#license-requirements).

* При получении сообщения об отказе в доступе при создании или просмотре пакетов доступа, если вы являетесь членом группы создателей каталога, необходимо [создать каталог](entitlement-management-catalog-create.md) перед созданием первого пакета Access.

## <a name="resources"></a>Ресурсы

* Роли для приложений определяются самим приложением и управляются в Azure AD. Если приложение не имеет ролей ресурсов, управление назначением назначает пользователям роль **доступа по умолчанию** .

    Обратите внимание, что в портал Azure также могут отображаться субъекты-службы для служб, которые не могут быть выбраны в качестве приложений.  В частности, **Exchange Online** и **SharePoint Online** являются службами, а не приложениями, имеющими роли ресурсов в каталоге, поэтому они не могут быть включены в пакет Access.  Вместо этого используйте Групповое лицензирование, чтобы установить подходящую лицензию для пользователя, которому нужен доступ к этим службам.

* Чтобы группа была ресурсом в пакете Access, она должна быть доступна для изменения в Azure AD.  Группы, поступающие в локальную Active Directory, не могут быть назначены как ресурсы, так как их владелец или атрибуты членов нельзя изменять в Azure AD.   Группы, созданные в Exchange Online в качестве групп распространения, не могут быть изменены в Azure AD. 

* Библиотеки документов SharePoint Online и отдельные документы не могут быть добавлены в качестве ресурсов.  Вместо этого создайте [группу безопасности Azure AD](../fundamentals/active-directory-groups-create-azure-portal.md), включите эту группу и роль сайта в пакет Access, а в SharePoint Online используйте эту группу для управления доступом к библиотеке документов или документу.

* Если есть пользователи, которые уже назначены ресурсу, которым вы хотите управлять с помощью пакета Access, убедитесь, что пользователи назначены пакету доступа с соответствующей политикой. Например, может потребоваться включить группу в пакет Access, который уже содержит пользователей в группе. Если пользователям в группе требуется постоянный доступ, они должны иметь соответствующую политику для пакетов доступа, чтобы они не потеряли доступ к группе. Пакет доступа можно назначить, запросив пользователей запросить пакет доступа, содержащий этот ресурс, или напрямую назначить его пакету доступа. Дополнительные сведения см. в разделе [изменение параметров запросов и утверждений для пакета Access](entitlement-management-access-package-request-policy.md).

* При удалении члена команды они также удаляются из группы Office 365. Удаление из функции разговора команды может быть отложено. Дополнительные сведения см. в разделе [членство в группе](https://docs.microsoft.com/microsoftteams/office-365-groups#group-membership).

* Убедитесь, что каталог не настроен для работы с несколькими географическими настройками. Сейчас Управление назначениями не поддерживает несколько географических расположений для SharePoint Online. Сайты SharePoint Online должны находиться в географическом расположении по умолчанию, чтобы управлять ими. Дополнительные сведения см. [в статье поддержка нескольких географических возможностей в OneDrive и SharePoint Online](https://docs.microsoft.com/Office365/Enterprise/multi-geo-capabilities-in-onedrive-and-sharepoint-online-in-office-365).

## <a name="external-users"></a>Внешние пользователи

* Когда внешний пользователь хочет запросить доступ к пакету Access, убедитесь, что он использует **ссылку на портал доступа** для пакета Access. Дополнительные сведения см. [в разделе Совместное использование ссылки для запроса пакета Access](entitlement-management-access-package-settings.md). Если внешний пользователь просто посещает **myaccess.Microsoft.com** и не использует ссылку на портал «полный доступ», он увидит доступные пакеты доступа в своей организации, а не в вашей организации.

* Если внешний пользователь не может запросить доступ к пакету Access или не может получить доступ к ресурсам, обязательно проверьте [параметры для внешних пользователей](entitlement-management-external-users.md#settings-for-external-users).

* Если новый внешний пользователь, который ранее не входил в каталог, получает пакет Access, включая сайт SharePoint Online, его пакет доступа будет отображаться как не полностью доставленный, пока учетная запись не будет подготовлена в SharePoint Online. Дополнительные сведения о параметрах общего доступа см. [в статье Просмотр параметров внешнего общего доступа в SharePoint Online](entitlement-management-external-users.md#review-your-sharepoint-online-external-sharing-settings).

## <a name="requests"></a>Requests (Запросы)

* Когда пользователь хочет запросить доступ к пакету Access, убедитесь, что он использует **ссылку на портал доступа** для пакета Access. Дополнительные сведения см. [в разделе Совместное использование ссылки для запроса пакета Access](entitlement-management-access-package-settings.md).

* При открытии портала My Access с браузером, настроенным в режим "в частном режиме" или "режиме инкогнито", это может конфликтовать с поведением входа. При посещении портала "мой доступ" рекомендуется не использовать режим "в частном режиме" или "режиме инкогнито" для браузера.

* Если пользователь, который еще не находится в каталоге, входит в систему на портале доступа для запроса пакета Access, убедитесь, что он прошел проверку подлинности с помощью учетной записи организации. Учетная запись организации может быть либо учетной записью в каталоге ресурсов, либо каталогом, включенным в одну из политик пакета Access. Если учетная запись пользователя не является учетной записью организации или каталог, в котором она проверяются, не включена в политику, пользователь не увидит пакет Access. Дополнительные сведения см. [в разделе запрос доступа к пакету Access](entitlement-management-request-access.md).

* Если пользователю запрещено входить в каталог ресурсов, он не сможет запросить доступ на портале My Access. Прежде чем пользователь сможет запросить доступ, необходимо удалить блок входа из профиля пользователя. Чтобы удалить блок входа, в портал Azure щелкните **Azure Active Directory**, выберите **Пользователи**, щелкните пользователя, а затем щелкните **профиль**. Измените раздел **Параметры** и измените параметр **блокировать вход** на **нет**. Дополнительные сведения см. в разделе [Добавление или обновление сведений о профиле пользователя с помощью Azure Active Directory](../fundamentals/active-directory-users-profile-azure-portal.md).  Можно также проверить, заблокирован ли пользователь из-за [политики защиты идентификации](../identity-protection/howto-unblock-user.md).

* Если пользователь одновременно является инициатором и утверждающим, на портале доступа он не увидит запрос пакета Access на странице **утверждения** . Это намеренное поведение — пользователь не может утверждать собственный запрос. Убедитесь, что запрашиваемый пакет доступа имеет дополнительные утверждающие, настроенные для политики. Дополнительные сведения см. в разделе [изменение параметров запросов и утверждений для пакета Access](entitlement-management-access-package-request-policy.md).

### <a name="view-a-requests-delivery-errors"></a>Просмотр ошибок доставки запроса

**Предварительная роль:** Глобальный администратор, администратор пользователей, владелец каталога или диспетчер пакетов Access

1. На портале Azure щелкните **Azure Active Directory** и выберите **Управление удостоверениями**.

1. В меню слева щелкните **доступ к пакетам** и откройте пакет Access.

1. Щелкните **запросы**.

1. Выберите запрос, который требуется просмотреть.

    Если запрос содержит ошибки доставки, состояние запроса будет **не доставлено или** **частично доставлено**.

    При наличии ошибок доставки в области сведений о запросе отображается число ошибок доставки.

1. Щелкните число, чтобы просмотреть все ошибки доставки запроса.

### <a name="reprocess-a-request"></a>Повторная обработка запроса

Если запрос встречает ошибку, можно повторно обработать запрос, чтобы повторить попытку. Можно выполнить только повторную обработку запроса, состояние **доставки которого не выполнено** или **частично доставлено** , а Дата завершения — менее одной недели.

**Предварительная роль:** Глобальный администратор, администратор пользователей, владелец каталога или диспетчер пакетов Access

1. На портале Azure щелкните **Azure Active Directory** и выберите **Управление удостоверениями**.

1. В меню слева щелкните **доступ к пакетам** и откройте пакет Access.

1. Щелкните **запросы**.

1. Щелкните запрос, который требуется повторно обработать.

1. В области сведения о запросе щелкните **повторно обработать запрос**.

    ![Повторная обработка невыполненного запроса](./media/entitlement-management-troubleshoot/reprocess-request.png)

### <a name="cancel-a-pending-request"></a>Отмена ожидающего запроса

Можно отменять только ожидающий запрос, который еще не был доставлен или с которым произошел сбой доставки.

**Предварительная роль:** Глобальный администратор, администратор пользователей, владелец каталога или диспетчер пакетов Access

1. На портале Azure щелкните **Azure Active Directory** и выберите **Управление удостоверениями**.

1. В меню слева щелкните **доступ к пакетам** и откройте пакет Access.

1. Щелкните **запросы**.

1. Щелкните запрос, который нужно отменить.

1. В области сведения о запросе щелкните **отменить запрос**.

## <a name="multiple-policies"></a>Несколько политик

* Управление назначениями следует минимальным рекомендациям по обеспечению привилегий. Когда пользователь запрашивает доступ к пакету Access, имеющему несколько политик, которые применяются, управление назначением включает логику, позволяющую обеспечить более высокий приоритет или более конкретные политики на основе универсальных политик. Если политика является универсальной, управление назначением может не отображать политику для запрашивающей стороны или автоматически выбирать более ограниченную политику.

* Например, рассмотрим пакет Access с двумя политиками для внутренних сотрудников, в которых обе политики применяются к запрашивающему. Первая политика предназначена для конкретных пользователей, которые включают запрашивающий. Вторая политика предназначена для всех пользователей в каталоге, членом которых является запрашивающий. В этом сценарии первая политика автоматически выбирается для запрашивающего, так как он более четкий. Запрашивающему не предоставляется возможность выбора второй политики.

* Если применяются несколько политик, то политика, которая автоматически выбирается, или политики, отображаемые запрашивающему, основаны на следующей логике приоритета:

    | Приоритет политики | Область действия |
    | --- | --- |
    | P1 | Определенные пользователи и группы в каталоге или в конкретных подключенных организациях |
    | P2 | Все элементы в каталоге (за исключением гостей) |
    | P3 | Все пользователи в вашем каталоге (включая гостей) или определенные подключенные Организации |
    | P4 | Все подключенные организации или все пользователи (все подключенные Организации + все новые внешние пользователи) |
    
    Если любая политика имеет более высокий приоритет, категории с низким приоритетом не учитываются. Пример того, как в запрашивающей части отображается несколько политик с одинаковым приоритетом, см. в разделе [Выбор политики](entitlement-management-request-access.md#select-a-policy).
