# <a name="troubleshoot-azure-ad-entitlement-management-preview"></a>Устранение неполадок управления назначениями Azure AD (Предварительная версия)

> [!IMPORTANT]
> Управление правами Azure Active Directory (Azure AD) сейчас предоставляется в общедоступной предварительной версии.
> Эта предварительная версия предоставляется без соглашения об уровне обслуживания и не рекомендована для использования рабочей среде. Некоторые функции могут не поддерживаться или их возможности могут быть ограничены.
> Дополнительные сведения см. в статье [Дополнительные условия использования предварительных выпусков Microsoft Azure](https://azure.microsoft.com/support/legal/preview-supplemental-terms/).

В этой статье описываются некоторые элементы, которые необходимо проверить, чтобы помочь в устранении неполадок управления обслуживанием Azure Active Directory (Azure AD).

## <a name="checklist-for-entitlement-management-administration"></a>Контрольный список для администрирования управления обслуживанием

* Если при настройке управления назначением вы получаете сообщение об отказе в доступе и являетесь глобальным администратором, убедитесь, что в каталоге есть [лицензия Azure AD Premium P2 (или EMS)](entitlement-management-overview.md#license-requirements).  
* При получении сообщения об отказе в доступе при создании или просмотре пакетов доступа, если вы являетесь членом группы создателей каталога, необходимо создать каталог перед созданием первого пакета Access.

## <a name="checklist-for-adding-a-resource"></a>Контрольный список для добавления ресурса

* Чтобы приложение было ресурсом в пакете Access, оно должно иметь по крайней мере одну роль ресурса, которая может быть назначена. Роли определяются самим приложением и управляются в Azure AD. Обратите внимание, что в портал Azure также могут отображаться субъекты-службы для служб, которые не могут быть выбраны в качестве приложений.  В частности, **Exchange Online** и **SharePoint Online** являются службами, а не приложениями, имеющими роли ресурсов в каталоге, поэтому они не могут быть включены в пакет Access.  Вместо этого используйте Групповое лицензирование, чтобы установить подходящую лицензию для пользователя, которому нужен доступ к этим службам.

* Чтобы группа была ресурсом в пакете Access, она должна быть доступна для изменения в Azure AD.  Группы, поступающие в локальную Active Directory, не могут быть назначены как ресурсы, так как их владелец или атрибуты членов нельзя изменять в Azure AD.   Группы, созданные в Exchange Online в качестве групп распространения, не могут быть изменены в Azure AD. 

* Библиотеки документов SharePoint Online и отдельные документы не могут быть добавлены в качестве ресурсов.  Вместо этого создайте группу безопасности Azure AD, включите эту группу и роль сайта в пакет Access, а в SharePoint Online используйте эту группу для управления доступом к библиотеке документов или документу.

* Если есть пользователи, которые уже назначены ресурсу, которым вы хотите управлять с помощью пакета Access, убедитесь, что пользователи назначены пакету доступа с соответствующей политикой. Например, может потребоваться включить группу в пакет Access, который уже содержит пользователей в группе. Если пользователям в группе требуется постоянный доступ, они должны иметь соответствующую политику для пакетов доступа, чтобы они не потеряли доступ к группе. Пакет доступа можно назначить, запросив пользователей запросить пакет доступа, содержащий этот ресурс, или напрямую назначить его пакету доступа. Дополнительные сведения см. в разделе [изменение и Управление существующим пакетом Access](entitlement-management-access-package-edit.md).

## <a name="checklist-for-providing-external-users-access"></a>Контрольный список для предоставления доступа внешним пользователям

* Если имеется [список разрешений](../b2b/allow-deny-list.md)B2B, то пользователи, чьи каталоги запрещены, не смогут запрашивать доступ.

* Убедитесь, что [политики условного доступа](../conditional-access/require-managed-devices.md) не позволяют внешним пользователям запрашивать доступ или иметь возможность использовать приложения в пакетах доступа.

## <a name="checklist-for-request-issues"></a>Контрольный список проблем запроса

* Когда пользователь хочет запросить доступ к пакету Access, убедитесь, что он использует **ссылку на портал доступа** для пакета Access. Дополнительные сведения см. в разделе " [копирование портала доступа](entitlement-management-access-package-edit.md#copy-my-access-portal-link)".  Если внешний пользователь посещает **myaccess.Microsoft.com**, он увидит доступные пакеты доступа в своей организации.

* Если пользователь, который еще не находится в каталоге, входит в систему на портале доступа для запроса пакета Access, убедитесь, что он прошел проверку подлинности с помощью учетной записи организации. Учетная запись организации может быть либо учетной записью в каталоге ресурсов, либо каталогом, включенным в одну из политик пакета Access. Если учетная запись пользователя не является учетной записью организации или каталог, в котором она проверяются, не включена в политику, пользователь не увидит пакет Access. Дополнительные сведения см. [в разделе запрос доступа к пакету Access](entitlement-management-request-access.md).

* Если пользователю запрещено входить в каталог ресурсов, он не сможет запросить доступ на портале My Access. Прежде чем пользователь сможет запросить доступ, необходимо удалить блок входа из профиля пользователя. Чтобы удалить блок входа, в портал Azure щелкните **Azure Active Directory**, выберите **Пользователи**, щелкните пользователя, а затем щелкните **профиль**. Измените раздел **Параметры** и измените параметр **блокировать вход** на **нет**. Дополнительные сведения см. в разделе [Добавление или обновление сведений о профиле пользователя с помощью Azure Active Directory](../fundamentals/active-directory-users-profile-azure-portal.md).  Можно также проверить, заблокирован ли пользователь из-за [политики защиты идентификации](../identity-protection/howto-unblock-user.md).

* Если пользователь одновременно является инициатором и утверждающим, на портале доступа он не увидит запрос пакета Access на странице **утверждения** . Это намеренное поведение — пользователь не может утверждать собственный запрос. Убедитесь, что запрашиваемый пакет доступа имеет дополнительные утверждающие, настроенные для политики. Дополнительные сведения см. в разделе [изменение существующей политики](entitlement-management-access-package-edit.md#edit-an-existing-policy).

* Если новый внешний пользователь, который ранее не входил в каталог, получает пакет Access, включая сайт SharePoint Online, его пакет доступа будет отображаться как не полностью доставленный, пока учетная запись не будет подготовлена в SharePoint Online.
