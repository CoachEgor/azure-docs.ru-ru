# <a name="troubleshoot-azure-ad-entitlement-management"></a>Управление правами на устранение проблем Azure AD

В этой статье описаны некоторые элементы, которые следует проверить, чтобы помочь вам устранить устранение неполадок в управлении ресурсами Azure Active Directory (Azure AD).

## <a name="administration"></a>Администрирование

* Если при настройке управления правами вы получаете отказано в доступе и вы являетесь глобальным администратором, убедитесь, что в вашем каталоге есть [лицензия Azure AD Premium P2 (или EMS E5).](entitlement-management-overview.md#license-requirements)

* Если при создании или просмотре пакетов доступа вы получаете отказано в доступе и являетесь членом группы создателей Каталога, необходимо [создать каталог](entitlement-management-catalog-create.md) до создания первого пакета доступа.

## <a name="resources"></a>Ресурсы

* Роли для приложений определяются в самом приложении, и управление ими осуществляется в Azure AD. Если приложение не имеет ролей ресурсов, управление правами назначает пользователям роль **доступа по умолчанию.**

    Обратите внимание, что портал Azure также может отображать принципы служб ы и услуг, которые не могут быть выбраны в качестве приложений.  В частности, **Exchange Online** и **SharePoint Online** — это сервисы, а не приложения, которые имеют роли ресурса в каталоге, поэтому они не могут быть включены в пакет доступа.  Вместо этого используйте групповое лицензирование и установите подходящую лицензию для пользователей, которым нужен доступ к этим службам.

* Чтобы группа была ресурсом в пакете для доступа, она должна быть доступна для изменения в Azure AD.  Группы из локального каталога Active Directory нельзя назначить в качестве ресурсов, так как их атрибуты владельца и членов нельзя изменить в Azure AD.   Группы, созданные в Exchange Online в качестве списков рассылки, нельзя изменить в Azure AD. 

* Библиотеки документов SharePoint Online и отдельные документы нельзя добавить в качестве ресурсов.  Вместо этого создайте [группу безопасности Azure AD,](../fundamentals/active-directory-groups-create-azure-portal.md)включите эту группу и роль сайта в пакете доступа, а в SharePoint Online используйте эту группу для управления доступом к библиотеке документов или документу.

* Если есть пользователи, которые уже назначены ресурсу, которым вы хотите управлять с помощью пакета для доступа, убедитесь, что пользователи назначены пакету с использованием соответствующей политики. Например, может потребоваться включить группу в пакет для доступа, в котором уже есть пользователи в группе. Если пользователям в группе требуется постоянный доступ, в их отношении должна действовать соответствующая политика для пакетов для доступа, чтобы они не утратили доступ к группе. Чтобы назначить пакет для доступа, можно уведомить пользователей о необходимости запросить пакет, содержащий этот ресурс, или напрямую назначить их пакету. Для получения дополнительной информации смотрите [настройки запроса и утверждения изменений для пакета доступа.](entitlement-management-access-package-request-policy.md)

* Если удалить члена из команды, он также будет удален из группы Office 365. С удалением из чата команды может возникнуть задержка. Для получения дополнительной информации [см.](https://docs.microsoft.com/microsoftteams/office-365-groups#group-membership)

* Убедитесь, что каталог не настроен для работы в нескольких регионах. Сейчас для управления правами не поддерживаются несколько регионов для SharePoint Online. Чтобы можно было управлять правами для сайтов SharePoint Online, эти сайты должны принадлежать к географическому расположению по умолчанию. Для получения дополнительной информации смотрите [Multi-Geo возможности в OneDrive и SharePoint Online](https://docs.microsoft.com/Office365/Enterprise/multi-geo-capabilities-in-onedrive-and-sharepoint-online-in-office-365).

## <a name="access-packages"></a>Пакеты для доступа

* Если вы попытаетесь удалить пакет доступа или политику и увидеть сообщение об ошибке, в которое говорится, что есть активные назначения, если вы не видите пользователей с назначениями, проверьте, есть ли у недавно удаленных пользователей назначения. В течение 30-дневного окна после удаления пользователя учетная запись пользователя может быть восстановлена.   

## <a name="external-users"></a>Внешние пользователи

* Если внешний пользователь хочет запросить доступ к пакету доступа, убедитесь, что он использует **ссылку портала My Access** для пакета доступа. Для получения дополнительной информации [см. Ссылку «Поделиться», чтобы запросить пакет доступа.](entitlement-management-access-package-settings.md) Если внешний пользователь просто посещает **myaccess.microsoft.com** и не использует полную ссылку портала My Access, то он увидит пакеты доступа, доступные им в их собственной организации, а не в вашей организации.

* Если внешний пользователь не может запросить доступ к пакету доступа или не может получить доступ к ресурсам, обязательно проверьте [настройки для внешних пользователей.](entitlement-management-external-users.md#settings-for-external-users)

* Если новый внешний пользователь, который ранее не входил в каталог, получит пакет для доступа, в который входит сайт SharePoint Online, его пакет будет отображаться как предоставленный неполностью, пока в SharePoint Online не будет подготовлена учетная запись. Для получения дополнительной информации о настройках совместного использования смотрите [параметры совместного использования SharePoint Online.](entitlement-management-external-users.md#review-your-sharepoint-online-external-sharing-settings)

## <a name="requests"></a>Requests

* Если пользователь хочет запросить доступ к пакету доступа, убедитесь, что он использует **ссылку портала My Access** для пакета доступа. Для получения дополнительной информации [см. Ссылку «Поделиться», чтобы запросить пакет доступа.](entitlement-management-access-package-settings.md)

* Если открыть портал "Мой доступ" в браузере с настроенным закрытым или анонимным режимом, может возникнуть конфликт при входе. Мы рекомендуем не использовать закрытый или анонимный режим для посещения портала "Мой доступ".

* Если пользователь, которого еще нет в каталоге, войдет на портал "Мой доступ" и запросит пакета для доступа, убедитесь, что он прошел проверку подлинности с использованием учетной записи организации. Учетная запись организации может быть в каталоге ресурсов или в каталоге, включенном в одну из политик пакета для доступа. Если учетная запись пользователя не принадлежит к организации или каталог проверки подлинности не включен в политику, пользователь не увидит пакет для доступа. Для получения дополнительной информации [см.](entitlement-management-request-access.md)

* Если для пользователя заблокирован вход в каталог ресурсов, пользователь не сможет запросить доступ на портале "Мой доступ". Чтобы пользователь мог запросить доступ, необходимо удалить блокировку входа из его профиля. Чтобы удалить блок ввне записи, на портале Azure нажмите **Azure Active Directory,** нажмите **на пользователя,** нажмите на пользователя, а затем нажмите **профиль**. Отспособите раздел **«Настройки»** и **измените ввесение блока в** **No.** Для получения дополнительной информации [см. Добавить или обновить информацию о профиле пользователя с помощью Active Directory Azure.](../fundamentals/active-directory-users-profile-azure-portal.md)  Вы также можете проверить, был ли пользователь заблокирован из-за [политики защиты идентификации.](../identity-protection/howto-unblock-user.md)

* На портале My Access, если пользователь является одновременно запрашивателем и утверждателем, он не увидит свой запрос на пакет доступа на странице **Утверждения.** Это предусмотренное поведение — пользователь не может утвердить свой запрос. Убедитесь, что для запрашиваемого пакета для доступа согласно политике настроены дополнительные утверждающие. Для получения дополнительной информации смотрите [настройки запроса и утверждения изменений для пакета доступа.](entitlement-management-access-package-request-policy.md)

### <a name="view-a-requests-delivery-errors"></a>Просмотр ошибок доставки запроса

**Роль предпосылки:** Глобальный администратор, администратор пользователей, владелец каталога или менеджер пакетов доступа

1. На портале Azure щелкните **Azure Active Directory** и выберите **Управление удостоверениями**.

1. В левом меню щелкните **пакеты Access,** а затем откройте пакет доступа.

1. Нажмите **Запросы**.

1. Выберите запрос, который вы хотите просмотреть.

    Если в запросе есть ошибки с доставкой, статус запроса будет **не доставлен** или **частично доставлен.**

    Если есть какие-либо ошибки доставки, в панели детали запроса, будет подсчет ошибок доставки.

1. Нажмите на счет, чтобы увидеть все ошибки доставки запроса.

### <a name="reprocess-a-request"></a>Переработать запрос

Если запрос столкнулся с ошибкой, можно переработать запрос, чтобы попробовать его еще раз. Вы можете переработать запрос, который имеет статус **доставки не удалось** или **частично доставлены** и завершена дата менее одной недели.

**Роль предпосылки:** Глобальный администратор, администратор пользователей, владелец каталога или менеджер пакетов доступа

1. На портале Azure щелкните **Azure Active Directory** и выберите **Управление удостоверениями**.

1. В левом меню щелкните **пакеты Access,** а затем откройте пакет доступа.

1. Нажмите **Запросы**.

1. Нажмите на запрос, который необходимо переработать.

1. В панели запроса детали, нажмите **Reprocess запрос**.

    ![Переработать неудавшийся запрос](./media/entitlement-management-troubleshoot/reprocess-request.png)

### <a name="cancel-a-pending-request"></a>Отмена ожидающего запроса

Вы можете отменить только отложенный запрос, который еще не был доставлен или доставка которого не удалась.

**Роль предпосылки:** Глобальный администратор, администратор пользователей, владелец каталога или менеджер пакетов доступа

1. На портале Azure щелкните **Azure Active Directory** и выберите **Управление удостоверениями**.

1. В левом меню щелкните **пакеты Access,** а затем откройте пакет доступа.

1. Нажмите **Запросы**.

1. Нажмите на запрос, который вы хотите отменить.

1. В панели запроса детали, нажмите **Отмена запроса**.

## <a name="multiple-policies"></a>Несколько политик

* Управление правами следует наименее привилегиям передовой практики. Когда пользователь запрашивает доступ к пакету доступа, который имеет несколько политик, которые применяются, управление правами включает логику, чтобы гарантировать, что более строгие или более конкретные политики приоритеты над общими политиками. Если политика является универсальной, управление правами может не отображать политику запрашиванию или автоматически выбрать более строгую политику.

* Например, рассмотрим пакет доступа с двумя политиками для внутренних сотрудников, в которых обе политики применяются к запрашиванию. Первая политика предназначена для конкретных пользователей, включающих запросщика. Вторая политика предназначена для всех пользователей в каталоге, членом которого является запрашиватель. В этом случае первая политика автоматически выбирается для запрашивателя, поскольку она является более строгой. Запрашиваеку не предоставляется возможность выбора второй политики.

* При подаче нескольких политик автоматически выбраны политики или политики, отображаемые запрашиванию, основаны на следующей логике приоритетов:

    | Приоритет политики | Область |
    | --- | --- |
    | P1 | Конкретные пользователи и группы в каталоге или конкретных связанных организаций |
    | P2 | Все участники в каталоге (за исключением гостей) |
    | P3 | Все пользователи в вашем каталоге (включая гостей) или конкретные связанные организации |
    | P4 | Все подключенные организации или Все пользователи (все подключенные организации и любые новые внешние пользователи) |
    
    Если какая-либо политика находится в категории более высокого приоритета, то категории с более низким приоритетом игнорируются. Например, как несколько политик с одинаковым приоритетом отображаются в запросе, [см.](entitlement-management-request-access.md#select-a-policy)
