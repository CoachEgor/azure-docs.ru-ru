# <a name="use-azure-ad-access-reviews-to-manage-users-excluded-from-conditional-access-policies"></a>Используйте обзоры доступа Azure AD для управления пользователями, исключенными из политик условного доступа

В идеальном мире все пользователи выполняют политики доступа для защиты доступа к ресурсам организации. Но иногда в интересах бизнеса приходится делать исключения. В этой статье описано несколько примеров, в которых могут потребоваться исключения, и предложены методы, позволяющие ИТ-администратору с помощью проверок доступа Azure Active Directory решить такую задачу, избежать упущений в исключениях из политик и предоставить аудиторам доказательства регулярной проверки.

> [!NOTE]
> Для использования проверок доступа Azure AD требуется действительная платная лицензия Azure AD Premium P2, Enterprise Mobility + Security E5 или пробная лицензия. Дополнительные сведения см. в разделе [Выпуски Azure Active Directory](../fundamentals/active-directory-whatis.md).

## <a name="why-would-you-exclude-users-from-policies"></a>Зачем нужно исключать пользователей из политики?

Как ИТ-администратор, можно использовать [Azure AD Conditional Access,](../conditional-access/overview.md) чтобы требовать от пользователей аутентификации с помощью многофакторной аутентификации (MFA) или входить в систему из доверенной сети или устройства. При планировании развертывания вы можете столкнуться с тем, что такие требования не могут быть выполнены некоторыми пользователями. Например, кто-то работает в удаленном офисе, который не входит во внутреннюю сеть организации, или один из руководителей использует старый и неподдерживаемый телефон. Бизнес требует, чтобы эти пользователи могли войти в систему и выполнять свою работу, поэтому они исключены из политики условного доступа.

В качестве другого примера можно использовать [указанные местоположения](../conditional-access/location-condition.md) в Условиях Доступа для настройки набора округов и регионов, из которых не требуется, чтобы пользователи могли получить доступ к своему арендатору.

![Названные места в условном доступе](./media/conditional-access-exclusion/named-locations.png)

Однако в некоторых случаях у пользователей могут быть законные основания для ввода из этих заблокированных стран/регионов. Например, во время деловой или личной поездки. В этом примере политика условного доступа для блокировки этих стран/регионов может иметь специальную группу облачной безопасности для пользователей, исключенных из политики. Пользователи, которым нужен доступ на время поездок, смогут сами добавлять себя в эту группу с помощью [самостоятельного управления группами AAD](../users-groups-roles/groups-self-service-management.md).

Другим примером может быть то, что у вас есть политика условного доступа, которая [блокирует устаревшую аутентификацию для подавляющего большинства пользователей.](https://cloudblogs.microsoft.com/enterprisemobility/2018/06/07/azure-ad-conditional-access-support-for-blocking-legacy-auth-is-in-public-preview/) Корпорация Майкрософт настоятельно рекомендует блокировать в клиенте устаревшие протоколы, чтобы повысить уровень безопасности. Если же у вас есть пользователи, которым абсолютно необходимо использовать старые методы проверки подлинности для доступа к ресурсам из Office 2010 или через клиенты IMAP, SMTP или POP, вы можете исключить этих пользователей из политики, которая блокирует старые методы проверки подлинности.

## <a name="why-are-exclusions-challenging"></a>Почему исключения вызывают трудности?

В Azure AD можно развязать политику условного доступа к набору пользователей. Вы также можете исключить некоторых из этих пользователей, выбрав роли Azure AD, отдельных пользователей или гостей пользователей. Важно помнить, что при наличии исключений невозможно обязать таких пользователей соблюдать требования политики. Если исключения настроены для списка отдельных пользователей или через устаревшие локальные группы безопасности, видимость такого списка (пользователи могут даже не знать о его существовании) и возможности ИТ-администратора по его контролю (пользователи могут включать себя в группу безопасности и обходить политику) ограничены. Кроме того, даже обоснованно настроенное для пользователя исключение может вскоре оказаться ненужным или неправильным.

Когда создаются исключения, список включенных в них пользователей очень мал. Но с течением времени число исключений и размер списка растут. Следует время от времени просматривать такие списки и проверять, всем ли включенным в них пользователям действительно нужны исключения. С технической точки зрения управление списком не вызывает особых трудностей. Но кто принимает деловые решения и как обеспечить возможность аудита?

Однако, если настроить исключение в политику условного доступа с помощью группы Azure AD, то можно использовать просмотры доступа в качестве компенсирующего элемента управления, чтобы повысить видимость и сократить число пользователей, имеющих исключение.

## <a name="how-to-create-an-exclusion-group-in-a-conditional-access-policy"></a>Как создать группу исключения в политике условного доступа

Выполните следующие действия, чтобы создать новую группу Azure AD и политику условного доступа, которая не применяется к этой группе.

### <a name="create-an-exclusion-group"></a>Создание группы исключения

1. Войдите на портал Azure.

1. В области навигации слева щелкните **Azure Active Directory** и **Группы**.

1. В меню сверху щелкните **Новая группа**, чтобы открыть панель группы.

1. В списке **Тип группы** выберите **Безопасность**. Введите имя и описание.

1. Для параметра **Членство** обязательно укажите тип **Назначено**.

1. Выберите пользователей, которые должны входить в эту группу исключения, и щелкните **Создать**.

    ![Новое групповое стекло в Active Directory Azure](./media/conditional-access-exclusion/new-group.png)

### <a name="create-a-conditional-access-policy-that-excludes-the-group"></a>Создание политики условного доступа, исключающая группу

Теперь можно создать политику условного доступа, используюую эту группу исключений.

1. В левой навигации щелкните **Active Directory Azure,** а затем нажмите **«Условный доступ»,** чтобы открыть лезвие **политики.**

1. Щелкните **Новая политика**, чтобы открыть панель **Создание**.

1. Задайте имя.

1. В разделе "Назначения" выберите **Пользователи и группы**.

1. На вкладке **Включить** выберите **Все пользователи**.

1. На вкладке **Исключить** установите флажок **Пользователи и группы** и щелкните **Выбрать исключенных пользователей**.

1. Выберите ранее созданную группу исключения.

    > [!NOTE]
    > Мы рекомендуем на время тестирования внести исключить из политики хотя бы одну учетную запись администратора, чтобы случайно не заблокировать весь доступ к клиенту.

1. Продолжайте настраивать политику условного доступа на основе ваших организационных требований.

    ![Выберите панель исключенных пользователей в условном доступе](./media/conditional-access-exclusion/select-excluded-users.png)

Рассмотрим два примера, где можно использовать обзоры доступа для управления исключениями в политиках условного доступа.

## <a name="example-1-access-review-for-users-accessing-from-blocked-countriesregions"></a>Пример 1: Обзор доступа для пользователей, обращающихся из заблокированных стран/регионов

Допустим, у вас есть политика условного доступа, которая блокирует доступ из определенных стран/регионов. В этой политике также настроена группа исключений. Для членов этой группы мы рекомендуем применить следующую проверку доступа.

> [!NOTE]
> Для создания обзоров доступа требуется роль глобального администратора или администратора пользователя.

1. Проверка будет повторяться каждую неделю.

2. Время завершения не устанавливается, чтобы группа исключения всегда содержала самые актуальные сведения.

3. В область действия этой проверки включаются все члены этой группы.

4. Каждый пользователь должен будет самостоятельно засвидетельствовать, что он все еще должен иметь доступ из этих заблокированных стран/регионов, поэтому он все еще должен быть членом группы.

5. Если пользователь не ответит на запрос об отзыве, он будет автоматически удален из группы и, следовательно, больше не может получить доступ к арендатору во время путешествия в эти страны/регионы.

6. Включите уведомления по электронной почте, чтобы пользователи получали информацию о начале и завершении проверки доступа.

    ![Создание панели обзора доступа, например, 1](./media/conditional-access-exclusion/create-access-review-1.png)

## <a name="example-2-access-review-for-users-accessing-with-legacy-authentication"></a>Пример 2. Проверка доступа для пользователей, использующих устаревшие методы проверки подлинности

Допустим, у вас есть политика условного доступа, которая блокирует доступ для пользователей с помощью устаревшей аутентификации и старых версий клиентов. В этой политике также настроена группа исключений. Для членов этой группы мы рекомендуем применить следующую проверку доступа.

1. Эта проверка должна выполняться регулярно.

2. Проверка применяется к каждому члену группы.

3. Можно настроить ее так, чтобы рецензентами проверки являлись руководители бизнес-подразделений.

4. Автоматическое применение результатов позволяет автоматически удалять пользователей, для которых не подтверждена необходимость продолжать использование устаревших методов проверки подлинности.

5. Также будет полезно включить рекомендации, чтобы рецензентам больших групп было легче принимать решения.

6. Включите уведомления по электронной почте, чтобы пользователи получали информацию о начале и завершении проверки доступа.

    ![Создание панели обзора доступа, например 2](./media/conditional-access-exclusion/create-access-review-2.png)

**Совет для профессионалов**. Если у вас много групп исключений, а значит есть необходимость создавать много проверок доступа, вы можете выполнять эти действия программным способом через API в бета-версии конечной точки Microsoft Graph. Чтобы воспользоваться этими возможностями, изучите [документацию по API проверки доступа AAD](https://developer.microsoft.com/graph/docs/api-reference/beta/resources/accessreviews_root) и [примеры проверок доступа для AAD через Microsoft Graph](https://techcommunity.microsoft.com/t5/Azure-Active-Directory/Example-of-retrieving-Azure-AD-access-reviews-via-Microsoft/td-p/236096).

## <a name="access-review-results-and-audit-logs"></a>Результаты проверки доступа и журналы аудита

Теперь, когда у вас есть все на месте, группа, политика условного доступа и обзоры доступа, пришло время отслеживать и отслеживать результаты этих обзоров.

1. На портале Azure откройте колонку **Проверки доступа**.

1. Откройте элемент управления и программу, которые вы создали для управления группой исключений.

1. Щелкните **Результаты**, чтобы увидеть списки подтвержденных и удаленных пользователей.

    ![Результаты обзоров доступа показывают, кто был утвержден](./media/conditional-access-exclusion/access-reviews-results.png)

1. Теперь щелкните **Журналы аудита**, чтобы увидеть список выполненных в ходе этой проверки действий.

    ![Доступ проверяет действия журналов аудита](./media/conditional-access-exclusion/access-reviews-audit-logs.png)

Вы как ИТ-администратор должны знать, что иногда невозможно избежать создания групп исключений для политик. Но проверки доступа AAD существенно упростят обслуживание таких групп, регулярную проверку руководителями компании или самими пользователями, а также аудит изменений.
