# <a name="use-azure-ad-access-reviews-to-manage-users-excluded-from-conditional-access-policies"></a>Использование проверок доступа Azure AD для управления пользователями, исключенными из политик условного доступа

В идеальном мире все пользователи следуют политикам доступа для защиты доступа к ресурсам Организации. Но иногда в интересах бизнеса приходится делать исключения. В этой статье рассматриваются некоторые примеры ситуаций, в которых могут потребоваться исключения. Вы, как ИТ-администратор, можете управлять этой задачей, избежать подробного анализа исключений политик и предоставить аудиты, подтверждающие, что эти исключения регулярно просматриваются с помощью Azure Active Directory (Azure AD).

>[!NOTE]
> Для использования проверок доступа Azure AD требуется действительная платная лицензия Azure AD Premium P2, Enterprise Mobility + Security E5 или пробная лицензия. Дополнительные сведения см. в разделе [Выпуски Azure Active Directory](../fundamentals/active-directory-whatis.md).

## <a name="why-would-you-exclude-users-from-policies"></a>Зачем нужно исключать пользователей из политики?

Предположим, что в качестве администратора вы решили использовать [Условный доступ Azure AD](../conditional-access/overview.md) , чтобы требовать многофакторную проверку подлинности (MFA), и ограничить запросы проверки подлинности конкретными сетями или устройствами. Во время планирования развертывания вы понимаете, что не все пользователи могут удовлетворить эти требования. Например, пользователи могут работать с удаленными офисами, а не с частью внутренней сети. Кроме того, может потребоваться обеспечить доступ пользователей, подключающихся с помощью неподдерживаемых устройств, при ожидании замены этих устройств. Вкратце, бизнесу нужны эти пользователи для входа в систему и их работы, чтобы исключить их из политик условного доступа.

В качестве другого примера вы можете использовать [именованные расположения](../conditional-access/location-condition.md) в условном доступе, чтобы указать набор стран и регионов, из которых вы не хотите предоставлять пользователям доступ к своим клиентам.

![Именованные расположения в условном доступе](./media/conditional-access-exclusion/named-locations.png)

К сожалению, у некоторых пользователей по-прежнему есть веская причина для входа из этих заблокированных стран или регионов. Например, пользователи могут быть в командировке для работы и нуждаются в доступе к корпоративным ресурсам. В этом случае политика условного доступа для блокировки этих стран и регионов может использовать группу безопасности облака для исключенных пользователей из политики. Пользователи, которым нужен доступ на время поездок, смогут сами добавлять себя в эту группу с помощью [самостоятельного управления группами AAD](../users-groups-roles/groups-self-service-management.md).

Другим примером может быть то, что у вас есть политика условного доступа, [блокирующая устаревшую проверку подлинности для подавляющего большинства пользователей](https://cloudblogs.microsoft.com/enterprisemobility/2018/06/07/azure-ad-conditional-access-support-for-blocking-legacy-auth-is-in-public-preview/). Однако если у вас есть несколько пользователей, которым необходимо использовать устаревшие методы проверки подлинности для доступа к ресурсам через клиенты Office 2010 или IMAP/SMTP/POP, можно исключить этих пользователей из политики, блокирующей устаревшие методы проверки подлинности.

>[!NOTE]
>Корпорация Майкрософт настоятельно рекомендует блокировать в клиенте устаревшие протоколы, чтобы повысить уровень безопасности.

## <a name="why-are-exclusions-challenging"></a>Почему исключения вызывают трудности?

В Azure AD можно ограничить политику условного доступа набором пользователей. Кроме того, исключения можно настроить, выбрав роли Azure AD, отдельные пользователи или гости. Следует помнить, что при настройке исключений цель политики не может быть применена к исключенным пользователям. Если исключения настраиваются с помощью списка пользователей или локальных групп безопасности, вы получите ограниченную видимость исключений. В результате имеем следующее:

- Пользователи могут не быть уверены, что они исключены.

- Пользователи могут присоединиться к группе безопасности, чтобы обойти политику.

- Исключенные пользователи могут быть квалифицированы для исключения раньше, но могут больше не соответствовать требованиям.

Часто при первой настройке исключения существует списке пользователей, которые обходят политику. Со временем все больше и больше пользователей добавляются к исключению, а список растет. В некоторый момент необходимо просмотреть список и убедиться, что каждый из этих пользователей по-прежнему имеет право на исключение. Управление списком исключений с технической точки зрения может быть относительно простым, но кто принимает бизнес-решения и как убедиться в том, что все они доступны для аудита? Однако если вы настраиваете исключение с помощью группы Azure AD, вы можете использовать проверки доступа как компенсирующий элемент управления, чтобы обеспечить видимость и сократить число исключенных пользователей.

## <a name="how-to-create-an-exclusion-group-in-a-conditional-access-policy"></a>Создание группы исключений в политике условного доступа

Выполните следующие действия, чтобы создать новую группу Azure AD и политику условного доступа, которая не применяется к этой группе.

### <a name="create-an-exclusion-group"></a>Создание группы исключения

1. Войдите на портал Azure.

2. В области навигации слева щелкните **Azure Active Directory** и **Группы**.

3. В меню сверху щелкните **Новая группа**, чтобы открыть панель группы.

4. В списке **Тип группы** выберите **Безопасность**. Введите имя и описание.

5. Для параметра **Членство** обязательно укажите тип **Назначено**.

6. Выберите пользователей, которые должны входить в эту группу исключения, и щелкните **Создать**.

![Новая область группы в Azure Active Directory](./media/conditional-access-exclusion/new-group.png)

### <a name="create-a-conditional-access-policy-that-excludes-the-group"></a>Создание политики условного доступа, которая исключает группу

Теперь можно создать политику условного доступа, которая использует эту группу исключений.

1. На панели навигации слева щелкните **Azure Active Directory** , а затем выберите **Условный доступ** , чтобы открыть колонку **политики** .

2. Щелкните **Новая политика**, чтобы открыть панель **Создание**.

3. Задайте имя.

4. В разделе "Назначения" выберите **Пользователи и группы**.

5. На вкладке **Включить** выберите **Все пользователи**.

6. На вкладке **исключить** добавьте флажок для **пользователей и групп** , а затем щелкните **выбрать исключенных пользователей**.

7. Выберите ранее созданную группу исключения.

   > [!NOTE] 
   > Мы рекомендуем на время тестирования внести исключить из политики хотя бы одну учетную запись администратора, чтобы случайно не заблокировать весь доступ к клиенту.

8. Продолжайте настройку политики условного доступа в соответствии с требованиями Организации.

![Выбор области исключенных пользователей в условном доступе](./media/conditional-access-exclusion/select-excluded-users.png)
  
Рассмотрим два примера использования проверок доступа для управления исключениями в политиках условного доступа.

## <a name="example-1-access-review-for-users-accessing-from-blocked-countriesregions"></a>Пример 1. Проверка доступа для пользователей, обращающихся к заблокированным странам или регионам

Предположим, что у вас есть политика условного доступа, которая блокирует доступ из определенных стран или регионов. В этой политике также настроена группа исключений. Для членов этой группы мы рекомендуем применить следующую проверку доступа.

> [!NOTE] 
> Для создания проверок доступа требуется роль глобального администратора или администратора пользователей.

1. Проверка будет выполняться еженедельно.

2. Никогда не будет заканчиваться, чтобы обеспечить актуальность этой группы исключений.

3. В область действия этой проверки включаются все члены этой группы.

4. Каждый пользователь должен самостоятельно подтверждать, что им по-прежнему нужен доступ из этих заблокированных стран или регионов, поэтому им по-прежнему нужно быть членом группы.

5. Если пользователь не отвечает на запрос на проверку, он автоматически удаляется из группы и больше не будет иметь доступа к клиенту при командировке в эти страны или регионы.

6. Включите уведомления по электронной почте, чтобы пользователи могли узнать о начале и завершении проверки доступа.

    ![Создайте панель проверки доступа, например 1](./media/conditional-access-exclusion/create-access-review-1.png)

## <a name="example-2-access-review-for-users-accessing-with-legacy-authentication"></a>Пример 2. Проверка доступа для пользователей, использующих устаревшие методы проверки подлинности

Предположим, что у вас есть политика условного доступа, которая блокирует доступ для пользователей, использующих устаревшую проверку подлинности и более старую версию клиента, и включает группу, исключенную из политики. Для членов этой группы мы рекомендуем применить следующую проверку доступа.

1. Эта проверка должна выполняться регулярно.

2. Проверка применяется к каждому члену группы.

3. Можно настроить ее так, чтобы рецензентами проверки являлись руководители бизнес-подразделений.

4. Автоматическое применение результатов позволяет автоматически удалять пользователей, для которых не подтверждена необходимость продолжать использование устаревших методов проверки подлинности.

5. Также будет полезно включить рекомендации, чтобы рецензентам больших групп было легче принимать решения.

6. Включите уведомления по электронной почте, чтобы пользователи получали информацию о начале и завершении проверки доступа.

    ![Создайте панель проверки доступа, например 2](./media/conditional-access-exclusion/create-access-review-2.png)

>[!IMPORTANT] 
>Если у вас много групп исключений и, следовательно, нужно создать несколько проверок доступа, теперь у нас есть API в конечной точке Microsoft Graph Beta, который позволяет создавать их и управлять ими программными средствами. Чтобы воспользоваться этими возможностями, изучите [документацию по API проверки доступа AAD](https://developer.microsoft.com/graph/docs/api-reference/beta/resources/accessreviews_root) и [примеры проверок доступа для AAD через Microsoft Graph](https://techcommunity.microsoft.com/t5/Azure-Active-Directory/Example-of-retrieving-Azure-AD-access-reviews-via-Microsoft/td-p/236096).

## <a name="access-review-results-and-audit-logs"></a>Результаты проверки доступа и журналы аудита

Теперь, когда у вас есть все что угодно, группы, политики условного доступа и проверки доступа, пришло время отслеживать и отслеживать результаты этих проверок.

1. В портал Azure откройте колонку проверки **доступа** .

2. Откройте элемент управления и программу, которые вы создали для управления группой исключений.

3. Щелкните **Результаты**, чтобы увидеть списки подтвержденных и удаленных пользователей.

    ![Результаты проверок доступа показывает, кто был утвержден](./media/conditional-access-exclusion/access-reviews-results.png)

4. Теперь щелкните **Журналы аудита**, чтобы увидеть список выполненных в ходе этой проверки действий.

    ![Проверки доступа Просмотр журналов аудита действия](./media/conditional-access-exclusion/access-reviews-audit-logs.png)

Вы как ИТ-администратор должны знать, что иногда невозможно избежать создания групп исключений для политик. Тем не менее, поддержание этих групп на регулярной основе их владельца или самих пользователей, а аудит этих изменений можно упростить с помощью проверок доступа Azure AD.
