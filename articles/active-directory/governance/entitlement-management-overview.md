# <a name="what-is-azure-ad-entitlement-management-preview"></a>Что такое управление правами Azure AD? (предварительная версия)

> [!IMPORTANT]
> Управление правами Azure Active Directory (Azure AD) сейчас предоставляется в общедоступной предварительной версии.
> Эта предварительная версия предоставляется без соглашения об уровне обслуживания и не рекомендована для использования рабочей среде. Некоторые функции могут не поддерживаться или их возможности могут быть ограничены.
> Дополнительные сведения см. в статье [Дополнительные условия использования предварительных выпусков Microsoft Azure](https://azure.microsoft.com/support/legal/preview-supplemental-terms/).

Сотрудники в организации требуется доступ к различных групп, приложений и узлов для выполнения их работы. Управление такой доступ является сложной задачей. В большинстве случаев отсутствует упорядоченный список всех ресурсов, необходимые пользователю для проекта. Руководитель проекта имеет хорошее понимание ресурсов, необходимых, лиц запутанным и как долго будут продолжаться в проект. Тем не менее руководитель проекта обычно не имеет разрешений для утверждения или предоставлять доступ другим пользователям. Этот сценарий становится более сложным, при попытке работы с внешних отдельным лицам или компаниям.

Управление правами Azure Active Directory (Azure AD) может помочь управлять доступом к группам, приложений и сайтов SharePoint Online для внутренних пользователей, а также пользователей за пределами вашей организации.

## <a name="why-use-entitlement-management"></a>Зачем использовать управление правами?

Предприятиям часто сталкиваются с управлением доступом к ресурсам, таких как:

- Пользователи могут не знать, какой доступ, они должны иметь
- Пользователи испытывают трудности с определением местоположения соответствующим лицам или нужные ресурсы
- После того как пользователям найти и получать доступ к ресурсу, они может удерживать доступ к больше, чем требуется для бизнес-целей

Эти проблемы сочетаются для пользователей, которым требуется доступ из другого каталога, таких как внешние пользователи из организации цепочки поставок или партнерам. Пример:

- Организации могут не знать всех конкретных пользователей в других каталогах, чтобы иметь возможность их приглашение
- Даже если организации смогли пригласить этих пользователей, организаций, может не сохраняться согласованно управлять всеми доступ пользователя

Управление правами Azure AD может помочь решить эти проблемы.

## <a name="what-can-i-do-with-entitlement-management"></a>Что можно делать с помощью прав управления?

Ниже приведены некоторые из возможностей, право управления.

- Создание пакетов связанных ресурсов, которые пользователи могут запрашивать
- Определите правила для запроса ресурсов и истечения срока действия доступа
- Управляют жизненным циклом доступ для внутренних и внешних пользователей
- Делегировать управление ресурсами
- Назначить утверждающих для утверждения запросов
- Создавать отчеты для отслеживания журнала

## <a name="what-resources-can-i-manage"></a>Какие ресурсы можно управлять?

Ниже приведены типы ресурсов, можно управлять доступом к с помощью прав управления.

- Группы безопасности Azure AD
- Группы Office 365
- Azure AD корпоративные приложения, включая приложения SaaS и встроенные пользовательские приложения, поддерживающие федерацию или подготовки
- SharePoint Online семейств веб-сайтов

Также можно управлять доступом к другим ресурсам, которые зависят от группы безопасности Azure AD или группах Office 365.  Пример:

- Вы можете передать лицензий для пользователей для Microsoft Office 365 с помощью группы безопасности Azure AD в пакете доступа, настроив [группового лицензирования](../users-groups-roles/licensing-groups-assign.md) для этой группы
- Вы можете предоставить пользователям доступ для управления ресурсами Azure с помощью группы безопасности Azure AD в пакете доступ и создание [назначений ролей Azure](../../role-based-access-control/role-assignments-portal.md) для этой группы

## <a name="what-are-access-packages-and-policies"></a>Что такое пакеты и политик?

Управление правами введено понятие *доступа пакета*. Пакет доступа — это пакет, все ресурсы, которые пользователю нужно работать над проектом или выполнять свою работу. Ресурсы включают в себя доступ к группам, приложений или сайтов. Пакеты доступа используются для управления доступом для внутренних сотрудников и также пользователей за пределами вашей организации. Пакеты для доступа добавляются в контейнеры, называемые *каталогами*.

Пакеты также включать один или несколько *политики*. Политика определяет правила или защитных функций для доступа к пакету доступ. При включении политики обеспечивает, что только нужные пользователи, предоставляется доступ к ресурсам, а также для правой количество времени.

![Пакет доступа и политики](./media/entitlement-management-overview/elm-overview-access-package.png)

Пакет доступа и связанные с ним политики диспетчер пакетов доступ определяет:

- Ресурсы
- Роли пользователей необходимые для ресурсов
- Внутренних пользователей и внешних пользователей, которые могут запросить доступ
- Процесс утверждения и пользователей, которые могут утверждать или отклонять доступ
- Продолжительность доступа пользователя

В примере ниже показан пример различных элементов в управление правами. В нем показано два пакета доступ на примере.

- **Пакет доступу к 1** включает одну группу в качестве ресурса. Доступ определяется с помощью политики, которая включает набор пользователей в каталоге, чтобы запросить доступ.
- **Пакет доступа 2** включает как ресурсы группы, приложения и сайту SharePoint Online. Доступ определяется с помощью двух разных политик. Первая политика позволяет реализовывать набор пользователей в каталоге, чтобы запросить доступ. Вторая политика позволяет пользователям в каталоге, внешний запрос на доступ.

## <a name="external-users"></a>Внешние пользователи

При использовании [Azure AD предприятие предприятие (B2B)](../b2b/what-is-b2b.md) пригласить опыту, адреса электронной почты внешних гостевых пользователей, вы хотите перенести в каталоге ресурсов и работы с уже необходимо знать. Это отлично работает при работе над проектом краткосрочное или меньшего размера и вы уже знаете, все участники, но это становится все сложнее управлять, если у вас есть множество пользователей, которых вы хотите работать с, или если участники изменяться со временем.  Например вы может работать с другой организацией и одна точка контакта с этой организации, но со временем дополнительных пользователей из этой организации также потребуется доступ.

С помощью прав управления можно определить политику, которая позволяет пользователей из организаций, которые вы указали, которые также используют Azure AD, чтобы иметь возможность запросить пакет доступа. Можно указать ли утверждение является обязательным и дату окончания срока действия для доступа. Если утверждение требуется, можно также назначить в качестве утверждающего одного или нескольких пользователей из внешней организации ранее приглашенным - так, как они, скорее всего знать, какие внешние пользователи из организации необходим доступ. После настройки доступа пакета, можно отправить им ссылку к пакету доступ для контактного лица в внешней организации. Такой контакт можно поделиться с другими пользователями в внешней организации, и они могут использовать эту ссылку для запроса доступа пакета.  Пользователи этой организации, которые уже были приглашены в каталог можно также использовать эту ссылку.

Если запрос утвержден, управление правами подготовит пользователя с помощью необходимых прав доступа, которые могут включать пригласить пользователя в том случае, если они еще не развернуты в вашем каталоге. Azure AD автоматически создаст учетную запись B2B для них.  Обратите внимание, что администратор ранее ограничен организаций разрешены для совместной работы, задав [B2B разрешить или запретить списка](../b2b/allow-deny-list.md) для разрешения или запрета приглашает для других организаций.  Если пользователь не разрешен в списки разрешений и блокировок, то они не может присоединиться.

Так как не требуется доступ внешних пользователей до получения последнего неограниченно долго, можно указать дату истечения в политике, например 180 дней. Через 180 дней Если доступ не обновлена, управление правами удалит все доступа, связанные с пакетом, доступ.  Если пользователь, который был приглашен через управление правами не имеет других доступа пакета назначений, затем после потери их последнее присваивание свою учетную запись B2B будут блокироваться войдите в систему в течение 30 дней и впоследствии удаляется.  Это предотвращает распространение ненужные учетные записи.  

## <a name="terminology"></a>Терминология

Чтобы лучше понять управление правами и сопутствующую документацию, изучите следующие термины.

| Термин или понятие | Описание |
| --- | --- |
| Управление правами | Служба, которая назначает, отменяет и администрирует пакеты. |
| пакет доступа | Коллекция разрешений и политик к ресурсам, которые пользователи могут запрашивать. Пакет доступа всегда содержится в каталоге. |
| запрос на доступ | Запрос на доступ к пакету доступ. Запрос обычно проходит через рабочий процесс. |
| policy | Набор правил, определяющий жизненный цикл доступа, например, как пользователи получают доступ, кто может утверждать и как долго пользователи имеют доступ. Среди примеров политик — доступом сотрудников и внешнего доступа. |
| catalog | Контейнер связанные ресурсы и пакеты. |
| Общие каталога | Встроенные каталога, которая всегда доступна. Для добавления ресурсов в каталоге Общие требуются определенные разрешения. |
| resource | Ресурса или службы (например, группы, приложения или узла), пользователю могут быть предоставлены разрешения для. |
| Тип ресурса | Тип ресурса, который содержит группы, приложений и сайтов SharePoint Online. |
| роль ресурса | Коллекция разрешений, связанных с ресурсом. |
| каталог ресурсов | Каталог, содержащий один или несколько ресурсов для совместного использования. |
| Назначенные пользователи | Назначение пакета доступ пользователю или группе. |
| enable | Процесс предоставления пакету доступ для пользователей, для запроса. |
