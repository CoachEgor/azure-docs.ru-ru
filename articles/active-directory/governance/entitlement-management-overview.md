# <a name="what-is-azure-ad-entitlement-management"></a>Что собой представляет управление правами Azure AD?

Управление правами Azure Active Directory (Azure AD) — это функция [управления идентификацией,](identity-governance-overview.md) которая позволяет организациям управлять жизненным циклом идентификации и доступа в масштабе, автоматизируя рабочие процессы запроса доступа, назначения доступа, обзоры и истечение срока действия.

Сотрудникам организаций необходим доступ к различным группам, приложениям и сайтам для выполнения своей работы. Управление этим доступом является сложной задачей, так как требования меняются - добавляются новые приложения или пользователям нужны дополнительные права доступа.  Этот сценарий усложняется при сотрудничестве с внешними организациями - вы можете не знать, кому в другой организации нужен доступ к ресурсам вашей организации, и они не будут знать, какие приложения, группы или сайты используют сяора.

Управление правами Azure AD может помочь вам более эффективно управлять доступом к группам, приложениям и сайтам SharePoint Online для внутренних пользователей, а также для пользователей за пределами организации, которым необходим доступ к этим ресурсам.

## <a name="why-use-entitlement-management"></a>Зачем использовать управление правами?

Корпоративные организации часто сталкиваются с проблемами при управлении доступом сотрудников к таким ресурсам, как:

- Пользователи могут не знать, какой доступ они должны иметь, и даже если они это сделают, они могут испытывать трудности с поиском нужных лиц, чтобы утвердить их доступ
- Как только пользователи находят и получают доступ к ресурсу, они могут удерживать доступ дольше, чем требуется для деловых целей

Эти проблемы усугубляются для пользователей, которым необходим доступ от другой организации, таких как внешние пользователи из организаций цепочки поставок или других деловых партнеров. Пример:

- Никто не может не знать всех конкретных лиц в каталогах других организаций, чтобы иметь возможность пригласить их
- Даже если они смогли пригласить этих пользователей, никто в этой организации не может помнить, чтобы управлять всем доступом пользователя последовательно

Управление правами Azure AD может помочь в решении этих проблем. ## <a name="what-can-i-do-with-entitlement-management"></a>Что я могу сделать с управлением правами?

Вот некоторые из возможностей управления правами:

- Делегируйте неадминистраторам возможность создания пакетов доступа. Эти пакеты доступа содержат ресурсы, которые пользователи могут запросить, и менеджеры пакетов доступа могут определять политики с правилами, для которых пользователи могут запрашивать, кто должен утвердить их доступ, и когда срок доступа истекает.
- Выберите подключенные организации, пользователи которых могут запрашивать доступ.  Когда пользователь, еще не входящий в ваш каталог, запрашивает доступ и утверждается, он автоматически предлагается в ваш каталог и заданный доступ.  По истечении срока их доступа, если у них нет других назначений пакета доступа, их учетная запись B2B в вашем каталоге может быть автоматически удалена.

## <a name="what-are-access-packages-and-what-resources-can-i-manage-with-them"></a>Что такое пакеты доступа и какими ресурсами можно управлять с ними?

Управление правами вводит в Azure AD концепцию *пакета доступа.* Пакет доступа представляет собой набор всех ресурсов с доступом, который необходим пользователю для работы над проектом или выполнения своей задачи. Пакеты доступа используются для управления доступом для ваших внутренних сотрудников, а также пользователей за пределами организации.

 Ниже приведены типы ресурсов, к которым можно управлять доступом пользователя с помощью управления правами:

- Членство в группах безопасности Azure AD
- Членство в Office 365 групп и команд
- Назначение корпоративных приложений Azure AD, включая приложения SaaS и пользовательские интегрированные приложения, поддерживающие федеративное/единое включение и/или подготовку
- Членство в сайтах SharePoint Online

Вы также можете контролировать доступ к другим ресурсам, которые опираются на группы безопасности Azure AD или Группы Office 365.  Пример:

- Вы можете предоставить пользователям лицензии для Microsoft Office 365, используя группу безопасности Azure AD в пакете доступа и настраивая [групповое лицензирование](../users-groups-roles/licensing-groups-assign.md) для этой группы
- Вы можете предоставить пользователям доступ к управлению ресурсами Azure, используя группу безопасности Azure AD в пакете доступа и создав [назначение ролей Azure](../../role-based-access-control/role-assignments-portal.md) для этой группы

## <a name="how-do-i-control-who-gets-access"></a>Как упроизваю, кто получает доступ?

С помощью пакета доступа администратор или менеджер пакетов доступа перечисляет ресурсы (группы, приложения и сайты) и роли, необходимые пользователям для этих ресурсов.

Пакеты доступа также включают одну или несколько *политик.* Политика определяет правила или ограждения для назначения для доступа к пакету. Каждая политика может быть использована для обеспечения того, чтобы только соответствующие пользователи могли запрашивать доступ, что есть утверждающие для их запроса, и что их доступ к этим ресурсам ограничен по времени и истечет, если не будет продлен.

![Пакет доступа и политики](./media/entitlement-management-overview/elm-overview-access-package.png)

В рамках каждой политики администратор или менеджер пакетов доступа определяет

- Либо уже существующие пользователи (как правило, сотрудники или уже приглашенные гости), либо организации-партнеры внешних пользователей, которые имеют право запросить доступ
- Процесс утверждения и пользователи, которые могут одобрить или отказать в доступе
- Продолжительность назначения доступа пользователя, после утверждения, до истечения срока действия назначения

На следующей диаграмме показан пример различных элементов управления правами. Он показывает один каталог с двумя примерами пакетов доступа.

- **Пакет доступа 1** включает в себя одну группу в качестве ресурса. Доступ определяется с помощью политики, которая позволяет набору пользователей в каталоге запрашивать доступ.
- **Пакет доступа 2** включает в себя группу, приложение и сайт SharePoint Online в качестве ресурсов. Доступ определяется с помощью двух различных политик. Первая политика позволяет набору пользователей в каталоге запрашивать доступ. Вторая политика позволяет пользователям во внешнем каталоге запрашивать доступ.

## <a name="when-should-i-use-access-packages"></a>Когда следует использовать пакеты доступа?

Пакеты доступа не заменяют другие механизмы назначения доступа.  Они наиболее уместны в таких ситуациях, как:

- Сотрудникам необходим ограниченный по времени доступ к конкретной задаче.  Например, можно использовать групповое лицензирование и динамическую группу, чтобы обеспечить всем сотрудникам почтовый ящик Exchange Online, а затем использовать пакеты доступа для ситуаций, в которых сотрудникам нужен дополнительный доступ, например, для чтения ведомственных ресурсов из другого Отдел.
- Доступ должен быть одобрен менеджером сотрудника или другими назначенными лицами.
- Департаменты хотят управлять своими собственными политиками доступа для своих ресурсов без участия ИТ-отделов.  
- Две или более организаций сотрудничают над проектом, и в результате несколько пользователей из одной организации должны быть привлечены через Azure AD B2B для доступа к ресурсам другой организации.

## <a name="how-do-i-delegate-access"></a>Как делегировать доступ?

 Пакеты для доступа добавляются в контейнеры, называемые *каталогами*.  Вы можете иметь единый каталог для всех пакетов доступа, или вы можете назначить лиц для создания и собственных каталогов. Администратор может добавлять ресурсы в любой каталог, но неадминистратор может только добавить в каталог ресурсы, которыми он владеет. Владелец каталога может добавлять других пользователей в качестве совладельцев каталога или в качестве менеджеров пакетов доступа.  Эти сценарии описаны далее в [статье делегирования и роли в управлении правами Azure AD](entitlement-management-delegate.md).

## <a name="summary-of-terminology"></a>Резюме терминологии

Чтобы лучше понять управление правами и его документацию, вы можете вернуться к следующему списку терминов.

| Термин | Описание |
| --- | --- |
| пакет доступа | Пакет ресурсов, в которые нуждается команда или проект, и регулируется политиками. Пакет доступа всегда содержится в каталоге. Можно создать новый пакет доступа для сценария, в котором пользователи должны запрашивать доступ.  |
| запрос доступа | Запрос на доступ к ресурсам в пакете доступа. Запрос обычно проходит через рабочий процесс утверждения.  В случае одобрения запрашивающий пользователь получает назначение пакета доступа. |
| присваивание | Назначение пакета доступа пользователю гарантирует, что пользователь имеет все роли ресурса этого пакета доступа.  Назначения пакетов доступа обычно имеют ограничение по времени до истечения срока их действия. |
| catalog | Контейнер с сопутствующими ресурсами и пакетами доступа.  Каталоги используются для делегирования, так что неадминистраторы могут создавать свои собственные пакеты доступа. Владельцы каталогов могут добавлять ресурсы, которые у них есть, в каталог. |
| создатель каталога | Коллекция пользователей, которые уполномочены создавать новые каталоги.  Когда пользователь, не являющийся администратором, получивший право быть создателем каталога, создает новый каталог, он автоматически становится владельцем этого каталога. |
| подключенная организация | Внешний каталог Azure AD или домен, с которым вы имеете отношения. Пользователи из подключенной организации могут быть указаны в политике как разрешенные для запроса доступа. |
| policy | Набор правил, определяющий жизненный цикл доступа, например, как пользователи получают доступ, кто может утверждать и как долго пользователи имеют доступ через назначение. Политика связана с пакетом доступа. Например, пакет доступа может иметь две политики: одну для сотрудников, чтобы запросить доступ, и второй для внешних пользователей, чтобы запросить доступ. |
| ресурс | Актив, например группа Office, группа безопасности, приложение или сайт SharePoint Online, с ролью, на которую пользователь может получить разрешения. |
| каталог ресурсов | Каталог, который имеет один или несколько ресурсов для совместного обмена. |
| роль ресурса | Коллекция разрешений, связанных с ресурсом и определяемых ею. Группа имеет две роли - член и владелец. Сайты SharePoint обычно имеют 3 роли, но могут иметь дополнительные пользовательские роли. Приложения могут иметь пользовательские роли. |


## <a name="license-requirements"></a>Требования лицензий

[!INCLUDE [Azure AD Premium P2 license](../../../includes/active-directory-p2-license.md)]

Специализированные облака, такие как Azure Government, Azure Germany и Azure China 21Vianet, в настоящее время недоступны для использования.

### <a name="how-many-licenses-must-you-have"></a>Сколько лицензий должно быть у вас?

Убедитесь, что в каталоге есть не менее столько лицензий Azure AD Premium P2, сколько у вас есть сотрудники, которые будут выполнять следующие задачи:

- Пользователи-члены, которые **могут** запросить пакет доступа.
- Пользователь и приглашенные пользователи, запрашивающие пакет доступа.
- Пользователь и приглашенные пользователи, которые одобряют запросы на пакет доступа.

Лицензии Azure AD Premium P2 **не** требуются для следующих задач:

- Лицензии не требуются пользователям с ролью Глобального администратора, которые настраивают начальные каталоги, пакеты доступа и политики и делегируют административные задачи другим пользователям.
- Для пользователей, которым были делегированы административные задачи, такие как создатель каталога, владелец каталога и менеджер пакетов доступа не требуется никаких лицензий.
- Для гостей, которые **могут** запросить пакеты доступа, не требуются лицензии, но **не** запрашивают пакет доступа.

Для каждой оплаченной лицензии Azure AD Premium P2, которую вы приобретаете для пользователей -участницы (сотрудников), можно использовать Azure AD B2B для приглашения до 5 гостевых пользователей. Эти гостевые пользователи также могут использовать функции Azure AD Premium P2. Для получения дополнительной информации смотрите [руководство по лицензированию совместной работы Azure AD B2B см.](../b2b/licensing-guidance.md)

Для получения дополнительной информации о лицензиях см. [Назначьте или удалите лицензии с помощью портала Active Directory Azure.](../fundamentals/license-users-groups.md)
