# <a name="azure-active-directory-b2b-best-practices"></a>Лучшие практики Azure Active Directory B2B
В этой статье содержатся рекомендации и рекомендации для совместной работы между бизнесом (B2B) в Active Directory Azure (Azure AD).

   > [!IMPORTANT]
   > **Начиная с 31 марта 2021**г. корпорация Майкрософт больше не будет поддерживать выкуп приглашений путем создания неуправляемых учетных записей Azure AD и арендаторов для сценариев совместной работы B2B. В процессе подготовки мы рекомендуем клиентам выбрать [одноразовый выбор в пользу электронной почты.](one-time-passcode.md) Мы приветствуем ваши отзывы об этой общедоступной функции предварительного просмотра и рады создать еще больше способов сотрудничества.

## <a name="b2b-recommendations"></a>Рекомендации B2B
| Рекомендация | Комментарии |
| --- | --- |
| Для оптимального опыта вхождения в систему, federate с поставщиками идентификационных данных | По мере возможности, федеративируйте непосредственно с поставщиками идентификационных данных, чтобы позволить приглашенным пользователям войти в общие приложения и ресурсы без создания учетных записей Майкрософт (MSA) или Azure AD. Вы можете использовать [функцию Федерации Google,](google-federation.md) чтобы позволить гостевым пользователям B2B войти в систему со своими учетными записями Google. Или можно использовать [функцию Direct federation (preview)](direct-federation.md) для создания прямой федерации с любой организацией, поставщик идентификационных данных (IdP) поддерживает протокол SAML 2.0 или WS-Fed. |
| Используйте функцию одноразовый пароль электронной почты (предварительный просмотр) для гостей B2B, которые не могут проверить подлинность другими средствами | Функция электронного почтового [индекса доступа (предварительный просмотр)](one-time-passcode.md) проверяет подлинность пользователей B2B, когда они не могут быть проверены с помощью других средств, таких как Azure AD, учетная запись Майкрософт (MSA) или федерация Google. Если гостевой пользователь активирует приглашение или обращается к общему ресурсу, он может запросить временный код, который будет отправлен на адрес электронной почты этого пользователя. Этот код нужно ввести, чтобы войти в систему. |
| Добавление фирменной символики организации на страницу входа | Вы можете настроить страницу ввнужного варианта, чтобы она была более интуитивной для ваших пользователей B2B. Узнайте, как [добавить брендинг компании для вхинга и страниц группы доступа.](../fundamentals/customize-branding.md) |
| Добавьте заявление о конфиденциальности в опыт выкупа гостевых пользователей B2B | Вы можете добавить URL-адрес заявления о конфиденциальности вашей организации в первый процесс выкупа приглашений, чтобы приглашенный пользователь согласился на ваши условия конфиденциальности для продолжения. Смотрите [сведения о конфиденциальности вашей организации в Active Directory Azure Active.](https://aka.ms/adprivacystatement) |
| Используйте функцию массового приглашения (предварительного просмотра), чтобы пригласить несколько гостевых пользователей B2B одновременно | Пригласите нескольких гостевых пользователей в организацию одновременно, используя функцию предварительного просмотра на портале Azure. Эта функция позволяет загружать файл CSV для создания гостевых пользователей B2B и отправки приглашений оптом. Смотрите [учебник для оптовых приглашающих пользователей B2B](tutorial-bulk-invite.md). |
| Применение политик условного доступа для многофакторной аутентификации (MFA) | Мы рекомендуем применять политики MFA в приложениях, которыми вы хотите поделиться с пользователями B2B-партнеров. Таким образом, МИД будет последовательно применяться к приложениям в вашем арендаторе, независимо от того, использует ли организация-партнер MFA. Смотрите [Условный доступ для пользователей совместной работы B2B](conditional-access.md). |
| Если вы применяете политики условного доступа на основе устройств, используйте списки исключений, чтобы обеспечить доступ к пользователям B2B | Если политики условного доступа на основе устройств включены в организации, устройства гостевых пользователей B2B будут заблокированы, поскольку они не управляются вашей организацией. Можно создать списки исключений, содержащие определенных пользователей-партнеров, чтобы исключить их из политики условного доступа на основе устройства. Смотрите [Условный доступ для пользователей совместной работы B2B](conditional-access.md). |
| Используйте URL-адрес, в частности для арендаторов, при предоставлении прямых ссылок на ваших гостевых пользователей B2B | В качестве альтернативы пригласительной электронной почте вы можете предоставить гостю прямую ссылку на ваше приложение или портал. Эта прямая ссылка должна быть конкретной для арендатора, то есть она должна включать идентификатор клиента или проверенный домен, чтобы гость мог быть проверен в вашем арендаторе, где находится общее приложение. Просмотрите [опыт выкупа для гостевого пользователя.](redemption-experience.md) |
| При разработке приложения используйте UserType для определения впечатления гостевого пользователя  | Если вы разрабатываете приложение и хотите предоставить различные возможности для пользователей-арендаторов и гостевых пользователей, используйте свойство UserType. Утверждение UserType сейчас не включено в токен. Приложениядолжны использовать API Microsoft Graph для запроса каталога для пользователя, чтобы получить свой UserType. |
| Изменение свойства UserType *только* в том случае, если отношение пользователя к организации изменяется | Хотя PowerShell может преобразовать свойство UserType для пользователя из пользователя в гость (и наоборот), это свойство следует изменить только в том случае, если отношение пользователя к вашей организации изменяется. Смотрите [Свойства гостевого пользователя B2B](user-properties.md).|
