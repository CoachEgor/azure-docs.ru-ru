# <a name="azure-active-directory-b2b-collaboration-invitation-redemption"></a>Активация приглашения службы совместной работы Azure Active Directory B2B

В этой статье описываются способы гостевых пользователей можно получить доступ к ресурсами и процессе получения согласия, их придется столкнуться. Если вы отправляете сообщение с приглашением в гостевой ОС, приглашение со ссылкой гостевой можно активировать для получения доступа, приложения или портала. Приглашение по электронной почте — один из способов, которыми гостевые системы могут получить доступ к ресурсам. Кроме того можно добавлять гостей в каталог и предоставить им прямую ссылку на портал или приложение, которое вы хотите поделиться. Независимо от метода, которые они используют Гости пройдете процесс согласия первый раз. Этот процесс гарантирует, что Гости согласны с условиями конфиденциальности и принимать рабочую [условия использования](https://docs.microsoft.com/azure/active-directory/governance/active-directory-tou) настройки.

При добавлении гостевого пользователя в каталоге учетной записи гостя имеет (можно просмотреть в PowerShell) согласия статус, первоначально задается **PendingAcceptance**. Этот параметр, пока гостевой принимает ваше приглашение и соглашается с тем, чтобы политика конфиденциальности и условия использования. После этого согласия приобретает значение **принято**, и на страницах согласия больше не представлены в гостевом компьютере.

## <a name="redemption-through-the-invitation-email"></a>Активация с помощью приглашения по электронной почте

При добавлении гостевого пользователя в своем каталоге, [с помощью портала Azure](https://docs.microsoft.com/azure/active-directory/b2b/b2b-quickstart-add-guest-users-portal), отправляется сообщение с приглашением гостевых в процессе. Вы также можете отправить приглашения по электронной почте, когда вы будете [с помощью PowerShell](https://docs.microsoft.com/azure/active-directory/b2b/b2b-quickstart-invite-powershell) добавить гостевых пользователей в каталог. Вот описание работы гостевой пользователь активировал ссылку в сообщение электронной почты.

1. Получает гостевой [приглашение по электронной почте](https://docs.microsoft.com/azure/active-directory/b2b/invitation-email-elements) , присланные **приглашения Microsoft**.
2. Выбирает гостевой **приступить к работе** в сообщении электронной почты.
3. Если в федеративной организации гостевой нет учетной записи Azure AD, учетной записи Майкрософт (MSA) или учетная запись электронной почты, ему предлагается создавать MSA (если не [одноразовый пароль](https://docs.microsoft.com/azure/active-directory/b2b/one-time-passcode) параметр, который не требует MSA ).
4. По ходу гостевой [согласие качества](#consent-experience-for-the-guest) описано ниже.

## <a name="redemption-through-a-direct-link"></a>Активация с помощью прямой ссылки

Как альтернативу приглашение по электронной почте можно присвоить гостевой прямую ссылку в приложении или на портале. Сначала необходимо добавить гостевого пользователя в каталог с помощью [портала Azure](https://docs.microsoft.com/azure/active-directory/b2b/b2b-quickstart-add-guest-users-portal) или [PowerShell](https://docs.microsoft.com/azure/active-directory/b2b/b2b-quickstart-invite-powershell). После этого можно использовать любой из [настраиваемых способов развертывания приложений для пользователей](https://docs.microsoft.com/azure/active-directory/manage-apps/end-user-experiences), включая прямые ссылки единого входа. Когда виртуальная машина использует прямую ссылку вместо приглашение по электронной почте, будет по-прежнему интерактивной через первого согласия.

> [!IMPORTANT]
> Прямая ссылка должен быть конкретного клиента. Другими словами он должен содержать идентификатор клиента или проверенного домена, поэтому Гость может пройти проверку подлинности в клиенте, где находятся приложения. Общие URL-адрес, например https://myapps.microsoft.com не будет работать для гостя, так как вы будете перенаправлены к их домашнего клиента для проверки подлинности. Ниже приведены некоторые примеры прямые ссылки в контексте клиента.
 > - Приложения панели доступа: https://myapps.microsoft.com/?tenantid=&lt; идентификатор клиента&gt; 
 > - Приложения панели доступа для проверенного домена: https://myapps.microsoft.com/&lt; проверенного домена&gt;
 > - Портал Azure: https://portal.azure.com/&lt; идентификатор клиента&gt;
 > - Отдельных приложений: сведения об использовании [прямую ссылку единого входа](../manage-apps/end-user-experiences.md#direct-sign-on-links)

Существуют случаи, где приглашение по электронной почте предпочтительнее, чем прямую ссылку. Если эти особые случаи важны для вашей организации, мы рекомендуем пригласить пользователей с помощью методов, которые по-прежнему отправлять сообщения с приглашением:
 - У пользователя нет учетной записи Azure AD, MSA или учетная запись электронной почты в федеративной организации. Если вы используете функцию одноразовый секретный код, Гость необходимо активировать приглашение по электронной почте, пошаговые инструкции по созданию MSA.
 - Иногда объект приглашенного пользователя не может использовать адрес электронной почты из-за конфликта с контактным объектом (например, контактным объектом Outlook). В этом случае пользователю необходимо щелкнуть URL-адрес активации в приглашении, полученном по электронной почте.
 - Пользователь может выполнить вход с псевдонимом адреса электронной почты, по которому поступило приглашение (псевдоним — это дополнительный адрес электронной почты, связанный с учетной записью электронной почты). В этом случае пользователю необходимо щелкнуть URL-адрес активации в приглашении, полученном по электронной почте.

## <a name="consent-experience-for-the-guest"></a>Взаимодействие согласия для гостей

При входе гостя для доступа к ресурсам в партнерской организации в первый раз, они будете направлены на следующих страницах. 

1. Обзоры гостевой **просмотрите разрешения** страницу, описывающую заявление о конфиденциальности в приглашающей организации. Пользователь должен **Accept** использование информацию о них в соответствии с политиками конфиденциальности в приглашающей организации, по-прежнему.

   ![Снимок экрана со страницей "Проверить разрешения"](media/redemption-experience/review-permissions.png) 

   > [!NOTE]
   > Сведения о том, как администратор клиента может создать ссылку на заявление о конфиденциальности вашей организации, см. в разделе [Практическое руководство. Добавление сведений о конфиденциальности организации в Azure Active Directory](https://aka.ms/adprivacystatement).

2. Если настроены условия использования, Гость открывает и проверяет условия использования, а затем выбирает **Accept**. 

   ![Снимок экрана с новыми условиями использования](media/redemption-experience/terms-of-use-accept.png) 

   > [!NOTE]
   > Можно настроить, см. в разделе [условия использования](../governance/active-directory-tou.md) в **управление** > **организационных взаимосвязей** > **условия использования**.

3. Если не указано иначе, гостевой перенаправляется на панели доступа приложения список приложений, которые можно получить доступ к гостевой ОС.

   ![Снимок экрана, показывающий на панели доступа приложений](media/redemption-experience/myapps.png) 

В каталоге, Гость на **приглашение принято** примет значение **Да**. Если создан MSA, гостя **источника** показывает **учетной записи Майкрософт**. Дополнительные сведения о свойств учетных записей гостевых пользователей, см. в разделе [свойства пользователя службы совместной работы Azure AD B2B](user-properties.md).
