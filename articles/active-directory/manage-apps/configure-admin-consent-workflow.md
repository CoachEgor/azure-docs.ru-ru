# <a name="configure-the-admin-consent-workflow-preview"></a>Настройка рабочего процесса согласия админа (предварительный просмотр)

В этой статье описывается, как включить функцию рабочего процесса (предварительного просмотра запроса) запроса согласия на запрос у админ-, которая предоставляет конечным пользователям возможность запрашивать доступ к приложениям, требующим согласия админа.

Без рабочего процесса согласия на запрос от админирования пользователь в наниматели, где согласие пользователя отключено, будет заблокирован, когда он попытается получить доступ к любому приложению, которое требует разрешения на доступ к организационным данным. Пользователь видит общее сообщение об ошибке, в которое говорится, что он не имеет доступа к приложению, и он должен обратиться за помощью к своему админу. Но часто пользователь не знает, к кому обратиться, поэтому они либо сдаются, либо создают новую локальную учетную запись в приложении. Даже если админ уведомляется об этом, не всегда существует упрощенный процесс, который поможет админу предоставить доступ и уведомить своих пользователей.
 
Рабочий процесс согласия админа дает админмам безопасный способ предоставления доступа к приложениям, требующим одобрения админа. Когда пользователь пытается получить доступ к приложению, но не может дать согласие, он может отправить запрос на утверждение админа. Запрос отправляется по электронной почте адм-серверам, которые были назначены рецензентами. Рецензент принимает меры по запросу, и пользователь уведомляется об действии.

Для утверждения запросов рецензент должен быть глобальным администратором, администратором облачных приложений или администратором приложений. Рецензент уже должен иметь одну из этих ролей админ назначен; простое назначение их в качестве рецензента не повышает их привилегии.

## <a name="enable-the-admin-consent-workflow"></a>Включить рабочий процесс согласия админа

Для обеспечения рабочего процесса согласия админа и выбора рецензентов:

1. Вопиюсь на [портал Azure](https://portal.azure.com) в качестве глобального администратора.
2. Щелкните **Все службы** в верхней части меню навигации слева. Откроется **расширение Azure Active Directory**.
3. В поле поиска фильтров введите **«Активный каталог Azure»** и выберите элемент **Active Directory Azure.**
4. Щелкните **Корпоративные приложения** в меню навигации. 
5. Под **управлением**выберите **настройки пользователя.**
6. В соответствии с **запросами на согласие админа (Preview)**, установить **Пользователи могут запросить согласие админа на приложения, которые они не могут дать согласие на** **Да**.

   ![Настройка параметров рабочего процесса согласия админа](media/configure-admin-consent-workflow/admin-consent-requests-settings.png)
 
6. Настройте следующие параметры:

   * **Выберите пользователей для просмотра запросов на согласие админ.** Выберите рецензентов для этого рабочего процесса из набора пользователей, которые имеют роли глобального администратора, администратора облачных приложений и администратора приложений.
   * **Выбранные пользователи будут получать уведомления по электронной почте для запросов.** Включить или отключить уведомления по электронной почте рецензентам при запросе.  
   * **Выбранные пользователи получат напоминания об истечении срока действия запроса.** Включить или отключить уведомления об электронной почте для рецензентов, когда запрос истекает.  
   * **Запрос на согласие истекает через (дней)**. Укажите, как долго запросы остаются действительными.

7. Нажмите кнопку **Сохранить**. Включена функция может занять до часа.

> [!NOTE]
> Вы можете добавить или удалить рецензентов для этого рабочего процесса, изменив список **рецензентов запросов запросов запросов запросов на выбор.** Обратите внимание, что текущее ограничение этой функции заключается в том, что рецензенты могут сохранять возможность просмотра запросов, которые были сделаны в то время как они были назначены в качестве рецензента.

## <a name="how-users-request-admin-consent"></a>Как пользователи запрашивают согласие на участие в амини

После включения рабочего процесса согласия админа пользователи могут запросить одобрение приложения, на которое они не могут дать согласие. Следующие шаги описывают опыт пользователя при запросе утверждения. 

1. Пользователь пытается войти в приложение.

2. Отображается **требуемое** утверждение сообщения. Пользователь вводит обоснование необходимости доступа к приложению, а затем выбирает **утверждение Запроса.**

   ![Запрос и обоснование запроса пользователя на согласие админа](media/configure-admin-consent-workflow/end-user-justification.png)

3. **Отправляемый** запрос подтверждает, что запрос был отправлен админу. Если пользователь отправляет несколько запросов, только первый запрос отправляется к аминину.

   ![Запрос и обоснование запроса пользователя на согласие админа](media/configure-admin-consent-workflow/end-user-sent-request.png)

 4. Пользователь получает уведомление по электронной почте, когда его запрос одобрен, отклонен или заблокирован. 

## <a name="review-and-take-action-on-admin-consent-requests"></a>Просмотр и принятие мер по запросам на согласие админа

Для рассмотрения запросов на согласие админа и принятия мер:

1. Вопийте на [портале Azure](https://portal.azure.com) в качестве одного из зарегистрированных рецензентов рабочего процесса согласия админа.
2. Выберите **все службы** в верхней части меню левой навигации. Откроется **расширение Azure Active Directory**.
3. В поле поиска фильтров введите **«Активный каталог Azure»** и выберите элемент **Active Directory Azure.**
4. Щелкните **Корпоративные приложения** в меню навигации.
5. В соответствии с **activity,** выберите **запросы на согласие админа (Preview)**.

   > [!NOTE]
   > Рецензенты будут видеть запросы на админ, созданные после того, как они были назначены рецензентом.

1. Выберите запрашиваемый приложение.
2. Просмотрите сведения о запросе:  

   * Чтобы узнать, кто запрашивает доступ и почему, выберите **забрасываемую** вкладкой.
   * Чтобы узнать, какие разрешения запрашиваются приложением, выберите **разрешения на просмотр и согласие.**

8. Оцените запрос и примите соответствующие меры:

   * **Утвердить запрос**. Чтобы утвердить запрос, предоставите запросу согласие на применение. После утверждения запроса все запрашивающие уведомляются о том, что им был предоставлен доступ.  
   * **Запретить запрос**. Чтобы отклонить запрос, вы должны предоставить обоснование, которое будет предоставлено всем запрашивателям. После отказа в запросе все запрашивающие уведомляются о том, что им было отказано в доступе к приложению. Отказ в запросе не помешает пользователям запрашивать согласие на приложение в будущем.  
   * **Блокировка запроса**. Чтобы заблокировать запрос, необходимо предоставить обоснование, которое будет предоставлено всем запрашивателям. После блокировки запроса все запрашиваемые поставщики уведомляются об отказе в доступе к приложению. Блокировка запроса создает основной объект службы для приложения в вашем арендаторе в отключенном состоянии. Пользователи не смогут запрашивать согласие админа на приложение в будущем.
 
## <a name="email-notifications"></a>Уведомления по электронной почте
 
При настройке все рецензенты будут получать уведомления по электронной почте, когда:

* Создан новый запрос
* Срок действия запроса истек
* Запрос приближается к сроку годности  
 
Запросы будут получать уведомления по электронной почте, когда:

* Они подают новый запрос на доступ
* Срок их запроса истек
* Их просьба была отклонена или заблокирована
* Их просьба была одобрена

## <a name="faq"></a>часто задаваемые вопросы 

**Я включил этот рабочий процесс, но при тестировании функциональности, почему я не могу увидеть новый "Утверждение требуется" запрос позволяет мне запрашивать доступ?**

После включения функции для просмотра обновления конечным пользователям может потребоваться до 60 минут. Вы можете проверить, что конфигурация вступила в силу, просмотрев `https://graph.microsoft.com/beta/settings` значение **EnableAdminConsentRequests В** API.

**Почему я не могу просмотреть все ожидающие запросы?**

Рецензенты могут видеть запросы на админ, созданные после того, как они были назначены рецензентом. Так что если вы недавно были добавлены в качестве рецензента, вы не увидите никаких запросов, которые были созданы до вашего назначения.

**Почему, как рецензент, я вижу несколько запросов на одно и то же приложение?**
  
Если разработчик приложения настроил свое приложение для использования статического и динамического согласия для запроса доступа к данным его конечных пользователей, вы увидите два запроса на согласие админа. Один запрос представляет статические разрешения, а другой — динамические разрешения.

**Могу ли я, как запрашивать, могу ли я проверить статус моего запроса?**  

Нет, на данный момент запрашивающие могут получать обновления только с помощью уведомлений по электронной почте.

**Как рецензент, можно ли утвердить заявку, но не для всех?**
 
Если вы обеспокоены предоставлением согласия на прием к аминину и разрешением всем пользователям в арендаторе использовать приложение, мы рекомендуем вам отклонить запрос. Затем вручную предоставляете согласие админа, ограничивая доступ к приложению, требуя назначения пользователя, и назначая пользователям или группам приложение. Подробнее см. статью [Методы назначения пользователей и групп](methods-for-assigning-users-and-groups.md).
