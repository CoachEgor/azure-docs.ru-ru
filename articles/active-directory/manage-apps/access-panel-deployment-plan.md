# <a name="plan-an-azure-active-directory-access-panel-deployment"></a>Планирование развертывания панели активного доступа к каталогам Azure

Панель доступа Azure Active Directory (Azure AD) — это веб-портал, который помогает снизить затраты на поддержку, повысить производительность и безопасность и уменьшить разочарование пользователей. Система включает в себя подробную отчетность, которая отслеживает при доступе к системе и уведомляет администраторов о неправомерном использовании или злоупотреблении.

Используя панель доступа Azure AD, вы можете:

* Откройте для себя и получите доступ ко всем ресурсам, связанным с Azure AD, таким и применяемым к их компании, например, приложениям
* Запрос доступа к новым приложениям и группам
* Управление доступом к этим ресурсам для других
* Управление сбросами паролей самообслуживания и настройками многофакторной аутентификации Azure
* Управление своими устройствами

Он также позволяет администраторам управлять:

* Условия обслуживания
* Организации
* Проверки доступа


## <a name="benefits-of-azure-ad-access-panel-integration"></a>Преимущества интеграции панели доступа Azure AD

Панель доступа Azure AD access помогает предприятиям следующим образом:

**Обеспечивает интуитивно понятный пользовательский интерфейс:** Панель доступа предоставляет вам единую платформу для всех приложений, подключенных к подключенным к Windows.ru. У вас есть единый портал для поиска существующих настроек и новых возможностей, таких как управление группой и сброс паролей самообслуживания по мере их добавления. Интуитивно понятный опыт позволяет пользователям быстрее вернуться к работе и быть более продуктивными, уменьшая при этом их разочарование.

**Повышает производительность**: Все пользовательские приложения в панели доступа включены SSO. Включение SSO в корпоративные приложения и Office 365 создает превосходный опыт вхождения за счет сокращения или устранения дополнительных запросов. Панель доступа использует самообслуживание и динамическое членство и повышает общую безопасность вашей системы идентификации. Он делает это, гарантируя, что нужные люди управляют доступом к приложениям. Панель доступа служит согласованной целевой страницей для быстрого поиска ресурсов и продолжения рабочих задач.

**Управление затратами**: Включение панели доступа с Azure AD может помочь с изъятием инфраструктуры. Это снижает затраты на поддержку, предоставляя вам согласованный портал для поиска всех ваших приложений, запроса доступа к ресурсам и управления учетными записями.

**Повышает гибкость и безопасность:** Панель доступа предоставляет вам доступ к безопасности и гибкости, которые обеспечивает облачная платформа. Администраторы могут легко изменять настройки приложений и ресурсов и удовлетворять новые требования безопасности, не затрагивая пользователей.

**Обеспечивает надежный аудит и отслеживание использования:** Аудит и отслеживание использования для всех пользовательских возможностей позволяют вам знать, когда пользователи используют свои ресурсы и гарантирует, что вы можете оценить безопасность.

* [Самостоятельный сброс пароля](https://docs.microsoft.com/azure/active-directory/authentication/quickstart-sspr)

* [Защита идентификации Azure Active Directory](https://docs.microsoft.com/azure/active-directory/active-directory-identityprotection)

Смотрите [полное руководство по лицензированию Azure AD.](https://azure.microsoft.com/pricing/details/active-directory/)

## <a name="plan-your-sso-configuration"></a>Планирование конфигурации SSO

Когда пользователь входит в приложение, он проходит процесс проверки подлинности и должен доказать, кто он. Без SSO в приложении хранится пароль, и пользователь должен знать этот пароль. С помощью SSO учетные данные пользователей передаются в приложение, поэтому им не нужно повторно вводить пароли для каждого приложения.

Для запуска приложений в моих приложениях sSO должна быть включена.

Azure AD поддерживает три различных способа включения [приложений с одним включением в систему:](what-is-single-sign-on.md)

* **Федеративный единый вход** 
    * Позволяет приложению перенаправлять на Azure AD для проверки подлинности пользователя, а не подсказывать пароль. 
    * Поддерживается для приложений, которые используют протоколы, такие как SAML 2.0, WS-Federation или OpenID Connect, и является самым богатым способом одного вхождения.

* **Одиночный вписаться на основе пароля** 
    * Обеспечивает безопасное хранение паролей приложения и воспроизведение с помощью расширения веб-браузера или мобильного приложения. 
    * Использует существующий процесс вступления, предоставляемый приложением, но позволяет администратору управлять паролями. Пользователь не обязан знать пароль.

* **Существующий единый входной знак** 
    * Позволяет Azure AD использовать любой существующий единый знак, настроенный для приложения.
    * Позволяет увязывать эти приложения с порталами Office 365 или Azure AD Access Panel. 
    * Позволяет предоставлять дополнительные отчеты в Azure AD при запуске приложений. 
    * Включает в себя использование Azure Application Proxy и подключенного режима единого ввоза.

Узнайте, как настроить sSO-режим приложения здесь: [Единый подгон приложений в Active Directory Azure.](https://docs.microsoft.com/azure/active-directory/active-directory-appssoaccess-whatis)

Для лучшего опыта со страницы My Apps начните с интеграции облачных приложений, доступных для федеративного SSO. Federated SSO позволяет пользователям иметь последовательный один клик опыт через их поверхности запуска приложения и, как правило, более надежны в управлении конфигурацией.

Используйте Federated SSO с Azure AD (OpenID Connect/SAML), когда приложение поддерживает его, а не на основе паролей SSO и ADFS.

Для получения дополнительной информации о том, как развертывать и настраивать приложения SaaS, можно найти [в плане развертывания SaaS SSO.](https://aka.ms/deploymentplans/sso)

#### <a name="plan-to-deploy-the-my-apps-browser-extension"></a>Планирование развертывания расширения браузера My Apps

При входе пользователей в SSO-приложения на основе паролей им необходимо установить и использовать безопасное расширение для вхаской My Apps. Расширение выполняет скрипт, который передает пароль в форме входной системы приложения. Пользователям предлагается установить расширение при первом запуске приложения SSO на основе паролей. Более подробную информацию о расширении можно найти в этой документации об [установке расширения браузера Access Panel.](https://docs.microsoft.com/azure/active-directory/application-access-panel-extension-problem-installing?/?WT.mc_id=DMC_AAD_Manage_Apps_Troubleshooting_Nav)

Если необходимо интегрировать SSO-приложения на основе паролей, следует определить механизм развертывания расширения в масштабе с [помощью поддерживаемых браузеров.](https://docs.microsoft.com/azure/active-directory/active-directory-saas-access-panel-introduction) Доступные параметры: 

* [Групповая политика для Internet Explorer](https://azure.microsoft.com/documentation/articles/active-directory-saas-ie-group-policy/)

* [Менеджер конфигурации для Internet Explorer](https://docs.microsoft.com/configmgr/core/clients/deploy/deploy-clients-to-windows-computers)

* [Загрузка и конфигурация для Chrome, Firefox, Microsoft Edge или IE](https://docs.microsoft.com/azure/active-directory/active-directory-saas-access-panel-introduction)

Подробнее: [Как настроить пароль одного знака](https://docs.microsoft.com/azure/active-directory/application-config-sso-how-to-configure-password-sso-non-gallery).

Пользователи, которые не используют приложения SSO на основе паролей, также получают выгоду от расширения. Эти преимущества включают возможность запуска любого приложения из панели поиска, поиск доступа к недавно используемым приложениям и наличие ссылки на страницу My Apps.

#### <a name="plan-for-mobile-access"></a>План мобильного доступа

Браузер, защищенный политикой Intune (Microsoft Edge или Intune Managed Browser), необходим для мобильных пользователей, запускающих приложения SSO на основе паролей. Защищенный политикой браузер позволяет передавать пароль, сохраненный для приложения. Microsoft Edge или управляемый браузер предоставляет набор функций защиты веб-данных. Вы также можете использовать Microsoft Edge для корпоративных сценариев на устройствах iOS и Android. Microsoft Edge поддерживает те же сценарии управления, что и управляемый браузер Intune, и улучшает взаимодействие с пользователем. Подробнее: [Управление веб-доступом с помощью защищенного от политики браузера Microsoft Intune.](https://docs.microsoft.com/intune/app-configuration-managed-browser)

## <a name="plan-your-my-apps-deployment"></a>Планирование развертывания моих приложений

Основой панели доступа является приложение пусковая My [https://myapps.microsoft.com](https://myapps.microsoft.com/)Apps, к которой пользователи имеют доступ в . Страницы My Apps дают пользователям единое место, где они начинают свою работу и попадают в нужные приложения. Здесь пользователи находят список всех приложений, к которых они имеют доступ к одним подписным знакам. 

![Скриншот панели приложений](media/access-panel-deployment-plan/ap-dp-app-launcher.png)

Те же приложения будут отображаться в приложении Office 365, когда пользователи используют портал Office 365.

Запланируйте заказ, в котором вы добавите приложения в пусковую установку My Apps, и решите, будете ли вы раскатывать их постепенно или все сразу. Для этого создайте инвентаризацию приложений, в которых перечислен тип аутентификации и любые существующие интеграции SSO для каждого приложения.

#### <a name="add-applications-to-the-my-apps-panel"></a>Добавление приложений в панель My Apps

Любое приложение с поддержкой Azure AD SSO может быть добавлено в пусковую установку My Apps. Другие приложения добавляются с помощью опции Linked SSO. Можно настроить плитку приложения, которая ссылается на URL существующего веб-приложения. Связанные SSO позволяют начать направлять пользователей на портал My Apps без переноса всех приложений в Azure AD SSO. Вы можете постепенно переходить к SSO-настроенным приложениям Azure AD, не нарушая работу пользователей.

#### <a name="use-my-apps-collections"></a>Используйте коллекции моих приложений

По умолчанию все приложения перечислены вместе на одной странице. Но вы можете использовать коллекции для объединения связанных приложений и представления их на отдельной вкладке, что упрощает их поиск. Например, можно использовать коллекции для создания логических групп приложений для определенных ролей заданий, задач, проектов и так далее. Для получения информации см. [Как использовать коллекции My Apps для настройки панелей пользовательского доступа.](access-panel-collections.md) 

#### <a name="plan-whether-to-use-my-apps-or-an-existing-portal"></a>Планирование использования моих приложений или существующего портала

У пользователей уже может быть приложение или портал, кроме моих приложений. Если да, то решите, поддерживать ли оба портала или использовать только один.

Если существующий портал уже используется в качестве отправной точки для пользователей, можно интегрировать функциональность My Apps, используя URL-адреса пользовательского доступа. URL-адреса доступа пользователей функционируют как прямые ссылки на приложения, доступные на портале My Apps. Эти URL-адреса могут быть встроены в любой существующий веб-сайт. Когда пользователь выбирает ссылку, он открывает приложение с портала My Apps.

Вы можете найти свойство URL-адреса пользователя в области **свойств приложений На** портале Azure.

## <a name="plan-self-service-application-discovery-and-access"></a>Планирование обнаружения приложений самообслуживания и доступа

После развертывания основного набора приложений на странице «Мои приложения» пользователя следует включить функции управления приложениями самообслуживания. Обнаружение приложения для самообслуживания позволяет пользователям:

* Найдите новые приложения для добавления в свои мои приложения. 
* Добавьте дополнительные приложения, которые они могут не знать, что им нужно во время настройки.

Рабочие процессы утверждения доступны для явного утверждения для доступа к приложениям. Пользователи, которые являются одобрительными пользователями, будут получать уведомления на портале My Apps, когда будет отложен запрос на доступ к приложению.

## <a name="plan-self-service-group-membership"></a>Планирование членства в группе самообслуживания 

Вы можете позволить пользователям создавать и управлять своими собственными группами безопасности или Office 365 групп в Azure AD. Владелец группы может одобрить или отклонить запросы на членство и делегировать контроль членства в группе. Функции управления группами самообслуживания недоступны для групп безопасности с поддержкой почты или списков рассылки.

Чтобы спланировать членство в группе самообслуживания, определите, разрешите ли вы всем пользователям в вашей организации создавать и управлять группами или только подмножеством пользователей. Если вы разрешаете подмножество пользователей, вам нужно настроить группу, к которой добавляются эти люди. Подробнее о возможности этих сценариев можно узнать [в каталоге Azure Active Directory.](https://docs.microsoft.com/azure/active-directory/users-groups-roles/groups-self-service-management)

## <a name="plan-reporting-and-auditing"></a>Отчетность и аудит плана

Azure AD предоставляет [отчеты, которые предлагают технические и бизнес-идеи.](https://azure.microsoft.com/documentation/articles/active-directory-view-access-usage-reports/) Работайте с владельцами ваших деловых и технических приложений, чтобы взять на себя ответственность за эти отчеты и регулярно потреблять их. В следующей таблице приводятся некоторые примеры типичных сценариев отчетности.

|   | Управление рисками.| Повышение производительности труда| Система управления и соответствие требованиям |
|  - |- | - | - |
| Типы отчетов|  Разрешения и использование приложений| Действия по подготовке учетных записей| Просмотрите доступ к приложениям |
| Потенциальные действия| Доступ к аудиту; аннулировать разрешения| Исправление любых ошибок подготовки| «Отозвать доступ» |

Azure AD хранит большинство аудиторских данных в течение 30 дней. Данные доступны через Azure Admin Portal или API для загрузки в системы анализа.

#### <a name="auditing"></a>Аудит

Аудиторские журналы для доступа к приложениям доступны в течение 30 дней. Если аудит безопасности в вашем предприятии требует более длительного хранения, журналы должны быть экспортированы в инструмент событий и управления безопасностью (SIEM), такой как Splunk или ArcSight.

Для аудита, отчетности и резервного копирования аварийного восстановления документируйте требуемую частоту загрузки, целевую систему и кто отвечает за управление каждым резервным копированием. Возможно, вам не потребуется отдельное аудит и резервное копирование. Резервное копирование аварийного восстановления должно быть отдельной сущностью.

Если вы хотите скрыть от пользователей набор приложений Office, есть возможность скрыть приложения с портала My Apps, при этом разрешая доступ с портала Office 365. ### <a name="deploy-application-self-service-capabilities"></a>Развертывание возможностей самообслуживания приложений

Доступ к приложениям самообслуживания позволяет пользователям самостоятельно открывать и запрашивать доступ к приложениям. Пользователи имеют свободу доступа к приложениям, которые им нужны, не проходя через ИТ-группы каждый раз, чтобы запросить доступ. Когда пользователь запрашивает доступ и утверждается автоматически или вручную владельцем приложения, он добавляется в группу на задней части. Включена отчетность о том, кто запросил, одобрил или удалил доступ, и это дает вам контроль над управлением назначенными ролями.

Вы можете делегировать утверждение запросов на доступ к приложениям бизнес-одобрители. Бизнес-одобритель может установить пароли доступа к приложению со страницы «Мои приложения» компании.

Подробнее: [Как использовать доступ к приложениям самообслуживания.](https://docs.microsoft.com/azure/active-directory/application-access-panel-self-service-applications-how-to)
