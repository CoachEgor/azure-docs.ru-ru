# <a name="application-management-best-practices"></a>Лучшие практики управления приложениями
Эта статья содержит рекомендации и рекомендации по управлению приложениями в Azure Active Directory (Azure AD), с использованием автоматического подготовки и публикации корпоративных приложений с помощью Application Proxy.

## <a name="cloud-app-and-single-sign-on-recommendations"></a>Облачное приложение и рекомендации по единому вводом
| Рекомендация | Комментарии |
| --- | --- |
| Проверьте галерею приложений Azure AD для приложений  | Azure AD имеет галерею, которая содержит тысячи предварительно интегрированных приложений, которые включены с единым ввозам Enterprise (SSO). Для руководства по настройке приложения смотрите [руководство по приложению SaaS.](https://azure.microsoft.com/documentation/articles/active-directory-saas-tutorial-list/)  | 
| Используйте федеративное SAML-основе SSO  | Когда приложение поддерживает его, используйте Federated, SAML основе SSO с Azure AD вместо паролей на основе SSO и ADFS.  | 
| Используйте SHA-256 для подписания сертификата  | Azure AD использует алгоритм SHA-256 по умолчанию для подписания ответа SAML. Используйте SHA-256, если приложение не требует SHA-1 (см. [Параметры подписания сертификата](certificate-signing-options.md) и [проблема регистрации приложения.)](application-sign-in-problem-application-error.md)  | 
| Требуется назначение пользователя  | По умолчанию пользователи могут получить доступ к корпоративным приложениям без их назначения. Однако, если приложение предоставляет роли, или если вы хотите, чтобы приложение появилось на панели доступа пользователя, требуется назначение пользователя. (См. [руководство разработчика для интеграции приложений.)](developer-guidance-for-integrating-applications.md)  | 
| Развертывание панели доступа My Apps для пользователей | [Панель доступа](end-user-experiences.md) `https://myapps.microsoft.com` — это веб-портал, который предоставляет пользователям единую точку входа для назначенных ими облачных приложений. По мере добавления дополнительных возможностей, таких как управление группой и сбросить пароли самообслуживания, пользователи могут найти их в панели доступа. [См.](access-panel-deployment-plan.md)
| Использование группового назначения  | Если она включена в подписку, назначьте группы приложению, чтобы вы могли делегировать владельцу группы текущее управление доступом. (См. [руководство разработчика для интеграции приложений.)](developer-guidance-for-integrating-applications.md)   | 
| Создание процесса управления сертификатами | Максимальный срок действия сертификата подписи составляет три года. Чтобы предотвратить или свести к минимуму сбой из-за истечения срока действия сертификата, используйте роли и списки рассылки электронной почты, чтобы обеспечить тщательный контроль за уведомлениями об изменениях, связанными с сертификатом. |

## <a name="provisioning-recommendations"></a>Подготовка рекомендаций
| Рекомендация | Комментарии |
| --- | --- |
| Используйте учебники для настройки подготовки с облачными приложениями | Проверьте [список учебников приложений SaaS,](https://azure.microsoft.com/documentation/articles/active-directory-saas-tutorial-list/) чтобы получить пошаговое руководство по настройке подготовки к приложению галереи, которое вы хотите добавить. |
| Используйте журналы подготовки (предварительный просмотр) для мониторинга состояния | В [журналах подготовки](../reports-monitoring/concept-provisioning-logs.md?context=azure/active-directory/manage-apps/context/manage-apps-context) приводится подробная информация обо всех действиях службы обеспечения, включая статус для отдельных пользователей. |
| Назначить группу дистрибуции на электронную почту уведомления о подготовке | Чтобы повысить видимость критических оповещений, отправляемых службой обеспечения, присвоите группе дистрибутива настройки Уведомлений. |


## <a name="application-proxy-recommendations"></a>Рекомендации по прокси-серверам приложений
| Рекомендация | Комментарии |
| --- | --- |
| Использование приложения Прокси для удаленного доступа к внутренним ресурсам | Приложение Прокси рекомендуется для предоставления удаленным пользователям доступ к внутренним ресурсам, заменив необходимость VPN или обратного прокси. Он не предназначен для доступа к ресурсам внутри корпоративной сети, поскольку он может добавить задержки.
| Используйте пользовательские домены | Настройка пользовательских доменов для приложений [(см. Настройка пользовательских доменов),](application-proxy-configure-custom-domain.md)чтобы URL-адреса для пользователей и между приложениями работали внутри или за пределами вашей сети. Вы также сможете контролировать свой брендинг и настроить URL-адреса.  При использовании пользовательских доменных имен планируйте получить общедоступный сертификат у доверенного сертификата, не являющихся корпорацией Майкрософт. Azure Application Proxy поддерживает стандартные сертификаты на[основе подстановочных знаков](application-proxy-wildcard.md)или сертификаты на основе SAN. (См. [Планирование прокси-приложений](application-proxy-deployment-plan.md).) |
| Синхронизация пользователей перед развертыванием прокси-сервера приложений | Перед развертыванием прокси-сервера приложения синхронизируйте идентификаторы пользователей из предварительного каталога или создайте их непосредственно в Azure AD. Синхронизация идентификационных данных позволяет Azure AD предварительно аутентифицировать пользователей, прежде чем предоставить им доступ к приложениям App Proxy. Он также предоставляет необходимую информацию об идентификаторах пользователей для выполнения одного входного (SSO). (См. [Планирование прокси-приложений](application-proxy-deployment-plan.md).) |
| Следуйте нашим советам по высокой доступности и балансировке нагрузки | Чтобы узнать, как [потоки](application-proxy-high-availability-load-balancing.md)трафика среди пользователей, разъемы Приложения Прокси, и серверы приложений бэк-энда, а также получить советы по оптимизации производительности и балансировки нагрузки, см. |
| Используйте несколько разъемов | Используйте два или более разъемов Application Proxy для большей устойчивости, доступности и масштабирования (см. [разъемы Application Proxy).](application-proxy-connectors.md) Создавайте группы разъемов и убедитесь, что каждая группа разъемов имеет по крайней мере два разъема (три разъема оптимальны). |
| Найдите серверы разъемов рядом с серверами приложений и убедитесь, что они находятся в одном домене | Для оптимизации производительности физически найдите сервер разъема рядом с серверами приложений [(см. соображения топологии сети).](application-proxy-network-topology.md) Кроме того, сервер разъема и серверы веб-приложений должны принадлежать к тому же домену Active Directory, или они должны охватывать доверительные домены. Эта конфигурация необходима для SSO с интегрированной аутентификацией Windows (IWA) и ограниченной делегацией Kerberos (KCD). Если серверы находятся в разных доменах, вам нужно будет использовать ресурсоемкую делегацию для SSO (см. [KCD для одного входного с Application Proxy).](application-proxy-configure-single-sign-on-with-kcd.md) |
| Включить автоматическое обновление для разъемов | Включите автоматические обновления для разъемов для последних функций и исправлений ошибок. Корпорация Майкрософт предоставляет непосредственную поддержку последней версии разъема и одной версии. (См. [историю версии релиза прокси-сервера.)](application-proxy-release-version-history.md) |
| Обойти свой предприимки прокси | Для более простого обслуживания назначьте разъем, чтобы обойти свой прокси-сервер, чтобы он непосредственно подключается к службам Azure. (См. [разъемы прокси-серверов приложений и прокси-серверов](application-proxy-configure-connectors-with-proxy-servers.md).) |
| Используйте прокси-приложение Azure AD через прокси-приложение Web | Используйте Azure AD Application Proxy для большинства сценариев. Прокси-сервер Web Application предпочтительнее только в сценариях, требующих прокси-сервера для AD FS и где вы не можете использовать пользовательские домены в Active Directory Azure. (См. [Миграцию прокси-приложений](application-proxy-migration.md).) |
