# <a name="add-an-unlisted-non-gallery-application-to-your-azure-ad-organization"></a>Добавление приложения, не включенного в список (не галерейных) в организацию Azure AD

В дополнение к выбору в [галерее приложений Azure AD,](https://azure.microsoft.com/documentation/articles/active-directory-saas-tutorial-list/)у вас есть возможность добавить **негалерей приложение.** Можно добавить любое приложение, уже существовавщее в вашей организации, или любое стороннее приложение от поставщика, который еще не входит в галерею Azure AD. В зависимости от [лицензионного соглашения](https://azure.microsoft.com/pricing/details/active-directory/)доступны следующие возможности:

- Интеграция самообслуживания любого приложения, поддерживающего язык идентификации [безопасности (SAML) 2.0](https://wikipedia.org/wiki/SAML_2.0) (по инициативе SP или IdP)
- самостоятельная интеграция любого веб-приложения, где есть HTML-страница входа, с использованием функции [единого входа на основе пароля](what-is-single-sign-on.md#password-based-sso)
- Самообслуживание приложений, которые используют систему управления [кросс-доменом (SCIM) для подготовки пользователей](../app-provisioning/use-scim-to-provision-users-and-groups.md)
- Возможность добавлять ссылки на любые приложения в [Средство запуска приложений Office 365](https://www.microsoft.com/microsoft-365/blog/2014/10/16/organize-office-365-new-app-launcher-2/) или [Панель доступа Azure AD](what-is-single-sign-on.md#linked-sign-on).

В этой статье описывается, как добавить приложение, не являющийся галереей, в корпоративные **приложения** на портале Azure без написания кода. Если вместо этого вы ищете рекомендации для разработчиков по интеграции пользовательских приложений с Azure AD, см. [Сценарии аутентификации для Azure AD.](../develop/authentication-scenarios.md) При разработке приложения, использующего современный протокол, такой как [OpenId Connect/OAuth](../develop/active-directory-v2-protocols.md) для проверки подлинности пользователей, вы можете зарегистрировать его на платформе идентификации Майкрософт, используя опыт [регистрации приложений](../develop/quickstart-register-app.md) на портале Azure.

## <a name="add-a-non-gallery-application"></a>Добавление приложения не из коллекции

1. Вопийте на [портале Active Directory Azure,](https://aad.portal.azure.com/) используя учетную запись администратора платформы Майкрософт.

2. Выберите**новое приложение** **Enterprise Applications** > .

3. (Необязательно, но рекомендуется) В поле поиска **Browse Azure AD Gallery** введите имя дисплея приложения. 

4. Выберите **Создать свое собственное приложение.** **Отображается собственная** страница приложения.

   ![Добавить приложение](media/add-non-gallery-app/create-your-own-application.png)

5. Начните вводить имя дисплея для нового приложения. Если есть какие-либо приложения галереи с похожими именами, они будут отображаться в списке результатов поиска.

   > [!NOTE]
   > Мы рекомендуем использовать галерейную версию приложения, когда это возможно. Если приложение, которое вы хотите добавить, отображается в результатах поиска, выберите приложение и пропустите остальную часть этой процедуры.

6. В соответствии **с Что вы хотите сделать с вашим приложением?** выберите **Интеграция любого другого приложения вы не найдете в галерее**. Эта опция обычно используется для приложений SAML и WS-Fed.

   > [!NOTE]
   > Два других варианта используются в следующих сценариях:
   >* **Настройка приложения Proxy для безопасного удаленного доступа к штатном приложению** открывает страницу конфигурации для прокси-серверов Azure AD И разъемов.
   >* **Регистрация приложения, над которым вы работаете, чтобы интегрироваться с Azure AD,** открывает страницу **регистрации приложений.** Эта опция обычно используется для приложений OpenID Connect.

7. Выберите **Создать**. Откроется страница приложения **Общие сведения**.

    - Параметр **Видимый пользователю** определяет, могут ли пользователи, назначенные приложению, видеть его на панели доступа и в средстве запуска O365.
