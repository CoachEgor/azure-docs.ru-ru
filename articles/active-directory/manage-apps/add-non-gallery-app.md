# <a name="add-an-unlisted-non-gallery-application-to-your-azure-ad-organization"></a>Добавление неупорядоченного приложения (не из коллекции) в организацию Azure AD

Помимо выбора в [коллекции приложений Azure AD](https://azure.microsoft.com/documentation/articles/active-directory-saas-tutorial-list/), можно добавить **приложение, не являющееся коллекцией**. Можно добавить любое приложение, которое уже существует в вашей организации, или любое стороннее приложение от поставщика, который еще не является частью коллекции Azure AD. В зависимости от [лицензионного соглашения](https://azure.microsoft.com/pricing/details/active-directory/)доступны следующие возможности.

- Самостоятельная интеграция любого приложения, которое поддерживает поставщики удостоверений [язык разметки зявлений системы безопасности (SAML) (SAML) 2,0](https://wikipedia.org/wiki/SAML_2.0) (инициированные с использованием SP или IDP)
- самостоятельная интеграция любого веб-приложения, где есть HTML-страница входа, с использованием функции [единого входа на основе пароля](what-is-single-sign-on.md#password-based-sso)
- Самостоятельное подключение приложений, использующих [систему для использования протокола управления удостоверениями доменов (scim) для подготовки пользователей](use-scim-to-provision-users-and-groups.md)
- Возможность добавлять ссылки на любые приложения в [Средство запуска приложений Office 365](https://www.microsoft.com/microsoft-365/blog/2014/10/16/organize-office-365-new-app-launcher-2/) или [Панель доступа Azure AD](what-is-single-sign-on.md#linked-sign-on).

В этой статье описывается добавление приложения не из коллекции в **корпоративные приложения** в портал Azure без написания кода. Если вместо этого вы ищете руководство для разработчиков о том, как интегрировать пользовательские приложения с Azure AD, см. статью [сценарии проверки подлинности в Azure AD](../develop/authentication-scenarios.md). При разработке приложения, использующего современный протокол, например [OpenID Connect Connect/OAuth](../develop/active-directory-v2-protocols.md) для проверки подлинности пользователей, можно зарегистрировать его на платформе Microsoft Identity с помощью [Регистрация приложенийного](../develop/quickstart-register-app.md) интерфейса в портал Azure.

## <a name="add-a-non-gallery-application"></a>Добавление приложения не из коллекции

1. Войдите на [портал Azure Active Directory](https://aad.portal.azure.com/) с помощью учетной записи администратора платформы удостоверений Майкрософт.
1. Выберите **корпоративные приложения** > **создать приложение**.
2. (Необязательно, но рекомендуется) В поле поиска **Добавить из коллекции** введите отображаемое имя приложения. Если приложение отображается в результатах поиска, выберите его и пропустите оставшуюся часть этой процедуры.
3. Выберите **приложение не из коллекции**. Откроется страница **Добавление собственного приложения** .

   ![Добавить приложение](./media/configure-single-sign-on-non-gallery-applications/add-your-own-application.png)
5. Введите отображаемое имя нового приложения.
6. Выберите **Добавить**. Откроется страница **Обзор** приложения.

## <a name="configure-user-sign-in-properties"></a>Настройка свойства входа пользователя

1. Выберите **Свойства**, чтобы открыть панель "Свойства" для редактирования.

    ![Изменение панели "Свойства"](media/add-application-portal/edit-properties.png)

1. Задайте следующие параметры, чтобы определить, как пользователи, назначенные или неназначенные приложению, могут входить в приложение, а пользователь может видеть приложение на панели доступа.

    - Параметр **Включено для входа пользователя** определяет, могут ли пользователи, назначенные приложению, выполнять вход в приложение.
    - Параметр **Требуется назначение пользователей** определяет, могут ли пользователи, не назначенные приложению, выполнять вход.
    - Параметр **Видимый пользователю** определяет, могут ли пользователи, назначенные приложению, видеть его на панели доступа и в средстве запуска O365.
