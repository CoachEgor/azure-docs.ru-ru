# <a name="baseline-policy-block-legacy-authentication-preview"></a>Базовая политика: Блок проверки подлинности (Предварительная версия)

Чтобы предоставить пользователям удобный доступ к облачным приложениям, Azure Active Directory (AAD) поддерживает широкий набор протоколов проверки подлинности, часть из которых уже устарела. Устаревшая проверка подлинности — это термин, который ссылается на запрос проверки подлинности, сделанных:

* Старых клиентов Office, которые не используют современную проверку подлинности (например, клиент Office 2010)
* Любой клиент, использующий устаревший mail протоколы, такие как IMAP, SMTP и POP3

В настоящее время большинство все угрозы попытки входа в систему поступают из устаревшей проверки подлинности. Устаревшая проверка подлинности не поддерживает многофакторную проверку подлинности (MFA). Даже если у вас есть политику многофакторной Идентификации, включен в вашем каталоге, вредоносный субъект можно проверку подлинности с помощью устаревших протоколов и обхода многофакторной проверки Подлинности.

Для защиты вашей учетной записи от вредоносных проверки подлинности запросов, выполняемых устаревшие протоколы рекомендуется заблокировать все вместе эти попытки. Чтобы упростить блокирует все запросы на вход, сделанных устаревшие протоколы, мы создали базовую политику, который делает именно это.

**Устаревшая проверка подлинности блок** — [базовую политику](concept-baseline-protection.md) блокирует все запросы проверки подлинности от устаревших протоколов. Чтобы успешно войти в систему для всех пользователей, необходимо использовать современную проверку подлинности. Используется в сочетании с другими политиками базовых показателей, будут заблокированы все запросы, поступающие из устаревших протоколов и все пользователи должны будут пройти многофакторную Идентификацию при необходимости. Эта политика не блокирует Exchange ActiveSync.

## <a name="identify-legacy-authentication-use"></a>Определение использования проверки подлинности

Прежде чем вы можете заблокировать устаревшей проверки подлинности в вашем каталоге, необходимо сначала понять, если у пользователей есть приложения, использующие устаревшие проверки подлинности и как оно влияет на общую каталога. Azure AD вход журналы можно использовать для понимания того, при использовании проверки подлинности.

1. Перейдите к **портала Azure** > **Azure Active Directory** > **входы в систему**.
1. Добавить столбец клиентского приложения, если он не отображается, щелкнув **столбцы** > **клиентское приложение**.
1. Фильтровать по **клиентское приложение** > **другие клиенты** и нажмите кнопку **применить**.

Только show, вы можете войти попыток, фильтрации, выполненных в устаревшие протоколы аутентификации. Щелкнув каждой отдельной попытки входа в систему будет отображение подробных сведений. **Клиентское приложение** под **основные сведения** вкладке укажет, использованный протокол проверки подлинности.

Эти журналы укажет, какие пользователи по-прежнему находятся в зависимости от проверки подлинности и приложения, использующие устаревшие протоколы для запросов проверки подлинности. Для пользователей, которые не будут отображаться в этих журналах и подтверждается не будет использовать проверки подлинности, реализовать политику условного доступа или включить **базовая политика: блок проверки подлинности** эти только для пользователей.

## <a name="moving-away-from-legacy-authentication"></a>Переход от устаревшей проверки подлинности

Когда лучше представить себе, кто использует проверки подлинности в вашем каталоге и приложения, которые зависят от нее, следующим шагом является обновление используют современную проверку подлинности пользователей. Современная проверка подлинности — это метод управления удостоверениями, которая позволяет создавать более безопасные проверки подлинности пользователя и авторизации. При наличии политику многофакторной проверки Подлинности на месте от каталога, современная проверка подлинности гарантирует, что пользователю предлагается для многофакторной проверки Подлинности, при необходимости. Это более безопасной альтернативой устаревшие протоколы аутентификации.

В этом разделе дается пошаговое Обзор о том, как обновить среду для современной проверки подлинности. Прочитайте описанные ниже шаги перед включением устаревшей проверки подлинности, блокирует политику в вашей организации.

### <a name="step-1-enable-modern-authentication-in-your-directory"></a>Шаг 1. Включение современной проверки подлинности в вашем каталоге

Первым шагом Включение современной проверки подлинности нужно убедиться, что каталог поддерживает современную проверку подлинности. Современная проверка подлинности включена по умолчанию для каталоги, созданные в течение или после 1 августа 2017 г. Если каталог был создан до этой даты, необходимо вручную включить современную проверку подлинности для каталога, выполнив следующие действия:

1. Проверьте, поддерживает ли каталог уже современную проверку подлинности, выполнив `Get-CsOAuthConfiguration` из [Скайп для бизнеса Online PowerShell модуля](https://docs.microsoft.com/office365/enterprise/powershell/manage-skype-for-business-online-with-office-365-powershell).
1. Если команда возвращает пустую коллекцию `OAuthServers` свойства, а затем современная проверка подлинности отключена. Обновите параметр, чтобы включить современную проверку подлинности с помощью `Set-CsOAuthConfiguration`. Если ваш `OAuthServers` свойство содержит запись, вы готовы к работе.

Убедитесь в том, что для выполнения этого шага, прежде чем двигаться вперед. Крайне важно, что в каталоге конфигурации сначала изменяются так, как они указывают, какой протокол будет использоваться всеми клиентами Office. Даже если вы используете клиентах Office, поддерживающих современную проверку подлинности, они по умолчанию будет использовать устаревшие протоколы Если современная проверка подлинности отключена от каталога.

### <a name="step-2-office-applications"></a>Шаг 2. Приложения Office

После включения современную проверку подлинности в вашем каталоге, можно начать обновление приложений по включению современной аутентификации для клиентов Office. Office 2016 или более новые клиенты поддерживают современную проверку подлинности по умолчанию. Никакие дополнительные действия не требуются.

Если вы используете клиенты Windows Office 2013 или более ранней версии, рекомендуется обновить Office 2016 или более поздней версии. Даже после завершения предыдущего шага по включению современной проверки подлинности в вашем каталоге, более старые приложения Office будет продолжать использовать устаревшие протоколы аутентификации. Если вы используете клиенты Office 2013 и не смогут сразу обновите Office 2016 или более поздней версии, выполните действия, описанные в следующей статье, чтобы [включить современную проверку подлинности для Office 2013 на устройствах Windows](https://docs.microsoft.com/office365/admin/security-and-compliance/enable-modern-authentication). Для защиты вашей учетной записи при использовании проверки подлинности, рекомендуется использовать надежные пароли в вашем каталоге. Ознакомьтесь с [защита паролей Azure AD](../authentication/concept-password-ban-bad.md) запрет слабые пароли в вашем каталоге.

Office 2010 не поддерживает современную проверку подлинности. Будет необходимо обновить все пользователи с Office 2010 до более новой версии Office. Рекомендуется обновить Office 2016 или более поздней версии, так как он блокирует проверки подлинности по умолчанию.

Если вы используете MacOS, рекомендуется выполнить обновление до Office для Mac 2016 или более поздней версии. Если вы используете встроенный почтовый клиент, необходимо иметь MacOS версии 10.14 или более поздней версии на всех устройствах.

### <a name="step-3-exchange-and-sharepoint"></a>Шаг 3. Exchange и SharePoint

Для клиентам Outlook на основе Windows, использующие современную проверку подлинности Exchange Online необходимо включить современную проверку подлинности. Если современная проверка подлинности отключена для Exchange Online, клиенты Outlook на основе Windows, поддерживающих современную проверку подлинности (Outlook 2013 и выше) будет использовать обычную проверку подлинности для подключения к почтовым ящикам Exchange Online.

SharePoint Online включена для современной проверки подлинности по умолчанию. Для каталогов, созданных после 1 августа 2017 г. по умолчанию в Exchange Online включена современная проверка подлинности. Тем не менее, если ранее отключено современную проверку подлинности или вы используете каталога, созданного до этой даты, выполните действия, описанные в следующей статье, чтобы [Включение современной проверки подлинности в Exchange Online](https://docs.microsoft.com/exchange/clients-and-mobile-in-exchange-online/enable-or-disable-modern-authentication-in-exchange-online).

### <a name="step-4-skype-for-business"></a>Шаг 4. Skype для бизнеса

Чтобы избежать запросов проверки подлинности, выполняемых Скайп для бизнеса, бывает необходимо включить современную аутентификацию для Скайп для бизнеса Online. Для каталогов, созданных после 1 августа 2017 г. по умолчанию включена современная проверка подлинности для Скайп для бизнеса.

Чтобы включить современную проверку подлинности в Скайп для бизнеса, мы рекомендуем переход к Microsoft Teams, который поддерживает современную проверку подлинности по умолчанию. Тем не менее если вы не сможете tr в данный момент, будет необходимо включить современную аутентификацию для Скайп для бизнеса Online, чтобы Скайп пользователей начинается, использующими современную проверку подлинности. Выполните следующие действия в этой статье [Скайп для топологий бизнеса, поддерживаемые современной проверкой подлинности](https://docs.microsoft.com/skypeforbusiness/plan-your-deployment/modern-authentication/topologies-supported), для действия, чтобы включить современную проверку подлинности для Скайп для бизнеса.

В дополнение к включению современной аутентификации для Скайп для бизнеса Online, мы рекомендуем современных включить проверку подлинности для Exchange Online при включении современная проверка подлинности для Скайп для бизнеса. Этот процесс поможет синхронизировать состояние современную проверку подлинности в Exchange Online и Скайп для бизнеса online и предотвратить несколько приглашения входа в систему для Скайп для бизнес-клиентами.

### <a name="step-5-using-mobile-devices"></a>Шаг 5. С помощью мобильных устройств

Приложения на мобильном устройстве должны блокировать также устаревшей проверки подлинности. Мы рекомендуем использовать Outlook для мобильных устройств. Outlook Mobile поддерживает современную проверку подлинности по умолчанию и будет удовлетворять другие политики защиты базовых показателей многофакторной проверки Подлинности.

Чтобы использовать собственный почтовый клиент iOS, необходимо будет работать под управлением iOS версии 11.0 или более поздней версии, чтобы убедиться, что почтовый клиент был обновлен для блокировки проверки подлинности.

### <a name="step-6-on-premises-clients"></a>Шаг 6. Локальные клиенты

Если вы являетесь клиентом гибридного, с помощью локального сервера Exchange и Скайп для бизнеса локально, обе службы необходимо обновить, чтобы включить современную проверку подлинности. При использовании современной аутентификации в гибридной среде, вы по-прежнему проверку подлинности пользователей в локальной. Рассказ о авторизации их доступа к ресурсам (файлы или сообщения электронной почты) изменений.

Перед началом работы, включение современной проверки подлинности в локальной, убедитесь, что вы выполнили theIf соответствуют требованиям, теперь вы готовы включить современную проверку подлинности в локальной.

Инструкции по включению современной проверки подлинности см. в следующих статьях:

* [Настройка Exchange на локальном сервере для использования гибридной современной проверкой подлинности](https://docs.microsoft.com/office365/enterprise/configure-exchange-server-for-hybrid-modern-authentication)
* [Использование Скайп современной проверки подлинности (ADAL) для бизнеса](https://docs.microsoft.com/skypeforbusiness/manage/authentication/use-adal)

## <a name="enable-the-baseline-policy"></a>Включить базовую политику

Политика **базовая политика: Блок проверки подлинности (Предварительная версия)** поставляется предварительно настроенным и будут отображаться в верхней при переходе к колонка условного доступа на портале Azure.

Чтобы включить эту политику и обеспечения защиты организации:

1. Войдите в **портала Azure** как глобального администратора, администратора безопасности или администратора условного доступа.
1. Перейдите к **Azure Active Directory** > **условного доступа**.
1. В списке политик выберите **базовая политика: Блок проверки подлинности (Предварительная версия)** .
1. Задайте **включить политику** для **использовать политику немедленно**.
1. Нажмите кнопку **Сохранить**.
