# <a name="what-are-baseline-policies"></a>Что такое базовые политики?

Базовые политики представляют собой набор предопределенных политик, которые помогают защитить организации от многих распространенных атак. К таким распространенным атакам могут относиться распыление паролей, повторное воспроизведение и фишинг. Базовые политики доступны во всех изданиях Azure AD. Корпорация Майкрософт предоставляет эти базовые политики защиты всем, так как за последние несколько лет заметно участились атаки на основе удостоверений. Цель этих четырех политик заключается в обеспечении того, чтобы все организации имели базовый уровень безопасности, включенный без каких-либо дополнительных затрат.

Для управления индивидуальными политиками условного доступа требуется лицензия Azure AD Premium.

> [!IMPORTANT]
> Базовые политики унижаются. [Узнайте, что нового в Active Directory Azure?](../fundamentals/whats-new.md#replacement-of-baseline-policies-with-security-defaults)

## <a name="baseline-policies"></a>Базовые политики

![Базовые политики условного доступа на портале Azure](./media/concept-baseline-protection/conditional-access-baseline-policies.png)

Существует четыре базовые политики:

* Требуется МИД для админов (предварительный просмотр)
* Защита конечных пользователей (предварительный просмотр)
* Блокировка устаревшей аутентификации (предварительный просмотр)
* Требуется МИД для управления сервисом (предварительный просмотр)

Все четыре эти политики повлияют на устаревшие потоки аутентификации, такие как POP, IMAP и старые клиенты настольных компьютеров Office.

### <a name="exclusions"></a>Исключения

Когда базовые политики вошли в их первоначальный публичный предварительный просмотр, был вариант исключить пользователей из политики. Эта возможность развивалась через предварительный просмотр и была удалена в июле 2019 года. Организации, которые уже создали исключения, смогли продолжать удерживать их новыми пользователями, не могут добавить исключения в политику.

### <a name="require-mfa-for-admins-preview"></a>Требуется МИД для админов (предварительный просмотр)

Из-за мощности и доступа, которые имеют учетные записи администратора, вы должны относиться к ним с особой осторожностью. Одним из распространенных методов повышения защиты привилегированных учетных записей является требование более сильной формы проверки учетной записи при их входе. В Active Directory Azure вы можете получить более сильную проверку учетной записи, потребовав от администраторов зарегистрироваться и использовать многофакторную аутентификацию Azure.

Требование МИД для админов (предварительный просмотр) является базовой политикой, которая требует многофакторной аутентификации (MFA) для следующих ролей каталога, которые считаются наиболее привилегированными ролями Azure AD:

* Глобальный администратор.
* администратор SharePoint;
* администратор Exchange;
* Администратор условного доступа
* администратор безопасности;
* Администратор службы поддержки / администратор паролей.
* Администратор выставления счетов
* Администратор пользователей

Если в вашей организации эти учетные записи используются в сценариях или коде, рассмотрите возможность замены их [управляемыми удостоверениями](../managed-identities-azure-resources/overview.md).

### <a name="end-user-protection-preview"></a>Защита конечных пользователей (предварительный просмотр)

Высокопривилегированные администраторы не единственные, кто стал объектом атак. Плохие актеры, как правило, ориентированы на обычных пользователей. После получения доступа эти плохие субъекты могут запросить доступ к конфиденциальной информации от имени первоначального владельца учетной записи или загрузить весь каталог и совершить фишинг-атаку на всю организацию. Одним из распространенных методов повышения защиты для всех пользователей является необходимость более сильной формы проверки учетной записи при обнаружении рискованного ввоза.

**Защита конечных пользователей (предварительный просмотр)** — базовая политика, которая защищает всех пользователей в каталоге. Включение этой политики требует, чтобы все пользователи зарегистрировались для получения многофакторной аутентификации Azure в течение 14 дней. После регистрации пользователям будет предложено выполнять MFA только во время попыток рискованного входа. Скомпрометированные учетные записи пользователей будут заблокированы до тех пор, пока пользователь не сбросит пароль и не отклонит события риска. 

> [!NOTE]
> Любые пользователи, ранее помеченные для риска, блокируются до сбросить пароль и риск увольнения после активации политики.

### <a name="block-legacy-authentication-preview"></a>Блокировка устаревшей аутентификации (предварительный просмотр)

Устаревшие протоколы аутентификации (набы: IMAP, SMTP, POP3) — это протоколы, обычно используемые старыми почтовыми клиентами для проверки подлинности. Устаревшие протоколы не поддерживают многофакторную аутентификацию. Даже если у вас есть политика, требующая многофакторной аутентификации для вашего каталога, плохой актер может аутентифицировать с помощью одного из этих устаревших протоколов и обойти многофакторную аутентификацию.

Лучший способ защитить свою учетную запись от вредоносных запросов на аутентификацию, сделанных устаревшими протоколами, — это заблокировать их.

Базовая политика **аутентификации наследия Block (предварительный просмотр)** блокирует запросы на проверку подлинности, которые делаются с использованием устаревших протоколов. Современная аутентификация должна использоваться для успешного вхластания для всех пользователей. Запросы, поступающие из устаревших протоколов, будут заблокированы в сочетании с другими базовыми политиками. Кроме того, все пользователи будут обязаны МИД, когда это необходимо. Эта политика не блокирует Exchange ActiveSync.

### <a name="require-mfa-for-service-management-preview"></a>Требуется МИД для управления сервисом (предварительный просмотр)

Организации используют различные службы Azure и управляют ими с помощью инструментов, основанных на ресурсах Azure, таких как:

* Портал Azure
* Azure PowerShell
* Azure CLI

Использование любого из этих инструментов для управления ресурсами является весьма привилегированным действием. Эти инструменты могут изменять конфигурации в масштабах всей подписки, такие как настройки служб и выставление счетов за подписку.

Для защиты привилегированных действий эта политика **«Требуя, чтобы МИД» для управления службами (предварительный просмотр)** требовала многофакторной аутентификации для любого пользователя, доступного для портала Azure, Azure PowerShell или Azure CLI.
