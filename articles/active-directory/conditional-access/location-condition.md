# <a name="what-is-the-location-condition-in-azure-active-directory-conditional-access"></a>Каково состояние местоположения в условном доступе к каталогу Azure Active Directory? 

С помощью условного доступа К вашего облачного каталога [(Azure AD)](../active-directory-conditional-access-azure-portal.md)можно управлять тем, как авторизованные пользователи могут получить доступ к облачным приложениям. Условие местоположения политики условного доступа позволяет привязывать настройки элементов управления доступом к местоположению пользователей в сети.

В этой статье приведены сведения, необходимые для настройки условия расположения.

## <a name="locations"></a>Расположения

Azure AD позволяет одним образом входить в систему устройств, приложений и служб из любой точки интернета. Условие расположения позволяет управлять доступом к облачным приложениям на основе расположения пользователя в сети. Ниже приведены наиболее распространенные варианты использования условия расположения.

- Требование многофакторной проверки подлинности для пользователей, которые пытаются получить доступ к службе, но пока отключены от корпоративной сети.
- Блокировка доступа для пользователей, которые пытаются получить доступ к службе из определенных стран или регионов.

Местоположение — это метка для местоположения сети, представляющая либо названное местоположение, либо многофакторная аутентификация Trusted IPs.

## <a name="named-locations"></a>Именованные расположения

С названными местоположениями можно создавать логические группирования диапазонов IP-адресов или стран и регионов.

Вы можете получить доступ к указанным местоположениям в разделе **«Управление»** страницы «Условный доступ».

![Названные места в условном доступе](./media/location-condition/02.png)

Именованное расположение состоит из следующих компонентов.

![Создание нового имени](./media/location-condition/42.png)

- **Имя** — отображаемое имя именованного расположения.
- **Диапазоны IP-адресов** — один или не сколько диапазонов IPv4-адресов в формате CIDR. Указание диапазона адресов IPv6 не поддерживается.

   > [!NOTE]
   > Диапазоны адресов IPv6 в настоящее время не могут быть включены в названное местоположение. Это означает, что диапазоны IPv6 не могут быть исключены из политики условного доступа.

- **Отметить как надежное расположение** — флажок, который можно установить для именованного расположения, чтобы указать надежное расположение. Как правило, надежные расположения являются областями сети, которые контролируются ИТ-отделом. Помимо условного доступа, доверенные именные местоположения также используются в отчетах по защите идентификационных данных Azure и AD Azure AD для уменьшения [ложных срабатываний.](../reports-monitoring/concept-risk-events.md#impossible-travel-to-atypical-locations-1)
- **Страны и регионы**. Этот параметр позволяет выбрать одну или несколько стран или регионов для определения именованного расположения.
- **Включите неизвестные области** - Некоторые IP-адреса не отображаются в конкретной стране или регионе. Этот параметр позволяет указать, нужно ли включать эти IP-адреса в именованное расположение. Используйте данный параметр, когда политика, использующая именованное расположение, должна применяться к неизвестным расположениям.

Количество именованных расположений, которые можно настроить, ограничено размером соответствующего объекта в Azure AD. Вы можете настроить местоположения на основе следующих ограничений:

- Именованное расположение с диапазонами IP-адресов до 1200.
- Максимум 90 именованных расположений, каждому из которых назначен один диапазон IP-адресов.

Политика условного доступа применяется к трафику IPv4 и IPv6. В настоящее время названные места не позволяют настроить диапазоны IPv6. Это ограничение вызывает следующие ситуации:

- Политика условного доступа не может быть ориентирована на определенные диапазоны IPv6
- Политика условного доступа не может исключать определенные диапазоны IPV6

Если политика настроена для применения к "Любое местоположение", она будет применяться к трафику IPv4 и IPv6. Названные местоположения, настроенные для определенных стран и регионов, поддерживают только адреса IPv4. Трафик IPv6 включен только в том случае, если выбрана возможность "включить неизвестные области".

## <a name="trusted-ips"></a>Надежные IP-адреса

Вы также можете настроить диапазоны IP-адресов, представляющие локальную интрасеть в [параметрах службы многофакторной проверки подлинности](https://account.activedirectory.windowsazure.com/usermanagement/mfasettings.aspx). Эта функция позволяет настроить до 50 диапазонов IP-адресов. Диапазоны IP-адресов имеют формат CIDR. Для получения дополнительной информации [см.](../authentication/howto-mfa-mfasettings.md#trusted-ips)  

Если у вас настроены доверенные IP-адреса, они отображаются как **MFA Trusted IPS** в списке мест для состояния местоположения.

### <a name="skipping-multi-factor-authentication"></a>Пропуск многофакторной проверки подлинности

На странице параметров службы многофакторной проверки подлинности можно определить пользователей корпоративной интрасети, установив флажок **Пропустить многофакторную проверку подлинности для запросов от федеративных пользователей из моей интрасети**. Этот параметр указывает, что внутреннее утверждение корпоративной сети, выдаваемое сервером служб федерации Active Directory (AD FS), должно быть надежным и использоваться для идентификации пользователя в корпоративной сети. Для получения дополнительной информации [см. Функцию «Включить доверенные IPs» с помощью «Условного доступа».](../authentication/howto-mfa-mfasettings.md#enable-the-trusted-ips-feature-by-using-conditional-access)

После проверки этой опции, в ключая названное местоположение **MFA Trusted IPS,** будут применяться к любым политикам с выбранной этой опцией.

Для мобильных и настольных приложений, которые уже давно проживают в течение сессии, Условный доступ периодически пересматривается. По умолчанию — один раз в час. Если внутренне утверждение корпоративной сети выпускается только во время первоначальной проверки подлинности, Azure AD может не иметь списка диапазонов надежных IP-адресов. В этом случае трудно определить, находится ли по-прежнему пользователь в корпоративной сети.

1. Проверьте, входит ли IP-адрес пользователя в один из диапазонов надежных IP-адресов.
2. Проверьте, соответствуют ли первые три окте IP-адреса пользователя первым трем окте ip-адреса первоначальной аутентификации. IP-адрес сравнивается с первоначальной аутентификацией, когда внутренняя претензия корпоративной сети была первоначально выдана и местоположение пользователя было проверено.

Если оба этапа завершатся ошибкой, считается, что пользователь больше не использует надежный IP-адрес.

## <a name="location-condition-configuration"></a>Настройка условия расположения

При настройке условия расположения вы можете выбрать:

- Любое расположение
- Все надежные расположения
- Выбранные расположения

![Настройка условия расположения](./media/location-condition/01.png)

### <a name="any-location"></a>Любое расположение

По умолчанию выбор варианта **Любое местонахождение** приводит к тому, что политика применяется ко всем IP-адресам (то есть к любому адресу в Интернете). Этот параметр не ограничивается IP-адресами, настроенными как именованное расположение. Выбрав вариант **Любое местонахождение**, вы все равно можете исключить определенные расположения из политики. Например, можно применить политику ко всем расположениям (за исключением надежных расположений), чтобы задать для области все расположения (за исключением корпоративной сети).

### <a name="all-trusted-locations"></a>Все надежные расположения

Этот параметр применяется:

- ко всем расположениям, помеченным как надежное расположение;
- **надежным IP-адресам MFA** (если настроены).

### <a name="selected-locations"></a>Выбранные расположения

С помощью этого параметра можно выбрать одно или несколько именованных расположений. Для применения политики с этим параметром пользователю необходимо подключиться из любого из выбранных расположений. Если щелкнуть **Выбрать**, откроется элемент управления со списком именованных сетей. В списке также показано, является ли надежным расположение в сети. Именованное расположение **Надежные IP-адреса MFA** используется для включения параметров IP-адресов, которые можно настроить на странице параметров службы многофакторной проверки подлинности.

## <a name="what-you-should-know"></a>Необходимая информация

### <a name="when-is-a-location-evaluated"></a>Когда выполняется проверка расположения

Политики условного доступа оцениваются при:

- Изначально пользователь входит в веб-приложение, мобильное или классическое приложение.
- Мобильное или классическое приложение, в котором применяется современная аутентификация, использует маркер обновления для получения нового маркера доступа. По умолчанию эта проверка один раз в час.

Эта проверка означает для мобильных и настольных приложений с использованием современной аутентификации, изменение местоположения будет обнаружено в течение часа после изменения местоположения сети. Для мобильных и классических приложений, которые не используют современную аутентификацию, к каждому запросу на маркер применяется политика. Частота выполнения запроса зависит от приложения. Для веб-приложений политика также применяется во время начального входа и действует в течение времени существования сеанса веб-приложения. Так как в разных приложениях время сеанса будет разным, время между оценкам политики также может изменяться. Каждый раз, когда приложение запрашивает новый маркер входа, применяется политика.

По умолчанию Azure AD выдает токен на почасовой основе. После перемещения из корпоративной сети в течение часа политика применяется для приложений, использующих современную проверку подлинности.

### <a name="user-ip-address"></a>IP-адрес пользователя

При оценке политики используется общедоступный IP-адрес пользователя. Для устройств в частной сети этот IP-адрес не является клиентским IP-адреса устройства пользователя в интрасети, а адресом, используемым сетью для подключения к общедоступному интернету.

> [!WARNING]
> Если у вашего устройства есть только адрес IPv6, настройка состояния местоположения не поддерживается.

### <a name="bulk-uploading-and-downloading-of-named-locations"></a>Массовое обновление и загрузка именованных расположений

При создании или обновлении именованных расположений для массовых обновлений можно передать или скачать CSV-файл с диапазонами IP-адресов. При передаче диапазоны IP-адресов заменяются адресами, указанными в файле. Каждая строка файла содержит один диапазон IP-адресов в формате CIDR.

### <a name="cloud-proxies-and-vpns"></a>Облачные прокси-серверы и виртуальные частные сети (VPN)

Если используется прокси-сервер, размещенный в облаке, или решение VPN, при оценке политики Azure AD использует IP-адрес прокси-сервера. Заголовок X-Forwarded-For (XFF), содержащий общедоступный IP-адрес пользователей, не используется, так как нет подтверждения того, что он поступает из надежного источника, а значит может быть поддельным.

При наличии облачного прокси-сервера можно использовать политику, требующую присоединенного к домену устройства, или внутреннее утверждение корпоративной сети из AD FS.

### <a name="api-support-and-powershell"></a>Поддержка API и PowerShell

API и PowerShell пока не поддерживаются для названных местоположений или для политик условного доступа.
