# <a name="configure-authentication-session-management-with-conditional-access"></a>Настройка управления сеанса проверки подлинности с помощью условного доступа

В сложном развертывании организации могут иметь необходимость ограничить сеансов проверки подлинности. Некоторые сценарии могут включать:

* Доступ к ресурсам с неуправляемых или общего устройства
* Доступ к конфиденциальной информации из внешней сети
* Высокая степень влияния на пользователей
* Важных бизнес-приложений

Элементы управления условного доступа позволяют создавать политики, предназначенных конкретные варианты использования в вашей организации, не влияя на всех пользователей.

Прежде чем углубляться в подробности о том, как настроить политику, давайте рассмотрим конфигурацию по умолчанию.

## <a name="user-sign-in-frequency"></a>Частота связи входа пользователя

Частота входа определяет период времени, прежде чем пользователю будет предложено снова войти в систему при попытке доступа к ресурсу.

Azure Active Directory (Azure AD) по умолчанию для пользователя при входе частота устанавливается скользящее окно 90 дней. Запрашивает у пользователей учетные данные часто может показаться разумным, что необходимо сделать, но его можно но: пользователей, которые обучаются вводить учетные данные без думать может непреднамеренно предоставить им строку вредоносных учетных данных.

Это может казаться огромной не запрашивать пользователя заново выполнить вход в течение 90 дней, на самом деле, все нарушения политики ИТ отменит сеанс. Некоторые примеры причинами (но не ограничиваются) изменение пароля, несоответствия устройства или отключить учетную запись. Можно также явно [отозвать сеансов пользователей с помощью PowerShell](https://docs.microsoft.com/powershell/module/azuread/revoke-azureaduserallrefreshtoken?view=azureadps-2.0). Конфигурация по умолчанию Azure AD зависит от «пользователям вводить свои учетные данные, если не был изменен уровень защищенности свои сеансы не спрашивайте».

Частота вход задание работает с приложениями, которые реализованы протоколы OATH2 или OIDC в соответствии со стандартами. Большинство собственных приложений Microsoft для Windows, Mac и мобильных устройств соответствует параметру.

## <a name="persistence-of-browsing-sessions"></a>Сохраняемость сеансы просмотра

Сеанс постоянных обозревателя позволяет пользователям оставаться в системе после закрытия и повторного открытия окна обозревателя.

Сохранение сеанса браузера по умолчанию Azure AD позволяет пользователям на личных устройствах, чтобы выбрать, следует ли сохранять сеанс, отображая «оставаться в системе?» запрос после успешной проверки подлинности. Если браузер сохраняемость можно настроить в AD FS, следуя указаниям из статьи [параметры AD FS единый вход](https://docs.microsoft.com/windows-server/identity/ad-fs/operations/ad-fs-single-sign-on-settings#enable-psso-for-office-365-users-to-access-sharepoint-online
), мы будет соответствовать этой политике и сохранение сеанса AD, а также Azure. Можно также настроить ли пользователи в клиенте см. в разделе «Веб-узел Stay входа?» запрос, изменив соответствующий параметр в фирменной символики область на портале Azure, следуя указаниям из статьи [Настройка страницы входа в Azure AD](../fundamentals/customize-branding.md).

## <a name="configuring-authentication-session-controls"></a>Настройка элементов управления проверки подлинности сеанса

Условный доступ — это функция Azure AD Premium и требует лицензии уровня "премиум". Дополнительные сведения см. в статье [Условный доступ в Azure Active Directory](overview.md#license-requirements)

> [!WARNING]
> Если вы используете [настройки времени жизни маркера](../develop/active-directory-configurable-token-lifetimes.md) функция в настоящее время в общедоступной предварительной версии, обратите внимание, что мы не поддерживаем создание двух разных политик для каждого сочетания пользователя или приложения: один с использованием этой функции и еще один с возможность настройки времени жизни маркера. Корпорация Майкрософт планирует снять с учета функцию настройки времени жизни маркера на 15 октября 2007 г и замените его компонент управления сеанса проверки подлинности условного доступа.  

### <a name="policy-1-sign-in-frequency-control"></a>Политика 1: Частота входа в систему управления

1. Создать новую политику
1. Выберите все необходимые условия для среды клиента, включая целевой облачных приложений.

   > [!NOTE]
   > Рекомендуется, чтобы задать частоту prompt равно проверки подлинности для ключа приложения Microsoft Office, например Exchange Online и SharePoint Online для удобства работы пользователей.

1. Перейдите к **элементы управления доступом** > **сеанса** и нажмите кнопку **частота входа в систему**
1. Введите требуемое значение свойства дни и часы в первое текстовое поле
1. Выберите для параметра **часов** или **дней** из раскрывающегося списка
1. Сохраните политику

![Политики условного доступа, настроенных для входа в частота](media/howto-conditional-access-session-lifetime/conditional-access-policy-session-sign-in-frequency.png)

В Azure AD зарегистрированных Windows устройства входа на устройство считается запрос. Например если вы настроили знак частотой до 24 часов для приложений Office, пользователи Azure AD зарегистрировано Windows устройства будет удовлетворять входа политика частоты при входе на устройство и не предлагается повторно при открытии приложения Office.

Если вы настроили другой вход частотой для разных веб-приложений, запущенных в одном сеансе браузера, строгой политики применяется в обоих приложениях так как маркер один сеанс, совместно используют все приложения, работающие в одном сеансе браузера.

### <a name="policy-2-persistent-browser-session"></a>Политика 2: Сеанс постоянных браузера

1. Создать новую политику
1. Выберите все обязательные условия.

   > [!NOTE]
   > Обратите внимание на то, что данного элемента управления требуется выбрать «Все облачные приложения» в условии. Сохранение сеанса браузера управляется токена сеанса проверки подлинности. Все вкладки в сеансе браузера, совместно используют один сеанс маркер и поэтому все они должны совместно использовать состояние сохраняемости.

1. Перейдите к **элементы управления доступом** > **сеанса** и нажмите кнопку **сеанс постоянных браузера**
1. Выберите значение из раскрывающегося списка
1. Сэкономить политики

![Политики условного доступа, настроенного для постоянных браузера](media/howto-conditional-access-session-lifetime/conditional-access-policy-session-persistent-browser.png)

> [!NOTE]
> Постоянные конфигурации сеанс браузера в условного доступа Azure AD приведет к перезаписи «оставаться в системе?» Настройка фирменной символики панели на портале Azure для одного пользователя, если вы настроили обе политики компании.
