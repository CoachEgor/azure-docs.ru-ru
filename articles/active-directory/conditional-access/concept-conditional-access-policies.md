# <a name="building-a-conditional-access-policy"></a>Создание политики условного доступа

Как поясняется в статье [Что такое условный доступ](overview.md), политика условного доступа является заявлением if-then, **заданий** и **элементов управления доступом.** Политика условного доступа объединяет сигналы для принятия решений и обеспечения соблюдения организационных политик.

Как организация создает эти политики? Что требуется?

![Условный доступ (Сигналы и решения - Правоприменение и политика)](./media/concept-conditional-access-policies/conditional-access-signal-decision-enforcement.png)

## <a name="assignments"></a>Назначения

Часть назначений контролирует, кто, что и где политики условного доступа.

### <a name="users-and-groups"></a>Пользователи и группы

[Пользователи и группы](concept-conditional-access-users-groups.md) назначают, кто будет включать или исключать политику. Это назначение может включать всех пользователей, определенные группы пользователей, роли каталогов или внешних гостевых пользователей. 

### <a name="cloud-apps-or-actions"></a>Облачные приложения или действия

[Облачные приложения или действия](concept-conditional-access-cloud-apps.md) могут включать или исключать облачные приложения или действия пользователей, на которые будет распространяться политика.

### <a name="conditions"></a>Условия

Политика может содержать несколько [условий.](concept-conditional-access-conditions.md)

#### <a name="sign-in-risk"></a>Риск при входе

Для организаций с [Azure AD Identity Protection,](../identity-protection/overview.md)обнаружение рисков, генерируемых там, может повлиять на политики условного доступа.

#### <a name="device-platforms"></a>Платформы устройств

Организации с несколькими платформами операционных систем устройств могут пожелать применять определенные политики на разных платформах. 

Информация, используемая для расчета платформы устройства, поступает из непроверенных источников, таких как строки агента пользователя, которые могут быть изменены.

#### <a name="locations"></a>Расположения

Данные о местоположении предоставляются на данных геолокации IP. Администраторы могут выбрать для определения местоположения и выбрать, чтобы отметить некоторые из них как доверенные, как те, для их сети местах организации.

#### <a name="client-apps"></a>Клиентские приложения

По умолчанию политики условного доступа применяются к приложениям для браузеров, мобильным приложениям и настольным клиентам, поддерживающим современную аутентификацию. 

Это условие назначения позволяет политикам условного доступа ориентироваться на определенные клиентские приложения, не использующие современную аутентификацию. Эти приложения включают в себя клиентов Exchange ActiveSync, старые приложения Office, которые не используют современную аутентификацию, и почтовые протоколы, такие как IMAP, MAPI, POP и SMTP.

#### <a name="device-state"></a>Состояние устройства

Этот элемент управления используется для исключения устройств, которые являются гибридными Azure AD, объединенными или отмеченными совместимыми в Intune. Это исключение может быть сделано для блокировки неуправляемых устройств. 

## <a name="access-controls"></a>Элементы управления доступом

Контроль доступа часть политики условного доступа контролирует, как выполняется политика.

### <a name="grant"></a>Предоставить

[Грант](concept-conditional-access-grant.md) предоставляет администраторам средства обеспечения соблюдения политики, где они могут блокировать или предоставлять доступ.

#### <a name="block-access"></a>Заблокировать доступ

Блок-доступ делает именно это, он будет блокировать доступ в рамках указанных назначений. Контроль блока является мощным и должны быть орудуты с соответствующими знаниями.

#### <a name="grant-access"></a>Предоставление доступа

Контроль грантов может привести к принудительному исполнению одного или нескольких элементов управления. 

- Требуется многофакторная аутентификация (Многофакторная аутентификация Azure)
- Требуется, чтобы устройство было помечено как совместимое (Intune)
- Требование устройства с гибридным присоединением к Azure AD
- Требование утвержденного клиентского приложения
- Требование политики защиты приложений

Администраторы могут потребовать одного из предыдущих элементов управления или всех выбранных элементов управления, используя следующие параметры. По умолчанию для нескольких элементов управления требуется все.

- Требуются все выбранные элементы управления (управление и управление)
- Требуется один из выбранных элементов управления (управление или управление)

### <a name="session"></a>Сеанс

[Элементы управления сеансами](concept-conditional-access-session.md) могут ограничить 

- Использование примененных к приложению ограничений
   - В настоящее время работает только с Exchange Online и SharePoint Online.
      - Передает информацию об устройстве, чтобы контролировать опыт предоставления полного или ограниченного доступа.
- Используйте управление приложениями условного доступа
   - Использует сигналы от Microsoft Cloud App Security для таких вещей, как: 
      - Блокируйте загрузку, вырез, копирование и печать конфиденциальных документов.
      - Мониторинг рискованного поведения сеанса.
      - Требуется маркировка конфиденциальных файлов.
- Частота входиной регистрации
   - Возможность изменения знака по умолчанию в частоте для современной аутентификации.
- Стойкая сессия браузера
   - Позволяет пользователям оставаться в сети после закрытия и повторного открытия окна браузера.

## <a name="simple-policies"></a>Простые политики

Политика условного доступа должна содержать как минимум следующее, необходимое для принудительного исполнения:

- **Название** политики.
- **Назначения**
   - **Пользователи и/или группы** для применения политики.
   - **Облачные приложения или действия,** к применению политики.
- **Элементы управления доступом**
   - **Элементы** управления грантом или **блоком**

![Политика условного доступа пустой](./media/concept-conditional-access-policies/conditional-access-blank-policy.png)
