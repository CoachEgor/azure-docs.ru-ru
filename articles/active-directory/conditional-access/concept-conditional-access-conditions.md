# <a name="conditional-access-conditions"></a>Условный доступ: условия

В политике условного доступа администратор может использовать сигналы из таких условий, как риск, платформа устройства или местоположение, чтобы улучшить свои политические решения. 

![Определение политики условного доступа и определение условий](./media/concept-conditional-access-conditions/conditional-access-conditions.png)

Несколько условий могут быть объединены для создания мелкозернистых и конкретных политик условного доступа.

Например, при доступе к конфиденциальному приложению администратор может учесть информацию о рисках, всданных в систему Identity Protection и местоположение, в решение о доступе в дополнение к другим элементам управления, таким как многофакторная аутентификация.

## <a name="sign-in-risk"></a>Риск при входе

Для клиентов, имеющих доступ к [identity Protection,](../identity-protection/overview-identity-protection.md)риск входной системы может быть оценен как часть политики условного доступа. Риск ввоза представляет вероятность того, что данный запрос на проверку подлинности не авторизован владельцем удостоверения личности. Более подробную информацию о риске входе можно найти в статьях ["Что такое риск](../identity-protection/concept-identity-protection-risks.md#sign-in-risk) и [как: настроить и включить политику риска.](../identity-protection/howto-identity-protection-configure-risk-policies.md)

## <a name="device-platforms"></a>Платформы устройств

Платформа устройства характеризуется операционной системой, которая работает на устройстве. Azure AD идентифицирует платформу с помощью информации, предоставляемой устройством, например строки агента пользователя. Поскольку строки агента пользователя могут быть изменены, эта информация не проверена. Платформа устройств должна использоваться совместно с политиками соответствия требованиям устройств Microsoft Intune или как часть оператора блока. По умолчанию применяются ко всем платформам устройств.

Условный доступ Azure AD поддерживает следующие платформы устройств:

- Android
- iOS
- Windows Phone
- Windows
- macOS

Если вы блокируете устаревшую аутентификацию с помощью условия **других клиентов,** можно также установить условие платформы устройства.

## <a name="locations"></a>Расположения

При настройке местоположения в качестве условия организации могут включать или исключать местоположения. Эти названные места могут включать общедоступную информацию о сети IPv4, страну или регион или даже неизвестные области, которые не отображаются в конкретных странах или регионах. Только диапазоны IP могут быть помечены как доверенное местоположение.

При включении **любого местоположения,** эта опция включает в себя любой IP-адрес в Интернете, а не просто настроены имени места. При выборе **любого местоположения**администраторы могут исключить **все доверенные** или **выбранные места.**

Например, некоторые организации могут отказаться от многофакторной аутентификации, когда их пользователи подключены к сети в надежном месте, например в их физической штаб-квартире. Администраторы могут создать политику, которая включает в себя любое местоположение, но исключает выбранные места для своих сетей штаб-квартиры.

Более подробную информацию о местоположениях можно найти в статье ["Каково состояние местоположения в условном доступе к каталогу Azure Active Directory"](location-condition.md).

## <a name="client-apps-preview"></a>Клиентские приложения (предварительный просмотр)

Политики условного доступа по умолчанию применяются к браузерным приложениям и приложениям, использующим современные протоколы аутентификации. В дополнение к этим приложениям, администраторы могут включить Exchange ActiveSync клиентов и других клиентов, которые используют устаревшие протоколы.

- Браузер
   - К ним относятся веб-приложения, которые используют протоколы, такие как SAML, WS-Federation, OpenID Connect, или услуги, зарегистрированные в качестве конфиденциального клиента OAuth.
- Мобильные приложения и настольные клиенты
   - Современные клиенты аутентификации
      - Эта опция включает приложения, такие как настольные и телефонные приложения Office.
   - Клиенты Exchange ActiveSync
      - По умолчанию это включает в себя все использование протокола Exchange ActiveSync (EAS). Выбор **политики применения только для поддерживаемых платформ** будет ограничиваться поддерживаемыми платформами, такими как iOS, Android и Windows.
      - Когда политика блокирует использование Exchange ActiveSync, пострадавший пользователь получит одно карантинное письмо. Это письмо с предоставлением информации о том, почему они заблокированы и включают инструкции по исправлению положения, если это возможно.
   - Другие клиенты
      - Эта опция включает клиентов, которые используют основные/устаревшие протоколы аутентификации, которые не поддерживают современную аутентификацию.
         - Аутентифицированный SMTP - Используется клиентом POP и IMAP для отправки сообщений электронной почты.
         - Autodiscover - Используется клиентами Outlook и EAS для поиска и подключения к почтовым ящикам в Exchange Online.
         - Exchange Online PowerShell - Используется для подключения к Exchange Online с удаленным PowerShell. Если вы блокируете базовую аутентификацию для Exchange Online PowerShell, вам необходимо использовать модуль Exchange Online PowerShell для подключения. Для получения инструкций [см. Подключитесь к Exchange Online PowerShell с помощью многофакторной аутентификации.](/powershell/exchange/exchange-online/connect-to-exchange-online-powershell/mfa-connect-to-exchange-online-powershell)
         - Exchange Web Services (EWS) - интерфейс программирования, используемый Outlook, Outlook для Mac и сторонними приложениями.
         - IMAP4 - Используется клиентами электронной почты IMAP.
         - MAPI над HTTP (MAPI/HTTP) - Используется Outlook 2010 и позже.
         - Offline Address Book (OAB) - копия коллекций адресного списка, которые загружаются и используются Outlook.
         - Outlook Anywhere (RPC over HTTP) - Используется Outlook 2016 и ранее.
         - Outlook Service - Используется приложением "Почта" и "Календарь" для Windows 10.
         - POP3 - Используется клиентами электронной почты POP.
         - Отчетность веб-служб - Используется для получения данных отчета в Exchange Online.

Эти условия обычно используются при необходимости управляемого устройства, блокирования устаревшей аутентификации и блокировки веб-приложений, но допускающих мобильные или настольные приложения.

#### <a name="why-do-i-see-a-certificate-prompt-in-the-browser"></a>Почему я вижу запрос сертификата в браузере

На Windows 7, iOS, Android и macOS Azure AD идентифицируют устройство с помощью сертификата клиента, который предоставляется при регистрации устройства в Azure AD.  Когда пользователь впервые вписывается через браузер, пользователю предлагается выбрать сертификат. Пользователь должен выбрать этот сертификат перед использованием браузера.

#### <a name="chrome-support"></a>Поддержка Chrome

Для поддержки Chrome в **Windows 10 Создатели Обновление (версия 1703)** или позже, установите [расширение Windows 10 Учетные записи](https://chrome.google.com/webstore/detail/windows-10-accounts/ppnbnpeolgkicgegkbkbjmhlideopiji). Это расширение требуется, когда политика условного доступа требует конкретных деталей устройства.

Чтобы автоматически развернуть это расширение в браузере Chrome, создайте следующий раздел реестра:

|    |    |
| --- | --- |
| путь | HKEY_LOCAL_MACHINE\Software\Policies\Google\Chrome\ExtensionInstallForcelist |
| name | 1 |
| Тип | REG_SZ (String) |
| Данные | ppbnpeolgkicgegkbbjmhlideopi;https//clients2.google.com/service/update2/crx\: |

Для поддержки Chrome в **Windows 8.1 и Windows 7** создайте следующий раздел реестра:

|    |    |
| --- | --- |
| путь | HKEY_LOCAL_MACHINE\SOFTWARE\Policies\Google\Chrome\AutoSelectCertificateForUrls |
| name | 1 |
| Тип | REG_SZ (String) |
| Данные | {"pattern":"https://device.login.microsoftonline.com","filter":{"ISSUER":{"CN":"MS-Organization-Access"}}} |

Эти браузеры поддерживают аутентификацию устройств, позволяя идентифицировать устройство и проверить, соответствует ли оно политике. Если браузер работает в частном режиме, проверка устройства завершается ошибкой.

### <a name="exchange-activesync-clients"></a>Клиенты Exchange ActiveSync

- Организации могут выбирать клиентов Exchange ActiveSync только при назначении политики пользователям или группам. Выбор **всех пользователей,** **всех гостевых и внешних пользователей**или **ролей каталога** приведет к блокировке всех пользователей.
- При создании политики, назначенной клиентам Exchange ActiveSync, **Office 365 Exchange Online** должен быть единственным облачным приложением, назначенным для политики. 
- Организации могут сузить область действия этой политики до определенных платформ, используя условие **платформ ы устройства.**

Если элемент доступа, назначенный для политики, использует **утвержденное клиентское приложение,** пользователю предписано установить и использовать мобильный клиент Outlook. В случае необходимости **многофакторной аутентификации** пострадавшие пользователи блокируются, поскольку базовая аутентификация не поддерживает многофакторную аутентификацию.

### <a name="other-clients"></a>Другие клиенты

Выберите вариант **Другие клиенты**, чтобы указать условие, которое влияет на приложения, использующие обычную проверку подлинности с протоколами электронной почты IMAP, MAPI, POP, SMTP, и приложения Office более ранних версий, которые не используют современные методы проверки подлинности.

## <a name="device-state-preview"></a>Состояние устройства (предварительный просмотр)

Условие состояния устройства может использоваться для исключения устройств, которые являются гибридными AD-каналом Azure, объединенными и/или устройствами, помеченными как совместимые с политикой соответствия Microsoft Intune, из политики условного доступа организации.

Например, *все пользователи,* обращающиеся к облачному приложению *Microsoft Azure Management,* включая все состояния **устройств,** за исключением **устройства Hybrid Azure AD, присоединились и** **устройства, отмеченные как совместимые** и для *элементов управления доступом,* **Блок.** 
   - В этом примере будет создана политика, позволяющая получить доступ к microsoft Azure Management только с устройств, которые являются гибридными AD Azure, объединенными и/или устройствами, помеченными как совместимые.
