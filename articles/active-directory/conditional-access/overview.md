# <a name="what-is-conditional-access-in-azure-active-directory"></a>Что представляет собой условный доступ в Azure Active Directory?

Безопасность очень важна для организаций, использующих облако. Ключевым аспектом облачной безопасности является идентификация и доступ при управлении облачными ресурсами. В мире мобильных и облачных технологий пользователи могут откуда угодно обращаться к ресурсам организации с помощью различных устройств и приложений. Поэтому уже недостаточно просто указать, кто имеет доступ к ресурсу. Чтобы управлять балансом между безопасностью и производительностью, в решении по управлению доступом вам также нужно учитывать способ доступа к ресурсам. Это требование можно выполнить с помощью условного доступа Azure Active Directory (Azure AD). Условный доступ — это возможность Azure Active Directory. Благодаря условному доступу вы можете внедрять автоматизированные решения управления доступом для доступа к облачным приложениям на основе условий.

Политики условного доступа применяются после того, как пользователь прошел однофакторную аутентификацию. Поэтому условный доступ не может предотвратить атаки типа "отказ в обслуживании" (DoS). Но этот механизм может использовать сигналы, поступающие от этих событий (например, уровень риска при входе, расположение запроса и т. д.), для определения доступа.  

![Контроль](./media/overview/81.png)

В этой статье приведены общие сведения об условном доступе в Azure AD.

## <a name="common-scenarios"></a>Распространенные сценарии

В эпоху мобильных и облачных технологий Azure Active Directory обеспечивает единый вход для устройств, приложений и служб из любого расположения. В связи с увеличением количества устройств (включая BYOD) и сторонних приложений SaaS, а также растущей популярностью работы за пределами корпоративных сетей вы все чаще сталкиваетесь с двумя противоположными задачами:

- продуктивная работа пользователей в любом месте и в любое время;
- постоянная защита корпоративных ресурсов.

С помощью политик условного доступа можно применить элементы управления доступом в соответствии с необходимыми условиями. Условный доступ Azure AD позволяет при необходимости повысить безопасность и не влияет на работу пользователей, когда это не требуется.

Ниже приведено несколько распространенных проблем доступа, которые может устранить условный доступ.

- **[Риск при входе](conditions.md#sign-in-risk)**. Служба "Защита идентификации Azure Active Directory" обнаруживает риски при входе. Как ограничить доступ, если обнаруженный риск при входе указывает на несоответствующий субъект? Что делать, если вы хотите получить более убедительные доказательства того, что вход выполнен полномочным пользователем? Что делать, чтобы заблокировать доступ определенных пользователей к приложению?  

- **[Расположение в сети](location-condition.md)**. Служба Azure AD доступна откуда угодно. Что, если выполняется попытка доступа из расположения в сети, которое не находится под управлением вашего ИТ-отдела? Использование сочетания имени пользователя и пароля может быть достаточным для подтверждения подлинности при доступе из корпоративной сети. Что, если требуется более надежное подтверждение подлинности при попытках доступа, инициируемых из других непредвиденных стран или регионов мира? Что, если необходимо даже блокировать попытки доступа из определенных расположений?  

- **[Управление устройствами](conditions.md#device-platforms)**. В Azure AD пользователи могут обращаться к облачным приложениям, используя широкий спектр устройств, включая мобильные, а также используя личные устройства. Что, если эти попытки доступа должны будут предприниматься только с устройств, управляемых вашим ИТ-отделом? Что, если требуется даже блокировать доступ с устройств определенных типов к облачным приложениям в вашей среде?

- **[Клиентское приложение](conditions.md#client-apps)**. Сейчас для доступа ко многим облачным приложениям используются разные типы приложений, в том числе веб-приложения, мобильные приложения или классические приложения. Что, если попытка доступа выполняется с использованием типа клиентского приложения, вызывающего известные проблемы? Что, если для определенных типов приложений требуется устройство, которым управляет ваш ИТ-отдел?

Эти вопросы и ответы на них дают представление о распространенных сценариях доступа, в которых применяется условный доступ Azure AD.
Условный доступ — это компонент Azure Active Directory, который позволяет обрабатывать сценарии доступа, используя подход на основе политик.

## <a name="conditional-access-policies"></a>Политики условного доступа

Политика условного доступа — это определение сценария доступа с использованием следующего шаблона:

![Контроль](./media/overview/10.png)

Элемент **Then do this** (Сделать это) определяет ответ политики. Важно отметить, что цель политики условного доступа — не предоставление доступа к облачному приложению. В Azure AD для предоставления доступа к облачным приложениям используется назначение пользователей. С помощью политики условного доступа можно управлять способом доступа авторизованных пользователей (пользователей, которым предоставлен доступ к облачному приложению) к облачным приложениям при определенных условиях. С помощью ответа можно принудительно применить дополнительные требования, например прохождение многофакторной проверки подлинности, использование управляемого устройства и т. д. В контексте условного доступа Azure AD требования, которые принудительно применяет политика, называются элементами управления доступом. Наиболее строгая форма политики может блокировать доступ. Дополнительные сведения см. в разделе [Элементы управления условным доступом в Azure Active Directory](controls.md).

Элемент **When this happens** (В этом случае) определяет причину активации политики. Эта причина определяется группой условий, которые должны быть соблюдены. В условном доступе Azure AD особую роль играют два условия назначения.

- **[Пользователи](conditions.md#users-and-groups)**. Пользователи, выполняющие попытку доступа (**кто**).

- **[Облачные приложения](conditions.md#cloud-apps-and-actions)**. Целевые объекты для попытки доступа (**что**).

Эти два условия являются обязательными в политике условного доступа. Помимо этих двух обязательных условий можно также добавить дополнительные условия, которые описывают, как выполняется попытка доступа. К распространенным примерам можно отнести использование мобильных устройств или расположений, которые находятся за пределами корпоративной сети. Дополнительные сведения см. в разделе [Условия в условном доступе Azure Active Directory](conditions.md).

Сочетание условий с элементами управления доступом и представляет собой политику условного доступа.

![Контроль](./media/overview/51.png)

С помощью условного доступа Azure AD можно контролировать доступ авторизованных пользователей к облачным приложениям. Цель политики условного доступа — применение дополнительных элементов управления доступом при попытке доступа к облачному приложению на основе способа, которым осуществляется эта попытка.

Подход на основе политик для защиты доступа к облачным приложениям позволяет начать проектировать требования политики для своей среды, используя структуру, описанную в этой статье, и не беспокоясь о технической реализации.

## <a name="azure-ad-conditional-access-and-federated-authentication"></a>Условный доступ Azure AD и федеративная проверка подлинности

Политики условного доступа могут выполняться вместе с [федеративной проверкой подлинности](../../security/azure-ad-choose-authn.md#federated-authentication). Поддерживаются все применимые условия и элементы управления, а [отчеты Azure AD](../reports-monitoring/concept-sign-ins.md) обеспечивают видимость применения политики к операциям входа, которые выполняет активный пользователь.

*Федеративная проверка подлинности с помощью Azure AD* означает, что доверенная служба аутентификации осуществляет проверку подлинности пользователя в Azure AD. Доверенной службой аутентификации могут выступать службы федерации Active Directory (AD FS) или любая другая служба федерации. В этой конфигурации основная проверка подлинности пользователя осуществляется в службе, а затем Azure AD используется для выполнения входа в отдельные приложения. Условный доступ Azure AD применяется перед предоставлением пользователю доступа к запрашиваемому приложению. 

Если настроенная политика условного доступа требует многофакторной проверки подлинности, Azure AD по умолчанию использует Azure MFA. Если для многофакторной проверки подлинности используется служба федерации, службу Azure AD можно настроить так, чтобы она перенаправляла в службу федерации, когда требуется MFA. Для этого задайте для параметра `-SupportsMFA` значение `$true` в [PowerShell](https://docs.microsoft.com/powershell/module/msonline/set-msoldomainfederationsettings). Этот параметр действует для служб федеративной проверки подлинности, поддерживающих запрос на MFA, выданный Azure AD с помощью `wauth= http://schemas.microsoft.com/claims/multipleauthn`.

После того как пользователь выполнил вход в службу федеративной проверки подлинности, Azure AD обрабатывает другие требования политики, например требование соответствия устройства или утверждения приложения.
