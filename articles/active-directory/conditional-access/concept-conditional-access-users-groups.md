# <a name="conditional-access-users-and-groups"></a>Условный доступ: пользователи и группы

Политика условного доступа должна включать назначение пользователя в качестве одного из сигналов в процессе принятия решения. Пользователи могут быть включены или исключены из политики условного доступа. 

![Пользователь как сигнал в решениях, принятых Условным Доступом](./media/concept-conditional-access-users-groups/conditional-access-users-and-groups.png)

## <a name="include-users"></a>Включить пользователей

Этот список пользователей обычно включает в себя всех пользователей, на которые ориентирована организация в политике условного доступа. 

При создании политики условного доступа доступны следующие варианты.

- Отсутствуют
   - Пользователи не выбраны
- все пользователи
   - Все пользователи, которые существуют в каталоге, включая b2B гостей.
- Выбор элемента "Пользователи и группы"
   - Все гостевые и внешние пользователи (предварительный просмотр)
      - Этот выбор включает в себя любые B2B `user type` гостей `guest`и внешних пользователей, включая любого пользователя с атрибутом набор . Этот выбор также применяется к любому внешнему пользователю, вписавому в другую организацию, например поставщик облачных решений (CSP). 
   - Роли каталога (предварительный просмотр)
      - Позволяет администраторам выбирать определенные роли каталога Azure AD, используемые для определения назначения. Например, организации могут создать более ограничительную политику в отношении пользователей, которым назначена роль глобального администратора.
   - Пользователи и группы
      - Позволяет таргетинг на определенные наборы пользователей. Например, организации могут выбрать группу, содержащую всех сотрудников отдела управления персоналом при выборе приложения hr в качестве облачного приложения. Это может быть группа любого типа в Azure AD, включая динамические группы, назначенные группы безопасности и группы рассылки.

## <a name="exclude-users"></a>Исключить пользователей

Когда организации включают и исключают пользователя или группу, пользователь или группа исключаются из политики, поскольку действие, исключаемые, переопределяет включение в политику. Исключения обычно используются для экстренного доступа или взломных счетов. Более подробную информацию о учетных записях экстренного доступа и о том, почему они важны, можно найти в следующих статьях: 

* [Управление учетными записями для аварийного доступа в Azure AD](../users-groups-roles/directory-emergency-access.md)
* [Создание устойчивой стратегии управления доступом с помощью Azure Active Directory](../authentication/concept-resilient-controls.md)

При создании политики условного доступа можно исключить следующие варианты.

- Все гостевые и внешние пользователи (предварительный просмотр)
   - Этот выбор включает в себя любые B2B `user type` гостей `guest`и внешних пользователей, включая любого пользователя с атрибутом набор . Этот выбор также применяется к любому внешнему пользователю, вписавому в другую организацию, например поставщик облачных решений (CSP). 
- Роли каталога (предварительный просмотр)
   - Позволяет администраторам выбирать определенные роли каталога Azure AD, используемые для определения назначения. Например, организации могут создать более ограничительную политику в отношении пользователей, которым назначена роль глобального администратора.
- Пользователи и группы
   - Позволяет таргетинг на определенные наборы пользователей. Например, организации могут выбрать группу, содержащую всех сотрудников отдела управления персоналом при выборе приложения hr в качестве облачного приложения. Это может быть группа любого типа в Azure AD, включая динамические группы, назначенные группы безопасности и группы рассылки.

### <a name="preventing-administrator-lockout"></a>Предотвращение блокировки администратора

Чтобы администратор не заблокировал себя в каталоге при создании политики, применяемой ко **всем пользователям** и **всем приложениям,** он увидит следующее предупреждение.

> Не запирайся! Мы рекомендуем применять политику для небольшого набора пользователей, чтобы сначала проверить, что она ведет себя так, как ожидалось. Мы также рекомендуем исключить из этой политики по крайней мере одного администратора. Это гарантирует, что у вас есть доступ и может обновить политику, если требуется изменение. Пожалуйста, просмотрите пострадавших пользователей и приложения.

По умолчанию политика предоставляет возможность исключить текущего пользователя из политики, но этот по умолчанию может быть отменен администратором, как показано на следующем изображении. 

![Предупреждение, не запирайся!](./media/concept-conditional-access-users-groups/conditional-access-users-and-groups-lockout-warning.png)
