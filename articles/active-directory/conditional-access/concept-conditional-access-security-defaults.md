# <a name="what-are-security-defaults"></a>Что такое параметры безопасности по умолчанию?

Управление безопасностью может оказаться сложной задачей, когда распространенные атаки, связанные с удостоверениями, такие как распыление паролей, воспроизведение и фишинг, становятся все более популярными. Создание более простого способа защиты Организации от таких распространенных атак является целью обеспечения безопасности по умолчанию в Azure Active Directory (AD). Параметры безопасности по умолчанию упрощают защиту и позволяют защитить организацию от распространенных атак. Параметры безопасности по умолчанию содержат предварительно настроенные настройки безопасности для этих распространенных атак. Корпорация Майкрософт делает параметры безопасности по умолчанию доступными для всех. Цель состоит в том, чтобы гарантировать, что все организации имеют базовый уровень безопасности без дополнительной оплаты.

![Снимок экрана нового UX по умолчанию для интерфейса безопасности](./media/concept-conditional-access-security-defaults/security-defaults-azure-ad-portal.png)
 
В клиенте будут включены следующие конфигурации безопасности. 

## <a name="unified-mfa-registration"></a>Единая регистрация MFA

Все пользователи в клиенте будут обязательно зарегистрированы для многофакторной идентификации Azure (MFA). Для регистрации Azure MFA с помощью Microsoft Authenticator приложения у пользователей будет 14 дней. По истечении 14 дней пользователь не сможет войти в систему, пока не будет выполнена регистрация MFA.

Мы понимаем, что некоторые пользователи могут выйти из Office и (или) не будут входить в систему в течение 14 дней сразу после включения параметров безопасности по умолчанию. Чтобы предоставить каждому пользователю достаточно времени для регистрации MFA, 14-дневный период уникален для каждого пользователя. 14-дневный период пользователя начинается после первого успешного интерактивного входа после включения параметров безопасности по умолчанию.

## <a name="mfa-enforcement"></a>Принудительное применение MFA

### <a name="protecting-administrators"></a>Защита администраторов

Пользователи с доступом к привилегированным учетным записям имеют повышенный доступ к вашей среде. Из-за возможностей этих учетных записей при работе с ними следует соблюдать осторожность. Наиболее распространенный способ повышения защиты привилегированных учетных записей — использовать более строгую форму проверки учетной записи при входе в систему. В Azure Active Directory можно получить более надежную проверку учетной записи, запросив многофакторную проверку подлинности.

После завершения регистрации MFA необходимо выполнить следующие девять ролей администратора Azure AD, чтобы выполнять MFA каждый раз, когда они входят в систему.

- Глобальный администратор.
- администратор SharePoint;
- администратор Exchange;
- Администратор условного доступа
- Администратор безопасности
- Администратор службы технической поддержки и администратор паролей
- Администратор выставления счетов
- Администратор пользователей
- Администратор проверки подлинности

### <a name="protecting-all-users"></a>Защита всех пользователей

Мы будем думать, что учетные записи администратора являются единственными учетными записями, требующими защиты с помощью многофакторной проверки подлинности (MFA). Администраторы имеют широкие возможности доступа к конфиденциальным сведениям и могут вносить изменения в параметры, относящиеся к подписке. Однако неплохая субъекты, как правило, предназначены для конечных пользователей. После получения доступа эти недопустимые субъекты могут запросить доступ к привилегированным сведениям от имени оригинального владельца учетной записи или скачать весь каталог, чтобы выполнить фишинговую атаку для всей Организации. Одним из распространенных способов улучшения защиты для всех пользователей является требование более надежной проверки учетной записи, например многофакторной проверки подлинности (MFA) для всех. После завершения регистрации MFA пользователи будут получать запрос на MFA при необходимости.

### <a name="blocking-legacy-authentication"></a>Блокирование устаревшей проверки подлинности

Чтобы предоставить пользователям удобный доступ к облачным приложениям, Azure Active Directory (AAD) поддерживает широкий набор протоколов проверки подлинности, часть из которых уже устарела. Устаревшая проверка подлинности — это термин, который ссылается на запрос проверки подлинности, сделанный:

- Более старые клиенты Office, не использующие современную проверку подлинности (например, Office 2010 Client)
- Любой клиент, использующий устаревшие почтовые протоколы, такие как IMAP/SMTP/POP3

В настоящее время большинство возможных попыток входа в систему поступают от устаревшей проверки подлинности. Устаревшая проверка подлинности не поддерживает многофакторную проверку подлинности (MFA). Даже если в вашем каталоге включена политика MFA, Недопустимый субъект может пройти проверку подлинности с помощью устаревшего протокола и обойти MFA. После включения параметров безопасности по умолчанию в клиенте все запросы проверки подлинности, сделанные устаревшим протоколом, будут заблокированы. Параметры безопасности по умолчанию не блокируют Exchange ActiveSync.

### <a name="protecting-privileged-actions"></a>Защита привилегированных действий

Организации используют различные службы Azure, управляемые через API Azure Resource Manager, включая:

- портала Azure 
- Azure PowerShell 
- Azure CLI

Использование Azure Resource Manager для управления службами — это действие с высоким уровнем привилегий. Azure Resource Manager может изменять конфигурации на уровне клиента, такие как параметры службы и выставление счетов по подписке. Однофакторная проверка подлинности уязвима для различных атак, таких как фишинг и распыление паролей. Поэтому важно проверить подлинность пользователей, желающих получить доступ к Azure Resource Manager и конфигурациям обновлений, путем требования многофакторной идентификации перед предоставлением доступа.

После включения в клиенте параметров безопасности по умолчанию любой пользователь, обращающийся к портал Azure, Azure PowerShell или Azure CLI, должен будет выполнить многофакторную проверку подлинности. Эта политика применяется ко всем пользователям, обращающимся к Azure Resource Manager, независимо от того, являетесь ли они администратором или пользователем. Если пользователь не зарегистрирован для MFA, ему потребуется зарегистрироваться с помощью Microsoft Authenticator приложения, чтобы продолжать работу. Не будет предоставляться 14-дневный период регистрации MFA.

## <a name="deployment-considerations"></a>Рекомендации по развертыванию

Ниже приведены некоторые дополнительные соображения, связанные с развертыванием параметров безопасности для клиента.

### <a name="legacy-protocols"></a>Устаревшие протоколы

Устаревшие протоколы проверки подлинности (IMAP, SMTP, POP3 и т. д.) используются почтовыми клиентами для выполнения запросов проверки подлинности. Эти протоколы не поддерживают MFA. Большая часть учетных записей, обнаруженных корпорацией Майкрософт, вызвана неправильными субъектами, которые выполняют атаки на устаревшие протоколы, пытающиеся обходить MFA. Чтобы убедиться, что MFA требуется при входе в учетную запись администратора, и неудачные субъекты не могут обходить MFA, параметры безопасности по умолчанию блокируют все запросы проверки подлинности, произведенные учетным записям администратора из устаревших

> [!WARNING]
> Перед включением этого параметра убедитесь, что администраторы не используют устаревшие протоколы проверки подлинности. Дополнительные сведения см. в статье [как перейти от устаревшей проверки подлинности](concept-conditional-access-block-legacy-authentication.md).

### <a name="conditional-access"></a>Условный доступ

Условный доступ можно использовать для настройки политик, обеспечивающих то же поведение, что и параметры безопасности по умолчанию. Если вы используете условный доступ и в вашей среде включены политики условного доступа, параметры безопасности по умолчанию будут недоступны. Если у вас есть лицензия, которая предоставляет условный доступ, но в вашей среде не включены политики условного доступа, вы можете использовать параметры безопасности по умолчанию, пока не будут включены политики ЦС.

## <a name="enabling-security-defaults"></a>Включение параметров безопасности по умолчанию

Чтобы включить параметры безопасности по умолчанию в каталоге, выполните следующие действия.

1. Войдите в [портал Azure](https://portal.azure.com) As администратор безопасности, администратор условного доступа или глобальный администратор.
1. Перейдите к **Azure Active Directory**  @ no__t-2 **свойства**
1. Выберите **Управление параметрами безопасности по умолчанию**
1. Установите переключатель **включить параметры безопасности по умолчанию** в положение **Да**.
1. Нажмите кнопку Сохранить.
