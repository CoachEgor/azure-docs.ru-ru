# <a name="conditional-access-grant"></a>Условный доступ: Грант

В рамках политики условного доступа администратор может использовать элементы управления доступом для предоставления или блокировки доступа к ресурсам.

![Политика условного доступа с контролем грантов, требующим многофакторной аутентификации](./media/concept-conditional-access-grant/conditional-access-grant.png)

## <a name="block-access"></a>Заблокировать доступ

Блок учитывает все назначения и предотвращает доступ в зависимости от конфигурации политики условного доступа.

Блок является мощным контролем, который должен быть орудовал с соответствующими знаниями. Администраторы должны использовать [режим report-only](concept-conditional-access-report-only.md) для тестирования перед включением.

## <a name="grant-access"></a>Предоставление доступа

Администраторы могут выбрать применение одного или нескольких элементов управления при предоставлении доступа. Эти элементы управления включают в себя следующие варианты: 

- [Требуется многофакторная аутентификация (Многофакторная аутентификация Azure)](../authentication/concept-mfa-howitworks.md)
- [Требуется, чтобы устройство было помечено как совместимое (Microsoft Intune)](/intune/protect/device-compliance-get-started)
- [Требуется гибридное устройство Azure AD](../devices/concept-azure-ad-join-hybrid.md)
- [Требование утвержденного клиентского приложения](app-based-conditional-access.md)
- [Требование политики защиты приложений](app-protection-based-conditional-access.md)

Когда администраторы решают объединить эти параметры, они могут выбрать следующие методы:

- Требуются все выбранные элементы управления (управление **и** управление)
- Требуется один из выбранных элементов управления (управление **или** управление)

По умолчанию Условный доступ требует всех выбранных элементов управления.

### <a name="require-multi-factor-authentication"></a>Требование многофакторной идентификации

Выбор этого флажка потребует от пользователей выполнения Azure Multi-Фактораута. Более подробную информацию о развертывании многофакторной аутентификации Azure можно найти в статье [Планирование многофакторного развертывания Azure Multi-Factor Authentication.](../authentication/howto-mfa-getstarted.md)

### <a name="require-device-to-be-marked-as-compliant"></a>Требовать, чтобы устройство было отмечено как соответствующее

Организации, развернувшие Microsoft Intune, могут использовать информацию, возвращенную с их устройств, для идентификации устройств, отвечающих определенным требованиям соответствия требованиям. Эта информация о соответствии политике перенаправляется из Intune в Azure AD, где условный доступ может принимать решения о предоставлении или блокировании доступа к ресурсам. Для получения дополнительной информации о политиках соответствия с мнитесь в статье [«Набор правил на устройствах», позволяющий получить доступ к ресурсам в организации с помощью Intune.](/intune/protect/device-compliance-get-started)

Устройство может быть помечено как совместимое Intune (для любого устройства ОС) или сторонней системой MDM для устройств Windows 10. Jamf pro является единственной поддерживаемой сторонней системой MDM. Более подробную информацию об интеграции можно найти в статье [«Интеграция Jamf Pro с Intune для соответствия».](/intune/protect/conditional-access-integrate-jamf)

Устройства должны быть зарегистрированы в Azure AD, прежде чем они могут быть помечены как совместимые. Более подробную информацию о регистрации устройства можно найти в статье, [Что такое личность устройства](../devices/overview.md).

### <a name="require-hybrid-azure-ad-joined-device"></a>Требуется гибридное устройство Azure AD

Организации могут использовать итог устройства как часть своей политики условного доступа. Организации могут требовать, чтобы устройства были гибридными Azure AD, объединенными с помощью этого флажка. Для получения дополнительной информации об идентификационных данных устройства смотрите статью [Что такое идентификация устройства?.](../devices/overview.md)

### <a name="require-approved-client-app"></a>Требование утвержденного клиентского приложения

Организации могут потребовать, чтобы попытка доступа к выбранным облачным приложениям была сделана из утвержденного клиентского приложения. Эти одобренные клиентские приложения поддерживают [политики защиты приложений Intune,](/intune/app-protection-policy) независимо от любого решения для управления мобильными устройствами (MDM).

Для того, чтобы использовать этот контроль гранта, Условный доступ требует, чтобы устройство было зарегистрировано в Active Directory Azure, что требует использования приложения брокера. В качестве приложения посредника могут выступать решения Microsoft Authenticator для iOS или корпоративный портал Майкрософт для устройств Android. Если приложение брокера не установлено на устройстве, когда пользователь пытается проверить подлинность, пользователь перенаправляется в магазин приложений для установки приложения брокера.

Эта настройка применяется к следующим приложениям для iOS и Android:

- Microsoft Azure Information Protection.
- Microsoft Bookings
- Кортана от Microsoft
- Microsoft Dynamics 365
- Microsoft Edge
- Microsoft Excel
- Microsoft Flow
- Microsoft Intune Managed Browser
- Microsoft Invoicing.
- Microsoft Kaizala
- Microsoft Launcher;
- Microsoft Office
- Концентратор офиса Майкрософт
- Microsoft OneDrive
- Microsoft OneNote;
- Microsoft Outlook
- Планировщик (Майкрософт);
- Microsoft PowerApps
- Microsoft Power BI
- Microsoft PowerPoint
- Microsoft SharePoint
- Microsoft Skype для бизнеса;
- Microsoft StaffHub
- Microsoft Stream;
- Microsoft Teams
- Microsoft To-Do
- Microsoft Visio;
- Microsoft Word
- Microsoft Yammer
- Доска Майкрософт

**Замечания**

- Утвержденные клиентских приложения поддерживают функцию управления мобильными приложениями Intune.
- Функция **Требовать утвержденное клиентское приложение**:
   - поддерживает только iOS и Android для условия платформы устройства;
   - Для регистрации устройства требуется приложение брокера. На iOS, брокер приложение Microsoft Authenticator и на Android, это Intune компании портал приложение.
- Условный доступ не может считать Microsoft Edge в режиме InPrivate утвержденным клиентским приложением.

Смотрите статью [«Как: Требуют одобренных клиентских приложений для доступа к облачным приложениям с условным доступом](app-based-conditional-access.md) для примеров конфигурации».

### <a name="require-app-protection-policy"></a>Требование политики защиты приложений

В политике условного доступа необходимо, чтобы [политика защиты приложений Intune](/intune/app-protection-policy) присутствовала в клиентском приложении, прежде чем доступ к выбранным облачным приложениям будет доступен. 

Для того, чтобы использовать этот контроль гранта, Условный доступ требует, чтобы устройство было зарегистрировано в Active Directory Azure, что требует использования приложения брокера. В качестве приложения посредника могут выступать решения Microsoft Authenticator для iOS или корпоративный портал Майкрософт для устройств Android. Если приложение брокера не установлено на устройстве, когда пользователь пытается проверить подлинность, пользователь перенаправляется в магазин приложений для установки приложения брокера.

Этот параметр применяется к следующим клиентским приложениям.

- Кортана от Microsoft
- Microsoft OneDrive
- Microsoft Outlook
- Планировщик (Майкрософт);

**Замечания**

- Приложения для политики защиты приложений поддерживают функцию управления мобильными приложениями Intune с защитой политики.
- Требования **политики защиты приложений требуют:**
    - поддерживает только iOS и Android для условия платформы устройства;
    - Для регистрации устройства требуется приложение брокера. На iOS, брокер приложение Microsoft Authenticator и на Android, это Intune компании портал приложение.

Смотрите статью [«Как: требуйте политики защиты приложений и утвержденное клиентское приложение для доступа к облачным приложениям с условным доступом](app-protection-based-conditional-access.md) для примеров конфигурации.
