# <a name="deploy-a-secure-azure-managed-workstation"></a>Развертывание безопасной рабочей станции, управляемой Azure

Теперь, когда вы [понимаете безопасные рабочие станции,](concept-azure-managed-workstation.md)пришло время начать процесс развертывания. С помощью этого руководства вы используете определенные профили для создания рабочей станции, которая является более безопасной с самого начала.

![Развертывание безопасной рабочей станции](./media/howto-azure-managed-workstation/deploying-secure-workstations.png)

Выберите профиль перед развертыванием решения. Можно использовать несколько профилей одновременно в развертывании и назначать их тегами или группами.

> [!NOTE]
> Примените любой из профилей по мере необходимости ваших требований. Вы можете перейти на другой профиль, назначив его в Microsoft Intune.

> [!NOTE]
> В безопасных **рабочих станциях устройства** наведения будут назначены с профилями и политиками. Пользователи не будут применять к ним политики напрямую, что позволяет осуществлять совместное использование устройств (общие устройства). Если безопасная рабочая станция не используется в развертывании или необходимы отдельные политики пользователя, назначение профилей политики пользователя может быть назначено пользователю и устройству.

## <a name="azure-active-directory-configuration"></a>Конфигурация активного каталога Azure

Активный каталог Azure (Azure AD) управляет пользователями, группами и устройствами для рабочих станций администратора. Включить службы идентификации и функции с [учетной записью администратора.](../users-groups-roles/directory-assign-admin-roles.md)

При создании защищенной учетной записи администратора рабочей станции вы подвергаете учетную запись текущей рабочей станции. Убедитесь, что вы используете известное безопасное устройство для этой начальной конфигурации и всей глобальной конфигурации. Чтобы уменьшить воздействие атаки для первого опыта, подумайте о [том, чтобы следовать рекомендациям по предотвращению заражения вредоносными программами.](/windows/security/threat-protection/intelligence/prevent-malware-infection)

Требуется многофакторная аутентификация, по крайней мере для администраторов. Смотрите руководство [по внедрению облачную систему Deploy.](../authentication/howto-mfa-getstarted.md)

### <a name="azure-ad-users-and-groups"></a>Пользователи и группы Azure AD

1. С портала Azure просматривайте > **пользователи** >  **Active Directory Azure****New user.**
1. Создайте администратора устройства, выяснив шаги в [учебнике по созданию пользователя.](/Intune/quickstart-create-user)
1. Введите:

   * **Название** - Администратор безопасных рабочих станций
   * **Имя пользователя** - `secure-ws-admin@identityitpro.com`
   * **Роль** - администратора**каталога** ограниченного администратора и выберите роль **администратора Intune.**

1. Выберите **Создать**.

Далее вы создаете две группы: пользователей рабочих станций и устройства рабочих станций.

С портала Azure просмотрите новую**группу**Active > Directory**Groups** >  **Azure.**

1. Для группы пользователей рабочих станций может потребоваться настроить [групповое лицензирование](../users-groups-roles/licensing-groups-assign.md) для автоматизации предоставления лицензий пользователям.
1. Для группы пользователей рабочей станции введите:

   * **Тип группы** - Безопасность
   * **Название группы** - Безопасные пользователи рабочих станций
   * **Тип членства** - Назначено

1. Добавьте пользователя администратора безопасной рабочей станции:`secure-ws-admin@identityitpro.com`
1. Вы можете добавить любых других пользователей, которые будут управлять безопасными рабочими станциями.
1. Выберите **Создать**.
1. Для группы рабочих станций введите:

   * **Тип группы** - Безопасность
   * **Название группы** - Безопасные рабочие станции
   * **Тип членства** - Назначено

1. Выберите **Создать**.

### <a name="azure-ad-device-configuration"></a>Конфигурация aD-устройства Azure

#### <a name="specify-who-can-join-devices-to-azure-ad"></a>Укажите, кто может присоединиться к устройствам Azure AD

Настроили настройку устройств в Active Directory, чтобы позволить вашей группе административной безопасности присоединение устройств к вашему домену. Чтобы настроить этот параметр с портала Azure:

1. Перейдите к**настройкам устройств активных****устройств** >  **каталога Azure** > Active Directory Device.
1. Выбранный под **пользователями** **может присоединиться к устройствам Azure AD,** а затем выбрать группу "Безопасные пользователи рабочей станции".

#### <a name="removal-of-local-admin-rights"></a>Удаление прав местного админа

Этот метод требует, чтобы пользователи рабочих станций VIP, DevOps и безопасных уровней не имели прав администратора на своих машинах. Чтобы настроить этот параметр с портала Azure:

1. Перейдите к**настройкам устройств активных****устройств** >  **каталога Azure** > Active Directory Device.
1. Выберите **Нет** под **дополнительными локальными администраторами на устройствах, связанных с Azure AD.**

#### <a name="require-multi-factor-authentication-to-join-devices"></a>Требуется многофакторная аутентификация для присоединения устройств

Для дальнейшего укрепления процесса присоединения устройств к Azure AD:

1. Перейдите к**настройкам устройств активных****устройств** >  **каталога Azure** > Active Directory Device.
1. Выберите **Да** под **Требование Multi-Фактор Аут для присоединения устройств.**
1. Нажмите кнопку **Сохранить**.

#### <a name="configure-mobile-device-management"></a>Настройка управления мобильными устройствами

На портале Azure выполните следующие действия.

1. Просмотрите информацию о мобильности **active Directory** > Azure **(MDM и MAM)** > **Microsoft Intune.**
1. Измените настройку **параметра пользователя MDM** на **all.**
1. Нажмите кнопку **Сохранить**.

Эти шаги позволяют управлять любым устройством с помощью Intune. Для получения дополнительной информации, [см. Intune Быстрый запуск: Настройка автоматической регистрации для устройств Windows 10](/Intune/quickstart-setup-auto-enrollment). На будущем вы создаете политики конфигурации и соответствия требованиям Intune.

#### <a name="azure-ad-conditional-access"></a>Условный доступ AD Azure

Условный доступ Azure AD может помочь ограничить привилегированные административные задачи совместимыми устройствами. Предопределенные члены группы **пользователей безопасных рабочих станций** обязаны выполнять многофакторную аутентификацию при входе в облачные приложения. Наилучшей практикой является исключение учетных записей экстренного доступа из политики. Для получения дополнительной информации смотрите [Управление учетными записями экстренного доступа в Azure AD](../users-groups-roles/directory-emergency-access.md).

## <a name="intune-configuration"></a>Конфигурация неустое

### <a name="configure-enrollment-status"></a>Настройка статуса регистрации

Важно убедиться, что ваша безопасная рабочая станция является надежным чистым устройством. При покупке новых устройств, вы можете настаивать на том, что они завод набор [для Windows 10 Pro в режиме S](/Windows/deployment/Windows-10-pro-in-s-mode), который ограничивает уязвимости во время управления цепочкой поставок. После получения устройства от поставщика, вы можете использовать Autopilot, чтобы изменить его от режима S. В следующем руководстве содержится подробная информация о применении процесса трансформации.

Чтобы убедиться, что устройства полностью настроены перед использованием, Intune предоставляет средства для **блокировки использования устройства до тех пор, пока не будут установлены все приложения и профили.**

С **портала Azure**:

1. **Enrollment Status Page** > **Default** > Перейдите к**настройкам****регистрации** > страницы регистрации на**странице** > регистрации **Устройств Microsoft Intune** > Device.
1. Установить **Показать завершение установки профиля приложения** **да**.
1. Установите **блок устройства использования до тех пор, пока все приложения и профили установлены** на **Да**.

### <a name="create-an-autopilot-deployment-profile"></a>Создание профиля развертывания Autopilot

После создания группы устройств необходимо создать профиль развертывания для настройки устройств автопилота.

В Intune на портале Azure:

1. Выберите **профили** > **развертывания регистрации** > **Deployment Profiles** > устройств в**Windows.**
1. Введите:

   * Имя - **Безопасный профиль развертывания рабочей станции.**
   * Описание - **Развертывание безопасных рабочих станций.**
   * Для параметра **Convert all targeted devices to Autopilot** (Преобразовать все целевые устройства в Autopilot) выберите вариант **Да**. Этот параметр гарантирует, что все устройства из списка будут зарегистрированы в службе развертывания Autopilot. Регистрация завершится в течение 48 часов.

1. Нажмите кнопку **Далее**.

   * Для **режима развертывания**выберите **саморазвертывание (Предварительный просмотр).** Устройства с этим профилем связаны с пользователем, который регистрирует устройство. Для регистрации устройства нужны учетные данные пользователя. Важно отметить, что развертывание устройства в режиме **самостоятельной развертывания** позволит вам развернуть ноутбуки в общей модели. Назначение пользователя не произойдет до тех пор, пока устройство не будет назначено пользователю в первый раз. В результате любые пользовательские политики, такие как BitLocker, не будут включены до завершения назначения пользователя. Для получения дополнительной [информации](/intune/device-profile-assign)о том, как войти на защищенное устройство, см.
   * **Присоединиться к Azure AD в качестве** коробки должно показать **Azure AD присоединился** и быть серым.
   * Выберите свой язык (регион), **стандарт**типа учетной записи пользователя. 

1. Нажмите кнопку **Далее**.

   * Выберите тег области, если вы предварительно настроили его.

1. Нажмите кнопку **Далее**.
1. Выберите **Назначения** > **Assign to** > **для выбранных групп.** В **группы Select, чтобы включить,** выберите **Безопасные рабочие станции.**
1. Нажмите кнопку **Далее**.
1. Нажмите кнопку **Создать**, чтобы создать профиль. Теперь профиль развертывания Autopilot можно назначать устройствам.

Регистрация устройств в Autopilot обеспечивает другой пользовательский опыт в зависимости от типа и роли устройства. В нашем примере развертывания мы иллюстрируем модель, в которой защищенные устройства развернуты навалом и могут быть общими, но при первом использовании устройство назначается пользователю. Для получения дополнительной информации, [см.](/intune/device-enrollment)

### <a name="configure-windows-update"></a>Настройка обновления Windows

Ведение Windows 10 в актуальном состоянии является одним из самых важных вещей, которые вы можете сделать. Чтобы поддерживать Windows в безопасном состоянии, развертывайте кольцо обновления для управления темпами, когда обновления применяются к рабочим станциям. 

В этом руководстве рекомендуется создать новое кольцо обновления и изменить следующие настройки по умолчанию:

На портале Azure выполните следующие действия.

1. Перейти к **Microsoft Intune** > **Программное обеспечение обновления** > **Windows 10 Обновление Кольца**.
1. Введите:

   * Имя - **Обновления рабочей станции, управляемые Azure**
   * Канал обслуживания - **Инсайдерская Windows - Быстрый**
   * Отсрочка обновления качества (дни) - **3**
   * Период отсрочки обновления (дней) - **3**
   * Автоматическое обновление - **Автоматическая установка и перезагрузка без контроля над конечными пользователями**
   * Блокировка пользователя от приостановки обновлений Windows - **Блокировка**
   * Требуется разрешение пользователя для перезагрузки вне рабочего времени - **Требуется**
   * Разрешить пользователю перезагрузку (включенную перезагрузку) - **Требуется**
   * Переход пользователей к включенному перезапуску после автоматического перезапуска (дней) - **3**
   * Отложить занимается перезагрузить напоминание (дней) - **3**
   * Установить крайний срок для ожидающих перезагрузки (дней) - **3**

1. Выберите **Создать**.
1. На вкладке **«Назначения»** добавьте группу **безопасных рабочих станций.**

Для получения дополнительной информации о [Policy CSP - Update](/windows/client-management/mdm/policy-csp-update)политиках обновления Windows см.

### <a name="windows-defender-atp-intune-integration"></a>Windows Defender ATP Intune интеграция

Windows Defender ATP и Microsoft Intune работают вместе, чтобы предотвратить нарушения безопасности. Они также могут ограничить воздействие нарушений. Возможности СПС обеспечивают обнаружение угроз в режиме реального времени, а также позволяют проводить обширный аудит и регистрацию устройств конечной точки.

Чтобы настроить интеграцию Windows Defender ATP и Intune, перейдите на портал Azure.

1. Просмотр **Microsoft Intune** > **Устройства соответствия** > **Windows Defender ATP**.
1. В шаге 1 под **настройкой Windows Defender ATP**, выберите **Подключите Windows Defender ATP к Microsoft Intune в Центре безопасности защиты Windows Defender.**
1. В центре безопасности Защитника Windows:

   1. Выберите **функции Настройки** > **Расширенные.**
   1. Для **подключения Microsoft Intune**выберите **On**.
   1. Выберите **Сохранить параметры**.

1. После установления соединения вернитесь в Intune и выберите **Refresh** в верхней части.
1. Установите параметр **Подключение устройств Windows версии 10.0.15063 и более поздней к ATP в Защитнике Windows** в значение **Вкл**.
1. Нажмите кнопку **Сохранить**.

Дополнительные сведения см. в статье [Advanced Threat Protection в Защитнике Windows](/Windows/security/threat-protection/windows-defender-atp/windows-defender-advanced-threat-protection).

Таким образом, вы можете определить дополнительное затвердевание, которое может потребоваться, как указано в [документации SECCON.](/windows/security/threat-protection/windows-security-configuration-framework/windows-security-configuration-framework)

## <a name="additional-configurations-and-hardening-to-consider"></a>Дополнительные конфигурации и затвердевания для рассмотрения

Следуя инструкциям, вы развернули безопасную рабочую станцию. Тем не менее, вы также должны рассмотреть дополнительные элементы управления. Пример:

* ограничить доступ к альтернативным браузерам
* позволяют исходящие HTTP
* блокировать избранные веб-сайты
* набор разрешений для запуска пользовательских скриптов PowerShell

### <a name="set-rules-in-the-firewall-configuration-service-provider-csp"></a>Установите правила в поставщике услуг по настройке брандмауэра (CSP)

Вы можете внести дополнительные изменения в управление как входящими, так и исходящими правилами по мере необходимости для разрешенных и заблокированных конечных точек. По мере того как вы продолжаете затвердевать безопасную рабочую станцию, вы можете ослабить ограничение, которое лишает себя входящего и исходящего трафика. Можно добавить разрешенные исходящие сайты, чтобы включить общие и доверенные веб-сайты. Для получения дополнительной [Firewall configuration service](/Windows/client-management/mdm/firewall-csp)информации см.

Ограничительное управление трафиком URL включает в себя:

* Отказать всем входящим трафиком - Установить в скрипте профиля защищенной рабочей станции.
* Запретить все исходящие, за исключением отдельных служб Azure и Microsoft, включая Azure Cloud Shell и возможность сбросить пароль Azure.

```
[HKEY_CURRENT_USER\Software\Microsoft\Windows\CurrentVersion\Internet Settings]
"ProxyEnable"=dword:00000001
"ProxyServer"="127.0.0.1:80"
"ProxyOverride"="*.azure.com;*.azure.net;*.microsoft.com;*.windowsupdate.com;*.microsoftonline.com;*.microsoftonline.cn;*.windows.net;*.windowsazure.com;*.windowsazure.cn;*.azure.cn;*.loganalytics.io;*.applicationinsights.io;*.vsassets.io;*.azure-automation.net;*.visualstudio.com,portal.office.com;*.aspnetcdn.com;*.sharepointonline.com;*.msecnd.net;*.msocdn.com;*.webtrends.com"
"AutoDetect"=dword:00000000
```

Если вы хотите обеспечить более детализацию ваших правил блокировки, вы можете обратиться к проекту Spamhaus, который поддерживает [список блоков доменов (DBL):](https://www.spamhaus.org/dbl/)хороший ресурс для использования в качестве расширенного набора правил для реализации для блокировок сайтов.

### <a name="manage-local-applications"></a>Управление локальными приложениями

Безопасная рабочая станция перемещается в действительно закаленном состоянии при удалении локальных приложений, включая приложения для повышения производительности. Здесь вы добавляете Chrome в качестве браузера по умолчанию и используете Intune, чтобы ограничить возможность пользователя модифицировать браузер и его плагины.

#### <a name="deploy-applications-using-intune"></a>Развертывание приложений с помощью Intune

В некоторых ситуациях приложения, такие как браузер Google Chrome, необходимы на защищенной рабочей станции. В следующем примере содержатся инструкции по установке Chrome на устройства в группе безопасности **Secure Workstations.**

1. Скачать оффлайн установщик [Chrome расслоение для Windows 64-битный](https://cloud.google.com/chrome-enterprise/browser/download/).
1. Извлеките файлы и обратите `GoogleChromeStandaloneEnterprise64.msi` внимание на местоположение файла.
1. На **портале Azure** просматривайте**приложения** >  **Microsoft Intune** > **Client Apps.** > **Add**
1. Под **типом приложения**выберите **Line-of-business.**
1. Под **файлом пакета** `GoogleChromeStandaloneEnterprise64.msi` App выберите файл из извлеченного местоположения и выберите **OK.**
1. В соответствии с **информацией App**, предоставить описание и издателя. Нажмите кнопку **ОК**.
1. Нажмите кнопку **Добавить**.
1. На вкладке **«Назначения»** выберите **Доступно для зарегистрированных устройств** под **типом «Назначение».**
1. Под **включенными группами**добавьте группу **безопасных рабочих станций.**
1. Нажмите кнопку **ОК**, а затем нажмите кнопку **Сохранить**.

Для получения дополнительной информации о настройке настроек Chrome [см.](https://support.google.com/chrome/a/answer/9102677)

#### <a name="configuring-the-company-portal-for-custom-apps"></a>Настройка портала компании для пользовательских приложений

В защищенном режиме установка приложения ограничена порталом компании Intune. Однако для установки портала требуется доступ к Microsoft Store. В защищенном решении вы можете сделать портал компании доступным для всех устройств в автономном режиме.

Управляемая Intune копия [портала компании](/Intune/store-apps-company-portal-app) предоставляет вам по требованию доступ к дополнительным инструментам, которые можно подтолкнуть к пользователям защищенных рабочих станций.

Возможно, потребуется установить 32-разрядные приложения С Windows или другие приложения, развертывание которых требует специальной подготовки. В таких случаях [инструмент подготовки к содержимому Microsoft win32](https://github.com/Microsoft/Microsoft-Win32-Content-Prep-Tool) может предоставить готовый к использованию файл формата `.intunewin` для установки.

### <a name="conditional-access-only-allowing-secured-workstation-ability-to-access-azure-portal"></a>Условный доступ только позволяет защищенной рабочей станции получить доступ к порталу Azure

Azure AD предлагает возможность управлять и ограничивать, кто и что может получить доступ к порталу управления облаками Azure. Включение [условия доступа](../conditional-access/overview.md) гарантирует, что только ваша безопасная рабочая станция может управлять или изменять ресурсы. Важно, чтобы при развертывании этой функции вы учитывали, если функциональность [экстренного доступа](../users-groups-roles/directory-emergency-access.md) может или должна использоваться только в крайних случаях, а учетная запись управляется через политику.

> [!NOTE]
> Вам нужно будет создать группу пользователей и включить в нештатный пользователь, который может обойти политику условного доступа. Например, у нас есть группа безопасности под названием **Emergency BreakGlass**

1. Ознакомьтесь с **порталом** > Azure**Microsoft Intune** > **Conditional Access - Политика** > **Новая политика**.
1. Укажите **имя** для политики.
1. Выберите **пользователей и групп,** > **выберите пользователей и групп** 
1. Выберите **Включить** > **роли каталога** > Выберите роли > Глобального администратора, Привилегированном администраторе ролей, привилегированном администраторе аутентификации, администраторе безопасности, администраторе соответствия, администраторе условного доступа, администраторе приложений, администраторе облачных приложений, администраторе службы Intune
1. Выберите **Исключить** > Выберите **пользователей и группы** > **Выберите Избранные пользователи,** > Выберите группу **аварийных BreakGlass.**
1. Выберите **облачные приложения или действия** > Выберите **все облачные приложения**
1. Выберите **условия** > выберите **платформы устройств** > Выберите настройку **Да** > Выберите **платформы Выберите устройство** Выберите **Windows**
1. Выберите **элементы управления доступом** > Выберите **доступ к гранту** **Да** > Выберите **Устройство требование, которое будет помечено как совместимое.** 
1. Выберите политику включения > **в** **программу**
 
Этот набор мер политик гарантирует, что администраторы должны использовать совместимое устройство Windows, которое устанавливается Intune, и WDATP. 

Организациям следует также рассмотреть возможность блокировки устаревших протоколов проверки подлинности в своих средах. Существует несколько способов выполнения этой задачи, для получения дополнительной информации о блокировке устаревших протоколов проверки подлинности смотрите статью [«Как: блокировать устаревшую аутентификацию в Azure AD с условным доступом](../conditional-access/block-legacy-authentication.md)».

## <a name="using-sentinel-and-windows-defender-atp-to-monitor-and-respond-to-security-incidents"></a>Использование СПС Sentinel и Windows Defender для мониторинга инцидентов, связанных с безопасностью, и реагирования на них

Мониторинг безопасного развертывания рабочих станций может быть осуществлен путем включения «Sentinel» и использования [управления угрозами и уязвимостью](/windows/security/threat-protection/microsoft-defender-atp/next-gen-threat-and-vuln-mgt) Руководство не обеспечит исчерпывающую охоту на угрозы, но обеспечит хорошие усилия здравого смысла для мониторинга и реагирования на потенциальные инциденты безопасности.

Мы будем использовать **Azure Sentinel** для: 

* Сбор данных с портала Intune, Azure и Azure AD для мониторинга пользователей и устройств
* Помощь В расследовании подозрительной активности пользователя и устройства
* И диск возможность изучить расследования безопасности с помощью WDATP

Мониторинг Sentinel требует настройки разъемов для источников данных, таких как Azure AD.

1. На **портале Azure**перейдите на **Azure Sentinel (Предварительный просмотр)** > Выберите **Добавить**
1. В **«Выбрать рабочее пространство для добавления в Azure Sentinel** Выберите **Создать новое рабочее пространство»**
1. Введите:
   * **Рабочее пространство для аналитики журналов** - "Мониторинг безопасных рабочих станций"
   * **Подписка** - Выберите активную подписку
   * **Группа ресурсов** - выберите «Создать новую» > Безопасную рабочую станцию RG > **Ok**
   * **Местоположение** - Выберите место, которое географически лучше всего подходит для развертывания
   * **Цена уровень** - Выберите **за ГБ (2018)**
1. Выберите **Ok**.

Далее мы подключим к мониторингу доступные безопасные источники данных рабочих станций.

1. На **портале Azure**перейдите на **рабочее пространство Azure Sentinel** > выберите безопасное рабочее пространство **мониторинга рабочих станций**
1. Выберите **разъемы данных**
1. Выберите **Azure Active Directory** > страницу Open Connector > После проверения предпосылки. Приступить к настройке и выбрать **Подключение** как для журналов входа в Azure AD, так и для журналов аудита Azure AD.
1. Выберите **Azure Activity** > открытую страницу соединителя > После проверения предпосылки. Приступить к настройке журналов активности Azure > Выберите подписку > Выберите **Подключение**

По мере сбора данных Sentinel вы сможете наблюдать за активностью, выбирая **портал Azure,** перейдите на **Azure Sentinel Overview** 

Мы будем использовать **Windows Defender ATP (WDATP)** для:

* Постоянное наблюдение и мониторинг уязвимостей и неправильной настройки
* Используйте WDATP для определения приоритетов динамических угроз в дикой природе
* Диск корреляции уязвимостей с конечными точками обнаружения и реагирования (EDR) оповещения
* Используйте панель мониторинга для выявления уязвимости машинного уровня во время расследований
* Вытолкните восстановительные работы в Intune

Наверскакните [панель мониторинга Defender ATP.](https://securitycenter.windows.com/machines) Использование рекомендаций в [обзоре панели мониторинга управления & угрозой &.](/windows/security/threat-protection/microsoft-defender-atp/tvm-dashboard-insights)

## <a name="monitoring-application-activity-using-microsoft-monitoring-agent-mma"></a>Мониторинг активности приложений с помощью агента мониторинга Майкрософт (MMA)
Начиная с специализированной рабочей станции, шкафчик приложения включен для мониторинга активности приложения на рабочей станции. Для интеграции мониторинга в рабочее пространство Log Analytics необходимо следовать агенту ММА и конфигурации. 

> [!NOTE]
> Профиль специализированной рабочей станции содержит политики мониторинга AppLocker. Развертывание политик и средств требуется для мониторинга активности приложения на клиенте

Развертывание агента ММА с помощью сценария Intune PowerShell

1. Загрузите сценарий настройки [на локальное устройство.](https://aka.ms/securedworkstationgit)
1. Обновление параметров, **$WorkSpaceID** и **$WorkSpaceKey**
1. Просмотрите **на портале** > Azure**Microsoft Intune** > **Device конфигурацию** > **Скриптов PowerShell Добавление.** > **Add**
1. Укажите **имя** скрипта и укажите **местоположение скрипта.**
1. Выберите **настройку.**
   1. Установите **выполнить этот скрипт, используя зарегистрированные на учетные данные** **Yes**.
   1. Нажмите кнопку **ОК**.
1. Выберите **Создать**.
1. Выберите**группы** **назначений.** > 
   1. Добавьте группу безопасности **Безопасные рабочие станции.**
   1. Нажмите кнопку **Сохранить**.

Далее необходимо настроить log Analytics для получения новых журналов
1. На **портале Azure**перейдите на **рабочее пространство для аналитики журналов** > Выберите - "Безопасный мониторинг рабочих станций"
1. Выберите **Расширенные настройки** > **журналов событий** **Windows** > Windows
1. В **сборе событий из следующих журналов событий** 
1. Введите:
   * 'Microsoft-Windows-AppLocker/EXE и DLL' > **Неотборный информационный**
   * 'Microsoft-Windows-AppLocker/MSI и сценарий' > неизбранных **информационных**
   * 'Microsoft-Windows-AppLocker/Packaged app-Deployment' > Невыберите **Информационный**
   * 'Microsoft-Windows-AppLocker/Packaged app-Execution' > Неизбранный **информационный**
1. Выберите **Сохранить**

Регистрация приложений будет доступна в выбранном рабочей области Log Analytics.
