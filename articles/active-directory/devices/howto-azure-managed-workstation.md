## <a name="azure-active-directory-configuration"></a>Настройка Azure Active Directory

Настройка каталога Azure Active Directory (Azure AD), который управляет пользователей, групп и устройств для рабочие станции администратора необходимо включить службы удостоверений и компоненты с [учетной записи администратора](https://docs.microsoft.com/azure/active-directory/users-groups-roles/directory-assign-admin-roles).

Рекомендуется сделать этой начальной конфигурации и все глобальные конфигурации из известных устройств безопасный. Вы можете по [предотвращения заражения вредоносными программами](https://docs.microsoft.com/windows/security/threat-protection/intelligence/prevent-malware-infection) для снижения риска раскрытия сначала процесс первого от атаки.

Организации должны требовать многофакторную проверку подлинности, по крайней мере для администраторов. См. в разделе [развертывание MFA в облаке](https://docs.microsoft.com/azure/active-directory/authentication/howto-mfa-getstarted) для руководство по реализации.

### <a name="azure-ad-device-configuration"></a>Настройка устройства Azure AD

#### <a name="specify-who-can-join-devices-to-azure-ad"></a>Укажите, могут присоединять устройства к Azure AD

Настройка устройств параметр в Active Directory, чтобы разрешить вашей группы безопасности администрирования присоединить устройства к домену. Чтобы настроить этот параметр на портале Azure, перейдите к **Azure Active Directory** > **устройств** > **параметры устройства**. Выберите **выбранные** под **пользователи могут присоединять устройства к Azure AD** и выберите группу «Пользователи рабочей станции Secure».

#### <a name="removal-of-local-admin-rights"></a>Удаление прав локального администратора

В процессе развертывания пользователи рабочих станций виртуальных IP-адресов, DevOps и безопасного уровня рабочих станций будет иметь без прав администратора на своих компьютерах. Чтобы настроить этот параметр на портале Azure, перейдите к **Azure Active Directory** > **устройств** > **параметры устройства**. Выберите **None** под **присоединение дополнительных локальных администраторов в Azure AD устройств, присоединенных к**.

#### <a name="require-multi-factor-authentication-to-join-devices"></a>Требовать многофакторную проверку подлинности, чтобы присоединить устройства

Для дальнейшего усиления процесс присоединения устройства к Azure AD, перейдите к **Azure Active Directory** > **устройств** > **параметры устройства** Выберите **Да** под **требует многофакторной проверки подлинности для присоединения устройств** выберите **Сохранить**.

#### <a name="configure-mdm"></a>Настройка управления мобильными Устройствами

На портале Azure перейдите к **Azure Active Directory** > **мобильность (MDM и MAM)**  > **Microsoft Intune**. #### <a name="azure-ad-conditional-access"></a>Условный доступ Azure AD

Azure AD условный доступ можно защитить эти привилегированные задачи администрирования на совместимых устройствах. Пользователи, которые мы определили как члены **защиты рабочей станции пользователей** группы нужно будет выполнить многофакторную проверку подлинности при входе в систему облачных приложений. Мы выполните руководство по лучшим методикам и исключить из политики нашей учетные записи аварийного доступа. Дополнительные сведения можно найти в статье [управление учетные записи аварийного доступа в Azure AD](https://docs.microsoft.com/azure/active-directory/users-groups-roles/directory-emergency-access)

Чтобы настроить условный доступ на портале Azure, перейдите к **Azure Active Directory** > **условного доступа** > **новая политика**.

* **Имя** -политика обязательности устройство обеспечения безопасности
* Назначения
   * **Пользователи и группы**
      * Включить - **пользователей и групп** — выберите **защиты рабочей станции пользователей** ранее созданной группе
      * Исключить - **пользователей и групп** -выберите учетные записи аварийного доступа в организации
   * **Облачные приложения**
      * Включить - **все облачные приложения**
* Элементы управления доступом
   * **Предоставление** — выберите **предоставить доступ** "переключатель"
      * **Требовать многофакторную проверку подлинности**
      * **Требовать, чтобы устройство было отмечено как соответствующее**
      * Для нескольких элементов управления - **требовать все выбранные элементы управления**
* Включить политику - **на**

При необходимости, организации могут создавать политики для блока странах, где бы не доступ к ресурсам компании. Дополнительные сведения о политиках условного доступа на основе расположения IP-адрес можно найти в статье [Какова условия расположения в Azure Active Directory условного доступа?](https://docs.microsoft.com/azure/active-directory/conditional-access/location-condition)

### <a name="configure-enrollment-status"></a>Настройка состояния регистрации

Как описано в управлении логистическими цепочками, обеспечение безопасной рабочей станции является доверенным устройством чистой рекомендуется при покупке новых устройств, устройства быть фабрики, равным [Windows 10 Pro, в режиме S](https://docs.microsoft.com/Windows/deployment/Windows-10-pro-in-s-mode), который ограничивает возможность раскрытия и уязвимостей во время управления цепочкой поставок. После получения устройство у вашего поставщика устройства будут удалены из S режиме, с помощью Autopilot. Далее представлены сведения о применении процесса преобразования.

Мы хотим убедиться, что перед использованием полностью настройки устройств. Intune предоставляет средства для **заблокировать использование устройств, пока не будут установлены все приложения и профили**. 

1. Из **портала Azure** перейдите к:
1. **Microsoft Intune** > **регистрации устройств** > **регистрации Windows** > **странице состояния регистрации(Предварительнаяверсия)**  >  **По умолчанию** > **параметры**.
1. Задайте **отображаться ход выполнения установки профиля приложения** для **Да**.
1. Задайте **заблокировать использование устройств, пока не будут установлены все приложения и профили** для **Да**.

### <a name="create-an-autopilot-deployment-profile"></a>Создание профиля развертывания Autopilot

1. Введите имя **Secure профиля развертывания рабочей станции**. В поле Описание введите **развертывания безопасных рабочих станций**.

1. Для режима развертывания выберите **самостоятельного развертывания (Предварительная версия)** . Устройства с этим профилем, связаны с пользователем, регистрации устройства. Для регистрации устройства необходимы учетные данные пользователя.
1. В соединении с Azure AD в качестве», вкладка « **присоединенные к Azure AD** должен быть выбран и серым цветом.
1. Выберите Out-of-box experience (OOBE), настройте следующие параметры и оставить другие значение по умолчанию, а затем выберите **ОК**:
   1. Тип учетной записи пользователя: **Стандартный**
1. Нажмите кнопку **Создать**, чтобы создать профиль. Теперь профиль развертывания Autopilot доступна для назначения на устройствах.
1. Выберите **назначения** > **назначить** > **выбранные группы**
   1. **Выберите группы для включения** -Secure пользователи рабочей станции

### <a name="configure-windows-update"></a>Настройка центра обновления Windows

Чтобы поддерживать Windows 10 в безопасное состояние, мы Развернем круг обновления для управления скоростью, по которому обновления применяются на рабочих станциях. Эту конфигурацию можно найти в **портала Azure** > **Microsoft Intune** > **обновления программного обеспечения**  >  **Круги обновлений Windows 10**.

Мы выполним **создать** новое кольцо обновления со следующими параметрами, изменить значения по умолчанию.

* Имя — **обновления рабочей станции под управлением Azure**
* Канал - обслуживания **программа предварительной оценки Windows — Fast**
* Отсрочка для обновления (в днях) - **3**
* Период отсрочки обновления компонентов (в днях) - **3**
* Режим автоматического обновления - **автоматическая установка и перезагрузка без управления конечным пользователем**
* Запретить пользователю Приостановка обновления Windows — **блока**
* Требовать утверждения пользователя для перезапуска вне рабочих часов - **требуется**
* Разрешить пользователю перезагрузить (занято) - **требуется**
   * Переход пользователей заинтересованности перезапуска после автоматическую перезагрузку (в днях) - **3**
   * Отложить заинтересованности перезапуска напоминания (в днях) - **3**
   * Установить крайний срок для ожидающие перезагрузки (в днях) - **3**

Нажмите кнопку **создать** на **назначения** добавить вкладку **защиты рабочих станций** группа как включенную группу.

### <a name="windows-defender-atp-intune-integration"></a>Интеграция с ATP в Защитнике Windows Intune

ATP в Защитнике Windows и Microsoft Intune работают вместе для предотвращения нарушений безопасности и ограничить влияние брешей. Возможности обеспечит обнаружение в реальном времени. ATP направим наши развертывания обширной аудит и ведение журнала устройств конечной точке.

Чтобы настроить интеграцию ATP в Защитнике Windows Intune на портале Azure, перейдите к **Microsoft Intune** > **соответствие устройства политике** > **ATP в Защитнике Windows** .

1. На шаге 1 в разделе **Настройка ATP в Защитнике Windows**, нажмите кнопку **подключения ATP в Защитнике Windows в Microsoft Intune в центре безопасности Защитника Windows**.
1. В центре безопасности Защитника Windows:
   1. Выберите **параметры** > **дополнительные функции**
   1. Для **подключения Microsoft Intune**, выберите **на**
   1. Выберите **сохранить параметры**
1. После установки подключения, вернитесь в Intune и нажмите кнопку **обновить** вверху.
1. Задайте **подключения Windows версии устройств 10.0.15063 и более поздние версии ATP в Защитнике Windows** для **на**.
1. **Сохранить**

Дополнительные сведения можно найти в статье [Windows Defender Advanced Threat Protection](https://docs.microsoft.com/Windows/security/threat-protection/windows-defender-atp/windows-defender-advanced-threat-protection).

Под управлением Intune данных экспорта скрипта `DeviceConfiguration_Export.ps1` из [GiuHub конфигурация устройства репозитория](https://github.com/microsoftgraph/powershell-intune-samples/tree/master/DeviceConfiguration) предоставит текущего Экспорт всех существующих профилей Intune.

## <a name="additional-configurations-and-hardening-to-consider"></a>Дополнительные конфигурации и усиление безопасности, которые следует учитывать

Инструкции с включенной безопасной станции, дополнительные элементы управления также следует рассматривать, такие, как получить доступ к альтернативный браузеров, исходящий HTTP разрешенных и заблокированных веб-сайтов и возможность запуска сценария PowerShell.

### <a name="restrictive-inbound-and-outbound-rules-in-firewall-configuration-service-provider-csp"></a>Строгие правила входящего и исходящего трафика в брандмауэре поставщика службы конфигурации (CSP)

Дополнительное управление как входящего, так и исходящих правил могут обновляться для отражения разрешенных и заблокированных конечных точек. По мере усиления защиты безопасной рабочей станции, мы переместить ограничение для запрета всех входящих и исходящих по умолчанию и добавить сайты, разрешенные для исходящего трафика в соответствии с распространенных и надежных веб-сайтов. Дополнительные сведения о конфигурации поставщика службы конфигурации брандмауэра можно найти в статье [CSP брандмауэра](https://docs.microsoft.com/Windows/client-management/mdm/firewall-csp).

Ниже приведены рекомендации по умолчанию ограничены.

* Запретить весь входящий трафик
* Запретить весь исходящий трафик

Проект Spamhaus ведет список хорошо, организации могут использовать в качестве отправной точки для блокирующих сайтов, известный как [списка блок домена (Двойной)](https://www.spamhaus.org/dbl/).

### <a name="managing-local-applications"></a>Управление локальными приложениями

Удаление локальных приложений, включая удаление офисные приложения будет перевести безопасной рабочей станции в по-настоящему усиленной защитой. В нашем примере мы добавим Chrome браузером по умолчанию и ограничить возможность изменить браузера, включая подключаемые модули с помощью Intune.

#### <a name="deploy-applications-using-intune"></a>Развертывание приложений с помощью Intune

В некоторых ситуациях приложений, таких как браузер Google Chrome, нужно на защищенной рабочей станции. Ниже приведены инструкции по установке Chrome для устройств в группе безопасности **защиты рабочих станций** созданного ранее.

1. Загрузите автономный установщик [пакет Chrome для Windows 64‑bit](https://cloud.google.com/chrome-enterprise/browser/download/)
1. Извлеките файлы и запишите расположение `GoogleChromeStandaloneEnterprise64.msi` для установки с помощью Intune
1. В **портала Azure** перейдите к **Microsoft Intune** > **клиентских приложений** > **приложений**  >  **Добавить**
1. В разделе **тип приложения**, выберите **Line-of-business**
1. В разделе **файл пакета приложения**выберите `GoogleChromeStandaloneEnterprise64.msi` места извлечения и нажмите кнопку **ОК**
1. В разделе **сведения о приложении**, введите описание и издатель и выберите **ОК**
1. Щелкните **Добавить**.
1. На **назначения** выберите **доступно для зарегистрированных устройств** под **тип назначения**
1. В разделе **включенные группы**, добавьте **защиты рабочих станций** ранее созданной группе
1. Нажмите кнопку **ОК** затем **сохранить**

Дополнительные рекомендации по настройке параметров Chrome можно найти в своей статье службы поддержки [управления браузера Chrome с помощью Microsoft Intune](https://support.google.com/chrome/a/answer/9102677).

#### <a name="configuring-the-company-portal-for-custom-apps"></a>Настройка корпоративного портала для пользовательских приложений

В безопасном режиме Установка приложений будет ограничен корпоративного портала Intune. Тем не менее при установке портала требуется доступ к Microsoft Store. В нашем решении защищенной будет доступна портале ко всем устройствам в режиме вне сети на корпоративном портале.

Установка Intune управляемых копии [корпоративного портала](https://docs.microsoft.com/Intune/store-apps-company-portal-app) позволит возможность присвоения дополнительные средства по запросу для защищенных рабочих станциях пользователей.

В некоторых организациях может потребоваться установить Windows 32-разрядные приложения или приложения, которым требуются другие подготовительные действия для развертывания. Для этих приложений [Microsoft win32 содержимого prep tool](https://github.com/Microsoft/Microsoft-Win32-Content-Prep-Tool) предоставит готовые к использованию `.intunewin` файла форматирования для установки.

## <a name="enroll-and-validate-your-first-device"></a>Зарегистрировать и проверить первого устройства

1. Чтобы зарегистрировать устройство, вам потребуется следующее:
   * **Серийный номер** - найден в корпусе устройства
   * **Идентификатор продукта Windows** - найден в разделе **системы** > **о** из меню "Параметры Windows".
   * Под управлением [Get WindowsAutoPilotInfo](https://aka.ms/Autopilotshell) предоставит хэш CSV-файл для регистрации устройств с помощью все необходимые сведения. 
      * Запустите `Get-WindowsAutoPilotInfo – outputfile device1.csv` для вывода данных в CSV-файл, который можно импортировать в Intune.

   > [!NOTE]
   > Сценарий требовало повышенных прав и запустить в качестве удаленных подписи. Для правильной работы сценария можно использовать следующую команду. `Set-ExecutionPolicy -ExecutionPolicy RemoteSigned`

   *  Эти данные можно собирать при входе в Windows 10 версии 1809 или более поздней версии устройства для сбора информации или к поставщику оборудования способны предоставить эти данные при заказе новых устройств.
1. Сбор необходимых сведений и вернуться к **портала Azure**. Перейдите к **Microsoft Intune** > **регистрации устройств** > **регистрации Windows** > **устройствами. Управление устройствами Windows Autopilot**выберите **импорта**и выберите CSV-файл, вы создали или были предоставлены.
1. Добавить в группу безопасности теперь зарегистрированном устройстве **защиты рабочих станций** созданного ранее.
1. На устройстве Windows 10, вы хотите настроить, откройте **параметры Windows** > **обновление и безопасность** > **восстановления**. Выберите **приступить к работе** под **сбросить этот компьютер** и следуйте инструкциям на экране для сброса и изменить конфигурацию устройства с помощью профиля и соответствие требованиям политики, настроенные.

После настройки устройства выполните проверку и проверьте конфигурацию. Убедитесь, что первое устройство настроен правильно, прежде чем продолжить развертывание.
