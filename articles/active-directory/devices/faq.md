# <a name="azure-active-directory-device-management-faq"></a>Часто задаваемые вопросы об управлении устройствами в Azure Active Directory

## <a name="general-faq"></a>Общие часто задаваемые вопросы

### <a name="q-i-registered-the-device-recently-why-cant-i-see-the-device-under-my-user-info-in-the-azure-portal-or-why-is-the-device-owner-marked-as-na-for-hybrid-azure-active-directory-azure-ad-joined-devices"></a>Вопрос. Мной недавно было зарегистрировано устройство. Почему я не могу увидеть устройство под информацией о пользователях на портале Azure? Почему владелец устройства отмечен как "Недоступен" для устройств, использующих гибридное присоединение к Azure Active Directory (Azure AD)?

**A:** Устройства Windows 10, которые являются гибридными Azure AD присоединились не отображаются под **user устройств**.
Откройте представление **Все устройства** на портале Azure. Вы также можете использовать командлет PowerShell [Get-MsolDevice](/powershell/module/msonline/get-msoldevice?view=azureadps-1.0).

В списке **устройств пользователей** перечислены только следующие устройства:

- Все личные устройства без гибридного присоединения к Azure AD. 
- Все устройства не на платформе Windows 10 или Windows Server 2016.
- Все устройства не на платформе Windows.

### <a name="q-how-do-i-know-what-the-device-registration-state-of-the-client-is"></a>Вопрос. Как узнать состояние регистрации устройства клиента?

**A:** На портале Azure перейдите на **все устройства.** Найдите устройство по его идентификатору. Посмотрите значение в столбце "Тип соединения". Иногда может оказаться, что выполнялся сброс или пересоздание образа устройства. Таким образом, важно также проверить состояние регистрации устройства на самом устройстве:

- Для устройств с Windows 10 и Windows Server 2016 или более поздней версии запустите `dsregcmd.exe /status`.
- Для версий ОС нижнего уровня запустите `%programFiles%\Microsoft Workplace Join\autoworkplace.exe`.

**A:** Для устранения неполадок информации, см.
- [Устройства для устранения неполадок с использованием dsregcmd команды](troubleshoot-device-dsregcmd.md)
- [Устранение неполадок на устройствах под управлением Windows 10 и Windows Server 2016 с гибридным присоединением к Azure Active Directory](troubleshoot-hybrid-join-windows-current.md)
- [Неуступчивый гибридный каталог Azure Active](troubleshoot-hybrid-join-windows-legacy.md)

Для устройства отображается состояние "Зарегистрировано". Правильно ли я настроен для использования условия доступа?

### <a name="q-why-do-my-users-see-an-error-message-saying-your-organization-has-deleted-the-device-or-your-organization-has-disabled-the-device-on-their-windows-10-devices"></a>В: Почему пользователи видят сообщение об ошибке: «Ваша организация удалила устройство» или «Ваша организация отключила устройство» на своих устройствах с Windows 10?

**A:** На устройствах Windows 10, объединенных или зарегистрированных в Azure AD, пользователям выдается [токен первичного обновления (PRT),](concept-primary-refresh-token.md) который позволяет написать один знак. Действительность ГВП основана на действительности самого устройства. Пользователи видят это сообщение, если устройство удалено или отключено в Azure AD, не инацизивая действие с самого устройства. Устройство может быть удалено или отключено в Azure AD один из следующих сценариев: 

- Пользователь отсутвает устройство с портала My Apps. 
- Администратор (или пользователь) удаляет или отключает устройство на портале Azure или с помощью PowerShell
- Гибридное AD Azure присоединилось только к соединению: администратор удаляет устройства OU из области синхронизации, в результате чего устройства будут удалены из Azure AD
- Обновление Azure AD подключается к версии 1.4.xx.x. [Понимание Azure AD Подключите 1.4.xx.x и исчезновение устройства](/azure/active-directory/hybrid/reference-connect-device-disappearance).


Ниже приведены данные о том, как можно исправить эти действия.

### <a name="q-i-disabled-or-deleted-my-device-in-the-azure-portal-or-by-using-windows-powershell-but-the-local-state-on-the-device-says-its-still-registered-what-should-i-do"></a>В: Я отключил или удалил устройство на портале Azure или с помощью Windows PowerShell. Но в локальном состоянии устройства указано, что оно по-прежнему зарегистрировано. Что делать?

**A:** Эта операция по замыслу. В этом случае устройство не имеет доступа к ресурсам в облаке. Администраторы могут выполнять это действие для устаревших, потерянных или украденных устройств, чтобы предотвратить несанкционированный доступ. Если это действие было выполнено непреднамеренно, необходимо повторно включить или перерегистрировать устройство, как описано ниже

- Если устройство было отключено в Azure AD, администратор с достаточными привилегиями может включить его с портала Azure AD  
  > [!NOTE]
  > При синхронизации устройств с помощью Azure AD Connect гибридные устройства Azure AD будут автоматически перевключены в течение следующего цикла синхронизации. Таким образом, если вам нужно отключить гибридное устройство Azure AD, вам нужно отключить его от вашего собственного AD

 - Если устройство удалено в Azure AD, необходимо перерегистрировать устройство. Для перерегистрации необходимо принять ручное действие на устройстве. Ниже приведены инструкции по перерегистрации на основе состояния устройства. 

      Для перерегистрации гибридных Azure AD, присоединившихся к устройствам Windows 10 и Windows Server 2016/2019, предпримите следующие шаги:

      1. Запустите командную строку от имени администратора.
      1. Введите `dsregcmd.exe /debug /leave`.
      1. Выполните выход и вход, чтобы запустить запланированную задачу, которая повторно регистрирует устройство в Azure AD. 

      Для версий Windows OS низкого уровня, которые являются гибридными Azure AD, присоединитесь, предпримите следующие шаги:

      1. Запустите командную строку от имени администратора.
      1. Введите `"%programFiles%\Microsoft Workplace Join\autoworkplace.exe /l"`.
      1. Введите `"%programFiles%\Microsoft Workplace Join\autoworkplace.exe /j"`.

      Для устройств, объединенных Совместно с Устройствами Windows 10, Присоединенные устройства Azure AD, выполняйте следующие действия:

      1. Запустите командную строку от имени администратора.
      1. Введите `dsregcmd /forcerecovery` (Примечание: Для выполнения этого действия необходимо быть администратором).
      1. Нажмите кнопку "Войти" в диалоге, который открывается и продолжается со знаком в процессе.
      1. Выйдите и войдите обратно в устройство, чтобы завершить восстановление.

      Для устройств, зарегистрированных Windows 10, Azure AD, выполняйте следующие действия:

      1. Перейти к **настройкам учетных** > **записей** > Доступ к работе или**школы**. 
      1. Выберите учетную запись и выберите **Отключить.**
      1. Нажмите на кнопку "Подключите" и зарегистрируйте устройство снова, пройдя через знак в процессе.

### <a name="q-why-do-i-see-duplicate-device-entries-in-the-azure-portal"></a>В: Почему на портале Azure можно увидеть дубликаты записей устройств?

**A:**

- Несколько попыток отсоединить, а затем присоединить одно и то же устройство Windows 10 или Windows Server 2016 могут привести к появлению повторяющихся записей. 
- Каждый пользователь Windows, нажавший кнопку **Add Work or School Account** (Добавить рабочую или учебную учетную запись), создает запись устройства с тем же именем.
- Для версий ОС Windows нижнего уровня, присоединенных к локальному домену Azure Directory, с помощью автоматической регистрации для каждого пользователя домена, который входит в устройство, создается запись устройства с тем же именем. 
- Присоединенный к Azure AD компьютер, на котором были выполнены очистка, повторная установка и повторное присоединение с тем же именем, будет отображен как еще одна запись с тем же именем устройства.

### <a name="q-does-windows-10-device-registration-in-azure-ad-support-tpms-in-fips-mode"></a>В: Поддерживает ли регистрация устройств Windows 10 в Azure AD TPMs в режиме FIPS?

**A:** Регистрация устройств Windows 10 поддерживается только для FIPS-совместимых TPM 2.0 и не поддерживается для TPM 1.2. Если на устройствах есть TPM 1.2, совместимый с FIPS, вы должны отключить их, прежде чем приступить к присоединению Azure AD или к гибридному AD Azure AD. Корпорация Майкрософт не предоставляет никаких инструментов для отключения режима FIPS для TPMs, поскольку она зависит от производителя TPM. Свяжитесь с вашим оборудованием OEM для поддержки.

### <a name="q-why-are-there-devices-marked-as-pending-under-the-registered-column-in-the-azure-portal"></a>В: Почему в колонке REGISTERED на портале Azure устройства помечены как «В ожидании»?

**A**: Ожидание указывает на то, что устройство не зарегистрировано. Это состояние указывает на то, что устройство было синхронизировано с помощью Azure AD-соединения с предприимченным AD и готово к регистрации устройства. Эти устройства имеют набор JOIN TYPE для "Гибридная Azure AD присоединился". Узнайте больше о [том, как спланировать гибридную реализацию Active Directory Azure.](hybrid-azuread-join-plan.md)

>[!NOTE]
>Устройство также может измениться с зарегистрированного состояния на "В ожидании"
>* Если устройство сначала удалено из Azure AD и повторно синхронизировано с предприимченным AD.
>* Если устройство удалено из области синхронизации в Azure AD Connect и добавлено обратно.
>
>В обоих случаях необходимо перерегистрировать устройство вручную на каждом из этих устройств. Чтобы проверить, было ли устройство ранее зарегистрировано, можно [устранить неполадки устройств с помощью команды dsregcmd.](troubleshoot-device-dsregcmd.md)

## <a name="azure-ad-join-faq"></a>Часто задаваемые вопросы о присоединении к Azure AD

### <a name="q-how-do-i-unjoin-an-azure-ad-joined-device-locally-on-the-device"></a>В: Как развернуть устройство Azure AD локально на устройстве?

**A:** Для чистых устройств, связанных с Azure AD, убедитесь, что у вас есть учетная запись локального администратора в автономном режиме или создайте ее. Вы не сможете выполнить вход с учетными данными пользователя Azure AD. Далее перейдите в **Настройки учетных** > **записей** > Доступ к работе или**школы**. Выберите учетную запись и щелкните **Отключить**. Следуйте подсказкам и укажите учетные данные локального администратора. Перезапустите устройство, чтобы завершить процесс отсоединения.

### <a name="q-can-my-users-sign-in-to-azure-ad-joined-devices-that-are-deleted-or-disabled-in-azure-ad"></a>В: Могут ли пользователи войти в доступ к устройствам Azure AD, которые удалены или отключены в Azure AD?

**A:** Да. В Windows есть возможность кэширования имени пользователя и пароля, позволяющая пользователям, которые выполнили вход ранее, быстро получать доступ к рабочему столу даже без подключения к сети. 

Если устройство удалено или отключено в Azure AD, то это неизвестно на устройстве Windows. Таким образом, пользователи, которые выполнили вход ранее, по-прежнему могут получать доступ к рабочему столу с использованием кэшированного имени пользователя и пароля. Но поскольку устройство удалено или отключено, пользователи не могут получить доступ к любым ресурсам, защищенным условным доступом на основе устройства. 

У пользователей, которые не выполнили вход ранее, нет доступа к устройству. Для них не включено кэширование имени пользователя и пароля. 

---

### <a name="q-can-a-disabled-or-deleted-user-sign-in-to-an-azure-ad-joined-devices"></a>В: Может ли отключенный или удаленный пользователь войти в систему с устройствами, объединенными AD Azure

**A:** Да, но только на ограниченное время. Когда пользователь удален или отключен в Azure AD, то об этом не сразу становится известно на устройстве Windows. Таким образом, у пользователей, которые выполнили вход ранее, есть доступ к рабочему столу с помощью кэшированного имени пользователя и пароля. 

Как правило, устройство получает сведения о состоянии пользователя менее чем через четыре часа. Затем Windows блокирует доступ этих пользователей к рабочему столу. После удаления или отключения пользователя в Azure AD все его токены отменяются. Таким образом, у него нет доступа к ресурсам. 

У удаленных или отключенных пользователей, которые не выполнили вход ранее, нет доступа к устройству. Для них не включено кэширование имени пользователя и пароля.

### <a name="q-why-do-my-users-have-issues-on-azure-ad-joined-devices-after-changing-their-upn"></a>В: Почему у моих пользователей возникают проблемы с устройствами, применяемыми AD Azure, после изменения UPN?

**A:** В настоящее время изменения UPN не полностью поддерживаются на устройствах, связанных с Azure AD. Поэтому их проверка подлинности с Azure AD завершается ошибкой после изменения UPN. Таким образом, у пользователей возникают проблемы с единым входом и условным доступом на своих устройствах. Сейчас пользователи должны входить в Windows с помощью плитки "Другой пользователь", используя новое UPN для устранения этой проблемы. Сейчас мы работаем над решением этой проблемы. Тем не менее пользователи, которые входят в систему с помощью Windows Hello для бизнеса, не сталкиваются с этой проблемой.

### <a name="q-why-dont-some-of-my-users-get-azure-multi-factor-authentication-prompts-on-azure-ad-joined-devices"></a>В: Почему некоторые из моих пользователей не получают запросы azure Multi-Factor Authentication на устройствах, связанных с Azure AD?

**A:** Пользователь может присоединиться или зарегистрировать устройство с Azure AD с помощью Multi-Factor Authentication. Само устройство становится вторым надежным фактором для этого пользователя. Каждый раз, когда этот пользователь входит на устройство и получает доступ к приложению, Azure AD рассматривает устройство как второй фактор. Таким образом, этот пользователь может легко получать доступ к приложениям без дополнительных запросов Многофакторной идентификации. 

Эта реакция на событие

- Применяется для присоединенных или зарегистрированных устройств Azure AD, а не для присоединенных устройств Azure AD с гибридным развертыванием.
- не применяется к остальным пользователям, вошедшим в учетную запись устройства. Поэтому все остальные пользователи, получившие доступ к этому устройству, получают запрос на Многофакторную идентификацию. Они могут обращаться к приложениям, для которых требуется Многофакторная идентификация.

### <a name="q-why-do-i-get-a-username-or-password-is-incorrect-message-for-a-device-i-just-joined-to-azure-ad"></a>В: Почему я получаю *имя пользователя или пароль неправильное* сообщение для устройства, которое я только что присоединился к Azure AD?

**A:** Общие причины для этого сценария таковы:

- Ваши учетные данные недействительны.
- Вашему компьютеру не удалось связаться с Azure Active Directory. Наличие проблем с сетевым подключением.
- Для использования федеративных имен для входа требуется, чтобы сервер федерации поддерживал включенные и доступные конечные точки WS-Trust. 
- Вы включили сквозную аутентификацию. Таким образом, при входе в систему необходимо изменить временный пароль.

### <a name="q-why-do-i-see-the-oops-an-error-occurred-dialog-when-i-try-to-azure-ad-join-my-pc"></a>Вопрос: Почему я вижу *Oops ... произошла ошибка!* диалог, когда я пытаюсь Azure AD присоединиться к моему компьютеру?

**A:** Эта ошибка возникает при настройке регистрации Active Directory Azure с помощью Intune. Убедитесь в том, что пользователю, который пытается выполнить операцию присоединения к Azure AD, назначена соответствующая лицензия Intune. Дополнительные сведения см. в статье [Настройка регистрации для устройств Windows](/intune/windows-enroll).

### <a name="q-why-did-my-attempt-to-azure-ad-join-a-pc-fail-although-i-didnt-get-any-error-information"></a>В: Почему моя попытка присоединения Azure AD к ПК не была отклонена, хотя я не получил никакой информации об ошибках?

**A:** Вероятной причиной является то, что вы вписались в устройство с помощью локальной встроенной учетной записи администратора. Создайте другую локальную учетную запись, прежде чем использовать присоединение к Azure Active Directory, чтобы завершить настройку.

### <a name="qwhat-are-the-ms-organization-p2p-access-certificates-present-on-our-windows-10-devices"></a>Вопрос. Что такое сертификаты MS-Organization-P2P-Access, представленные на наших устройствах Windows 10?

**A:** Сертификаты MS-Organization-P2P-Access выдаются Azure AD для обоих, присоединившихся Azure AD и гибридных устройств Azure AD. Эти сертификаты используются для включения доверия между устройствами в одном и том же клиенте для сценариев удаленного рабочего стола. Один сертификат выдается устройству, а другой — пользователю. Сертификат устройства находится в папке `Local Computer\Personal\Certificates` и действителен в течение одного дня. Этот сертификат обновляется (путем выдачи нового сертификата), если устройство по-прежнему активно в Azure AD. Сертификат пользователя находится в папке `Current User\Personal\Certificates` и также является действительным в течение одного дня, но он выдается по требованию, когда пользователь пытается создать сеанс подключения к удаленному рабочему столу другого устройства, присоединенного к Azure AD. Он не обновляется по истечении срока действия. Оба эти сертификата выдаются с использованием сертификата MS-Organization-P2P-Access, который находится в папке `Local Computer\AAD Token Issuer\Certificates`. Этот сертификат выдается Azure AD во время регистрации устройства. 

---

### <a name="qwhy-do-i-see-multiple-expired-certificates-issued-by-ms-organization-p2p-access-on-our-windows-10-devices-how-can-i-delete-them"></a>Вопрос. Почему отображается несколько просроченных сертификатов, выданных MS-Organization-P2P-Access на наших устройствах Windows 10? Как их удалить?

**A:** Существовал вопрос, выявленный на Windows 10 версия 1709 и ниже, где истек MS-Организация-P2P-Доступ сертификаты продолжали существовать в компьютерном магазине из-за криптографических проблем. Ваши пользователи могут столкнуться с проблемами с подключением к сети, если вы используете любые VPN-клиенты (например, Cisco AnyConnect), которые не могут обрабатывать большое количество сертификатов с истекшим сроком действия. Эта проблема исправлена в выпуске Windows 10 версии 1803, что подразумевает автоматическое удаление таких просроченных сертификатов MS-Organization-P2P-Access. Эту проблему можно устранить, обновив устройства до Windows 10 версии 1803. Если выполнить обновление невозможно, можно удалить эти сертификаты без какого-либо неблагоприятного воздействия.

## <a name="hybrid-azure-ad-join-faq"></a>Часто задаваемые вопросы о гибридном присоединении к Azure AD

### <a name="q-how-do-i-unjoin-a-hybrid-azure-ad-joined-device-locally-on-the-device"></a>В: Как объединить устройство, объединенное гибридным Azure AD локально на устройстве?

**A:** Для гибридных устройств Azure AD, которые присоединились к aD, обязательно отключите автоматическую регистрацию. Таким образом запланированная задача не зарегистрирует его снова. Затем запустите командную строку от имени администратора и введите `dsregcmd.exe /debug /leave`. Вместо этого можно выполнить следующую команду в качестве сценария на нескольких устройствах, чтобы отсоединить устройства в пакетном режиме.

### <a name="q-where-can-i-find-troubleshooting-information-to-diagnose-hybrid-azure-ad-join-failures"></a>В: Где можно найти информацию об устранении неполадок для диагностики сбоев гибридного соединения Azure AD?

**A:** Для устранения неполадок информации, см.

- [Устранение неполадок на устройствах под управлением Windows 10 и Windows Server 2016 с гибридным присоединением к Azure Active Directory](troubleshoot-hybrid-join-windows-current.md)
- [Неуступчивый гибридный каталог Azure Active](troubleshoot-hybrid-join-windows-legacy.md)
 
### <a name="q-why-do-i-see-a-duplicate-azure-ad-registered-record-for-my-windows-10-hybrid-azure-ad-joined-device-in-the-azure-ad-devices-list"></a>В: Почему я вижу в списке устройств Azure AD двойную запись Azure AD для моего гибридного устройства Azure AD с Windows 10?

**A:** Когда пользователи добавляют свои учетные записи в приложения на устройстве, присоединенном к домену, они могут быть предложены с **добавлением учетной записи в Windows?** Если ввести **Yes** (Да) в командной строке, устройство будет зарегистрировано в Azure AD. Тип доверия помечается как зарегистрированный в Azure AD. Включив гибридное присоединение к Azure AD для вашей организации, вы также присоедините устройство к гибридной Azure AD. После этого для одного устройства будут отображаться два состояния устройства. 

Присоединение к гибридной Azure AD имеет приоритет над регистрацией в Azure AD. Таким образом, ваше устройство считается гибридным Azure AD, объединенным для любой проверки подлинности и оценки условного доступа. Вы можете удалить запись устройства о регистрации в Azure AD на портале Azure AD. Узнайте, как [избежать двух состояний или удалить их на компьютере с Windows 10](hybrid-azuread-join-plan.md#review-things-you-should-know). 

---

### <a name="q-why-do-my-users-have-issues-on-windows-10-hybrid-azure-ad-joined-devices-after-changing-their-upn"></a>В: Почему у моих пользователей возникают проблемы на устройствах с гибридными Azure AD windows 10 после изменения UPN?

**A:** В настоящее время изменения UPN не полностью поддерживаются гибридными устройствами Azure AD. Если пользователи могут войти на устройство и имеют доступ к своим локальным приложениям, проверка подлинности с помощью Azure AD завершится ошибкой после изменения UPN. Таким образом, у пользователей возникают проблемы с единым входом и условным доступом на своих устройствах. В это время необходимо разсоединить устройство из Azure AD (запуск "dsregcmd/leave" с повышенными привилегиями) и воссоединиться (происходит автоматически) для устранения проблемы. Сейчас мы работаем над решением этой проблемы. Тем не менее пользователи, которые входят в систему с помощью Windows Hello для бизнеса, не сталкиваются с этой проблемой. 

---

### <a name="q-do-windows-10-hybrid-azure-ad-joined-devices-require-line-of-sight-to-the-domain-controller-to-get-access-to-cloud-resources"></a>Вопрос: Требуют ли гибридные устройства Azure AD для Windows 10 гибридных Azure AD для доступа к облачным ресурсам прямой видимости для контроллера домена?

**A:** Нет, за исключением случаев изменения пароля пользователя. После того, как соединение Сwindows 10 гибридAzры AD завершено, и пользователь зарегистрировался по крайней мере один раз, устройство не требует прямой видимости для контроллера домена для доступа к облачным ресурсам. Windows 10 может получить один пакет приложений Azure AD из любой точки мира с подключением к Интернету, за исключением случаев изменения пароля. Пользователи, вслушавшие windows Hello for Business, продолжают получать одноразовые приложения Azure AD даже после изменения пароля, даже если у них нет прямой видимости для контроллера домена. 

---

### <a name="q-what-happens-if-a-user-changes-their-password-and-tries-to-login-to-their-windows-10-hybrid-azure-ad-joined-device-outside-the-corporate-network"></a>В: Что произойдет, если пользователь изменит свой пароль и попытается войти в свой гибридный Azure AD с Windows 10, который присоединился к устройству за пределами корпоративной сети?

**A:** Если пароль изменен за пределами корпоративной сети (например, с помощью Azure AD SSPR), то пользователь, восстав в снадобье, не сможет войти в систему. Для гибридных устройств, соединенных Azure AD, основным органом является on-premises Active Directory. Если устройство не имеет прямой видимости для контроллера домена, оно не может проверить новый пароль. Таким образом, пользователю необходимо установить связь с контроллером домена (либо через VPN, либо находясь в корпоративной сети), прежде чем они смогут войти в устройство со своим новым паролем. В противном случае, они могут войти только со своим старым паролем из-за кэшированного знака в возможности в Windows. Однако старый пароль аннулируется Azure AD во время запросов токенов и, следовательно, предотвращает один вход и не выполняет никаких политик условного доступа на основе устройств. Эта проблема не возникает, если вы используете Windows Hello для бизнеса.

## <a name="azure-ad-register-faq"></a>Часто задаваемые вопросы о регистрации Azure AD

### <a name="q-how-do-i-remove-an-azure-ad-registered-state-for-a-device-locally"></a>В: Как удалить зарегистрированное состояние Azure AD для локального устройства?

**A:** 
- Для устройств, зарегистрированных На Windows 10 Azure AD, перейдите к **настройкам учетных** > **записей,** > доступ к работе или**школе.** Выберите учетную запись и щелкните **Отключить**. Регистрация устройств проводится в профиле пользователя на Windows 10.
- Для iOS и Android можно использовать приложение Microsoft Authenticator **Настройки Регистрация** > **устройств** и выбрать устройство **Unregister.**
- Для macOS можно использовать приложение Microsoft Intune Company Portal, чтобы отстранить устройство от управления и удалить любую регистрацию. 

---
### <a name="q-how-can-i-block-users-from-adding-additional-work-accounts-azure-ad-registered-on-my-corporate-windows-10-devices"></a>В: Как я могу запретить пользователям добавлять дополнительные рабочие учетные записи (зарегистрировано AD Azure) на моих корпоративных устройствах Windows 10?

**A:** Включите следующий реестр, чтобы блокировать пользователям добавление дополнительных рабочих учетных записей в корпоративный домен, присоедининный к нему Azure AD или гибридный Azure AD, присоединившись к устройствам Windows 10. Эта политика также может быть использована для блокирования доменов, объединенных в работу машин, от непреднамеренного регистрации Azure AD с той же учетной записью пользователя. 

`HKLM\SOFTWARE\Policies\Microsoft\Windows\WorkplaceJoin, "BlockAADWorkplaceJoin"=dword:00000001`

### <a name="q-can-i-register-android-or-ios-byod-devices"></a>Вопрос. Можно ли зарегистрировать устройства Android или iOS BYOD?

**A:** Да, но только с помощью службы регистрации устройств Azure и для гибридных клиентов. Не поддерживается с локальной службой регистрации устройств в службах федерации Active Directory (AD FS).
