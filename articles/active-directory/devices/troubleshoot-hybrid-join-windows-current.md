# <a name="troubleshooting-hybrid-azure-active-directory-joined-devices"></a>Неуступчивые гибридные гибридные гибридные гибридные каталоги Azure Active 

Содержание этой статьи применимо к устройствам под управлением Windows 10 или Windows Server 2016.

Для других клиентов Windows смотрите статью [Troubleshooting гибридного Azure Active Directory, объединяемого устройствами уровня вниз.](troubleshoot-hybrid-join-windows-legacy.md)

В этой статье предполагается, что вы [настроили гибридное присоединение устройств к Azure Active Directory](hybrid-azuread-join-plan.md) для поддержки следующих сценариев:

- Условный доступ на основе информации об устройстве
- [Корпоративное перемещение параметров](../active-directory-windows-enterprise-state-roaming-overview.md)
- [Настройка Windows Hello для бизнеса](../active-directory-azureadjoin-passport-deployment.md)

В этом документе содержатся рекомендации по устранению неполадок для устранения потенциальных проблем. 

Для Windows 10 и Windows Server 2016 гибридное присоединение к Azure Active Directory поддерживает обновление Windows 10 от ноября 2015 г.

### <a name="step-2-evaluate-the-join-status"></a>Шаг 2. Анализ состояния присоединения 

Просмотрите следующие поля и убедитесь, что для них заданы ожидаемые значения.

#### <a name="domainjoined--yes"></a>DomainJoined: YES  

Это поле показывает, присоединено ли устройство к локальному каталогу Active Directory или нет. Если отображается значение **NO**, то устройство не может выполнить гибридное присоединение к Azure AD.  

#### <a name="workplacejoined--no"></a>WorkplaceJoined: NO  

Это поле показывает, зарегистрировано ли устройство в Azure AD в качестве личного устройства (с пометкой *Присоединено к рабочей области*). Этот параметр должен иметь значение **NO** для присоединенных к домену компьютеров, на которых также настроено гибридное присоединение к Azure AD. Если этот параметр имеет значение **YES**, то рабочая или учебная учетная запись была добавлена до завершения гибридного присоединения к Azure AD. В таком случае при использовании версии юбилейного обновления Windows 10 (1607) учетная запись игнорируется.

#### <a name="azureadjoined--yes"></a>AzureAdJoined: YES  

Это поле указывает, соединено ли устройство. Значение будет **ДА,** если устройство является либо устройством Azure AD, либо гибридным устройством Azure AD.
Если отображается значение **NO**, то присоединение к Azure AD еще не завершено. 

Приступайке к следующим шагам для дальнейшего устранения неполадок.

### <a name="step-4-check-for-possible-causes-and-resolutions-from-the-lists-below"></a>Шаг 4: Проверьте возможные причины и разрешения из списков ниже

#### <a name="pre-check-phase"></a>Фаза предварительной проверки

Возможные причины сбоя:

- Устройство не имеет прямой видимости для контроллера домена.
   - Устройство должно находиться во внутренней сети организации или на VPN с сетевой линией видимости для контроллера домена Active Directory (AD).

#### <a name="discover-phase"></a>Откройте для себя фазу

Возможные причины сбоя:

- Объект Подключения службы (SCP) неправильно настроен/не в состоянии прочитать объект SCP с DC.
   - В лесу AD, к которому принадлежит устройство, требуется действительный объект SCP, который указывает на проверенное доменное имя в Azure AD.
   - Подробности можно найти в разделе [Настройка точки подключения службы](hybrid-azuread-join-federated-domains.md#configure-hybrid-azure-ad-join).
- Неспособность подключить и получить метаданные обнаружения из конечных точек обнаружения.
   - Устройство должно иметь доступ `https://enterpriseregistration.windows.net`к, в контексте SYSTEM, чтобы обнаружить регистрацию и авторизацию конечных точек. 
   - Если в предварительной среде требуется исходящий прокси, ИТ-админ должен убедиться, что компьютерная учетная запись устройства может обнаружить и бесшумно проверить подлинность исходящего прокси.
- Неспособность подключиться к конечной точке области пользователя и выполнить обнаружение области. (Windows 10 версия 1809 и только позже)
   - Устройство должно иметь доступ `https://login.microsoftonline.com`к, в контексте SYSTEM, для выполнения обнаружения области для проверенного домена и определения типа домена (управляемого/федеративного).
   - Если в предварительной среде требуется исходящий прокси, ИТ-админ должен убедиться, что контекст SYSTEM на устройстве способен обнаружить и бесшумно проверить подлинность исходящего прокси.

###### <a name="http-errors"></a>Ошибки HTTP

- **DSREG_DISCOVERY_TENANT_NOT_FOUND** (0x801c003a/-2145648582)
   - Причина: объект SCP настроен с неправильным идентификатором клиента. Или активных подписок в арендаторе не нашлись.
   - Разрешение: Убедитесь, что объект SCP настроен с правильным идентификатором клиента Azure AD и активными подписками или присутствует в арендаторе.
- **DSREG_SERVER_BUSY** (0x801c0025/-2145648603)
   - Причина: HTTP 503 с сервера DRS.
   - Разрешение: Сервер в настоящее время недоступен. будущие попытки присоединиться, скорее всего, увенчаются успехом, как только сервер вернется в строй.

- **E_INVALIDDATA** (0x8007000d/-2147024883)
   - Причина: Ответ сервера JSON не может быть разогнан. Скорее всего, из-за прокси возвращения HTTP 200 со страницей HTML Auth.
   - Разрешение: Если в предварительной среде требуется исходящий прокси, ИТ-админ должен убедиться, что контекст SYSTEM на устройстве может обнаружить и бесшумно аутентифицировать исходящий прокси.

- **ERROR_ADAL_COULDNOT_DISCOVER_USERNAME_PASSWORD_ENDPOINT** (0xcaa90023/-894894045)

##### <a name="tpm-errors"></a>Ошибки TPM

- **NTE_BAD_KEYSET** (0x80090016/-2146893802)
   - Причина: операция TPM не сработала или была недействительной
   - Разрешение: Вероятно, из-за плохого изображения sysprep. Убедитесь, что машина, из которой создавалось изображение sysprep, не присоединилась Azure AD, присоединился гибридный Azure AD или был зарегистрирован Azure AD.
- **TPM_E_PCP_INTERNAL_ERROR** (0x80290407/-2144795641)
   - Причина: Общая ошибка TPM. 
   - Разрешение: отсражайте TPM на устройствах с этой ошибкой. Версия 1809 и выше Windows 10 и выше автоматически обнаруживает сбои TPM и завершает гибридное соединение Azure AD без использования TPM.
- **TPM_E_NOTFIPS** (0x80280036/-2144862154)
   - Причина: TPM в режиме FIPS в настоящее время не поддерживается.
   - Разрешение: отсражайте TPM на устройствах с этой ошибкой. Windows 1809 автоматически обнаруживает сбои TPM и завершает гибридное соединение Azure AD без использования TPM.
- **NTE_AUTHENTICATION_IGNORED** (0x80090031/-2146893775)
   - Причина: TPM заблокирован.
   - Разрешение: Переходная ошибка. Дождитесь периода охлаждения. Присоединиться к попытке через некоторое время должны увенчаться успехом. Более подробную информацию можно найти в статье [TPM основы](/windows/security/information-protection/tpm/tpm-fundamentals#anti-hammering)

##### <a name="network-errors"></a>Ошибки сети

- **WININET_E_TIMEOUT** (0x80072ee2/-2147012894)
   - Причина: Общая сеть тайм-аут, пытаясь зарегистрировать устройство в DRS
   - Разрешение: Проверьте `https://enterpriseregistration.windows.net`подключение к сети.
- **WININET_E_NAME_NOT_RESOLVED** (0x80072ee7/-2147012889)
   - Причина: имя сервера или адрес не может быть решена.
   - Разрешение: Проверьте `https://enterpriseregistration.windows.net`подключение к сети. Убедитесь, что разрешение DNS для хоста точное в n/w и на устройстве.
- **WININET_E_CONNECTION_ABORTED** (0x80072efe/-2147012866)
   - Причина: Соединение с сервером было прервано ненормально.
   - Разрешение: Повторная попытка через некоторое время или попытка присоединения из альтернативного стабильного местоположения сети.

#### <a name="wamdefaultset-yes-and-azureadprt-yes"></a>WamDefaultSet: ДА и AzureADPrt: ДА
  
Эти поля показывают, прошел ли пользователь аутентификацию в Azure AD при входе в устройство. Если эти параметры имеют значение **NO**, то причины могут быть следующими:

- Плохой ключ хранения в TPM, связанный с устройством при регистрации (проверить KeySignTest во время работы повышенной).
- Наличие альтернативного имени пользователя.
- Прокси-сервер HTTP не найден.
