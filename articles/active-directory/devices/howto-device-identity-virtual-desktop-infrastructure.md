# <a name="device-identity-and-desktop-virtualization"></a>Идентификация устройств и виртуализация рабочего стола

Администраторы обычно развертывают платформы виртуальной настольной инфраструктуры (VDI), размещающие операционные системы Windows в своих организациях. Администраторы разворачивают VDI для:

- Оптимизация управления.
- Снижение затрат за счет консолидации и централизации ресурсов.
- Обеспечить мобильность конечных пользователей и свободу доступа к виртуальным рабочим столам в любое время, из любого места, на любом устройстве.

Существует два основных типа виртуальных настольных компьютеров:

- Постоянный
- Временный

Постоянные версии используют уникальное изображение рабочего стола для каждого пользователя или пул пользователей. Эти уникальные рабочие столы могут быть настроены и сохранены для использования в будущем. 

Непостоянные версии используют набор настольных компьютеров, к которым пользователи могут получить доступ по мере необходимости. Эти непостоянные рабочие столы возвращаются в исходное состояние после того, как пользователь вырегистрируется.

В этой статье будут указаны рекомендации корпорации Майкрософт для администраторов по поддержке идентификации устройств и VDI. Для получения дополнительной информации об идентификации устройства смотрите статью [Что такое идентификация устройства](overview.md).

## <a name="microsofts-guidance"></a>Руководство корпорации Майкрософт

Администраторам следует ссылаться на следующие статьи, основанные на их инфраструктуре идентификации, чтобы узнать, как настроить гибридное соединение Azure AD.

- [Настройка гибридного каталога Azure Active Directory для федеративной среды](hybrid-azuread-join-federated-domains.md)
- [Настройка гибридного каталога Azure Active Directory для управляемой среды](hybrid-azuread-join-managed-domains.md)

Если вы полагаетесь на инструмент подготовки системы (sysprep.exe) и если вы используете предварительное изображение Windows 10 1809 для установки, убедитесь, что изображение не с устройства, которое уже зарегистрировано в Azure AD, как гибрид Azure AD присоединился.

Если вы полагаетесь на снимок виртуальной машины (VM) для создания дополнительных виртуальных высважитесь, убедитесь, что снимок не от VM, который уже зарегистрирован в Azure AD как гибридное соединение Azure AD.

При развертывании непостоянного VDI ИТ-администраторам следует уделять пристальное внимание управлению устаревшими устройствами в Azure AD. Корпорация Майкрософт рекомендует ИТ-администраторам реализовать приведенное ниже руководство. Невыполнение этого требования приведет к тому, что в вашем каталоге будет много устаревших устройств Hybrid Azure AD, которые были зарегистрированы с вашей непостоянной платформы VDI.

- Создайте и используйте префикс для имени дисплея компьютера, который указывает на рабочий стол как на Основе VDI.
- Реализация следующей команды в рамках сценария входа в систему. Эта команда вызовет наилучший вызов усилий для Azure AD для удаления устройства.
   - Для устройств уровня Windows - autoworkplace.exe /leave
- Определите и внедрить процесс [управления устаревшими устройствами.](manage-stale-devices.md)
   - Если у вас есть стратегия для идентификации ваших непостоянных гибридных устройств Azure AD, вы можете быть более агрессивными в очистке этих устройств, чтобы убедиться, что ваш каталог не потребляется с большим количеством устаревших устройств.
