# <a name="controlled-validation-of-hybrid-azure-ad-join"></a>Контролируемая проверка гибридного присоединения к Azure AD

Когда все предварительные требования будут на месте, устройства Windows автоматически регистрируются в качестве устройств в вашем арендаторе Azure AD. Состояние идентификаторов этих устройств в Azure AD называется гибридным соединением Azure AD. Более подробную информацию о концепциях, охватываемых в этой статье, можно найти в статьях [«Введение в управление устройствами» в Active Directory Azure](overview.md) и [запланируйте гибридную программу Azure Active Directory.](hybrid-azuread-join-plan.md)

Организации могут захотеть сделать контролируемую проверку гибридного соединения Azure AD, прежде чем включить его во всю организацию одновременно. В этой статье будет объяснено, как выполнить контролируемую проверку гибридного соединения Azure AD.

## <a name="controlled-validation-of-hybrid-azure-ad-join-on-windows-current-devices"></a>Контролируемая проверка гибридного Azure AD на текущих устройствах Windows

Для устройств под управлением Windows Desktop поддерживается юбилейное обновление Windows 10 (версия 1607) или более поздняя версия. Рекомендуем выполнить обновление до последней версии Windows 10.

Чтобы сделать контролируемую проверку гибридного соединения Azure AD на текущих устройствах Windows, необходимо:

1. Очистить точку подключения службы (SCP) от Active Directory (AD), если она существует
1. Настройка настройки реестра на стороне клиента для SCP на компьютерах, связанных с доменом, с помощью объекта групповой политики (GPO)
1. Если вы используете AD FS, вы также должны настроить настройку реестра на стороне клиента для SCP на вашем сервере AD FS с помощью GPO  
1. Возможно, потребуется [настроить параметры синхронизации](../hybrid/how-to-connect-post-installation.md#additional-tasks-available-in-azure-ad-connect) в Azure AD Connect, чтобы обеспечить синхронизацию устройств. 


### <a name="clear-the-scp-from-ad"></a>Очистить SCP от AD

Используйте редактор интерфейсов Служб активных каталогов (ADSI Edit) для изменения объектов SCP в AD.

1. Запустите настольное приложение **ADSI Edit** с административной рабочей станции или контроллер домена в качестве администратора предприятия.
1. Подключитесь к **контексту именования конфигурации** вашего домена.
1. Просмотр **конфигурации регистрации cn-Configuration,DC-contoso,DC-com** > **CN-Services** > **CN-Устройства Регистрация Конфигурация**
1. Нажмите правой кнопкой мыши на объекте **листа CN-62a0ff2e-97b9-4513-943f-0d221bd30080** и выберите **Свойства**
   1. Выберите **ключевые слова** из окна **редактора Attribute** и нажмите **«Отредки»**
   1. Выберите значения **azureADId** и **azureADName** (по одному за раз) и нажмите **Удалить**
1. Закрыть **ADSI Edit**


### <a name="configure-client-side-registry-setting-for-scp"></a>Настройка настройки реестра на стороне клиента для SCP

Используйте следующий пример для создания объекта групповой политики (GPO) для развертывания настройки реестра, настраивающей запись SCP в реестре ваших устройств.

1. Откройте консоль управления групповой политикой и создайте новый объект групповой политики в вашем домене.
   1. Предоставьте новое имя GPO (например, ClientSideSCP).
1. Отсеяните GPO и найдите следующий путь: > **Реестр**  > **настроек настройки конфигурации** > **Windows Settings** **компьютера**
1. Нажмите на реестр правой кнопкой мыши и выберите **новый** > **пункт регистрации**
   1. На **вкладке «Общая»** назначаем следующее
      1. Действие: **Обновление**
      1. Улей: **HKEY_LOCAL_MACHINE**
      1. Ключевой путь: **ПРОГРАММА-Microsoft-Windows**
      1. Значение имени: **TenantId**
      1. Тип значения: **REG_SZ**
      1. Данные значений: **Идентификатор** GUID или каталога экземпляра Azure AD (это значение можно найти в > **идентификаторе**каталога **Active****Directory Directory)** > **Properties** > 
   1. Нажмите кнопку **ОК**
1. Нажмите на реестр правой кнопкой мыши и выберите **новый** > **пункт регистрации**
   1. На **вкладке «Общая»** назначаем следующее
      1. Действие: **Обновление**
      1. Улей: **HKEY_LOCAL_MACHINE**
      1. Ключевой путь: **ПРОГРАММА-Microsoft-Windows**
      1. Значение имени: **TenantName**
      1. Тип значения: **REG_SZ**
      1. Значение данных: Ваше проверенное **доменное имя,** если вы используете федеративной среде, такой как AD FS. Ваше проверенное **доменное имя** или `contoso.onmicrosoft.com` onmicrosoft.com доменное имя, например, если вы используете управляемую среду
   1. Нажмите кнопку **ОК**
1. Закройте редактор для недавно созданного GPO
1. Связать недавно созданную GPO с желаемым OU, содержащим компьютеры, связанные с доменом, которые принадлежат к вашей контролируемой популяции развертывания

### <a name="configure-ad-fs-settings"></a>Настройка параметров AD FS

Если вы используете AD FS, сначала необходимо настроить SCP на стороне клиента с помощью инструкций, упомянутых выше, связав GPO с серверами AD FS. Объект SCP определяет источник полномочий для объектов устройства. Это может быть на месте или Azure AD. Когда SCP на стороне клиента настроен для AD FS, источник объектов устройства устанавливается как Azure AD.

> [!NOTE]
> Если вы не смогли настроить клиентскую сторону SCP на серверах AD FS, источник идентификационных данных устройства будет рассматриваться как предварительный. Затем ADFS начнет удалять объекты устройств из предварительного каталога после установленного периода, определенного в атрибуте регистрации устройств ADFS "MaximumInactiveDays". Объекты регистрации устройств ADFS можно найти с помощью [cmdlet Get-AdfsDeviceRegistration.](/powershell/module/adfs/get-adfsdeviceregistration?view=win10-ps)

## <a name="controlled-validation-of-hybrid-azure-ad-join-on-windows-down-level-devices"></a>Контролируемая проверка гибридного Azure AD на устройствах уровня снисху

Чтобы зарегистрировать устройства Windows нижнего уровня, необходимо установить в организации [Microsoft Workplace Join для компьютеров, на которых не используется Windows 10](https://www.microsoft.com/download/details.aspx?id=53554). Это средство доступно в Центре загрузки Майкрософт.

Развернуть пакет можно с помощью системы распространения программного обеспечения, например  [Microsoft Endpoint Configuration Manager](/configmgr/). Этот пакет поддерживает параметры стандартной автоматической установки с использованием параметра quiet. В текущей ветви System Center Configuration Manager доступны дополнительные преимущества предыдущих версий, такие как возможность отслеживать ход регистрации.

Установщик создает в системе запланированную задачу, которая выполняется в контексте пользователя. Задача запускается в момент входа пользователя в систему Windows. Эта задача автоматически присоединяет устройство к Azure AD, используя учетные данные пользователя, после аутентификации с помощью Azure AD.

Чтобы контролировать регистрацию устройства, необходимо развернуть пакет установки Windows на выбранной группе устройств на уровне низшения Windows.

> [!NOTE]
> Если SCP не настроен в AD, то следует следовать тому же подходу, который описан для [настройки настройки реестра на стороне клиента для SCP](#configure-client-side-registry-setting-for-scp)) на компьютерах, связанных с доменом, с помощью объекта групповой политики (GPO).


После проверки того, что все работает в штатном режиме, можно автоматически зарегистрировать остальные устройства текущего и нистого уровня Windows с помощью Azure AD, [настроив SCP с помощью Azure AD Connect.](hybrid-azuread-join-managed-domains.md#configure-hybrid-azure-ad-join)
