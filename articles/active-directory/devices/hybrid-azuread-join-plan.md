# <a name="how-to-plan-your-hybrid-azure-active-directory-join-implementation"></a>Как: Планирование гибридного каталога Azure Active

Подобно пользователю, устройство — это еще одно основное удостоверение, которое необходимо защитить, чтобы с его помощью обеспечить постоянную защиту ресурсов независимо от того, где вы находитесь. Для этого разместите удостоверения своих устройств в Azure AD и управляйте такими удостоверениями с помощью одного из следующих методов:

- присоединение к Azure AD;
- гибридное присоединение к Azure AD;
- регистрация в Azure AD.

Размещение устройств в Azure AD позволяет увеличить эффективность работы пользователей благодаря выполнению единого входа для облачных и локальных ресурсов. В то же время, вы можете обеспечить доступ к облачным и предприимчать ресурсы с [помощью условия доступа.](../active-directory-conditional-access-azure-portal.md)

Если у вас есть среда Active Directory (AD) и вы хотите присоединиться к компьютеру, связанному с aD доменом, к Azure AD, вы можете сделать это, присоединившись к гибридному AD-аду. В этой статье приведены соответствующие шаги для реализации гибридного присоединения устройств к Azure AD в вашей среде.

## <a name="review-things-you-should-know"></a>Ознакомление с важными сведениями

### <a name="unsupported-scenarios"></a>Неподдерживаемые сценарии
- Гибридное соединение Azure AD в настоящее время не поддерживается, если среда состоит из одного леса AD, синхронизирующего идентификационные данные с более чем одним арендатором Azure AD.

- Гибридное соединение Azure AD не поддерживается для Windows Server, работая в роли контроллера домена (DC).

- Гибридное соединение Azure AD не поддерживается на устройствах уровня Windows при использовании роуминга учетных данных или пользовательского профиля или обязательного профиля.

- Server Core OS не поддерживает регистрацию устройств любого типа.

### <a name="os-imaging-considerations"></a>Соображения изображения ОС
- Если вы полагаетесь на инструмент подготовки системы (Sysprep) и используете для установки **изображение предварительноwindows 10 1809,** убедитесь, что изображение не с устройства, которое уже зарегистрировано в Azure AD в качестве гибридного Azure AD.

- Если вы полагаетесь на снимок виртуальной машины (VM) для создания дополнительных виртуальных высважитесь, убедитесь, что снимок не от VM, который уже зарегистрирован в Azure AD как гибридное соединение Azure AD.

- Если вы используете [Unified Write Filter](/windows-hardware/customize/enterprise/unified-write-filter) и аналогичные технологии, которые осваивают изменения на диске при перезагрузке, они должны быть применены после присоединения устройства к гибридной Azure AD. Включение таких технологий до завершения гибридного соединения Azure AD приведет к тому, что устройство будет объединено при каждой перезагрузке

### <a name="handling-devices-with-azure-ad-registered-state"></a>Обработка устройств с состоянием, зарегистрированным Azure AD
Если устройства, объединенные доменами Windows 10, являются [Azure AD, зарегистрированными](overview.md#getting-devices-in-azure-ad) на вашего арендатора, это может привести к двойному состоянию гибридного Azure AD, объединенного и зарегистрированного устройства Azure AD. Мы рекомендуем обновить до Windows 10 1803 (с KB4489894 применяется) или выше, чтобы автоматически рассмотреть этот сценарий. В релизах, выпускавшихся до 1803 года, необходимо удалить состояние, зарегистрированное Azure AD, вручную, прежде чем включить гибридную Azure AD. В 1803 и выше релизы, следующие изменения были сделаны, чтобы избежать этого двойного состояния:

- Любое существующее состояние, зарегистрированное Azure AD для пользователя, будет автоматически удалено <i>после присоединения устройства к гибридному Azure AD и в входе в систему того же пользователя.</i> Например, если пользователь A зарегистрировал состояние Azure AD на устройстве, двойное состояние пользователя A очищается только при входе пользователя A в устройство. если на одном устройстве несколько пользователей, двойное состояние очищается индивидуально при входе в систему.
- Вы можете предотвратить, чтобы устройство, объединенное доменом, было зарегистрировано в Azure AD, добавив этот ключ реестра - HKLM-SOFTWARE-Политики,Microsoft/Windows-WorkplaceJoin, "BlockAADWorkplaceJoin":dword:00000001.
- В Windows 10 1803, если у вас есть Windows Hello for Business настроен, пользователю необходимо перенастроить Windows Hello for Business после очистки двойного состояния. Этот вопрос был решен с KB4512509

> [!NOTE]
> Зарегистрированное устройство Azure AD не будет автоматически удалено, если им управляет Intune.

### <a name="additional-considerations"></a>Дополнительные сведения
- Если в вашей среде используется [Device identity and desktop virtualization](/azure/active-directory/devices/howto-device-identity-virtual-desktop-infrastructure)виртуальная инфраструктура настольных компьютеров (VDI), см.

- Гибридное соединение Azure AD поддерживается для FIPS-совместимых TPM 2.0 и не поддерживается для TPM 1.2. Если на устройствах есть TPM 1.2, совместимый с FIPS, вы должны отключить их, прежде чем приступить к гибридному ad Azure. Корпорация Майкрософт не предоставляет никаких инструментов для отключения режима FIPS для TPMs, поскольку она зависит от производителя TPM. Пожалуйста, свяжитесь с вашим оборудованием OEM для поддержки. 

- Начиная с windows 10 1903 релиз, TPMs 1.2 не используются с гибридной Azure AD соединения и устройства с этими TPMs будут рассматриваться как если бы они не имеют TPM.

## <a name="review-controlled-validation-of-hybrid-azure-ad-join"></a>Обзор контролируемой проверки гибридного соединения Azure AD

Когда все предварительные требования будут на месте, устройства Windows автоматически регистрируются в качестве устройств в вашем арендаторе Azure AD. Состояние идентификаторов этих устройств в Azure AD называется гибридным соединением Azure AD. Более подробную информацию о концепциях, охватываемых в этой статье, можно найти в статье [«Введение в управление идентификацией устройств» в Active Directory Azure Active.](overview.md)

Организации могут захотеть сделать контролируемую проверку гибридного соединения Azure AD, прежде чем включить его во всю организацию одновременно. Просмотрите [статью, контролируемую проверку гибридного соединения Azure AD,](hybrid-azuread-join-control.md) чтобы понять, как это сделать.

## <a name="select-your-scenario-based-on-your-identity-infrastructure"></a>Выберите сценарий на основе инфраструктуры идентификации

Гибридное соединение Azure AD работает как с, управляемыми, так и с федеративной средой в зависимости от того, является ли UPN руемым или неуправляемым. Смотрите нижнюю часть страницы таблицы по поддерживаемым сценариям.  

### <a name="managed-environment"></a>Управляемая среда

Управляемую среду можно развернуть при помощи [синхронизации хэша паролей](/azure/active-directory/hybrid/whatis-phs) или [сквозной аутентификации](/azure/active-directory/hybrid/how-to-connect-pta) с [единым входом](/azure/active-directory/hybrid/how-to-connect-sso).

В этих сценариях не требуется настраивать сервер федерации для проверки подлинности.

### <a name="federated-environment"></a>Федеративная среда

В федеративной среде должен быть поставщик удостоверений, поддерживающий следующие требования. Это не относится к федеративной среде, в которой используются службы федерации Active Directory (AD FS).

- **Претензия WIAORMULTIAUTHN:** Эта претензия необходима для соединения гибридного Azure AD для устройств на уровне низом Windows.
- **Протокол WS-Trust:** Этот протокол необходим для проверки подлинности гибридных устройств Azure AD с Azure AD. При использовании AD FS нужно включить следующие конечные точки WS-Trust: `/adfs/services/trust/2005/windowstransport`  
`/adfs/services/trust/13/windowstransport`  
  `/adfs/services/trust/2005/usernamemixed` 
  `/adfs/services/trust/13/usernamemixed`
  `/adfs/services/trust/2005/certificatemixed` 
  `/adfs/services/trust/13/certificatemixed` 

> [!WARNING] 
> Также нужно включить **adfs/services/trust/2005/windowstransport** или **adfs/services/trust/13/windowstransport**, но только в качестве конечных точек с подключением к интрасети. Их НЕЛЬЗЯ предоставлять как конечные точки с подключением к экстрасети через прокси-сервер веб-приложения. Чтобы узнать больше о том, как отключить конечные точки WS-Trust Windows, [см.](/windows-server/identity/ad-fs/deployment/best-practices-securing-ad-fs#disable-ws-trust-windows-endpoints-on-the-proxy-ie-from-extranet) Вы можете увидеть, какие конечные точки включены через консоль управления AD FS в **области конечных** > **точек службы.**

> [!NOTE]
> Azure AD не поддерживает смарт-карты и сертификаты в управляемых доменах.

Начиная с версии 1.1.819.0 Azure AD Connect предоставляет мастер для настройки гибридного присоединения к Azure AD. Этот мастер позволяет значительно упростить настройку. Если установка нужной версии Azure AD Connect для вас не подходит, перейдите к [руководству по настройке гибридного присоединения устройств к Azure AD вручную](hybrid-azuread-join-manual.md).

## <a name="review-on-premises-ad-users-upn-support-for-hybrid-azure-ad-join"></a>Просмотр нахотраса пользователей AD UPN поддержки для гибридного Azure AD присоединиться

Иногда ваши предприимченные AD-пользователи UPN могут отличаться от UPNs Azure AD. В некоторых случаях присоединение к гибридной Azure Active Directory в Windows 10 обеспечивает ограниченную поддержку имен участников-пользователей локальной службы AD, которая определяется [методом проверки подлинности](/azure/security/fundamentals/choose-ad-authn), типом домена и версией Windows 10. Существуют два типа имен участников-пользователей локальной службы AD, которые могут существовать в вашей среде.

- Routable пользователей UPN: routable UPN имеет действительный проверенный домен, который зарегистрирован в регистраторе домена. Например, если основным доменом является contoso.com в Azure Active Directory, то contoso.org является основным доменом в локальной службе AD, принадлежащим компании Contoso и [проверенным в Azure Active Directory](/azure/active-directory/fundamentals/add-custom-domain).
- Нерулебивые пользователи UPN: не-routable UPN не имеет проверенного домена. Эго возможно использовать только в частной сети вашей организации. Например, если основным доменом в Azure Active Directory является contoso.com, то contoso.local является основным доменом в локальной службе AD, но не является проверяемым доменом в Интернете и используется только в сети компании Contoso.

> [!NOTE]
> Информация в этом разделе распространяется только на пользователей UPN. Это не применимо к локальному суффиксу домена компьютера (пример: computer1.contoso.local).

В приведенной ниже таблице приведены сведения о поддержке имен участников-пользователей локальной службы AD для гибридного присоединения к Azure Active Directory в Windows 10.

| Тип имени участника-пользователя локальной службы AD | Тип домена | Версия Windows 10 | Описание |
| ----- | ----- | ----- | ----- |
| Маршрутизируемый | Федеративные | Начиная с выпуска 1703 | Общедоступная версия |
| Немаршрутизируемый | Федеративные | Начиная с выпуска 1803 | Общедоступная версия |
| Маршрутизируемый | Управляемые | Начиная с выпуска 1803 | Как правило, Azure AD SSPR на экране блокировки Windows не поддерживается |
| Немаршрутизируемый | Управляемые | Не поддерживается | |
