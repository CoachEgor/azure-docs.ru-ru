# <a name="sign-in-to-windows-virtual-machine-in-azure-using-azure-active-directory-authentication-preview"></a>Вопиюсь в виртуальной машине Windows в Azure с помощью проверки подлинности Active Directory Azure (Предварительный просмотр)

Теперь организации могут использовать проверку подлинности Azure Active Directory (AD) для своих виртуальных машин Azure (VMs) под управлением **Windows Server 2019 Издания дата-центра данных** или **Windows 10 1809** и позже. Использование Azure AD для аутентификации в отношении вс-м информации предоставляет возможность централизованного контроля и обеспечения соблюдения политик. Такие инструменты, как Azure Role-Based Control (RBAC) и Azure AD Conditional Access позволяют контролировать, кто может получить доступ к VM. В этой статье показано, как создать и настроить VM Windows Server 2019 для использования аутентификации Azure AD.

| --- |
| Регистрация Azure AD для VMs-ми с Windows Azure — это общедоступное функцию предварительного просмотра в каталоге Azure Active. См. дополнительные сведения о [дополнительных условиях использования предварительных выпусков Microsoft Azure](https://azure.microsoft.com/support/legal/preview-supplemental-terms/).|
|     |

Использование аутентификации Azure AD для входа в систему для windows VMs в Azure имеет много преимуществ, включая:

- Используйте те же федеративные или управляемые учетные данные Azure AD, которые обычно используются.
- Больше не нужно управлять учетными записями локальных администраторов.
- Azure RBAC позволяет предоставлять соответствующий доступ к вм-технологий в зависимости от потребностей и удалять их, когда в этом больше нет необходимости.
- Прежде чем разрешить доступ к VM, Azure AD Conditional Access может применять дополнительные требования, такие как: 
   - Многофакторная Идентификация
   - Проверка риска ввне
- Автоматизация и масштабирование Azure AD объединяют сярвые к vMs-мизантам Windows Azure, которые являются частью развертывания VDI.

> [!NOTE]
> Как только вы включите эту возможность, к вашим вмчатся windows в Azure присоединятся к Azure AD. Вы не можете присоединиться к нему в другой домен, такой как on-premises AD или Azure AD DS. Если это необходимо, вам нужно отключить VM от арендатора Azure AD, унавив расширение.

В Cloud Shell предварительно установлены и настроены общие инструменты Azure для использования с вашей учетной записью. Нажмите кнопку "Копировать", чтобы скопировать код. Вставьте его в Cloud Shell и нажмите клавишу "ВВОД", чтобы выполнить код. Cloud Shell можно открыть разными способами:

Нажмите кнопку Попробовать в правом верхнем углу блока с кодом.
Откройте Cloud Shell в браузере.
Нажмите кнопку меню Cloud Shell в правом верхнем углу окна [портала Azure](https://portal.azure.com).

Если вы решили установить и использовать CLI локально, эта статья требует, чтобы вы запускали версию Azure CLI 2.0.31 или позже. Чтобы узнать версию, выполните команду az --version. Если вам нужно установить или обновить, смотрите статью [Установить Azure CLI](/cli/azure/install-azure-cli).

1. Создайте группу ресурсов с помощью команды [az group create](/cli/azure/group#az-group-create). 
1. Создайте VM с [помощью az vm, создавая](/cli/azure/vm#az-vm-create) с помощью поддерживаемого дистрибутива в поддерживаемом регионе. 
1. Установите расширение входа в VM Azure AD. 

Следующий пример развертывает VM под названием myVM, который использует Win2019Datacenter, в группу ресурсов под названием myResourceGroup, в южном центральном регионе. В следующих примерах можно при необходимости указать собственную группу ресурсов и имена виртуальных машин.

```AzureCLI
az group create --name myResourceGroup --location southcentralus

az vm create \
    --resource-group myResourceGroup \
    --name myVM \
    --image Win2019Datacenter \
    --assign-identity \
    --admin-username azureuser \
    --admin-password yourpassword
```

> [!NOTE]
> Необходимо включить систему, назначенную управляемой личностью на виртуальной машине, прежде чем установить расширение входа в VM Azure AD.

Создание виртуальной машины и вспомогательных ресурсов занимает несколько минут.

Наконец, установите расширение входа в VM Azure AD, чтобы включить в систему журнал Azure AD для Windows VM. Расширения виртуальных машин — это небольшие приложения, которые выполняют задачи настройки и автоматизации после развертывания виртуальных машин Azure. Используйте набор [расширений az vm](/cli/azure/vm/extension#az-vm-extension-set) для установки расширения AADLoginForWindows на VM под названием myVM в группе ресурсов myResourceGroup:

> [!NOTE]
> Вы можете установить расширение AADLoginForWindows на существующий Windows Server 2019 или Windows 10 1809, а затем VM, чтобы включить его для проверки подлинности Azure AD. Ниже приведен пример АК CLI.

```AzureCLI
az vm extension set \
    --publisher Microsoft.Azure.ActiveDirectory \
    --name AADLoginForWindows \
    --resource-group myResourceGroup \
    --vm-name myVM
```

`Succeeded` Показано, `provisioningState` как только расширение установлено на VM.

## <a name="configure-role-assignments-for-the-vm"></a>Настройка назначений ролей для виртуальной машины

Теперь, когда вы создали VM, вам нужно настроить политику Azure RBAC, чтобы определить, кто может войти в Систему VM. Для авторизации входа на виртуальную машину используются две роли RBAC:

- Войти в систему **виртуальной машины Администратор:** Пользователи с этой ролью могут войти в виртуальную машину Azure с привилегиями администратора.
- **Имя для входа пользователя виртуальной машины.** Пользователи с этой ролью могут входить на виртуальную машину Azure с правами обычного пользователя.

> [!NOTE]
> Чтобы позволить пользователю войти в VM через RDP, необходимо назначить либо виртуальную машину Администратор Войти или Виртуальная машина Пользователя Войти роль. Пользователь Azure с ролями владельца или вкладчика, назначенными для VM, не имеет автоматически привилегий входа в VM через RDP. Это необходимо для обеспечения проверенного разделения между набором людей, которые управляют виртуальными машинами, по сравнению с набором людей, которые могут получить доступ к виртуальным машинам.

Существует несколько способов настройки ролевых заданий для VM:

- Использование опыта Azure AD Portal
- Использование опыта работы с облачными оболочками Azure

### <a name="using-azure-ad-portal-experience"></a>Использование опыта Azure AD Portal

Настройка ролевых заданий для AD Споддержки Windows Server 2019 ВМ:

1. Перейдите на конкретную страницу обзора виртуальной машины
1. Выберите **элемент управления доступом (IAM)** из вариантов меню
1. Выберите **Добавить,** **Добавить назначение ролей,** чтобы открыть панель назначения роли Добавить.
1. В списке выпадающих **ролей** выберите такую роль, как **Логин администратора виртуальной машины** или **Логин пользователя виртуальной машины.**
1. В поле **Select** выберите пользователя, группу, основной службы или управляемую идентификацию. Если субъект безопасности не отображается в списке, введите его имя в поле **Выбор**, чтобы найти в каталоге отображаемые имена, адреса электронной почты и идентификаторы объектов.
1. Выберите **Сохранить,** чтобы назначить роль.

Через несколько секунд субъекту безопасности будет назначена роль в выбранной области.

![Назначить роли пользователям, которые получат доступ к VM](./media/howto-vm-sign-in-azure-ad-windows/azure-portal-access-control-assign-role.png)

### <a name="using-the-azure-cloud-shell-experience"></a>Использование опыта работы с облачными оболочками Azure

В следующем примере используется команда [az role assignment create](/cli/azure/role/assignment#az-role-assignment-create) для назначения роли Имя для входа администратора виртуальной машины виртуальной машине текущего пользователя Azure. Имя пользователя активной учетной записи Azure можно получить с помощью команды [az account show](/cli/azure/account#az-account-show). В качестве области задается виртуальная машина, созданная на предыдущем шаге с помощью команды [az vm show](/cli/azure/vm#az-vm-show). Область также можно назначить на уровне группы ресурсов или подписки. При этом применяются обычные разрешения наследования RBAC. Для получения дополнительной [Role-Based Access Controls](../../virtual-machines/linux/login-using-aad.md)информации см.

```   AzureCLI
username=$(az account show --query user.name --output tsv)
vm=$(az vm show --resource-group myResourceGroup --name myVM --query id -o tsv)

    --role "Virtual Machine Administrator Login" \
    --assignee $username \
    --scope $vm
```

Идентификатор объекта учетной записи пользователя можно получить с помощью команды [az ad user list](/cli/azure/ad/user#az-ad-user-list).

## <a name="using-conditional-access"></a>Использование условного доступа

Вы можете применять политики условного доступа, такие как многофакторная аутентификация или проверка риска вхинга пользователя, прежде чем авторизовать доступ к Windows VMs в Azure, которые включены при всколых Azure AD. Чтобы применить политику условного доступа, необходимо выбрать приложение "Azure Windows VM Sign-In" из облачных приложений или опции назначения действий, а затем использовать риск ввески в качестве условия и/или потребовать многофакторной аутентификации в качестве элемента контроля доступа к гранту. 

> [!NOTE]
> Если вы используете "Требовать многофакторной аутентификации" в качестве элемента контроля доступа к гранту для запроса доступа к приложению "Azure Windows VM Sign-In", то вы должны предоставить многофакторную претензию аутентификации как часть клиента, который инициирует сеанс RDP для целевой Windows ВМ в Лазурном берегу. Единственный способ достичь этого на Windows 10 клиента является использование Windows Hello для бизнеса PIN или биометрической authenication с клиентом RDP. Поддержка биометрической аутентификации была добавлена клиенту RDP в версии Windows 10 1809. Удаленный рабочий стол с использованием Windows Hello for Business аутентификации доступен только для развертывания, использующих модель доверия сертификата, и в настоящее время недоступен для ключевой модели доверия.

## <a name="log-in-using-azure-ad-credentials-to-a-windows-vm"></a>Войти в систему с использованием учетных данных Azure AD для Windows VM

> [!IMPORTANT]
> Удаленное подключение к ВМ, присоединенному к Azure AD, разрешено только с ПК с Windows 10, к которым присоединяются Azure AD или гибридный Azure AD, соединенные к тому **же** каталогу, что и VM. Кроме того, для RDP, использующего учетные данные Azure AD, пользователь должен принадлежать к одной из двух ролей RBAC- Virtual Machine Administrator Login или Virtual Machine User Login. В настоящее время Azure Bastion не может использоваться для входа в систему с помощью проверки подлинности Active Directory Azure с расширением AADLoginForWindows. Поддерживается только прямой RDP.

Чтобы войти в виртуальную машину Windows Server 2019 с помощью Azure AD: 

1. Перейдите на страницу обзора виртуальной машины, включенную с помощью логона Azure AD.
1. Выберите **Connect,** чтобы открыть подключение к виртуальному лезвию машины.
1. Выберите **Скачать RDP файл**.
1. Выберите **Открытый** для запуска удаленного клиента соединения рабочего стола.
1. Выберите **Connect** для запуска диалога о входах в Windows.
1. Logon, используя учетные данные Azure AD.

Теперь вы вписаны в виртуальную машину Windows Server 2019 Azure с назначенными разрешениями ролей, такими как VM User или VM Administrator. 

> [!NOTE]
> Вы можете сохранить . RDP файл локально на вашем компьютере для запуска будущих удаленных настольных соединений на ваш виртуальный компьютер вместо того, чтобы перейти на виртуальную страницу обзора машины на портале Azure и с помощью опции подключения.

## <a name="troubleshoot"></a>Устранение неполадок

### <a name="troubleshoot-deployment-issues"></a>Устранение неполадок развертывания

Расширение AADLoginForWindows должно быть успешно установлено для того, чтобы VM завершил процесс присоединения Azure AD. Выполните следующие действия, если расширение VM не удается установить правильно.

1. RDP к VM с помощью учетной записи локального администратора и изучить CommandExecuti'n.log под  
   
   C: »WindowsAzure-Logs-Plugins-Microsoft.Azure.ActiveDirectory.AADLoginForWindows-0.3.1.0. 

   > [!NOTE]
   > При повторном запуске расширения после первоначального сбоя журнал с ошибкой развертывания будет сохранен как CommandExecution_YYYYMMDDHHMMSSSSS.log. "
1. Откройте запрос команды на VM и проверьте эти запросы в отношении службы метаданных инстанции (IMDS), работающая на хост-узла Azure:

   | Команда для запуска | Ожидаемые выходные данные |
   | --- | --- |
   | `curl -H @{"Metadata"="true"} "http://169.254.169.254/metadata/instance?api-version=2017-08-01"` | Правильная информация о Azure VM |
   | `curl -H @{"Metadata"="true"} "http://169.254.169.254/metadata/identity/info?api-version=2018-02-01"` | Идентификатор действительного клиента, связанный с подпиской Azure |
   | `curl -H @{"Metadata"="true"} "http://169.254.169.254/metadata/identity/oauth2/token?resource=urn:ms-drs:enterpriseregistration.windows.net&api-version=2018-02-01"` | Действительный токен доступа, выпущенный Active Directory Azure для управляемого итогового удостоверения, назначенного этому VM |

   > [!NOTE]

#### <a name="unauthorized-client"></a>Несанкционированный клиент

Если при инициировании удаленного подключения к VM вы видите следующее сообщение об ошибке: 

- Ваши учетные данные не сработали

![Ваши учетные данные не сработали](./media/howto-vm-sign-in-azure-ad-windows/your-credentials-did-not-work.png)

Убедитесь, что компьютер Windows 10, который вы используете для инициирования удаленного соединения рабочего стола, является либо подключенным Azure AD, либо гибридным Azure AD, объединенным с тем же каталогом Azure AD, к которому присоединяется ваш VM. Для получения дополнительной информации об идентификации устройства смотрите статью [Что такое идентификация устройства](/azure/active-directory/devices/overview).

> [!NOTE]
> Windows 10 20H1, добавит поддержку Azure AD Зарегистрированный компьютер, чтобы инициировать удаленное подключение к вашему VM. Присоединяйтесь к программе предварительной оценки Windows, чтобы попробовать это и изучить новые возможности Windows 10.

Кроме того, проверка расширения AADLoginForWindows не была неустановлена после завершения соединения Azure AD.
 
#### <a name="mfa-sign-in-method-required"></a>Требуется метод регистрации МИД

Если при инициировании удаленного подключения к VM вы видите следующее сообщение об ошибке: 

- Метод вхаски, который вы пытаетесь использовать, не допускается. Попробуйте другой способ регистрации или свяжитесь с системным администратором.

![Метод вхаски, который вы пытаетесь использовать, не допускается.](./media/howto-vm-sign-in-azure-ad-windows/mfa-sign-in-method-required.png)

Если вы настроили политику условного доступа, которая требует многофакторной аутентификации (MFA), прежде чем вы сможете получить доступ к ресурсу, то вам нужно убедиться, что Windows 10 PC инициирует удаленное соединение рабочего стола с вашими vM-знаками при использовании сильного метод аутентификации, такой как Windows Hello. Если вы не используете сильный метод проверки подлинности для удаленного соединения рабочего стола, вы увидите предыдущую ошибку.

Если вы не развернули Windows Hello for Business и если это пока не вариант, можно исключить требование МИД, нанастройки политики условного доступа, которая исключает приложение "Azure Windows VM Sign-In" из списка облачных приложений, требующих MFA. Чтобы узнать больше о Windows [Windows Hello for Business Overview](/windows/security/identity-protection/hello-for-business/hello-identity-verification)Hello for Business, см.

> [!NOTE]
> Windows Hello for Business PIN-аутентификация с ПОМОЩЬю RDP была поддержана Windows 10 для нескольких версий, однако поддержка биометрической аутентификации с ПОМОЩЬю RDP была добавлена в версию Windows 10 1809. Использование Windows Hello for Business auth во время RDP доступно только для развертываний, использующих модель доверия сертификата, и в настоящее время недоступно для ключевой модели доверия.
