# <a name="combined-security-information-registration-overview"></a>Комбинированный обзор регистрации информации о безопасности

Перед комбинированной регистрацией пользователи отдельно зарегистрировали методы аутентификации для Azure Multi-Factor Authentication и сбросить паролем самообслуживания (SSPR). Люди были смущены тем, что аналогичные методы были использованы для Multi-Фактор аутентификации и SSPR, но они должны были зарегистрироваться для обеих функций. Теперь, с комбинированной регистрацией, пользователи могут зарегистрироваться один раз и получить преимущества как Multi-фактор аутентификации и SSPR.

В этой статье излагается, что такое комбинированная регистрация безопасности. Для начала с комбинированной регистрации безопасности, см.

> [!div class="nextstepaction"]
> [Включить комбинированную регистрацию безопасности](howto-registration-mfa-sspr-combined.md)

![Мой профиль, показывающий зарегистрированную информацию о безопасности для пользователя](media/concept-registration-mfa-sspr-combined/combined-security-info-defualts-registered.png)

Прежде чем включить новый опыт, просмотрите эту документацию, ориентированную на администратора, и документацию, ориентированную на пользователя, чтобы убедиться, что вы понимаете функциональность и эффект этой функции. Основывайте обучение на [пользовательской документации,](../user-help/user-help-security-info-overview.md) чтобы подготовить пользователей к новому опыту и помочь обеспечить успешное развертывание.

Комбинированная регистрация информации о безопасности Azure AD в настоящее время недоступна для национальных облаков, таких как Azure US Government, Azure Germany или Azure China 21Vianet.

Новый мой профиль согласуется с внешний вид комбинированной регистрации и обеспечивает бесшовный опыт для пользователей. Пользователи могут видеть мой [https://myprofile.microsoft.com](https://myprofile.microsoft.com)профиль, перейдя на .

## <a name="methods-available-in-combined-registration"></a>Методы, доступные при комбинированной регистрации

Комбинированная регистрация поддерживает следующие методы и действия аутентификации:

|   | Зарегистрировать | Change | DELETE |
| --- | --- | --- | --- |
| Microsoft Authenticator | Да (максимум 5) | нет | Да |
| Другое приложение аутентификатора | Да (максимум 5) | нет | Да |
| Токен оборудования | нет | нет | Да |
| Номер телефона | Да | Да | Да |
| Дополнительный телефон | Да | Да | Да |
| Рабочий телефон | нет | нет | нет |
| Email | Да | Да | Да |
| Контрольные вопросы | Да | нет | Да |
| Пароли приложений | Да | нет | Да |
| Ключи безопасности FIDO2<br />*Управляемый режим только со страницы [информации о безопасности](https://mysignins.microsoft.com/security-info)*| Да | Да | Да |

> [!NOTE]
> Пароли приложений доступны только для пользователей, которые были принуждены для мультифакторной аутентификации. Пароли приложений недоступны пользователям, включенным для мультифакторной аутентификации с помощью политики условного доступа.

Пользователи могут установить один из следующих вариантов в качестве метода многофакторной аутентификации по умолчанию:

- Microsoft Authenticator - уведомление.
- Приложение authenticator или аппаратный токен - код.
- Телефонный звонок.
- Текстовое сообщение.

По мере того, как мы продолжаем добавлять в Azure AD больше методов проверки подлинности, эти методы будут доступны в комбинированной регистрации.

## <a name="combined-registration-modes"></a>Комбинированные режимы регистрации

Существует два способа комбинированной регистрации: прерывание и управление.

- **Режим прерывания** — это опыт, подобный мастеру, представленный пользователям при регистрации или обновлении информации о безопасности при вхвочённом веском.
- **Режим управления** является частью профиля пользователя и позволяет пользователям управлять своей информацией о безопасности.

Для обоих режимов пользователи, которые ранее зарегистрировали метод, который может быть использован для мультифакторной аутентификации, должны будут выполнить Многофакторную аутентификацию, прежде чем они смогут получить доступ к информации о безопасности.

### <a name="interrupt-mode"></a>Режим прерывания

Комбинированная регистрация уважает как многофакторную аутентификацию, так и политики SSPR, если и то, и другое включено для вашего арендатора. Эти политики контролируют, прерывается ли пользователь для регистрации во время вступления в систему и какие методы доступны для регистрации.

Вот несколько сценариев, в которых пользователям может быть предложено зарегистрироваться или обновить информацию о безопасности:

- Многофакторная регистрация аутентификации осуществляется через Identity Protection: Пользователям предлагается зарегистрироваться во время вхинга. Они регистрируют многофакторные методы аутентификации и методы SSPR (если пользователь включен для SSPR).
- Многофакторная регистрация аутентификации осуществляется через мультифакторную аутентификацию для пользователей: пользователям предлагается зарегистрироваться во время входного ввена. Они регистрируют многофакторные методы аутентификации и методы SSPR (если пользователь включен для SSPR).
- Регистрация мультифакторной аутентификации осуществляется через Условный доступ или другие политики: пользователям предлагается зарегистрироваться при использовании ресурса, требующего многофакторной аутентификации. Они регистрируют многофакторные методы аутентификации и методы SSPR (если пользователь включен для SSPR).
- Регистрация SSPR принудительная: Пользователям предлагается зарегистрироваться во время входиний. Они регистрируют только методы SSPR.
- Обновление SSPR применяется: пользователи обязаны просматривать свои данные о безопасности с интервалом, установленным админом. Пользователям показывают свою информацию и могут подтвердить текущую информацию или внести изменения, если это необходимо.

При принудительном оформлении регистрации пользователям отображается минимальное количество методов, необходимых для соответствия политикам Multi-Factor Authentication и SSPR, от большинства до наименее безопасных.

Пример:

- Пользователь включен для SSPR. Политика SSPR требовала сбросить два метода и включила код мобильного приложения, электронную почту и телефон.
   - Этот пользователь должен зарегистрировать два метода.
      - Пользователю по умолчанию отображается приложение и телефон аутентификатора.
      - Пользователь может зарегистрировать электронную почту вместо приложения или телефона аутентификатора.

В этом графике описывается, какие методы отображаются пользователю при прерывании регистрации во время входе в систему:

![Комбинированная информация о безопасности flowchart](media/concept-registration-mfa-sspr-combined/combined-security-info-flow-chart.png)

Если у вас включена мультифакторная аутентификация и SSPR, мы рекомендуем вам обеспечить регистрацию мультифакторной аутентификации.

Если политика SSPR требует от пользователей регулярно просматривать свои данные о безопасности, пользователи прерываются при входе в систему и отображаются все зарегистрированные методы. Они могут подтвердить текущую информацию, если она актуальна, или они могут внести изменения, если это необходимо. Пользователи должны выполнять многофакторную аутентификацию при доступе к этой странице.

### <a name="manage-mode"></a>Режим управления

Пользователи могут получить доступ [https://aka.ms/mysecurityinfo](https://aka.ms/mysecurityinfo) к режиму управления, перейдя к или выбрав **информацию о безопасности** из моего профиля. Оттуда пользователи могут добавлять методы, удалять или изменять существующие методы, изменять метод по умолчанию и многое другое.

## <a name="key-usage-scenarios"></a>Ключевые сценарии использования

### <a name="set-up-security-info-during-sign-in"></a>Настройка сведений для защиты при входе в систему

Админ принудительной регистрации.

Пользователь не настроил всю необходимую информацию о безопасности и перешел на портал Azure. После ввода имени пользователя и пароля пользователю предлагается настроить информацию о безопасности. Затем пользователь следует за шагами, показанным в мастере, чтобы настроить требуемую информацию о безопасности. Если эти настройки позволяют, пользователь может настроить другие методы, понесемые по умолчанию. После завершения мастера пользователи просматривают настроимые методы и методы по умолчанию для мультифакторной аутентификации. Для завершения процесса настройки пользователь подтверждает информацию и продолжает на портале Azure.

### <a name="set-up-security-info-from-my-profile"></a>Настройка информации о безопасности из моего профиля

Админ не принудительной регистрации.

Пользователь, который еще не создал всю необходимую [https://myprofile.microsoft.com](https://myprofile.microsoft.com)информацию о безопасности идет на . Пользователь выбирает **информацию о безопасности** в левом стеле. Оттуда пользователь выбирает способ добавления метода, выбирает любой из доступных методов и следует шагам по настройке этого метода. После завершения пользователь видит метод, который только что был настроен на странице информации о безопасности.

### <a name="delete-security-info-from-my-profile"></a>Удаление информации о безопасности из моего профиля

Пользователь, который ранее настроил по крайней [https://aka.ms/mysecurityinfo](https://aka.ms/mysecurityinfo)мере один метод переходит к . Пользователь выбирает удалить один из ранее зарегистрированных методов. После завершения пользователь больше не видит этот метод на странице информации о безопасности.

### <a name="change-the-default-method-from-my-profile"></a>Изменение метода по умолчанию из моего профиля

Пользователь, который ранее создал по крайней мере один метод, который может [https://aka.ms/mysecurityinfo](https://aka.ms/mysecurityinfo)быть использован для многофакторной аутентификации переходит к . Пользователь изменяет текущий метод по умолчанию на другой метод по умолчанию. После завершения пользователь видит новый метод по умолчанию на странице информации о безопасности.
