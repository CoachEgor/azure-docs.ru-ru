# <a name="deploy-azure-ad-self-service-password-reset"></a>Развертывание Azure AD самостоятельного сброса пароля

Самостоятельный сброс пароля (SSPR) — это функция Azure Active Directory, позволяющей сотрудникам сбрасывать свои пароли без необходимости обратитесь к ИТ-специалистам. Сотрудников необходимо зарегистрировать для или быть зарегистрирован для самостоятельно сбрасывать пароль перед использованием службы. Во время регистрации сотрудник выбирает один или несколько методов проверки подлинности, включаемые в своей организации.

SSPR позволяет сотрудникам быстро разблокировки и продолжить работу независимо от того, где они находятся и время суток. Позволяя пользователям разблокировать себя, вашей организации можно сократить непроизводительное время и затраты на высокий уровень поддержки для наиболее распространенных проблем, связанные с паролем.

Предоставьте пользователям возможность быстро зарегистрироваться, развернув SSPR вместе с другим приложением или службой в вашей организации. Это действие создает большое количество входов и определят требования регистрации.

Прежде чем развертывать SSPR, организации может потребоваться определить, сколько сброса пароля связанных обращений произойдет временем и средняя стоимость каждого вызова. Они могут использовать этот данных после развертывания для отображения значения, что SSPR предоставляет вашей организации.  

## <a name="how-sspr-works"></a>Как работает SSPR

1. Когда пользователь пытается сбросить пароль, они проверяют их ранее зарегистрированных метод или методы, чтобы подтвердить свою личность.
1. Затем пользователь вводит новый пароль.
   1. Для облачных пользователей новый пароль хранится в Azure Active Directory. Дополнительные сведения см. в статье [как самостоятельный](concept-sspr-howitworks.md#how-does-the-password-reset-portal-work).
   1. Для гибридных пользователей пароль записывается обратно в локальной Active Directory с помощью службы Azure AD Connect. Дополнительные сведения см. в статье [новые возможности обратной записи паролей](concept-sspr-writeback.md#how-password-writeback-works).

## <a name="licensing-considerations"></a>Сведения о лицензировании

Azure Active Directory — смысл лицензии на пользователя, каждый пользователь должен иметь соответствующая лицензия для функции, которые они используют.

- Самостоятельного сброса пароля для облачных пользователей доступна с Azure AD Basic или более поздней.
- Самостоятельный сброс пароля с локальной обратной записи для гибридных сред требуется Azure AD Premium P1 или более поздней версии.

Дополнительные сведения о лицензировании можно найти на [странице цен на Azure Active Directory](https://azure.microsoft.com/pricing/details/active-directory/)

### <a name="sspr-properties-recommendations"></a>Рекомендации свойства SSPR

При включении функции самостоятельного сброса пароля, выберите группу безопасности, используемый во время пилотной программы.

Если вы планируете запустить службу, более широко, рекомендуется использовать параметр «все» для принудительного применения SSPR для всех пользователей в организации. Если вы не присвоено значение all, группы безопасности выберите соответствующие Azure AD или AD синхронизируется с Azure AD.

### <a name="authentication-methods"></a>Методы проверки подлинности

Задайте методы проверки подлинности, необходимые для регистрации для по крайней мере на единицу больше, чем количество, необходимое для выполнения сброса. Позволяя нескольким обеспечивает пользователям гибкость при им потребуется сбросить.

Задайте **несколько методов, необходимых для сброса** до уровня, подходящие для вашей организации. Один требует минимальных сопротивление, хотя две может повысить уровень безопасности.

См. в разделе [методы проверки подлинности](concept-authentication-methods.md) подробные сведения для проверки подлинности, которые методы доступны для SSPR, предварительно определенные секретные вопросы и как создать настраиваемые контрольные вопросы.

### <a name="registration-settings"></a>Параметры регистрации

Задайте **требовать регистрацию пользователей при входе в** для **Да**. Этот параметр означает, что пользователи принуждаются к регистрации при входе, гарантируя, что все пользователи будут защищены.

Задайте **число дней до пользователям будет предложено повторно подтвердить данные проверки подлинности** для между **90** и **180** дней, если вашей организации есть компании необходимости для более короткий интервал времени.

### <a name="notifications-settings"></a>Параметры уведомлений

Настроить оба **уведомлять пользователей о сбросе пароля** и **уведомлять всех администраторов, если другие Администраторы сбрасывают свои пароли** для **Да**. Выбрав **Да** на обоих повышает уровень безопасности, гарантируя, что пользователи извещены, когда пароль был сброшен и что всем администраторам известно, когда администратор изменяет пароль. Если пользователи или Администраторы получать такие уведомления, и они не инициировали изменения, они сразу же сообщить потенциальные нарушения безопасности.

### <a name="customization"></a>Настройка

Крайне важно настроить **службы технической поддержки по электронной почте или URL-адрес** для обеспечения пользователям, которые проблем можно быстро получить справку. Для этого параметра общий адрес электронной почты службы технической поддержки или веб-страницы, который пользователи уже есть опыт.

### <a name="on-premises-integration"></a>Интеграция с локальной средой

Если у вас есть гибридная среда, убедитесь, что **обратная запись паролей в локальный AD** присваивается **Да**. Также можно задайте, чтобы разрешить пользователям разблокировать учетную запись без сброса пароля Yes, поскольку он даст им большую гибкость.

### <a name="changingresetting-passwords-of-administrators"></a>Изменение или сброс паролей для администраторов

Учетные записи администратора — специальные учетные записи с повышенным уровнем разрешений. Чтобы защитить их, чтобы изменение паролей администраторов применяются следующие ограничения:

- Администраторы предприятия в локальной или Администраторы домена не может сбросить свой пароль через SSPR. Их можно изменить только свой пароль в своей локальной среде. Таким образом мы рекомендуем не выполняется синхронизация локальных учетных записей администраторов AD в Azure AD.
- Администратор не может использовать секретные вопросы и ответы как метод для сброса пароля.

### <a name="environments-with-multiple-identity-management-systems"></a>Среды с помощью нескольких систем управления удостоверениями

При наличии нескольких удостоверений систем управления в среде, такие как диспетчеры удостоверений на предприятии, такие как Oracle AM, SiteMinder, или других систем, а затем пароли, записи в Active Directory может должны быть синхронизированы в другие системы, с помощью такого средства, как Пароль службы уведомлений о смене (PCNS) с помощью Microsoft Identity Manager (MIM). Чтобы найти сведения об этом более сложные сценарии, см. в статье [развертывание службы уведомлений о смене паролей MIM на контроллере домена](https://docs.microsoft.com/microsoft-identity-manager/deploying-mim-password-change-notification-service-on-domain-controller).

Реализация выполняется в три этапа:

- Настройка пользователей и лицензии
- Настройка Azure AD SSPR для регистрации и самообслуживания
- Настройка Azure AD Connect для обратной записи паролей

### <a name="communicate-the-change"></a>Сообщить об изменении

Начните реализацию план обмена информацией, который вы разработали на этапе планирования.

### <a name="ensure-groups-are-created-and-populated"></a>Убедитесь, создаются и заполняются группы

Ссылки в разделе методов проверки подлинности пароля планирование и гарантировать доступность в группах для реализации пилотного проекта или рабочей среде, и все соответствующие пользователи добавляются в группы.

### <a name="apply-licenses"></a>Применение лицензии

Нужно реализовать группы должны иметь Azure AD назначенную им лицензию уровня "премиум". Вы можете назначать лицензии непосредственно к группе, или можно использовать существующих политик лицензии как с помощью PowerShell или группового лицензирования.

### <a name="configure-sspr"></a>Настройка SSPR

#### <a name="enable-groups-for-sspr"></a>Включите группы для SSPR

1. Доступ к порталу Azure с учетной записью администратора.
1. Выберите все службы и в поле фильтра введите Azure Active Directory и выберите Azure Active Directory.
1. В колонке "Active Directory" выберите сброс пароля.
1. В панели «Свойства» выберите выбранные. Если вы хотите, чтобы все пользователи включена, выбрать все.
1. Пароль по умолчанию сбросить колонка "политика", введите имя первой группы, выберите его и затем нажмите кнопку выбрать в нижней части экрана и выберите для сохранения в верхней части экрана.
1. Повторите эту процедуру для каждой группы.

#### <a name="configure-the-authentication-methods"></a>Настройка методов проверки подлинности

Ссылаться на планирования из раздела «Планирование методов проверки подлинности пароль» в этом документе.

1. Выберите регистрацию, в разделе пользователь должен зарегистрировать при входе в систему, выберите "Да" и задайте число дней до истечения срока действия и нажмите кнопку Save.
1. Выберите уведомление и настроить на ваш план и нажмите кнопку Save.
1. Выберите настройки и настроить на ваш план и нажмите кнопку Save.
1. Выберите локальную интеграцию и настроить на ваш план и нажмите кнопку Save.

Сведения и действия, чтобы настроить эту возможность можно найти в статье [на экране входа в сброса пароля Azure AD](tutorial-sspr-windows.md)

| Роль последующих бизнеса | Роль Azure AD (при необходимости) |
| :---: | :---: |
| Службы поддержки уровня 1 | Администратор паролей |
| Службы поддержки уровня 2 | Администратор пользователей |
| Администратор SSPR | Глобальный администратор. |

### <a name="support-scenarios"></a>Сценарии поддержки

Чтобы включить ваш успех группы поддержки, можно создать вопросы и ответы на вопросы, получаемых от пользователей основе. В следующей таблице приведены распространенные сценарии поддержки.

| Сценарии | Описание |
| --- | --- |
| Пользователь не имеет зарегистрированных методы проверки подлинности доступны | Пользователь пытается сбросить пароль, но нет других методов проверки подлинности, что они зарегистрированы доступных (пример: они оставить их сотовый телефон дома и не может получить доступ к электронной почте) |
| Пользователь не является получение текстового или вызывать в их рабочий или мобильный телефон | Пользователь пытается пройти проверку подлинности через SMS или звонок, но не получает текст/вызов. |
| Пользователь не может получить доступ к портала сброса паролей | Пользователь хочет сбросить пароль, но не включена для сброса пароля и таким образом, невозможно просмотреть страницу, чтобы обновить пароли. |
| Пользователь не может задать новый пароль | Пользователь завершает проверку во время процесс сброса пароля, но не может установить новый пароль. |
| Пользователь не видит ссылку сброса пароля на устройстве Windows 10 | Пользователь пытается сбросить пароль с экрана блокировки Windows 10, но устройство либо не присоединено к Azure AD или не включена политика устройств Intune |

Можно также включать сведения, например следующие действия для устранения неисправностей.

- Какие группы используются для SSPR.
- Какие методы проверки подлинности, настроены.
- Политики доступа, связанные с по или корпоративной сети.
- Действия по устранению неполадок для распространенных сценариев.

Можно также ссылаться на нашей электронной документации по устранению неполадок самостоятельного сброса пароля для понять общие действия по устранению неполадок для наиболее распространенных сценариев SSPR.
