[Self-Service Password Reset (SSPR)](https://www.youtube.com/watch?v=tnb2Qf4hTP8) — это функция Active Directory (AD), которая позволяет пользователям сбросить свои пароли без обращения за помощью к ИТ-персоналу. Пользователи могут быстро разблокировать себя и продолжать работать независимо от того, где они находятся или время суток. Позволяя сотрудникам разблокировать себя, ваша организация может сократить непродуктивное время и высокие затраты на поддержку наиболее распространенных проблем, связанных с паролем.

SSPR имеет следующие ключевые возможности:

* Самообслуживание позволяет конечным пользователям сбросить свои просроченные или неистекающие пароли без обращения к администратору или службе поддержки для поддержки.
* [Пароль Списания](https://docs.microsoft.com/azure/active-directory/authentication/concept-sspr-writeback) позволяет управление локальных паролей и разрешение блокировки учетной записи, хотя облако.
* Отчеты об управлении паролями дают администраторам представление о сбое паролей и регистрационных действиях, происходящих в их организации.

В этом руководстве по развертыванию показано, как планировать, а затем тестировать развертывание SSPR.

Чтобы быстро увидеть SSPR в действии, а затем вернуться, чтобы понять дополнительные соображения развертывания:

Узнайте больше о SSPR. [Посмотрите, как это работает: сбросить пароль самообслуживания Azure AD.](https://docs.microsoft.com/azure/active-directory/authentication/concept-sspr-howitworks)

### <a name="key-benefits"></a>Основные преимущества

Ключевыми преимуществами обеспечения SSPR являются:

* **Управление затратами.** SSPR снижает затраты на ИТ-поддержку, позволяя пользователям самостоятельно сбросить пароли. Это также снижает стоимость потерянного времени из-за потерянных паролей и блокировок. 

* **Интуитивное взаимодействие с пользователем**. Он обеспечивает интуитивно понятный одноразовый процесс регистрации пользователей, который позволяет пользователям сбросить пароли и разблокировать учетные записи по требованию с любого устройства или местоположения. SSPR позволяет пользователям быстрее вернуться к работе и быть более продуктивными.

* **Гибкость и безопасность**. SSPR позволяет предприятиям получить доступ к безопасности и гибкости, которые обеспечивает облачная платформа. Администраторы могут изменять настройки с учетом новых требований безопасности и распоряжаться этими изменениями для пользователей, не нарушая их ввоза.

* **Надежный аудит и отслеживание использования**. Организация может гарантировать, что бизнес-системы остаются безопасными, в то время как ее пользователи сбросить свои собственные пароли. Надежные журналы аудита содержат информацию о каждом этапе процесса сбросить пароль. Эти журналы доступны в API и позволяют пользователю импортировать данные в систему мониторинга инцидентов безопасности и событий (SIEM).

### <a name="solution-architecture"></a>Архитектура решения

Ниже приводится пример архитектуры решения для сбросить пароль для общих гибридных сред.

![диаграмма архитектуры решений](./media/howto-sspr-deployment//solutions-architecture.png)

Описание рабочего процесса

Чтобы сбросить пароль, пользователи переходят на [портал сготовона паролей.](https://aka.ms/sspr) Они должны проверить ранее зарегистрированный метод проверки подлинности или методы, чтобы доказать свою личность. Если они успешно сбросить пароль, они начинают процесс сбросить.

* Для пользователей, накоторыехающих только в облаке, SSPR хранит новый пароль в Azure AD. 

* Для гибридных пользователей SSPR записывает пароль к каталогу On-prem Active Directory через службу Azure AD Connect. 

Примечание: Для пользователей, которые имеют пароль [синхронизации хэша (PHS)](https://docs.microsoft.com/azure/active-directory/hybrid/whatis-phs) отключен, SSPR хранит пароли в on-prem Active Directory только.

### <a name="authentication-methods"></a>Методы проверки подлинности

При включении SSPR пользователи могут сбросить пароль только в том случае, если у них есть данные, присутствующие в методах проверки подлинности, которые администратор включил. Методы включают телефон, уведомление приложения Authenticator, вопросы безопасности и т.д. Для получения дополнительной [What are authentication methods?](https://docs.microsoft.com/azure/active-directory/authentication/concept-authentication-methods)информации см.

Мы рекомендуем следующие настройки метода аутентификации:

* Установите **методы аутентификации, необходимые для регистрации,** по крайней мере на один больше, чем число, необходимое для сбросить. Разрешение нескольких аутентификаций дает пользователям гибкость при смножестве.

* Установите **количество методов, необходимых для сбросить** до уровня, соответствующего вашей организации. Один требует наименьшего трения, в то время как два могут увеличить вашу позу безопасности. 

Примечание: Пользователь должен иметь методы проверки подлинности, настроенные в [политиках Паролей и ограничения в каталоге Azure Active.](https://docs.microsoft.com/azure/active-directory/authentication/concept-sspr-policy)

### <a name="registration-settings"></a>Параметры регистрации

Установить **Требование пользователей зарегистрироваться при вхвочне в** **Yes**. Эта настройка требует, чтобы пользователи регистрировались при входной связи, гарантируя, что все пользователи защищены.

Установите **количество дней, прежде чем пользователям будет предложено подтвердить свою информацию о проверке подлинности,** до **90-180** дней, если только у вашей организации нет необходимости в более коротких сроках. **180**

### <a name="notifications-settings"></a>Настройки уведомлений

Настроили **как пользователей Notify на сброс пароля,** так и **уведомите всех админинов, когда другие админы сбросили свой пароль** к **Yes.** Выбор **Yes** на обоих повышает безопасность, гарантируя, что пользователи знают, когда их пароль сбросить. Это также гарантирует, что все админы знают, когда админ меняет пароль. Если пользователи или админы получают уведомление и не инициировали изменение, они могут немедленно сообщить о потенциальной проблеме безопасности.

### <a name="customization-settings"></a>Настройки настройки

Очень важно настроить электронную почту или URL-адрес службы поддержки, чтобы пользователи, испытывающие проблемы, могли немедленно получить помощь. Установите эту опцию на общий адрес электронной почты справочной службы или веб-страницу, с которой знакомы ваши пользователи. 

Для получения дополнительной [информации](https://docs.microsoft.com/azure/active-directory/authentication/concept-sspr-customization)см.

### <a name="password-writeback"></a>Обратная запись паролей

**Сноверный пароль** включен с [помощью Azure AD Connect](https://docs.microsoft.com/azure/active-directory/hybrid/whatis-hybrid-identity) и записывает сброс пароля в облаке обратно в существующий каталог в режиме реального времени. Для получения дополнительной информации, смотрите [Что такое пароль списания?](https://docs.microsoft.com/azure/active-directory/authentication/concept-sspr-writeback)

Мы рекомендуем следующие настройки:

* Убедитесь, что **списать пароли к предварительной AD** установлен **на Да**. 
* Установите **Разрешите пользователям разблокировать учетную запись без сброса пароля** **на Yes**.

По умолчанию Azure AD разблокирует учетные записи при выполнении сброса пароля.

### <a name="administrator-password-setting"></a>Настройка пароля администратора

Учетные записи администраторов имеют повышенные разрешения. Администраторы на месте предприятия или домена не могут сбросить свои пароли через SSPR. Учетные записи на месте имеют следующие ограничения:

* может изменить свой пароль только в среде на прем.
* никогда не может использовать секретные вопросы и ответы в качестве способа сбросить свой пароль.

Мы рекомендуем не синхронизировать учетные записи ремин-ремин на prem Active Directory с Azure AD.

### <a name="environments-with-multiple-identity-management-systems"></a>Среды с несколькими системами управления идентификационными данными

Некоторые среды имеют несколько систем управления идентификацией. Руководители личных данных, такие как Oracle AM и SiteMinder, требуют синхронизации с AD для паролей. Это можно сделать с помощью такого инструмента, как Служба уведомления об изменении пароля (PCNS) с помощью менеджера идентификации Майкрософт (MIM). Чтобы найти информацию об этом более [Deploy the MIM Password Change Notification Service on a domain controller](https://docs.microsoft.com/microsoft-identity-manager/deploying-mim-password-change-notification-service-on-domain-controller)сложном сценарии, см.

### <a name="plan-support"></a>Поддержка плана

Хотя SSPR обычно не создает проблем с пользователями, важно подготовить вспомогательный персонал для решения проблем, которые могут возникнуть. В то время как администратор может сбросить пароль для конечных пользователей через портал Azure AD, лучше помочь решить проблему с помощью процесса самообслуживания.

Чтобы обеспечить успех вашей группы поддержки, вы можете создать часто задаваемые вопросы на основе вопросов, которые вы получаете от ваших пользователей. Вот несколько таких случаев.

| Сценарии| Описание |
| - | - |
| У пользователя нет доступных зарегистрированных методов проверки подлинности| Пользователь пытается сбросить свой пароль, но не имеет ни одного из методов проверки подлинности, которые они зарегистрировали (Пример: они оставили свой мобильный телефон дома и не могут получить доступ к электронной почте) |
| Пользователь не получает смС или вызова на свой офис или мобильный телефон| Пользователь пытается проверить свою личность с помощью текста или вызова, но не получает текст /звонок. |
| Пользователь не может получить доступ к порталу сготовона паролей| Пользователь хочет сбросить свой пароль, но не включен для сбросить пароль и не может получить доступ к странице для обновления паролей. |
| Пользователь не может установить новый пароль| Пользователь завершает проверку во время потока сбросить пароль, но не может установить новый пароль. |
| Пользователь не видит ссылку на пароль снавпана на устройстве Windows 10| Пользователь пытается сбросить пароль с экрана блокировки Windows 10, но устройство либо не присоединилось к Azure AD, либо политика устройства Intune не включена |

### <a name="plan-rollback"></a>Откат плана

Для отката развертывания:

* для одного пользователя удалите пользователя из группы безопасности 

* для группы удалите группу из конфигурации SSPR

* Для всех отожмите SSPR для арендатора Azure AD

## <a name="manage-sspr"></a>Управление SSPR

Azure AD может предоставить дополнительную информацию о производительности SSPR с помощью аудитов и отчетов.

### <a name="password-management-activity-reports"></a>Отчеты об управлении паролями 

Для измерения производительности SSPR можно использовать готовые отчеты на портале Azure. При наличии соответствующей лицензии можно также создавать пользовательские запросы. Для получения дополнительной [Reporting options for Azure AD password management](https://docs.microsoft.com/azure/active-directory/authentication/howto-sspr-reporting) информации см.

> [!NOTE]
>  Вы должны быть [глобальным администратором,](https://docs.microsoft.com/azure/active-directory/users-groups-roles/directory-assign-admin-roles)и вы должны отказаться от сбора этих данных для вашей организации. Чтобы зарегистрироваться, необходимо хотя бы один раз посетить вкладку отчетов или журналаудита на портале Azure. До тех пор данные не собирают для вашей организации.

Аудиторские журналы для регистрации и сбросить пароль доступны в течение 30 дней. Если аудит безопасности в корпорации требует более длительного хранения, журналы должны быть экспортированы и использованы в инструмент SIEM, такой как [Azure Sentinel,](https://docs.microsoft.com/azure/sentinel/connect-azure-active-directory)Splunk или ArcSight.
