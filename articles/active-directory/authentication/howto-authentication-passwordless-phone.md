# <a name="enable-passwordless-sign-in-with-the-microsoft-authenticator-app-preview"></a>Включить без пароль войти в приложение Microsoft Authenticator (предварительный просмотр)

Приложение Microsoft Authenticator позволяет войти в любую учетную запись Azure AD, не используя пароль. Как и аналогичная технология [Windows Hello для бизнеса](/windows/security/identity-protection/hello-for-business/hello-identity-verification), Microsoft Authenticator использует аутентификацию на основе ключей, привязывает учетные данные пользователя к определенному устройству и применяет биометрические данные или ПИН-код. Этот метод проверки подлинности может быть использован на любой платформе устройства, включая мобильный, а также с любым приложением или веб-сайтом, который интегрируется с библиотеками аутентификации Майкрософт. 

![Пример входиного пользователя браузера с просьбой одобрить входе](./media/howto-authentication-passwordless-phone/phone-sign-in-microsoft-authenticator-app.png)

Вместо того, чтобы видеть запрос на пароль после ввода имени пользователя, человек, который включил телефон, входящий в приложение Microsoft Authenticator, увидит сообщение, в котором им будет спроявлеться номер в приложении. Пользователь должен передать этот номер в приложение, нажать кнопку подтверждения и предоставить ПИН-код или биометрические данные, чтобы завершить процесс аутентификации.

> [!NOTE]
> Эта возможность находится в приложении Microsoft Authenticator с марта 2017 года, поэтому существует вероятность того, что при включении политики для каталога пользователи могут немедленно столкнуться с этим потоком и увидеть сообщение об ошибке, если они не были включены политикой. Учитывайте это и не забудьте предупредить пользователей.

## <a name="prerequisites"></a>Предварительные требования

- Многофакторная аутентификация Azure с push-уведомлениями, разрешенными в качестве метода проверки 
- Последняя версия Microsoft Authenticator на устройствах под управлением iOS версии 8.0 или более поздней версии или Android 6.0 или более поздней версии.

> [!NOTE]
> Если вы включили предыдущее приложение Microsoft Authenticator без паролей, не вспомненное с помощью Azure AD PowerShell, оно было включено для всего каталога. Если вы позволите использовать этот новый метод, он будет заменять политику PowerShell. Мы рекомендуем включить всех пользователей в вашем арендатора с помощью новых методов аутентификации, в противном случае пользователи, не включая новую политику, больше не смогут войти в систему без пароль. 

## <a name="enable-passwordless-authentication-methods"></a>Включить методы проверки подлинности без паролей

### <a name="enable-the-combined-registration-experience"></a>Включить опыт комбинированной регистрации

Функции регистрации методов проверки подлинности без паролей опираются на комбинированный предварительный просмотр регистрации. Следуйте шагам в статье [Включить комбинированную регистрацию информации безопасности (предварительный просмотр),](howto-registration-mfa-sspr-combined.md)чтобы включить комбинированный предварительный просмотр регистрации.

### <a name="enable-passwordless-phone-sign-in-authentication-methods"></a>Включить методы проверки подлинности без паролей телефона

1. Войти на [портал Azure](https://portal.azure.com)
1. Найдите и выберите *Azure Active Directory*. Выберите**методы проверки подлинности** >  **безопасности** > **политики метода проверки (Предварительный просмотр)**
1. Под **Passwordless телефон входе в**систему, выбрать следующие варианты
   1. **Включить** - Да или Нет
   1. **Цель** - Все пользователи или выберите пользователей
1. **Сохранить** для установки новой политики

## <a name="user-registration-and-management-of-microsoft-authenticator-app"></a>Регистрация пользователей и управление приложением Microsoft Authenticator

1. Просмотр[https://aka.ms/mysecurityinfo](https://aka.ms/mysecurityinfo)
1. Войти, если не уже
1. Добавить приложение аутентификатора, нажав метод **добавления,** выбрав приложение Authenticator и нажав **Добавить** **Add method**
1. Следуйте инструкциям по установке и настройке приложения Microsoft Authenticator на вашем устройстве
1. Нажмите **Сделано,** чтобы завершить поток настройки приложения Authenticator MFA. 
1. В **Microsoft Authenticator**выберите **Включить телефон в входе из** меню выпадения учетной записи
1. Следуйте инструкциям в приложении, чтобы закончить регистрацию для безпадоного телефонного ввоза. 

Организации могут указать своим пользователям на статью [Войти с вашего телефона,](../user-help/microsoft-authenticator-app-phone-signin-faq.md) а не пароль для дальнейшей помощи в настройке в приложении Microsoft Authenticator и включение телефонного входе. Для того, чтобы применить эти настройки, возможно, потребуется выйти из системы и войти обратно в арендатора. 

## <a name="sign-in-with-passwordless-credential"></a>Вопинье с учетными данными без паролей

На этапе общедоступной предварительной версии у вас нет возможности обязать пользователей создавать или использовать новый тип учетных данных. Пользователь столкнется без паролей, встретившись только после того, как админ включил своего клиента, **и** пользователь обновил свое приложение Microsoft Authenticator, чтобы включить телефон в систему.

После ввода вашего имени пользователя в Интернете и выбора **Next,** пользователям предоставляется номер и предлагается в их приложении Microsoft Authenticator, чтобы выбрать соответствующий номер для проверки подлинности вместо использования пароля. 

![Пример входиного браузера с помощью приложения Microsoft Authenticator](./media/howto-authentication-passwordless-phone/web-sign-in-microsoft-authenticator-app.png)

## <a name="known-issues"></a>Известные проблемы

### <a name="user-is-not-enabled-by-policy-but-still-has-passwordless-phone-sign-in-method-in-microsoft-authenticator"></a>Пользователь не включен политикой, но по-прежнему имеет метод вхлажения без паролей телефона в Microsoft Authenticator

Вполне возможно, что пользователь в какой-то момент создал учетные данные без паролей телефона в своем текущем приложении Microsoft Authenticator, или на более раннем устройстве. Как только админ развяжет политику метода проверки подлинности для входе телефона без паролей, любой пользователь, имеющем зарегистрированный учетный акт, начнет испытывать новый запрос на входе, независимо от того, были ли они включены в использование политики или нет. Если пользователю не разрешили использовать учетные данные по политике, он увидит ошибку после завершения потока проверки подлинности. 

Админ может выбрать возможность использовать пароль без паролем телефон входе, или пользователь должен удалить метод. Если у пользователя больше нет зарегистрированного устройства, он может пойти [https://aka.ms/mysecurityinfo](https://aka.ms/mysecurityinfo) и удалить его. Если они все еще используют Authenticator для МИД, они могут выбрать **телефон-ввод от Disable** из Microsoft Authenticator.  

### <a name="ad-fs-integration"></a>Интеграция AD FS

Когда пользователь включит учетные данные Microsoft Authenticator без пароля, аутентификация этого пользователя по умолчанию всегда будет выполняться через уведомление для подтверждения. Эта логика предотвращает перенаправление пользователя гибридного клиента в ADFS для проверки входа, если этот пользователь не выбрал действие "Использовать свой пароль". Этот процесс также будут обходить любые локальные политики условного доступа и потоки сквозной проверки подлинности. 

Если у пользователя есть неотвеченная проверка входа по телефону без ответа и попытки войти снова, пользователь может быть доставлен в ADFS, чтобы ввести пароль вместо этого.  

### <a name="azure-mfa-server"></a>Сервер Azure MFA

Конечные пользователи, включенные для MFA через внутренний сервер Azure MFA организации, могут по-прежнему создавать и использовать один телефонный знак без паролей в учетных данных. Если пользователь введет одну учетную запись в несколько (5 и более) установок Microsoft Authenticator, это может привести к ошибке.  

### <a name="device-registration"></a>Регистрация устройства

Одним из предварительных условий для создания этой новой сильной учетной записи является то, что устройство, на котором установлено приложение Microsoft Authenticator, также должно быть зарегистрировано в рамках арендатора Azure AD для отдельного пользователя. В связи с действующими ограничениями на регистрацию устройств устройство может быть зарегистрировано только у одного арендатора. Это означает, что в приложении Microsoft Authenticator можно настроить только одну рабочую или учебную учетную запись для входа с телефона.

### <a name="intune-mobile-application-management"></a>Управление мобильными приложениями Intune 

Конечные пользователи, на которых распространяется политика, требующая управления мобильными приложениями (MAM), не могут зарегистрировать учетные данные без паролей в приложении Microsoft Authenticator. 

> [!NOTE]
> Регистрация устройств не такая же, как управление устройствами или "MDM". Он связывает идентификатор устройства и идентификатор пользователя только вместе в каталоге Azure AD.
