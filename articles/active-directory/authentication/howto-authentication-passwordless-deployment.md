# <a name="plan-a-passwordless-authentication-deployment-in-azure-active-directory"></a>Планирование развертывания проверки подлинности без паролей в active-каталоге Azure

> [!NOTE]
> Чтобы создать офлайн-версию этого плана развертывания, используйте функцию PDF-версии браузера Print.

Большинство кибератак начинаются с скомпрометированного имени пользователя и пароля. Организации пытаются противостоять угрозе, требуя от пользователей использовать один из следующих подходов:

- Длинные пароли
- Сложные пароли
- Частые изменения пароля
- Многофакторная идентификация (MFA)

Исследования Корпорации Майкрософт [показывают,](https://aka.ms/passwordguidance) что эти усилия раздражают пользователей и приводят к повышанию расходов на поддержку. Для получения дополнительной информации, см [$word.](https://techcommunity.microsoft.com/t5/Azure-Active-Directory-Identity/Your-Pa-word-doesn-t-matter/ba-p/731984)

### <a name="benefits-of-passwordless-authentication"></a>Преимущества проверки подлинности без паролей

- **Повышенная безопасность**. Уменьшите риск фишинговых атак и атак спрей паролей, удалив пароли в качестве поверхности атаки.
-  **Улучшение пользовательского опыта.** Предоставь пользователям удобный способ доступа к данным из любого места. Обеспечить легкий доступ к приложениям и услугам, таким как Outlook, OneDrive или Office во время мобильных устройств.
-  **Надежные идеи**. Получите представление о деятельности пользователей без паролей с помощью надежного ведения журнала и аудита.

С passwordless, пароль заменяется чем-то у вас есть плюс то, что вы или что-то вы знаете. Например, Windows Hello for Business может использовать биометрический жест, например, лицо или отпечаток пальца, или PIN-код для конкретного устройства, который не передается по сети.

## <a name="passwordless-authentication-methods"></a>Методы безпадной аутентификации
Корпорация Майкрософт предлагает три варианта проверки подлинности без паролей, которые охватывают многие сценарии. Эти методы могут быть использованы в тандеме:

- [Windows Hello for Business](https://docs.microsoft.com/azure/security/fundamentals/ad-passwordless) лучше всего подходит для пользователей на выделенных компьютерах Windows.
- Ключевой вокбезопасности с [ключами безопасности FIDO2](https://docs.microsoft.com/azure/security/fundamentals/ad-passwordless) особенно полезен для пользователей, которые вписываются в общие машины, такие как киоски, в ситуациях, когда использование телефонов ограничено, и для высоко привилегированных идентификационных данных.
- Телефон, всданный в [приложение Microsoft Authenticator,](https://docs.microsoft.com/azure/security/fundamentals/ad-passwordless) полезен для предоставления пользователям с мобильными устройствами безпачной опции. Приложение Authenticator превращает любой iOS или Android телефон в надежный, без паролей учетных данных, позволяя пользователям войти в любую платформу или браузер. Пользователи войти, получив уведомление на свой телефон, сопоставляя номер отображается на экране, чтобы один на своем телефоне, а затем с помощью их биометрических данных или PIN-кода для подтверждения.

### <a name="technical-considerations-for-the-microsoft-authenticator-app"></a>Технические соображения для приложения Microsoft Authenticator

**Интеграция DD FS** - Когда пользователь позволяет учетным данным Microsoft Authenticator без паролей, аутентификация для этого пользователя по умолчанию отправляет уведомление для утверждения. Пользователи в гибридном арендаторе не могут быть направлены в ADFS для входним в ввод, если они не выберут "Использовать пароль вместо". Этот процесс также обходит любые политики условного доступа и сквозной аутентификации. Однако, если *login_hint* указан, пользователь направляется в ADFS и обойдя возможность использовать учетные данные без паролей.

**Сервер azure Multi-factor authentication** - Конечные пользователи, включенные для многофакторной аутентификации через местный сервер Azure MFA организации, могут создавать и использовать один учетный учетный данный без паролей. Если пользователь пытается обновить несколько установок (5 или более) учетных данных Microsoft Authenticator, это изменение может привести к ошибке.

**Регистрация устройств** - Для использования приложения Authenticator для проверки подлинности без паролей устройство должно быть зарегистрировано в арендаторе Azure AD и не может быть общим устройством. Устройство может быть зарегистрировано только в одном арендаторе. Это ограничение означает, что только одна рабочая или школьная учетная запись поддерживается для входного телефона с помощью приложения Authenticator.

### <a name="security-key-life-cycle"></a>Ключевой жизненный цикл безопасности

Ключи безопасности обеспечивают доступ к вашим ресурсам, и вы должны планировать управление этими физическими устройствами.

1. **Распределение ключей**: Планируйте, как предоставить ключи вашей организации. Возможно, у вас есть централизованный процесс подготовки или позволить конечным пользователям приобрести фидо 2.0-совместимые ключи.
1. **Активация ключа**: Конечные пользователи должны самостоятельно активировать ключ безопасности. Конечные пользователи регистрируют свои ключи безопасности [https://aka.ms/mysecurityinfo](https://aka.ms/mysecurityinfo) и позволяют использовать второй фактор (PIN или биометрический) при первом использовании.
1. **Отключение ключа:** В то время как функциональность ключа безопасности находится в стадии предварительного просмотра, администратор не может удалить ключ из учетной записи пользователя. Пользователь должен удалить его. Если ключ потерян или украден:
   1. Удалите пользователя из любой группы, включенной для проверки подлинности без паролей.
   1. Убедитесь, что они удалили ключ в качестве метода проверки подлинности.
   1. Выдать новый ключ. **Замена ключа:** Пользователи могут включить два ключа безопасности одновременно. При замене ключа безопасности убедитесь, что пользователь также удалил заменяемый ключ.

- [Включить провайдера учетных данных с пакетом подготовки](howto-authentication-passwordless-security-key-windows.md#enable-with-a-provisioning-package)
   - Если развертывание Intune невозможно, администраторы должны развернуть пакет на каждой машине, чтобы включить функциональность поставщика учетных данных. Установка упаковки может осуществляться одним из следующих вариантов:
      - Менеджер группы или настройки
      - Локальная установка на машине Windows 10
- [Включить провайдера учетных данных с помощью групповой политики](howto-authentication-passwordless-security-key-windows.md#enable-with-group-policy)
   - Поддерживается только для гибридных устройств Azure AD.

## <a name="plan-auditing-security-and-testing"></a>Планирование аудита, безопасности и тестирования
Планирование аудита, отвечающего вашим организационным и нормативным требованиям, является неотъемлемой частью развертывания.

### <a name="auditing-passwordless"></a>Аудит без паролей

Azure AD имеет отчеты, которые предоставляют технические и бизнес-информацию. Владельцы ваших бизнес-и технических приложений возьмут на себя право собственности на эти отчеты и будут использовать их в соответствии с требованиями организации.

Раздел методов **проверки подлинности** в портале Active Directory Azure — это место, где администраторы могут включать и управлять настройками учетных данных без паролей.

Azure AD добавляет записи в журналы аудита, когда:

- Админ вносит изменения в раздел «Методы аутентификации».
- Пользователь вносит какие-либо изменения в свои учетные данные в каталоге Azure Active.

В следующей таблице приводятся некоторые примеры типичных сценариев отчетности:

|   | Управление рисками. | Повышение производительности труда | Система управления и соответствие требованиям |
| --- | --- | --- | --- |
| **Типы отчетов** | Методы аутентификации - пользователи, зарегистрированные для комбинированной регистрации безопасности | Методы аутентификации - пользователи, зарегистрированные для уведомления приложения | Вход: просмотрите, кто и как имеет доступ к арендатору |
| **Потенциальные действия** | Целевые пользователи еще не зарегистрированы | Привод принятия Microsoft Authenticator приложение или ключи безопасности | Отмена доступа или применение дополнительных политик безопасности для админ-неусданным |

**Azure AD хранит большинство аудиторских данных** в течение 30 дней и делает данные доступными через портал Azure Admin или API для загрузки в системы анализа. Если требуется более длительное удержание, экспортите и потребляйте журналы в инструменте SIEM, таком как [Azure Sentinel,](../../sentinel/connect-azure-active-directory.md)Splunk или Sumo Logic. [Подробнее о просмотре отчетов о доступе и использовании.](../reports-monitoring/overview-reports.md)

Пользователи могут зарегистрироваться и управлять своими [https://aka.ms/mysecurityinfo](https://aka.ms/mysecurityinfo)учетными данными, перемещаясь в . Эта ссылка направляет пользователей к опыту управления учетными данными конечных пользователей, который был включен с помощью комбинированного опыта регистрации подлинности SSPR/Multi-factor. Azure AD регистрирует регистрацию устройств безопасности FIDO2 и изменяет методы проверки подлинности пользователем.

### <a name="plan-security"></a>Планирование безопасности
В рамках этого плана развертывания корпорация Майкрософт рекомендует, чтобы для всех привилегированных учетных записей админ-аккаунтов была включена проверка подлинности без паролей.

Когда пользователи включат или отключают учетную запись на ключе безопасности или сбросили второй фактор для ключа безопасности на своих компьютерах с Windows 10, запись добавляется в журнал безопасности и находится под следующими идентизациями событий: *4670* и *5382.*

### <a name="plan-for-rollback"></a>План отката

Хотя безпадочная аутентификация является легкой функцией с минимальным воздействием на конечных пользователей, может потребоваться откат.

Откат требует, чтобы администратор вписался на портал Active Directory Azure, выбрал желаемые сильные методы проверки подлинности и изменил опцию включения на **No.** Этот процесс отключает безпарольную функциональность для всех пользователей.

Пользователям, которые уже зарегистрировали устройства безопасности FIDO2, предлагается использовать устройство безопасности при следующем входе в систему, а затем увидеть следующую ошибку:
