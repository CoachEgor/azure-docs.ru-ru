# <a name="enable-on-premises-azure-active-directory-password-protection"></a>Включить предприимчивую защиту паролем Azure Active Directory

Пользователи часто создают пароли, которые используют общие местные слова, такие как школа, спортивная команда или известный человек. Эти пароли легко угадать, и слабы против словарных атак. Для обеспечения надежных паролей в вашей организации, Azure Active Directory (Azure AD) Password Protection предоставляет глобальный и пользовательский список запрещенных паролей. Запрос на изменение пароля не удается, если есть совпадение в этих запрещенных список паролей.

Для защиты среды Active Directory Domain Services (AD DS) можно установить и настроить Azure AD Password Protection для работы с вашим д-р на прем DC. В этой статье показано, как включить защиту паролем Azure AD для вашей среды.

Для получения дополнительной информации о том, как Azure AD Password Protection работает в предварительной среде, [см.](concept-password-ban-bad-on-premises.md)

## <a name="before-you-begin"></a>Перед началом

В этой статье показано, как включить защиту паролем Azure AD для вашей среды. Перед тем, как завершить эту статью, [установите и зарегистрируйте прокси-сервис Azure AD Password Protection и агентов постоянного тока](howto-password-ban-bad-on-premises-deploy.md) в среде AD DS.

## <a name="enable-on-premises-password-protection"></a>Включить защиту от предварительных паролей

1. Вопийте на**Password protection** [портале Azure](https://portal.azure.com) и просматривайте**методы проверки безопасности** > **Security** >  **Azure Active Directory** > Security Protection.
1. Установите опцию **защиты паролей Enable в Active Directory Windows Server** для *Yes.*

    Когда эта настройка установлена *в No,* все развернутые агенты Azure AD Password Protection DC переходят в режим покоя, где все пароли принимаются как есть. Действия проверки не выполняются, а события аудита не генерируются.

1. Рекомендуется сначала установить **режим** *аудита.* После того, как вы знакомы с функцией и влиянием на пользователей в вашей организации, вы можете переключить **режим** на *принудительное*. Для получения дополнительной информации смотрите следующий раздел о [режимах работы](#modes-of-operation).
1. Когда все будет готово, щелкните **Сохранить**.

    [![](media/howto-password-ban-bad-on-premises-operations/enable-configure-custom-banned-passwords-cropped.png "Enable on-premises password protection under Authentication Methods in the Azure portal")](media/howto-password-ban-bad-on-premises-operations/enable-configure-custom-banned-passwords.png#lightbox)

## <a name="modes-of-operation"></a>Режимы работы

При включании предварительной защиты паролем Azure AD можно использовать режим *аудита* или *режим принудительного исполнения.* Мы рекомендуем, чтобы первоначальное развертывание и тестирование всегда начинались в режиме аудита. Записи в журнале событий должны отслеживаться, чтобы предвидеть, будут ли нарушены какие-либо существующие операционные процессы после включения режима *принудительного исполнения.*

### <a name="audit-mode"></a>Режим аудита

*Режим аудита* предназначен как способ запуска программного обеспечения в режиме "что если". Каждая служба агента Azure AD Password Protection DC оценивает входящий пароль в соответствии с действующей в настоящее время политикой.

Если текущая политика настроена в режиме аудита, "плохие" пароли приводят к сообщениям журнала событий, но обрабатываются и обновляются. Это поведение является единственным различием между режимом аудита и принудительного исполнения. Все остальные операции работают одинаково.

### <a name="enforced-mode"></a>Принудительный режим

*Принудительный* режим предназначен как окончательная конфигурация. Например, в режиме аудита каждая служба Azure AD Password Protection DC оценивает входящие пароли в соответствии с действующей в настоящее время политикой. При включении принудительного режима пароль, который считается небезопасным в соответствии с политикой, отклоняется.

Когда пароль отклоняется в принудительном режиме агентом Azure AD Password Protection, конечный пользователь видит аналогичную ошибку, как будто он увидит, был ли его пароль отклонен традиционным исправным применением сложности пароля. Например, пользователь может видеть следующее традиционное сообщение об ошибке в журнале Windows или изменять экран пароля:

*"Не удается обновить пароль. Значение, предусмотренное для нового пароля, не соответствует требованиям длины, сложности или истории домена»...*

Это сообщение является лишь одним из примеров нескольких возможных результатов. Конкретное сообщение об ошибке может варьироваться в зависимости от фактического программного обеспечения или сценария, который пытается установить небезопасный пароль.

Пострадавшим конечным пользователям может потребоваться работать со своим ИТ-персоналом, чтобы понять новые требования и выбрать безопасные пароли.

> [!NOTE]
> Azure AD Password Protection не контролирует конкретное сообщение об ошибке, отображаемые клиентской машиной при отклонении слабого пароля.
