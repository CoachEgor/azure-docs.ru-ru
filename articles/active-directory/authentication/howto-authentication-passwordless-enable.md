# <a name="enable-passwordless-sign-in-for-azure-ad-preview"></a>Включение без пароля входа для Azure AD (Предварительная версия)

## <a name="requirements"></a>Требования

* Многофакторная идентификация Azure
* В сочетании регистрации предварительной версии
* Просмотр ключей безопасности FIDO2 требуются совместимые ключи безопасности FIDO2
* WebAuthN требует Microsoft Edge в Windows 10 1809 или более поздней версии
* FIDO2 на основе Windows входа требуется Azure AD присоединенных к Windows 10 1809 или более поздней версии

## <a name="prepare-devices-for-preview"></a>Подготовка устройств для предварительной версии

Устройства, которые будут Пилотное развертывание с должна быть запущена Windows 10 версии 1809 или более поздней версии. Получения наилучших результатов — в Windows 10 1903 или более поздней версии.

## <a name="enable-security-keys-for-windows-sign-in"></a>Включить ключей безопасности для входа в систему с Windows

Организациям может потребоваться использовать один или несколько из следующих методов, чтобы включить использование ключей безопасности для Windows вход.

### <a name="enable-credential-provider-via-intune"></a>Включить поставщик учетных данных с помощью Intune

1. Войдите на [портале Azure](https://portal.azure.com).
1. Перейдите к **Microsoft Intune** > **регистрации устройств** > **регистрации Windows** > **Windows Hello для бизнеса** > **свойства**.
1. В разделе **параметры** задать **использовать ключи безопасности для входа в систему** для **включено**.

Настройка ключей безопасности для входа, не зависит от того, Настройка Windows Hello для бизнеса.

#### <a name="enable-targeted-intune-deployment"></a>Включите целевое развертывание Intune

Чтобы ориентироваться групп конкретного устройства, чтобы включить поставщик учетных данных, используйте следующие пользовательские параметры с помощью Intune. 

1. Войдите на [портале Azure](https://portal.azure.com).
1. Перейдите к **Microsoft Intune** > **конфигурации устройства** > **профили** > **создать профиль**.
1. Настроить новый профиль со следующими параметрами
   1. Имя: Ключи безопасности для входа в Windows
   1. Описание: Включает ключи безопасности FIDO для использования во время входа Windows
   1. Платформа: Windows 10 и более поздние версии
   1. Тип платформы: Пользовательская
   1. Параметры настраиваемых OMA-URI:
      1. Имя: Включить ключей безопасности FIDO для входа в Windows
      1. OMA-URI: ./Device/Vendor/MSFT/PassportForWork/SecurityKey/UseSecurityKeyForSignin
      1. Тип данных: Целое число
      1. Значение: 1 
1. Эту политику можно назначить для конкретных пользователей, устройств или групп. Дополнительные сведения можно найти в статье [назначение профилей пользователей и устройств в Microsoft Intune](https://docs.microsoft.com/intune/device-profile-assign).

![Создание политики конфигурации Intune пользовательское устройство](./media/howto-authentication-passwordless-enable/intune-custom-profile.png)

### <a name="enable-credential-provider-via-provisioning-package"></a>Включить поставщик учетных данных с помощью пакета подготовки

Для устройств не под управлением Intune можно установить пакет подготовки для включения функциональных возможностей. Конструктор конфигураций Windows приложения могут устанавливаться с [Microsoft Store](https://www.microsoft.com/store/apps/9nblggh4tx22).

1. Запустите конструктор конфигурации Windows.
1. Выберите **файл** > **новый проект**.
1. Присвойте проекту имя и запишите путь которой создается проект.
1. Щелкните **Далее**.
1. Оставьте **Подготовка пакета** выбранное в качестве **рабочего процесса проекта выбранные** и выберите **Далее**.
1. Выберите **настольные версии Windows все** под **выбрать параметры для просмотра и настройки** и выберите **Далее**.
1. Выберите **Готово**.
1. В только что созданный проект, перейдите к **параметры среды выполнения** > **WindowsHelloForBusiness** > **SecurityKeys**  >  **UseSecurityKeyForSignIn**.
1. Задайте **UseSecurityKeyForSignIn** для **включен**.
1. Выберите **Экспорт** > **пакета подготовки**
1. Оставьте значения по умолчанию в **построения** под **описания подготовки пакета** и выберите **Далее**.
1. Оставьте значения по умолчанию в **построения** под **выбрать сведения о безопасности для подготовки пакета** и выберите **Далее**.
1. Запишите или измените путь в **построения** windows откройте раздел **для сохранения пакета подготовки** и выберите **Далее**.
1. Выберите **построения** на **сборки пакета подготовки** страницы.
1. Сохраните две созданные файлы (ppkg и cat) в расположение, где их можно применить к машинам более поздней версии.
1. Следуйте указаниям в статье [применить пакет подготовки](https://docs.microsoft.com/windows/configuration/provisioning-packages/provisioning-apply-package), чтобы применить пакет установки, вы создали.

## <a name="enable-passwordless-authentication-methods"></a>Включить методы проверки подлинности без пароля

### <a name="enable-the-combined-registration-experience"></a>Включить процесс объединенный регистрации

Возможности регистрации для ключей безопасности FIDO2 зависят от предварительной версии объединенного регистрации. Выполните следующие действия, чтобы включить предварительную версию объединенный регистрации.

1. Войдите на [портал Azure](https://portal.azure.com)
1. Перейдите к **Azure Active Directory** > **параметров пользователя**
   1. Щелкните **Управление параметрами для компонентов предварительной версии панель доступа**
   1. В разделе **пользователи могут использовать функции предварительной версии для регистрации и управления ими сведения о безопасности — расширенный**.
      1. Выберите **выбранные** и выберите группу пользователей, которые будут участвовать в предварительной версии.
      1. Или выберите **все** включить для всех пользователей в вашем каталоге.
1. Нажмите кнопку **Сохранить**

### <a name="enable-new-passwordless-authentication-methods"></a>Включение новых методов проверки подлинности без пароля

1. Войдите на [портал Azure](https://portal.azure.com)
1. Перейдите к **Azure Active Directory** > **методы проверки подлинности** > **политика метод проверки подлинности (Предварительная версия)**
1. Под каждым **метод**, выберите следующие параметры
   1. **Включить** — Да или нет
   1. **Целевой объект** -всех пользователей или выбрать пользователей
1. **Сохранить** каждого метода

> [!WARNING]
> FIDO2, «Ключ политики ограниченного использования программ» пока не работает. Эта функция будут доступны до выхода общедоступной версии, не изменяйте эти политики по умолчанию.

> [!NOTE]
> Не нужно использовать оба метода без пароля (Если вы хотите просмотреть только один метод без пароля, вы можете включить только этот метод). Мы рекомендуем вам опробовать оба метода, так как они имеют свои преимущества.

## <a name="user-registration-and-management-of-fido2-security-keys"></a>Регистрация пользователей и управление ключами безопасности FIDO2

1. Перейдите на сайт [https://myprofile.microsoft.com](https://myprofile.microsoft.com).
1. Войти, если это еще не уже
1. Нажмите кнопку **сведения о безопасности**
   1. Если у пользователя уже есть хотя бы один метод многофакторной идентификации Azure, которые зарегистрированы, они сразу же можно зарегистрировать ключ безопасности FIDO2.
   1. Если у них нет хотя бы один метод многофакторной идентификации Azure, которые зарегистрированы, их необходимо добавить.
1. Добавить ключ безопасности FIDO2, щелкнув **добавьте метод** и выбрав **ключ безопасности**
1. Выберите **USB-устройство** или **NFC-устройство**
1. Располагать собственным ключом готовых и выберите **Далее**
1. Бокс будет отображаться и запрос на создание или ввод ПИН-код для ключа безопасности, а затем выполнить необходимые жест ключа либо биометрические не касайтесь ее.
1. Вам будет возвращен процесс объединенный регистрации и требуется указать понятное имя для токена, и какой из них позволяют определить, если у вас есть несколько. Нажмите кнопку **Далее**.
1. Нажмите кнопку **сделать** для завершения процесса

### <a name="manage-security-key-biometric-pin-or-reset-security-key"></a>Управление ключ биометрических параметров, ПИН-код, или сброса ключа безопасности

## <a name="known-issues"></a>Известные проблемы

### <a name="fido2-security-keys"></a>Ключи безопасности FIDO2

#### <a name="security-key-provisioning"></a>Подготовки ключей безопасности

Администратору при подготовке и отзыве ключей безопасности не доступен в общедоступной предварительной версии.

#### <a name="hybrid-azure-ad-join"></a>гибридное присоединение к Azure AD;

Пользователи, полагаясь на WIA единого входа, использующих управляемые учетные данные, такие как ключи безопасности FIDO2 или без пароля входа через приложение Microsoft Authenticator необходимо гибридного присоединения Windows 10 получить преимущества единого входа. Тем не менее работают только для Azure Active Directory присоединены машин сейчас ключей безопасности. Мы рекомендуем вам опробовать FIDO2 ключей безопасности для экрана блокировки Windows на чисто компьютерам, присоединенным Azure Active Directory. Это ограничение не применяется для Интернета.

#### <a name="upn-changes"></a>Изменения имени участника-пользователя

Мы работаем над поддержкой — средство, позволяющее изменение имени участника-пользователя в гибридных AADJ и AADJ устройств. При изменении имени участника-пользователя, не сможет изменять FIDO2 ключей безопасности для учетной записи для этого. Поэтому единственный подход — сбросить устройство и пользователь должен повторно зарегистрировать.

### <a name="authenticator-app"></a>Приложение Authenticator

#### <a name="ad-fs-integration"></a>Интеграция AD FS

Когда пользователь включит учетные данные Microsoft Authenticator без пароля, аутентификация этого пользователя по умолчанию всегда будет выполняться через уведомление для подтверждения. Эта логика запрещает пользователям в гибридный клиент перенаправляется в ADFS для входа в проверке без пользователя, используя дополнительный шаг нажмите кнопку «Используйте пароль.» Этот процесс также будут обходить любые локальные политики условного доступа и потоки сквозной проверки подлинности. Исключением из этого процесса является при login_hint указан, пользователь будет автоматически переадресовано в службы AD FS, а обход вариант использования учетных данных без пароля.

#### <a name="azure-mfa-server"></a>Сервер Azure MFA

Конечным пользователям, которые включены для многофакторной проверки Подлинности через сервер Azure MFA в локальной организации по-прежнему можно создать и использовать знак один телефонный без пароля в учетных данных. Если пользователь введет одну учетную запись в несколько (5 и более) установок Microsoft Authenticator, это может привести к ошибке.  

#### <a name="device-registration"></a>Регистрация устройства

Одним из обязательных условий для создания защищенных учетных данных нового типа является регистрация устройства, на котором они размещаются, в клиенте Azure AD для конкретного пользователя. Ограничения по регистрации устройств позволяют зарегистрировать устройство только в одном клиенте. Это ограничение означает, что можно включить только одну рабочую или учебную учетную запись в приложении Microsoft Authenticator для входа в систему с телефона.
