# <a name="planning-a-cloud-based-azure-multi-factor-authentication-deployment"></a>Планирование развертывания облачной многофакторной идентификации Azure

Пользователи подключаются к ресурсам организации на все более сложные сценарии. Людей используют для подключения из принадлежащие организации, личных и общих устройств на и вне корпоративной сети с использованием смарт-телефоны, планшеты, ПК и ноутбуки, часто на нескольких платформах. В этом мире всегда подключены, нескольких устройств и нескольких платформ безопасность учетных записей пользователей важнее, чем когда-либо. Пароли, независимо от их сложности, используемый между устройствами, сетями и платформами, больше не достаточно для обеспечения безопасности учетной записи пользователя, особенно в том случае, если пользователи склонны используйте пароли повторно для учетных записей. Сложных фишинг и другие методы социальной инженерии, можно привести имена пользователей и пароли, ID и проданных в публичного атак.

[Azure многофакторной идентификации (MFA)](concept-mfa-howitworks.md) помогает защитить доступ к данным и приложениям. Она обеспечивает дополнительный уровень безопасности, с использованием второй формы проверки подлинности. Организации могут использовать [условного доступа](../conditional-access/overview.md) для обеспечения удовлетворения их конкретных нужд.

| Локальной устаревших приложений, опубликованных для доступа к облаку | Azure AD [прокси приложения](../manage-apps/application-proxy.md) развертывается. |

## <a name="plan-user-rollout"></a>Планирование развертывания пользователей

План развертывания многофакторной проверки Подлинности должен включать следуют во время развертывания, которые находятся в емкости поддержка пилотного развертывания. Применение политики условного доступа для небольшой группы пользователей экспериментальной перед началом процесса развертывания. После оценки влияния на пилотных пользователей, процесс, используемый и регистрации поведения, можно добавить дополнительные группы в политику или добавление дополнительных пользователей к существующим группам.

Мы рекомендуем communications разрабатываются в сочетании с представителями из вашей организации, например, взаимодействие, управление изменениями или отдела кадров отделах.

## <a name="deployment-considerations"></a>Рекомендации по развертыванию

Многофакторная Идентификация Azure развертывается путем применения политик с помощью условного доступа. Объект [политики условного доступа](../conditional-access/overview.md) может требоваться выполнить многофакторную проверку подлинности при выполнении определенных условий такие как:

* Все пользователи, определенного пользователя, член группы или назначенные роли
* Доступ к конкретного облачного приложения
* Платформа устройства
* Состояние устройства
* Сетевое расположение или расположены IP-адрес
* Клиентские приложения
* Риска входа (требуется защита идентификации)
* Устройства, соответствующие требованиям
* Гибридные устройства, присоединенные к Azure AD
* Утвержденные клиентские приложения

Используйте настраиваемые плакатов и шаблонов писем в [материалы внедрение многофакторной проверки подлинности](https://www.microsoft.com/download/details.aspx?id=57600&WT.mc_id=rss_alldownloads_all) для развертывания многофакторной проверки подлинности для вашей организации.

## <a name="enable-multi-factor-authentication-with-conditional-access"></a>Включение службы Многофакторной идентификации Azure с условным доступом

Политики условного доступа принудительная регистрация, требуя незарегистрированных пользователей завершить регистрацию при первом входе в систему, рассмотрения важный элемент обеспечения безопасности.

[Защита идентификации Azure AD](../identity-protection/howto-configure-risk-policies.md) вносит политики регистрации и политики автоматических риска обнаружение и исправление в истории многофакторной идентификации Azure. Политики могут быть созданы для принудительного изменения пароля, когда существует угроза скомпрометированного удостоверения или требовать многофакторную Идентификацию, если вход в систему считается рискованные ниже [события](../reports-monitoring/concept-risk-events.md):

* Утерянные учетные данные
* Попытки входа с анонимных IP-адресов
* Невозможно переместиться в нетипичные расположения
* Попытки входа из неизвестных расположений
* Попытки входа с инфицированных устройств
* Входы в систему с IP-адресов с подозрительными действиями

Некоторые из событий риска, обнаруживаемые защитой идентификации Azure Active Directory происходит в реальном времени, а некоторые требуют автономную обработку. Администраторы могут выбрать для блокирования пользователей, которые демонстрируют схем опасного поведения и выполнил компенсацию вручную, требует изменения пароля или требовать многофакторную проверку подлинности как часть свои политики условного доступа.

## <a name="plan-authentication-methods"></a>Планирование методов проверки подлинности

Администраторы могут выбрать [методы проверки подлинности](../authentication/concept-authentication-methods.md) они хотят сделать доступными для пользователей. Важно разрешить более одного метода проверки подлинности, чтобы пользователи могли метод резервного копирования на случай недоступности их основного метода. Администраторы могут включать следующие методы:

### <a name="notification-through-mobile-app"></a>Уведомление в мобильном приложении

Push-уведомление отправляется в приложение Microsoft Authenticator на своем мобильном устройстве. Пользователь просматривает уведомление и выбирает **утвердить** полной проверки. Push-уведомлений через мобильное приложение предоставляют параметр наименее заметна для пользователей. Они также параметр наиболее надежным и безопасным, так как они используют подключение к данным, а не телефонии.

> [!NOTE]
> Если у организации есть сотрудники, работа или во время поездок в Китай, **уведомление в мобильном приложении** метод **устройств Android** не работает в этой стране. Альтернативные методы должны быть доступны для этих пользователей.

### <a name="verification-code-from-mobile-app"></a>Код проверки от мобильного приложения

Мобильное приложение, например приложение Microsoft Authenticator создает код проверки OATH каждые 30 секунд. Пользователь вводит этот код проверки в интерфейсе входа. Можно использовать параметр мобильного приложения, ли телефон может данных или сигнала сотовой связи.

### <a name="call-to-phone"></a>Звонок на телефон

Автоматический голосовой вызов направляется на пользователя. Пользователь отвечает на вызов и нажала **#** на клавиатуре телефона, чтобы утвердить свою проверку подлинности. По номеру телефона — это отличный способ резервного копирования для уведомлением или кодом подтверждения из мобильного приложения.

### <a name="text-message-to-phone"></a>SMS на телефон

Для пользователя отправляется текстовое сообщение, которое содержит код проверки, пользователю предлагается ввести код проверки в интерфейсе входа.

## <a name="plan-registration-policy"></a>Планирование политики регистрации

Администраторы должны определить, как пользователи будут регистрировать свои методы. В организациях нужно [предоставления объединенный регистрации](howto-registration-mfa-sspr-combined.md) для Azure MFA и пароля reset (SSPR). SSPR позволяет пользователям для сброса пароля в безопасным способом, используя те же методы, которые они используют для многофакторной проверки подлинности. Мы рекомендуем этот параметр совместно регистрации, в настоящее время в общедоступной предварительной версии, так как он является удобной для пользователей с возможность зарегистрировать один раз для обеих служб. Включение те же методы для SSPR и многофакторной Идентификации Azure позволит пользователям зарегистрировать, чтобы использовать обе эти функции.

### <a name="registration-with-identity-protection"></a>Регистрация с помощью защиты идентификации

Если ваша организация использует Azure Active Directory Identity Protection, [Настройка политики регистрации многофакторной проверки Подлинности](../identity-protection/howto-mfa-policy.md) запрашивать пользователям зарегистрироваться при следующем входе в интерактивном режиме.

### <a name="registration-without-identity-protection"></a>Регистрация без защиты идентификации

Если вашей организации нет лицензий, включение защиты идентификации Azure, пользователям предлагается зарегистрироваться при следующем обязательна многофакторная проверка Подлинности при входе в систему. Пользователи могут быть не зарегистрирована для многофакторной проверки Подлинности, если они не используют приложения, защищенные с помощью многофакторной проверки Подлинности. Очень важно для получения всех пользователей, зарегистрированных недобросовестных сторон не может подобрать пароль пользователя и регистрации для многофакторной проверки Подлинности от их имени, эффективно контролирует учетной записи.

#### <a name="enforcing-registration"></a>Принудительной регистрацией

Выполнив следующие действия условного доступа политики можно принудительно регистрации для многофакторной проверки подлинности пользователей

1. Создайте группу, добавить всех пользователей, не зарегистрирован.
2. С помощью условного доступа, применять многофакторную проверку подлинности для этой группы, для доступа ко всем ресурсам.
3. Периодически оценить членства в группах и удалить из группы пользователей, зарегистрированных.

Незарегистрированные и зарегистрированные пользователи многофакторной Идентификации Azure может быть зарегистрирована с помощью команд PowerShell, которые зависят от [модуля MSOnline PowerShell](https://docs.microsoft.com/powershell/azure/active-directory/install-msonlinev1?view=azureadps-1.0).

#### <a name="identify-registered-users"></a>Определите зарегистрированных пользователей

```PowerShell
Get-MsolUser -All | where {$_.StrongAuthenticationMethods -ne $null} | Select-Object -Property UserPrincipalName | Sort-Object userprincipalname 
```

#### <a name="identify-non-registered-users"></a>Определите незарегистрированных пользователей

```PowerShell
Get-MsolUser -All | where {$_.StrongAuthenticationMethods.Count -eq 0} | Select-Object -Property UserPrincipalName | Sort-Object userprincipalname 
```

### <a name="convert-users-from-per-user-mfa-to-conditional-access-based-mfa"></a>Convert пользователей из многофакторной проверки Подлинности пользователя для условного доступа на основе многофакторной проверки Подлинности

Если пользователи были включены с помощью включена для пользователя и принудительная многофакторная Идентификация Azure следующую команду PowerShell можно поможет выполнить преобразование в условного доступа на основе многофакторной идентификации Azure.

```PowerShell
# Disable MFA for all users, keeping their MFA methods intact
Get-MsolUser -All | Disable-MFA -KeepMethods

# Enforce MFA for all users
Get-MsolUser -All | Set-MfaState -State Enforced

# Wrapper to disable MFA with the option to keep the MFA
# methods (to avoid having to proof-up again later)
function Disable-Mfa {

    [CmdletBinding()]
    param(
        [Parameter(ValueFromPipeline=$True)]
        $User,
        [switch] $KeepMethods
    )

    Process {

        Write-Verbose ("Disabling MFA for user '{0}'" -f $User.UserPrincipalName)
        $User | Set-MfaState -State Disabled

        if ($KeepMethods) {
            # Restore the MFA methods which got cleared when disabling MFA
            Set-MsolUser -ObjectId $User.ObjectId `
                         -StrongAuthenticationMethods $User.StrongAuthenticationMethods
        }
    }
}

# Sets the MFA requirement state
function Set-MfaState {

    [CmdletBinding()]
    param(
        [Parameter(ValueFromPipelineByPropertyName=$True)]
        $ObjectId,
        [Parameter(ValueFromPipelineByPropertyName=$True)]
        $UserPrincipalName,
        [ValidateSet("Disabled","Enabled","Enforced")]
        $State
    )

    Process {
        Write-Verbose ("Setting MFA state for user '{0}' to '{1}'." -f $ObjectId, $State)
        $Requirements = @()
        if ($State -ne "Disabled") {
            $Requirement =
                [Microsoft.Online.Administration.StrongAuthenticationRequirement]::new()
            $Requirement.RelyingParty = "*"
            $Requirement.State = $State
            $Requirements += $Requirement
        }

        Set-MsolUser -ObjectId $ObjectId -UserPrincipalName $UserPrincipalName `
                     -StrongAuthenticationRequirements $Requirements
    }
}

```

## <a name="plan-conditional-access-policies"></a>Планирование политик условного доступа

Спланируйте стратегию политики условного доступа, которая будет определить, когда необходимы многофакторной проверки Подлинности и другие элементы управления, см. в статье [что такое условный доступ в Azure Active Directory?](../conditional-access/overview.md).

Очень важно предотвратить непреднамеренное блокирование вашего клиента Azure AD. Можно уменьшить ущерб от этого случайного блокирования административного доступа, [создания двух или более учетных записей доступа для чрезвычайных ситуаций в клиенте](../users-groups-roles/directory-emergency-access.md) и их исключение из политики условного доступа.

### <a name="create-conditional-access-policy"></a>Создание политики условного доступа

1. Войдите на [портал Azure](https://portal.azure.com), используя учетную запись глобального администратора.
1. Перейдите к **Azure Active Directory**, **условного доступа**.
1. Выберите **Новая политика**.
1. Введите понятное имя для политики.
1. В разделе **Пользователи и группы** сделайте следующее.
   * На вкладке **Включить** щелкните переключатель **Все пользователи**.
   * На **исключить** вкладке, установите флажок для **пользователей и групп** и выберите учетные записи аварийного доступа.
   * Нажмите кнопку **Done**(Готово).
1. В разделе **Облачные приложения** щелкните переключатель **Все облачные приложения**.
   * Необязательно. На вкладке **Исключить** выберите облачные приложения в своей организации, для которых не требуется MFA.
   * Нажмите кнопку **Done**(Готово).
1. В разделе **Условия** сделайте следующее.
   * Необязательно. Если вы включили службу "Защита идентификации Azure", вы можете включить оценку риска входа как часть политики.
   * Необязательно. Если у вас настроены надежные расположения или именованные расположения, их можно включить в политику или исключить из нее.
1. В разделе **Предоставление** убедитесь, что выбран переключатель **Предоставить доступ**.
    * Установите флажок **Требовать многофакторную проверку подлинности**.
    * Нажмите кнопку **Выбрать**.
1. Пропустите раздел **Сеанс**.
1. Для переключателя **Включить политику** установите значение **Вкл.**
1. Нажмите кнопку **Создать**.

## <a name="plan-integration-with-on-premises-systems"></a>Планирование интеграции с локальными системами

Некоторые прежних версий и локальные приложения, которые не прошли проверку подлинности непосредственно Azure AD требуются дополнительные действия, чтобы использовать в том числе многофакторной проверки Подлинности:

* Для прежних версий локальных приложений, которые нужно будет использовать прокси приложения.
* Локальные приложения RADIUS, которые нужно будет использовать адаптер многофакторной проверки Подлинности с NPS-сервер.
* Локальных приложений AD FS, которые потребуются для использования адаптером MFA с AD FS 2016.

Приложения, которые прямой проверки подлинности в Azure AD и современную аутентификацию (WS-Fed, SAML, OAuth, OpenID Connect) можно воспользоваться условного доступа политики напрямую.

### <a name="use-azure-mfa-with-azure-ad-application-proxy"></a>Использование Azure MFA с помощью прокси приложения Azure AD

Приложения, хранящимся в локальном можно опубликовать в Azure AD клиента через [прокси приложения Azure AD](../manage-apps/application-proxy.md) и воспользоваться преимуществами многофакторной идентификации Azure, если они настроены для использования Azure AD предварительной проверки подлинности.

Эти приложения распространяются политики условного доступа, принудительное применение многофакторной идентификации Azure, так же, как и любого другого AD-интегрированные приложения Azure.

Аналогично Если многофакторная Идентификация Azure применяется для всех входов пользователей, локальных приложений, опубликованных с помощью прокси приложения Azure AD будут защищены.

### <a name="integrating-azure-multi-factor-authentication-with-network-policy-server"></a>Интеграция Azure многофакторной проверки подлинности сервера политики сети

Расширение сервера политики сети (NPS) для MFA Azure добавляет облачные функции многофакторной идентификации в инфраструктуру проверки подлинности на базе существующих серверов. Расширение NPS можно добавить телефонный звонок, текстовое сообщение или Проверка телефона в приложении в существующую последовательность аутентификации. Такая интеграция имеет следующие ограничения:

* С помощью протокола CHAPv2 поддерживаются только Push-уведомления приложения структуры проверки подлинности и голосовой вызов.
* Невозможно применить политики условного доступа.

Расширение NPS выполняет функции адаптера между RADIUS и облачной службе Azure MFA предоставлять второй фактор проверки подлинности для защиты [VPN](howto-mfa-nps-extension-vpn.md), [подключений шлюза удаленных рабочих столов](howto-mfa-nps-extension-rdg.md), или другие поддерживает RADIUS приложения. Пользователи, что регистрация для Azure MFA в этой среде запрашиваются все попытки проверки подлинности, отсутствие политики условного доступа означает, что всегда обязательна многофакторная проверка Подлинности.

#### <a name="implementing-your-nps-server"></a>Реализация сервера политики сети

Если у вас развернут экземпляр сервера политики сети и используется уже, ссылаться на [интеграция существующей инфраструктуры NPS с многофакторной идентификацией Azure](howto-mfa-nps-extension.md). Если вы настраиваете сервер политики сети в первый раз, см. раздел [сервера политики сети (NPS)](https://docs.microsoft.com/windows-server/networking/technologies/nps/nps-top) инструкции. Рекомендации по устранению неполадок можно найти в статье [Устранение ошибок с расширением NPS для многофакторной идентификации Azure](howto-mfa-nps-extension-errors.md).

#### <a name="prepare-nps-for-users-that-arent-enrolled-for-mfa"></a>Подготовка службы NPS для пользователей, которые не зарегистрированы для многофакторной проверки Подлинности

Выберите, что происходит при попытке проверки подлинности пользователей, которые не зарегистрированы с многофакторной проверкой Подлинности. Используйте параметр реестра `REQUIRE_USER_MATCH` в пути реестра `HKLM\Software\Microsoft\AzureMFA` для управления поведением функции. Этот параметр имеет только один параметр конфигурации.

| Ключ | Значение | значение по умолчанию |
| --- | --- | --- |
| `REQUIRE_USER_MATCH` | TRUE / FALSE | Не задано (эквивалентно значению True) |

Этот параметр определяет, что нужно делать при проверке подлинности пользователя, не зарегистрированного для MFA. Последствия изменения этого параметра, перечислены в следующей таблице.

| Параметры | Состояние многофакторной проверки Подлинности пользователя | Эффекты |
| --- | --- | --- |
| Ключ не существует | Не зарегистрировано | Запрос MFA завершается неудачно |
| Значение присвоено значение True или не задано | Не зарегистрировано | Запрос MFA завершается неудачно |
| Ключ, значение False | Не зарегистрировано | Проверка подлинности без многофакторной проверки Подлинности |
| Ключ в значение False или True | Зарегистрировано | Должен пройти проверку подлинности с многофакторной проверкой Подлинности |

### <a name="integrate-with-active-directory-federation-services"></a>Интеграция со службами федерации Active Directory

Если ваша организация использует федерацию с Azure AD, вы можете использовать [многофакторную идентификацию Azure для защиты ресурсов AD FS](multi-factor-authentication-get-started-adfs.md), так и локально и в облаке. Azure MFA можно уменьшить пароли и обеспечить более безопасный способ проверки подлинности. Начиная с Windows Server 2016, теперь можно настроить Azure MFA для основной проверки подлинности.

В отличие от и в случае с AD FS в Windows Server 2012 R2, адаптер Azure MFA AD FS 2016 напрямую интегрируется с Azure AD и не требует многофакторной проверки Подлинности Azure на локальном сервере. Адаптер Azure MFA встроена в Windows Server 2016, и нет необходимости дополнительной установки.

При использовании Azure MFA с AD FS 2016 и целевым приложением, регулируется политикой условного доступа, существуют дополнительные рекомендации:

* Условный доступ доступен в том случае, когда приложение проверяющей стороны для Azure AD в федерацию с AD FS 2016.
* Условный доступ не предоставляется, когда приложение проверяющей стороны в AD FS 2016 и является управляемый или федерацию с AD FS 2016.
* Условный доступ не также доступны при настройке AD FS 2016 для использования Azure MFA как основной метод аутентификации.

#### <a name="ad-fs-logging"></a>Ведение журнала AD FS

Стандартный AD FS 2016 ведение журнала в журнале безопасности Windows и в журнал администратора AD FS содержит сведения о запросы на проверку подлинности и их успеха или сбоя. Данные журнала событий в рамках этих событий укажет, использовалось ли Azure MFA. Например AD FS аудит кодом 1200 может содержать:

```
<MfaPerformed>true</MfaPerformed>
<MfaMethod>MFA</MfaMethod>
```

#### <a name="renew-and-manage-certificates"></a>Продление сертификатов и управление ими

На каждом сервере AD FS в локальном My Store будет самозаверяющий Azure MFA сертификатов под названием OU = Microsoft AD FS Azure MFA, которая содержит дату окончания срока действия сертификата. Проверьте срок действия этого сертификата на каждом сервере AD FS, чтобы определить дату истечения срока действия.

Если срок действия сертификатов приближается истечение срока действия, [создания и проверки новый сертификат на каждом сервере AD FS многофакторной проверки Подлинности](https://docs.microsoft.com/windows-server/identity/ad-fs/operations/configure-ad-fs-and-azure-mfa#configure-the-ad-fs-servers).

Следующее руководство подробно описано, как управление сертификатами Azure MFA на серверы AD FS. При настройке AD FS с Azure MFA, сформированный с помощью сертификатов `New-AdfsAzureMfaTenantCertificate` командлет PowerShell действительны в течение 2 года. Обновление и установить обновленные сертификаты до истечения срока действия в ovoid нарушений в работе в службе многофакторной проверки Подлинности.
