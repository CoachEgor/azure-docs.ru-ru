# <a name="planning-a-cloud-based-azure-multi-factor-authentication-deployment"></a>Планирование развертывания облачной службы Многофакторной идентификации Azure

Люди подключаются к организационным ресурсам во все более сложных сценариях. Люди подключаются к устройствам, принадлежащим организации, личным и общедоступным устройствам, на и вне корпоративной сети, используя смартфоны, планшеты, ПК и ноутбуки, часто на нескольких платформах. В этом всегда подключенном, многофункциональном и многоплатформеном мире безопасность учетных записей пользователей как никогда важна. Пароли, независимо от их сложности, используемые на устройствах, сетях и платформах, больше не достаточны для обеспечения безопасности учетной записи пользователя, особенно когда пользователи склонны повторно использовать пароли в разных учетных записях. Сложные фишинг и другие атаки социальной инженерии могут привести к тому, что имена пользователей и пароли будут размещаться и продаваться по темной сети.

[Multi-Factor Authentication (MFA) Azure Multi-Factor Authentication (MFA)](concept-mfa-howitworks.md) помогает защитить доступ к данным и приложениям. Он обеспечивает дополнительный уровень безопасности с помощью второй формы проверки подлинности. Организации могут использовать [Условный доступ,](../conditional-access/overview.md) чтобы сделать решение соответствовать их конкретным потребностям.

| Натерритории устаревших приложений, опубликованных для доступа к облаку | Развертывается [прокси-приложение](../manage-apps/application-proxy.md) Azure AD. |

## <a name="plan-user-rollout"></a>Планирование развертывания пользователей

План развертывания МИД должен включать пилотное развертывание с последующими волнами развертывания, которые находятся в пределах вашей емкости поддержки. Начните развертывание, применяя политики условного доступа к небольшой группе пилотных пользователей. Оценив аяру пользователей, используемых в процессе и поведение регистрации, можно либо добавить в политику больше групп, либо добавить больше пользователей к существующим группам.

### <a name="user-communications"></a>Коммуникации пользователей

В запланированных сообщениях крайне важно информировать пользователей о предстоящих изменениях, требованиях к регистрации Azure MFA и любых необходимых действиях пользователей. Мы рекомендуем совместно с представителями вашей организации, такими как отделы коммуникаций, управления изменениями или отделы кадров, развивать коммуникации.

Корпорация Майкрософт предоставляет [шаблоны связи](https://aka.ms/mfatemplates) и [документацию конечных пользователей,](../user-help/security-info-setup-signin.md) чтобы помочь подготовить ваши сообщения. Вы можете отправить [https://myprofile.microsoft.com](https://myprofile.microsoft.com) пользователей для регистрации непосредственно, выбрав ссылки **Security Info** на этой странице.

## <a name="deployment-considerations"></a>Рекомендации по развертыванию

Многофакторная аутентификация Azure развертывается путем обеспечения политик и условного доступа. [Политика условного доступа](../conditional-access/overview.md) может потребовать от пользователей выполнять многофакторную аутентификацию при выполнении определенных критериев, таких как:

* Все пользователи, конкретный пользователь, член группы или назначенная роль
* Доступ к конкретному облачному приложению
* Платформа устройства
* Состояние устройства
* Расположение сети или гео-расположенный IP-адрес
* Клиентские приложения
* Риск ввоза (требует защиты личных данных)
* Устройства, соответствующие требованиям
* Гибридные устройства, присоединенные к Azure AD
* Утвержденное клиентское приложение

Используйте настраиваемые плакаты и шаблоны электронной почты в [многофакторных материалах развертывания аутентификации](https://www.microsoft.com/download/details.aspx?id=57600&WT.mc_id=rss_alldownloads_all) для развертывания многофакторной аутентификации в организацию.

## <a name="enable-multi-factor-authentication-with-conditional-access"></a>Включение службы Многофакторной идентификации Azure с условным доступом

Политики условного доступа обеспечивают регистрацию, требуя от незарегистрированных пользователей завершить регистрацию при первом вписании, что является важным соображением безопасности.

[Защита идентификационных данных Azure AD](../identity-protection/howto-configure-risk-policies.md) вносит как политику регистрации, так и автоматизированную политику обнаружения рисков и исправлений в историю многофакторной аутентификации Azure. Политики могут быть созданы, чтобы заставить изменения пароля, когда есть угроза скомпрометированной идентичности или требуют МИД, когда входе считается рискованным [следующие события:](../reports-monitoring/concept-risk-events.md)

* Утерянные учетные данные
* Попытки входа с анонимных IP-адресов
* Невозможно переместиться в нетипичные расположения
* Попытки входа из неизвестных расположений
* Попытки входа с инфицированных устройств
* Регистрация с IP-адресов с подозрительной деятельностью

Некоторые из обнаружений рисков, обнаруженных Azure Active Directory Identity Protection, происходят в режиме реального времени, а некоторые требуют автономной обработки. Администраторы могут блокировать пользователей, которые демонстрируют рискованное поведение и исправления вручную, требуют изменения пароля или требуют многофакторной аутентификации в рамках своих политик условного доступа.

## <a name="define-network-locations"></a>Определение местоположения сети

Мы рекомендуем организациям использовать Условный доступ для определения своей сети с помощью [названных местоположений.](../conditional-access/location-condition.md#named-locations) Если ваша организация использует Identity Protection, рассмотрите возможность использования политик, основанных на риске, вместо названных местоположений.

### <a name="configuring-a-named-location"></a>Настройка названного местоположения

1. Откройте **активный каталог Azure** на портале Azure
2. Выберите **Безопасность**.
3. Под **управлением**выберите **именные места**
4. Выберите **новое местоположение**
5. В поле **«Имя»** предоставьте значимое имя
6. Выберите, определяете ли вы местоположение с помощью *диапазонов IP* или *стран/регионов*
   1. При *использовании диапазона IP*
      1. Решите, следует ли *Марк, как доверенное местоположение*. Вход из надежного расположения снижает риски входа пользователя. Только пометить это место как доверенный, если вы знаете, IP диапазоны введены установлены и надежными в вашей организации.
      2. Указать диапазоны ИС
   2. При *использовании стран/регионов*
      1. Расширьте меню выпадающих и выберите страны или регионы, которые вы хотите определить для этого названного местоположения.
      2. Решите, включать ли *неизвестные области.* Неизвестные области являются IP-адресами, которые не могут быть отображены в стране/регионе.
7. Выберите **Создать**

## <a name="plan-authentication-methods"></a>Планирование методов проверки подлинности

Администраторы могут выбрать [методы проверки подлинности,](../authentication/concept-authentication-methods.md) которые они хотят сделать доступными для пользователей. Важно разрешить более одного метода проверки подлинности, чтобы пользователи имели доступный метод резервного копирования в случае недоступности основного метода. Администраторам доступны следующие методы, позволяющие:

### <a name="notification-through-mobile-app"></a>Уведомление в мобильном приложении

Push-уведомление отправляется в приложение Microsoft Authenticator на вашем мобильном устройстве. Пользователь просматривает уведомление и выбирает **Утверждение** для завершения проверки. Push-уведомления через мобильное приложение предоставляют наименее интрузивный вариант для пользователей. Они также являются наиболее надежным и безопасным вариантом, поскольку они используют подключение к данным, а не телефонию.

> [!NOTE]
> Если в вашей организации работает персонал или он путешествует в Китай, метод **уведомления с помощью мобильного приложения** на **устройствах Android** не работает в этой стране. Для этих пользователей должны быть доступны альтернативные методы.

### <a name="verification-code-from-mobile-app"></a>Код проверки от мобильного приложения

Мобильное приложение, такое как приложение Microsoft Authenticator, генерирует новый код проверки OATH каждые 30 секунд. Пользователь вводит этот код проверки в интерфейсе входа. Опция мобильного приложения может быть использована независимо от того, есть ли у телефона данные или сотовый сигнал.

### <a name="call-to-phone"></a>Звонок на телефон

Пользователю размещается автоматический голосовой вызов. Пользователь отвечает на звонок **#** и нажимает на клавиатуру телефона, чтобы утвердить их аутентификацию. Звонок на телефон — отличный метод резервного копирования для уведомления или кода проверки из мобильного приложения.

### <a name="text-message-to-phone"></a>SMS на телефон

Пользователю отправляется текстовое сообщение, содержащее код проверки, и пользователю предлагается ввести код проверки в интерфейс входа.

## <a name="plan-registration-policy"></a>Политика регистрации плана

Администраторы должны определить, как пользователи будут регистрировать свои методы. Организации должны [включить новый опыт комбинированной регистрации](howto-registration-mfa-sspr-combined.md) для Azure MFA и сбросить пароли самообслуживания (SSPR). SSPR позволяет пользователям безопасно сбросить свой пароль, используя те же методы, которые они используют для многофакторной аутентификации. Мы рекомендуем эту комбинированную регистрацию, в настоящее время в открытом предварительном просмотре, потому что это большой опыт для пользователей, с возможностью зарегистрироваться один раз для обеих услуг. Включение одних и тех же методов для SSPR и Azure MFA позволит пользователям зарегистрироваться для использования обеих функций.

### <a name="registration-with-identity-protection"></a>Регистрация с защитой личных данных

Если организация использует Azure Active Directory Identity Protection, [назначьте политику регистрации МИД,](../identity-protection/howto-mfa-policy.md) чтобы побудить пользователей зарегистрироваться при следующем входе в систему.

### <a name="registration-without-identity-protection"></a>Регистрация без защиты личных данных

Если в вашей организации нет лицензий, позволяющих защитить личность, пользователям предлагается зарегистрироваться при входе во всможное займом. Пользователи не могут быть зарегистрированы в МИД, если они не используют приложения, защищенные МИД. Важно, чтобы все пользователи были зарегистрированы так, чтобы плохие актеры не могли угадать пароль пользователя и зарегистрироваться в МИД от их имени, эффективно взяв под контроль учетную запись.

#### <a name="enforcing-registration"></a>Обеспечение регистрации

Используя следующие шаги, политика условного доступа может заставить пользователей зарегистрироваться для многофакторной аутентификации

1. Создайте группу, добавьте всех пользователей, которые в настоящее время не зарегистрированы.
2. Используя Условный доступ, принудить многофакторную аутентификацию для этой группы для доступа ко всем ресурсам.
3. Периодически пересматривайте членство в группе и удаляйте пользователей, зарегистрировавшихся в группе.

Вы можете идентифицировать зарегистрированных и незарегистрированных пользователей Azure MFA с помощью команд PowerShell, которые опираются на [модуль MSOnline PowerShell.](https://docs.microsoft.com/powershell/azure/active-directory/install-msonlinev1?view=azureadps-1.0)

#### <a name="identify-registered-users"></a>Выявление зарегистрированных пользователей

```PowerShell
Get-MsolUser -All | where {$_.StrongAuthenticationMethods -ne $null} | Select-Object -Property UserPrincipalName | Sort-Object userprincipalname 
```

#### <a name="identify-non-registered-users"></a>Выявление незарегистрированных пользователей

```PowerShell
Get-MsolUser -All | where {$_.StrongAuthenticationMethods.Count -eq 0} | Select-Object -Property UserPrincipalName | Sort-Object userprincipalname 
```

### <a name="convert-users-from-per-user-mfa-to-conditional-access-based-mfa"></a>Преобразование пользователей из МИД на одного пользователя в МИД на основе условного доступа

Если пользователи были включены с помощью включенной и принудительной azure Multi-Фактор аутентификации, следующая PowerShell может помочь вам сделать преобразование в условную azure Multi-Факторную аутентификацию.

Выполнить эту PowerShell в окне ISE или сохранить как . Файл PS1 для локального запуска.

```PowerShell
# Sets the MFA requirement state
function Set-MfaState {

    [CmdletBinding()]
    param(
        [Parameter(ValueFromPipelineByPropertyName=$True)]
        $ObjectId,
        [Parameter(ValueFromPipelineByPropertyName=$True)]
        $UserPrincipalName,
        [ValidateSet("Disabled","Enabled","Enforced")]
        $State
    )

    Process {
        Write-Verbose ("Setting MFA state for user '{0}' to '{1}'." -f $ObjectId, $State)
        $Requirements = @()
        if ($State -ne "Disabled") {
            $Requirement =
                [Microsoft.Online.Administration.StrongAuthenticationRequirement]::new()
            $Requirement.RelyingParty = "*"
            $Requirement.State = $State
            $Requirements += $Requirement
        }

        Set-MsolUser -ObjectId $ObjectId -UserPrincipalName $UserPrincipalName `
                     -StrongAuthenticationRequirements $Requirements
    }
}

# Disable MFA for all users
Get-MsolUser -All | Set-MfaState -State Disabled
```

> [!NOTE]
> Недавно мы изменили поведение и сценарий PowerShell выше, соответственно. Ранее скрипт спас от методов МИД, отключил МИД и восстановил методы. Это больше не является необходимым теперь, когда поведение по умолчанию для оттачивать не очищает методы.

## <a name="plan-conditional-access-policies"></a>Планирование политик условного доступа

Чтобы спланировать стратегию политики условного доступа, которая определит, когда требуется FAFA и другие элементы управления, обратитесь к [Что такое условный доступ в Active Directory Azure?.](../conditional-access/overview.md)

Важно, чтобы вы не допустили непреднамеренной блокировки арендатора Azure AD. Вы можете смягчить последствия этого непреднамеренного отсутствия административного доступа, [создав две или более учетных записей экстренного доступа в вашем арендаторе](../users-groups-roles/directory-emergency-access.md) и исключив их из политики условного доступа.

### <a name="create-conditional-access-policy"></a>Создание политики условного доступа

1. Вопиюсь на [портал Azure](https://portal.azure.com) с помощью глобальной учетной записи администратора.
1. Просмотрите доступ к > **условную безопасность****безопасности** **Azure Active Directory.** > 
1. Выберите **новую политику**.
   ![Создание политики условного доступа для включения пользователей портала Azure в пилотную группу MFA](media/howto-mfa-getstarted/conditionalaccess-newpolicy.png)
1. Введите понятное имя для политики.
1. Под **пользователями и группами:**
   * На вкладке **Включить** щелкните переключатель **Все пользователи**.
   * На вкладке **«Исключить»** проверьте поле для **пользователей и групп** и выберите учетные записи экстренного доступа.
   * Нажмите кнопку **Готово**.
1. В **приложениях Cloud**выберите кнопку радио **«Все облачные приложения».**
   * Необязательно. На вкладке **Исключить** выберите облачные приложения в своей организации, для которых не требуется MFA.
   * Нажмите кнопку **Готово**.
1. В разделе **Условия:**

## <a name="plan-integration-with-on-premises-systems"></a>Планирование интеграции с помощью систем на месте

Некоторые устаревшие и корпоративные приложения, которые не подтверждаются непосредственно в отношении Azure AD, требуют дополнительных шагов для использования МИД, включая:

* Наследие на месте приложений, которые должны будут использовать приложение прокси.
* В помещении приложения RADIUS, которые должны будут использовать адаптер MFA с сервером NPS.
* На-предах AD FS приложений, которые должны будут использовать MFA адаптер с AD FS 2016 или новые.

Приложения, которые непосредственно аутентифицируются с Azure AD и имеют современную аутентификацию (WS-Fed, SAML, OAuth, OpenID Connect) могут непосредственно использовать политики условного доступа.

### <a name="use-azure-mfa-with-azure-ad-application-proxy"></a>Используйте Azure MFA с прокси-сервером Azure AD

Приложения, проживающие в помещениях, могут быть опубликованы арендатору Azure AD через [Azure AD Application Proxy](../manage-apps/application-proxy.md) и могут воспользоваться преимуществами azure Multi-Factor Authentication, если они настроены на использование предварительной аутентификации Azure AD.

Эти приложения подчиняются политикам условного доступа, которые обеспечивают многофакторную аутентификацию Azure, как и любое другое приложение, интегрированное в Azure AD.

Аналогичным образом, если Azure Multi-Factor Authentication обеспечивается для всех пользовательских всподанных, корпоративные приложения, опубликованные с помощью Azure AD Application Proxy, будут защищены.

### <a name="integrating-azure-multi-factor-authentication-with-network-policy-server"></a>Интеграция многофакторной аутентификации Azure с сервером сетевой политики

Расширение сервера политики сети (NPS) для MFA Azure добавляет облачные функции многофакторной идентификации в инфраструктуру проверки подлинности на базе существующих серверов. С расширением NPS вы можете добавить телефонный звонок, текстовое сообщение или проверку приложения телефона в существующий поток аутентификации. Эта интеграция имеет следующие ограничения:

* С протоколом CHAPv2 поддерживаются только push-уведомления и голосовой вызов приложения authenticator.
* Политики условного доступа не могут быть применены.

Расширение NPS выступает в качестве адаптера между RADIUS и облачным Azure MFA, чтобы обеспечить второй фактор аутентификации для защиты [VPN,](howto-mfa-nps-extension-vpn.md) [удаленного соединения шлюза для настольных](howto-mfa-nps-extension-rdg.md)компьютеров или других приложений, способных РАЗRADI. Пользователи, зарегистрировавшиеся в Azure MFA в этой среде, будут оспорены за все попытки проверки подлинности, отсутствие политики условного доступа означает, что МИД всегда требуется.

#### <a name="implementing-your-nps-server"></a>Реализация сервера NPS

Если экземпляр NPS уже развернут и используется, обратитесь [к существующей инфраструктуре NPS с многофакторной аутентификацией Azure Multi-Factor Authentication.](howto-mfa-nps-extension.md) Если вы впервые настраиваете NPS, обратитесь за инструкциями к [серверу сетевой политики (NPS).](https://docs.microsoft.com/windows-server/networking/technologies/nps/nps-top) Руководство по устранению неполадок можно найти в статье [Разрешить сообщения об ошибках из расширения NPS для Azure Multi-фактор аутентификации](howto-mfa-nps-extension-errors.md).

#### <a name="prepare-nps-for-users-that-arent-enrolled-for-mfa"></a>Подготовка NPS для пользователей, которые не зарегистрированы для МИД

Выберите, что происходит, когда пользователи, не зарегистрированные в МИД, пытаются проверить подлинность. Используйте настройки `REQUIRE_USER_MATCH` реестра `HKLM\Software\Microsoft\AzureMFA` в пути реестра для управления поведением функции. Этот параметр имеет единую опцию конфигурации.

| Ключ | Значение | Значение по умолчанию |
| --- | --- | --- |
| `REQUIRE_USER_MATCH` | ПРАВДА / ЛОЖНО | Не задано (эквивалентно значению True) |

Этот параметр определяет, что нужно делать при проверке подлинности пользователя, не зарегистрированного для MFA. Последствия изменения этого параметра перечислены в таблице ниже.

| Параметры | Статус МИД пользователя | Произведенный эффект |
| --- | --- | --- |
| Ключ не существует | Не зарегистрировано | Вызов МИД не удался |
| Значение, установленное на True / не установлено | Не зарегистрировано | Вызов МИД не удался |
| Ключ установлен на ложные | Не зарегистрировано | Аутентификация без МИД |
| Ключ установлен на ложные или правда | Зарегистрировано | Должна быть проверена с МИД |

### <a name="integrate-with-active-directory-federation-services"></a>Интеграция с службами Федерации активного каталога

Если ваша организация используется с помощью Azure AD, вы можете использовать [Azure Multi-Factor Authentication для защиты ресурсов AD FS](multi-factor-authentication-get-started-adfs.md)как в помещениях, так и в облаке. Azure MFA позволяет уменьшить пароли и обеспечить более безопасный способ проверки подлинности. Начиная с Windows Server 2016, теперь вы можете настроить Azure MFA для первичной аутентификации.

В отличие от AD FS в Windows Server 2012 R2, адаптер AD FS 2016 Azure MFA интегрируется непосредственно с Azure AD и не требует внутреннего сервера Azure MFA. Адаптер Azure MFA встроен в Windows Server 2016, и нет необходимости в дополнительной установке.

При использовании Azure MFA с AD FS 2016 и целевым приложением распространяется политика условного доступа, существуют дополнительные соображения:

* Условный доступ доступен, когда приложение является полагающейся стороной Azure AD, федеративной aD FS 2016 или более новой.
* Условный доступ недоступен, когда приложение является полагающейся стороной AD FS 2016 или AD FS 2019 и управляется или федеративается с AD FS 2016 или AD FS 2019.
* Условный доступ также недоступен, когда AD FS 2016 или AD FS 2019 настроен ы на использование Azure MFA в качестве основного метода проверки подлинности.

#### <a name="ad-fs-logging"></a>AD FS лесозаготовки

Стандартный AD FS 2016 и 2019 вход в журнал безопасности Windows и журнал AD FS Ad AdIn содержит информацию о запросах на аутентификацию и их успехе или сбое. Данные журнала событий в этих событиях будут указывать, использовался ли Azure MFA. Например, AD FS Auditing Event ID 1200 может содержать:

```
<MfaPerformed>true</MfaPerformed>
<MfaMethod>MFA</MfaMethod>
```

#### <a name="renew-and-manage-certificates"></a>Обновление и управление сертификатами

На каждом сервере AD FS, на локальном компьютере My Store, будет размещен сертификат Azure MFA под названием OU-Microsoft AD FS Azure MFA, который содержит срок действия сертификата. Проверьте срок действия этого сертификата на каждом сервере AD FS, чтобы определить дату истечения срока действия.

Если срок действия сертификатов приближается к истечению срока действия, [создайте и проверьте новый сертификат MFA на каждом сервере AD FS.](https://docs.microsoft.com/windows-server/identity/ad-fs/operations/configure-ad-fs-and-azure-mfa#configure-the-ad-fs-servers)

Ниже приводится информация о том, как управлять сертификатами Azure MFA на серверах AD FS. При настройке AD FS с Azure MFA сертификаты, генерируемые через cmdlet `New-AdfsAzureMfaTenantCertificate` PowerShell, действительны в течение 2 лет. Обновить и установить обновленные сертификаты до истечения срока действия сбоев в работе службы МИД.
