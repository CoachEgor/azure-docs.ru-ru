# <a name="azure-ad-administrative-units-troubleshooting-and-faq"></a>Административные подразделения Azure AD: Устранение проблем и часто задаваемые вопросы

Для более детального административного управления в Active Directory Azure (Azure AD) можно назначить пользователям роль Azure AD с областью, ограниченной одним или несколькими административными единицами (AUs). Для примеров скриптов PowerShell для общих задач [см. Работа с административными единицами.](https://docs.microsoft.com/powershell/azure/active-directory/working-with-administrative-units?view=azureadps-2.0)

## <a name="frequently-asked-questions"></a>Часто задаваемые вопросы

**В: Почему я не могу создать административную единицу?**

**A:** Только *Глобальный администратор* или *привилегированный администратор ролей* может создать административную единицу в Azure AD. Убедитесь, что пользователю, пытавшемуся создать административное подразделение, присваивается либо *роль Глобального администратора,* либо *привилегировая роль администратора ролей.*

**В: Я добавил группу в административную единицу. Почему члены группы до сих пор не появляются там?**

**A:** При добавлении группы в административную единицу это не приводит к добавлению в нее всех членов группы. Пользователи должны быть непосредственно назначены административной единице.

**В: Я только что добавил (или удалил) члена административной единицы. Почему участник не появляется (или все еще появляется) в пользовательском интерфейсе?**

**A:** Иногда обработка добавления или удаления одного или нескольких членов административного подразделения может занять несколько минут, чтобы отразиться на странице **административных единиц.** Кроме того, можно перейти непосредственно к свойствам связанного ресурса и посмотреть, завершено ли действие. Для получения дополнительной информации о пользователях и группах [List administrative units for a group](roles-admin-units-add-manage-groups.md)в AUs [см.](roles-admin-units-add-manage-users.md)

**В: Я являюсь делегированным администратором паролей в административном блоке. Почему я не могу сбросить пароль конкретного пользователя?**

**A:** Как администратор административного подразделения, вы можете сбросить пароли только для пользователей, которые назначены в ваш административный блок. Убедитесь, что пользователь, сбросить пароль которого не удается, принадлежит к административной единице, которой вы были назначены. Если пользователь принадлежит к той же административной единице, но вы все еще не можете сбросить его пароль, проверьте роли, которые назначены пользователю. 

Чтобы предотвратить повышение привилегий, администратор с охватом административного подразделения не может сбросить пароль пользователя, назначенного на роль с общеорганизационным охватом.

**В: Зачем нужны административные единицы? Разве мы не могли использовать группы безопасности как способ определения области?**

**A:** Группы безопасности имеют существующую модель назначения и авторизации. *Администратор пользователей,* например, может управлять членством всех групп безопасности в организации Azure AD. Роль может использовать группы для управления доступом к приложениям, таким как Salesforce. *Администратор пользователя* не должен управлять самой моделью делегирования, что было бы результатом, если бы группы безопасности были расширены для поддержки сценариев «группирования ресурсов». Административные единицы, такие как организационные единицы в Active Directory Windows Server, предназначены для обеспечения способа администрирования сферы действия широкого спектра объектов каталога. Сами группы безопасности могут быть членами областей ресурсов. Использование групп безопасности для определения набора групп безопасности, которыми может управлять администратор, может ввести в заблуждение.

**В: Что значит добавить группу в административную единицу?**

**A:** Добавление группы в административное подразделение приводит саму группу в сферу управления любого *Администратора Пользователя,* который также относится к этому административному блоку. Администраторы пользователей административного подразделения могут управлять именем и членством самой группы. Он не предоставляет *Администратору пользователя* разрешения административного блока на управление пользователями группы (например, для сбросить их пароли). Чтобы предоставить *Администратору Пользователя* возможность управлять пользователями, пользователи должны быть прямыми членами административного блока.

**В: Может ли ресурс (пользователь или группа) быть членом более чем одной административной единицы?**

**A:** Да, ресурс может быть членом более чем одной административной единицы. Ресурсможет управляться всеми администраторами, занимающиеся общеорганизационными и административными подразделениями, имеющими разрешения на ресурс.

**В: Доступны ли административные единицы в организациях B2C?**

**A:** Нет, административные единицы недоступны для организаций B2C.

**В: Поддерживаются ли вложенные административные единицы?**

**A:** Нет, вложенные административные единицы не поддерживаются.

**В: Поддерживаются ли административные единицы в PowerShell и Graph API?**

**A:** Да. Вы найдете поддержку для административных единиц в [документации PowerShell cmdlet](https://docs.microsoft.com/powershell/module/Azuread/?view=azureadps-2.0-preview) и [образцах скриптов.](https://docs.microsoft.com/powershell/azure/active-directory/working-with-administrative-units?view=azureadps-2.0-preview) 

Найти поддержку [для типа ресурсов административного блока](https://developer.microsoft.com/graph/docs/api-reference/beta/resources/administrativeunit) Unit в Microsoft Graph.
