# <a name="custom-administrator-roles-in-azure-active-directory-preview"></a>Роли настраиваемого администратора в Azure Active Directory (Предварительная версия)

В этой статье описывается, как определить новый пользовательский RBAC (Управление доступом на основе ролей) и области ресурсов в Azure Active Directory (Azure AD). Пользовательские роли RBAC выявляются базовыми разрешениями [встроенных ролей](directory-assign-admin-roles.md) , что позволяет создавать и упорядочивать собственные пользовательские роли. Области ресурсов дают возможность назначить пользовательскую роль для управления некоторыми ресурсами (например, одним приложением) без предоставления доступа ко всем ресурсам (все приложения).

Предоставление разрешения с помощью настраиваемых ролей RBAC — двухэтапный процесс. Сначала необходимо создать пользовательское определение роли и добавить в него разрешения из списка предустановок. Это те же разрешения, которые используются во встроенных ролях. Создав роль, вы назначаете ее пользователю, создавая назначение роли. Этот двухэтапный процесс позволяет создать одну роль и назначить ее несколько раз в разных областях. Пользовательская роль может быть назначена в области каталога или ее можно назначить в области объекта. Примером области объекта может быть одно приложение. Таким образом можно назначить одну и ту же роль Светлана для всех приложений в каталоге, а затем Навин только для приложения "отчеты о расходах Contoso".

Первый выпуск настраиваемых ролей RBAC включает возможность создания роли для назначения разрешений на управление регистрациями приложений. Со временем будут добавлены дополнительные разрешения для ресурсов Организации, таких как корпоративные приложения, пользователи и устройства.

Предварительные версии функций:

- Обновление пользовательского интерфейса портала для создания пользовательских ролей и управления ими, а затем назначение их пользователям в области всей Организации
- Модуль предварительной версии PowerShell с новыми командлетами для:
  - Создание настраиваемых ролей и управление ими
  - Назначение пользовательских ролей с областью регистрации в масштабе всей организации или на уровне приложения
  - Назначение встроенных ролей в области всей Организации (четность с помощью командлетов в общедоступной версии)
  - Поддержка API Graph Azure AD

Управление доступом на основе ролей Azure AD — это общедоступная Предварительная версия Azure AD. она доступна с любым платным планом лицензирования Azure AD. См. подробные сведения о [дополнительных условиях использования предварительных выпусков Microsoft Azure](https://azure.microsoft.com/support/legal/preview-supplemental-terms/).

## <a name="understand-azure-ad-role-based-access-control"></a>Общие сведения об управлении доступом на основе ролей в Azure AD

Управление доступом на основе ролей в Azure AD позволяет назначать роли, настроенные для разрешения разрешенных действий только для одного типа ресурса Azure AD. Доступ на основе ролей Azure AD работает с такими понятиями, как управление доступом на основе ролей в Azure ([Azure RBAC](../../role-based-access-control/overview.md) для доступа к ресурсам Azure), но управление доступом на основе РОЛЕЙ Azure AD основано на Microsoft Graph и Azure RBAC основаны на Azure Resource Manager. Однако обе системы основаны на назначениях ролей.

### <a name="role-assignments"></a>Назначения ролей

Способ управления доступом с помощью управления доступом на основе ролей Azure AD заключается в создании **назначений ролей**, которые используются для принудительного применения разрешений. Назначение роли состоит из трех элементов:

- Субъект безопасности
- Определение роли
- Область действия ресурса

Доступ предоставляется путем создания назначения ролей, а отзывается путем его удаления. [Назначения ролей можно создавать](roles-create-custom.md) с помощью портал Azure, Azure AD PowerShell и API Graph. [Назначения для пользовательской роли можно просмотреть](roles-view-assignments.md#view-the-assignments-of-a-role-with-single-application-scope-using-the-azure-ad-portal-preview)отдельно.

На приведенной ниже схеме показан пример назначения ролей. В этом примере Олегу «зеленый» назначена роль « [Администратор приложения](directory-assign-admin-roles.md#application-administrator) » в области приложения Salesforce. Крис не имеет доступа к управлению любым другим приложением, если только они не являются частью назначения ролей.

### <a name="security-principal"></a>Субъект безопасности

Субъект безопасности представляет пользователя или субъекта-службы, которым должен быть назначен доступ к ресурсам Azure AD. Пользователь является отдельным *пользователем* , у которого есть профиль пользователя в Azure Active Directory. *Субъект-служба* — это удостоверение безопасности, используемое приложениями или службами для доступа к конкретным ресурсам Azure AD.

Субъект безопасности похож на удостоверение пользователя в том, что он представляет имя пользователя и пароль или сертификат, но для приложения или службы, а не для пользователя.

### <a name="role"></a>Role

Определение роли, или роль, представляет собой набор разрешений. В определении роли перечислены операции, которые можно выполнять с ресурсами Azure AD, такие как создание, чтение, обновление и удаление. В Azure AD существует два типа ролей:

- Встроенные роли, созданные корпорацией Майкрософт, которые невозможно изменить. Встроенная роль глобального администратора имеет все разрешения на все ресурсы Azure AD.
- Настраиваемые роли, созданные и управляемые вашей организацией.

### <a name="scope"></a>`Scope`

Область — это ограничение разрешенных действий для конкретного ресурса Azure AD. При назначении роли можно указать область, которая ограничивает разрешенные действия администратора конкретным ресурсом. Например, если необходимо предоставить разработчику настраиваемую роль, но только для управления регистрацией конкретного приложения, можно включить регистрацию конкретного приложения в качестве области назначения роли.

  > [!Note]
  > Пользовательские роли можно назначать в области действия каталога и ресурса. Они еще не могут быть назначены в области административной единицы.
  > Встроенные роли могут быть назначены в области каталога, а в некоторых случаях — в области административной единицы. Они еще не могут быть назначены в области объекта.
