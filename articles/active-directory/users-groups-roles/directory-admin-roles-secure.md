# <a name="securing-privileged-access-for-hybrid-and-cloud-deployments-in-azure-ad"></a>Защита привилегированного доступа для гибридных и облачных развертываний в Azure AD

Безопасность бизнес-активов зависит от целостности привилегированных учетных записей, которые управляют системами ИТ. Кибератак-злоумышленники используют атаки кражи учетных данных для целевых учетных записей администраторов и другие привилегированные права доступа, чтобы попытаться получить доступ к конфиденциальным данным.

Для облачных служб предотвращение и реагирование являются общими обязанностями клиента и поставщика облачных служб. Дополнительные сведения о последних угрозах в отношении конечных точек и облачной среды см. в [этом отчете корпорации Майкрософт](https://www.microsoft.com/security/operations/security-intelligence-report). Эта статья может помочь вам разработать стратегию развития в соответствии с вашими текущими планами и описанными здесь указаниями.

> [!NOTE]
> Корпорация Майкрософт считает своим долгом обеспечивать высокий уровень доверия, прозрачность, соответствие стандартам и нормативным требованиям. Узнайте больше о том, как глобальная команда реагирования на инциденты Майкрософт уменьшает риск, связанный с атаками на облачные службы, и как система безопасности встроена в бизнес-продукты Майкрософт и облачные службы, в [разделе "Безопасность" в Центре управления безопасностью Майкрософт](https://www.microsoft.com/trustcenter/security), а также ознакомьтесь с целями соответствия Майкрософт в разделе [Соответствие](https://www.microsoft.com/trustcenter/compliance).

Обычно безопасность организации сосредоточена на точках входа и выхода сети в качестве периметра безопасности. Однако приложения SaaS и персональные устройства в Интернете сделали этот подход менее эффективным. В Azure AD мы заменяем периметр безопасности сети на проверку подлинности на уровне удостоверений вашей организации, при этом пользователи, которым назначены привилегированные административные роли, находятся в элементе управления. Их доступ должен быть защищен, независимо от того, является ли среда локальной, облачной или гибридной.

Для защиты привилегированного доступа необходимо внести изменения в:

* процессы, административные методы и управление знаниями.
* технические компоненты, такие как средства защиты узлов, защиту учетных записей и управление удостоверениями.

Обеспечьте безопасность привилегированного доступа способом, который управляется и сообщается в службах Майкрософт, о которых вы интересуют. Если у вас есть локальные учетные записи администратора, см. Руководство по локальному и гибридному привилегированному доступу в Active Directory в разделе [Защита привилегированного доступа](https://docs.microsoft.com/windows-server/identity/securing-privileged-access/securing-privileged-access).

> [!NOTE]
> Рекомендации в этой статье относятся преимущественно к функциям Azure Active Directory, которые включены в планы Azure Active Directory Premium P1 и P2. Azure Active Directory Premium P2 входит в пакет EMS E5 и пакет Microsoft 365 E5. В этом руководстве предполагается, что ваша организация уже приобрела для своих пользователей лицензии Azure AD Premium P2. Если у вас нет этих лицензий, некоторые из рекомендаций могут не распространяться на вашу организацию. Кроме того, в этой статье термин глобальный администратор (или глобальный администратор) означает то же самое, что и «администратор компании» или «администратор клиента».

## <a name="develop-a-roadmap"></a>Разработка стратегии

Корпорация Майкрософт рекомендует разработать и следовать стратегии для обеспечения защиты привилегированного доступа от кибератак. Вы всегда можете изменить свою стратегию с учетом текущих возможностей и конкретных требований в вашей организации. Каждый этап стратегии должен повышать затратность и сложность атаки на получение привилегированного доступа к вашим локальным, облачным и гибридным активам. Корпорация Майкрософт рекомендует следующие четыре этапа плана. Запланируйте наиболее эффективные и самые быстрые реализации первыми. Эта статья может быть вашим руководством, основанным на опытах корпорации Майкрософт в отношении кибератак и реагирования на атаки. Временные шкалы для этой стратегии являются приближениями.

![Этапы стратегии со сроками выполнения](./media/directory-admin-roles-secure/roadmap-timeline.png)

* Этап 1 (24–48 часов). Критические действия, которые рекомендуется выполнить сразу.

* Этап 2 (2–4 недели). Устранение рисков наиболее часто используемых методов атак.

* Этап 3 (1–3 месяца). Обеспечение прозрачности и полного контроля активности администратора.

* Этап 4 (шесть месяцев и более). Дальнейшее создания средств защиты для укрепления платформы безопасности.

Эта стратегия предназначена для эффективного использования технологий Майкрософт, которые возможно уже развернуты. Рассмотрите возможность привязки к любым средствам безопасности других поставщиков, которые уже развернуты или рассматривают развертывание.

## <a name="stage-1-critical-items-to-do-right-now"></a>Этап 1. критически важные элементы

![Первая стадия 1. критически важные элементы](./media/directory-admin-roles-secure/stage-one.png)

Первый этап стратегии сосредоточен на критических задачах, которые быстро и легко реализовать. Мы рекомендуем выполнить его в течение первых 24–48 часов, чтобы обеспечить базовый уровень безопасности привилегированного доступа. Этот этап стратегии по защите привилегированного доступа включает в себя приведенные ниже действия.

### <a name="general-preparation"></a>Общая подготовка

#### <a name="turn-on-azure-ad-privileged-identity-management"></a>Включение управления привилегированными пользователями Azure AD

Рекомендуется включить Azure AD Privileged Identity Management (PIM) в рабочей среде Azure AD. После включения PIM вы получите уведомления по электронной почте об изменениях роли привилегированного доступа. Уведомления предоставляют раннее предупреждение о добавлении дополнительных пользователей в роли с высоким уровнем привилегий.

Azure AD PIM входит в состав Azure AD Premium P2 и EMS E5. Чтобы обеспечить защиту доступа к приложениям и ресурсам локально и в облаке, подпишитесь на [Enterprise Mobility + Security бесплатную пробную версию в 90 дней](https://www.microsoft.com/cloud-platform/enterprise-mobility-security-trial). Azure AD Privileged Identity Management и защита идентификации Azure AD отслеживать действия по обеспечению безопасности с помощью отчетов, аудита и оповещений Azure AD.

После включения Azure AD Privileged Identity Management:

1. Войдите в [портал Azure](https://portal.azure.com/) с помощью учетной записи глобального администратора рабочей организации Azure AD.

2. Чтобы выбрать организацию Azure AD, в которой вы хотите использовать управление привилегированными пользователями, выберите имя пользователя в правом верхнем углу портал Azure.

3. В меню портал Azure выберите **все службы** и отфильтруйте список для **Azure AD privileged Identity Management**.

4. Выберите Privileged Identity Management в списке **Все службы** и закрепите плитку на панели мониторинга.

Первый пользователь, использующий PIM в вашей организации, назначается ролям администратора **безопасности** и **администратора привилегированных ролей** . Только администраторы привилегированных ролей могут управлять назначением ролей в каталоге Azure AD для пользователей. Мастер безопасности PIM поможет вам выполнить начальное обнаружение и назначение. Вы можете выйти из мастера, пока не внося каких-либо дополнительных изменений.

#### <a name="identify-and-categorize-accounts-that-are-in-highly-privileged-roles"></a>Определение и классифицирование учетных записей, которые находятся в привилегированных ролях

После включения Azure AD Privileged Identity Management просмотрите пользователей, которые находятся в следующих ролях Azure AD:

* Глобальный администратор.
* администратор привилегированных ролей;
* Администратор Exchange Online
* Администратор SharePoint Online

Если у вас нет Azure AD Privileged Identity Management в вашей организации, можно использовать [API PowerShell](https://docs.microsoft.com/powershell/module/azuread/get-azureaddirectoryrolemember?view=azureadps-2.0). Начните с роли глобального администратора, так как глобальный администратор имеет те же разрешения во всех облачных службах, на которые подписана ваша организация. Эти разрешения предоставляются независимо от того, где они были назначены: в центре администрирования Microsoft 365, портал Azure или модулем Azure AD для Microsoft PowerShell.

Удалите все учетные записи, которые больше не нужны в этих ролях. Затем классифицируйте оставшиеся учетные записи, назначенные ролям администраторов:

* Назначено пользователям с правами администратора, но также используется для неадминистративных целей (например, личной электронной почты).
* Назначено пользователям с правами администратора и используется только для административных целей
* совместно используется несколькими пользователями;
* для сценариев аварийного доступа;
* для автоматических скриптов;
* для внешних пользователей.

#### <a name="define-at-least-two-emergency-access-accounts"></a>Определение по крайней мере двух учетных записей аварийного доступа

Пользователь может случайно заблокировать свою роль. Например, если федеративный локальный поставщик удостоверений недоступен, пользователи не смогут войти в систему или активировать существующую учетную запись администратора. Вы можете подготовиться к случайному отсутствию доступа, сохранив две или более учетных записей для аварийного доступа.

Учетные записи аварийного доступа помогают ограничить привилегированный доступ в Организации Azure AD. Эти учетные записи имеют высокий уровень привилегий и не назначаются конкретным лицам. Учетные записи аварийного доступа ограничены аварийным выполнением сценариев, в которых нельзя использовать обычные административные учетные записи. Убедитесь, что учетная запись аварийного пользования контролируется и сокращается только на время, в течение которого она необходима.

Оцените учетные записи, которые назначены или могут получить роль глобального администратора. Если вы не видите облачных учетных записей с помощью \*домена. onmicrosoft.com (для аварийного доступа с эффектом "взломать"), создайте их. Дополнительные сведения см. в статье [Управление учетными записями администратора для аварийного доступа в Azure AD](directory-emergency-access.md).

#### <a name="turn-on-multi-factor-authentication-and-register-all-other-highly-privileged-single-user-non-federated-admin-accounts"></a>Включить многофакторную проверку подлинности и зарегистрировать все другие нефедеративные учетные записи администратора с высоким уровнем привилегий

Требуйте прохождения Многофакторной идентификации Azure (MFA) при входе в систему для всех отдельных пользователей, которым постоянно назначена одна или несколько из ролей администраторов Azure AD: глобальный администратор, администратор привилегированных ролей, администратор Exchange Online и администратор SharePoint Online. Используйте руководство по включению [MFA для ваших учетных записей администратора](../authentication/howto-mfa-userstates.md) и зарегистрируйте всех этих пользователей на странице [https://aka.ms/mfasetup](https://aka.ms/mfasetup). Более подробные сведения можно найти в шаге 2 и 3 руководства [по защите доступа к данным и службам в Office 365](https://support.office.com/article/Protect-access-to-data-and-services-in-Office-365-a6ef28a4-2447-4b43-aae2-f5af6d53c68e). 

## <a name="stage-2-mitigate-frequently-used-attacks"></a>Этап 2. Устранение часто используемых атак

![Этап 2. Устранение часто используемых атак](./media/directory-admin-roles-secure/stage-two.png)

Второй этап стратегии направлен на предотвращение наиболее часто используемых методов атак с целью кражи и злоумышленного использования учетных данных и может быть реализован примерно в течение 2–4 недель. Этот этап стратегии по защите привилегированного доступа включает в себя приведенные ниже действия.

### <a name="general-preparation"></a>Общая подготовка

#### <a name="conduct-an-inventory-of-services-owners-and-admins"></a>Инвентаризация служб, владельцев и администраторов

Увеличение "использования собственного устройства" и работа из домашних политик, а также увеличение беспроводного подключения делают критически важным отслеживание пользователей, подключающихся к сети. Аудит безопасности может раскрывать устройства, приложения и программы в сети, которые не поддерживаются Организацией и которые представляют высокий риск. Дополнительные сведения см. в статье [Обзор управления безопасностью Azure и мониторинга](../../security/fundamentals/management-monitoring-overview.md). Выполните все следующие задачи в рамках процесса инвентаризации.

* Определите пользователей, у которых есть административные роли, и службы, которыми они могут управлять.
* Используйте Azure AD PIM, чтобы узнать, какие пользователи в вашей организации имеют доступ администратора к Azure AD.
* Помимо ролей, определенных в Azure AD, Office 365 поставляется с набором ролей администратора, которые вы можете назначить пользователям в вашей организации. Каждая роль администратора сопоставляется с общими бизнес-функциями и дает пользователям в вашей организации разрешение выполнять определенные задачи в [центре администрирования Microsoft 365](https://admin.microsoft.com). Используйте центр администрирования Microsoft 365, чтобы узнать, какие пользователи в вашей организации имеют доступ администратора к Office 365, включая роли, не управляемые в Azure AD. Дополнительные сведения см. в статье [о ролях администраторов office 365](https://support.office.com/article/About-Office-365-admin-roles-da585eea-f576-4f55-a1e0-87090b6aaa9d) и [методиках безопасности для Office 365](https://docs.microsoft.com/office365/servicedescriptions/office-365-platform-service-description/office-365-securitycompliance-center).
* Выполните инвентаризацию в службах, от которых зависит ваша организация, например Azure, Intune или Dynamics 365.
* Убедитесь, что учетные записи используются в целях администрирования:

  * Подключены рабочие адреса электронной почты
  * Вы зарегистрировались для использования многофакторной идентификации Azure или используете локальный сервер MFA.
* Просите пользователей указать коммерческое обоснование для административного доступа.
* Удалите доступ администратора для тех лиц и служб, которым он не требуется.

#### <a name="identify-microsoft-accounts-in-administrative-roles-that-need-to-be-switched-to-work-or-school-accounts"></a>Определение учетных записей Майкрософт в административных ролях, которые необходимо переключить на использование рабочих или учебных учетных записей

Если начальные глобальные администраторы повторно используют существующие учетные данные учетная запись Майкрософт, когда они начали использовать Azure AD, замените учетные записи Майкрософт отдельными облачными или синхронизированными учетными записями.

#### <a name="ensure-separate-user-accounts-and-mail-forwarding-for-global-administrator-accounts"></a>Обеспечение отдельных учетных записей пользователей и перенаправление почты для глобальных учетных записей администратора

Личные учетные записи электронной почты регулярно подвергаются phishing-атакам кибератак, что делает неприемлемыми личные адреса электронной почты для учетных записей глобального администратора. Чтобы отделить Интернет от прав администратора, создайте выделенные учетные записи для каждого пользователя с правами администратора.

* Не забудьте создать отдельные учетные записи для пользователей, которые будут выполнять глобальные задачи администрирования.
* Убедитесь, что глобальные администраторы не открывают сообщения электронной почты и не запускают программы с учетными записями администратора.
* Убедитесь, что эти учетные записи имеют адрес электронной почты, пересылаемый в рабочий почтовый ящик.

#### <a name="ensure-the-passwords-of-administrative-accounts-have-recently-changed"></a>Проверка того, что пароли административных учетных записей недавно изменялись

Убедитесь, что все пользователи вошли в свои учетные записи администратора и изменили пароли по крайней мере один раз за последние 90 дней. Кроме того, убедитесь, что пароли в общих учетных записях были изменены недавно.

#### <a name="turn-on-password-hash-synchronization"></a>Включение синхронизации хэша паролей

Azure AD Connect синхронизирует хэш-значение хэша пароля пользователя из локальной Active Directory с облачной организацией Azure AD. Вы можете использовать синхронизацию хэша паролей в качестве резервной копии, если вы используете Федерацию с службы федерации Active Directory (AD FS) (AD FS). Эта резервная копия может быть полезной, если локальные Active Directory или AD FS серверы временно недоступны.

Синхронизация хэша паролей позволяет пользователям входить в службу, используя тот же пароль, который используется для входа в локальный экземпляр Active Directory. Синхронизация хэша паролей позволяет службе защиты идентификации обнаруживать скомпрометированные учетные данные путем сравнения хэшей паролей с паролями, которые считаются скомпрометированными. Дополнительные сведения см. в статье о [реализации синхронизации хэша паролей с помощью службы синхронизации Azure AD Connect](../hybrid/how-to-connect-password-hash-synchronization.md).

#### <a name="require-multi-factor-authentication-for-users-in-privileged-roles-and-exposed-users"></a>Требовать многофакторную проверку подлинности для пользователей привилегированных ролей и пользователей, имеющих доступ к ним

Azure AD рекомендует требовать многофакторную проверку подлинности (MFA) для всех пользователей. Не забудьте учесть пользователей, которые могли бы оказать значительные последствия, если их учетная запись была скомпрометирована (например, финансовые сотрудники). MFA снижает риск атаки из-за скомпрометированного пароля.

Включите:

* [MFA с использованием политик условного доступа](../authentication/howto-mfa-getstarted.md) для всех пользователей в Организации.

### <a name="general-preparation"></a>Общая подготовка

#### <a name="complete-an-access-review-of-users-in-administrator-roles"></a>Проверка доступа пользователей с ролью администратора

Другие корпоративные пользователи получают привилегированный доступ через облачные службы, что может привести к неуправляемому доступу. Сегодня пользователи могут стать глобальными администраторами для Office 365, администраторами подписки Azure или иметь доступ администратора к виртуальным машинам или через приложения SaaS.

В Организации должны быть все сотрудники обрабатывали обычные бизнес-транзакции как непривилегированные пользователи, а затем предоставлять права администратора по мере необходимости. Полный обзор доступа позволяет определить и подтвердить пользователей, которым разрешено активировать права администратора.

Мы рекомендуем следующее:

1. Определите, какие пользователи являются администраторами Azure AD, включите административный JIT-доступ по запросу и средства управления безопасностью на основе ролей.
2. Назначьте пользователям, у которых нет явного обоснования для использования привилегированного доступа администратора, другую роль (если нет допустимой роли, удалите их).

#### <a name="continue-rollout-of-stronger-authentication-for-all-users"></a>Развертывание более строгой аутентификации для всех пользователей

Требовать от высокодоступных пользователей использовать современные и строгие проверки подлинности, такие как Azure MFA или Windows Hello. Примерами высокодоступных пользователей являются:

* Руководители C Suite
* Диспетчеры высокого уровня
* Важная ИТ и персонал безопасности

#### <a name="use-dedicated-workstations-for-administration-for-azure-ad"></a>Использование выделенных рабочих станций для администрирования Azure AD

Злоумышленники могут попытаться выбрать привилегированные учетные записи, чтобы они могли нарушить целостность и подлинность данных. Они часто используют вредоносный код, который изменяет логику программы или отслеживает, что администратор вводит учетные данные. Рабочие станции с привилегированным доступом предоставляют выделенную операционную систему для конфиденциальных задач, защищенную от атак из Интернета и векторов угроз. Отделение этих конфиденциальных задач и учетных записей от ежедневно используемых рабочих станций и устройств обеспечивает надежную защиту от следующих компонентов:

* Фишинговые атаки
* Уязвимости в приложениях и операционных системах
* Атаки с олицетворением
* Атаки путем кражи учетных данных, такие как ведение журнала нажатия клавиш, передача-сигнал-hash и передача билета

Развертывание рабочих станций с привилегированным доступом позволяет снизить риск ввода учетных данных администраторами в настольной среде, которая не была зафиксирована. Дополнительные сведения см. в статье [Privileged Access Workstations](https://docs.microsoft.com/windows-server/identity/securing-privileged-access/privileged-access-workstations) (Рабочие станции с привилегированным доступом).

#### <a name="review-national-institute-of-standards-and-technology-recommendations-for-handling-incidents"></a>Просмотр рекомендаций по реагированию на инциденты системы безопасности компьютера Национального института стандартов и технологий

Национальный институт стандартов и технологий (NIST) предоставляет рекомендации по обработке инцидентов, в частности по анализу данных, связанных с инцидентами, и определению соответствующих мер реагирования на каждый из них. Дополнительные сведения см. в [руководстве по реагированию на инциденты системы безопасности компьютера Национального института стандартов и технологий (SP 800-61, 2 редакция)](https://nvlpubs.nist.gov/nistpubs/SpecialPublications/NIST.SP.800-61r2.pdf).

#### <a name="implement-privileged-identity-management-pim-for-jit-to-additional-administrative-roles"></a>Реализация PIM для JIT-доступа к дополнительным административным ролям

* Настроить принудительное прохождение MFA во время процесса активации.
* Использовать оповещения для информирования администраторов о внештатных изменениях.
* Предоставление пользователям возможности сохранения привилегированного доступа для предварительно настроенного времени
* Разрешить администраторам безопасности:

  * Обнаружение всех привилегированных удостоверений
  * Просмотр отчетов аудита
  * Создание проверок доступа для обнаружения всех пользователей, которым разрешено активировать права администратора

Если вы уже используете Azure AD PIM, установите сроки для ограниченных по времени привилегий (например, окна обслуживания).

Сканируйте и классифицируйте файлы в облаке и применяйте метки защиты данных Azure. #### <a name="monitor-activity-in-connected-cloud-apps"></a>Мониторинг действий в подключенных облачных приложениях

Рекомендуется использовать [Microsoft Cloud App Security](https://docs.microsoft.com/cloud-app-security/what-is-cloud-app-security) , чтобы обеспечить защиту доступа пользователей в подключенных приложениях. Эта функция обеспечивает безопасность корпоративного доступа к облачным приложениям и защищает учетные записи администратора, позволяя выполнять следующие задачи:

* расширить возможности видимости и управления облачными приложениями;
* создать политики для доступа, действий и совместного использования данных;
* автоматически идентифицировать рискованные действия, ненормальное поведение и угрозы;
* предотвратить утечку данных;
* минимизировать риск, автоматизировать предотвращение угроз и обеспечить принудительное применение политики.

Агент Cloud App Security SIEM интегрирует Cloud App Security с вашим сервером SIEM, чтобы обеспечить централизованный мониторинг оповещений и действий Office 365. Он запускается на вашем сервере, получает оповещения и действия из Cloud App Security и передает их на сервер SIEM. Дополнительные сведения см. в статье [Интеграция SIEM](https://docs.microsoft.com/cloud-app-security/siem).

Этап 4 стратегии должен быть реализован в течение шести месяцев и больше. Завершите свой план, чтобы усилить защиту привилегированного доступа от потенциальных атак, известных сегодня. Для угроз безопасности завтра рекомендуется просматривать систему безопасности в качестве непрерывного процесса, чтобы повысить затраты и сократить количество успешных злоумышленников, нацеленных на среду.

Защита привилегированного доступа важна для обеспечения безопасности ваших бизнес-ресурсов. Однако она должна быть частью полной программы безопасности, предоставляющей текущие гарантии безопасности. Эта программа должна включать такие элементы, как:

* Политика
* Операции
* Безопасность информации
* Серверы
* Приложения
* Компьютеры
* Устройства
* Cloud Fabric

При управлении учетными записями привилегированного доступа рекомендуется учитывать следующие рекомендации.

* Следите за тем, чтобы администраторы выполняли свои повседневные бизнес-действия в качестве непривилегированных пользователей
* Предоставьте привилегированный доступ только при необходимости и удалите его позже (JIT).
* Вести журналы действий аудита, связанные с привилегированными учетными записями

## <a name="break-glass-what-to-do-in-an-emergency"></a>Аварийный режим: что делать в чрезвычайной ситуации

![Учетные записи для доступа к аварийному прозрачному доступу](./media/directory-admin-roles-secure/emergency.jpeg)

## <a name="faq-answers-for-securing-privileged-access"></a>Часто задаваемые вопросы: ответы на вопросы о защите привилегированного доступа  

**Вопрос.** Что делать, если некоторые компоненты обеспечения безопасного доступа еще не реализованы?

**Ответ.** Определите не менее двух учетных записей для аварийного доступа, назначьте MFA привилегированным учетным записям администратора и отделите учетные записи пользователей от глобальных учетных записей администраторов.

**Вопрос.** После нарушения безопасности, что является главной проблемой, которую следует решить в первую очередь?

**Ответ.** Убедитесь, что вам необходима самая надежная проверка подлинности для пользователей с высокой доступностью.

**Вопрос.** Что произойдет, если наши привилегированные администраторы будут деактивированы?

**Ответ.** Создайте учетную запись глобального администратора, которая всегда обновляется.

**Вопрос.** Что произойдет, если остался только один глобальный администратор, и к нему нельзя получить доступ?

**Ответ.** Используйте одну из своих учетных записей для аварийного доступа, чтобы быстро получить привилегированный доступ.

**Вопрос.** Как защитить администраторов внутри моей организации?

**Ответ.** Администраторы всегда должны выполнять свою повседневную работу в качестве стандартных "непривилегированных" пользователей.

**Вопрос.** Каковы рекомендации для создания учетных записей администратора в Azure AD?

**Ответ.** Используйте привилегированный доступ для определенных задач администрирования.

**Вопрос.** Какие существуют средства для снижения постоянного административного доступа?

**Ответ.** Использование PIM и ролей администраторов Azure AD.

**Вопрос.** Какова позиция Майкрософт относительно синхронизации учетных записей администраторов в Azure AD?

**Ответ.** Учетные записи администратора уровня 0 используются только для локальных учетных записей AD. Такие учетные записи обычно не синхронизируются с Azure AD в облаке. Учетные записи администратора уровня 0 включают учетные записи, группы и другие ресурсы с прямым или косвенным администрированием локального Active Directory леса, доменов, контроллеров домена и ресурсов.

**Вопрос.** Как можно предотвратить случайное назначение административного доступа на портале?

**Ответ.** Используйте непривилегированные учетные записи для всех пользователей и большинства администраторов. Начните с разработки схемы сотрудников организации, чтобы определить, какие из немногих учетных записей администратора должны быть привилегированными. И отслеживать только что созданные административные пользователи.
