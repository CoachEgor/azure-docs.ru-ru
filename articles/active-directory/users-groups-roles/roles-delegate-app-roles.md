# <a name="delegate-app-registration-permissions-in-azure-active-directory"></a>Делегаты разрешения на регистрацию приложений в Active Directory Azure

В этой статье описывается, как использовать разрешения, выданные пользовательскими ролями, в каталоге Azure Active Directory (Azure AD) для удовлетворения потребностей управления приложениями. В Azure AD можно делегировать разрешения на создание приложений и управление следующим образом:

- [Ограничение того, кто может создавать приложения](#restrict-who-can-create-applications) и управлять создаваемыми приложениями. По умолчанию в Azure AD все пользователи могут регистрировать регистрации приложений и управлять всеми аспектами создаваемых ими приложений. Это может быть ограничено только отдельными людьми, что разрешение.
- [Назначение одного или нескольких владельцев в приложение.](#assign-application-owners) Это простой способ предоставить кому-либо возможность управлять всеми аспектами конфигурации Azure AD для конкретного приложения.
- [Назначение встроенной административной роли,](#assign-built-in-application-admin-roles) которая предоставляет доступ к конфигурации управления в Azure AD для всех приложений. Это рекомендуемый способ предоставления ИТ-специалистам доступа к управлению широкими разрешениями конфигурации приложений без предоставления доступа к управлению другими частями Azure AD, не связанными с конфигурацией приложений.
- [Создание пользовательской роли,](#create-and-assign-a-custom-role-preview) определяющей очень конкретные разрешения, и присвоение ее кому-либо либо в области одного приложения в качестве ограниченного владельца, либо в области каталога (все приложения) в качестве ограниченного администратора.

Важно рассмотреть вопрос о предоставлении доступа с помощью одного из вышеуказанных методов по двум причинам. Во-первых, делегирование возможности выполнения административных задач уменьшает накладные расходы глобального администратора. Во-вторых, использование ограниченных разрешений улучшает вашу безопасность и снижает вероятность несанкционированного доступа. Вопросы делегирования и общие рекомендации рассматриваются в статье [Делегирование администрирования в Azure Active Directory](roles-concept-delegation.md).

## <a name="restrict-who-can-create-applications"></a>Ограничьте возможность создания приложений

По умолчанию в Azure AD все пользователи могут регистрировать регистрации приложений и управлять всеми аспектами создаваемых ими приложений. Каждый человек также имеет возможность дать согласие на доступ к приложениям, доступным для данных компании от их имени. Вы можете выборно предоставлять эти разрешения, установив глобальные коммутаторы на "Нет" и добавив выбранных пользователей к роли разработчика приложений.

### <a name="to-disable-the-default-ability-to-create-application-registrations-or-consent-to-applications"></a>Отключить возможность по умолчанию создавать регистрации приложений или согласие на приложения

1. Вопийте в организации Azure AD с учетной записью, имеющей право на роль глобального администратора в вашей организации Azure AD.
1. Установите один или оба из следующих:

    - На [странице настроек пользователей для организации,](https://portal.azure.com/#blade/Microsoft_AAD_IAM/ActiveDirectoryMenuBlade/UserSettings)установить **пользователи могут зарегистрировать** настройки приложений в No. Это отсваит возможность по умолчанию для пользователей создавать регистрации приложений.
    - В [настройках пользователя для корпоративных приложений,](https://portal.azure.com/#blade/Microsoft_AAD_IAM/StartboardApplicationsMenuBlade/UserSettings/menuId/)установить **Пользователи могут дать согласие на доступ к данным компании от их имени** настройки No. Это отсутопустит возможность по умолчанию для пользователей дать согласие на доступ к данным компании от их имени.

### <a name="grant-individual-permissions-to-create-and-consent-to-applications-when-the-default-ability-is-disabled"></a>Предоставлять индивидуальные разрешения на создание и согласие на приложения, когда возможность по умолчанию отключена

Назначить роль разработчика приложения, чтобы предоставить возможность создавать регистрации **приложений,** когда пользователи могут зарегистрировать настройки приложений установлен в No. Эта роль также дает разрешение на согласие от своего имени, когда **пользователи могут дать согласие на приложения, имеющие доступ к данным компании от их имени** настройки установлен на нет. Как поведение системы, когда пользователь создает новую регистрацию приложения, он автоматически добавляется в качестве первого владельца. Разрешения на владение дают пользователю возможность управлять всеми аспектами регистрации приложения или корпоративного приложения, которым он владеет.

## <a name="assign-application-owners"></a>Назначить владельцев приложений

Назначение владельцев — это простой способ предоставить возможность управлять всеми аспектами конфигурации Azure AD для конкретной регистрации приложений или корпоративного приложения. Как поведение системы, когда пользователь создает новую регистрацию приложения, он автоматически добавляется в качестве первого владельца. Разрешения на владение дают пользователю возможность управлять всеми аспектами регистрации приложения или корпоративного приложения, которым он владеет. Первоначальный владелец может быть удален и дополнительные владельцы могут быть добавлены.

### <a name="enterprise-application-owners"></a>Владельцы корпоративных приложений

Как владелец, пользователь может управлять конфигурацией корпоративного приложения для организации, такой как единая конфигурация, подготовка и назначения пользователей. Владелец также может добавлять или удалять других владельцев. В отличие от глобальных администраторов, владельцы могут управлять только корпоративными приложениями, которыми они владеют.

В некоторых случаях корпоративные приложения, созданные из галереи приложений, включают как корпоративное приложение, так и регистрацию приложений. Когда это верно, добавление владельца в корпоративное приложение автоматически добавляет владельца к соответствующей регистрации приложения в качестве владельца.

### <a name="to-assign-an-owner-to-an-enterprise-application"></a>Назначить владельца корпоративному приложению

1. Восвай в свою  [организацию Azure AD](https://portal.azure.com/#blade/Microsoft_AAD_IAM/ActiveDirectoryMenuBlade/Overview)с учетной записью, имеющей право на администратора приложения или администратора облачных приложений для организации.
1. На  [странице регистрации приложений](https://portal.azure.com/#blade/Microsoft_AAD_IAM/StartboardApplicationsMenuBlade/AllApps/menuId/)для организации выберите приложение, чтобы открыть страницу Обзора для приложения.
1. Выберите **Владельцы** для просмотра списка владельцев в приложении.
1. Выберите **Добавить** для выбора одного или нескольких владельцев для добавления в приложение.

> [!IMPORTANT]
> Пользователи и директора служб могут быть владельцами регистрации заявлений. Владельцами корпоративных приложений могут быть только пользователи. Группы не могут быть назначены в качестве владельцев либо.
>
> Владельцы могут добавлять учетные данные в приложение и использовать эти учетные данные для выдачи себя за личность приложения. Приложение может иметь больше разрешений, чем владелец, и, таким образом, будет повышение привилегии над тем, что владелец имеет доступ к как пользователь или основной службы. Владелец приложения потенциально может создавать или обновлять пользователей или другие объекты, выдавая себя за приложение, в зависимости от разрешений приложения.

## <a name="assign-built-in-application-admin-roles"></a>Назначить встроенные роли админа приложений

Azure AD имеет набор встроенных ролей админов для предоставления доступа к конфигурации управления в Azure AD для всех приложений. Эти роли являются рекомендуемым способом предоставления ИТ-специалистам доступа к управлению широкими разрешениями конфигурации приложений без предоставления доступа к управлению другими частями Azure AD, не связанными с конфигурацией приложений.

- Администратор приложения: пользователи с этой ролью могут создавать и контролировать все аспекты корпоративных приложений, регистрации приложений, а также параметры прокси приложения. Эта роль также предоставляет возможность дать согласие на делегированные разрешения и разрешения приложений, исключая Microsoft Graph. Пользователи, назначенные на эту роль, не добавляются в качестве владельцев при создании новых регистраций приложений или корпоративных приложений.
- Администратор облачных приложений: пользователи с этой ролью имеют те же разрешения, что и для роли администратора приложений, за исключением возможности управления прокси приложения. Пользователи, назначенные на эту роль, не добавляются в качестве владельцев при создании новых регистраций приложений или корпоративных приложений.

Для получения дополнительной информации и просмотра описания для этих ролей [см.](directory-assign-admin-roles.md#available-roles)

Следуйте инструкциям в роли Назначать пользователям с помощью руководства [Azure Active Directory](../fundamentals/active-directory-users-assign-role-azure-portal.md) по присвоению функций администратора приложений или администратора облачных приложений.

> [!IMPORTANT]
> Администраторы приложений и администраторы облачных приложений могут добавлять учетные данные в приложение и использовать эти учетные данные для олицетворения личности приложения. Приложение может иметь разрешения, которые являются высотой привилегий над разрешениями роли админ. Админ в этой роли может потенциально создавать или обновлять пользователей или другие объекты, выдавая себя за приложение, в зависимости от разрешений приложения.
> Ни одна из ролей не предоставляет возможность управления параметрами условного доступа.

## <a name="create-and-assign-a-custom-role-preview"></a>Создание и присвоение пользовательской роли (предварительный просмотр)

Создание пользовательских ролей и назначение пользовательских ролей — это отдельные шаги:

- [Создайте пользовательское *определение роли* ](roles-create-custom.md) и [добавьте к нему разрешения из заранее установленного списка.](roles-custom-available-permissions.md) Это те же разрешения, которые используются во встроенных ролях.
- [Создайте *назначение роли* ](roles-assign-powershell.md) для назначения пользовательской роли.

Это разделение позволяет создать единое определение ролей, а затем назначить его много раз в разных *областях.* Пользовательская роль может быть назначена в общеорганизационном диапазоне или может быть назначена в области, если один объект Azure AD. Примером области объектов является регистрация единого приложения. Используя различные области, одно и то же определение роли может быть назначено Салли по всем регистрациям приложений в организации, а затем Naveen только по регистрации приложения Contoso Expense Reports.

Советы при создании и использовании пользовательских ролей для делегирования управления приложениями:
- Пользовательские роли предоставляют доступ только в самых современных лопастях регистрации приложений портала Azure AD. Они не предоставляют доступ к устаревшей лезвиям регистраций приложений.
- Пользовательские роли не предоставляют доступа к порталу Azure AD, когда настройка пользователя "Ограничить доступ к порталу администрирования Azure AD" установлена в "Да".
- Регистрации приложений, к которые пользователь имеет доступ с помощью ролевых заданий, отображаются только во вкладке "Все приложения" на странице регистрации приложения. Они не отображаются во вкладке "Принадлежавшие приложения".

Для получения дополнительной информации об основах пользовательских ролей, см [пользовательский обзор ролей,](roles-custom-overview.md)а также как [создать пользовательскую роль](roles-create-custom.md) и как [назначить роль.](roles-assign-powershell.md)
