# <a name="tutorial-configure-g-suite-for-automatic-user-provisioning"></a>Руководство по настройке G Suite для автоматической подготовки пользователей

Цель этого руководства состоит в том, чтобы продемонстрировать шаги, которые будут выполнены в G Suite и Azure Active Directory (Azure AD) для настройки Azure AD для автоматического предоставления и де-предоставления пользователям и/или группам G Suite.

> [!NOTE]
> В этом руководстве рассматривается соединитель, созданный на базе службы подготовки пользователей Azure AD. Подробные сведения о том, что делает эта служба, как она работает, и часто задаваемые вопросы см. в статье [Автоматическая подготовка пользователей и ее отзыв для приложений SaaS в Azure Active Directory](../app-provisioning/user-provisioning.md).

> [!NOTE]
> Разъем G Suite был недавно обновлен в октябре 2019 года. Изменения, внесенные в разъем G Suite, включают:
> - Добавлена поддержка дополнительных атрибутов G Suite и группы. 
> - Обновленные имена атрибутов целевых атрибутов G Suite, чтобы соответствовать тому, что определено [здесь.](https://developers.google.com/admin-sdk/directory)
> - Обновлено отображение атрибутов по умолчанию.

## <a name="prerequisites"></a>Предварительные требования

Чтобы настроить интеграцию Azure AD с G Suite, вам потребуется:

- клиент Azure AD;
- [Арендатор G Suite](https://gsuite.google.com/pricing.html)
- Учетная запись пользователя в G Suite с разрешениями Админа.

## <a name="assign-users-to-g-suite"></a>Назначение пользователей в G Suite

Активный каталог Azure использует концепцию, называемую назначениями, чтобы определить, какие пользователи должны получить доступ к выбранным приложениям. В контексте автоматической подготовки пользователей синхронизируются только пользователи и/или группы, назначенные приложению azure AD.

Прежде чем настроить и включить автоматическую подготовку пользователей, необходимо решить, какие пользователи и/или группы в Azure AD нуждаются в доступе к G Suite. После решения вы можете назначить этих пользователей и/или групп G Suite, следуя инструкциям здесь:

* [Назначение корпоративному приложению пользователя или группы](../manage-apps/assign-user-or-group-access-portal.md)

### <a name="important-tips-for-assigning-users-to-g-suite"></a>Важные советы по назначению пользователей G Suite

* Для тестирования автоматической конфигурации подготовки пользователей рекомендуется назначить одного пользователя Azure AD G Suite. Дополнительные пользователи и/или группы можно назначить позднее.

* При назначении пользователя G Suite необходимо выбрать любую действительную роль приложения (если она имеется) в диалоге о назначении. Пользователи с ролью **доступа по умолчанию** исключены из подготовки.

## <a name="setup-g-suite-for-provisioning"></a>Настройка G Suite для подготовки

Прежде чем настроить G Suite для автоматического предоставления пользовательского запроса с Azure AD, необходимо включить SCIM-подготовку на G Suite.

1. Воссоединитесь на [консоль G Suite Admin](https://admin.google.com/) с учетной записью администратора, а затем выберите **Security.** Если эта ссылка не отображается, она может быть скрыта в меню **More Controls** (Другие элементы управления) в нижней части экрана.

    ![Выбор страницы Security (Безопасность)][10]

1. На странице **Security** (Безопасность) выберите **API Reference** (Справочник по API).

    ![Выбор справочника по API][15]

1. Выберите **Включить доступ через API**.

    ![Выбор справочника по API][16]

   > [!IMPORTANT]
   > Для каждого пользователя, которого вы собираетесь предоставить G Suite, его имя пользователя в Azure AD **должно** быть привязано к пользовательскому домену. Например, такие имена пользователей, как bob@contoso.onmicrosoft.com, не будут приняты G Suite. а bob@contoso.com — будут. Вы можете изменить существующий домен пользователя, следуя инструкциям [здесь](https://docs.microsoft.com/azure/active-directory/fundamentals/add-custom-domain).

1. После добавления и проверки желаемых пользовательских доменов с помощью Azure AD вы должны проверить их снова с помощью G Suite. Для проверки доменов в G Suite, обратитесь к следующим шагам:

    а. В [консоли G Suite Admin](https://admin.google.com/)выберите **домены.**

    ![Выбор доменов][20]

    b. Выберите **Add a domain or a domain alias** (Добавить домен или псевдоним домена).

    ![Добавление нового домена][21]

    c. Выберите **Add another domain** (Добавить другой домен), а затем введите имя домена, который нужно добавить.

    ![Ввод доменного имени][22]

    d. Выберите **Continue and verify domain ownership** (Перейти к проверке владельца домена). Затем выполните действия, подтверждающие, что вы владеете доменным именем. Для получения исчерпывающих инструкций о том, как проверить свой домен с помощью Google, [см.](https://support.google.com/webmasters/answer/35179)

    д) Повторите предыдущие шаги для любых дополнительных доменов, которые вы собираетесь добавить в G Suite.

1. Затем определите, какую учетную запись админ вы хотите использовать для управления подготовкой пользователей в G Suite. Перейдите к **ролям админа**.

    ![Выбор Google Apps][26]

1. Для **роли админа** этой учетной записи отодвините **Привилегии** для этой роли. Убедитесь, что для учетной записи выбраны все параметры в разделе **Admin API Privileges** (Административные права доступа к API). Это позволит использовать ее для подготовки.

    ![Выбор Google Apps][27]

## <a name="add-g-suite-from-the-gallery"></a>Добавить G Suite из галереи

Для настройки G Suite для автоматического оформления пользовательского интерфейса с помощью Azure AD необходимо добавить G Suite из галереи приложений Azure AD в список управляемых приложений SaaS. 

1. На **[портале Azure](https://portal.azure.com)** на левой навигационной панели выберите **Active Directory Azure.**

    ![Кнопка Azure Active Directory](common/select-azuread.png)

1. Перейдите к **корпоративным приложениям,** а затем выберите **все приложения.**

    ![Колонка "Корпоративные приложения"](common/enterprise-applications.png)

1. Чтобы добавить новое приложение, выберите кнопку **Нового приложения** в верхней части панели.

    ![Кнопка "Создать приложение"](common/add-new-app.png)

1. В поле поиска введите **G Suite,** выберите **G Suite** в панели результатов, а затем нажмите кнопку **Добавить,** чтобы добавить приложение.

    ![G Suite в списке результатов](common/search-new-app.png)

## <a name="configuring-automatic-user-provisioning-to-g-suite"></a>Настройка автоматического обеспечения пользователя к G Suite 

В этом разделе вы проведет шаги по настройке службы предоставления AD Azure AD для создания, обновления и отработки пользователей и/или групп в G Suite на основе пользовательских и/или групповых заданий в Azure AD.

> [!TIP]
> Вы также можете включить SAML на основе одного знака для G Suite , следуя инструкциям, представленным в [G Suite Единый знак на учебник](https://docs.microsoft.com/azure/active-directory/saas-apps/google-apps-tutorial). Единый вход можно настроить независимо от автоматической подготовки пользователей, хотя эти две возможности дополняют друг друга.

> [!NOTE]
> Чтобы узнать больше о aPI API-аниди G Suite, обратитесь к [API каталога.](https://developers.google.com/admin-sdk/directory)

### <a name="to-configure-automatic-user-provisioning-for-g-suite-in-azure-ad"></a>Для настройки автоматического набора пользователей для G Suite в Azure AD:

1. Войдите на [портал Azure](https://portal.azure.com). Выберите **корпоративные приложения,** затем выберите **все приложения.**

    ![Колонка "Корпоративные приложения"](common/enterprise-applications.png)

1. В списке приложений выберите **G Suite**.

    ![Ссылка на G Suite в списке "Приложения"](common/all-applications.png)

1. Выберите вкладку **«Подготовка».**

    ![Вкладка подготовки](common/provisioning.png)

1. Установите **режим обеспечения** **автоматического.**

    ![Вкладка подготовки](common/provisioning-automatic.png)

1. В разделе **Учетные данные администратора** выберите **Авторизовать**. В новом окне браузера откроется диалоговое окно авторизации Google.

    ![G Suite Авторизация](media/google-apps-provisioning-tutorial/authorize.png)

1. Подтвердите, что вы хотите дать AD разрешений Azure AD для внесения изменений в ваш арендатор G Suite. Нажмите кнопку **Принять**.

    ![Подтверждение разрешения][28]

1. На портале Azure выберите **Проверка подключения**, чтобы убедиться, что Azure AD может подключиться к вашему приложению. Если подключение отсутствует, убедитесь, что у учетной записи G Suite есть разрешения администратора команды. Затем повторите шаг **авторизации**.

1. В поле **Почтовое уведомление** введите адрес электронной почты пользователя или группы, которые должны получать уведомления об ошибках подготовки, а также установите флажок **Send an email notification when a failure occurs** (Отправить уведомление по электронной почте при сбое).

    ![Почтовое уведомление](common/provisioning-notification-email.png)

1. Нажмите **Сохранить**.

1. В разделе **Картпинги** выберите **синхронизацию активных пользователей каталога Azure в G Suite.**

    ![G Suite Пользовательские картографы](media/google-apps-provisioning-tutorial/usermappings.png)

1. Просмотрите атрибуты пользователя, синхронизированные с Azure AD на G Suite в разделе **Атрибуты.** Атрибуты, выбранные в качестве свойств **соответствия,** используются для сопоставления учетных записей пользователей в G Suite для операций обновления. Нажмите кнопку **Сохранить**, чтобы зафиксировать все изменения.

    ![Атрибуты пользователей G Suite](media/google-apps-provisioning-tutorial/userattributes.png)

1. В разделе **Картпинги** выберите **синхронизацию активных групп каталогов Azure в G Suite.**

    ![G Suite Group Картпинги](media/google-apps-provisioning-tutorial/groupmappings.png)

1. Просмотрите атрибуты группы, синхронизированные с Azure AD с G Suite в разделе **Атрибуты Mapping.** Атрибуты, выбранные в качестве свойств **соответствия,** используются для соответствия группам в G Suite для операций обновления. Нажмите кнопку **Сохранить**, чтобы зафиксировать все изменения. В ui отображается набор отображений атрибутов по умолчанию между Azure AD и G Suite. Вы можете добавить дополнительные атрибуты, такие как Org Unit, нажав на добавление нового отображения.

    ![Атрибуты G Suite Group](media/google-apps-provisioning-tutorial/groupattributes.png)

1. Чтобы настроить фильтры области, ознакомьтесь со следующими инструкциями, предоставленными в [руководстве по фильтрам области](../app-provisioning/define-conditional-rules-for-provisioning-user-accounts.md).

1. Чтобы включить службу предоставления AD Azure AD для G Suite, измените **статус подготовки** в разделе **«Настройки».** **On**

    ![Состояние подготовки "Включено"](common/provisioning-toggle-on.png)

1. Определите пользователей и/или группы, которые вы хотели бы предоставить G Suite, выбрав желаемые значения в **разделе Область** в разделе **Параметры.**

    ![Область действия подготовки](common/provisioning-scope.png)

1. Когда будете готовы выполнить подготовку, нажмите кнопку **Сохранить**.

    ![Сохранение конфигурации подготовки](common/provisioning-configuration-save.png)

После этого начнется начальная синхронизация пользователей и (или) групп, определенных в поле **Область** раздела **Параметры**. Начальная синхронизация занимает больше времени, чем последующие операции синхронизации. Если служба запущена, они выполняются примерно каждые 40 минут. Раздел **«Детали синхронизации»** можно использовать для мониторинга прогресса и отслеживания ссылок на отчет о деятельности, в котором описаны все действия, выполняемые службой предоставления AD Azure AD в G Suite.

> [!NOTE]
> Если у пользователей уже есть существующая учетная запись с личным и потребителем, используя адрес электронной почты пользователя Azure AD, это может привести к некоторой проблеме, которая может быть решена с помощью инструмента передачи Google до выполнения синхронизации каталога.

Дополнительные сведения о чтении журналов подготовки Azure AD см. в руководстве по [отчетам об автоматической подготовке учетных записей](../app-provisioning/check-status-user-account-provisioning.md).

## <a name="common-issues"></a>Распространенные проблемы
* Сбои авторизации могут возникнуть, когда учетная запись, используемая для создания соединения, не предназначена для администратора в GSuite. Убедитесь, что учетная запись, используемая для авторизации доступа, имеет разрешения для админизации на **всех доменах,** с которыми необходимо предоставить пользователям. 
* Azure AD поддерживает отключение пользователей в GSuite, чтобы они не могли получить доступ к приложению, но не удаляет пользователей в GSuite.
