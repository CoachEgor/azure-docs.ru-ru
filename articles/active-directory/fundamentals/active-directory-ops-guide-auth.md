# <a name="azure-active-directory-authentication-management-operations-reference-guide"></a>Справочный справочник по операциям управления операциями по управлению операциями Azure Active Directory

В этом разделе [справочного руководства по операциям Azure AD](active-directory-ops-guide-intro.md) описаны проверки и действия, которые следует предпринять для защиты и управления учетными данными, определения опыта проверки подлинности, назначения делегирования, измерения использования и определения политик доступа на основе осанки безопасности предприятия.

> [!NOTE]
> Эти рекомендации актуальны на дату публикации, но могут меняться с течением времени. Организации должны постоянно оценивать свою практику идентификации по мере развития продуктов и служб Майкрософт с течением времени.

## <a name="key-operational-processes"></a>Ключевые операционные процессы

### <a name="assign-owners-to-key-tasks"></a>Назначать владельцам ключевые задачи

Управление Active Directory Azure требует непрерывного выполнения ключевых рабочих задач и процессов, которые могут не быть частью проекта развертывания. По-прежнему важно настроить эти задачи для оптимизации среды. Ключевыми задачами и рекомендуемыми владельцами являются:

| Задача | Владелец |
| :- | :- |
| Управление жизненным циклом конфигурации одного входного знака (SSO) в Azure AD | Оперативная группа IAM |
| Разработка политик условного доступа для приложений Azure AD | Архитектурная команда InfoSec |
| Архивная активность в системе SIEM | Оперативная группа InfoSec |
| Архивирование событий риска в системе SIEM | Оперативная группа InfoSec |
| Изучение и расследование отчетов о безопасности | Оперативная группа InfoSec |
| Провести исследование и исследовать события риска | Оперативная группа InfoSec |
| Проверка и исследование пользователей, помеченных для отчетов о рисках и уязвимостях из Azure AD Identity Protection | Оперативная группа InfoSec |

> [!NOTE]
> Защита идентификационных данных Azure AD требует лицензии Azure AD Premium P2. Чтобы найти правильную лицензию для ваших требований, [см. Сравнение общедоступных функций выпусков Azure AD Free и Azure AD Premium.](https://azure.microsoft.com/pricing/details/active-directory/)

При просмотре списка может потребоваться либо назначить владельца для задач, которые отсутствуют у владельца, либо настроить право собственности для задач с владельцами, которые не соответствуют рекомендациям выше.

## <a name="credentials-management"></a>Управление учетными данными

### <a name="password-policies"></a>Политики паролей

Безопасное управление паролями является одной из наиболее важных частей управления идентификацией и доступом и часто самой большой мишенью атак. Azure AD поддерживает несколько функций, которые могут помочь предотвратить успешное нападение.

Используйте таблицу ниже, чтобы найти рекомендуемое решение для смягчения проблемы, которая должна быть решена:

| Проблемы | Рекомендация |
| :- | :- |
| Нет механизма защиты от слабых паролей | Включить сбросить пароля самообслуживания Azure AD [(SSPR)](https://docs.microsoft.com/azure/active-directory/authentication/concept-sspr-howitworks) и [защиту паролем](https://docs.microsoft.com/azure/active-directory/authentication/concept-password-ban-bad-on-premises) |
| Нет механизма обнаружения утечек паролей | Включить [синхронизацию хэша паролей](https://docs.microsoft.com/azure/active-directory/hybrid/how-to-connect-password-hash-synchronization) (PHS) для получения информации |
| Использование AD FS и невозможность перехода к управляемой аутентификации | Включить [AD FS Extranet Smart Локаут](https://docs.microsoft.com/windows-server/identity/ad-fs/operations/configure-ad-fs-extranet-smart-lockout-protection) и / или [Azure AD Smart Локаут](https://docs.microsoft.com/azure/active-directory/authentication/howto-password-smart-lockout) |
| Политика паролей использует правила, основанные на сложности, такие как длина, несколько наборов символов или истечение срока действия | Пересмотрите в пользу [рекомендуемых методов Microsoft](https://www.microsoft.com/research/publication/password-guidance/?from=http%3A%2F%2Fresearch.microsoft.com%2Fpubs%2F265143%2Fmicrosoft_password_guidance.pdf) и переключите свой подход к управлению паролями и разключите [защиту паролей Azure AD.](https://docs.microsoft.com/azure/active-directory/authentication/concept-password-ban-bad) |
| Пользователи не зарегистрированы для использования многофакторной аутентификации (MFA) | [Зарегистрируйте всю информацию о безопасности пользователя,](https://docs.microsoft.com/azure/active-directory/identity-protection/howto-mfa-policy) чтобы она могла быть использована в качестве механизма для проверки личности пользователя вместе с его паролем |
| Нет отзыва паролей в зависимости от риска пользователя | Развертывание политик и риска пользователей Azure AD [Identity Protection,](https://docs.microsoft.com/azure/active-directory/identity-protection/howto-user-risk-policy) чтобы заставить изменить пароль на просочившихся учетных данных с помощью SSPR |
| Нет интеллектуального механизма блокировки для защиты вредоносной аутентификации от плохих субъектов, поступающих от идентифицированных IP-адресов | Развертывание облачной аутентификации с синхронизацией хэша пароля или [сквозной аутентификацией](https://docs.microsoft.com/azure/active-directory/hybrid/how-to-connect-pta-quick-start) (PTA) |

### <a name="enable-self-service-password-reset-and-password-protection"></a>Включить сбросить паролем самообслуживания и защиту паролем

Пользователи, нуждающиеся в изменении или сбросить свои пароли является одним из крупнейших источников объема и стоимости вызовов службы поддержки. В дополнение к стоимости, изменение пароля в качестве инструмента для снижения риска пользователя является фундаментальным шагом в улучшении безопасности позы вашей организации.

Как минимум, рекомендуется развернуть [сбросить пароля самообслуживания](https://docs.microsoft.com/azure/active-directory/authentication/concept-sspr-howitworks) Azure AD (SSPR) и [защиту от предварительных паролей](https://docs.microsoft.com/azure/active-directory/authentication/howto-password-ban-bad-on-premises-deploy) для выполнения:

- Отклонение вызовов службы поддержки.
- Замените использование временных паролей.
- Замените любое существующее решение для управления паролем самообслуживания, которое опирается на собственное решение.
- [Устранить слабые пароли](https://docs.microsoft.com/azure/active-directory/authentication/concept-password-ban-bad) в вашей организации.

> [!NOTE]
> Для организаций с подпиской Azure AD Premium P2 рекомендуется развертывать SSPR и использовать ее как часть [политики защиты пользователей](https://docs.microsoft.com/azure/active-directory/identity-protection/howto-user-risk-policy)от идентификации.

### <a name="strong-credential-management"></a>Сильное управление полномочиями

Пароли сами по себе не являются достаточно безопасными, чтобы предотвратить плохие субъекты от получения доступа к вашей среде. Как минимум, любой пользователь с привилегировой учетной записью должен быть включен для многофакторной аутентификации (MFA). В идеале, вы должны включить [комбинированную регистрацию](https://docs.microsoft.com/azure/active-directory/authentication/concept-registration-mfa-sspr-combined) и потребовать от всех пользователей зарегистрироваться для MFA и SSPR, используя [комбинированный опыт регистрации.](https://docs.microsoft.com/azure/active-directory/user-help/user-help-security-info-overview) В конце концов, мы рекомендуем вам принять стратегию, чтобы [обеспечить устойчивость,](https://docs.microsoft.com/azure/active-directory/authentication/concept-resilient-controls) чтобы уменьшить риск локаута из-за непредвиденных обстоятельств.

![Комбинированный поток пользовательского опыта](./media/active-directory-ops-guide/active-directory-ops-img4.png)

### <a name="on-premises-outage-authentication-resiliency"></a>Устойчивость проверки подлинности на месте

В дополнение к преимуществам простоты и позволяет обнаружение утечки учетных данных, Azure AD Password Hash Sync (PHS) и Azure MFA позволяют пользователям получить доступ к приложениям SaaS и Office 365, несмотря на перебои в работе из-за кибератак, таких как [NotPetya.](https://www.microsoft.com/security/blog/2018/02/05/overview-of-petya-a-rapid-cyberattack/) Кроме того, можно включить PHS в то время как в сочетании с федерацией. Включение PHS позволяет использовать резервную проверку подлинности, когда службы федерации недоступны.

Если в вашей организации нет стратегии устойчивости к простою или она не интегрирована с Azure AD, следует развернуть PHS Azure AD и определить план аварийного восстановления, который включает PHS. Включение Azure AD PHS позволит пользователям аутентифицировать сядую проверку в отношении Azure AD, если ваш вонщённый Active Directory будет недоступен.

![поток синхронизации хэша паролей](./media/active-directory-ops-guide/active-directory-ops-img5.png)

Чтобы лучше понять параметры аутентификации, [см.](../hybrid/choose-ad-authn.md)

### <a name="programmatic-usage-of-credentials"></a>Программное использование учетных данных

Скрипты Azure AD с использованием PowerShell или приложения, использующие API Microsoft Graph, требуют безопасной аутентификации. Плохое управление учетными данными, выполнение этих скриптов и инструментов, увеличивает риск кражи учетных данных. Если вы используете скрипты или приложения, которые полагаются на жесткие пароли или подсказки паролей, вы должны сначала просмотреть пароли в файлах для проверки или исходном коде, а затем заменить эти зависимости и использовать Управляемые идентификаторы Azure, интегрированную аутентификацию Windows или [сертификаты,](../reports-monitoring/tutorial-access-api-with-certificates.md) когда это возможно. Для приложений, где предыдущие решения невозможны, рассмотрите возможность использования [Azure Key Vault.](https://azure.microsoft.com/services/key-vault/)

Если вы определите, что существуют директора служб с учетными данными паролей, и вы не уверены, как эти учетные данные паролей защищены скриптами или приложениями, свяжитесь с владельцем приложения, чтобы лучше понять шаблоны использования.

Корпорация Майкрософт также рекомендует вам связаться с владельцами приложений, чтобы понять шаблоны использования, если есть принципы службы с учетными данными паролей.

## <a name="authentication-experience"></a>Опыт аутентификации

### <a name="on-premises-authentication"></a>Проверка подлинности на месте

Федеративная аутентификация с интегрированной аутентификацией Windows (IWA) или бесшовная единая проверка (SSO) управления аутентификацией с синхронизацией хэш-паролей или сквозной аутентификацией является лучшим пользовательским опытом, когда внутри корпоративной сети с прямой видимости для контроллеров доменов. Это сводит к минимуму быструю усталость учетных данных и снижает риск того, что пользователи становятся жертвами фишинговых атак. Если вы уже используете облачную аутентификацию с помощью PHS или PTA, но пользователям все еще необходимо ввести свой пароль при проверке подлинности на месте, то вы должны немедленно [развернуть Seamless SSO.](../hybrid/how-to-connect-sso.md) С другой стороны, если в настоящее время вы удовлетворены планами в конечном итоге перейти на облачную аутентификацию, то вы должны реализовать Seamless SSO в рамках проекта миграции.

### <a name="device-trust-access-policies"></a>Политики доверия устройств

Подобно пользователю в вашей организации, устройство — это еще одно основное удостоверение, которое необходимо защитить. Удостоверение устройства можно использовать для защиты ресурсов в любое время и из любого расположения.Аутентификации устройства и учета его типа доверия улучшает вашу безопасность и удобство использования:

- Избегая трения, например, с МИД, когда устройству доверяют
- Блокирование доступа от ненадежных устройств
- Для устройств Windows 10 организуем [единое ввоза на местные ресурсы.](../devices/azuread-join-sso.md)

Вы можете выполнить эту цель, привнося идентификаторы устройств и управляя ими в Azure AD, используя один из следующих методов:

- Организации могут использовать [Microsoft Intune](https://docs.microsoft.com/intune/what-is-intune) для управления устройством и обеспечения соблюдения политики соответствия, аттестировать работоспособность устройства и устанавливать условные политики доступа в зависимости от соответствия устройства. Microsoft Intune может управлять устройствами iOS, настольными компьютерами Mac (интеграция JAMF), настольными компьютерами Windows (насегодняшнив анамбудет с использованием управления мобильными устройствами для Windows 10 и совместного управления с Microsoft Endpoint Configuration Manager) и мобильными устройствами Android.
- [Гибридное соединение Azure AD](../devices/hybrid-azuread-join-managed-domains.md) обеспечивает управление с помощью групповых политик или менеджера конфигурации конечных точек Майкрософт в среде с устройствами, работающими на домене Active Directory. Организации могут развертывать управляемую среду либо через PHS, либо PTA с помощью Seamless SSO. Приведение устройств в Azure AD максимизирует производительность пользователя через SSO через облачные и внутренние ресурсы, позволяя одновременно обеспечивать доступ к облачным и наемным ресурсам с [помощью Условия доступа.](../conditional-access/overview.md) 

Если у вас есть устройства Windows, зарегистрированные в облаке, или устройства Windows, зарегистрированные в облаке, но без условных политик доступа, следует зарегистрировать незарегистрированные устройства и в любом случае [использовать соединение Hybrid Azure AD](../conditional-access/require-managed-devices.md) в качестве элемента управления в политике условного доступа.

![Скриншот гранта в политике условного доступа, требующей гибридного устройства](./media/active-directory-ops-guide/active-directory-ops-img6.png)

Если вы управляете устройствами с ПОМОЩЬю MDM или Microsoft Intune, но не используете элементы управления устройствами в условных политиках доступа, мы рекомендуем использовать [устройство Require, чтобы быть помеченным как соответствующий](../conditional-access/require-managed-devices.md#require-device-to-be-marked-as-compliant) элемент управления в этих политиках.

### <a name="windows-hello-for-business"></a>Windows Hello для бизнеса

В Windows 10 [Windows Hello for Business](https://docs.microsoft.com/windows/security/identity-protection/hello-for-business/hello-identity-verification) заменяет пароли сильной двухфакторной аутентификацией на ПК. Windows Hello for Business обеспечивает пользователям более рациональный интерфейс MFA и снижает зависимость от паролей. Если вы еще не начали развертывание устройств Windows 10 или лишь частично развернули их, мы рекомендуем вам обновить сядок до Windows 10 и [включить Windows Hello for Business](https://docs.microsoft.com/windows/security/identity-protection/hello-for-business/hello-manage-in-organization) на всех устройствах.

## <a name="application-authentication-and-assignment"></a>Аутентификация и назначение приложений

### <a name="single-sign-on-for-apps"></a>Единый входной знак для приложений

Предоставление стандартизированного единого механизма регистрации для всего предприятия имеет решающее значение для наилучшего пользовательского опыта, снижения риска, способности отчитываться и управления. Если вы используете приложения, поддерживающие SSO с Azure AD, но в настоящее время настроены на использование локальных учетных записей, следует перенастроить эти приложения для использования SSO с Azure AD. Аналогичным образом, если вы используете какие-либо приложения, которые поддерживают SSO с Azure AD, но используете другого поставщика идентификационных данных, следует перенастроить эти приложения для использования SSO с Azure AD. Для приложений, которые не поддерживают протоколы федерации, но поддерживают проверку подлинности на основе форм, мы рекомендуем настроить приложение для использования [хранилища паролей](../manage-apps/application-proxy-configure-single-sign-on-password-vaulting.md) с помощью Azure AD Application Proxy.

![Регистрация на основе пароля на AppProxy](./media/active-directory-ops-guide/active-directory-ops-img8.png)

> [!NOTE]
> Если у вас нет механизма обнаружения неуправляемых приложений в вашей организации, мы рекомендуем внедрить процесс обнаружения с помощью решения для брокера безопасности доступа к облаку (CASB), такого как [Microsoft Cloud App Security.](https://www.microsoft.com/enterprise-mobility-security/cloud-app-security)

### <a name="conditional-access-implementation"></a>Реализация условного доступа

Условный доступ является важным инструментом для улучшения безопасности вашей организации. Поэтому важно следовать этим рекомендациям:

- Убедитесь, что все приложения SaaS имеют по крайней мере одну применяемую политику
- Избегайте объединения фильтра **All apps** с управлением **блоком,** чтобы избежать риска блокировки
- Избегайте использования **всех пользователей** в качестве фильтра и непреднамеренного добавления **гостей**
- **Перенести все "устаревшие" политики на портал Azure**
- Поймать все критерии для пользователей, устройств и приложений
- Используйте политики условного доступа для [реализации МИД,](../conditional-access/plan-conditional-access.md)а не использовать **МИД на пользователя**
- Иметь небольшой набор основных политик, которые могут применяться к нескольким приложениям
- Определите пустые группы исключений и добавьте их в политики для стратегии исключения
- План для [разбивки стекла](../users-groups-roles/directory-admin-roles-secure.md#break-glass-what-to-do-in-an-emergency) счета без контроля МИД
- Обеспечьте согласованный опыт работы в приложениях Для клиентов Office 365, например, В группах команд, OneDrive для бизнеса, Outlook и т.д.) путем внедрения одного и того же набора элементов управления для таких сервисов, как Exchange Online и Sharepoint Online
- Назначение политики должно осуществляться группами, а не отдельными лицами
- Регулярно проводите обзоры групп исключений, используемых в политиках для ограничения времени, времени, необходимого для пользователей вне поля зрения безопасности. Если у вас есть Azure AD P2, то вы можете использовать обзоры доступа для автоматизации процесса

### <a name="legacy-authentication"></a>Устаревшие способы аутентификации

Сильные учетные данные, такие как MFA, не могут защитить приложения, используя устаревшие протоколы аутентификации, которые делают их предпочтительным вектором атаки злонамеренных субъектов. Блокировка устаревшей аутентификации имеет решающее значение для улучшения положения в области безопасности доступа.

Наследие аутентификации — это термин, относядайся к протоколам аутентификации, используемым приложениями, такими как:

- Пожилые клиенты Office, которые не используют современную аутентификацию (например, клиент Office 2010)
- Клиенты, которые используют почтовые протоколы, такие как IMAP/SMTP/POP

Злоумышленники решительно предпочитают эти протоколы - на самом деле, почти [100% атак спрей паролей](https://techcommunity.microsoft.com/t5/Azure-Active-Directory-Identity/Your-Pa-word-doesn-t-matter/ba-p/731984) используют устаревшие протоколы аутентификации! Хакеры используют устаревшие протоколы аутентификации, потому что они не поддерживают интерактивный входной, который необходим для дополнительных проблем безопасности, таких как многофакторная аутентификация и аутентификация устройства.

Если устаревшая аутентификация широко используется в вашей среде, следует запланировать как можно скорее перенести устаревших клиентов клиентам, [поддерживающим современную аутентификацию.](https://docs.microsoft.com/office365/enterprise/modern-auth-for-office-2013-and-2016) В то же время, если у вас есть некоторые пользователи, уже использующие современную аутентификацию, но другие, которые все еще используют устаревшую аутентификацию, вы должны принять следующие меры для блокировки устаревших клиентов аутентификации:

1. Используйте [отчеты о действиях Sign-In](https://docs.microsoft.com/azure/active-directory/reports-monitoring/concept-sign-ins) для идентификации пользователей, которые все еще используют устаревшую аутентификацию и планируют исправление:

   а. Обновление до современной аутентификации способных клиентов пострадавших пользователей.
   
   b. Запланируйте временный промежуток сокращения, чтобы заблокировать на ступеньки ниже.
   
   c. Определите, какие устаревшие приложения имеют жесткую зависимость от устаревшей аутентификации. Смотрите шаг 3 ниже.

2. Отменяйте устаревшие протоколы в источнике (например, Exchange Mailbox) для пользователей, которые не используют устаревший auth, чтобы избежать большего воздействия.
3. Для остальных учетных записей (в идеале нечеловеческих идентификаторов, таких как учетные записи служб) используйте [условный доступ для ограничения устаревших протоколов](https://techcommunity.microsoft.com/t5/Azure-Active-Directory-Identity/Azure-AD-Conditional-Access-support-for-blocking-legacy-auth-is/ba-p/245417) после проверки подлинности.

### <a name="traffic-from-unexpected-locations"></a>Трафик из неожиданных мест

Злоумышленники происходят из разных частей мира. Управление этим риском с помощью условных политик доступа с местом в качестве условия. [Условие местоположения](https://docs.microsoft.com/azure/active-directory/conditional-access/location-condition) политики условного доступа позволяет блокировать доступ к местам, откуда нет бизнес-причин для ввода.

![Создание нового имени](./media/active-directory-ops-guide/active-directory-ops-img14.png)

При наличии таковой используйте решение по вопросам безопасности и управлению событиями (SIEM) для анализа и поиска закономерностей доступа в разных регионах. Если вы не используете продукт SIEM или не используете информацию о проверке подлинности из Azure AD, мы рекомендуем использовать [Azure Monitor](https://docs.microsoft.com/azure/azure-monitor/overview) для выявления закономерностей доступа в разных регионах.

## <a name="access-usage"></a>Использование доступа

### <a name="azure-ad-logs-archived-and-integrated-with-incident-response-plans"></a>Журналы Azure AD, архивированные и интегрированные с планами реагирования на инциденты

Наличие доступа к активности, аудиту и событиям риска для Azure AD имеет решающее значение для устранения неполадок, анализа использования и проведения судебно-медицинских исследований. Azure AD предоставляет доступ к этим источникам через API REST, которые имеют ограниченный период хранения. Система управления информацией безопасности и событиями (SIEM), или эквивалентная архивная технология, является ключевым фактором для долгосрочного хранения аудита и поддержки. Для обеспечения долгосрочного хранения журналов Azure AD необходимо либо добавить их в существующее решение SIEM, либо использовать [Azure Monitor.](https://docs.microsoft.com/azure/active-directory/reports-monitoring/concept-activity-logs-azure-monitor) Архивные журналы, которые могут быть использованы в рамках ваших планов реагирования на инциденты и расследований.
