# <a name="what-are-security-defaults"></a>Что такое по умолчанию?

Управление безопасностью может быть затруднено, когда общие атаки, связанные с идентификацией, становятся все более и более популярными. Эти атаки включают спрей паролей, воспроизведение и фишинг.

По умолчанию безопасности в Active Directory Azure (Azure AD) упрощают безопасность и помогают защитить организацию. По умолчанию безопасности содержат преднастроенные настройки безопасности для общих атак. 

Корпорация Майкрософт делает систему безопасности доступными для всех. Цель состоит в том, чтобы обеспечить, чтобы все организации имели базовый уровень безопасности, включенный без каких-либо дополнительных затрат. Вы включите по умолчанию на портале Azure.

![Скриншот портала Azure с переключателем для обеспечения по умолчанию](./media/concept-fundamentals-security-defaults/security-defaults-azure-ad-portal.png)
 
> [!TIP]
> Если ваш арендатор был создан 22 октября 2019 года или после него, возможно, вы испытываете новое поведение по умолчанию и уже включили по умолчанию в вашем арендаторе. В целях защиты всех наших пользователей, по умолчанию безопасности в настоящее время развертывается для всех новых арендаторов, созданных.

Более подробную информацию о том, почему по умолчанию безопасности в настоящее время доступны можно найти в блоге Алекса Weinert, [Введение по умолчанию безопасности](https://techcommunity.microsoft.com/t5/azure-active-directory-identity/introducing-security-defaults/ba-p/1061414).

## <a name="unified-multi-factor-authentication-registration"></a>Единая многофакторная регистрация аутентификации

Все пользователи в вашем арендаторе должны зарегистрироваться для многофакторной аутентификации (MFA) в виде службы многофакторной аутентификации Azure. У пользователей есть 14 дней, чтобы зарегистрироваться для мультифакторной аутентификации с помощью приложения Microsoft Authenticator. По истечении 14 дней пользователь не сможет войти в систему до тех пор, пока не будет завершена регистрация мультифакторной аутентификации.

Мы понимаем, что некоторые пользователи могут быть вне офиса или не входить в систему в течение 14 дней сразу после включения по умолчанию. Для того, чтобы у каждого пользователя было достаточно времени, чтобы зарегистрироваться для Multi-Factor Authentication, 14-дневный период уникален для каждого пользователя. 14-дневный период пользователя начинается после первого успешного интерактивного вхламыва после включения по умолчанию.

## <a name="multi-factor-authentication-enforcement"></a>Многофакторная аутентификация

### <a name="protecting-administrators"></a>Защита администраторов

Пользователи, имеющих доступ к привилегированным учетным записям, имеют более широкий доступ к вашей среде. Из-за возможностей этих учетных записей при работе с ними следует соблюдать осторожность. Одним из распространенных методов повышения защиты привилегированных учетных записей является необходимость более сильной формы проверки счетов для входе в систему. В Azure AD вы можете получить более сильную проверку учетной записи, требуя многофакторной аутентификации.

После завершения регистрации с Multi-Factor Authentication следующие девять ролей администратора Azure AD должны будут выполнять дополнительную аутентификацию каждый раз, когда они вступят в систему:

- Глобальный администратор.
- администратор SharePoint;
- администратор Exchange;
- Администратор условного доступа
- администратор безопасности;
- Администратор службы поддержки или администратор паролей
- Администратор выставления счетов
- Администратор пользователей
- Администратор аутентификации

### <a name="protecting-all-users"></a>Защита всех пользователей

Мы склонны думать, что учетные записи администратора являются единственными учетными записями, которые нуждаются в дополнительных уровнях аутентификации. Администраторы имеют широкий доступ к конфиденциальной информации и могут вносить изменения в настройки по подписке. Но злоумышленники, как правило, нацелены на конечных пользователей. 

После того, как эти злоумышленники получат доступ, они могут запросить доступ к конфиденциальной информации от имени первоначального владельца учетной записи. Они даже могут загрузить весь каталог для выполнения фишинговой атаки на всю организацию. 

Одним из распространенных методов повышения защиты для всех пользователей является необходимость более сильной формы проверки учетной записи, такой как Multi-Factor Authentication, для всех. После того, как пользователи завершат регистрацию мультифакторной аутентификации, им будет предложено провести дополнительную аутентификацию, когда это необходимо.

### <a name="blocking-legacy-authentication"></a>Блокировка устаревшей аутентификации

Чтобы предоставить пользователям легкий доступ к облачным приложениям, Azure AD поддерживает различные протоколы аутентификации, включая устаревшую аутентификацию. *Наследие аутентификации* — это термин, относядайся к запросу на аутентификацию, сделанным:

- Клиенты, которые не используют современную аутентификацию (например, клиент Office 2010).
- Любой клиент, который использует старые почтовые протоколы, такие как IMAP, SMTP или POP3.

Сегодня большинство компрометирующих попыток ввода в систему происходят из устаревшей аутентификации. Наследие аутентификации не поддерживает многофакторную аутентификацию. Даже если в каталоге включена политика мультифакторной аутентификации, злоумышленник может проверить подлинность с помощью старого протокола и обойти многофакторную аутентификацию. 

После включения по умолчанию в вашем арендаторе все запросы аутентификации, сделанные старым протоколом, будут заблокированы. По умолчанию безопасности блокирует базовую аутентификацию Exchange Active Sync.

> [!WARNING]
> Прежде чем включить по умолчанию, убедитесь, что администраторы не используют старые протоколы проверки подлинности. Для получения дополнительной информации [см.](concept-fundamentals-block-legacy-authentication.md)

### <a name="protecting-privileged-actions"></a>Защита привилегированных действий

Организации используют различные службы Azure, управляемые через API-аналитику ресурсов Azure, включая:

- Портал Azure 
- Azure PowerShell 
- Azure CLI

Использование менеджера ресурсов Azure для управления службами является очень привилегированным действием. Менеджер ресурсов Azure может изменять конфигурации в масштабах всего клиента, такие как настройки служб и выставление счетов за подписку. Однофакторная аутентификация уязвима для различных атак, таких как фишинг и спрей паролей. 

Важно проверить личность пользователей, которые хотят получить доступ к менеджеру ресурсов Azure и обновить конфигурации. Вы проверяете их личность, требуя дополнительной аутентификации, прежде чем разрешить доступ.

После включения по умолчанию в вашем арендаторе, любой пользователь, который получает доступ к порталу Azure, Azure PowerShell или Azure CLI, должен будет завершить дополнительную аутентификацию. Эта политика распространяется на всех пользователей, которые имеют доступ к менеджеру ресурсов Azure, независимо от того, являются ли они администратором или пользователем. 

> [!NOTE]
> До 2017 Exchange Online арендаторы имеют современную аутентификацию отключена по умолчанию. Для того, чтобы избежать возможности цикла входа при аутентификации через этих арендаторов, необходимо [включить современную аутентификацию.](https://docs.microsoft.com/exchange/clients-and-mobile-in-exchange-online/enable-or-disable-modern-authentication-in-exchange-online)

> [!NOTE]
> Учетная запись синхронизации Azure AD Connect исключена из по умолчанию безопасности и не будет предложено зарегистрироваться для многофакторной аутентификации или выполнить ее. Организации не должны использовать эту учетную запись для других целей.

### <a name="authentication-methods"></a>Методы проверки подлинности

По умолчанию безопасности позволяют регистрации и использования Azure Multi-фактор аутентификации **с помощью только приложения Microsoft Authenticator с помощью уведомлений.** Условный доступ позволяет использовать любой метод проверки подлинности, который администратор выбирает для включения.

|   | Параметры безопасности по умолчанию | Условный доступ |
| --- | --- | --- |
| Уведомление в мобильном приложении | X | X |
| Код проверки из мобильного приложения или токен оборудования |   | X |
| SMS на телефон |   | X |
| Звонок на телефон |   | X |
| Пароли приложений |   | ХЗЗ |

Пароли приложений доступны только в МИД для пользователей с устаревшими сценариями проверки подлинности только в том случае, если они включены администраторами.

### <a name="conditional-access"></a>Условный доступ

Условный доступ можно использовать для настройки политик, аналогичных по умолчанию безопасности, но с большей детализацией, включая исключения пользователей, которые не доступны в по умолчанию безопасности. Если вы используете Условный доступ и в вашей среде включены политики условного доступа, по умолчанию безопасности не будут доступны для вас. Если у вас есть лицензия, обеспечивающая условный доступ, но не имеюющая политики условного доступа в вашей среде, вы можете использовать по умолчанию безопасности до тех пор, пока не включите политики условного доступа. Более подробную информацию о лицензировании Azure AD можно найти на [странице ценообразования Azure AD.](https://azure.microsoft.com/pricing/details/active-directory/)

## <a name="enabling-security-defaults"></a>Включение по умолчанию безопасности

Для включения по умолчанию в каталоге:

1. Вопиюсь на [портал](https://portal.azure.com) Azure в качестве администратора безопасности, администратора условного доступа или глобального администратора.
1. Просмотрите свойства **Properties** **активных каталогов Azure Active Directory.** >
1. Выберите **Управление по умолчанию безопасности.**
1. Установите **переключатель по умолчанию безопасности Включить** на **Yes.**
1. Щелкните **Сохранить**.

## <a name="disabling-security-defaults"></a>Отключение по умолчанию безопасности

Организации, выбирающие реализацию политик «Условный доступ», заменяющие по умолчанию безопасности, должны отключить по умолчанию. 

![Предупреждение сообщения отключает по умолчанию безопасности для включения условного доступа](./media/concept-fundamentals-security-defaults/security-defaults-disable-before-conditional-access.png)

Чтобы отключить по умолчанию в каталоге:

1. Вопиюсь на [портал](https://portal.azure.com) Azure в качестве администратора безопасности, администратора условного доступа или глобального администратора.
1. Просмотрите свойства **Properties** **активных каталогов Azure Active Directory.** >
1. Выберите **Управление по умолчанию безопасности.**
1. Установите **переключатель по умолчанию включить систему безопасности** в **No.**
1. Щелкните **Сохранить**.
