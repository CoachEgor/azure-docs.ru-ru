# <a name="blocking-legacy-authentication"></a>Блокировка устаревшей аутентификации
 
Чтобы предоставить пользователям удобный доступ к облачным приложениям, Azure Active Directory (AAD) поддерживает широкий набор протоколов проверки подлинности, часть из которых уже устарела. Наследие аутентификации — это термин, относядайся к запросу на аутентификацию, сделанным:

- Пожилые клиенты Office, которые не используют современную аутентификацию (например, клиент Office 2010)
- Любой клиент, который использует устаревшие почтовые протоколы, такие как IMAP/SMTP/POP3

Сегодня большинство попыток компрометации ввода приходят из устаревшей аутентификации. Наследие аутентификации не поддерживает многофакторную аутентификацию (MFA). Даже если в каталоге включена политика МИД, плохой актер может проверить подлинность с помощью устаревшего протокола и обойти МИД. Лучший способ защитить свою учетную запись от вредоносных запросов на аутентификацию, сделанных устаревшими протоколами, — это полностью заблокировать эти попытки.

## <a name="identify-legacy-authentication-use"></a>Определение устаревшего использования аутентификации

Прежде чем вы сможете заблокировать устаревшую аутентификацию в каталоге, необходимо сначала понять, есть ли у пользователей приложения, которые используют устаревшую аутентификацию, и как она влияет на общий каталог. Для понимания того, используете ли вы устаревшую аутентификацию, можно использовать журналы входа в систему Azure AD.

1. Перейдите на портал Azure > Active Directory > входящих в систему.
1. Добавьте столбец приложения Клиента, если оно не отображается, нажав на колонки > клиентское приложение.
1. Фильтр по клиенту App > проверить все другие варианты клиента, представленные и нажмите Применить.
1. Фильтр по статусу > успех и нажмите Применить. 
1. При необходимости расширьте диапазон дат с помощью фильтра Date.

Фильтрация покажет только успешные попытки входной системы, предпринятые выбранными устаревшими протоколами аутентификации. Нажатие на каждую индивидуальную попытку входного в ввешания покажет вам дополнительные детали. Столбец клиентского приложения или поле приложения клиента под вкладкой Basic Info после выбора отдельного ряда данных будет указывать, какой устаревший протокол проверки подлинности был использован. Эти журналы будут указывать, какие пользователи по-прежнему в зависимости от устаревшей аутентификации и какие приложения используют устаревшие протоколы для запросов на аутентификацию. Для пользователей, которые не отображаются в этих журналах и подтверждены не использовать устаревшую аутентификацию, реализуйте политику условного доступа или включите политику Базовой линии: блокировать устаревшую аутентификацию только для этих пользователей.

## <a name="moving-away-from-legacy-authentication"></a>Отход от устаревшей аутентификации 

Если у вас есть лучшее представление о том, кто использует устаревшую аутентификацию в каталоге и какие приложения зависят от него, следующим шагом является обновление пользователей с использованием современной аутентификации. Современная аутентификация — это метод управления идентификацией, который обеспечивает более безопасную аутентификацию и авторизацию пользователей. Если в каталоге есть политика МИД, современная аутентификация гарантирует, что пользователь будет предложен для МИД, когда это необходимо. Это более безопасная альтернатива устаревшим протоколам аутентификации.

В этом разделе дается пошаговое обзор того, как обновить среду до современной аутентификации. Прочитайте приведенные ниже шаги, прежде чем включить в организацию политику блокировки устаревшей проверки подлинности.

### <a name="step-1-enable-modern-authentication-in-your-directory"></a>Шаг 1: Включить современную аутентификацию в каталоге

Первым шагом в обеспечении современной аутентификации является обеспечение того, чтобы ваш каталог поддерживал современную аутентификацию. Современная аутентификация включена по умолчанию для каталогов, созданных 1 августа 2017 года или после нее. Если каталог был создан до этой даты, вам необходимо вручную включить современную аутентификацию для вашего каталога, используя следующие действия:

1. Проверьте, поддерживает ли ваш каталог современную `Get-CsOAuthConfiguration` аутентификацию, запустив [модуль Skype для Business Online PowerShell.](https://docs.microsoft.com/office365/enterprise/powershell/manage-skype-for-business-online-with-office-365-powershell)
1. Если ваша команда `OAuthServers` возвращает пустое свойство, то современная аутентификация отключена. Обновление настройки, чтобы позволить современную аутентификацию с помощью. `Set-CsOAuthConfiguration` Если `OAuthServers` ваша собственность содержит запись, вы хорошо идти.

Не забудьте завершить этот шаг, прежде чем двигаться вперед. Очень важно сначала изменить конфигурации каталога, поскольку они диктуют, какой протокол будет использоваться всеми клиентами Office. Даже если вы используете клиентов Office, поддерживающих современную аутентификацию, они будут по умолчанию использовать устаревшие протоколы, если современная аутентификация отключена в вашем каталоге.

### <a name="step-2-office-applications"></a>Шаг 2: Офисные приложения

После включения современной аутентификации в каталоге можно начать обновление приложений, включив современную аутентификацию для клиентов Office. Office 2016 или более поздние клиенты поддерживают современную аутентификацию по умолчанию. Дополнительных шагов не требуется.

Если вы используете клиенты Office 2013 Windows или старше, мы рекомендуем перейти на Office 2016 или позже. Даже после завершения предыдущего шага обеспечения современной аутентификации в каталоге старые приложения Office будут продолжать использовать устаревшие протоколы аутентификации. Если вы используете клиентов Office 2013 и не можете немедленно перейти на Office 2016 или позже, выполните следующие действия в следующей статье, чтобы [включить современную аутентификацию для Office 2013 на устройствах Windows.](https://docs.microsoft.com/office365/admin/security-and-compliance/enable-modern-authentication) Чтобы защитить вашу учетную запись во время использования устаревшей аутентификации, мы рекомендуем использовать надежные пароли в каталоге. Ознакомьтесь с  [защитой паролей Azure AD,](../authentication/concept-password-ban-bad.md)чтобы запретить слабые пароли в каталоге.

Office 2010 не поддерживает современную аутентификацию. Вам нужно будет обновить любых пользователей с Office 2010 до более последней версии Office. Мы рекомендуем перейти на Office 2016 или позже, так как он блокирует устаревшую аутентификацию по умолчанию.

Если вы используете MacOS, мы рекомендуем перейти на Office для Mac 2016 или позже. Если вы используете родной почтовый клиент, вам нужно будет иметь версию MacOS 10.14 или позже на всех устройствах.

### <a name="step-3-exchange-and-sharepoint"></a>Шаг 3: Обмен и SharePoint

Для того чтобы клиенты Outlook на базе Windows использовали современную аутентификацию, Exchange Online также должна быть включена в современную аутентификацию. Если современная аутентификация отключена для Exchange Online, клиенты Outlook на базе Windows, поддерживающие современную аутентификацию (Outlook 2013 или более поздний), будут использовать базовую аутентификацию для подключения к почтовым ящикам Exchange Online.

SharePoint Online включен для современной аутентификации по умолчанию. Для каталогов, созданных после 1 августа 2017 года, современная аутентификация включена по умолчанию в Exchange Online. Однако, если вы ранее отключили современную аутентификацию или используете каталог, созданный до этой даты, выполните следующие действия в следующей статье, чтобы [включить современную аутентификацию в Exchange Online.](https://docs.microsoft.com/exchange/clients-and-mobile-in-exchange-online/enable-or-disable-modern-authentication-in-exchange-online)

### <a name="step-4-skype-for-business"></a>Шаг 4: Skype для бизнеса

Чтобы предотвратить устаревшие запросы на аутентификацию, сделанные Skype для бизнеса, необходимо включить современную аутентификацию для Skype для Business Online. Для каталогов, созданных после 1 августа 2017 года, современная аутентификация для Skype для бизнеса включена по умолчанию.

Мы предлагаем вам перейти на Microsoft Teams, которая поддерживает современную аутентификацию по умолчанию. Однако, если вы не можете мигрировать в это время, вам нужно будет включить современную аутентификацию для Skype для Business Online, чтобы Skype для бизнес-клиентов начать использовать современную аутентификацию. Следуйте шагам в этой статье [Skype для бизнес-топологий поддерживается современной аутентификации](https://docs.microsoft.com/skypeforbusiness/plan-your-deployment/modern-authentication/topologies-supported), чтобы современная аутентификация для Skype для бизнеса.

В дополнение к обеспечению современной аутентификации для Skype для Business Online, мы рекомендуем включить современную аутентификацию для Exchange Online при включении современной аутентификации для Skype для бизнеса. Этот процесс поможет синхронизировать состояние современной аутентификации в Exchange Online и Skype для бизнеса онлайн и предотвратит несколько запросов для Skype для бизнес-клиентов.

### <a name="step-5-using-mobile-devices"></a>Шаг 5: Использование мобильных устройств

Приложения на мобильном устройстве также должны блокировать устаревшую аутентификацию. Мы рекомендуем использовать Outlook для мобильных устройств. Outlook for Mobile поддерживает современную аутентификацию по умолчанию и удовлетворит другие базовые политики защиты МИД.

Для того, чтобы использовать родной почтовый клиент iOS, вам нужно будет запустить версию iOS 11.0 или позже, чтобы убедиться, что почтовый клиент был обновлен, чтобы заблокировать устаревшую аутентификацию.

### <a name="step-6-on-premises-clients"></a>Шаг 6: Клиенты на территории

Если вы являетесь гибридным клиентом, использующим Exchange Server на месте и Skype для бизнеса, обе службы должны быть обновлены, чтобы обеспечить современную аутентификацию. При использовании современной аутентификации в гибридной среде вы по-прежнему проверяете подлинность пользователей на месте. История авторизации их доступа к ресурсам (файлам или электронным письмам) меняется.

Прежде чем вы сможете начать включение современной аутентификации на месте, пожалуйста, убедитесь, что вы выполнили предварительные требования. Теперь вы готовы включить современную аутентификацию на месте.

Шаги для обеспечения современной аутентификации можно найти в следующих статьях:

* [Как настроить Exchange Server на месте, чтобы использовать гибридную современную аутентификацию](https://docs.microsoft.com/office365/enterprise/configure-exchange-server-for-hybrid-modern-authentication)
* [Как использовать современную аутентификацию (ADAL) с Помощью Skype для бизнеса](https://docs.microsoft.com/skypeforbusiness/manage/authentication/use-adal)
