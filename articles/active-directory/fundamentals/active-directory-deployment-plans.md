# <a name="azure-active-directory-deployment-plans"></a>Планы развертывания Azure Active Directory
Ищете сквозные рекомендации по развертыванию возможностей Active Directory Azure (Azure AD)? Планы развертывания Azure AD проработави бизнес-ценность, соображения планирования и оперативные процедуры, необходимые для успешного развертывания общих возможностей Azure AD.

На любой странице плана используйте возможность печати вашего браузера для создания актуальной автономной версии документации.
## <a name="include-the-right-stakeholders"></a>Включить нужные заинтересованные стороны

При начале планирования развертывания новой возможности важно включить ключевые заинтересованные стороны в организацию. Мы рекомендуем вам определить и задокументировать человека или людей, выполняющие каждую из следующих ролей, и работать с ними, чтобы определить их участие в проекте.  

Роли могут включать в себя следующие 

|Роль |Описание |
|-|-|
|Конечный пользователь|Репрезентативная группа пользователей, для которой эта возможность будет реализована. Часто просматривает изменения в пилотной программе.
|Менеджер иТ-поддержки|Представитель организации ИТ-поддержки, который может предоставить данные о поддержке этого изменения с точки зрения справочной службы.  
|Архитектор идентификации или глобальный администратор Azure|Представитель группы управления идентификационными удостоверениями, отвечающий за определение соответствия этого изменения основной инфраструктуре управления идентификационными данных в вашей организации.|
|Владелец бизнеса приложений |Общий владелец бизнеса пострадавшего приложения (ы), который может включать управление доступом.Может также предоставить данные о пользовательском опыте и полезности этого изменения с точки зрения конечного пользователя.
|Владелец безопасности|Представитель группы безопасности, который может подписать, что план будет соответствовать требованиям безопасности вашей организации.|
|Диспетчер соответствия|Лицо, внее вашей организации, ответственное за обеспечение соответствия корпоративным, отраслевым или правительственным требованиям.|

**Уровни участия могут включать:**

- **R**esponsible для реализации плана и результатов проекта 

- **Pproval**плана проекта и результатов 

- **C**ontributor к плану проекта и результату 

- **Я**nformed плана проекта и результатов


## <a name="best-practices-for-a-pilot"></a>Лучшие практики для пилота
Пилот позволяет проверить с небольшой группой, прежде чем включить возможность для всех. Убедитесь, что в рамках тестирования каждый случай использования в вашей организации тщательно проверяется. Лучше всего ориентироваться на определенную группу пилотных пользователей, прежде чем выкатывать это в вашу организацию в целом.

В первой волне нацелитесь на ИТ,, удобство использования и других подходящих пользователей, которые могут протестировать и предоставить обратную связь. Эта обратная связь должна использоваться для дальнейшей разработки сообщений и инструкций, которые вы отправляете своим пользователям, а также для того, чтобы дать представление о типах вопросов, которые могут видеть сотрудники службы поддержки. 

Расширение развертывания для более крупных групп пользователей должно осуществляться за счет увеличения охвата целевой группы(ы) целевых. Это может быть сделано путем [динамического членства](../users-groups-roles/groups-dynamic-membership.md)в группе или путем ручного добавления пользователей в целевую группу (ы).
