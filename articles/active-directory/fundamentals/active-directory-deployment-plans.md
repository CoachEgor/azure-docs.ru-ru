# <a name="azure-active-directory-deployment-plans"></a>Планы развертывания Azure Active Directory
Ищете комплексное руководство по развертыванию некоторых возможностей Azure Active Directory (Azure AD)? В следующих планах развертывания рассматриваются бизнес-преимущества, вопросы планирования, разработки и операционные процедуры, необходимые для успешного развертывания нескольких наиболее распространенных возможностей Azure AD. 

В документах вы найдете шаблоны электронной почты, схемы системной архитектуры, общие тестовые случаи и многое другое. 

Мы будем рады вашим отзывам. Пройдите этот короткий [опрос](https://aka.ms/deploymentplanfeedback), чтобы оценить пользу приведенных документов. 

## <a name="include-the-right-stakeholders"></a>Включение нужных заинтересованных лиц

При планировании развертывания для новой возможности важно включить ключевые заинтересованные лица в Организации. Мы рекомендуем определить и задокументировать человека или людей, которые выполняют каждую из следующих ролей, и работать с ними, чтобы определить их участие в проекте.  

К ролям могут относиться следующие 

|Role |Описание |
|-|-|
|Конечный пользователь|Репрезентативная группа пользователей, для которых будет реализована возможность. Часто предварительный просмотр изменений в пилотной программе.
|Диспетчер технической поддержки|Он поддерживает представителя Организации, который может предоставлять входные данные о поддержке этого изменения с точки зрения службы поддержки.  
|Архитектор удостоверений или глобальный администратор Azure|Специалист группы управления удостоверениями, отвечающий за определение того, как это изменение согласуется с основной инфраструктурой управления удостоверениями в вашей организации.|
|Владелец бизнес-приложения |Общий бизнес-владелец затронутых приложений, который может включать в себя управление доступом.  Может также предоставить входные данные о работе пользователя и полезность этого изменения с точки зрения конечного пользователя.
|Владелец безопасности|Специалист из группы безопасности, который может подписаться на то, что план будет соответствовать требованиям безопасности вашей организации.|
|Диспетчер соответствия|Лицо в Организации несет ответственность за обеспечение соответствия требованиям Организации, отрасли или правительственных органов.|

**Уровни участия могут включать:**

- **R**еспонсибле для реализации плана проекта и результата 

- **Ппровал плана**и результата проекта 

- План и результат в плане проекта **C** 

- **Я**нформед план и результат проекта
 
## <a name="deployment-plans"></a>Планы развертывания



|Сценарий |Описание |
|-|-|
|[Многофакторная идентификация](../authentication/howto-mfa-getstarted.md)|Многофакторная идентификация (MFA) Azure — решение Майкрософт для двухэтапной проверки. С помощью утвержденных администратором методов проверки подлинности Azure MFA помогает защитить доступ к данным и приложениям, при этом не усложняя простой процесс входа в систему.|
|[Условный доступ](https://aka.ms/deploymentplans/ca)|С помощью условного доступа можно реализовать автоматизированные решения по контролю доступа, которые могут получать доступ к облачным приложениям на основе условий.|
|[Самостоятельный сброс пароля](https://aka.ms/deploymentplans/sspr)|Самостоятельный сброс пароля дает пользователям возможность сбрасывать пароль без вмешательства администратора, где и когда им нужно.|
|[Управление привилегированными пользователями](https://aka.ms/deploymentplans/pim)|Azure AD Privileged Identity Management (PIM) позволяет управлять привилегированными административными ролями в Azure AD, ресурсах Azure и других службах Microsoft Online Services. PIM предоставляет такие решения, как JIT-доступ, рабочие процессы утверждения запросов и полностью интегрированные проверки доступа, что позволяет выявлять, обнаруживать и предотвращать вредоносные действия привилегированных ролей в режиме реального времени.|
|[Единый вход](https://aka.ms/deploymentplans/sso)|Единый вход позволяет вам получить доступ ко всем приложениям и ресурсам, которые необходимы для ведения бизнеса. При этом выполняется однократный вход с использованием одной учетной записи пользователя. После входа в систему вы можете перейти из Microsoft Office в SalesForce, а затем в Box без повторной проверки подлинности (например, ввода пароля).|
|[Простой единый вход](https://aka.ms/SeamlessSSODPDownload)|Простой единый вход Azure Active Directory автоматически обеспечивает пользователям вход в систему, когда они работают на корпоративных устройствах, подключенных к корпоративной сети. Если он включен, пользователям не нужно вводить пароль для входа в Azure AD, а в большинстве случаев — даже вводить имя пользователя. Эта функция предоставляет пользователям удобный доступ к облачным приложениям и не требует установки каких-либо дополнительных локальных компонентов.|
|[Панель доступа](https://aka.ms/AccessPanelDPDownload)|Предоставляет пользователям простой центр для обнаружения приложений и доступа к ним. Улучшите продуктивность сотрудников, предоставив им возможности самообслуживания, например запрашивание доступа к новым приложениям и группам или управление доступом к этим ресурсам от лица других пользователей.|
|[ADFS для синхронизации хэша паролей](https://aka.ms/deploymentplans/adfs2phs)|При синхронизации хэша паролей хэши паролей пользователей синхронизируются из локальной службы Active Directory в Azure AD, позволяя Azure AD выполнять проверку подлинности пользователей без взаимодействия с локальной службой Active Directory.|
|[ADFS для сквозной проверки подлинности](https://aka.ms/deploymentplans/adfs2pta)|Сквозная проверка подлинности Azure AD позволяет пользователям входить в локальные и облачные приложения, используя те же пароли. Эта функция облегчает работу пользователей, так как им не нужно запоминать на один пароль больше, и сокращает затраты на ИТ-поддержку, так как снижается вероятность того, что пользователи забудут процедуру входа. Если пользователи выполняют вход с помощью Azure AD, эта функция проверяет пароли пользователей непосредственно в локальной службе Active Directory.|
|[Azure Active Directory Application Proxy](https://aka.ms/deploymentplans/appproxy)|Сегодня сотрудникам нужно продуктивно работать без привязки к расположению, времени и оборудованию. Они предпочитают использовать собственные устройства — планшеты, телефоны и ноутбуки. Кроме того, они хотят иметь постоянный доступ к своим приложениям, как к приложениям SaaS в облаке, так и к приложениям в корпоративной локальной сети. Традиционно для доступа к локальным приложениям требуются виртуальные частные сети (VPN) или сети периметра. Помимо того, что такие решения довольно сложны и их непросто защитить, на их настройку и управление требуются немалые средства. Но есть более эффективный способ! Это Azure Active Directory Application Proxy.|
|[Подготовка пользователей](https://aka.ms/UserProvisioningDPDownload)|Служба Azure AD помогает автоматизировать процесс создания, обслуживания и удаления удостоверений пользователей в облачных приложениях (SaaS), таких как Dropbox, Salesforce, ServiceNow и других.|
|[Входящая подготовка пользователя на основе Workday](https://aka.ms/WorkdayDeploymentPlan)|Входящая подготовка пользователя на основе Workday для Active Directory образует основу для текущих процессов управления удостоверениями и улучшает бизнес-процессы, которые зависят от достоверных данных удостоверений. С помощью этой возможности вы можете легко управлять жизненным циклом удостоверений сотрудников и временных работников, настроив правила, которые сопоставляют процессы приема на работу, перехода в другой отдел и увольнения (Joiner-Mover-Leaver) с действиями по подготовке, которую выполняют сотрудники ИТ-отдела (создание, включение, отключение, удаление учетных записей).|
|[Отчеты и мониторинг](https://aka.ms/deploymentplans/reporting)| Разработка решения для создания отчетов и мониторинга Azure AD зависит от вашего законодательства, безопасности и эксплуатационных требований, а также от существующей среды и процессов. В этой статье представлены различные варианты проектирования и руководство по правильной стратегии развертывания.|
