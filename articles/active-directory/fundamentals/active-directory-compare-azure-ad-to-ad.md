# <a name="compare-active-directory-to-azure-active-directory"></a>Сравните Active Directory с Active Directory

Azure Active Directory — это следующая эволюция решений для управления идентификацией и доступом для облака. Корпорация Майкрософт представила службы домена Active Directory в Windows 2000, чтобы дать организациям возможность управлять несколькими компонентами и системами внутренней инфраструктуры, используя единую идентификацию на пользователя.

Azure AD переносит этот подход на новый уровень, предоставляя организациям решение Identity as a Service (IDaaS) для всех своих приложений в облаке и в помещениях.

Большинство ИТ-администраторов знакомы с концепциями доменных служб Active Directory. В следующей таблице описаны различия и сходства между концепциями Active Directory и Active Directory.

|Концепция|Active Directory|Azure Active Directory |
|:-|:-|:-|
|**Пользователи**|||
|Подготовка: пользователи | Организации создают внутренних пользователей вручную или используют внутреннюю или автоматизированную систему подготовки, такую как менеджер идентификации Майкрософт, для интеграции с системой управления персоналом.|Существующие организации Адъютина используют [Azure AD Connect](https://docs.microsoft.com/azure/active-directory/hybrid/how-to-connect-sync-whatis) для синхронизации идентификаторов в облаке.</br> Azure AD добавляет поддержку для автоматического создания пользователей из [облачных HR-систем.](https://docs.microsoft.com/azure/active-directory/saas-apps/workday-tutorial) </br>Azure AD может предоставлять идентификаторы в [приложениях SCIM, что позволяет](https://docs.microsoft.com/azure/active-directory/manage-apps/use-scim-to-provision-users-and-groups) Приложениям SaaS автоматически предоставлять приложениям необходимые сведения для доступа пользователей. |
|Подготовка: внешние идентификаторы| Организации создают внешних пользователей вручную в качестве обычных пользователей в выделенном внешнем лесу AD, что приводит к накладным расходам администрирования для управления жизненным циклом внешних идентификационных данных (гостевые пользователи)| Azure AD предоставляет особый класс идентификации для поддержки внешних идентификаторов. [Azure AD B2B](https://docs.microsoft.com/azure/active-directory/b2b/) будет управлять ссылкой на внешний иждиверей, чтобы убедиться, что они действительны. |
| Управление правами и группы| Администраторы делают пользователей членами групп. Владельцы приложений и ресурсов затем дают группам доступ к приложениям или ресурсам.| [Группы](https://docs.microsoft.com/azure/active-directory/fundamentals/active-directory-groups-create-azure-portal) также доступны в Azure AD, и администраторы также могут использовать группы для выдачи разрешений ресурсам. В Azure AD администраторы могут назначать членство группам вручную или использовать запрос для динамического включения пользователей в группу. </br> Администраторы могут использовать [управление правами](https://docs.microsoft.com/azure/active-directory/governance/entitlement-management-overview) в Azure AD, чтобы предоставить пользователям доступ к набору приложений и ресурсов, использующих рабочие процессы и, при необходимости, критерии, основанные на времени. |
| Админ менеджмент|Организации будут использовать сочетание доменов, организационных единиц и групп в АД для делегирования административных прав для управления каталогом и ресурсами, которые они контролируют.| Azure AD предоставляет [встроенные роли](https://docs.microsoft.com/azure/active-directory/fundamentals/active-directory-users-assign-role-azure-portal) с системой управления доступом на основе ролей (RBAC) с ограниченной поддержкой [создания пользовательских ролей](https://docs.microsoft.com/azure/active-directory/users-groups-roles/roles-custom-overview) для делегирования привилегированного доступа к системе идентификации, приложениям и ресурсам, которые она контролирует.</br>Управление ролями может быть улучшено с помощью [Привилегированного управления идентификацией (PIM),](https://docs.microsoft.com/azure/active-directory/privileged-identity-management/pim-configure) чтобы обеспечить доступ к привилегированным ролям, ограниченный по времени, ограниченный по времени или основанный на рабочем процессе. |
| Управление учетными данными| Учетные данные в Active Directory основаны на паролях, аутентификации сертификатов и аутентификации смарт-карт. Пароли управляются с помощью политики паролей, основанных на длине пароля, сроке действия и сложности.|Azure AD использует интеллектуальную [защиту паролей](https://docs.microsoft.com/azure/active-directory/authentication/concept-password-ban-bad) для облачных и предварительных. Защита включает в себя интеллектуальную блокировку плюс блокирование общих и пользовательских фраз паролей и замен. </br>Azure AD значительно повышает безопасность [благодаря многофакторной аутентификации](https://docs.microsoft.com/azure/active-directory/authentication/concept-mfa-howitworks) и [технологиям без паролей,](https://docs.microsoft.com/azure/active-directory/authentication/concept-authentication-passwordless) таким как FIDO2. </br>Azure AD снижает затраты на поддержку, предоставляя пользователям систему [снава паролей самообслуживания.](https://docs.microsoft.com/azure/active-directory/authentication/concept-sspr-howitworks) |
| **Приложения**|||
| Приложения для инфраструктуры|Active Directory составляет основу для многих компонентов инфраструктуры, например, DNS, DHCP, IPSec, WiFi, NPS и VPN-доступа|В новом облачном мире Azure AD является новой плоскостью управления для доступа к приложениям по сравнению с использованием сетевых элементов управления. Когда пользователи проверяют[подлинность, Условный доступ (CA)](https://docs.microsoft.com/azure/active-directory/conditional-access/overview), будет контролировать, какие пользователи, будет иметь доступ к каким приложениям при необходимых условиях.|
| Традиционные и устаревшие приложения| Большинство наземных приложений используют LDAP, интегрированную аутентификацию Windows (NTLM и Kerberos) или аутентификацию на основе заголовка для контроля доступа к пользователям.| Azure AD может предоставлять доступ к этим типам приложений с использованием прокси-агентов [приложения Azure AD,](https://docs.microsoft.com/azure/active-directory/manage-apps/application-proxy) работающих на месте. С помощью этого метода Azure AD позволяет аутентифицировать пользователей Active Directory на месте, используя Kerberos во время миграции или необходимости сосуществовать с устаревшими приложениями. |
| Приложения SaaS|Active Directory не поддерживает приложения SaaS нативной поддержке и требует системы федерации, такой как AD FS.|Приложения SaaS, поддерживающие OAuth2,\* SAML и WS- authentication, могут быть интегрированы в использование Azure AD для проверки подлинности. |
| Бизнес-приложения (LOB) с современной аутентификацией|Организации могут использовать AD FS с Active Directory для поддержки LOB-приложений, требующих современной аутентификации.| Lob-приложения, требующие современной аутентификации, могут быть настроены для использования Azure AD для проверки подлинности. |
| Услуги среднего уровня/Daemon|Службы, работающие в условиях, обычно используют учетные записи служб AD или групповые учетные записи службы (gMSA) для запуска. Затем эти приложения унаследуют разрешения учетной записи службы.| Azure AD предоставляет [управляемые идентификаторы](https://docs.microsoft.com/azure/active-directory/managed-identities-azure-resources/index) для выполнения других рабочих нагрузок в облаке. Жизненный цикл этих идентификаторов управляется Azure AD и привязан к поставщику ресурсов, который не может быть использован для других целей для получения бэкдорного доступа.|
| **Устройства**|||
| Мобильный|Active Directory не поддерживает мобильные устройства без сторонних решений.| Решение Microsoft по управлению мобильными устройствами, Microsoft Intune, интегрировано с Azure AD. Корпорация Майкрософт Intune предоставляет информацию о состоянии устройства системе идентификации для оценки при проверке подлинности. |
| Рабочие столы windows|Active Directory предоставляет возможность домена присоединиться к устройствам Windows для управления ими с помощью групповой политики, менеджер конфигурации System Center или других сторонних решений.|Устройства Windows можно [присоединить к Azure AD.](https://docs.microsoft.com/azure/active-directory/devices/) Условный доступ может проверить, является ли устройство Azure AD, к которого присоединился Azure AD в рамках процесса проверки подлинности. Устройствами Windows также можно управлять с [помощью Microsoft Intune.](https://docs.microsoft.com/intune/what-is-intune) В этом случае, условный доступ, будет рассматривать ли устройство жалобы (например, современные патчи безопасности и вирусных подписей), прежде чем разрешить доступ к приложениям.|
| Серверы Windows| Active Directory предоставляет надежные возможности управления для находящихся на территории Серверов Windows с использованием групповой политики или других управленческих решений.| Виртуальные компьютеры серверов Windows в Azure можно управлять с помощью [служб доменов Azure AD.](https://docs.microsoft.com/azure/active-directory-domain-services/) [Управляемые идентификаторы](https://docs.microsoft.com/azure/active-directory/managed-identities-azure-resources/index) могут использоваться, когда им. ВМ необходим доступ к каталогу или ресурсам системы идентификации.|
| Рабочие нагрузки Linux/Unix|Active Directory не поддерживает не-Windows без сторонних решений, хотя linux-машины могут быть настроены для проверки подлинности с Active Directory как область Kerberos.|Linux/Unix VMs могут использовать [управляемые идентификаторы](https://docs.microsoft.com/azure/active-directory/managed-identities-azure-resources/index) для доступа к системе идентификации или ресурсам. Некоторые организации перекладывают эти рабочие нагрузки на облачные контейнерные технологии, которые также могут использовать управляемые идентификаторы.|
