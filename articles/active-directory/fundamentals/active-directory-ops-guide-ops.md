# <a name="azure-active-directory-general-operations-guide-reference"></a>Ссылка на общие операции по общим операциям Azure Active

В этом разделе [справочного руководства по операциям Azure AD](active-directory-ops-guide-intro.md) описаны проверки и действия, которые следует предпринять для оптимизации общих операций Active Directory Azure (Azure AD).

> [!NOTE]
> Эти рекомендации актуальны на дату публикации, но могут меняться с течением времени. Организации должны постоянно оценивать свои оперативные практики по мере развития продуктов и служб Майкрософт с течением времени.

## <a name="key-operational-processes"></a>Ключевые операционные процессы

### <a name="assign-owners-to-key-tasks"></a>Назначать владельцам ключевые задачи

Управление Active Directory Azure требует непрерывного выполнения ключевых рабочих задач и процессов, которые могут не быть частью проекта развертывания. По-прежнему важно настроить эти задачи для оптимизации среды. Ключевыми задачами и рекомендуемыми владельцами являются:

| Задача | Владелец |
| :- | :- |
| Привод Улучшения на идентичность Безопасный результат | Оперативная группа InfoSec |
| Поддержание серверов подключения к рекламе Azure | Оперативная группа IAM |
| Регулярно выполнять и сортировать отчеты IdFix | Оперативная группа IAM |
| Triage Azure AD Connect Оповещения здоровья для синхронизации и AD FS | Оперативная группа IAM |
| Если не использовать Azure AD Connect Health, то у клиента есть эквивалентный процесс и инструменты для мониторинга пользовательской инфраструктуры | Оперативная группа IAM |
| Если не использовать AD FS, то клиент имеет эквивалентный процесс и инструменты для мониторинга пользовательской инфраструктуры | Оперативная группа IAM |
| Мониторинг гибридных журналов: Прокси-соединения приложения Azure AD | Оперативная группа IAM |
| Мониторинг гибридных журналов: Сквозная аутентификация агентов | Оперативная группа IAM |
| Мониторинг гибридных журналов: Служба списания паролей | Оперативная группа IAM |
| Мониторинг гибридных журналов: шлюз защиты паролей | Оперативная группа IAM |
| Мониторинг гибридных журналов: Расширение NPS Azure MFA (если это применимо) | Оперативная группа IAM |

При просмотре списка может потребоваться либо назначить владельца для задач, которые отсутствуют у владельца, либо настроить право собственности для задач с владельцами, которые не соответствуют рекомендациям выше.

### <a name="on-premises-agents-management"></a>Управление агентами на территории

Внедрение передового опыта может помочь оптимальной работе на месте агентов. Рассмотрим следующие рекомендации:

- Несколько прокси-разъемов Azure AD Application в одну группу разъемов рекомендуется обеспечивать бесшовную балансировку нагрузки и высокую доступность, избегая отдельных точек сбоя при доступе к прокси-приложениям. Если в настоящее время в группе разъема, который обрабатывает приложения в рабочей группе, есть только один разъем, следует развернуть по крайней мере два разъема для избыточности.
- Создание и использование группы прокси-разъемов приложений для отладки может быть полезно для устранения неполадок и при подключении новых приложений. Мы также рекомендуем устанавливать сетевые инструменты, такие как Message Analyzer и Fiddler в разъеме.
- Нескольким агентам по сквозной аутентификации рекомендуется обеспечивать бесшовную балансировку нагрузки и высокую доступность, избегая одной точки сбоя во время потока аутентификации. Не забудьте развернуть по крайней мере два сквозных агента аутентификации для избыточности.

## <a name="operational-surface-area"></a>Зона рабочей поверхности

### <a name="ad-fs-lockdown"></a>Блокировка AD FS

Организации, которые настраивают приложения для проверки подлинности непосредственно в Azure AD, получают преимущества от [интеллектуальной блокировки Azure AD.](https://docs.microsoft.com/azure/active-directory/active-directory-secure-passwords) Если вы используете AD FS в Windows Server 2012 R2, внедрите [защиту блокировки](https://docs.microsoft.com/windows-server/identity/ad-fs/operations/configure-ad-fs-extranet-soft-lockout-protection)AD FS. Если вы используете AD FS на Windows Server 2016 или позже, реализуем [смарт-блокировку экстрасети.](https://support.microsoft.com/help/4096478/extranet-smart-lockout-feature-in-windows-server-2016) Как минимум, мы рекомендуем включить блокировку экстрасети, чтобы сдержать риск атак грубой силы против находящихся в ней Active Directory. Однако, если у вас есть AD FS в Windows 2016 или выше, вы также должны включить extranet смарт блокировки, которые помогут смягчить атаки [спрей паролей.](https://www.microsoft.com/microsoft-365/blog/2018/03/05/azure-ad-and-adfs-best-practices-defending-against-password-spray-attacks/)

Если AD FS используется только для федерации Azure AD, есть некоторые конечные точки, которые могут быть выключены, чтобы свести к минимуму площадь поверхности атаки. Например, если AD FS используется только для Azure AD, следует отключить конечные точки WS-Trust, помимо конечных точек, включенных для **usernamemixed** и **windowstransport.**

### <a name="access-to-machines-with-on-premises-identity-components"></a>Доступ к машинам с компонентами предприимки

Организации должны блокировать доступ к машинам с помощью предприимчиваемых гибридных компонентов так же, как и ваш домен. Например, оператор резервного копирования или администратор Hyper-V не должны иметь возможность войти в AD-сервер Azure Connect Server для изменения правил.

Модель административного уровня Active Directory была разработана для защиты систем идентификации с использованием набора буферных зон между полным контролем над окружающей средой (Tier 0) и высокорисковыми рабочими станциями, которые злоумышленники часто компрометируют. ![Схема, показывающая три уровня модели уровней](./media/active-directory-ops-guide/active-directory-ops-img18.png)

[Модель уровня](https://docs.microsoft.com/windows-server/identity/securing-privileged-access/securing-privileged-access-reference-material) состоит из трех уровней и включает только административные учетные записи, а не стандартные учетные записи пользователей.

- **Уровень 0** - Прямой контроль корпоративных идентификаторов в среде. К уровню 0 принадлежат учетные записи, группы и другие ресурсы с прямым или непрямым административным контролем над лесами, доменами или контроллерами доменов Active Directory, а также над их ресурсами. Степень безопасности всех ресурсов уровня 0 одинакова, так как фактически все они находятся под контролем друг друга.
- **Уровень 1** - Контроль корпоративных серверов и приложений. К ресурсам уровня 1 принадлежат серверные операционные системы, облачные службы и корпоративные приложения. Учетные записи администраторов уровня 1 предоставляют права административного управления значительной частью бизнеса, в основе которого лежат эти ресурсы. Распространенный пример такой роли — администратор сервера. Он занимается обслуживанием операционных систем этих ресурсов и может повлиять на все службы предприятия.
- **Tier 2** - Управление рабочими станциями и устройствами пользователей. Учетные записи администраторов уровня 2 предоставляют права административного управления значительной частью бизнеса, а именно над рабочими станциями и устройствами, лежащими в его основе. К их примерам можно отнести администраторов службы технической поддержки или компьютеров, так как они могут повлиять на целостность данных практически любого пользователя.

Блокировка доступа к компонентам идентификации, таким как Azure AD Connect, AD FS и S'L, так же, как и для контроллеров доменов.
