# <a name="azure-active-directory-identity-and-access-management-operations-reference-guide"></a>Справочный справочник по операциям управления именем Azure Active Directory и управлению доступом

В этом разделе [справочного руководства по операциям Azure AD](active-directory-ops-guide-intro.md) описаны проверки и действия, которые следует учитывать для обеспечения и управления жизненным циклом идентификационных данных и их назначениями.

> [!NOTE]
> Эти рекомендации актуальны на дату публикации, но могут меняться с течением времени. Организации должны постоянно оценивать свою практику идентификации по мере развития продуктов и служб Майкрософт с течением времени.

## <a name="key-operational-processes"></a>Ключевые операционные процессы

### <a name="assign-owners-to-key-tasks"></a>Назначать владельцам ключевые задачи

Управление Active Directory Azure требует непрерывного выполнения ключевых рабочих задач и процессов, которые могут не быть частью проекта развертывания. По-прежнему важно настроить эти задачи для поддержания среды. Ключевыми задачами и рекомендуемыми владельцами являются:

| Задача | Владелец |
| :- | :- |
| Определите процесс создания подписок Azure | Вариации по организации |
| Решите, кто получает лицензии Enterprise Mobility и Security | Оперативная группа IAM |
| Решите, кто получит лицензии Office 365 | Команда производительности |
| Решите, кто получит другие лицензии, например, Dynamics, VSO | Application Owner; |
| Назначение лицензий | Оперативная группа IAM |
| Устранение неполадок и исправление ошибок в назначении лицензии | Оперативная группа IAM |
| Предоставление идентификаторов для приложений в Azure AD | Оперативная группа IAM |

При просмотре списка может потребоваться либо назначить владельца для задач, которые отсутствуют у владельца, либо настроить право собственности для задач с владельцами, которые не соответствуют рекомендациям выше.

## <a name="on-premises-identity-synchronization"></a>Синхронизация личных данных в помещениях

### <a name="identify-and-resolve-synchronization-issues"></a>Выявление и решение проблем синхронизации

Корпорация Майкрософт рекомендует вам иметь хороший базовый упор и понимание проблем в среде, что может привести к проблемам синхронизации в облаке. Поскольку автоматизированные инструменты, такие как [IdFix](https://docs.microsoft.com/office365/enterprise/prepare-directory-attributes-for-synch-with-idfix) и [Azure AD Connect Health,](https://docs.microsoft.com/azure/active-directory/hybrid/whatis-azure-ad-connect#why-use-azure-ad-connect-health) могут генерировать большой объем ложных срабатываний, мы рекомендуем вам определить ошибки синхронизации, которые были оставлены без внимания в течение более 100 дней, очищая эти объекты по ошибке. Долгосрочные нерешенные ошибки синхронизации могут генерировать инциденты поддержки. [Ошибки для устранения неполадок во время синхронизации](https://docs.microsoft.com/azure/active-directory/hybrid/tshoot-connect-sync-errors) предоставляют обзор различных типов ошибок синхронизации, некоторых возможных сценариев, которые вызывают эти ошибки, и возможных способов исправления ошибок.

### <a name="azure-ad-connect-sync-configuration"></a>Конфигурация синхронизации подключения к рекламе Azure

Для обеспечения всех гибридных возможностей, безопасности на основе устройств и интеграции с Azure AD необходимо синхронизировать учетные записи пользователей, которые сотрудники используют для входа в свои рабочие столы.

Если вы не синхронизируете лесопользователей войти в систему, то вы должны изменить синхронизацию, чтобы прийти из надлежащего леса.

#### <a name="synchronization-scope-and-object-filtering"></a>Область синхронизации и фильтрация объектов

Удаление известных ведер объектов, которые не требуют синхронизации, имеет следующие эксплуатационные преимущества:

- Меньше источников ошибок синхронизации
- Быстрые циклы синхронизации
- Меньше "мусора" для переноса из местных, например, загрязнения глобального списка адресов для учетных записей на местах, которые не имеют отношения к облаку

> [!NOTE]
> Если вы обнаружите, что импортируете много объектов, которые не экспортируются в облако, следует фильтровать OU или определенные атрибуты.

Примерами объектов, которые следует исключить, являются:

- Учетные записи служб, которые не используются для облачных приложений
- Группы, которые не предназначены для использования в сценариях облачных технологий, таких как те, которые используются для предоставления доступа к ресурсам
- Пользователи или контакты, которые являются внешними идентификаторами, которые предназначены для представления в сотрудничестве Azure AD B2B
- Компьютерные учетные записи, где сотрудники не предназначены для доступа к облачным приложениям, например, с серверов

> [!NOTE]
> Если одна человеческая личность имеет несколько учетных записей, подготовленных из чего-то, например, миграции домена, слияния или приобретения, следует синхронизировать учетную запись, используемую пользователем, на ежедневной основе, например, то, что они используют для входа в свой компьютер .

В идеале необходимо достичь баланса между уменьшением количества объектов для синхронизации и сложностью правил. Как правило, комбинация между [фильтрацией](https://docs.microsoft.com/azure/active-directory/hybrid/how-to-connect-sync-configure-filtering) OU/container плюс простое отображение атрибута с атрибутом cloudFiltered является эффективной комбинацией фильтрации.

> [!IMPORTANT]
> Если вы используете групповую фильтрацию в рабочей процессе, следует перейти к другому подходу фильтрации.

#### <a name="sync-failover-or-disaster-recovery"></a>Синхронизация сбоя или аварийное восстановление

Azure AD Connect играет ключевую роль в процессе подготовки. Если Sync Server по какой-либо причине переходит в автономный режим, изменения в помещениях не могут быть обновлены в облаке и могут привести к проблемам с доступом пользователей. Поэтому важно определить стратегию неудачи, которая позволяет администраторам быстро возобновить синхронизацию после того, как сервер синхронизации перейдет в автономный режим. Такие стратегии могут подпадать под следующие категории:

- **Развертывание Azure AD Connect Server (ы) в режиме постановки** - позволяет администратору "продвигать" промежуточный сервер к производству простым переключателем конфигурации.
- **Используйте виртуализацию** - Если подключение Azure AD развернуто в виртуальной машине (VM), админы могут использовать свой стек виртуализации для живой миграции или быстрой передислокации VM и, следовательно, возобновления синхронизации.

Если в организации отсутствует стратегия аварийного восстановления и сбоя в синхронизации, не стоит стесняться развертывать Azure AD Connect в режиме постановки. Аналогичным образом, при наличии несоответствия между конфигурацией производства и постановки следует перестроить базовый режим Azure AD Connect в соответствии с конфигурацией производства, включая версии программного обеспечения и конфигурации.

#### <a name="stay-current"></a>Последние новости

Корпорация Майкрософт регулярно обновляет Azure AD Connect. Оставайтесь в курсе, чтобы воспользоваться преимуществами улучшений производительности, исправлений ошибок и новых возможностей, которые предоставляет каждая новая версия.

Если версия Azure AD Connect отстает более чем на шесть месяцев, следует обновить ее до самой последней версии.

#### <a name="source-anchor"></a>Источник якоря

Использование **ms-DS-consistencyguid** в качестве [исходного якоря](https://docs.microsoft.com/azure/active-directory/hybrid/plan-connect-design-concepts) позволяет легче осуществлять миграцию объектов в лесах и областях, что является общим для консолидации/очистки доменов AD, слияний, поглощений и изъятий.

Если в настоящее время вы используете **ObjectGuid** в качестве исходного якоря, мы рекомендуем вам перейти к использованию **ms-DS-ConsistencyGuid.**

#### <a name="custom-rules"></a>Настраиваемые правила

Пользовательские правила Azure AD Connect обеспечивают возможность управления потоком атрибутов между объектами, находивными в непредпомяне, и облачными объектами. Однако чрезмерное использование или неправильное использование пользовательских правил может ввести следующие риски:

- Сложность устранения неполадок
- Ухудшение производительности при выполнении сложных операций между объектами
- Более высокая вероятность расхождения конфигурации между производственным сервером и трансайд-сервером
- Дополнительные накладные расходы при обновлении Azure AD Connect, если пользовательские правила создаются в пределах приоритета более 100 (используется встроенными правилами)

Если вы используете слишком сложные правила, следует изучить причины сложности и найти возможности для упрощения. Аналогичным образом, если вы создали пользовательские правила с значением приоритета более 100, вы должны исправить правила, чтобы они не подвержены риску или конфликт с набором по умолчанию.

Примеры неправильного использования пользовательских правил включают в себя:

- **Компенсировать грязные данные в каталоге** - В этом случае рекомендуется работать с владельцами команды AD и очищать данные в каталоге в качестве задачи восстановления, а также корректировать процессы, чтобы избежать повторного введения плохих данных.
- **Одноразовые исправления отдельных пользователей** - Это часто, чтобы найти правила, которые специальные выбросы случае, как правило, из-за проблемы с конкретным пользователем.
- **Чрезмерно сложное "CloudFiltering"** - Хотя сокращение числа объектов является хорошей практикой, существует риск создания и чрезмерносложной области синхронизации с использованием многих правил синхронизации. При наличии сложной логики включения/исключения объектов за пределы фильтрации OU рекомендуется иметь дело с этой логикой вне синхронизации и маркировать объекты простым атрибутом "cloudFiltered", который может передаваться с помощью простого правила синхронизации.

#### <a name="azure-ad-connect-configuration-documenter"></a>Документатор конфигурации Azure AD Connect

[Документатор конфигурации Azure AD Connect](https://github.com/Microsoft/AADConnectConfigDocumenter) — это инструмент, который можно использовать для создания документации установки Azure AD Connect для лучшего понимания конфигурации синхронизации, укрепления уверенности в том, что все изменилось, когда вы применили новую сборку или конфигурацию Azure AD Connect или добавили или обновили пользовательские правила синхронизации. Текущие возможности инструмента включают в себя:

- Документация полной конфигурации синхронизации Azure AD Connect.
- Документация любых изменений в конфигурации двух серверов синхронизации Azure AD Connect или изменений из заданной базовой настройки конфигурации.
- Генерация сценария развертывания PowerShell для переноса различий правил синхронизации или настройки с одного сервера на другой.

## <a name="assignment-to-apps-and-resources"></a>Назначение приложений и ресурсов

### <a name="group-based-licensing-for-microsoft-cloud-services"></a>Групповое лицензирование облачных служб Майкрософт

Активный каталог Azure упрощает управление лицензиями с [помощью группового лицензирования](https://docs.microsoft.com/azure/active-directory/fundamentals/active-directory-licensing-whatis-azure-portal) облачных служб Майкрософт. Таким образом, IAM обеспечивает групповую инфраструктуру и делегирует управление этими группами соответствующим группам в организациях. Существует несколько способов настройки членства групп в Azure AD, включая:

- **Синхронизированные с наемными -** Группы могут поступать из наемных каталогов, которые могут быть хорошо подходят для организаций, которые создали процессы управления группами, которые могут быть расширены, чтобы назначить лицензии в офисе 365.

- **Attribute-based/dynamic** - Группы могут создаваться в облаке на основе выражения, основанного на атрибутах пользователя, например, Отдел равен "продажам". Azure AD поддерживает членов группы, сохраняя ее в соответствии с определенным выражением. Использование такого рода групп ы для назначения лицензии позволяет выполнять назначение лицензий на основе атрибутов, что хорошо подходит для организаций, имеющих высокое качество данных в каталоге.

- **Делегированная собственность** - Группы могут быть созданы в облаке и могут быть назначены владельцами. Таким образом, вы можете предоставить владельцам бизнеса, например, группе совместной работы или группе BI, определить, кто должен иметь доступ.

Если в настоящее время вы используете ручной процесс для присвоения пользователям лицензий и компонентов, мы рекомендуем вам реализовать групповое лицензирование. Если текущий процесс не отслеживает ошибки лицензирования или то, что назначено по сравнению с доступным, следует определить улучшения процесса для устранения ошибок лицензирования и мониторинга назначения лицензирования.

Другим аспектом управления лицензией является определение планов обслуживания (компонентов лицензии), которые должны быть включены на основе рабочих функций в организации. Предоставление доступа к планам обслуживания, которые не являются необходимыми, может привести к тому, что пользователи увидят на портале Office инструменты, которые они не прошли или не должны использовать. Это может привести к дополнительному объему справочной службы, ненужной подготовке и поставить под угрозу соответствие требованиям и управление, например, при подготовке OneDrive для бизнеса для частных лиц, которым может быть запрещено делиться контентом.

Используйте следующие рекомендации для определения планов обслуживания для пользователей:

- Администраторам следует определить "пакеты" планов обслуживания, которые будут предложены пользователям на основе их роли, например, работника "белых воротничков" по сравнению с напольным работником.
- Создавайте группы по группам и присваивайте лицензию плану обслуживания.
- В качестве опционов может быть определен атрибут для хранения пакетов для пользователей.

> [!IMPORTANT]
> Групповое лицензирование в Azure AD вводит понятие пользователей в состоянии ошибки лицензирования. Если вы заметили какие-либо ошибки в лицензировании, то вы должны немедленно [определить и устранить](https://docs.microsoft.com/azure/active-directory/users-groups-roles/licensing-groups-resolve-problems) любые проблемы с назначением лицензии.

#### <a name="lifecycle-management"></a>Управление жизненным циклом

Если в настоящее время вы используете инструмент, например [Microsoft Identity Manager](https://docs.microsoft.com/microsoft-identity-manager/) или стороннюю систему, которая опирается на локта-инфраструктуру, мы рекомендуем вам разгрузить назначение из существующего инструмента, реализовать групповое лицензирование и определить управление жизненным циклом группы на основе [групп.](https://docs.microsoft.com/azure/active-directory/users-groups-roles/licensing-group-advanced#use-group-based-licensing-with-dynamic-groups) Аналогичным образом, если существующий процесс не учитывает новых сотрудников или сотрудников, которые покидают организацию, следует развернуть групповое лицензирование на основе динамических групп и определить жизненный цикл членства в группе. Наконец, если групповое лицензирование развертывается против групп, не использующих управление жизненным циклом, рассмотрите возможность использования облачных групп для включения таких возможностей, как делегированное владение или динамическое членство на основе атрибутов.

### <a name="assignment-of-apps-with-all-users-group"></a>Назначение приложений с группой "Все пользователи"

Владельцы ресурсов могут полагать, что группа **всех пользователей** содержит только **сотрудников предприятия,** когда они могут содержать как **корпоративных сотрудников,** так и **гостей.** В результате следует особо еснять сядевидность при использовании группы **Всех пользователей** для назначения приложений и предоставления доступа к ресурсам, таким как содержимое SharePoint или приложения.

> [!IMPORTANT]
> Если группа **всех пользователей** включена и используется для политик и условного доступа, приложения или назначения ресурсов, убедитесь, что [защитите группу,](https://docs.microsoft.com/azure/active-directory/b2b/use-dynamic-groups) если вы не хотите, чтобы она включала в нее гостевых пользователей. Кроме того, следует исправить назначения лицензирования, создавая и назначая группам, в которых содержатся только **корпоративные сотрудники.** С другой стороны, если вы обнаружите, что группа **всех пользователей** включена, но не используется для предоставления доступа к ресурсам, убедитесь, что оперативное руководство вашей организации заключается в намеренном использовании этой группы (которая включает в себя как **корпоративных сотрудников, так** и **гостей).**

### <a name="automated-user-provisioning-to-apps"></a>Автоматизированная подготовка пользователей к приложениям

[Автоматизированная подготовка пользователей](https://docs.microsoft.com/azure/active-directory/manage-apps/user-provisioning) к приложениям является наилучшим способом создания последовательной подготовки, декручения и жизненного цикла идентификационных данных в нескольких системах.

Если в настоящее время вы предоставляете приложения на специальной основе или используете такие вещи, как CSV-файлы, JIT или локальные решения, которые не касаются управления жизненным циклом, мы рекомендуем вам [реализовать подготовку приложений](https://docs.microsoft.com/azure/active-directory/manage-apps/user-provisioning#how-do-i-set-up-automatic-provisioning-to-an-application) с Azure AD для поддерживаемых приложений и определить согласованный шаблон для приложений, которые еще не поддерживаются Azure AD.

### <a name="azure-ad-connect-delta-sync-cycle-baseline"></a>Базовый цикл синхронизации дельты Azure AD Connect

Важно понимать объем изменений в организации и убедиться, что не занимает слишком много времени, чтобы иметь предсказуемое время синхронизации.

Частота [синхронизации дельты по умолчанию](https://docs.microsoft.com/azure/active-directory/hybrid/how-to-connect-sync-feature-scheduler) составляет 30 минут. Если синхронизация дельты занимает более 30 минут последовательно или существуют значительные расхождения между производительностью дельта-синхронизации постановки и производства, следует изучить и просмотреть [факторы, влияющие на производительность Azure AD Connect.](https://docs.microsoft.com/azure/active-directory/hybrid/plan-connect-performance-factors)
