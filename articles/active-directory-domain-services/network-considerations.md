# <a name="networking-considerations-for-azure-ad-domain-services"></a>Рекомендации по сетям для доменных служб Azure AD
## <a name="how-to-select-an-azure-virtual-network"></a>Выбор виртуальной сети Azure
При выборе виртуальной сети для использования с доменными службами Azure AD придерживайтесь следующих рекомендаций.

### <a name="type-of-azure-virtual-network"></a>Тип виртуальной сети Azure
* **Виртуальные сети Resource Manager**. Доменные службы Azure AD можно включить в виртуальных сетях, созданных с помощью Azure Resource Manager.
* Нельзя включить доменные службы Azure AD в классической виртуальной сети Azure.
* К виртуальной сети, в которой включены доменные службы Azure AD, можно подключить другие виртуальные сети. Дополнительные сведения см. в разделе [Сетевое подключение](network-considerations.md#network-connectivity).

### <a name="azure-region-for-the-virtual-network"></a>Регион Azure для виртуальной сети
* Управляемый домен доменных служб Azure AD развертывается в том же регионе Azure, что и виртуальная сеть, в которой необходимо включить службу.
* Выберите виртуальную сеть в регионе Azure, который поддерживается доменными службами Azure AD.
* На странице [служб Azure по регионам](https://azure.microsoft.com/regions/#services/) можно узнать, в каких регионах Azure доступны доменные службы Azure AD.

### <a name="requirements-for-the-virtual-network"></a>Требования к виртуальной сети
* **Близкое расположение к рабочим нагрузкам Azure**. Выберите виртуальную сеть, где размещаются или будут размещены виртуальные машины, которым требуется доступ к доменным службам Azure AD. Если рабочие нагрузки развертываются в виртуальной сети, отличной от виртуальной сети управляемого домена, вы также можете подключаться к виртуальным сетям.
* **Пользовательский или собственный DNS-сервер**. Убедитесь, что для виртуальной сети нет настроенных пользовательских DNS-серверов. Пример пользовательского DNS-сервера является экземпляром DNS Windows Server, запущенным на виртуальной машине Windows Server, развернутой в виртуальной сети. Доменные службы Azure AD не поддерживают интеграцию с пользовательскими DNS-серверами в виртуальной сети.
* **Существующие домены с одним доменным именем**. Убедитесь, что в этой виртуальной сети нет домена с таким же доменным именем. Например, предположим, что в выбранной виртуальной сети уже есть домен с именем contoso.com, Позже вы попытаетесь включить в ней управляемый домен доменных служб Azure AD с таким же доменным именем (contoso.com). В таком случае попытка включить доменные службы Azure AD приведет к ошибке. Причина заключается в конфликте доменных имен в этой виртуальной сети. В такой ситуации для настройки управляемого домена доменных служб Azure AD необходимо использовать другое имя. Кроме того, можно отменить подготовку существующего домена и включить доменные службы Azure AD.

> [!WARNING]
> После включения службы нельзя переместить доменные службы в другую виртуальную сеть.
>
>


## <a name="guidelines-for-choosing-a-subnet"></a>Рекомендации по выбору подсети

![Рекомендуемая конфигурация подсетей](./media/active-directory-domain-services-design-guide/vnet-subnet-design.png)

* Развертывайте доменные службы Azure AD в **отдельную выделенную подсеть** в виртуальной сети Azure.
* Не применяйте группы безопасности сети к выделенной подсети для управляемого домена. Если нужно применить группы безопасности сети к выделенной подсети, убедитесь, что **порты, необходимые для обслуживания вашего домена и управления им, не заблокированы**.
* Не стоит чрезмерно ограничивать число IP-адресов, доступных в рамках выделенной подсети для вашего управляемого домена. Из-за этого ограничения служба не сможет предоставить управляемому домену доступ к двум контроллерам домена.
* **Не включайте доменные службы Azure AD в подсети шлюза** виртуальной сети.

> [!WARNING]
> При связывании NSG с подсетью, в которой включены доменные службы Azure AD, можно нарушить обслуживание домена и управление им в корпорации Майкрософт. Кроме того, произойдет сбой синхронизации между клиентом Azure AD и управляемым доменом. **Соглашение об уровне обслуживания не применяется к развертываниям, в которых применена группа безопасности сети, блокирующая обновление доменных служб Azure AD и не позволяющая им управлять вашим доменом.**
>
>

## <a name="ports-required-for-azure-ad-domain-services"></a>Порты, необходимые для доменных служб Azure AD
Следующие порты требуются доменным службам Azure AD для обслуживания и поддержки управляемого домена. Убедитесь, что эти порты не заблокированы для подсети, в которой включен управляемый домен.

| Номер порта | Обязательный? | Назначение |
| --- | --- | --- |
| 443 | Обязательно |Синхронизация с клиентом Azure AD |
| 5986 | Обязательно | Управление доменом |
| 3389 | Обязательно | Управление доменом |
| 636 | Необязательно | Безопасный доступ LDAP (LDAPS) к управляемому домену |

**Порт 443 (синхронизация с Azure AD)**
* Используется для синхронизации каталога Azure AD с управляемым доменом.
* Разрешить доступ к этому порту в NSG нужно обязательно. Без доступа к этому порту управляемый домен не будет синхронизироваться с каталогом Azure AD. Возможно, пользователи не смогут войти в систему, так как изменения их паролей не будут синхронизированы с управляемым доменом.
* Доступ входящего трафика к этому порту можно ограничить IP-адресами, относящимися к диапазону IP-адресов Azure. Обратите внимание на то, что диапазон адресов Azure отличается от диапазона PowerShell, показанного в правиле ниже.

**Порт 5986 (удаленное взаимодействие PowerShell)**
* Используется для выполнения задач управления с помощью удаленного взаимодействия PowerShell, выполняемого для управляемого домена.
* Разрешить доступ через этот порт в NSG нужно обязательно. Без доступа к этому порту невозможно обновить, настроить или отслеживать управляемый домен, а также создать его резервную копию.
* Для всех новых доменов и доменов с виртуальной сетью Azure Resource Manager доступ для входящего трафика к этому порту можно ограничить следующими исходными IP-адресами: 52.180.179.108, 52.180.177.87, 13.75.105.168, 52.175.18.134, 52.138.68.41, 52.138.65.157, 104.41.159.212, 104.45.138.161, 52.169.125.119, 52.169.218.0, 52.187.19.1, 52.187.120.237, 13.78.172.246, 52.161.110.169, 52.174.189.149, 40.68.160.142, 40.83.144.56, 13.64.151.161, 52.180.183.67, 52.180.181.39, 52.175.28.111, 52.175.16.141, 52.138.70.93, 52.138.64.115, 40.80.146.22, 40.121.211.60, 52.138.143.173, 52.169.87.10, 13.76.171.84, 52.187.169.156, 13.78.174.255, 13.78.191.178, 40.68.163.143, 23.100.14.28, 13.64.188.43, 23.99.93.197.
* Для доменов с классической виртуальной сетью доступ для входящего трафика к этому порту можно ограничить следующими исходными IP-адресами: 52.180.183.8, 23.101.0.70, 52.225.184.198, 52.179.126.223, 13.74.249.156, 52.187.117.83, 52.161.13.95, 104.40.156.18, 104.40.87.209.
* Контроллеры домена для управляемого домена обычно не прослушивают этот порт. Служба открывает этот порт на контроллерах управляемого домена, только если для управляемого домена необходимо выполнить операции управления или обслуживания. Сразу после завершения операции служба закрывает этот порт на контроллерах управляемого домена.

**Порт 3389 (удаленный рабочий стол)**
* Используется для подключений удаленного рабочего стола к контроллерам управляемого домена.
* Доступ для входящего трафика можно ограничить следующими входящими IP-адресами: 207.68.190.32/27, 13.106.78.32/27, 13.106.174.32/27, 13.106.4.96/27.
* Этот порт остается отключенным для управляемого домена. Этот механизм не используется регулярно, так как задачи управления и мониторинга выполняются с помощью удаленного взаимодействия PowerShell. Этот порт используется только в тех редких случаях, когда корпорации Майкрософт требуется удаленное подключение к управляемому домену для расширенного устранения неполадок. Порт будет закрыт сразу после устранения неполадок.

**Порт 636 (защищенный протокол LDAP)**
* Используется для включения доступа по защищенному протоколу LDAP к управляемому домену через Интернет.
* Открывать этот порт через NSG не обязательно. Этот порт следует открыть только в том случае, когда включен защищенный доступ по протоколу LDAP через Интернет.
* Доступ входящего трафика к этому порту можно ограничить IP-адресами источников, от которых ожидается подключение по защищенному протоколу LDAP.


## <a name="network-security-groups"></a>группы сетевой безопасности;
[Группа безопасности сети (NSG)](../virtual-network/virtual-networks-nsg.md) содержит правила из списка управления доступом, которые разрешают или запрещают передачу сетевого трафика на экземпляры виртуальных машин в виртуальной сети. Группы безопасности сети можно связать с подсетями или отдельными экземплярами виртуальных машин в одной из подсетей. Когда NSG связана с подсетью, правила списка управления доступом применяются ко всем экземплярам виртуальной машины в этой подсети. Кроме того, трафик на отдельные виртуальные машины можно дополнительно ограничить, связав NSG непосредственно с нужной виртуальной машиной.

### <a name="sample-nsg-for-virtual-networks-with-azure-ad-domain-services"></a>Пример NSG для виртуальных сетей с доменными службами Azure AD
В следующей таблице показан пример группы безопасности сети (NSG), которую можно настроить для виртуальной сети с управляемым доменом доменных служб Azure AD. Это правило разрешает входящий трафик из требуемых портов. Это необходимо для установки исправлений и обновлений для управляемого домена, а также для его отслеживания корпорацией Майкрософт. Правило "DenyAll" по умолчанию применяется ко всему входящему трафику из Интернета.

Кроме того, данная NSG также показывает, как можно заблокировать защищенный доступ LDAP через Интернет. Пропустите это правило, если вы не включили защищенный доступ LDAP к управляемому домену через Интернет. Группа безопасности сети содержит набор правил, которые разрешают входящий доступ по протоколу LDAPS через TCP-порт 636 только для указанного набора IP-адресов. Правило группы безопасности сети, которое разрешает доступ по протоколу LDAPS через Интернет только с указанных IP-адресов, имеет более высокий приоритет по сравнению с правилом группы безопасности сети "DenyAll".

![Пример группы безопасности сети для доступа по защищенному протоколу LDAPS через Интернет](./media/active-directory-domain-services-alerts/default-nsg.png)

## <a name="network-connectivity"></a>Сетевое подключение
Управляемый домен доменных служб Azure Active Directory можно включить только в пределах одной виртуальной сети в Azure.

### <a name="scenarios-for-connecting-azure-networks"></a>Сценарии для подключения сетей Azure
Виртуальные сети Azure можно подключать для использования управляемого домена в любом из следующих сценариев развертывания.

#### <a name="use-the-managed-domain-in-more-than-one-azure-virtual-network"></a>Использование управляемого домена в нескольких виртуальных сетях Azure
Вы можете подключить другие виртуальные сети Azure к виртуальной сети Azure, в которой включены доменные службы Azure Active Directory. Благодаря подключениям VPN и пирингу виртуальных сетей управляемый домен можно использовать с рабочими нагрузками, развернутыми в других виртуальных сетях.

![Подключение к классической виртуальной сети](./media/active-directory-domain-services-design-guide/classic-vnet-connectivity.png)

#### <a name="use-the-managed-domain-in-a-resource-manager-based-virtual-network"></a>Использование управляемого домена в виртуальной сети на основе Resource Manager
Вы можете подключить виртуальную сеть на основе Resource Manager к классической виртуальной сети Azure, в которой включены доменные службы Azure AD. Благодаря этому управляемый домен можно использовать с рабочими нагрузками, развернутыми в виртуальных сетях на основе Resource Manager.

![Подключение Resource Manager к классической виртуальной сети](./media/active-directory-domain-services-design-guide/classic-arm-vnet-connectivity.png)

### <a name="network-connection-options"></a>Параметры сетевого подключения
* **Подключение между виртуальными сетями с использованием пиринга**. Пиринговая связь между виртуальными сетями — это механизм подключения между двумя виртуальными сетями, находящимися в одном регионе, через магистральную сеть Azure. После создания пиринговой связи две виртуальные сети выглядят как одна при любом подключении. Они по-прежнему управляются как отдельные ресурсы, но теперь находящиеся в них виртуальные машины могут взаимодействовать друг с другом напрямую по частным IP-адресам.

    ![Подключение к виртуальной сети с помощью пиринга](./media/active-directory-domain-services-design-guide/vnet-peering.png)

    [Дополнительные сведения. Пиринговая связь между виртуальными сетями](../virtual-network/virtual-network-peering-overview.md)

* **Подключение между виртуальными сетями с использованием VPN-подключения типа "сеть — сеть"** . Подключение одной виртуальной сети к другой похоже на подключение виртуальной сети к локальному сайту. В обоих типах подключений используется VPN-шлюз для создания защищенного туннеля, использующего IPsec/IKE.

    ![Подключение к виртуальной сети с помощью VPN-шлюза](./media/active-directory-domain-services-design-guide/vnet-connection-vpn-gateway.jpg)

    [Дополнительные сведения. Подключение виртуальных сетей с помощью VPN-шлюза](../vpn-gateway/virtual-networks-configure-vnet-to-vnet-connection.md)
