# <a name="virtual-network-design-considerations-and-configuration-options-for-azure-ad-domain-services"></a>Соображения виртуального проектирования сети и параметры конфигурации для служб доменов Azure AD

Поскольку службы функционального домена Azure Active Directory Domain Services (AD DS) предоставляют услуги аутентификации и управления другимприложениям и рабочим нагрузкам, подключение к сети является ключевым компонентом. Без правильно настроенных виртуальных сетевых ресурсов приложения и рабочие нагрузки не могут общаться с функциями Azure AD DS и использовать их. Планируйте свои требования к виртуальной сети, чтобы убедиться, что Azure AD DS может обслуживать ваши приложения и рабочие нагрузки по мере необходимости.

В этой статье излагаются соображения проектирования и требования к виртуальной сети Azure для поддержки Azure AD DS.

## <a name="azure-virtual-network-design"></a>Виртуальный сетевой дизайн Azure

Для обеспечения подключения к сети и обеспечения аутентификации приложений и служб в отношении DS Azure AD используется виртуальная сеть и подсеть Azure. В идеале Azure AD DS должна быть развернута в собственной виртуальной сети. Можно включить отдельную подсеть приложений в ту же виртуальную сеть для размещения ваших рабочих нагрузок управления VM или легких приложений. Отдельная виртуальная сеть для больших или сложных рабочих нагрузок приложений, внедряемый в виртуальную сеть Azure AD DS, обычно является наиболее подходящей конструкцией. Другие варианты проектирований являются действительными, при условии, что вы отвечаете требованиям, изложенным в следующих разделах для виртуальной сети и подсети.

При проектировании виртуальной сети для Azure AD DS применяются следующие соображения:

* DS Azure AD должен быть развернут в том же регионе Azure, что и ваша виртуальная сеть.
    * В это время можно развернуть только один разыскинадля dS Azure AD в одном арендаторе Azure AD. Управляемый домен Azure AD DS развертывается в одном регионе. Убедитесь, что вы создаете или выбираете виртуальную сеть в [регионе, поддерживающем Azure AD DS.](https://azure.microsoft.com/global-infrastructure/services/?products=active-directory-ds&regions=all)
* Рассмотрим близость других регионов Azure и виртуальных сетей, в которые размещаются рабочие нагрузки приложения.
    * Чтобы свести к минимуму задержку, держите основные приложения близко к или в том же регионе, что и виртуальная сетевая подсеть для домена Azure AD DS. Вы можете использовать виртуальные сети пиринговых или виртуальных частных сетей (VPN) соединения между виртуальными сетями Azure. Эти варианты подключения обсуждаются в следующем разделе.
* Виртуальная сеть не может полагаться на DNS-сервисы, кроме тех, которые предоставляются Azure AD DS.
    * Azure AD DS предоставляет собственную службу DNS. Виртуальная сеть должна быть настроена для использования этих адресов службы DNS. Разрешение имен для дополнительных пространстваимен может быть выполнено с помощью условных форвардов.
    * Вы не можете использовать пользовательские настройки DNS сервера для направления запросов с других DNS-серверов, в том числе на VMs. Ресурсы виртуальной сети должны использовать службу DNS, предоставляемую Azure AD DS.

> [!IMPORTANT]
> Вы не можете переместить Azure AD DS в другую виртуальную сеть после включения службы.

Управляемый домен Azure AD DS подключается к подсети в виртуальной сети Azure. Дизайн этой подсети для Azure AD DS со следующими соображениями:

* Azure AD DS должен быть развернут в собственной подсети. Не используйте существующую подсеть или подсеть шлюза.
* Группа сетевой безопасности создается во время развертывания домена Azure AD DS. Эта группа сетевой безопасности содержит необходимые правила для правильного сообщения службы.
    * Не создавайте и не используйте существующую группу сетевой безопасности с вашими собственными правилами.
* Azure AD DS требует 3-5 IP-адресов. Убедитесь, что диапазон IP-адресов подсети может предоставить такое количество адресов.
    * Ограничение доступных IP-адресов может помешать службам домена Azure AD поддерживать два контроллера домена.

В следующем примере диаграмма описывает действительную конструкцию, в которой Azure AD DS имеет свою собственную подсеть, есть подсеть шлюза для внешнего подключения, а рабочие нагрузки приложения находятся в подключенной подсети в виртуальной сети:

## <a name="connections-to-the-azure-ad-ds-virtual-network"></a>Подключение к виртуальной сети Azure AD DS

Как отмечалось в предыдущем разделе, можно создать домен Отдела АДУ только в одной виртуальной сети в Azure, и только один управляемый домен может быть создан в арендаторе Azure AD. На основе этой архитектуры может потребоваться подключение одной или нескольких виртуальных сетей, в которые размещаются рабочие нагрузки приложения, к виртуальной сети Azure AD DS.

Вы можете подключить рабочие нагрузки приложений, размещенные в других виртуальных сетях Azure, с помощью одного из следующих методов:

* Пиринг между виртуальными сетями
* Виртуальная частная сеть (VPN)

### <a name="virtual-network-peering"></a>Пиринг между виртуальными сетями

Пиринговая связь между виртуальными сетями — это механизм подключения между двумя виртуальными сетями, находящимися в одном регионе, через магистральную сеть Azure. Глобальная виртуальная сеть пиринга может подключить виртуальную сеть в регионах Azure. После того, как заглянул, две виртуальные сети позволяют ресурсам, таким как виртуальные, общаться друг с другом непосредственно с помощью частных IP-адресов. Использование виртуальной сети пиринга позволяет развертывать доступный dS-канал Azure AD с рабочими нагрузками приложений, развернутыми в других виртуальных сетях.

![Подключение к виртуальной сети с помощью пиринга](./media/active-directory-domain-services-design-guide/vnet-peering.png)

Для получения дополнительной информации смотрите [обзор внизимового обзора виртуальной сети Azure.](../virtual-network/virtual-network-peering-overview.md)

### <a name="virtual-private-networking-vpn"></a>Виртуальная частная сеть (VPN)

Вы можете подключить виртуальную сеть к другой виртуальной сети (VNet-to-VNet) таким же образом, чтобы настроить виртуальную сеть на локальный сайт. Оба соединения используют VPN шлюз для создания безопасного туннеля с помощью IPsec/IKE. Эта модель соединения позволяет развертывать Azure AD DS в виртуальную сеть Azure, а затем подключать места расположения помещений или другие облака.

![Виртуальное подключение к сети с помощью VPN-шлюза](./media/active-directory-domain-services-design-guide/vnet-connection-vpn-gateway.jpg)

Для получения дополнительной информации об использовании виртуальных частных сетей, прочитайте [Накнаждись на vpn-соединение VNet с помощью портала Azure.](../vpn-gateway/vpn-gateway-howto-vnet-vnet-resource-manager-portal.md)

## <a name="name-resolution-when-connecting-virtual-networks"></a>Разрешение имен при подключении виртуальных сетей

Виртуальные сети, подключенные к виртуальной сети Azure AD Domain Services, как правило, имеют свои собственные настройки DNS. При подключении виртуальных сетей не настраивается автоматически разрешение имен для подключения виртуальной сети для решения служб, предоставляемых доменом Azure AD DS. Разрешение имен на подключенных виртуальных сетях должно быть настроено, чтобы рабочие нагрузки приложений могли находить службы домена Azure AD.

Вы можете включить разрешение имен с помощью условных дислиний DNS на сервере DNS, поддерживающих подключение виртуальных сетей, или с помощью тех же IP-адресов DNS из виртуальной сети Azure AD Domain Service.

## <a name="network-resources-used-by-azure-ad-ds"></a>Сетевые ресурсы, используемые DS Azure AD

Управляемый домен Azure AD DS создает некоторые сетевые ресурсы во время развертывания. Эти ресурсы необходимы для успешной работы и управления доменом Azure AD DS и не должны быть настроены вручную.

| Ресурс Azure                          | Описание |
|:----------------------------------------|:---|
| Карта сетевого интерфейса                  | Azure AD DS размещает управляемый домен на двух контроллерах доменов (DCs), которые работают на Windows Server как Azure VMs. Каждый VM имеет виртуальный сетевой интерфейс, который подключается к вашей виртуальной сетевой подсети. |
| Динамический стандартный публичный IP-адрес      | Azure AD DS общается с службой синхронизации и управления с помощью стандартного публичного IP-адреса SKU. Для получения дополнительной информации об общедоступных IP-адресах смотрите [типы IP-адресов и методы распределения в Azure.](../virtual-network/virtual-network-ip-addresses-overview-arm.md) |
| Стандартный балансилер нагрузки Azure            | Azure AD DS использует стандартный балансера нагрузки SKU для перевода сетевых адресов (NAT) и балансировки нагрузки (при использовании с защищенным LDAP). Для получения дополнительной информации о [What is Azure Load Balancer?](../load-balancer/load-balancer-overview.md) балансисаторах нагрузки Azure см. |
| Правила перевода адресов сети (NAT) | Azure AD DS создает и использует три правила NAT на балансе груза - одно правило для безопасного трафика HTTP и два правила для безопасного remoting PowerShell. |
| Правила балансировщика нагрузки                     | Когда управляемый домен Azure AD DS настроен для безопасного LDAP в порту TCP 636, создаются и используются на балансере нагрузки для распределения трафика. |

> [!WARNING]
> Не удаляйте и не изменяйте ни один из сетевого ресурса, созданного DD Azure AD, например, вручную настраивая баланселист или правила нагрузки. При удалении или изменении каких-либо сетевых ресурсов может произойти сбой службы Azure AD DS.

## <a name="network-security-groups-and-required-ports"></a>Группы сетевой безопасности и требуемые порты

[Группа сетевой безопасности (NSG)](../virtual-network/virtual-networks-nsg.md) содержит список правил, которые позволяют или отказывают сетевому трафику в трафике в виртуальной сети Azure. Группа сетевой безопасности создается при развертывании DS Azure AD, который содержит набор правил, которые позволяют службе предоставлять функции аутентификации и управления. Эта группа сетевой безопасности по умолчанию связана с виртуальной сетевой подсетью, в которой развернут исктые dS Azure AD.

Для обеспечения аутентификации и услуг по управлению службами безопасности Azure AD DS требуются следующие правила группы сетевой безопасности. Не изменяйте и не удаляйте эти правила группы сетевой безопасности для виртуальной сетевой подсети, в которой развернут амеленный домен Azure AD DS.

| Номер порта | Протокол | Источник                             | Назначение | Действие | Обязательно | Назначение |
|:-----------:|:--------:|:----------------------------------:|:-----------:|:------:|:--------:|:--------|
| 443         | TCP      | AzureActiveDirectoryDomainServices | Любой         | Allow  | Да      | Синхронизация с вашим арендатором Azure AD. |
| 3389        | TCP      | CorpNetSaw                         | Любой         | Allow  | Да      | Управление вашим доменом. |
| 5986        | TCP      | AzureActiveDirectoryDomainServices | Любой         | Allow  | Да      | Управление вашим доменом. |

> [!WARNING]
> Не отодевайте вручную эти сетевые ресурсы и конфигурации. При связывании неправильно настроенной группы сетевой безопасности или определенной пользователем таблицы маршрутов с подсетью, в которой развернутА DS Azure AD, вы можете нарушить возможность Microsoft обслуживать и управлять доменом. Синхронизация между вашим арендатором Azure AD и доменом Azure AD DS также нарушена.
>
> Если вы используете безопасный LDAP, вы можете добавить требуемое правило TCP port 636, чтобы при необходимости разрешить внешний трафик. Добавление этого правила не размещает правила группы сетевой безопасности в неподдерживаемом состоянии. Для получения дополнительной [информации, см. Блокировка вниз безопасный доступ LDAP через Интернет](tutorial-configure-ldaps.md#lock-down-secure-ldap-access-over-the-internet)
>
> По умолчанию правила для *AllowVnetInBound*, *AllowAzureLoadBalancerInBound*, *DenyAllInBound*, *AllowVnetOutBound*, *AllowInternetOutBound*, и *DenyAllOutBound* также существуют для группы сетевой безопасности. Не издавайте и не удаляйте эти правила по умолчанию.
>
> La La Azure не применяется к развертываниям, где неправильно настроенная группа безопасности сети и/или таблицы маршрутов с определенным пользователем были применены, что блокирует обновление DS Azure AD и управление доменом.

### <a name="port-443---synchronization-with-azure-ad"></a>Порт 443 - синхронизация с Azure AD

* Используется для синхронизации арендатора Azure AD с доменом Azure AD DS.
* Без доступа к этому порту управляемый dS Azure AD не может синхронизироваться с вашим арендатором Azure AD. Пользователи могут не иметь возможности войти в систему, так как изменения в их паролях не будут синхронизированы с вашим доменом Azure AD DS.
* Входящий доступ к этому порту на IP-адреса ограничен по умолчанию с помощью тега службы **AzureActiveDirectoryDomainServices.**
* Не ограничивайте исходящий доступ из этого порта.

### <a name="port-3389---management-using-remote-desktop"></a>Порт 3389 - управление с помощью удаленного рабочего стола

* Используется для удаленных подключений к настольным компьютерам для контроллеров доменов в домене Azure AD DS.
* Правило группы безопасности сети по умолчанию использует тег службы *CorpNetSaw* для дальнейшего ограничения трафика.
    * Этот тег службы позволяет использовать удаленный рабочий стол только для доступа к домену Azure AD DS только для обеспечения безопасности рабочих станций доступа.
    * Доступ разрешен только с обоснованием бизнеса, например, для управления или устранения неполадок.
* Это правило может быть установлено для *отрицания,* и только установить, чтобы *разрешить,* когда это необходимо. Большинство задач управления и мониторинга выполняются с помощью ремотинга PowerShell. RDP используется только в тех редких случаях, когда Microsoft необходимо удаленно подключиться к управляемому домену для продвинутого устранения неполадок.

> [!NOTE]
> Вы не можете вручную выбрать тег службы *CorpNetSaw* из портала, если вы попытаетесь отсеить это правило группы безопасности сети. Для настройки правила, использующему тег службы *CorpNetSaw,* необходимо использовать Azure PowerShell или Azure CLI.

### <a name="port-5986---management-using-powershell-remoting"></a>Порт 5986 - управление с помощью PowerShell remoting

* Используется для выполнения задач управления с помощью ремемки PowerShell в домене Azure AD DS.
* Без доступа к этому порту правительственный домен Azure AD DS не может быть обновлен, настроен, подкреплен или проверен.
* Для доменов, управляемых Azure AD DS, которые используют виртуальную сеть на основе менеджера ресурсов, можно ограничить входящему доступ к этому порту сервисным тегом *AzureActiveDirectoryDomainServices.*
    * Для устаревших доменов Azure AD DS, управляемых с помощью виртуальной сети на основе классической, Вы можете ограничить входявший доступ к этому порту следующим IP-адресом источника: *52.180.183.8*, *23.101.0.70*, *52.225.184.198*, *52.179.126.223*, *13.74.249.156*, *52.187.117.83,* *52.161.13.95*, *104.40.156.18*, и *104.40.87.209*.

    > [!NOTE]
    > В 2017 году службы домена Azure AD стали доступны для размещения в сети управления ресурсами Azure. С тех пор мы смогли создать более безопасную службу, используя современные возможности менеджера ресурсов Azure. Поскольку развертывание ресурсов Azure полностью заменит классические развертывания, классические виртуальные сетевые развертывания Azure AD DS будут отменены 1 марта 2023 года.
    >
    > Для получения дополнительной информации [см.](https://azure.microsoft.com/updates/we-are-retiring-azure-ad-domain-services-classic-vnet-support-on-march-1-2023/)

## <a name="user-defined-routes"></a>Определяемые пользователем маршруты

Маршруты, определяемые пользователями, не создаются по умолчанию и не нужны для правильной работы DS Azure AD. Если требуется использовать таблицы маршрутов, избегайте внесения каких-либо изменений в маршрут *0.0.0.0.* Изменения в этом маршруте нарушают работу служб доменов Azure AD и переносят управляемый домен в неподдерживаемое состояние.

Необходимо также направлять входящий трафик с IP-адресов, включенных в соответствующие теги службы Azure, в подсеть Доменных служб Azure AD. Для получения дополнительной информации о тегах услуг [Azure IP Ranges and Service Tags - Public Cloud](https://www.microsoft.com/en-us/download/details.aspx?id=56519)и связанных с ними IP-адресах см.

> [!CAUTION]
> Эти диапазоны IP-адресов Центра обработки данных Azure могут изменяться без предварительного уведомления. Убедитесь, что у вас есть процессы для проверки последних IP-адресов.
