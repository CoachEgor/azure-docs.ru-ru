# <a name="management-concepts-for-user-accounts-passwords-and-administration-in-azure-active-directory-domain-services"></a>Концепции управления учетными записями пользователей, паролями и администрированием в службах домена Active Directory

При создании и запуске домена Active Directory Domain Services (AD DS) и запускаются, по сравнению с традиционной предварительной средой AD DS существуют некоторые различия в поведении. В Azure AD DS вы используете те же административные инструменты в качестве автономного домена, но вы не можете напрямую получить доступ к контроллерам доменов (DC). Существуют также некоторые различия в поведении политик и хэшов паролей в зависимости от источника создания учетной записи пользователя.

В этой концептуальной статье подробно описано, как управлять доменом Azure AD DS и о различном поведении учетных записей пользователей в зависимости от способа их создания.

## <a name="domain-management"></a>управление доменами;

В Azure AD DS контроллеры домена (DCs), содержащие все ресурсы, такие как пользователи и группы, учетные данные и политики, являются частью управляемой службы. Для избыточности два DCs создаются как часть домена Azure AD DS. Вы не можете войти в эти DCs для выполнения задач управления. Вместо этого вы создаете VM управления, который присоединился к домену Azure AD DS, а затем устанавливаете обычные инструменты управления AD DS. Вы можете использовать, например, привязки с функциями Active Directory Administrative Center или Microsoft Management Console (MMC), например, dNS или объекты групповой политики.

## <a name="user-account-creation"></a>Создание учетной записи пользователя

Учетные записи пользователей могут быть созданы в Azure AD DS несколькими способами. Большинство учетных записей пользователей синхронизированы с Azure AD, что также может включать учетную запись пользователя, синхронизированную с предварительной средой AD DS. Вы также можете вручную создавать учетные записи непосредственно в Azure AD DS. Некоторые функции, такие как первоначальная синхронизация пароля или политика пароля, ведут себя по-разному в зависимости от того, как и где создаются учетные записи пользователей.

* Учетная запись пользователя может быть синхронизирована с Azure AD. Это включает в себя учетные записи пользователей только для облаков, созданные непосредственно в Azure AD, и гибридные учетные записи пользователей, синхронизированные с предварительной средой DD DS с помощью Azure AD Connect.
    * Большинство учетных записей пользователей в Azure AD DS создаются в процессе синхронизации с Azure AD.
* Учетная запись пользователя может быть создана вручную в домене Azure AD DS, управляемом DS, и не существует в Azure AD.
    * Если необходимо создать учетные записи служб для приложений, которые работают только в Azure AD DS, их можно вручную создать в управляемом домене. Поскольку синхронизация — это один из способов от Azure AD, учетные записи пользователей, созданные в DD Azure AD, не синхронизированы с Azure AD.

## <a name="password-policy"></a>Политика паролей

DS Azure AD включает в себя политику паролей по умолчанию, которая определяет настройки для таких вещей, как блокировка учетной записи, максимальный возраст паролей и сложность паролей. Такие настройки, как политика блокировки учетных записей, применяются ко всем пользователям в Azure AD DS, независимо от того, как пользователь был создан, как указано в предыдущем разделе. Некоторые параметры, такие как минимальная длина пароля и сложность паролей, применяются только к пользователям, созданным непосредственно в Azure AD DS.

Можно создать собственные пользовательские политики паролей для отмены политики по умолчанию в Azure AD DS. Эти пользовательские политики могут быть применены к определенным группам пользователей по мере необходимости.

Для получения дополнительной информации о различиях в том, как политики паролей применяются в зависимости от источника создания пользователя, [см.][password-policy]

## <a name="password-hashes"></a>Хэши паролей

Чтобы выполнять аутентификацию пользователей в управляемом домене, Azure AD DS использует хэши паролей в формате, требуемом для аутентификации NTLM и Kerberos. Пока вы не включите Azure AD DS для клиента, Azure AD не будет создавать и хранить хэши паролей в формате, требуемом для аутентификации NTLM или Kerberos. Из соображений безопасности Azure AD никогда не хранит пароли в виде открытого текста. Поэтому Azure AD не может автоматически создать хэши паролей в формате NTLM или Kerberos по уже существующим учетным данным пользователей.

При использовании облачных учетных записей пользователям нужно сменить пароль, чтобы получить доступ к Azure AD DS. При смене пароля хэши паролей автоматически создаются и сохраняются в Azure AD для аутентификации Kerberos и NTLM. Учетная запись не синхронизируется с Azure AD на DS Azure AD до тех пор, пока пароль не будет изменен.

Для пользователей, синхронизированных с предварительной средой AD DS с помощью Azure AD Connect, [позволяет синхронизировать хэши паролей.][hybrid-phs]

> [!IMPORTANT]
> Azure AD Connect синхронизирует устаревшие хэши паролей только при вхотворе DS Azure AD для вашего aure AD-нанимателя. Устаревшие хэши паролей не используются, если использовать Azure AD Connect только для синхронизации предварительной среды DD DS с Azure AD.
>
> Если ваши устаревшие приложения не используют ntLM аутентификацию или простые привязки LDAP, мы рекомендуем отключить синхронизацию хэша паролем NTLM для Azure AD DS. Для получения дополнительной [информации][secure-domain]см.

После настройки хэши паролей в требуемом формате сохраняются в управляемом домене Azure AD DS. Если вы удалите управляемый домен Azure AD DS, вместе с ним будут удалены все сохраненные хэши паролей. Синхронизированная информация о учетных данных в Azure AD не может быть повторно использована, если позже вы создадите домена Azure AD DS - необходимо перенастроить синхронизацию хэша паролей для повторного хранения хэшов паролей. Ранее присоединенные к домену виртуальные машины и пользователи не смогут сразу выполнить аутентификацию, пока Azure AD не создаст и не сохранит хэши паролей в новом управляемом домене Azure AD DS. См. сведения о [синхронизации хэшей паролей в Azure AD DS и Azure AD Connect][azure-ad-password-sync].

> [!IMPORTANT]
> Azure AD Connect следует устанавливать и настраивать только для синхронизации с локальными средами AD DS. Установка Azure AD Connect в управляемом домене Azure AD DS для синхронизации объектов обратно в Azure AD не поддерживается.

## <a name="forests-and-trusts"></a>Леса и отношения доверия

*Лес* — это логическая конструкция, используемая службами доменов Active Directory Domain Services (AD DS) для группы одного или нескольких *доменов.* Затем домены хранят объекты для пользователей или групп и предоставляют услуги аутентификации.

В Azure AD DS лес содержит только один домен. В помещениях AD DS леса часто содержат много доменов. В крупных организациях, особенно после слияний и поглощений, вы можете в конечном итоге с несколькими на территории леса, которые каждый затем содержат несколько доменов.

По умолчанию управляемый dS-домен Azure AD создается как лес *пользователя.* Лес этого типа синхронизирует все объекты из Azure AD, включая учетные записи пользователей, созданные в локальной среде AD DS. Учетные записи пользователей могут напрямую аутентифицироваться на домене Azure AD DS, например, для вступления в vM, объединенный доменом. Лес пользователя работает, когда хэши паролей могут быть синхронизированы и пользователи не используют эксклюзивные методы регистрации, такие как проверка подлинности смарт-карт.

В *лесном* массиве ресурсов Azure AD DS пользователи удостоверяют подлинность в отношении одностороннего лесного *траста* из своего собственного AD DS. При таком подходе объекты пользователя и хэши паролей не синхронизируются с Azure AD DS. Объекты пользователя и учетные данные существуют только в предварительном AD DS. Такой подход позволяет предприятиям размещать ресурсы и прикладные платформы в Azure, которые зависят от классической аутентификации, таких как LDAPS, Kerberos или NTLM, но любые проблемы проверки подлинности или проблемы удаляются. Леса ресурсов Azure AD DS в настоящее время доступны в предварительной версии.

Более подробную информацию о типах лесов в Azure AD DS можно узнать: [Что такое ресурсные леса?][concepts-forest] и [Как лесные трасты работают в Azure AD DS?][concepts-trust]

## <a name="azure-ad-ds-skus"></a>Azure AD DS SKUs

В Azure AD DS доступная производительность и функции основаны на SKU. Вы выбираете SKU при создании управляемого домена и можете переключать SkUS по мере изменения бизнес-требований после развертывания управляемого домена. В следующей таблице излагаются имеющиеся СКУ и различия между ними:

| Номер SKU   | Максимальное количество объектов | Частота резервного копирования | Максимальное количество исходящих лесных трастов |
|------------|----------------------|------------------|----|
| Standard   | Неограниченно            | Каждые 7 дней     | 0  |
| Enterprise | Неограниченно            | Каждые 3 дня     | 5  |
| Premium    | Неограниченно            | Ежедневно            | 10 |

До этих SkUs Azure AD DS использовалась модель биллинга, основанная на количестве объектов (учетных записей пользователей и компьютеров) в домене Azure AD DS. Больше не существует переменных цен, основанных на количестве объектов в управляемом домене.

Для получения дополнительной информации смотрите [страницу ценообразования Azure AD DS][pricing].

### <a name="managed-domain-performance"></a>Управляемая производительность домена

Производительность домена зависит от того, как осуществляется аутентификация для приложения. Дополнительные вычислительные ресурсы могут помочь улучшить время отклика запроса и сократить время, затрачиваемые на операции синхронизации. По мере повышения уровня SKU увеличивается объем вычислительных ресурсов, доступных для управляемого домена. Отслеживайте производительность приложений и планируйте необходимые ресурсы.

Если ваш бизнес или приложение требует изменений и вам требуется дополнительная вычислительная мощность для вашего домена Azure AD DS, вы можете переключиться на другой SKU.

### <a name="backup-frequency"></a>Частота резервного копирования

Частота резервного копирования определяет частоту частого снимка управляемого домена. Резервное копирование является автоматизированным процессом, управляемым платформой Azure. В случае возникновения проблемы с управляемым доменом поддержка Azure может помочь вам восстановить резервное копирование. Поскольку синхронизация происходит только в одну сторону *от* Azure AD, любые проблемы в домене Azure AD DS не повлияют на среды и функциональность AD DS в предварительном помещении.

По мере повышения уровня SKU увеличивается частота снимков резервного копирования. Просмотрите ваши бизнес-требования и цель точки восстановления (RPO), чтобы определить требуемую частоту резервного копирования для управляемого домена. Если ваши требования к бизнесу или заявкам изменятся и вам нужны более частые резервные копирования, вы можете переключиться на другой SKU.

### <a name="outbound-forests"></a>Исходящие леса

В предыдущем разделе подробно описаны односторонние исходящие лесные трасты от управляемого домена Azure AD DS до предварительной среды AD DS (в настоящее время в предварительном просмотре). SKU определяет максимальное количество лесных трастов, которые можно создать для домена Azure AD DS. Просмотрите требования к бизнесу и приложениям, чтобы определить, сколько трастов вам действительно нужно, и выберите подходящий SKU Azure AD DS. Опять же, если ваши бизнес-требования меняются, и вам нужно создать дополнительные лесные трасты, вы можете переключиться на другой SKU.
