# <a name="azure-active-directory-domain-services-deployment-and-management-for-azure-cloud-solution-providers"></a>Развертывание и управление системами управления ресурсами для систем управления ресурсами для облачных решений Azure

Поставщики облачных решений Azure (CSP) — программа для партнеров Майкрософт и предоставляет лицензионный канал для различных облачных служб Майкрософт. Azure CSP дает партнерам возможность управлять продажами и выставлением счетов, обеспечивать поддержку по выставлению счетов и техническую поддержку и быть единственным непосредственным контактом клиентов. Кроме того, Azure CSP предоставляет полный набор инструментов, включая портал самообслуживания и сопутствующие интерфейсы API. Эти инструменты позволяют партнерам CSP легко подготавливать ресурсы Azure и управлять ими, а также выставлять счета для клиентов и их подписок.

[Портал Partner Center](https://docs.microsoft.com/azure/cloud-solution-provider/overview/partner-center-overview) является отправной точкой для всех партнеров Azure CSP и обеспечивает богатые возможности управления клиентами, автоматизированную обработку и многое другое. Партнеры Azure CSP могут использовать возможности Центра партнеров с помощью пользовательского веб-интерфейса или с помощью PowerShell и различных вызовов API.

На приведенной ниже схеме показан общий принцип работы модели CSP. Здесь у Contoso есть арендатор Active Directory (Azure AD). Она является партнером CSP, и служба Azure CSP развертывает ресурсы в ее подписке Azure CSP и управляет ими. У компании Contoso также могут быть обычные (прямые) подписки Azure, счета за которые выставляются непосредственно этой компании.

![Общие сведения о модели CSP](./media/csp/csp_model_overview.png)

Арендатор партнера CSP имеет три специальные группы агентов - *администраторов,* *агентов Helpdesk* и *агентов по продажам.*

Группа агентов *Администратора* назначается администратору-администратору в аттенере Azure AD Contoso. В результате пользователь, принадлежащий к группе агентов-админ-агентов партнера CSP, имеет привилегии админа клиента в атаренде Azure AD cond компании Contoso.

Когда партнер CSP подготавливает подписку Azure CSP для компании Contoso, их группе агентов администрирования назначается роль владельца для этой подписки. В результате агенты администрирования партнера CSP получают необходимые привилегии для подготовки ресурсов Azure, таких как виртуальные машины, виртуальные сети и доменные службы Azure AD, от имени компании Contoso.

Дополнительные сведения см. в [обзоре Azure CSP](https://docs.microsoft.com/azure/cloud-solution-provider/overview/azure-csp-overview).

## <a name="benefits-of-using-azure-ad-ds-in-an-azure-csp-subscription"></a>Преимущества использования Azure AD DS в подписке Azure CSP

Службы функционального домена Azure Active Directory Domain Services (Azure AD DS) предоставляют управляемые доменные службы, такие как соединение доменов, групповая политика, ldAP, проверка подлинности Kerberos/NTLM, которая полностью совместима с службами активного домена Windows Server Active Directory. На протяжении десятилетий было создано множество приложений для работы с AD и использования этих возможностей. Многие независимые поставщики программного обеспечения создавали и развертывали приложения в локальной среде клиентов. Эти приложения трудно поддерживать, так как часто требуется доступ к различным средам, где развертываются приложения. Благодаря подпискам Azure CSP вы получаете более простую альтернативу с возможностями масштабирования и гибкости Azure.

Azure AD DS поддерживает подписку Azure CSP. Приложение можно развернуть в подписке Azure CSP, привязанной к клиенту, арендатору Azure AD. В результате ваши сотрудники (сотрудники службы поддержки) могут управлять, администрировать и обслуживать вво-пог. м, на которых развертывается ваше приложение, используя корпоративные учетные данные вашей организации.

Можно также развернуть домен Azure AD DS, управляемый DS, в клиенте AD-арендатора. Затем приложение подключается к управляемому домену клиента. Возможности в ашего приложении, которые полагаются на Kerberos / NTLM, LDAP или [System.DirectoryServices API,](/dotnet/api/system.directoryservices) работают без проблем с доменом клиента. Конечные клиенты выигрывают от потребления вашего приложения в качестве службы, не беспокоясь о поддержании инфраструктуры, на которое развернуто приложение.

Все выставления счетов за ресурсы Azure, потребляемые в этой подписке, включая Azure AD DS, взимается обратно к вам. Вы сохраняете полный контроль над отношениями с клиентом, когда дело доходит до продаж, выставления счетов, технической поддержки и т.д. Благодаря гибкости платформы Azure CSP небольшая группа агентов поддержки может обслуживать многих таких клиентов, у которых развернуты экземпляры приложения.

## <a name="csp-deployment-models-for-azure-ad-ds"></a>Модели развертывания CSP для Azure AD DS

Существует два способа использования Azure AD DS с подпиской Azure CSP. Выберите соответствующий способ, приняв во внимание требования к безопасности и простоте клиентов.

### <a name="direct-deployment-model"></a>Модель прямого развертывания

В этой модели развертывания Azure AD DS включен в виртуальную сеть, которая принадлежит подписке Azure CSP. Агенты администрирования партнера CSP имеют следующие привилегии:

* *Глобальные* привилегии администратора в клиентском атакжеме Azure AD.
* Привилегии *владельца подписки* в подписке Azure CSP.

![Модель прямого развертывания](./media/csp/csp_direct_deployment_model.png)

В этой модели развертывания агенты администрирования поставщика CSP могут администрировать удостоверения клиента. Эти агенты-админы могут выполнять такие задачи, как предоставление новых пользователей или групп, или добавлять приложения в аттепользователя Azure AD клиента.

Эта модель развертывания может подходить для небольших организаций, которые не имеют специального администратора идентификации или предпочитают партнеру CSP управлять идентификационные данные от их имени.

### <a name="peered-deployment-model"></a>Пиринговая модель развертывания

В этой модели развертывания Azure AD DS включен в виртуальную сеть, принадлежащую клиенту, - прямую подписку Azure, оплаченную клиентом. Партнер CSP может развертывать приложения в виртуальной сети, принадлежащей подписке CSP клиента. Эти виртуальные сети можно подключить с помощью пиринга виртуальных сетей Azure.

При этом развертывании рабочие нагрузки или приложения, развернутые партнером CSP в подписке Azure CSP, могут подключаться к управляемому домену клиента, предусмотренному в прямой подписке клиента Azure.

![Пиринговая модель развертывания](./media/csp/csp_peered_deployment_model.png)

Такая модель развертывания обеспечивает разделение привилегий и позволяет агентам службы технической поддержки партнера CSP администрировать подписку Azure, а также развертывать в ней ресурсы и управлять ими. Однако агентам службы поддержки CSP-партнеров не нужно иметь глобальные привилегии администратора в каталоге Azure AD клиента. Администраторы удостоверений клиента могут по-прежнему управлять удостоверениями в своей организации.

Эта модель развертывания может подходить для сценариев, в которых ISV предоставляет размещенную версию своего собственного приложения, которая также должна быть подключена к Azure AD клиента.

## <a name="administer-azure-ad-ds-in-csp-subscriptions"></a>Администрирование Azure AD DS в подписках CSP

Ниже приведены важные рекомендации по администрированию управляемого домена в подписке Azure CSP.

* **Агенты CSP-админ могут предоставить управляемый домен, используя свои учетные данные:** Azure AD DS поддерживает подписку Azure CSP. Пользователи, принадлежащие к группе агентов-админ-агентов партнера CSP, могут предоставить новый домен Azure AD DS.

* **КСП могут создавать новые управляемые домены для своих клиентов с помощью PowerShell:** [Узнайте, как включить Azure AD DS с помощью PowerShell](powershell-create-instance.md) для получения подробной информации.

* **Агенты CSP-админ не могут выполнять текущие задачи управления в управляемом домене, используя свои учетные данные:** Пользователи CSP-админ не могут выполнять рутинные задачи управления в управляемом домене, используя свои учетные данные. Эти пользователи являются внешними для клиента ВАЗУ AD-налажима, и их учетные данные недоступны в клиенте AD-арендатора Azure AD. Azure AD DS не имеет доступа к хэшам паролей Kerberos и NTLM для этих пользователей, поэтому пользователи не могут быть проверены на доменах Azure AD DS.

  > [!WARNING]
  > Необходимо создать учетную запись пользователя в каталоге клиента для выполнения повседневных задач администрирования в управляемом домене.
  >
  > Вы не можете войти в управляемый домен, используя учетные данные пользователя CSP. Для этого используйте учетные данные учетной записи пользователя, принадлежащей арендатору Azure AD клиента. Эти учетные данные нужны для таких задач, как присоединение вс-мв. к управляемому домену, администрирование DNS или администрирование групповой политики.

* **Учетная запись пользователя, созданная для постоянного администрирования, должна быть добавлена в группу *администраторов AAD DC:* ** группа *администраторов AAD DC* имеет привилегии для выполнения определенных делегированных задач администрирования в управляемом домене. Эти задачи включают настройку DNS, создание организационных подразделений и администрирование групповой политики.
    
    Для выполнения этих задач в управляемом домене партнеру CSP должна быть создана учетная запись пользователя в рамках ad-нанимателя Azure AD клиента. Учетные данные для этой учетной записи должны совместно использоваться агентами администрирования партнера CSP. Кроме того, эта учетная запись пользователя должна быть добавлена в группу *администраторов AAD DC,* чтобы выполнять задачи конфигурации в управляемом домене с помощью этой учетной записи пользователя.
