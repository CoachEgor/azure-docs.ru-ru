# <a name="tutorial-create-an-outbound-forest-trust-to-an-on-premises-domain-in-azure-active-directory-domain-services-preview"></a>Учебник: Создание исходящего лесного траста для локтевого домена в службах домена Active Directory (предварительный просмотр)

В средах, где вы не можете синхронизировать хэши паролей, или у вас есть пользователи, которые только вписываются в использование смарт-карт, чтобы они не знали свой пароль, вы можете использовать лес ресурсов в Azure Active Directory Domain Services (AD DS). Ресурсный лес использует одностороннее исходящие довершиемые средства из Azure AD DS в одну или несколько находящихся в ней средах AD DS. Это доверительное отношение позволяет пользователям, приложениям и компьютерам аутентифицироваться на основе доменов из домена Azure AD DS. Ресурсные леса Azure AD DS в настоящее время находятся в предварительном просмотре.

![Диаграмма лесного траста от Azure AD DS до находинного AD DS](./media/concepts-resource-forest/resource-forest-trust-relationship.png)

В этом руководстве описано следующее:

> [!div class="checklist"]
> * Настройка DNS в среде AD DS для поддержки подключения Azure AD DS
> * Создание одностороннего входящего лесного доверия в предварительной среде AD DS
> * Создание одностороннего исходящего лесного доверия в Azure AD DS
> * Проверка и проверка доверительного отношения для проверки подлинности и доступа к ресурсам

Если у вас еще нет подписки Azure, создайте [учетную запись](https://azure.microsoft.com/free/?WT.mc_id=A261C142F), прежде чем начинать работу.

## <a name="prerequisites"></a>Предварительные требования

Для работы с этим учебником требуются следующие ресурсы и разрешения:

* Активная подписка Azure.
    * Если у вас еще нет подписки Azure, создайте [учетную запись](https://azure.microsoft.com/free/?WT.mc_id=A261C142F).
* Связанный с вашей подпиской клиент Azure Active Directory, синхронизированный с локальным или облачным каталогом.
    * Если потребуется, [создайте клиент Azure Active Directory][create-azure-ad-tenant] или [свяжите подписку Azure со своей учетной записью][associate-azure-ad-tenant].
* Управляемый доменом Azure Active Directory Domain, созданный с помощью ресурсного леса и настроенный в вашем арендаторе Azure AD.
    * Если потребуется, [создайте и настройте экземпляр доменных служб Azure Active Directory][create-azure-ad-ds-instance-advanced].
    
    > [!IMPORTANT]
    > Убедитесь, что с помощью *ресурсного* леса можно убеждаться в создании домена Azure AD DS. Параметр по умолчанию создает лес *пользователя.* Только ресурсные леса могут создавать трасты для сред aD DS на prem. Вы также должны использовать минимум *предприятия* SKU для управляемого домена. При [необходимости измените SKU для домена Azure AD DS, управляемого DS.][howto-change-sku]

## <a name="sign-in-to-the-azure-portal"></a>Вход на портал Azure

В этом учебнике вы создаете и настраиваете исходящий лесной траст из Azure AD DS с помощью портала Azure. Чтобы начать работу, войдите на [портал Azure](https://portal.azure.com).

## <a name="networking-considerations"></a>Рекомендации по работе с сетями

Виртуальная сеть, в размещениивого ресурсного леса Azure AD DS, нуждается в подключении к сети в вашем предпосылок Active Directory. Приложениям и службам также необходимо подключение к виртуальной сети, худеяное лесовой ресурсов Azure AD DS. Подключение к ресурсам ресурсов Azure AD DS должно быть всегда и стабильным, в противном случае пользователи могут не получить аутентификации или получить доступ к ресурсам.

Прежде чем настроить лесной траст в Azure AD DS, убедитесь, что сеть между Azure и предварительной средой соответствует следующим требованиям:

* Используйте частные IP-адреса. Не полагайтесь на DHCP с динамическим назначением IP-адреса.
* Избегайте перекрытия пространства IP-адресов, чтобы позволить виртуальной сети вглядываться и разгрома успешно общаться между Azure и на местах.
* Виртуальная сеть Azure нуждается в подсети шлюза для настройки VPN-соединения сайта (S2S) или ExpressRoute
* Создавайте подсети с достаточным количеством IP-адресов для поддержки сценария.
* Убедитесь, что Azure AD DS имеет свою собственную подсеть, не делитесь этой виртуальной сетевой подсетью с приложениями VMs и службами.
* Вглядываемые виртуальные сети НЕ являются транзитными.
    * Виртуальные сети Azure должны создаваться между всеми виртуальными сетями, которые вы хотите использовать в лесном доверии ресурсов Azure AD DS, для создания в непредпозренных средах AD DS.
* Обеспечить непрерывное подключение к сети в вашем на территории Active Directory forest. Не используйте соединения по требованию.
* Убедитесь, что между вашим лесным именем ресурса Azure AD DS и вашим напредстательным названием Active Directory forest имеется непрерывное разрешение имен (DNS).

## <a name="configure-dns-in-the-on-premises-domain"></a>Настройка DNS в домен

Чтобы правильно разрешить управляемый доменом Azure AD DS из локальной среды, может потребоваться добавление переадресов к существующим DNS-серверам. Если вы не настройлили среду на настройку для связи с доменом Azure AD DS, выполните следующие шаги от рабочей станции управления для домена AD DS:

1. Выберите **«Старт» (англ.) Административные инструменты DNS**
1. Правый выбор DNS-сервера, например *myAD01,* выберите **Свойства**
1. Выберите **нападающие,** затем **отодвите,** чтобы добавить дополнительные нападающие.
1. Добавьте IP-адреса домена Azure AD DS, такие как *10.0.1.4* и *10.0.1.5*.

## <a name="create-inbound-forest-trust-in-the-on-premises-domain"></a>Создание входящего лесного траста в предварительном домене

В домене AD DS требуется входящий лесной траст для домена Azure AD DS. Это доверие должно быть создано вручную в домене AD DS, он не может быть создан на портале Azure.

Чтобы настроить входяельное доверие на домен AD DS, выполните следующие шаги от рабочей станции управления для предварительного домена AD DS:

1. Выберите **«Старт» (англ.) Административные инструменты Активные домены каталога и трасты**
1. Правый выбор домена, *например, onprem.contoso.com,* выберите **Свойства**
1. Выберите вкладку **Trusts,** а затем **New Trust**

   > [!NOTE]
   > Если вы не видите опцию меню **Trusts,** проверьте **свойства** для *типа Forest.* Только *ресурсные* леса могут создавать трасты. Если тип леса является *пользователем,* вы не можете создать трасты. В настоящее время нет возможности изменить тип лесного типа домена Azure AD DS. Вам необходимо удалить и воссоздать управляемый домен в качестве ресурсного леса.

1. Введите имя на доменном имени Azure AD DS, *например, aaddscontoso.com,* затем выберите **Next**
1. Выберите опцию для создания **Лесного траста,** затем создайте **один способ: входящий** траст.
1. Выберите, чтобы создать доверие только для **этого домена**. На следующем этапе вы создадите доверие к порталу Azure для домена Azure AD DS.
1. Выберите для использования **Лесной всей проверки подлинности,** а затем введите и подтвердите пароль доверия. Этот же пароль также вводится на портале Azure в следующем разделе.
1. Шаг через следующие несколько окон с опциями по умолчанию, а затем выбрать вариант для **Нет, не подтверждайте исходящие доверия.**
1. Выберите **отделку**

## <a name="create-outbound-forest-trust-in-azure-ad-ds"></a>Создание исходящего лесного доверия в Azure AD DS

С созданием локального домена AD DS, настроенного для решения управляемого домена Azure AD DS, и созданным входящим лесным трастом, теперь создается исходящий лесной траст. Этот исходящий лесной траст завершает доверительное отношение между доменом AD DS и доменом Azure AD DS.

Чтобы создать исходящий траст для управляемого домена Azure AD DS на портале Azure, выполните следующие шаги:

1. На портале Azure ищите и выберите **службы домена Azure AD,** а затем выберите управляемый домен, например *aaddscontoso.com*
1. Из меню на левой стороне домена Azure AD DS выберите **Трасты,** затем выберите **«Добавить** доверие».
1. Введите имя дисплея, которое идентифицирует ваше доверие, а затем на месте доверенное имя DNS леса, *например, onprem.contoso.com*
1. Предоставьте тот же пароль доверия, который использовался при настройке входящего лесного траста для предварительного домена AD DS в предыдущем разделе.
1. Предоставьте по крайней мере два DNS-сервера для доменa AD DS, таких как *10.0.2.4* и *10.0.2.5*
1. Когда они будут готовы, **сохраните** исходящий лесной траст

    [Создание исходящего лесного доверия на портале Azure](./media/create-forest-trust/portal-create-outbound-trust.png)

### <a name="on-premises-user-authentication-from-the-azure-ad-ds-resource-forest"></a>Проверка подлинности пользователей на территории ресурса Azure AD DS

Виртуальная машина Windows Server должна быть объединена с доменом ресурсов Azure AD DS. Используйте эту виртуальную машину для проверки подлинности вашего собственного пользователя на виртуальной машине.

1. Подключайтесь к Windows Server VM, объединенному к ресурсу Azure AD DS, используя remote Desktop и учетные данные администратора Azure AD DS. Если вы получаете ошибку аутентификации сетевого уровня (NLA), проверьте, что используемая учетная запись пользователя не является учетной записью домена.

    > [!NOTE]
    > Для безопасного подключения к вс-м информации, объединенным в службу домена Azure AD, можно использовать [службу хостов Azure Bastion](https://docs.microsoft.com/azure/bastion/bastion-overview) в поддерживаемых регионах Azure.

1. Откройте запрос команды `whoami` и используйте команду, чтобы показать выдающееся имя в настоящее время проверенного пользователя:

    ```console
    whoami /fqdn
    ```

1. Используйте `runas` команду для проверки подлинности в качестве пользователя из предварительного домена. В следующей команде `userUpn@trusteddomain.com` замените UPN пользователя из надежного доменового помещения. Команда подсказывает вам пароль пользователя:

    ```console
    Runas /u:userUpn@trusteddomain.com cmd.exe
    ```

1. Если проверка подлинности является успешной, открывается новый запрос команды. Название нового запроса команды `running as userUpn@trusteddomain.com`включает в себя.
1. Используйте `whoami /fqdn` в новой команде запрос для просмотра имени подлинного пользователя из предварительного Active Directory.

### <a name="access-resources-in-the-azure-ad-ds-resource-forest-using-on-premises-user"></a>Доступ к ресурсам в ресурсном лесу Azure AD DS с помощью пользователя

Используя Windows Server VM, объединенный в ресурсный лес Azure AD DS, можно протестировать сценарий, при котором пользователи смогут получить доступ к ресурсам, размещенным в лесоводском ресурсе, когда они проверяют подлинность с компьютеров в домене с пользователями из предварительного домена. Следующие примеры показывают, как создавать и тестировать различные общие сценарии.

#### <a name="enable-file-and-printer-sharing"></a>Включить совместное использование файлов и принтеров

1. Подключайтесь к Windows Server VM, объединенному к ресурсу Azure AD DS, используя remote Desktop и учетные данные администратора Azure AD DS. Если вы получаете ошибку аутентификации сетевого уровня (NLA), проверьте, что используемая учетная запись пользователя не является учетной записью домена.

    > [!NOTE]
    > Для безопасного подключения к вс-м информации, объединенным в службу домена Azure AD, можно использовать [службу хостов Azure Bastion](https://docs.microsoft.com/azure/bastion/bastion-overview) в поддерживаемых регионах Azure.

1. Откройте **настройки Windows,** затем выберите и выберите **Центр сети и обмена.**
1. Выберите опцию для расширенных настроек **совместного использования Change.**
1. Под **профилем домена**выберите **«Включите файл и совместное использование принтера»,** а затем **сохраните изменения.**
1. Закрыть **сеть и обмен центр**.

#### <a name="create-a-security-group-and-add-members"></a>Создание группы безопасности и добавление участников

1. Откройте оснастку **Пользователи и компьютеры Active Directory**.
1. Правильно выберите доменное имя, выберите **новое,** а затем выберите **Организационную единицу.**
1. В поле имени введите *LocalObjects,* затем выберите **OK.**
1. Выберите и щелкните правой кнопкой **LocalObjects** в навигационном стеле. Выберите **новые,** а затем **группы**.
1. Тип *FileServerAccess* в поле **имени группы.** Для **группы область,** выберите **локальный домен,** затем выберите **OK.**
1. В панели содержимого, дважды щелкните **FileServerAccess**. Выберите **участников,** выберите **добавить,** а затем выберите **Места.**
1. Выберите локальный активный каталог из представления **«Местоположение»,** а затем выберите **OK.**
1. Введите *пользователей домена* в **введите имена объектов для выбора** коробки. Выберите **Проверить имена**, предоставить учетные данные для находящихся в предварительном Active Directory, а затем выберите **OK**.

    > [!NOTE]
    > Вы должны предоставить учетные данные, потому что доверительные отношения только один способ. Это означает, что пользователи из DS Azure AD не могут получить доступ к ресурсам или искать пользователей или групп в надежном (в непредподанном) домене.

1. Группа **пользователей домена** из вашего внутреннего каталога должна быть членом группы **FileServerAccess.** Выберите **OK,** чтобы сохранить группу и закрыть окно.

#### <a name="create-a-file-share-for-cross-forest-access"></a>Создание файла для доступа к кросс-фолесу

1. На Windows Server VM, примкнув к ресурсу Azure AD DS, создайте папку и предоставьте имя, такое как *CrossForestShare.*
1. Правильный выбор папки и выбрать **Свойства**.
1. Выберите вкладку **безопасности,** а затем выберите **Edit.**
1. В поле диалога *«Разрешения на CrossForestShare»* выберите **Добавить.**
1. Введите *FileServerAccess* в **Введите имена объектов, чтобы выбрать,** затем выберите **OK**.
1. Выберите *FileServerAccess* из групп или списка **имен пользователей.** В списке **Разрешений для FileServerAccess** выберите *Разрешить* для **изменения** и **написать** разрешения, а затем выберите **OK**.
1. Выберите вкладку **«Общий доступ»,** а затем выберите **Расширенный Обмен...**
1. Выберите **Разделите эту папку,** а затем введите памятное имя для файла доля в **имя доля,** такие как *CrossForestShare*.
1. Выберите **Разрешения**. В списке **Разрешений для всех** выберите **Разрешение «Разрешить»** для **изменения.**
1. Выберите **OK** два раза, а затем **закрыть**.

#### <a name="validate-cross-forest-authentication-to-a-resource"></a>Проверка проверки подлинности кросс-фоссов на ресурсе

1. Войти в windows-компьютер, соединенный с вашим предприимчивым Active Directory, используя учетную запись пользователя из вашего собственного Active Directory.
1. Используя **Windows Explorer,** подключитесь к созданной вами акции с `\\fs1.aaddscontoso.com\CrossforestShare`использованием полностью квалифицированного имени хоста и такой доли, как.
1. Для проверки разрешения на запись, правильно выбрать в папке, выберите **новый,** а затем выберите **текстовый документ**. Используйте имя по умолчанию **Новый текстовый документ**.

    При правильном настройке разрешений на запись создается новый текстовый документ. Следующие шаги будут открываться, отсваивать и удалять файл по мере необходимости.
1. Для проверки разрешения на чтение откройте **новый текстовый документ**.
1. Чтобы проверить разрешение на изменение, добавьте текст в файл и закройте **блокнот.** При попросевом для сохранения изменений, выберите **Сохранить**.
1. Для проверки разрешения на удаление, право выбрать **новый текстовый документ** и выбрать **Удалить**. Выберите **«Да»,** чтобы подтвердить удаление файла.
