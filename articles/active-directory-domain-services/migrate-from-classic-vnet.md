# <a name="migrate-azure-ad-domain-services-from-the-classic-virtual-network-model-to-resource-manager"></a>Переносите службы домена Azure AD из классической виртуальной сетевой модели в менеджер ресурсов

Службы домена Active Directory Domain Services (AD DS) поддерживают одноразовый переход для клиентов, использующих в настоящее время классическую виртуальную сетевую модель для виртуальной сетевой модели Resource Manager. Управляемые доменами Azure AD DS, которые используют модель развертывания ресурсного менеджера, предоставляются дополнительные функции, такие как мелкозернистой политики паролей, журналы аудита и защита блокировки учетных записей.

В этой статье излагаются преимущества и соображения для миграции, а затем необходимые шаги для успешной миграции существующего экземпляра Azure AD DS.

> [!NOTE]
> В 2017 году службы домена Azure AD стали доступны для размещения в сети управления ресурсами Azure. С тех пор мы смогли создать более безопасную службу, используя современные возможности менеджера ресурсов Azure. Поскольку развертывание ресурсов Azure полностью заменит классические развертывания, классические виртуальные сетевые развертывания Azure AD DS будут отменены 1 марта 2023 года.
>
> Для получения дополнительной информации [см.](https://azure.microsoft.com/updates/we-are-retiring-azure-ad-domain-services-classic-vnet-support-on-march-1-2023/)

## <a name="overview-of-the-migration-process"></a>Обзор процесса миграции

Процесс миграции занимает существующий экземпляр Azure AD DS, который работает в виртуальной сети Classic, и перемещает его в существующую виртуальную сеть Resource Manager. Миграция выполняется с помощью PowerShell и имеет два основных этапа исполнения - *подготовка* и *миграция.*

![Обзор процесса миграции для Azure AD DS](media/migrate-from-classic-vnet/migration-overview.png)

На этапе *подготовки* Azure AD DS занимает резервную поддержку домена, чтобы получить последний снимок пользователей, групп и паролей, синхронизированных с управляемым доменом. Синхронизация затем отключается, а облачный сервис, в котором размещается домен Azure AD DS, удаляется. На этапе подготовки управляемый домен Azure AD DS не может проверить подлинность пользователей.

![Этап подготовки к миграции Azure AD DS](media/migrate-from-classic-vnet/migration-preparation.png)

На этапе *миграции* скопированы базовые виртуальные диски для контроллеров доменов из домена Classic Azure AD DS для создания виртуальных технологий с использованием модели развертывания ресурсного менеджера. Затем воссоздан управляемый домен Azure AD DS, который включает конфигурацию LDAPS и DNS. Синхронизация с Azure AD перезапущена, а сертификаты LDAP восстановлены. Нет необходимости присоединяться к любым машинам в домене Azure AD DS — они по-прежнему соединяются с управляемым доменом и работают без изменений.

![Миграция Azure AD DS](media/migrate-from-classic-vnet/migration-process.png)

## <a name="migration-benefits"></a>Миграционные льготы

При перемещении управляемого домена Azure AD DS с помощью этого процесса миграции необходимо воссоединение машин с управляемым доменом или удаление экземпляра DS Azure AD и создание его с нуля. В конце процесса миграции vM-чаты по-прежнему присоединяются к управляемому домену Azure AD DS.

После миграции Azure AD DS предоставляет множество функций, доступных только для доменов с использованием виртуальных сетей Resource Manager, таких как:

* Поддержка политики паролей с тонкой зернисто-зернисто-зернисто-сули
* Защита блокировки учетной записи AD.
* Уведомления о предупреждениях по электронной почте на домене Azure AD DS.
* Аудит журналов с помощью Azure Monitor.
* Интеграция файлов Azure
* Интеграция HD Insights

Управляемые доменами Azure AD DS, которые используют виртуальную сеть Resource Manager, помогают вам быть в курсе последних новых функций. Поддержка Azure AD DS с использованием виртуальных сетей Classic будет универслена в будущем.

## <a name="example-scenarios-for-migration"></a>Примеры сценариев миграции

Некоторые распространенные сценарии миграции домена Azure AD DS включают следующие примеры.

> [!NOTE]
> Не конвертировать классическую виртуальную сеть до тех пор, пока вы не подтвердите успешную миграцию. Преобразование виртуальной сети удаляет возможность отката или восстановления управляемого домена Azure AD DS при наличии каких-либо проблем на этапах миграции и проверки.

### <a name="migrate-azure-ad-ds-to-an-existing-resource-manager-virtual-network-recommended"></a>Перенести Azure AD DS в существующую виртуальную сеть менеджера ресурсов (рекомендуется)

Распространенным сценарием является то, что вы уже перевели другие существующие классические ресурсы в модель развертывания диспетчера ресурсов и виртуальную сеть. Затем пиринг используется из виртуальной сети Resource Manager и переходят к виртуальной сети Classic, которая продолжает запускать Azure AD DS. Этот подход позволяет приложениям и службам управления ресурсами использовать функции аутентификации и управления доменом Azure AD DS в виртуальной сети Classic. После переноса все ресурсы работают с помощью модели развертывания ресурсного менеджера и виртуальной сети.

![Перенести Azure AD DS в существующую виртуальную сеть менеджера ресурсов](media/migrate-from-classic-vnet/migrate-to-existing-vnet.png)

Шаги высокого уровня, участвующие в этом примере сценария миграции, включают следующие части:

1. Удалите существующие VPN шлюзы или виртуальную сеть пиринга, настроенную на классическую виртуальную сеть.
1. Воспроизвносите управляемый домен Azure AD DS, используя шаги, изложенные в этой статье.
1. Проверьте и подтвердите успешную миграцию, а затем удалите классическую виртуальную сеть.

### <a name="migrate-multiple-resources-including-azure-ad-ds"></a>Мигрировать несколько ресурсов, включая Azure AD DS

В этом примере можно перенести DS Azure AD и другие связанные с ними ресурсы из модели развертывания Classic в модель развертывания диспетчеров ресурсов. Если некоторые ресурсы продолжали работать в виртуальной сети Classic наряду с доменом Azure AD DS, все они могут извлечь выгоду из перехода на модель развертывания диспетчера ресурсов.

![Перемотка нескольких ресурсов в модель развертывания диспетчера ресурсов](media/migrate-from-classic-vnet/migrate-multiple-resources.png)

Шаги высокого уровня, участвующие в этом примере сценария миграции, включают следующие части:

1. Удалите существующие VPN шлюзы или виртуальную сеть пиринга, настроенную на классическую виртуальную сеть.
1. Воспроизвносите управляемый домен Azure AD DS, используя шаги, изложенные в этой статье.
1. Настройка виртуальной сети между виртуальной сетью Classic и сетью Resource Manager.
1. Проверьте и подтвердите успешную миграцию.
1. [Перемещение дополнительных классических ресурсов, таких как VMs][migrate-iaas].

### <a name="migrate-azure-ad-ds-but-keep-other-resources-on-the-classic-virtual-network"></a>Мигрируйте Azure AD DS, но сохраните другие ресурсы в виртуальной сети Classic

В этом примере сценария у вас есть минимальное количество простоев в одном сеансе. Вы только переносите Azure AD DS в виртуальную сеть менеджера ресурсов и сохраняете существующие ресурсы в модели классического развертывания и виртуальной сети. В следующий период обслуживания можно перенести дополнительные ресурсы из модели развертывания Classic и виртуальной сети по желанию.

![Мигрировать только DS Azure AD в модель развертывания диспетчера ресурсов](media/migrate-from-classic-vnet/migrate-only-azure-ad-ds.png)

Шаги высокого уровня, участвующие в этом примере сценария миграции, включают следующие части:

1. Удалите существующие VPN шлюзы или виртуальную сеть пиринга, настроенную на классическую виртуальную сеть.
1. Воспроизвносите управляемый домен Azure AD DS, используя шаги, изложенные в этой статье.
1. Настройка виртуальной сети между виртуальной сетью Classic и новой виртуальной сетью Resource Manager.
1. [Позже, перенести дополнительные ресурсы][migrate-iaas] из виртуальной сети Classic по мере необходимости.

## <a name="before-you-begin"></a>Перед началом

При подготовке и переносе домена Azure AD DS возникают некоторые соображения, учитывающие доступность служб аутентификации и управления. Управляемый домен Azure AD DS недоступен в течение определенного периода времени во время миграции. Приложения и службы, опирающиеся на aD-интерфейс Azure AD, испытывают время простоя во время миграции.

> [!IMPORTANT]
> Прочитайте все эти статьи и рекомендации по миграции, прежде чем начать процесс миграции. Процесс миграции влияет на доступность контроллеров доменов Azure AD DS на периодвремени. Пользователи, службы и приложения не могут проверить подлинность в отношении управляемого домена во время процесса миграции.

### <a name="ip-addresses"></a>IP-адреса

IP-адреса контроллера домена для DS Azure AD управляемые изменения домена после миграции. Это изменение включает общедоступный IP-адрес для безопасной конечной точки LDAP. Новые IP-адреса находятся внутри диапазона адресов для новой подсети в виртуальной сети Resource Manager.

В случае отката IP-адреса могут меняться после отката.

Azure AD DS обычно использует первые два доступных IP-адреса в диапазоне адресов, но это не гарантируется. В настоящее время нельзя указать IP-адреса для использования после миграции.

### <a name="downtime"></a>Простой

Процесс миграции включает в себя, что контроллеры доменов в автономном режиме в течение определенного периода времени. Контроллеры доменов недоступны, в то время как Azure AD DS переносится на модель развертывания диспетчера ресурсов и виртуальную сеть. В среднем время простоя составляет от 1 до 3 часов. Этот период времени от того, когда контроллеры домена принимаются в автономном режиме до момента первого контроллера домена возвращается в Интернете. Это среднее значение не включает время, необходимое для репликации второго контроллера домена, или время, которое может потребоваться для переноса дополнительных ресурсов в модель развертывания диспетчера ресурсов.

### <a name="account-lockout"></a>Блокировка учетной записи

Управляемые доменами Azure AD DS, которые работают в виртуальных сетях Classic, не имеют политикблокировки блокировки учетной записи AD. Если ВМ подвергаются воздействию Интернета, злоумышленники могут использовать методы спрей паролей, чтобы грубо проложить свой путь в учетные записи. Нет политики блокировки учетной записи, чтобы остановить эти попытки. Для управляемых доменов Azure AD DS, которые используют модель развертывания ресурсного менеджера и виртуальные сети, политики блокировки учетной записи AD защищают от этих атак с прерывок паролей.

По умолчанию 5 неудачных попыток пароля за 2 минуты заблокировать учетную запись на 30 минут.

Заблокированная учетная запись не может быть использована для входини, что может помешать управлению доменом Azure AD DS или приложениями, управляемыми учетной записью. После того, как управляемый домен Azure AD DS будет перенесен, учетные записи могут испытывать то, что кажется постоянным блокировкой из-за неоднократных неудачных попыток войти в систему. Два распространенных сценария после миграции включают в себя следующие:

* Учетная запись службы, используюва просроченный пароль.
    * Учетная запись сервиса неоднократно пытается войти с просроченным паролем, который блокирует учетную запись. Чтобы исправить это, найдите приложение или VM с истекшим сроком действия учетных данных и обновите пароль.
* Вредоносная сущность использует попытки грубой силы войти в учетные записи.
    * Когда вс-м внешёвжок подвергаются воздействию Интернета, злоумышленники часто пробуют общие комбинации имени пользователя и паролей при попытке подписать. Эти неоднократные неудачные попытки входной системы могут блокировать учетные записи. Не рекомендуется использовать учетные записи администраторов с общими именами, такими как *администратор* или *администратор,* например, чтобы свести к минимуму блокировку административных учетных записей.
    * Свести к минимуму количество вс-миноностов, которые подвергаются воздействию Интернета. С помощью портала Azure Bastion можно использовать [Azure Bastion][azure-bastion] для безопасного подключения к вм-исключам.

Если вы подозреваете, что некоторые учетные записи могут быть заблокированы после миграции, в заключительных шагах миграции описано, как включить аудит или изменить параметры тонкозернистого параметра политики паролей.

### <a name="roll-back-and-restore"></a>Откат и восстановление

Если миграция не удалась, происходит процесс отката или восстановления домена Azure AD DS. Откат — это опция самообслуживания, которая немедленно возвращает состояние управляемого домена до попытки миграции. Инженеры службы поддержки Azure также могут восстановить управляемый домен из резервного копирования в качестве последнего средства. Для получения дополнительной информации узнайте, [как откатить или восстановить из неисправной миграции.](#roll-back-and-restore-from-migration)

### <a name="restrictions-on-available-virtual-networks"></a>Ограничения на доступные виртуальные сети

Существуют некоторые ограничения для виртуальных сетей, в которые можно перенести управляемый dS Azure AD. Виртуальная сеть destination Resource Manager должна соответствовать следующим требованиям:

* Виртуальная сеть Resource Manager должна находиться в той же подписке Azure, что и классическая виртуальная сеть, в которую в настоящее время развернута DS Azure AD.
* Виртуальная сеть Resource Manager должна находиться в том же регионе, что и классическая виртуальная сеть, в которую в настоящее время развернута Azure AD DS.
* Подсеть виртуальной сети Resource Manager должна иметь не менее 3-5 доступных IP-адресов.
* Подсеть виртуальной сети Resource Manager должна быть выделенной подсетью для Azure AD DS и не должна размещать другие рабочие нагрузки.

## <a name="update-and-verify-virtual-network-settings"></a>Обновление и проверка виртуальных настроек сети

Прежде чем начать процесс миграции, выполните следующие первоначальные проверки и обновления. Эти действия могут произойти в любое время до миграции и не повлиять на работу домена Azure AD DS.

1. Обновите местную среду Azure PowerShell до последней версии. Для выполнения этапов миграции требуется, по крайней мере, версия *2.3.2*.

    Информацию о том, как проверить и обновить версию PowerShell, можно узнать в [обзоре Azure PowerShell.][azure-powershell]

1. Создайте или выберите существующую виртуальную сеть Resource Manager.

    Убедитесь, что настройки сети не блокируют необходимые порты, необходимые для DS Azure AD. Порты должны быть открыты как в виртуальной сети Classic, так и в виртуальной сети Resource Manager. Эти параметры включают таблицы маршрутов (хотя не рекомендуется использовать таблицы маршрутов) и группы сетевой безопасности.

    Для просмотра требуемых портов см. [группы безопасности сети и требуемые порты.][network-ports] Чтобы свести к минимуму проблемы сетевой связи, рекомендуется подождать и применить группу сетевой безопасности или таблицу маршрутов к виртуальной сети Resource Manager после успешного завершения миграции.

    Обратите внимание на эту целевую группу ресурсов, таргетинг на виртуальную сеть и на виртуальную сетевую подсеть. Эти имена ресурсов используются в процессе миграции.

1. Проверьте управляемое здоровье домена Azure AD DS на портале Azure. Если у вас есть какие-либо оповещения для управляемого домена, удалите их, прежде чем начать процесс миграции.
1. В обязательном порядке, если вы планируете переместить другие ресурсы в модель развертывания диспетчера ресурсов и виртуальную сеть, подтвердите, что эти ресурсы могут быть перенесены. Для получения дополнительной [Platform-supported migration of IaaS resources from Classic to Resource Manager][migrate-iaas]информации см.

    > [!NOTE]
    > Не преобразуйте виртуальную сеть Classic в виртуальную сеть Resource Manager. Если вы это сделаете, нет возможности откатить или восстановить домен Azure AD DS.

## <a name="prepare-the-managed-domain-for-migration"></a>Подготовка управляемого домена для миграции

Azure PowerShell используется для подготовки домена Azure AD DS, управляемого для миграции. Эти шаги включают в себя резервное копирование, приостановку синхронизации и удаляние облачного сервиса, в котором размещается DD Azure AD. Когда этот шаг завершается, Azure AD DS снимается в автономном режиме в течение определенного периода времени. Если шаг подготовки не удается, вы можете [откатить к предыдущему состоянию.](#roll-back)

Чтобы подготовить к миграции управляемый домен Azure AD DS, выполните следующие шаги:

1. Установите `Migrate-Aaads` скрипт из [галереи PowerShell.][powershell-script] Этот скрипт миграции PowerShell — цифровой контракт, подписанный командой разработчиков Azure AD.

    ```powershell
    Install-Script -Name Migrate-Aadds
    ```

1. Создайте переменную для удержания учетных данных по скрипту миграции с помощью cmdlet [Get-Credential.][get-credential]

    Указано, что учетная запись пользователя требует глобальных привилегий *администратора* в вашем клиенте Azure AD для включения DS Azure AD, а затем привилегий *contributor* в подписке Azure для создания необходимых ресурсов Azure AD DS.

    По запросу введите соответствующую учетную запись пользователя и пароль:

    ```powershell
    $creds = Get-Credential
    ```

1. Теперь `Migrate-Aadds` запустите cmdlet с использованием *параметра -Подготовители.* Предоставьте *-ManagedDomainFqdn* для вашего собственного домена Azure AD DS, например, *aaddscontoso.com:*

    ```powershell
    Migrate-Aadds `
        -Prepare `
        -ManagedDomainFqdn aaddscontoso.com `
        -Credentials $creds
    ```

## <a name="migrate-the-managed-domain"></a>Перенести управляемый домен

При подготовке и резервном копировании домена Azure AD DS можно перенести домен. Этот шаг воссоздает контроллер домена Доменных служб Azure AD Services с помощью модели развертывания ресурсного менеджера. Этот шаг может занять от 1 до 3 часов.

Выполнить `Migrate-Aadds` cmdlet с помощью параметра *-Commit.* Предоставьте *-ManagedDomainFqdn* для вашего собственного домена Azure AD DS, подготовленного в предыдущем разделе, например, *aaddscontoso.com:*

Укажите группу целевых ресурсов, содержащую виртуальную сеть, в которую вы хотите перенести DD Azure, например *myResourceGroup.* Предоставьте целевую виртуальную сеть, такую как *myVnet,* и подсеть, например *DomainServices.*

После запуска этой команды вы не можете откатить:

```powershell
Migrate-Aadds `
    -Commit `
    -ManagedDomainFqdn aaddscontoso.com `
    -VirtualNetworkResourceGroupName myResourceGroup `
    -VirtualNetworkName myVnet `
    -VirtualSubnetName DomainServices `
    -Credentials $creds
```

После проверки скрипта управляемый домен подготовлен к миграции, введите *Y,* чтобы начать процесс миграции.

> [!IMPORTANT]
> Не преобразовывайте виртуальную сеть Classic в виртуальную сеть Resource Manager во время процесса миграции. При преобразовании виртуальной сети нельзя откатить или восстановить управляемый dS-домен Azure AD, поскольку исходная виртуальная сеть больше не будет существовать.

Каждые две минуты в процессе миграции индикатор прогресса сообщает о текущем состоянии, как показано в следующем примере вывода:

![Индикатор прогресса миграции Azure AD DS](media/migrate-from-classic-vnet/powershell-migration-status.png)

Процесс миграции продолжает работать, даже если вы закрываете сценарий PowerShell. На портале Azure состояние управляемого домена сообщается как *Migrating.*

Когда миграция успешно завершается, можно просмотреть IP-адрес ip-адреса первого контроллера домена на портале Azure или через Azure PowerShell. Также отображается смета времени на втором контроллере домена.

На этом этапе можно дополнительно переместить другие существующие ресурсы из модели развертывания Classic и виртуальной сети. Или можно сохранить ресурсы в модели развертывания Classic и свести виртуальные сети друг к другу после завершения миграции DS Azure AD.

## <a name="test-and-verify-connectivity-after-the-migration"></a>Тестирование и проверка подключения после миграции

Успешное развертывание и использование в домене Azure AD DS может занять некоторое время.

С помощью модели развертывания диспетчера ресурсов сетевые ресурсы для домена Azure AD DS отображаются на портале Azure или Azure PowerShell. Подробнее о том, что представляют собой эти сетевые ресурсы, можно узнать, смотрите [в сетевых ресурсах, используемых DD Azure AD.][network-resources]

Когда доступен хотя бы один контроллер домена, выполните следующие этапы конфигурации для подключения сети к вс-м в.

* **Обновление настроек DNS сервера** Чтобы другие ресурсы виртуальной сети Resource Manager разрешали и использовали управляемый домен Azure AD DS, обновите настройки DNS с помощью IP-адресов новых контроллеров доменов. Портал Azure может автоматически настроить эти настройки для вас. Чтобы узнать больше о том, как настроить виртуальную сеть Resource Manager, смотрите [настройки обновления DNS для виртуальной сети Azure.][update-dns]
* **Перезапуск книжки с миноносными вводом** - По мере изменения IP-адресов сервера DNS для контроллеров доменов Azure AD DS перезапускают любые конменты, объединенные vMs, чтобы они затем использовали новые настройки DNS-сервера. Если приложения или визг и ручно настроены настройки DNS, вручную обновляйте их с помощью новых IP-адресов DNS-сервера контроллеров домена, отображаемых на портале Azure.

Теперь проверьте виртуальное сетевое соединение и разрешение имени. На VM, который подключен к виртуальной сети Resource Manager или заглянул к ней, попробуйте следующие тесты сетевой коммуникации:

1. Проверьте, можно ли пинговать IP-адрес одного из контроллеров домена, например`ping 10.1.0.4`
    * IP-адреса контроллеров доменов отображаются на странице **Свойств** для домена Azure AD DS на портале Azure.
1. Проверить разрешение имен управляемого домена, например`nslookup aaddscontoso.com`
    * Укажите имя DNS для собственного домена Azure AD DS, чтобы убедиться, что настройки DNS являются правильными и разрешаются.

Второй контроллер домена должен быть доступен через 1-2 часа после завершения миграционного cmdlet. Чтобы проверить, доступен ли второй контроллер домена, на портале Azure посетите страницу **Свойств** для управляемого домена Azure AD DS. Если отображается два IP-адреса, второй контроллер домена готов.

## <a name="optional-post-migration-configuration-steps"></a>Дополнительные постмиграционные шаги конфигурации

Когда процесс миграции успешно завершен, некоторые дополнительные шаги конфигурации включают включение журналов аудита или уведомлений по электронной почте или обновление мелкозернистой политики паролей.

### <a name="subscribe-to-audit-logs-using-azure-monitor"></a>Подписаться на журналы аудита с помощью Azure Monitor

Azure AD DS предоставляет журналы аудита, помогая устранить неполадки и просматривать события на контроллерах домена. Для получения дополнительной [информации см.][security-audits]

Шаблоны можно использовать для мониторинга важной информации, разослаенных в журналах. Например, шаблон рабочей книги журналов аудита может отслеживать возможные блокировки учетных записей в домене Azure AD DS.

### <a name="configure-azure-ad-domain-services-email-notifications"></a>Настройка уведомлений по электронной почте Службы домена Azure AD

Чтобы уведомлять о обнаружении проблемы в домене Azure AD DS, обновите настройки уведомления электронной почты на портале Azure. Для получения дополнительной [информации см.][notifications]

### <a name="update-fine-grained-password-policy"></a>Обновление тонкозернистого политики паролей

При необходимости можно обновить мелкозернистой политики паролей, чтобы быть менее ограничительной, чем конфигурация по умолчанию. Можно использовать журналы аудита, чтобы определить, имеет ли смысл менее ограничительный параметр, а затем настроить политику по мере необходимости. Используйте следующие шаги высокого уровня для проверки и обновления параметров политики для учетных записей, которые неоднократно блокируются после миграции:

1. [Найдите политику паролей][password-policy] для меньшего количества ограничений на домене Azure AD DS и наблюдайте за событиями в журналах аудита.
1. Если какие-либо учетные записи служб используют просроченные пароли, указанные в журналах аудита, обновите эти учетные записи с помощью правильного пароля.
1. Если VM подвергается воздействию Интернета, просмотрите общие имена учетных записей, таких как *администратор,* *пользователь*или *гость* с высокими попытками входной связи. Там, где это возможно, обновите эти VMs, чтобы использовать менее общеназванные учетные записи.
1. Используйте сетевой след на VM, чтобы найти источник атак и заблокировать эти IP-адреса от возможности попытки входного вбрасывания.
1. При минимальных проблемах блокировки обновляйте полис блокировки, чтобы быть максимально ограничительным по мере необходимости.

### <a name="creating-a-network-security-group"></a>Создание группы безопасности сети

DS Azure AD нуждается в группе сетевой безопасности для защиты портов, необходимых для управляемого домена, и блокирования всего входящего трафика. Эта группа сетевой безопасности действует как дополнительный уровень защиты для блокировки доступа к управляемому домену и не создается автоматически. Чтобы создать группу сетевой безопасности и открыть необходимые порты, просмотрите следующие шаги:

1. На портале Azure выберите ресурс Azure AD DS. На странице обзора отображается кнопка для создания группы сетевой безопасности, если с службами домена Azure AD нет никаких связей.
1. Если вы используете безопасный LDAP, добавьте правило в группу сетевой безопасности, чтобы разрешить входящий трафик для порта *TCP* *636.* Для получения дополнительной информации [см.][secure-ldap]

## <a name="roll-back-and-restore-from-migration"></a>Откат назад и восстановление от миграции

До определенного момента в процессе миграции можно откатить или восстановить управляемый доменом Azure AD DS.

### <a name="roll-back"></a>Откат

Если есть ошибка при запуске cmdlet PowerShell для подготовки к миграции в шаге 2 или к самой миграции в шаге 3, управляемый домен Azure AD DS может откатиться к исходной конфигурации. Этот откат требует оригинальной виртуальной сети Classic. Обратите внимание, что IP-адреса могут по-прежнему меняться после отката.

Запустите `Migrate-Aadds` cmdlet с помощью параметра *- Abort.* Предоставьте *-ManagedDomainFqdn* для вашего собственного домена Azure AD DS, подготовленного в предыдущем разделе, например, *aaddscontoso.com*и название классической виртуальной сети, например, *myClassicVnet:*

```powershell
Migrate-Aadds `
    -Abort `
    -ManagedDomainFqdn aaddscontoso.com `
    -ClassicVirtualNetworkName myClassicVnet `
    -Credentials $creds
```

### <a name="restore"></a>Восстановить

В крайнем случае, службы домена Azure AD могут быть восстановлены из последнего доступного резервного копирования. Резервное копирование принимается в шаге 1 миграции, чтобы убедиться, что наиболее текущая резервная информация доступна. Эта резервная часть хранится в течение 30 дней.

Чтобы восстановить управляемый доменом Azure AD DS из резервного копирования, [откройте билет для обеспечения поддержки с помощью портала Azure.][azure-support] Предоставьте свой идентификатор каталога, доменное имя и причину восстановления. Процесс поддержки и восстановления может занять несколько дней.
