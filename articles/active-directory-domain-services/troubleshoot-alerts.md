# <a name="known-issues-common-alerts-and-resolutions-in-azure-active-directory-domain-services"></a>Известные проблемы: Общие оповещения и разрешения в службах домена Active Directory

В качестве центральной части идентификации и аутентификации приложений службы azure Active Directory Domain Domain Services (Azure AD DS) иногда возникает проблемы. Если вы столкнетесь с проблемами, есть некоторые общие оповещения и связанные с ними шаги по устранению неполадок, которые помогут вам снова запустить вещи. В любое время вы также можете [открыть запрос службы поддержки Azure][azure-support] на дополнительную помощь в устранении неполадок.

В этой статье содержится информация об устранении неполадок для общих оповещений в Azure AD DS.

## <a name="aadds100-missing-directory"></a>AADDS100: отсутствующий каталог

### <a name="alert-message"></a>Сообщение оповещения

*Каталог Azure AD, связанный с управляемым доменом, возможно, был удален. Управляемый домен больше не находится в поддерживаемой конфигурации. Корпорация Майкрософт не может контролировать, управлять, исправлять и синхронизировать управляемый домен.*

### <a name="resolution"></a>Решение

Эта ошибка обычно возникает при перемещении подписки Azure в новый каталог Azure AD и удаляется старый каталог Azure AD, связанный с DD Azure AD.

Эта ошибка является неисправимой. Чтобы разрешить оповещение, [удалите существующий домен Azure AD DS](delete-aadds.md) и воссоздайте его в новом каталоге. Если у вас возникли проблемы с удалянием домена Azure AD DS, [откройте запрос службы поддержки Azure][azure-support] для дополнительной помощи в устранении неполадок.

## <a name="aadds101-azure-ad-b2c-is-running-in-this-directory"></a>AADDS101: Azure AD B2C выполняется в этом каталоге

### <a name="alert-message"></a>Сообщение оповещения

*В каталоге Azure AD B2C невозможно включить доменные службы Azure AD*

### <a name="resolution"></a>Решение

Azure AD DS автоматически синхронизируется с каталогом Azure AD. Если каталог Azure AD настроен для B2C, DS Azure AD не может быть развернут и синхронизирован.

Чтобы использовать Azure AD DS, необходимо воссоздать управляемый домен в каталоге не Azure AD B2C, используя следующие действия:

1. [Удалите управляемый доменом Azure AD DS](delete-aadds.md) из существующего каталога Azure AD.
1. Создайте новый каталог Azure AD, который не является каталогом Azure AD B2C.
1. [Создайте заменяющий домен Azure AD DS.](tutorial-create-instance.md)

Здоровье домена Azure AD DS автоматически обновляется в течение двух часов и удаляет оповещение.

## <a name="aadds103-address-is-in-a-public-ip-range"></a>AADDS103: адрес находится в общедоступном диапазоне IP-адресов

### <a name="alert-message"></a>Сообщение оповещения

*Диапазон IP-адресов для виртуальной сети, в которой вы включили службы доменов Azure AD, находится в общедоступном диапазоне IP. Службы домена Azure AD должны быть включены в виртуальную сеть с частным диапазоном IP-адресов. Эта конфигурация влияет на способность корпорации Майкрософт контролировать, управлять, исправлять и синхронизировать управляемый домен.*

### <a name="resolution"></a>Решение

Прежде чем начать, убедитесь, что вы [понимаете, частные IP v4 адрес пространства.](https://en.wikipedia.org/wiki/Private_network#Private_IPv4_address_spaces)

Внутри виртуальной сети виртуальные виртуальные компании могут делать запросы на ресурсы Azure в том же диапазоне IP-адресов, что и настроены для подсети. Если настроить общедоступный диапазон IP-адресов для подсети, запросы, маршрутизиваемые в виртуальной сети, могут не достичь предполагаемых веб-ресурсов. Такая конфигурация может привести к непредсказуемым ошибкам с Azure AD DS.

> [!NOTE]
> Если вам принадлежит диапазон IP-адресов в Интернете, настроенный в виртуальной сети, то это оповещение можно игнорировать. Однако службы домена Azure AD не могут подставить [sLA](https://azure.microsoft.com/support/legal/sla/active-directory-ds/v1_0/)с этой конфигурацией, так как это может привести к непредсказуемым ошибкам.

Чтобы разрешить это оповещение, удалите существующий домен Azure AD DS и воссоздайте его в виртуальной сети с частным диапазоном IP-адресов. Этот процесс является разрушительным, так как домен Azure AD DS недоступен, а все созданные пользовательские ресурсы, такие как OUs или учетные записи служб, потеряны.

1. [Удалите из каталога управляемый доменом Azure AD DS.](delete-aadds.md)
1. Чтобы обновить диапазон IP-адресов виртуальной сети, ищите и выберите *виртуальную сеть* на портале Azure. Выберите виртуальную сеть для Azure AD DS, которая неправильно имеет общедоступный набор IP-адресов.
1. Под **настройками**выберите *адресное пространство.*
1. Обновите диапазон адресов, выбрав существующий диапазон адресов и отредактировав его или добавив дополнительный диапазон адресов. Убедитесь, что новый диапазон IP-адресов находится в частном диапазоне IP. При готовности **сохраните** изменения.
1. Выберите **подсети** в левой навигации.
1. Выберите подсеть, которая вы хотите отсеить, или создайте дополнительную подсеть.
1. Обновление или указано частный диапазон IP-адресов, а затем **сохранить** ваши изменения.
1. [Создайте заменяющий домен Azure AD DS.](tutorial-create-instance.md) Убедитесь, что вы выбираете обновленную виртуальную сетевую подсеть с частным диапазоном IP-адресов.

Здоровье домена Azure AD DS автоматически обновляется в течение двух часов и удаляет оповещение.

## <a name="aadds106-your-azure-subscription-is-not-found"></a>AADDS106. Подписка Azure не найдена

### <a name="alert-message"></a>Сообщение оповещения

*Ваша подписка Azure, связанная с управляемым доменом, удалена.  Служба домена Azure AD domain Services требует активной подписки для нормального функционирования.*

### <a name="resolution"></a>Решение

DD Azure AD требует активной подписки и не может быть перемещен в другую подписку. Если подписка Azure, с которой был связан с управляемым доменом Azure AD DS, удалена, необходимо воссоздать подписку Azure и домен Azure AD DS.

1. [Создайте подписку Azure.](../cost-management-billing/manage/create-subscription.md)
1. [Удалите управляемый доменом Azure AD DS](delete-aadds.md) из существующего каталога Azure AD.
1. [Создайте заменяющий домен Azure AD DS.](tutorial-create-instance.md)

## <a name="aadds107-your-azure-subscription-is-disabled"></a>AADDS107. Подписка Azure отключена

### <a name="alert-message"></a>Сообщение оповещения

*Ваша подписка Azure, связанная с управляемым доменом, не активна.  Служба домена Azure AD domain Services требует активной подписки для нормального функционирования.*

### <a name="resolution"></a>Решение

Azure AD DS требует активной подписки. Если подписка Azure, с которой был связан домен Azure AD DS, не активна, необходимо продлить ее для повторной активации подписки.

1. [Почему подписка Azure отключена и как активировать ее повторно?](https://docs.microsoft.com/azure/billing/billing-subscription-become-disable)
2. После обновления подписки уведомление Azure AD DS позволяет повторно включить управляемый домен.

При повторном включении управляемого домена работоспособность домена Azure AD DS автоматически обновляется в течение двух часов и удаляет оповещение.

## <a name="aadds108-subscription-moved-directories"></a>AADDS108. Подписка перемещенная в каталоги

### <a name="alert-message"></a>Сообщение оповещения

*Подписка, используемая службами домена Azure AD, была перемещена в другой каталог. Служба домена Azure AD должна иметь активную подписку в том же каталоге, чтобы нормально функционировать.*

### <a name="resolution"></a>Решение

DD Azure AD требует активной подписки и не может быть перемещен в другую подписку. Если подписка Azure, с которой был связан управляемый домен Azure AD DS, будет перемещена, переместите подписку обратно в предыдущий каталог или [удалите управляемый домен](delete-aadds.md) из существующего каталога и [создайте в выбранной подписке блокированный домен Azure AD DS.](tutorial-create-instance.md)

## <a name="aadds109-resources-for-your-managed-domain-cannot-be-found"></a>AADDS109. Ресурсы для управляемого домена не найдены

### <a name="alert-message"></a>Сообщение оповещения

*Ресурс, используемый для управляемого домена, был удален. Этот ресурс необходим для правильного функционирования служб доменов Azure AD.*

### <a name="resolution"></a>Решение

Azure AD DS создает дополнительные ресурсы для нормального функционирования, такие как общедоступные IP-адреса, виртуальные сетевые интерфейсы и балансилер нагрузки. Если какой-либо из этих ресурсов удален, управляемый домен находится в неподдерживаемом состоянии и предотвращает управление доменом. Более подробную информацию об этих ресурсах можно узнать в [сетевых ресурсах, используемых DD Azure AD](network-considerations.md#network-resources-used-by-azure-ad-ds).

Это предупреждение генерируется при удалении одного из этих необходимых ресурсов. Если ресурс был удален менее 4 часов назад, есть вероятность, что платформа Azure может автоматически воссоздать удаленный ресурс. Ниже приведены следующие шаги, на котором излагается, как проверить состояние работоспособности и отметку времени для удаления ресурсов:

1. На портале Azure ищите и выберите **domain Services.** Выберите свой домена Azure AD DS, например *aaddscontoso.com.*
1. В левой навигации выберите **Здоровье**.
1. На странице работоспособности выберите оповещение с ID *AADDS109.*
1. Предупреждение имеет метку времени, когда он был впервые найден. Если эта метка времени была менее чем 4 часа назад, платформа Azure может автоматически воссоздать ресурс и решить оповещение самостоятельно.

    Если оповещению более 4 часов, управляемый dS Azure AD находится в невосстановимом состоянии. [Удалите управляемый домен Azure AD DS,](delete-aadds.md) а затем [создайте управляемый для замены домен.](tutorial-create-instance.md)

## <a name="aadds110-the-subnet-associated-with-your-managed-domain-is-full"></a>AADDS110. Подсеть, связанная с управляемым доменом, переполнена

### <a name="alert-message"></a>Сообщение оповещения

*Подсеть, выбранная для развертывания доменных служб Azure AD, переполнена и не обладает достаточным местом для контроллера домена, который необходимо создать.*

### <a name="resolution"></a>Решение

Виртуальная сетевая подсеть для Azure AD DS нуждается в достаточном количестве IP-адресов для автоматически созданных ресурсов. Это пространство IP-адреса включает в себя необходимость создания ресурсов замены, если есть событие обслуживания. Чтобы свести к минимуму риск иссякнуть из доступных IP-адресов, не развертывайте дополнительные ресурсы, такие как собственные виртуальные, в ту же виртуальную сетевую подсеть, что и Azure AD DS.

Эта ошибка является неисправимой. Чтобы разрешить оповещение, [удалите существующий домен Azure AD DS](delete-aadds.md) и воссоздайте его. Если у вас возникли проблемы с удалянием домена Azure AD DS, [откройте запрос службы поддержки Azure][azure-support] для дополнительной помощи в устранении неполадок.

## <a name="aadds111-service-principal-unauthorized"></a>AADDS111: Основной несанкционированный сервис

### <a name="alert-message"></a>Сообщение оповещения

*Принцип службы, который используют службы домена Azure AD для обслуживания домена, не уполномочен управлять ресурсами в подписке Azure. Директор службы должен получить разрешения на обслуживание управляемого домена.*

### <a name="resolution"></a>Решение

Некоторые автоматически генерируемые принципы служб используются для управления и создания ресурсов для домена Azure AD DS. Если разрешения доступа для одного из этих принципов службы изменены, домен не может правильно управлять ресурсами. Следующие шаги показывают, как понять, а затем предоставить разрешения на доступ к директору службы:

1. Узнайте о [элементах управления доступом на основе ролей и о том, как предоставить доступ к приложениям, на портале Azure.](https://docs.microsoft.com/azure/role-based-access-control/role-assignments-portal)
2. Просмотрите доступ, который имеет директор службы с ID *abba844e-bc0e-44b0-947a-dc74e5d09022,* и предоставите доступ, который был отказано ранее.

## <a name="aadds112-not-enough-ip-address-in-the-managed-domain"></a>AADDS112. Недостаточно IP-адресов в управляемом домене

### <a name="alert-message"></a>Сообщение оповещения

*Мы определили, что подсеть виртуальной сети в этом домене может не иметь достаточного количества IP-адресов. Службам домена Azure AD необходимо как минимум два доступных IP-адреса в подсети, в которых она включена. Мы рекомендуем иметь не менее 3-5 запасных IP-адресов в подсети. Это могло произойти, если другие виртуальные машины развернуты в подсети, тем самым исчерпав количество доступных IP-адресов или если есть ограничение на количество доступных IP-адресов в подсети.*

### <a name="resolution"></a>Решение

Виртуальная сетевая подсеть для Azure AD DS нуждается в достаточном количестве IP-адресов для автоматически созданных ресурсов. Это пространство IP-адреса включает в себя необходимость создания ресурсов замены, если есть событие обслуживания. Чтобы свести к минимуму риск иссякнуть из доступных IP-адресов, не развертывайте дополнительные ресурсы, такие как собственные виртуальные, в ту же виртуальную сетевую подсеть, что и Azure AD DS.

Чтобы разрешить это оповещение, удалите существующий домен Azure AD DS и воссоздайте его в виртуальной сети с достаточно большим диапазоном IP-адресов. Этот процесс является разрушительным, так как домен Azure AD DS недоступен, а все созданные пользовательские ресурсы, такие как OUs или учетные записи служб, потеряны.

1. [Удалите из каталога управляемый доменом Azure AD DS.](delete-aadds.md)
1. Чтобы обновить диапазон IP-адресов виртуальной сети, ищите и выберите *виртуальную сеть* на портале Azure. Выберите виртуальную сеть для Azure AD DS с небольшим диапазоном IP-адресов.
1. Под **настройками**выберите *адресное пространство.*
1. Обновите диапазон адресов, выбрав существующий диапазон адресов и отредактировав его или добавив дополнительный диапазон адресов. Убедитесь, что новый диапазон IP-адресов достаточно велик для подсетевого диапазона Azure AD DS. При готовности **сохраните** изменения.
1. Выберите **подсети** в левой навигации.
1. Выберите подсеть, которая вы хотите отсеить, или создайте дополнительную подсеть.
1. Обновление или указать достаточно большой диапазон IP-адресов, а затем **сохранить** изменения.
1. [Создайте заменяющий домен Azure AD DS.](tutorial-create-instance.md) Убедитесь, что вы выбираете обновленную виртуальную сетевую подсеть с достаточно большим диапазоном IP-адресов.

Здоровье домена Azure AD DS автоматически обновляется в течение двух часов и удаляет оповещение.

## <a name="aadds115-resources-are-locked"></a>AADDS115. Ресурсы заблокированы

### <a name="alert-message"></a>Сообщение оповещения

*Один или несколько сетевых ресурсов, используемых в управляемом домене, не могут быть использованы, так как целевая область была заблокирована.*

### <a name="resolution"></a>Решение

Блокировки ресурсов могут быть применены к ресурсам Azure для предотвращения изменений или удаления. Поскольку Azure AD DS является управляемой службой, платформе Azure необходима возможность внесения изменений конфигурации. Если блокировка ресурсов применяется на некоторых компонентах Azure AD DS, платформа Azure не может выполнять задачи управления.

Чтобы проверить блокировки ресурсов на компонентах Azure AD DS и удалить их, выполните следующие шаги:

1. Для каждого из компонентов сети Azure AD DS в группе ресурсов, таких как виртуальная сеть, сетевой интерфейс или общедоступный IP-адрес, проверьте журналы операций на портале Azure. Эти журналы операций должны указывать, почему операция не сбой и где блокировка ресурса применяется.
1. Выберите ресурс, где блокировка применяется, то под **замки**, выберите и удалите блокировку (ы).

## <a name="aadds116-resources-are-unusable"></a>AADDS116. Ресурсы недоступны для использования

### <a name="alert-message"></a>Сообщение оповещения

*Один или несколько сетевых ресурсов, используемых в управляемом домене, не могут быть использованы из-за ограничений политики.*

### <a name="resolution"></a>Решение

Политики применяются к ресурсам Azure и группам ресурсов, которые контролируют, какие действия конфигурации разрешены. Поскольку Azure AD DS является управляемой службой, платформе Azure необходима возможность внесения изменений конфигурации. Если политика применяется к некоторым компонентам Azure AD DS, платформа Azure может оказаться не в состоянии выполнять свои задачи управления.

Чтобы проверить прикладные политики на компонентах Azure AD DS и обновить их, выполните следующие шаги:

1. Для каждого из компонентов сети Azure AD DS в группе ресурсов, таких как виртуальная сеть, NIC или общедоступный IP-адрес, проверьте журналы операций на портале Azure. Эти журналы операций должны указывать, почему операция не сбой и где применяется ограничительная политика.
1. Выберите ресурс, в котором применяется политика, затем в соответствии с **политиками,** выберите и отображайте политику, чтобы она была менее ограничительной.

## <a name="aadds500-synchronization-has-not-completed-in-a-while"></a>AADDS500: синхронизация не выполнена спустя некоторое время

### <a name="alert-message"></a>Сообщение оповещения

*Последний раз управляемый домен был синхронизирован с Azure AD на «дате». Пользователи могут быть не в состоянии войти в управляемый домен или членство в группе не может быть синхронизировано с Azure AD.*

### <a name="resolution"></a>Решение

[Проверьте работоспособность Azure AD DS](check-health.md) на наличие оповещений, указывающих на проблемы в конфигурации управляемого домена. Проблемы с конфигурацией сети могут блокировать синхронизацию aD Azure. Если вы можете разрешить оповещения, указывающие на проблему конфигурации, подождите два часа и проверьте, успешно ли завершена синхронизация.

Следующие распространенные причины приводят к тому, что синхронизация останавливается в доменах, управляемых Azure AD DS:

* Требуемое подключение к сети заблокировано. Чтобы узнать больше о том, как проверить виртуальную сеть Azure на наличие проблем и того, что требуется, смотрите [группы безопасности сети](alert-nsg.md) и [требования к сети для служб доменов Azure AD.](network-considerations.md)
*  Синхронизация паролей не была настроена или успешно завершена при развертывании домена Azure AD DS. Можно настроить синхронизацию паролей только для пользователей из [облачных](tutorial-create-instance.md#enable-user-accounts-for-azure-ad-ds) или [гибридных пользователей с on-prem.](tutorial-configure-password-hash-sync.md)

## <a name="aadds503-suspension-due-to-disabled-subscription"></a>AADDS503: блокировка из-за неактивной подписки

### <a name="alert-message"></a>Сообщение оповещения

*Управляемый домен блокируется, так как подписка Azure, связанная с доменом, неактивна.*

### <a name="resolution"></a>Решение

> [!WARNING]
> Если управляемый dS AD Azure приостанавливается на длительный период времени, существует опасность его удаления. Устранить причину приостановки как можно быстрее. Для получения дополнительной информации [см. Поймите приостановленные состояния для Azure AD DS](suspension.md).

Azure AD DS требует активной подписки. Если подписка Azure, с которой был связан домен Azure AD DS, не активна, необходимо продлить ее для повторной активации подписки.

1. [Почему подписка Azure отключена и как активировать ее повторно?](https://docs.microsoft.com/azure/billing/billing-subscription-become-disable)
2. После обновления подписки уведомление Azure AD DS позволяет повторно включить управляемый домен.

При повторном включении управляемого домена работоспособность домена Azure AD DS автоматически обновляется в течение двух часов и удаляет оповещение.
