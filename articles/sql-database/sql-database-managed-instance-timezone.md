# <a name="time-zones-in-azure-sql-database-managed-instance"></a>Часовые пояса в управляемой базе данных Azure S'L

Скоординированное универсальное время (UTC) является рекомендуемым часовом поясом для уровня данных облачных решений. Управляемая группа данных Azure S'L также предлагает выбор часовых поясов для удовлетворения потребностей существующих приложений, которые хранят значения даты и времени, а также функции даты и времени вызова с неявным контекстом определенного часового пояса.

Функции T-S'L, такие как [GETDATE()](https://docs.microsoft.com/sql/t-sql/functions/getdate-transact-sql) или код CLR, наблюдают за часового поясом, установленным на уровне экземпляра. Задания серверного агента также следуют расписанию в соответствии с часовой поясом экземпляра.

  >[!NOTE]
  > Managed Instance — это единственный вариант развертывания базы данных Azure S'L, поддерживающий настройки часового пояса. Другие варианты развертывания всегда следуют UTC.
Используйте [ВРЕМЯ ЗОНЕ](https://docs.microsoft.com/sql/t-sql/queries/at-time-zone-transact-sql) в единичных и объединенных базах данных S'L, если вам необходимо интерпретировать информацию о дате и времени в часовом поясе, не входящих в UTC.

## <a name="supported-time-zones"></a>Поддерживаемые часовые пояса

Набор поддерживаемых часовых поясов передается по наследству от базовой операционной системы управляемого экземпляра. Он регулярно обновляется, чтобы получить новые определения часового пояса и отражать изменения в существующих.

[Летнее время/часовой пояс изменения политики](https://aka.ms/time) гарантирует историческую точность с 2010 года вперед.

Список с именами поддерживаемых часовых поясов разоблачается через представление [системы sys.time_zone_info.](https://docs.microsoft.com/sql/relational-databases/system-catalog-views/sys-time-zone-info-transact-sql)

## <a name="set-a-time-zone"></a>Установка часового пояса

Часовой пояс управляемого экземпляра может быть установлен только при создании экземпляра. Часовой пояс по умолчанию — UTC.

  >[!NOTE]
  > Часовой пояс существующего управляемого экземпляра не может быть изменен.

### <a name="set-the-time-zone-through-the-azure-portal"></a>Установите часовой пояс через портал Azure

При вводе параметров для нового экземпляра выберите часовой пояс из списка поддерживаемых часовых поясов.
  
![Установка часового пояса при создании экземпляра](media/sql-database-managed-instance-timezone/01-setting_timezone-during-instance-creation.png)

### <a name="azure-resource-manager-template"></a>Шаблон Azure Resource Manager

Укажите свойство timezoneId в [шаблоне менеджера ресурсов,](https://aka.ms/sql-mi-create-arm-posh) чтобы установить часовой пояс во время создания экземпляра.

```json
"properties": {
                "administratorLogin": "[parameters('user')]",
                "administratorLoginPassword": "[parameters('pwd')]",
                "subnetId": "[parameters('subnetId')]",
                "storageSizeInGB": 256,
                "vCores": 8,
                "licenseType": "LicenseIncluded",
                "hardwareFamily": "Gen5",
                "collation": "Serbian_Cyrillic_100_CS_AS",
                "timezoneId": "Central European Standard Time"
            },

```

Список поддерживаемых значений для свойства timezoneId находится в конце этой статьи.

Если не указано, часовой пояс устанавливается utC.

## <a name="check-the-time-zone-of-an-instance"></a>Проверьте часовой пояс экземпляра

Функция [CURRENT_TIMEZONE](https://docs.microsoft.com/sql/t-sql/functions/current-timezone-transact-sql) возвращает имя отображения часового пояса экземпляра.

## <a name="cross-feature-considerations"></a>Кросс-функциональные соображения

### <a name="restore-and-import"></a>Восстановление и импорт

Можно восстановить файл резервного копирования или импортировать данные в управляемый экземпляр из экземпляра или сервер с различными настройками часового пояса. Убедитесь в том, чтобы сделать это с осторожностью. Проанализируйте поведение приложения и результаты запросов и отчетов, точно так же, как при передаче данных между двумя экземплярами S'L Server с различными настройками часового пояса.

### <a name="point-in-time-restore"></a>Восстановление на момент времени

При выполнении момента восстановления время восстановления интерпретируется как время UTC. Таким образом, любую двусмысленность из-за летнего времени и его потенциальных изменений избежать.

### <a name="auto-failover-groups"></a>Группы автоматической отработки отказа

Использование одного и того же часового пояса в первичном и вторичном экземпляре в группе неудач не обеспечивается, но мы настоятельно рекомендуем его.

  >[!WARNING]
  > Мы настоятельно рекомендуем использовать один и тот же часовой пояс для первичного и вторичного экземпляра в группе неудач. Из-за некоторых случаев редкого использования один и тот же часовой пояс в первичных и вторичных экземплярах не соблюдается. Важно понимать, что в случае ручного или автоматического сбоя, вторичный экземпляр сохранит свой первоначальный часовой пояс.

## <a name="limitations"></a>Ограничения

- Часовой пояс существующего управляемого экземпляра не может быть изменен.
- Внешние процессы, запущенные с заданий серверного агента S'L, не соблюдают часовой пояс экземпляра.
