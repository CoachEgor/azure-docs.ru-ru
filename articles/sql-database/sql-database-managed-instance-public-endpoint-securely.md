# <a name="use-an-azure-sql-database-managed-instance-securely-with-public-endpoints"></a>Безопасное использование управляемого экземпляра базы данных Azure S'L с общедоступными конечными точками

Управляемые экземпляры базы данных Azure S'L могут обеспечить подключение пользователей к [общедоступным конечным точкам.](../virtual-network/virtual-network-service-endpoints-overview.md) В этой статье объясняется, как сделать эту конфигурацию более безопасной.

## <a name="scenarios"></a>Сценарии

Управляемый экземпляр базы данных S'L обеспечивает частную конечную точку, позволяющую подключение внутри своей виртуальной сети. Опция по умолчанию заключается в обеспечении максимальной изоляции. Однако есть сценарии, в которых необходимо предоставить общедоступное соединение конечных точек:

- Управляемый экземпляр должен интегрироваться с предложениями платформы только для нескольких арендаторов (PaaS).
- Вам нужна более высокая пропускная информация, чем это возможно при использовании VPN.
- Политика компании запрещает PaaS внутри корпоративных сетей.

## <a name="deploy-a-managed-instance-for-public-endpoint-access"></a>Развертывание управляемого экземпляра для общедоступного доступа к конечным точкам

Хотя это и не является обязательным, общая модель развертывания для управляемого экземпляра с общедоступным конечным доступом заключается в создании экземпляра в выделенной изолированной виртуальной сети. В этой конфигурации виртуальная сеть используется только для виртуальной изоляции кластеров. Не имеет значения, перекрываетли ли пространство IP-адреса управляемого экземпляра пространство IP-адреса с пространством IP-адреса корпоративной сети.

## <a name="secure-data-in-motion"></a>Безопасные данные в движении

Трафик данных управляемого экземпляра всегда шифруется, если драйвер клиента поддерживает шифрование. Данные, отправляемые между управляемым экземпляром и другими виртуальными машинами Azure или службами Azure, никогда не покидают костяк Azure. При наличии связи между управляемым экземпляром и предварительной сетью мы рекомендуем вам использовать Azure ExpressRoute с помощью пиринга Майкрософт. ExpressRoute поможет вам избежать перемещения данных через общедоступный Интернет. Для управляемого примера частного подключения можно использовать только приватное пиринг.

## <a name="lock-down-inbound-and-outbound-connectivity"></a>Блокировка входящих и исходящих подключений

На следующей диаграмме показаны рекомендуемые конфигурации безопасности:

![Конфигурации безопасности для блокировки входящих и исходящих подключений](media/sql-database-managed-instance-public-endpoint-securely/managed-instance-vnet.png)

Управляемый экземпляр имеет [специальный общедоступный адрес конечных точек.](sql-database-managed-instance-find-management-endpoint-ip-address.md) В исходящих брандмауэрах на стороне клиента и в правилах группы безопасности сети установите этот общедоступный IP-адрес для ограничения исходящих подключений.

Для обеспечения трафика в управляемый экземпляр осуществляется из надежных источников, мы рекомендуем подключаться из источников с известными IP-адресами. Используйте группу сетевой безопасности для ограничения доступа к управляемому экземпляру общедоступной точки на порту 3342.

Когда клиентам необходимо инициировать подключение из предварительной сети, убедитесь, что исходный адрес переведен на известный набор IP-адресов. Если вы не можете сделать это (например, мобильная рабочая сила является типичным сценарием), мы рекомендуем вам использовать [VPN-соединения от точки к сайту и частную конечную точку.](sql-database-managed-instance-configure-p2s.md)

Если соединения запускаются с Azure, мы рекомендуем трафик исходить от хорошо известного [виртуального IP-адреса](../virtual-network/virtual-networks-reserved-public-ip.md) (например, виртуальной машины). Чтобы упростить управление виртуальными IP-адресами (VIP) — можно использовать [общедоступные префиксы IP-адресов.](../virtual-network/public-ip-address-prefix.md)
