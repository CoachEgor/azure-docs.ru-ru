# <a name="hyperscale-service-tier"></a>Уровень служб "Гипермасштабирование"

База данных SQL Azure основана на архитектуре ядра СУБД SQL Server, которая соответствует облачной среде, чтобы даже в случае сбоя инфраструктуры обеспечить доступность на уровне 99,99 %. Существует три модели архитектуры, которые используются в Базе данных SQL Azure.
- уровни "Общего назначения" или "Стандартный"; 
-  Уровень "Гипермасштабирование"
-  уровни "Критически важный для бизнеса" или "Премиум";

Уровень служб "Гипермасштабирование" в Базе данных SQL Azure — это новейший уровень в рамках модели приобретения на основе виртуальных ядер. Этот высокомасштабируемый уровень производительности ресурсов хранения и вычислений, который использует архитектуру Azure для масштабирования этих ресурсов для службы "База данных SQL Azure" за пределы, доступные для уровней служб "Общего назначения" и "Критически важный для бизнеса".

> 
> [!NOTE]
> См. подробнее об уровнях служб [Общего назначения](sql-database-service-tier-general-purpose.md) и [Критически важный для бизнеса](sql-database-service-tier-business-critical.md) в рамках модели приобретения на основе виртуальных ядер. Сравнение модели приобретения на основе виртуальных ядер и DTU см. в статье [Ресурсы и модели приобретения для Базы данных SQL Azure](sql-database-service-tiers.md).


## <a name="what-are-the-hyperscale-capabilities"></a>Каковы возможности ценовой категории "Гипермасштабирование"?

Уровень служб "Гипермасштабирование" в службе "База данных SQL Azure" предоставляет следующие дополнительные возможности:

- Поддержка размера базы данных до 100 ТБ.
- Почти мгновенное резервное копирование базы данных (на основе снимков файлов, хранящихся в хранилище Azure Blob) независимо от размера без влияния IO на ресурсы вычислений  
- Быстрое восстановление баз данных (на основе файлов моментальных снимков) за несколько минут, а не часов или дней (вне зависимости от размера операции с данными).
- Более высокая общая производительность благодаря высокой пропускной способности ведения журнала и ускоренному выполнению транзакций независимо от объемов данных.
- Быстрое горизонтальное масштабирование. Вы можете подготовить один или несколько узлов только для чтения, чтобы переносить на них рабочие нагрузки чтения и использовать эти узлы в качестве узлов горячей замены.
- Быстрое увеличение масштаба. Вы можете постоянно увеличивать масштаб вычислительных ресурсов, чтобы размещать крупные рабочие нагрузки по мере необходимости, а затем уменьшать их масштаб, когда они не нужны.

Уровень служб "Гипермасштабирование" устраняет многие практические ограничения, традиционно наблюдаемые в облачных базах данных. В то время как большинство других баз данных ограничены ресурсами, доступными на одном узле, базы данных на уровне служб "Гипермасштабирование" не имеют таких ограничений. Благодаря гибкой архитектуре хранилища его объем растет по мере необходимости. На самом деле базы данных уровня "Гипермасштабирование" не создаются с определенным максимальным размером. База данных уровня "Гипермасштабирование" растет по мере необходимости, поэтому плата взимается только за ту емкость, которую вы используете. Для рабочих нагрузок интенсивного чтения уровень служб "Гипермасштабирование" обеспечивает быстрое горизонтальное масштабирование путем подготовки дополнительных реплик чтения при необходимости перенести рабочие нагрузки.

Кроме того, время, необходимое для создания резервных копий базы данных или увеличения/уменьшения масштаба, больше не привязано к объему данных в базе данных. Резервные копии баз данных уровня "Гипермасштабирование" могут создаваться практически мгновенно. Увеличивать или уменьшать масштаб баз данных с десятками терабайт данных можно за считаные минуты. Вы больше не ограничены первоначальными вариантами конфигурации.

Дополнительные сведения о размерах вычислительных ресурсов для уровня служб "Гипермасштабирование" см. в разделе [Характеристики уровней служб](sql-database-service-tiers-vcore.md#service-tiers).

## <a name="who-should-consider-the-hyperscale-service-tier"></a>Кому рекомендуется использовать уровень служб "Гипермасштабирование"

Уровень обслуживания Hyperscale предназначен для большинства бизнес-нагрузок, поскольку обеспечивает большую гибкость и высокую производительность с независимыми масштабируемыми вычислительными и хранилищами. С возможностью автоматического масштабирования хранения до 100 ТБ, это отличный выбор для клиентов, которые:

- Иметь большие базы данных на месте и хотите модернизировать свои приложения, перейдя в облако
- Уже находятся в облаке и ограничены максимальными ограничениями размера базы данных других уровней обслуживания (1-4 ТБ)
- Иметь меньшие базы данных, но требуется быстрое вертикальное и горизонтальное масштабирование вычислений, высокая производительность, мгновенное резервное копирование и быстрое восстановление базы данных.

Уровень обслуживания Hyperscale поддерживает широкий спектр рабочих нагрузок s'L Server, от чистого OLTP до чистой аналитики, но в первую очередь оптимизирован для OLTP и гибридных транзакций и аналитической обработки (HTAP).

> [!IMPORTANT]
> Эластичные пулы не поддерживают уровень служб "Гипермасштабирование".

## <a name="hyperscale-pricing-model"></a>Модель ценообразования для ценовой категории "Гипермасштабирование"

Уровень служб "Гипермасштабирование" доступен только в [модели на основе виртуальных ядер](sql-database-service-tiers-vcore.md). Ввиду новой архитектуры модель ценообразования несколько отличается от уровней служб "Общего назначения" или "Критически важный для бизнеса".

- **Вычислить**:

  Цена за единицу вычислений ценовой категории "Гипермасштабирование" на реплику. Цена [Преимущества гибридного использования Azure](https://azure.microsoft.com/pricing/hybrid-benefit/) автоматически применяется к репликам с масштабированием для чтения. Мы создаем первичную реплику и одну реплику только для чтения в базе данных Hyperscale по умолчанию.  Пользователи могут настроить общее количество реплик, включая первичные от 1-5.

- **Хранилище**.

  Указывать максимальный объем данных при настройке базы данных ценовой категории "Гипермасштабирование" не нужно. В гиперуровне взимается плата за хранение для базы данных на основе фактического распределения. Хранение автоматически выделяется от 40 ГБ до 100 ТБ, с шагом в 10 ГБ 10 ГБ. Несколько файлов данных могут расти одновременно, если это необходимо. База данных Hyperscale создается со стартовым размером 10 ГБ и начинает расти на 10 ГБ каждые 10 минут, пока не достигнет размера 40 ГБ.

Дополнительные сведения о ценах категории "Гипермасштабирование" см. в разделе [цен на Базу данных SQL Azure](https://azure.microsoft.com/pricing/details/sql-database/single/).

## <a name="distributed-functions-architecture"></a>Архитектура распределенных функций

В отличие от традиционных ядер СУБД, в которых все функции управления данными централизованы в одном месте или процессе (даже в так называемых распределенных базах данных есть несколько копий монолитного механизма обработки данных), база данных уровня "Гипермасштабирование" отделяет механизм обработки запросов, где семантики различных модулей обработки данных расходятся, от компонентов, которые обеспечивают долговременное хранение и устойчивость данных. Таким образом, емкость хранилища может быть плавно масштабирована по мере необходимости (исходная цель составляет 100 ТБ). Только для чтения реплики имеют одни и те же компоненты хранения, поэтому никакая копия данных не требуется для создания новой читаемой реплики. 

На следующей схеме представлены различные типы узлов в базе данных уровня "Гипермасштабирование".

![архитектура](./media/sql-database-hyperscale/hyperscale-architecture.png)

База данных Hyperscale содержит следующие типы компонентов:

### <a name="compute"></a>Службы вычислений

На вычислительном узле расположен реляционный модуль, поэтому там находятся все элементы языка, операции обработки запросов и прочее. Все взаимодействия пользователя с базой данных уровня "Гипермасштабирование" происходят через эти вычислительные узлы. На вычислительных узлах выполняется кэширование на основе дисков SSD (именуемое на предыдущей схеме как RBPEX — отказоустойчивое расширение пула), чтобы минимизировать количество круговых путей в сети, необходимых для получения страницы данных. Существует один первичный вычислительный узел, где обрабатываются все транзакции и рабочие нагрузки чтения и записи. При этом есть один или несколько вторичных вычислительных узлов, которые выступают в качестве резервных узлов на случай отработки отказа, а также действуют как вычислительные узлы только для чтения для переноса рабочих нагрузок чтения (если эта функция необходима).

### <a name="page-server"></a>Сервер страницы

Серверы страниц — это системы, представляющие горизонтально масштабируемую подсистему хранилища.  Каждый сервер страниц отвечает за подмножество страниц в базе данных.  Номинально каждый сервер страницы контролирует от 128 ГБ до 1 ТБ данных. Данные совместно размещаются только на одном сервере страниц (за пределами реплик, созданных для обеспечения избыточности и доступности). Задача сервера страниц заключается в том, чтобы предоставлять страницы базы данных для вычислительных узлов по требованию и обновлять страницы, когда транзакции обновляют данные. Актуальность данных на серверах страниц обеспечивается за счет воспроизведения записей журнала из службы журналов. Серверы страниц также поддерживают кэширование на основе диска SSD, чтобы повысить производительность. Для обеспечения дополнительной надежности за долгосрочное хранение страниц данных отвечает служба хранилища Azure.

### <a name="log-service"></a>Служба журнала

Служба журнала принимает записи журналов из первичной реплики вычислений, сохраняет их в прочном кэше и направляет записи журнала на остальные реплики вычислений (чтобы они могли обновлять свои кэши), а также соответствующий сервер страницы (ы), так что данные могут быть обновлены там. Таким образом, все изменения данных из первичной реплики вычислений распространяются через службу журнала на все вторичные вычислительные реплики и серверы страниц. Наконец, записи журналов выталкиваются на долгосрочное хранение в Хранилище Azure, которое является практически бесконечным хранилищем хранилища. Этот механизм устраняет необходимость частого усечения журналов. Служба журнала также имеет локальный кэш для ускорения доступа к записям журналов.

### <a name="azure-storage"></a>Хранилище Azure

Azure Storage содержит все файлы данных в базе данных. Серверы страниц обновляют файлы данных в Azure Storage. Это хранилище используется для целей резервного копирования, а также для репликации между регионами Azure. Резервное копирование осуществляется с помощью моментальных снимков данных. Восстановление операций с помощью моментальных снимков быстро, независимо от размера данных. Данные могут быть восстановлены в любой момент времени в период хранения резервного копирования базы данных.

## <a name="backup-and-restore"></a>Резервное копирование и восстановление

Резервное копирование на основе файла и, следовательно, они почти мгновенно. Разделение хранения и вычислений позволяет снизить работу резервного копирования/восстановления на уровень хранения, чтобы уменьшить нагрузку на обработку первичной вычислительной реплики. В результате резервное копирование базы данных не влияет на производительность первичного вычислительного узла; аналогичным образом, восстановление выполняется путем возврата к файловым снимкам, и как таковые не являются размером операции данных. Восстановление — это постоянная операция, и даже многотерабайтные базы данных могут быть восстановлены в течение нескольких минут, а не часов или дней. Создание новых баз данных путем восстановления существующей резервной копии также использует эту функцию: создание копий баз данных для целей разработки или тестирования, даже терабайтных баз данных, вырабатывается в считанные минуты.

## <a name="scale-and-performance-advantages"></a>Преимущества масштабирования и производительности

Благодаря возможности быстро развертывать и свертывать дополнительные вычислительные узлы, предназначенные только для чтения, архитектура уровня "Гипермасштабирование" предоставляет значительные возможности масштабирования для чтения и также может освобождать первичный вычислительный узел для обслуживания большего количества запросов на запись. Кроме того, вычислительные узлы можно быстро увеличивать или уменьшать в масштабе благодаря архитектуре уровня "Гипермасштабирование" с общим хранилищем.

## <a name="create-a-hyperscale-database"></a>Создание базы данных Hyperscale

База данных Hyperscale может быть создана с помощью [портала Azure,](https://portal.azure.com) [T-S'L,](https://docs.microsoft.com/sql/t-sql/statements/create-database-transact-sql?view=azuresqldb-current) [PowerShell](https://docs.microsoft.com/powershell/module/azurerm.sql/new-azurermsqldatabase) или [CLI.](https://docs.microsoft.com/cli/azure/sql/db#az-sql-db-create) Гипермасштабные базы данных доступны только с помощью [модели покупки на основе vCore.](sql-database-service-tiers-vcore.md)

Следующая команда T-SQL создает базу данных ценовой категории "Гипермасштабирование". Необходимо указать целевые выпуск и службу в инструкции `CREATE DATABASE`. Обратитесь к [лимитам ресурсов](https://docs.microsoft.com/azure/sql-database/sql-database-vcore-resource-limits-single-databases#hyperscale---provisioned-compute---gen4) для списка действительных целей службы.

```sql
-- Create a Hyperscale Database
CREATE DATABASE [HyperscaleDB1] (EDITION = 'Hyperscale', SERVICE_OBJECTIVE = 'HS_Gen5_4');
GO
```
Это позволит создать базу данных Hyperscale на оборудовании Gen5 с 4 ядрами.

## <a name="migrate-an-existing-azure-sql-database-to-the-hyperscale-service-tier"></a>Перенос имеющейся базы данных Azure SQL на уровень служб ценовой категории "Гипермасштабирование"

Вы можете переместить существующие базы данных Azure S'L в Hyperscale с помощью [портала Azure,](https://portal.azure.com) [T-S'L,](https://docs.microsoft.com/sql/t-sql/statements/alter-database-transact-sql?view=azuresqldb-current) [PowerShell](https://docs.microsoft.com/powershell/module/azurerm.sql/set-azurermsqldatabase) или [CLI.](https://docs.microsoft.com/cli/azure/sql/db#az-sql-db-update) В настоящее время это одностороннее переселение. Вы не можете переместить базы данных из Hyperscale в другой уровень обслуживания, кроме как путем экспорта и импорта данных. Для подтверждения концепции (POCs) мы рекомендуем сделать копию ваших производственных баз данных и перейти копию в Hyperscale. Миграция существующей базы данных Azure S'L на уровень Hyperscale — это размер операции данных.

Следующие команды T-SQL перемещает базу данных на уровень служб ценовой категории "Гипермасштабирование". Необходимо указать целевые выпуск и службу в инструкции `ALTER DATABASE`.

```sql
-- Alter a database to make it a Hyperscale Database
ALTER DATABASE [DB2] MODIFY (EDITION = 'Hyperscale', SERVICE_OBJECTIVE = 'HS_Gen5_4');
GO
```

## <a name="connect-to-a-read-scale-replica-of-a-hyperscale-database"></a>Подключение к реплике базы данных ценовой категории "Гипермасштабирование", доступной только для чтения

В базах данных `ApplicationIntent` Hyperscale аргумент в строке соединения, предоставляемой клиентом, определяет, направляется ли соединение к реплике записи или к вторичной реплике только для чтения. Если параметру `ApplicationIntent` присвоено значение `READONLY` и у базы данных нет вторичной реплике, подключение будет направляться к первичной реплике в режиме по умолчанию `ReadWrite`.

```cmd
-- Connection string with application intent
Server=tcp:<myserver>.database.windows.net;Database=<mydatabase>;ApplicationIntent=ReadOnly;User ID=<myLogin>;Password=<myPassword>;Trusted_Connection=False; Encrypt=True;
```

Гипермасштабные вторичные реплики идентичны, используя ту же цель уровня обслуживания, что и основная реплика. При наличии более одной вторичной реплики рабочая нагрузка распределяется по всем имеющимся второстепенным вариантам. Каждая второстепенная реплика обновляется независимо, таким образом, различные реплики могут иметь различную задержку данных по отношению к основной реплике.

## <a name="database-high-availability-in-hyperscale"></a>Высокая доступность базы данных в гипермасштабе

Как и во всех других уровнях обслуживания, Hyperscale гарантирует долговечность данных для совершенных транзакций, независимо от наличия вычислимых реплик. Степень простоя из-за того, что основная реплика становится недоступной, зависит от типа сбоя (запланированного против незапланированного) и наличия хотя бы одной вторичной реплики. При запланированном сбою (т.е. событии обслуживания) система либо создает новую первичную реплику перед началом сбоя, либо использует существующую вторичную реплику в качестве цели неудачи. При незапланированном сбое (т.е. аппаратном сбое на основной реплике) система использует вторичную реплику в качестве цели неудачи, если она существует, или создает новую первичную реплику из пула доступных вычислительных мощностей. В последнем случае продолжительность простоя больше из-за дополнительных шагов, необходимых для создания новой основной реплики.

Для Hyperscale SLA [см.](https://azure.microsoft.com/support/legal/sla/sql-database/)

## <a name="disaster-recovery-for-hyperscale-databases"></a>Аварийное восстановление для гипермасштабных баз данных

### <a name="restoring-a-hyperscale-database-to-a-different-geography"></a>Восстановление базы данных Hyperscale в другой географии
Если вам необходимо восстановить Hyperscale DB базы данных Azure s'L в регион, за исключением той, в которой она в настоящее время размещается, в рамках операции аварийного восстановления или сверла, перемещения или любой другой причины, основной метод заключается в геовосстановлении базы данных.  Это включает в себя точно такие же шаги, что и то, что вы могли бы использовать для восстановления любого другого DB АЗУРЕ СЗЛ в другой регион:
1. Создайте сервер базы данных S'L в целевом регионе, если у вас еще нет соответствующего сервера.  Этот сервер должен принадлежать той же подписке, что и исходный (источник) сервер.
2. Следуйте инструкциям в теме [геовосстановления](https://docs.microsoft.com/azure/sql-database/sql-database-recovery-using-backups#geo-restore) страницы по восстановлению баз данных Azure S'L из автоматических резервных копирований.

> [!NOTE]
> Поскольку источник и цель находятся в отдельных регионах, база данных не может обмениваться моментальным хранилищем с исходной базой данных, как в негеовосстановлениях, которые выполняются очень быстро. В случае геовосстановления базы данных Hyperscale это будет операция размера данных, даже если цель находится в парной области геореплицированного хранилища.  Это означает, что для геовосстановления потребуется время, пропорциональное размеру восстанавливаемых баз данных.  Если цель находится в парном регионе, копия будет находиться в пределах области, которая будет значительно быстрее, чем копия кросс-региона, но она все равно будет операцией размера данных.

## <a name="known-limitations"></a>Известные ограничения

Это текущие ограничения для уровня обслуживания Hyperscale по состоянию на ГА.  Мы активно работаем над устранением как можно большего числа этих ограничений.

| Проблема | Описание |
| :---- | :--------- |
| Панель резервного копирования управления для логического сервера не показывает, что базы данных Hyperscale будут отфильтро:  | Hyperscale имеет отдельный метод управления резервными копированиями, и как таковой настройки сохранения долгосрочного хранения и точки во времени не применяются / не являются недействительными. Следовательно, базы данных ценовой категории "Гипермасштабирование" не отображаются в области "Управление резервными копиями". |
| Восстановление на момент времени | После того, как база данных будет перенесена в уровень службы Hyperscale, восстановление в точке времени до миграции не поддерживается.|
| Восстановление негипермасштабного DB в Гипермасштабии и наоборот | Вы не можете восстановить базу данных Hyperscale в базу данных, не отношушься к гипермасштабу, и не можете восстановить базу данных, не относясь к гипермасштабу, в базу данных Hyperscale.|
| Если база данных имеет один или несколько файлов данных больше, чем 1 ТБ, миграция завершается неудачей | В некоторых случаях можно обойти эту проблему, сократив число больших файлов до менее 1 ТБ. При миграции базы данных, используемой в процессе миграции, убедитесь, что ни один файл не становится больше, чем 1 ТБ. Используйте следующий запрос для определения размера файлов базы данных. `SELECT *, name AS file_name, size * 8. / 1024 / 1024 AS file_size_GB FROM sys.database_files WHERE type_desc = 'ROWS'`;|
| Базы данных SQL | Управляемая инстанция по базам данных Azure S'L в настоящее время не поддерживается базами данных Hyperscale. |
| Эластичные пулы |  Упругие пулы в настоящее время не поддерживаются с помощью гипермасштаба базы данных S'L.|
| Сейчас переход на ценовую категорию "Гипермасштабирование" является необратимой операцией. | После переноса базы данных на уровень служб ценовой категории "Гипермасштабирование" ее невозможно перенести непосредственно на другой уровень служб. В настоящее время единственным способом переноса базы данных из Hyperscale в негипермасштаб является экспорт/импорт с помощью файла BACPAC или других технологий передачи данных (Bulk Copy, Azure Data Factory, Azure Databricks, SSIS и т.д.)|
| Миграция баз данных с постоянными объектами памяти | Hyperscale поддерживает только непостоянные объекты In-Memory (типы таблиц, родные SPs и функции).  Стойкие таблицы in-Memory и другие объекты должны быть удалены и воссозданы в качестве объектов, не входящих в память, прежде чем перейти базу данных на уровень службы Hyperscale.|
| Отслеживание изменений | Отслеживание изменений в настоящее время находится в открытом доступе и может быть включено в новые или существующие базы данных Hyperscale. |
| Репликация гео  | Вы пока не можете настроить георепликацию для гипермасштаба базы данных Azure S'L. |
| Копирование базы данных | Вы пока не можете использовать копию базы данных для создания новой базы данных в гипермасштабе Azure S'L. |
| Интеграция TDE/AKV | Прозрачное шифрование базы данных с помощью Azure Key Vault (обычно называемое Bring-Your-Own-Key или BYOK) еще не поддерживается для hyperscale базы данных Azure S'L, однако TDE с управляемыми ключами службы полностью поддерживается. |
|Интеллектуальные функции базы данных | За исключением опции "Force Plan", все остальные параметры автоматической настройки еще не поддерживаются на Hyperscale: параметры могут казаться включенными, но не будет никаких рекомендаций или действий. |
|Сведения о производительности запросов | Исследования производительности запросов в настоящее время не поддерживаются для баз данных Hyperscale. |
| Сжатие базы данных | DBCC SHRINKDATABASE или DBCC SHRINKFILE в настоящее время не поддерживаются для баз данных Hyperscale. |
| Проверка целостности базы данных | DBCC CHECKDB в настоящее время не поддерживается для гипермасштабных баз данных. Подробную информацию об управлении целостностью данных можно узнать [в](https://azure.microsoft.com/blog/data-integrity-in-azure-sql-database/) базе данных Azure S'L. |
