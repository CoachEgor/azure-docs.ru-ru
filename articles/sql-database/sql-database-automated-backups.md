# <a name="automated-backups"></a>Автоматическое резервное копирование

База данных Azure S'L автоматически создает резервные данные базы данных, которые хранятся в течение периода настройки. Он использует [геоизмерное хранилище считываемым доступом Azure (RA-GRS)](../storage/common/storage-redundancy.md) для обеспечения сохранения резервных копий, даже если центр обработки данных недоступен.

Резервные копии базы данных являются важной частью любой стратегии непрерывности бизнес-процессов и аварийного восстановления, так как они защищают данные от случайного повреждения или удаления. Если правила безопасности требуют, чтобы резервные копирования были доступны в течение длительного времени (до 10 лет), можно настроить [долгосрочное удержание](sql-database-long-term-retention.md) на однотонных базах данных и эластичных пулах баз данных.

## <a name="what-is-a-sql-database-backup"></a>Что такое резервная копия Базы данных SQL?

База данных S'L использует технологию S'L Server для создания [полных резервных копий](https://docs.microsoft.com/sql/relational-databases/backup-restore/full-database-backups-sql-server) каждую неделю, [дифференциальных резервных копий](https://docs.microsoft.com/sql/relational-databases/backup-restore/differential-backups-sql-server) каждые 12 часов, а [также резервного копирования журналов транзакций](https://docs.microsoft.com/sql/relational-databases/backup-restore/transaction-log-backups-sql-server) каждые 5-10 минут. Резервные копии хранятся в [каплях хранения данных RA-GRS,](../storage/common/storage-redundancy.md) которые реплицируются в [парный центр обработки данных](../best-practices-availability-paired-regions.md) для защиты от сбоа в центре обработки данных. При восстановлении базы данных служба определяет, какие полные, дифференциальные и транзакционные резервные копии регистра должны быть восстановлены.

Эти резервные копии позволяют выполнить следующие операции:

- **Восстановить существующую базу данных в точку времени в период** хранения, используя портал Azure, Azure PowerShell, Azure CLI или REST API. Для отдельных баз данных и эластичных пулов баз данных эта операция создаст новую базу данных на том же сервере, что и исходная база данных. В управляемом экземпляре эта операция может создать копию базы данных или тот же или другой управляемый экземпляр под одной подпиской.
- **Восстановление удаленной базы данных к моменту удаления** или в любое время в период удержания. Удаленная база данных может быть восстановлена только на том же логическом сервере или управляемом экземпляре, где была создана исходная база данных.
- **Восстановление базы данных в другой географический регион.** Геовосстановление позволяет восстановиться после географической катастрофы, когда вы не можете получить доступ к серверу и базе данных. Он создает новую базу данных на любом существующем сервере, в любой точке мира.
- **Восстановление базы данных из определенного долгосрочного резервного копирования** в одной базе данных или эластичной пуле баз данных, если база данных настроена с долгосрочной политикой удержания (LTR). LTR позволяет восстановить старую версию базы данных с помощью [портала Azure](sql-database-long-term-backup-retention-configure.md#using-azure-portal) или [Azure PowerShell](sql-database-long-term-backup-retention-configure.md#using-powershell) для удовлетворения запроса на соответствие требованиям или запуска старой версии приложения. Для получения дополнительной информации [см.](sql-database-long-term-retention.md)

Для выполнения восстановления смотрите [базу данных Restore из резервных ups.](sql-database-recovery-using-backups.md)

## <a name="backup-frequency"></a>Частота резервного копирования

### <a name="point-in-time-restore"></a>Восстановление на момент времени

База данных S'L поддерживает самообслуживание для восстановления точек времени (PITR) путем автоматического создания полных резервных копий, дифференциальных резервных копий и резервных копий журналов транзакций. Полное резервное копирование базы данных создается еженедельно, и резервные данные с дифференциалами обычно создаются каждые 12 часов. Резервное копирование журналов транзакций обычно создается каждые 5-10 минут. Частота резервных копий журналов транзакций зависит от размера вычисления и объема активности базы данных. 

Первое полное резервное копирование планируется сразу после создания базы данных. Эта резервная работа обычно завершается в течение 30 минут, но это может занять больше времени, когда база данных большая. Например, начальное резервное копирование будет выполняться дольше для восстановленной базы данных или копии базы данных. После первого полного резервного копирования все последующие операции резервного копирования планируются автоматически и управляются без участия пользователя в фоновом режиме. Точное время создания всех копий базы данных определяет служба Базы данных SQL с учетом распределения общей рабочей нагрузки в системе. Вы не можете изменить или отключить задания резервного копирования.

Резервное копирование PITR защищено георезервным хранилищем. Дополнительные сведения см. в статье [Репликация службы хранилища Azure](../storage/common/storage-redundancy.md).

Для получения дополнительной информации о PITR, [см.](sql-database-recovery-using-backups.md#point-in-time-restore)

### <a name="long-term-retention"></a>Длительный период удержания

Для единичных и объединенных баз данных можно настроить долгосрочное хранение (LTR) полных резервных копий на срок до 10 лет в хранилище Azure Blob. Если вы включаете политику LTR, еженедельные полные резервные копии автоматически копируются в другой контейнер хранения RA-GRS. Чтобы соответствовать различным требованиям соответствия, можно выбрать различные периоды хранения для еженедельного, ежемесячного и/или ежегодного резервного копирования. Потребление хранилища зависит от выбранной частоты резервных копий и периода или периодов хранения. Для оценки стоимости хранилища LTR можно использовать [калькулятор ценообразования LTR.](https://azure.microsoft.com/pricing/calculator/?service=sql-database)

Как и резервные копии PITR, резервные копии LTR защищены георезервным хранилищем. Дополнительные сведения см. в статье [Репликация службы хранилища Azure](../storage/common/storage-redundancy.md).

Для получения дополнительной информации о LTR [см.](sql-database-long-term-retention.md)

## <a name="backup-storage-consumption"></a>Потребление резервного хранилища

Для отдельных баз данных это уравнение используется для расчета общего использования резервного хранилища:

`Total backup storage size = (size of full backups + size of differential backups + size of log backups) – database size`

Для эластичных пулов баз данных общий размер хранилища резервного копирования агрегируется на уровне пула и рассчитывается следующим образом:

`Total backup storage size = (total size of all full backups + total size of all differential backups + total size of all log backups) - allocated pool data storage`

Резервное копирование, возникает до периода хранения, автоматически очищается в зависимости от их метки времени. Поскольку резервные копии дифференциалов и резервных копий журналов требуют более раннего полного резервного копирования, чтобы быть полезными, они удаляются вместе еженедельными кусками.

База данных Azure S'L вычисляет общее хранилище резервного копирования в запасе в качестве кумулятивного значения. Каждый час это значение сообщается конвейеру выставления счетов Azure, который отвечает за агрегирование этого почасового использования для расчета потребления в конце каждого месяца. После того, как база данных отбрасывается, потребление уменьшается по мере старения резервных служб. После того, как резервные ожидания становятся старше периода хранения, выставление счетов останавливается.

   > [!IMPORTANT]
   > Резервное копирование базы данных сохраняется в течение указанного периода хранения, даже если база данных была удалена. Хотя удаление и повторное создание базы данных часто может сэкономить на расходах на хранение и вычисление, это может увеличить затраты на резервное копирование, потому что Microsoft сохраняет резервную копию для указанного периода удержания (который составляет минимум 7 дней) для каждой упавшей базы данных, каждой время он упал.

### <a name="fine-tune-backup-storage-consumption"></a>Потребление резервного копирования тонкой настройки

Избыточное потребление резервного хранилища будет зависеть от рабочей нагрузки и размера отдельных баз данных. Рассмотрим некоторые из следующих методов настройки для снижения потребления резервного хранилища:

* Сократите [период хранения резервного копирования](#change-the-pitr-backup-retention-period-by-using-the-azure-portal) до минимально возможного для ваших нужд.
* Избегайте выполнения больших операций записи, таких как восстановление индекса, чаще, чем вам нужно.
* Для больших операций загрузки данных рассмотрите возможность использования [кластерных индексов столбцов,](https://docs.microsoft.com/sql/database-engine/using-clustered-columnstore-indexes)уменьшите число некластерных индексов и рассмотрите операции навалочных нагрузок с числом строк около 1 миллиона.
* На уровне общего назначения сервисное хранилище данных обходится дешевле, чем цена избыточного резервного хранилища. Если у вас постоянно высокие избыточные затраты на хранение резервного копирования, можно увеличить объем хранения данных, чтобы сэкономить на резервном хранении.
* Используйте TempDB вместо постоянных таблиц в логике ETL для хранения временных результатов. (Применяется только к управляемому экземпляру.)
* Рассмотрите возможность отключить шифрование TDE для баз данных, не содержащих конфиденциальных данных (например, базы данных разработки или тестирования). Резервное копирование для незашифрованных баз данных обычно сжимается с более высоким коэффициентом сжатия.

> [!IMPORTANT]
> Для анализа данных Mart и хранилища данных мы настоятельно рекомендуем использовать [кластерные индексы столбцов,](https://docs.microsoft.com/sql/database-engine/using-clustered-columnstore-indexes)уменьшить количество некластерных индексов и рассмотреть операции навалом нагрузки с числом строк около 1 миллиона для уменьшения избыточного потребления резервного хранилища.

## <a name="storage-costs"></a>Затраты на хранение

Цена для хранения варьируется в зависимости от того, используете ли вы модель DTU или модель vCore.

### <a name="dtu-model"></a>Модель с DTU

При использовании модели DTU дополнительная плата за резервное хранение данных и эластичных пулов баз данных не взимается.

### <a name="vcore-model"></a>Модель с виртуальными ядрами

Для отдельных баз данных минимальная сумма хранения резервного копирования, равная 100 процентам размера базы данных, предоставляется без дополнительной платы. Для эластичных пулов баз данных и управляемых экземпляров минимальная сумма хранения резервного копирования, равная 100 процентам выделенного хранилища данных для пула или размера экземпляра, соответственно, предоставляется без дополнительной платы. Плата за дополнительное пространство хранилища резервных копий будет взиматься за количество ГБ в месяц. Это дополнительное потребление будет зависеть от рабочей нагрузки и размера отдельных баз данных.

База данных Azure S'L вычислит общее хранилище резервного копирования в запасе в качестве совокупного значения. Каждый час это значение сообщается конвейеру выставления счетов Azure, который отвечает за агрегирование этого почасового использования для получения потребления в конце каждого месяца. После того, как база данных удалена, корпорация Майкрософт уменьшает потребление по мере старения резервных служб. После того, как резервные ожидания становятся старше периода хранения, выставление счетов останавливается. Поскольку все резервные копии журнала и дифференциальные резервные копии сохраняются в течение полного периода хранения, сильно измененные базы данных будут иметь более высокие сборы за резервное копирование.

Предположим, что база данных накопила 744 ГБ резервного хранилища и что эта сумма остается постоянной в течение всего месяца. Чтобы преобразовать это кумулятивное потребление хранилища в почасовое использование, разделите его на 744,0 (31 день в месяц и 24 часа в сутки). Таким образом, база данных S'L сообщит, что база данных потребляется 1 ГБ резервного копирования PITR каждый час. Счет наяне будет агрегировать это потребление и отображать использование 744 ГБ за весь месяц. Стоимость будет основываться на $ / ГБ / месяц ставка в вашем регионе.

Теперь, более сложный пример. Предположим, что в середине месяца срок ее хранения увеличился до 14 дней. Предположим, что это увеличение (гипотетически) приводит к удвоению общего объема резервного хранилища до 1488 ГБ. База данных S'L сообщит о 1 ГБ использования в течение часов с 1 по 372. Он будет сообщать об использовании в качестве 2 ГБ в течение нескольких часов 373 до 744. Это использование будет агрегировано с окончательным счетом 1116 ГБ/месяц.

### <a name="monitor-costs"></a>Мониторинг затрат

Чтобы понять затраты на хранение резервного копирования, перейдите на портал **управления затратами и выставление счетов** на портале Azure, выберите **управление затратами,** а затем выберите **анализ затрат.** Выберите нужную подписку в качестве **области,** а затем отфильтруйте для периода времени и службы, которые вас интересуют.

Добавьте фильтр для **имени службы,** а затем выберите **базу данных sql** в списке выпадающих. Используйте фильтр **подкатегории счетчика,** чтобы выбрать счетчик выставления счетов для службы. Для единой базы данных или упругого пула базы данных выберите **резервное хранилище для резервного копирования одного/упругого пула.** Для управляемого экземпляра выберите **хранилище резервного копирования mi pitr.** Подкатегории **хранения** и **вычислений** могут вас также заинтересовать, но они не связаны с затратами на хранение резервного копирования.

## <a name="backup-retention"></a>Хранение резервных копий

Все базы данных Azure S'L (единичные, объединенные и управляемые базы данных экземпляров) имеют период хранения резервного копирования по умолчанию 7 дней. Период [хранения резервного копирования](#change-the-pitr-backup-retention-period) можно изменить на 35 дней.

При удалении базы данных резервные копии для базы данных SQL будут храниться точно так же, как и для базы данных в Интернете. Например, если вы удалите базовую базу данных с периодом хранения семь дней, резервная часть, которая четыре дня сохраняется в течение еще трех дней.

Если вам нужно сохранить резервные копии на более долгий срок по сравнению с максимальным периодом хранения, вы можете изменить свойства резервной копии, добавив в базу данных один или несколько периодов долгосрочного хранения. Для получения дополнительной информации [см.](sql-database-long-term-retention.md)

> [!IMPORTANT]
> При удалении сервера Azure S'L, в котором размещены базы данных S'L, все эластичные пулы баз данных и базы данных, принадлежащие серверу, также удаляются. Они не могут быть восстановлены. Вы не можете восстановить удаленный сервер. Но если вы настроили долгосрочное удержание, резервные ожидания для баз данных с LTR не будут удалены, и эти базы данных могут быть восстановлены.

## <a name="encrypted-backups"></a>Зашифрованные резервные копии

Если база данных зашифрована с помощью TDE, резервные копирования автоматически шифруются в состоянии покоя, включая резервные копирования LTR. Когда для базы данных Azure SQL включено прозрачное шифрование данных, также шифруются резервные копии. Прозрачное шифрование данных включено по умолчанию для всех новых баз данных SQL Azure. Дополнительные сведения о прозрачном шифровании данных см. в статье [Прозрачное шифрование данных в базе данных SQL Azure](/sql/relational-databases/security/encryption/transparent-data-encryption-azure-sql).

## <a name="backup-integrity"></a>Целостность резервного копирования

На постоянной основе инженерная группа базы данных Azure S'L автоматически тестирует восстановление автоматизированных резервных данных баз данных, размещенных в логических серверах и эластичных пулах баз данных. (Это тестирование недоступно в управляемом экземпляре.) При восстановлении времени базы данных также получают проверку целостности DBCC CHECKDB.

Управляемый экземпляр осуществляет автоматическое начальное резервное копирование с `CHECKSUM` базами данных, восстановленными с помощью нативной `RESTORE` команды или с помощью службы миграции данных Azure после завершения миграции.

При обнаружении ошибок в процессе проверки целостности команда разработки получает оповещения. Для получения дополнительной информации посетите [базу данных «Целостность данных» в базе данных Azure S'L.](https://azure.microsoft.com/blog/data-integrity-in-azure-sql-database/)

## <a name="compliance"></a>Соответствие нормативным требованиям

При переносе базы данных из уровня службы на основе DTU в уровень обслуживания на основе vCore сохраняется сохранение PITR, чтобы гарантировать, что политика восстановления данных приложения не будет нарушена. Если сохранение по умолчанию не соответствует вашим требованиям соответствия требованиям, можно изменить период хранения PITR с помощью PowerShell или REST API. ## <a name="change-the-pitr-backup-retention-period"></a>Изменение периода хранения резервного копирования PITR

Можно изменить период хранения резервного копирования PITR по умолчанию с помощью портала Azure, PowerShell или API REST. Ниже приведены примеры, как изменить удержание PITR до 28 дней.

> [!WARNING]
> Если сократить текущий период хранения, все существующие резервные отстания, которые старше нового периода удержания, больше не доступны. Если увеличить текущий период удержания, база данных S'L сохранит существующие резервные данные до конца более длительного периода хранения.

> [!NOTE]
> Эти AA будут влиять только на период удержания PITR. Если вы настроили LTR для базы данных, это не повлияет. Для получения информации о том, как изменить периоды удержания LTR, [см.](sql-database-long-term-retention.md)

### <a name="change-the-pitr-backup-retention-period-by-using-the-azure-portal"></a>Измените период хранения резервного копирования PITR с помощью портала Azure

Чтобы изменить период хранения резервного копирования PITR с помощью портала Azure, перейдите на объект сервера, период хранения которого вы хотите изменить в портале. Затем выберите подходящий вариант на основе изменяемого объекта сервера.

#### <a name="single-database-and-elastic-database-pools"></a>[Единая база данных и эластичные пулы баз данных](#tab/single-database)

Изменения в сохранении резервного копирования PITR для отдельных баз данных Azure S'L выполняются на уровне сервера. Изменения, внесенные на уровне сервера, применяются к базам данных на сервере. Чтобы изменить удержание PITR для сервера базы данных Azure s'L с портала Azure, перейдите на лопасти обзора сервера. Выберите **управление резервными кавер-нет** в левом стекле, а затем выберите **удержание настройки** в верхней части экрана:

Изменения в хранении резервного копирования PITR для управляемых экземпляров базы данных S'L выполняются на отдельном уровне базы данных. Чтобы изменить сохранение резервного копирования PITR для базы данных экземпляров с портала Azure, перейдите на отдельный лопасти обзора базы данных. ### <a name="change-the-pitr-backup-retention-period-by-using-powershell"></a>Измените период хранения резервного копирования PITR с помощью PowerShell

[!INCLUDE [updated-for-az](../../includes/updated-for-az.md)]
> [!IMPORTANT]
> Модуль PowerShell AzureRM по-прежнему поддерживается базой данных Azure S'L, но все будущие разработки предназначены для модуля Az.Sql. Для получения дополнительной информации посетите [AzureRM.Sql](https://docs.microsoft.com/powershell/module/AzureRM.Sql/). Аргументы в пользу команд в модуле Az существенно идентичны тем, которые есть в модулях AzureRm.

```powershell
Set-AzSqlDatabaseBackupShortTermRetentionPolicy -ResourceGroupName resourceGroup -ServerName testserver -DatabaseName testDatabase -RetentionDays 28
```

### <a name="change-the-pitr-backup-retention-period-by-using-the-rest-api"></a>Изменение периода хранения резервного копирования PITR с помощью REST API

#### <a name="sample-request"></a>Пример запроса

```http
PUT https://management.azure.com/subscriptions/00000000-1111-2222-3333-444444444444/resourceGroups/resourceGroup/providers/Microsoft.Sql/servers/testserver/databases/testDatabase/backupShortTermRetentionPolicies/default?api-version=2017-10-01-preview
```

#### <a name="request-body"></a>Тело запроса

```json
{
  "properties":{
    "retentionDays":28
  }
}
```

#### <a name="sample-response"></a>Пример ответа

Код состояния: 200.

```json
{
  "id": "/subscriptions/00000000-1111-2222-3333-444444444444/providers/Microsoft.Sql/resourceGroups/resourceGroup/servers/testserver/databases/testDatabase/backupShortTermRetentionPolicies/default",
  "name": "default",
  "type": "Microsoft.Sql/resourceGroups/servers/databases/backupShortTermRetentionPolicies",
  "properties": {
    "retentionDays": 28
  }
}
```

Дополнительные сведения о политиках краткосрочного хранения резервных копий см. [здесь](https://docs.microsoft.com/rest/api/sql/backupshorttermretentionpolicies).
