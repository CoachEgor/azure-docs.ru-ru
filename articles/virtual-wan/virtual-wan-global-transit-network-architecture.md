# <a name="global-transit-network-architecture-and-virtual-wan"></a>Архитектура глобальной транзитной сети и Виртуальная глобальная сеть

Глобальная транзитная архитектура сети принята компаниями для консолидации, подключения и контроля облачного ИТ-уровня ИТ. В современных облачных организациях сетевой трафик не требуется получать в головном офисе. Глобальная транзитная архитектура сети основана на двух знакомых сетевых понятиях и новых концепциях, уникальных для облачных и облачных архитектур.

![архитектура](./media/virtual-wan-global-transit-network-architecture/architecture2.png)

**Рис. 1. Глобальная транзитная сеть с виртуальной глобальной сетью @ no__t-0

Для современных предприятий требуется повсеместное подключение между распределенными приложениями, данными и пользователями в облаке и локальной среде. Виртуальная глобальная сеть Azure позволяет использовать глобальную транзитную сетевую архитектуру, обеспечивая повсеместное подключение между глобально распределенными наборами виртуальных сетей, сайтами, приложениями и пользователями. Виртуальная глобальная сеть Azure — это служба, управляемая корпорацией Майкрософт. Все сетевые компоненты, из которых состоит эта служба, размещаются и управляются корпорацией Майкрософт. Дополнительные сведения о виртуальной глобальной сети см. в статье [Обзор виртуальной глобальной сети](virtual-wan-about.md) .

В архитектуре виртуальной глобальной сети Azure регионы Azure служат центрами, к которым можно подключить ветви. После подключения ветвей можно использовать магистраль Azure для создания ветви между виртуальными сетями и, при необходимости, подключения ветвления к ветви.

Виртуальную сеть WAN можно установить, создав один виртуальный концентратор глобальной сети в регионе, который имеет наибольшее количество периферийных серверов (ветвей, виртуальных сетей, пользователей), а затем подключаясь к периферийным серверам, которые находятся в других регионах, в центр. Кроме того, если периферийные серверы географически распределены, можно также создать экземпляры региональных концентраторов и соединить концентраторы. Концентраторы являются частью одной виртуальной глобальной сети, но их можно связать с разными региональными политиками.

## <a name="hub"></a>Транзитный транспорт (hub-and-лучевой)

Глобальная транзитная архитектура сети основана на классической модели подключения "звезда — сеть", где центр размещенной в облаке сети обеспечивает транзитивное подключение между конечными точками, которые могут распределяться по разным типам "периферийных".
  
В этой модели спица может быть:

* Виртуальная сеть (виртуальных сетей)
* Сайт физической ветви
* Удаленный пользователь
* Интернет

![Глобальная и резервная схема в глобальной транспорте](./media/virtual-wan-global-transit-network-architecture/architecture.png)

**Рис. 2. Hub-and-лучевой @ no__t-0

На рис. 2 показано логическое представление глобальной сети, где географически распределенные пользователи, физические сайты и виртуальных сетей взаимосвязаны через сетевой концентратор, размещенный в облаке. Эта архитектура обеспечивает возможность логического транзитного подключения с одним прыжком между конечными точками сети. Периферийные серверы подключаются к концентратору различными сетевыми службами Azure, такими как ExpressRoute или VPN типа "сеть — сеть" для физических ветвей, подключений к виртуальной сети для виртуальных сетей и VPN-подключения "точка — сеть" для удаленных пользователей.

## <a name="crossregion"></a>Подключение между регионами

Для предприятия облачное хранилище обычно следует за физическим местом. Большинство предприятий обращаются к облаку из региона, ближайшего к физическому сайту и пользователям. Одним из основных субъектов глобальной сетевой архитектуры является включение межрегионового подключения между сетевыми сущностями и конечными точками. Облачная область может охватывать несколько регионов. Это означает, что трафик из ветви, подключенной к облаку в одном регионе, может достигать другой ветви или виртуальной сети в другом регионе с помощью подключения концентратора к концентратору, которое в настоящее время находится в нашей стратегии.

## <a name="any"></a>Подключение "любой к любому"

Глобальная транзитная архитектура сети позволяет *Подключиться* через центральный сетевой концентратор. Эта архитектура устраняет или уменьшает потребность в полной модели подключения к сети или частичной сетке, которая сложнее в разработке и обслуживании. Кроме того, Управление маршрутизацией в сетях с центральными и периферийными сетями проще в настройке и обслуживании.

Любое подключение в контексте глобальной архитектуры позволяет предприятиям с глобально распределенными пользователями, ветвями, центрами обработки данных, виртуальных сетей и приложениями подключаться друг к другу через транзитный центр. Транзитный центр выступает в качестве глобальной транзитной системы.

### <a name="branchvnet"></a>Ветвление в виртуальную сеть

Branch-to-VNet — это основной путь, поддерживаемый виртуальной глобальной сетью Azure. Этот путь позволяет подключать ветви к рабочим нагрузкам Azure IAAS Enterprise, развернутым в Azure виртуальных сетей. Ветви можно подключить к виртуальной глобальной сети через ExpressRoute или VPN типа "сеть — сеть". Трафик передается в виртуальных сетей, подключенные к виртуальным концентраторам WAN через подключения к виртуальной сети. Для виртуальной глобальной сети не требуется [транзитный шлюз](../virtual-network/virtual-network-peering-overview.md#gateways-and-on-premises-connectivity) , так как виртуальная сеть WAN автоматически включает транзит шлюза для сайтов ветвей.

### <a name="branchbranch"></a>Ветвь-ветвь

Ветви можно подключить к виртуальному концентратору глобальной сети Azure с помощью каналов ExpressRoute и/или VPN-подключений типа "сеть — сеть". Вы можете подключить ветви к виртуальному концентратору глобальной сети, расположенному в регионе, ближайшем к ветви.

Этот параметр позволяет компаниям использовать магистраль Azure для подключения ветвей. Однако несмотря на то, что эта возможность доступна, следует взвесить преимущества подключения ветвей через виртуальную глобальную сеть Azure по сравнению с использованием частной глобальной сети.

### <a name="usertovnet"></a>Удаленный пользователь-виртуальная сеть

Вы можете включить прямой и безопасный удаленный доступ к Azure с помощью подключений типа "точка — сеть" от клиента удаленного пользователя к виртуальной глобальной сети. Удаленным пользователям предприятия больше не нужно хаирпин в облако, используя корпоративную VPN.

### <a name="usertobranch"></a>Удаленный пользователь-ветвь

Удаленный путь "пользователь-ветвь" позволяет удаленным пользователям, которые используют подключение типа "точка — сеть" к Azure и локальным рабочим нагрузкам и приложениям, переходящим через облако. Этот путь предоставляет удаленным пользователям возможность доступа к рабочим нагрузкам, которые развернуты в Azure и локально. Предприятия могут включить централизованную службу удаленного доступа на основе облака в виртуальной глобальной сети Azure.

### <a name="vnetvnet"></a>Транзит между виртуальными сетями с помощью пиринга виртуальных сетей

Чтобы подключить виртуальных сетей к друг другу для поддержки многоуровневых приложений, реализованных в нескольких виртуальных сетей, используйте пиринг виртуальной сети. Транзитный сценарий "Виртуальная сеть — виртуальная сеть" с помощью виртуальной глобальной сети Azure в настоящее время не поддерживается, но находится в стратегии Azure. Подключение виртуальных сетей через пиринг виртуальных сетей — это рекомендуемое решение для виртуальных сетей, которое должно быть подключено друг к другу. 

### <a name="globalreach"></a>Global Reach ExpressRoute

ExpressRoute — это частный и устойчивый способ подключения локальных сетей к Microsoft Cloud. ExpressRoute Global Reach — это дополнительный компонент для ExpressRoute. С помощью Global Reach можно связать каналы ExpressRoute, чтобы создать частную сеть между локальными сетями. Для ветвей, подключенных к виртуальной глобальной сети Azure с помощью ExpressRoute, требуется, чтобы Global Reach ExpressRoute взаимодействовал друг с другом.

В этой модели каждая ветвь, подключенная к виртуальному концентратору глобальной сети с помощью ExpressRoute, может подключаться к виртуальных сетей с помощью пути Branch-to-VNet. Трафик ветвления в ветви не переносится в центр, так как ExpressRoute Global Reach обеспечивает более оптимальный путь через глобальную сеть Azure.

## <a name="security"></a>Безопасность и управление политиками

Концентратор виртуальной сети взаимоподключает и потенциально видит весь транзитный трафик. Это может быть место для размещения центральных сетевых функций и служб, таких как облачная маршрутизация, политика сети и безопасность, а также управление доступом к Интернету.
