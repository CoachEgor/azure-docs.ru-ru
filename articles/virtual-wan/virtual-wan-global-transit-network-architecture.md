# <a name="global-transit-network-architecture-and-virtual-wan"></a>Глобальные транзитный сетевой архитектуры и виртуальной глобальной сети

Глобальные транзитный сетевой архитектуры внедряется организациям консолидировать, подключения и управления, ориентированные на облако современных корпоративных ИТ-инфраструктуры. В современных корпоративных ориентированные на облако сетевой трафик не нужно быть backhauled для головного Офиса. Архитектура сети глобального передаче основана на привычными понятиями сети и новыми понятиями, которые являются уникальными для облака и облачных архитектур.

![архитектура](./media/virtual-wan-global-transit-network-architecture/architecture2.png)

**Рис. 1. Глобальные транзитной сети с виртуальной глобальной сети**

Современных предприятиях требуется повсеместно подключение между hyper распределенных приложений, данных и пользователей в облаке и локальной. Azure виртуальной глобальной сети доступна глобального транзитный сетевой архитектуры, предоставляющая повсеместно, любой к любому подключение между наборов глобально распределенных виртуальных сетей, сайтов, приложений и пользователей. Azure виртуальной глобальной сети — это служба, управляемых корпорацией Майкрософт. Все сетевые компоненты, которые эта служба представляет собой хранимых и управляемых корпорацией Майкрософт. Дополнительные сведения о виртуальной глобальной сети, см. в разделе [Общие сведения о виртуальных глобальной сети](virtual-wan-about.md) статьи.

В Azure виртуальной глобальной СЕТЬЮ архитектуре регионов Azure служат в качестве концентраторов, к которым вы можете выбрать подключение ветвей. Подключившись ветви, вы можете использовать магистральную сеть Azure, чтобы установить эту ветвь to-VNet и, возможно, подключения к ветви в ветвь.

Виртуальная глобальная сеть можно установить путем создания в одном центре виртуальной глобальной сети в регионе с самым большим количеством периферийных зон (ветви, виртуальные сети, пользователей) и затем так соедините периферийных зон, которые находятся в других регионах к концентратору. Кроме того Если периферийные зоны географически распределены, можно также создать экземпляр региональные концентраторы и соединения, которые используются концентраторы. Концентраторы являются частью одной виртуальной глобальной сети, но они могут быть связаны с помощью различных региональных политик.

## <a name="hub"></a>Концентратор и спица пути

Архитектура сети глобального передаче основывается на модели классического концентратор и спица подключения, где размещенной облачной сети «концентратор» обеспечивает транзитивное подключение между конечными точками, которые могут быть распределены на разных типах «периферийные зоны».
  
В этой модели может быть периферийной зоны:

* Виртуальная сеть (Vnet)
* Сайт физической ветви
* Удаленный пользователь
* Интернет

![звездообразной схеме глобальные пути](./media/virtual-wan-global-transit-network-architecture/architecture.png)

**Рис. 2. Концентратор и спица**

Рис. 2 показано логическое представление глобальной сети, где географически распределенных пользователей, физические узлы и виртуальные сети связаны между собой с помощью сетевых вещей, размещенных в облаке. Эта архитектура предоставляет логический одной ретрансляцией транзитными подключениями между сетевые конечные точки. Периферийные зоны в концентратор соединены различных сетевых службах Azure например ExpressRoute или сайт для сайта VPN для физического ветвей, Пиринг виртуальных сетей для виртуальных сетей и VPN между сайтами для удаленных пользователей.

## <a name="crossregion"></a>Межрегиональное подключение

Для предприятия для объема данных облака, как правило, следует занимаемый. Большинство предприятий доступ к облаку из регион, ближайший к их физических узлов и пользователей. Одно из ключевых участников архитектуры глобальной сети является возможность межрегиональное подключение между сущностями сети и конечных точек. Объем облачных может охватывать несколько регионов. Это означает, что трафик из ветви, подключенных к облаку в одном регионе можно связаться с другой ветви или виртуальной сети в другом регионе с помощью подключения концентратора в концентратор, которая в настоящее время в предварительной версии.

## <a name="any"></a>Любой к любому подключения

Архитектура сети глобальные пути позволяет *любой к любому подключения* через концентратор центральной сети. Эта архитектура полностью устранить или снизить потребность в полной сетки или модели подключения к частичной сетки, которые являются более сложными для создания и поддержки. Кроме того маршрутизации в концентратор и спица и сетки сети он упрощает настройку и обслуживание.

Любой к любому подключение в контексте архитектуры глобального позволяет организациям с глобально распределенных пользователей, ветви, центрах обработки данных, виртуальные сети и приложений, соединяющихся друг с другом через Центр пути. Центр передаче выступает в качестве системы глобального пути.

### <a name="branchvnet"></a>Ветвь to-VNet

Ветвь to-VNet — это основной путь, поддерживаемый Azure виртуальной глобальной сети. Этот путь позволяет подключаться ветвей для корпоративных рабочих нагрузок Azure IAAS, развернутых в виртуальных сетях Azure. Ветви можно подключить к виртуальной глобальной сети с помощью ExpressRoute или VPN-Подключение сайт сайт. Изменится трафик к виртуальным сетям, подключенных к виртуальной глобальной сети центров через подключения виртуальной сети.

### <a name="branchbranch"></a>Ветви в ветвь

Ветви можно подключить к концентратор Azure виртуальной глобальной сети с помощью каналов ExpressRoute или VPN-подключения сайт сайт. Ветви можно подключиться к концентратору виртуальной глобальной сети в регион, ближайший к ветви.

Этот параметр позволяет предприятиям эффективно использовать магистральную сеть Azure для подключения ветвей. Тем не менее несмотря на то, что эта возможность доступна, вам следует взвесить преимущества подключении ветвей через Azure виртуальной глобальной сети и с помощью частной глобальной сети.

### <a name="usertovnet"></a>Удаленный пользователь to-VNet

Вы можете включить прямой защищенного удаленного доступа в Azure с помощью подключения удаленного пользователя клиента точка сеть к виртуальной глобальной сети. Удаленные пользователи предприятия больше не нужно hairpin в облако с помощью VPN предприятия.

### <a name="usertobranch"></a>Удаленный пользователь для ветви

Путь удаленного пользователя к ветви позволяет удаленным пользователям, использующим подключение точка сеть Azure доступ для локальных рабочих нагрузок и приложений с транзитом через облако. Этот путь позволяет удаленным пользователям для рабочих нагрузок, развернутых в Azure и локальных. Предприятия могут предоставлять службы центра облачной безопасный удаленный доступ в Azure виртуальной глобальной сети.

### <a name="vnetvnet"></a>Время передачи виртуальными сетями с помощью пиринга

Для подключения виртуальных сетей друг к другу для поддержки многоуровневых приложений, которые реализуются для нескольких виртуальных сетей, используйте пиринг виртуальной сети. Сценарий передачи виртуальными сетями с помощью Azure виртуальной глобальной сети сейчас не поддерживается, но на странице стратегии развития Azure. Подключение виртуальных сетей через Пиринг виртуальной сети — в качестве решения рекомендуется для виртуальных сетей, должны быть подключены друг к другу. [Транзит через шлюз](../virtual-network/virtual-network-peering-overview.md#gateways-and-on-premises-connectivity) (в рамках пиринга виртуальной сети) не является обязательным для виртуальной глобальной сети, так как виртуальной глобальной сети автоматически включает Транзит через шлюз.

### <a name="globalreach"></a>ExpressRoute в мире

ExpressRoute — частный и устойчивые способ подключения вашей локальной сети к Microsoft Cloud. ExpressRoute глобальным доступом является функцией расширения для ExpressRoute. С глобальным доступом вы можете связать каналов ExpressRoute вместе, чтобы сделать частной сети между вашей локальной сети. Ветви, которые подключены к Azure виртуальной глобальной сети с помощью ExpressRoute требуется ExpressRoute глобальным доступом для взаимодействия друг с другом.

В этой модели каждая ветвь, подключенный к концентратору виртуальной глобальной сети с помощью ExpressRoute можно подключиться к виртуальным сетям, используя путь ветви to-VNet. Трафик ветви в ветвь не транзитной концентратора, так как глобальным доступом ExpressRoute обеспечивает более оптимальный путь через глобальную сеть Azure.

## <a name="security"></a>Безопасность и управление политиками
