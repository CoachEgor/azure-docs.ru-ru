# <a name="global-transit-network-architecture-and-virtual-wan"></a>Глобальная архитектура транзитной сети и виртуальный WAN

Современные предприятия требуют повсеместной связи между гиперраспределенными приложениями, данными и пользователями в облаке и в помещениях. Глобальная архитектура транзитных сетей внедряется предприятиями для консолидации, подключения и контроля за облачным современным и глобальным иТ-ней.

Архитектура глобальной транзитной сети основана на классической модели подключения концентратора и спица, где облачный хаб сети обеспечивает транзитную связь между конечными точками, которые могут быть распределены между различными типами "спичек".

В этой модели спица может быть:
* Виртуальная сеть (VNets)
* Физический сайт филиала
* Удаленный пользователь
* Интернет

![концентратор и говорил](./media/virtual-wan-global-transit-network-architecture/figure1.png)

**Рисунок 1: Глобальная сеть транзитных узлов и спиц**

На рисунке 1 показано логическое представление о глобальной транзитной сети, где географически распределенные пользователи, физические сайты и VNets взаимосвязаны через сетевой концентратор, размещенный в облаке. Эта архитектура обеспечивает логическое одно-хоп транзитное подключение между конечными точками сети.

## <a name="global-transit-network-with-virtual-wan"></a><a name="globalnetworktransit"></a>Глобальная транзитная сеть с Virtual WAN

Azure Virtual WAN — это облачный сетевой сервис, управляемый корпорацией Майкрософт. Все сетевые компоненты, из которой состоит эта служба, размещаются и управляются корпорацией Майкрософт. Для получения дополнительной информации о виртуальном WAN, [см.](virtual-wan-about.md)

Azure Virtual WAN позволяет глобальную архитектуру транзитной сети, обеспечивая повсеместное, любое подключение между глобально распределенными наборами облачных рабочих нагрузок в VNets, филиалах, приложениях SaaS и PaaS и пользователях.

![Виртуальная глобальная сеть Azure (WAN)](./media/virtual-wan-global-transit-network-architecture/figure2.png)

**Рисунок 2: Глобальная транзитная сеть и виртуальный WAN**

В архитектуре Virtual WAN, виртуальные концентраторы WAN подготовлены в регионах Azure, к которым можно подключить филиалы, VNets и удаленных пользователей. Физические ветви сайты подключены к концентратору Premium ExpressRoute или сайт-к сайту-VPN, VNets подключены к концентратору с помощью VNet соединений, и удаленные пользователи могут непосредственно подключиться к концентратору с помощью vpn пользователя (точка-к-сайт VPN). Virtual WAN также поддерживает межрегиональное соединение VNet, где VNet в одном регионе может быть подключен к виртуальному узлу WAN в другом регионе.

Вы можете создать виртуальный WAN, создав один виртуальный WAN-концентратор в регионе, который имеет наибольшее количество спицов (филиалы, VNets, пользователи), а затем соединив спицы, которые находятся в других регионах, к концентратору. Это хороший вариант, когда предприятие след в основном в одном регионе с несколькими удаленными спицы.  
  
## <a name="hub-to-hub-connectivity-preview"></a><a name="hubtohub"></a>Подключение к концентратору (Предварительный просмотр)

Область корпоративного облака может охватывать несколько областей облаков, и оптимально (задержка-мудрый) для доступа к облаку из региона, ближайшего к их физическому сайту и пользователям. Одним из ключевых принципов глобальной архитектуры транзитных сетей является обеспечение межрегионального подключения между всеми облачными и конечными точками сети. Это означает, что трафик из ветки, подключенной к облаку в одном регионе, может достичь другой ветки или VNet в другом регионе с помощью подключения к концентратору, включенного [Azure Global Network.](https://azure.microsoft.com/global-infrastructure/global-network/)

![кросс-регион](./media/virtual-wan-global-transit-network-architecture/figure3.png)

**Рисунок 3: Виртуальное кросс-региональное подключение WAN**

Когда несколько концентраторов включены в одном виртуальном WAN, концентраторы автоматически взаимосвязаны через ссылки к онцентру, что позволяет глобальную связь между ветвями и Vnets, которые распределены между несколькими регионами. 

Кроме того, концентраторы, которые являются частью одного и того же виртуального WAN, могут быть связаны с различными региональными политиками доступа и безопасности. Для получения дополнительной информации в этой статье смотрите [вопросы безопасности и контроля политики.](#security)

## <a name="any-to-any-connectivity"></a><a name="anytoany"></a>Любое подключение к любому

Глобальная архитектура транзитной сети обеспечивает любое подключение через виртуальные концентраторы WAN. Эта архитектура устраняет или уменьшает потребность в полной сетке или частичном подключении сетки между спицы, которые являются более сложными для создания и поддержания. Кроме того, управление повыки в сети концентратор-и-смежные сети проще настраивать и поддерживать.

Любое подключение (в контексте глобальной архитектуры) позволяет предприятию с глобально распределенными пользователями, филиалами, центрами обработки данных, VNets и приложениями подключаться друг к другу через "транзитный" хаб (ы). Azure Virtual WAN выступает в качестве глобальной транзитной системы.

### <a name="branch-to-vnet-a-and-branch-to-vnet-cross-region-g"></a>Кросс-регион от филиала к VNet (a) и Межрегиональный регион от филиала до VNet (g)

Branch-to-VNet — основной путь, поддерживаемый Azure Virtual WAN. Этот путь позволяет подключать ветви к рабочим нагрузкам предприятия Azure IAAS, развернутым в внедряемых в Azure VNets. Филиалы могут быть подключены к виртуальному WAN через ExpressRoute или от сайта к сайту VPN. Трафик проходит через VNets, которые подключены к виртуальным концентратам WAN через VNet Connections. Явный [транзит шлюзов](../virtual-network/virtual-network-peering-overview.md#gateways-and-on-premises-connectivity) не требуется для виртуального WAN, потому что Virtual WAN автоматически позволяет шлюз транзита на сайт филиала. Смотрите статью [Virtual WAN Partners](virtual-wan-configure-automation-providers.md) о том, как подключить SD-WAN CPE к virtual WAN.

### <a name="expressroute-global-reach-and-virtual-wan"></a>ExpressRoute Глобальный охват и виртуальный WAN

ExpressRoute — это частный и устойчивый способ подключения ваших сетей к облаку Майкрософт. Виртуальный WAN поддерживает соединения схемы экспресс-маршрута. Подключение филиала к виртуальному WAN с экспресс-маршрутом требует 1) Premium Circuit 2) Circuit, чтобы быть в месте с поддержкой Global Reach.

ExpressRoute Global Reach — это дополнительная функция для ExpressRoute. С Global Reach вы можете связать схемы ExpressRoute вместе, чтобы создать частную сеть между вашими сетевыми сетями. Филиалы, подключенные к Virtual WAN Azure с помощью ExpressRoute, требуют, чтобы ExpressRoute Global Reach общался друг с другом.

В этой модели каждая ветвь, подключенная к виртуальному концентратору WAN с помощью ExpressRoute, может подключаться к VNets, используя путь ветки к VNet. Трафик от филиала к филиалу не будет проходить через узел, потому что ExpressRoute Global Reach обеспечивает более оптимальный путь через Azure WAN.

### <a name="branch-to-branch-b-and-branch-to-branch-cross-region-f"></a>Межрегиональный филиал (b) и межрегиональный межрегиональный межрегиональный регион от филиала до отделения (f)

Филиалы могут быть подключены к виртуальному концентратору WAN Azure с помощью схем ExpressRoute и/или VPN-соединений сайта. Вы можете подключить ветви к виртуальному концентратору WAN, который находится в регионе, близком к ветке.

Эта опция позволяет предприятиям использовать магистраль Azure для подключения ветвей. Однако, несмотря на то, что эта возможность доступна, следует взвесить преимущества подключения ветвей через Azure Virtual WAN по сравнению с использованием частного WAN.  

### <a name="remote-user-to-vnet-c"></a>Удаленный пользователь-к-VNet (c)

Вы можете включить прямой, безопасный удаленный доступ к Azure, используя связь от точки к сайту от удаленного клиента пользователя к виртуальному WAN. Корпоративные удаленные пользователи больше не должны шпильки в облаке с помощью корпоративного VPN.

### <a name="remote-user-to-branch-d"></a>Удаленный пользователь к ветви (d)

Путь удаленного пользователя к ветвям позволяет удаленным пользователям, использующим подключение к сайту к работе с Azure, получить доступ к местным рабочим нагрузкам и приложениям, транзитом через облако. Этот способ предоставляет удаленным пользователям возможность доступа к рабочим нагрузкам, развернутым как в Azure, так и в помещениях. Предприятия могут включить централизованную облачную службу удаленного доступа в Azure Virtual WAN.

### <a name="vnet-to-vnet-transit-e-and-vnet-to-vnet-cross-region-h"></a>Транзит VNet-to-VNet (e) и Кросс-регион VNet-to-VNet (h)

Транзит VNet-to-VNet позволяет VNets подключаться друг к другу, чтобы соединить многоуровневые приложения, которые реализуются через несколько VNets. Дополнительно, вы можете подключить VNets друг к другу через VNet Peering, и это может быть подходящим для некоторых сценариев, где транзит через концентратор VWAN не является необходимым.

## <a name="security-and-policy-control"></a><a name="security"></a>Безопасность и контроль политики

Виртуальные концентраторы Azure WAN соединяют все конечные точки сети в гибридной сети и потенциально видят весь транзитный сетевой трафик. Виртуальные концентраторы WAN могут быть преобразованы в защищенные виртуальные концентраторы путем развертывания брандмауэра Azure в концентратах VWAN для обеспечения облачной безопасности, доступа и управления политикой. Оркестрирование брандмауэров Azure в виртуальных концентралях WAN может выполняться менеджером брандмауэра Azure.

[Azure Firewall Manager](https://go.microsoft.com/fwlink/?linkid=2107683) предоставляет возможности для управления и масштабирования безопасности глобальных транзитных сетей. Менеджер брандмауэра Azure предоставляет возможность централизованного управления трассированием, глобального управления политиками, передовых служб безопасности в Интернете через сторонних групп вместе с Azure Firewall.

![защищенный виртуальный концентратор с Брандмауэром Azure](./media/virtual-wan-global-transit-network-architecture/figure5.png)

**Рисунок 5: Защищенный виртуальный концентратор с брандмауэром Azure**

* Безопасный транзит VNet-to-VNet (e)

### <a name="vnet-to-vnet-secured-transit-e"></a>Защищенный транзит VNet-to-VNet (e)

Защищенный транзит VNet-v-VNet позволяет VNets подключаться друг к другу через брандмауэр Azure в виртуальном концентраторе WAN.

### <a name="vnet-to-internet-or-third-party-security-service-i"></a>Служба безопасности VNet-to-Internet или сторонние (i)

Защищенный транзит VNet-to-Internet или сторонних сторонних сетей позволяет VNets подключаться к Интернету или поддерживаемым сторонним службам безопасности через брандмауэр Azure Firewall в виртуальном концентраторе WAN.

### <a name="branch-to-internet-or-third-party-security-service-j"></a>Служба безопасности от филиала в Интернете или сторонние (j)
Транзит от ветки в Интернет или сторонний Безопасный позволяет филиалам подключаться к Интернету или поддерживаемым сторонним службам безопасности через брандмауэр Azure firewall в виртуальном концентраторе WAN.
