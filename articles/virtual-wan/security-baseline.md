# <a name="azure-security-baseline-for-virtual-wan"></a>Базовый уровень безопасности Azure для виртуальной глобальной сети

Этот базовый план безопасности применяет рекомендации из [контрольного показателя безопасности Azure версии 2,0](../security/benchmarks/overview.md) к Microsoft Azure виртуальной глобальной сети. Azure Security Benchmark содержит рекомендации по обеспечению безопасности облачных решений в Azure. Содержимое группируются по **элементам управления безопасностью** , определенным в ходе тестирования системы безопасности Azure, и связанным рекомендациям, применимым к виртуальной глобальной сети. **Элементы управления** , неприменимые к виртуальной глобальной сети, были исключены.

Сведения о том, как виртуальная Глобальная сеть полностью сопоставляется с тестовым уровнем безопасности Azure, см. в [полном файле сопоставления базовых показателей безопасности виртуальной глобальной сети](https://github.com/MicrosoftDocs/SecurityBenchmarks/tree/master/Azure%20Offer%20Security%20Baselines).

### <a name="ns-1-implement-security-for-internal-traffic"></a>NS-1: реализация безопасности для внутреннего трафика

**Руководство**. Виртуальная глобальная сеть Azure не поддерживает развертывание непосредственно в виртуальной сети. Тем не менее вы по-прежнему можете применять правила группы безопасности сети к ресурсам периферийных ресурсов, использовать защиту брандмауэра Azure или создавать пользовательские таблицы маршрутов, чтобы запретить или разрешить частный трафик.

### <a name="ns-2-connect-private-networks-together"></a>NS-2. Совместное подключение частных сетей 

**Руководство**. Использование Azure ExpressRoute или виртуальной частной сети Azure (VPN) для создания частных подключений между центрами обработки данных Azure и локальной инфраструктурой в среде совместного размещения. Подключения ExpressRoute не проходят через общедоступный Интернет, обеспечивают повышенную надежность, скорость и низкие задержки, чем обычные подключения к Интернету. 

Для VPN типа "точка — сеть" и VPN-подключения "сеть — сеть" подключите локальные устройства или сети к виртуальной сети, используя любое сочетание параметров VPN и ExpressRoute. Для совместного подключения нескольких виртуальных сетей в Azure используйте пиринг виртуальных сетей. Сетевой трафик между одноранговыми виртуальными сетями является частным и хранится в магистральной сети Azure.

### <a name="ns-4-protect-applications-and-services-from-external-network-attacks"></a>NS-4: Защита приложений и служб от внешних сетевых атак

**Рекомендации**. Защитите виртуальные глобальные ресурсы Azure от атак из внешних сетей, в том числе от распределенных атак типа "отказ в обслуживании" (от атак DDoS), атак, зависящих от приложения, а также незапрошенных и потенциально вредоносных интернет-трафика. 

Используйте брандмауэр Azure для защиты приложений и служб от потенциально вредоносного трафика из Интернета и других внешних расположений. Выберите Azure от атак DDoS Protection для своих ресурсов на основе атак в виртуальных сетях Azure. Используйте центр безопасности Azure для обнаружения рисков с несоответствующими настройками, связанными с сетевыми ресурсами.

### <a name="ns-6-simplify-network-security-rules"></a>NS-6: Упростите правила безопасности сети.

### <a name="ns-7-secure-domain-name-service-dns"></a>NS-7: служба защиты доменных имен (DNS)

**Руководство**. Виртуальная глобальная сеть Azure предлагает пользовательские DNS-серверы для VPN-шлюзов типа "точка — сеть". 

Следуйте рекомендациям по обеспечению безопасности DNS, чтобы избежать распространенных атак, таких как висячие DNS, атаки путем взлома DNS, подмена DNS и подделка и т. д.

Если в качестве полномочной службы DNS используется Azure DNS, убедитесь, что зоны и записи DNS защищены от случайного или вредоносного изменения с помощью управления доступом на основе ролей Azure (Azure RBAC) и блокировки ресурсов.

### <a name="im-1-standardize-azure-active-directory-as-the-central-identity-and-authentication-system"></a>IM-1: стандартизация Azure Active Directory в качестве централизованной системы идентификации и проверки подлинности

**Руководство**. VPN-подключение типа "точка — сеть" виртуальной сети Azure интегрировано с Azure Active Directory (Azure AD), которое является службой управления удостоверениями и доступом Azure по умолчанию. Необходимо стандартизировать Azure AD, чтобы управлять удостоверениями и управлением доступом организации в.

- Microsoft Cloud ресурсы, такие как портал Azure, служба хранилища Azure, виртуальные машины Azure (Linux и Windows), Azure Key Vault, PaaS и приложения SaaS.
- Ресурсы Организации, такие как приложения в Azure или ресурсы корпоративной сети.

Обеспечьте безопасность Azure AD с высоким приоритетом в целях безопасности в облаке вашей организации. Оцените свои удостоверения и безопасность с помощью функции оценки безопасности центра безопасности Azure, чтобы оценить, насколько близко ваша конфигурация соответствует рекомендациям. Реализуйте рекомендации Майкрософт для улучшения безопасности.

Azure AD также поддерживает внешние удостоверения, которые позволяют пользователям без учетная запись Майкрософт входить в свои приложения и ресурсы с внешними удостоверениями. ### <a name="im-4-use-strong-authentication-controls-for-all-azure-active-directory-based-access"></a>IM-4. использование строгих средств проверки подлинности для всех Azure Active Directory доступом

**Руководство**. VPN-подключение типа "точка — сеть" виртуальной сети Azure интегрировано с Azure Active Directory (Azure AD), которое поддерживает надежные элементы управления проверкой подлинности с многофакторной проверкой подлинности и надежные методы без пароля.

- Многофакторная проверка подлинности. Включите многофакторную проверку подлинности Azure AD и следуйте рекомендациям по управлению удостоверениями и доступом в центре безопасности Azure, чтобы получить рекомендации по настройке многофакторной проверки подлинности. Многофакторная проверка подлинности может быть применена ко всем, выбирать пользователей или на уровне отдельных пользователей в зависимости от условий входа и факторов риска.

- Проверка подлинности с паролем — доступны три варианта проверки подлинности с паролем. К ним относятся, Windows Hello для бизнеса, Microsoft Authenticator приложение и локальные методы проверки подлинности, такие как смарт-карты.

Для администраторов и привилегированных пользователей убедитесь, что используется высший уровень надежной проверки подлинности, а затем развертывание соответствующей политики надежной проверки подлинности для других пользователей.

### <a name="dp-4-encrypt-sensitive-information-in-transit"></a>DP-4: шифрование конфиденциальной информации во время передачи

**Руководство**. шифрование является критически важным для трафика во внешних и общедоступных сетях.

- Используйте элементы управления доступом, транзитные данные должны быть защищены от атак с использованием аппаратного контроллера управления (например, для записи трафика) с помощью шифрования, чтобы гарантировать, что злоумышленники не смогут легко считывать или изменять данные.

- Убедитесь, что для трафика HTTP все клиенты, подключающиеся к ресурсам Azure, могут согласовать TLS v 1.2 или более поздней версии.

- Для удаленного управления используйте SSH (для Linux) или RDP/TLS (для Windows) вместо незашифрованного протокола. Устаревшие версии SSL/TLS/SSH, протоколы и слабые шифры должны быть отключены.

- В базовой инфраструктуре Azure предоставляет данные в транзитном шифровании по умолчанию для трафика между центрами обработки данных Azure.
