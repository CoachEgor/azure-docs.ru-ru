# <a name="scenario-any-to-any"></a>Сценарий: "любой к любому"

При работе с маршрутизацией виртуальных сетей виртуальной глобальной сети существует несколько доступных сценариев. В любом сценарии любой периферийный сервер может достигать другой. При наличии нескольких концентраторов маршрутизация между концентраторами (также называемая межцентральным концентратором) включается по умолчанию в стандартную виртуальную сеть WAN. Дополнительные сведения о маршрутизации виртуальных концентраторов см. в статье [о маршрутизации виртуальных концентраторов](about-virtual-hub-routing.md).

## <a name="design"></a><a name="design"></a>Конструирование

Чтобы определить, сколько таблиц маршрутов потребуется в сценарии виртуальной глобальной сети, можно создать матрицу подключения, где каждая ячейка показывает, может ли источник (строка) обмениваться данными с целевым объектом (столбцом).

| От |   Кому |  *Виртуальных сетей* | *Ветви* |
| -------------- | -------- | ---------- | ---|
| Виртуальные сети     | &#8594;| Прямой доступ | Прямой доступ |
| Ветви   | &#8594;| Прямой доступ  | Прямой доступ |

Каждая ячейка в предыдущей таблице описывает, взаимодействует ли виртуальная Глобальная сеть (сторона "from", заголовки строк) с префиксом назначения (стороны "to" последовательности, заголовки столбцов, выделенные курсивом). В этом сценарии отсутствуют брандмауэры или сетевые виртуальные устройства, поэтому обмен данными осуществляется напрямую через виртуальную глобальную сеть (следовательно, слово «Direct» в таблице).

Так как все подключения из виртуальных сетей и ветвей (VPN, ExpressRoute и VPN пользователей) имеют те же требования к подключению, требуется одна таблица маршрутов. В результате все соединения будут связаны и распространены в одну и ту же таблицу маршрутов по умолчанию:

* Виртуальные сети:
  * Связанная таблица маршрутов: **по умолчанию**
  * Распространение на таблицы маршрутов: **по умолчанию**
* Ветк
  * Связанная таблица маршрутов: **по умолчанию**
  * Распространение на таблицы маршрутов: **по умолчанию**

Дополнительные сведения о маршрутизации виртуальных концентраторов см. в статье [о маршрутизации виртуальных концентраторов](about-virtual-hub-routing.md).

## <a name="architecture"></a><a name="architecture"></a>Архитектура

На **рис. 1**все виртуальных сетей и ветви (VPN, EXPRESSROUTE, P2S) могут быть доступны друг другу. В виртуальном концентраторе подключения работают следующим образом:

* Подключение VPN подключает VPN-сайт к VPN-шлюзу.
* Подключение виртуальной сети подключает виртуальную сеть к виртуальному концентратору. Маршрутизатор виртуального концентратора обеспечивает транзитную функциональность между виртуальных сетей.
* Подключение ExpressRoute подключает канал ExpressRoute к шлюзу ExpressRoute.

Эти соединения (по умолчанию при создании) связаны с таблицей маршрутов по умолчанию, если конфигурация маршрутизации не настроена для подключения ни к **одному**, ни к пользовательской таблице маршрутов. Эти соединения также распространяют маршруты по умолчанию в таблицу маршрутов по умолчанию. Это позволяет выполнять любые сценарии, где любой периферийный сервер (виртуальная сеть, VPN, ER, P2S) может быть доступен друг за другом.

## <a name="workflow"></a><a name="workflow"></a>Рабочий процесс

Этот сценарий включен по умолчанию для стандартной виртуальной глобальной сети. Если параметр ветвления-to-Branch отключен в конфигурации глобальной сети, то он запрещает подключение между периферийными сетями. VPN-подключения ExpressRoute и пользователей в виртуальной глобальной сети считаются периферийными.
