# <a name="linux-performance-troubleshooting"></a>Linux Производительности Устранение

Исчерпание ресурсов на машинах Linux является общей проблемой и может проявляться через широкий спектр симптомов. В этом документе содержится обзор на высоком уровне инструментов, доступных для диагностики таких проблем.

Многие из этих инструментов принимают интервал, на котором для производства подвижного производства. Этот формат вывода обычно значительно упрощает выявление шаблонов. В случае принятия пример вызова. `[interval]`

Многие из этих инструментов имеют обширную историю и широкий набор параметров конфигурации. Эта страница предоставляет только простой подмножество призывов, чтобы выделить общие проблемы. Канонический источник информации всегда является эталонной документацией для каждого конкретного инструмента. Эта документация будет гораздо более тщательной, чем то, что представлено здесь.

## <a name="guidance"></a>Руководство

Будьте системными в своем подходе к расследованию проблем производительности. Двумя общими подходами являются USE (использование, насыщенность, ошибки) и RED (ставка, ошибки, продолжительность). RED обычно используется в контексте служб мониторинга на основе запросов. USE обычно используется для мониторинга ресурсов: для каждого ресурса в машине, мониторинга использования, насыщения и ошибок. Четыре основных вида ресурсов на любой машине являются cpu, память, диск и сеть. Высокая частота использования, насыщения или ошибок любого из этих ресурсов указывает на возможную проблему с системой. Когда проблема существует, исследуйте первопричину: почему задержка в итога диска высока? Диски или виртуальная машина SKU задушены? Какие процессы пишут на устройства и в какие файлы?

Некоторые примеры общих проблем и показателей для их диагностики:
- IoPS регулирования: использовать йостат для измерения на устройство IOPS. Убедитесь, что ни один диск не превышает своего предела, а сумма для всех дисков меньше лимита для виртуальной машины.
- Мощность регулирования: используйте йостат как для IOPS, но измерения чтения / записи пропускной способности. Убедитесь, что пропускная способность как на устройство, так и на совокупную пропускную способность ниже пределов пропускной способности.
- Исчерпание SNAT: это может проявляться как высокие активные (исходящие) соединения в SAR. 
- Потеря пакета: это может быть измерено по прокси через TCP retransmit графа относительно отправлены / получил кол. Оба `sar` `netstat` и может показать эту информацию.

## <a name="general"></a>Общие сведения

Эти инструменты являются общим назначением и охватывают основную системную информацию. Они являются хорошей отправной точкой для дальнейшего расследования.

### <a name="uptime"></a>Работоспособности

```
$ uptime
 19:32:33 up 17 days, 12:36,  0 users,  load average: 0.21, 0.77, 0.69
```

время простоя обеспечивает время простоя системы и 1, 5 и 15-минутные средние нагрузки. Эти средние нагрузки примерно соответствуют потокам выполнения работ или ожидания непрерывной работы. В абсолютном этих цифрах может быть трудно интерпретировать, но измеряется с течением времени они могут сказать нам полезную информацию:

- 1-минутный средний > 5-минутный средний означает, что нагрузка увеличивается.
- 1-минутный средний < 5-минутный средний означает снижение нагрузки.

время простоя может также пролить свет на то, почему информация недоступна: проблема может быть решена сама по себе или путем перезагрузки, прежде чем пользователь сможет получить доступ к машине.

Значение нагрузки выше, чем количество доступных потоков процессора, может указывать на проблему производительности с заданной рабочей нагрузкой.

`iostat`обеспечивает глубокое понимание использования дисков. Этот призыв `-x` проходит для `-y` расширенной статистики, чтобы пропустить начальные средние значения системы печати выходной связи с момента загрузки, и `1 1` указать, что нам нужен 1-секундный интервал, заканчивающийся после одного блока вывода. 

`iostat`предоставляет много полезных статистических данных:

- `r/s`и `w/s` считываются в секунду и пишет в секунду. Сумма этих значений составляет IOPS.
- `rkB/s`и `wkB/s` килобайты считываются/пишут в секунду. Сумма этих значений является пропускной стоимостью.
- `await`это среднее время iowait в миллисекундах для запросов в очереди.
- `avgqu-sz`— средний размер очереди по сравнению с предоставленным интервалом.

На Azure VM:

- сумма и `r/s` `w/s` для отдельного блока устройства не может превышать sKU предела этого диска.
- сумма и `rkB/s` `wkB/s` для отдельного блока устройства не может превышать пределы SKU этого диска
- сумма `r/s` и `w/s` для всех блок-устройств не может превышать пределы для VM SKU.
- сумма `rkB/s` и 'wkB/s для всех блок-устройств не могут превышать пределы для VM SKU.

Обратите внимание, что диск ОС считается управляемым диском самого маленького SKU, соответствующим его емкости. Например, диск ОС 1024 ГБ соответствует диску P30. Эфемерные диски ОС и временные диски не имеют отдельных ограничений на диски; они ограничены только полными лимитами VM.

Ненулевые значения ожидания или avgqu-sz также являются хорошими индикаторами утверждения IO.

Исчерпание порта SNAT будет проявляться как высокие "активные" значения, так как проблема обусловлена высокой скоростью исходящих, локально инициированных соединений TCP.

`netstat`может introspect широкий спектр сетевой статистики, здесь ссылается с резюме вывода. Есть много полезных полей здесь в зависимости от вопроса. Одним из полезных полей в разделе TCP является "неудачные попытки подключения". Это может быть признаком истощения порта SNAT или других проблем, применяющих исходящие соединения. Высокая частота ретранслируемых сегментов (также в разделе TCP) может указывать на проблемы с доставкой пакетов.
