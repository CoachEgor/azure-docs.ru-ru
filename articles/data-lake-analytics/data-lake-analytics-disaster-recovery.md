# <a name="disaster-recovery-guidance-for-azure-data-lake-analytics"></a>Руководство по аварийному восстановлению Azure Data Lake Analytics

Azure Data Lake Analytics — это служба обработки заданий аналитики по запросу, позволяющая упростить большие данные. Вместо развертывания и настройки оборудования вы пишете запросы для преобразования данных и получения ценных выводов. Служба аналитики способна мгновенно выполнять задания любого масштаба, для чего нужно указать необходимый объем ресурсов. Вы оплачиваете только время выполнения задания, что приводит к значительной экономии средств. Статья содержит рекомендации о том, как защитить свои задания в случае редких сбоев оборудования во всем регионе или случайного удаления.

## <a name="disaster-recovery-guidance"></a>Руководство по аварийному восстановлению

При использовании Azure Data Lake Analytics, крайне важно для подготовки собственного плана аварийного восстановления. Эта статья поможет вам составить план аварийного восстановления. Существуют дополнительные ресурсы, которые помогут вам создать свою собственную схему:
- [Восстановление после сбоев и аварийное восстановление для приложений Azure](/azure/architecture/reliability/disaster-recovery)
- [Техническое руководство по обеспечению устойчивости в Azure](/azure/architecture/reliability)

## <a name="best-practices-and-scenario-guidance"></a>Советы и рекомендации для сценария

Повторяющееся задание U-SQL можно выполнить в учетной записи ADLA в регионе, который считывает и записывает таблицы U-SQL, а также неструктурированных данных.  Подготовка к аварии, выполнив следующие действия:

1. Создание учетных записей ADLA и ADLS в дополнительный регион, который будет использоваться во время сбоя.

   > [!NOTE]
   > Поскольку имена учетных записей являются глобально уникальными, необходимо используйте согласованную схему именования, которая указывает учетную запись, которая является вторичной.

2. Для неструктурированных данных, укажите ссылку [руководство по аварийному восстановлению данных в Gen1 хранилища Озера данных Azure](../data-lake-store/data-lake-store-disaster-recovery-guidance.md)

3. Для структурированных данных, хранящихся в ADLA таблиц и баз данных создание копий метаданные артефактов, таких как базы данных, таблицы, возвращающих табличные значения функций и сборки. Необходимо периодически resync эти артефакты, когда изменения вступают в силу в рабочей среде. Например вставленные данные должен реплицировать в дополнительный регион с при копировании и вставке в вспомогательной таблицы.

   > [!NOTE]
   > Имена этих объектов относятся к дополнительной учетной записи и не являются глобально уникальными, поэтому они могут иметь те же имена, как и в основной рабочей учетной записи.

Во время сбоя необходимо обновить скрипты, поэтому точки входных путей для вторичной конечной точки. Затем пользователи отправьте свои задания в учетной записи ADLA в дополнительном регионе. Выходные данные задания записывается в учетную запись ADLA и ADLS в дополнительном регионе.
