# <a name="disaster-recovery-guidance-for-azure-data-lake-analytics"></a>Руководство по восстановлению стихийных бедствий для аналитики озер данных Azure

Azure Data Lake Analytics — это служба обработки заданий аналитики по запросу, позволяющая упростить большие данные. Вместо развертывания и настройки оборудования вы пишете запросы для преобразования данных и получения ценных выводов. Служба аналитики способна мгновенно выполнять задания любого масштаба, для чего нужно указать необходимый объем ресурсов. Вы оплачиваете только время выполнения задания, что приводит к значительной экономии средств. В этой статье содержатся рекомендации о том, как защитить работу от редких перебоев в работе по всему региону или случайных удаляний.

## <a name="disaster-recovery-guidance"></a>Руководство по аварийному восстановлению

При использовании Аналитики озер данных Azure очень важно, чтобы вы подготовили свой собственный план аварийного восстановления. Эта статья поможет вам составить план аварийного восстановления. Есть дополнительные ресурсы, которые могут помочь вам создать свой собственный план:
- [Восстановление после сбоев и аварийное восстановление для приложений Azure](/azure/architecture/reliability/disaster-recovery)
- [Техническое руководство по обеспечению устойчивости в Azure](/azure/architecture/checklist/resiliency-per-service)

## <a name="best-practices-and-scenario-guidance"></a>Рекомендации по практике и руководству по сценарию

Вы можете запустить повторяющуюся работу U-S'L в учетной записи ADLA в регионе, который читает и записывает таблицы U-S'L, а также неструктурированные данные.  Подготовка к катастрофе, предпринимая следующие шаги:

1. Создание учетных записей ADLA и ADLS во вторичном регионе, которые будут использоваться во время простоя.

   > [!NOTE]
   > Поскольку имена учетных записей являются глобально уникальными, используйте последовательную схему именования, которая указывает, какая учетная запись является вторичной.

2. Для неструктурированных данных ссылки [на рекомендации по восстановлению стихийных бедствий для данных в Azure Data Lake Storage Gen1](../data-lake-store/data-lake-store-disaster-recovery-guidance.md)

3. Для структурированных данных, хранящихся в таблицах и базах данных ADLA, создайте копии артефактов метаданных, таких как базы данных, таблицы, функции, оцененные по таблице, и сборки. Необходимо периодически пересинхронизировать эти артефакты, когда происходят изменения в производстве. Например, вновь вставленные данные должны быть воспроизведены во вторичную область путем копирования данных и вставки во вторичную таблицу.

   > [!NOTE]
   > Эти имена объектов относятся к вторичной учетной записи и не являются глобально уникальными, поэтому они могут иметь те же имена, что и в первичном производственном счете.

Во время сделя, необходимо обновить скрипты, чтобы пути ввода указывают на вторичную конечную точку. Затем пользователи отправляют свои рабочие места на счет ADLA во вторичном регионе. Затем результаты задания будут записаться на счет ADLA и ADLS во вторичном регионе.
