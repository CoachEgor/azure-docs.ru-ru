# <a name="integrate-3270-screen-driven-apps-on-ibm-mainframes-with-azure-by-using-azure-logic-apps-and-ibm-3270-connector"></a>Интеграция 3270 приложений с экраном на мэйнфреймах IBM с помощью приложений Azure Logic Apps и разъема IBM 3270

> [!NOTE]
> Этот разъем находится в [*общедоступном предварительном просмотре.*](https://azure.microsoft.com/support/legal/preview-supplemental-terms/) 

С помощью приложений Azure Logic Apps и разъема IBM 3270 вы можете получить доступ и запустить приложения IBM mainframe, которые обычно можно управлять, перемещаясь по экранам эмуляторов 3270. Таким образом, вы можете интегрировать приложения IBM mainframe с Azure, Microsoft и другими приложениями, службами и системами, создавая автоматизированные рабочие процессы с приложениями Azure Logic Apps. Разъем общается с мэйнфреймами IBM с помощью протокола TN3270 и доступен во всех регионах Azure Logic Apps, за исключением Azure Government и Azure China 21Vianet. Если вы новичок в логических приложениях, просмотрите [Что такое приложения логики Azure?](../logic-apps/logic-apps-overview.md)

В этой статье описаны эти аспекты для использования 3270 разъем: 

* Зачем использовать разъем IBM 3270 в приложениях логики Azure и как разъем работает 3270 управляемых экраном приложений

* Предпосылки и настройки для использования разъема 3270

* Шаги для добавления 3270 действий разъема в приложение логики

## <a name="why-use-this-connector"></a>Зачем использовать этот разъем?

Для доступа к приложениям на мэйнфреймах IBM обычно используется эмулятор терминала 3270, который часто называют «зеленым экраном». Этот метод является проверенным временем способом, но имеет ограничения. Хотя Host Integration Server (HIS) помогает вам работать непосредственно с этими приложениями, иногда разделение экрана и бизнес-логики может оказаться невозможным. Или, может быть, у вас больше нет информации о том, как работают приложения хоста.

Чтобы расширить эти сценарии, разъем IBM 3270 в Azure Logic Apps работает с инструментом 3270 Design Tool, который используется для записи или "захвата", экранов хоста, используемых для конкретной задачи, определить поток навигации для этой задачи через приложение mainframe и определить методы с входными и выходными параметрами для этой задачи. Инструмент проектирования преобразует эту информацию в метаданные, которые используется разъемом 3270 при вызове действия, представляющего эту задачу из приложения логики.

После создания файла метаданных из инструмента проектирования вы добавляете этот файл в учетную запись интеграции в Azure. Таким образом, приложение логики может получить доступ к метаданным вашего приложения при добавлении действия разъема 3270. Разъем считывает файл метаданных с вашей учетной записи интеграции, обрабатывает навигацию через 3270 экранов и динамически представляет параметры для действия разъема 3270. Затем можно предоставить данные в приложение-хоста, и разъем возвращает результаты в приложение логики. Таким образом, вы можете интегрировать устаревшие приложения с Azure, Microsoft и другими приложениями, службами и системами, которые поддерживают Сядкие ВИС.

## <a name="prerequisites"></a>Предварительные требования

* Подписка Azure. Если у вас еще нет подписки Azure, [зарегистрируйтесь для получения бесплатной учетной записи Azure](https://azure.microsoft.com/free/).

* Основные знания о [том, как создавать логические приложения](../logic-apps/quickstart-create-first-logic-app-workflow.md)

* Рекомендуется: [Среда интеграционных услуг (ISE)](../logic-apps/connect-virtual-network-vnet-isolated-environment.md) 

  Эту среду можно выбрать в качестве местоположения для создания и запуска приложения логики. ISE предоставляет доступ из приложения логики к ресурсам, защищенным внутри виртуальных сетей Azure.

* Приложение логики для автоматизации и запуска приложения с экраном 3270

  Разъем IBM 3270 не имеет триггеров, поэтому используйте другой триггер, чтобы запустить логическое приложение, например триггер **recurrence.** Затем можно добавить 3270 действий разъема. Для начала [создайте пустое приложение логики.](../logic-apps/quickstart-create-first-logic-app-workflow.md) 
  Если вы используете ISE, выберите ISE в качестве местоположения приложения логики.

* [Скачать и установить 3270 Дизайн инструмент](https://aka.ms/3270-design-tool-download).
Единственным предварительным условием является [Microsoft .NET Framework 4.6.1](https://aka.ms/net-framework-download).

  Этот инструмент помогает записывать экраны, пути навигации, методы и параметры задач в приложении, которые вы добавляете и запускаете как действия разъема 3270. Инструмент генерирует файл Host Integration Designer XML (HIDX), который предоставляет необходимые метаданные для разъема для использования вашего приложения мейнфреймов.
  
  После загрузки и установки этого инструмента выполните следующие действия для подключения к хосту:

  1. Откройте инструмент 3270 Design Tool. Из меню **сессии** выберите **сессии host**.
  
  1. Предоставьте информацию о сервере-хозяине TN3270.

* [Интеграционная учетная запись](../logic-apps/logic-apps-enterprise-integration-create-integration-account.md), которая является местом, где вы храните ваш файл HIDX в качестве карты, чтобы ваше приложение логики может получить доступ к метаданным и определениям методов в этом файле. 

  Убедитесь, что ваша учетная запись интеграции связана с приложением логики, в который вы используете. Кроме того, если вы используете ISE, убедитесь, что местоположение учетной записи вашей интеграции является тем же ISE, которое использует ваше приложение логики.

* Доступ к серверу TN3270, на котором размещено ваше приложение мейнфреймов

<a name="define-app-metadata"></a>

## <a name="create-metadata-overview"></a>Создание обзора метаданных

В приложении 3270 с экраном экраны и поля данных уникальны для ваших сценариев, поэтому разъем 3270 нуждается в этой информации о вашем приложении, которую вы можете предоставить в виде метаданных. Этот метаданный описывает информацию, которая помогает приложению логики идентифицировать и распознавать экраны, описывает, как перемещаться между экранами, где ввести данные и где ожидать результатов. Чтобы указать и создать эти метаданные, вы используете инструмент 3270 Design Tool, который проведет вас через эти конкретные режимы или *этапы,* как описано позже в более подробной информации:

* **Захват**: В этом режиме вы записываете экраны, необходимые для выполнения конкретной задачи с помощью приложения мэйнфрейм, например, получая банковский баланс.

* **Навигация**: В этом режиме вы указываете план или путь для навигации по экранам приложения мейнфрейма для конкретной задачи.

* **Методы**: В этом режиме, вы `GetBalance`определяете метод, например, который описывает путь навигации экрана. Вы также выбираете поля на каждом экране, которые становятся параметрами ввода и вывода метода.

### <a name="unsupported-elements"></a>Неподдерживаемые элементы

Инструмент проектирования не поддерживает следующие элементы:

* Частичная базовая карта IBM Basic Mapping Support (BMS): Если вы импортируете карту BMS, инструмент проектирования игнорирует частичные определения экрана.
* Параметры In/Out: Параметры In/Out не могут определить параметры In/Out.
* Обработка меню: Не поддерживается во время предварительного просмотра
* Обработка массива: не поддерживается во время предварительного просмотра

<a name="capture-screens"></a>

## <a name="capture-screens"></a>Захват экранов

В этом режиме вы отмечаете элемент на каждом экране 3270, который однозначно идентифицирует этот экран. Например, можно указать строку текста или более сложный набор условий, таких как конкретный текст и непустое поле. Вы можете записать эти экраны через живое соединение с сервером-хозяином, либо импортировать эту информацию с карты IBM Basic Mapping Support (BMS). Для подключения к хосту используется эмулятор TN3270. Каждое действие разъема должно отобразить одну задачу, которая начинается с подключения к сеансу и заканчивается отключением от сеанса.

1. Если вы еще не сделали, откройте 3270 Дизайн инструмент. На панели инструментов выберите **Capture,** чтобы войти в режим Захвата.

1. Чтобы начать запись, нажмите клавишу F5 или из меню **Recording,** выберите **Start Recording.** 

1. Из меню **сеанса** выберите **Connect**.

1. В панели **захвата,** начиная с первого экрана в приложении, шаг через приложение для конкретной задачи, которую вы записываете.

1. После завершения задания выйти из приложения, как вы обычно делаете.

1. Из меню **сеанса** выберите **Разъединение.**

1. Чтобы остановить запись, нажмите клавиши Shift и F5 или из меню **Записи,** выберите **Stop Recording.**

   После захвата экранов для выполнения задачи инструмент дизайнера показывает эскизы, которые представляют эти экраны. Некоторые заметки об этих эскизах:

   * В комплекте с захваченными экранами, у вас есть экран, который называется "Пустой".

     При первом подключении к [CICS](https://www.ibm.com/it-infrastructure/z/cics)необходимо отправить ключ "Clear", прежде чем вы сможете ввести имя транзакции, которая вы хотите запустить. Экран, на котором вы отправляете ключ "Clear", не имеет *атрибутов распознавания,* таких как название экрана, которое можно добавить с помощью редактора распознавания экрана. Чтобы представить этот экран, эскизы включает в себя экран под названием "Пустой". Позже этот экран можно использовать для представления экрана, на котором вы вводите имя транзакции.

   * По умолчанию имя захваченного экрана использует первое слово на экране. Если это имя уже существует, инструмент проектирования придукует имя с подчеркиваем и номером, например, "WBGB" и "WBGB_1".

1. Чтобы дать более значимое имя захваченному экрану, выполните следующие действия:

   1. В панели **экранов хоста** выберите экран, который вы хотите переименовать.

   1. В том же стекле, в нижней части в том же панели, найти свойство **имя экрана.**

   1. Измените текущее имя экрана на более описательное имя.

1. Теперь укажите поля для идентификации каждого экрана.

   С потоком данных 3270 экраны не имеют идентификаторов по умолчанию, поэтому вам нужно выбрать уникальный текст на каждом экране. Для сложных сценариев можно указать несколько условий, например, уникальный текст и поле с определенным условием.

После завершения выбора полей распознавания перейдите в следующий режим.

### <a name="conditions-for-identifying-repeated-screens"></a>Условия для идентификации повторяющихся экранов

Для того, чтобы разъем мог перемещаться и различать экраны, обычно на экране можно найти уникальный текст, который можно использовать в качестве идентификатора среди захваченных экранов. Для повторных экранов может потребоваться больше методов идентификации. Например, предположим, что у вас есть два экрана, которые выглядят одинаково, за исключением того, что один экран возвращает допустимое значение, в то время как другой экран возвращает сообщение об ошибке.

В инструменте проектирования можно добавить *атрибуты распознавания,* например, заголовок экрана, например "Получить баланс учетной записи", с помощью редактора распознавания экрана. Если у вас раздвищенный путь и обе ветви возвращают один и тот же экран, но с разными результатами, вам нужны другие атрибуты распознавания. Во время выполнения разъем использует эти атрибуты для определения текущей ветви и вилки. Вот условия, которые вы можете использовать:

* Конкретное значение: Это значение соответствует указанной строке в указанном месте.
* НЕ определенное значение: Это значение не соответствует указанной строке в указанном месте.
* Пусто: это поле пусто.
* НЕ пусто: Это поле не пусто.

Чтобы узнать больше, смотрите [план навигации Пример](#example-plan) позже в этой теме.

<a name="define-navigation"></a>

## <a name="define-navigation-plans"></a>Определение планов навигации

В этом режиме вы определяете поток или шаги для навигации по экранам приложения мейнфрейма для вашей конкретной задачи. Например, иногда может быть несколько путей, которые может пройти приложение, где один путь дает правильный результат, в то время как другой путь приводит к ошибке. Для каждого экрана укажите нажатия клавиш, необходимые для `CICSPROD <enter>`перехода на следующий экран, например.

> [!TIP]
> Если вы автоматируете несколько задач, которые используют одни и те же экраны подключения и отключения, инструмент проектирования предоставляет специальные типы планов Подключения и Отключения. Когда вы определяете эти планы, вы можете добавить их в начало и конец вашего плана навигации.

### <a name="guidelines-for-plan-definitions"></a>Руководящие принципы определения плана

* Включите все экраны, начиная с подключения и заканчивая отключением.

* Можно создать автономный план или использовать планы Connect и Disconnect, которые позволяют повторно использовать серию экранов, общих для всех транзакций.

  * Последний экран в плане Connect должен быть тем же экраном, что и первый экран в плане навигации.

  * Первый экран в плане разъединения должен быть тем же экраном, что и последний экран в плане навигации.

* Ваши захваченные экраны могут содержать много повторяющихся экранов, поэтому выберите и используйте только один экземпляр любых повторяющихся экранов в вашем плане. Вот несколько примеров повторяющихся экранов:

  * Экран ввне, например, экран **MSG-10**
  * Экран приветствия для CICS
  * "Чистый" или **пустой** экран

<a name="create-plans"></a>

### <a name="create-plans"></a>Создание планов

1. На панели инструментов 3270 Design Tool выберите **навигацию,** чтобы войти в режим навигации.

1. Для начала плана в **навигационном** стеле выберите **новый план.**

1. Под **названием «Выбрать новое план»** введите имя для вашего плана. Из списка **Тип** выберите тип плана:

   | Тип плана | Описание |
   |-----------|-------------|
   | **Процесс** | Для автономных или комбинированных планов |
   | **Подключить** | Для планов подключения |
   | **Отключить** | Для планов разъединения |
   |||

1. От панели **экранов хоста** перетащите захваченные эскизы к поверхности плана навигации в панели **навигации.**

   Чтобы представить пустой экран, на котором вы вводите имя транзакции, используйте экран "Пустой".

1. Упорядочить экраны в порядке, описываемом задаче, которую вы определяете.

1. Чтобы определить траекторию потока между экранами, включая вилки и соединения, на панели инструментов инструментария design tool, выберите **Flow.**

1. Выберите первый экран в потоке. Перетащите и нарисуйте соединение со следующим экраном в потоке.

1. Для каждого экрана укажите значения для свойства **КЛЮЧА AID** (Идентификатор внимания) и для свойства **Fixed Text,** которое перемещает поток на следующий экран.

   Возможно, у вас есть только ключ AID, или ключ AID и фиксированный текст.

После завершения навигационного плана можно [определить методы в следующем режиме.](#define-method)

#### <a name="example-identify-repeated-screens"></a>Пример: Определить повторяющиеся экраны

Для того, чтобы разъем мог перемещаться и различать экраны, обычно на экране можно найти уникальный текст, который можно использовать в качестве идентификатора на захваченных экранах. Для повторных экранов может потребоваться больше методов идентификации. Пример плана имеет вилку, где вы можете получить экраны, которые похожи. Один экран возвращает баланс счета, в то время как другой экран возвращает сообщение об ошибке.

Инструмент проектирования позволяет добавлять атрибуты распознавания, например, название экрана под названием "Получить баланс учетной записи", с помощью редактора распознавания экрана. В случае с аналогичными экранами, вам нужны другие атрибуты. Во время выполнения разъем использует эти атрибуты для определения ветки и вилки.

* В ветке, которая возвращает действительный вход, который является экраном с балансом счета, можно добавить поле, которое имеет условие "не пустое".

* В ветке, которая возвращается с ошибкой, можно добавить поле с "пустым" условием.

<a name="define-method"></a>

## <a name="define-methods"></a>Определение методов

В этом режиме вы определяете метод, связанный с планом навигации. Для каждого параметра метода вы указываете тип данных, например строку, целый ряд, дату или время и так далее. Когда вы закончите, вы можете протестировать свой метод на живой хост и подтвердить, что метод работает как ожидалось. Затем вы создаете файл метаданных или файл Host Integration Designer XML (HIDX), который теперь имеет определения метода для создания и запуска разъема IBM 3270.

1. На панели инструментов 3270 Design Tool выберите **Методы,** чтобы войти в режим Методы. 

1. В панели **навигации** выберите экран с нужными полями ввода.

1. Чтобы добавить первый параметр ввода для вашего метода, выполните следующие действия:

   1. В панели **захвата,** на экране эмулятора 3270, выберите все поле, а не только текст внутри поля, который вы хотите в качестве первого ввода.

      > [!TIP]
      > Чтобы отобразить все поля и убедиться, что вы выберете полное поле в меню **View,** выберите **все поля.**

   1. На панели инструментов инструментария design tool выберите **поле ввода.** 

   Чтобы добавить больше входных параметров, повторите предыдущие шаги для каждого параметра.

1. Чтобы добавить первый параметр вывода для метода, выполните следующие действия:

   1. В панели **захвата,** на экране эмулятора 3270, выберите все поле, а не только текст внутри поля, который вы хотите в качестве первого вывода.

      > [!TIP]
      > Чтобы отобразить все поля и убедиться, что вы выберете полное поле в меню **View,** выберите **все поля.**

   1. На панели инструментов инструментария design tool выберите **поле вывода.**

   Чтобы добавить больше параметров вывода, повторите предыдущие шаги для каждого параметра.

1. После добавления всех параметров метода определите эти свойства для каждого параметра:

   | Имя свойства | Возможные значения | 
   |---------------|-----------------|
   | **Тип данных** | Байт, Дата Время, Десятичная, Int, Длинные, Короткие, Строка |
   | **Филд Филь Техника** | Параметры поддерживают эти типы заполнения, при необходимости заполняя пробелами: <p><p>- **Тип**: Введите символы последовательно в поле. <p>- **Заменить**содержимое поля символами, при необходимости заполнив пробелы. <p>- **EraseEofType:** Очистить поле, а затем ввести символы последовательно в поле. |
   | **Строка формата** | Некоторые типы параметров используют строку формата, которая информирует разъем 3270 о том, как преобразовывать текст с экрана в тип данных .NET: <p><p>- **DateTime**: Строка формата DateTime следует [за строками даты и формата времени .NET.](https://docs.microsoft.com/dotnet/standard/base-types/custom-date-and-time-format-strings) Например, дата `06/30/2019` использует строку `MM/dd/yyyy`формата. <p>- **Десятичная**: Строка десятичной формата использует [оговорку COBOL Picture.](https://www.ibm.com/support/knowledgecenter/SS6SG3_5.2.0/com.ibm.cobol52.ent.doc/PGandLR/ref/rlddepic.html) Например, в `100.35` номере `999V99`используется строка формата. |
   |||

## <a name="save-and-view-metadata"></a>Сохранение и просмотр метаданных

После определения метода, но перед тестированием метода, сохраните всю информацию, которую вы определили до сих пор, в файл RAP (.rap).
Вы можете сохранить этот файл RAP в любое время в любом режиме. Инструмент проектирования также включает в себя образец RAP файл, который вы можете открыть и просмотреть, просматривая папку установки инструмента проектирования в этом месте и открытие "WoodgroveBank.rap" файл:

`..\Program Files\Microsoft Host Integration Server - 3270 Design Tool\SDK\WoodgroveBank.rap`

Однако, если вы попытаетесь сохранить изменения в примере файла RAP или создать файл HIDX из образца файла RAP, пока файл остается в папке установки инструмента проектирования, вы можете получить ошибку "отказ в доступе". По умолчанию инструмент проектирования устанавливается в папку «Файлы программы» без повышенных разрешений. Если вы ошибетесь, попробуйте одно из следующих решений:

* Копируйте образец файла в другом месте.
* Запустите инструмент проектирования в качестве администратора.
* Сделайте себя владельцем папки SDK.

## <a name="test-your-method"></a>Проверьте свой метод

1. Чтобы запустить метод против живого хоста, пока он находится в режиме Methods, нажмите клавишу F5 или из панели инструментов инструментария design tool, выберите **Run.**

   > [!TIP]
   > Вы можете изменить режимы в любое время. В меню **файла** выберите **режим,** а затем выберите выбранный режим.

1. Введите значения параметров и выберите **OK**.

1. Чтобы продолжить работу на следующем экране, выберите **Next**.

1. Когда вы закончите, выберите **Done**, который показывает значения параметра вывода.

<a name="add-metadata-integration-account"></a>

## <a name="generate-and-upload-hidx-file"></a>Создание и загрузка файла HIDX

Когда вы будете готовы, создайте файл HIDX, чтобы вы могли загрузить на свою учетную запись интеграции. Инструмент 3270 Design Tool создает файл HIDX в новом подфедере, где вы сохранили файл RAP.

1. На панели инструментов 3270 Design Tool выберите **Код generate.**

1. Перейдите в папку, содержащую файл RAP, и откройте субфолдер, созданный инструментом после создания файла HIDX. Подтвердите, что инструмент создал файл HIDX.

1. Войдите на [портал Azure](https://portal.azure.com)и найдите учетную запись интеграции.

1. Добавьте файл HIDX в качестве карты в свою учетную запись интеграции, [проследя за этими аналогичными шагами для добавления карт,](../logic-apps/logic-apps-enterprise-integration-liquid-transform.md)но при выборе типа карты выберите **HIDX.**

Позже, когда вы впервые добавите действие IBM 3270 в логическое приложение, вам будет предложено создать соединение между приложением логики и принимающим сервером, предоставляя информацию о подключении, например имена для учетной записи интеграции и сервера-хозяина . После создания соединения можно выбрать ранее добавленный файл HIDX, метод выполнения и параметры для использования.

Когда вы закончите все эти шаги, вы можете использовать действие, которое вы создаете в приложении логики для подключения к вашему мэйнфрейму IBM, дисковые экраны для вашего приложения, ввод данных, результаты возврата и так далее. Вы также можете продолжить добавление других действий в логическое приложение для интеграции с другими приложениями, службами и системами.
