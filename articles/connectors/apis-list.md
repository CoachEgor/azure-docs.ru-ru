# <a name="connectors-for-azure-logic-apps"></a>Соединители для Azure Logic Apps

Соединители обеспечивают быстрый доступ из Azure Logic Apps к событиям, данным и действиям в других приложениях, службах, системах, протоколах, а также на других платформах. Используя соединители в приложениях логики, вы можете расширить возможности своих облачных и локальных приложений, чтобы выполнять задачи с создаваемыми и существующими данными.

В то время как Logic Apps предлагает [сотни разъемов,](https://docs.microsoft.com/connectors)эта статья описывает *популярные и более часто используемые* разъемы, которые успешно используются тысячами приложений и миллионами казней для обработки данных и информации. Чтобы найти полный список разъемов и справочную информацию каждого разъема, такую как триггеры, действия и ограничения, просмотрите страницы ссылки на разъем ы в [обзоре Connectors.](https://docs.microsoft.com/connectors) Кроме того, узнать больше о [триггеров и действий](#triggers-actions), [Логика Apps модели ценообразования](../logic-apps/logic-apps-pricing.md), и [логика Apps цен детали](https://azure.microsoft.com/pricing/details/logic-apps/).

> [!TIP]
> Для интеграции с службой или API, у которых нет разъема, можно либо напрямую вызвать службу по протоколу, такому как HTTP, или создать [пользовательский разъем.](#custom)

## <a name="connector-types"></a>Типы соединительных соединений

Соединительы доступны как встроенные триггеры и действия или как управляемые разъемы.

<a name="built-in"></a>

* [**Встроенные**](#built-ins)триггеры и действия являются «родными» для приложений Azure Logic Apps и помогают выполнять эти задачи для логических приложений:

  * Запуск по пользовательским и расширенным графикам.

  * Организуйте и контролируйте рабочий процесс приложения логики, например, циклы и условия, а также для работы с переменными и операциями данных.

  * Общайтесь с другими конечными точками.

  * Получайте и отвечайте на запросы.

  * Вызов функций Azure, Azure API Apps (Web Apps), собственные API, управляемые и опубликованные с помощью Управления API Azure, и вложенные логические приложения, которые могут получать запросы.

<a name="managed-connectors"></a>

* [**Управляемые разъемы:**](#managed-api-connectors)развернутые и управляемые корпорацией Майкрософт, эти разъемы обеспечивают триггеры и действия для доступа к облачным службам, предприимжению в новых системах или обоим, включая Office 365, Azure Blob Storage, S'L Server, Dynamics, Salesforce, SharePoint и многое другое. Некоторые разъемы специально поддерживают сценарии коммуникации между бизнесом (B2B) и требуют [интеграционной учетной записи,](../logic-apps/logic-apps-enterprise-integration-create-integration-account.md) связанной с вашим логическим приложением. Прежде чем использовать некоторые разъемы, возможно, придется сначала создать соединения, которые управляются приложениями Azure Logic Apps.

  Например, если вы используете Microsoft BizTalk Server, ваши логические приложения могут подключаться к вашему BizTalk Server и общаться с вашим сервером BizTalk с помощью [предварительного разъема BizTalk Server.](#on-premises-connectors) Затем вы можете расширить или выполнить операции, подобные BizTalk, в своих приложениях логики с помощью [соединителей для учетной записи интеграции](#integration-account-connectors).

  Соединения классифицируются как стандартные или корпоративные. [Корпоративные разъемы](#enterprise-connectors) обеспечивают доступ к корпоративным системам, таким как SAP, IBM M и IBM 3270, за дополнительную плату. Чтобы определить, является ли разъем стандартным или корпоративным, см. технические детали на странице ссылки каждого разъема под [обзором Connectors.](https://docs.microsoft.com/connectors)

  Вы также можете определить разъемы с помощью этих категорий, хотя некоторые разъемы могут пересекать несколько категорий. Например, SAP является разъемом Enterprise и предприимчивым разъемом:

  |   |   |
  |---|---|
  | [**Управляемые разъемы**](#managed-api-connectors) | Создавайте приложения логики, которые используют такие службы, как хранилище BLOB-объектов, Office 365, Dynamics, Power BI, OneDrive, Salesforce, SharePoint Online и многие другие. |
  | [**Локальные соединители**](#on-premises-connectors) | После установки и настройки [локального шлюза данных][gateway-doc] эти соединители помогают приложениям логики получать доступ к локальным системам, таким как SQL Server, SharePoint Server, Oracle DB, общие файловые ресурсы и другие. |
  | [**Соединители для учетной записи интеграции**](#integration-account-connectors) | Доступные при создании и оплате учетной записи интеграции. Эти соединители преобразуют и проверяют XML, кодируют и декодируют неструктурированные файлы, а также обрабатывают сообщения типа "бизнес-бизнес" (B2B) с помощью протоколов AS2, EDIFACT и X12. |

### <a name="connect-from-an-integration-service-environment"></a>Подключение из среды служб интеграции

Для логических приложений, которым необходим прямой доступ к ресурсам в виртуальной сети Azure, можно создать изолированную [среду служб ы интеграции (ISE),](../logic-apps/connect-virtual-network-vnet-isolated-environment-overview.md) где можно создавать, развертывать и запускать логические приложения на выделенных ресурсах. В Logic App Designer при просмотре разъемов, которые вы хотите использовать для логических приложений в ISE, метка **CORE** отображается на встроенных триггерах и действиях, в то время как метка **ISE** отображается на некоторых разъемах:

* **CORE**: Встроенные триггеры и действия с этой меткой работают в том же ISE, что и ваши логические приложения, например:

  ![Пример разъема ISE](./media/apis-list/example-core-connector.png)

* **ISE**: Управляемые разъемы с этой меткой работают в том же ISE, что и ваши логические приложения, например:

  ![Пример разъема ISE](./media/apis-list/example-ise-connector.png)

  Если у вас есть предприимчная система, подключенная к виртуальной сети Azure, ISE позволяет вашим логическим приложениям получить прямой доступ к этой системе без [предварительного шлюза данных.](../logic-apps/logic-apps-gateway-connection.md) Вместо этого, вы можете использовать разъем **ISE** этой системы, если это возможно, действие HTTP, или [пользовательский разъем.](#custom) Для штатных систем, не имеющие разъемов **ISE,** используйте шлюз данных. Чтобы просмотреть доступные разъемы ISE, [см.](#ise-connectors)

* Все остальные разъемы без метки **CORE** или **ISE,** которые вы можете продолжать использовать, работают в глобальной, мультитенантной службе логических приложений, например:

  ![Пример мультитенантного разъема](./media/apis-list/example-multi-tenant-connector.png)

Логические приложения, которые работают в ISE и их разъемы, независимо от того, где эти разъемы работают, следуют плану фиксированного ценообразования по сравнению с планом ценообразования на основе потребления. Для получения дополнительной информации смотрите следующие страницы:

* [Модель ценообразования приложений логики](../logic-apps/logic-apps-pricing.md)
* [Детали ценообразования Logic Apps](https://azure.microsoft.com/pricing/details/logic-apps/)
* [Подключение к виртуальным сетям Azure из Azure Logic Apps](../logic-apps/connect-virtual-network-vnet-isolated-environment.md)

<a name="triggers-actions"></a>

## <a name="triggers-and-action-types"></a>Триггеры и типы действий

Соединения могут обеспечить *триггеры,* *действия*или оба. Триггер *trigger* является первым шагом в любом приложении логики, обычно указывая событие, которое запускает триггер и запускает приложение логики. Например, разъем FTP имеет триггер, который запускает приложение логики ", когда файл добавлен или изменен". Некоторые триггеры регулярно проверяют указанное событие или данные, а затем стреляют, когда они обнаруживают указанное событие или данные. Другие триггеры ждут, но стреляйте мгновенно, когда происходит определенное событие или когда новые данные доступны. Триггеры также передают любые необходимые данные в приложение логики. Приложение логики может читать и использовать эти данные на протяжении всего рабочего процесса. Например, разъем Twitter имеет триггер "Когда новый твит размещен", который передает содержимое твита в рабочий процесс вашего приложения логики.

После срабатывания триггера приложения Azure Logic Apps создают экземпляр приложения логики и запускают *действия* в рабочем процессе приложения логики. Действия являются шагами, которые следуют за триггером и выполняют задачи в рабочем процессе приложения логики. Например, можно создать логическое приложение, которое получает данные клиентов из базы данных S'L и обрабатывает эти данные в последующих действиях.

Ниже приведены общие типы триггеров, которые предоставляют azure Logic Apps:

* *Повторяющийся триггер*: Этот триггер выполняется по определенному графику и не тесно связан с определенной службой или системой.

* *Триггер опроса*: Этот триггер регулярно опрашивает конкретную службу или систему на основе указанного графика, проверки новых данных или того, произошло ли конкретное событие. Если доступны новые данные или произошло конкретное событие, триггер создает и запускает новый экземпляр приложения логики, которое теперь может использовать данные, передаваемые в качестве ввода.

* *Нажмите триггер*: Этот триггер ждет и слушает новые данные или событие произойдет. Когда новые данные доступны или когда происходит событие, триггер создает и запускает новый экземпляр приложения логики, которое теперь может использовать данные, которые передаются в качестве ввода.

<a name="connections"></a>

## <a name="connector-configuration"></a>Конфигурация соединителя

Триггеры и действия каждого разъема обеспечивают свои собственные свойства для настройки. Многие разъемы также требуют, чтобы сначала создать *подключение* к целевой службе или системе и предоставить учетные данные аутентификации или другие детали конфигурации, прежде чем вы сможете использовать триггер или действие в приложении логики. Например, необходимо авторизовать подключение к учетной записи Twitter для доступа к данным или для публикации от вашего имени.

Для разъемов, которые используют Azure Active Directory (Azure AD) OAuth, создание соединения означает подписание в службе, например Office 365, Salesforce или GitHub, где ваш токен доступа [зашифрован](../security/fundamentals/encryption-overview.md) и надежно хранится в секретном хранилище Azure. Другие разъемы, такие как FTP и S'L, требуют подключения с деталями конфигурации, такими как адрес сервера, имя пользователя и пароль. Эти сведения о конфигурации подключения также безопасно хранятся в зашифрованном виде. Подробнее о [шифровании читайте в azure](../security/fundamentals/encryption-overview.md).

Соединения могут получить доступ к целевой службе или системе до тех пор, пока эта служба или система позволяет. Для служб, которые используют подключения Azure AD OAuth, такие как Office 365 и Dynamics, Приложения Логики Azure обновляют токены доступа на неопределенный срок. Другие службы могут иметь ограничения на то, как долго приложения Azure Logic Могут использовать токен без обновления. Как правило, некоторые действия аннулируют все маркеры доступа, например изменение пароля.

<a name="custom"></a>

## <a name="custom-apis-and-connectors"></a>Пользовательские API и соединители

Чтобы вызывать API-интерфейсы, которые запускают пользовательский код или недоступны в качестве соединителей, можно расширить платформу Logic Apps, [создавая настраиваемые приложения API](../logic-apps/logic-apps-create-api-app.md). Кроме того, вы можете [создавать пользовательские соединители](../logic-apps/custom-connector-overview.md) для *любых* API на основе REST или SOAP, которые делают эти API доступными для любого приложения логики в вашей подписке Azure. Чтобы пользовательские приложения API или соединители были доступными для любого пользователя в Azure, вы можете [отправить соединители для сертификации Майкрософт](../logic-apps/custom-connector-submit-certification.md).

> [!NOTE]
> Логические приложения, развертываемые и запускаемые в [среде интеграционных служб (ISE),](../logic-apps/connect-virtual-network-vnet-isolated-environment-overview.md) могут напрямую получать доступ к ресурсам в виртуальной сети Azure. Если у вас есть пользовательские разъемы, которые требуют предварительного шлюза данных, и вы создали эти разъемы за пределами ISE, логические приложения в ISE также могут использовать эти разъемы.
>
> Пользовательские разъемы, созданные в ISE, не работают с шлюзом данных. Однако эти разъемы могут напрямую получить доступ к источникам данных, подключенным к виртуальной сети Azure, принимающей ISE. Таким образом, логические приложения в ISE, скорее всего, не нуждаются в шлюзе данных при общении с этими ресурсами.
>
> Для получения дополнительной информации о создании ISEs [см.](../logic-apps/connect-virtual-network-vnet-isolated-environment.md)
