# <a name="connectors-for-azure-logic-apps"></a>Соединители для Azure Logic Apps

Соединители обеспечивают быстрый доступ из Azure Logic Apps для событий, данных и действия для других приложений, систем, протоколы и службы платформ. С помощью соединителей в logic apps позволяет расширить возможности для облачных и локальных приложений для выполнения задач с данными, уже есть.

Хотя система Logic Apps предлагает [свыше 200 соединителей](https://docs.microsoft.com/connectors), в этой статье описаны популярные соединители, чаще всего используемые во многих приложениях и выполнениях для обработки данных и информации. Полный список соединителей и каждый соединитель справочные сведения, такие как триггеры, действия и ограничения, см. в статье на страницах справочника соединитель в разделе [Общие сведения о соединителях](https://docs.microsoft.com/connectors). Кроме того, Дополнительные сведения о [триггеры и действия](#triggers-actions), [модель ценообразования приложений логики](../logic-apps/logic-apps-pricing.md), и [Logic Apps, сведения о ценах на](https://azure.microsoft.com/pricing/details/logic-apps/). 

> [!NOTE]
> Для интеграции со службой или API, который не имеет соединителя, можно непосредственно вызывать службу с помощью протокола, например HTTP или создать [Настраиваемый соединитель](#custom).

Соединители доступны как встроенные триггеры и действия или управляемые соединители.

* [**Встроенные объекты**](#built-ins): Эти встроенные триггеры и действия «native» для Azure Logic Apps и помогут создать приложения логики, которые работают на пользовательские расписания, взаимодействовать с другими конечными точками, получать и реагировать на запросы и вызова функции Azure, приложения API Azure (веб-приложения), собственные интерфейсы API управляемые, так и опубликованные с помощью управления API Azure и вложенных приложений логики, которые могут получать запросы. Кроме того, с помощью встроенных действий можно организовать и контролировать рабочий процесс приложения логики, а также работать с данными.

  > [!NOTE]
  > Приложения логики в [среда службы интеграции (ISE)](../logic-apps/connect-virtual-network-vnet-isolated-environment-overview.md) могут напрямую обращаться к ресурсам виртуальной сети Azure.
  > Если вы используете интегрированную среду Сценариев, встроенных триггеров и действий с параметром **Core** метку, выполните в одной интегрированной среде Сценариев, как приложения логики. Logic apps, триггеры, встроенные и встроенные действия, которые выполняются в интегрированной среде Сценариев использования ценовой план отличается от тарифный план с оплатой.
  >
  > Дополнительные сведения о создании ISEs см. в разделе [подключение к виртуальным сетям Azure из Azure Logic Apps](../logic-apps/connect-virtual-network-vnet-isolated-environment.md#create-logic-apps-environment). 
  > Дополнительные сведения о ценах см. в разделе [модель ценообразования приложений логики](../logic-apps/logic-apps-pricing.md).

* **Управляемые соединители.** Развертываются и управляются корпорацией Майкрософт, эти соединители предоставляют триггеры и действия для доступа к облачным службам, в локальных системах или оба, включая Office 365, хранилище BLOB-объектов Azure, SQL Server, Dynamics, Salesforce, SharePoint и многое другое. Некоторые соединители специально поддерживают сценарии взаимодействия предприятие предприятие (B2B) и требуют [учетной записи интеграции](../logic-apps/logic-apps-enterprise-integration-create-integration-account.md) , связанный с вашим приложением логики. Прежде чем использовать некоторые соединители, необходимо сначала создать подключения, которые управляются Azure Logic Apps. 

  Например, при использовании Microsoft BizTalk Server, приложения логики можно подключения и взаимодействия с сервером BizTalk с помощью [BizTalk Server локальный соединитель](#on-premises-connectors). 
  Затем вы можете расширить или выполнить операции, подобные BizTalk, в своих приложениях логики с помощью [соединителей для учетной записи интеграции](#integration-account-connectors).

  Соединители подразделяются на выпуск Standard или Enterprise. 
  [Корпоративные соединители](#enterprise-connectors) предоставляют доступ к корпоративным системам, таких как SAP, IBM MQ и IBM 3270 за дополнительную плату. Чтобы определить, является ли соединитель Standard или Enterprise, см. в разделе технических сведений в каждый соединитель справочной странице в разделе [Общие сведения о соединителях](https://docs.microsoft.com/connectors). 

  Соединители также можно определить с помощью этих категорий, несмотря на то, что некоторые соединители может пересекать несколько категорий.

<a name="on-premises-connectors"></a>

## <a name="on-premises-connectors"></a>Локальные соединители

<a name="triggers-actions"></a>

## <a name="triggers-and-actions---more-info"></a>Триггеры и действия — Дополнительные сведения

Соединители можно предоставить *триггеры*, *действия*, или оба. Объект *триггера* является первым шагом в любом приложении логики, обычно указание событие, которое активирует триггер и запускает выполнение приложения логики. Например соединитель FTP имеет триггер, который запускает приложение логики «при добавлении или изменении файла». Некоторые триггеры Регулярная проверка для указанного события или данные, а затем обработать при обнаружении указанного события или данные. Другие триггеры ожидания, но срабатывают мгновенной когда происходит определенное событие или когда доступны новые данные. Триггеры также передавать необходимые данные для приложения логики. Приложение логики можно читать и использовать эти данные через рабочий процесс.
Например, соединитель Twitter предоставляет триггера «При публикации нового твита», который передает твита содержимого в рабочий процесс приложения логики. 

После срабатывания триггера Azure Logic Apps создает экземпляр приложения логики и запускает выполнение *действия* в рабочем процессе приложения логики. Действия — действия, выполните триггера и выполнения задач в рабочем процессе приложения логики. Например можно создать приложение логики, которое получает данные клиентов из базы данных SQL и обработки этих данных в последующих действиях. 

Ниже приведены общие виды триггеры, которые предоставляет Azure Logic Apps.

* *Триггер повторения*: Этот триггер выполняется по указанному расписанию и не тесно связана с конкретной службы или системы.

* *Опрашивающий триггер*: Этот триггер регулярно опрашивает определенной службы или системы, на основе заданного расписания, проверка наличия новых данных или ли определенное событие произошло. Если новые данные или произошло определенное событие, триггер создает и запускает новый экземпляр приложения логики, который теперь можно использовать данные, передается в качестве входных данных.

* *Извещающий триггер*: Этот триггер ожидает и ожидает передачи данных для новых данных или же планируемое событие. Когда доступны новые данные, или когда происходит событие, триггер создает и запускает новый экземпляр приложения логики, который теперь можно использовать данные, передается в качестве входных данных.

<a name="custom"></a>

## <a name="connector-configuration"></a>Настройка соединителя

Триггеры и действия для каждого соединителя предоставляют свои собственные свойства для настройки. Множество соединителей также требуется, необходимо сначала создать *подключения* для целевой службы или системы и укажите учетные данные проверки подлинности или другие данные конфигурации, прежде чем триггер или действие, можно использовать в приложении логики. Например необходимо авторизовать подключение к учетной записи Twitter для доступа к данным или к публикации от вашего имени. 

Для соединителей, использующих OAuth создав подключение значит войти в службу, например Office 365, Salesforce или GitHub, где маркер доступа шифруется и безопасно хранится в секретном хранилище Azure. Других соединителей, таких как FTP и SQL, требуется подключение, которое содержит сведения о конфигурации, такие как адрес сервера, имя пользователя и пароль. Эти сведения о конфигурации подключения также безопасно хранятся в зашифрованном виде. 

Подключения можно получить доступ к целевой службы или системы для до тех пор, пока этой службы или системы позволяет. Для служб, использующих подключений OAuth Azure Active Directory (AD), например Office 365 и Dynamics Azure Logic Apps обновляет маркеры доступа бессрочно. Другие службы могут иметь ограничения на время в Azure Logic Apps может применять маркер без обновления. Как правило некоторые действия недействительными все маркеры доступа, такие как смена пароля.

<a name="custom"></a>

## <a name="custom-apis-and-connectors"></a>Пользовательские API и соединители

Чтобы вызывать API-интерфейсы, которые запускают пользовательский код или недоступны в качестве соединителей, можно расширить платформу Logic Apps, [создавая настраиваемые приложения API](../logic-apps/logic-apps-create-api-app.md). Кроме того, вы можете [создавать пользовательские соединители](../logic-apps/custom-connector-overview.md) для *любых* API на основе REST или SOAP, которые делают эти API доступными для любого приложения логики в вашей подписке Azure.
Чтобы пользовательские приложения API или соединители были доступными для любого пользователя в Azure, вы можете [отправить соединители для сертификации Майкрософт](../logic-apps/custom-connector-submit-certification.md).

> [!NOTE]
> Приложения логики в [среда службы интеграции (ISE)](../logic-apps/connect-virtual-network-vnet-isolated-environment-overview.md) могут напрямую обращаться к ресурсам виртуальной сети Azure.
> Если у вас есть настраиваемые соединители, которые требуется локальный шлюз данных, и вы создали эти соединители за пределами интегрированную среду Сценариев, эти соединители можно также использовать в приложениях логики в интегрированную среду Сценариев.
>
> Настраиваемые соединители, созданные в интегрированную среду Сценариев не работают с локальным шлюзом данных. Тем не менее эти соединители можно непосредственно получить доступ к локальным источникам данных, подключенных к виртуальной сети Azure размещение в интегрированную среду Сценариев. Таким образом приложение логики в интегрированную среду Сценариев скорее не нуждается шлюз данных при взаимодействии с этими ресурсами.
>
> Дополнительные сведения о создании ISEs см. в разделе [подключение к виртуальным сетям Azure из Azure Logic Apps](../logic-apps/connect-virtual-network-vnet-isolated-environment.md#create-logic-apps-environment).
