# <a name="data-security-in-azure-australia"></a>Безопасность данных в Azure для Австралии

К ключевым принципам обеспечения безопасности данных клиента относится:

* защита данных с помощью шифрования;
* управление секретами;
* Запрещение доступа к данным

## <a name="encrypting-your-data"></a>Шифрование данных

Шифрование данных может быть применено на уровне диска (при хранении), в базах данных (неактивных и транзитных), в приложениях (транзитных) и в сети (транзитном). Существует несколько способов достижения шифрования в Azure.

|Служба или функция|Описание|
|---|---|
|Шифрование службы хранения|Шифрование службы хранилища Azure осуществляется на уровне учетной записи хранения. При записи в службу хранилища Azure для блочных и страничных BLOB-объектов выполняется автоматическое шифрование. При считывании данных они расшифровываются службой хранилища Azure перед возвратом. Используйте SSE для защиты данных без необходимости изменения или добавления кода в приложения.|
|Шифрование диска Azure|Шифрование дисков Azure обеспечивает шифрование дисков ОС и дисков данных, используемых виртуальной машиной Azure. Интеграция с хранилищем ключей Azure позволяет получить контроль и управлять ключами шифрования дисков.|
|Шифрование на уровне приложения на стороне клиента|Шифрование на стороне клиента встроено в клиентские библиотеки хранилища Java и .NET, которые могут использовать Azure Key Vault API, упрощая реализацию. Используйте Azure Key Vault, чтобы получить доступ к секретам в Azure Key Vault для конкретных пользователей, использующих Azure Active Directory.|
|Шифрование при передаче|Базовое шифрование, доступное для подключения к Azure Австралии, поддерживает протокол TLS 1,2 и сертификаты X. 509. Алгоритмы шифрования FIPS 140-2 уровня 1 также используются для сетевых подключений инфраструктуры между центрами обработки данных Azure Австралии.  Windows Server 2016, Windows 10, Windows Server 2012 R2, Windows 8.1 и общие файловые ресурсы Azure могут использовать SMB 3,0 для шифрования между виртуальной машиной и общей папкой. Используйте шифрование на стороне клиента для шифрования данных перед их передачей в хранилище в клиентском приложении, а также для расшифровки данных после их передачи из хранилища.|
|Виртуальные машины IaaS|Используйте шифрование дисков Azure. Вы можете включить шифрование службы хранения для шифрования файлов VHD, используемых для резервирования этих дисков в службе хранения Azure, но при этом шифроваться будут только данные, записываемые с этого момента. Это означает, что если вы создадите виртуальную машину, а затем включите шифрование службы хранения в учетной записи хранения, содержащей файл VHD, шифроваться будут только изменения, но не исходный файл VHD.|
|Шифрование на стороне клиента|Это наиболее безопасный метод шифрования данных, так как он шифруется перед передачей и шифрует неактивных данные. Однако оно требует добавления кода в приложения, использующие хранилище, что может быть нежелательным. В таких случаях можно использовать протокол HTTPS для передачи данных и Шифрование службы хранилища для шифрования неактивных данных. Шифрование на стороне клиента также требует больше нагрузки на клиент. это необходимо учитывать в плане масштабируемости, особенно при шифровании и передаче больших объемов данных.|
|

Дополнительные сведения о параметрах шифрования в Azure см. в разделе с [руководством по безопасности хранилища](https://docs.microsoft.com/azure/storage/storage-security-guide).

## <a name="protecting-data-by-managing-secrets"></a>Защита данных путем управления секретами

Безопасное управление ключами очень важно для защиты данных в облаке. Клиенты должны стремиться упростить управление ключами и поддерживать контроль над ключами, которые используются облачными приложениями и службами для шифрования данных.

### <a name="managing-secrets"></a>управление секретами;

* Чтобы свести к минимуму риск доступа к секретам, представленным в жестко заданных файлах конфигурации, скриптах и в исходном коде, можно использовать хранилище ключей. Хранилище ключей Azure шифрует ключи (например, ключи шифрования для шифрования дисков Azure) и секреты (например, пароли), сохраняя их в аппаратных модулях безопасности, отвечающих стандартам FIPS 140-2 уровня 2. Чтобы обеспечить более высокий уровень защиты, ключи можно импортировать или создать в аппаратных модулях безопасности.
* Код приложения и шаблоны должны содержать только URI-ссылки на секреты (что означает, что фактические секреты не находятся в коде, конфигурации или репозиториях исходного кода). Это позволяет предотвращать основные фишинговые атаки на внутренние или внешние репозитории, например с использованием ботов-собирателей на GitHub.
* В хранилище ключей можно использовать надежные средства управления RBAC. Если доверенный оператор уходит из компании или переходит в другую группу, ему должен быть запрещен доступ к секретам.  

Дополнительные сведения см. в разделе [Azure Key Vault](azure-key-vault.md)

## <a name="isolation-to-restrict-data-access"></a>изоляция для ограничения доступа к данным.

Для обеспечения изоляции применяются границы, сегментация и контейнеры, чтобы предоставлять доступ к данным только авторизованным пользователям, службам и приложениям. Например, разделение между клиентами является основным механизмом безопасности для облачных платформ с несколькими клиентами, таких как Microsoft Azure. Логическая изоляция позволяет предотвратить вмешательство одного клиента в операции другого.

#### <a name="per-customer-isolation"></a>Изоляция по клиентам

Azure реализует контроль доступа к сети и разделение через изоляцию VLAN уровня 2, списки управления доступом, подсистемы балансировки нагрузки и IP-фильтры.

Клиенты могут изолировать ресурсы в подписках, группах ресурсов, виртуальных сетях и подсетях.

Дополнительные сведения об изоляции в Microsoft Azure см. в разделе [изоляция в общедоступном облаке Azure](../security/fundamentals/isolation-choices.md).
