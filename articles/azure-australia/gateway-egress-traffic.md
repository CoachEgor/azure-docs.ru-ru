# <a name="controlling-egress-traffic-in-azure-australia"></a>Управление трафиком исходящего трафика в Azure Австралии

Основным компонентом защиты систем ICT является управление сетевым трафиком. Возможность обмена данными только с трафиком, необходимым для функционирования системы, снижает вероятность компрометации. Видимость и контроль над внешними системами, с которыми ваши приложения и службы взаимодействуют, помогает обнаруживать скомпрометированные системы, а также пытаться успешно или утечка данные. В этой статье приводятся сведения о том, как исходящий сетевой трафик работает в Azure и предоставляет рекомендации по реализации средств управления безопасностью сети для подключенной к Интернету системы, которая соответствует кибератак центру безопасности Австралии (ACSC). Руководство для потребителей и цель руководства по информационной безопасности ACSC (ISM).

## <a name="architecture"></a>Архитектура

Для надлежащего управления трафиком исходящего трафика при проектировании и реализации сетевой безопасности необходимо понять, как исходящий сетевой трафик работает в Azure как в IaaS, так и в PaaS. В этом разделе приводятся общие сведения о том, как обрабатываются исходящий трафик, создаваемый ресурсом, размещенным в Azure, а также элементы управления безопасностью, доступные для ограничения и управления этим трафиком.

### <a name="architecture-components"></a>Компоненты архитектуры

Схема архитектуры, показанная здесь, описывает возможные пути, которые может выполнять сетевой трафик при выходе из системы, развернутой в подсети в виртуальной сети. Трафик в виртуальной сети управляется и регулируется на уровне подсети, при этом правила маршрутизации и безопасности применяются к ресурсам, содержащимся в. Компоненты, связанные с исходящим трафиком, делятся на системы, действующие маршруты, типы следующего прыжка, элементы безопасности и исходящие данные PaaS.

### <a name="systems"></a>X64

Системы — это ресурсы Azure и связанные компоненты, которые создают исходящий трафик в IP-подсети, которая является частью виртуальной сети.

| Компонент | Описание |
| --- | ---|
|Виртуальная сеть (VNet) | Виртуальная сеть — это базовый ресурс в Azure, предоставляющий платформу и границу для развертывания ресурсов и включения взаимодействия. Виртуальная сеть находится в регионе Azure и определяет пространство IP-адресов и границы маршрутизации для ресурсов, интегрированных с виртуальной сетью, таких как виртуальные машины.|
|Subnet | Подсеть — это диапазон IP-адресов, который создается в виртуальной сети. Для сегментации сети можно создать несколько подсетей в виртуальной сети.|
|Сетевой интерфейс| Сетевой интерфейс — это ресурс, который существует в Azure. Он присоединяется к виртуальной машине и назначает частный IP-адрес, не поддерживающий маршрутизацию в Интернете, из подсети, с которой он связан. Этот IP-адрес динамически или статически назначается с помощью Azure Resource Manager.|
|Общедоступные IP-адреса| Общедоступный IP-адрес — это ресурс, который резервирует один из общедоступных маршрутизируемых IP-адресов Майкрософт из указанного региона для использования в виртуальной сети. Он может быть связан с конкретным сетевым интерфейсом или ресурсом PaaS, который позволяет ресурсу взаимодействовать с Интернетом, ExpressRoute и другими системами PaaS.|
|

### <a name="routes"></a>Маршруты

Путь, используемый для исходящего трафика, зависит от действующих маршрутов для этого ресурса, который является результирующим набором маршрутов, определяемых сочетанием маршрутов из всех возможных источников и применением логики маршрутизации Azure.

| Компонент | Описание |
|--- | ---|
|Системные маршруты| Azure автоматически создает системные маршруты и назначает их каждой подсети в виртуальной сети. Системные маршруты не могут быть созданы или удалены, но некоторые из них можно переопределить с помощью настраиваемых маршрутов. Azure создает системные маршруты по умолчанию для каждой подсети и добавляет дополнительные необязательные маршруты по умолчанию в определенные подсети или каждую подсеть, когда используются определенные возможности Azure.|
|Конечные точки службы| Конечные точки службы обеспечивают прямое, частное подключение из подсети к определенной возможности PaaS. Конечные точки службы, доступные только для подмножества возможностей PaaS, обеспечивают повышенную производительность и безопасность для ресурсов в виртуальной сети, обращающейся к PaaS.|
|таблицы маршрутов;| Таблица маршрутов — это ресурс в Azure, который можно создать для указания определяемых пользователем маршрутов (определяемые пользователем маршруты), которые могут дополнять или переопределять системные маршруты или маршруты для обучения через протокол BGP. Каждый UDR указывает сеть, маску сети и следующий прыжок. Таблица маршрутов может быть связана с подсетью, и одна и та же таблица маршрутов может быть связана с несколькими подсетями, но в подсети может быть только одна таблица маршрутов.|
|Протокол BGP| BGP — это протокол маршрутизации между автономными системами. Автономная система — это сеть или группа сетей под общим администрированием и общие политики маршрутизации. BGP используется для обмена данными маршрутизации между автономными системами. Протокол BGP можно интегрировать в виртуальные сети через шлюзы виртуальной сети.|
|

### <a name="next-hop-types-defined"></a>Типы следующего прыжка определены

Каждый маршрут в Azure включает диапазон сети и связанную маску подсети и следующий прыжок, который определяет способ обработки трафика.

Тип следующего прыжка | Описание
---- | ----
**Виртуальная сеть** | Направляет трафик между диапазонами адресов в адресном пространстве виртуальной сети. Azure создает маршрут с префиксом адреса, который соответствует каждому диапазону адресов, определенному в адресном пространстве виртуальной сети. Если в адресном пространстве виртуальной сети определено несколько диапазонов адресов, Azure создает отдельный маршрут для каждого диапазона адресов. Azure автоматически направляет трафик между подсетями в виртуальной сети с помощью маршрутов, созданных для каждого диапазона адресов.
**Пиринговая связь между виртуальными сетями** | При создании пиринга между двумя виртуальными сетями для каждого диапазона адресов каждой виртуальной сети добавляется маршрут к виртуальной сети, к которой он подсоединен. Трафик направляется между одноранговыми виртуальными сетями так же, как подсети в виртуальной сети.
**Шлюз виртуальной сети** | При добавлении шлюза виртуальной сети в виртуальную сеть добавляются один или несколько маршрутов с шлюзом виртуальной сети, указанным в качестве типа следующего прыжка. К ним относятся маршруты, настроенные в ресурсе шлюза локальной сети, и все маршруты, обученные через BGP.
**Виртуальное устройство** | Виртуальное устройство обычно запускает сетевое приложение, например брандмауэр. Виртуальное устройство допускает дополнительную обработку трафика, например фильтрацию, проверку или преобразование адресов. Каждый маршрут с типом прыжка виртуального устройства должен также указывать IP-адрес следующего прыжка.
**VirtualNetworkServiceEndpoint** | Общедоступные IP-адреса для конкретной службы добавляются в качестве маршрутов в подсеть со следующим прыжком VirtualNetworkServiceEndpoint при включении конечной точки службы. Конечные точки службы включены для отдельных служб в отдельных подсетях в виртуальной сети. Общедоступные IP-адреса служб Azure периодически изменяются. При изменении адресов Azure автоматически управляет ими в таблице маршрутов.
**Интернет**; | Трафик со следующим прыжком Интернета выполнит выход из виртуальной сети и автоматически преобразует его в общедоступный IP-адрес из динамического пула, доступного в соответствующем регионе Azure, или с помощью общедоступного IP-адреса, настроенного для этого ресурса. Если адрес назначения предназначен для одной из служб Azure, трафик направляется непосредственно в службу через магистральную сеть Azure вместо маршрутизации трафика в Интернет. Трафик между службами Azure не проходит через Интернет, независимо от того, в каком регионе Azure находится виртуальная сеть или в каком регионе развернута служба Azure.
**None** | Трафик со следующим прыжком "нет" удаляется. Azure создает системные маршруты по умолчанию для зарезервированных префиксов адресов без типа следующего прыжка. Маршруты со следующим прыжком "нет" можно также добавлять с помощью таблиц маршрутов, чтобы предотвратить маршрутизацию трафика в определенные сети.
|

### <a name="security-controls"></a>Средства управления безопасностью

Элемент управления | Описание
----- | -----
**Группы безопасности сети (группы безопасности сети)** | Группы безопасности сети управление трафиком в ресурсы виртуальной сети в Azure и из них. Группы безопасности сети применяются правила для разрешенных или запрещенных потоков трафика, включая трафик в Azure и между Azure и внешними сетями, такими как локальные или Интернет. Группы безопасности сети применяются к подсетям в виртуальной сети или к отдельным сетевым интерфейсам.
**Брандмауэр Azure** | Брандмауэр Azure — это управляемая облачная служба безопасности сети, которая защищает ресурсы виртуальной сети Azure. Брандмауэр разработан как служба с полным отслеживанием состояния со встроенной высокой доступностью и неограниченной облачной масштабируемостью. Брандмауэр Azure можно настроить с помощью традиционных правил фильтрации сети, основанных на IP-адресах, протоколах и портах, но также поддерживает фильтрацию на основе полных доменных имен (FQDN), тегов служб и встроенных средств анализа угроз.
**Сетевое виртуальное устройство (NVA)** | Сетевые виртуальные устройства — это носители виртуальных машин, которые могут предоставлять функции сети, безопасности и других функций в Azure. NVA поддерживает сетевые функции и службы в виде виртуальных машин в виртуальных сетях и развертываниях. NVA можно использовать для решения конкретных требований, интеграции с инструментами управления и эксплуатации, а также для обеспечения согласованности существующих продуктов. Azure поддерживает широкий перечень сторонних сетевых виртуальных устройств, включая брандмауэры веб-приложений (WAF), брандмауэры, маршрутизаторы и шлюзы, контроллеры доставки приложений (ADC) и оптимизаторы глобальной сети.
**Политики конечной точки службы (Предварительная версия)** | Политики конечной точки службы виртуальной сети позволяют фильтровать трафик виртуальной сети в службах Azure, разрешая только определенные ресурсы службы Azure, через конечные точки службы. Политики конечных точек предоставляют возможность детального контроля доступа трафика из виртуальной сети к службам Azure.
**Политика Azure** | Политика Azure — это служба в Azure для создания, назначения политик и управления ими. Эти политики используют правила для управления типами ресурсов, которые могут быть развернуты, и конфигурацией этих ресурсов. Политики можно использовать для принудительного обеспечения соответствия, предотвращая развертывание ресурсов, если они не соответствуют требованиям или могут использоваться для наблюдения за состоянием соответствия.
|

### <a name="paas-egress"></a>Исходящий трафик PaaS

Большинство ресурсов PaaS не создают исходящий трафик, но либо отвечают на входящие запросы (например, шлюз приложений, хранилище, базу данных SQL и т. д.), либо отправляют данные из других ресурсов (например, служебная шина и Azure Relay). Сетевая связь между ресурсами PaaS, такими как службы приложений в хранилище или базы данных SQL, управляется и содержится в Azure и защищена с помощью удостоверений и других элементов управления конфигурацией ресурсов, а не сегментации или разделения сети.

Ресурсы PaaS, развернутые в виртуальной сети, получают выделенные IP-адреса и подчиняются всем элементам управления маршрутизации и группы безопасности сети таким же образом, как и другие ресурсы в виртуальной сети. Ресурсы PaaS, которые не существуют в виртуальной сети, будут использовать пул IP-адресов, которые являются общими для всех экземпляров ресурса, которые либо публикуются в документации Майкрософт, либо могут быть определены с помощью Azure Resource Manager.

## <a name="general-guidance"></a>Общие рекомендации

Для проектирования и создания безопасных решений в Azure очень важно понимать и контролировать сетевой трафик, чтобы могли возникнуть только идентифицированные и созданные взаимосвязи. Цель данного руководства и конкретные рекомендации по компонентам в последующих разделах — описание средств и служб, которые можно применять для применения принципов, описанных в [разделе Защита ACSC: Реализация сегментации сети и разделения](https://acsc.gov.au/publications/protect/network_segmentation_segregation.htm) между рабочими нагрузками Azure. Сюда входит подробное описание создания виртуальной архитектуры для защиты ресурсов, когда невозможно применить те же традиционные физические и сетевые элементы управления, которые можно использовать в локальной среде.

### <a name="guidance"></a>Руководство

* Ограничение числа точек исходящего трафика для виртуальных сетей
* Переопределите системный маршрут по умолчанию для всех подсетей, которым не требуется прямая Исходящая связь с Интернетом.
* Разработайте и реализуйте полную архитектуру сети для обнаружения и управления всеми точками входящего и исходящего трафика в ресурсах Azure.
* Рассмотрите возможность использовать архитектуру основной и периферийной сети для виртуальных сетей, как описано в документации по виртуальному центру обработки данных (Microsoft Virtual Data Center).
* Применяйте продукты с встроенными функциями безопасности для исходящих подключений к Интернету (например, брандмауэр Azure, виртуальные сетевые устройства или веб-прокси).
* Используйте элементы управления удостоверениями, такие как доступ на основе ролей, условный доступ и многофакторная идентификация (MFA), чтобы ограничить привилегии конфигурации сети.
* Реализуйте блокировки, чтобы предотвратить изменение или удаление ключевых элементов конфигурации сети.
* Развертывание PaaS в интегрированной конфигурации виртуальной сети для повышения разделения и контроля
* Реализация ExpressRoute для подключения к локальным сетям
* Реализация VPN для интеграции с внешними сетями
* С помощью политики Azure можно ограничить регионы и ресурсы только теми, которые необходимы для работы функций системы.
* Использование политики Azure для принудительного применения базовой конфигурации безопасности для ресурсов
* Использование наблюдателя за сетями и Azure Monitor для ведения журнала, аудита и видимости сетевого трафика в Azure

### <a name="virtual-network-vnet"></a>Виртуальная сеть (VNet)

Виртуальные сети — это один из основных стандартных блоков для работы с сетью в Azure. Виртуальные сети определяют пространство IP-адресов и границы маршрутизации для использования в разных системах. Виртуальные сети делятся на подсети, а все подсети в виртуальной сети имеют прямой сетевой маршрут. Используя шлюзы виртуальной сети (ExpressRoute или VPN), системы в виртуальной сети могут интегрироваться с локальными и внешними средами, а также с помощью преобразования сетевых адресов Azure (NAT) и распределения общедоступных IP-адресов. системы могут Подключитесь к Интернету или другим регионам и службам Azure. Основные сведения о виртуальных сетях и связанных параметрах конфигурации и маршрутизации очень важны для понимания и контроля исходящего сетевого трафика.

Поскольку виртуальные сети формируют базовое адресное пространство и границы маршрутизации в Azure, их можно использовать в качестве основного стандартного блока сегментации и разделения сети.

## <a name="effective-routes"></a>Фактические маршруты

Действующие маршруты — это результирующий набор маршрутов, определяемый сочетанием системных маршрутов, конечных точек служб, таблиц маршрутов и протокола BGP и приложения логики маршрутизации Azure. Когда исходящий трафик передается из подсети, Azure выбирает маршрут на основе целевого IP-адреса, используя самый длинный алгоритм сопоставления префикса. Если несколько маршрутов содержат один и тот же префикс адреса, Azure выбирает тип маршрута на основе следующего приоритета:

1. определяемый пользователем маршрут;
2. маршрут BGP;
3. Системные маршруты

Важно отметить, что системные маршруты для трафика, связанного с виртуальной сетью, пирингами виртуальных сетей или конечными точками службы виртуальной сети, являются предпочтительными маршрутами, даже если маршруты BGP являются более конкретными.

Люди, участвующие в проектировании или реализации топологий маршрутизации в Azure, должны понимать, как Azure маршрутизирует трафик и разрабатывать архитектуру, которая распределяет необходимые функции систем с необходимой безопасностью и видимостью. Следует соблюдать осторожность при планировании среды, чтобы избежать чрезмерного вмешательства и исключений при маршрутизации, так как это повысит сложность и может привести к устранению неполадок и сбоям при поиске трудовых и трудоемких операций.

### <a name="service-endpoints"></a>Конечные точки службы

Включение [конечных точек службы](https://docs.microsoft.com/azure/virtual-network/virtual-network-service-endpoints-overview) в подсети предоставляет прямой путь взаимодействия с соответствующим ресурсом PaaS. Это обеспечивает повышенную производительность и безопасность за счет ограничения доступного пути взаимодействия только этой службе. Использование конечных точек службы приводит к вводу пути к утечка данных, так как конфигурация по умолчанию обеспечивает доступ ко всем экземплярам службы PaaS, а не к конкретным экземплярам, необходимым для приложения или системы.

Сущности Содружество должны оценивать риск, связанный с предоставлением прямого доступа к ресурсу PaaS, включая вероятность и последствия неверного использования пути.

Чтобы уменьшить потенциальные риски, связанные с конечными точками служб, реализуйте политики конечных точек службы там, где это возможно, или рассмотрите возможность включения конечных точек службы в брандмауэре Azure или подсети NVA и маршрутизации трафика из указанных подсетей, где дополнительно можно применять фильтрацию, мониторинг и проверку.

### <a name="border-gateway-protocol-bgp"></a>Протокол BGP

Протокол BGP может использоваться шлюзами виртуальной сети для динамического обмена данными маршрутизации с локальными или другими внешними сетями. BGP применяется к виртуальной сети при настройке через шлюз виртуальной сети ExpressRoute через частный пиринг ExpressRoute и при включении на VPN-шлюзе Azure.

Люди, участвующие в проектировании или реализации виртуальных сетей и шлюзов виртуальных сетей в Azure, должны иметь время, чтобы понять поведение и параметры конфигурации, доступные для BGP в Azure.

## <a name="next-hop-types"></a>Типы следующего прыжка

### <a name="virtual-network"></a>Виртуальная сеть

Маршруты со следующим прыжком виртуальной сети добавляются автоматически как системные маршруты, но также можно добавить в определяемые пользователем маршруты, чтобы направить трафик обратно в виртуальную сеть в экземплярах, в которых системный маршрут был переопределен.

### <a name="vnet-peering"></a>Пиринг VNet

Пиринг виртуальной сети обеспечивает связь между двумя разнородными виртуальными сетями. Настройка пиринга виртуальных сетей должна быть включена в каждой виртуальной сети, но виртуальные сети не обязательно должны находиться в одном регионе, подписке или связаны с одним и тем же клиентом Azure Active Directory (Azure AD).

При настройке пиринга между виртуальными сетями важно, чтобы пользователи, участвующие в проектировании или реализации пиринга виртуальных сетей, понимали четыре связанных параметра конфигурации и их применение к каждой стороне однорангового узла:

1. **Разрешить доступ к виртуальным сетям**: Выберите **включено** (по умолчанию), чтобы разрешить обмен данными между двумя виртуальными сетями. Если разрешить обмен данными между виртуальными сетями, ресурсы, подключенные к каждой из них, взаимодействуют между собой с той же пропускной способностью и задержкой, что и ресурсы, подключенные к одной виртуальной сети.
2. **Разрешить перенаправленный трафик**. Установите этот флажок, чтобы разрешить трафик, перенаправленный сетевым трафиком, который не был отправлен из виртуальной сети, для передачи в эту виртуальную сеть через пиринг. Этот параметр является фундаментальным для реализации топологии концентратора и периферийной сети.
3. **Разрешить транзит шлюзов.** Установите этот флажок, чтобы разрешить одноранговой виртуальной сети использовать шлюз виртуальной сети, подключенный к этой виртуальной сети. *Разрешение транзита шлюза* включено в виртуальной сети с ресурсом шлюза виртуальной сети, но применяется только в том случае, если в другой виртуальной сети включен параметр *использовать удаленные шлюзы* .
4. **Использовать удаленные шлюзы.** Установите этот флажок, чтобы разрешить передачу трафика из этой виртуальной сети через шлюз виртуальной сети, подключенный к виртуальной сети, с которой выполняется пиринг. *Используйте удаленные шлюзы* в виртуальной сети без шлюза виртуальной сети и применяются только в том случае, если в другой виртуальной сети включен параметр *Разрешить транзит шлюза* .

### <a name="virtual-network-gateway"></a>Шлюз виртуальной сети

Шлюзы виртуальной сети обеспечивают механизм интеграции виртуальных сетей с внешними сетями, такими как локальные среды, партнерские среды и другие облачные развертывания. Двумя типами шлюза виртуальной сети являются ExpressRoute и VPN.

#### <a name="expressroute-gateway"></a>Шлюз ExpressRoute

Шлюзы ExpressRoute предоставляют точку выхода из виртуальной сети в локальную среду и должны быть развернуты в соответствии с требованиями к безопасности, доступности, финансовым требованиям и производительности. Шлюзы ExpressRoute обеспечивают определенную пропускную способность сети и стоимость использования после развертывания. У виртуальных сетей может быть только один шлюз ExpressRoute, но он может быть подключен к нескольким каналам ExpressRoute и может использоваться несколькими виртуальными сетями через пиринг виртуальных сетей, что позволяет совместно использовать пропускную способность и возможность подключения. Следует соблюдать осторожность, чтобы настроить маршрутизацию между локальными средами и виртуальными сетями с помощью шлюзов ExpressRoute, чтобы обеспечить сквозное подключение с помощью известных, контролируемых точек исходящего трафика сети. Содружество сущности, использующие шлюз ExpressRoute через частный пиринг ExpressRoute, также должны развертывать сетевые виртуальные устройства (NVA), чтобы установить VPN-подключение к локальной среде для обеспечения соответствия с руководством потребителя ACSC.

Важно отметить, что шлюзы ExpressRoute имеют ограничения на диапазоны адресов, сообщества и другие конкретные элементы конфигурации, которыми обмениваются по протоколу BGP.

#### <a name="vpn-gateway"></a>VPN-шлюз

VPN-шлюз Azure предоставляет сетевую точку выхода из виртуальной сети в внешнюю сеть для безопасного подключения типа "сеть — сеть". VPN-шлюзы обеспечивают определенную пропускную способность сети и стоимость использования после развертывания. Сущности Содружество, в которых используется VPN-шлюз, должны убедиться, что она настроена в соответствии с руководством потребителя ACSC. Виртуальные сети могут иметь только один VPN-шлюз, но его можно настроить с помощью нескольких туннелей и использовать в нескольких виртуальных сетях с помощью пиринга виртуальной сети, что позволяет нескольким виртуальным сетям совместно использовать пропускную способность и возможность подключения. VPN-шлюзы можно установить через Интернет или через ExpressRoute через пиринг Майкрософт.

Следует соблюдать осторожность при создании маршрутов со следующим прыжком "нет", особенно при применении его к маршруту по умолчанию 0.0.0.0/0, так как это может привести к непредвиденным последствиям и может привести к устранению сложных и длительных проблем системы.

## <a name="security"></a>Безопасность

Реализация сегментации сети и элементов управления разделениями в IaaS и PaaS обеспечивается за счет обеспечения безопасности самих возможностей и путем реализации управляемых путей взаимодействия из систем, которые будут взаимодействовать с возможностями.

Проектирование и разработка решений в Azure — это процесс создания логической архитектуры для понимания, контроля и отслеживания сетевых ресурсов во всем присутствии Azure. Эта логическая архитектура — это программное обеспечение, определенное на платформе Azure, которое принимает место физической топологии сети, реализованной в традиционных сетевых средах.

Создаваемая логическая архитектура должна обеспечивать функциональность, необходимую для удобства использования, но также должна обеспечивать видимость и контроль, необходимые для обеспечения безопасности и целостности.

Достижение этого результата основано на реализации необходимых средств сегментации и разделения сети, а также в защите и применении топологии сети и реализации этих средств.

### <a name="network-security-groups-nsgs"></a>Группы безопасности сети (NSG)

Группы безопасности сети используются для указания входящего и исходящего трафика, разрешенного для подсети или определенного сетевого интерфейса. При настройке группы безопасности сети сущности Содружество должны использовать подход список разрешений, где правила настроены на разрешение необходимого трафика с правилом по умолчанию, настроенного на запрет всего трафика, который не соответствует конкретной инструкции разрешения. При планировании и настройке группы безопасности сети необходимо соблюдать осторожность, чтобы убедиться в том, что весь необходимый входящий и исходящий трафик захватывается соответствующим образом. Сюда входит определение и понимание всех диапазонов частных IP-адресов, заносящихся в виртуальных сетях и локальной среде, а также конкретных служб Майкрософт, таких как требования к управлению Azure Load Balancer и PaaS. Лица, участвующие в проектировании и реализации группы безопасности сети, должны также понимать использование тегов служб и групп безопасности приложений для создания детализированных, сервисных и зависящих от приложений правил безопасности.

Важно отметить, что конфигурация по умолчанию для NSG разрешает исходящий трафик по всем адресам в виртуальной сети и всем общедоступным IP-адресам.

### <a name="azure-policy"></a>Политики Azure

Политика Azure — это ключевой компонент для обеспечения целостности логической архитектуры среды Azure. В службах Azure предусмотрены различные службы и исходящие пути сетевого трафика. Очень важно, чтобы Содружество сущности осведомлены о ресурсах, существующих в их среде и доступных сетевых исходящих точках. Чтобы гарантировать, что в среде Azure не создаются несанкционированные сетевые исходящие точки, Содружествоные сущности должны использовать политику Azure для управления типами ресурсов, которые могут быть развернуты, и конфигурацией этих ресурсов. Практические примеры включают в себя ограниченные ресурсы только авторами и утверждены для использования и требуют добавления группы безопасности сети в подсети.

## <a name="paas-egress-capabilities"></a>Возможности исходящего трафика PaaS

Возможности PaaS предоставляют возможности для повышения функциональности и упрощения управления, но предоставляют сложности, необходимые для сегментации и разделения сети. Возможности PaaS обычно настраиваются с использованием общедоступных IP-адресов и доступны из Интернета.  Если вы используете возможности PaaS в системах и решениях, следует соблюдать осторожность, чтобы определять потоки связи между компонентами и создавать правила безопасности сети, чтобы разрешить только это взаимодействие. В рамках многофункционального подхода к безопасности возможности PaaS должны быть настроены с шифрованием, проверкой подлинности и соответствующими элементами управления доступом и разрешениями.  

### <a name="public-ip-for-paas"></a>Общедоступный IP-адрес для PaaS

Общедоступные IP-адреса для возможностей PaaS выделяются в зависимости от региона, в котором размещается или развертывается служба. Понимание распределения общедоступных IP-адресов и регионов требуется, если вы собираетесь создать соответствующие правила безопасности сети и топологию маршрутизации для сегментации сети и разделения, охватывающих виртуальные сети Azure, PaaS и ExpressRoute. Подключение к Интернету. Azure выделяет IP-адреса из пула, выделенного для каждого региона Azure. Корпорация Майкрософт делает адреса, используемые в каждом регионе, доступными для загрузки, который обновляется обычным образом и управляется. Службы, доступные в каждом регионе, также часто изменяются по мере выпуска новых служб или более широкого развертывания служб. Содружество сущности должны регулярно просматривать эти материалы и использовать автоматизацию для обслуживания систем по мере необходимости. Конкретные IP-адреса для некоторых служб, размещенных в каждом регионе, можно получить, обратившись в службу поддержки Майкрософт.
