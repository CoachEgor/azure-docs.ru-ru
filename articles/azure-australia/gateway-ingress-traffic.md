# <a name="controlling-ingress-traffic-in-azure-australia"></a>Управление трафиком входящих данных в Azure Австралии

Основным элементом защиты систем ICT является управление сетевым трафиком. Трафик должен быть ограничен только тем, что необходим для работы системы, чтобы снизить вероятность компрометации.

В этом руководство приводятся подробные сведения о работе входящего (исходящего) сетевого трафика в Azure, а также рекомендации по реализации средств управления безопасностью сети для подключенной к Интернету системы.

Элементы управления сетью согласованы с руководством для потребителей Австралии кибератак Security Center (ACSC) и намерением руководства по информационной безопасности ACSC (ISM).

## <a name="requirements"></a>Требования

Общие требования к безопасности для систем Содружество определяются в ISM. Чтобы помочь Содружество сущности в реализации сетевой безопасности, ACSC опубликовал [ACSC Protect: Реализация сегментации и разделения](https://www.acsc.gov.au/publications/protect/network_segmentation_segregation.htm)сети, а также помощь в обеспечении безопасности систем в облачных средах ACSC опубликовала [Безопасность облачных вычислений для клиентов](https://www.acsc.gov.au/publications/protect/Cloud_Computing_Security_for_Tenants.pdf).

Эти руководства описывают контекст для реализации сетевой безопасности и управления трафиком и включают практические рекомендации по проектированию и настройке сети.

В статье [Безопасность облачных вычислений Майкрософт для клиентов Microsoft Azure](https://aka.ms/au-irap) Guide на странице "Австралия" на портале доверия службы выделяются конкретные технологии Майкрософт, которые позволяют выполнить рекомендации в публикациях ACSC.

## <a name="architecture"></a>Архитектура

Если вы участвуете в проектировании или реализации управления сетевыми безопасностью и входящими трафиками, необходимо сначала понять, как входящий сетевой трафик работает в Azure как в IaaS, так и в PaaS. В этом разделе представлен обзор возможных точек входа, в которых сетевой трафик может дойти до ресурса, размещенного в Azure, а также элементы управления безопасностью, доступные для ограничения и управления трафиком. Каждый из этих компонентов подробно описан в остальных разделах этого руководств.

Схема архитектуры, показанная здесь, описывает возможные пути, которые сетевой трафик может предпринять для подключения к службе, размещенной в Azure. ### <a name="azure-components"></a>Компоненты Azure

|Компонент | Описание|
|---|---|
|**Защита от атак DDoS** | Распределенные атаки типа "отказ в обслуживании" (от атак DDoS) пытаются выпустить ресурсы приложения, что делает приложение недоступным для легальных пользователей. Атаку DDoS можно направить на любую конечную точку, публично доступную через Интернет. Azure включает защиту от атак DDoS автоматически с помощью платформы Azure и предоставляет дополнительные возможности по устранению рисков, которые можно включить для конкретных приложений для более детального контроля.|
| **Диспетчер трафика** | Диспетчер трафика Azure — это балансировщик нагрузки на основе службы доменных имен (DNS), который может оптимально распределять трафик между службами в регионах Azure, обеспечивая высокий уровень доступности и скорость реагирования. Диспетчер трафика использует DNS для направления клиентских запросов к наиболее подходящей конечной точке на основе метода маршрутизации трафика и работоспособности конечных точек.|
| **ExpressRoute** | ExpressRoute — это выделенное сетевое подключение для использования облачных служб Майкрософт. Он подготавливается через поставщика услуг подключения и обеспечивает повышенную надежность, скорость, низкие задержки и повышенную безопасность по сравнению с обычными подключениями через Интернет. Канал ExpressRoute представляет собой логическое подключение между локальной инфраструктурой и облачными службами Майкрософт через поставщика услуг подключения.|
| **Частный пиринг ExpressRoute** | Частный пиринг ExpressRoute — это подключение между локальной средой и частными виртуальными сетями Azure. Частный пиринг обеспечивает доступ к службам Azure, таким как виртуальные машины, развернутым в виртуальной сети. Ресурсы и виртуальные сети, доступ к которым осуществлялся через частный пиринг, считаются расширением основной сети Организации. Частный пиринг обеспечивает двунаправленное подключение между локальной сетью и виртуальными сетями Azure с помощью частных IP-адресов.|
| **Пиринг Майкрософт для ExpressRoute** | ExpressRoute Microsoft пиринг — это подключение между локальной средой и общедоступными службами Майкрософт и Azure. Сюда входит подключение к Office 365, Dynamics 365 и службам PaaS Azure. Пиринг устанавливается через общедоступные IP-адреса, которые принадлежат организации или поставщику услуг подключения. По умолчанию доступ к службам через пиринг Майкрософт ExpressRoute невозможен, и Организация должна принять участие в необходимых службах. Затем этот процесс обеспечивает подключение к тем же конечным точкам, которые доступны в Интернете.|
|

### <a name="iaas-ingress-components"></a>Входящие компоненты IaaS

|Компонент | Описание|
|---|---|
|**Сетевой интерфейс** | Сетевой интерфейс — это ресурс, который существует в Azure. Он присоединяется к виртуальной машине и назначает частный IP-адрес, не поддерживающий маршрутизацию в Интернете, из подсети, с которой он связан. Этот IP-адрес динамически или статически назначается с помощью Azure Resource Manager.|
|**Подсеть** | Подсеть — это диапазон IP-адресов, который создается в виртуальной сети. Для сегментации сети можно создать несколько подсетей в виртуальной сети.|
| **Виртуальная сеть (VNet)** | Виртуальная сеть — это базовый ресурс в Azure, предоставляющий платформу и границу для развертывания ресурсов и включения взаимодействия. Виртуальная сеть находится в регионе Azure и определяет пространство IP-адресов и границы маршрутизации для ресурсов, интегрированных с виртуальной сетью, таких как виртуальные машины.|
| **Пиринг между виртуальными сетями** | Пиринг виртуальных сетей — это параметр конфигурации Azure, обеспечивающий прямой обмен данными между двумя виртуальных сетей без необходимости в шлюзе виртуальной сети. После пиринга эти две виртуальных сетей могут напрямую обмениваться данными, а дополнительная настройка может управлять использованием шлюзов виртуальной сети и других вариантов передачи.|
| **Общедоступный IP-адрес** | Общедоступный IP-адрес — это ресурс, который резервирует один из общедоступных маршрутизируемых IP-адресов Майкрософт из указанного региона для использования в виртуальной сети. Он может быть связан с конкретным сетевым интерфейсом, который обеспечивает доступ к ресурсу из систем Интернета, ExpressRoute и PaaS.|
| **Шлюз ExpressRoute** | Шлюз ExpressRoute — это объект в виртуальной сети, обеспечивающий подключение и маршрутизацию из виртуальной сети в локальные сети через частный пиринг в канале ExpressRoute.|
| **VPN-шлюз** | VPN-шлюз — это объект в виртуальной сети, предоставляющий зашифрованный туннель от виртуальной сети к внешней сети. Зашифрованный туннель может быть "сеть-сеть" для двунаправленной связи с локальной средой, другой виртуальной сетью, облачной средой или "точка — сеть" для обмена данными с одной конечной точкой.|
| **Интеграция виртуальной сети PaaS** | Многие возможности PaaS можно развернуть в виртуальной сети или интегрировать с ней. Некоторые возможности PaaS могут быть полностью интегрированы с виртуальной сетью и доступны только через частные IP-адреса. Другие, такие как Azure Load Balancer и шлюз приложений Azure, могут иметь внешний интерфейс с общедоступным IP-адресом и внутренним интерфейсом с частным IP-адресом в виртуальной сети. В этом случае трафик может входить в виртуальную сеть с помощью возможности PaaS.|
|

### <a name="paas-ingress-components"></a>Компоненты входящих данных PaaS

|Компонент | Описание|
|---|---|
|**Имя узла** | Возможности Azure PaaS определяются уникальным общедоступным именем узла, которое назначается при создании ресурса. Затем это имя узла регистрируется в общедоступном DNS-домене, где его можно разрешить в общедоступный IP-адрес.|
|**Общедоступный IP-адрес** | Если развертывание в интегрированной конфигурации виртуальной сети не выполняется, доступ к возможностям PaaS Azure осуществляется через общедоступный IP-адрес, поддерживающий маршрутизацию через Интернет. Этот адрес может быть выделен для конкретных ресурсов, например общедоступного Load Balancer, или может быть связан с определенной возможностью, имеющей общую точку входа для нескольких экземпляров, таких как хранилище или SQL. Этот общедоступный IP-адрес можно получить из Интернета, ExpressRoute или общедоступных IP-адресов IaaS через магистральную сеть Azure.|
|**Конечные точки службы** | Конечные точки службы предоставляют прямое, частное подключение из виртуальной сети к определенной возможности PaaS. Конечные точки службы, доступные только для подмножества возможностей PaaS, обеспечивают повышенную производительность и безопасность для ресурсов в виртуальной сети, обращающейся к PaaS.|
|

### <a name="security-controls"></a>Средства управления безопасностью

|Компонент | Описание|
|---|---|
|**Группы безопасности сети (группы безопасности сети)** | Группы безопасности сети управление трафиком в ресурсы виртуальной сети в Azure и из них. Группы безопасности сети применяются правила для разрешенных или запрещенных потоков трафика, включая трафик в Azure и между Azure и внешними сетями, такими как локальные или Интернет. Группы безопасности сети применяются к подсетям в виртуальной сети или к отдельным сетевым интерфейсам.|
|**Брандмауэр PaaS** | Многие возможности PaaS, такие как хранилище и SQL, имеют встроенный брандмауэр для управления входящим сетевым трафиком к конкретному ресурсу. Можно создать правила, разрешающие или запрещающие подключения с конкретных IP-адресов и (или) виртуальных сетей.|
|**Проверка подлинности и управление доступом в PaaS** | В рамках многоуровневого подхода к безопасности возможности PaaS предоставляют несколько механизмов для проверки подлинности пользователей и управления привилегиями и доступом.|
|**Политика Azure** | Политика Azure — это служба в Azure для создания, назначения политик и управления ими. Эти политики используют правила для управления типами ресурсов, которые могут быть развернуты, и конфигурацией этих ресурсов. Политики можно использовать для принудительного обеспечения соответствия, предотвращая развертывание ресурсов, если они не соответствуют требованиям или могут использоваться для наблюдения за состоянием соответствия.|
|

## <a name="general-guidance"></a>Общие рекомендации

Для проектирования и создания безопасных решений в Azure очень важно понимать и контролировать сетевой трафик, чтобы могли возникнуть только идентифицированные и созданные взаимосвязи. Цель данного руководства и конкретные рекомендации по компонентам в последующих разделах описывают средства и службы, которые можно применять для применения принципов, описанных в разделе _защита ACSC: Реализация сегментации сети и разделения_ между рабочими нагрузками Azure. Сюда входит подробное описание создания виртуальной архитектуры для защиты ресурсов, когда невозможно применить те же традиционные физические и сетевые элементы управления, которые можно использовать в локальной среде.

### <a name="specific-focus-areas"></a>Конкретные области фокуса

* Ограничение числа точек входа для виртуальных сетей
* Ограничение числа общедоступных IP-адресов
* Рассмотрите возможность использовать архитектуру основной и периферийной сети для виртуальных сетей, как описано в документации по виртуальному центру обработки данных (Microsoft Virtual Data Center).
* Применяйте продукты с встроенными функциями безопасности для входящих подключений из Интернета (например, шлюз приложений, шлюз API, сетевые виртуальные устройства).
* Ограничение потоков связи возможностями PaaS только теми, которые необходимы для функционирования системы
* Развертывание PaaS в интегрированной конфигурации виртуальной сети для повышения разделения и контроля
* Настройка систем для использования механизмов шифрования в соответствии с руководством потребителя ACSC и ISM
* Использование средств защиты на основе удостоверений, таких как проверка подлинности и управление доступом на основе ролей, в дополнение к традиционным элементам управления сетью
* Реализация ExpressRoute для подключения к локальным сетям
* Реализация VPN для административного трафика и интеграция с внешними сетями
* С помощью политики Azure можно ограничить регионы и ресурсы только теми, которые необходимы для работы функций системы.
* Использование политики Azure для принудительного применения базовой конфигурации безопасности для ресурсов, доступных из Интернета

### <a name="ddos-protection"></a>Защита от атак DDoS

Ресурсы, доступные в Интернете, подвержены атакам от атак DDoS. Для защиты от этих атак Azure обеспечивает защиту от атак DDoS на уровне Basic и Standard.

Базовый компонент автоматически включается в составе платформы Azure, включая мониторинг трафика Always on, и устранение распространенных атак на уровне сети в реальном времени, предоставляя те же средства защиты, которые используются веб-службы корпорации Майкрософт. Для снижения трафика атаки в регионах можно использовать всю глобальную сеть Azure. Предоставляется защита для общедоступных IP-адресов Azure IPv4 и IPv6.

Standard предоставляет дополнительные возможности по устранению рисков по сравнению с уровнем служб "базовый", которые настроены специально для ресурсов виртуальной сети Azure. Политики защиты корректируются на основе мониторинга выделенного трафика и алгоритмов машинного обучения. Защита обеспечивается для общедоступных IP-адресов Azure IPv4.

Это обеспечивает повышенную надежность, производительность и конфиденциальность подключений. Используя фильтры маршрутов, сущности Содружество могут ограничить обмен данными только с требуемыми регионами Azure, но сюда входят службы, размещенные в других организациях и требующие дополнительных возможностей фильтрации или проверки между локальная среда и корпорация Майкрософт.

В этой модели отсутствует активное взаимодействие внутренней локальной сети с общедоступными службами Azure через ExpressRoute, но безопасное подключение к частным виртуальным сетям достигается в соответствии с руководством потребителя ACSC.

## <a name="iaas-ingress"></a>Входящий трафик IaaS

В этом разделе содержится руководство по компонентам для управления входящим трафиком в компоненты IaaS. Сетевые интерфейсы виртуальных машин должны быть запланированы и настроены соответствующим образом для согласования с общими сегментами сети и целями разделения.

### <a name="virtual-network-vnet"></a>Виртуальная сеть (VNet)

Виртуальных сетей — это один из основных стандартных блоков для работы с сетью в Azure. Виртуальные сети определяют пространство IP-адресов и границы маршрутизации для использования в разных системах. Виртуальные сети делятся на подсети, а все подсети в виртуальной сети имеют прямой сетевой маршрут. Используя шлюзы виртуальной сети (ExpressRoute или VPN), системы в виртуальной сети могут предоставлять доступ к локальным и внешним средам. Основные сведения о виртуальных сетях и связанных параметрах конфигурации и маршрутизации очень важны для понимания и управления входящим сетевым трафиком.

### <a name="expressroute-gateway"></a>Шлюз ExpressRoute

Шлюзы ExpressRoute предоставляют точку входа из локальной среды и должны быть развернуты в соответствии с требованиями к безопасности, доступности, финансовым требованиям и производительности. Шлюзы ExpressRoute обеспечивают определенную пропускную способность сети и стоимость использования после развертывания. У виртуальных сетей может быть только один шлюз ExpressRoute, но он может быть подключен к нескольким каналам ExpressRoute и может использоваться несколькими виртуальными сетями через пиринг виртуальных сетей, что позволяет нескольким виртуальным сетям совместно использовать пропускную способность и возможности подключения. Следует соблюдать осторожность, чтобы настроить маршрутизацию между локальными средами и виртуальными сетями с помощью шлюзов ExpressRoute, чтобы обеспечить сквозное подключение с помощью известных управляемых сетевых входных точек. Содружество сущности, использующие шлюз ExpressRoute, также должны развертывать сетевые виртуальные устройства, чтобы установить VPN-подключение к локальной среде для обеспечения соответствия с руководством потребителя ACSC.

### <a name="paas-vnet-integration"></a>Интеграция виртуальной сети PaaS

Использование PaaS может обеспечить расширенные функциональные возможности и доступность, а также снизить расходы на управление, но должно быть обеспечено соответствующим образом. Чтобы повысить уровень контроля, обеспечить сегментацию сети или обеспечить безопасную входную точку входа для приложений и служб, многие возможности PaaS можно интегрировать с виртуальной сетью.

Чтобы обеспечить безопасную точку входа, для возможностей PaaS, таких как шлюз приложений, можно настроить внешний, общедоступный интерфейс и внутренний частный интерфейс для взаимодействия со службами приложений. Это предотвращает необходимость настройки серверов приложений с общедоступными IP-адресами и их предоставления внешним сетям.

Чтобы использовать PaaS в качестве интегрированной части архитектуры системы или приложения, корпорация Майкрософт предоставляет несколько механизмов развертывания PaaS в виртуальной сети. Методология развертывания позволяет ограничивать входящий доступ из внешних сетей, таких как Интернет, обеспечивая подключение и интеграцию с внутренними системами и приложениями. Примеры включают среды службы приложений, управляемые экземпляры SQL и многое другое.

## <a name="paas-ingress"></a>Входящий трафик PaaS

Возможности PaaS предоставляют возможности для повышения производительности и упрощения управления, но предоставляют сложности, необходимые для сегментации и разделения сети. Возможности PaaS обычно настраиваются с использованием общедоступных IP-адресов и доступны из Интернета.  При создании систем, использующих возможности PaaS, следует соблюдать осторожность, чтобы определять все необходимые потоки связи между компонентами в системе и правилами безопасности сети, созданными для разрешения только этой связи. В рамках многофункционального подхода к безопасности возможности PaaS должны быть настроены с шифрованием, проверкой подлинности и соответствующими элементами управления доступом и разрешениями.  

### <a name="hostname"></a>Имя узла

Возможности PaaS однозначно идентифицируются именами узлов, чтобы разрешить размещение нескольких экземпляров одной и той же службы на одном общедоступном IP-адресе. Уникальные имена узлов указываются при создании ресурсов и существуют в доменах DNS, принадлежащих корпорации Майкрософт. Конкретные имена узлов для разрабатываемых служб можно использовать в средствах безопасности с возможностями фильтрации на уровне приложения. При необходимости для некоторых служб также можно настроить личные домены.

### <a name="public-ip-for-paas"></a>Общедоступный IP-адрес для PaaS

Общедоступные IP-адреса для возможностей PaaS выделяются в зависимости от региона, в котором размещается или развертывается служба. Чтобы создать соответствующие правила безопасности сети и топологию маршрутизации для сегментации и разделения сетей, охватывающих виртуальные сети Azure, PaaS и ExpressRoute и подключение к Интернету, следует понимать, как выделять общедоступные IP-адреса и регионы. Обязательно. Azure выделяет IP-адреса из пула, выделенного для каждого региона Azure. Корпорация Майкрософт делает адреса, используемые в каждом регионе, доступными для загрузки, который обновляется обычным образом и управляется. Службы, доступные в каждом регионе, также часто изменяются по мере выпуска новых служб или более широкого развертывания служб. Содружество сущности должны регулярно просматривать эти материалы и использовать автоматизацию для обслуживания систем по мере необходимости. Конкретные IP-адреса для некоторых служб, размещенных в каждом регионе, можно получить, обратившись в службу поддержки Майкрософт.

### <a name="service-endpoints"></a>Конечные точки службы

Конечные точки службы виртуальной сети обеспечивают высокоскоростное, частное входное сетевое подключение для подсетей в виртуальной сети для использования конкретных возможностей PaaS. Для полной сегментации сети и разделения возможности PaaS необходимо настроить функцию PaaS, чтобы принимать подключения только из необходимых виртуальных сетей. Не все возможности PaaS поддерживают сочетание правил брандмауэра, включающих конечные точки службы и традиционные правила на основе IP-адресов, поэтому следует соблюдать осторожность, чтобы понять поток обмена данными, необходимый для функциональности приложения и администрирования. Поэтому реализация этих средств безопасности не влияет на доступность службы.

## <a name="security"></a>Безопасность

Реализация сегментации сети и элементов управления разделениями в IaaS и PaaS обеспечивается за счет обеспечения безопасности самих возможностей и путем реализации управляемых путей взаимодействия из систем, которые будут взаимодействовать с возможностями.

Проектирование и разработка решений в Azure — это процесс создания логической архитектуры для понимания, контроля и отслеживания сетевых ресурсов во всем присутствии Azure. Эта логическая архитектура — это программное обеспечение, определенное на платформе Azure, которое принимает место физической топологии сети, реализованной в традиционных сетевых средах.

Создаваемая логическая архитектура должна обеспечивать функциональность, необходимую для удобства использования, но также должна обеспечивать видимость и контроль, необходимые для обеспечения безопасности и целостности.

Достижение этого результата основано на реализации необходимых средств сегментации и разделения сети, а также в защите и применении топологии сети и реализации этих средств.

Сведения, приведенные в этом пошаговом окне, можно использовать для определения источников входящего трафика, которые необходимо разрешить, а также способов управления трафиком.

### <a name="network-security-groups-nsgs"></a>Группы безопасности сети (NSG)

Группы безопасности сети используются для указания входящего и исходящего трафика, разрешенного для подсети или определенного сетевого интерфейса. При настройке группы безопасности сети сущности Содружество должны использовать подход список разрешений, где правила настроены на разрешение необходимого трафика с правилом по умолчанию, настроенного на запрет всего трафика, который не соответствует конкретной инструкции разрешения. Необходимо соблюдать осторожность при планировании и настройке группы безопасности сети, чтобы обеспечить правильную запись всех необходимых входящих и исходящих данных. Сюда входит определение и понимание всех диапазонов частных IP-адресов, заносящихся в виртуальных сетях Azure и локальной среде, а также конкретных служб Майкрософт, таких как требования к управлению Azure Load Balancer и PaaS. Лица, участвующие в проектировании и реализации групп безопасности сети, также должны понимать использование тегов служб и групп безопасности приложений для создания детализированных, сервисных и зависящих от приложения правил безопасности.

## <a name="paas-firewall"></a>Брандмауэр PaaS

Брандмауэр PaaS — это возможность управления доступом к сети, которую можно применить к определенным службам PaaS. Он позволяет настроить фильтрацию IP-адресов или фильтрацию из определенных виртуальных сетей, чтобы ограничить входящий трафик для конкретного экземпляра PaaS. Для возможностей PaaS, включающих брандмауэр, политики управления доступом к сети должны быть настроены на разрешение только необходимого входящего трафика в зависимости от требований приложения.
