# <a name="secure-remote-administration-of-your-gateway-in-azure-australia"></a>Безопасное удаленное администрирование шлюза в Azure Австралии

Очень важно обеспечить доступность и целостность любой системы, которая безопасно и управляется административными действиями. Административные действия следует выполнять с защищенного устройства, через безопасное подключение и с помощью надежных процессов проверки подлинности и авторизации. Безопасное удаленное администрирование гарантирует, что только выполняемые действия выполняются только разработчиками, которые являются администраторами.

В этой статье приводятся сведения о реализации безопасного удаленного администрирования для системы, доступной в Azure, которая соответствует рекомендациям потребителя кибератак Security Center (ACSC) и намерениям о ACSC Руководство по безопасности (ISM).

## <a name="australian-cyber-security-centre-acsc-requirements"></a>Требования кибератак Security Center (ACSC) Австралии

Общие требования к безопасности для систем Содружество определяются в ISM. Чтобы помочь Содружество сущности в обеспечении безопасного администрирования, ACSC опубликовал [ACSC защиту: Безопасное администрирование](https://www.acsc.gov.au/publications/protect/secure-administration.htm)

В этом документе обсуждается важность безопасного администрирования и предлагается один из способов реализации среды безопасного администрирования. В документе описаны элементы решения безопасного администрирования следующим образом.

|Элемент   |Описание   |
|---|---|
|Контроль привилегированного доступа   |Управление доступом к привилегированным учетным записям — это фундаментальный контроль безопасности, который защищает привилегированные учетные записи от несанкционированного использования. Методология управления доступом будет охватывать концепции "минимальных прав" и "требуется", а также процессов и процедур для управления учетными записями служб и персоналом.   |
|Многофакторная проверка подлинности   |Реализация дополнительных факторов проверки подлинности за пределами имен пользователей и парольных фраз, таких как физические маркеры или SmartCards, может помочь защитить критические ресурсы. Если злоумышленник нарушает учетные данные для привилегированных учетных записей, так как все административные действия должны сначала пройти некоторую форму многофакторной проверки подлинности, последствия могут значительно снизиться.|
|Привилегированные рабочие станции|Использование известной безопасной среды для административных задач может привести к снижению риска компрометации сети из-за реализации дополнительных средств управления безопасностью.|
|Ведение журналов и аудит   |Автоматическое создание, сбор и анализ событий безопасности и администрирования, связанных с рабочими станциями, серверами, сетевыми устройствами и полями перехода, позволяет обнаруживать нарушения и попытки взлома. Автоматизация позволяет организациям быстрее реагировать на запросы, уменьшая последствия компрометации.|
|Сегментация и разделение сети|Сегментирование сети на логические зоны, такие как разные домены безопасности, и дальнейшее разделение этих логических сетей за счет ограниченных типов данных, переходящих из одной зоны в другую, позволяет ограничить перемещение бокового смещения. Сегментация не позволяет злоумышленнику получить доступ к дополнительным ресурсам.|
|Поля перехода|Поле перехода — это защищенный сервер удаленного доступа, который обычно используется по службы удаленных рабочих столов или Secure Shell (SSH) корпорации Майкрософт. Поля перехода действуют как пошаговые точки для администраторов, обращающихся к критическим системам со всеми административными действиями, выполненными с выделенного узла.|
|

## <a name="administration-workflow"></a>Рабочий процесс администрирования

Администрирование систем, развернутых в Azure, делится на две категории: администрирование конфигурации Azure и Администрирование рабочих нагрузок, развернутых в Azure. Настройка Azure выполняется с помощью портал Azure и администрирование рабочей нагрузки осуществляется с помощью административных механизмов, таких как протокол удаленного рабочего стола (RDP), Secure Shell (SSH) или для возможностей PaaS, с помощью таких средств, как SQL Management Studio.

Получение доступа для администрирования — это многоэтапный процесс, включающий компоненты, перечисленные в архитектуре, и требующий доступа к портал Azure и конфигурации Azure, прежде чем можно будет получить доступ к рабочим нагрузкам Azure.

>[!NOTE]
> Описанные здесь действия относятся к общему процессу, в котором используются графические компоненты интерфейса пользователя (GUI) Azure. Эти действия также можно выполнить с помощью других интерфейсов, таких как PowerShell.

Групповая политика элементы управления предотвращают проверку подлинности учетных записей без прав администратора на привилегированной рабочей станции и запрещают учетным записям администратора выполнять проверку подлинности на рабочих станциях без привилегий. Microsoft Intune управляет соответствием привилегированной рабочей станции, обеспечивая актуальность обновлений программного обеспечения, защиты от вредоносных программ и других требований к соответствию. |
|портал Azure войти |Администратор открывает веб-браузер для портал Azure, который шифруется с помощью протокола TLS, и входит в систему с использованием учетных данных администратора. Запрос проверки подлинности обрабатывается через Azure Active Directory напрямую или с помощью механизмов проверки подлинности, таких как службы федерации Active Directory (AD FS) (AD FS) или сквозная проверка подлинности. |

Администратор должен сначала пройти проверку подлинности на мобильном устройстве, а затем в Microsoft Authenticator приложении с помощью ПИН-кода или биометрической системы, прежде чем попытка проверки подлинности будет выполнена в Azure MFA. |
|Условный доступ |Политики условного доступа проверяют попытку проверки подлинности, чтобы убедиться, что она соответствует необходимым требованиям, таким как IP-адрес, с которого поступает подключение, членство в группе привилегированной учетной записи, а также состояние управления и соответствия требованиям привилегированная Рабочая станция, сообщаемая Intune. |
|Управление привилегированными пользователями (PIM) |С помощью портал Azure администратор может активировать или запросить активацию для привилегированных ролей, для которых они авторизации через PIM. PIM гарантирует, что привилегированные учетные записи не имеют зафиксированных административных привилегий и что все запросы к привилегированному доступу выполняются только в течение времени, необходимого для выполнения административных задач. PIM также обеспечивает ведение журнала всех запросов и активаций в целях аудита. |

### <a name="azure-workload-administration"></a>Администрирование рабочих нагрузок Azure

|Шаг |Описание|
|---|---|
|JIT-доступ|Для получения доступа к виртуальным машинам администратор использует JIT для запроса доступа к серверу переходов с IP-адреса шлюза удаленных рабочих столов, а также RDP или SSH с сервера переходов к соответствующим виртуальным машинам рабочей нагрузки.|
|VPN-шлюз Azure|Теперь администратор устанавливает VPN-подключение типа "точка — сеть" IPSec с привилегированной рабочей станции на VPN-шлюз Azure, который выполняет проверку подлинности на основе сертификата для установления соединения.|
|Шлюз удаленных рабочих столов|Теперь администратор пытается подключиться к серверу переходов по протоколу RDP с помощью шлюза удаленных рабочих столов, указанного в конфигурации подключение к удаленному рабочему столу. Шлюз удаленных рабочих столов имеет частный IP-адрес, доступ к которому осуществляется через VPN-шлюз Azure. Политики на шлюзе удаленных рабочих столов контролируют, разрешена ли привилегированная учетная запись для доступа к запрошенному серверу переходов. Шлюз удаленных рабочих столов запрашивает у администратора учетные данные и перенаправляет запрос на проверку подлинности на сервер политики сети (NPS).|
|Сервер политики сети (NPS)|NPS получает запрос на проверку подлинности от шлюза удаленных рабочих столов и проверяет имя пользователя и пароль в Active Directory перед отправкой запроса на Azure Active Directory для активации запроса проверки подлинности Azure MFA.|
|Azure MFA|Azure MFA отправляет запрос на проверку подлинности на зарегистрированное мобильное устройство привилегированной учетной записи. Мобильное устройство управляется Intune для обеспечения соответствия требованиям безопасности. Администратор должен сначала пройти проверку подлинности на мобильном устройстве, а затем в Microsoft Authenticator приложении с помощью ПИН-кода или биометрической системы, прежде чем попытка проверки подлинности будет выполнена в Azure MFA.|
|Сервер переходов|После успешной проверки подлинности RDP-подключение шифруется с помощью протокола TLS, а затем отправляется через зашифрованный туннель IPSec в VPN-шлюз Azure через шлюз удаленных рабочих столов и на сервер переходов. С сервера переходов теперь администратор может выполнять RDP или SSH для рабочей нагрузки виртуальных машин, как указано в JIT-запросе.|
|

## <a name="general-guidance"></a>Общие рекомендации

При реализации компонентов, перечисленных в этой статье, применяются следующие общие рекомендации.

* Проверка доступности региона для служб, гарантирующая, что все данные остаются в развернутых расположениях и развертываются в Центрально центре au или Au Central 2 в качестве первого предпочтения для защищенных рабочих нагрузок.

* Сведения о состоянии сертификации отдельных служб и проведении самооценки на всех соответствующих компонентах, не включенных в отчет, *2018* см. в руководстве по ACSC.  *Microsoft Azure с ЗАЩИТой*

* Обеспечьте сетевое подключение и необходимую конфигурацию прокси-сервера для доступа к необходимым компонентам проверки подлинности, таким как Azure AD, ADFS и PTA.

* Использование политики Azure для мониторинга и обеспечения соответствия требованиям

* Убедитесь, что виртуальные машины, особенно домен Active Directory контроллеры, хранятся в зашифрованных учетных записях хранения и используют шифрование дисков Azure.

* Создание и поддержка надежных процессов управления удостоверениями и административными привилегиями, а также управление ими для лежат технических элементов управления, перечисленных в этой статье.

## <a name="secure-devices"></a>Защита устройств

Физические устройства, используемые привилегированными пользователями для выполнения административных задач, — это ценные цели для вредоносных субъектов. Обеспечение безопасности и целостности физических устройств и обеспечение их доступности от вредоносных программ и защита их от компрометации — ключевая часть обеспечения безопасного удаленного администрирования. Это относится к конфигурации безопасности с высоким приоритетом, как указано в восьми ACSC важных стратегиях, позволяющих снизить риски безопасности кибератак, такие как список разрешений приложений, приложения для исправлений, усиление защиты приложений и обновление операционных систем. Чтобы убедиться, что состояние устройства соответствует требованиям к Организации, эти возможности необходимо установить, настроить, проконтролировать, проверить и включить в отчет.

### <a name="privileged-workstation"></a>Привилегированная Рабочая станция

Привилегированная Рабочая станция — это защищенный компьютер, который можно использовать для выполнения административных обязанностей и доступен только административным учетным записям. Привилегированная Рабочая станция должна иметь политики и конфигурацию, чтобы ограничить программное обеспечение, которое может быть запущено, его доступ к сетевым ресурсам и Интернету и учетные данные следует защищать в случае кражи или компрометации устройства. |

### <a name="mobile-device"></a>Мобильное устройство

Мобильное устройство повышает риск случайной потери или кражи из-за его переносимости и размера, поэтому его необходимо защищать соответствующим образом. Мобильное устройство предоставляет надежный дополнительный фактор для проверки подлинности, учитывая возможность принудительного применения проверки подлинности для доступа к устройству, отслеживания с помощью служб определения местоположения, функций шифрования и возможности удаленной очистки. При использовании мобильного устройства в качестве дополнительного фактора проверки подлинности для Azure устройство должно быть настроено на использование Microsoft Authenticator приложения с ПИН-кодом или биометрической проверкой подлинности, а не через телефонные звонки или текстовые сообщения.

## <a name="secure-communication"></a>Безопасная связь

Коммуникационный трафик для действий администрирования может содержать строго конфиденциальную информацию, например учетные данные администратора, и должна быть управляема и защищена соответствующим образом. Обеспечение безопасной связи включает в себя надежные возможности шифрования для предотвращения перехвата и сегментации сети и ограничений, которые ограничивают административный трафик для создания конечных точек и управляют бокового смещения перемещением при компрометации системы.

### <a name="azure-portal"></a>портала Azure

Обмен данными с портал Azure шифруется с помощью протокола TLS, а использование портал Azure сертифицировано ACSC. Содружество сущности должны следовать рекомендациям из *руководства пользователя ACSC* и настроить их веб-браузеры, чтобы убедиться, что они используют последнюю версию TLS и поддерживаемые алгоритмы шифрования.

## <a name="strong-authentication"></a>Строгая проверка подлинности

Безопасная идентификация привилегированных пользователей перед предоставлением доступа к системам является основным компонентом безопасного администрирования. Необходимо обеспечить защиту учетных данных, связанных с привилегированной учетной записью, и предотвратить получение вредоносными субъектами доступа к системам посредством олицетворения или кражи учетных данных.

### <a name="domain-controller-dc"></a>Контроллер домена (DC)

На высоком уровне контроллер домена размещает копию Active Directory базы данных, которая содержит всех пользователей, компьютеры и группы в пределах домена. Контроллеры домена выполняют проверку подлинности для пользователей и компьютеров. Контроллеры домена в этой архитектуре размещаются как виртуальные машины в Azure и предоставляют службы проверки подлинности для привилегированных учетных записей, подключающихся к серверам переходов и рабочим нагрузкам.

Azure AD может быть синхронизированы с Active Directoryом с помощью Azure AD Connect и может обеспечивать Федеративные проверки подлинности с помощью службы федерации Active Directory (AD FS) (AD FS) и Azure AD Connect. Azure AD — это основной компонент безопасного администрирования.

Azure MFA поддерживает ряд методов проверки подлинности, и эта архитектура опирается на Microsoft Authenticator приложение для усиления безопасности и интеграции с NPS.

## <a name="strong-authorisation"></a>Strong авторизации

После надежной идентификации привилегированной учетной записи ей может быть предоставлен доступ к ресурсам. Авторизации управляет привилегиями, назначенными конкретной учетной записи, и управляет ими. Strong процессы авторизации согласованы с 8 важными стратегиями ACSC, чтобы устранить кибератак инциденты безопасности с ограниченными правами администратора.

### <a name="identity-and-access-management"></a>Управление удостоверениями и доступом

Доступ для выполнения привилегированных действий в Azure основан на ролях, назначенных этой учетной записи. Azure включает обширный и детализированный диапазон ролей с конкретными разрешениями для выполнения конкретных задач. Эти роли могут быть предоставлены на нескольких уровнях, например в подписке или группе ресурсов. Назначение ролей и управление разрешениями основаны на учетных записях и группах в Azure Active Directory и управляются с помощью управления доступом (IAM) в Azure.

### <a name="privileged-identity-management-pim"></a>Управление привилегированными пользователями (PIM)

Привилегированным учетным записям не требуется постоянный или незащищенный привилегированный доступ, но вместо этого ему может быть предоставлена возможность запросить привилегированный доступ в течение определенного периода времени, чтобы обеспечить выполнение привилегированных действий. PIM предоставляет дополнительные элементы управления для поддержки и ограничения привилегированного доступа, а также ведения журнала и аудита для контроля экземпляров использования привилегий.

Условный доступ используется для принудительного применения MFA, управления устройствами и обеспечения соответствия с помощью Intune и членства в группах административных учетных записей.
