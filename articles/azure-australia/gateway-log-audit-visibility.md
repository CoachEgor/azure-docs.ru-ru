# <a name="gateway-logging-auditing-and-visibility-in-azure-australia"></a>Ведение журнала шлюза, аудит и видимость в Австралии Azure

Обнаружение угроз безопасности кибератак и реагирование на них полагается на создание, сбор и анализ данных, связанных с работой системы.

Корпорация Майкрософт имеет встроенные средства Azure, которые помогают реализовать ведение журналов, аудит и видимость для управления безопасностью систем, развернутых в Azure. Кроме того, существует Эталонная архитектура, которая соответствует рекомендациям потребителя кибератак Security Center (ACSC) и намерениям по информационной безопасности (ISM).

Шлюзы действуют как механизмы управления потоком информации на уровне сети, а также могут контролировать сведения на более высоких уровнях модели Open System Interconnect (OSI). Шлюзы необходимы для управления потоками данных между доменами безопасности и предотвращения несанкционированного доступа из внешних сетей. Учитывая важность шлюзов в управлении потоком информации между доменами безопасности, любые сбои, особенно при более высоких классификациях, могут иметь серьезные последствия. Таким образом, надежные механизмы для оповещений сотрудников о ситуациях, которые могут вызвать кибератакные инциденты безопасности, особенно важны для шлюзов.

Реализация возможностей ведения журналов и оповещений для шлюзов может помочь в обнаружении инцидентов безопасности кибератак, попыток вторжений и необычных шаблонах использования. Кроме того, хранение журналов событий на отдельном защищенном сервере журналов повышает сложность удаления данных журнала, чтобы уничтожить свидетельство целевого кибератак вторжения.

## <a name="australian-cyber-security-centre-acsc-requirements"></a>Требования кибератак Security Center (ACSC) Австралии

Общие требования к безопасности для систем Содружество определяются в руководстве по безопасности ACSC Information Security (ISM). Чтобы помочь Содружество сущности в соответствии с этими требованиями в Azure, *руководство по потребителю ACSC — Microsoft Azure в защищенном* *отчете о сертификации ACSC — Microsoft Azure* публикации подробно описываются следующие конкретные требования. для ведения журнала, аудита и видимости:

1. Чтобы снизить риски, возникающие при использовании общих облачных ресурсов, Содружество сущности должны принять участие в Microsoft Azure предоставляемых им возможностях, включая центр безопасности Azure, Azure Monitor, политику Azure и помощник Azure, чтобы помочь сущностям выполнять Мониторинг рабочих нагрузок Azure в режиме реального времени

2. ACSC также рекомендует, чтобы Содружествоные сущности перенаправляли все обязательные журналы безопасности в ACSC для всего мониторинга Австралии для государственных организаций.

3. Чтобы помочь в устранении рисков, Содружество сущности должны настраиваться в своих подписках Azure:

    * Включение центра безопасности Azure
    * Обновление до уровня "Стандартный"
    * Включение автоматической подготовки Microsoft Monitoring Agent для поддерживаемых виртуальных машин Azure
    * Регулярно проверяйте, приоритисе и снижая рекомендации и оповещения безопасности на панели мониторинга центра безопасности.

4. Правительственные учреждения должны включить пересылку журналов и событий из своей подписки Azure в ACSC, чтобы предоставить ACSC со сведениями о несоответствии этим рекомендациям. Концентраторы событий Azure предоставляют возможность выполнять потоковую передачу внешних журналов в ACSC или локальные системы, принадлежащие сущности Содружество.

5. Сущности Содружество должны согласовать ведение журнала, которое они включают в Azure, в соответствии с требованиями, указанными в ISM.

6. Корпорация Майкрософт оставляет журналы в Azure в течение 90 дней. Сущности Customer должны реализовать Архив журнала региме, чтобы гарантировать, что журналы могут храниться в течение семи лет, необходимых в УДАЛЯЕМ АФДА

7. Сущности Содружество, имеющие локальные или основанные на Azure сведения о безопасности и управлении событиями (SIEM), также могут пересылать журналы в эти системы.

8. Сущности Содружество должны реализовывать журналы потоков наблюдателя за сетями для групп безопасности сети (группы безопасности сети) и виртуальных машин. Эти журналы должны храниться в выделенной учетной записи хранения, содержащей только журналы безопасности, а доступ к учетной записи хранения должен быть защищен с помощью элементов управления доступом на основе ролей.

9. Сущности Содружество должны реализовывать руководство потребителя ACSC, чтобы обеспечить соответствие рабочих нагрузок Azure цели ISM для ведения журнала и мониторинга. Содружество сущности также должны использовать возможности Azure, которые помогают ACSC получать мониторинг, оповещения и журналы в режиме реального времени, связанные с использованием Azure для государственных организаций.

* используется службами Azure Analysis Services для получения аналитики.

При реализации компонентов, перечисленных в этой статье, применяются следующие общие рекомендации.

* Сведения о состоянии сертификации отдельных служб и проведении самооценки на всех соответствующих компонентах, не включенных в отчет, *2018* см. в руководстве по ACSC.  *Microsoft Azure с ЗАЩИТой*

* Выявление и приоритисе ведение журнала, аудит и видимость для систем с большими значениями, а также всех входящих и исходящих точек сети в системах, размещенных в Azure.

Шлюз приложений может предоставлять важные сведения об использовании веб-приложений, а также помогает обнаруживать инциденты безопасности кибератак. Шлюз приложений отправляет метаданные в журнал действий и журналы диагностики в Azure Monitor, где их можно использовать в Log Analytics или распространять в концентратор событий или учетную запись хранения.

### <a name="vpn-gateway"></a>VPN-шлюз

VPN-шлюз — это потенциальная точка входа для широкого спектра обмена данными с средой Azure, например подключение к локальной среде и административный трафик. Регистрация в VPN-шлюзах обеспечивает аналитику и возможность трассировки подключений к среде Azure. Ведение журнала может обеспечить аудит и анализ, а также помочь в обнаружении или исследовании вредоносных или аномальных подключений. Журналы VPN-шлюза отправляются в журнал действий Azure Monitor, где их можно использовать в Log Analytics или распространять в концентратор событий или учетную запись хранения.

### <a name="azure-firewall"></a>Брандмауэр Azure

Брандмауэр Azure предоставляет управляемую точку выхода из среды Azure, а созданные журналы, которые включают сведения о предпринятых и успешных исходящих подключениях, являются важным элементом стратегии ведения журнала. Эти журналы могут проверять, работают ли системы в соответствии с разработкой, а также помогают обнаруживать вредоносный код или субъекты, пытающиеся подключиться к неразрабатываемым внешним системам. Брандмауэр Azure записывает журналы в журнал действий и журналы диагностики в Azure Monitor, где их можно использовать в Log Analytics или распространять в концентратор событий или учетную запись хранения.

### <a name="virtual-machines"></a>Виртуальные машины

Виртуальные машины — это конечные точки, которые отправляют и получают сетевые подключения, обрабатывают данные и предоставляют службы. Так как виртуальные машины могут размещать данные или важные системные службы, убедитесь, что они работают правильно и обнаружение инцидентов безопасности кибератак может быть критически важным. Виртуальные машины собираются различные журналы событий и аудита, которые могут отслеживать работу системы и действия, выполняемые в этой системе. Журналы, собранные на виртуальных машинах, можно перенаправлять в Log Analytics рабочую область, используя Microsoft Monitoring Agent, где их можно проанализировать с помощью центра безопасности Azure и применимых решений Log Analytics. Виртуальные машины также могут напрямую интегрироваться с концентраторами событий Azure или SIEM, либо напрямую, либо через сервер сбора журналов.

## <a name="log-retention"></a>Хранение журналов

Для государственных организаций правительства США журналы, захваченные в Azure, должны храниться в соответствии с национальными архивами [административных функций Австралии (АФДА)](http://www.naa.gov.au/information-management/records-authorities/types-of-records-authorities/AFDA/index.aspx), которые указывают, что журналы сохраняются до семи лет.

|---|---|
|Журнал действий Azure|До 90 дней|
|Рабочая область Log Analytics|До двух лет|
|Концентратор событий|До семи дней|
|

Вы обязаны убедиться, что журналы архивируются соответствующим образом, чтобы они соответствовали АФДА и другим законодательным требованиям.

SIEM — это система, которая обеспечивает централизованное хранение, аудит и анализ журналов безопасности с помощью определенных механизмов для приема широкого спектра данных журнала и интеллектуальных средств для анализа, создания отчетов и обнаружения инцидентов и реагирования. Вы можете использовать возможности SIEM, включающие сведения о ведении журналов Azure, чтобы расширить возможности безопасности, предоставляемые в Azure изначально. Сущности Содружество могут полагаться на SIEM, размещенные на виртуальных машинах в Azure, в локальной среде или в виде программного обеспечения как услуги (SaaS) в зависимости от конкретных требований.

### <a name="australian-cyber-security-centre"></a>Центр безопасности Австралии кибератак

Кибератак Security Center (ACSC) — это ведущий специалист в Австралии для государственных организаций в области безопасности National кибератак. Он объединяет кибератакные возможности безопасности из Австралии правительства США, чтобы улучшить устойчивость кибератак для Австралии, а также поддержку экономических и социальных процветание Австралии в цифровом возрасте. ACSC рекомендует, чтобы сущности Содружество перенаправляли все создаваемые системой файлы журналов, события и журналы в ACSC для всего мониторинга Австралии для государственных организаций.

## <a name="incident-response"></a>Реагирование на инциденты

Создание соответствующих журналов, их сбор в централизованные репозитории и выполнение анализа увеличивают понимание систем и предоставляет механизмы для обнаружения инцидентов безопасности кибератак. После обнаружения инцидентов или событий следующий шаг заключается в реагировании на эти события и выполнении действий для поддержания работоспособности системы и защиты служб и данных от компрометации. Azure предоставляет сочетание служб для эффективного реагирования на любые происходящие события.

### <a name="azure-alerts"></a>Оповещения Azure

### <a name="azure-automation"></a>Служба автоматизации Azure

Служба автоматизации Azure позволяет сущностям Содружество запускать действия в ответ на события. Это может привести к запуску записи пакетов на виртуальных машинах, запуску рабочего процесса, прерыванию или запуску виртуальных машин или служб, а также ряда других задач. Автоматизация позволяет быстро реагировать на предупреждения без вмешательства пользователя, таким образом уменьшая время отклика и серьезность инцидента или события.
