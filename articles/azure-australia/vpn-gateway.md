# <a name="azure-vpn-gateway-in-azure-australia"></a>VPN-шлюз Azure в Австралии Azure

Критической службой с любым общедоступным облаком является безопасное подключение облачных ресурсов и служб к существующим локальным системам.  Служба, которая предоставляет эту возможность в Azure, является VPN-шлюзом Azure (VPN-шлюзом). В этой статье описываются основные моменты, касающиеся настройки VPN-шлюза в соответствии со стандартными элементами управления "ручное" (ASD) принимало [Information Security](https://acsc.gov.au/infosec/ism/) (ISM).

VPN-шлюз используется для отправки зашифрованного трафика между виртуальной сетью в Azure и другой сетью.  VPN-шлюзы решают три сценария:

- **Сеть — сеть** S2S
- **Точка-сеть** P2S
- **Подключение между виртуальными сетями**

В этой статье основное внимание уделяется VPN-шлюзам S2S. На схеме 1 показан пример конфигурации VPN-шлюза типа "сеть — сеть".

## <a name="key-design-considerations"></a>Основные рекомендации по проектированию

Существует три сетевых способа подключения Azure к клиентам Австралии для государственных организаций.

- **ЗНАЧОК**
- **ExpressRoute**
- **Общедоступный Интернет**

В статье о потребителях кибератак Security Center [для Azure](https://servicetrust.microsoft.com/viewpage/Australia) рекомендуется использовать VPN-шлюз (или АНАЛОГИЧную защищенную сертифицированную службу стороннего поставщика) в сочетании с тремя сетевыми параметрами, чтобы обеспечить соответствие подключений Элементы управления ISM для шифрования и целостности.

### <a name="encryption-and-integrity"></a>Шифрование и целостность

По умолчанию VPN согласовывает алгоритмы шифрования и целостности и параметры во время установки соединения в рамках подтверждения IKE.  В ходе подтверждения IKE конфигурация и порядок предпочтения будут зависеть от того, является ли VPN-шлюз инициатором или ответчиком (имена NetBIOS управляются через VPN-устройство).  Окончательная конфигурация подключения управляется конфигурацией VPN-устройства.  Дополнительные сведения о проверенных VPN-устройствах и их конфигурации см. здесь: [О VPN-устройствах](https://docs.microsoft.com/azure/vpn-gateway/vpn-gateway-about-vpn-devices)

VPN-шлюзы могут управлять шифрованием и целостностью, настраивая пользовательскую политику IPsec/IKE для соединения.

### <a name="resource-operations"></a>Операции с ресурсами

VPN-шлюзы создают подключение между средами Azure и средой, отличной от Azure, через общедоступный Интернет.  ISM имеет элементы управления, связанные с явной авторизацией подключений.  По умолчанию VPN-шлюзы можно использовать для создания неавторизованных туннелей в безопасных средах.  Поэтому очень важно, чтобы организации использовали управление доступом на основе ролей Azure (RBAC) для управления тем, кто может создавать и изменять VPN-шлюзы и их подключения.  В Azure нет встроенной роли для управления VPN-шлюзами, поэтому для этого потребуется настраиваемая роль.

Доступ к ролям "владелец", "участник" и "участник сети" тесно контролируется.  Также рекомендуется использовать Azure AD Privileged Identity Management для более детального контроля доступа.

### <a name="high-availability"></a>Высокая надежность

VPN-шлюзы Azure могут иметь несколько подключений (см. схему 1) и поддерживают несколько локальных VPN-устройств в одной локальной среде.  

Виртуальные сети в Azure могут иметь несколько VPN-шлюзов, которые можно развертывать в независимых, активных и активных конфигурациях.

Рекомендуется, чтобы все VPN-шлюзы были развернуты в [конфигурации с высокой доступностью](https://docs.microsoft.com/azure/vpn-gateway/vpn-gateway-highlyavailable): например, два локальных VPN-устройства подключены к двум VPN-шлюзам в режиме "активный — пассивный" или "активный — активный" (см. диаграмму 2).

### <a name="forced-tunneling"></a>Принудительное туннелирование

Принудительное туннелирование перенаправления или "принудительное" перенаправление всех трафика, привязанного к Интернету, обратно в локальную среду через VPN-шлюз для проверки и аудита. Без принудительного туннелирования Интернет-трафик с виртуальных машин в Azure просматривает инфраструктуру сети Azure напрямую в общедоступный Интернет без возможности проверки или аудита трафика.  Это важно, если организации требуется использовать безопасный Шлюз Интернета (SIG) для среды.

### <a name="service-attributes"></a>Атрибуты службы

VPN-шлюзы для подключений S2S, настроенных для государственных организаций Австралии, должны иметь следующие атрибуты:

|Атрибут | ПРИХОДИТ|
|--- | --- |
|gatewayType | ЧЕРЕЗ|
|

Параметры атрибутов, необходимые для обеспечения соответствия элементам управления ISM для защиты,:

|Атрибут | ПРИХОДИТ|
|--- |---|
|vpnType |RouteBased|
|vpnClientConfiguration/Впнклиентпротоколс | IkeV2|
|

VPN-шлюзы Azure поддерживают ряд алгоритмов шифрования из стандартов протокола IPsec и IKE.  Политика по умолчанию делает максимально возможным взаимодействие с широким диапазоном VPN-устройств сторонних производителей.  В результате может быть согласована несоответствующая конфигурация в ходе подтверждения IKE.  Поэтому настоятельно рекомендуется применять [Настраиваемые параметры политики IPSec/IKE](https://docs.microsoft.com/azure/vpn-gateway/vpn-gateway-ipsecikepolicy-rm-powershell) к vpnClientConfiguration в VPN-шлюзах, чтобы обеспечить соответствие подключений ЭЛЕМЕНТАМ управления ISM для локальных сред подключения к Azure.  Ключевые атрибуты:

|Атрибут|СЛЕДУЕТ|ПРИХОДИТ|
|---|---|---|
|салифетимесекондс|< 14400 сек|> 300 сек|
|садатасизекилобитес| |> 1024 КБ|
|ипсеценкриптион| |AES256 — GCMAES256|
|ипсеЦинтегрити| |SHA256 — GCMAES256|
|икинкриптион| |AES256 — GCMAES256|
|икеинтегрити| |SHA256 — GCMAES256|
|дхграуп|DHGroup14, DHGroup24, ECP256, ECP384|DHGroup2|
|пфсграуп|PFS2048, PFS24, ECP256, ECP384||
|

*Для Дхграуп и Пфсграуп в приведенной выше таблице предпочтительнее использовать ECP256 и ECP384, даже если другие параметры могут быть использованы.*

|Общедоступный IP-адрес | VPN-шлюзам S2S требуется общедоступный IP-адрес для установления подключения между локальным VPN-устройством и VPN-шлюзом.  Прежде чем создавать VPN-шлюз S2S, необходимо создать общедоступный IP-адрес.|

|

   Microsoft. Network/virtualNetworkGateways/*  
   Microsoft. Network/Connections/*  
   Microsoft. Network/локалнетворкгатевайс/*  
   Microsoft. Network/virtualNetworks/подсети/*  
   Microsoft. Network/publicIPAddresses/*  
   Microsoft. Network/ПублиЦиппрефиксес/*  
   Microsoft. Network/Раутетаблес/*  

2. Добавьте пользовательскую роль для пользователей, которым разрешено создавать VPN-шлюзы и подключения к локальным средам и управлять ими.

### <a name="example-powershell-script"></a>Пример скрипта PowerShell

Пример скрипта PowerShell для создания пользовательской политики IPSEC/IKE, которая соответствует элементам управления ISM для безопасной защиты в Австралии.

Предполагается, что виртуальная сеть, VPN-шлюз и локальные шлюзы существуют.

#### <a name="create-an-ipsecike-policy"></a>Создайте политику IPsec/IKE

Следующий пример скрипта создает политику IPsec/IKE со следующими параметрами и алгоритмами:

- IKEv2: AES256, SHA256, Дхграуп ECP256
- IPsec: AES256, SHA256, PFS ECP256, время жизни SA 14 400 секунд & 102400000 КБ

```powershell
$custompolicy = New-AzIpsecPolicy `
                    -IkeEncryption AES256 `
                    -IkeIntegrity SHA256 `
                    -DhGroup ECP256 `
                    -IpsecEncryption AES256 `
                    -IpsecIntegrity SHA256 `
                    -PfsGroup ECP256 `
                    -SALifeTimeSeconds 14400 `
                    -SADataSizeKilobytes 102400000
```

#### <a name="create-a-s2s-vpn-connection-with-the-custom-ipsecike-policy"></a>Создание VPN-подключения S2S с помощью настраиваемой политики IPsec/IKE

```powershell
$vpngw = Get-AzVirtualNetworkGateway `
                    -Name "<yourVPNGatewayName>" `
                    -ResourceGroupName "<yourResourceGroupName>"
$localgw = Get-AzLocalNetworkGateway  `
                    -Name "<yourLocalGatewayName>" `
                    -ResourceGroupName "<yourResourceGroupName>"

New-AzVirtualNetworkGatewayConnection `
                    -Name "ConnectionName" `
                    -ResourceGroupName "<yourResourceGroupName>" `
                    -VirtualNetworkGateway1 $vpngw `
                    -LocalNetworkGateway2 $localgw `
                    -Location "Australia Central" `
                    -ConnectionType IPsec `
                    -IpsecPolicies $custompolicy `
                    -SharedKey "AzureA1b2C3"
```
