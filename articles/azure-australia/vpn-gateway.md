# <a name="azure-vpn-gateway-in-azure-australia"></a>VPN-шлюз Azure в Австралии Azure

Критической службой с любым общедоступным облаком является безопасное подключение облачных ресурсов и служб к существующим локальным системам. Служба, которая предоставляет эту возможность в Azure, — это VPN-шлюз Azure. В этой статье описываются ключевые моменты, которые следует учитывать при настройке VPN-шлюза в соответствии с [элементами управления ручной информационной безопасности (ISM](https://acsc.gov.au/infosec/ism/)) принимало.

VPN-шлюз используется для отправки зашифрованного трафика между виртуальной сетью в Azure и другой сетью. VPN-шлюзы решают три сценария:

- "Сеть — сеть" (S2S)
- Точка-сеть (P2S)
- Сеть — сеть

В этой статье основное внимание уделяется VPN-шлюзам S2S. На схеме 1 показан пример конфигурации VPN-шлюза S2S.

![VPN-шлюз с многосайтовыми подключениями](media/vpngateway-multisite-connection-diagram.png)

## <a name="key-design-considerations"></a>Основные рекомендации по проектированию

Существует три сетевых способа подключения Azure к клиентам Австралии для государственных организаций.

- ЗНАЧОК
- Azure ExpressRoute
- Общедоступный Интернет

В статье о потребителе кибератак Security Center [для Azure](https://servicetrust.microsoft.com/viewpage/Australia) рекомендуется использовать VPN-шлюз (или ЭКВИВАЛЕНТную защищенную сертифицированную службу стороннего поставщика) в сочетании с тремя сетевыми параметрами. Эта рекомендация заключается в том, чтобы обеспечить соответствие подключений элементам управления ISM для шифрования и целостности.

### <a name="encryption-and-integrity"></a>Шифрование и целостность

По умолчанию VPN согласовывает алгоритмы шифрования и целостности и параметры во время установки соединения в рамках подтверждения IKE. В ходе подтверждения IKE конфигурация и порядок предпочтения зависят от того, является ли VPN-шлюз инициатором или респондентом. Эта схема управляется через VPN-устройство. Окончательная конфигурация подключения управляется конфигурацией VPN-устройства. Дополнительные сведения о проверенных VPN-устройствах и их конфигурации см. в статье [о службах VPN](https://docs.microsoft.com/azure/vpn-gateway/vpn-gateway-about-vpn-devices).

VPN-шлюзы могут управлять шифрованием и целостностью, настраивая пользовательскую политику IPsec/IKE для соединения.

### <a name="resource-operations"></a>Операции с ресурсами

VPN-шлюзы создают подключение между средами Azure и средой, отличной от Azure, через общедоступный Интернет. ISM имеет элементы управления, связанные с явной авторизацией подключений. По умолчанию VPN-шлюзы можно использовать для создания неавторизованных туннелей в безопасных средах. Очень важно, чтобы организации использовали управление доступом на основе ролей (RBAC) Azure, чтобы контролировать, кто может создавать и изменять VPN-шлюзы и их подключения. В Azure нет встроенной роли для управления VPN-шлюзами, поэтому требуется настраиваемая роль.

Доступ к ролям "владелец", "участник" и "участник сети" тесно контролируется. Также рекомендуется использовать Azure Active Directory Privileged Identity Management для более детального контроля доступа.

### <a name="high-availability"></a>Высокая надежность

VPN-шлюзы Azure могут иметь несколько подключений и поддерживать несколько локальных VPN-устройств в одной локальной среде. См. схему 1.

Виртуальные сети в Azure могут иметь несколько VPN-шлюзов, которые можно развертывать в независимых, активных и активных конфигурациях.

Рекомендуется развернуть все VPN-шлюзы в [конфигурации высокой доступности](https://docs.microsoft.com/azure/vpn-gateway/vpn-gateway-highlyavailable). В качестве примера можно привести два локальных VPN-устройства, подключенные к двум VPN-шлюзам в режиме "активный — пассивный" или "активный — активный". См. схему 2.

### <a name="forced-tunneling"></a>Принудительное туннелирование

Принудительное туннелирование перенаправляет или заставляет весь Интернет-трафик обратно в локальную среду через VPN-шлюз для проверки и аудита. Без принудительного туннелирования Интернет-трафик с виртуальных машин в Azure просматривает инфраструктуру сети Azure напрямую в общедоступный Интернет без возможности проверки или аудита трафика. Принудительное туннелирование является критически важным, если организация требует использовать безопасный Шлюз Интернета (SIG) для среды.

## <a name="detailed-configuration"></a>Подробные сведения

### <a name="service-attributes"></a>Атрибуты службы

VPN-шлюзы для подключений S2S, настроенных для государственных организаций Австралии, должны иметь следующие атрибуты:

|Атрибут | Приходит|
|--- | --- |
|gatewayType | ЧЕРЕЗ|
|

Параметры атрибутов, необходимые для обеспечения соответствия элементам управления ISM для защиты,:

|Атрибут | Приходит|
|--- |---|
|vpnType |RouteBased|
|vpnClientConfiguration/Впнклиентпротоколс | IkeV2|
|

VPN-шлюзы Azure поддерживают ряд алгоритмов шифрования из стандартов протокола IPsec и IKE. Политика по умолчанию задает максимальную совместимость с широким диапазоном VPN-устройств сторонних производителей. В результате может быть согласована несоответствующая конфигурация в ходе подтверждения IKE. Настоятельно рекомендуется применять [Настраиваемые параметры политики IPSec/IKE](https://docs.microsoft.com/azure/vpn-gateway/vpn-gateway-ipsecikepolicy-rm-powershell) к vpnClientConfiguration в VPN-шлюзах, чтобы обеспечить соответствие подключений ЭЛЕМЕНТАМ управления ISM для подключений к локальной среде в Azure. Ключевые атрибуты показаны в следующей таблице.

|Атрибут|Следует|Приходит|
|---|---|---|
|салифетимесекондс|< 14400 сек|> 300 сек|
|садатасизекилобитес| |> 1024 КБ|
|ипсеценкриптион| |AES256 — GCMAES256|
|ипсеЦинтегрити| |SHA256 — GCMAES256|
|икинкриптион| |AES256 — GCMAES256|
|икеинтегрити| |SHA256 — GCMAES256|
|дхграуп|DHGroup14, DHGroup24, ECP256, ECP384|DHGroup2|
|пфсграуп|PFS2048, PFS24, ECP256, ECP384||
|

Для Дхграуп и Пфсграуп в предыдущей таблице предпочтительными являются ECP256 и ECP384, даже если можно использовать другие параметры.

## <a name="implementation-steps-using-powershell"></a>Действия по реализации с помощью PowerShell

### <a name="role-based-access-control"></a>Управление доступом на основе ролей

1. Здесь описывается создание настраиваемой роли. Например, участник шлюза виртуальной сети. Создайте роль, назначаемую пользователям, которым будет разрешено создавать и изменять VPN-шлюзы. Пользовательская роль должна допускать следующие операции:

   Microsoft. Network/virtualNetworkGateways/*  
   Microsoft. Network/Connections/*  
   Microsoft. Network/локалнетворкгатевайс/*  
   Microsoft. Network/virtualNetworks/подсети/*  
   Microsoft. Network/publicIPAddresses/*  
   Microsoft. Network/ПублиЦиппрефиксес/*  
   Microsoft. Network/Раутетаблес/*  

2. Добавьте пользовательскую роль для пользователей, которым разрешено создавать VPN-шлюзы и подключения к локальным средам и управлять ими.

### <a name="create-a-vpn-gateway"></a>Создание VPN-шлюза

В этих шагах предполагается, что вы уже создали виртуальную сеть.

1. Создайте новый общедоступный IP-адрес.
2. Создайте подсеть VPN-шлюза.
3. Создайте файл конфигурации IP VPN-шлюза.
4. Создайте VPN-шлюз.
5. Создайте локальный сетевой шлюз для локального VPN-устройства.
6. Создайте политику IPsec. В этом шаге предполагается, что вы используете пользовательские политики IPsec/IKE.
7. Создайте подключение между VPN-шлюзом и шлюзом локальной сети с помощью политики IPsec.

### <a name="enforce-tunneling"></a>Принудительное туннелирование

Если требуется принудительное туннелирование, перед созданием VPN-шлюза выполните следующие действия.

1. Создайте таблицу маршрутов и правила маршрута.
2. Свяжите таблицу маршрутов с соответствующими подсетями.

### <a name="example-powershell-script"></a>Пример сценария PowerShell

Пример сценария PowerShell, используемого для создания пользовательской политики IPsec/IKE, соответствует элементам управления ISM для безопасной защиты в Австралии.

Предполагается, что виртуальная сеть, VPN-шлюз и локальные шлюзы существуют.

#### <a name="create-an-ipsecike-policy"></a>Создайте политику IPsec/IKE

Следующий пример скрипта создает политику IPsec/IKE со следующими параметрами и алгоритмами:

- IKEv2: AES256, SHA256, Дхграуп ECP256
- IPsec: AES256, SHA256, PFS ECP256, время жизни SA 14 400 с и 102 400 000 КБ

```powershell
$custompolicy = New-AzIpsecPolicy `
                    -IkeEncryption AES256 `
                    -IkeIntegrity SHA256 `
                    -DhGroup ECP256 `
                    -IpsecEncryption AES256 `
                    -IpsecIntegrity SHA256 `
                    -PfsGroup ECP256 `
                    -SALifeTimeSeconds 14400 `
                    -SADataSizeKilobytes 102400000
```

#### <a name="create-a-s2s-vpn-connection-with-the-custom-ipsecike-policy"></a>Создание VPN-подключения S2S с помощью настраиваемой политики IPsec/IKE

```powershell
$vpngw = Get-AzVirtualNetworkGateway `
                    -Name "<yourVPNGatewayName>" `
                    -ResourceGroupName "<yourResourceGroupName>"
$localgw = Get-AzLocalNetworkGateway  `
                    -Name "<yourLocalGatewayName>" `
                    -ResourceGroupName "<yourResourceGroupName>"

New-AzVirtualNetworkGatewayConnection `
                    -Name "ConnectionName" `
                    -ResourceGroupName "<yourResourceGroupName>" `
                    -VirtualNetworkGateway1 $vpngw `
                    -LocalNetworkGateway2 $localgw `
                    -Location "Australia Central" `
                    -ConnectionType IPsec `
                    -IpsecPolicies $custompolicy `
                    -SharedKey "AzureA1b2C3"
```
