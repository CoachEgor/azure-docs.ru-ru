# <a name="actions-in-the-standard-rules-engine-for-azure-cdn"></a>Действия в движке стандартных правил для Azure CDN

В [движке стандартных правил](cdn-standard-rules-engine.md) для сети доставки содержимого Azure (Azure CDN) правило состоит из одного или нескольких условий соответствия и действия. В этой статье приводится подробное описание действий, которые можно использовать в движке стандартных правил для Azure CDN.

Вторая часть правила – это действие. Действие определяет поведение, применяемое к типу запроса, которое определяет условие соответствия или набор условий соответствия.

## <a name="actions"></a>Действия

Следующие действия доступны для использования в движке стандартных правил для Azure CDN. 

### <a name="cache-expiration"></a>Срок действия кэша

Используйте это действие, чтобы перезаписать значение тихих действий (TTL) конечной точки для запросов, которые указаны условиями соответствия правил.

#### <a name="required-fields"></a>Обязательные поля

Поведение кэша |  Описание              
---------------|----------------
Обход кэша | Когда этот параметр выбран и правило совпадает, содержимое не кэшируется.
Переопределение | Когда этот параметр выбран и правило совпадает, значение TTL, возвращенное из вашего происхождения, перезаписано со значением, указанным в действии.
Установить, если отсутствует | Когда этот параметр выбран и правило совпадает, если значение TTL не было возвращено из вашего происхождения, правило устанавливает TTL к значению, указанному в действии.

#### <a name="additional-fields"></a>Дополнительные поля

Дни | Часы | Минуты | Секунды
-----|-------|---------|--------
Int | Int | Int | Int 

### <a name="cache-key-query-string"></a>Строка запроса ключа кэша

Используйте это действие, чтобы изменить ключ кэша на основе строк запроса.

#### <a name="required-fields"></a>Обязательные поля

Поведение | Описание
---------|------------
Включить | При выборе этой опции и соответствии правила строки запроса, указанные в параметрах, включаются при генерировании ключа кэша. 
кэшировать все уникальные URL-адреса. | Когда эта опция выбрана и правило совпадает, каждый уникальный URL имеет свой собственный ключ кэша. 
Исключить | При выборе этого параметра и соответствии правила строки запроса, указанные в параметрах, исключаются при генерировании ключа кэша.
игнорировать строки запросов; | При выборе этой опции и соответствии правил строки запроса не учитываются при сгенерировании ключа кэша. 

### <a name="modify-request-header"></a>Изменение заголовка запроса

Используйте это действие для изменения заголовков, которые присутствуют в запросах, отправленных вашему происхождению.

#### <a name="required-fields"></a>Обязательные поля

Действие | Имя заголовка HTTP | Значение
-------|------------------|------
Append | При выборе этой опции и соответствии правила, заголовок, указанный в **имени заголовка,** добавляется в запрос с указанным значением. Если заголовок уже присутствует, значение прикладивается к существующему значению. | Строка
Overwrite | При выборе этой опции и соответствии правила, заголовок, указанный в **имени заголовка,** добавляется в запрос с указанным значением. Если заголовок уже присутствует, указанное значение перезаписывает существующее значение. | Строка
DELETE | При выборе этой опции правило совпадает, и заголовок, указанный в правиле, присутствует, заголовок удаляется из запроса. | Строка

### <a name="modify-response-header"></a>Изменение заголовка ответа

Используйте это действие для изменения заголовков, которые присутствуют в ответах, возвращенных вашим клиентам.

#### <a name="required-fields"></a>Обязательные поля

Действие | Имя заголовка HTTP | Значение
-------|------------------|------
Append | При выборе этой опции и соответствии правила, заголовок, указанный в **имени заголовка,** добавляется в ответ с помощью указанного **значения.** Если заголовок уже присутствует, **значение** прикладивается к существующему значению. | Строка
Overwrite | При выборе этой опции и соответствии правила, заголовок, указанный в **имени заголовка,** добавляется в ответ с помощью указанного **значения.** Если заголовок уже присутствует, **значение** перезаписывает существующее значение. | Строка
DELETE | При выборе этого параметра правило совпадает, и заголовок, указанный в правиле, присутствует, заголовок удаляется из ответа. | Строка

### <a name="url-redirect"></a>Перенаправление URL-адреса

Используйте это действие, чтобы перенаправить клиентов на новый URL. 

#### <a name="required-fields"></a>Обязательные поля

Поле | Описание 
------|------------
Тип | Выберите тип ответа, чтобы вернуться к запрашиванию: Найдено (302), Перемещено (301), Временное перенаправление (307) и Постоянное перенаправление (308).
Протокол | Запрос на матч, HTTP, HTTPS.
Имя узла | Выберите имя хоста, на которое вы хотите перенаправить запрос. Оставьте пустым, чтобы сохранить входящий узла.
путь | Определите путь для использования в перенаправлении. Оставьте пустым, чтобы сохранить входящий путь.  
Строка запроса | Определите строку запроса, используемую в перенаправлении. Оставьте пустым, чтобы сохранить строку входящего запроса. 
Fragment (Фрагмент) | Определите фрагмент для использования в перенаправлении. Оставьте пустым, чтобы сохранить входящий фрагмент. 

Мы настоятельно рекомендуем использовать абсолютный URL- АДРЕС. Использование относительного URL-адреса может перенаправить URL-адреса Azure CDN на недействительный путь. 

### <a name="url-rewrite"></a>Переопределение URL-адреса

Используйте это действие, чтобы переписать путь запроса, который на пути к вашему происхождению.

#### <a name="required-fields"></a>Обязательные поля

Поле | Описание 
------|------------
Шаблон источника | Определите шаблон исходного кода в пути URL для замены. В настоящее время шаблон исходного кода использует совпадение на основе префикса. Чтобы соответствовать всем траекториям URL, используйте передний слэш ()**/** в качестве значения исходного шаблона.
Назначение | Определите путь назначения для использования в перезаписи. Путь назначения перезаписывает шаблон исходного кода.
Сохранить непревзойденный путь | Если установить **да,** оставшийся путь после шаблона исходного кода пригожен к новому пути назначения.
