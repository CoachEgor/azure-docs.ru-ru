# <a name="expressroute-workflows-for-circuit-provisioning-and-circuit-states"></a>Процедуры ExpressRoute для подготовки каналов и состояний каналов
На этой странице описаны процедуры подготовки служб и настройки маршрутизации на высоком уровне.

![Рабочий процесс настройки канала](./media/expressroute-workflows/expressroute-circuit-workflow.png)

На следующих цифрах и соответствующих шагах излагаются задачи по обеспечению сквозной схемы ExpressRoute. 

1. Настройте канал ExpressRoute с помощью PowerShell. Для этого выполните инструкции в статье [Создание каналов ExpressRoute](expressroute-howto-circuit-classic.md) .
2. Закажите подключение у поставщика услуг. Порядок действий может варьироваться. За подробной информацией о заказе подключения обратитесь к поставщику услуг подключения.
3. Убедитесь, что канал был подготовлен, проверив состояние подготовки канала ExpressRoute с помощью PowerShell. 
4. Настройте домены маршрутизации. Если поставщик подключений управляет конфигурацией уровня 3, он настраивает маршрутизатор для вашей цепи. Если поставщик услуг подключения предлагает только услуги уровня 2, необходимо настроить routing в соответствии с рекомендациями, описанными в [требованиях к реукторе](expressroute-routing.md) и страницах [конфигурации.](expressroute-howto-routing-classic.md)
   
   * Включите частный пиринг Azure — он требуется для подключения к виртуальным машинам и (или) облачным службам, развернутым в виртуальных сетях.

   * Включите Microsoft peering - Включите это для доступа к онлайн-сервисам Майкрософт, таким как Office 365. Все службы Azure PaaS доступны через пиринг Майкрософт.
     
     > [!IMPORTANT]
     > Для подключения к Майкрософт нельзя использовать прокси-сервер или ресурс, который используется для подключения к Интернету. Подключение к ExpressRoute и к Интернету через один и тот же ресурс приведет к асимметричной маршрутизации и вызовет проблемы подключения в вашей сети.
     > 
     > 
     
     ![Рабочие процессы маршрутизации](./media/expressroute-workflows/routing-workflow.png)
5. Свяжите виртуальные сети с каналами ExpressRoute. Виртуальные сети можно связать с каналом ExpressRoute. Выполните инструкции по [связыванию виртуальных сетей](expressroute-howto-linkvnet-arm.md) с каналом. Эти виртуальные сети могут входить в ту же самую подписку, что и канал ExpressRoute, или в другую.

## <a name="expressroute-circuit-provisioning-states"></a>Состояния подготовки канала ExpressRoute
У каждого канала ExpressRoute имеется два состояния.

* Состояние подготовки поставщика услуг
* Состояние

Состояние указывает на состояние подготовки Майкрософт. При создании канала Expressroute это свойство имеет значение "Включен".

Состояние подготовки поставщика услуг подключения обозначает состояние на стороне поставщика. Возможные состояния: *NotProvisioned* (Не подготовлен), *Provisioning* (Идет подготовка) и *Provisioned* (Подготовлен). Схема ExpressRoute должна находиться в состоянии, предусмотренном для настройки пиринга.

### <a name="possible-states-of-an-expressroute-circuit"></a>Возможные состояния канала ExpressRoute
В этом разделе перечислены возможные состояния схемы ExpressRoute.

**В момент создания**

Схема ExpressRoute сообщит о следующих состояниях при создании ресурсов.

    ServiceProviderProvisioningState : NotProvisioned
    Status                           : Enabled


**Пока поставщик услуг подключения выполняет подготовку канала**

Схема ExpressRoute сообщит о следующих состояниях, в то время как поставщик подключения работает над обеспечением цепи.

    ServiceProviderProvisioningState : Provisioning
    Status                           : Enabled


**После завершения процесса подготовки поставщиком услуг подключения**

Схема ExpressRoute сообщит о следующих состояниях после того, как поставщик подключений успешно проработает схему.

    ServiceProviderProvisioningState : Provisioned
    Status                           : Enabled


**Пока поставщик услуг подключения отзывает канал**

Если схема ExpressRoute должна быть дезавуирована, схема сообщит о следующих состояниях, как только поставщик услуг завершит процесс деобеспечения.

    ServiceProviderProvisioningState : NotProvisioned
    Status                           : Enabled


При необходимости вы можете снова включить канал или удалить его с помощью командлетов PowerShell.  

> [!IMPORTANT]
> Схема не может быть удалена, когда ServiceProvideringState предоставляетили или предостережали. Поставщику подключения необходимо дезавуировать схему, прежде чем она может быть удалена. Корпорация Майкрософт будет продолжать выставлять счета за схему до тех пор, пока в Azure не будет удален ресурс схемы ExpressRoute.
> 

## <a name="routing-session-configuration-state"></a>Состояние конфигурации сеанса маршрутизации
Положение о состоянии BGP сообщает, включена ли сессия BGP на краю Майкрософт. Состояние должно быть включено в использование приватного или электронного пиринга Майкрософт.

Обязательно проверьте состояние сеанса BGP, особенно для пиринга Майкрософт. Еще одно состояние, которое нужно учитывать, называется *состояние объявленных общедоступных префиксов*. Состояние рекламируемых общедоступных префиксов должно находиться в *настроенном* состоянии, как для того, чтобы сессия BGP была настроена, так и для того, чтобы ваша реуктора работала сквозной. 

Если для состояния объявленных общедоступных префиксов установлено значение *Validation needed* (Требуется проверка), сеанс BGP не включается, поскольку объявленные префиксы не совпадают с номером AS ни в одном реестре маршрутизации. 

> [!IMPORTANT]
> Если рекламируемое состояние общедоступных префиксов находится в состоянии *ручной проверки,* необходимо открыть билет поддержки с [поддержкой Майкрософт](https://portal.azure.com/?#blade/Microsoft_Azure_Support/HelpAndSupportBlade) и предоставить доказательства того, что вы владеете IP-адресами, рекламируемыми вместе с соответствующим номером Автономной системы.
>
