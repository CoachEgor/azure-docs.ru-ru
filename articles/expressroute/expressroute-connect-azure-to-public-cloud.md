# <a name="connecting-azure-with-public-clouds"></a>Соединение Azure с общедоступными облаками

Многие предприятия проводят многооблачную стратегию из-за бизнес-целей и технических целей. К ним относятся затраты, гибкость, доступность функций, избыточность, суверенитет данных и т.д. Эта стратегия помогает им использовать лучшие из обоих облаков. 

Такой подход также создает проблемы для предприятия с точки зрения архитектуры сети и приложений. Некоторые из этих проблем являются задержка и пропускная стоимость данных. Для решения этих проблем клиенты хотят подключиться к нескольким облакам напрямую. Некоторые поставщики услуг предоставляют решение для подключения нескольких поставщиков облачных услуг для клиентов. В других случаях клиент может развернуть свой собственный маршрутизатор для подключения нескольких общедоступных облаков.
## <a name="connectivity-via-expressroute"></a>Связь через ExpressRoute
ExpressRoute позволяет клиентам расширять свои сети в облаке Майкрософт по частному подключению, которому способствует поставщик подключения. С помощью ExpressRoute клиенты могут устанавливать связи с облачными службами Майкрософт.

Есть три способа подключения через ExpressRoute.

1. Поставщик layer3
2. Поставщик layer2
3. Прямое соединение

### <a name="layer3-provider"></a>Поставщик layer3

Поставщики Layer3 широко известны как IP VPN или MPLS VPN провайдеров. Клиенты используют этих поставщиков для многоточечной связи между центрами обработки данных, филиалами и облаком. Клиенты подключаются к поставщику L3 через BGP или по статическому маршруту по умолчанию. Поставщик услуг рекламирует маршруты между сайтами клиентов, центрами обработки данных и общедоступным облаком. 
 
При подключении через поставщика Layer3 корпорация Майкрософт будет рекламировать маршруты клиентов VNET поставщику услуг через BGP. Поставщик может иметь две разные реализации.

![](media/expressroute-connect-azure-to-public-cloud/azure-to-public-clouds-l3.png)

Поставщик может высадить каждого поставщика облачных услуг в отдельный VRF, если трафик от всех поставщиков облачных услуг достигнет маршрутизатора клиента. Если клиент работает BGP с поставщиком услуг, то эти маршруты будут повторно объявлены другим поставщикам облачных услуг по умолчанию. 

Если поставщик услуг выставляет всех поставщиков облачных услуг в тот же VRF, то маршруты будут рекламироваться другим поставщикам облачных услуг непосредственно от поставщика услуг. Это предполагает стандартную операцию BGP, когда маршруты eBGP рекламируются другим соседям eBGP по умолчанию.

Каждое общедоступное облако имеет различные пределы префикса, поэтому при распространении маршрутов поставщик услуг должен проявлять осторожность при распределении маршрутов.

### <a name="layer2-provider-and-direct-connection"></a>Поставщик и прямое подключение Layer2

Хотя физическое подключение в обеих моделях отличается, но на уровне 3 BGP устанавливается непосредственно между MSEE и маршрутизатором клиента. Для ExpressRoute Прямая клиент подключается к MSEE напрямую. В случае layer2 поставщик услуг расширяет VLAN из помещений клиента до облака. Клиенты заключают BGP в верхней части сети layer2 для подключения своих ОК к облаку.
![](media/expressroute-connect-azure-to-public-cloud/azure-to-public-clouds-l2.png)
В обоих случаях клиент будет иметь точечные соединения с каждым из общедоступных облаков. Клиент установит отдельное соединение BGP с каждым общедоступным облаком. Маршруты, полученные одним поставщиком облачных услуг, будут рекламироваться другому поставщику облачных услуг по умолчанию. Каждый поставщик облачных услуг имеет различные ограничения префикса, поэтому при рекламе маршрутов клиент должен заботиться об этих ограничениях. Клиент может использовать обычные ручки BGP с Microsoft во время рекламных маршрутов из других общедоступных облаков.

## <a name="direct-connection-with-expressroute"></a>Прямая связь с ExpressRoute

Клиенты могут подключить ExpressRoute непосредственно к предложению прямого подключения поставщика облачных услуг. Два облачных провайдера будут подключены к спине и BGP будет создан непосредственно между их маршрутизаторами. Этот тип связи доступен с Oracle сегодня.

## <a name="site-to-site-vpn"></a>VPN "сайт — сайт"

Клиенты могут использовать Интернет для подключения своих экземпляров в Azure с другими общедоступными облаками. Почти все поставщики облачных услуг предлагают VPN-возможности от сайта. Однако из-за отсутствия определенных вариантов могут возникнуть несовместимость. Например, некоторые поставщики облачных услуг поддерживают только IKEv1, поэтому в этом облаке требуется конечная точка завершения VPN. Для тех поставщиков облачных услуг, поддерживающих IKEv2, может быть установлен прямой туннель между VPN шлюзами у обоих поставщиков облачных услуг.

VPN от сайта к сайту не считается решением с высокой пропускной устойчивостью и низкой задержкой. Тем не менее, он может быть использован в качестве резервного копирования для физического подключения.
