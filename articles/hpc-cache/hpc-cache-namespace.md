# <a name="plan-the-aggregated-namespace"></a>Проектирование агрегированного пространства имен

Azure HPC Cache позволяет клиентам получить доступ к различным системам хранения через виртуальное пространство имен, которое скрывает детали системы хранения бэк-энда.

При добавлении цели хранения вы устанавливаете путь файла, обращенного к клиенту. Клиентские машины устанавливают этот путь файла и могут делать запросы на чтение файлов в кэш вместо непосредственного монтажа системы хранения.

Поскольку Azure HPC Cache управляет этой виртуальной файловой системой, можно изменить цель хранения, не изменяя траекторию ожидания клиента. Например, можно заменить систему хранения оборудования на облачное хранилище без необходимости переписать процедуры, стоящие перед клиентами.

## <a name="aggregated-namespace-example"></a>Пример агрегированного пространства имен

Планируйте агрегированное пространство имен таким образом, чтобы клиентские машины могли удобно добираться до необходимой информации, а также чтобы администраторы и инженеры рабочего процесса могли легко различать пути.

Например, рассмотрим систему, в которой экземпляр Azure HPC Cache используется для обработки данных, хранящихся в Azure Blob. Для анализа требуются файлы шаблонов, которые хранятся в локальном центре обработки данных.

Данные шаблона хранятся в центре обработки данных, а информация, необходимая для этой работы, хранится в следующих подкаталогах:

    /goldline/templates/acme2017/sku798
    /goldline/templates/acme2017/sku980 

Система хранения центров обработки данных предоставляет следующие экспортные товары:

    /
    /goldline
    /goldline/templates

Анализируемые данные были скопированы в контейнер для хранения Azure Blob под названием "источник" с помощью [утилиты CLFSLoad.](hpc-cache-ingest.md#pre-load-data-in-blob-storage-with-clfsload)

Чтобы обеспечить легкий доступ через кэш, рассмотрите возможность создания целей хранения с помощью этих виртуальных путей пространства имен:

| Система хранения бэк-энда <br/> (Путь файла NFS или контейнер Blob) | Виртуальный путь пространства имен |
|-----------------------------------------|------------------------|
| /goldline/templates/acme2017/sku798     | /templates/sku798      |
| /goldline/templates/acme2017/sku980     | /шаблоны/sku980      |
| источникколлекции                        | /источник/               |

Цель хранения NFS может иметь несколько виртуальных путей пространства имен, если каждый из них ссылается на уникальный путь экспорта.

Поскольку пути источника NFS являются субдиректорами одного и того же экспорта, необходимо определить несколько путей пространства имен из одной и той же цели хранения.

| Целевой хост-хост хранилища  | Путь экспорта NFS      | Путь субдиректоров | Путь пространства имен    |
|--------------------------|----------------------|-------------------|-------------------|
| *IP-адрес или хост-имя* | /goldline/templates  | acme2017/sku798   | /templates/sku798 |
| *IP-адрес или хост-имя* | /goldline/templates  | acme2017/sku980   | /шаблоны/sku980 |

Клиентское приложение может смонтировать кэш и легко получить ``/source`` ``/templates/sku798``доступ ``/templates/sku980``к агрегированным пути файла мэш-пространства и.
