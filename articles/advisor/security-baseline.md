# <a name="azure-security-baseline-for-azure-advisor"></a>Базовый план безопасности Azure для помощника по Azure

Этот базовый план безопасности применяет рекомендации из [тестового показателя безопасности Azure версии 2,0](../security/benchmarks/overview.md) к советнику по Azure. Azure Security Benchmark содержит рекомендации по обеспечению безопасности облачных решений в Azure. Содержимое группируются по **элементам управления безопасностью** , определенным в тестовом производительности системы безопасности Azure, и связанным рекомендациям, применимым к помощнику по Azure. **Элементы управления** , неприменимые к помощнику по Azure, были исключены.

Чтобы узнать, как Azure Advisor полностью сопоставляется с тестовым показателем безопасности Azure, ознакомьтесь с [полным файлом сопоставления базовых показателей безопасности Azure Advisor](https://github.com/MicrosoftDocs/SecurityBenchmarks/tree/master/Azure%20Offer%20Security%20Baselines).

### <a name="im-1-standardize-azure-active-directory-as-the-central-identity-and-authentication-system"></a>IM-1: стандартизация Azure Active Directory в качестве централизованной системы идентификации и проверки подлинности

**Руководство**. Помощник по azure использует Azure Active Directory (Azure AD) в качестве службы управления удостоверениями и доступом по умолчанию. Стандартизация Azure AD для управления удостоверениями и доступом в вашей организации в:

- Microsoft Cloud ресурсы, такие как портал Azure, служба хранилища Azure, виртуальная машина Azure (Linux и Windows), Azure Key Vault, PaaS и приложения SaaS
- Ресурсы Организации, такие как приложения в Azure или ресурсы корпоративной сети;

Обеспечьте безопасность Azure AD с высоким приоритетом в целях безопасности в облаке вашей организации. Azure AD также предоставляет удостоверение безопасного удостоверения, помогающее оценить безопасность идентификации по сравнению с рекомендациями Майкрософт. Используйте оценку, чтобы оценить, насколько близко ваша конфигурация соответствует рекомендациям, и улучшить безопасность.

Обратите внимание, что Azure AD поддерживает внешние удостоверения, которые позволяют пользователям без учетная запись Майкрософт входить в свои приложения и ресурсы с их внешним удостоверением.

- [Использование внешних поставщиков удостоверений для приложения](/azure/active-directory/b2b/identity-providers)

Единый вход используется для управления доступом к данным и ресурсам Организации в локальной среде и в облаке, а также для обеспечения безопасного доступа к ним. Подключайте пользователей, приложения и устройства к Azure AD, чтобы обеспечить простой, безопасный доступ и более широкие возможности управления.

### <a name="im-4-use-strong-authentication-controls-for-all-azure-active-directory-based-access"></a>IM-4. использование строгих средств проверки подлинности для всех Azure Active Directory доступом

**Руководство**. Помощник по azure использует Azure Active Directory (Azure AD), который поддерживает надежные элементы управления проверкой подлинности с помощью многофакторной проверки подлинности и надежные методы, не требующие пароля.
- Многофакторная проверка подлинности. Включите многофакторную проверку подлинности Azure AD и следуйте рекомендациям по управлению удостоверениями и доступом центра безопасности Azure для некоторых рекомендаций в настройке многофакторной проверки подлинности. Многофакторная проверка подлинности может быть применена ко всем, выбирать пользователей или на уровне отдельных пользователей в зависимости от условий входа и факторов риска.
- Проверка подлинности с паролем — доступны три варианта проверки подлинности с паролем. К ним относятся, Windows Hello для бизнеса, Microsoft Authenticator приложение и локальные методы проверки подлинности, такие как смарт-карты.

Для администраторов и привилегированных пользователей убедитесь, что используется самый высокий уровень надежной проверки подлинности, а затем выполнив соответствующую политику строгой проверки подлинности другим пользователям.

### <a name="im-5-monitor-and-alert-on-account-anomalies"></a>IM-5: отслеживание и оповещение об аномалиях учетной записи

**Руководство**. Помощник по Azure интегрируется с Azure Active Directory (Azure AD), в котором предоставляются следующие источники данных:
- Вход. отчет о входе содержит сведения об использовании управляемых приложений и действиях входа пользователей.
- Журналы аудита — возможность отслеживания с помощью журналов всех изменений, внесенных при использовании разных функций в Azure AD. Примеры журналов аудита включают изменения, внесенные в такие ресурсы в Azure AD, как добавление или удаление пользователей, приложений, групп, ролей и политик.
- Рискованный вход. рискованный вход — это индикатор попытки входа, которая могла быть выполнена пользователем, который не является законным владельцем учетной записи пользователя.
- Пользователи, находящиеся в группе риска. Такая пометка означает, что конфиденциальность учетной записи пользователя, возможно, нарушена.

Используйте эти источники данных для интеграции с Azure Monitor, Sentinel Azure или сторонними системами SIEM. Настройка оповещений в центре безопасности Azure для определенных подозрительных действий, например чрезмерного числа неудачных попыток проверки подлинности, нерекомендуемых учетных записей в подписке.

Azure Advanced Threat protection (ATP) — это решение для обеспечения безопасности, которое может использовать Active Directory сигналы для определения, обнаружения и исследования сложных угроз, скомпрометированных удостоверений и действий по предварительной проверке.

### <a name="pa-2-restrict-administrative-access-to-business-critical-systems"></a>PA-2: ограничение административного доступа к системам, критически важным для бизнеса

### <a name="gs-7-define-logging-and-threat-response-strategy"></a>GS-7: определение стратегии ведения журнала и реагирования на угрозы

**Руководство**. Создание стратегии регистрации и реагирования на угрозы для быстрого обнаружения и устранения угроз при соблюдении требований соответствия требованиям. Определение приоритетов для аналитиков с помощью высококачественных оповещений и беспрепятственного взаимодействия, чтобы они могли сосредоточиться на угрозах, а не на интеграции и ручных действиях. 

Эта стратегия должна включать документированное руководство, политику и стандарты для следующих элементов: 

-   Роль и обязанности Организации по обеспечению безопасности (SecOps) 

-   Четко определенный процесс реагирования на инциденты, согласованный с NIST или другой платформой отрасли 

-   Сбор и хранение журналов для поддержки обнаружения угроз, реагирования на инциденты и соответствия требованиям

-   Централизованная видимость и корреляция сведений о угрозах, использование SIEM, собственные возможности Azure и другие источники 

-   Связь и план уведомлений с вашими клиентами, поставщиками и общедоступными сторонами

-   Использование платформ Azure native и сторонних производителей для обработки инцидентов, таких как ведение журнала и обнаружение угроз, судебные расследования, исправление атак и удаление

-   Процессы обработки инцидентов и действий, выполняемых после инцидента, таких как выводы и хранение доказательств
