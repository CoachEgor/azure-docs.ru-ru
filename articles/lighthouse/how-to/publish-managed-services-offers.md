# <a name="publish-a-managed-services-offer-to-azure-marketplace"></a>Публикация предложения управляемых служб в Azure Marketplace

Из этой статьи вы узнаете, как опубликовать общедоступное или частное предложение управляемых служб [в Azure Marketplace](https://azuremarketplace.microsoft.com) с помощью [Портала Cloud Partner](https://cloudpartner.azure.com/), обеспечивая подключение заказчика, который приобретает предложение, к системе делегированного управления ресурсами Azure. 

> [!NOTE]
> Для создания и публикации этих предложений необходима действительная [учетная запись в Центре партнеров](https://docs.microsoft.com/azure/marketplace/partner-center-portal/create-account). Если у вас еще нет учетной записи, в [процессе регистрации](https://aka.ms/joinmarketplace) вы сможете выполнить действия по созданию учетной записи в Центре партнеров и регистрации в программе Commercial Marketplace. Ваш идентификатор Microsoft Partner Network будет [автоматически связан](https://docs.microsoft.com/azure/billing/billing-partner-admin-link-started) с публикуемыми предложениями, чтобы отследить влияние на привлечение клиентов.
>
> Если вы не хотите публиковать предложение в Azure Marketplace, вы можете вручную подключать клиентов с помощью шаблонов Azure Resource Manager. Дополнительные сведения см. в статье [Onboard a customer to Azure delegated resource management](onboard-customer.md) (Подключение клиента к системе делегированного управления ресурсами Azure)

Публикация предложения управляемых служб аналогична публикации предложения любого другого типа в Azure Marketplace. Чтобы узнать об этом процессе, ознакомьтесь [с руководством по публикации в Azure Marketplace и AppSource](https://docs.microsoft.com/azure/marketplace/marketplace-publishers-guide) и статьей об [управлении предложениями Azure и AppSource Marketplace](https://docs.microsoft.com/azure/marketplace/cloud-partner-portal/manage-offers/cpp-manage-offers).

> [!IMPORTANT]
> Каждый план в предложении управляемых служб содержит раздел **Manifest Details** (Сведения о манифесте), в котором можно определить сущности Azure Active Directory (Azure AD) в клиенте, который будет иметь доступ к делегированным группам ресурсов и (или) подпискам для клиентов, купившим этот план. Важно помнить, что любая группа (или пользователь, или субъект-служба), добавленные на этом этапе, будут иметь одинаковые разрешения для всех клиентов, купивших план. Чтобы назначить разные группы для каждого клиента, необходимо опубликовать отдельный закрытый план, который является эксклюзивным для каждого клиента.

## <a name="create-your-offer-in-the-cloud-partner-portal"></a>Создание предложения на Портале Cloud Partner

1. Войдите на [облачный портал для партнеров](https://cloudpartner.azure.com/).
2. В меню навигации слева выберите **Новое предложение**, а затем выберите **Управляемые службы**.
3. Вы увидите раздел **Редактор** для вашего предложения с четырьмя частями, которые необходимо заполнить: **Параметры предложения**, **Планы**, **Marketplace** и **Поддержка**. Прочтите инструкции по заполнению этих разделов.

## <a name="enter-offer-settings"></a>Ввод параметров предложения

В разделе **Параметры предложения** укажите следующие данные:

|Поле  |ОПИСАНИЕ  |
|---------|---------|
|**Идентификатор предложения**     | Уникальный идентификатор предложения (в профиле издателя). Этот идентификатор может содержать только алфавитно-цифровые знаки в нижнем регистре, тире и символы подчеркивания, не более 50 символов. Помните, что идентификатор предложения может отображаться для клиентов в таких местах, как URL-адреса продуктов и отчеты о выставлении счетов. После публикации предложения это значение изменить нельзя.        |
|**Идентификатор издателя**     | Идентификатор издателя, который будет связан с предложением. При наличии нескольких идентификаторов издателей можно выбрать тот, который вы хотите использовать для этого предложения.       |
|**Имя**     | Имя (до 50 символов) вашего предложения, отображаемое для клиентов в Azure Marketplace и на портале Azure. Используйте узнаваемое название торговой марки, которое будет понятно клиентам. Если вы продвигаете это предложение через собственный веб-сайт, обязательно используйте здесь точное название.        |

По завершении нажмите кнопку **Сохранить**. Теперь вы готовы перейти к разделу **Планы**.

## <a name="create-plans"></a>Создание планов

В каждом предложении должен быть один или несколько планов (иногда их называют номерами SKU). Вы можете добавить несколько планов для поддержки различных наборов функций по разным ценам или для настройки определенного плана для ограниченной аудитории конкретных клиентов. Клиенты могут просматривать планы, доступные для них в родительском предложении.

В разделе "Планы" для каждого плана, который вы хотите создать, выберите **Создать план**. Заполните поле **Идентификатор плана**. Этот идентификатор может содержать только алфавитно-цифровые знаки в нижнем регистре, тире и символы подчеркивания, не более 50 символов. Идентификатор плана может отображаться для клиентов в таких местах, как URL-адреса продуктов и отчеты о выставлении счетов. После публикации предложения это значение изменить нельзя.

Далее заполните следующие разделы в разделе **Сведения о плане**:

|Поле  |ОПИСАНИЕ  |
|---------|---------|
|**Заголовок**     | Понятное имя плана для отображения. Максимальная длина составляет 50 символов.        |
|**Сводка**     | Краткое описание плана для отображения под заголовком. Максимальная длина составляет 100 символов.        |
|**Описание**     | Текст описания, который предоставляет более подробные характеристики плана.         |
|**Модель выставления счетов**     | Здесь показано 2 модели выставления счетов, но для предложений управляемых служб вам необходимо выбрать **С использованием собственной лицензии**. Это означает, что вы будете выставлять клиентам счета непосредственно по затратам, связанным с этим предложением, и корпорация Майкрософт не будет взимать с вас плату.   |
|**Это частный план?**     | Определяет частный или общедоступный режим использования номера SKU. По умолчанию используется значение **No** (Нет). Это значит, что номер является общедоступным. Если вы оставите этот вариант, ваш план не будет ограничен конкретными клиентами (или определенным числом клиентов). После публикации общедоступного плана его нельзя будет изменить на частный. Чтобы сделать этот план доступным только для конкретных клиентов, выберите **Да**. После этого необходимо определить клиентов, указав их идентификаторы подписок. Их можно ввести по одному (до 10 подписок) или путем отправки CSV-файла (до 20 000 подписок). Обязательно включите собственные подписки, чтобы можно было протестировать предложение. Дополнительные сведения см. в статье о [частных номерах SKU и планах](https://docs.microsoft.com/azure/marketplace/cloud-partner-portal-orig/cloud-partner-portal-azure-private-skus).  |

Наконец, заполните раздел **Manifest Details** (Сведения о манифесте). При этом создается манифест с данными авторизации для управления ресурсами клиента. Указанные здесь сведения необходимы для подключения клиентов к делегированному управлению ресурсами Azure. Как отмечалось выше, эти разрешения применяются ко всем клиентам, которые приобрели план, поэтому если вы хотите предоставить доступ конкретному клиенту, вам потребуется опубликовать частный план для эксклюзивного использования.

- Сначала укажите **версию** манифеста. Используйте формат *n. n. n* (например, 1.2.5).
- Затем заполните поле **Идентификатор клиента**. Это глобальный уникальный идентификатор, связанный с идентификатором клиента Azure Active Directory вашей организации (т. е. клиентом, с помощью которого вы будете управлять ресурсами клиентов). Если у вас нет его под рукой, его можно найти, наведя указатель мыши на имя учетной записи в правом верхнем углу портала Azure либо выбрав **Переключение каталога**. 
- Наконец, добавьте в план одну или несколько записей **Авторизация**. Авторизации определяют сущности, которые имеют доступ к ресурсам и подпискам для клиентов, купивших план. Эти сведения необходимо указать для доступа к ресурсам от имени клиента с помощью делегированного управления ресурсами Azure.
  Для каждой авторизации укажите следующие сведения. После этого вы можете выбирать **Новая авторизация** столько раз, сколько потребуется для добавления дополнительных пользователей или определений ролей.
  - **Идентификатор объекта Azure AD**. Идентификатор пользователя, группы пользователей или приложения в Azure AD, которому предоставляются определенные разрешения (как описано в пункте определения роли) на ресурсы ваших клиентов.
  - **Отображаемое имя объекта Azure AD**. Понятное имя, помогающее клиенту понять назначение авторизации. Клиент увидит это имя при делегировании ресурсов.
  - **Определение роли**. Выберите из списка одну из доступных встроенных ролей Azure AD. Эта роль определит разрешения, которые пользователь в поле **Идентификатор объекта Azure AD** будет иметь на ресурсы ваших клиентов. Сведения об этих ролях см. в статье [Built-in roles for Azure resources](https://docs.microsoft.com/azure/role-based-access-control/built-in-roles) (Встроенные роли для ресурсов Azure).
  - **Назначаемые роли**. Если вы выбрали администратора доступа пользователей в разделе **Определение роли** для этой авторизации, здесь вы можете добавить одну или несколько назначаемых ролей. Пользователь в поле **Идентификатор объекта Azure AD** сможет назначить эти **Назначаемые роли** [управляемым удостоверениям](https://docs.microsoft.com/azure/managed-applications/publish-managed-identity). Обратите внимание, что к этому пользователю не будут применяться никакие другие разрешения, обычно связанные с ролью администратора доступа пользователей. (Если вы не выбрали администратора доступа пользователей для определения роли этого пользователя, это поле не будет действовать.)

> [!TIP]
> В большинстве случаев разрешения необходимо назначить группе пользователей или субъекту-службе Azure AD, а не ряду отдельных учетных записей пользователей. В этом случае вы сможете добавлять или удалять доступ для отдельных пользователей, не обновляя и повторно публикуя план при изменении требований к доступу.

После добавления планов выберите **Сохранить**, а затем перейдите к разделу **Marketplace**.

## <a name="provide-marketplace-text-and-images"></a>Предоставление текста и изображений Marketplace

В разделе **Marketplace** указывается текст и изображения, которые клиенты увидят в Azure Marketplace и на портале Azure.

Укажите сведения в следующих полях в разделе **Обзор**:

|Поле  |ОПИСАНИЕ  |
|---------|---------|
|**Заголовок**     |  Название предложения (часто длинное официальное имя). Этот заголовок будет отображаться на видном месте в marketplace. Максимальная длина составляет 50 символов. В большинстве случаев это **имя** должно совпадать с именем, введенным в разделе **Параметры предложения**.       |
|**Сводка**     | Краткое описание назначения или функции предложения. Как правило, оно отображается под заголовком. Максимальная длина составляет 100 символов.        |
|**Long Summary** (Развернутая сводка)     | Более развернутая сводка назначения или функции вашего предложения. Максимальная длина составляет 256 символов.        |
|**Описание**     | Дополнительные сведения о вашем предложении. Максимальная длина этого поля составляет 3000 символов. Оно поддерживает простое HTML-форматирование.        |
|**Marketing Identifier** (Маркетинговый идентификатор)     | Уникальный идентификатор, доступный для использования в URL-адресе. Он будет использоваться в URL-адресах Marketplace для этого предложения. Если идентификатор издателя имеет значение *contoso*, и вы назначили маркетинговый идентификатор *sampleApp*, для вашего предложения в Azure Marketplace будет создан URL-адрес *https://azuremarketplace.microsoft.com/marketplace/apps/contoso.sampleApp* .        |
|**Идентификаторы подписок для предварительного просмотра**     | Добавьте от 1 до 100 идентификаторов подписок. Клиенты, связанные с этими подписками, смогут просматривать предложение в Azure Marketplace, прежде чем оно станет активным. Мы рекомендуем здесь добавить ваши собственные подписки, чтобы вы могли предварительно просмотреть, как ваше предложение будет выглядеть в Azure Marketplace, прежде чем вы сделаете его доступным для клиентов.  (Специалисты службы поддержки Майкрософт и команды разработчиков также смогут просматривать ваше предложение в течение периода предварительной версии.)   |
|**Полезные ссылки**     | URL-адреса, связанные с вашим предложением, например документация, заметки о выпуске, вопросы и ответы и т. д.        |
|**Suggested Categories (Max 5)** (Предлагаемые категории, макс. 5)     | Одна или несколько категорий (до пяти), которые применяются к вашему предложению. Благодаря этим категориям клиенты могут найти ваше предложение в Azure Marketplace и на портале Azure.        |

В разделе **Marketing Artifacts** (Маркетинговые артефакты) вы можете отправить логотипы и другие ресурсы, которые будут отображаться в предложении. При необходимости можно отправить снимки экрана или ссылки на видео, которые помогут клиентам составить представление о вашем предложении.

Требуются четыре размера логотипа: **малый (40x40)** , **средний (90x90)** , **крупный (115x115)** и **широкий (255x155)** . Следуйте приведенным ниже рекомендациям для ваших логотипов:

- Дизайн Azure отличается простой цветовой палитрой. Используйте в логотипах ограниченное число основных и дополнительных цветов.
- Цвета темы портала — белый и черный. Постарайтесь не использовать эти цвета для фона своего логотипа. Используйте цвет, который выделяет ваш логотип на портале. Мы рекомендуем простые основные цвета.
- При использовании прозрачного фона убедитесь, что для логотипа и текста не используется белый, черный или синий цвет.
- Логотип должен быть сплошным, избегайте градиентов. Не используйте в логотипе градиентный фон.
- Не размещайте в логотипе текст, даже название своей компании или торговой марки.
- Логотип не должен быть растянут.

Логотип **Имиджевый (815x290)** необязателен, но рекомендуется. Если вы добавили имиджевый логотип, следуйте приведенным ниже рекомендациям:

- Не добавляйте текст в имиджевый логотип и обязательно оставьте пустым пространство в 415 пикселей в правой части логотипа. Это необходимо для того, чтобы оставить место для текстовых элементов, которые будут встроены программным образом: отображаемое имя издателя, название плана, развернутая сводка предложения.
- Фон имиджевого логотипа не может быть черным, белым или прозрачным. Убедитесь, что цвет фона не слишком светлый, так как встроенный текст будет отображаться белым цветом.
- После публикации предложения с имиджевым логотипом его нельзя удалить (хотя при необходимости вы можете изменить его на другую версию).

В разделе **Управление интересами** можно выбрать систему CRM, в которой будут храниться ваши интересы, если это необходимо. 

Наконец, заполните поля **URL-адрес политики конфиденциальности** и **Условия использования** в разделе **Юридические сведения**. Здесь также можно указать, следует ли использовать [Стандартный контракт](https://docs.microsoft.com/azure/marketplace/standard-contract) для этого предложения.

Не забудьте сохранить изменения, прежде чем перейти к разделу **Поддержка**.

## <a name="add-support-info"></a>Добавление сведений о поддержке

В разделе **Поддержка** укажите имя, адрес электронной почты и номер телефона технического специалиста и сотрудника службы поддержки. Вам также необходимо предоставить URL-адреса поддержки. Корпорация Майкрософт может использовать эти сведения, если нам нужно связаться с вами по вопросам бизнеса и технической поддержки.

Добавив эти сведения, нажмите кнопку **Сохранить**.

## <a name="publish-your-offer"></a>Публикация вашего предложения

Когда вы предоставите все сведения, вашим следующим шагом будет публикация предложения в Azure Marketplace. Нажмите кнопку **Опубликовать**, чтобы запустить процесс активации предложения. Дополнительные сведения об этом процессе см. в статье [Publish Azure Marketplace and AppSource offers](https://docs.microsoft.com/azure/marketplace/cloud-partner-portal/manage-offers/cpp-publish-offer) (Публикация предложений Azure Marketplace и AppSource).
