# <a name="azure-blockchain-service-security"></a>Azure Blockchain службы безопасности

Служба Azure Blockchain использует несколько возможностей Azure для хранения данных, безопасности и доступности. Данные защищены с помощью изоляции, шифрования и проверки подлинности.

## <a name="isolation"></a>Изоляция

Ресурсы Azure Blockchain службы изолируются в виртуальной частной сети. Каждый узел транзакции и проверки является виртуальной машины (VM). Виртуальные машины в одной виртуальной сети не могут взаимодействовать напрямую с виртуальными машинами в другой виртуальной сети. Изоляция гарантирует, что обмен данными останутся закрытыми в пределах виртуальной сети. Дополнительные сведения об изоляции виртуальной сети Azure, см. в разделе [изоляция в общедоступном облаке Azure](../../security/azure-isolation.md#networking-isolation).

![Схема виртуальной сети](./media/data-security/vnet.png)

## <a name="encryption"></a>Шифрование

Данные пользователя хранятся в службе хранилища Azure. Данные пользователя шифруются при передаче и при хранении для безопасности и конфиденциальности. Дополнительные сведения можно найти в разделе  [Руководство по безопасности хранилища Azure](../../storage/common/storage-security-guide.md).

## <a name="authentication"></a>Authentication

Транзакции могут отправляться на узлы блокчейн через конечную точку RPC. Клиенты взаимодействуют с узла транзакции с помощью обратного прокси-сервера, проверку подлинности маркеров и шифрует данные по протоколу SSL.

![Схема проверки подлинности](./media/data-security/authentication.png)

Существует три режима проверки подлинности для доступа RPC.

### <a name="basic-authentication"></a>Обычная аутентификация

Обычная проверка подлинности использует заголовок проверки подлинности HTTP, содержащий имя пользователя и пароль. Имя пользователя является имя узла блокчейна. Пароль задается во время подготовки члена или узел. Пароль можно изменить с помощью портала Azure или интерфейса командной строки.

### <a name="access-keys"></a>Ключи доступа

Ключи доступа используйте случайный строку, включенную в URL-адрес конечной точки. Два ключа доступа помогают обеспечить смены ключей. Можно повторно создать ключи с портала Azure и интерфейса командной строки.

### <a name="azure-active-directory"></a>Azure Active Directory

Azure Active Directory (Azure AD) на основе утверждений используется механизм аутентификации, где происходит проверка подлинности с помощью Azure AD, используя учетные данные пользователя Azure AD. Azure AD обеспечивает управление удостоверениями на основе облака и позволяет клиентам использовать единое удостоверение для всего предприятия и доступ приложений в облаке. Служба Azure Blockchain интегрируется с Azure AD, включение идентификатор федерации, единого входа и многофакторной проверки подлинности. Можно назначить пользователей, групп и ролей приложения в вашей организации для доступа члена и узел блокчейна.

Прокси клиента Azure AD можно найти в [GitHub](https://github.com/Microsoft/azure-blockchain-connector/releases). Прокси клиента направляет пользователя на страницу входа в Azure AD и получает токен носителя после успешной проверки подлинности. Как следствие Ethereum клиентского приложения, например Geth или Truffle подключении пользователя к конечной точке прокси клиента. Наконец при отправке транзакции прокси клиента внедряет токен носителя в заголовке http, и обратный прокси-сервер проверяет маркер, с помощью протокола OAuth.

## <a name="keys-and-ethereum-accounts"></a>Ключи и учетные записи Ethereum

При подготовке член службы Azure Blockchain, Ethereum учетной записи и пару открытого и закрытого ключей создается. Закрытый ключ используется для отправки транзакции до блокчейна. Ethereum является последние 20 байт хэш открытого ключа. Учетная запись, Ethereum также называется бумажник.

Пары закрытого и открытого ключей хранится в виде файла ключа в формате JSON. Закрытый ключ шифруется с помощью пароль, введенный при создании службы ГК блокчейна.

Закрытые ключи используются для создания цифровой подписи транзакции. В частных блокчейна смарт-контрактов, подписанные закрытым ключом представляет о подписавшем. Чтобы проверить действительность подписи, получатель можно сравнить открытый ключ подписывающего лица с адресом, вычисляемым на основе подписи.

Несчастливых ключи используются для однозначной идентификации узлов кворума. Несчастливых ключи создаются во время подготовки узла и указаны в параметре privateFor закрытый транзакции в кворуме.
