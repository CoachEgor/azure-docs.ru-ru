# <a name="limits-in-azure-blockchain-service"></a>Ограничения в блокчейн-сервисе Azure

Служба блокчейн Azure имеет такие ограничения обслуживания и функциональные возможности, как количество узлов, которые может иметь участник, ограничения консорциума и объемы хранения.

## <a name="pricing-tier"></a>Ценовая категория

Максимальные ограничения на транзакции и узлы валидатора зависят от того, предоставляете ли вы Azure Blockchain Service на базовых или стандартных уровнях ценообразования.

| Ценовая категория | Узлы транзакции Max | Узлы max валидатора |
|:---|:---:|:---:|
| Basic | 10 | 1 |
| Standard | 10 | 2 |

Ваша сеть консорциума должна иметь как минимум два стандартных узла уровня Azure Blockchain Service. Стандартные узлы уровня включают два узла валидатора. Четыре узла валидатора необходимы для [удовлетворения Стамбуле византийский консенсус толерантности к ошибкам.](https://github.com/jpmorganchase/quorum/wiki/Quorum-Consensus)

Использование базового уровня предназначено для разработки, тестирования и доказательства концепций. Используйте стандартный уровень для развертывания производственного класса. Вы также должны использовать *стандартный* уровень, если вы используете Blockchain Data Manager или отправляете большой объем частных транзакций.

Изменение уровня ценообразования между базовым и стандартным после создания члена не поддерживается.

## <a name="storage-capacity"></a>Емкость хранилища

Максимальное количество хранилища, которое может быть использовано на узла для данных и журналов реестра реестра, составляет 1,8 терабайта.

Уменьшение размера регистра и хранилища журналов не поддерживается.
## <a name="consortium-limits"></a>Ограничения консорциума

* **Имена консорциумов и членов должны быть уникальными** для других консорциумов и имен членов службы блокчейн Azure.

* **Имена членов и консорциумов не могут быть изменены**

* **Все члены консорциума должны быть в одном ценовом уровне**

* **Все члены, участвующие в консорциуме, должны проживать в одном регионе**

    Первый член, созданный в консорциуме, диктует регион. Приглашенные члены консорциума должны проживать в том же регионе, что и первый член. Ограничение всех членов в одном регионе помогает обеспечить не негативное влияние сетевого консенсуса.

* **Консорциум должен иметь по крайней мере одного администратора**

    Если в консорциуме есть только один администратор, он не может удалить себя из консорциума или удалить своего участника до тех пор, пока другой администратор не будет добавлен или повышен в консорциуме.

* **Члены, удаленные из консорциума, не могут быть добавлены снова**

    Скорее, они должны быть повторно приглашены присоединиться к консорциуму и создать нового члена. Существующие ресурсы-члены не удаляются для сохранения исторических транзакций.

* **Все члены консорциума должны использовать одну и ту же версию книги**

    Для получения дополнительной информации о версиях исправления, обновлений [Patching, updates, and versions](ledger-versions.md)и книг, доступных в Azure Blockchain Service, см.

## <a name="performance"></a>Производительность

Не используйте газовую *функцию eth.estimate* для каждой передачи транзакции. Функция *eth.estimate* является интенсивной памятью. Вызов функции в несколько раз резко сокращает транзакции в секунду.

Если возможно, используйте консервативную стоимость газа для подачи сделок и минимизировать использование *eth.estimate*.
