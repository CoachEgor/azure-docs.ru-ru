# <a name="what-are-trusted-security-partners-preview"></a>Что такое доверенные партнеры безопасности (Предварительная версия)?

> [!IMPORTANT]
> Эта общедоступная предварительная версия предоставляется без соглашения об уровне обслуживания и не должна использоваться для производственных рабочих нагрузок. Некоторые функции могут не поддерживаться, иметь ограничения и быть доступными не во всех расположениях Azure. См. [дополнительные условия использования для предварительных версий Microsoft Azure](https://azure.microsoft.com/support/legal/preview-supplemental-terms/).

*Доверенные партнеры безопасности (Предварительная версия)* в диспетчере брандмауэра Azure позволяют использовать привычные, лучшие в своем классе предложения безопасности как услуги (секаас) для защиты доступа к Интернету для пользователей.

С помощью быстрой настройки можно защитить центр с помощью поддерживаемого партнера по безопасности, а также перенаправлять и фильтровать Интернет-трафик из виртуальных сетей (виртуальных сетей) или расположений ветвей в регионе. Это делается с помощью автоматизированного управления маршрутами без настройки и управления определяемыми пользователем маршрутами (определяемые пользователем маршруты).

Вы можете развернуть защищенные центры, настроенные с помощью выбранного партнера по безопасности в нескольких регионах Azure, чтобы обеспечить подключение и безопасность пользователей в любом месте по всему миру в этих регионах. Благодаря возможности использовать предложение партнера по безопасности для трафика приложений Интернета или SaaS, а также брандмауэр Azure для частного трафика в защищенных центрах, теперь можно приступить к созданию приложения безопасности в Azure, близкого к глобально распределенным пользователям, и приложений.

Для этой предварительной версии поддерживаемые партнеры по обеспечению безопасности — **ZScaler** и **ибосс**. Поддерживаемые регионы: WestCentralUS, NorthCentralUS, WestUS, WestUS2 и EastUS.

## <a name="key-scenarios"></a>Ключевые сценарии

Для фильтрации трафика Интернета в следующих случаях можно использовать партнеров по безопасности:

- Виртуальная сеть (VNet) в Интернет

   Используйте расширенную защиту в Интернете с учетом пользователей для облачных рабочих нагрузок, работающих в Azure.

- Переход к Интернету

   Используйте возможности подключения к Azure и глобального распределения, чтобы легко добавить сторонние фильтры Нсаас для ветвления в Интернет-сценарии. Вы можете создать глобальную транзитную сеть и безопасность с помощью виртуальной глобальной сети Azure.

Следующие сценарии поддерживаются:
-   Использование виртуальной сети для Интернета с помощью сторонних предложений партнеров.
-   Подветвление к Интернету с помощью стороннего предложения партнеров.
-   Разветвление в Интернет с помощью стороннего предложения партнеров, остальной объем частного трафика (периферийных, периферийных, филиалов и ветвей) через брандмауэр Azure.

Следующий сценарий не поддерживается:

- Виртуальную сеть для Интернета через предложение партнера нельзя сочетать с брандмауэром Azure для частного трафика. См. следующие ограничения.

## <a name="current-limitations"></a>Текущие ограничения

- Для виртуальной сети в Интернете нельзя смешивать Добавление брандмауэра Azure для частного трафика и предложения партнера по интернет-трафику. Вы можете отправить Интернет-трафик в брандмауэр Azure или предложение партнера по безопасности стороннего производителя в защищенном виртуальном концентраторе, но не в обоих. 
- Вы можете развернуть не более одного партнера по безопасности на каждом виртуальном концентраторе. Если необходимо изменить поставщика, необходимо удалить существующего партнера и добавить новый.

## <a name="best-practices-for-internet-traffic-filtering-in-secured-virtual-hubs"></a>Рекомендации по фильтрации трафика Интернета в защищенных виртуальных концентраторах

Трафик Интернета обычно включает веб-трафик. Но он также включает трафик, предназначенный для приложений SaaS, таких как Office 365 (O365) и общедоступные службы PaaS Azure, такие как служба хранилища Azure, SQL Azure и т. д. Ниже приведены рекомендации по обработке трафика для этих служб.

### <a name="handling-azure-paas-traffic"></a>Обработка трафика Azure PaaS
 
- Используйте брандмауэр Azure для защиты, если ваш трафик состоит в основном из Azure PaaS, а доступ к ресурсам для приложений можно фильтровать с помощью IP-адресов, полных доменных имен, тегов служб или тегов полного доменного имени.

- Используйте стороннее решение на своих концентраторах, если ваш трафик состоит из доступа к приложениям SaaS, или требуется фильтрация с учетом пользователей (например, для рабочих нагрузок инфраструктуры виртуальных рабочих столов (VDI)) или требуются расширенные возможности фильтрации в Интернете.

## <a name="handling-office-365-o365-traffic"></a>Обработка трафика Office 365 (O365)

В сценариях с глобальным распределенным размещением необходимо перенаправить трафик Office 365 непосредственно в ветви перед отправкой оставшегося Интернет-трафика в центр безопасности Azure.

Для пользователей Office 365 задержка и производительность сети являются критически важными для успешного взаимодействия с пользователем. Чтобы достичь этих целей в целях достижения оптимальной производительности и удобства взаимодействия с пользователем, клиенты должны внедрить прямую и локальную escape-версию Office 365 перед тем, как рассмотреть возможность маршрутизации остальной части Интернет-трафика через Azure.

[Принципы сетевого подключения office 365](https://docs.microsoft.com/office365/enterprise/office-365-network-connectivity-principles) вызывают подключение к key Office 365, чтобы они направлялись локально из ветви пользователя или мобильного устройства и напрямую через Интернет в ближайшую сетевую точку присутствия (Майкрософт).

Кроме того, подключения Office 365 строго шифруются для обеспечения конфиденциальности и используют эффективные, частные протоколы для повышения производительности. Это непрактично и не влияет на эти подключения к традиционным решениям безопасности на уровне сети. По этим причинам мы настоятельно рекомендуем клиентам отправлять трафик Office 365 непосредственно из ветвей перед отправкой оставшейся части трафика через Azure. Корпорация Майкрософт сотрудничает с несколькими поставщиками решений SD-WAN, которые интегрируются с Azure и Office 365 и облегчают клиентам поддержку прямой и локальной организации в Интернете с помощью Office 365. Дополнительные сведения см [. в разделе разделы справки Установка политик Office 365 через виртуальную глобальную сеть?](https://docs.microsoft.com/azure/virtual-wan/virtual-wan-office365-overview#how-do-i-set-my-o365-policies-via-virtual-wan)
