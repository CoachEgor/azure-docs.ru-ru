# <a name="what-are-trusted-security-partners-preview"></a>Что собой представляет функция доверенных партнеров по безопасности (предварительная версия)?

> [!IMPORTANT]
> Эта общедоступная предварительная версия предоставляется без соглашения об уровне обслуживания и не должна использоваться для производственных рабочих нагрузок. Некоторые функции могут не поддерживаться, иметь ограничения и быть доступными не во всех расположениях Azure. См. [дополнительные условия использования для предварительных версий Microsoft Azure](https://azure.microsoft.com/support/legal/preview-supplemental-terms/).

*Доверенные партнеры по безопасности (предварительный просмотр)* в Azure Firewall Manager позволяют использовать знакомые, лучшие в породе, сторонние средства безопасности в качестве службы (SECaaS) для защиты доступа в Интернет для пользователей.

С быстрой конфигурацией можно обеспечить доступ к узлу с поддерживаемым партнером по безопасности, а также маршрутификатор и фильтрацию интернет-трафика из ваших виртуальных сетей (VNets) или ветвей в регионе. Это делается с помощью автоматизированного управления маршрутами, без настройки и управления пользовательскими определенными маршрутами (UDRs).

Можно развернуть защищенные концентраторы, настроенные с вашим партнером по безопасности в нескольких регионах Azure, чтобы обеспечить подключение и безопасность пользователей в любой точке земного шара в этих регионах. Благодаря возможности использовать предложение партнера по безопасности для трафика приложений для Интернета/SaaS и Azure Firewall для частного трафика в защищенных концентралях, теперь можно начать строить преимущество безопасности в Azure, близкое к пользователям и приложениям, которые будут распределены по всему миру.

Для этого предварительного просмотра, поддерживаемые партнеры по безопасности являются **«Scaler»** и **iboss**. Поддерживаемые регионы: ЗападныйЦентральный, Северо-Центральный, Западный, Западный, Западный И Восточный.

## <a name="key-scenarios"></a>Ключевые сценарии

Вы можете использовать партнеров по безопасности для фильтрации интернет-трафика в следующих сценариях:

- Виртуальная сеть (VNet) в Интернет

   Используйте расширенную защиту интернет-трафика с учетом пользователей для облачных рабочих нагрузок в Azure.

- Филиал в Интернете

   Используйте подключение Azure и глобальное распространение, чтобы легко добавить стороннюю фильтрацию NSaaS для ветки в интернет-сценарии. Вы можете создать глобальную транзитную сеть и преимущество безопасности с помощью Azure Virtual WAN.

Поддерживаются следующие сценарии:
-   VNet в Интернет через стороннего партнера предложение.
-   Филиал в Интернет через стороннего партнера предложение.
-   Филиал в Интернетчерез сторонний партнер предлагает остальную часть частного трафика (Spoke-to-Spoke, Spoke-to-Branches, Branch-to-Spokes) через Azure Firewall.

Следующий сценарий не поддерживается:

- VNet для Интернета через предложение партнера не может быть объединена с Azure Firewall для частного трафика. Посмотреть следующие ограничения.

## <a name="current-limitations"></a>Текущие ограничения

- Для VNet в Интернете нельзя смешивать добавление Брандмауэра Azure для частного трафика и партнерскую предложение для интернет-трафика. Вы можете отправить интернет-трафик в Azure Firewall или стороннего партнера по безопасности, предлагая в защищенном виртуальном концентраторе, но не на обоих. 
- Вы можете развернуть не более одного партнера по безопасности в виртуальном концентраторе. Если вам необходимо изменить поставщика, необходимо удалить существующего партнера и добавить новый.

## <a name="best-practices-for-internet-traffic-filtering-in-secured-virtual-hubs"></a>Лучшие практики фильтрации интернет-трафика в защищенных виртуальных центрах

Интернет-трафик обычно включает в себя веб-трафик. Но он также включает трафик, предназначенный для приложений SaaS, таких как Office 365 (O365) и общедоступные службы PaaS Azure, Azure Sql и так далее. Ниже приведены рекомендации наилучшей практики для обработки трафика на эти услуги:

### <a name="handling-azure-paas-traffic"></a>Обработка трафика Azure PaaS
 
- Используйте Azure Firewall для защиты, если трафик состоит в основном из Azure PaaS, а доступ к ресурсам для приложений можно отфильтровать с помощью IP-адресов, F-DN, тегов службы или тегов ФЗН.

- Используйте решение стороннего партнера в концентратах, если ваш трафик состоит из доступа к приложениям SaaS или вам нужна фильтрация пользователей (например, для рабочих нагрузок виртуальной настольной инфраструктуры (VDI) или вам нужны расширенные возможности фильтрации Интернета.

## <a name="handling-office-365-o365-traffic"></a>Обработка Office 365 (O365) трафик

В глобально распределенных сценариях расположения филиалов следует перенаправить трафик Office 365 непосредственно в ветку, прежде чем отправлять оставшийся интернет-трафик в защищенном узле Azure.

Для Office 365 задержка сети и производительность имеют решающее значение для успешного пользовательского опыта. Для достижения этих целей, учитывая оптимальную производительность и пользовательский опыт, клиенты должны реализовать Office 365 прямого и локального выхода, прежде чем рассматривать маршрутизирование остальной части интернет-трафика через Azure.

[Принципы подключения к сети Office 365](https://docs.microsoft.com/office365/enterprise/office-365-network-connectivity-principles) требуют, чтобы ключевые сетевые соединения Office 365 направлялись локально из ветки пользователя или мобильного устройства и непосредственно через Интернет в ближайшую сетевую точку присутствия Майкрософт.

Кроме того, подключения Office 365 сильно зашифрованы для конфиденциальности и используют эффективные, запатентованные протоколы по причинам производительности. Это делает нецелесообразным и результативным подвергать эти соединения традиционным решениям сетевого уровня безопасности. По этим причинам мы настоятельно рекомендуем клиентам отправлять трафик Office 365 непосредственно из филиалов, прежде чем отправлять остальную часть трафика через Azure. Корпорация Майкрософт сотрудничает с несколькими поставщиками решений SD-WAN, которые интегрируются с Azure и Office 365 и упрощают для клиентов возможность прямого и локального прорыва ВАИ 365. Для получения подробной информации смотрите [Как я могу настроить мои политики O365 через Virtual WAN?](https://docs.microsoft.com/azure/virtual-wan/virtual-wan-office365-overview)
