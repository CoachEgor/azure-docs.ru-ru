# <a name="shape-events-with-azure-time-series-insights-preview"></a>Формирование событий с помощью предварительной версии службы "Аналитика временных рядов Azure"

В этой статье определяются рекомендации по формированию полезной нагрузки JSON для приема в Azure Time Series Insights и максимальной эффективности запросов предварительного просмотра.

## <a name="best-practices"></a>Рекомендации

Лучше всего тщательно продумать, как вы отправляете события в среду просмотра Time Series Insights. 

Общие рекомендации включают:

* Отправлять данные по сети наиболее эффективным путем.
* Храните свои данные таким образом, чтобы упростить их статистическое вычисление для своего сценария.

Для лучшей производительности запроса придерживайтесь следующих правил:

* Не отправляйте необязательные свойства. Time Series Исследования Предварительный просмотр счетов по использованию. Лучше всего хранить и обрабатывать только те данные, которые вы будете зажимать.
* Используйте поля экземпляра для статических данных, Эта практика помогает избежать отправки статических данных по сети. Поля экземпляров, которые являются компонентом модели Time Series, работают как справочные данные в обычно доступном сервисе Time Series Insights. Чтобы узнать больше о полях экземпляров, прочитайте [модель таймрядных рядов](./time-series-insights-update-tsm.md).
* Два или несколько событий могут использовать одинаковые свойства измерения. Это позволит более эффективно отправлять данные по сети.
* Не применяйте глубокую вложенность массивов. Time Series Insights Preview поддерживает до двух уровней вложенных массивов, содержащих объекты. а также в сообщениях преобразует массивы в плоскую структуру, разбивая на несколько событий с парами "свойство — значение".
* Если для всех или большинства событий существует только несколько мер, лучше отправлять эти меры как отдельные свойства в одном и том же объекте. Отправка их отдельно сокращает количество событий и может повысить производительность запроса, поскольку требуется меньшее количество событий.

## <a name="column-flattening"></a>Уплощение колонны

Во время приема полезных нагрузок, содержащих вложенные объекты, будут сглажены таким образом, чтобы название столбца представляло собой единое значение с разграничателем.

* Например, следующий вложенный JSON:

   ```JSON
   "data": {
        "flow": 1.0172575712203979,
   },
   ```

   Становится: `data_flow` при выравнивании.

> [!IMPORTANT]
> * Azure Time Series Исследования Просмотр`_`использует подчеркивает ( ) для разграничения столбцов.
> * Обратите внимание на отличие от общей`.`доступности, которая использует периоды () вместо.

Ниже приведены более сложные сценарии.

#### <a name="example-1"></a>Пример 1:

Следующий сценарий имеет два (или более) устройств, которые отправляют измерения (сигналы): *Скорость потока,* *Давление масла двигателя*, *Температура,* и *влажность*.

Отправлено одно сообщение Azure IoT Hub, в котором внешний массив содержит общий раздел значений общего измерения (обратите внимание на две записи устройства, содержащиеся в сообщении).

```JSON
[
  {
    "deviceId":"FXXX",
    "timestamp":"2018-01-17T01:17:00Z",
    "series":[
      {
        "Flow Rate ft3/s":1.0172575712203979,
        "Engine Oil Pressure psi ":34.7
      },
      {
        "Flow Rate ft3/s":2.445906400680542,
        "Engine Oil Pressure psi ":49.2
      }
    ]
  },
  {
    "deviceId":"FYYY",
    "timestamp":"2018-01-17T01:18:00Z",
    "series":[
      {
        "Flow Rate ft3/s":0.58015072345733643,
        "Engine Oil Pressure psi ":22.2
      }
    ]
  }
]
```

**Вынос:**

* Пример JSON имеет внешний массив, который использует данные [Time Series Instance](./time-series-insights-update-tsm.md#time-series-model-instances) для повышения эффективности сообщения. Несмотря на то, что метаданные устройства Time Series Instances вряд ли изменятся, они часто предоставляют полезные свойства для анализа данных.

* JSON объединяет два или более сообщений (по одному от каждого устройства) в одну полезную нагрузку, экономя на пропускной способности с течением времени.

* Отдельные точки данных серий для каждого устройства объединяются в один атрибут **серии,** уменьшая необходимость непрерывного потока обновлений для каждого устройства.

> [!TIP]
> Чтобы уменьшить количество сообщений, необходимых для отправки данных, и сделать телеметрию более эффективной, рассмотрите пакетирование значений общего измерения и метаданных Time Series Instance в единую полезную нагрузку JSON.

#### <a name="time-series-instance"></a>Экземпляр временных рядов 

Давайте подробнее рассмотрим, как использовать [Time Series Instance](./time-series-insights-update-tsm.md#time-series-model-instances) для более оптимальной формы JSON. 

> [!NOTE]
> [Идентификаторы серии времени](./time-series-insights-update-how-to-id.md) ниже являются *deviceIds*.

```JSON
[
  {
    "timeSeriesId":[
      "FXXX"
    ],
    "typeId":"17150182-daf3-449d-adaf-69c5a7517546",
    "hierarchyIds":[
      "b888bb7f-06f0-4bfd-95c3-fac6032fa4da"
    ],
    "description":null,
    "instanceFields":{
      "L1":"REVOLT SIMULATOR",
      "L2":"Battery System"
    }
  },
  {
    "timeSeriesId":[
      "FYYY"
    ],
    "typeId":"17150182-daf3-449d-adaf-69c5a7517546",
    "hierarchyIds":[
      "b888bb7f-06f0-4bfd-95c3-fac6032fa4da"
    ],
    "description":null,
    "instanceFields":{
      "L1":"COMMON SIMULATOR",
      "L2":"Battery System"
    }
  }
]
```

Предварительная версия службы "Аналитика временных рядов" объединяет таблицу (после преобразования в плоскую структуру) во время запроса. В таблице содержатся дополнительные столбцы, такие как **Тип**.

| deviceId  | Тип | L1 | L2 | TIMESTAMP | series_Flow ставка ft3/s | series_Engine нефтяное давление пси |
| ---- | ---- | ---- | ---- | ---- | ---- | ---- |
| `FXXX` | Default_Type | Имитирующее устройство | Система аккумулятора | 2018-01-17T01:17:00Z |   1.0172575712203979 |    34.7 |
| `FXXX` | Default_Type | Имитирующее устройство |   Система аккумулятора |    2018-01-17T01:17:00Z | 2.445906400680542 |  49.2 |
| `FYYY` | LINE_DATA    COMMON | Имитирующее устройство |    Система аккумулятора |    2018-01-17T01:18:00Z | 0.58015072345733643 |    22.2 |

> [!NOTE]
>  Предыдущая таблица представляет представление запроса в [Обзоре Временных рядов.](./time-series-insights-update-explorer.md)

**Вынос:**

* В предыдущем примере статические свойства хранятся в Time Series Insights Preview для оптимизации данных, отправляемых по сети.
* Данные Time Series Insights Preview соединяются во время запроса через идентификатор Time Series, определяемый в экземпляре.
* Используются два слоя гнездования. Это число является наиболее, что Время Серии Исследования Предварительный поддерживает. Критически важно избегать глубоко вложенных массивов.
* Так как у нас есть несколько мер, они передаются в виде отдельных свойств в одном и том же объекте. В этом примере, **series_Flow Оценить psi**, **series_Engine давление масла psi**, и series_Flow скорость **ft3/s** являются уникальными столбцов.

>[!IMPORTANT]
> Поля экземпляров не хранятся с помощью телеметрии. Они хранятся с метаданными в модели временных рядов.

#### <a name="example-2"></a>Пример 2.

Рассмотрим следующее JSON:

```JSON
{
  "deviceId": "FXXX",
  "timestamp": "2019-01-18T01:17:00Z",
  "data": {
        "flow": 1.0172575712203979,
    },
  "data_flow" : 1.76435072345733643
}
```

В приведенном выше примере `data["flow"]` сплющенное свойство `data_flow` представляет собой столкновение с именованием с собственностью.

В этом случае *последнее* значение свойства перезапишет предыдущее значение. 

> [!TIP]
> Свяжитесь с командой Time Series Insights для получения дополнительной помощи!

> [!WARNING] 
> * В тех случаях, когда дубликаты свойств присутствуют в одной и той же (единственной) полезной нагрузке события из-за уплощения или другого механизма, сохраняется последняя > значения свойства, перезаписи любых предыдущих значений.
> * Серия комбинированных событий не перекроет друг друга.
