# <a name="enable-replication-to-azure-for-vmware-vms"></a>Включение репликации в Azure для виртуальных машин VMware

В этой статье описано, как включить репликацию локальных виртуальных машин VMware в Azure.

## <a name="prerequisites"></a>Технические условия

В этой статье предполагается, что вы:

- [Настройка исходной среды в локальной](vmware-azure-set-up-source.md).
- [Настройка целевой среды в Azure](vmware-azure-set-up-target.md).

## <a name="before-you-start"></a>Перед началом работы
При репликации виртуальных машин VMware, учитывайте эти сведения:

* Учетная запись пользователя Azure должна содержать определенные [разрешения](site-recovery-role-based-linked-access-control.md#permissions-required-to-enable-replication-for-new-virtual-machines) для включения репликации новой виртуальной машины в Azure.
* Виртуальные машины VMware обнаруживаются каждые 15 минут. Может занять 15 минут или более виртуальных машин для отображения на портале Azure после обнаружения. Аналогично, обнаружение может также занять 15 минут или больше времени, при добавлении нового узла vCenter server или vSphere.
* Может занять 15 минут или больше времени для изменения среды на виртуальной машине (например, установка средств VMware) для обновления в на портале.
* Можно проверить время последнего обнаружения для виртуальных машин VMware: См. в разделе **последний контакт в** на **серверы конфигурации** страницы для vCenter server или узла vSphere.
* Чтобы добавить виртуальные машины для репликации, не дожидаясь запланированного обнаружения, выделите сервер конфигурации (но не щелкая его) и выберите **обновить**.
* При включении репликации, если виртуальная машина уже подготовлена, сервер обработки автоматически устанавливает службу Azure Site Recovery Mobility на него.

## <a name="enable-replication"></a>Включение репликации

Прежде чем выполнять действия, описанные в этом разделе, обратите внимание на следующее:
* Azure Site Recovery реплицирует теперь непосредственно для использования управляемых дисков для всех новых репликациями. Сервер обработки записывает журналы репликации для учетной записи хранения кэша в целевом регионе. Эти журналы используются для создания точек восстановления на управляемых дисках реплики.
* Во время отработки отказа, выбранная точка восстановления используется для создания управляемого целевого диска.
* Виртуальные машины, которые были настроены для репликации в целевые учетные записи хранения не затрагиваются.
* Репликация в учетные записи хранения для новой виртуальной машины доступен только через Representational State Transfer (REST) API и Powershell. Используйте Azure REST API версии 2016-08-10 "или" 2018-01-10 для репликации в учетные записи хранения.

1. Перейдите к **шаг 2: Репликация приложения** > **Источник**. Включив репликацию впервые, выберите **+ реплицировать** в хранилище, чтобы включить репликацию для дополнительных виртуальных машин.
2. На странице **Источник** щелкните **Источник** и выберите нужный сервер конфигурации.
3. Для **тип компьютера**выберите **виртуальных машин** или **физических компьютеров**.
4. В поле **vCenter/vSphere Hypervisor** (Гипервизор vCenter или vSphere) выберите сервер vCenter, который управляет узлом vSphere, или сам узел. Этот параметр не учитывается, если вы реплицируете физические компьютеры.
5. Выберите сервер обработки. Если нет дополнительных серверов обработки создан, сервер приемникам без встроенных возможностей обработки сервера конфигурации появится в раскрывающемся списке. Состояние работоспособности каждого процесса сервера указывается согласно рекомендуемые ограничения и другие параметры. Выберите сервер обработки работоспособное. Объект [критических](vmware-physical-azure-monitor-process-server.md#process-server-alerts) нельзя выбрать сервер обработки. Вы можете либо [решения](vmware-physical-azure-troubleshoot-process-server.md) ошибки **или** Настройка [сервера обработки масштабирования](vmware-azure-set-up-process-server-scale.md).

Обновите компоненты Site Recovery для 9.24 версии или более поздней версии для всех предупреждений будет создан.

2. Выберите сеть Azure и подсеть, к которой будут подключаться виртуальные машины Azure после отработки отказа. Сеть должна располагаться в том же регионе, что и хранилище службы Site Recovery.

Выберите **отложить настройку** выбрать сеть Azure для каждой виртуальной машины. Если у вас нет сети, ее необходимо создать. Чтобы создать сеть с помощью Azure Resource Manager, выберите **создать**. Выберите подсеть, если это применимо, а затем выберите **ОК**.

### <a name="configure-networks-and-ip-addresses"></a>Настройка сетей и IP-адресов

Вы можете задать целевой IP-адрес. Если не указать его, выполнена отработка отказа виртуальной машины использует DHCP. Если задать адрес, недоступный при отработке отказа, произойдет сбой. Если он доступен в сети тестовой отработки отказа, же целевой IP-адрес можно использовать для тестовой отработки отказа.

Количество сетевых адаптеров зависит от размера, указанного для целевой виртуальной машины следующим образом:

- Если число сетевых адаптеров на исходной виртуальной машины меньше или равно числу адаптеров, которые разрешены для размера целевой Виртуальной машины, целевой объект имеет одинаковое количество адаптеров как источник.
- Если число адаптеров исходной виртуальной машины превышает значение, разрешенному для целевой Виртуальной машины, используется максимальный размер целевой. Например если исходная виртуальная машина имеет два сетевых адаптера, а размер целевой Виртуальной машины поддерживает четыре, целевой виртуальной машине есть два адаптера. Если исходная виртуальная машина содержит два адаптера, но целевой размер поддерживает только один, целевой виртуальной Машине будет только один адаптер.
- Если для виртуальной машины настроено несколько сетевых адаптеров, все они будут подключены к одной сети. Кроме того, становится первый адаптер, который отображается в списке *по умолчанию* сетевой адаптер в виртуальной машине Azure. 

### <a name="azure-hybrid-benefit"></a>Преимущество гибридного использования Azure

Клиенты Microsoft Software Assurance можно использовать преимущество гибридного использования Azure, чтобы сэкономить на стоимости лицензирования компьютеров Windows Server, которые переносятся в Azure. Преимущество также применяется к аварийного восстановления Azure. Если вы имеете право, преимущество можно назначить виртуальной машине, Site Recovery создает в случае отработки отказа. Для этого выполните следующие действия:
1. Перейдите к **свойства компьютера и сети** реплицированной виртуальной машины.
2. Ответ на запрос, если у вас есть лицензия Windows Server, дающая право на Преимущество гибридного использования Azure.
3. Убедитесь, что у вас есть соответствующая лицензия Windows Server с Software Assurance, который можно использовать для применения преимущества к виртуальной Машине, создаваемой при отработке отказа.
4. Сохраните параметры для реплицируемой виртуальной машины.

См. дополнительные сведения о программе [преимуществ гибридного использования Azure](https://aka.ms/azure-hybrid-benefit-pricing).

## <a name="resolve-common-issues"></a>Устранение распространенных проблем

* Каждый диск должен быть меньше, чем 4 ТБ.
* Диск операционной системы должен представлять собой базовый диск, не динамический диск.
* Для создания поддержкой 2/UEFI виртуальных машин должна быть из семейства операционной системы Windows и загрузочный диск должен быть меньше, чем 300 ГБ.
