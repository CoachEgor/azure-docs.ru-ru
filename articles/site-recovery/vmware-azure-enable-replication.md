# <a name="enable-replication-to-azure-for-vmware-vms"></a>Включение репликации в Azure для виртуальных машин VMware

В этой статье описано, как включить репликацию локальных виртуальных машин VMware в Azure.

## <a name="resolve-common-issues"></a>Решение общих проблем

* Каждый диск должен быть меньше 4 ТБ.
* Диск ОС должен быть базовым диском, а не динамическим диском.
* Для виртуальных машин с поддержкой поколения 2/UEFI семейство операционных систем должно быть Windows, а загрузочный диск должен быть меньше 300 ГБ.

## <a name="prerequisites"></a>Предварительные требования

В этой статье предполагается, что вы:

- [Настройка исходной среды.](vmware-azure-set-up-source.md)
- [Настройка целевой среды в Azure.](vmware-azure-set-up-target.md)
- [Проверьте требования и предпосылки](vmware-physical-azure-support-matrix.md) перед запуском. Важные вещи, чтобы отметить:
    - [Поддерживаемые операционные системы](vmware-physical-azure-support-matrix.md#replicated-machines) для реплицированных машин.
    - [Поддержка хранения/диска.](vmware-physical-azure-support-matrix.md#storage)
    - [Требования Azure,](vmware-physical-azure-support-matrix.md#azure-vm-requirements) которым должны соответствовать приемние помещений.


## <a name="before-you-start"></a>Перед началом работы
При репликации виртуальных машин VMware, имейте эту информацию в виду:

* Учетная запись пользователя Azure должна содержать определенные [разрешения](site-recovery-role-based-linked-access-control.md#permissions-required-to-enable-replication-for-new-virtual-machines) для включения репликации новой виртуальной машины в Azure.
* Виртуальные машины VMware обнаруживаются каждые 15 минут. Появление вв.М. на портале Azure после обнаружения может занять 15 минут или больше. Кроме того, открытие может занять 15 минут или больше при добавлении нового сервера vCenter или хоста vSphere.
* Обновление средств виртуальной машины (например, установка инструментов VMware) может занять 15 минут или больше.
* Вы можете проверить последнее время для VMware VMs: Смотрите **последний контакт на** поле на странице **конфигурационных серверов** для vCenter server/vSphere host.
* Чтобы добавить виртуальные машины для репликации, не дожидаясь запланированного открытия, выделите сервер конфигурации (но не нажимайте на него) и выберите **Refresh.**
* При включении репликации, если виртуальная машина подготовлена, сервер процесса автоматически устанавливает на нее службу восстановления azure Site Recovery Mobility.

## <a name="enable-replication"></a>Включение репликации

Прежде чем выполнить шаги в этом разделе, обратите внимание на следующую информацию:
* Восстановление сайта Azure теперь реплицируется непосредственно на управляемые диски для всех новых репликаций. Сервер процесса записывает журналы репликации в учетную запись хранения кэша в целевом регионе. Эти журналы используются для создания точек восстановления в управляемых дисках реплики, которые имеют именования конвенции asrseeddisk.
* Поддержка Powershell для репликации управляемых дисков доступна в [версии модуля Az.RecoveryServices 2.0.0](https://www.powershellgallery.com/packages/Az.RecoveryServices/2.0.0-preview) 
* Во время сбоя выбранная точка восстановления используется для создания диска, управляемого целевым, .
* VMs, которые ранее были настроены для репликации в целевые учетные записи хранилища, не затрагиваются.
* Репликация учетных записей для хранения новой виртуальной машины доступна только через API-и Powershell. Используйте версию Azure REST API 2016-08-10 или 2018-01-10 для репликации на учетные записи.

2. На странице **Источник** щелкните **Источник** и выберите нужный сервер конфигурации.
3. Для **типа машины**, выберите **виртуальные машины** или **физические машины.**
4. В поле **vCenter/vSphere Hypervisor** (Гипервизор vCenter или vSphere) выберите сервер vCenter, который управляет узлом vSphere, или сам узел. Эта настройка не актуальна, если вы реплицируете физические компьютеры.
5. Выберите сервер обработки. Если нет дополнительных серверов процесса, встроенный сервер процесса сервера конфигурации будет доступен в выпадающем. ### <a name="configure-networks-and-ip-addresses"></a>Настройка сетей и IP-адресов

Вы можете задать целевой IP-адрес. Если вы не укажите адрес, несшименная виртуальная машина использует DHCP. Если задать адрес, недоступный при отработке отказа, произойдет сбой. Если адрес доступен в сети тестового совмещении, можно использовать тот же целевой IP-адрес для сбоя теста.

Количество сетевых адаптеров определяется размером, который вы указываете для целевой виртуальной машины, следующим образом:

- Если количество сетевых адаптеров на исходной виртуальной машине меньше или равно числу адаптеров, которые разрешены для размера целевого VM, цель имеет такое же количество адаптеров, как и источник.
- Если количество адаптеров для исходной виртуальной машины превышает количество, которое разрешено для размера целевого VM, используется максимальный размер цели. Например, если исходная виртуальная машина имеет два сетевых адаптеров, а размер цели VM поддерживает четыре, то целевая виртуальная машина имеет два адаптера. Если источник VM имеет два адаптера, но размер цели поддерживает только один, целевой VM имеет только один адаптер.
- Если для виртуальной машины настроено несколько сетевых адаптеров, все они будут подключены к одной сети. Кроме того, первым адаптером, показанным в списке, становится сетевой адаптер *по умолчанию* в виртуальной машине Azure. 

### <a name="azure-hybrid-benefit"></a>Преимущество гибридного использования Azure

Клиенты Microsoft Software Assurance могут использовать Azure Hybrid Benefit для экономии на затратах на лицензирование компьютеров Windows Server, которые переходят в Azure. Преимущество также распространяется на аварийное восстановление Azure. Если вы имеете право, вы можете назначить преимущество виртуальной машины, которую создает восстановление сайта, если произошел сбой. Для этого выполните следующие действия:
1. Перейти к **компьютеру и сети свойства** реплицируемой виртуальной машины.
2. Отвечайте на вопрос, есть ли у вас лицензия Windows Server, которая дает вам право на получение гибридной выгоды Azure.
3. Подтвердите, что у вас есть лицензия Windows Server с Software Assurance, которую вы можете использовать для применения преимущества к VM, который будет создан при сбою.
4. Сохраните настройки для реплицируемой виртуальной машины.

См. дополнительные сведения о программе [преимуществ гибридного использования Azure](https://aka.ms/azure-hybrid-benefit-pricing).
