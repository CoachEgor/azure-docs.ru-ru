# <a name="enable-replication-to-azure-for-vmware-vms"></a>Включение репликации в Azure для виртуальных машин VMware

В этой статье описывается, как включить репликацию виртуальных машин VMware в Azure.

## <a name="prerequisites"></a>Предварительные требования

Эта статья предполагает, что ваша система отвечает следующим критериям:

- [Настройка исходной среды.](vmware-azure-set-up-source.md)
- [Настройка целевой среды в Azure.](vmware-azure-set-up-target.md)
- [Проверьте требования и предпосылки](vmware-physical-azure-support-matrix.md) перед запуском. Важные вещи, чтобы отметить:
  - [Поддерживаемые операционные системы](vmware-physical-azure-support-matrix.md#replicated-machines) для реплицированных машин.
  - [Поддержка хранения/диска.](vmware-physical-azure-support-matrix.md#storage)
  - [Требования Azure,](vmware-physical-azure-support-matrix.md#azure-vm-requirements) которым должны соответствовать приемние помещений.

### <a name="resolve-common-issues"></a>Решение общих проблем

- Каждый диск должен быть меньше 4 ТБ.
- Диск операционной системы должен быть базовым диском, а не динамическим диском.
- Для виртуальных машин с поддержкой 2 поколения UEFI семейство операционных систем должно быть Windows, а загрузочный диск должен быть меньше 300 ГБ.

## <a name="before-you-start"></a>Перед началом работы

При репликации виртуальных машин VMware имейте эту информацию в уме:

- Учетная запись пользователя Azure должна содержать определенные [разрешения](site-recovery-role-based-linked-access-control.md#permissions-required-to-enable-replication-for-new-virtual-machines) для включения репликации новой виртуальной машины в Azure.
- Виртуальные машины VMware обнаруживаются каждые 15 минут. Появление вв.М. на портале Azure после обнаружения может занять 15 минут или более. При добавлении нового сервера vCenter или vSphere host открытие может занять 15 минут или более.
- Обновление виртуальной машины на портале может занять 15 минут или более. Например, установка инструментов VMware.
- Вы можете проверить последнее время для VMware VMs: Смотрите **последний контакт на** поле на странице **конфигурационных серверов** для vCenter server/vSphere host.
- Чтобы добавить виртуальные машины для репликации, не дожидаясь запланированного открытия, выделите сервер конфигурации (но не нажимайте на него) и выберите **Refresh.**
- При включении репликации, если виртуальная машина подготовлена, сервер процесса автоматически устанавливает службу восстановления сайта Azure на VM.

## <a name="enable-replication"></a>Включение репликации

Прежде чем делать шаги в этом разделе, просмотрите следующую информацию:

- Восстановление сайта Azure теперь реплицируется непосредственно на управляемые диски для всех новых репликаций. Сервер процесса записывает журналы репликации в учетную запись хранения кэша в целевом регионе. Эти журналы используются для создания точек восстановления в `asrseeddisk`управляемых дисках реплики, которые имеют именования конвенции.
- Поддержка PowerShell для репликации управляемых дисков доступна начиная с [версии модуля Az.RecoveryServices 2.0.0](https://www.powershellgallery.com/packages/Az.RecoveryServices/2.0.0-preview)
- Во время сбоя выбранная точка восстановления используется для создания диска, управляемого целевым, .
- VMs, которые ранее были настроены для репликации в целевые учетные записи хранилища, не затрагиваются.
- Репликация учетных записей для хранения новой виртуальной машины доступна только через API-доступную государственную передачу (REST) и PowerShell. Используйте версию Azure REST API 2016-08-10 или 2018-01-10 для репликации для учетных записей.

Для репликации выполните следующие действия:

1. Перейти к **шагу 2: Replicate приложение** > **Источник**. После того, как вы впервые включить репликацию, выберите **«Replicate»** в хранилище, чтобы включить репликацию для дополнительных виртуальных машин.
1. На странице **Источник** щелкните **Источник** и выберите нужный сервер конфигурации.
1. Для **типа машины**, выберите **виртуальные машины** или **физические машины.**
1. В поле **vCenter/vSphere Hypervisor** (Гипервизор vCenter или vSphere) выберите сервер vCenter, который управляет узлом vSphere, или сам узел. Эта настройка не актуальна, если вы реплицируете физические компьютеры.
1. Выберите сервер обработки. Если нет дополнительных серверов процесса, встроенный сервер процесса сервера конфигурации будет доступен в меню выпадения. Состояние работоспособности каждого сервера процесса указывается в соответствии с рекомендуемыми ограничениями и другими параметрами. Выберите работоспособный сервер обработки. [Критический](vmware-physical-azure-monitor-process-server.md#process-server-alerts) сервер процесса не может быть выбран. Вы можете [устранить неполадки и ошибки](vmware-physical-azure-troubleshoot-process-server.md)**или**[настроить сервер обработки масштабирования](vmware-azure-set-up-process-server-scale.md).

   :::image type="content" source="./media/vmware-azure-enable-replication/ps-selection.png" alt-text="Включить окно источника репликации":::

   > [!NOTE]
   > Начиная с [версии 9.24,](site-recovery-whats-new.md)дополнительные оповещения вводятся для улучшения оповещений о работоспособности сервера процесса. Обновление компонентов восстановления сайта до версии 9.24 или выше для всех предупреждений, которые будут созданы.

1. Для **Target**выберите группу подписки и ресурсов, где вы хотите создать сбой над виртуальными машинами. Выберите модель развертывания, которую требуется использовать в Azure для сбой в использовании визу множеся.
1. Выберите сеть и подсеть Azure, к которым будут подключаться VMs Azure после сбоя. Сеть должна находиться в том же регионе, что и хранилище службы восстановления сайта.

   Выберите **настройку для выбранных машин,** чтобы применить настройки сети для всех виртуальных машин, которые вы выбираете для защиты. Выберите **нанастройку позже,** чтобы выбрать сеть Azure на виртуальную машину. Если у вас нет сети, ее необходимо создать. Чтобы создать сеть с помощью менеджера ресурсов Azure, выберите **Создать новый.** Выберите подсеть, если это применимо, а затем выберите **OK.**

   :::image type="content" source="./media/vmware-azure-enable-replication/enable-rep3.png" alt-text="Включить целевое окно репликации":::

Вы можете выбрать только виртуальные машины, для которых можно было бы выключать репликацию. Нажмите кнопку **ОК**. Если вы не видите или не выбираете какую-либо конкретную виртуальную машину, см. [Машина Исхода не указана на портале Azure](vmware-azure-troubleshoot-replication.md#step-3-troubleshoot-source-machines-that-arent-available-for-replication) для устранения проблемы.

1. По умолчанию все диски источника VM реплицируются. Чтобы исключить диски из репликации, очистите флажок **Include** для любых дисков, которые вы не хотите реплицировать. Нажмите кнопку **ОК**.    > - Объедините виртуальные машины и физические сервера, чтобы они могли зеркалировать рабочие нагрузки. Включение согласованности нескольких виртуальных машин может повлиять на производительность рабочей нагрузки, Делайте это только в том случае, если виртуальные машины работают с той же рабочей нагрузкой, и вам нужна последовательность.

## <a name="view-and-manage-vm-properties"></a>Просмотр свойств виртуальной машины и управление ими

Затем проверьте свойства исходной виртуальной машины. Помните, что имя виртуальной машины Azure должно соответствовать [требованиям к виртуальным машинам Azure после отработки отказа](vmware-physical-azure-support-matrix.md#replicated-machines).

1. Перейти к **настройкам** > **Реплицированные элементы**, а затем выбрать виртуальную машину. Страница **Essentials** показывает информацию о настройках и статусе VM.
1. На разделе **Свойства** можно просмотреть сведения о репликации и отработке отказа для виртуальной машины.
1. В свойствах **Compute и Network** > **Compute**можно изменить несколько свойств VM.

   :::image type="content" source="./media/vmware-azure-enable-replication/vmproperties.png" alt-text="Окно вычислительных и сетевых свойств":::

   - **Имя Azure VM:** При необходимости измените имя в соответствии с требованиями Azure.
   - **Целевой размер VM или тип VM:** Размер VM по умолчанию выбирается на основе параметров, которые включают количество дисков, количество NIC, количество ядра процессора, память и доступные размеры ролей VM в целевой области Azure. Восстановление сайта Azure выбирает первый доступный размер VM, который удовлетворяет всем критериям. Вы можете выбрать другой размер VM в зависимости от ваших потребностей в любое время до неудачи. Размер диска VM также основан на размере исходного диска, и он может быть изменен только после неудачи. Узнайте больше о размерах дисков и ставках IOPS на [целевых показателях масштабируемости и производительности для VM-дисков на Windows.](/azure/virtual-machines/windows/disk-scalability-targets)
   - **Группа ресурсов**: Вы можете выбрать [группу ресурсов,](/azure/azure-resource-manager/management/overview#resource-groups)из которой виртуальная машина становится частью сбоя публикации. Вы можете изменить эту настройку в любое время до неудачи. После неудачи, если вы переносите виртуальную машину в другую группу ресурсов, настройки защиты для этого виртуального автомата перерыв.

     - Перечислены только наборы доступности, относяющиеся к указанной группе ресурсов.
     - Виртуальные вывизм, которые находятся в разных виртуальных сетях, не могут быть частью одного и того же набора доступности.
     - В одну группу доступности могут входить только виртуальные машины одинакового размера.

### <a name="configure-networks-and-ip-addresses"></a>Настройка сетей и IP-адресов

Вы можете установить целевой IP-адрес:

- Если вы не укажите адрес, сбой в VM использует DHCP.
- Если задать адрес, недоступный при отработке отказа, произойдет сбой.
- Если адрес доступен в сети тестового совмещении, можно использовать тот же целевой IP-адрес для сбоя теста.

Количество сетевых адаптеров определяется размером, который вы указываете для целевой виртуальной машины, следующим образом:

- Если количество сетевых адаптеров на исходной виртуальной машине меньше или равно числу адаптеров, которые разрешены для размера целевого VM, цель имеет такое же количество адаптеров, как и источник.
- Если количество адаптеров для исходной виртуальной машины превышает количество, которое разрешено для размера целевого VM, используется максимальный размер цели. Например, если исходная виртуальная машина имеет два сетевых адаптеров, а размер цели VM поддерживает четыре, то целевая виртуальная машина имеет два адаптера. Если источник VM имеет два адаптера, но размер цели поддерживает только один, целевой VM имеет только один адаптер.
- Если для виртуальной машины настроено несколько сетевых адаптеров, все они будут подключены к одной сети. Кроме того, первым адаптером, показанным в списке, становится сетевой адаптер по умолчанию в виртуальной машине Azure.

### <a name="azure-hybrid-benefit"></a>Преимущество гибридного использования Azure

Клиенты Microsoft Software Assurance могут использовать Azure Hybrid Benefit для экономии на затратах на лицензирование компьютеров Windows Server, которые переходят в Azure. Преимущество также распространяется на аварийное восстановление Azure. Если вы имеете право, вы можете назначить преимущество виртуальной машины, которую создает восстановление сайта, если произошел сбой.

1. Перейти к **компьютеру и сети свойства** реплицируемой виртуальной машины.
1. Отвечайте на вопрос, есть ли у вас лицензия Windows Server, которая дает вам право на получение гибридной выгоды Azure.
1. Подтвердите, что у вас есть лицензия Windows Server с Software Assurance, которую вы можете использовать для применения преимущества к VM, который будет создан при сбою.
1. Сохраните настройки для реплицируемой виртуальной машины.

[Узнайте больше](https://azure.microsoft.com/pricing/hybrid-benefit/) о преимуществах Azure Hybrid.
