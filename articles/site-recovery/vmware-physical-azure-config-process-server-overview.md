# <a name="about-site-recovery-components-configuration-process-master-target"></a>О компонентах восстановления сайта (конфигурация, процесс, главная цель)

В этой статье описаны конфигурация, процесс и главные целевые серверы, используемые службой [восстановления сайта](site-recovery-overview.md) для репликации VMware VMs и физических серверов в Azure.

## <a name="configuration-server"></a>Сервер конфигурации

Для аварийного восстановления предприимных VM-м и физических серверов развернуть сервер конфигурации восстановления сайта.

**Параметр** | **Подробно** | **Ссылки**
--- | --- | ---
**Компоненты**  | Машина сервера конфигурации выполняет все компоненты восстановления сайта, которые включают сервер конфигурации, сервер процесса и основной целевой сервер.<br/><br/> При настройке сервера конфигурации все компоненты устанавливаются автоматически. | [Прочитайте](vmware-azure-common-questions.md#configuration-server) часто задаваемые вопросы о конфигурации сервера.
**Role** | Сервер конфигурации используется для управления обменом данными между локальным источником и Azure, а также репликацией данных. | Узнайте больше об архитектуре [Для VMware](vmware-azure-architecture.md) и [восстановлении аварийного спасения физического сервера](physical-azure-architecture.md) в Azure.
**Требования к VMware** | Для аварийного восстановления предприимчивых VM VMw, вы должны установить и запустить сервер конфигурации в качестве внутреннего, высокодоступного VMware VM. | [Узнайте о](vmware-azure-deploy-configuration-server.md#prerequisites) предпосылках.
**Развертывание VMware** | Мы рекомендуем развертывать сервер конфигурации с помощью загруженного шаблона OVA. Этот метод обеспечивает простой способ настройки сервера конфигурации, который соответствует всем требованиям и предпосылкам.<br/><br/> Если по какой-либо причине вы не можете развернуть VMware VM с помощью шаблона OVA, вы можете настроить настройку серверных машин вручную, как описано ниже для восстановления аварийности физического аппарата. | [Развертывание](vmware-azure-deploy-configuration-server.md#deploy-a-configuration-server-through-an-ova-template) с шаблоном OVA.
**Требования для физических серверов** | Для восстановления аварийного засечника на внутренних физических серверах вы развертываете сервер конфигурации вручную. | [Узнайте о](physical-azure-set-up-source.md#prerequisites) предпосылках.
**Физическое развертывание сервера** | Если он не может быть установлен в качестве VMware VM, вы можете установить его на физическом сервере. | [Развертывайте](physical-azure-set-up-source.md#set-up-the-source-environment) сервер конфигурации вручную.

## <a name="process-server"></a>Сервер обработки

Сервер процесса обрабатывает данные репликации во время сбоя и сбоя, а также устанавливает службу мобильности для внутренних VM VMw и физических серверов.

**Параметр** | **Подробно** | **Ссылки**
--- | --- | ---
**Развертывание**  | По умолчанию при развертывании сервера конфигурации устанавливается сервер процесса. <br/><br/> Для аварийного восстановления и репликации vM-м и физических серверов vmwи vm и физических серверов необходим сервер для обработки процессов. | Ознакомьтесь с [дополнительными сведениями](vmware-azure-architecture.md#architectural-components).
**Роль (на территории)** | Получает данные репликации с машин, включенных для репликации. <br/><br/> Оптимизирует данные репликации с помощью кэширования, сжатия и шифрования и отправляет их в Хранилище Azure. <br/><br/> Выполняет push-установку службы восстановления сайта Mobility на предприимки VMware и физические серверы, которые вы хотите воспроизвести. <br/><br/> Выполняет автоматическое обнаружение припредельных машин. | Ознакомьтесь с [дополнительными сведениями](vmware-azure-enable-replication.md).
**Роль (неудача из Azure)** | После сбоя с вашего локальных сайтов вы настроили сервер процессов в Azure, как Azure VM, для обработки сбоев в вашем месте.<br/><br/> Сервер процесса в Azure является временным. VM Azure может быть удален после сбоя. | Ознакомьтесь с [дополнительными сведениями](vmware-azure-set-up-process-server-azure.md).
**Масштабирование** | Для более крупных развертываний можно настроить дополнительные серверы масштабирования. Дополнительные серверы масштабируют емкость, обрабатывая большее количество репликационных машин и большие объемы трафика репликации.<br/><br/> Вы можете перемещать машины между двумя серверами процессов, чтобы загрузить трафик репликации баланса. | Ознакомьтесь с [дополнительными сведениями](vmware-azure-set-up-process-server-scale.md).

## <a name="master-target-server"></a>Главный целевой сервер

Главный целевой сервер обрабатывает данные репликации при восстановлении размещения с переносом из Azure.

- По умолчанию на сервере конфигурации устанавливается основной целевой сервер.
- Для крупных развертываний вы можете добавить отдельный главный целевой сервер для восстановления размещения.
