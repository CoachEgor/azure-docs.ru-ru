# <a name="customize-networking-configurations-of-the-target-azure-vm"></a>Настройка сетевых конфигураций целевой виртуальной машины Azure

В этой статье содержатся рекомендации по настройке конфигураций сетей на целевой виртуальной машине Azure (VM) при репликации и восстановлении виртуальных технологий Azure из одного региона в другой с помощью [Azure Site Recovery.](site-recovery-overview.md)

## <a name="before-you-start"></a>Перед началом работы

Узнайте, как Site Recovery обеспечивает аварийное восстановление для [этого сценария](azure-to-azure-architecture.md).

## <a name="supported-networking-resources"></a>Поддерживаемые сетевые ресурсы

Вы можете предоставить следующие ключевые конфигурации ресурсов для failover VM при воспроизведении VMs Azure:

- [Внутренний балансер нагрузки](https://docs.microsoft.com/azure/load-balancer/load-balancer-overview)
- [Общественная ИС](https://docs.microsoft.com/azure/virtual-network/virtual-network-ip-addresses-overview-arm#public-ip-addresses)
- [Группа сетевой безопасности](https://docs.microsoft.com/azure/virtual-network/manage-network-security-group) как для подсети, так и для NIC

## <a name="prerequisites"></a>Предварительные требования

- Убедитесь, что вы заранее спланируете конфигурацию стороны восстановления.
- Создавайте сетевые ресурсы заранее. Предоставьте его в качестве вхотливки, чтобы служба восстановления сайта Azure смогли соблюдать эти настройки и гарантировать, что failover VM будет соответствовать этим настройкам.

## <a name="customize-failover-and-test-failover-networking-configurations"></a>Настройка конфигураций сетей безрезультатно и тестировать

1. Перейти к **Реплицированные элементы**. 
2. Выберите желаемый Azure VM.
3. Выберите **вычисления и сети** и выберите **Edit**. Обратите внимание, что параметры конфигурации NIC включают соответствующие ресурсы в источнике. 

     ![Настройка конфигураций сетей failover](media/azure-to-azure-customize-networking/edit-networking-properties.png)

4. Выберите виртуальную сеть тестового отказа. Вы можете оставить его пустым и выбрать один во время теста failover.
5. Сеть Failover — это Select **Edit** рядом с ником, который вы хотите настроить. В следующем лезвии, которое открывается, выберите соответствующие заранее созданные ресурсы в месте отказа теста и неудачи.

    ![Оторите конфигурацию NIC](media/azure-to-azure-customize-networking/nic-drilldown.png) 

6. Нажмите кнопку **ОК**.

Восстановление сайта теперь будет соблюдать эти настройки и гарантировать, что VM на failover подключен к выбранного ресурса через соответствующий NIC.

При запуске теста сотказа с помощью плана восстановления он всегда запрашивает виртуальную сеть Azure. Эта виртуальная сеть будет использоваться для тестирования failover для машин, которые не имеют настройки теста failover предварительно настроены.

## <a name="troubleshooting"></a>Устранение неполадок

### <a name="unable-to-view-or-select-a-resource"></a>Невозможно просмотреть или выбрать ресурс

Если вы не можете выбрать сетевой ресурс или просмотреть его, пройдите следующие проверки и условия:

- Целевое поле для сетевого ресурса включено только в том случае, если источник VM имел соответствующий вход. Это основано на принципе, что для сценария аварийного восстановления, вы хотели бы либо точное или уменьшенную версию вашего источника.
- Для каждого сетевого ресурса некоторые фильтры применяются в списке выпадающих, чтобы гарантировать, что сбой VM может прикрепиться к выбранному ресурсу и надежность failover сохраняется. Эти фильтры основаны на тех же сетевых условиях, которые были бы проверены при настройке исходного VM.

Внутренние проверки баланса нагрузки:

- Подписка и регион балансиста нагрузки и целевой VM должны быть одинаковыми.
- Виртуальная сеть, связанная с внутренним балансером нагрузки и с целевым VM, должна быть одинаковой.
- Целевой VM в общественных IP SKU и внутренней нагрузки балансоровыs SKU должны быть одинаковыми.
- Если целевой VM настроен для размещения в зоне доступности, проверьте, является ли балансослер нагрузки резервным или частью любой зоны доступности. (Основные балансирумы нагрузки SKU не поддерживают зоны и в этом случае не будут отображаться в списке выпадающих выбросов.)
- Убедитесь, что внутренний балансоровщик нагрузки имеет предварительно созданный пул бэк-энда и конфигурацию переднего конца.

Общественный IP-адрес:

- Подписка и регион общественного IP и целевой VM должны быть одинаковыми.
- Целевой VM в общественных IP SKU и внутренней нагрузки балансоровыs SKU должны быть одинаковыми.

Группа безопасности сети.
- Подписка и регион группы сетевой безопасности и целевой VM должны быть одинаковыми.


> [!WARNING]
> Если целевой VM связан с набором доступности, то необходимо связать общественный ip и внутренний балансер нагрузки того же SKU с балансом других ВМ и внутренней нагрузки в наборе доступности. Если вы этого не сможете, неудача может не увенчаться успехом.
