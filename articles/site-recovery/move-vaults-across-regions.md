# <a name="moving-recovery-services-vault-and--azure-site-recovery-configuration-to-another-azure-region"></a>Перемещение конфигурации хранилища Служб восстановления и Azure Site Recovery в другой регион Azure

Существуют различные сценарии, для которых необходимо выполнять перемещение имеющихся ресурсов Azure из одного региона в другой. Причины могут быть связаны с обеспечением управляемости, системой управления, слиянием или поглощением компаний и т. д. При перемещении виртуальных машин Azure может также понадобиться перенести конфигурацию аварийного восстановления. Совершенного способа перемещения имеющейся конфигурации аварийного восстановления из одного региона в другой не существует. Главной причиной является необходимость настройки целевого региона на основе исходного региона виртуальной машины. Если вы решите изменить регион, имеющиеся конфигурации целевого региона использовать будет уже нельзя, их нужно будет сбросить. В этой статье описан пошаговый процесс повторной настройки аварийного восстановления и перемещения в другой регион.

В рамках этого документа вы будете выполнять следующие операции:

> [!div class="checklist"]
> * проверять предварительные требования для переноса;
> * Определение ресурсов, которые использовались службой Azure Site Recovery. 
> * Отключение репликации
> * Удаление ресурсов 
> * Повторная настройка восстановления сайта на основе нового исходного региона для виртуальных машин.

> [!IMPORTANT]
> В настоящее время не существует совершенного способа перемещения конфигурации хранилища Служб восстановления и аварийного восстановления в другой регион. С помощью сведений, приведенных в этом документе, клиент сможет отключить репликацию и повторно настроить ее в новом регионе.

## <a name="prerequisites"></a>Предварительные требования

- Удалите конфигурацию аварийного восстановления, прежде чем пытаться переместить виртуальные машины Azure в другой регион. 

> [!NOTE]
> Если ваш целевой регион для виртуальной машины Azure совпадает с целевым регионом аварийного восстановления, вы можете использовать существующую конфигурацию репликации и выполнить перемещение, следуя шагам, указанным в [этой статье](azure-to-azure-tutorial-migrate.md). 

- Убедитесь, что вы принимаете обоснованное решение и заинтересованные лица проинформированы об изменениях, так как ваша виртуальная машина не будет защищена от сбоев до тех пор, пока перемещение не будет завершено. 

## <a name="identify-the-resources-that-were-used-by-azure-site-recovery"></a>Определение ресурсов, которые использовались службой Azure Site Recovery
Мы рекомендуем выполнить этот шаг, прежде чем переходить к следующему. Таким образом вам будет легче определить имеющиеся ресурсы, пока виртуальные машины все еще реплицируются.

В каждой реплицируемой виртуальной машине Azure выберите **Защищенные элементы** > **Реплицированные элементы**>**Свойства** и найдите следующие ресурсы:

- Целевая группа ресурсов
- Учетная запись хранения кэша.
- Целевая учетная запись хранения (в случае виртуальной машины Azure на основе неуправляемого диска). 
- Целевая сеть.


## <a name="disable-existing-disaster-recovery-configuration"></a>Отключение имеющейся конфигурации аварийного восстановления

1. Перейдите в раздел **Хранилище Служб восстановления**. 
2.  Выберите **Защищенные элементы** > **Реплицированные элементы**, щелкните виртуальную машину правой кнопкой мыши и выберите **Отключить репликацию**.
3. Сделайте это для всех виртуальных машин, которые необходимо переместить.
> [!NOTE]
> Служба мобильности не будет удалена с защищенных серверов, вам необходимо удалить ее вручную. Если вы планируете повторно защитить сервер, то удаление службы мобильности можно пропустить.

## <a name="delete-the-resources"></a>Удаление ресурсов

1. Перейдите в раздел **Хранилище Служб восстановления**.
2. Нажмите кнопку **Удалить**.
3. Удалите все ресурсы, указанные на [предыдущем шаге](#identify-the-resources-that-were-used-by-azure-site-recovery).
