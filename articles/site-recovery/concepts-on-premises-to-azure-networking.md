# <a name="connect-to-azure-vms-after-failover-from-on-premises"></a>Подключение к Пазур VMs после сбоя из наместераренного 


В этой статье описывается, как настроить подключение, чтобы можно было успешно подключиться к VMs Azure после сбоя.

При настройке аварийного восстановления виртуальных машин (Виртуальных технологий) и физических серверов в Azure [восстановление сайта Azure](site-recovery-overview.md) начинает репликацию машин в Azure. Затем, когда происходят перебои, вы можете не перейти на Azure с вашего сайта. При сбою происходит восстановление сайта, создает Azure VMs, используя реплицированные данные. В рамках планирования аварийного восстановления необходимо выяснить, как подключиться к приложениям, работающим на этих MMs Azure после сбоя.

В этой статье раскрываются следующие темы:

> [!div class="checklist"]
> * Подготовьте наемные машины до сбоя.
> * Подготовьте VMs Azure после сбоя. 
> * Сохранение IP-адресов на M Ms Azure после сбоя.
> * Присваивай новые IP-адреса в Адрес Azure VMs после сбоя.

## <a name="prepare-on-premises-machines"></a>Подготовка наместерачных машин

Чтобы обеспечить подключение к VMs Azure, подготовьте свои предприимченные машины перед сбой.

### <a name="prepare-windows-machines"></a>Подготовка компьютеров с Windows

На локальных компьютерах с Windows выполните следующие действия:

1. Настройте параметры Windows. Они включают в себя удаление любых статических постоянных маршрутов или прокси WinHTTP, а также настройка политики s-диска SAN на **OnlineAll.** [Следуйте](../virtual-machines/windows/prepare-for-upload-vhd-image.md#set-windows-configurations-for-azure) этим инструкциям.

2. Убедитесь, что [эти службы](../virtual-machines/windows/prepare-for-upload-vhd-image.md#check-the-windows-services) работают.

3. Включите удаленный рабочий стол (RDP), чтобы разрешить установку удаленных подключений к локальному компьютеру. Сведения о включении RDP с помощью PowerShell см. в [этой статье](../virtual-machines/windows/prepare-for-upload-vhd-image.md#update-remote-desktop-registry-settings).

4. Чтобы получить доступ к Azure VM через Интернет после сбоя, в Windows Firewall на предприимчивой машине, разрешить TCP и UDP в общедоступном профиле и установить RDP в качестве разрешенного приложения для всех профилей.

5. Если вы хотите получить доступ к VM Azure через VPN от сайта к сайту после сбоя, в Windows Firewall на предприимчивой машине, разрешите RDP для домена и частных профилей. Ознакомьтесь со сведениями о разрешении трафика RDP на [этой странице](../virtual-machines/windows/prepare-for-upload-vhd-image.md#configure-windows-firewall-rules).
6. Убедитесь, что на предварительном VM нет обновлений Windows, когда вы вызываете сбой. В этом случае обновления могут начать сяду на Azure VM после сбоя, и вы не сможете войти в VM до тех пор, пока обновления не закончатся.

### <a name="prepare-linux-machines"></a>Подготовка компьютеров с Linux

На локальных компьютерах с Linux выполните следующие действия:

1. Настройте автоматический запуск службы SSH при загрузке системы (если он еще не настроен).
2. Убедитесь, что правила брандмауэра разрешают SSH-подключение.


## <a name="configure-azure-vms-after-failover"></a>Настройка VMs Azure после сбоя

После неудачи сделайте следующее на созданных VMs-ми зачетах Azure.

1. Чтобы подключиться к виртуальной машине через Интернет, назначьте ей общедоступный IP-адрес. Для виртуальной машины Azure нельзя указывать общедоступный IP-адрес, ранее используемый для локального компьютера. [Дополнительные сведения](../virtual-network/virtual-network-public-ip-address.md)
2. Убедитесь, что правила группы безопасности сети (NSG) на виртуальной машине разрешают входящие подключения к порту RDP или SSH.
3. Перейдите в раздел [Диагностика загрузки](../virtual-machines/troubleshooting/boot-diagnostics.md#enable-boot-diagnostics-on-existing-virtual-machine), чтобы просмотреть сведения о виртуальной машине.


> [!NOTE]
> Служба "Бастион Azure" предоставляет закрытый доступ по RDP и SSH к виртуальным машинам Azure. [Ознакомьтесь с дополнительными сведениями](../bastion/bastion-overview.md) об этой службе.

## <a name="set-a-public-ip-address"></a>Установка общедоступного IP-адреса

В качестве альтернативы назначению общедоступного IP-адреса вручную VM Azure можно назначить адрес во время сбоя с помощью сценария или запуска автоматизации Azure в [плане восстановления](site-recovery-create-recovery-plans.md)сайта, или настроить routing уровня DNS с помощью azure Traffic Manager. [Подробнее](concepts-public-ip-address-with-site-recovery.md) о настройке публичного адреса.


## <a name="assign-an-internal-address"></a>Назначить внутренний адрес

Чтобы установить внутренний IP-адрес Azure VM после сбоя, у вас есть несколько вариантов:

- **Сохранить тот же IP-адрес:** Вы можете использовать тот же IP-адрес на Azure VM, что и тот, который выделен для внепредки.
- **Используйте другой IP-адрес:** Вы можете использовать другой IP-адрес для Azure VM.


## <a name="retain-ip-addresses"></a>Использование тех же IP-адресов

Восстановление сайта позволяет сохранить те же IP-адреса при сбое в Azure. Сохранение одного и того же IP-адреса позволяет избежать потенциальных проблем сети после сбоя, но вносит некоторые сложности.

- Если целевой Azure VM использует тот же IP-адрес/подсеть, что и ваш сайт, вы не можете подключиться между ними с помощью VPN-соединения сайта или ExpressRoute из-за перекрытия адреса. Подсети должны быть уникальными.
- Вам необходимо подключение от нахастого к Azure после сбоя, так что приложения доступны на Azure VMs. Azure не поддерживает растянутые VLAN, поэтому, если вы хотите сохранить IP-адреса, вам необходимо перебрать IP-пространство в Azure, не сумев провести всю подсеть, в дополнение к предварительной машине.
- Сбой Subnet гарантирует, что конкретная подсеть недоступна одновременно на местах и в Azure.

Сохранение IP-адресов требует следующих шагов:

- В вычислительной & свойствами реплицируемого элемента, установленной сети и IP-адреса для целевого Azure VM для отражения настройки на месте.
- Подсети должны управляться в процессе аварийного восстановления. Для соответствия сети Azure VNet требуется Alure VNet, а после сбоя сетевые маршруты должны быть изменены, чтобы отразить, что подсеть переехала в Azure и новые места IP-адресов.  

### <a name="failover-example"></a>Пример отработки отказа

Давайте рассмотрим пример.

- Фиктивная компания Woodgrove Bank размещает свои бизнес-приложения в помещениях, где они размещают свои мобильные приложения в Azure.
- Они подключаются от наместа к Azure через VPN от сайта к сайту. 
- Woodgrove использует восстановление сайта для воспроизведения наместах машин в Azure.
- Их приложения используют ЖЕСТКИе IP-адреса, поэтому они хотят сохранить те же IP-адреса в Azure.
- На месте машины, работающие с приложениями, работают в трех подсетях:
    - 192.168.1.0/24.
    - 192.168.2.0/24
    - 192.168.3.0/24
- Их приложения, работающие в Azure, расположены в сети Azure VNet **Azure** в двух подсетях:
- 172.16.1.0/24
- 172.16.2.0/24.

Для того, чтобы сохранить адреса, вот что они делают.

1. При вращении репликации указывается, что машины должны реплицироваться в **сеть Azure.**
2. Они создают **сеть восстановления** в Azure. Это VNet отражает 192.168.1.0/24 подсеть в их предварительной сети.
3. Woodgrove устанавливает [соединение VNet-to-VNet](../vpn-gateway/vpn-gateway-howto-vnet-vnet-resource-manager-portal.md) между двумя сетями. 

    > [!NOTE]
    > В зависимости от требований к заявке, соединение VNet-to-VNet может быть настроено до сбоя, так как в [плане восстановления](site-recovery-create-recovery-plans.md)восстановления сайта завершается выполнение ручного шага/сценарного шага/azure автоматизации автоматизации или после завершения сбоя.

4. Перед неудачей, на свойствах машины в восстановлении сайта, они устанавливают целевой IP-адрес на адрес наемного аппарата, как описано в следующей процедуре.
5. После сбоя в создании VMs Azure с тем же IP-адресом создаются с тем же IP-адресом. Woodgrove подключается от **Сети Azure** к **сети восстановления** VNet с помощью внимательного внизания VNet (с включенным подключением транзита).
6. В помещениях Woodgrove необходимо внести изменения в сеть, втомив изменение маршрутов, чтобы отразить, что 192.168.1.0/24 переместился в Azure.

### <a name="set-target-network-settings"></a>Установка параметров целевой сети

Перед сбоем укажите настройки сети и IP-адрес для целевого Azure VM.

1.  В хранилище служб восстановления - > **Реплицированные элементы,** выберите предприимки.
2. На странице **вычислений и сети** для машины щелкните **Edit,** чтобы настроить настройки сети и адаптера для целевого Azure VM.
3. В **свойствах Сети**выберите целевую сеть, в которой будет находиться VM Azure, когда он будет создан после сбоя.
4. В **сетевых интерфейсах**настраиваем сетевые адаптеры в целевой сети. По умолчанию Восстановление сайта показывает все обнаруженные NICs на предприимчивой машине.
    - В **типе интерфейса Целевой сети** вы можете установить каждый NIC как **Основной,** **Вторичный,** или **не создавать,** если вам не нужно, что конкретные NIC в целевой сети. Один сетевой адаптер должен быть установлен в качестве основного для сбоя. Обратите внимание, что изменение целевой сети влияет на все NICs для Azure VM.
    - Нажмите на имя NIC, чтобы указать подсеть, в которой будет развернут VM Azure.
    - Перезаписи **Dynamic** с помощью частного IP-адреса, который вы хотите назначить для целевой Azure VM. Если IP-адрес не указан, восстановление сайта назначит следующий доступный IP-адрес в подсети NIC при сбою.
    - [Подробнее](site-recovery-manage-network-interfaces-on-premises-to-azure.md) об управлении НИК для локтевых сбоев в Azure.
