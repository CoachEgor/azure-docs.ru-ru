## <a name="replication"></a>Репликация

### <a name="can-i-replicate-vms-enabled-through-azure-disk-encryption"></a>Можно ли реплицировать виртуальные машины, включенные с помощью шифрования дисков Azure?

Да. Восстановление сайта поддерживает аварийное восстановление vMs с включенным шифрованием Azure Disk. При вращении репликации Azure копирует все необходимые ключи шифрования дисков и секреты из исходного региона в целевой регион в контексте пользователя. Если у вас нет соответствующих разрешений, администратор безопасности может использовать скрипт для копирования ключей и секретов.

- Восстановление сайта поддерживает шифрование azure Disk для vMs-технологий Azure, работающих под управлением Windows.
- Восстановление сайта поддерживает версию шифрования дисков Azure 0.1, которая имеет схему, требующая активного каталога Azure (Azure AD). Восстановление сайта также поддерживает версию 1.1, которая не требует Azure AD. [Подробнее о схеме расширения для шифрования дисков Azure.](../virtual-machines/extensions/azure-disk-enc-windows.md#extension-schema)
  - Для версии 1.1 шифрования дисков Azure необходимо использовать Windows VMs с управляемыми дисками.
  - [Подробнее](azure-to-azure-how-to-enable-replication-ade-vms.md) о включении репликации для зашифрованных VM.

### <a name="can-i-replicate-vms-to-another-subscription"></a>Можно ли реплицировать виртуальные машины в другую подписку?

Да, вы можете реплицировать виртуальные машины Azure в другую подписку в том же клиенте Azure AD.

Нанастройка аварийного восстановления [по подписке,](https://azure.microsoft.com/blog/cross-subscription-dr) выбрав другую подписку во время репликации.

### <a name="can-i-replicate-zone-pinned-azure-vms-to-another-region"></a>Можно ли реплицировать виртуальные машины Azure, прикрепленные к зонам, в другой регион?

Да, вы можете [реплицировать виртуальные машины, прикрепленные к зонам](https://azure.microsoft.com/blog/disaster-recovery-of-zone-pinned-azure-virtual-machines-to-another-region), в другой регион.

### <a name="can-i-exclude-disks"></a>Можно ли исключать диски?

Да, вы можете исключить диски во время защиты с помощью PowerShell. Для получения дополнительной информации см., [как исключить диски из репликации](azure-to-azure-exclude-disks.md).

### <a name="can-i-add-new-disks-to-replicated-vms-and-enable-replication-for-them"></a>Могу ли я добавить новые диски для реплицированных VMs и включить репликацию для них?

Да, добавление новых дисков для реплицированных VMs и включение репликации для них поддерживается для Azure VMs с управляемыми дисками. При добавлении нового диска в Azure VM, включенный для репликации, работоспособность репликации для VM показывает предупреждение. Это предупреждение гласит, что один или несколько дисков на VM доступны для защиты. Можно включить репликацию для добавленных дисков.

- Если вы включите защиту добавленных дисков, предупреждение исчезнет после первоначального репликации.
- Если вы не включите репликацию для диска, вы можете отклонить предупреждение.
- Если вы не сможете сделать vM с включенным диском и репликацией, есть точки репликации. Точки репликации будут отображать диски, доступные для восстановления.

Например, предположим, что у VM есть один диск, и вы добавляете новый. Перед добавлением диска может быть создана точка репликации. Эта точка репликации покажет, что она состоит из "1 из 2 дисков".

Восстановление сайта не поддерживает "горячее удаление" диска из реплицируемого VM. Если вы удалите VM-диск, вам нужно отключить, а затем повторно включить репликацию для VM.

### <a name="how-often-can-i-replicate-to-azure"></a>Как часто можно выполнять репликацию в Azure?

Этот процесс является непрерывным при репликации виртуальных машин Azure в другой регион Azure. Дополнительные сведения см. в статье [Архитектура аварийного восстановления из Azure в Azure](https://docs.microsoft.com/azure/site-recovery/azure-to-azure-architecture#replication-process).

### <a name="can-i-replicate-virtual-machines-within-a-region-i-need-this-functionality-to-migrate-vms"></a>Можно ли реплицировать виртуальные машины в одном регионе? Мне нужна эта функциональность для переноса виз.

Для репликации вс-м в регионе нельзя использовать решение восстановления дисков Azure-to-Azure.

### <a name="can-i-replicate-vm-instances-to-any-azure-region"></a>Могу ли я воспроизвести экземпляры VM в любом регионе Azure?

Используя восстановление сайта, можно реплицировать и восстанавливать визмежду нами между любыми двумя регионами в пределах одного и того же географического кластера. Географические кластеры определены с учетом задержки и независимости данных. Дополнительные сведения см. в разделе [Поддержка регионов](https://docs.microsoft.com/azure/site-recovery/azure-to-azure-support-matrix#region-support).

### <a name="does-site-recovery-require-internet-connectivity"></a>Требуется ли подключение к Интернету для Site Recovery?

Нет, восстановление сайта не требует подключения к Интернету. Но это требует доступа к URL-адресам восстановления сайта и диапазонам IP, о чем говорится в [сети в аварийном восстановлении Azure VM.](https://docs.microsoft.com/azure/site-recovery/azure-to-azure-about-networking#outbound-connectivity-for-ip-address-ranges)

### <a name="can-i-replicate-an-application-that-has-a-separate-resource-group-for-separate-tiers"></a>Могу ли я воспроизвести приложение, которое имеет отдельную группу ресурсов для отдельных уровней?

Да, можно воспроизвести приложение и сохранить конфигурацию аварийного восстановления в отдельной группе ресурсов.

Например, если приложение имеет приложение каждого уровня, базу данных и веб-страницу в отдельной группе ресурсов, то вам придется выбрать [мастера репликации](https://docs.microsoft.com/azure/site-recovery/azure-to-azure-how-to-enable-replication#enable-replication) три раза, чтобы защитить все уровни. Восстановление сайта позволит воспроизвести эти три уровня на три различные группы ресурсов.

## <a name="replication-policy"></a>Политика репликации

### <a name="what-is-a-replication-policy"></a>Что такое политика репликации?

Политика репликации определяет параметры истории удержания точек восстановления. Политика также определяет частоту последовательных снимков приложений. По умолчанию Azure Site Recovery создает новую политику репликации с параметрами по умолчанию:

- 24 часа для хранения точек восстановления;
- 60 минут для периодичности создания моментальных снимков с согласованием приложений.

[Подробнее о настройках репликации.](https://docs.microsoft.com/azure/site-recovery/azure-to-azure-tutorial-enable-replication#configure-replication-settings)

### <a name="what-is-a-crash-consistent-recovery-point"></a>Что такое отказоустойчивая точка восстановления?

Точка аварийного восстановления имеет данные на диске, как если бы вы вытащили шнур питания с сервера во время снимка. Точка аварийного восстановления не включает ничего, что было в памяти при снимке.

В настоящее время большинство приложений может успешно восстанавливаться из отказоустойчивых моментальных снимков. Отказоустойчивой точки восстановления обычно достаточно для операционных систем (кроме баз данных), файловых серверов приложений, DHCP-серверов и серверов печати.

### <a name="what-is-the-frequency-of-crash-consistent-recovery-point-generation"></a>Какая частота создания отказоустойчивой точки восстановления?

Site Recovery создает отказоустойчивую точку восстановления каждые 5 минут.

### <a name="what-is-an-application-consistent-recovery-point"></a>Что такое точка восстановления, согласованная с приложением?

Согласованные с приложением точки восстановления создаются из согласованных с приложением моментальных снимков. Пункты восстановления, согласованные с приложением, захватывают те же данные, что и сочные моментальные снимки, а также захват данных в памяти и всех транзакций в процессе.

Из-за их дополнительного содержания, приложения-последовательные снимки являются наиболее вовлеченными и занимают самый длинный. Согласованные с приложениями точки восстановления рекомендуется использовать для операционных систем баз данных и таких приложений, как SQL Server.

### <a name="what-is-the-impact-of-application-consistent-recovery-points-on-application-performance"></a>Как влияют согласованные с приложениями точки восстановления на производительность приложения?

Пункты восстановления, согласованные с приложением, захватывают все данные в памяти и в процессе. Поскольку точки восстановления захватывают эти данные, они требуют фреймворка, такой как Volume Shadow Copy Service на Windows, чтобы потихоньку заполучить приложение. Если процесс захвата является частым, это может повлиять на производительность, когда рабочая нагрузка уже занята. Мы не рекомендуем использовать низкочастотные для согласованных пунктов восстановления приложений для рабочих нагрузок, не связанных с базой данных. Даже для рабочей нагрузки базы данных достаточно 1 часа.

### <a name="what-is-the-minimum-frequency-of-application-consistent-recovery-point-generation"></a>Какова минимальная частота создания точек восстановления, согласованных с приложением?

Восстановление сайта может создать точку восстановления приложения с минимальной частотой 1 час.

### <a name="how-are-recovery-points-generated-and-saved"></a>Каким образом точки восстановления создаются и сохраняются?

Чтобы понять, как восстановление сайта генерирует точки восстановления, рассмотрим пример политики репликации. Эта политика репликации имеет точку восстановления с 24-часовым окном удержания и совместимым частотным снимком частоты приложения 1 час.

Site Recovery создает отказоустойчивую точку восстановления каждые 5 минут. Вы не можете изменить эту частоту. За последний час вы можете выбрать из 12 аварийных точек и 1 точки, согласованных с приложением. С течением времени восстановление сайта подчеркает все точки восстановления после последнего часа и экономит только 1 точку восстановления в час.

На следующем снимке экрана показан пример. На снимке экрана:

- В течение последнего часа, Есть точки восстановления с частотой 5 минут.
- Помимо прошедшего часа, восстановление сайта сохраняет только 1 точку восстановления.

   ![Список созданных точек восстановления](./media/azure-to-azure-troubleshoot-errors/recoverypoints.png)

### <a name="how-far-back-can-i-recover"></a>До какой точки во времени можно выполнить восстановление?

Самая давняя доступная точка восстановления — 72 часа.

### <a name="i-have-a-replication-policy-of-24-hours-what-will-happen-if-a-problem-prevents-site-recovery-from-generating-recovery-points-for-more-than-24-hours-will-my-previous-recovery-points-be-lost"></a>У меня есть политика репликации 24 часов. Что произойдет, если проблема не позволит восстановить сайт от генерации точек восстановления в течение более 24 часов? Потеряю ли я предыдущие точки восстановления?

Нет. В этом случае Site Recovery будет хранить все предыдущие точки восстановления. В зависимости от окна удержания точек восстановления восстановление сайта заменяет старейшую точку только в том случае, если она генерирует новые точки. Из-за проблемы восстановление сайта не может генерировать новые точки восстановления. До тех пор, пока не появляются новые точки восстановления, все старые точки останутся после того, как вы достигнете окна удержания.

### <a name="after-replication-is-enabled-on-a-vm-how-do-i-change-the-replication-policy"></a>Как изменить политику репликации после включения репликации на виртуальной машине?

Перейдите к > **политикам репликации инфраструктуры восстановления инфраструктуры****инфраструктуры** >  **сайта.** Выберите политику, которую необходимо отсечь, и сохраните изменения. Изменения будут применяться и к существующим репликациям.

### <a name="are-all-the-recovery-points-a-complete-copy-of-the-vm-or-a-differential"></a>Все точки восстановления представляют полную или разностную копию виртуальной машины?

Первая сгенерированная точка восстановления имеет полную копию. Все последующие точки восстановления имеют разностные изменения.

### <a name="does-increasing-the-retention-period-of-recovery-points-increase-the-storage-cost"></a>Увеличивает ли возрастание периода хранения точки восстановления затраты на хранилище?

Да, если вы увеличите период удержания с 24 часов до 72 часов, восстановление сайта сохранит точки восстановления еще на 48 часов. За дополнительное время хранения будет взиматься плата. Например, в одной точке восстановления могут быть дельта-изменения на 10 ГБ с стоимостью за ГБ в размере 0,16 доллара в месяц. Дополнительные сборы будут составят $1.60 и 48 в месяц.

## <a name="multi-vm-consistency"></a>Согласованность нескольких виртуальных машин

### <a name="what-is-multi-vm-consistency"></a>Что такое мульти-VM согласованность?

Согласованность Multi-VM гарантирует согласованность точки восстановления во всех реплицированных виртуальных машинах.

Восстановление сайта обеспечивает опцию **последовательности Multi-VM,** которая создает группу репликации всех машин.

При сбое в виртуальных машинах они будут иметь общие точки восстановления, согласованные с хэшрами и приложениями.

Пройдите через учебник, чтобы [включить мульти-VM согласованности.](https://docs.microsoft.com/azure/site-recovery/azure-to-azure-tutorial-enable-replication#enable-replication-for-a-vm)

### <a name="can-i-fail-over-a-single-virtual-machine-within-a-multi-vm-consistency-replication-group"></a>Могу ли я выполнить неудачу в одной виртуальной машине в группе репликации последовательности multi-VM?

При выборе опции **согласованности Multi-VM** вы заявляете, что приложение зависит от всех виртуальных машин в группе. Одиночная виртуальная машина сбой не допускается.

### <a name="how-many-virtual-machines-can-i-replicate-as-a-part-of-a-multi-vm-consistency-replication-group"></a>Сколько виртуальных машин я могу воспроизвести как часть группы репликации последовательности multi-VM?

Вы можете совместно реплицировать 16 виртуальных машин в группе репликации.

### <a name="when-should-i-enable-multi-vm-consistency"></a>Когда следует включить консистенцию multi-VM?

Поскольку согласованность мульти-VM является интенсивной процессором, что может повлиять на производительность рабочей нагрузки. Используйте согласованность мульти-VM только в том случае, если машины работают с одной и той же рабочей нагрузкой и вам нужна последовательность в нескольких машинах. Например, если в приложении есть два экземпляра сервера S'L и два веб-сервера, то должна быть согласованность нескольких VM только для экземпляров сервера S'L.

### <a name="can-you-add-an-already-replicating-vm-to-a-replication-group"></a>Можете ли вы добавить уже реплицируя VM в группу репликации?

Вы можете добавить VM в новую группу репликации, одновременно включив репликацию. Вы также можете добавить VM в существующую группу репликации, одновременно включив репликацию. Однако нельзя добавлять уже реплицируемый VM в новую группу репликации или существующую группу репликации.

## <a name="failover"></a>Отработка отказа

### <a name="how-is-capacity-ensured-in-the-target-region-for-azure-vms"></a>Как обеспечивается пропускная способность в целевом регионе для VMs Azure?

Команда восстановления сайта и группа управления емкостями Azure планируют достаточную инфраструктурную емкость. При запуске сбоя группы также помогают обеспечить развертывание экземпляров VM, защищенных восстановлением сайта, в целевом регионе.

### <a name="is-failover-automatic"></a>Отработка отказа выполняется автоматически?

Отработка отказа не выполняется автоматически. Вы можете начать неудачи одним щелчком мыши на портале, или вы можете использовать [PowerShell,](azure-to-azure-powershell.md) чтобы вызвать неудачу.

### <a name="can-i-keep-a-public-ip-address-after-a-failover"></a>Могу ли я сохранить общедоступный IP-адрес после сбоя?

Вы не можете сохранить общедоступный IP-адрес производственного приложения после сбоя.

При составлении рабочей нагрузки в рамках процесса сбоя необходимо присвоить рабочую нагрузку общедоступному иИПой Azure. Общедоступный ресурс ИС Azure должен быть доступен в целевом регионе. Присвоить общедоступный ip-ресурс Azure можно вручную или автоматизировать его с помощью плана восстановления. Узнайте, как [настроить общедоступные IP-адреса после сбоя.](concepts-public-ip-address-with-site-recovery.md#public-ip-address-assignment-using-recovery-plan)

### <a name="can-i-keep-a-private-ip-address-during-a-failover"></a>Могу ли я сохранить личный IP-адрес во время сбоя?

Да, вы можете сохранить личный IP-адрес. По умолчанию при включении аварийного восстановления для виртуальных машин Azure служба Site Recovery создает целевые ресурсы на основе параметров исходных ресурсов. Для виртуальных машин Azure, настроенных со статичными IP-адресами, Восстановление сайта пытается предоставить тот же IP-адрес для целевого VM, если он не используется.
Подробнее о [сохранении IP-адресов во время сбоя.](site-recovery-retain-ip-azure-vm-failover.md)

### <a name="after-a-failover-why-is-the-server-assigned-a-new-ip-address"></a>Почему серверу присваивается новый IP-адрес?

Site Recovery пытается предоставить IP-адрес во время отработки отказа. Если другая виртуальная машина принимает этот адрес, Site Recovery устанавливает следующий доступный IP-адрес.

Подробнее о [настройке сетевого картирования и IP-адресов для виртуальных сетей.](azure-to-azure-network-mapping.md#set-up-ip-addressing-for-target-vms)

### <a name="what-are-latest-lowest-rpo-recovery-points"></a>Что такое точка восстановления с меткой **Последняя (наименьшее значение RPO)**?

**Последний (самый низкий RPO)** параметр сначала обрабатывает все данные, отправленные на восстановление сайта. После обработки данных служба создает точку восстановления для каждого VM, прежде чем перейти к VM. Эта опция обеспечивает самую низкую цель точки восстановления (RPO). VM, созданный после неудачи, имеет все данные, реплицированные на восстановление сайта с момента срабатывания.

### <a name="do-latest-lowest-rpo-recovery-points-have-an-impact-on-failover-rto"></a>Влияет ли точка восстановления с меткой **Последняя (наименьшее значение RPO)** на RTO отработки отказа?

Да. Восстановление сайта обрабатывает все ожидающие данные до сбоя, поэтому эта опция имеет более высокую цель времени восстановления (RTO) по сравнению с другими параметрами.

### <a name="what-does-the-latest-processed-option-in-recovery-points-mean"></a>Что означает параметр точки восстановления **Последняя обработанная**?

**Последняя обработанная** опция не справляется со всеми вмвм в плане до последней точки восстановления, которую обрабатывают сярвосстановления сайта. Чтобы увидеть последнюю точку восстановления для конкретного VM, проверьте **последние точки восстановления** в настройках VM. Этот вариант обеспечивает низкий показатель RTO, так как не требует времени на обработку данных.

### <a name="what-happens-if-my-primary-region-experiences-an-unexpected-outage"></a>Что произойдет, если в моем первичном регионе произойдет неожиданный сбой?

Вы можете активировать отработку отказа после сбоя. Восстановление сайта не требует подключения из основной области, чтобы сделать сбой.

### <a name="what-is-an-rto-of-a-vm-failover"></a>Что такое RTO неудачи VM?

Восстановление сайта имеет [RTO SLA 2 часа](https://azure.microsoft.com/support/legal/sla/site-recovery/v1_2/). Тем не менее, большую часть времени, восстановление сайта не удается над виртуальными машинами в течение нескольких минут. Вы можете рассчитать RTO, перейдя на сбой рабочих мест, которые показывают время, необходимое для воспитания VM. Для плана восстановления RTO, обратитесь к следующему разделу.

## <a name="recovery-plans"></a>Планы восстановления

### <a name="what-is-a-recovery-plan"></a>Что такое план восстановления?

План восстановления в Site Recovery управляет восстановлением виртуальных машин после отработки отказа. Они помогают реализовать точное, воспроизводимое и автоматическое восстановление. План восстановления удовлетворяет следующие потребности:

- Определение группы виртуальных машин, для которых выполняется единая отработка отказа
- Определение зависимостей между виртуальными машинами для точного восстановления приложения
- Автоматизация восстановления и пользовательских действий, выполняемых вручную, чтобы можно было выполнить на виртуальных машинах другие задачи, кроме отработки отказа

Подробнее [о создании планов восстановления.](site-recovery-create-recovery-plans.md)

### <a name="how-is-sequencing-achieved-in-a-recovery-plan"></a>Каким образом в плане восстановления достигается виртуализация?

В плане восстановления можно создать несколько групп для достижения виртуализации. Отработка отказа каждой группы выполняется по очереди. Виртуальные машины, которые являются частью одной группы, выходят из строя вместе, за ними следует другая группа. Чтобы узнать, как смоделировать приложение с помощью плана восстановления, обратитесь к статье [Сведения о планах восстановления](recovery-plan-overview.md#model-apps).

### <a name="how-can-i-find-the-rto-of-a-recovery-plan"></a>Как определить RTO плана восстановления?

Чтобы проверить RTO плана восстановления, протестируйте отработку отказа плана восстановления и перейдите к **заданиям Site Recovery**.
В следующем примере см. **SAPTestRecoveryPlan** Задание заняло 8 минут и 59 секунд, чтобы потерпеть неудачу над всеми виртуальными машинами и сделать определенные действия.

![Список заданий Site Recovery](./media/azure-to-azure-troubleshoot-errors/recoveryplanrto.PNG)

### <a name="can-i-add-automation-runbooks-to-the-recovery-plan"></a>Можно ли добавлять модули runbook службы автоматизации в планы восстановления?

Да, вы можете интегрировать в план восстановления модули runbook службы автоматизации Azure. Подробнее о [добавлении runbooks Azure Automation.](site-recovery-runbook-automation.md)

## <a name="reprotection-and-failback"></a>Повторное включение защиты и восстановление размещения

### <a name="i-failed-over-from-the-primary-region-to-a-disaster-recovery-region-are-vms-in-a-dr-region-protected-automatically"></a>Я потерпел неудачу из первичного региона в район аварийного восстановления. Защищены ли ВМ в регионе DR автоматически?

Нет. При выполнении [отработки отказа](https://docs.microsoft.com/azure/site-recovery/azure-to-azure-tutorial-failover-failback) виртуальных машин Azure из одного региона в другой их начальная загрузка в регионе аварийного восстановления не защищена. Чтобы восстановить размещение виртуальных машин в основном регионе, [повторно включите защиту](https://docs.microsoft.com/azure/site-recovery/azure-to-azure-how-to-reprotect) виртуальных машин в дополнительном регионе.

### <a name="at-the-time-of-reprotection-does-site-recovery-replicate-complete-data-from-the-secondary-region-to-the-primary-region"></a>Реплицирует ли Site Recovery во время повторного включения защиты все данные из дополнительного региона в основной регион?

Это зависит от ситуации. Если область источника VM существует, то синхронизируются только изменения между исходным диском и целевым диском. Site Recovery вычисляет различия путем сравнения дисков, а затем передает данные. Этот процесс обычно занимает несколько часов. Для получения дополнительной информации о том, что происходит во время резащиты, см. [Reprotect не удалось над экземплярами Azure VM в основной области.](https://docs.microsoft.com/azure/site-recovery/azure-to-azure-how-to-reprotect#what-happens-during-reprotection)

### <a name="how-much-time-does-it-take-to-fail-back"></a>Сколько времени занимает восстановление размещения?

После повторной защиты отказ занимает примерно столько же времени, что требуется, чтобы сбой из первичного региона во вторичный регион.

## <a name="capacity"></a><a name="capacity"></a>Capacity

### <a name="how-is-capacity-ensured-in-the-target-region-for-azure-vms"></a>Как обеспечивается пропускная способность в целевом регионе для VMs Azure?

Команда восстановления сайта и группа управления емкостями Azure планируют достаточную инфраструктурную емкость. При запуске сбоя группы также помогают обеспечить развертывание экземпляров VM, защищенных восстановлением сайта, в целевом регионе.

### <a name="does-site-recovery-work-with-reserved-instances"></a>Работает ли восстановление сайта с зарезервированными экземплярами?

Да, вы можете приобрести [зарезервированные VMs Azure](https://azure.microsoft.com/pricing/reserved-vm-instances/) в зоне аварийного восстановления, и операции по восстановлению сайта будут использовать их. Никаких дополнительных настроек не требуется.

## <a name="security"></a>Безопасность

### <a name="is-replication-data-sent-to-the-site-recovery-service"></a>Отправляются ли данные репликации в службу Site Recovery?

Нет, восстановление сайта не перехватывает реплицированные данные, и у него нет никакой информации о том, что работает на ваших ВМ. В службу Site Recovery передаются только метаданные, необходимые для управления репликацией и отработкой отказа.

Восстановление сайта ISO 27001:2013, 27018, HIPAA и DPA сертифицированы. Служба проходит оценку SOC2 и FedRAMP JAB.

### <a name="does-site-recovery-encrypt-replication"></a>Выполняет ли служба Site Recovery шифрование репликации?

Да, поддерживаются как шифрование в пути, так и [шифрование в Azure.](https://docs.microsoft.com/azure/storage/storage-service-encryption)
