# <a name="support-matrix-for-azure-vm-disaster-recovery-between-azure-regions"></a>Матрица поддержки для аварийного восстановления Azure VM между регионами Azure

В этой статье кратко излагаются поддержка и предпосылки для аварийного восстановления вс-мизаносцев Azure из одного региона Azure в другой с помощью службы [восстановления сайтов Azure.](site-recovery-overview.md)


## <a name="deployment-method-support"></a>Поддержка метода развертывания

**Развертывание** |  **Поддержка**
--- | ---
**Портал Azure** | Поддерживается.
**PowerShell** | Поддерживается. [Дополнительные сведения](azure-to-azure-powershell.md)
**REST API** | Поддерживается.
**Cli** | Сейчас не поддерживается


## <a name="resource-support"></a>Поддержка ресурсов

**Действие с ресурсом** | **Подробно**
--- | ---
**Перемещение хранилищ в группах ресурсов** | Не поддерживается
**Перемещение вычислительных ресурсов, ресурсов хранилищ и сетевых ресурсов между группами ресурсов** | Не поддерживается.<br/><br/> Если вы перемещаете виртуальную машину или связанные компоненты, такие как хранилище или сеть, после ее репликации, вам необходимо отключить репликацию и повторно включить ее для виртуальной машины.
**Репликация виртуальных машин Azure из одной подписки в другую для обеспечения аварийного восстановления** | Поддерживается в том же клиенте Azure Active Directory.
**Миграция виртуальных машин между регионами в рамках поддерживаемых географических кластеров (в пределах подписок и между ними)** | Поддерживается в том же клиенте Azure Active Directory.
**Перемещение виртуальных машин внутри региона** | Не поддерживается.

>[!NOTE]
>
> - Для **Бразилии на юге**, вы можете повторить и потерпеть неудачу в этих регионах: южная центральная часть США, Западный Центральный США, Восток США, Восток США 2, Западная ЧАСТЬ США, Западная США 2, и Северо-Центральный США.
> - Бразилия на юг может быть использована только в качестве исходного региона, из которого вМ могут реплицироваться с помощью восстановления сайта. Он не может выступать в качестве целевого региона. Это связано с проблемами задержки из-за географических расстояний. Обратите внимание, что если вы не более из Бразилии на юг в качестве источника региона к цели, failback в Бразилию на юг от целевого региона поддерживается.
> - Вы можете работать в регионах, к которым у вас есть соответствующий доступ.
> - Если область, в которой вы хотите создать хранилище, не отображалась, убедитесь, что ваша подписка имеет доступ к созданию ресурсов в этом регионе.
> - Если вы не видите область в географическом кластере при вменении репликации, убедитесь, что в подписке есть разрешения на создание вневых компьютеров в этом регионе.



## <a name="cache-storage"></a>Хранилище кэша

В следующей таблице приведены сведения о поддержке учетных записей хранения кэша, используемые Azure Site Recovery для репликации.

**Параметр** | **Поддержка** | **Подробно**
--- | --- | ---
Учетные записи хранения общего назначения версии 2 ("горячего" и "холодного" уровней доступа) | Поддерживается | Использование GPv2 не рекомендуется, поскольку транзакционные издержки для V2 значительно выше, чем счета хранения V1.
Хранилище уровня "Премиум" | Не поддерживается | Стандартные учетные записи хранения используются для хранения кэша, чтобы помочь оптимизировать затраты.
Брандмауэры службы хранилища Azure для виртуальных сетей  | Поддерживается | Если в целевой учетной записи хранения или в учетной записи хранения кэша включен брандмауэр, убедитесь, что задан параметр [Разрешить доверенным службам Майкрософт обходить этот брандмауэр](https://docs.microsoft.com/azure/storage/common/storage-network-security#exceptions).<br></br>Кроме того, убедитесь, что разрешен доступ по меньшей мере к одной подсети исходной виртуальной сети.
