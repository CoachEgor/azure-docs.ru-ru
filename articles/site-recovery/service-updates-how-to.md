# <a name="service-updates-in-site-recovery"></a>Обновления службы в восстановлении сайта

В этой статье содержится обзор обновлений [восстановления сайта Azure](site-recovery-overview.md) и описано, как обновить компоненты восстановления сайта.

Восстановление сайта регулярно публикует обновления служб. Обновления включают новые функции, улучшения поддержки, обновления компонентов и исправления ошибок. Для того, чтобы воспользоваться последними функциями и исправлениями, мы рекомендуем запускать последние версии компонентов восстановления сайта. 
 
 
## <a name="updates-support"></a>Поддержка обновлений

### <a name="support-statement-for-azure-site-recovery"></a>Заявление о поддержке для Azure Site Recovery

Мы рекомендуем всегда переходить на последние версии компонентов:

**С каждой новой версией 'N' выпущенного компонента восстановления сайта Azure все версии ниже 'N-4' считаются неподвлатыми.** 

> [!IMPORTANT]
> Официальная поддержка предназначена для обновления с версии N-4 > N-4 до N версии. Например, если вы работаете вы находитесь на N-6, вам нужно сначала обновить до N-4, а затем перейти на N.


### <a name="links-to-currently-supported-update-rollups"></a>Ссылки на поддерживаемые накопительные пакеты обновления

 Просмотрите последнее обновление свертывания (версия N) в [этой статье](site-recovery-whats-new.md). Помните, что восстановление сайта обеспечивает поддержку версий N-4.



## <a name="component-expiry"></a>Срок действия компонента

Восстановление сайта уведомляет вас о просроченных компонентах (или близится к истечению) по электронной почте (если вы подписались на уведомления по электронной почте) или на панели мониторинга хранилища на портале.

- Кроме того, при наличии обновлений в представлении инфраструктуры для вашего сценария на портале рядом с компонентом появляется **доступная кнопка обновления.** Эта кнопка перенаправляет вас на ссылку для загрузки последней версии компонента.
-  Уведомления панели мониторинга Vaults недоступны при репликации Hyper-V VM. 

Уведомления об электронных письмах отправляются следующим образом.

**Время** | **Частота**
--- | ---
60 дней до истечения срока действия компонента | Раз в две недели
Следующие 53 дня | Раз в неделю
за последние 7 дней; | один раз в день
После истечения срока действия | Раз в две недели


### <a name="upgrading-outside-official-support"></a>Модернизация внешней официальной поддержки

Если разница между компонентной версией и последней версией релиза больше четырех, это считается неподвлатым. В этом случае обновление следующим образом: 

1. Обновление установленного в настоящее время компонента для текущей версии плюс четыре. Например, если версия, если 9.16, то обновить до 9.20.
2. Затем переключите на следующую совместимую версию. Так в нашем примере, после обновления 9.16 до 9.20, обновление до 9.24. 

Следуйте тому же процессу для всех соответствующих компонентов.

### <a name="support-for-latest-operating-systemskernels"></a>Поддержка новейших операционных систем/ядер

> [!NOTE]
> Если у вас запланировано окно обслуживания, и в него включена перезагрузка, мы рекомендуем сначала обновить компоненты восстановления сайта, а затем продолжить остальные запланированные действия в окне обслуживания.

1. Перед обновлением операционных версий операционной системы/ядра убедитесь, поддерживается ли целевая версия Восстановление сайта. 

    - Поддержка [Azure VM.](azure-to-azure-support-matrix.md#replicated-machine-operating-systems)
    - [Поддержка VMware/физическая поддержка сервера](vmware-physical-azure-support-matrix.md#replicated-machines)
    - [Поддержка Hyper-V.](hyper-v-azure-support-matrix.md#replicated-vms)
2. Просмотрите [доступные обновления,](site-recovery-whats-new.md) чтобы узнать, что вы хотите обновить.
3. Обновление до последней версии восстановления сайта.
4. Обновление операционной системы/ядра до требуемых версий.
5. Перезагрузите систему.


Этот процесс гарантирует, что операционная система/ядро машины будет обновлена до последней версии, и что последние изменения восстановления сайта, необходимые для поддержки новой версии, будут загружены на машину.

## <a name="azure-vm-disaster-recovery-to-azure"></a>Аварийное восстановление виртуальной машины Azure в Azure

В этом случае мы настоятельно рекомендуем [включить автоматические обновления.](azure-to-azure-autoupdate.md) Вы можете разрешить восстановлению сайта управлять обновлениями следующим образом:

- Во время процесса репликации включить.
- Устанавливая настройки обновления расширения внутри хранилища.

Если вы хотите управлять обновлениями вручную, сделайте следующее:

1. В хранилище > **реплицированных элементов,** нажмите это уведомление в верхней части экрана: 
    
    **Доступно обновление нового агента репликации восстановления сайта. Нажмите, чтобы установить ->**

4. Выберите vMs, для которых вы хотите применить обновление, а затем нажмите **OK**.


## <a name="vmware-vmphysical-server-disaster-recovery-to-azure"></a>VMware VM/физическое восстановление аварийного сервера в Azure

1. На основе текущей версии и [оператора поддержки](#support-statement-for-azure-site-recovery)установите обновление сначала на сервер конфигурации, используя [эти инструкции.](vmware-azure-deploy-configuration-server.md#upgrade-the-configuration-server) 
2. Если у вас есть серверы масштабирования процесса, обновите их следующим образом, используя [эти инструкции.](vmware-azure-manage-process-server.md#upgrade-a-process-server)
3. Чтобы обновить агент Mobility на каждой защищенной машине, обратитесь к [этой](vmware-physical-manage-mobility-service.md#update-mobility-service-from-azure-portal) статье.

### <a name="reboot-after-mobility-service-upgrade"></a>Перезагрузка после обновления службы мобильности

Перезагрузка рекомендуется после каждого обновления службы Мобильность, чтобы убедиться, что все последние изменения загружаются на исходную машину.

Перезагрузка не является обязательной, если разница между версией агента во время последней перезагрузки и текущей версией больше, чем четыре.

Пример в таблице показывает, как это работает.

|**Версия агента (последняя перезагрузка)** | **Обновление до** | **Обязательная перезагрузка?**|
|---------|---------|---------|
|9.16 |  9.18 | Не обязательно|
|9.16 | 9.19 | Не обязательно|
| 9.16 | 9.20 | Не обязательно
 | 9.16 | 9.21 | Mandatory.<br/><br/> Обновление до 9.20, затем перезагрузка перед обновлением до 9.21.

## <a name="hyper-v-vm-disaster-recovery-to-azure"></a>Аварийное восстановление виртуальной машины Hyper-V в Azure

### <a name="between-a-hyper-v-site-and-azure"></a>Между сайтом Hyper-V и Azure

1. Загрузите обновление для поставщика восстановления сайта Microsoft Azure.
2. Установите поставщика на каждый сервер Hyper-V, зарегистрированный в Восстановлении Сайта. Если вы работаете кластер, обновите все кластерные узлы.


## <a name="between-an-on-premises-vmm-site-and-azure"></a>Между локальным сайтом VMM и Azure
1. Загрузите обновление для поставщика восстановления сайта Microsoft Azure.
2. Установите поставщика на сервер VMM. Если VMM развернут в кластере, установите Поставщика на все кластерные узлы.
3. Установите новейший агент служб восстановления Microsoft Azure на все узлы Hyper-V или кластерные узлы.


## <a name="between-two-on-premises-vmm-sites"></a>Между двумя локальными сайтами VMM
1. Загрузите последнее обновление для поставщика восстановления сайтов Microsoft Azure.
2. Установите последний провайдер на сервере VMM, управляющем сайтом вторичного восстановления. Если VMM развернут в кластере, установите Поставщика на все кластерные узлы.
3. После обновления сайта восстановления установите поставщика на сервер VMM, который управляет основным сайтом.
