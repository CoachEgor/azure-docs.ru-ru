# <a name="fail-back-vmware-vms-to-on-premises-site"></a>Fail назад VMware VMs на месте на месте

В этой статье описывается, как вывести из строя VMs Azure на сайт, после [отказа](site-recovery-failover.md) от новких ВМ в Azure с [помощью восстановления сайта Azure.](site-recovery-overview.md) После сбоя в предварительном помещении можно включить репликацию, чтобы предприимчивые ВМ начали реплицироваться в Azure.

## <a name="before-you-start"></a>Перед началом работы

1. Узнайте больше о [отказов VMware.](failover-failback-overview.md#vmwarephysical-reprotectionfailback) 
2. Убедитесь, что вы рассмотрели и завершили шаги для [подготовки к отказу,](vmware-azure-prepare-failback.md)и что все необходимые компоненты развернуты. Компоненты включают сервер процесса в Azure, внутренний основной целевой сервер и VPN-соединение от сайта к сайту (или приватное пиринг ExpressRoute) для сбоя.
3. Убедитесь, что вы выполнили [требования](vmware-azure-reprotect.md#before-you-begin) для повторной защиты и отказов, и что вы [включили резащиту](vmware-azure-reprotect.md#enable-reprotection) VMs Azure, чтобы они реплицироваться из Azure в сайт. VMs должны быть в реплицируемом состоянии является заказ на неудачу обратно.




## <a name="run-a-failover-to-fail-back"></a>Выполнить неудачу, чтобы потерпеть неудачу назад

1. Убедитесь, что VMs Azure повторно защищены и реплицируются на сайте. 
    - VM необходимо по крайней мере одну точку восстановления для того, чтобы выйти из строя.
    - Если вы не сможете вернуть план восстановления, то все машины в плане должны иметь по крайней мере одну точку восстановления.
2. В хранилище > **Реплицированные элементы,** выберите VM. Справа щелкните VM > **незапланированный Failover**.
3. На странице **Подтверждение отработки отказа** должно быть указано направление отработки отказа "Из Azure".
4. Выберите точку восстановления, которую вы хотите использовать для отработки отказа.
    - Мы рекомендуем использовать **последнюю** точку восстановления. Последовательная точка приложения отстает от последней точки времени и приводит к потере некоторых данных.
    - **Последний** является аварийно-последовательной точки восстановления.
    - С **последним**, VM не удается к своей последней доступной точке времени. Если у вас есть группа репликации для согласованности мульти-VM в плане восстановления, каждый VM в группе не доводит свою независимую последнюю точку времени.
    - Если вы используете точку восстановления приложения, каждый VM не возвращается к своей последней доступной точке. Если в плане восстановления есть группа репликации, каждая группа восстанавливается в общей доступной точке восстановления.
5. Failover начинается. Восстановление сайта выключает VMs Azure.
6. После завершения сбоя проверьте, что все работает как ожидалось. Убедитесь, что Чаты закрыты. 
7. Со всем проверенным, правой кнопкой мыши VM > **Commit**, чтобы закончить процесс неудачи. Commit удаляет неудавшийся VM Azure. 

> [!NOTE]
> Для Windows VMs восстановление сайта отсутсвие vMware во время сбоя. Во время отказа от Windows VM инструменты VMware снова включаются. 




## <a name="reprotect-from-on-premises-to-azure"></a>Повторное включение защиты из локальной среды в Azure

После фиксации сбоя высоцдляются, vMs Azure удаляются. VM вернулся в месте, но он не защищен. Чтобы снова начать репликацию ВМ в Azure:

1. В хранилище > **Реплицированные элементы,** выберите не удалось обратно VMs, а затем выберите **Re-Protect.**
2. Укажите сервер процесса, используемый для отправки данных обратно в Azure.
3. Нажмите кнопку **ОК**, чтобы запустить задание повторной защиты.

> [!NOTE]
> После запуска внештатного VM агенту требуется до 15 минут, чтобы зарегистрироваться обратно на сервер конфигурации. Если включить повторную защиту в течение этого времени, произойдет сбой и на экране отобразится сообщение о том, что агент не установлен. Если это происходит, подождите несколько минут и защитите.

## <a name="next-steps"></a>Дальнейшие действия

После завершения задания повторной защиты предплещный VM реплицируется в Azure. При необходимости можно [запустить еще один сбой](site-recovery-failover.md) в Azure.
