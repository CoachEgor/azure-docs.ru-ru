# <a name="fail-back-vmware-vms-to-on-premises-site"></a>Восстановить при сбое виртуальные машины VMware на локальном сайте

В этой статье описывается, как восстановить работоспособность виртуальных машин Azure на локальном сайте после [отработки отказа](site-recovery-failover.md) локальных виртуальных машин в Azure с [Azure Site Recovery](site-recovery-overview.md). После восстановления размещения в локальной среде вы включите репликацию, чтобы локальные виртуальные машины начали реплицироваться в Azure.

## <a name="before-you-start"></a>Перед началом работы

1. Сведения о [восстановлении размещения VMware](failover-failback-overview.md#vmwarephysical-reprotectionfailback). 
2. Убедитесь, что вы проверили и выполнили шаги для [подготовки к восстановлению размещения](vmware-azure-prepare-failback.md)и развертывания всех необходимых компонентов. Компоненты включают в себя сервер обработки в Azure, локальный главный целевой сервер и подключение VPN типа "сеть — сеть" (или частный пиринг ExpressRoute) для восстановления размещения.
3. Убедитесь, что вы выполнили [требования](vmware-azure-reprotect.md#before-you-begin) к повторной защите и восстановлению размещения, а также [включили повторное включение защиты](vmware-azure-reprotect.md#enable-reprotection) виртуальных машин Azure, чтобы они выполняли репликацию из Azure на локальный сайт. Виртуальные машины должны находиться в реплицируемом состоянии, чтобы восстановить их.




## <a name="run-a-failover-to-fail-back"></a>Запуск отработки отказа для обратного переключения

1. Убедитесь, что виртуальные машины Azure повторно защищены и реплицируются на локальный сайт.
    - Виртуальной машине требуется по крайней мере одна точка восстановления, чтобы выполнить восстановление после сбоя.
    - При восстановлении после сбоя плана восстановления все компьютеры в плане должны иметь по крайней мере одну точку восстановления.
2. В хранилище > **реплицированные элементы**выберите виртуальную машину. Щелкните правой кнопкой мыши виртуальную машину > **внеплановая отработка отказа**.
3. На странице **Подтверждение отработки отказа** должно быть указано направление отработки отказа "Из Azure".
4. Выберите точку восстановления, которую вы хотите использовать для отработки отказа.
    - Рекомендуется использовать **последнюю** точку восстановления. Точка, соответствующая приложению, находится за последней точкой во времени и приводит к потери данных.
    - **Последняя** — это отказоустойчивая точка восстановления.
    - В **последнем**случае виртуальная машина переходит на последнюю доступную точку во времени. При наличии группы репликации для согласованности нескольких виртуальных машин в плане восстановления каждая виртуальная машина в группе переходит на независимый последний момент времени.
    - При использовании точки восстановления, соответствующей приложению, каждая виртуальная машина возвращается в последнюю доступную точку. Если план восстановления содержит группу репликации, каждая группа восстанавливает общую доступную точку восстановления.
5. Начинается отработка отказа. Site Recovery завершает работу виртуальных машин Azure.
6. После завершения отработки отказа проверьте, что все работает правильно. Убедитесь, что работа виртуальных машин Azure завершена. 
7. Если все проверено, щелкните правой кнопкой мыши виртуальную машину > **фиксация**, чтобы завершить процесс отработки отказа. Commit удаляет виртуальную машину Azure, для которой выполнена отработка отказа. 

> [!NOTE]
> Для виртуальных машин Windows Site Recovery отключает средства VMware во время отработки отказа. При восстановлении размещения виртуальной машины Windows средства VMware снова активируются. 




## <a name="reprotect-from-on-premises-to-azure"></a>Повторное включение защиты из локальной среды в Azure

После фиксации восстановления размещения виртуальные машины Azure удаляются. Виртуальная машина возвращается на локальный сайт, но она не защищена. Чтобы снова приступить к репликации виртуальных машин в Azure, выполните следующие действия.

1. В хранилище > **реплицированные элементы**, выберите неисправные виртуальные машины, а затем щелкните **повторно защитить**.
2. Укажите сервер обработки, используемый для отправки данных обратно в Azure.
3. Нажмите кнопку **ОК**, чтобы запустить задание повторной защиты.

> [!NOTE]
> После запуска локальной виртуальной машины для регистрации агента на сервере конфигурации потребуется до 15 минут. Если включить повторную защиту в течение этого времени, произойдет сбой и на экране отобразится сообщение о том, что агент не установлен. В таком случае подождите несколько минут и повторно включите защиту.

## <a name="next-steps"></a>Дальнейшие шаги

После завершения задания повторной защиты локальная виртуальная машина реплицируется в Azure. При необходимости можно [запустить другую отработку отказа](site-recovery-failover.md) в Azure.
