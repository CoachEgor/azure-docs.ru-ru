# <a name="monitor-the-process-server"></a>Мониторинг сервера обработки

В этой статье описаны способы отслеживания [Site Recovery](site-recovery-overview.md) сервер обработки.

- Сервер обработки используется при настройке аварийного восстановления локальных виртуальных машин VMware и физических серверов в Azure.
- По умолчанию сервер обработки выполняется на сервере конфигурации. Он устанавливается по умолчанию при развертывании сервера конфигурации.
- При необходимости для масштабирования и дескриптор большее количество реплицированные машины и большие объемы трафика репликации, можно развернуть серверы дополнительные, горизонтальное масштабирование обработки.

[Дополнительные сведения](vmware-physical-azure-config-process-server-overview.md) о роли и развертывания серверов обработки.

## <a name="monitoring-overview"></a>Обзор мониторинга

Так как на сервере обработки включено множество ролей, особенно в реплицированные данные кэширования, сжатия и передачи в Azure, важно наблюдать за состоянием сервера обработки на постоянной основе.

Существует множество ситуаций, которые часто влияют на производительность процесса сервера. Проблемы, влияющие на производительность будет иметь каскадный эффект на работоспособность виртуальной Машины, в конечном итоге принудительной отправки на сервер обработки и его реплицированные машины в критическое состояние. В частности:

- Слишком большого числа виртуальных машин использовать сервер обработки, приближается или превышение рекомендуемые ограничения.
- Виртуальные машины, с помощью сервера обработки имеют большая скорость изменения данных.
- Пропускная способность сети между виртуальными машинами и сервер обработки недостаточно для отправки данных репликации на сервер обработки.
- Пропускная способность сети между сервером обработки и Azure не позволяет отправлять данные репликации с сервера обработки в Azure.

Все эти проблемы могут влиять на целевую точку восстановления (RPO) виртуальных машин. 

**Почему?** Так как для создания точки восстановления для виртуальной Машины требуется все диски на виртуальной Машине иметь общую точку. Если один диск содержит большая скорость изменения данных, репликация выполняется медленно или сервер обработки не оптимален, это влияет на эффективность создания точек восстановления.

## <a name="monitor-proactively"></a>Упреждающий мониторинг

Чтобы избежать проблем с сервером обработки, очень важно:

- Понять определенные требования для сервера обработки с помощью [емкости и руководство по сквозному масштабированию](site-recovery-plan-capacity-vmware.md#capacity-considerations)и убедитесь, что развернуты серверы обработки и в соответствии с рекомендациями.
- Мониторинг оповещений и устранение неполадок при их возникновении, для обеспечения эффективной работы серверов процесса.

## <a name="monitor-process-server-health"></a>Наблюдение за исправностью сервера обработки

Вы можете отслеживать состояние работоспособности серверов процесс следующим образом: 

1. Чтобы отслеживать работоспособность репликации и состояние реплицированный компьютер и его сервер обработки, в хранилище > **реплицированные элементы**, щелкните машину, вы хотите отслеживать.
2. В **работоспособность репликации**, можно отслеживать состояние работоспособности виртуальной Машины. Выберите состояние для детализации углублением сведения об ошибке.

    ![Работоспособность сервера обработки в панели мониторинга виртуальной Машины](./media/vmware-physical-azure-monitor-process-server/vm-ps-health.png)

4. В **работоспособность сервера процесса**, можно отслеживать состояние процесса сервера. Детализировать подробности.

    ![Сведения о сервере обработки в панели мониторинга виртуальной Машины](./media/vmware-physical-azure-monitor-process-server/ps-summary.png)

5. Также работоспособности можно отслеживать с помощью графического представления на странице виртуальной Машины.
    - Сервер обработки масштабирования будет выделенные оранжевым цветом, при наличии предупреждений, связанных с ним, а красным, если он имеет все критические проблемы. 
    - Если сервер обработки работает в стандартном развертывании на сервере конфигурации, сервер конфигурации будет выделена соответствующим образом.
    - Чтобы перейти вниз, щелкните сервер конфигурации или сервер обработки. Обратите внимание на то, все ошибки и рекомендации по исправлению.

Вы также можете отслеживать серверы в хранилище в разделе обработки **инфраструктура Site Recovery**. В **управлять инфраструктурой Site Recovery**, нажмите кнопку **серверы конфигурации**. Выберите сервер конфигурации, связанные с сервером обработки и переход вниз в сведения о сервере обработки.
