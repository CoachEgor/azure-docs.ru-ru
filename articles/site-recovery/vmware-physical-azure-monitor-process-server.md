# <a name="monitor-the-process-server"></a>Мониторинг сервера обработки

В этой статье описывается наблюдение за сервером обработки [Site Recovery](site-recovery-overview.md) .

- Сервер обработки используется при настройке аварийного восстановления локальных виртуальных машин VMware и физических серверов в Azure.
- По умолчанию сервер обработки запускается на сервере конфигурации. Он устанавливается по умолчанию при развертывании сервера конфигурации.
- Кроме того, для масштабирования и обработки большого количества реплицируемых компьютеров и более высоких томов трафика репликации можно развернуть дополнительные серверы обработки масштабирования.

Дополнительные [сведения](vmware-physical-azure-config-process-server-overview.md) о роли и развертывании серверов обработки.

## <a name="monitoring-overview"></a>Общие сведения о мониторинге

Поскольку сервер обработки имеет настолько много ролей, особенно в реплицированных кэшированных данных, сжатии и передаче в Azure, важно отслеживать работоспособность сервера обработки на постоянной основе.

Существует несколько ситуаций, которые обычно влияют на производительность сервера обработки. Проблемы, влияющие на производительность, будут иметь каскадное влияние на работоспособность виртуальной машины, в конечном итоге отправляя как сервер обработки, так и его реплицированные компьютеры в критическое состояние. Существуют следующие ситуации.

- Большое число виртуальных машин использует сервер обработки, который приближается к рекомендуемому или превышает его.
- Для виртуальных машин, использующих сервер обработки, используется высокий уровень интенсивности обновлений.
- Пропускная способность сети между виртуальными машинами и сервером обработки недостаточно для передачи данных репликации на сервер обработки.
- Пропускная способность сети между сервером обработки и Azure недостаточно для передачи данных репликации с сервера обработки в Azure.

Все эти проблемы могут повлиять на целевую точку восстановления (RPO) виртуальных машин. 

**Важно?** Так как для создания точки восстановления для виртуальной машины требуется общая точка на всех дисках виртуальной машины. Если один диск имеет высокий уровень интенсивности обновлений, репликация замедлена или сервер обработки не является оптимальным, это влияет на эффективность создания точек восстановления.

## <a name="monitor-proactively"></a>Упреждающий мониторинг

Чтобы избежать проблем с сервером обработки, важно:

- Ознакомьтесь с конкретными требованиями к серверам обработки, используя [рекомендации по емкости и выбору размеров](site-recovery-plan-capacity-vmware.md#capacity-considerations), и убедитесь, что серверы обработки развернуты и выполняются в соответствии с рекомендациями.
- Мониторинг предупреждений и устранение неполадок по мере их возникновения для обеспечения эффективной работы серверов обработки.


## <a name="process-server-alerts"></a>Предупреждения сервера обработки

Сервер обработки создает ряд предупреждений о работоспособности, приведенных в следующей таблице.

**Тип оповещения** | **Сведения**
--- | ---
![Работоспособна][green] | Сервер обработки подключен и является работоспособным.
![Предупреждение][yellow] | Загрузка ЦП > 80% за последние 15 минут
![Предупреждение][yellow] | Использование памяти > 80% за последние 15 минут
![Предупреждение][yellow] | Свободное место в папке кэша < 30% за последние 15 минут
![Предупреждение][yellow] | Службы сервера обработки не работают за последние 15 минут
![Critical][red] | Загрузка ЦП > 95% за последние 15 минут
![Critical][red] | Использование памяти > 95% за последние 15 минут
![Critical][red] | Свободное место в папке кэша < 25% за последние 15 минут
![Critical][red] | Нет пульса от сервера обработки в течение 15 минут.

![Ключ таблицы](./media/vmware-physical-azure-monitor-process-server/table-key.png)

> [!NOTE]
> Общее состояние работоспособности сервера обработки зависит от наихудшего созданного предупреждения.



## <a name="monitor-process-server-health"></a>Мониторинг работоспособности сервера обработки

Состояние работоспособности серверов обработки можно отслеживать следующим образом. 

1. Чтобы отслеживать работоспособность репликации и состояние реплицированного компьютера, а также его сервер обработки, в хранилище > реплицированные **элементы**, щелкните компьютер, который требуется отслеживать.
2. В области **работоспособность репликации**можно отслеживать состояние работоспособности виртуальной машины. Щелкните состояние, чтобы детализировать подробные сведения об ошибке.

    ![Работоспособность сервера обработки на панели мониторинга виртуальной машины](./media/vmware-physical-azure-monitor-process-server/vm-ps-health.png)

4. В области **работоспособность сервера обработки**можно отслеживать состояние сервера обработки. Детализация для получения подробных сведений.

    ![Сведения о сервере обработки на панели мониторинга виртуальной машины](./media/vmware-physical-azure-monitor-process-server/ps-summary.png)

5. Работоспособность также можно отслеживать с помощью графического представления на странице виртуальной машины.
    - Сервер обработки масштабирования будет выделяться оранжевым цветом, если с ним связаны предупреждения, и красным, если имеются какие-либо критические проблемы. 
    - Если сервер обработки выполняется в развертывании по умолчанию на сервере конфигурации, сервер конфигурации будет выделен соответствующим образом.
    - Чтобы выполнить детализацию, щелкните сервер конфигурации или сервер обработки. Обратите внимание на все проблемы и рекомендации по исправлению.

Вы также можете отслеживать серверы обработки в хранилище в разделе **Site Recovery Infrastructure (инфраструктура**). В окне **Управление инфраструктурой Site Recovery**щелкните **серверы конфигурации**. Выберите сервер конфигурации, связанный с сервером обработки, и перейдите к сведениям о сервере обработки.
