# <a name="deprecation-of-disaster-recovery-between-customer-managed-sites-with-vmm-using-azure-site-recovery"></a>Изнурительный процесс аварийного восстановления между объектами, управляемыми клиентами (с VMM) с использованием Azure Site Recovery

В этой статье описывается предстоящий план амортизации, соответствующие последствия и альтернативные варианты, доступные для клиентов для следующего сценария:

DR между клиентами сайтов, управляемых System Center Виртуальный менеджер машины (SCVMM) с использованием восстановления сайта

> [!IMPORTANT]
> Клиентам рекомендуется принять меры по исправлению положения в ближайшее время, чтобы избежать каких-либо нарушений в их среде. 

## <a name="what-changes-should-you-expect"></a>Какие изменения следует ожидать?

- Начиная с марта 2020 года, вы будете получать уведомления портала Azure & сообщения электронной почты с предстоящим амортизацию репликации hyper-V V. Амортизм запланирован на март 2023 года.

- Если у вас есть существующая конфигурация, настройка не повлияет.

- Как только сценарии унижаются, если клиент не следует альтернативным подходам, существующие репликации могут быть нарушены. Клиенты не смогут просматривать, управлять или выполнять любые операции, связанные с DR, через опыт восстановления сайтов Azure на портале Azure.
 
## <a name="alternatives"></a>Альтернативные варианты 

Ниже приведены альтернативы, которые клиент может выбрать, чтобы гарантировать, что их стратегия DR не влияет, как только сценарий утихнет. 

- Вариант 1 (Рекомендуется): Выберите [начать использовать Azure в качестве цели DR.](hyper-v-vmm-azure-tutorial.md)


- Вариант 2: Выберите продолжение репликации сайта с помощью базового [решения Hyper-V Replica,](https://docs.microsoft.com/windows-server/virtualization/hyper-v/manage/set-up-hyper-v-replica)но вы не сможете управлять конфигурациями DR с помощью восстановления сайта Azure на портале Azure. 


## <a name="remediation-steps"></a>Шаги восстановления

Если вы решили пойти с вариантом 1, пожалуйста, выполните следующие шаги:

1. [Отключить защиту всех виртуальных машин, связанных с VMMs.](site-recovery-manage-registration-and-protection.md#disable-protection-for-a-hyper-v-virtual-machine-replicating-to-secondary-vmm-server-using-the-system-center-vmm-to-vmm-scenario) Используйте **репликацию "Отключить" и удалите** опцию или запустите упомянутые скрипты, чтобы обеспечить очистку параметров репликации в помещениях. 

2. [Отменить регистрацию всех серверов VMM](site-recovery-manage-registration-and-protection.md#unregister-a-vmm-server) из конфигурации репликации от сайта к сайту.

3. [Подготовьте ресурсы Azure](tutorial-prepare-azure-for-hyperv.md) для включения репликации виз.
4. [Подготовка локальных серверов Hyper-V](hyper-v-prepare-on-premises-tutorial.md)
5. [Настройка репликации для vMs в облаке VMM](hyper-v-vmm-azure-tutorial.md)
6. Необязательный, но [рекомендуемый: Выполнить DR сверло](tutorial-dr-drill-azure.md)

Если вы решили пойти с вариантом 2 использования Hyper-V реплики, выполнить следующие шаги:

1. В **защищенных элементах** > **Реплицированные элементы,** правой кнопкой мыши машина > отключить **репликацию.**
2. В **отстраните репликацию,** выберите **Удалить**.

    Он позволяет удалить реплицированный элемент из Azure Site Recovery (выставление счетов останавливается). Конфигурация репликации на локальной виртуальной машине **не будет** очищена.
