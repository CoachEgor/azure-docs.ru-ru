# <a name="frequently-asked-questions-about-azure-container-instances"></a>Часто задаваемые вопросы о экземпляры контейнеров Azure

В этой статье рассматриваются часто задаваемые вопросы о экземпляры контейнеров Azure.

## <a name="deployment"></a>Развертывание

### <a name="how-large-can-my-container-image-be"></a>Размер может быть образе контейнера

Максимальный размер для создания образа развертывания контейнера в экземплярах контейнера Azure составляет 15 ГБ. Можно развертывать большие изображения в зависимости от точного доступности в данный момент развертывается, но это не гарантируется.

Размер образа контейнера влияет на время, необходимое для развертывания, и обычно будут храниться образы контейнера как можно меньшего размера.

### <a name="how-can-i-speed-up-the-deployment-of-my-container"></a>Как можно ускорить развертывание Мой контейнер?

Так как одна из основных определителями времени развертывания размер изображения, ищите способы уменьшения размера. Удалите слои, вам не нужна, или уменьшить размер слоев в образе (комплектации светлее базовый образ операционной системы). Например если вы используете контейнеры Linux, рассмотрите возможность использования Alpine как базового образа, а не всего сервера Ubuntu. Аналогичным образом для контейнеров Windows, используйте базовый образ Nano Server, если это возможно. 

Следует также проверить список предварительно кэшированного изображений в образах контейнеров Azure, доступных через [вывод списка кэшированных образов](/rest/api/container-instances/listcachedimages) API. Возможно, вы сможете исходящего слой изображения для одного из предварительно кэшированного изображения переключения. 

Показать больше [подробное руководство](container-instances-troubleshooting.md#container-takes-a-long-time-to-start) сокращение время запуска контейнера.

### <a name="what-windows-base-os-images-are-supported"></a>Какие базовые образы ОС Windows поддерживаются?

#### <a name="windows-server-2016-base-images"></a>Базовые образы Windows Server 2016

* [Nano Server](https://hub.docker.com/_/microsoft-windows-nanoserver): `10.0.14393.x`, `sac2016`
* [Windows Server Core](https://hub.docker.com/_/microsoft-windows-servercore): `ltsc2016`,  `10.0.14393.x`

> [!NOTE]
> Образы Windows, в зависимости от выпуска Semi-Annual Channel 1709 или 1803 не поддерживаются.

#### <a name="windows-server-2019-and-client-base-images-preview"></a>Windows Server 2019 и клиента базовые образы (Предварительная версия)

* [Nano Server](https://hub.docker.com/_/microsoft-windows-nanoserver): `1809`, `10.0.17763.x`
* [Windows Server Core](https://hub.docker.com/_/microsoft-windows-servercore): `ltsc2019`, `1809`, `10.0.17763.x`
* [Windows](https://hub.docker.com/_/microsoft-windows): `1809`, `10.0.17763.x` 

### <a name="what-net-or-net-core-image-layer-should-i-use-in-my-container"></a>Какие слой образа .NET или .NET Core следует использовать в своем контейнере? 

Используйте наименьшее образ, который удовлетворяет вашим требованиям. Для Linux, можно использовать *alpine среды выполнения* образа .NET Core, который поддерживался с момента выпуска .NET Core 2.1. Для Windows, если вы используете полную версию .NET Framework, то необходимо использовать образ Windows Server Core (образ только для среды выполнения, такие как *4.7.2-windowsservercore-ltsc2016*). Только для среды выполнения изображения имеют меньший размер, но не поддерживают рабочие нагрузки, требующие пакета SDK для .NET.

## <a name="availability-and-quotas"></a>Доступность и квоты

### <a name="how-many-cores-and-memory-should-i-allocate-for-my-containers-or-the-container-group"></a>Сколько ядер и памяти нужно выделять для моей контейнеров или группе контейнеров?

Это зависит от рабочей нагрузки. Начните с малого и тестирования производительности, чтобы увидеть, как сделать контейнеры. [Наблюдение за использованием ресурсов ЦП и памяти](container-instances-monitor.md), а затем добавьте ядра или объем памяти в зависимости от типа процессов, развертываемых в контейнере. 

Не забудьте также проверить [доступность ресурсов](container-instances-region-availability.md#availability---general) для региона, при развертывании в верхней границы на число ядер ЦП и памяти, доступной в группе контейнеров. 

### <a name="what-underlying-infrastructure-does-aci-run-on"></a>Какие базовой инфраструктуры выполняется ACI?

Экземпляры контейнеров Azure позволит понять, бессерверные службы контейнеров по запросу, поэтому мы хотим, чтобы сосредоточиться на разработке контейнеров вы, а не беспокоиться насчет инфраструктуры! Для тех, кто интересно или wanting для сравнений на производительность, ACI запускает на наборы виртуальных машин Azure из различных номеров SKU, главным образом из F и серия d. Ожидается, что изменение в будущем, пока идет разработка и оптимизация службы. 

### <a name="i-want-to-deploy-thousand-of-cores-on-aci---can-i-get-my-quota-increased"></a>Мне нужно развернуть тысяч ядер в ACI - получить Мои увеличить квоту?
 
Да (иногда). См. в разделе [квоты и ограничения](container-instances-quotas.md) статьи в параметре текущие квоты и какие ограничения можно увеличить по запросу.

### <a name="can-i-deploy-with-more-than-4-cores-and-16-gb-of-ram"></a>Можно ли развернуть более чем в 4-ядерный процессор и 16 ГБ ОЗУ

Пока нет. В настоящее время это максимальные ограничения для группы контейнеров. С определенными требованиями или запросы в службу поддержки Azure. 

### <a name="when-will-aci-be-in-a-specific-region"></a>Когда будет ACI в определенном регионе?

Публикуется текущей доступности региона [здесь](container-instances-region-availability.md#availability---general), и актуальные сведения доступен через [возможности списка](/rest/api/container-instances/listcapabilities) API. Если у вас есть потребность в определенном регионе, обратитесь в службу поддержки Azure.

## <a name="features-and-scenarios"></a>Функции и сценарии

### <a name="how-do-i-scale-a-container-group"></a>Как масштабировать группу контейнеров?

В настоящее время масштабирование недоступно для контейнеров "или" группы контейнеров. Если вам нужно запустить несколько экземпляров, используйте наш API для автоматизации и создать дополнительные запросы на создание группы контейнера в службу. 

### <a name="what-features-are-available-to-instances-running-in-a-custom-vnet"></a>Какие функции доступны для экземпляров, работающих в виртуальной сети пользовательские?

Развертывание группы контейнеров в виртуальной сети Azure по своему усмотрению и делегировать частные IP-адреса в контейнере группы маршрутизации трафика в виртуальной сети между ресурсами Azure. Развертывание группы контейнеров в виртуальной сети в данный момент в предварительной версии, и некоторые аспекты этой функции могут измениться до выпуска общедоступной версии (GA). См. в разделе [ограничения предварительной версии](container-instances-vnet.md#preview-limitations) обновленные сведения.

## <a name="pricing"></a>Цены

### <a name="when-does-the-meter-start-running"></a>Когда счетчик начинается выполнение?

Длительность использования группы контейнеров рассчитывается с момента мы начнем, чтобы извлечь образ свой первый контейнер (для нового развертывания) или группы контейнеров перезапускается (если они уже развернуты), до остановки группы контейнеров. Подробнее см. в [цены на экземпляры контейнеров](https://azure.microsoft.com/pricing/details/container-instances/).

### <a name="do-i-stop-being-charged-when-my-containers-are-stopped"></a>Остановить, и плата взимается при остановке Мои контейнеры?

Счетчики останавливаться после остановки отделу весь контейнер. До тех пор, пока выполняется контейнер в вашей группе контейнеров, мы проводим ресурсы, в случае, если вы хотите снова запускаются контейнеры.
