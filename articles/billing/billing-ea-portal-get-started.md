# <a name="get-started-with-the-azure-ea-portal"></a>Начало работы с порталом Azure EA

Эта статья поможет прямым и косвенным клиентам Azure EA получить базовые сведения о [портале Azure EA](https://ea.azure.com) для начала работы:

- структура портала Azure EA;
- роли, применяемые на портале Azure EA;
- процедура создания подписок;
- анализ затрат на портале Azure EA и на портале Azure.

## <a name="azure-ea-portal-hierarchy"></a>Иерархия портала Azure EA

Иерархия портала Azure EA включает следующие компоненты:

**Портал Microsoft Azure EA**. Портал Azure EA предназначен для управления через Интернет и поможет вам контролировать затраты на службы Azure EA. Здесь вы можете создать иерархию Azure EA, назначив отделы, учетные записи и подписки. Здесь же вы можете выверить затраты на использование служб, скачать отчеты об использовании и просмотреть прайс-листы. Вы также создаете здесь ключи API для регистрации.

**Отделы**. Вы можете создать отделы, чтобы разделить затраты на логические группы, а затем задать бюджет или квоту для каждого отдела.

**Учетные записи**. На портале Azure EA учетные записи выполняют роль организационных подразделений и используются для управления подписками. Также они применяются для создания отчетов.

**Подписки**. На портале Azure EA подписки являются наименьшей организационной единицей. По сути это контейнеры для служб Azure, которыми управляет администратор служб.

## <a name="enterprise-user-roles"></a>Роли корпоративных пользователей

Для администрирования служб Azure, относящихся к вашему соглашению о регистрации, можно использовать пять ролей администраторов предприятия с четким разделением обязанностей:

- Администратор предприятия
- Администратор отдела
- Владелец учетной записи
- Администратор службы
- Контактное лицо для уведомлений

Роли используются для выполнения задач на двух разных порталах Microsoft Azure. Портал Azure EA (https://ea.azure.com) используется для управления выставлением счетов и затратами. Портал Azure (https://portal.azure.com) позволяет управлять службами Azure.

Роли пользователей связываются с учетной записью пользователя. Для проверки подлинности каждый пользователь должен иметь действительную рабочую, учебную учетную запись или учетную запись Майкрософт. Следите за тем, чтобы каждая учетная запись была связана с активно отслеживаемым адресом электронной почты. На эти адреса электронной почты отправляются уведомления об учетной записи.

При настройке пользователей вы можете назначить роли администратора предприятия нескольким рабочим, учебным учетным записям или учетным записям Майкрософт. Но роли владельца учетной записи можно назначить лишь одну рабочую, учебную учетную запись или учетную запись Майкрософт. Кроме того, одна рабочая, учебная учетная запись или учетная запись Майкрософт может одновременно иметь роли администратора предприятия и владельца учетной записи.

### <a name="enterprise-administrator"></a>Администратор предприятия

Роль администратора предприятия имеет наивысший уровень доступа. Пользователи с этой ролью могут:

- управлять учетными записями и их владельцами;
- управлять другими администраторами предприятия;
- управлять администраторами подразделений;
- управлять контактными данными для отправки уведомлений;
- просматривать сведения об использовании всех учетных записей;
- просматривать неоплачиваемые расходы для всех учетных записей.

Корпоративное соглашение о регистрации может включать нескольких администраторов предприятия. Вы можете предоставить администраторам предприятия доступ только на чтение. Все они наследуют роль администратора подразделения.

### <a name="department-administrator"></a>Администратор отдела

Пользователи с этой ролью могут:

- создавать отделы и управлять ими;
- создавать учетные записи;
- просматривать сведения об использовании для отделов, которыми они управляют;
- просматривать затраты, если предоставлены необходимые разрешения.

Корпоративное соглашение о регистрации может включать нескольких администраторов отделов.

Вы можете предоставить администраторам отделов доступ только на чтение. Чтобы предоставить доступ только на чтение, измените учетную запись администратора отдела или создайте новую, указав значение **Да** для параметра "Только для чтения".

### <a name="account-owner"></a>Владелец учетной записи

Пользователи с этой ролью могут:

- создавать подписки и управлять ими;
- управлять администраторами служб;
- просматривать сведения об использовании для подписок.

Каждая учетная запись должна иметь уникальную учетную запись Майкрософт, рабочую или учебную учетную запись. Дополнительные сведения см. в описании [административных ролей в Azure при использовании Соглашения Azure Enterprise](billing-understand-ea-roles.md).

### <a name="service-administrator"></a>Администратор службы

Администратор службы получает разрешения управлять службами на портале Azure и назначать роль соадминистратора другим пользователям.

### <a name="notification-contact"></a>Контактное лицо для уведомлений

Контакт для уведомлений получает уведомления об использовании, связанные с регистрацией.

## <a name="activate-your-enrollment"></a>Активация соглашения о регистрации

Чтобы активировать службу, первоначальный администратор предприятия должен открыть портал Azure EA по адресу [https://ea.azure.com](https://ea.azure.com) и выполнить вход с тем адресом электронной почты, который указан в сообщении с приглашением.

Если вы настроены в качестве администратора EA, не ждите получения сообщения для активации, чтобы войти на портал Azure EA. Вы можете перейти на страницу [https://ea.azure.com](https://ea.azure.com) и войти в систему, используя свой адрес электронной почты (рабочий, учебный или Live ID) и пароль.

Если у вас есть несколько соглашений о регистраций, выберите то, которое вы намерены активировать. По умолчанию отображаются только активные соглашения о регистрации. Чтобы просмотреть журнал соглашений о регистрации, снимите флажок **Активные** в правом верхнем углу портала Azure EA.

В разделе "Соглашение о регистрации" отображается состояние **Активно**.

![Пример отображения активного соглашения о регистрации](./media/billing-ea-portal-get-started/ea-enrollment-status.png)

Только существующие администраторы предприятия Azure могут создавать других администраторов предприятия.

### <a name="create-another-enterprise-admin"></a>Создание администратора предприятия

- Войдите на [портал Azure EA](https://ea.azure.com) и откройте раздел **Manage** > **Enrollment Detail** (Управление > Параметры соглашения о регистрации), а на этой странице щелкните **+ Добавить администратора** в верхнем правом углу.

Убедитесь, что вам известны адрес электронной почты пользователя и предпочтительный метод проверки подлинности (например, рабочая, учебная учетная запись или учетная запись Майкрософт). Эта информация является обязательной для добавления пользователя.

Если вы не являетесь администратором EA, свяжитесь с администратором EA и попросите добавить вас в соглашение о регистрации. Когда вас добавят в соглашение о регистрации, вы получите электронное сообщение для активации.

Если администратор EA не сможет помочь вам, создайте [запрос на поддержку на портале Azure EA](https://support.microsoft.com/supportrequestform/cf791efa-485b-95a3-6fad-3daf9cd4027c). Введите следующие сведения:

- номер соглашения о регистрации;
- добавляемый адрес электронной почты и тип проверки подлинности (рабочая, учебная учетная запись или учетная запись Майкрософт);
- сообщение электронной почты с подтверждением от действующего администратора EA.
  - Если с действующим администратором EA невозможно связаться, обратитесь к партнеру или консультанту по программному обеспечению, чтобы они изменили контактную информацию через средство VLSC.

Дополнительные сведения см. в описании [административных ролей в Azure при использовании Соглашения Azure Enterprise](billing-understand-ea-roles.md).

## <a name="create-an-azure-ea-department"></a>Создание отдела Azure EA

Администраторы предприятия и отделов используют отделы, чтобы организовать управление и отчетность для корпоративных служб Azure, и распределять сведения об использовании по отделам и местам возникновения затрат. Администратор предприятия может:

- добавлять и удалять отделы;
- связывать учетные записи с отделами;
- создавать администраторов отделов;
- разрешать администраторам отделов просматривать цены и затраты.

Администратор отдела может добавлять новые учетные записи в свои отделы. Они могут удалять учетные записи из своих отделов, но не из соглашения о регистрации.

Чтобы добавить отдел, сделайте следующее:

1. На панели навигации слева щелкните **Manage** (Управление).
2. Перейдите на вкладку **Отдел** и щелкните **+ Add Department** (+ Добавить отдел), а затем введите необходимые сведения.
3. Единственное поле, которое обязательно для заполнения — это имя отдела. Длина имени должна быть не менее трех символов.
4. Завершив ввод, щелкните **Add** (Добавить).

## <a name="add-a-department-admin"></a>Добавление администратора отдела

После завершения создания отдела, администратор предприятия Azure может добавить администраторов отделов и связать их с отделами. Администратор отдела может:

- создавать других администраторов отделов;
- просматривать и изменять свойства отдела, например имя или связь с местом возникновения затрат;
- добавлять в свои отделы новые учетные записи;
- удалять учетные записи из своих отделов;
- скачивать сведения об использовании для своих отделов;
- просматривать ежемесячные сведения об использования и расходах для своего отдела, если администратор предприятия предоставил соответствующее разрешение <sup>1</sup>

### <a name="to-add-a-department-admin"></a>Добавление администратора отдела

Как администратор предприятия сделайте следующее:

1. На панели навигации слева щелкните **Manage** (Управление).
2. Откройте вкладку **Department** (Отдел) и выберите нужный отдел.
3. Щелкните **+ Add Administrator** (+ Добавить администратора) и введите необходимые сведения.
4. Чтобы предоставить доступ только для чтения, для параметра **Только для чтения** выберите значение **Да**, а затем щелкните **Добавить**.

![Пример диалогового окна для добавления администратора отдела](./media/billing-ea-portal-get-started/ea-create-add-department-admin.png)

### <a name="to-set-read-only-access"></a>Настройка доступа только на чтение

Вы можете предоставить администраторам отделов доступ только на чтение. Для этого выполните следующее при создании администратора отдела:

- Для параметра "Только для чтения" выберите значение **Да**.

Чтобы изменить существующего администратора отдела, выполните:

1. Выберите нужный отдел и щелкните значок карандаша рядом с именем **администратора отдела**, которого вы хотите заменить.
2. Для параметра "Только для чтения" выберите значение **Да**. Щелкните **Сохранить**.

Пользователи с ролью администратора предприятия автоматически получают разрешения администратора отдела.

<sup>1</sup> Если вы получили разрешение на просмотр данных об использовании и расходах для отдела за месяц, но не можете их просмотреть, обратитесь к партнеру.

## <a name="add-an-account"></a>Добавить учетную запись

Структура учетной записи и подписки влияет на то, как они управляются и как отображаются в счетах и отчетах. Обычно эта структура включает разбивку по корпоративным подразделениям, функциональным командам и географическим расположениям.

Чтобы добавить учетную запись, сделайте следующее:

1. На портале Azure EA щелкните **Manage** (Управление) на панели навигации слева.
2. Выберите вкладку **Account** (Учетная запись) и щелкните **Add Account** (Добавить учетную запись) на странице **Account** (Учетная запись).
3. Выберите отдел или оставьте это поле пустым, а затем выберите подходящий тип проверки подлинности.
4. Укажите понятное имя для идентификации учетной записи в отчетах.
5. Введите **адрес электронной почты владельца учетной записи**, чтобы установить вязь между ним и новой учетной записью.
6. Подтвердите адрес электронной почты и щелкните **Добавить**.

![Пример списка учетных записей с действием "+ Добавить учетную запись"](./media/billing-ea-portal-get-started/create-ea-add-an-account.png)

Вы можете добавить еще одну учетную запись, щелкнув элемент **Another Account** (Еще одна учетная запись) или кнопку **Добавить** в правом нижнем углу на панели инструментов слева.

Чтобы подтвердить владение учетной записью, сделайте следующее:

1. Войдите на портал Azure EA.
1. Просмотрите состояние учетной записи, чтобы проверить владение ею. Состояние должно измениться с **Pending** (Ожидание) на **Start/End date** (Дата начала и окончания). Даты начала и окончания обозначают дату первого входа пользователя и завершение срока действия соглашения.
1. Когда появится предупреждающее сообщение, владельцу учетной записи нужно нажать кнопку **Продолжить**, чтобы активировать учетную запись при первом входе на портал Azure EA Portal.


## <a name="change-account-owner"></a>Изменение владельца учетной записи

Администраторы предприятия могут с помощью портала Azure EA передать права на владение учетной записью подписки в пределах соглашения о регистрации. Это действие перемещает все подписки из исходной учетной записи пользователя в новую учетную запись пользователя.

Важные сведения о передаче данных об учетной записи пользователя:

- поддерживается передача из рабочей или учебной учетной записи в другую рабочую или учебную учетную запись;
- поддерживается передача из учетной записи Майкрософт в рабочую или учебную учетную запись;
- не поддерживается передача из рабочей или учебной учетной записи в учетную запись Майкрософт;
- поддерживается передача из учетной записи Майкрософт в другую учетную запись Майкрософт; чтобы поддерживать передачу, целевая учетная запись должна быть допустимой учетной записью Azure Commerce. Для новых учетных записей вам будет предложено создать учетную запись Azure Commerce при входе на портал Azure EA. Для существующих учетных записей вам придется создать новую подписку Azure, чтобы использовать эту учетную запись.
- Когда вы завершите перемещение подписки, корпорация Майкрософт обновит сведения о владельце учетной записи.

Политики управления доступом на основе ролей:

- Существующие политики управления доступом на основе ролей (RBAC) Azure, назначения ролей администратора службы и соадминистратора сохраняются только в том случае, если передача подписки Azure выполняется между двумя идентификаторами организации, относящимися к одному клиенту. Другие передачи подписок приводят к потере политик RBAC и назначений ролей администратора службы. Политики и роли администратора не передаются между каталогами. Администратором службы автоматически назначается владелец целевой учетной записи.
- Если передача подписки выполняется между двумя идентификаторами организации, относящимися к одному клиенту, в ней сохраняются политики управления доступом на основе ролей (RBAC) Azure, имеющиеся назначения ролей администратора службы и соадминистратора.

Перед изменением владельца учетной записи выполните следующие действия.

1. Откройте вкладку **Account** (Учетная запись) и найдите исходную учетную запись. Исходная учетная запись должна быть активной.
2. Найдите конечную учетную запись. Она также должна быть активной.

Чтобы передать владение учетной записью для всех подписок, выполните следующие действия.

1. На панели навигации слева щелкните **Manage** (Управление).
2. Откройте вкладку **Account** (Учетная запись) и наведите указатель мыши на учетную запись.
3. Щелкните значок изменения владельца учетной записи справа. Это символ в форме человека.
4. Выберите нужную учетную запись и щелкните **Далее**.
5. Подтвердите передачу и щелкните **Submit** (Отправить).

![Изображение со значком изменения владельца учетной записи](./media/billing-ea-portal-get-started/create-ea-create-sub-transfer-account-ownership-of-sub.png)

Чтобы передать владение учетной записью для одной подписки, выполните следующие действия.

1. На панели навигации слева щелкните **Manage** (Управление).
2. Откройте вкладку **Account** (Учетная запись) и наведите указатель мыши на учетную запись.

## <a name="create-a-subscription"></a>Создание подписки

Владельцы учетных записей могут просматривать подписки и управлять ими. Подписки позволяют предоставить разным группам в организации доступ к средам разработки и проектам. Например, вы можете создать подписки для тестирования, рабочей среды, разработки и промежуточных процессов. Наличие отдельных подписок для каждой среды приложений помогает защитить все эти среды. Вы также можете назначить разные учетные записи администраторов служб для каждой подписки. Подписки можно связывать с любым количеством служб. Владелец учетной записи создает подписки и назначает учетные записи администраторов служб для каждой подписки в своей учетной записи.

### <a name="add-a-subscription"></a>Добавление подписки

Для создания подписки важно учитывать следующие сведения.

При первом добавлении подписки в учетную запись вам будет предложено принять соглашение MOSA и тарифный план. Они не применимы к клиентам Соглашения Enterprise, но это необходимый шаг для создания подписки. Поправки, вступающие в силу при заключении соглашения Microsoft Azure Enterprise, имеют более высокий приоритет, чем указанные выше элементы, а значит контрактные обязательства не изменяются. При появлении запроса установите флажок, подтверждающий согласие с условиями.

Все новые подписки создаются с именем подписки по умолчанию _Microsoft Azure Enterprise_. Вы можете изменить имя подписки, чтобы отличать ее от других подписок в вашем соглашении о регистрации. Это также поможет идентифицировать ее в отчетах на уровне предприятия.

Чтобы добавить подписку, сделайте следующее:

1. Войдите в учетную запись на портале Azure EA.
2. Откройте вкладку **Администратор** и щелкните элемент **Подписка** в верхней части страницы.
2. Убедитесь, что вы вошли с правами владельца этой учетной записи.
3. Щелкните **+Добавить подписку**, а затем **Купить**.
  При первом добавлении подписки в учетную запись необходимо указать контактные данные. При добавлении последующих подписок контактные данные вносятся автоматически.
4. Щелкните **Подписки** и выберите созданную подписку, а затем щелкните **Изменить сведения о подписке**.
5. Измените **имя подписки** и **администратора службы**, затем щелкните символ галочки.
  Это имя подписки будет отображаться в отчетах, а также будет назначено как имя проекта, связанного с подпиской на портале разработки.

Для появления новой подписки в списке подписок может потребоваться до 24 часов. После создания подписки вы сможете выполнить следующие действия:

- [изменить сведения о подписке](https://account.azure.com/Subscriptions);
- [управлять службами в подписке](https://portal.azure.com/#home).

## <a name="transfer-ea-subscription-to-pay-as-you-go-subscription"></a>Переход с подписки EA на подписку с оплатой по мере использования

Чтобы перейти с подписки EA на отдельную подписку с оплатой по мере использования, необходимо создать запрос на поддержку на Azure EA Portal. Для этого щелкните **Новый запрос на поддержку** в разделе "Справка и поддержка".

## <a name="associate-an-existing-account-with-your-pay-as-you-go-subscription"></a>Связывание имеющейся учетной записи с подпиской с оплатой по мере использования

Если у вас уже есть учетная запись Microsoft Azure на портале Microsoft Azure, введите связанную учетную запись Майкрософт либо рабочую или учебную учетную запись, чтобы связать ее с регистрацией Соглашения Enterprise.

### <a name="associate-an-existing-account"></a>Связывание имеющейся учетной записи

1. На Enterprise Portal щелкните **Управление**.
1. Перейдите на вкладку **Учетная запись**.
1. Щелкните **+Добавить учетную запись**.
1. Введите учетную запись Майкрософт либо рабочую или учебную учетную запись, связанную с имеющейся учетной записью.
1. Подтвердите учетную запись Майкрософт либо рабочую или учебную учетную запись, связанную с имеющейся учетной записью.
1. Укажите имя, которое требуется использовать для идентификации этой учетной записи в отчетах.
1. Нажмите кнопку **Добавить**.
1. Вы можете добавить дополнительную учетную запись, снова выбрав параметр **+Добавить учетную запись**, или вернитесь на домашнюю страницу, нажав кнопку **Администратор**.
1. Если нажать кнопку для просмотра страницы **Учетная запись**, только что добавленная учетная запись будет отображаться в состоянии **В ожидании**.

### <a name="confirm-account-ownership"></a>Подтверждение владения учетной записью

1. Войдите в учетную запись электронной почты, связанную с указанной учетной записью Майкрософт либо рабочей или учебной учетной записью.
1. Откройте уведомление по электронной почте с _приглашением активировать учетную запись в службе Microsoft Azure из программы корпоративного лицензирования Майкрософт_.
1. Щелкните ссылку **Log into the Microsoft Azure Enterprise Portal** (Войти в Microsoft Azure Enterprise Portal) в приглашении.
1. Нажмите кнопку **Войти**.
1. Введите учетную запись Майкрософт либо рабочую или учебную учетную запись и пароль, чтобы войти и подтвердить владение учетной записью.

### <a name="azure-marketplace"></a>Azure Marketplace

Хотя большинство подписок переходят из среды с оплатой по мере использования в Enterprise Azure, службы Azure Marketplace не переходят. Чтобы получить одно представление всех подписок и расходов, мы рекомендуем добавить службы Azure Marketplace на Enterprise Portal.

1. В области навигации слева щелкните **Управление**.
1. Выберите вкладку **Регистрация**.
1. Просмотрите раздел со сведениями о регистрации.
1. Справа от поля Azure Marketplace щелкните значок с изображением карандаша, чтобы включить это поле, и щелкните **Сохранить**.

Теперь владелец учетной записи сможет приобрести подписки Azure Marketplace, которые ранее принадлежали среде с оплатой по мере использования.

После активации новых подписок Azure Marketplace в рамках вашей регистрации отмените подписки Marketplace, созданные в среде с оплатой по мере использования. Этот шаг очень важен, чтобы подписки Marketplace не перешли в неработоспособное состояние по истечении срока действия платежного средства с оплатой по мере использования.

### <a name="msdn"></a>MSDN

Подписки MSDN автоматически преобразуются в MSDN для разработки и тестирования с оплатой по мере использования, а предложение EA теряет все имеющиеся средства на счете.

### <a name="azure-in-open"></a>Azure с открытой лицензией;

Связь подписки Azure с открытой лицензией с Соглашением EA приведет к потере неиспользованных средств на счете Azure. Чтобы избежать потери средств на счете, рекомендуется использовать все кредиты по подписке Azure с открытой лицензией, прежде чем добавлять эту учетную запись в EA.  

### <a name="accounts-with-support-subscriptions"></a>Учетные записи с подписками на поддержку

Обратите внимание, что при добавлении на Enterprise Portal имеющихся учетных записей, в которых есть подписка на поддержку (но еще нет подписки на поддержку EA), подписка на поддержку MOSA не будет передаваться автоматически и ее нужно будет повторно купить в EA. Льготный период действия поддержки будет предоставляться в конце следующего месяца, чтобы предоставить время на переупорядочение поддержки.

## <a name="ea-term-glossary"></a>Глоссарий терминов EA

- **Учетная запись** — подразделение на Azure EA Portal, используемое для администрирования подписок и создания отчетов.
- **Владелец учетной записи** — пользователь, управляющий подписками и администраторами служб в Microsoft Azure. Он может просматривать данные об использовании этой учетной записи и связанных с ней подписок.
- **Подписка на поправки** — одна подписка на один год или взаимосвязанная подписка в рамках поправки на регистрацию.
- **Обязательство** — обязательство за годовую денежную сумму для служб Microsoft Azure с тарифом по обязательствам со скидкой за использование в рамках этой предоплаты.
- **Администратор подразделения** — пользователи, управляющие подразделениями, создающие учетные записи и владельцев учетных записей, просматривающие сведения об использовании для подразделений, которыми они управляют, и затраты при предоставлении разрешений.
- **Номер регистрации** — уникальный идентификатор, предоставляемый корпорацией Майкрософт для идентификации конкретной регистрации, связанной с Соглашением Enterprise.
- **Администратор предприятия** — пользователи, управляющие подразделениями и владельцами подразделений, а также учетными записями и владельцами учетных записей в Microsoft Azure. Они могут управлять администраторами предприятия, а также просматривать данные об использовании, оплаченные ресурсы и неоплаченные счета по всем учетным записям и подпискам, связанным с корпоративной регистрацией.
- **Соглашение Enterprise** — лицензионное соглашение Майкрософт для клиентов с централизованным приобретением, которые хотят стандартизировать всю организацию на базе технологий Майкрософт и поддерживать инфраструктуру отдела информационных технологий на стандартном программном обеспечении Майкрософт.
- **Регистрация Соглашения Enterprise** — регистрация в программе Соглашения Enterprise, предоставляющая продукты Майкрософт со скидкой.
- **Учетная запись Майкрософт**: веб-служба, которая позволяет веб-узлам выполнять аутентификацию пользователя с помощью одного набора учетных данных.
- **Поправка регистрации Microsoft Azure Enterprise (поправка регистрации)**  — поправка, подписанная предприятием, которая обеспечивает доступ к Microsoft Azure в рамках корпоративной регистрации.
- **Azure EA Portal** — портал, на котором корпоративные клиенты управляют учетными записями Microsoft Azure и связанными с ними подписками.
- **Количество потребляемых ресурсов** — количество отдельных служб Microsoft Azure, использованных в месяц.
- **Администратор служб** — пользователь, который может получать доступ к подпискам и проектам разработки и управлять ими на Azure EA Portal.
- **Подписка**: представляет собой подписку Azure EA Portal и является контейнером служб Microsoft Azure, которыми управляет один и тот же администратор служб.
- **Рабочая или учебная учетная запись** — для организаций, которые настроили Active Directory с федерацией в облаке и все учетные записи которых находятся в отдельном клиенте.

### <a name="enrollment-statuses"></a>Состояния регистрации:

- **Pending**. Администратор регистрации должен войти на Azure EA Portal. Когда он войдет в систему, состояние регистрации изменится на "Активно".
- **Активно**. Регистрация активна. На Azure EA Portal можно создавать учетные записи и подписки. Регистрация будет оставаться активной до наступления даты окончания срока действия Соглашения Enterprise.
- **Неопределенный продленный период**. Неопределенный продленный период возникает после наступления даты окончания срока действия Соглашения Enterprise. Он позволяет клиентам EA, которые согласились на продленный период, продолжать использовать Azure в течение срока действия Соглашения Enterprise. До наступления даты окончания срока действия Соглашения Enterprise для регистрации EA администратор регистрации должен решить, что делать дальше: продлевать регистрацию, добавив дополнительные денежные обязательства, перенести ресурсы в новую регистрацию, перейти на программу Microsoft Online Subscription (MOSP) или подтвердить отключение всех служб, связанных с регистрацией.
- **Срок действия истек**. Клиент EA отказался от продленного периода, а для регистрации EA наступила дата окончания срока действия Соглашения Enterprise. Срок действия регистрации истечет, и все связанные службы будут отключены.
- **Перенесено**. Регистрации, связанные учетные записи и службы которых перенесены в новую регистрацию, будут отображаться с состоянием "Перенесено". Обратите внимание, что регистрация не переносится автоматически, если при возобновлении действия создается номер регистрации. Для упрощения автоматического перемещения клиент должен добавить предыдущий номер регистрации в документы возобновления действия.

## <a name="get-started-on-azure-ea-faq"></a>Вопросы и ответы по началу работы в Azure EA

В этом документе приводятся сведения о типичных вопросах, которые задают клиенты в процессе подключения.  

### <a name="can-i-associate-my-existing-azure-account-to-enterprise-enrollment"></a>Можно ли связать имеющуюся учетную запись Azure с Соглашением о регистрации Enterprise?

Да, можно. Важно отметить, что все подписки Azure, для которых вы являетесь владельцем учетной записи, будут преобразованы в Соглашение Enterprise. Сюда входят подписки, использующие ежемесячные средства на счете (например, Visual Studio, AzurePass, MPN, BizSpark и т. д.), а это означает, что вы потеряете ежемесячные средства на счете.

### <a name="i-accidentally-associated-my-existing-azure-account-with-enterprise-enrollment-as-a-result-i-lost-my-monthly-credit-is-it-possible-to-get-my-monthly-credit-back"></a>Моя имеющаяся учетная запись Azure была случайно связана с Соглашением о регистрации Enterprise. В результате мои ежемесячные средства на счете утеряны. Можно ли вернуть ежемесячные средства на счете?

Чтобы восстановить ваше индивидуальное преимущество Azure для подписки Visual Studio после аутентификации в качестве владельца учетной записи EA и использования того входа в EA, что и для подписки Visual Studio, необходимо выполнить одно из следующих действий.
1. Удалите этого владельца учетной записи на EA Portal после удаления или перемещения любых подписок Azure, которыми он владеет, и зарегистрируйте его для получения индивидуальных преимуществ Azure для Visual Studio заново.
 OR
1. Удалите подписчика Visual Studio с сайта администрирования в VLSC и переназначьте подписку, чтобы в этот раз он выполнил другой вход. После этого он сможет зарегистрироваться для получения индивидуальных преимуществ Azure для Visual Studio заново.

### <a name="what-type-of-subscription-should-i-create"></a>Какого типа подписку следует создать?

EA Portal предлагает два типа подписок для корпоративных клиентов:

- Microsoft Azure Enterprise — идеально подходит для:
  - всех видов использования в рабочей среде;
  - получения лучших цен на основе расходов на инфраструктуру;
  - дополнительные сведения см. в статье https://azure.microsoft.com/pricing/enterprise-agreement/.
- Enterprise — разработка и тестирование идеально подходит для:
  - всех групповых рабочих нагрузок разработки и тестирования;
  - средних и тяжелых индивидуальных рабочих нагрузок разработки и тестирования;
  - получения доступа к специальным образам MSDN и тарифным ценам на службы;
  - дополнительные сведения см. в статье https://azure.microsoft.com/offers/ms-azr-0148p/.

### <a name="is-it-possible-to-transfer-subscription-ownership-to-another-account"></a>Можно ли передать права владения подпиской Azure другой учетной записи?

Да, права владения подпиской Azure можно передать другой учетной записи. Например, если у учетной записи A есть три подписки, администратор предприятия может переместить одну подписку в учетную запись B, одну в учетную запись C и одну в учетную запись D либо все подписки в учетную запись E.

Вы можете перейти на страницу EA и щелкнуть "Управление" > "Учетная запись", навести указатель на элемент **Учетная запись** (справа), и вы увидите параметр Transfer Ownership (Передать владение) (значок выстрела в голову) и "Переместить подписку" (значок списка).

Этот параметр будет отображаться только для активных учетных записей.

### <a name="i-see-subscription-name-defaults-to-offer-name-should-i-change-the-subscription-name-to-something-meaningful-to-my-organization"></a>Подписке по умолчанию присвоено название предложения. Можно ли изменить название подписки на что-нибудь понятное для моей организации?

Для любой созданной подписки по умолчанию будет использоваться выбранный вами тип предложения. Мы рекомендуем изменить название подписки на что-то понятное, чтобы упростить ее отслеживание.

**Чтобы изменить имя, сделайте следующее**:
1. Войдите на портал [https://account.windowsazure.com](https://account.windowsazure.com).
1. Щелкните список "Подписка".
1. Выберите "Подписка".
1. Щелкните значок **Управление подпиской**.
1. Измените сведения о подписке.

### <a name="how-can-i-track-cost-incurred-by-cost-center"></a>Как можно отследить затраты в месте возникновения затрат?

Для контроля затрат в месте возникновения затрат необходимо определить его на любом из следующих уровней:
- отдел;
- Учетная запись
- Subscription

В зависимости от ваших потребностей вы можете использовать одно и то же место возникновения затрат для мониторинга потребления и затрат, связанных с конкретным местом возникновения затрат.

Например, для отслеживания затрат для специального проекта, в котором участвуют несколько подразделений, может потребоваться использовать место возникновения затрат на уровне подписки, чтобы отследить потребление и затраты.

Вы не можете определить место возникновения затрат на уровне службы, и если вы хотите отследить потребление на уровне службы, можно использовать функцию "Тег", доступную на уровне службы.

### <a name="how-do-i-track-usage-and-spend-by-different-departments-in-my-organization"></a>Как отследить потребление и затраты в различных подразделениях в моей организации?

Вы можете создать столько подразделений, сколько требуется для регистрации EA. Чтобы правильно отследить потребление, необходимо убедиться, что подписки не являются общими для разных подразделений.

После создания подразделения и подписки можно просмотреть сведения в отчете о потреблении, который поможет отслеживать потребление и управлять затратами и расходами на уровне подразделения.

Вы также можете получить доступ к данным о потреблении с помощью API, раздела подробной информации и примера кода, которые можно найти по адресу [https://ea.azure.com/helpdocs/reportingAPI](https://ea.azure.com/helpdocs/reportingAPI).

### <a name="can-i-set-the-spending-quota-and-get-alerts-as-i-approach-my-limit"></a>Можно ли задать квоту расходов и получать оповещения о достижении ограничения?

Вы можете задать квоту расходов на уровне подразделения, и система будет автоматически сообщать вам, когда предельные суммы расходов будут достигать 50 %, 75 %, 90 % и 100 % от заданной квоты.

Чтобы определить квоту расходов, щелкните подразделение, в которое нужно добавить предельную сумму расходов, и значок редактирования. Нажмите кнопку **Сохранить**, чтобы сохранить сведения.

### <a name="i-used-resource-groups-rgs-to-implement-rbac-and-track-usage-how-can-i-view-the-associated-usage-details"></a>Я использовал группы ресурсов для реализации RBAC и мониторинга потребления. Как можно просмотреть связанные сведения о потреблении?

Такие сведения, как "Группы ресурсов" и "Тег", если они используются, отслеживаются на уровне службы. Сведения доступны в файле с подробными сведениями о потреблении (CSV), который можно скачать на Azure EA Portal [https://ea.azure.com/report/downloadusage](https://ea.azure.com/report/downloadusage).

Вы также можете получить доступ к данным о потреблении с помощью API, раздела подробной информации и примера кода, которые можно найти по адресу [https://ea.azure.com/helpdocs/reportingAPI](https://ea.azure.com/helpdocs/reportingAPI).

Обратите внимание, что теги можно применять только к ресурсам, которые поддерживают операции Resource Manager. Если вы создали виртуальную машину, виртуальную сеть или хранилище при помощи модели классического развертывания (например, через классический портал), то к таким ресурсам нельзя применить теги. Эти ресурсы необходимо развернуть повторно с помощью Resource Manager для поддержки тегов. Все остальные ресурсы поддерживают теги.

### <a name="can-i-perform-analyses-using-power-bi"></a>Можно ли выполнять анализ с помощью Power BI?

Да. С пакетом содержимого Microsoft Azure Enterprise для Power BI вы можете быстро импортировать и анализировать потребление ресурсов Azure по Соглашению о регистрации Enterprise, узнавать, какой отдел, учетная запись или подписка использовали больше всего ресурсов, какую службу ваша организация использовала больше всего или отслеживать тренды расходов и потребления.

**На веб-сайте Power BI:**

 1. Войдите с помощью допустимой рабочей или учебной учетной записи.
    - Рабочая или учебная учетная запись может быть той же или отличаться от используемой для доступа к регистрации через Azure EA Portal.
 1. На панели мониторинга служб:
    - Выберите плитку Microsoft Azure Enterprise.
    - Нажмите кнопку **Соединить**.
 1. На экране Connect to Azure Enterprise (Подключение к Azure Enterprise):
    - Выберите URL-адрес окружения Azure: [https://ea.azure.com](https://ea.azure.com).
    - Укажите число месяцев: от 1 до 36.
    - Введите номер регистрации.
    - Щелкните **Далее**.
 1. Введите ключ API в поле ключа аутентификации. Ключ API можно получить на портале Azure EA Portal в разделе "Скачать сведения об использовании". Щелкните **API Access Key** (Ключ доступа к API).
    - Скопируйте и вставьте ключ в поле "Ключ учетной записи".
    - Загрузка данных в Power BI займет примерно 5–30 минут в зависимости от размера набора данных.

Отчеты Power BI доступны для прямых, партнерских и косвенных клиентов EA, которые могут просматривать сведения о выставлении счетов.
