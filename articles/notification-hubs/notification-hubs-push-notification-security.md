# <a name="notification-hubs-security"></a>Безопасность Центров уведомлений

## <a name="overview"></a>Обзор

В этой статье описывается модель безопасности Центров уведомлений Azure.

## <a name="shared-access-signature-security-sas"></a>Безопасность на базе подписанного URL-адреса (SAS)

Центры уведомлений реализуют схему безопасности на уровне сущностей под названием SAS (подписанный URL-адрес). Эта схема позволяет сущности обмена сообщениями объявлять до 12 правил авторизации в своем описании, которое предоставляет права на эту сущность.

Каждое правило содержит имя, значение ключа (общий секрет) и набор прав, как описано в [утверждений безопасности](#security-claims). При создании концентратора уведомлений, автоматически создаются два правила: одно с **прослушивания** правами (клиентское приложение), а второй для **все** правами (используемое внутренним сервером приложений).

При управлении регистрациями из клиентских приложений, если данные, отправляемые посредством уведомлений, не являются конфиденциальными (например, новости погоды), то обычный способ доступа к Центру уведомлений состоит в передаче значения ключа для правила "Доступ только для прослушивания" в клиентское приложение, а также передаче значения ключа для правила "Полный доступ" в серверную часть приложения.

Приложения не должны включать значение ключа в клиентских приложениях Windows Store, вместо этого должны получить его из серверной части приложения при запуске клиентского приложения.

Ключ с **прослушивания** доступ позволяет клиентскому приложению регистрироваться на любой тег. Если в приложении регистрации должны ограничиваться конкретными тегами для конкретных клиентов (например, когда теги отражают идентификаторы пользователей), регистрацию должна проводить серверная часть вашего приложения. Дополнительные сведения см. в разделе [управления регистрацией](notification-hubs-push-notification-registration-management.md). в статье "Управление регистрациями" Обратите внимание, что в этом случае клиентское приложение не имеет прямого доступа к Центрам уведомлений.

## <a name="security-claims"></a>Утверждения безопасности

Аналогично другим сущностям для операций Центра уведомлений разрешены три типа утверждений безопасности: **Прослушивание**, **отправки**, и **управление**.

| Утверждение   | Описание                                          | Разрешенные операции |
| ------- | ---------------------------------------------------- | ------------------ |
| Прослушивание  | Создание, обновление, чтение и удаление отдельных регистраций | Создание и обновление регистрации<br><br>Чтение регистрации<br><br>Чтение всех регистраций для обработки<br><br>Удаление регистрации |
| Отправка    | Отправка сообщений в Центр уведомлений                | Отправить сообщение |
| Управление  | Операции CRUD в Центрах уведомлений (включая обновление учетных данных PNS и ключей безопасности) и чтение регистраций на основе тегов |Создание, обновление, чтение, удаление Центров уведомлений<br><br>Чтение регистраций по тегу |

Уведомление концентраторов принимает маркеры созданы с помощью подписанных URL-общие ключи, настроенные напрямую в центре уведомлений.

Не поддерживается для отправки уведомления в нескольких пространствах имен. Пространства имен логический контейнер для центров уведомлений и не участвует в отправке уведомлений.
Политики доступа на уровне пространства имен (учетные данные) могут использоваться для операций уровня пространства имен, например: список центров уведомлений, создание или удаление центров уведомлений, и т.д. Только политики доступа на уровне концентратора позволит вам отправлять уведомления.
