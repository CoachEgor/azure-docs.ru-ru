# <a name="integration-of-azure-devtest-labs-and-azure-devops"></a>Интеграция Azure DevTest Labs и Azure DevOps
DevOps — это методологии разработки программного обеспечения, которое интегрируется с операциями (Ops) для системы разработки программного обеспечения (Dev). Эта система может предоставлять новые функции, обновления и исправления, согласованные с бизнес-целей. Эту методику включает в себя все — от разработки новых функций, основанных на цели, шаблоны использования и отзывы клиентов. исправление, восстановление и ограниченный режим работы системы, в случае возникновения ошибки. Компонент легко идентифицируемыми этой методики является непрерывной интеграции (CI) / конвейер непрерывной поставки (CD). Конвейер Непрерывной интеграции и принимает сведения, кода и ресурсов из фиксации через ряд шагов, которые включают построение, тестирование и развертывание, для создания системы. В этой статье рассматриваются различные способы эффективного использования labs в конвейер в корпоративной среде. 

## <a name="benefits-of-using-labs-in-devops-workflow"></a>Преимущества использования labs в рабочем процессе DevOps 

### <a name="focused-access"></a>Доступом 
Использование лаборатории как компонент позволяет конкретных экосистемы, связываемый с ограниченной группой пользователей. Как правило команда или группу, которая работает в общую область или была включена отдельная функция будет иметь лаборатории, назначенные им.   

### <a name="infrastructure-replication-in-the-cloud"></a>Инфраструктуры репликации в облаке 
Разработчик может быстро настроить экосистемы разработки, которая содержит поле с исходным кодом и средства разработки. Разработчик также может создать среду, которая будет практически идентичен тому рабочую конфигурацию. Он помогает быстрее во внутреннем цикле разработки. 

### <a name="pre-production"></a>Подготовительной 
Наличие в лабораторию в конвейере Непрерывной интеграции и облегчает несколько разных подготовительных средах или машин, работающих в то же время для асинхронной проверки. Поддержка различных средах, например, агенты сборки можно развернуть и управление ими осуществляется в лаборатории. 

## <a name="devops-with-devtest-labs"></a>DevOps с помощью DevTest Labs 

### <a name="development--operation"></a>Разработка и операции 
Лаборатории должны концентрироваться в группе, которая работает в функциональной области. Это общие цели допускает совместное использование ресурсов конкретных областей, таких как инструменты, сценарии или шаблонов Resource Manager. Он позволяет быстрее изменения при ограничении негативное воздействие до одной группы. Этих общих ресурсов позволяют разработчикам создание виртуальных машин с необходимый код, средства и конфигурации. Они могут быть созданы динамически или система, которая создает базовые образы с настройками. Не только разработчик можно создавать виртуальные машины, но также они могут создавать среды DevTest Labs, на основе шаблонов, необходимые для создания необходимых ресурсов Azure в лаборатории. Любые изменения или уничтожения данных рабочих может выполняться в лабораторной среде без влияния на кто-то еще. Рассмотрим сценарий, где этот продукт является отдельная система, установленная на компьютере клиента. В этом сценарии DevTest Labs повысила создания виртуальной Машины, который включает в себя установку дополнительного программного обеспечения с помощью артефактов и предварительное создание конфигурации клиента для ускорить внутренний цикл тестирования кода. 
  
## <a name="cicd-pipeline"></a>Конвейер CI/CD 
Конвейер Непрерывной интеграции и является одним из важных компонентов из DevOps, переместить код из запроса на Вытягивание разработчика, он интегрируется с существующим кодом и его развертывание в рабочей среде экосистему. Все ресурсы не нужно быть в лаборатории. Например узел Jenkins может настроить за пределами лаборатории как ресурс постоянных. Ниже приведены некоторые конкретные примеры интеграции labs в конвейер. 

### <a name="build"></a>Создание 
Конвейер сборки занимается созданием пакета компонентов, которые будут одновременно проверить передаваемые в конвейер выпуска. Лабораторные занятия по могут быть частью конвейера сборки как расположение для агентов сборки и другие ресурсы поддержки. У меня есть возможность динамически создать инфраструктуру позволяет расширить возможности управления. Возможность использовать несколько сред в лаборатории каждая сборка может выполняться асинхронно при использовании идентификатор сборки как часть сведений о среде для уникальной идентификации ресурсы для определенной сборки.   

Агенты построения лаборатории возможность ограничить доступ, повышает безопасность и снижает вероятность случайного повреждения.  

### <a name="test"></a>Тест 
DevTest Labs позволяет конвейера CI/CD для автоматизации создания ресурса Azure (виртуальные машины, среды), который может использоваться для автоматического и ручного тестирования. Виртуальные машины будут создаваться с помощью артефактов или формул, использующих данные из процесса сборки для создания необходимой для тестирования различных пользовательских конфигураций.   

### <a name="release"></a>Release 
DevTest Labs обычно используется для проверки подлинности в разделе выпуска, перед развертыванием кода. Это аналогично тестированию в области построения. Рабочие ресурсы не должны быть развернуты в DevTest Labs. 

### <a name="customization"></a>Настройка 
В DevOps в Azure есть существующие задачи, которые позволяют манипуляции виртуальных машин и сред в пределах определенного labs. Хотя службы DevOps Azure являются одним из способов управления конвейера CI/CD, вы можете интегрировать лаборатории в любой системе, которая поддерживает возможность вызова REST API, выполнения сценариев PowerShell или с помощью Azure CLI. 

Хотя некоторые диспетчеры конвейера Непрерывной интеграции и иметь существующий открытый подключаемые модули, которые могут управлять ресурсами в Azure и DevTest Labs. Может потребоваться использовать настраиваемые сценарии в соответствии с потребностями конкретного конвейера.  С учетом, при выполнении задачи субъекта-службы используется для получения доступа к лаборатории с соответствующей роли. Субъекту-службе обычно требуется участника роли доступ к лаборатории. Дополнительные сведения см. в разделе [интегрировать Azure DevTests Labs в конвейер непрерывной интеграции и поставки Azure DevOps](devtest-lab-integrate-ci-cd-vsts.md).
