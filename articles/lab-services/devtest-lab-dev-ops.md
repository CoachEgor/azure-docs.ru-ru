# <a name="integration-of-azure-devtest-labs-and-azure-devops"></a>Интеграция лабораторий Azure DevTest и Azure DevOps
DevOps — это методология разработки программного обеспечения, которая интегрирует разработку программного обеспечения (Dev) с операциями (Ops) для системы. Эта система может предоставлять новые функции, обновления и исправления в соответствии с бизнес-целями. Эта методология охватывает все, начиная от проектирования новых функций, основанных на целях, шаблонах использования и обратной связи с клиентами; для фиксации, восстановления и закалки системы при возникновении проблем. Легко идентифицируемым компонентом этой методологии является непрерывная интеграция (CI)/непрерывная доставка (CD) конвейера. Конвейер CI/CD принимает информацию, код и ресурсы из коммита через ряд шагов, которые включают создание, тестирование и развертывание для создания системы. В этой статье основное внимание уделяется различным способам эффективного использования лабораторий в конвейере в корпоративной среде. 

## <a name="benefits-of-using-labs-in-devops-workflow"></a>Преимущества использования лабораторий в рабочем процессе DevOps 

### <a name="focused-access"></a>Целенаправленный доступ 
Использование лаборатории в качестве компонента позволяет конкретной экосистеме ассоциироваться с ограниченной группой людей. Как правило, команда или группа, которая работает в общей области или конкретной функции будет иметь лабораторию назначены им.   

### <a name="infrastructure-replication-in-the-cloud"></a>Репликация инфраструктуры в облаке 
Разработчик может быстро настроить экосистему разработки, которая включает в себя коробку разработчиков с исходным кодом и инструментами. Разработчик также может создать среду, которая почти идентична конфигурации производства. Это помогает с более быстрым развитием внутренней петли. 

### <a name="pre-production"></a>Предварительный этап 
Наличие лаборатории в конвейере CI/CD упрощает наличие нескольких различных предсерийных сред или машин, работающих одновременно для асинхронного тестирования. Различные инфраструктуры поддержки, такие как агенты сборки, могут быть развернуты и управляется в лаборатории. 

## <a name="devops-with-devtest-labs"></a>DevOps с DevTest Labs 

### <a name="development--operation"></a>Развитие / Операция 
Лаборатория должна быть сосредоточена на команде, которая работает в определенной области. Эта общая направленность позволяет совместно использовать ресурсы, специфичные для конкретных областей, такие как инструменты, скрипты или шаблоны «Менеджер ресурсов». Это позволяет быстрее изменения при ограничении негативных последствий для меньшей группы. Эти общие ресурсы позволяют разработчику создавать всхожие ми-мы со всеми необходимыми кодами, инструментами и конфигурацией. Они могут быть созданы либо динамически, либо иметь систему, которая создает базовые изображения с настройками. Разработчик может не только создавать vMs, но и создавать среды DevTest Labs на основе необходимых шаблонов для создания соответствующих ресурсов Azure в лаборатории. Любые изменения или разрушительная работа может быть сделано против лабораторных условий, не затрагивая никого другого. Рассмотрим сценарий, при котором продукт представляет собой автономную систему, установленную на машине клиента. В этом сценарии DevTest Labs улучшила создание VM, которое включает в себя установку дополнительного программного обеспечения с использованием артефактов и предстроя конфигурации клиентов для более быстрого внутреннего тестирования цикла кода. 
  
## <a name="cicd-pipeline"></a>КОНВЕЙЕР CI/CD 
Конвейер CI/CD является одним из важнейших компонентов DevOps, который перемещает код из запроса на вытягивание разработчика, интегрирует его с существующим кодом и развертывает в производственной экосистеме. Все ресурсы не должны находиться в лаборатории. Например, хост Jenkins может быть настроен вне лаборатории в качестве более постоянного ресурса. Вот несколько конкретных примеров интеграции лабораторий в конвейер. 

### <a name="build"></a>Сборка 
Конвейер сборки ориентирован на создание пакета компонентов, которые будут протестированы вместе, чтобы быть переданы в конвейер выпуска. Лаборатории могут быть частью конвейера сборки в качестве места для агентов сборки и других вспомогательных ресурсов. Наличие способности динамически строить инфраструктуру позволяет осуществлять больший контроль. С возможностью иметь несколько сред в лаборатории, каждая сборка может быть запущена асинхронно при использовании идентификатора сборки как часть информации о среде, чтобы однозначно определить ресурсы для конкретной сборки.   

Для агентов сборки способность лаборатории ограничивать доступ повышает безопасность и снижает вероятность случайных поработок.  

### <a name="test"></a>Тест 
DevTest Labs позволяет конвейеру CI/CD автоматизировать создание ресурсов Azure (VMs, среды), которые могут быть использованы для автоматизированного и ручного тестирования. VMs будут создаваться с использованием артефактов или формул, использующих информацию из процесса сборки для создания различных пользовательских конфигураций, необходимых для тестирования.   

### <a name="release"></a>Release 
DevTest Labs обычно используется для проверки в разделе релиза перед развертыванием кода. Это похоже на тестирование в разделе Сборка. Производственные ресурсы не должны быть развернуты в DevTest Labs. 

### <a name="customization"></a>Настройка 
В Azure DevOps существуют задачи, позволяющие манипулировать визажательными и средами в определенных лабораториях. В то время как службы Azure DevOps являются одним из способов управления конвейером CI/CD, можно интегрировать лабораторию в любую систему, которая поддерживает возможность вызова REST AIS, выполнения скриптов PowerShell или использования Azure CLI. 

В то время как некоторые менеджеры конвейеров CI/CD имеют существующие плагины с открытым исходным кодом, которые могут управлять ресурсами в Azure и DevTest Labs. Возможно, вам придется использовать некоторые пользовательские сценарии в соответствии с конкретными потребностями конвейера.  Имея это в виду, при выполнении задачи, директор службы используется с соответствующей ролью, чтобы получить доступ к лаборатории.
