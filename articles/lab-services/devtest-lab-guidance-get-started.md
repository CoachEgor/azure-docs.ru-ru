# <a name="popular-scenarios-for-using-azure-devtest-labs"></a>Популярные сценарии использования лабораторий Azure DevTest
В зависимости от потребностей предприятия, DevTest Labs может быть настроена с различными требованиями.  В этой статье рассматриваются популярные сценарии. Каждый сценарий охватывает преимущества, принесенные с помощью DevTest Labs и ресурсов для использования для реализации этих сценариев.  

- Рабочие столы для разработки
- Среды тестирования
- Учебные занятия, практические лаборатории и хакатоны
- Расследования в песочнице
- Лаборатории для аудитории

## <a name="developer-desktops"></a>Рабочие столы для разработки
Разработчики часто имеют различные требования к компьютерам разработки для разных проектов. С DevTest Labs разработчики могут иметь доступ к виртуальным машинам по требованию, настроенным в соответствии с их наиболее распространенными сценариями. DevTest Labs обеспечивает следующие преимущества:

- Организации могут обеспечить общие машины разработки, обеспечивающие согласованность между группами.
- Разработчики могут быстро предоставить свои машины разработки по требованию или претендовать на [существующую ранее настроенную машину.](devtest-lab-add-claimable-vm.md)
- Разработчики могут предоставлять ресурсы в порядке самообслуживания без получения разрешений на уровне подписки.
- ИТ или админы могут [заранее определить топологию сетей,](devtest-lab-configure-vnet.md) и разработчики могут использовать ее простым и интуитивно понятным способом, не требуя особого доступа.
- Разработчики могут легко [настроить](devtest-lab-add-vm.md#add-an-existing-artifact-to-a-vm) свои машины разработки по мере необходимости.
- Администраторы могут контролировать затраты, следя за тем, чтобы:
    - Разработчики [не могут получить больше вымиков,](devtest-lab-set-lab-policy.md#set-virtual-machines-per-user) чем им нужно для разработки
    - [VMs выключаются,](devtest-lab-set-lab-policy.md#set-auto-shutdown) когда они не используются
    - Только [разрешение подмноженого набора экземпляров VM](devtest-lab-set-lab-policy.md#set-allowed-virtual-machine-sizes) для конкретных лабораторий
    - [Управление целевыми показателями затрат и уведомлениями](devtest-lab-configure-cost-management.md) для каждой лаборатории.

Для дальнейшего прочтения [см.](devtest-lab-developer-lab.md) 

## <a name="test-environments"></a>Среды тестирования
Создание тестовых сред и управление ими на предприятии может требовать существенных усилий. С DevTest Labs тестовые среды могут быть легко созданы, обновлены или дублированы. Это позволяет группам получить доступ к полностью настроенной среде, когда это необходимо. В этом сценарии DevTest Labs предоставляет следующие преимущества:

- Организации могут предоставлять общие среды тестирования, обеспечивая согласованность между группами.
- Тестеры могут протестировать последнюю версию своего приложения, быстро подгоняя среды Windows и Linux, используя многоразовые шаблоны.
- Администраторы могут подключить лабораторию к Azure DevOps, чтобы включить сценарии DevOps
- Владельцы лабораторий могут контролировать расходы, гарантируя, что:
    - [VMs в средах выключаются,](devtest-lab-set-lab-policy.md#set-auto-shutdown) когда они не используются
    - Только [разрешение подмноженого набора экземпляров VM для](devtest-lab-set-lab-policy.md#set-allowed-virtual-machine-sizes) конкретных лабораторий
    - [Управление целевыми показателями затрат и уведомлениями](devtest-lab-configure-cost-management.md) для каждой лаборатории.

Для получения дополнительной информации [см.](devtest-lab-test-env.md)

## <a name="sandboxed-investigations"></a>Расследования в песочнице
Разработчики часто изучают различные технологии или дизайн инфраструктуры. По умолчанию все среды, созданные с помощью DevTest Labs, создаются в их собственной группе ресурсов. Пользователь DevTest Labs получает только доступ к этим ресурсам. Однако для разработчиков, которым требуется больше контроля, можно обновить общелабораторную настройку, чтобы дать пользователю DevTest Labs права на получение [помощи](https://azure.microsoft.com/updates/azure-devtest-labs-view-and-set-access-rights-to-an-environment-rg/) для каждой создаваемой среды.  С DevTest Labs разработчики могут автоматически получить разрешение на участие в программе для сред, создаваемых ими в лаборатории.  Этот сценарий позволяет разработчикам добавлять и/или изменять ресурсы Azure по мере необходимости для их сред разработки или тестирования. Стоимость по странице [ресурса](devtest-lab-configure-cost-management.md#view-cost-by-resource) позволяет владельцам лабораторий отслеживать стоимость каждой среды, используемой для исследований.

Для получения дополнительной информации [см.](https://aka.ms/dtl-sandbox)

## <a name="trainings-hands-on-labs-and-hackathons"></a>Тренинги, практические лаборатории и хакатоны 
Лаборатория в Azure DevTest Labs выступает в качестве отличного контейнера для временных занятий, таких как семинары, практические лаборатории, тренинги или хакатоны.  Azure DevTest Labs позволяет создать лабораторию, куда можно добавить пользовательские шаблоны. Каждый обучаемый может использовать их для создания идентичных изолированных сред для обучения. В этом сценарии DevTest Labs предоставляет следующие преимущества:

- [Политики](devtest-lab-set-lab-policy.md) гарантируют, что слушатели получают только то количество ресурсов, таких как виртуальные машины, которые им нужны.
- Предварительно настроенные и созданные машины [востребованы](devtest-lab-add-claimable-vm.md) одним действием от стажера.
- Лаборатории совместно с стажерами, доступ к [URL для лаборатории](devtest-lab-faq.md#how-do-i-share-a-direct-link-to-my-lab).
- [Даты истечения](devtest-lab-add-vm.md#steps-to-add-a-vm-to-a-lab-in-azure-devtest-labs) срока действия на виртуальных машинах гарантируют, что машины будут удалены после того, как они больше не нужны.
- По окончании обучения легко [удалить лабораторию](devtest-lab-delete-lab-vm.md#delete-a-lab) и все [связанные с ней ресурсы.](devtest-lab-faq.md#how-do-i-automate-the-process-of-deleting-all-the-vms-in-my-lab)

Для получения дополнительной информации [см.](devtest-lab-training-lab.md)  

## <a name="proof-of-concept-vs-scaled-deployment"></a>Доказательство концепции против масштабирования развертывания
После того, как вы решите исследовать DevTest Labs, есть два общих пути вперед: Доказательство концепции против масштабируемого развертывания.  

**Масштабируемое развертывание** состоит из выполняемых неделями или месяцами проверки и планирования с целью развертывания DevTest Labs для всего предприятия с сотнями или тысячами разработчиков.

Доказательство развертывания **концепции** фокусируется на сосредоточенных усилиях одной группы по установлению организационной ценности. Хотя может быть заманчиво использовать масштабируемое развертывание, этот подход завершается неудачей чаще, чем подтверждение концепции. Таким образом, мы советуем начать с малого, учиться на опыте первой команды, повторить тот же подход с двумя или тремя дополнительными командами, а затем, основываясь на полученных знаниях, планировать масштабируемые развертывания. Для успешного подтверждения концепции мы рекомендуем выбрать одну или две команды и определить их сценарии (среды разработки и тестирования), задокументировать текущие варианты использования и развернуть DevTest Labs.
