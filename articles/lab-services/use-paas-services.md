# <a name="use-platform-as-a-service-paas-services-in-azure-devtest-labs"></a>Использование сервисов «Платформа как услуга» (PaaS) в Лабораториях Azure DevTest Labs
PaaS поддерживается в DevTest Labs с помощью функции среды. Среды в лабораториях DevTest поддерживаются предварительно настроенными шаблонами управления ресурсами Azure в репозитории Git. Окружающая среда может содержать ресурсы PaaS и IaaS. Они позволяют создавать сложные системы, которые могут включать ресурсы Azure, такие как виртуальные машины, базы данных, виртуальные сети и веб-приложения, которые настроены на совместную работу. Эти шаблоны позволяют последовательно развертывать и улучшать управление средами с помощью управления исходным кодом. 

Правильно настроенная система позволяет следующие сценарии: 

- Разработчики должны иметь независимые и несколько сред
- Тестирование на различных конфигурациях асинхронно
- Интеграция в постановочные и производственные конвейеры без изменений шаблонов
- Наличие машин и сред разработки в одной лаборатории улучшает простоту управления и отчетность о затратах.  

Поставщик ресурсов DevTest Labs создает ресурсы от имени пользователя лаборатории, поэтому для использования ресурсов PaaS не требуется никаких дополнительных разрешений. Следующее изображение показывает кластер Service Fabric как среду в лаборатории.

![Кластер Service Fabric как среда](./media/create-environment-service-fabric-cluster/cluster-created.png)

Для получения дополнительной информации о настройке сред [см. Создать мульти-VM-среды и ресурсы PaaS с шаблонами Azure Resource Manager.](devtest-lab-create-environment-from-arm.md) DevTest Labs имеет общедоступное хранилище шаблонов Azure Resource Manager, которые можно использовать для создания сред без подключения к внешнему источнику GitHub самостоятельно. Для получения дополнительной информации об общественных средах см. [Нанастройка и использование общедоступных средах в Лабораториях Azure DevTest Labs.](devtest-lab-configure-use-public-environments.md)

В крупных организациях группы разработки обычно предоставляют такие среды, как настраиваемые/изолированные среды тестирования. Могут быть среды, которые должны использоваться всеми командами в бизнес-подразделении или подразделении. ИТ-группа может захотеть предоставить свои среды, которые могут быть использованы всеми группами в организации.  

## <a name="customizations"></a>Настройки

#### <a name="sandbox"></a>Песочница 
Владелец лаборатории может настроить лабораторные среды, чтобы изменить роль пользователя от **читателя** к **вкладчику** в группе ресурсов. Эта возможность находится на странице **настройки лаборатории** в соответствии с **конфигурацией и политикой** лаборатории. Это изменение роли позволяет пользователю добавлять или удалять ресурсы в этой среде. Если требуется ограничить доступ, используйте политики Azure. Эта функциональность позволяет настроить ресурсы или конфигурацию без доступа на уровне подписки.

#### <a name="custom-tokens"></a>Пользовательские маркеры
Существует некоторая пользовательская лабораторная информация, которая находится за пределами группы ресурсов и специфична для сред, к которым шаблон может получить доступ. Вот некоторые из них: 

- Идентификация лабораторной сети
- Расположение
- Учетная запись хранения, в которой хранятся файлы шаблонов менеджера ресурсов. 
 
#### <a name="lab-virtual-network"></a>Лаборатория виртуальной сети
В [статье connecting environments to the virtual network](connect-environment-lab-virtual-network.md) статьи лаборатории `$(LabSubnetId)` описывается, как изменить шаблон ресурсного менеджера для использования маркера. При создании `$(LabSubnetId)` среды токен заменяется первым знаком подсети, где использование в виртуальном варианте **создания машины** соответствует **действительности.** Это позволяет нашей среде использовать ранее созданные сети. Если вы хотите использовать те же шаблоны resource Manager в средах тестирования, как постановка и производство, используйте `$(LabSubnetId)` в качестве значения по умолчанию в параметре шаблона ресурсного менеджера. 

#### <a name="environment-storage-account"></a>Учетная запись хранения среды
DevTest Labs поддерживает использование [вложенных шаблонов менеджера ресурсов.](../azure-resource-manager/templates/linked-templates.md) Шаблоны[диспетчера ресурсов Azure для шаблонов тестирования среды](deploy-nested-template-environments.md) объясняют, как использовать `_artifactsLocation` uri для `_artifactsLocationSasToken` шаблона диспетчера ресурсов в той же папке, что и вложенная папка основного шаблона. Более подробную информацию об этих двух маркерах можно узнать в разделе **«Руководство** по ресурсам [развертывания» —Руководство по рекомендациям](https://github.com/Azure/azure-quickstart-templates/blob/master/1-CONTRIBUTION-GUIDE/best-practices.md)по передовым практикам .

## <a name="user-experience"></a>Взаимодействие с пользователем

## <a name="developer"></a>Разработчик
Разработчики используют тот же рабочий процесс для создания VM для создания определенной среды. Они выбирают среду по сравнению с изображением машины и вводят необходимую информацию, требуемую шаблоном. Каждый разработчик, имеющий среду, позволяет развертывать изменения и улучшать внутреннюю отладку цикла. Среда может быть создана в любое время с помощью последнего шаблона.  Эта функция позволяет уничтожать и воссоздавать среды, чтобы сократить время простоя от необходимости вручную создавать систему или восстанавливаться после тестирования неисправностей.  

### <a name="testing"></a>Тестирование
Среды DevTest Labs позволяют проводить независимое тестирование определенного кода и конфигураций асинхронно. Общей практикой является настройка среды с кодом из индивидуального запроса на вытягивание и начало любого автоматизированного тестирования. После завершения автоматизированного тестирования любое ручное тестирование может быть выполнено в отношении конкретной среды. Обычно этот процесс выполняется в рамках конвейера CI/CD. 

## <a name="management-experience"></a>Опыт управления

### <a name="cost-tracking"></a>Отслеживание затрат
Функция отслеживания затрат включает ресурсы Azure в различных средах как часть общей тенденции затрат. Затраты ресурсов не выбивают различные ресурсы в среде, а отображают среду как единую стоимость.

### <a name="security"></a>Безопасность
Правильно настроенная подписка Azure с DevTest Labs может [ограничить доступ к ресурсам Azure только через лабораторию.](devtest-lab-add-devtest-user.md) В средах владелец лаборатории может разрешить пользователям доступ к ресурсам PaaS с утвержденными конфигурациями, не позволяя доступа к другим ресурсам Azure. В сценарии, где пользователи лаборатории настраивает среды, владелец лаборатории может разрешить доступ к вкладчику. Доступ к вкладчику позволяет пользователю лаборатории добавлять или удалять ресурс Azure только в управляемой группе ресурсов. Это позволяет более легко отслеживать и управления по сравнению с позволяют пользователю доступ к подписке.

### <a name="automation"></a>Служба автоматизации
Автоматизация является ключевым компонентом для крупномасштабной и эффективной экосистемы. Автоматизация необходима для управления или отслеживания нескольких сред в подписках и лабораториях.

### <a name="cicd-pipeline"></a>Конвейер CI/CD
Службы PaaS в DevTest Labs могут помочь улучшить конвейер CI/CD, сосредоточив развертывание, где доступ контролируется лабораторией.
