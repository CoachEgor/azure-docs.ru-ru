# <a name="best-practices-for-distributed-and-collaborative-development-of-azure-devtest-labs-resources"></a>Рекомендации по распределенной и совместной разработке ресурсов Azure DevTest Labs
Распределенная совместная разработка позволяет различным группам или людям разрабатывать и поддерживать кодовую базу. Чтобы быть успешным, процесс разработки зависит от способности создавать, обмениваться и интегрировать информацию. Этот ключевой принцип разработки может быть использован в Лабораториях Azure DevTest Labs. В лаборатории существует несколько типов ресурсов, которые обычно распределены между различными лабораториями на предприятии. Различные виды ресурсов сосредоточены на двух областях:

- Ресурсы, которые хранятся внутри лаборатории (на базе лаборатории)
- Ресурсы, хранящиеся во [внешних репозиториях, подключенных к лаборатории](devtest-lab-add-artifact-repo.md) (на основе хранилища кода). 

В этом документе описаны некоторые рекомендации, которые позволяют совместно работать и распространяться между несколькими группами, обеспечивая при этом настройку и качество на всех уровнях.

## <a name="lab-based-resources"></a>Лабораторные ресурсы

### <a name="custom-virtual-machine-images"></a>Пользовательские виртуальные изображения машины
Вы можете иметь общий источник пользовательских изображений, которые развернуты в лабораториях на ночной основе. Для получения подробной информации [см.](image-factory-create.md)    

### <a name="formulas"></a>Формулы
[Формулы](devtest-lab-manage-formulas.md) специфичны для лабораторий и не имеют механизма распределения. Члены лаборатории делают все разработки формул. 

## <a name="code-repository-based-resources"></a>Ресурсы, основанные на репозитории кода
Есть две различные функции, которые основаны на репозиториях кода, артефактах и средах. В этой статье представлены функции и способы наиболее эффективного настройки репозиторий и рабочего процесса, позволяющие настраивать доступные артефакты и среды на уровне организации или группы.  Этот рабочий процесс основан на стандартной [стратегии управления ветвлением исходного кода.](/azure/devops/repos/tfvc/branching-strategies-with-tfvc?view=azure-devops) 

### <a name="key-concepts"></a>Основные понятия
Исходная информация для артефактов включает метаданные, скрипты. Исходная информация для сред включает шаблоны метаданных и диспетчера ресурсов с любыми поддерживающими файлами, такими как скрипты PowerShell, dSC-скрипты, файлы и так далее.  

### <a name="repository-structure"></a>Структура репозитория  
Наиболее распространенной конфигурацией управления исходным кодом (SCC) является настройка многоуровневой структуры для хранения файлов кода (шаблоны управления ресурсами, метаданные и скрипты), которые используются в лабораториях. В частности, создание различных репозиторий для хранения ресурсов, управляемых различными уровнями бизнеса:   

- Ресурсы компании.
- Бизнес-подразделение/ресурсы в масштабах всего подразделения
- Ресурсы, специфичные для команд.

Каждый из этих уровней связан с другим репозиторием, где мастер-ветка должна быть качественной продукции. Ветви [branches](/azure/devops/repos/git/git-branching-guidance?view=azure-devops) в каждом репозитории будут предназначены для разработки этих конкретных ресурсов (артефактов или шаблонов). Эта структура хорошо согласуется с DevTest Labs, так как вы можете легко подключить несколько репозиторий и несколько ветвей одновременно с лабораториями организации. Имя репозитория включено в пользовательский интерфейс (UI), чтобы избежать путаницы при наличии одинаковых имен, описания и издателя.
     
На следующей диаграмме показаны два репозитория: репозиторий компании, который обслуживается ИТ-отделом, и репозиторий разделения, поддерживаемый подразделением R&D.

![Выборочная распределительной и совместная среда разработки](./media/best-practices-distributive-collaborative-dev-env/distributive-collaborative-dev-env.png)
   
Эта многоуровневая структура позволяет развиваться, сохраняя при этом более высокий уровень качества в основной ветке, а наличие нескольких репозиторий, подключенных к лаборатории, обеспечивает большую гибкость.
