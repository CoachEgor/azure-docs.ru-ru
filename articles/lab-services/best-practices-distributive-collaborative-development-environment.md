# <a name="best-practices-for-distributed-and-collaborative-development-of-azure-devtest-labs-resources"></a>Рекомендации по разработке распределенных и совместной работы Azure DevTest Labs ресурсов
Распределенные совместной разработки позволяет различными группами или людьми для разработки и обслуживания базовый код. Для достижения успеха процесс разработки зависит от возможности создания, совместного использования и интегрировать сведения. Этот принцип разработки ключа может использоваться в Azure DevTest Labs. Существует несколько типов ресурсов в лаборатории, которые часто распределяются между разных labs, в пределах предприятия. Различные типы ресурсов ориентированы на две области:

- Ресурсы, которые внутренне хранятся в лаборатории (на основе лаборатории)
- Ресурсы, которые хранятся в [внешних репозиториях, подключенных к лаборатории](devtest-lab-add-artifact-repo.md) (код основе репозитория). 

В этом документе описаны рекомендации, позволяющие совместной работы и распространения между несколькими командами гарантией того, настройки и качество на всех уровнях.

## <a name="lab-based-resources"></a>Ресурсы на основе лаборатории

### <a name="custom-virtual-machine-images"></a>Настраиваемые образы виртуальных машин
Может быть обычным источником пользовательских образов, развернутых на лабораторные занятия по ночам. Подробные сведения см. в разделе [Image factory](image-factory-create.md).    

### <a name="formulas"></a>Формулы
[Формулы](devtest-lab-manage-formulas.md) являются специфичными для лаборатории и не имеет механизма распространения. Элементы лаборатории выполняют вся разработка формул. 

## <a name="code-repository-based-resources"></a>Ресурсы на основе репозитория кода
Существуют две различные функции, основанные на репозиториев кода, артефакты и среды. В этой статье приведены функции, а также наиболее эффективно настроить репозитории и рабочий процесс чтобы обеспечить возможность настройки доступных артефактов и средах на уровне организации или команды.  Этот рабочий процесс основан на стандарте [исходного кода управлять стратегии ветвления](/devops/repos/tfvc/branching-strategies-with-tfvc?view=azure-devops). 

### <a name="key-concepts"></a>Основные понятия
Сведения об источнике для артефактов входят метаданные, скрипты. Сведения об источнике для сред включает в себя метаданные и шаблонов Resource Manager с все вспомогательные файлы, такие как сценарии PowerShell, скриптов DSC, ZIP-файлы и т. д.  

### <a name="repository-structure"></a>Структура репозитория  
Самая распространенная конфигурация, для управления исходным кодом (SCC) — настроить структуру многоуровневого для хранения файлов кода (шаблоны Resource Manager, метаданные и сценарии), которые позволяют в лабораториях. В частности создайте различные репозитории для хранения ресурсов, которые управляются различные уровни бизнес:   

- Ресурсы для всей компании.
- Единица/деления wide ресурсы для бизнеса
- Ресурсы для конкретной команды.

Каждый из этих уровней свяжите другого репозитория, где главной ветви должны быть производственного качества. [Ветви](/devops/repos/git/git-branching-guidance?view=azure-devops) в каждый репозиторий будет для разработки этих конкретных ресурсов (артефакты или шаблоны). Эта структура выравнивает с DevTest Labs, как можно с легкостью подключать несколько репозиториев и несколько ветвей в то же время, чтобы labs организации. Имя репозитория включается в пользовательский интерфейс (UI), чтобы избежать путаницы при наличии одинаковое имя, описание и издатель.
     
На следующей схеме показаны два репозитория: репозиторий компании, который обслуживается подразделения ИНФОРМАЦИОННЫХ и в репозиторий деления R & D деления.

![Пример среды разработки, распределенная и совместной работы](./media/best-practices-distributive-collaborative-dev-env/distributive-collaborative-dev-env.png)
   
Эта многоуровневая структура позволяет для разработки приложений, обеспечивая более высокий уровень качества в главную ветвь, хотя несколько репозиториев, подключенных к лаборатории обеспечивает большую гибкость.
