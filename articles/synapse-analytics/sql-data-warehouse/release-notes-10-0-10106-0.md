## <a name="check-your-azure-synapse-version"></a>Проверьте версию Azure Synapse

По мере развертывания новых функций во всех регионах проверьте развернутую версию в экземпляре и последние примечания к выпуску для доступности функций. Чтобы проверить версию, подключитесь к пулу S'L через студию управления серверами S'L (SSMS) и запустите, `SELECT @@VERSION;` чтобы вернуть текущую версию.

Используйте идентифицированную версию, чтобы подтвердить, какой выпуск был применен к вашему пулу S'L. Дата вывода определяет месяц выпуска, применяемый к пулу S'L.

> [!NOTE]
> Имя продукта, возвращенное SELECT,@VERSION будет меняться от хранилища данных Microsoft Azure S'L на Azure Synapse Analytics. Мы направим предварительное уведомление до того, как будет внесено изменение. Это изменение актуально для клиентов, которые сравнивают название@VERSION продукта с результатом SELECT в коде приложения. Чтобы избежать изменения кода приложения из-за ребрендинга продукта, пожалуйста, используйте эти команды для запроса SERVERPROPERTY для названия и версии продукта базы данных: Чтобы вернуть номер версии XX. X.XXXXX. X (без названия продукта) использует эту команду:
>
> ```sql
> SELECT SERVERPROPERTY('ProductVersion')
>
> --To return engine edition, use this command that returns 6 for Azure Synapse Analytics (Formerly SQL Data Warehouse):
>
> SELECT SERVERPROPERTY('EngineEdition')
> ```

Это позволяет последующим выполнениям запросов получать результаты непосредственно из сохраняемого кэша, поэтому повторная вычисления не требуется. Для обеспечения безопасности пользователи могут получить доступ к кэшированным результатам только в том случае, если у них есть те же разрешения на доступ к данным, что и у пользователей, создающих кэшированные результаты. Для получения дополнительной информации см. [Настройка производительности с кэшированием набора результатов.](performance-tuning-result-set-caching.md) Применяется к версии 10.0.10783.0 или выше.|

Аналогичным образом, экспорт данных из Azure Synapse в Azure Storage, защищенный VNet, также поддерживается через Polybase.<br/><br/>Для получения дополнительной информации о конечных точках службы VNet в Azure Synapse обратитесь к [сообщению в блоге](https://azure.microsoft.com/blog/general-availability-of-vnet-service-endpoints-for-azure-sql-data-warehouse/) или [документации](../../sql-database/sql-database-vnet-service-endpoint-rule-overview.md?toc=/azure/synapse-analytics/sql-data-warehouse/toc.json&bc=/azure/synapse-analytics/sql-data-warehouse/breadcrumb/toc.json).|
|**Автоматический мониторинг производительности (предварительная версия)**|[Магазин запросов](/sql/relational-databases/performance/monitoring-performance-by-using-the-query-store?toc=/azure/synapse-analytics/sql-data-warehouse/toc.json&bc=/azure/synapse-analytics/sql-data-warehouse/breadcrumb/toc.json&view=azure-sqldw-latest) теперь доступен в Предварительный просмотр в s'L Analytics в Azure Synapse. Хранилище запросов упрощает устранение проблем с производительностью запросов за счет отслеживания запросов, планов запросов, статистики среды выполнения и журнала запросов, с помощью которых можно выполнять мониторинг активности и производительности хранилища данных. Хранилище запросов — это набор внутренних хранилищ и динамических административных представлений (DMV), с помощью которых можно:<br/><br/>&bull;&nbsp; Определение и настройка лучших запросов, потребляющих ресурсы<br/>&bull;&nbsp; Выявление и улучшение незапланированных рабочих нагрузок<br/>&bull;&nbsp; Оценка производительности запроса и влияния на план с помощью изменений в статистике, индексах или размере системы (настройка DWU)<br/>&bull;&nbsp; Посмотреть полный текст запроса для всех выполненных запросов<br/><br/>Хранилище запросов содержит три фактических хранилища:<br/>&bull;&nbsp; Хранилище плана для сохранения информации о плане выполнения<br/>&bull;&nbsp; Хранилище статистики времени выполнения для сохранения информации о статистике выполнения<br/>&bull;&nbsp; Статистика ожидания хранит информацию о сохраняющейся статистике ожидания.<br/><br/>Аналитика s'L в Azure Synapse управляет этими магазинами автоматически и предоставляет неограниченное количество запросов, которые были запрошены в течение последних семи дней без дополнительной платы. Чтобы включить хранилище запросов, просто выполните инструкцию T-SQL ALTER DATABASE: <br/>sql ----ALTER DATABASE «DatabaseName» SET QUERY_STORE » ON;-------Для получения дополнительной информации о магазине запросов, см. статью, [мониторинг производительности с помощью магазина запросов](/sql/relational-databases/performance/monitoring-performance-by-using-the-query-store), и DMV магазина запросов, таких как [sys.query_store_query](/sql/relational-databases/system-catalog-views/sys-query-store-query-transact-sql?toc=/azure/synapse-analytics/sql-data-warehouse/toc.json&bc=/azure/synapse-analytics/sql-data-warehouse/breadcrumb/toc.json&view=azure-sqldw-latest). Вот [запись блога](https://azure.microsoft.com/blog/automatic-performance-monitoring-in-azure-sql-data-warehouse-with-query-store/) с объявлением о выпуске.|
|**Более низкие вычислительные уровни для аналитики S'L**|Аналитика S'L в Azure Synapse теперь поддерживает более низкие вычислительные уровни. Клиенты могут ознакомиться с ведущими функциями, гибкостью и безопасностью Azure Synapse, начиная со 100 cDWU[(единицы хранилища данных)](what-is-a-data-warehouse-unit-dwu-cdwu.md)и масштабироваться до 30 000 cDWU за считанные минуты. Начиная с середины декабря 2018 года клиенты могут использовать преимущества производительности и гибкость 2-го поколения на низких уровнях вычислений в [этих регионах](gen2-migration-schedule.md#automated-schedule-and-region-availability-table). Поддержка для остальных регионов станет доступна в течение 2019 года.<br/><br/>Обеспечив точку входа для хранилищ данных следующего поколения, корпорация Майкрософт открыла возможности для ориентированных на прибыль пользователей, которые хотят оценить все преимущества безопасного, высокопроизводительного хранилища данных, не тратя время на поиски наиболее подходящей для них пробной среды. Пользователи могут начать со 100 cDWU (ниже чем текущая точка входа в 500 cDWU). Аналитика продолжает поддерживать паузу и возобновить работу и выходит за рамки просто гибкости в вычислениях. 2 поколение также поддерживает неограниченную емкость хранилища столбцов, а также в 2,5 раза больше памяти для каждого запроса, до 128 одновременных запросов и функции [адаптивного кэширования](https://azure.microsoft.com/blog/adaptive-caching-powers-azure-sql-data-warehouse-performance-gains/). Эти функции в среднем приносят в пять раз больше производительности по сравнению с тем же хранилищем данных Unit на Gen1 по той же цене. Геоизбыточные резервные копии являются стандартными для 2-го поколения и обеспечивают встроенную гарантированную защиту данных. Аналитика s'L в Azure Synapse готова к масштабированию, когда вы находитесь.|
|**Слияния в фоновом режиме ColumnStore**|По умолчанию данные Azure S'L Data хранят данные в формате columnar, с микроразделами, называемыми [гребными группами.](sql-data-warehouse-memory-optimizations-for-columnstore-compression.md) Иногда, из-за ограничений памяти при построении индекса или времени загрузки данных, группы строк могут быть сжаты в размере (меньше оптимального размера миллионов строк). Группы строк могут также стать фрагментированными из-за удалений. Небольшие или фрагментированные группы строк приводят к более высокому потреблению памяти, а также к неэффективному выполнению запросов. С помощью этого выпуска задача фонового обслуживания в магазине столбцов объединяет небольшие сжатые группы строк для создания больших групп строк, чтобы лучше использовать память и ускорить выполнение запроса.
