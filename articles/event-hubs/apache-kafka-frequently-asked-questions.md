# <a name="frequently-asked-questions---event-hubs-for-apache-kafka"></a>Часто задаваемые вопросы - Концентраторы событий для Apache Kafka 
В этой статье содержатся ответы на некоторые часто задаваемые вопросы о переходе в центры событий для Apache Kafka.

## <a name="do-you-run-apache-kafka"></a>Ты управляешь Апач Кафкой?

Нет.  Мы выполняем операции API Kafka против инфраструктуры Event Hubs.  Поскольку существует тесная взаимосвязь между функциональностью Apache Kafka и Event Hubs AM-P (то есть производить, получать, управления и так далее),мы можем довести известную надежность концентраторов событий в пространстве Kafka PaaS.

## <a name="event-hubs-consumer-group-vs-kafka-consumer-group"></a>Группа потребителей Event Hubs против группы потребителей Кафки
В чем разница между группой потребителей Event Hub и группой потребителей Kafka в концентрах событий? Группы потребителей Kafka в концентрах событий полностью отличаются от стандартных групп потребителей Event Hubs.

**Группы потребителей Event Hubs**

- Они управляются с помощью создания, извлечения, обновления и удаления (CRUD) операций через шаблоны портала, SDK или Azure Resource Manager. Группы потребителей Event Hubs не могут быть автоматически созданы.
- Они являются детьми, в которых участвуют центр событий. Это означает, что одно и то же имя группы потребителей может быть повторно использовано между концентраторами событий в одном и том же пространстве имен, поскольку они разделены на сущности.
- Они не используются для хранения смещений. Организованное потребление АМЗП осуществляется с помощью внешнего офсетного хранилища, например, Хранилища Azure.

**Группы потребителей Кафки**

- Они автоматически созданы.  Группы Кафки могут управляться через AI потребительской группы Kafka.
- Они могут хранить смещения в службе концентраторов событий.
- Они используются в качестве ключей в том, что эффективно смещения ключ-значение магазина. Для уникальной `group.id` пары `topic-partition`и, мы храним смещения в Azure Storage (3x репликации). Пользователи event Hubs не несут дополнительных затрат на хранение от хранения смещений Kafka. Смещения манипулируются через AI группы потребителей Kafka, но *учетные записи* с компенсационным хранилищем не являются непосредственно видимыми или манипулируемыми для пользователей Event Hub.  
- Они охватывают пространство имен. Использование одного и того же названия группы Kafka для нескольких приложений по нескольким темам означает, что все приложения и их клиенты Kafka будут сбалансированы всякий раз, когда только одно приложение нуждается в перебалансировке.  Выберите имена групп с умом.
- Они полностью отличаются от групп потребителей Event Hubs. Вам не нужно использовать "$Default", и вам **не** нужно беспокоиться о клиентах Kafka, вмешивающихся в рабочие нагрузки АМЗП.
- Они не просматриваемы на портале Azure. Информация о группе потребителей доступна через AIS Kafka.
