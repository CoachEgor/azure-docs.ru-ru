# <a name="private-cloud-vmware-components"></a>Компоненты Private Cloud VMware

Частное облако — это изолированный стек VMware (хосты ESXi, vCenter, vSAN и NSX), управляемый сервером vCenter в домене управления.  Сервис CloudSimple позволяет развертывать VMware на голой металлической инфраструктуре Azure в местах Azure.  Частные облака интегрированы с остальной частью облака Azure.  Частное облако развернуто со следующими компонентами стека VMware:

* **VMware ESXi -** Гипервизор на Azure выделенные узлы
* **VMware vCenter -** Прибор для централизованного управления частной облачной средой vSphere
* **VMware vSAN -** Гиперконвергентное инфраструктурное решение
* **Центр обработки данных VMware NSX -** Виртуализация сети и программное обеспечение безопасности  

## <a name="vmware-component-versions"></a>Версии компонентов VMware

Стек Private Cloud VMware развертывается со следующей версией программного обеспечения.

| Компонент | Версия | Лицензированная версия |
|-----------|---------|------------------|
| Esxi | 6.7U2 | Предприятие Плюс |
| vCenter | 6.7U2 | vCenter Standard |
| vSAN | 6.7 | Enterprise |
| Центр обработки данных NSX | 2.4.1 | Дополнительно |

## <a name="esxi"></a>Esxi

VMware ESXi устанавливается на подготовленные узлы CloudSimple при создании частного облака.  ESXi предоставляет гипервизор для развертывания рабочих нагрузок виртуальных машин (VM).  Узлы обеспечивают гиперконвергентную инфраструктуру (вычисление и хранение) в частном облаке.  Узлы являются частью кластера vSphere в частном облаке.  Каждый узла имеет четыре интерфейса физических сетей, подключенных к сети подкладок.  Два физических сетевых интерфейса используются для создания **vSphere Distributed Switch (VDS)** на vCenter и два используются для создания **виртуального распределенного коммутатора, управляемого NSX (N-VDS).**  Сетевые интерфейсы настроены в активно-активном режиме для высокой доступности.

## <a name="vcenter-server-appliance"></a>vCenter серверная приборная контора

серверный прибор vCenter (VCSA) обеспечивает функции аутентификации, управления и оркестровки решения VMware от CloudSimple. VCSA со встроенным контроллером служб платформы (PSC) развертывается при создании частного облака.  VCSA развертывается в кластере vSphere, который создается при развертывании частного облака.  Каждое частное облако имеет свой собственный VCSA.  Расширение частного облака добавляет узлы к VCSA в частном облаке.

### <a name="vcenter-single-sign-on"></a>vCenter одиночный входной

Встроенный контроллер услуг платформы на VCSA связан с **доменом vCenter Single Sign-On.**  Доменное имя **cloudsimple.local**.  Пользователь **CloudOwner@cloudsimple.com** по умолчанию создан для доступа к vCenter.  Вы можете добавить свои активные источники идентификации каталога [для vCenter.](set-vcenter-identity.md)

## <a name="vsan-storage"></a>vSAN хранения

Частные облака создаются с полностью настроенным всефлядным хранилищем vSAN, локальным для кластера.  Минимум три узла одного и того же SKU необходимы для создания кластера vSphere с хранилищем данных vSAN.  Де-дублирование и сжатие включены в хранилище данных vSAN по умолчанию.  На каждом уотдете кластера vSphere создаются две дисковые группы. Каждая группа дисков содержит один кэш-диск и три диска емкости.

Политика хранения данных по умолчанию vSAN создается в кластере vSphere и применяется к магазину данных vSAN.  Эта политика определяет, как объекты хранения VM резервируются и распределяются в хранилище данных, чтобы гарантировать необходимый уровень обслуживания.  Политика хранения определяет **отказы в переносимости (FTT)** и **метод толерантности к сбоям.**  Можно создать новые политики хранения данных и применить их к визажам. Для поддержания SLA, 25% запасных мощностей должны быть сохранены на vSAN datastore.  

### <a name="default-vsan-storage-policy"></a>Политика хранения данных по умолчанию vSAN

В таблице ниже показаны параметры политики хранения vSAN по умолчанию.

| Количество узлов в кластере vSphere | FTT | Метод толерантности к отказу |
|------------------------------------|-----|--------------------------|
| 3 и 4 узла | 1 | RAID 1 (зеркало) - создает 2 экземпляра |
| От 5 до 16 узлов | 2 | RAID 1 (зеркало) - создает 3 экземпляра |

## <a name="nsx-data-center"></a>Центр обработки данных NSX

Центр обработки данных NSX предоставляет возможности виртуализации сети, микросегментации и сетевой безопасности в частном облаке.  Вы можете настроить все службы, поддерживаемые Центром обработки данных NSX, в частном облаке через NSX.  При создании частного облака устанавливаются и настраиваются следующие компоненты NSX.

* Менеджер NSXT
* Транспортные зоны
* Профиль хоста и края uplink
* Логический переключатель для транспортного транспорта edge, Ext1, и Ext2
* ПУЛ IP для транспортного узла ESXi
* IP бассейн для краевого транспортного узла
* Краевые узлы
* DrS Anti-affinity правило для контроллера и Edge VMs
* Маршрутизатор уровня 0
* Включить BGP на маршрутизаторе Tier0

## <a name="vsphere-cluster"></a>vСфера кластера

Хосты ESXi настроены как кластер для обеспечения высокой доступности частного облака.  При создании частного облака компоненты управления vSphere развертываются в первом кластере.  Для компонентов управления создается пул ресурсов, и все управляющие ввовы развертываются в этом пуле ресурсов. Первый кластер не может быть удален для сокращения частного облака.  vSphere кластер обеспечивает высокую доступность для VMs с использованием **vSphere HA**.  Отказы в переносительности основаны на количестве доступных узлов в кластере.  Вы можете использовать ```Number of nodes = 2N+1``` ```N``` формулу, где количество сбоев терпеть.

### <a name="vsphere-cluster-limits"></a>vСфера кластера

| Ресурс | Ограничение |
|----------|-------|
| Минимальное количество узлов для создания частного облака (первый кластер vSphere) | 3 |
| Максимальное количество узлов в кластере vSphere в частном облаке | 16 |
| Максимальное количество узлов в частном облаке | 64 |
| Максимальное количество кластеров vSphere в частном облаке | 21 |
| Минимальное количество узлов в новом кластере vSphere | 3 |

## <a name="vmware-infrastructure-maintenance"></a>Обслуживание инфраструктуры VMware

Иногда необходимо вносить изменения в конфигурацию инфраструктуры VMware. В настоящее время эти интервалы могут происходить каждые 1-2 месяца, но частота, как ожидается, будет снижаться с течением времени. Этот тип обслуживания обычно может быть выполнен без прерывания нормального потребления служб CloudSimple. Во время интервала обслуживания VMware следующие службы продолжают функционировать без какого-либо воздействия:

* Самолет управления vMware и приложения
* vCenter доступ
* Все сети и хранилища
* Весь трафик Azure

## <a name="updates-and-upgrades"></a>Обновления и обновления

CloudSimple отвечает за управление жизненным циклом программного обеспечения VMware (ESXi, vCenter, PSC и NSX) в частном облаке.

Обновления программного обеспечения включают в себя:

* **Патчи**. Патчи безопасности или исправления ошибок, выпущенные VMware.
* **Обновления**. Незначительное изменение версии компонента стека VMware.
* **Обновления**. Основные изменения версии компонента стека VMware.

CloudSimple тестирует критический патч безопасности, как только он становится доступным от VMware. В течение недели CloudSimple выкатывает патч безопасности в частные облачные среды.

CloudSimple обеспечивает ежеквартальные обновления технического обслуживания компонентов программного обеспечения VMware. Когда новая основная версия программного обеспечения VMware доступна, CloudSimple работает с клиентами, чтобы координировать подходящее окно обслуживания для обновления.
