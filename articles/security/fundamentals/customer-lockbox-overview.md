# <a name="customer-lockbox-for-microsoft-azure"></a>Блокировка клиентов для Microsoft Azure

> [!NOTE]
> Чтобы использовать эту функцию, ваша организация должна иметь [план поддержки Azure](https://azure.microsoft.com/support/plans/) с минимальным уровнем **разработчика.**

Клиентская блокировка для Microsoft Azure предоставляет клиентам интерфейс для рассмотрения и утверждения или отклонения запросов на доступ к данным клиентов. Он используется в тех случаях, когда инженеру Майкрософт необходимо получить доступ к данным клиентов во время запроса поддержки.

В этой статье рассказывается о том, как запросы клиентских блокировок инициируются, отслеживаются и хранятся для последующих обзоров и аудитов.

Клиент Lockbox теперь широко доступны и в настоящее время включен для удаленного доступа к рабочему столу виртуальных машин.

## <a name="workflow"></a>Рабочий процесс

Следующие шаги намечают типичный рабочий процесс для запроса блокировки клиентов.

1. У кого-то в организации есть проблемы с рабочей нагрузкой Azure.

2. После того, как этот человек устраняет проблему, но не может ее устранить, они открывают билет поддержки с [портала Azure.](https://ms.portal.azure.com/signin/index/?feature.settingsportalinstance=mpac) Билет назначается инженеру службы поддержки клиентов Azure.

3. Инженер службы поддержки Azure просматривает запрос службы и определяет дальнейшие шаги по устранению проблемы.

4. Если инженер службы поддержки не может устранить проблему с помощью стандартных инструментов и телеметрии, следующим шагом будет запрос повышенных разрешений с помощью службы доступа Just-In-Time (JIT). Этот запрос может быть получен от первоначального инженера поддержки. Или это может быть от другого инженера, потому что проблема перерастает в команду Azure DevOps.

5. После того, как запрос на доступ будет отправлен инженером Azure, служба Just-In-Time оценивает запрос с учетом таких факторов, как:
    - Сфера охвата ресурса
    - Является ли запрашивающий идентификационный или использует многофакторную аутентификацию
    - Уровни разрешений

    На основе правила JIT этот запрос может также включать одобрение внутренних утверждений корпорации Майкрософт. Например, утверждение может быть ведущим отделом поддержки клиентов или менеджером DevOps.

6. Когда запрос требует прямого доступа к данным клиентов, инициируется запрос на блокировку клиента. Например, удаленный доступ к виртуальному компьютеру клиента.

    Запрос теперь находится в **состоянии уведомления клиента,** ожидая одобрения клиента, прежде чем предоставить доступ.

7. В организации клиента пользователь, который имеет [роль Владельца](../../role-based-access-control/rbac-and-directory-admin-roles.md#azure-rbac-roles) для подписки Azure, получает электронное письмо от корпорации Майкрософт, чтобы уведомить его о готовящемся запросе доступа. Для запросов о блокировке клиентов этот человек является указанным утверждением.

    Пример сообщения электронной почты:

    ![Лазурный клиентский блокнот - уведомление по электронной почте](./media/customer-lockbox-overview/customer-lockbox-email-notification.png)

8. Уведомление по электронной почте содержит ссылку на лезвие **блокировки клиента** на портале Azure. Используя эту ссылку, назначенный одобритель регистрируется на портале Azure для просмотра любых ожидающих запросов, которые есть у их организации для lockbox клиента:

    ![Лазурный клиентский блокбокс - целевая страница](./media/customer-lockbox-overview/customer-lockbox-landing-page.png)

   Запрос остается в очереди клиента в течение четырех дней. По истечении этого времени автоматически истечет срок действия запроса на доступ, и инженерам Майкрософт не предоставляется доступ.

9. Чтобы получить подробную информацию о готовящемся запросе, назначенный утверждая может выбрать запрос на блокировку из **ожидающих запросов:**

    ![Локбокс клиента Azure - просмотр ожидающего запроса](./media/customer-lockbox-overview/customer-lockbox-pending-requests.png)

10. Назначенный одобритель также может выбрать **ИДЕНтификатор SERVICE REQUEST** для просмотра запроса на билет поддержки, который был создан исходным пользователем. Эта информация предоставляет контекст, почему корпорация Майкрософт занимается, и историю сообщаемой проблемы. Пример:

    ![Локтос клиента Azure - просмотр запроса на билет поддержки](./media/customer-lockbox-overview/customer-lockbox-support-ticket.png)

11. После рассмотрения запроса назначенный утверждающий выбирает **Утверждение** или **Отрицание:**

    ![Локток клиента Azure - выберите Утвердить или отказать](./media/customer-lockbox-overview/customer-lockbox-approval.png)

    В результате отбора:
    - **Утверждение**: Доступ предоставляется инженеру Майкрософт. Доступ предоставляется на период по умолчанию в восемь часов.
    - **Отказать**: Повышенный запрос доступа инженером Майкрософт отклоняется, и никаких дальнейших действий не предпринимается.

Для целей аудита действия, выполненные в этом рабочем процессе, регистрируются в [журналах запросов на работу клиента Lockbox.](#auditing-logs)

## <a name="auditing-logs"></a>Журналы аудита

Журналы клиентских блокировок хранятся в журналах активности. На портале Azure выберите **журналы активности** для просмотра информации о аудите, связанной с запросами на блокировку клиентов. Можно отфильтровать для определенных действий, таких как:
- **Отказать в запросе на блокировку**
- **Создание запроса на блокировку**
- **Утверждение запроса на блокировку**
- **Срок действия запроса lockbox**

Например:

![Локтос клиента Azure - журналы активности](./media/customer-lockbox-overview/customer-lockbox-activitylogs.png)

## <a name="supported-services-and-scenarios-in-general-availability"></a>Поддерживаемые услуги и сценарии в общей доступности

Следующие службы и сценарии в настоящее время в общей доступности для клиентского lockbox.

### <a name="remote-desktop-access-to-virtual-machines"></a>Удаленный доступ к виртуальным компьютерам

В настоящее время клиентская lockbox включена для удаленных запросов доступа к настольным компьютерам. Поддерживаются следующие рабочие нагрузки:
- Платформа как услуга (PaaS) - Облачные службы Azure (роль веб-службы и работы)
- Инфраструктура как услуга (IaaS) - Windows и Linux (только менеджер ресурсов Azure)
- Виртуальный набор масштабов машины - Windows и Linux

> [!NOTE]
> Экземпляры IaaS Classic не поддерживаются lockbox customer. Если в экземплярах IaaS Classic работают рабочие нагрузки, мы рекомендуем перенести их из моделей развертывания Classic в диспетчерские ресурсы. Для инструкций [см.](../../virtual-machines/windows/migration-classic-resource-manager-overview.md)

#### <a name="detailed-audit-logs"></a>Подробные журналы аудита

Для сценариев, связанных с удаленным доступом к рабочему столу, можно использовать журналы событий Windows для просмотра действий, предпринятых инженером Майкрософт. Рассмотрите возможность использования Центра безопасности Azure для сбора журналов событий и копирования данных в рабочее пространство для анализа. Для получения дополнительной информации смотрите [сбор данных в Центре безопасности Azure](../../security-center/security-center-enable-data-collection.md).

## <a name="supported-services-and-scenarios-in-preview"></a>Поддерживаемые службы и сценарии в предварительном просмотре

Следующие службы в настоящее время в предварительном просмотре для клиентского lockbox:

- Хранилище Azure

- БД SQL Azure

- Azure Data Explorer

- Виртуальные машины (теперь также охватывающих доступ к свалкам памяти и управляемым дискам)

- Переводы подписки Azure

Чтобы включить блокбокс клиента для этих предварительных предложений для вашей организации, зарегистрируйтесь в [клиентском блокпоре для публичного предварительного просмотра Azure.](https://aka.ms/customerlockbox/insiderprogram)


## <a name="exclusions"></a>Исключения

Запросы на блокировку клиентов не срабатывают в следующих сценариях инженерной поддержки:

- Инженеру Майкрософт необходимо выполнять действия, выходящие за рамки стандартных операционных процедур. Например, для восстановления или восстановления служб в непредвиденных или непредсказуемых сценариях.

- Инженер Майкрософт получает доступ к платформе Azure как часть устранения неполадок и непреднамеренно имеет доступ к данным клиентов. Например, команда сети Azure выполняет устранение неполадок, что приводит к захвату пакетов на сетевом устройстве. Однако, если клиент зашифровал данные во время их передачи, инженер не может прочитать данные.

## <a name="next-steps"></a>Дальнейшие действия

Клиентская коробка автоматически доступна для всех клиентов, имеющих [план поддержки Azure](https://azure.microsoft.com/support/plans/) с минимальным уровнем **разработчика.**

Если у вас есть подходящий план поддержки, вам не требуется никаких действий для включения почтового ящика клиентов. Запросы на блокировку клиентов инициируются инженером корпорации Майкрософт, если это действие необходимо для выполнения поддержки билета, который подается от кого-то в вашей организации.
