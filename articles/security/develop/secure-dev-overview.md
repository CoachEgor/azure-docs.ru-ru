# <a name="secure-development-best-practices-on-azure"></a>Лучшие практики безопасной разработки на Azure
В этой серии статей представлены действия и элементы управления безопасности, которые необходимо учитывать при разработке приложений для облака. Рассматриваются этапы жизненного цикла разработки microsoft Security Development (SDL) и вопросы и концепции безопасности, которые необходимо учитывать на каждом этапе жизненного цикла. Цель состоит в том, чтобы помочь определить действия и службы Azure, которые можно использовать на каждом этапе жизненного цикла для разработки, разработки и развертывания более безопасного приложения.

Рекомендации в статьях взяты из нашего опыта работы с безопасностью Azure и опыта наших клиентов. Вы можете использовать эти статьи в качестве ссылки на то, что вы должны рассмотреть на определенном этапе вашего проекта разработки, но мы предлагаем вам также прочитать все статьи от начала до конца, по крайней мере один раз. Чтение всех статей знакомит вас с концепциями, которые вы, возможно, пропустили на более ранних этапах вашего проекта. Внедрение этих концепций до выпуска продукта может помочь вам создать безопасное программное обеспечение, удовлетворить требования к соответствию требованиям безопасности и снизить затраты на разработку.

Эти статьи предназначены для создания ресурса для разработчиков программного обеспечения, разработчиков и тестировщиков на всех уровнях, которые создают и развертывают безопасные приложения Azure.

## <a name="overview"></a>Обзор

Безопасность является одним из наиболее важных аспектов любого приложения, и это не простая вещь, чтобы получить право. К счастью, Azure предоставляет множество служб, которые помогут вам защитить приложение в облаке. В этих статьях рассматриваются действия и службы Azure, которые можно реализовать на каждом этапе жизненного цикла разработки программного обеспечения, чтобы помочь вам разработать более безопасный код и развернуть более безопасное приложение в облаке.

## <a name="security-development-lifecycle"></a>Жизненный цикл разработки защищенных приложений

Следование передовым методам для безопасной разработки программного обеспечения требует интеграции безопасности в каждый этап жизненного цикла разработки программного обеспечения, от анализа требований до технического обслуживания, независимо от методологии проекта[(водопад,](https://en.wikipedia.org/wiki/Waterfall_model) [гибкая,](https://en.wikipedia.org/wiki/Agile_software_development)или [DevOps](https://en.wikipedia.org/wiki/DevOps)). В связи с громкими утечками данных и использованием недостатков оперативной безопасности все больше разработчиков понимают, что безопасность должна быть решена на протяжении всего процесса разработки.

Чем позже вы исправите проблему в жизненном цикле разработки, тем больше это исправление будет стоить вам. Вопросы безопасности не являются исключением. Если вы игнорируете проблемы безопасности на ранних этапах разработки программного обеспечения, каждый следующий этап может унаследовать уязвимости предыдущего этапа. Конечный продукт будет иметь накопившиеся многочисленные проблемы безопасности и возможность нарушения. Обеспечение безопасности на каждом этапе жизненного цикла разработки позволяет вылавливать проблемы на ранних стадиях и позволяет снизить затраты на разработку.

Мы следим за этапами [жизненного цикла разработки Microsoft Security Development (SDL)](https://msdn.microsoft.com/library/windows/desktop/84aed186-1d75-4366-8e61-8d258746bopq.aspx) для внедрения функций и служб Azure, которые можно использовать для выполнения безопасных методов разработки программного обеспечения на каждом этапе жизненного цикла.

Фазы SDL:

  - Обучение
  - Требования
  - Конструирование
  - Реализация
  - Проверка
  - Release
  - Ответ

![Жизненный цикл разработки защищенных приложений](./media/secure-dev-overview/01-sdl-phase.png)

В этих статьях мы группируем этапы SDL в проектирование, разработку и развертывание.

## <a name="engage-your-organizations-security-team"></a>Привлечение группы безопасности вашей организации

В вашей организации может быть официальная программа безопасности приложений, которая помогает вам в обеспечении безопасности от начала до конца во время жизненного цикла разработки. Если в вашей организации есть группы по обеспечению безопасности и соответствию требованиям, не забудьте задействовать их, прежде чем приступить к разработке приложения. Спросите их на каждом этапе SDL, есть ли какие-либо задачи, которые вы пропустили.

Мы понимаем, что у многих читателей может не быть группы безопасности или соответствия требованиям. Эти статьи могут помочь вам в вопросах безопасности и решениях, которые необходимо рассмотреть на каждом этапе SDL.

## <a name="resources"></a>Ресурсы

Используйте следующие ресурсы, чтобы узнать больше о разработке безопасных приложений и помочь обезопасить приложения в Azure:

[Microsoft Security Development Lifecycle (SDL)](https://msdn.microsoft.com/library/windows/desktop/84aed186-1d75-4366-8e61-8d258746bopq.aspx) - SDL — это процесс разработки программного обеспечения от корпорации Майкрософт, который помогает разработчикам создавать более безопасное программное обеспечение. Это помогает удовлетворить требования соответствия безопасности при одновременном снижении затрат на разработку.

[Проект по обеспечению безопасности приложений Open Web (OWASP)](https://www.owasp.org/index.php/Main_Page) - OWASP — это интернет-сообщество, которое производит свободно доступные статьи, методологии, документацию, инструменты и технологии в области безопасности веб-приложений.

[Pushing Left, Like a Boss](https://code.likeagirl.io/pushing-left-like-a-boss-part-1-80f1f007da95?WT.mc_id=docs-blog-tajanca) - Серия статей в Интернете, в которых излагаются различные типы действий по обеспечению безопасности приложений, которые разработчики должны завершить для создания более безопасного кода.

[Платформа идентификации Майкрософт](../../active-directory/develop/index.yml) — идентификационная платформа Майкрософт — это эволюция службы идентификации Azure AD и платформы разработчиков. Это полнофункционная платформа, которая состоит из службы аутентификации, библиотек с открытым исходным кодом, регистрации приложений и конфигурации, полной документации разработчика, образцов кода и другого контента разработчика. Платформа Microsoft по идентификации поддерживает отраслевые протоколы, такие как OAuth 2.0 и OpenID Connect.

[Рекомендации по безопасности для решений Azure](https://azure.microsoft.com/resources/security-best-practices-for-azure-solutions/) — набор рекомендаций по безопасности для разработки, развертывания и управления облачными решениями с помощью Azure. Этот документ предназначен для использования в ресурсе для ИТ-специалистов. К ним могут относиться проектировщики, архитекторы, разработчики и тестировщики, занимающиеся созданием и развертыванием безопасных решений в Azure.

[Планы безопасности и соответствия требованиям в Azure](https://servicetrust.microsoft.com/ViewPage/BlueprintOverview) — планы безопасности и соответствия Azure — это ресурсы, которые помогут создавать и запускать облачные приложения, соответствующие строгим правилам и стандартам.
