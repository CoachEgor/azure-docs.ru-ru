# <a name="secure-azure-computing-architecture"></a>Безопасная архитектура вычислительных ресурсов Azure

Данные учреждений Клиенты, которые развертывают рабочие нагрузки в Azure, запросили рекомендации по настройке защищенных виртуальных сетей и настройке средств и служб безопасности, оговоренных стандартами и практикой по требованию. 

Информационный орган по защите облачных вычислений (DISA) опубликовал [документ с требованиями к функциональным требованиям для архитектуры безопасности (фрд)](https://dl.dod.cyber.mil/wp-content/uploads/cloud/pdf/SCCA_FRD_v2-9.pdf) в 2017. СККА описывает функциональные цели для обеспечения безопасности точек подключения в сети информационной системы защиты (ДИСН) и коммерческих поставщиков облачных служб. СККА также описывает, как владельцы миссии защищают облачные приложения на границе подключения. Каждая сущность "DoD", которая подключается к коммерческому облаку, должна следовать правилам, установленным в СККА ФРД.
 
СККА состоит из четырех компонентов:
 
- Граница облачной точки доступа (БКАП)
- Стек безопасности виртуального центра обработки данных (ВДСС)
- Управляемая служба Virtual Datacenter (ВДМС)
- Диспетчер доверенных облачных учетных данных (ТККМ) 

Корпорация Майкрософт разработала решение, которое соответствует требованиям СККА для рабочих нагрузок IL4 и IL5, работающих в Azure. Это решение, связанное с Azure, называется безопасной вычислительной архитектурой Azure (Сака). Клиенты, которые развертывают Сака, соответствуют СККА ФРД. Они могут позволить клиентам с ограниченным подключением перемещать рабочие нагрузки в Azure после их подключения.

Рекомендации и архитектуры СККА относятся только к клиентам с нелегальными требованиями, но последние редакции Сака помогают из гражданских требованиям доверенного подключения к Интернету (КРЕСТИКИ). Последние редакции также помогают коммерческим клиентам реализовать защищенную сеть периметра для защиты своих сред Azure.


## <a name="secure-cloud-computing-architecture-components"></a>Безопасные облачные вычислительные компоненты

### <a name="bcap"></a>БКАП

Цель БКАП заключается в защите ДИСН от атак, исходящих в облачной среде. БКАП выполняет обнаружение вторжений и их предотвращение. Он также отфильтровывает несанкционированный трафик. Этот компонент можно разместить совместно с другими компонентами СККА. Рекомендуется развертывать этот компонент с помощью физического оборудования. Требования к безопасности БКАП перечислены в следующей таблице.

#### <a name="bcap-security-requirements"></a>Требования к безопасности БКАП

![Матрица требований БКАП](media/bcapreqs.png)


### <a name="vdss"></a>ВДСС

Цель ВДСС заключается в защите приложений, ответственных за DoD, которые размещены в Azure. ВДСС выполняет основную часть операций безопасности в СККА. Он проводит проверку трафика для защиты приложений, работающих в Azure. Этот компонент может быть предоставлен в среде Azure.

#### <a name="vdss-security-requirements"></a>Требования к безопасности ВДСС

![Матрица требований ВДСС](media/vdssreqs.png)

### <a name="vdms"></a>ВДМС

Цель ВДМС заключается в обеспечении безопасности узла и общих служб центра обработки данных. Функции ВДМС могут быть запущены в центре СККА, или владелец миссии может развернуть их в отдельной подписке Azure. Этот компонент может быть предоставлен в среде Azure.

#### <a name="vdms-security-requirements"></a>Требования к безопасности ВДМС

![Матрица требований ВДМС](media/vdmsreqs.png)


### <a name="tccm"></a>ТККМ

ТККМ является бизнес-ролью. Это лицо отвечает за управление СККА. Их обязанности: 

- Установите планы и политики для доступа учетной записи к облачной среде. 
- Убедитесь, что Управление удостоверениями и доступом работает правильно. 
- Обслуживание облачного плана управления учетными данными. 

Это лицо определяется официальным уполномоченным. БКАП, ВДСС и ВДМС предоставляют возможности, которые ТККМ должны выполнять свою работу.

## <a name="saca-components-and-planning-considerations"></a>Компоненты Сака и вопросы планирования 

Эталонная архитектура Сака предназначена для развертывания компонентов ВДСС и ВДМС в Azure и для включения ТККМ. Эта архитектура является модульной. Все части ВДСС и ВДМС могут находиться в централизованном центре. Некоторые элементы управления могут быть выполнены в пространстве владельца или даже в локальной среде. Корпорация Майкрософт рекомендует совместное размещение компонентов ВДСС и ВДМС в Центральной виртуальной сети, к которой могут подключаться все владельцы миссии. Эта архитектура показана на схеме ниже. 


![Схема эталонной архитектуры Сака](media/sacav2generic.png)

При планировании стратегии СККА соответствия требованиям и технической архитектуры следует учесть следующие темы с начала, поскольку они влияют на всех клиентов. Следующие проблемы возникают с клиентами с помощью DoD и замедляют планирование и выполнение. 

#### <a name="which-bcap-will-your-organization-use"></a>Какие БКАП будет использовать ваша организация?
   - DISA БКАП:
        - У DISA есть две рабочие Бкапс в пятиугольник и в Camp Робертс, CA. Третий план скоро станет доступен в сети. 
        - У Бкапс DISA есть каналы Azure ExpressRoute в Azure, которые могут использоваться клиентами с нелегальными подключениями. 
        - DISA имеет корпоративный сеанс пиринга Майкрософт для клиентов с нежелательной организацией, которые хотят подписываться на средства Microsoft Software как услуга (SaaS), такие как Office 365. С помощью DISA БКАП можно включить подключение и пиринг к экземпляру Сака. 
    - Создайте собственный БКАП:
        - Этот вариант требует аренды пространства в совместно размещенном центре обработки данных и настройки канала ExpressRoute в Azure. 
        - Этот параметр требует дополнительного утверждения. 
        - Из-за дополнительного утверждения и физической сборки этот параметр занимает больше всего времени. 
    - Рекомендуется использовать DISA БКАП. Этот вариант доступен, имеет встроенную избыточность и имеет клиентов, которые работают в рабочей среде уже сегодня.
- IP-пространство с маршрутизацией DoD:
    - Необходимо использовать IP-пространство с маршрутизацией по требованию на границе. Можно использовать NAT для подключения этих пространств к частному IP-пространству в Azure.
    - Чтобы получить IP-пространство, обратитесь в центр информации по сети DoD. Он необходим в рамках отправки системы или процесса утверждения сети (привязки) с помощью DISA. 
    - Если вы планируете использовать NAT для подключения к закрытому адресному пространству в Azure, вам потребуется не менее/24 подсеть адресного пространства, назначенного из сетевой карты для каждого региона, в котором планируется развернуть Сака.
- Избыточность:
    - Разверните экземпляр Сака по меньшей мере в двух регионах. В облаке с помощью DoD вы развернете его в обоих доступных регионах DoD.
    - Подключитесь по меньшей мере к двум Бкапс через отдельные каналы ExpressRoute. Затем оба подключения ExpressRoute можно связать с экземпляром Сака каждого региона. 
- Требования для конкретного компонента с вызовом DoD:
    - Есть ли у вашей организации какие – либо особые требования за пределами требований СККА? Некоторые организации имеют определенные требования к IP-адресам.
- Сака является модульной архитектурой:
    - Используйте только те компоненты, которые необходимы для вашей среды. 
        - Развертывайте сетевые виртуальные устройства на одном или многоуровневом уровне.
        - Используйте интегрированные IP-адреса или собственные IP-адреса.
- Уровень влияния приложений и данных на DoD:
    - Если у вас есть возможность запускать приложения в регионах Microsoft IL5, создайте экземпляр Сака в IL5. Экземпляр можно использовать перед IL4 приложениями и IL5. Экземпляр Сака IL4 перед приложением IL5, скорее всего, не получит аккредитации.

## <a name="most-common-deployment-scenario"></a>Наиболее распространенный сценарий развертывания

 Некоторые клиенты корпорации Майкрософт прошли полное развертывание или по крайней мере этапы планирования сред Сака. Их опыт выявил понимание наиболее распространенного сценария развертывания. На следующей схеме показана наиболее распространенная архитектура. 


![Схема эталонной архитектуры Сака](media/sacav2commonscenario.png) 


Как видно из диаграммы, клиенты с помощью DoD обычно подписываются на два DISA Бкапс. Один из них находится на Западном побережье, а другой — в Восточном регионе. Частный пиринг ExpressRoute включен в Azure в каждом расположении DISA БКАП. Затем эти одноранговые узлы ExpressRoute связываются с шлюзом виртуальной сети в центральных регионах Microsoft для восточной части и центра DoD. Экземпляр Сака развертывается в регионах "Восток" и "центр DOD" в области Azure. Весь входящий и исходящий трафик проходит через него к соединению ExpressRoute с DISA БКАП.

Приложения, ответственные за владение, затем выбирают регионы Azure, в которых планируется развертывать свои приложения. Они используют пиринг виртуальных сетей для подключения виртуальной сети приложения к виртуальной сети Сака. Затем они принудительно проходят весь трафик через экземпляр ВДСС.

Мы рекомендуем использовать эту архитектуру, так как она соответствует требованиям СККА. Он обладает высокой доступностью и легко масштабируется. Кроме того, это упрощает развертывание и управление.

## <a name="automated-saca-deployment-options"></a>Параметры автоматического развертывания Сака

 Как упоминалось ранее, корпорация Майкрософт сотрудничает с двумя поставщиками для создания шаблона автоматизированной инфраструктуры Сака. Оба шаблона развертывают следующие компоненты Azure: 

- Виртуальная сеть Сака
    - Подсеть управления
        - В этой подсети развертываются виртуальные машины и службы управления, также называемые полями перехода.
        - Подсеть ВДМС
            - В этой подсети развертываются виртуальные машины и службы, используемые для ВДМС.
        - Недоверенные и доверенные подсети
            - Эти подсети разворачивают виртуальные устройства.
        - Подсеть шлюза
            - В этой подсети развернут шлюз ExpressRoute.
- Виртуальные машины управления — окна переходов
    - Они используются для управления средой по внешнему каналу.
- Виртуальные сетевые модули
    - Вы используете Citrix или F5 в зависимости от того, какой шаблон развертывается.
- Общедоступные IP-адреса
    - Они используются для внешнего интерфейса до перевода ExpressRoute в режим «в сети». Эти IP-адреса переводятся в адресное пространство внутренних адресов Azure.
- Таблицы маршрутов 
    - Эти таблицы маршрутов, применяемые во время автоматизации, принудительно проходят весь трафик через виртуальное устройство.
- Подсистемы балансировки нагрузки Azure — SKU уровня "Стандартный"
    - Они используются для балансировки нагрузки трафика между устройствами.
- Группы безопасности сети
    - Они используются для управления типами трафика, которые могут перебрать определенные конечные точки.

Шаблон развертывания Citrix развертывает два уровня высокодоступных устройств Citrix ADC. Эта архитектура соответствует требованиям ВДСС. 

![Схема Citrix Сака](media/citrixsaca.png)


Документацию по Citrix и сценарий развертывания см. на [этой ссылке GitHub](https://github.com/citrix/netscaler-azure-templates/tree/master/templates/saca).


 ### <a name="f5-saca-deployment"></a>F5 Сака развертывание

Два отдельных шаблона развертывания F5 охватывают две различные архитектуры. Первый шаблон имеет только один уровень управляющих устройств F5 в конфигурации с высокой доступностью "активный — активный". Эта архитектура соответствует требованиям для ВДСС. Второй шаблон добавляет второй уровень с высокой доступностью "активный — активный" F5s. Этот второй уровень позволяет клиентам добавлять свои собственные IP-адреса отдельно от клавиши F5 между слоями F5. Не все компоненты DoD имеют определенные IP-адреса, предписанные для использования. В этом случае один слой устройств F5 работает в большинстве случаев, поскольку эта архитектура включает в себя IP-адреса на устройствах F5.

![Схема F5 Сака](media/f5saca.png)

Документацию по F5 и скрипт развертывания см. на [этой ссылке GitHub](https://github.com/f5devcentral/f5-azure-saca).
