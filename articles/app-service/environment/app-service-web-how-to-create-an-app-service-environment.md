# <a name="how-to-create-an-app-service-environment-v1"></a>Как создать среду службы приложений версии 1 

> [!NOTE]
> Эта статья посвящена среде службы приложений версии 1. Имеется более новая версия среды службы приложений, которая проще в использовании и которая работает на более мощной инфраструктуре. Чтобы узнать больше о новой версии, начните с изучения статьи [Введение в среду службы приложения](intro.md).
> 

### <a name="overview"></a>Обзор
Среда службы приложений (ASE) — это функция службы приложений Azure уровня служб "Премиум", которая предоставляет возможность расширенной настройки, недоступную в мультитенантных метках. Функция ASE фактически развертывает службу приложений Azure в виртуальной сети клиента. Чтобы получить более полное представление о возможностях, предлагаемых средами службы приложений, ознакомьтесь с документацией по [Среда службы приложений][WhatisASE] .

### <a name="before-you-create-your-ase"></a>Перед созданием среды службы приложений
Важно помнить о параметрах, которые нельзя будет изменить. Ниже перечислены параметры среды ASE, которые нельзя изменить после ее создания.

* Местоположение
* Подписка
* Группа ресурсов
* Используемая виртуальная сеть
* Используемая подсеть 
* Размер подсети

При выборе виртуальной сети и подсети убедитесь, что они достаточно большие для расширения в будущем. 

### <a name="creating-an-app-service-environment-v1"></a>Создание среды службы приложений версии 1
Чтобы создать среду службы приложений версии 1, вы можете выполнить поиск в Azure Marketplace по запросу ***App Service Environment v1*** или выбрать **Создать ресурс** -> **Интернет и мобильные устройства** -> **Среда службы приложений**. Вот как можно создать среду ASE версии 1.

1. Укажите имя среды ASE. Имя, указанное для ASE, будет использоваться для веб-приложений, созданных в этой среде ASE. Если для ASE указано имя appsvcenvdemo, то именем поддомена будет: *appsvcenvdemo.p.azurewebsites.net*. Если создать веб-приложение с именем *mytestapp*, то к нему можно будет обращаться по адресу *mytestapp.appsvcenvdemo.p.azurewebsites.net*. В имени ASE нельзя использовать пробелы. Если вы используете в имени символы верхнего регистра, в доменном имени будут использоваться такие же символы нижнего регистра. Если используется внутренний балансировщик нагрузки, то имя среды ASE не используется в поддомене, а указывается явно во время создания среды ASE.
   
    ![][1]
2. Выберите свою подписку. Подписка, которая используется для среды ASE, применяется для всех приложений, созданных в этой среде. Невозможно разместить среду ASE в виртуальной сети, которая находится в другой подписке.
3. Выберите или укажите новую группу ресурсов. Для среды ASE нужно использовать ту же группу ресурсов, что и для виртуальной сети. Если выбрать уже существующую виртуальную сеть, то это соответствующим образом изменит выбор группы ресурсов для среды ASE.
   
    ![][2]
4. Выберите виртуальную сеть и расположение. Можно выбрать существующую виртуальную сеть или создать новую. В случае выбора создания виртуальной сети можно указать ее имя и расположение. У новой виртуальной сети будет диапазон адресов 192.168.250.0/23 и подсеть **default**, определенная как 192.168.250.0/24. Можно также просто выбрать уже существующую классическую виртуальную сеть или виртуальную сеть Resource Manager. Параметр "Тип VIP" определяет, возможен ли доступ к среде ASE непосредственно из Интернета (внешний) или через внутренний балансировщик нагрузки (ILB). Дополнительные сведения о них см. в статье [использование внутреннего Load Balancer с среда службы приложений][ILBASE]. Если для параметра "Тип VIP" выбрать значение "Внешний", то можно будет выбрать количество внешних IP-адресов, с которыми создается система, предназначенных для подключений IPSSL. Если выбрать значение "Внутренний", то потребуется указать поддомен, который будет использовать среда ASE. Среды ASE можно развертывать в виртуальных сетях, использующих *либо* диапазоны общедоступных адресов, *либо* адресные пространства RFC1918 (т. е. частные адреса). Чтобы использовать виртуальную сеть с диапазоном общедоступных адресов, потребуется создать ее заранее. В случае выбора существующей виртуальной сети во время создания среды ASE необходимо будет создать новую подсеть. **На портале невозможно использовать предварительно созданную подсеть. Создать среду ASE с уже существующей подсетью можно с помощью шаблона Resource Manager.** Чтобы создать ASE на основе шаблона, используйте здесь сведения: [создание среда службы приложений из шаблона][ILBAseTemplate] и здесь [Создание ilB среда службы приложений из шаблона][ASEfromTemplate].

### <a name="details"></a>Подробнее
Среда ASE создается с двумя внешними интерфейсами и двумя рабочими ролями. Внешний интерфейс выступает в качестве конечных точек HTTP или HTTPS и отправляет трафик в рабочие роли. Это роли, в которых размещены ваши приложения. Их количество можно настроить после создания среды ASE. Более того, вы можете настроить правила автомасштабирования для этих пулов ресурсов. Дополнительные сведения о ручном масштабировании и мониторинге Среды службы приложений, а также управлении ею см. в статье [Настройка Среда службы приложений][ASEConfig] 

В подсети, используемой ASE, может существовать только одна среда ASE. Эту подсеть невозможно использовать для размещения чего-либо, кроме среды ASE.

### <a name="after-app-service-environment-v1-creation"></a>После создания среды службы приложений версии 1
После создания ASE можно настроить такие параметры:

* Количество внешних интерфейсов (минимальное: 2)
* Количество исполнителей (минимальное: 2)
* Количество IP-адресов, доступных для SSL на основе IP
* Размеры вычислительных ресурсов, используемых внешними интерфейсами или исполнителями (минимальный размер для внешнего интерфейса — P2)

Дополнительные сведения о ручном масштабировании и мониторинге Среды службы приложений, а также управлении ею см. в статье [Настройка Среда службы приложений][ASEConfig] 

Дополнительные сведения об автоматическом масштабировании см. в статье [Настройка автомасштабирования для Среда службы приложений][ASEAutoscale]

Существуют дополнительные зависимости, которые невозможно настроить, такие как база данных и хранилище. Они обрабатываются платформой Azure и поставляются с системой. Система хранения предоставляет до 500 ГБ для среды ASE, а Azure при необходимости регулирует размер базы данных в соответствии с масштабом системы.
