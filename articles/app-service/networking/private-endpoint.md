# <a name="using-private-endpoints-for-azure-web-app-preview"></a>Использование частных конечных точек для веб-приложения Azure (Предварительная версия)

> [!Note]
> Предварительная версия доступна в регионах "Восточная часть США" и "Западная часть США 2" для всех категории премиум v2 веб-приложений Windows и Linux и функций эластичного Premium. 

Вы можете использовать закрытую конечную точку для веб-приложения Azure, чтобы разрешить клиентам, находящимся в частной сети, безопасный доступ к приложению по частной ссылке. Частная конечная точка использует IP-адрес из адресного пространства виртуальной сети Azure. Сетевой трафик между клиентом в частной сети и веб-приложением проходит по виртуальной сети и частной ссылке на магистральную сеть Майкрософт, что устраняет уязвимость в общедоступном Интернете.

Использование частной конечной точки для веб-приложения позволяет:

- Защитите веб-приложение, настроив закрытую конечную точку, исключив открытую раскрытие.
- Безопасное подключение к веб-приложению из локальных сетей, которые подключаются к виртуальной сети с помощью частного пиринга VPN или ExpressRoute.

Если вам просто требуется безопасное подключение между виртуальной сетью и веб-приложением, конечной точкой службы является простейшее решение. Если вам также нужно подключиться к веб-приложению из локальной среды через шлюз Azure, виртуальную сеть с региональными сторонами или глобально одноранговую виртуальную сеть, это решение является частной конечной точкой.  

Дополнительные сведения см. в статье [конечные точки службы][serviceendpoint].

## <a name="conceptual-overview"></a>Общие сведения

Частная конечная точка — это специальный сетевой интерфейс (NIC) для веб-приложения Azure в подсети виртуальной сети (VNet).
При создании частной конечной точки для веб-приложения обеспечивается безопасное подключение между клиентами в частной сети и веб-приложением. Частной конечной точке назначается IP-адрес из диапазона IP-адресов виртуальной сети.
Соединение между частной конечной точкой и веб-приложением использует защищенную [закрытую ссылку][privatelink]. Частная конечная точка используется только для входящих потоков веб-приложения. Исходящие потоки не будут использовать эту закрытую конечную точку, но вы можете ввести исходящие потоки в сеть в другой подсети с помощью [функции интеграции с виртуальной][vnetintegrationfeature]сетью.

Подсеть, в которой вы подключаетесь к закрытой конечной точке, может содержать другие ресурсы, не нужна выделенная пустая подсеть.
Вы также можете развернуть частную конечную точку в регионе, отличном от региона веб-приложения. 

> [!Note]
>Функция интеграции с виртуальной сетью не может использовать ту же подсеть, что и частная конечная точка. это ограничение компонента интеграции с виртуальной сетью.

С точки зрения безопасности:

- При включении в веб-приложение частных конечных точек все открытые права доступа отключаются.
- В других виртуальных сетей и подсетях можно включить несколько частных конечных точек, в том числе виртуальных сетей в других регионах.
- IP-адрес NIC частной конечной точки должен быть динамическим, но останется прежним, пока не будет удалена частная конечная точка.
- Сетевой карте частной конечной точки не может быть назначен NSG.
- Подсеть, в которой размещается частная конечная точка, может иметь NSG, но необходимо отключить применение политик сети для частной конечной точки. см. раздел [отключение сетевых политик для частных конечных точек][disablesecuritype]. В результате вы не можете выполнять фильтрацию по любому NSG доступа к частной конечной точке.
- При включении закрытой конечной точки для веб-приложения конфигурация [ограничений доступа][accessrestrictions] веб-приложения не оценивается.
- Вы можете уменьшить риск утечка данных из виртуальной сети, удалив все правила NSG, в которых назначение — это Интернет или службы Azure. Но добавление частной конечной точки веб-приложения в подсеть позволит получить доступ к любому веб-приложению, размещенному в той же отметке развертывания и доступному для Интернета.

В журналах веб-протокола HTTP веб – приложения вы увидите исходный IP-адрес клиента. Это реализуется с помощью протокола TCP-прокси и пересылает свойство IP-адреса клиента в веб-приложение. Дополнительные сведения см. в разделе [Получение сведений о соединении с помощью прокси-сервера TCP версии 2][tcpproxy].

## <a name="dns"></a>DNS

Так как эта функция находится на этапе предварительной версии, мы не изменим DNS-запись во время предварительной версии. Необходимо самостоятельно управлять записью DNS в частном DNS-сервере или Azure DNS частной зоне.
Если необходимо использовать настраиваемое DNS-имя, необходимо добавить пользовательское имя в веб-приложение. Во время предварительной версии пользовательское имя должно проверяться как любое пользовательское имя, используя разрешение общедоступного DNS. Дополнительные сведения см. в разделе [Настраиваемая проверка DNS][dnsvalidation] .
