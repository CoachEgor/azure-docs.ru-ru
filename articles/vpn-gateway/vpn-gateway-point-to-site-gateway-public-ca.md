# <a name="transition-to-a-public-ca-gateway-certificate-for-p2s"></a>Переход на общедоступный сертификат шлюза от ЦС для P2S

VPN-шлюз Azure больше не выдает самозаверяющие сертификаты уровня Azure шлюзам для P2S-подключений. Теперь выданные сертификаты подписываются общедоступным центром сертификации (ЦС). Тем не менее некоторые шлюзы более ранних версий могут по-прежнему использовать самозаверяющие сертификаты. Срок действия этих сертификатов вскоре истечет, и их необходимо перевести на сертификаты общедоступных ЦС.

>[!NOTE]
> * Это изменение сертификата уровня Azure не влияет на самозаверяющие сертификаты, используемые для проверки подлинности клиента P2S. Вы можете продолжать выпускать и использовать самозаверяющие сертификаты в обычном режиме.
>

Сертификаты в этом контексте являются дополнительным сертификатом уровня Azure. Они не являются цепочками сертификатов, которые вы используете при создании собственных самозаверяющих корневых сертификатов и клиентских сертификатов для проверки подлинности. Эти сертификаты останутся без изменения, а их срок истекает по датам, созданным вами.

Ранее самозаверяющий сертификат для шлюза (выпущенный за пределами Azure) требовалось обновлять каждые 18 месяцев. После этого нужно было создать и повторно развернуть файлы конфигурации VPN-клиента для всех клиентов P2S. Переход на сертификаты общедоступных ЦС позволяет устранить это ограничение. Помимо перехода на лучшие сертификаты, это изменение также обеспечивает усовершенствования платформы, улучшенные метрики и повышенную стабильность.

Это изменение влияет только на шлюзы более ранних версий. Если ваш сертификат шлюза нужно перевести, вы получите сообщение или всплывающее уведомление на портале Azure. Вы можете проверить, касается ли это изменение вашего шлюза, следуя инструкциям в этой статье.

> [!IMPORTANT]
> Переход на открытые сертификаты ЦС была выполнена на 12 мая 2019 г. **В этом документе будут удалены на 31 мая 2019 г.**

## <a name="1-verify-your-certificate"></a>1. Проверка сертификата

### <a name="resource-manager"></a>Resource Manager

1. Проверьте, касается ли вас это обновление. Скачайте текущую конфигурацию VPN-клиента, следуя указаниям в [этой статье](point-to-site-vpn-client-configuration-azure-cert.md).

2. Откройте или извлеките ZIP-файл и перейдите к папке "Generic". В папке "Generic" вы увидите два файла, один из которых — *VPNSettings.xml*.
3. Откройте файл *VPNSettings.xml* в любом средстве просмотра или редакторе XML. В XML-файле найдите следующие поля:

   * `<ServerCertRootCn>DigiCert Global Root CA</ServerCertRootCn>`
   * `<ServerCertIssuerCn>DigiCert Global Root CA</ServerCertIssuerCn>`
4. Если для *ServerCertRotCn* и *ServerCertIssuerCn* указано значение "DigiCert Global Root CA", это обновление не касается вас и вам не нужно переходить к действиям в этой статье. Тем не менее, если отображается другой текст, сертификат шлюза является частью обновления и будет переведен.

### <a name="classic"></a>Классический

1. На клиентском компьютере, перейдите по пути `%appdata%/Microsoft/Network/Connections/Cm/<gatewayID>`. Вы можете просмотреть этот сертификат в папке идентификатора шлюза.
2. Для этого проверьте на вкладке "Общие", что центром выдачи является "DigiCert Global Root CA". Если у вас есть другой центр выдачи кроме этого, ваш сертификат шлюза является частью обновления и будет перенесен.

## <a name="2-check-certificate-transition-schedule"></a>2. Проверка расписания перехода сертификата

Если сертификат является частью обновления, сертификат шлюза будет переведен. См. сведения о времени перехода в примечании **Важно**. После обновления клиенты P2S не смогут подключиться с помощью старого профиля. Необходимо создать новые профили VPN-клиентов и установить их на клиентах.

## <a name="3-generate-vpn-client-configuration-profile"></a>3. Создание профиля конфигурации VPN-клиента

После перевода сертификата скачайте новый профиль VPN (файлы конфигурации VPN-клиента) на портале Azure. Дальнейшие инструкции см. в статье [Создание и установка файлов конфигурации VPN-клиента для настройки подключений типа "точка — сеть" с использованием собственной аутентификации Azure на основе сертификата](point-to-site-vpn-client-configuration-azure-cert.md). Вам не нужно создавать новые сертификаты клиента.

## <a name="4-deploy-vpn-client-profile"></a>4. Развертывание профиля VPN-клиента

Разверните новый профиль для всех VPN-клиентов "точка — сеть". VPN-клиенты потеряют подключение, пока новый профиль VPN для подключений "точка — сеть" не будет скачан и развернут на клиентских устройствах. Сертификаты клиентов, которые уже установлены на компьютерах VPN-клиентов, не нужно выдавать повторно.

## <a name="5-connect-the-vpn-client"></a>5. Подключение VPN-клиента

Подключитесь к Azure из VPN-клиента как обычно.
