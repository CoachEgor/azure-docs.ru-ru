# <a name="create-a-managed-service-offer"></a>Создание предложения управляемой службы

Управляемая служба позволяет включить сценарии [лигхсаусе для Azure](../../lighthouse/overview.md) . Когда клиент принимает предложение управляемой службы, он может подключить ресурсы для [управления делегированными ресурсами Azure](../../lighthouse/concepts/azure-delegated-resource-management.md). Прежде чем начать, [Создайте учетную запись коммерческого магазина в центре партнеров](https://docs.microsoft.com/azure/marketplace/partner-center-portal/create-account) , если вы еще этого не сделали. Убедитесь, что ваша учетная запись зарегистрирована в программе коммерческого рынка.

Для публикации предложения управляемой службы необходим [уровень компетенции "Серебряная облачная платформа" или](https://partner.microsoft.com/membership/cloud-platform-competency) " [специалист по Azure](https://partner.microsoft.com/membership/azure-expert-msp) ".

## <a name="create-a-new-offer"></a>Создание нового предложения

1. Войдите в [Центр партнеров](https://partner.microsoft.com/dashboard/home).
2. В меню слева выберите пункт**Обзор** **коммерческого рынка** > .
3. На странице Обзор выберите **+ создать** > **управляемую службу**предложения.

    ![Показывает меню навигации слева.](./media/new-offer-managed-service.png)

>[!NOTE]
>После публикации предложения изменения, внесенные в него в центре партнеров, отображаются только в витринах после повторной публикации предложения. Убедитесь, что после внесения изменений всегда выполняется повторная публикация.

## <a name="new-offer"></a>Новое предложение

Введите **идентификатор предложения**. Это уникальный идентификатор для каждого предложения в вашей учетной записи.

* Этот идентификатор отображается для клиентов в веб-адресе для предложения Marketplace и шаблонов Azure Resource Manager, если это применимо.
* Вы можете использовать только строчные буквы и цифры. Он может содержать дефисы и знаки подчеркивания, но не содержит пробелов и не должен превышать 50 символов. Например, если ввести **Test-предложение-1**, веб-адрес предложения будет иметь `https://azuremarketplace.microsoft.com/marketplace/../test-offer-1`значение.
* Идентификатор предложения нельзя изменить после нажатия **создать**.

Введите **псевдоним предложения**. Это имя, используемое для предложения в центре партнеров.

* Это имя не используется в Marketplace и отличается от имени предложения и других значений, отображаемых для клиентов.
* Невозможно изменить псевдоним предложения после нажатия на кнопку " **создать**".

Выберите **создать** , чтобы создать предложение и продолжить.

Выберите минимум одну и не более пяти категорий, которые будут использоваться для размещения предложения в соответствующих областях поиска Marketplace. Обязательно вызовите, как ваше предложение поддерживает эти категории в описании предложения.

## <a name="offer-listing"></a>Список предложений

> [!NOTE]
> Предложение с перечнем содержимого (например, описание, документы, снимки экрана и условия использования) не обязательно должно располагаться на английском языке, если описание предложения начинается с фразы «это приложение доступно только на языке, отличном от английского языка]». Также можно предоставить *полезный URL-адрес ссылки* , чтобы предложить содержимое на языке, отличном от того, который использовался в содержании предложения.

### <a name="name"></a>Название

Введенное здесь имя будет отображаться для клиентов в качестве заголовка списка предложений. Это поле заполняется текстом, введенным для параметра **псевдоним предложения** при создании предложения, но это значение можно изменить. Это имя может быть охраняемым товаром (и может содержать знаки товарных знаков или авторских прав). Имя не может содержать более 50 символов и не может включать символы эмодзи.

### <a name="search-results-summary"></a>Сводка по результатам поиска

Введите краткое описание предложения (до 100 символов), которое может использоваться в результатах поиска Marketplace.

### <a name="long-summary"></a>Длинная сводка

Введите более длинное описание предложения (до 256 символов). Эту длинную сводку можно также использовать в результатах поиска Marketplace.

### <a name="description"></a>Описание

Введите более длинное описание предложения (до 3 000 символов). Это описание будет отображаться для клиентов в обзоре списка Marketplace. Включите предложение ценности, ключевые преимущества, категории и (или) отраслевые связи, возможности приобретения в приложении и все необходимые раскрытия.

Некоторые советы по написанию описания:  

- Четко укажите преимущества, предоставляемые вашим предложением, в начале описания. Включите в свое выгодное предложение следующие элементы:
  - Описание предложения
  - Тип пользователя, который имеет преимущества от предложения
  - Потребности клиентов или проблемы с адресами предложения
- Имейте в виду, что первые несколько предложений могут отображаться в результатах поисковой системы.  
- Не полагайтесь на то, что функции и возможности помогут продавать ваш продукт. Вместо этого сосредоточьтесь на его преимуществах.  
- Используйте как можно больше отраслевого лексикона или формулировок на основе преимуществ.

Чтобы сделать описание предложения более привлекательным, используйте редактор форматированного текста для применения форматирования.

Следуйте приведенным ниже рекомендациям для ваших логотипов:

## <a name="preview"></a>Предварительный просмотр

Прежде чем публиковать предложение в более широком предложении Marketplace, сначала необходимо сделать его доступным для аудитории с ограниченной предварительной версией. Это позволяет подтвердить отображение предложения в Azure Marketplace, прежде чем сделать его доступным для клиентов. Группы поддержки и инженеров Майкрософт также смогут просматривать ваше предложение в течение периода предварительной версии.

Вы можете определить аудиторию предварительного просмотра, введя идентификатор подписки Azure в разделе **Предварительная версия аудитории** . Вы можете ввести до 10 идентификаторов подписок вручную или передать CSV-файл с идентификаторами подписки до 100.

Все клиенты, связанные с этими подписками, смогут просматривать предложение в Azure Marketplace, прежде чем оно станет активным. Не забудьте включить собственные подписки, чтобы можно было просмотреть предложение.

Прежде чем продолжить, выберите **Сохранить черновик** .

## <a name="plan-overview"></a>Общие сведения о плане

В каждом предложении должен быть один или несколько планов (иногда их называют номерами SKU). Вы можете добавить несколько планов для поддержки различных наборов функций по разным ценам или для настройки определенного плана для ограниченной аудитории конкретных клиентов. Клиенты могут просматривать планы, доступные для них в родительском предложении.

На странице **Обзор плана** выберите **+ создать новый план**. Затем введите **идентификатор плана** и **имя плана**. Оба эти значения могут содержать только буквы в нижнем регистре, тире и символы подчеркивания, не более 50 символов. Эти значения могут быть видимыми для клиентов, и их нельзя изменить после публикации предложения.

Нажмите кнопку **создать** после введения этих значений, чтобы продолжить работу с планом. Существует три раздела, которые необходимо выполнить: **составление списка планов**, **цен и доступности**, а затем **Техническая настройка**.

### <a name="plan-listing"></a>Список планов

Сначала предоставьте сводку по **результатам поиска** для плана. Это краткое описание вашего плана (до 100 символов), которое может использоваться в результатах поиска Marketplace.

Затем введите **Описание** , которое содержит более подробное объяснение плана.

### <a name="pricing-and-availability"></a>Цены и доступность

В настоящее время существует только одна модель ценообразования, которую можно использовать для предложения управляемой службы: использование **собственной лицензии (BYOL)**. Это означает, что вы будете выставлять клиентам счета непосредственно по затратам, связанным с этим предложением, и корпорация Майкрософт не будет взимать с вас плату.

Раздел **видимость плана** позволяет указать, должен ли этот план быть [закрытым](../../marketplace/private-offers.md). Если оставить поле " **частный план** " снятым, ваш план не будет ограничен конкретными клиентами (или определенным числом клиентов).

Чтобы сделать этот план доступным только для конкретных клиентов, выберите **Да**. После этого необходимо определить клиентов, указав их идентификаторы подписок. Их можно указать по одному (для 10 подписок) или путем отправки CSV-файла (максимум 10 000 подписок во всех планах). Обязательно включите собственные подписки, чтобы можно было протестировать предложение.

> [!IMPORTANT]
> После публикации плана как общедоступного вы не сможете изменить его на частный. Чтобы контролировать, какие клиенты могут принимать ваши предложения и делегировать ресурсы, используйте частный план. С помощью общедоступного плана вы не сможете ограничить доступность определенными клиентами или даже определенным числом клиентов (хотя вы можете не допустить продажи плана полностью, если вы решили это сделать). Вы можете [Удалить доступ к делегированию](../../lighthouse/how-to/remove-delegation.md) после того, как клиент примет предложение, только если вы включили **авторизацию** с **определением роли** , настроенным на [Удаление роли назначения регистрации управляемых служб](../../role-based-access-control/built-in-roles.md#managed-services-registration-assignment-delete-role) при публикации предложения. Вы также можете обратиться к клиенту и попросить его [Удалить доступ](../../lighthouse/how-to/view-manage-service-providers.md#add-or-remove-service-provider-offers).

### <a name="technical-configuration"></a>Техническая конфигурация

В этом разделе плана создается манифест с данными авторизации для управления ресурсами клиентов. Эти сведения необходимы для включения [управления делегированными ресурсами Azure](../../lighthouse/concepts/azure-delegated-resource-management.md).

Не забудьте проверить [клиентов, роли и пользователей в сценариях Azure лигхсаусе](../../lighthouse/concepts/tenants-users-roles.md#best-practices-for-defining-users-and-roles) , чтобы понять, какие роли поддерживаются, а также рекомендации по определению ваших авторизаций.

> [!NOTE]
> Как отмечалось выше, пользователи и роли в записях **Авторизация** будут действовать для всех клиентов, которые приобрели план. Если вы хотите предоставить доступ только конкретному клиенту, необходимо опубликовать частный план исключительно для него.

#### <a name="manifest"></a>манифеста

Сначала укажите **версию** манифеста. Используйте формат *n. n. n* (например, 1.2.5).

Затем заполните поле **Идентификатор клиента**. Это идентификатор GUID, связанный с ИДЕНТИФИКАТОРом клиента Azure Active Directory (Azure AD) вашей организации. то есть управление клиентом, из которого вы будете получать доступ к ресурсам клиентов. Если у вас нет его под рукой, его можно найти, наведя указатель мыши на имя учетной записи в правом верхнем углу портала Azure либо выбрав **Переключение каталога**.

Если вы публикуете новую версию предложения и хотите создать обновленный манифест, выберите **+ создать манифест**. Не забудьте увеличить номер версии из предыдущей версии манифеста.

#### <a name="authorization"></a>Авторизация

Авторизация определяет сущности в управляемом клиенте, которые имеют доступ к ресурсам и подпискам для клиентов, которые приобрели план. Каждой из этих сущностей назначается встроенная роль, предоставляющая определенные уровни доступа.

Для каждого плана можно создать до двадцати авторизаций.

> [!TIP]
> В большинстве случаев необходимо назначить роли группе пользователей или субъекту-службе Azure AD, а не к ряду отдельных учетных записей пользователей. В этом случае вы сможете добавлять или удалять доступ для отдельных пользователей, не обновляя и повторно публикуя план при изменении требований к доступу. При назначении ролей группам Azure AD [Убедитесь, что **Тип группы** — **Security** , а не **Office 365**](../../active-directory/fundamentals/active-directory-groups-create-azure-portal.md). Дополнительные рекомендации см. в разделе [Арендаторы, роли и пользователи в сценариях Azure Lighthouse](../../lighthouse/concepts/tenants-users-roles.md).

Для каждой **авторизации** укажите следующее. Затем можно выбрать **+ добавить авторизацию** столько раз, сколько требуется для добавления дополнительных пользователей и определений ролей.

- **Идентификатор объекта Azure AD**: идентификатор Azure AD пользователя, группы пользователей или приложения, которым будут предоставлены определенные разрешения (как определено определением роли) в ресурсы клиентов.
- **Отображаемое имя объекта Azure AD**: понятное имя, помогающее клиенту понять назначение авторизации. Клиент увидит это имя при делегировании ресурсов.
- **Определение роли**. Выберите из списка одну из доступных встроенных РОЛЕЙ Azure AD. Эта роль определит разрешения, которые пользователь в поле **Идентификатор объекта Azure AD** будет иметь на ресурсы ваших клиентов. Описание этих ролей см. в статьях [встроенные роли](../../role-based-access-control/built-in-roles.md) и [Поддержка ролей для управления делегированными ресурсами Azure](../../lighthouse/concepts/tenants-users-roles.md#role-support-for-azure-delegated-resource-management).
  > [!NOTE]
  > По мере добавления в Azure новых встроенных ролей они станут доступны здесь, хотя может быть задержка, прежде чем они будут отображаться.
- **Назначаемые роли**. Этот параметр отображается, только если в **определении роли** для этой авторизации выбрано значение администратор доступа пользователей. В таком случае здесь необходимо добавить одну или несколько назначаемых ролей. Пользователь в поле **идентификатор объекта Azure AD** сможет назначить эти роли [управляемым удостоверениям](../../active-directory/managed-identities-azure-resources/overview.md), которые необходимы для [развертывания политик, которые можно исправить](../../lighthouse/how-to/deploy-policy-remediation.md). Обратите внимание, что к этому пользователю не будут применяться никакие другие разрешения, обычно связанные с ролью администратора доступа пользователей.

> [!TIP]
> Чтобы при необходимости можно было [Удалить доступ к делегированию](../../lighthouse/how-to/remove-delegation.md) , включите **авторизацию** с **определением роли** , настроенным на [Удаление роли назначения регистрации управляемых служб](../../role-based-access-control/built-in-roles.md#managed-services-registration-assignment-delete-role). Если эта роль не назначена, делегированные ресурсы могут быть удалены только пользователем в арендаторе клиента.

После выполнения всех разделов плана можно выбрать вариант **+ создать план** столько раз, сколько нужно для создания дополнительных планов. Когда все будет готово, нажмите кнопку **Сохранить**.

Прежде чем продолжить, выберите **Сохранить черновик** .

## <a name="publish"></a>Публикация

### <a name="submit-offer-to-preview"></a>Отправить предложение в предварительную версию

Завершив все необходимые разделы предложения, выберите **проверить и опубликовать** в правом верхнем углу портала.

Если вы впервые публикуете это предложение, вы можете:

- См. сведения о состоянии завершения для каждого раздела предложения.
  - **Не запущено** — раздел не был затронут, и его необходимо завершить.
  - **Неполный** — раздел содержит ошибки, которые необходимо исправить, или требуется указать дополнительные сведения. Вернитесь в раздел (ы) и обновите его.
  - **Завершено** — раздел завершен, все необходимые данные были предоставлены и ошибки отсутствуют. Все разделы предложения должны находиться в полном состоянии, прежде чем вы сможете отправить предложение.
- В разделе **Примечания для сертификации** укажите инструкции по тестированию для группы сертификации, чтобы убедиться, что приложение правильно проверено, а также какие-либо дополнительные заметки, которые помогут понять ваше приложение.
- Отправьте предложение для публикации, выбрав **Отправить**. Мы отправим вам электронное сообщение, когда Предварительная версия предложения будет доступна для просмотра и утверждения. Вернитесь в центр партнеров и выберите **Go-Live** для предложения, чтобы опубликовать предложение в общедоступной аудитории (или в частном предложении).

### <a name="customer-experience-and-offer-management"></a>Взаимодействие с клиентами и Управление предложениями

Когда Клиент развертывает ваше предложение, он сможет делегировать подписки или группы ресурсов для [управления делегированными ресурсами Azure](../../lighthouse/concepts/azure-delegated-resource-management.md). Дополнительные сведения об этом процессе см. [в описании процесса адаптации клиентов](../../lighthouse/how-to/publish-managed-services-offers.md#the-customer-onboarding-process).

Вы можете в любое время [опубликовать обновленную версию предложения](update-existing-offer.md). Например, вам может потребоваться добавить новое определение роли к опубликованному ранее предложению. После этого клиенты, которые уже добавили предложение, увидят значок на странице [**Поставщики услуг**](../../lighthouse/how-to/view-manage-service-providers.md) на портале Azure, который сообщит им, доступно ли обновление. Каждый клиент сможет проверить изменения и решить, требуется ли им обновление до новой версии.
