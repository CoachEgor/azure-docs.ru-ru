# <a name="create-a-managed-service-offer"></a>Создание предложения управляемой службы

Управляемая служба предлагает поддержку сценариев [Azure Lighthouse](../../lighthouse/overview.md). Когда клиент принимает предложение управляемой службы, он может подключить ресурсы для [делегированного управления ресурсами Azure](../../lighthouse/concepts/azure-delegated-resource-management.md). Сначала [создайте учетную запись на коммерческой платформе в Центре партнеров](create-account.md), если вы еще этого не сделали. Убедитесь, что ваша учетная запись зарегистрирована в программе коммерческой платформы.

Для публикации предложения управляемой службы необходимо иметь [уровень компетенции облачной платформы "Серебряный" или "Золотой"](https://partner.microsoft.com/membership/cloud-platform-competency) или быть [Поставщиком управляемых служб Azure Expert](https://partner.microsoft.com/membership/azure-expert-msp).

## <a name="create-a-new-offer"></a>Создание нового предложения

1. Войдите в [Центр партнеров](https://partner.microsoft.com/dashboard/home).
2. В меню слева выберите **Коммерческая платформа** > **Обзор**.
3. На странице "Обзор" выберите **+ Новое предложение** > **Управляемая служба**.

    ![Отображается меню навигации слева.](./media/new-offer-managed-service.png)

>[!NOTE]
>После публикации предложения изменения, внесенные в него в Центре партнеров, отображаются на витрине только после повторной публикации. Если вы внесли изменения, обязательно выполните повторную публикацию.

## <a name="new-offer"></a>Новое предложение

Укажите **идентификатор предложения**. У каждого предложения в вашей учетной записи должен быть уникальный идентификатор.

* Идентификатор отображается для клиентов в веб-адресе предложения Marketplace, а также, если применимо, в веб-адресе шаблонов Resource Manager.
* Вы можете использовать только строчные буквы и цифры. Идентификатор может содержать дефисы и символы подчеркивания, но не пробелы, и включать не больше 50 символов. Например, если вы введете **test-offer-1**, веб-адрес предложения будет иметь следующий вид: `https://azuremarketplace.microsoft.com/marketplace/../test-offer-1`.
* Идентификатор предложения нельзя изменить после выбора команды **Создать**.

Укажите **псевдоним** для вашего предложения. Это имя, которое используется для предложения в Центре партнеров.

* Это имя не используется на Marketplace и отличается от имени предложения и других значений, отображаемых для клиентов.
* Псевдоним предложения нельзя изменить после выбора команды **Создать**.

Выберите **Создать**, чтобы создать предложение и продолжить.

## <a name="properties"></a>Свойства

### <a name="category"></a>Категория

Выберите хотя бы одну категорию (но не более пяти). Категории будут использоваться для размещения вашего предложения в соответствующих областях поиска на Marketplace. В описании предложения расскажите, как ваше предложение поддерживает эти категории.

Укажите свои юридические условия в поле **Условия использования**. Вы также можете предоставить URL-адрес, по которому можно найти условия. Клиентам будет предложено принять эти условия, прежде чем использовать предложение.

- Цвета темы портала — белый и черный. Постарайтесь не использовать эти цвета для фона своего логотипа. Используйте цвет, который выделяет ваш логотип на портале. Мы рекомендуем простые основные цвета.

- Логотип должен быть сплошным, избегайте градиентов. Не используйте в логотипе градиентный фон.
- Не размещайте в логотипе текст, даже название своей компании или торговой марки.
- Логотип не должен быть растянут.

## <a name="preview"></a>Preview (Предварительный просмотр)

Прежде чем публиковать предложение для всех, сделайте его доступным для ограниченной аудитории предварительной версии. Это позволяет проверить, как предложение отображается в Azure Marketplace, прежде чем сделать его доступным для клиентов. Специалисты службы поддержки Майкрософт и команды разработчиков также смогут просматривать ваше предложение в течение периода предварительной версии.

Вы можете определить аудиторию предварительного просмотра, введя идентификаторы подписки Azure в разделе **Preview аудитории**. Вы можете ввести до 10 идентификаторов подписок вручную или передать CSV-файл, содержащий до 100 идентификаторов подписок.

Клиенты, связанные с этими подписками, смогут просматривать предложение в Azure Marketplace, прежде чем оно станет активным. Обязательно включите собственные подписки, чтобы можно было выполнить предварительный просмотр предложения.

Чтобы продолжить, выберите **Сохранить черновик**.

## <a name="plan-overview"></a>Общие сведения о плане

Каждое предложение должно иметь один или несколько планов (ранее именуемых SKU). Вы можете добавить несколько планов для поддержки различных наборов функций по разным ценам или для настройки определенного плана для ограниченной аудитории конкретных клиентов. Клиенты могут просматривать планы, доступные для них в родительском предложении.

На странице **Обзор планов** выберите **+ Создать новый план**. Затем введите **Идентификатор плана** и **Имя плана**. Оба эти значения могут содержать только алфавитно-цифровые знаки в нижнем регистре, тире и символы подчеркивания, не более 50 символов. Эти значения могут быть видимыми для клиентов, и их нельзя изменить после публикации предложения.

Выберите **Создать** после введения этих значений, чтобы продолжить работу с планом. Для заполнения доступно три раздела: **Описание плана**, **Цены и доступность**, **Техническая конфигурация**.

### <a name="plan-listing"></a>Описание плана

Во-первых, укажите **Сводку результатов поиска** для плана. Это краткое (не более 100 символов) описание плана, которое будет отображаться в результатах поиска по Marketplace.

Далее введите **Описание**, которое предоставляет более подробные характеристики плана.

### <a name="pricing-and-availability"></a>Цены и доступность

В настоящее время существует только одна модель ценообразования, которую можно использовать для предложения управляемой службы: **С использованием собственной лицензии (BYOL)** . Это означает, что вы будете выставлять клиентам счета непосредственно по затратам, связанным с этим предложением, и корпорация Майкрософт не будет взимать с вас плату.

Раздел **Видимость плана** позволяет указать, должен ли этот план быть [частным](../../marketplace/private-offers.md). Если не выставлять флажок **Это частный план**, ваш план не будет ограничен конкретными клиентами (или определенным числом клиентов).

Чтобы сделать этот план доступным только для конкретных клиентов, выберите **Да**. После этого необходимо определить клиентов, указав их идентификаторы подписок. Их можно вводить по одному (до 10 подписок) или путем отправки CSV-файла (до 10 000 подписок для всех планов). Обязательно включите собственные подписки, чтобы можно было протестировать предложение.

> [!IMPORTANT]
> После опубликования плана как общедоступного его нельзя будет превратить в частный. Чтобы контролировать, какие клиенты могут принимать ваши предложения и делегировать ресурсы, используйте частный план. При использовании общедоступного плана нельзя будет ограничить его доступность для определенных клиентов или даже для определенного количества клиентов, хотя вы сможете при желании полностью прекратить продажу плана. Вы можете [удалить доступ к делегированию](../../lighthouse/how-to/remove-delegation.md) после того, как клиент принял предложение, только если при публикации предложения была включена **Авторизация** с **Определением роли**, установленным на [Роль для удаления назначения регистрации управляемых служб](../../role-based-access-control/built-in-roles.md#managed-services-registration-assignment-delete-role). Вы также можете обратиться к клиенту и попросите его [удалить ваш доступ](../../lighthouse/how-to/view-manage-service-providers.md#add-or-remove-service-provider-offers).

### <a name="technical-configuration"></a>Техническая конфигурация

В этом разделе плана создается манифест с данными авторизации для управления ресурсами клиента. Эти сведения необходимы для включения [делегированного управления ресурсами Azure](../../lighthouse/concepts/azure-delegated-resource-management.md).

Не забудьте изучить [Клиенты, роли и пользователи в сценариях Azure Lighthouse](../../lighthouse/concepts/tenants-users-roles.md#best-practices-for-defining-users-and-roles), чтобы понять, какие роли поддерживаются, а также рекомендации по определению ваших авторизаций.

> [!NOTE]
> Как отмечалось выше, пользователи и роли в записях **Авторизация** будут действовать для всех клиентов, которые приобрели план. Если вы хотите предоставить доступ только конкретному клиенту, необходимо опубликовать частный план исключительно для него.

#### <a name="manifest"></a>Manifest

Сначала укажите **версию** манифеста. Используйте формат *n. n. n* (например, 1.2.5).

Затем заполните поле **Идентификатор клиента**. Это глобальный уникальный идентификатор, связанный с идентификатором клиента Azure Active Directory (Azure AD) вашей организации (т. е. клиентом Azure, из которого вы будете получать доступ к ресурсам клиентов). Если у вас нет его под рукой, его можно найти, наведя указатель мыши на имя учетной записи в правом верхнем углу портала Azure либо выбрав **Переключение каталога**.

Если вы публикуете новую версию предложения и хотите создать обновленный манифест, выберите **+ Создать манифест**. Не забудьте увеличить номер версии по сравнению с предыдущей версией манифеста.

#### <a name="authorization"></a>Авторизация

Авторизации определяют сущности в управляющем клиенте Azure, которые имеют доступ к ресурсам и подпискам для клиентов, купивших план. Каждой из этих сущностей назначается встроенная роль, предоставляющая определенные уровни доступа.

Для каждого плана можно создать до двадцати авторизаций.

> [!TIP]
> В большинстве случаев роли необходимо назначить группе пользователей или субъекту-службе Azure AD, а не ряду отдельных учетных записей пользователей. В этом случае вы сможете добавлять или удалять доступ для отдельных пользователей, не обновляя и повторно публикуя план при изменении требований к доступу. При назначении ролей группам Azure AD [убедитесь, что **Тип группы** — это **Безопасность**, а не **Office 365**](../../active-directory/fundamentals/active-directory-groups-create-azure-portal.md). Дополнительные рекомендации см. в разделе [Арендаторы, роли и пользователи в сценариях Azure Lighthouse](../../lighthouse/concepts/tenants-users-roles.md).

Для каждой **авторизации** укажите следующее. После этого вы можете выбирать **+ Добавить авторизацию** столько раз, сколько потребуется для добавления дополнительных пользователей и определений ролей.

- **Идентификатор объекта Azure AD**. Идентификатор пользователя, группы пользователей или приложения в Azure AD, которому предоставляются определенные разрешения (как описано в пункте определения роли) на ресурсы ваших клиентов.
- **Отображаемое имя объекта Azure AD**. Понятное имя, помогающее клиенту понять назначение авторизации. Клиент увидит это имя при делегировании ресурсов.
- **Определение роли**. Выберите из списка одну из доступных встроенных ролей Azure AD. Эта роль определит разрешения, которые пользователь в поле **Идентификатор объекта Azure AD** будет иметь на ресурсы ваших клиентов. Описание этих ролей см. в статье [встроенные роли](../../role-based-access-control/built-in-roles.md) и [Поддержка ролей для Azure лигхсаусе](../../lighthouse/concepts/tenants-users-roles.md#role-support-for-azure-lighthouse).
  > [!NOTE]
  > По мере добавления в Azure новых встроенных ролей они станут доступны здесь, хотя возможна задержка, прежде чем они начнут отображаться.
- **Назначаемые роли**. Этот вариант появится только в том случае, если вы выбрали администратора доступа пользователей в разделе **Определение роли** для этой авторизации. В таком случае здесь необходимо добавить одну или несколько назначаемых ролей. Пользователь в поле **Идентификатор объекта Azure AD** сможет назначить эти роли [управляемым удостоверениям](../../active-directory/managed-identities-azure-resources/overview.md), которые нужны для [развертывания политик, которые можно исправить](../../lighthouse/how-to/deploy-policy-remediation.md). Обратите внимание, что к этому пользователю не будут применяться никакие другие разрешения, обычно связанные с ролью администратора доступа пользователей.

> [!TIP]
> Чтобы при необходимости можно было [удалить доступ к делегированию](../../lighthouse/how-to/remove-delegation.md), включите **Авторизацию** с **Определением роли**, установленным на [Роль удаления назначения регистрации управляемых служб](../../role-based-access-control/built-in-roles.md#managed-services-registration-assignment-delete-role). Если эта роль не назначена, делегированные ресурсы могут быть удалены только пользователем в арендаторе клиента.

После заполнения всех разделов плана можно выбирать **+ Создать новый план** столько раз, сколько необходимо для создания дополнительных планов. Когда все будет готово, нажмите кнопку **Сохранить**.

Чтобы продолжить, выберите **Сохранить черновик**.

## <a name="publish"></a>Публикация

### <a name="submit-offer-to-preview"></a>Отправка предварительной версии предложения

Заполнив все необходимые разделы предложения, выберите **Проверка и публикация** в правом верхнем углу портала.

Если вы впервые публикуете это предложение, вы можете:

- Проверить статус заполнения для каждого раздела предложения.
  - **Не начато** — вы еще не начали заполнять раздел.
  - **Не завершено** — раздел содержит ошибки, которые необходимо исправить, или требуется предоставить дополнительные сведения. Вернитесь в раздел и обновите его.
  - **Завершено** — раздел заполнен, были предоставлены все необходимые данные, ошибки отсутствуют. Вы сможете отправить предложение только после того, как все его разделы будут заполнены.
- В разделе **Примечания для сертификации** предоставьте инструкции по тестированию для группы сертификации, чтобы убедиться, что приложение правильно протестировано, а также дополнительные заметки, полезные для понимания приложения.
- Отправьте предложение для публикации, нажав **Отправить**. Мы сообщим по электронной почте, когда предварительная версия предложения станет доступна для проверки и утверждения. Вернитесь в Центр партнеров и выберите **Запустить**, чтобы опубликовать предложение для всех (или для частной аудитории).

### <a name="customer-experience-and-offer-management"></a>Взаимодействие с клиентами и управление предложениями

Когда клиент развертывает ваше предложение, он сможет делегировать одну или несколько подписок или групп ресурсов [системе делегированного управления ресурсами Azure](../../lighthouse/concepts/azure-delegated-resource-management.md). Дополнительные сведения об этом процессе см. в разделе [Процесс адаптации клиентов](../../lighthouse/how-to/publish-managed-services-offers.md#the-customer-onboarding-process).

Вы можете в любое время [опубликовать обновленную версию предложения](update-existing-offer.md). Например, вам может потребоваться добавить новое определение роли к опубликованному ранее предложению. После этого клиенты, которые уже добавили предложение, увидят значок на странице [**Поставщики услуг**](../../lighthouse/how-to/view-manage-service-providers.md) на портале Azure, который сообщит им, доступно ли обновление. Каждый клиент сможет проверить изменения и решить, требуется ли им обновление до новой версии.
