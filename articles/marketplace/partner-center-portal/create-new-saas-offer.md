# <a name="create-a-saas-offer-in-the-commercial-marketplace"></a>Создание предложения SaaS в коммерческом магазине

Чтобы приступить к созданию предложений SaaS (программное обеспечение как услуга) на коммерческой платформе, сначала необходимо [создать учетную запись Центра партнеров](./create-account.md) и открыть [панель мониторинга коммерческой платформы](https://partner.microsoft.com/dashboard/commercial-marketplace/offers), выбрав вкладку **Обзор**.

> [!NOTE]
> Если вы создаете предложение SaaS, которое можно создать, убедитесь, что вы реализуете интеграцию с [API-интерфейсами выполнения SaaS](./pc-saas-fulfillment-apis.md).  Интеграция с API — это единственный способ для правильной работы языка в Marketplace. Кроме того, необходимо убедиться, что приложение использует проверку подлинности Azure AD с единым входом (SSO). Ознакомьтесь с [Azure AD и предложениями SaaS, которые вы выдаете в коммерческом магазине](../azure-ad-saas.md).

## <a name="create-a-new-offer"></a>Создание нового предложения

1. Войдите в [Центр партнеров](https://partner.microsoft.com/dashboard/home).
2. В меню слева выберите **Коммерческая платформа** > **Обзор**.
3. На странице "Обзор" выберите **+ Новое предложение** > **Программное обеспечение как услуга**.

   ![Показано меню навигации слева.](./media/new-offer-saas.png)

> [!NOTE]
> После публикации предложения изменения, внесенные в него в центре партнеров, отображаются только в Интернет-магазинах после повторной публикации предложения. Если вы внесли изменения, обязательно выполните повторную публикацию.

## <a name="new-offer"></a>Новое предложение

Укажите **идентификатор предложения**. У каждого предложения в вашей учетной записи должен быть уникальный идентификатор.

- Идентификатор отображается для клиентов в веб-адресе предложения на Marketplace, а также, если применимо, шаблонов Azure Resource Manager.
- Вы можете использовать только строчные буквы и цифры. Идентификатор может содержать дефисы и символы подчеркивания, но не пробелы, и может включать не больше 50 символов. Например, если вы введете **test-offer-1**, веб-адрес предложения будет иметь следующий вид: `https://azuremarketplace.microsoft.com/marketplace/../test-offer-1`.
- Идентификатор предложения нельзя изменить после выбора команды **Создать**.

Заполните поле **Псевдоним предложения**. Это имя, которое используется для предложения в Центре партнеров.

- Это имя не используется в Marketplace и отличается от имени предложения и других значений, отображаемых для клиентов.
- Псевдоним предложения нельзя изменить после выбора команды **Создать**.

Выберите **Создать**, чтобы создать предложение и продолжить.

## <a name="offer-overview"></a>Обзор предложения

В окне **Состояние публикации** отображается визуальное представление шагов, необходимых для публикации этого предложения, и время для выполнения каждого шага. Значки незавершенных шагов публикации будут выделены серым.

В меню **Обзор предложения** содержится список ссылок для выполнения операций с этим предложением. Этот список операций изменится в зависимости от вашего выбора в отношении предложения.  

- Если предложение является черновиком — "Удалить черновик".
- Если это действующее предложение — Stop sell offer (Остановить продажу предложения).
- Если пока доступна только предварительная версия предложения — "Запустить".
- Если вы еще не завершили выход из Publisher, отмените публикацию

### <a name="sell-through-microsoft"></a>Продажа через корпорацию Майкрософт

Продажи через корпорацию Майкрософт предоставляют более удобные способы нахождения и привлечения клиентов, позволяя корпорации Майкрософт выполнять транзакции Marketplace от вашего имени и использовать преимущества своих глобально доступных коммерческих возможностей.

#### <a name="saas-offer-requirements"></a>Требования к предложению SaaS

Чтобы разместить предложения SaaS (программное обеспечение как услуга) на коммерческой платформе в Центре партнеров, нужно выполнить следующие условия:

- Для управления удостоверениями и проверки подлинности ваше предложение должно использовать [Azure Active Directory (Azure AD)](https://azure.microsoft.com/services/active-directory/).
- Для интеграции с Azure Marketplace ваше предложение должно использовать [API исполнения SaaS](pc-saas-fulfillment-api-v2.md).

#### <a name="saas-pricing-and-billing-options"></a>Цены на SaaS и варианты выставления счетов

Решения SaaS выполняются в подписке издателя Azure и плата клиентов за лицензирование включает стоимость инфраструктуры, на которой развернуто программное обеспечение. Использование инфраструктуры Azure контролируется, а счета выставляются непосредственно партнеру. Фактическая стоимость использования инфраструктуры не видна пользователю. Издатели добавляют плату за использование инфраструктуры Azure в счета за лицензирование программного обеспечения.

Предложения SaaS поддерживают ежемесячное или ежегодное выставление счетов с фиксированной платой на пользователя, а также плату за использование с помощью службы выставления счетов по тарифу. Коммерческая платформа Майкрософт использует в работе модель агентства, в которой издатели устанавливают цены, корпорация Майкрософт выставляет счета клиентам и выплачивает издателю доход, удерживая комиссию агентства.

> [!NOTE]
> **Сокращенная плата за обслуживание решений Marketplace** . для определенных предложений SaaS, опубликованных на коммерческом рынке, корпорация Майкрософт сокращает плату за службу Marketplace с 20% (как описано в соглашении об издателе Microsoft Publisher) до 10%. Чтобы ваши предложения были квалифицированы, ваши предложения должны быть обозначены корпорацией Майкрософт как выгодно Azure для совместной продажи. Право на выполнение должно быть соблюдаться по крайней мере за пять (5) рабочих дней до конца каждого календарного месяца, чтобы получить уменьшенный тариф на службу Marketplace в течение месяца. Сниженные сборы за службы Marketplace также применяются к виртуальным машинам Azure для совместной продажи выгодно, управляемым приложениям и любым другим квалифицированным предложениям IaaS, доступным в коммерческом магазине.

### <a name="list-through-microsoft"></a>Публикация через корпорацию Майкрософт

Развивайте свой бизнес вместе с корпорацией Майкрософт, публикуя предложения в Marketplace. Если вы публикуете предложение, но не выполняете транзакции через корпорацию Майкрософт, это означает, что корпорация Майкрософт не принимает прямого участия в транзакциях с лицензиями на программное обеспечение. За такие транзакции не взимается плата и издатель оставляет себе 100 % всех доходов за лицензирование программного обеспечения, полученных с клиента. Однако издатель несет ответственность за поддержку всех аспектов транзакции с лицензией на программу, в том числе выполнения заказа, измерения использования, выставления счетов, оплаты и сбора.

Например, `https://contoso.com/trial/saas-app`. Бесплатные пробные версии предложения создаются, управляются, настраиваются вашей службой и не имеют подписок, управляемых корпорацией Майкрософт.

> [!NOTE]
> Маркеры, которые приложение будет получать через пробную ссылку, можно использовать только для получения сведений о пользователях с помощью Azure Active Directory для автоматизации создания учетных записей в приложении. Учетные записи Майкрософт (MSA) не поддерживают проверку подлинности с помощью этого маркера.

### <a name="standard-contract-for-the-microsoft-commercial-marketplace"></a>Стандартный контракт для коммерческой платформы Майкрософт

Корпорация Майкрософт предоставляет шаблон Стандартного контракта.

- **Используете Стандартный контракт для коммерческой платформы Майкрософт?**

Чтобы упростить процесс приобретения для клиентов и уменьшить юридические сложности для разработчиков программного обеспечения, корпорация Майкрософт предлагает Стандартный контракт для коммерческой платформы Майкрософт, который помогает упростить транзакции на платформе. Вместо того чтобы создавать собственные условия, как издатель на коммерческой платформе вы можете предложить свое программное обеспечение на основе Стандартного контракта, который клиентам надо просмотреть и принять лишь один раз. Стандартный контракт можно найти по адресу https://go.microsoft.com/fwlink/?linkid=2041178.

Вы можете использовать Стандартный контракт вместо предоставления собственных условий, установив флажок "Использовать стандартный контракт для коммерческой платформы корпорации Майкрософт?".

![Установка флажка "Использовать стандартный контракт для коммерческой платформы корпорации Майкрософт?"](./media/use-standard-contract.png)

> [!NOTE]
> После публикации предложения с использованием Стандартного контракта для коммерческой платформы Майкрософт вы уже не сможете использовать собственные условия. Это сценарий "или". Или вы предлагаете свое решение по Стандартному контракту, **или** на собственных условиях. Если вы хотите изменить условия Стандартного контракта, это можно сделать с помощью поправок к Стандартному контракту.

#### <a name="standard-contract-amendments"></a>Поправки к Стандартному контракту

Поправки к Стандартному контракту позволяют издателю выбрать стандартные условия контракта для простоты и создать условия для своего продукта или бизнеса на его основе. Клиентам необходимо будет изучить только поправки к стандартному контракту, если они уже изучили и приняли Стандартный контракт Майкрософт.

Для издателей на коммерческой платформе доступно два вида поправок:

- Универсальные поправки: применяются к стандартному контракту для всех клиентов. Универсальные поправки демонстрируются каждому клиенту в процессе покупки предложения. Чтобы использовать ваше предложение, клиенты должны принять условия Стандартного контракта и поправки.
- Индивидуальные поправки: это особые поправки к Стандартному контракту, ориентированные на конкретных клиентов по их идентификаторам клиентов Azure. Издатели могут выбирать, к каким клиентам будут применяться такие поправки. В процессе покупки условия с индивидуальными поправками будут предлагаться только клиентам с соответствующими идентификаторами.  Чтобы использовать ваше предложение, клиенты должны принять условия стандартного контракта и поправки (-ок).

>[!NOTE]
> Эти два типа поправок накладываются друг на друга. Клиенты, для которых предназначены индивидуальные поправки, в процессе покупки также сталкиваются с универсальными поправками.

**Универсальные поправки к условиям в Стандартном контракте для коммерческой платформы Майкрософт**. Введите универсальные поправки к условиям в этом поле. Вы можете указать одну универсальную поправку для всех предложений. В этом поле можно ввести неограниченное количество символов. Данные условия демонстрируются клиентам в AppSource, Azure Marketplace и/или на портале Azure во время поиска и в процессе покупки.

**Индивидуальные поправки к условиям в Стандартном контракте для коммерческой платформы Майкрософт**. Начните с выбора параметра **Add custom amendment terms** (Добавить индивидуальные поправки к условиям). Вы можете указать до десяти индивидуальных поправок к условиям для каждого предложения.

- **Индивидуальные поправки к условиям**. Введите индивидуальные поправки к условиям в поле "Индивидуальные поправки к условиям". В этом поле можно ввести неограниченное количество символов. В процессе покупки предложения на портале Azure эти индивидуальные поправки к условиям будут предлагаться только клиентам с соответствующими идентификаторами.  
- **Идентификаторы клиентов** (обязательно). Для каждой индивидуальной поправки можно указать до 20 идентификаторов клиентов. При добавлении индивидуальной поправки необходимо указать хотя бы один идентификатор клиента для ваших клиентов в Azure. Вы можете запросить у клиента этот идентификатор. Чтобы его найти, необходимо перейти по адресу portal.azure.com и выбрать Active Directory Azure > Свойства. Значение идентификатора каталога — это идентификатор клиента (например, 50c464d3-4930-494c-963c-1e951d15360e). Вы также можете найти идентификатор клиента организации, используя URL-адрес доменного имени, где [находится мой Microsoft Azure и идентификатор клиента Office 365?](https://www.whatismytenantid.com)
- **Описание** (необязательно). При необходимости введите понятное описание идентификатора клиента, по которому вы сможете легко определить, для каких клиентов предназначена поправка.

#### <a name="terms-and-conditions"></a>Условия использования

Если вы хотите предоставить собственные условия, вы можете ввести их в поле условий. В этом поле можно ввести до 10 000 символов. Если для ваших условий требуется более длинное описание, введите в это поле ссылку с URL-адресом, по которой можно найти условия.

Клиентам будет предложено принять эти условия, прежде чем использовать это предложение.

## <a name="technical-configuration"></a>Техническая конфигурация

На вкладке **Техническая конфигурация** определяются технические сведения, используемые Marketplace для обмена данными со службой SaaS. Это подключение позволяет нам подготавливать ваше предложение для конечного клиента, если оно запрашивает и управляет им.

>[!NOTE]
>Перед настройкой сведений о предложении необходимо реализовать интеграцию с [API-интерфейсами выполнения SaaS](./pc-saas-fulfillment-api-v2.md) . Необходимо также создать целевую страницу, и приложение должно использовать проверку подлинности Azure AD с единым входом (SSO). Дополнительные сведения см. [в статье Azure AD и предложения SaaS, предоставляемые в коммерческом магазине](../azure-ad-saas.md).

Схемы и подробные пояснения, описывающие использование собранных полей, доступны в документации по [интерфейсам API](./pc-saas-fulfillment-api-v2.md).

- **URL-адрес целевой страницы** (обязательно) — определите URL-адрес сайта SaaS (например `https://contoso.com/signup` ,), который будут передаваться конечным пользователям после получения предложения из Marketplace и активации процесса настройки из вновь созданной подписки SaaS.  Этот URL-адрес будет вызван с параметром маркера идентификации покупки Marketplace, который уникальным образом идентифицирует покупку SaaS конкретного клиента.  Этот маркер необходимо отменять для получения соответствующих сведений о подписке SaaS с помощью API-интерфейса [Resolve](./pc-saas-fulfillment-api-v2.md#resolve-a-purchased-subscription) .  Эти сведения и все остальные, которые вы хотите собрать, следует использовать как часть интерактивной веб-страницы, созданной клиентом для выполнения регистрации клиента и активации покупки.  На этой странице пользователь должен зарегистрировать проверку подлинности одним щелчком с помощью Azure Active Directory (Azure AD). <br> <br> Этот URL-адрес с параметром маркера идентификации покупки Marketplace также будет вызываться, когда конечный клиент запускает управляемый интерфейс SaaS из центра администрирования портал Azure или M365. Оба потока следует обрабатывайте, когда маркер предоставляется в первый раз после приобретения новых клиентов и предоставляется для существующего клиента, управляющего SaaS. <br> <br> Целевая страница, которую вы настраиваете здесь, должна быть установлена и запущена 24/7. Это единственный способ уведомления о новых покупках предложений SaaS, сделанных в Marketplace, или запросы на настройку действующей подписки на предложение.

- **Веб-перехватчик подключения** (обязательно). для всех асинхронных событий, которые должны быть отправлены корпорацией Майкрософт (например, подписка SaaS была отменена), необходимо указать URL-адрес веб-перехватчика подключения. Мы будем называть этот URL-адрес для уведомления о событии. <br> <br> Веб-перехватчик, который вы задаете, должен быть установлен и работает 24/7, так как это единственный способ получать уведомления об обновлениях подписок SaaS, приобретенных в Marketplace.  Если у вас еще нет системы веб-перехватчика, простейшей конфигурацией является наличие приложения логики HTTP-конечной точки, которое будет прослушивать все события, отправляемые в него, а затем соответствующим образом обрабатывает их (например, `https://prod-1westus.logic.azure.com:443/work` ). Подробнее см. статью [Вызовы, срабатывания триггеров и создание вложенных рабочих процессов в приложениях логики](../../logic-apps/logic-apps-http-endpoint.md).

- **Идентификатор клиента Azure AD** (обязательный). в портал Azure необходимо [создать приложение Azure Active Directory (AD)](../../active-directory/develop/howto-create-service-principal-portal.md) , чтобы мы могли проверить подключение между двумя службами за счет прошедшего проверку подлинности соединения. Чтобы найти [идентификатор клиента](../../active-directory/develop/howto-create-service-principal-portal.md#get-tenant-and-app-id-values-for-signing-in), перейдите к Azure Active Directory и выберите **свойства**, а затем найдите **идентификатор каталога** (например, 50c464d3-4930-494c-963c-1e951d15360e).

- **Идентификатор приложения Azure AD** (обязательно) — также требуется [идентификатор приложения](../../active-directory/develop/howto-create-service-principal-portal.md#get-tenant-and-app-id-values-for-signing-in). Чтобы получить его значение, перейдите к Azure Active Directory и выберите **Регистрация приложений**, а затем найдите **идентификатор приложения** в списке (например, `50c464d3-4930-494c-963c-1e951d15360e` ).

>[!NOTE]
>Идентификатор приложения Azure AD связан с ИДЕНТИФИКАТОРом издателя в учетной записи центра партнеров.  Убедитесь, что в ваших предложениях используется один и тот же идентификатор приложения.

>[!NOTE]
>Если у издателя есть две или более разных учетных записей в центре партнеров, следует использовать два или более разных идентификатора приложения Azure AD, каждое для одной из учетных записей. Каждая учетная запись партнера в центре партнеров должна использовать уникальный идентификатор приложения Azure AD для всех предложений SaaS, опубликованных с помощью этой учетной записи.

Чтобы продолжить, выберите **Сохранить черновик**.

## <a name="plan-overview"></a>Общие сведения о плане

На этой странице можно указать различные параметры плана в рамках одного предложения. Эти планы (иногда называемые SKU) могут отличаться по версии, монетизации или уровням обслуживания. Чтобы опубликовать ваше предложение на Marketplace, необходимо настроить по крайней мере один план.

Вы можете создать до 100 планов для каждого предложения: до 45 могут быть частными. Узнайте больше о частных планах в [частных предложениях в коммерческом магазине Майкрософт](../private-offers.md).

После создания вы увидите имена планов, идентификаторы, модели ценообразования, доступность (открытый или частный), текущее состояние публикации и все доступные действия.

**Действия**, доступные в области **Обзор планов**, зависят от текущего состояния плана и могут быть следующими:

- Если план находится в состоянии **черновика** — "Удалить черновик".
- Если план **опубликован** — Stop sell plan (Закончить продажу плана) или Sync private audience (Синхронизировать частную аудиторию).

**Создание нового плана** (минимум один план для тех, кто осуществляет продажи через корпорацию Майкрософт)

- **Идентификатор плана.** Создайте уникальный идентификатор для каждого плана в этом предложении. Этот идентификатор будет отображаться для клиентов в URL-адресе продукта и шаблонах Azure Resource Manager (если применимо). Используйте только строчные буквы, цифры, дефисы и символы подчеркивания. Для этого идентификатора плана разрешено не более 50 символов. Этот идентификатор нельзя изменить после создания.
- **Имя плана.** Клиенты будут видеть это имя при выборе плана в вашем предложении. Создайте уникальное имя плана в предложении. Имя плана используется для различения планов ПО, которые могут быть частью одного предложения (например, имя предложения: Windows Server; планы: Windows Server 2016, Windows Server 2019).

#### <a name="pricing"></a>Цены

##### <a name="pricing-model"></a>Модель ценообразования

**Фиксированная ставка**. Предоставьте доступ к вашему предложению, используя одну ежемесячную или годовую фиксированную ставку. Это иногда называют ценообразованием на основе сайтов. С помощью этой модели ценообразования можно дополнительно определить тарифные планы, которые используют API службы контроля Marketplace для взимания платы с клиентов за использование нестандартных единиц.  Дополнительные сведения о выставлении счетов по тарифу с помощью службы контроля Marketplace см. в статье [Выставление счетов по тарифу с помощью службы из Marketplace для контроля потребления](./saas-metered-billing.md).  Этот параметр также следует использовать, если поведение используется в пакетах для службы SaaS.  Мы не рекомендуем клиентам часто переключать планы на ежедневной или ежечасной основе.

**На пользователя**. Предоставьте доступ к предложению по цене, основанной на количестве пользователей, которые получают к нему доступ и занимают места. Эта модель ценообразования на основе пользователей позволяет задать минимальное и максимальное количество допустимых пользователей в зависимости от цен. Таким образом с помощью нескольких планов можно настроить различные цены на основе числа пользователей.  Эти поля необязательные. Если значение слева не выбрано, число пользователей будет интерпретировано как не имеющее ограничения (минимум — 1, максимум — сколько поддерживает система). Эти поля могут быть изменены в рамках обновления плана.

После публикации невозможно изменить выбранную модель ценообразования для выставления счетов. Кроме того, все планы для одного предложения должны использовать одну и ту же модель ценообразования.

##### <a name="user-limits"></a>Ограничения пользователей

Если применимо, выберите и установите минимальные и максимальные ограничения пользователей.

##### <a name="billing-term-and-price"></a>Срок выставления счетов и цена

Выберите **срок** и **цену**, которую клиенты должны платить. Необходимо указать по крайней мере одну ежемесячную или годовую ставку либо предоставить клиентам оба варианта.

Цены, заданные в долларах США (USD), переводятся в локальную валюту всех выбранных рынков по текущему курсу во время сохранения. Проверяйте эти цены перед публикацией, экспортировав электронную таблицу цен и просмотрев цену на каждом рынке. Если вы хотите задать другие цены на отдельном рынке, измените и импортируйте электронную таблицу с ценами. Эти параметры вы определяете самостоятельно, поэтому мы рекомендуем проверить их.
*\*Сначала сохраните изменения в ценах, чтобы включить возможность экспорта данных о ценах.*

Внимательно проверяйте цены перед публикацией, так как существуют некоторые ограничения на изменения после публикации плана:

- После публикации плана модель ценообразования нельзя будет изменить.
- После публикации срока выставления счетов для плана его нельзя будет удалить.
- После публикации плана цены для отдельных рынков в плане нельзя будет изменить.

#### <a name="free-trial"></a>"Бесплатная пробная версия"

Предложения SaaS на коммерческом рынке позволяют предоставить бесплатную пробную версию на один месяц при продаже через корпорацию Майкрософт. Для всех моделей и условий выставления счетов, за исключением тарифных планов, поддерживаются бесплатные пробные версии. Месяц бесплатного доступа к пробным версиям позволяет клиентам иметь низкий порог вхождения.  Если вы решили включить бесплатную пробную версию для планов в рамках вашего предложения, клиент не сможет перейти на платную подписку до окончания начального периода в один месяц.  В течение этого времени клиенты, приобретающие предложение, могут испытать все поддерживаемые планы со включенной бесплатной пробной версией и преобразовывать их из одного в другой.  Преобразование в платную подписку выполняется автоматически в конце срока.

>[!NOTE]
>Если клиент решил преобразовать план без бесплатных пробных версий, преобразование произойдет, но возможность использования бесплатной пробной версии будет сразу же утрачена. Кроме того, после того как клиент начинает платить за план, он больше не сможет получить бесплатную пробную версию в той же подписке, даже если он перейдет на использование SKU с поддержкой бесплатных пробных версий.

Здесь вы можете настроить бесплатную пробную версию для каждого плана в предложении. Для этого установите следующий флажок, чтобы использовать пробную версию в течение одного месяца.

![Флажок "Разрешение бесплатной пробной версии на один месяц"](./media/free-trial-enable.png)

>[!NOTE]
>После публикации предложения с бесплатной пробной версией, поддерживающего транзакции предложения, его нельзя отключить для этого плана. Убедитесь, что при первой публикации этот параметр указан правильно, чтобы не создавать повторно план.

Чтобы получить сведения о подписках клиентов, которые работают с бесплатной пробной версией, используйте новое свойство API `isFreeTrial`, которое будет помечено как true или false. Дополнительные сведения см. в разделе об [API получения подписки SaaS](pc-saas-fulfillment-api-v2.md#get-subscription).

>[!NOTE]
>Бесплатные пробные версии не поддерживаются для планов, использующих службу из Marketplace для контроля потребления.

#### <a name="plan-visibility"></a>Видимость плана

Каждый план можно настроить для всех пользователей или только для конкретной аудитории по выбору. Для указания ограниченной аудитории используйте идентификаторы клиента Azure.

##### <a name="privacy"></a>Конфиденциальность

Выберите **Это частный план**, чтобы сделать ваш план закрытым и видимым только ограниченной аудитории по выбору. После публикации в качестве частного плана можно обновить аудиторию или сделать план доступным для всех. После того как план опубликован для всех, он должен оставаться видимым для всех. Его нельзя повторно преобразовать в частный план.

##### <a name="restricted-audience-tenant-ids"></a>**Ограниченная аудитория (идентификаторы клиента)**

Назначьте аудиторию, которая будет иметь доступ к этому частному плану. Доступ назначается с помощью идентификаторов клиента с возможностью добавить описание каждого назначенного идентификатора клиента. Вы можете добавить не более 10 идентификаторов клиента или 20 000 идентификаторов клиента заказчиков (при импорте CSV-файла электронной таблицы).

Клиент представляет организацию. Он имеет идентификатор в виде GUID (глобальный уникальный идентификатор, 128-разрядное целое число, используемое для идентификации ресурсов). Это выделенный экземпляр службы Azure AD, который организация или разработчик приложения получает после начала сотрудничества с корпорацией Майкрософт, то есть после регистрации в Azure, Microsoft Intune или Microsoft 365. Каждый клиент Azure AD отделен от остальных клиентов Azure AD. Чтобы проверить клиента, войдите на портал Azure под учетной записью, которую вы хотите использовать для управления приложением. Если у вас есть клиент, вы автоматически войдете в него и сможете увидеть имя клиента непосредственно под своей учетной записью. Наведите указатель мыши на имя учетной записи в верхней правой части портала Azure, чтобы увидеть свое имя, адрес электронной почты, идентификатор каталога или клиента (GUID), а также домен. Если ваша учетная запись связана с несколькими клиентами, выберите имя учетной записи, чтобы открыть меню, где можно переключаться между клиентами. У каждого клиента есть свой идентификатор. Вы также можете найти идентификатор клиента своей организации, используя URL-адрес доменного имени: [https://www.whatismytenantid.com](https://www.whatismytenantid.com).

В то время как в предложениях SaaS используются идентификаторы клиента для определения частной аудитории, в других типах предложений могут использоваться идентификаторы подписки Azure (которые также представляются в виде GUID).

> [!NOTE]
> Частная (или ограниченная) аудитория отличается от аудитории предварительной версии. На странице **[Предварительная версия](#preview-audience)** можно определить аудиторию предварительной версии. Аудитория предварительной версии может получить доступ к вашему предложению *до* его публикации на Marketplace. В то время как назначение частной аудитории относится только к определенному плану, аудитория предварительной версии может просматривать все планы (частные и общедоступные) только в течение ограниченного периода предварительной версии во время тестирования и проверки плана.

Чтобы продолжить, выберите **Сохранить черновик**.

### <a name="submit-offer-to-preview"></a>Отправка предварительной версии предложения

Если вы впервые публикуете это предложение, вы можете:

- Проверьте статус заполнения каждого раздела предложения.
  - **Не начато** — вы еще не начали заполнять раздел.
  - **Не завершено** — раздел содержит ошибки, которые необходимо исправить, или требуется предоставить дополнительные сведения. Необходимо вернуться к разделу и обновить его.
  - **Завершено** — раздел заполнен, были предоставлены все необходимые данные, ошибки отсутствуют. Вы сможете отправить предложение только после того, как все его разделы будут заполнены.
- Предоставьте инструкции по тестированию для группы сертификации, чтобы убедиться, что приложение правильно проверено, а также какие-либо дополнительные заметки, полезные для понимания приложения.
- Отправьте предложение для публикации, нажав **Отправить**. Мы сообщим, когда предварительная версия предложения станет доступна для проверки и утверждения, по электронной почте. Вернитесь в центр партнеров и выберите **Go-Live** , чтобы опубликовать предложение в общедоступной аудитории (или в частном предложении).
