# <a name="metered-billing-using-the-marketplace-metering-service"></a>Измеренный биллинг с помощью службы учета рынка

С помощью службы учета Marketplace вы можете создавать предложения по программному обеспечению как услуге (SaaS) в программе коммерческого рынка, которые заряжаются в соответствии с нестандартными единицами.  Перед публикацией этого предложения вы определяете размеры биллинга, такие как пропускная способность, билеты или обработанные электронные письма.  Затем клиенты платят в соответствии с их потреблением этих размеров, при этом ваша система информирует корпорацию Майкрософт через API учета Marketplace о платежных событиях по мере их возникновения.  

## <a name="prerequisites-for-metered-billing"></a>Предпосылки для дозированного выставления счетов

Для того, чтобы предложение SaaS использовать дозированный биллинг, оно должно:

* Удовлетворить все требования к [предложению](https://docs.microsoft.com/azure/marketplace/partner-center-portal/create-new-saas-offer#sell-through-microsoft) для продажи через предложение Microsoft, как указано в [Создать предложение SaaS](https://docs.microsoft.com/azure/marketplace/partner-center-portal/create-new-saas-offer).
* Интегрируйтесь с [AAS Fulfillment AIS](https://docs.microsoft.com/azure/marketplace/partner-center-portal/pc-saas-fulfillment-api-v2) для клиентов, чтобы обеспечить и подключиться к вашему предложению.  
* Будьте настроены для модели ценообразования **с фиксированной ставкой** для зарядки клиентов за ваш сервис.  Размеры являются дополнительным дополнением к модели ценообразования с фиксированной ставкой. 
* Интегрируйтесь с [AI-службы учета Marketplace](./marketplace-metering-service-apis.md) для информирования корпорации Майкрософт о платежных событиях.

>[!Note]
>Служба учета marketplace доступна только для модели выставления счетов по фиксированной ставке и не применяется к модели выставления счетов на одного пользователя.

## <a name="how-metered-billing-fits-in-with-pricing"></a>Как дозированные биллинга вписывается в ценообразование

Когда дело доходит до определения предложения вместе с его ценовыми моделями, важно понимать иерархию предложения.

* Каждое предложение SaaS настроено на продажу либо через Microsoft, либо нет.  Эта настройка не может быть изменена после публикации предложения.
* Каждое предложение SaaS, настроенное для продажи через корпорацию Майкрософт, может иметь один или несколько планов. Пользователь подписывается на предложение SaaS, но оно приобретается через корпорацию Майкрософт в контексте плана.
* Каждый план имеет модель ценообразования, связанную с ним: **фиксированная ставка** или **на пользователя.** Все планы в предложении должны быть связаны с одной и той же моделью ценообразования. Например, не может быть предложения, когда один из его планов является моделью ценообразования с фиксированной ставкой, а другой - моделью ценообразования для пользователей.
* В каждом плане, настроенном для модели выставления счетов с фиксированной ставкой, включается по крайней мере одна повторяющаяся плата (которая может составлять $0):
    * Повторяющаяся **ежемесячная** плата: фиксированная ежемесячная плата, которая предоплачивается за ежемесячный повторение при покупке пользователем плана.
    * Повторяющаяся **ежегодная** плата: фиксированная ежегодная плата, которая предоплачивается на ежегодном повторении при покупке пользователем плана.
* В дополнение к периодическим сборам, план может также включать дополнительные размеры, используемые для взимания платы с клиентов за использование, не включенное в фиксированную ставку.   Каждое измерение представляет собой единицу платежа, с помощью службы [измерения Marketplace](./marketplace-metering-service-apis.md)— это единица, которую служба службы службы связи будет сообщать корпорации Майкрософт.

## <a name="sample-offer"></a>Примерное предложение

Например, Contoso является издателем с сервисом SaaS под названием Contoso Notification Services (CNS). CNS позволяет клиентам отправлять уведомления либо по электронной почте или текст. Contoso зарегистрирована в качестве издателя в Партнерской программе коммерческого рынка для публикации предложений клиентам Azure.  Есть два плана, связанные с ЦНС, изложенные ниже:

* Базовый план
    * Отправить 10000 писем и 1000 текстов за $ 0/month
    * Помимо 10000 писем, платить $ 1 за каждые 100 писем
    * Помимо 1000 текстов, платить $ 0,02 за каждый текст
* План "Премиум"
    * Отправить 50000 писем и 10000 текстов за $ 350/month
    * Помимо 50000 писем, платить $ 0,5 за каждые 100 писем
    * Помимо 10000 текстов, платить $ 0,01 за каждый текст

Клиент Azure, подписавшийся на службу CNS, сможет отправлять включенное количество текстовых сообщений и электронных писем в месяц на основе выбранного плана.  Contoso измеряет использование до включенного количества без отправки каких-либо событий использования в корпорацию Майкрософт.  Когда клиенты потребляют больше, чем включено количество, они не должны менять планы или делать что-нибудь другое.  Contoso измерит перерасход за пределами включенного количества и начнет излучать события использования в корпорацию Майкрософт для дополнительного использования с помощью [API службы учета Marketplace.](./marketplace-metering-service-apis.md)  Корпорация Майкрософт, в свою очередь, взимает с клиента плату за дополнительное использование, указанное издателем.

## <a name="billing-dimensions"></a>Размеры биллинга

Размеры выставления счетов используются для передачи информации клиенту о том, как им будут выставлены счета за использование программного обеспечения, а также для передачи событий использования в корпорацию Майкрософт. Они определяются следующим образом:

* **Идентификатор размера**: неизменяемый идентификатор, на который ссылаются при испускании событий использования.
* **Имя размера**: имя дисплея связанное с размером, например «посланные текстовые сообщения».
* **Единица измерения**: описание платежного блока, например, "за текстовое сообщение" или "на 100 писем".
* **Цена за единицу:** цена за одну единицу измерения.  
* **Включено количество для ежемесячного термина**: количество измерения, включенного в месяц для клиентов, платящих повторяющуюся ежемесячную плату, должно быть бесчисленно.
* **Включено количество для годового срока**: количество измерений, включенных в месяц для клиентов, уплачивающих повторяющиеся ежегодные сборы, должны быть бесчислены.

Размеры биллинга распределяются по всем планам предложения.  Некоторые атрибуты применяются к измерению во всех планах, а другие атрибуты специфичны для планирования.

Атрибуты, определяющие само измерение, распределяются по всем планам предложения.  Перед публикацией предложения изменение этих атрибутов из контекста любого плана повлияет на определение измерения во всех планах.  Как только вы опубликуете предложение, эти атрибуты больше не будут отсечены.  Они перечислены ниже.

* Идентификатор
* name
* Единица измерения

Другие атрибуты измерения специфичны для каждого плана и могут иметь различные значения от плана к плану.  Перед публикацией плана вы можете отсеивать эти значения, и только этот план будет затронут.  После публикации плана эти атрибуты больше не будут отсечены.  Они перечислены ниже.

* цена за единицу;
* Включено количество для ежемесячных клиентов 
* Включено количество для годовых клиентов 

Размеры также имеют два специальных понятия: "включен" и "бесконечный":

* **Включено** указывает на то, что этот план участвует в этом измерении.  Возможно, вы захотите оставить это непроверенным, если вы создаете новый план, который не отправляет события использования на основе этого измерения.  Кроме того, любые новые измерения, добавленные после того, как план был впервые опубликован, будут отображаться как "не включенные" в уже опубликованном плане.  Отключенное измерение не будет отображаться ни в каких списках измерений для плана, просматриваемого клиентами.
* **Бесконечный**, представленный символом бесконечности «Я», указывает на то, что этот план участвует в этом измерении, но не измеряет использование против этого измерения.  Если вы хотите указать своим клиентам, что функциональность, представленная этим измерением, включена в план, но без ограничений на использование.  Измерение с бесконечным использованием будет отображаться в списках измерений для плана, увиденного клиентами, с указанием, что он никогда не понесет плату за этот план.

>[!Note] 
>Явно поддерживаются следующие сценарии: <br> - Вы можете добавить новое измерение в новый план.  Новое измерение не будет включено для любых уже опубликованных планов. <br> - Вы можете опубликовать план **с фиксированной ставкой** без каких-либо измерений, а затем добавить новый план и настроить новое измерение для этого плана. Новое измерение не будет включено для уже опубликованных планов.

## <a name="constraints"></a>Ограничения

### <a name="trial-behavior"></a>Пробное поведение

Измеренный биллинг с использованием службы учета рынка не совместим с предложением бесплатной пробной версии.  Невозможно настроить план для использования как дозированного биллинга, так и бесплатной пробной версии.

### <a name="locking-behavior"></a>Режим блокировки

Поскольку измерение, используемое в службе учета Marketplace, представляет собой понимание того, как клиент будет платить за услугу, все детали для измерения больше не могут быть отображены после публикации.  Важно, чтобы ваши размеры были полностью определены для плана перед публикацией.
  
После публикации предложения с измерением детали уровня предложения для этого измерения больше не могут быть изменены:

* Идентификатор
* name
* Единица измерения

После публикации плана детали уровня плана больше не могут быть изменены:

* цена за единицу;
* Включено количество для месячного срока
* Включено количество на годовой срок
* Включено ли измерение для плана

### <a name="upper-limits"></a>Верхние пределы

Максимальное количество измерений, которые могут быть настроены для одного предложения, составляет 18 уникальных измерений.

## <a name="get-support"></a>Получение поддержки

Если у вас есть один из следующих, вы можете открыть билет поддержки.

* Технические проблемы с API службы учета рынка.
* Проблема, которая должна быть эскалация из-за ошибки или ошибки на вашей стороне (например. неправильное событие использования).
* Любые другие вопросы, связанные с дозированным выставлением счетов. 

Следуйте ниже, чтобы отправить свой билет поддержки:

1. Перейдите на [страницу службы поддержки](https://support.microsoft.com/supportforbusiness/productselection?sapId=48734891-ee9a-5d77-bf29-82bf8d8111ff). Первые несколько меню выпадения автоматически заполняются для вас. Для поддержки Marketplace определите семейство продуктов как **облачные и онлайн-сервисы,** продукт как **издатель Marketplace.**  Не изменяйте преднаселенные выборы меню выпадающих.
2. В соответствии с "Выбрать версию продукта", выберите **Live предложение управления**.
3. В рубрике "Выберите категорию, которая лучше всего описывает проблему", выберите **приложения SaaS.**
4. В разделе "Выбрать проблему, которая лучше всего описывает проблему", выберите **дозированный биллинг.**
5. Выбрав кнопку **«Следующая»,** вы будете направлены на страницу **подробностей,** где вы можете ввести более подробную информацию о вашей проблеме.

Смотрите [Поддержку программы коммерческого рынка в Partner Center](https://docs.microsoft.com/azure/marketplace/partner-center-portal/support) для получения дополнительных вариантов поддержки издателей.
