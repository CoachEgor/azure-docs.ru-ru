# <a name="live-event-types-comparison"></a>Сравнение типов событий потоковой трансляции

В Службах мультимедиа Azure есть два типа [событий потоковой трансляции](https://docs.microsoft.com/rest/api/media/liveevents): кодирование в реальном времени или сквозной режим. 

## <a name="types-comparison"></a>Сравнение типов 

В следующей таблице сравниваются возможности типов событие прямой трансляции. Типы устанавливаются во время создания с помощью [LiveEventEncodingType](https://docs.microsoft.com/rest/api/media/liveevents/create#liveeventencodingtype):

* **LiveEventEncodingType.None** -локальный динамический кодировщик передает несколько скоростью потока. Переданные потоки проходят через событие прямой трансляции без дальнейшей обработки. 
* **LiveEventEncodingType.Standard** — это локальный динамический кодировщик передает односкоростной поток в событие прямой трансляции и служб мультимедиа создает несколько потоков битрейтов. Если веб-канала вклад 720p или более высокого разрешения **Default720p** Предустановка кодируют набор пар 6 разрешение и битрейт (Подробное описание ниже чуть позже).
* **LiveEventEncodingType.Premium1080p** — это локальный динамический кодировщик передает односкоростной поток в событие прямой трансляции и служб мультимедиа создает несколько потоков битрейтов. Предустановка Default1080p Указывает выходной набор пар разрешение и битрейт (Подробное описание ниже чуть позже).

## <a name="system-presets"></a>Системные предустановки

Разрешений и скоростей, содержащихся в выходных данных из динамического кодировщика определяются [presetName](https://docs.microsoft.com/rest/api/media/liveevents/create#liveeventencoding). При использовании **стандартный** live encoder (LiveEventEncodingType.Standard), а затем *Default720p* Предустановка определяет набор пар 6 разрешение и битрейт описано ниже. В противном случае, если с помощью **Premium1080p** live encoder (LiveEventEncodingType.Premium1080p), а затем *Default1080p* Предустановка определяет выходной набор пар разрешение и битрейт.

> [!NOTE]
> Невозможно применить Default1080p, стиль интерактивное событие, если он установлен и настроен для стандартных кодирование в реальном времени — вы получите ошибку. Вы также получите ошибку при попытке применить Default720p, стиль Premium1080p динамического кодировщика.

## <a name="implicit-properties-of-the-live-encoder"></a>Неявные свойства динамического кодировщика

Свойства динамический кодировщик, который можно управлять явно, с помощью стиля — например число уровней, разрешений и скоростей описано в предыдущем разделе. В этом разделе поясняются неявное свойства.

### <a name="group-of-pictures-gop-duration"></a>Группы изображений (GOP) длительности

Соответствует динамическому кодировщику [GOP](https://en.wikipedia.org/wiki/Group_of_pictures) структура вклада канал — то есть слоев выходных данных будет иметь одно и то же время GOP. Таким образом рекомендуется настроить локальный кодировщик для создания публикации веб-канала, с фиксированным длительность GOP (обычно составляющим 2 секунды). Это позволит гарантировать, что исходящие потоки HLS и MPEG-DASH в службе также была устранена длительность GOP. Небольшие вариации в длительность GOP, скорее всего скорректировать большинством устройств.

### <a name="frame-rate"></a>Частота кадров

Динамический кодировщик также соответствует продолжительность отдельные кадры в вклад канал — это означает, что слоев выходных данных будет иметь кадры с тем же длительности. Таким образом, рекомендуется настроить локальный кодировщик для создания веб-канала с фиксированным частота кадров (не более 30 кадров в секунду). Это позволит гарантировать исходящих потоков HLS и MPEG DASH от службы также исправление всех нарушений, тарифы длительности кадра. Небольшие вариации в частоты кадров может быть допустимой большинства устройств, но нет никакой гарантии, что динамический кодировщик будет преобразовано в выходное которая воспроизводится правильно. Локальный динамический кодировщик должен не пропуск кадров (например) При низком уровне заряда аккумулятора условиях) или различными частота кадров любым способом.

### <a name="resolution-of-contribution-feed-and-output-layers"></a>Разрешение публикации веб-канала и выходных данных слоев

Динамический кодировщик настроен во избежание upconverting вклад веб-канала. В результате максимальное разрешение слоев выходных данных не может превышать, вклада веб-канала.

Например при отправке веб-канала в 720p, чтобы Live события, настроенного для Default1080p кодированием в реальном времени, выходные данные будут иметь только 5 слоев, начиная с разрешением 720p 3 Мбит / с, перейдите вниз до 1080 p в 200 Кбит/с. Или при отправке веб-канала в 360p в реальном времени события, настроенного для Standard кодирования в реальном времени, выход будет содержать 3 уровней (с разрешением 288p, 216p и 192p). В вырожденного варианта при отправке веб-канал вклада точек, скажем, 160 x 90 динамического кодировщика — стандартный, выход будет содержать один слой с разрешением 160 x 90 же скоростью, что и вклад веб-канала.

### <a name="bitrate-of-contribution-feed-and-output-layers"></a>Скорость публикации веб-канала и выходных данных слоев

Динамический кодировщик настроен на станет поддерживать параметры битрейт в предустановке, вне зависимости от скорости вклада веб-канала. В результате битрейт выходных данных слоев могут выходить за границы домена вклад веб-канала. Например, при отправке веб-канала с разрешением 720p 1 Мбит/с, выходных данных слоев останется таким же, как и в [таблицы](live-event-types-comparison.md#output-video-streams-for-default720p) выше.
