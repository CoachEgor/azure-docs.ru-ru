# <a name="authorize-access-to-azure-app-configuration-using-azure-active-directory"></a>Разрешить доступ к конфигурации приложений Azure с помощью active-каталога Azure
Конфигурация приложений Azure поддерживает использование Active Directory Azure (Azure AD) для авторизации запросов в экземплярах конфигурации приложений.  Azure AD позволяет использовать элементы управления доступом на основе ролей (RBAC) для выдачи разрешений директору службы безопасности.  Директором безопасности может быть пользователь или [директор службы приложений.](../active-directory/develop/app-objects-and-service-principals.md)  Чтобы узнать больше о ролях и назначениях ролей, [см.](../role-based-access-control/overview.md)

## <a name="overview"></a>Обзор
Необходимо авторизовать запросы, сделанные директором службы безопасности (пользователем или приложением) для доступа к ресурсу конфигурации приложений.  С Azure AD доступ к ресурсу является двухэтапным процессом.
1. Личность директора службы безопасности проверяется, и жетон OAuth 2.0 возвращается.  Имя ресурса для запроса `https://login.microsoftonline.com/{tenantID}` `{tenantID}` маркера — это место, где совпадает идентификатор арендатора Azure Active Directory, к которому принадлежит основной сервис.
2. Токен передается как часть запроса в службу конфигурации приложений для авторизации доступа к указанному ресурсу.

Шаг проверки подлинности требует, чтобы запрос приложения содержит токен доступа OAuth 2.0 во время выполнения.  Если приложение работает в рамках сущности Azure, например приложения Azure Functions, Web App Azure или VM Azure, для доступа к ресурсам можно использовать управляемое удостоверение.  Чтобы узнать, как проверить подлинность запросов, сделанных [Authenticate access to Azure App Configuration resources with Azure Active Directory and managed identities for Azure Resources](howto-integrate-azure-managed-service-identity.md)управляемой идентификацией, в конфигурацию приложений Azure, см.

Шаг авторизации требует, чтобы одна или несколько ролей RBAC были назначены директору безопасности. Конфигурация приложений Azure предоставляет роли RBAC, которые охватывают наборы разрешений для ресурсов конфигурации приложений. Роли, назначенные директору безопасности, определяют разрешения, предоставленные директору. Для получения дополнительной информации [Built-in RBAC roles for Azure App Configuration](#built-in-rbac-roles-for-azure-app-configuration)о ролях RBAC см. 

## <a name="assign-rbac-roles-for-access-rights"></a>Назначать роли RBAC для прав доступа
Azure Active Directory (Azure AD) разрешает права доступа к защищенным ресурсам с помощью [управления доступом на основе ролей (RBAC)](../role-based-access-control/overview.md).

Когда роль RBAC назначается директору безопасности Azure AD, Azure предоставляет доступ к этим ресурсам для этого принципа безопасности. Доступ осуществляется к ресурсу конфигурации приложений. Принципом безопасности Azure AD может быть пользователь, директор службы приложений или [управляемый интимент для ресурсов Azure.](../active-directory/managed-identities-azure-resources/overview.md)

## <a name="built-in-rbac-roles-for-azure-app-configuration"></a>Встроенные роли RBAC для конфигурации приложений Azure
Azure предоставляет следующие встроенные роли RBAC для авторизации доступа к данным конфигурации приложений с помощью Azure AD и OAuth:

- Владелец конфигурации приложений Azure: Используйте эту роль, чтобы предоставить доступ к ресурсам конфигурации приложений.
- Лазурный читатель конфигурации приложений: Используйте эту роль, чтобы дать доступ к ресурсам конфигурации приложений.
- Автор: Используйте эту роль, чтобы предоставить админдоступу доступ к службе без предоставления доступа к данным, хранящимся в экземпляре конфигурации приложения.
