# <a name="assessments-in-azure-migrateserver-assessment"></a>Оценки в Azure Migrate:Оценка сервера

В этой статье содержится обзор оценок в инструменте [оценки Azure Migrate:Server.](migrate-services-overview.md#azure-migrate-server-assessment-tool) Инструмент оценки серверов может оценивать на месте VMware VMs, Hyper-V VMs и физические серверы для миграции в Azure.

## <a name="whats-an-assessment"></a>Что такое оценка?

Оценка с помощью инструмента оценки серверов измеряет готовность и оценивает влияние переноса внутренних серверов в Azure.

> [!NOTE]
> В правительстве Azure просмотрите поддерживаемые места [целевой](migrate-support-matrix.md#supported-geographies-azure-government) оценки. Обратите внимание, что рекомендации по размеру VM в оценках будут использовать серию VM специально для регионов правительственного облака. [Узнайте больше](https://azure.microsoft.com/global-infrastructure/services/?regions=usgov-non-regional,us-dod-central,us-dod-east,usgov-arizona,usgov-iowa,usgov-texas,usgov-virginia&products=virtual-machines) о типах VM.

## <a name="types-of-assessments"></a>Виды оценок

Оценки, которые вы создаете с помощью оценки серверов, являются моментальным моментом данных. Оценка серверов предоставляет два типа оценок.

**Тип оценки** | **Сведения** | **Data**
--- | --- | ---
**На основе производительности** | Оценки, которые делают рекомендации на основе собранных данных о производительности | Рекомендация по размеру VM основана на данных по использованию процессора и памяти.<br/><br/> Рекомендация типа диска (стандартные HDD/SSD или диски премиум-класса) основана на IOPS и пропускной их пропускной.
**Как-это на-предпосылки** | Оценки, которые не используют данные о производительности для выполнения рекомендаций. | Рекомендация по размеру VM основана на<br/><br> Рекомендуемый тип диска основан на выбранном типе хранилища для оценки.

## <a name="how-do-i-run-an-assessment"></a>Как провести оценку?

Существует несколько способов проведения оценки:

- Оцените машины с помощью метаданных серверов, собранных с помощью легкого прибора Azure Migrate. Прибор обнаруживает на месте машины и отправляет метаданные/данные о производительности машины в Azure Migrate.
- Оцените машины с помощью метаданных серверов, импортируемых в формате, разделенном запятой (CSV).

## <a name="how-do-i-assess-with-the-appliance"></a>Как оценить прибор?

Если вы развертываете прибор Azure Migrate для обнаружения штатных серверов, вы делаете следующее:

1. Вы настраиваете Azure и свою среду для работы с Server Assessment.
2. Для первой оценки необходимо создать проект Azure и добавить в него инструмент оценки серверов.
3. Развертываете легкий прибор Azure Migrate. Прибор постоянно обнаруживает на месте машины и отправляет метаданные и данные о производительности машины в Azure Migrate. Прибор развертывается в виде VM или физической машины. Нет необходимости устанавливать что-либо на машины, которые вы хотите оценить.
4. После того, как прибор начнет обнаружение машины, вы можете собрать машины, которые вы хотите оценить в группу, и запустить оценку для группы.

Вы можете следить за нашими учебниками для [VMware](tutorial-prepare-vmware.md), [Hyper-V](tutorial-prepare-hyper-v.md), или [физические серверы,](tutorial-prepare-physical.md) чтобы опробовать эти шаги.

## <a name="how-do-i-assess-with-imported-data"></a>Как оценить с помощью импортируемых данных?

Если вы оцениваете серверы с помощью файла CSV, вам не нужен прибор. Вместо этого вы делаете следующее:

1. Вы создали Azure для работы с оценкой серверов.
2. Для первой оценки необходимо создать проект Azure и добавить в него инструмент оценки серверов.
3. Вы загружаете шаблон CSV и добавляете к нему данные сервера.
4. Вы импортируете шаблон в оценку сервера.
5. Вы обнаружите серверы, добавленные с импортом, соберите затем в группу и проводите оценку для группы.

## <a name="what-data-does-the-appliance-collect"></a>Какие данные собирает прибор?

Если вы используете прибор Azure Migrate для оценки, узнайте о метаданных и данных о производительности, собранных для [VMware](migrate-appliance.md#collected-data---vmware) и [Hyper-V.](migrate-appliance.md#collected-data---hyper-v)

## <a name="how-does-the-appliance-calculate-performance-data"></a>Как прибор вычисляет данные о производительности?

При использовании прибора для обнаружения данные о производительности для параметров вычислений собираются следующим образом:

1. Прибор собирает точку выборки в режиме реального времени:

    - **VMware VMs**: Прибор собирает точку выборки в режиме реального времени каждые 20-секундный интервал.
    - **Гипер-V V M**: Точка выборки в реальном времени собирается каждые 30-секундный интервал.
    - **Физические серверы**: Точка выборки в режиме реального времени собирается каждые пять минут. 
    
2. Прибор сворачивает точки выборки (20 секунд, 30 секунд, пять минут), чтобы создать единую точку данных каждые 10 минут. Чтобы создать единую точку, прибор выбирает пиковое значение из всех образцов, а затем отправляет его в Azure.
3. Server Assessment хранит все 10-минутные точки выборки за последний месяц.
4. При создании оценки Оценка сервера определяет подходящую точку данных для использования для правильного размера, основываясь на значениях процентиля для *истории производительности* и *использования процентиля.*

    - Например, если история производительности установлена на одну неделю, а использование процентиля является 95-м процентилом, Оценка сервера сортирует 10-минутные точки выборки за последнюю неделю в порядке возрастания и выбирает 95-е процентильное значение для правильного размера. 
    - Значение 95-го процентиля гарантирует, что вы игнорируете любые выбросы, которые могут быть включены, если вы выберете 99-й процентиль.
    - Если вы хотите выбрать пиковое использование за период и не хотите пропустить выбросы, вы должны выбрать 99-й процентиль для использования процентиля.

5. Это значение умножается на коэффициент комфорта, чтобы получить эффективные данные об использовании производительности для каждой метрики (использование процессора, использование памяти, диск IOPS (читать и писать), пропускную способность диска (читать и писать) и пропускную способность сети (вход и выход), который собирает прибор.



## <a name="how-are-assessments-calculated"></a>Как рассчитываются оценки? 

Оценки в оценке серверов рассчитываются с использованием метаданных/данных о производительности для внутренних машин. Если развернуть прибор Azure Migrate, то оценивайте с помощью данных, собранных прибором. Если вы запустите оценку для машин, импортируемых с помощью . ФАЙЛ CSV предоставляет метаданные для расчета. Расчеты происходят в три этапа:

1. **Рассчитайте готовность Azure**: Оцените, подходят ли машины для миграции в Azure.
2. **Рассчитать рекомендации по размеру:** Оцените вычисления, хранение и размер сети. 
2. **Рассчитать ежемесячные расходы:** Рассчитать предполагаемые ежемесячные затраты на вычисления и хранение для работы машин в Azure после миграции.

Расчеты в порядке, и сервер машины перемещается на более поздний этап только в том случае, если он проходит предыдущий. Например, если сервер не выполняет готовность Azure, он помечен как неподходящий для Azure, а размериси и затраты не выполняются для этого сервера.

## <a name="calculate-readiness"></a>Рассчитать готовность

Не все машины подходят для запуска в Azure. Оценка сервера оценивает каждую местную машину и присваивает ей категорию готовности. 
- **Готова к Azure**: Машина может быть перенесена как в Azure без каких-либо изменений. Он начнется в Azure с полной поддержкой Azure.
- **Условно готова к Azure:** машина может начаться в Azure, но может не иметь полной поддержки Azure. Например, машина, работающего с старой версией Windows Server, не поддерживается в Azure. Вы должны быть осторожны, прежде чем перенести эти машины в Azure. Следуйте рекомендациям по восстановлению, предложенным в оценке для устранения проблем готовности.
- **Не готов к Azure**: Машина не будет запускаться в Azure. Например, если в помещении диск машины больше 64-TBs, он не может быть размещен в Azure. Следуйте инструкциям по исправлению ситуации, чтобы устранить проблему перед миграцией. 
- **Готовность неизвестна**: Azure Migrate не может определить готовность машины из-за недостаточного количества метаданных.

Для расчета готовности Server Assessment проверяет свойства машины и настройки операционной системы, суммированные в следующих таблицах. 

### <a name="machine-properties"></a>Свойства виртуальной машины

Оценка сервера анализирует следующие свойства внутреннего VM, чтобы определить, может ли он работать на Azure.

**Свойство** | **Сведения** | **Состояние готовности к работе в Azure**
--- | --- | ---
**Тип загрузки** | Azure поддерживает VMs с типом загрузки BIOS, а не UEFI. | Условно готов, если тип загрузки UEFI.
**Ядра** | Количество ядер в машинах должно быть равным или меньше, чем максимальное количество ядер (128), поддерживаемых для Azure VM.<br/><br/> Если доступен журнал производительности, служба "Миграция Azure" рассматривает используемые ядра для сравнения. Если в параметрах оценки указан фактор комфорта, то количество используемых ядер умножается на него.<br/><br/> Если нет истории производительности, Azure Migrate использует выделенные ядра без применения фактора комфорта. | Если меньше заданного ограничения или равно ему, виртуальная машина готова.
**Память** | Размер памяти машины должен быть равен или меньше, чем максимальная память (3892&nbsp;гигабайта (3892 гигабайта »GB» на серии Azure M Standard_M128m<sup>2)</sup>допускается для Azure VM. [Подробнее](https://docs.microsoft.com/azure/virtual-machines/windows/sizes).<br/><br/> Если доступен журнал производительности, служба "Миграция Azure" рассматривает используемую память для сравнения. Если задан фактор комфорта, объем используемой памяти умножается на него.<br/><br/> Если нет истории, выделенная память используется без применения фактора комфорта.<br/><br/> | Если не превышает ограничение, виртуальная машина готова.
**Диск хранилища** | Выделенный размер диска должен быть 32 ТБ или меньше. Хотя Azure поддерживает 64-ТБ-диски с ультра SSD-дисками, Azure Migrate: Server Assessment в настоящее время проверяет 32 ТБ, поскольку размер диска пока не поддерживает Ultra SSD. <br/><br/> Количество дисков, прикрепленных к машине, должно быть 65 или меньше, включая диск ОС. | Если не превышает ограничение, виртуальная машина готова.
**Сеть** | Машина должна иметь 32 или меньше сетевых интерфейсов (NICs) прилагается к нему. | Если не превышает ограничение, виртуальная машина готова.

### <a name="guest-operating-system"></a>Операционная система на виртуальной машине
Наряду с свойствами VM, Server Assessment рассматривает гостевую операционную систему машин, чтобы определить, может ли она работать на Azure.

> [!NOTE]
> Для VMware VMs, Оценка сервера использует операционную систему, указанную для VM в vCenter Server для обработки анализа гостевой ОС. Для Linux VMs, работающих на VMware, он в настоящее время не определяет точную версию ядра гостевой ОС.

Следующая логика используется Server Assessment для определения готовности Azure на основе операционной системы.

**Операционная система** | **Сведения** | **Состояние готовности к работе в Azure**
--- | --- | ---
Windows Server 2016 и все пакеты обновления | Azure обеспечивает полную поддержку. | Готово к работе в Azure
Windows Server 2012 R2 и все пакеты обновления | Azure обеспечивает полную поддержку. | Готово к работе в Azure
Windows Server 2012 и все пакеты обновления | Azure обеспечивает полную поддержку. | Готово к работе в Azure
Windows Server 2008 R2 со всеми пакетами обновления | Azure обеспечивает полную поддержку.| Готово к работе в Azure
Windows Server 2008 (32-разрядная и 64-разрядная) | Azure обеспечивает полную поддержку. | Готово к работе в Azure
Windows Server 2003, 2003 R2 | Эти операционные системы прошли дату окончания поддержки и нуждаются в [Соглашении о таможенной поддержке (CSA)](https://aka.ms/WSosstatement) для поддержки в Azure. | Условно готов к Azure. Перед переходом в Azure можно обновить ОС.

## <a name="calculating-sizing"></a>Расчет размеров


После того, как машина будет помечена как готовая к Azure, Server Assessment дает рекомендации по определению VM и диска SKU Azure. Расчеты размеров зависят от того, используете ли вы как-это на месте размера, или на основе производительности размеров.

### <a name="calculate-sizing-as-is-on-premises"></a>Рассчитать размер (как-это на-предпосылки)

 Если вы используете как-это на-предпосылки размеров, Оценка сервера не учитывает историю производительности VMs и дисков.

- **Вычислить размер**: Он выделяет Azure VM SKU на основе размера, выделенного на месте.
- **Хранение/дисковое размер:** Оценка сервера рассматривает тип хранилища, указанный в свойствах оценки (стандартный HDD/SSD/premium), и рекомендует тип диска соответственно. Тип хранилища по умолчанию — это диски премиум-класса.
- **Размер сети**: Оценка сервера учитывает сетевой адаптер на предприимчезной машине.


### <a name="calculate-sizing-performance-based"></a>Рассчитать размер (на основе производительности)

Если вы используете размеры, основанное на производительности, Оценка сервера дает рекомендации по размеру следующим образом:

- Оценка сервера учитывает историю производительности машины для определения размера VM и типа диска в Azure.
- Если серверы были импортированы с помощью файла CSV, используются указанные значения. Этот метод особенно полезен, если вы перераспределили локальная машина, использование невелико, и для экономии затрат необходимо использовать VM в Azure. 
- Если вы не хотите использовать данные о производительности, сбросить критерии размеров в качестве-это на месте, как описано в предыдущем разделе.

#### <a name="calculate-storage-sizing"></a>Рассчитать размер хранилища

Для размера хранилища Azure Migrate пытается сопоставить каждый диск, подключенный к машине, на диске в Azure и работает следующим образом:

1. Оценка сервера добавляет чтение и запись IOPS диска, чтобы получить общий требуемый IOPS. Аналогичным образом, он добавляет значения пропускной записи чтения и записи, чтобы получить общую пропускную стоимость каждого диска.
2. Если вы указали тип хранилища как автоматический, основываясь на эффективных значениях IOPS и пропускной стоимости, Оценка сервера определяет, следует ли отображивать диск на стандартный HDD, стандартный SSD или премиум-диск в Azure. Если тип хранилища установлен на стандартный HDD/SSD/Premium, Server Assessment пытается найти диск SKU в выбранном типе хранилища (Standard HDD/SSD/Premium).
3. Диски выбираются следующим образом:
    - Если Server Assessment не может найти диск с требуемым IOPS и пропускной памятью, он помечает машину как неподходящую для Azure.
    - Если Server Assessment находит набор подходящих дисков, он выбирает диски, поддерживающие месторасположение, указанное в настройках оценки.
    - Если существует несколько подходящих дисков, Server Assessment выбирает диск с наименьшими затратами.
    - Если данные о производительности для любого диска недоступны, данные конфигурации диска (размер диска) используются для поиска стандартного SSD-диска в Azure.

#### <a name="calculate-network-sizing"></a>Рассчитать размер сети

Server Assessment пытается найти VM Azure, который может поддерживать количество сетевых адаптеров, прикрепленных к предприимчивой машине, и производительность, требуемую этими сетевыми адаптерами.
- Чтобы получить эффективную сетевую производительность локального VM, Server Assessment агрегирует данные, передаваемые в секунду (MBps) из машины (сети из), во всех сетевых адаптеров, и применяет коэффициент комфорта. Он использует этот номер, чтобы найти VM Azure, который может поддерживать требуемую производительность сети.
- Наряду с производительностью сети, Server Assessment также рассматривает вопрос о том, может ли Azure VM поддерживать необходимое количество сетевых адаптеров.
- Если данные о производительности сети отсутствуют, Server Assessment учитывает только количество сетевого адаптера для размеров VM.


#### <a name="calculate-compute-sizing"></a>Рассчитать размеры вычислений

После расчета требований к хранению и сети Server Assessment учитывает требования к процессору и памяти для поиска подходящего размера VM в Azure.
- Azure Migrate рассматривает эффективные используемые ядра и память, чтобы найти подходящий размер VM в Azure.
- Если найти подходящий размер не удалось, компьютер помечается как неподходящий для Azure.
- Если найден подходящий размер, служба "Миграция Azure" применяет результаты вычисления параметров хранилища и сети. Затем он применяет настройки уровня расположения и ценообразования для окончательной рекомендации по размеру VM.
- Если доступно несколько соответствующих размеров виртуальных машин, служба выбирает рекомендуемый, который обеспечивает наименьшие затраты.


## <a name="confidence-ratings-performance-based"></a>Рейтинги доверия (на основе производительности)

Каждая оценка на основе производительности в Azure Migrate связана с рейтингом доверия, который колеблется от одной (самой низкой) до пяти звезд (самая высокая). Рейтинг доверия помогает оценить надежность рекомендаций по размеру, предоставляемых Azure Migrate.

- Оценка достоверности назначается на основе доступности точек данных, необходимых для вычисления оценки.
- Для размеров на основе производительности оценка сервера нуждается в:
    - Данные об использовании процессора и памяти VM.
    - Диск IOPS и данные пропускной их прокладки для каждого диска, прикрепленного к VM.
    - Сеть вв:2 для обработки размеров на основе производительности для каждого сетевого адаптера, прикрепленного к VM.
    - Если какой-либо из этих номеров использования недоступен, рекомендации по размеру могут быть ненадежными.

> [!NOTE]
> Рейтинги доверия не присваиваются серверам, оцениваемым с помощью импортируемого. Файл CSV. Рейтинги также не применимы для оценки как-есть на-предварительной.
   
### <a name="ratings"></a>Рейтинги

В зависимости от процентной доли имеющихся точек данных рейтинг доверия к оценке определяется следующим образом.

   **Уровень доступности точек данных** | **Оценка достоверности**
   --- | ---
   0-20% | 1 звезда
   21-40% | 2 звезды
   41-60% | 3 звезды
   61-80% | 4 звезды
   81-100% | 5 звезд

### <a name="low-confidence-ratings"></a>Низкие рейтинги доверия

Вот несколько причин, почему оценка может получить низкий рейтинг доверия:

- Вы не профилируют свою среду в течение всего срока, в течение которого вы создаете оценку. Например, если вы создаете оценку с продолжительностью производительности, установленной на один день, необходимо подождать по крайней мере день после начала обнаружения всех точек данных для сбора.
- В течение периода, для которого рассчитывалась оценка, некоторые виртуальные машины были отключены. Если какие-либо VM отключены на некоторое время, Оценка сервера не может собирать данные о производительности за этот период.
- Некоторые ВМ были созданы в течение периода, за который была рассчитана оценка. Например, если вы создали оценку для истории производительности за последний месяц, но некоторые ВМ были созданы в среде всего неделю назад, история производительности новых ВМ не будет существовать в течение всего срока действия.

> [!NOTE]
> Если рейтинг доверия любой оценки меньше пяти звезд, мы рекомендуем вам подождать по крайней мере день, пока прибор профилирует окружающую среду, а затем пересчитывает оценку. Если вы этого не сможете, размер на основе производительности может быть ненадежным. В этом случае мы рекомендуем переключить оценку на размер ы на местах.

## <a name="calculate-monthly-costs"></a>Рассчитать ежемесячные расходы

После завершения рекомендаций по размеру Azure Migrate вычисляет затраты на вычисления и хранение после миграции.

- **Стоимость вычислений**. Служба "Миграция Azure" использует API выставления счетов, чтобы на основании рекомендуемого размера виртуальной машины Azure вычислить ежемесячные затраты на виртуальную машину.
    - При этом учитывается операционная система, программа Software Assurance, зарезервированные экземпляры, время доступности виртуальных машин, расположение и параметры валюты.
    - Он агрегирует стоимость всех машин для расчета общей ежемесячной вычислительной стоимости.
- **Стоимость хранения**: Ежемесячная стоимость хранения для машины рассчитывается путем агрегирования ежемесячной стоимости всех дисков, прикрепленных к машине, следующим образом:
    - Оценка сервера вычисляет общие ежемесячные затраты на хранение, агрегируя затраты на хранение всех машин.
    - В настоящее время при расчете не учитываются предложения, указанные в настройках оценки.

Цены отображаются в валюте, заданной в настройках оценки.
