# <a name="about-assessments-in-azure-migrate"></a>Об оценках в службе "миграция Azure"

В этой статье описывается вычисление оценок в [службе "миграция Azure": Оценка серверов](migrate-services-overview.md#azure-migrate-server-assessment-tool). Вы можете выполнять оценку для групп локальных компьютеров, чтобы выяснить, готовы ли они к миграции в Azure Migration.

## <a name="how-do-i-run-an-assessment"></a>Разделы справки выполнить оценку?
Оценку можно выполнить с помощью службы "миграция Azure": "Оценка сервера" или другого средства Azure или стороннего производителя. После создания проекта службы "миграция Azure" необходимо добавить необходимое средство. [Дополнительные сведения](how-to-add-tool-first-time.md)

### <a name="collect-compute-data"></a>Получение данных вычислений

Данные производительности для параметров вычислений собираются следующим образом:

1. Устройство, выполняющее [миграцию Azure](migrate-appliance.md) , собирает образец точки в режиме реального времени:

    - **Виртуальные машины VMware**. для виртуальных машин VMware устройство "миграция Azure" собирает образец точки в реальном времени каждые 20 секунд.
    - **Виртуальные машины Hyper-v**. для виртуальных машин Hyper-v образец точки в реальном времени собирается каждые 30 секунд.
    - **Физические серверы**. для физических серверов образец точки в реальном времени собирается каждые пять минут. 
    
2. Устройство выполняет сведение образцов точек (20 секунд, 30 секунд, пять минут) для создания одной точки данных каждые 10 минут. Чтобы создать одну точку данных, устройство выбирает пиковое значение во всех примерах, а затем отправляет его в Azure.
3. В ходе оценки сервера хранятся все 10-минутные точки выборки за прошлый месяц.
4. При создании оценки Серверная Оценка определяет соответствующую точку данных, которая будет использоваться для правильного изменения размера, исходя из значений процентиля для *журнала производительности* и *использования процентилей*.

    - Например, если для журнала производительности задана одна неделя, а в качестве показателя использования процентиля является 95 процентиль процентиль, то оценка сервера сортирует 10-минутные точки выборки за последнюю неделю в возрастающем порядке и выбирает значение 95 процентиль процентиля для правильного изменения размера. 
    - Значение 95 процентиль процентиль гарантирует игнорирование любых выбросов, которые могут быть добавлены при выборе 99-м процентиль.
    - Если вы хотите выбрать пиковое использование для периода и не хотите пропускать выбросы, следует выбрать 99-м процентиль для использования процентиля.

5. Это значение умножается на эффективный фактор, чтобы получить эффективные данные об использовании производительности для каждой метрики (загрузка ЦП, использование памяти, операции ввода-вывода на диск (чтение и запись), пропускная способность диска (чтение и запись) и пропускная способность сети (в и в), устройство собирает.

Для выполнения оценки в оценке сервера вы готовитесь к оценке в локальной среде и в Azure, а также настроите устройство миграции Azure для непрерывного обнаружения локальных компьютеров. После обнаружения компьютеров их необходимо собрать в группы для их оценки. Для получения более подробных и высокодостоверных оценок можно визуализировать и сопоставлять зависимости между компьютерами, чтобы понять, как их перенести.

- Узнайте о выполнении оценок для [виртуальных машин VMware](tutorial-prepare-vmware.md), [виртуальных машин Hyper-V](tutorial-prepare-hyper-v.md)и [физических серверов](tutorial-prepare-physical.md).
- Узнайте, как оценивать серверы, [импортированные с помощью CSV-файла](tutorial-assess-import.md).
- Дополнительные сведения о настройке [визуализации зависимостей](concepts-dependency-visualization.md).

## <a name="how-are-assessments-calculated"></a>Как вычисляется Оценка? 

Оценки в службе "миграция Azure". Оценка сервера вычисляется с помощью метаданных, собранных о локальных компьютерах. При выполнении оценки на компьютерах, импортированных с помощью. CSV-файл, вы предоставляете метаданные для вычисления. Вычисления выполняются в три этапа:

1. **Рассчитайте готовность Azure**: оцените, подходят ли компьютеры для миграции в Azure.
2. **Рассчитайте рекомендации по выбору размера**: Оценка вычислений, хранилища и размера сети. 
2. **Вычисление месячных затрат**. Вычислите расчетные ежемесячные затраты на вычисления и хранение для запуска компьютеров в Azure после миграции.

Вычисления выполняются по порядку, и сервер машины перемещается на более позднюю стадию, только если он проходит предыдущее. Например, если сервер отказывается от готовности к работе с Azure, он помечается как неподходящий для Azure, а изменение размера и стоимости для этого сервера не выполняется.



## <a name="calculate-readiness"></a>Вычислить готовность

Не все компьютеры подходят для работы в Azure. Оценка сервера оценивает каждую локальную машину и назначает ей категорию готовности. 
- **Готово к работе с Azure**. Виртуальная машина может быть перенесена как есть в Azure без каких-либо изменений. Он будет запущен в Azure с полной поддержкой Azure.
- **Условно готово к работе**в Azure. компьютер может начаться в Azure, но может не поддерживать полную поддержку Azure. Например, компьютер, работающий под управлением более старой версии Windows Server, не поддерживается в Azure. Перед миграцией этих компьютеров в Azure необходимо соблюдать осторожность. Следуйте рекомендациям по исправлению, предлагаемым в ходе оценки, чтобы устранить проблемы с готовностью.
- **Не готово к работе в Azure**. компьютер не будет запущен в Azure. Например, если локальный диск компьютера больше 64-TBs, он не может быть размещен в Azure. Следуйте указаниям по исправлению, чтобы устранить проблему перед миграцией. 
- **Неизвестная готовность**: службе "миграция Azure" не удалось определить готовность компьютера из-за недостаточных метаданных.

Чтобы вычислить готовность, Оценка сервера проверяет свойства компьютера и параметры операционной системы, приведенные в следующих таблицах. 

### <a name="machine-properties"></a>Свойства виртуальной машины

Оценка сервера проверяет следующие свойства локальной виртуальной машины, чтобы определить, можно ли ее запустить в Azure.

**Свойство** | **Сведения** | **Состояние готовности к работе в Azure**
--- | --- | ---
**Тип загрузки** | Azure поддерживает виртуальные машины с типом загрузки BIOS, а не UEFI. | Условно готово, если тип загрузки — UEFI.
**Ядра** | Число ядер на компьютерах должно быть больше или равно максимальному количеству ядер (128), поддерживаемое для виртуальной машины Azure.<br/><br/> Если доступен журнал производительности, служба "Миграция Azure" рассматривает используемые ядра для сравнения. Если в параметрах оценки указан фактор комфорта, то количество используемых ядер умножается на него.<br/><br/> Если журнал производительности отсутствует, служба "миграция Azure" использует выделенные ядра без применения недостаточного коэффициента. | Если меньше заданного ограничения или равно ему, виртуальная машина готова.
**Память** | Объем памяти компьютера должен быть больше или равен максимальному объему памяти (3892 гигабайт [ГБ] в серии Azure M Standard_M128m&nbsp;<sup>2</sup>), разрешенном для виртуальной машины Azure. [Дополнительные сведения](https://docs.microsoft.com/azure/virtual-machines/windows/sizes)<br/><br/> Если доступен журнал производительности, служба "Миграция Azure" рассматривает используемую память для сравнения. Если задан фактор комфорта, объем используемой памяти умножается на него.<br/><br/> Если журнал отсутствует, выделенная память используется без применения коэффициента отдыха.<br/><br/> | Если не превышает ограничение, виртуальная машина готова.
**Диск хранилища** | Выделенный размер диска должен быть не менее 32 ТБ. Хотя Azure поддерживает диски 64-ТБ с SSD (цен. категория "Ультра") дисков, служба "миграция Azure" в настоящее время проверяет наличие на диске 32 ТБ в качестве ограничений размера диска, так как он еще не поддерживает SSD (цен. категория "Ультра"). <br/><br/> Число дисков, подключенных к компьютеру, должно быть не менее 65, включая диск операционной системы. | Если не превышает ограничение, виртуальная машина готова.
**Сеть** | К компьютеру должно быть подключено не менее 32 сетевых интерфейсов (NIC). | Если не превышает ограничение, виртуальная машина готова.

### <a name="guest-operating-system"></a>Операционная система на виртуальной машине

## <a name="calculate-sizing-as-is-on-premises"></a>Вычислить размер (как в локальной системе)

После того как компьютер помечается как готовый для Azure, средство оценки сервера вносит рекомендации по выбору размера для определения номера SKU виртуальной машины и диска Azure. Если вы используете "как есть" локальное изменение размера, Оценка сервера не учитывает журнал производительности виртуальных машин и дисков.

**Вычисление размера**. он выделяет номер SKU виртуальной машины Azure на основе размера, выделенного в локальной среде.
**Размер хранилища или диска**. Оценка сервера рассматривает тип хранилища, указанный в свойствах оценки ("Стандартный HDD" или "SSD/Премиум"), и соответствующим образом рекомендует тип диска. Типом хранилища по умолчанию являются диски уровня "Премиум".
**Размер сети**. при оценке сервера сетевой адаптер рассматривается на локальном компьютере.


## <a name="calculate-sizing-performance-based"></a>Вычисление размера (на основе производительности)

После того как компьютер помечается как готовый для Azure, при использовании размера, основанного на производительности, Оценка сервера делает следующее:

- При оценке сервера журнал производительности компьютера определяет размер виртуальной машины и тип диска в Azure.
- Если серверы были импортированы с помощью CSV-файла, используются указанные значения. Этот метод особенно полезен при чрезмерном выделении локального компьютера, на самом деле низкий уровень использования, и вам нужно щелкнуть правой кнопкой мыши размер виртуальной машины в Azure, чтобы сэкономить затраты. 
- Если вы не хотите использовать данные о производительности, сбросьте критерии изменения размера в "как есть локально", как описано в предыдущем разделе.

### <a name="calculate-storage-sizing"></a>Вычисление размера хранилища

Для размера хранилища служба "миграция Azure" пытается сопоставлять каждый диск, подключенный к компьютеру, к диску в Azure, и работает следующим образом:

1. При оценке сервера добавляются операции чтения и записи с диска, чтобы получить общее число операций ввода-вывода в секунду. Аналогичным образом для получения общей пропускной способности каждого диска добавляются значения пропускной способности чтения и записи.
2. Если вы указали тип хранения автоматически на основе действующих значений операций ввода-вывода и пропускной способности, то оценка сервера определяет, следует ли сопоставить диск со стандартным HDD, стандартным SSD или диском уровня "Премиум" в Azure. Если для типа хранилища задано значение HDD (цен. категория "Стандартный")/SSD/Premium, то средство оценки серверов пытается найти номер SKU диска в выбранном типе хранилища (HDD (цен. категория "Стандартный")/SSD/Premium).
3. Диски выбираются следующим образом:
    - Если серверной оценке не удается найти диск с необходимыми операциями ввода-вывода и пропускной способностью, он помечает компьютер как неподходящий для Azure.
    - Если серверная Оценка находит набор подходящих дисков, он выбирает диски, которые поддерживают расположение, указанное в параметрах оценки.
    - Если имеется несколько подходящих дисков, то оценка сервера выбирает диск с наименьшей стоимостью.
    - Если данные о производительности для любого диска недоступны, данные конфигурации диска (размер диска) используются для поиска в Azure стандартного SSD-диска.

### <a name="calculate-network-sizing"></a>Вычисление размера сети

Средство оценки серверов пытается найти виртуальную машину Azure, которая может поддерживать число сетевых адаптеров, подключенных к локальному компьютеру, и производительность, необходимую для этих сетевых адаптеров.
- Чтобы получить эффективную производительность сети для локальной виртуальной машины, в ходе оценки сервера выполняется статистическая обработка данных, передаваемых в секунду (вне сети), между всеми сетевыми адаптерами и применением эффективного фактора. Он использует это число для поиска виртуальной машины Azure, которая может поддерживать необходимую производительность сети.
- Помимо производительности сети, при оценке сервера также учитывается, может ли виртуальная машина Azure поддерживать требуемое количество сетевых адаптеров.
- Если данные о производительности сети недоступны, при оценке серверов в качестве размера виртуальной машины учитывается только число сетевых адаптеров.


### <a name="calculate-compute-sizing"></a>Вычисление размера вычислений

После вычисления требований к хранению и сети в ходе оценки сервера учитываются требования к ЦП и памяти, чтобы найти подходящий размер виртуальной машины в Azure.
- Служба "миграция Azure" рассматривает эффективные используемые ядра и память, чтобы найти подходящий размер виртуальной машины в Azure.
- Если найти подходящий размер не удалось, компьютер помечается как неподходящий для Azure.
- Если найден подходящий размер, служба "Миграция Azure" применяет результаты вычисления параметров хранилища и сети. Затем он применяет параметры расположение и ценовую категорию для окончательной рекомендации по размеру виртуальной машины.
- Если доступно несколько соответствующих размеров виртуальных машин, служба выбирает рекомендуемый, который обеспечивает наименьшие затраты.


### <a name="calculate-confidence-ratings"></a>Вычисление оценок достоверности

Каждая оценка на основе производительности в службе "миграция Azure" связана с рейтингом достоверности, который находится в диапазоне от одного (низший) до пяти звезд (самый высокий).
- Оценка достоверности назначается на основе доступности точек данных, необходимых для вычисления оценки.
- Оценка достоверности помогает определить надежность рекомендаций по выбору размера, предоставленных службой "Миграция Azure".
- Оценки достоверности не применяются *в качестве локальных* оценок.
- Для определения размера на основе производительности требуется оценка сервера:
    - Данные об использовании ЦП и памяти виртуальной машины.
    - Данные дискового ввода-вывода и пропускной способности для каждого диска, подключенного к виртуальной машине.
    - Сетевой ввод-вывод для управления размером на основе производительности для каждого сетевого адаптера, подключенного к виртуальной машине.

   Если любое из этих номеров использования недоступно в vCenter Server, рекомендация по размеру может быть ненадежной.

В зависимости от количества доступных точек данных Оценка достоверности для оценки выглядит следующим образом.

   **Уровень доступности точек данных** | **Оценка достоверности**
   --- | ---
   0-20% | 1 звезда
   21-40% | 2 звезды
   41-60% | 3 звезды
   61-80% | 4 звезды
   81-100% | 5 звезд

> [!NOTE]
> Оценки достоверности не назначаются оценкам серверов, импортированных с помощью. CSV-файл в службу "миграция Azure". 

#### <a name="low-confidence-ratings"></a>Низкая оценка достоверности

Ниже приведены несколько причин, по которым оценка может иметь низкую степень достоверности.

- Вы не проделали Профилирование среды в течение времени, в течение которого вы создаете оценку. Например, если вы создаете оценку с длительностью производительности в один день, необходимо подождать не менее одного дня после начала обнаружения всех точек данных для сбора.
- В течение периода, для которого рассчитывалась оценка, некоторые виртуальные машины были отключены. Если какие бы то ни было виртуальные машины отключаются в течение некоторого времени, при оценке сервера не удается получить данные о производительности за этот период.
- Некоторые виртуальные машины были созданы в течение периода, для которого была рассчитана оценка. Например, если вы создали оценку для журнала производительности за прошлый месяц, но некоторые виртуальные машины были созданы в среде только неделю назад, журнал производительности новых виртуальных машин не будет существовать в течение всего времени.

> [!NOTE]
> Если оценка достоверности для какой-либо оценки меньше пяти звезд, рекомендуется подождать по крайней мере один день, чтобы устройство пропрофилировать среду, а затем повторно вычислить оценку. В противном случае размер на основе производительности может быть ненадежным. В этом случае рекомендуется переключить оценку на локальное изменение размера.

## <a name="calculate-monthly-costs"></a>Вычисление месячных затрат

После выполнения рекомендаций по изменению размера служба "миграция Azure" вычисляет затраты на вычисления и хранение после миграции.

- **Стоимость вычислений**. Служба "Миграция Azure" использует API выставления счетов, чтобы на основании рекомендуемого размера виртуальной машины Azure вычислить ежемесячные затраты на виртуальную машину.
    - При этом учитывается операционная система, программа Software Assurance, зарезервированные экземпляры, время доступности виртуальных машин, расположение и параметры валюты.
    - Он объединяет затраты на все компьютеры для вычисления общей месячной стоимости вычислений.
- **Стоимость хранилища**. ежемесячные затраты на хранение на компьютере рассчитываются путем статистической обработки ежемесячных затрат на все диски, подключенные к компьютеру, следующим образом:
    - Оценка сервера вычисляет общую ежемесячную стоимость хранения за счет статистической обработки затрат на хранение всех компьютеров.
    - В настоящее время вычисления не учитывают предложения, указанные в параметрах оценки.

Цены отображаются в валюте, заданной в настройках оценки.
