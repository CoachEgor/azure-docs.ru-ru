# <a name="assessments-in-azure-migrate-server-assessment"></a>Оценки в Azure Migrate: Оценка серверов

В этой статье содержится обзор оценок в инструменте [Azure Migrate: Server Assessment.](migrate-services-overview.md#azure-migrate-server-assessment-tool) Инструмент может оценивать на месте виртуальные машины VMware, сверхv V M и физические серверы для миграции в Azure.

## <a name="whats-an-assessment"></a>Что такое оценка?

Оценка с помощью инструмента оценки серверов измеряет готовность и оценивает влияние переноса на лазурные серверы.

> [!NOTE]
> В правительстве Azure просмотрите поддерживаемые места [целевой](migrate-support-matrix.md#supported-geographies-azure-government) оценки. Обратите внимание, что рекомендации по размеру VM в оценках будут использовать серию VM специально для регионов правительственного облака. [Узнайте больше](https://azure.microsoft.com/global-infrastructure/services/?regions=usgov-non-regional,us-dod-central,us-dod-east,usgov-arizona,usgov-iowa,usgov-texas,usgov-virginia&products=virtual-machines) о типах VM.

## <a name="types-of-assessments"></a>Виды оценок

Оценки, которые вы создаете с помощью оценки серверов, являются моментальным моментом данных. Оценка серверов предоставляет два типа оценок.

**Тип оценки** | **Сведения** | **Data**
--- | --- | ---
**На основе производительности** | Оценки, которые делают рекомендации на основе собранных данных о производительности | Рекомендация по размеру VM основана на данных по использованию процессора и оперативной памяти.<br/><br/> Рекомендация типа диска основана на входных/выходных операциях в секунду (IOPS) и пропускной их пропускной. Типы дисков: Azure Standard HDD, Azure Standard SSD и Azure Premium.
**Как-это на-предпосылки** | Оценки, в отношении неиспользуемых данных о производительности для выносить рекомендации | Рекомендация по размеру VM основана на емкостном размере VM.<br/><br> Рекомендуемый тип диска основан на выбранном типе хранилища для оценки.

## <a name="how-do-i-run-an-assessment"></a>Как провести оценку?

Есть несколько способов провести оценку.

- Оцените машины с помощью метаданных серверов, собранных с помощью легкого прибора Azure Migrate. Устройство обнаруживает локальные компьютеры. Затем он отправляет метаданные машины и данные о производительности в Azure Migrate.
- Оцените машины с помощью метаданных серверов, импортируемых в формате, разделенном запятой (CSV).

## <a name="how-do-i-assess-with-the-appliance"></a>Как оценить прибор?

Если вы развертываете прибор Azure Migrate для обнаружения штатных серверов, сделайте следующие действия:

1. Навеяйте Azure и среду для работы с Server Assessment.
1. Для первой оценки создайте проект Azure и добавьте к нему инструмент оценки серверов.
1. Развертывание легкого прибора Azure Migrate. Прибор постоянно обнаруживает на месте машины и отправляет метаданные и данные о производительности машины в Azure Migrate. Развертывание прибора в виде VM или физической машины. Вам не нужно устанавливать что-либо на машины, которые вы хотите оценить.

После того, как прибор начнет обнаружение машины, вы можете собрать машины, которые вы хотите оценить, в группу и провести оценку для группы.

Следуйте нашим учебникам для [VMware](tutorial-prepare-vmware.md), [Hyper-V](tutorial-prepare-hyper-v.md), или [физические серверы,](tutorial-prepare-physical.md) чтобы опробовать эти шаги.

## <a name="how-do-i-assess-with-imported-data"></a>Как оценить с помощью импортируемых данных?

Если вы оцениваете серверы с помощью файла CSV, вам не нужен прибор. Вместо этого сделайте следующие шаги:

1. Навлаживайте Azure для работы с оценкой серверов.
1. Для первой оценки создайте проект Azure и добавьте к нему инструмент оценки серверов.
1. Загрузите шаблон CSV и добавьте к нему данные сервера.
1. Импортшаблона в оценку серверов.
1. Откройте для себя серверы, добавленные с импортом, соберите их в группу и запустите оценку для группы.

## <a name="what-data-does-the-appliance-collect"></a>Какие данные собирает прибор?

Если вы используете прибор Azure Migrate для оценки, узнайте о метаданных и данных о производительности, собранных для [VMware](migrate-appliance.md#collected-data---vmware) и [Hyper-V.](migrate-appliance.md#collected-data---hyper-v)

## <a name="how-does-the-appliance-calculate-performance-data"></a>Как прибор вычисляет данные о производительности?

Если вы используете прибор для обнаружения, он собирает данные о производительности для параметров вычислений с помощью следующих шагов:

1. Прибор собирает точку выборки в режиме реального времени.

    - **VMware VMs**: Образец точки собирается каждые 20 секунд.
    - **Hyper-V VMs**: Точка выборки собирается каждые 30 секунд.
    - **Физические серверы**: Образец точки собирается каждые пять минут.

1. Прибор объединяет точки выборки для создания одной точки данных каждые 10 минут. Для создания точки данных прибор выбирает пиковые значения из всех образцов. Затем он отправляет точку данных в Azure.
1. Server Assessment хранит все 10-минутные точки данных за последний месяц.
1. При создании оценки Оценка сервера определяет подходящую точку данных для использования для изация прав. Идентификация основана на значениях процентиля для *истории производительности* и *использования процентиля.*

    - Например, если история производительности составляет одну неделю, а использование процентиля — 95-й процентиль, оценка сервера сортирует 10-минутные баллы выборки за последнюю неделю. Он сортирует их в порядке возрастания и выбирает 95-й процентильной стоимости для прав.
    - Значение 95-го процентиля гарантирует, что вы игнорируете любые выбросы, которые могут быть включены, если вы выбрали 99-й процентиль.
    - Если вы хотите выбрать пиковое использование за период и не хотите пропустить выбросы, выберите 99-й процентиль для использования процентиля.

1. Это значение умножается на коэффициент комфорта, чтобы получить эффективные данные об использовании производительности для этих метрик, которые собирает прибор:

    - загрузка ЦП;
    - Использование оперативной памяти
    - Диск IOPS (читай и пиши)
    - Пропускная часть диска (читай и пиши)
    - Пропускная часть сети (вход и выход)

## <a name="how-are-assessments-calculated"></a>Как рассчитываются оценки?

Оценка сервера использует метаданные и данные о производительности на месте для расчета оценок. При развертывании прибора Azure Migrate оценка использует данные, собранные прибором. Но если вы выполняете оценку, импортируемую с помощью файла CSV, вы предоставляете метаданные для расчета.

Расчеты происходят в этих трех стадиях:

1. **Рассчитайте готовность Azure**: Оцените, подходят ли машины для миграции в Azure.
1. **Рассчитать рекомендации по размеру:** Оцените вычисления, хранение и размер сети.
1. **Рассчитать ежемесячные расходы:** Рассчитать предполагаемые ежемесячные затраты на вычисления и хранение для работы машин в Azure после миграции.

Расчеты находятся в предыдущем порядке. Сервер машины перемещается на более поздний этап только в том случае, если он проходит предыдущий. Например, если сервер не выполняет этап готовности Azure, он помечен как неподходящий для Azure. Для этого сервера не выполняется расчет размеров и затрат.

## <a name="calculate-readiness"></a>Рассчитать готовность

Не все машины подходят для запуска в Azure. Оценка сервера оценивает все внутренние машины и присваивает им категорию готовности.

- **Готова к Azure**: Машина может быть перенесена как в Azure без каких-либо изменений. Он начнется в Azure с полной поддержкой Azure.
- **Условно готова к Azure:** машина может начаться в Azure, но может не иметь полной поддержки Azure. Например, Azure не поддерживает машину, которая работает со старой версией Windows Server. Вы должны быть осторожны, прежде чем перенести эти машины в Azure. Чтобы устранить любые проблемы готовности, следуйте рекомендациям по исправлению положения, которые предлагает оценка.
- **Не готов к Azure**: Машина не будет запускаться в Azure. Например, если на диске в помещении машины хранится более 64 ТБ, Azure не может разместить машину. Следуйте инструкциям по исправлению положения, чтобы устранить проблему перед миграцией.
- **Готовность неизвестна**: Azure Migrate не может определить готовность машины из-за недостаточного количества метаданных.

Для расчета готовности Server Assessment проверяет свойства машины и настройки операционной системы, суммированные в следующих таблицах.

### <a name="machine-properties"></a>Свойства виртуальной машины

Оценка сервера анализирует следующие свойства внутреннего VM, чтобы определить, может ли он работать на Azure.

Свойство. | Сведения | Состояние готовности к работе в Azure
--- | --- | ---
**Тип загрузки** | Azure поддерживает VMs с типом загрузки BIOS, а не UEFI. | Условно готов, если тип загрузки UEFI
**Ядра** | Каждая машина должна иметь не более 128 ядер, что является максимальным числом, которое поддерживает Azure VM.<br/><br/> Если доступен журнал производительности, служба "Миграция Azure" рассматривает используемые ядра для сравнения. Если в настройках оценки указывается коэффициент комфорта, количество использованных ядер умножается на коэффициент комфорта.<br/><br/> Если нет истории производительности, Azure Migrate использует выделенные ядра без применения фактора комфорта. | Готовы, если количество ядер находится в пределах
**Оперативной памяти** | Каждая машина должна иметь не более 3892 ГБ оперативной памяти, что&nbsp;является максимальным размером azure M-серии Standard_M128m<sup>2</sup> ВМ поддерживает. [Подробнее](https://docs.microsoft.com/azure/virtual-machines/windows/sizes).<br/><br/> При наличии истории производительности Azure Migrate рассматривает используемую оперативную память для сравнения. Если указан коэффициент комфорта, используемая оперативная память умножается на коэффициент комфорта.<br/><br/> Если нет истории, выделенная оперативная память используется без применения фактора комфорта.<br/><br/> | Готов, если количество оперативной памяти находится в пределах предела
**Диск хранилища** | Выделенный размер диска должен быть не более 32 ТБ. Хотя Azure поддерживает 64-ТБ-диски с дисками Azure Ultra SSD, Azure Migrate: Server Assessment в настоящее время проверяет 32 ТБ как предел размера диска, потому что он еще не поддерживает Ultra SSD. <br/><br/> Количество дисков, прикрепленных к машине, включая диск ОС, должно быть 65 или меньше. | Готовы, если размер и номер диска находятся в пределах
**Сеть** | Машина должна иметь не более 32 сетевых интерфейсов (NICs) прилагается к нему. | Готовы, если количество NICs находится в пределах

### <a name="guest-operating-system"></a>Операционная система на виртуальной машине

Наряду с анализом свойств VM, Server Assessment рассматривает гостевую операционную систему машины, чтобы определить, может ли она работать на Azure.

> [!NOTE]
> Для обработки анализа гостей для VMware VMs, Server Assessment использует операционную систему, указанную для VM в vCenter Server. Для Linux VMs, работающих на VMware, Оценка сервера в настоящее время не определяет версию ядра гостевой ОС.

Оценка сервера использует следующую логику для определения готовности Azure на основе операционной системы:

**Операционная система** | **Сведения** | **Состояние готовности к работе в Azure**
--- | --- | ---
Windows Server 2016 и все пакеты обновления | Azure обеспечивает полную поддержку. | Готов к Azure.
Windows Server 2012 R2 и все пакеты обновления | Azure обеспечивает полную поддержку. | Готов к Azure.
Windows Server 2012 и все пакеты обновления | Azure обеспечивает полную поддержку. | Готов к Azure.
Windows Server 2008 R2 со всеми пакетами обновления | Azure обеспечивает полную поддержку.| Готов к Azure.
Windows Server 2008 (32-разрядная и 64-разрядная) | Azure обеспечивает полную поддержку. | Готов к Azure.
Windows Server 2003 и Windows Server 2003 R2 | Эти операционные системы прошли сроки окончания поддержки и нуждаются в [Соглашении о таможенной поддержке (CSA)](https://aka.ms/WSosstatement) для поддержки в Azure. | Условно готов к Azure. Перед переходом в Azure можно обновить ОС.
Windows 2000, Windows 98, Windows 95, Windows NT, Windows 3.1 и MS-DOS | Эти операционные системы прошли сроки окончания поддержки. Машина может быть приведена в начало в Azure, но Azure не предоставляет поддержки ОС. | Условно готов к Azure. Мы рекомендуем обновить ОС перед переходом в Azure.
Windows 7, Windows 8 и Windows 10 | Azure предоставляет поддержку [только подпиской Visual Studio.](https://docs.microsoft.com/azure/virtual-machines/windows/client-images) | Условно готов к Azure.

## <a name="calculating-sizing"></a>Расчет размеров

После того, как машина будет помечена как готовая к Azure, Server Assessment дает рекомендации по размеру. В этих рекомендациях определены Azure VM и disk SKU. Расчеты размеров зависят от того, используете ли вы как-это на месте размеров или на основе производительности размера.

### <a name="calculate-sizing-as-is-on-premises"></a>Рассчитать размер (как-это на-предпосылки)

 Если вы используете как-это на-предпосылки размеров, Оценка сервера не учитывает историю производительности VMs и дисков.

- **Вычисление размеров**: Оценка сервера выделяет Azure VM SKU в зависимости от размера, выделенного на месте.
- **Хранение и размер диска**: Оценка сервера рассматривает тип хранилища, указанный в свойствах оценки, и рекомендует соответствующий тип диска. Возможные типы хранилища: Standard HDD, Standard SSD и Premium. Тип хранилища по умолчанию — Премиум.
- **Размер сети**: Оценка сервера учитывает сетевой адаптер на предприимчезной машине.

### <a name="calculate-sizing-performance-based"></a>Рассчитать размер (на основе производительности)

Если вы используете размер на основе производительности, Server Assessment дает рекомендации по размеру следующим образом:

- Оценка сервера учитывает историю производительности машины для определения размера VM и типа диска в Azure.
- Если вы импортируете серверы с помощью файла CSV, используются указанные значения. Этот метод особенно полезен, если вы перераспределили локальная машина, использование является низким, и вы хотите, чтобы исправить Azure VM, чтобы сэкономить затраты.
- Если вы не хотите использовать данные о производительности, сбросить критерии размеров в качестве-это на месте, как описано в предыдущем разделе.

#### <a name="calculate-storage-sizing"></a>Рассчитать размер хранилища

Для размера хранилища Azure Migrate пытается сопоставить каждый диск, подключенный к машине, к диску Azure. Размер работает следующим образом:

1. Оценка сервера добавляет чтение и запись IOPS диска, чтобы получить общий требуемый IOPS. Аналогичным образом, он добавляет значения пропускной записи чтения и записи, чтобы получить общую пропускную стоимость каждого диска.
1. Если вы указали тип хранилища как автоматический, выбранный тип основан на эффективных значениях IOPS и пропускной стоимости. Оценка сервера определяет, следует ли отобразить диск со стандартным HDD, Стандартным SSD или Премиум-диском в Azure. Если тип хранилища установлен на один из этих типов дисков, Server Assessment пытается найти диск SKU в выбранном типе хранилища.
1. Диски выбираются следующим образом:
    - Если Server Assessment не может найти диск с требуемым IOPS и пропускной памятью, он помечает машину как неподходящую для Azure.
    - Если Server Assessment находит набор подходящих дисков, он выбирает диски, поддерживающие месторасположение, указанное в настройках оценки.
    - Если существует несколько подходящих дисков, Server Assessment выбирает диск с наименьшими затратами.
    - Если данные о производительности для любого диска недоступны, размер диска конфигурации используется для поиска стандартного SSD-диска в Azure.

#### <a name="calculate-network-sizing"></a>Рассчитать размер сети

Server Assessment пытается найти Azure VM, который поддерживает номер и требуемую производительность сетевых адаптеров, прикрепленных к предприимчивой машине.

- Чтобы получить эффективную сетевую производительность на месте VM, Server Assessment агрегирует скорость передачи данных из машины (сети из) во всех адаптерах сети. Затем он применяет фактор комфорта. Полученные значения используются для поиска VM Azure, который может поддерживать требуемую производительность сети.
- Наряду с производительностью сети, Server Assessment также рассматривает вопрос о том, может ли Azure VM поддерживать необходимое количество сетевых адаптеров.
- Если данные о производительности сети недоступны, Server Assessment учитывает только подсчет сетевого адаптера для размеров VM.

#### <a name="calculate-compute-sizing"></a>Рассчитать размеры вычислений

После расчета требований к хранению и сети Server Assessment учитывает требования к процессору и оперативной памяти для поиска подходящего размера VM в Azure.

- Azure Migrate рассматривает эффективные используемые ядра и оперативную память, чтобы найти подходящий размер Azure VM.
- Если найти подходящий размер не удалось, компьютер помечается как неподходящий для Azure.
- Если найден подходящий размер, служба "Миграция Azure" применяет результаты вычисления параметров хранилища и сети. Затем он применяет настройки местоположения и ценового уровня для окончательной рекомендации по размеру VM.
- Если доступно несколько соответствующих размеров виртуальных машин, служба выбирает рекомендуемый, который обеспечивает наименьшие затраты.

## <a name="confidence-ratings-performance-based"></a>Рейтинги доверия (на основе производительности)

Каждая оценка на основе производительности в Azure Migrate связана с рейтингом доверия. Рейтинг варьируется от одной (самой низкой) до пяти (самых высоких) звезд. Рейтинг доверия помогает оценить надежность рекомендаций по размеру, которые предоставляет Azure Migrate.

- Рейтинг доверия присваивается оценке. Рейтинг основан на наличии точек данных, необходимых для вычисления оценки.
- Для размеров на основе производительности оценка сервера нуждается в:
    - Данные об использовании процессора и оперативной памяти VM.
    - Диск IOPS и данные пропускной их прокладки для каждого диска, прикрепленного к VM.
    - Сеть вв:2 для обработки размеров на основе производительности для каждого сетевого адаптера, прикрепленного к VM.

Если какой-либо из этих номеров использования недоступен, рекомендации по размеру могут быть ненадежными.

> [!NOTE]
> Рейтинги доверия не присваиваются серверам, оцениваемым с помощью импортируемого файла CSV. Рейтинги также не применимы для оценки как-есть на-предварительной.

### <a name="ratings"></a>Рейтинги

В этой таблице показаны оценки доверия, которые зависят от процентной доли имеющихся точек данных:

   **Уровень доступности точек данных** | **Оценка достоверности**
   --- | ---
   0-20% | 1 звезда
   21-40% | 2 звезды
   41-60% | 3 звезды
   61-80% | 4 звезды
   81-100% | 5 звезд

### <a name="low-confidence-ratings"></a>Низкие рейтинги доверия

Вот несколько причин, почему оценка может получить низкий рейтинг доверия:

- Вы не профилируйте свою среду в течение всего срока, в течение которого вы создаете оценку. Например, если вы создаете оценку с продолжительностью производительности, установленной на один день, необходимо подождать по крайней мере через день после начала обнаружения всех точек данных для сбора.
- Некоторые ВМ были закрыты в то время, за которое была рассчитана оценка. Если какие-либо VM отключены на некоторое время, Оценка сервера не может собирать данные о производительности за этот период.
- Некоторые ВМ были созданы в то время, за которое была рассчитана оценка. Например, предположим, что вы создали оценку для истории производительности за последний месяц, но некоторые ВМ были созданы всего неделю назад. История производительности новых ВМ не будет существовать в течение всего срока действия.

> [!NOTE]
> Если рейтинг доверия любой оценки меньше пяти звезд, мы рекомендуем вам подождать по крайней мере день, пока прибор профилирует окружающую среду, а затем пересчитывает оценку. В противном случае размерна на основе производительности может быть ненадежной. В этом случае мы рекомендуем переключить оценку на размер ы на местах.

## <a name="calculate-monthly-costs"></a>Рассчитать ежемесячные расходы

После завершения рекомендаций по размеру Azure Migrate вычисляет затраты на вычисления и хранение после миграции.

- **Затраты на вычисления:** Azure Migrate использует рекомендуемый размер Azure VM и API выставления счетов Azure для расчета ежемесячной стоимости VM.

    При расчете учитываются:
    - Операционная система
    - Гарантия программного обеспечения
    - Зарезервированные экземпляры
    - Время доступности виртуальной машины
    - Расположение
    - Настройки валюты

    Оценка сервера агрегирует стоимость всех машин для расчета общей ежемесячной вычислительной стоимости.

- **Стоимость хранения**: Ежемесячная стоимость хранения для машины рассчитывается путем агрегирования ежемесячной стоимости всех дисков, которые подключены к машине.

    Оценка сервера вычисляет общие ежемесячные затраты на хранение, агрегируя затраты на хранение всех машин. В настоящее время при расчете не учитываются предложения, указанные в настройках оценки.

Цены отображаются в валюте, заданной в настройках оценки.
