# <a name="troubleshoot-the-azure-migrate-appliance-and-discovery"></a>Устранение неполадок прибора и обнаружения Azure Migrate

Эта статья помогает выснять проблемы при развертывании прибора [Azure Migrate](migrate-services-overview.md) и использовать прибор для обнаружения на месте машин.


## <a name="whats-supported"></a>Поддерживаемые функции

[Просмотрите](migrate-appliance.md) требования к поддержке приборов.


## <a name="invalid-ovf-manifest-entry"></a>"Недействительный OVF манифест вступления"

Если вы получили ошибку "Предоставленный файл манифеста недействителен: недействительный ввод OVF", сделайте следующее:

1. Убедитесь, что файл OVA с качается правильно, проверяя его значение хэша. [Подробнее](https://docs.microsoft.com/azure/migrate/tutorial-assessment-vmware). Если значение хэша не совпадает, загрузите файл OVA снова и повторите попытку развертывания.
2. Если развертывание по-прежнему не удается, и вы используете клиента VMware vSphere для развертывания файла OVF, попробуйте развернуть его через веб-клиент vSphere. Если развертывание по-прежнему не удается, попробуйте использовать другой веб-браузер.
3. Если вы используете веб-клиент vSphere и пытаетесь развернуть его на vCenter Server 6.5 или 6.7, попробуйте развернуть OVA непосредственно на хосте ESXi:
   - Подключайтесь к хосту ESXi напрямую (вместо vCenter Server) с веб-клиентом (https://<*хост IP-адрес*>/ui).
   - В **home** > **Inventory**выберите шаблон **«Развертывание файлов»** > **OVF.** Просмотрите OVA и завершите развертывание.
4. Если развертывание по-прежнему не удается, обратитесь в поддержку службы "Миграция Azure".

## <a name="cant-connect-to-the-internet"></a>Не устоияните к интернету

Это может произойти, если аппарат прибора находится за прокси.

- Убедитесь, что вы предоставляете учетные данные авторизации, если они нужны прокси.
- Если вы используете прокси-сервер на основе URL-адреса для управления исходящим подключением, добавьте [эти URL-адреса](migrate-appliance.md#url-access) в список разрешений.
- Если вы используете перехватный прокси для подключения к Интернету, импортировать прокси-сертификат на прибор VM, используя [эти шаги.](https://docs.microsoft.com/azure/migrate/concepts-collector)

## <a name="cant-sign-into-azure-from-the-appliance-web-app"></a>Не упускайте регистрацию в Azure из веб-приложения прибора

Ошибка "Извините, но у нас возникли проблемы с ввозам" появляется, если вы используете неправильную учетную запись Azure для вхотворки в Azure. Эта ошибка возникает по нескольким причинам:

- Если вы входите в веб-приложение для общедоступного облака, с помощью учетных данных пользователей для правительственного облачного портала.
- Если вы входите в веб-приложение для правительственного облака, используя учетные данные пользователей для частного облачного портала.

Убедитесь, что вы используете правильные учетные данные.

##  <a name="datetime-synchronization-error"></a>Ошибка синхронизации даты/времени

Ошибка синхронизации даты и времени (802) указывает на то, что часы сервера могут быть не синхронизированы с текущим временем более чем на пять минут. Измените время часы на сборщике VM, чтобы соответствовать текущему времени:

1. Откройте командную строку с правами администратора на этой виртуальной машине.
2. Чтобы проверить часовой пояс, запустите **w32tm /tz**.
3. Чтобы синхронизировать время, запустите **w32tm /resync.**


## <a name="unabletoconnecttoserver"></a>"UnabletoConnectToServer"

Если вы обнаружите эту ошибку соединения, вы можете быть не в состоянии подключиться к vCenter Server *Servername*.com:9443. Детали ошибки указывают на то, что `https://\*servername*.com:9443/sdk` нет никакой точки прослушивания, которая может принять сообщение.

- Проверьте, используете ли вы последнюю версию прибора. Если это не так, обновите прибор до [последней версии.](https://docs.microsoft.com/azure/migrate/concepts-collector)
- Если проблема все еще возникает в последней версии, прибор может быть не в состоянии решить указанное имя vCenter Server, или указанный порт может быть неправильным. По умолчанию, если порт не указан, коллектор попытается подключиться к порту номер 443.

    1. Ping *Servername*.com от прибора.
    2. Если шаг 1 не удается, попробуйте подключиться к серверу vCenter с помощью IP-адреса.
    3. Определите правильный номер порта для подключения к vCenter Server.
    4. Проверить, что vCenter Server запущен и запущен.


## <a name="error-6005260039-appliance-might-not-be-registered"></a>Ошибка 60052/60039: Прибор может не быть зарегистрирован

- Ошибка 60052, "Прибор может быть незарегистрирован успешно в проекте Azure Migrate" происходит, если учетная запись Azure, используемая для регистрации прибора, имеет недостаточно ещр.
    - Убедитесь, что учетная запись пользователя Azure, используемая для регистрации прибора, имеет по крайней мере разрешения на подписку.
    - [Узнайте больше](https://docs.microsoft.com/azure/migrate/migrate-appliance#appliance---vmware) об необходимых ролях и разрешениях Azure.
- Ошибка 60039 " Прибор может быть незарегистрирован успешно в проекте Azure Migrate" может произойти, если регистрация не удается, потому что проект Azure Migrate, используемый для регистра, прибор не может быть найден.
    - На портале Azure и проверить, существует ли проект в группе ресурсов.
    - Если проекта не существует, создайте новый проект Azure Migrate в группе ресурсов и снова зарегистрируйте прибор. [Узнайте, как](https://docs.microsoft.com/azure/migrate/how-to-add-tool-first-time#create-a-project-and-add-a-tool) создать новый проект.

## <a name="error-6003060031-key-vault-management-operation-failed"></a>Ошибка 60030/60031: Операция управления Ключом Vault не удалась

Если вы получили ошибку 60030 или 60031, "Операция управления ключем Azure Vault не удалось", сделайте следующее:
- Убедитесь, что учетная запись пользователя Azure, используемая для регистрации прибора, имеет по крайней мере разрешения на подписку.
- Убедитесь, что учетная запись имеет доступ к хранилищу ключей, указанному в сообщении об ошибке, а затем повторно попробуйте операцию.
- Если проблема будет повторяться, обратитесь в службу поддержки Майкрософт.
- [Узнайте больше](https://docs.microsoft.com/azure/migrate/migrate-appliance#appliance---vmware) об необходимых ролях и разрешениях Azure.

## <a name="error-60028-discovery-couldnt-be-initiated"></a>Ошибка 60028: Открытие не может быть начато

Ошибка 60028: "Открытие не может быть начато из-за ошибки. Операция, сдавшая сярот для указанного списка узлов или кластеров, указывает на то, что обнаружение не может быть запущено на узлах, перечисленных в ошибке, из-за проблемы с доступом к информации VM или извлечением. Остальные хозяева были успешно добавлены.

- Добавление узлов, перечисленных в ошибке, снова, используя опцию **Добавить хост.**
- Если есть ошибка проверки, просмотрите руководство по исправлению, чтобы исправить ошибки, а затем попробуйте **Сохранить и начать** опцию обнаружения снова.

## <a name="error-60025-azure-ad-operation-failed"></a>Ошибка 60025: работа Azure AD не удалась 
Ошибка 60025: "Операция Azure AD не удалась. Ошибка произошла при создании или обновлении приложения Azure AD, когда учетная запись пользователя Azure, используемая для инициирования обнаружения, отличается от учетной записи, используемой для регистрации прибора. Выполните одно из следующих действий.

- Убедитесь, что учетная запись пользователя, использовавающая открытие, такая же, как и та, которая используется для регистрации прибора.
- Предоставьте разрешения приложения Azure Active Directory на учетную запись пользователя, для которой операция обнаружения не работает.
- Удалите группу ресурсов, ранее созданную для проекта Azure Migrate. Создайте другую группу ресурсов, чтобы начать снова.
- [Узнайте больше](https://docs.microsoft.com/azure/migrate/migrate-appliance#appliance---vmware) о разрешениях приложений Azure Active Directory.


## <a name="error-50004-cant-connect-to-host-or-cluster"></a>Ошибка 50004: не усомниться в хостеле или кластере

Ошибка 50004: "Не может подключиться к хосту или кластеру, потому что имя сервера не может быть решена. Код ошибки WinRM: 0x803381B9" может произойти, если служба Azure DNS для прибора не сможет решить предоставленное вами название кластера или узла.

- Если вы видите эту ошибку в кластере, кластер F-DN.
- Вы также можете увидеть эту ошибку для узлов в кластере. Это означает, что прибор может подключиться к кластеру, но кластер возвращает имена хостеров, которые не являются F-DNs. Чтобы устранить эту ошибку, обновите файл хоста на приборе, добавив отображение IP-адреса и имен узлов:
    1. Откройте блокнот в качестве админа.
    2. Откройте файл C:'Windows-System32,Драйверы и хосты.
    3. Добавить IP-адрес и имя хоста в строку. Повторите для каждого узла или кластера, где вы видите эту ошибку.
    4. Сохраните и закройте файл hosts.
    5. Проверьте, может ли прибор подключиться к хостам с помощью приложения для управления приборами. Через 30 минут вы должны увидеть последнюю информацию об этих хостах на портале Azure.

## <a name="discovered-vms-not-in-portal"></a>Обнаруженные VMs не на портале

Если состояние обнаружения находится "Открытие в процессе", но еще не видите виз на портале, подождите несколько минут:
- Это займет около 15 минут для VMware VM .
- Для каждого добавленного хоста требуется около двух минут для обнаружения Hyper-V VM.

Если вы ждете и состояние не изменится, выберите **Обновление** на вкладке **«Серверы».** Это должно показать количество обнаруженных серверов в Azure Migrate: Оценка серверов и Миграция Azure: Миграция серверов.

Если это не работает, и вы обнаружите серверы VMware:

- Убедитесь, что указанная вами учетная запись vCenter имеет правильное разрешение, с доступом по крайней мере к одному VM.
- Azure Migrate не может обнаружить VMware VMs, если учетная запись vCenter имеет доступ на уровне папок vCenter VM. [Узнайте больше](set-discovery-scope.md) об открытии.

## <a name="vm-data-not-in-portal"></a>Данные ВМ не на портале

Если обнаруженные VMs не отображаются на портале или данные VM устарели, подождите несколько минут. Изменение обнаруженных данных конфигурации VM на портале занимает до 30 минут. Для появления изменений в данных приложения может потребоваться несколько часов. Если по истечение этого времени нет данных, попробуйте обновиться следующим образом

1. В **серверах** > **Azure мигрирует оценка серверов,** выберите **Обзор**.
2. Под **управлением,** выберите **Здоровье агента**.
3. Выберите **агент обновления**.
4. Подождите, пока операция обновления будет завершена. Теперь вы должны увидеть актуальную информацию.

## <a name="deleted-vms-appear-in-portal"></a>Удаленные VM-м внедряются на портале

Если вы удалите ВМ и они все еще появятся на портале, подождите 30 минут. Если они все еще появляются, освежите, как описано выше.
