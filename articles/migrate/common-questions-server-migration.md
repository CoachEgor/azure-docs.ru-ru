## <a name="how-does-agentless-vmware-replication-work"></a>Как работает репликация VMware без агента?

Метод репликации без агента для VMware использует моментальные снимки VMware, а VMware — отслеживание измененных блоков (CBT).

1. При запуске репликации планируются начальные циклы репликации. В начальном цикле создается моментальный снимок виртуальной машины. Этот моментальный снимок используется для репликации VMDK-дисков виртуальных машин. 
2. По завершении цикла начальной репликации разностные циклы репликации планируются периодически.
    - Во время разностной репликации создается моментальный снимок, а также блоки данных, которые изменились с момента предыдущего цикла репликации.
    - VMware CBT используется для определения блоков, которые были изменены с момента последнего цикла.
    - Частота периодических циклов репликации автоматически управляется службой "миграция Azure" в зависимости от того, сколько других виртуальных машин и дисков одновременно реплицируются из одного хранилища данных. В идеальном случае репликация в конечном итоге сходится на один цикл в час для каждой виртуальной машины.

При миграции на компьютер планируется цикл репликации по запросу для записи оставшихся данных. Вы можете отключить компьютер во время миграции, чтобы обеспечить нулевую целостность данных и согласованность приложений.

## <a name="why-isnt-resynchronization-exposed"></a>Почему повторная синхронизация не предоставляется?

Во время миграции без агента в каждом разностном цикле записывается разница между текущим моментальным снимком и ранее сделанным моментальным снимком. Так как это всегда различие между моментальными снимками, сворачивание данных в. Так что если определенный сектор записывается в N раз между моментальными снимками, необходимо передать только последнюю запись, так как мы заинтересованы только в последней синхронизации. Это отличается от репликации на основе агентов, в которой мы относимся к записи и применяем каждую запись. Это означает, что каждый разностный цикл является повторной синхронизацией. Поэтому параметр повторной синхронизации не предоставляется. Если все диски не синхронизированы из-за сбоя, они фиксируются в следующем цикле. 

## <a name="how-does-churn-rate-impact-agentless-replication"></a>Как частота обработки влияет на репликацию без агента?

Так как репликация без агента не имеет даты, шаблон обновления важнее, что скорость изменения. Когда файл снова записывается и снова, скорость не оказывает существенного воздействия. Однако шаблон, в котором записывается каждый другой сектор, вызывает высокое обновление в следующем цикле. Так как мы максимально сокращаем объем передаваемых данных, мы разрешают, чтобы данные выходили как можно больше, прежде чем планировать следующий цикл.  

## <a name="how-frequently-is-a-replication-cycle-scheduled"></a>Как часто планируется цикл репликации?

Формула для планирования следующего цикла репликации: (предыдущее время цикла/2) или 1 час, в зависимости от того, что больше.

Например, если для разностного цикла виртуальная машина занимает четыре часа, следующий цикл планируется в два часа, а не в течение следующего часа. Это происходит сразу же после начальной репликации, когда первый разностный цикл запланирован немедленно.

## <a name="how-does-agentless-replication-impact-vmware-servers"></a>Как репликация без агента влияет на серверы VMware?

На узлы vCenter Server и ESXi возникает некоторое влияние на производительность. Так как репликация без агента использует моментальные снимки, она потребляет операции ввода-вывода в хранилище и требует некоторых операций ввода-вывода в секунду. Не рекомендуется использовать репликацию без агента, если в вашей среде есть ограничения на хранилище или операции ввода-вывода.

## <a name="can-i-do-agentless-migration-of-uefi-vms-to-azure-gen-2"></a>Можно ли выполнять миграцию виртуальных машин UEFI без агента в Azure Gen 2?

Нет. Используйте Azure Site Recovery для переноса этих виртуальных машин на виртуальные машины Azure поколения 2. 

## <a name="can-i-pin-vms-to-azure-availability-zones-when-i-migrate"></a>Можно ли закрепить виртуальные машины для Зоны доступности Azure при миграции?

Нет, Зоны доступности Azure не поддерживаются.

## <a name="which-transport-protocol-is-used-by-azure-migrate-during-replication"></a>Какой транспортный протокол используется при миграции Azure во время репликации?

Служба "миграция Azure" использует протокол сетевого блока (NBD) с шифрованием SSL.

## <a name="what-is-the-minimum-vcenter-server-version-required-for-migration"></a>Какова минимальная версия vCenter Server, необходимая для миграции?

Необходимо иметь по крайней мере vCenter Server 5,5 и VMware vSphere узла ESXi версии 5,5.

## <a name="can-customers-migrate-their-vms-to-unmanaged-disks"></a>Могут ли клиенты переносить свои виртуальные машины на неуправляемые диски?

Нет. Служба "миграция Azure" поддерживает миграцию только на управляемые диски (стандартный HDD, SSD уровня "Премиум").

## <a name="how-many-vms-can-i-replicate-together-with-agentless-migration"></a>Сколько виртуальных машин можно реплицировать вместе с миграцией без агента?

Сейчас можно одновременно перенести 100 виртуальных машин на vCenter Server. Выполните миграцию в пакетах по 10 виртуальным машинам.
