## <a name="how-does-agentless-vmware-replication-work"></a>Как работает репликация VMware без агента?

Метод репликации без агента для VMware использует моментальные снимки VMware и VMware Change Block Tracking (CBT).

Вот как это делается:

1. При запуске репликации планируются начальные циклы репликации. В начальном цикле создается моментальный снимок виртуальной машины. Моментальный снимок используется для репликации VMDK-дисков виртуальных машин. 
2. По завершении цикла начальной репликации разностные циклы репликации планируются периодически.
    - Во время разностной репликации создается моментальный снимок, а также блоки данных, которые изменились с момента предыдущего цикла репликации.
    - VMware CBT используется для определения блоков, которые были изменены с момента последнего цикла.
    - Частота периодических циклов репликации автоматически управляется службой "миграция Azure" и зависит от того, сколько других виртуальных машин и дисков параллельно реплицируются из одного хранилища данных. В идеальном случае репликация в конечном итоге сходится на один цикл в час для каждой виртуальной машины.

Во время миграции планируется цикл репликации по запросу для записи оставшихся данных на компьютере. Чтобы обеспечить нулевую целостность данных и согласованность приложений, можно выбрать завершение работы компьютера во время миграции.

## <a name="why-isnt-resynchronization-exposed"></a>Почему повторная синхронизация не предоставляется?

Во время миграции без агента в каждом цикле Дельта записывается разница между текущим моментальным снимком и ранее сделанным моментальным снимком. Это всегда разница между моментальными снимками, а также сворачиванием данных в. Если конкретный сектор записывается в *N* раз между моментальными снимками, необходимо передать только последнюю запись, так как мы заинтересованы только в последней синхронизации. Процесс отличается от репликации на основе агента, в течение которого мы отслеживающим и применяя каждую запись. В этом процессе каждый разностный цикл является повторной синхронизацией. Таким образом, параметр повторной синхронизации не предоставляется. Если диски никогда не синхронизируются из-за сбоя, они фиксируются в следующем цикле. 

## <a name="how-does-churn-rate-affect-agentless-replication"></a>Как частота изменения влияет на репликацию без агента?

Поскольку репликация без агента складывает данные, *шаблон* обновления важнее, чем *скорость*изменения. Когда файл снова записывается и снова, скорость не оказывает существенного воздействия. Однако шаблон, в котором записывается каждый другой сектор, вызывает высокое обновление в следующем цикле. Так как мы максимально сокращаем объем передаваемых данных, мы разрешаете, что данные будут по возможности максимально точны до планирования следующего цикла.  

## <a name="how-frequently-is-a-replication-cycle-scheduled"></a>Как часто планируется цикл репликации?

Формула для планирования следующего цикла репликации — (предыдущее время цикла/2) или один час, в зависимости от того, что больше.

Например, если для разностного цикла виртуальная машина занимает четыре часа, следующий цикл планируется в два часа, а не в течение следующего часа. Процесс будет отличаться сразу после начальной репликации, если первый разностный цикл запланирован немедленно.

## <a name="how-does-agentless-replication-affect-vmware-servers"></a>Как репликация без агента влияет на серверы VMware?

Репликация без агента приводит к снижению производительности на узлах VMware vCenter Server и VMware ESXi. Поскольку репликация без агента использует моментальные снимки, она потребляет операции ввода-вывода в хранилище, поэтому требуется пропускная способность хранилища операций ввода-вывода в секунду. Не рекомендуется использовать репликацию без агента при наличии ограничений на хранилище или операции ввода-вывода в среде.

## <a name="can-i-do-agentless-migration-of-uefi-vms-to-azure-gen-2"></a>Можно ли выполнять миграцию виртуальных машин UEFI без агента в Azure Gen 2?

Нет. Используйте Azure Site Recovery для переноса этих виртуальных машин на виртуальные машины Azure поколения 2. 

## <a name="can-i-pin-vms-to-azure-availability-zones-when-i-migrate"></a>Можно ли закрепить виртуальные машины для Зоны доступности Azure при миграции?

Нет. Зоны доступности Azure не поддерживаются для миграции службы "миграция Azure".

## <a name="what-transport-protocol-does-azure-migrate-use-during-replication"></a>Какой транспортный протокол используется для миграции Azure во время репликации?

Служба "миграция Azure" использует протокол сетевого блока (NBD) с шифрованием TLS.

## <a name="what-is-the-minimum-vcenter-server-version-required-for-migration"></a>Какова минимальная версия vCenter Server, необходимая для миграции?

Необходимо иметь по меньшей мере vCenter Server 5,5 и vSphere ESXi узла версии 5,5.

## <a name="can-customers-migrate-their-vms-to-unmanaged-disks"></a>Могут ли клиенты переносить свои виртуальные машины на неуправляемые диски?

Нет. Служба "миграция Azure" поддерживает миграцию только на управляемые диски (HDD (цен. категория "Стандартный"), SSD (цен. категория "Премиум")).

## <a name="how-many-vms-can-i-replicate-at-one-time-by-using-agentless-migration"></a>Сколько виртуальных машин можно реплицировать одновременно с помощью миграции без агента?

Сейчас можно одновременно перенести 100 виртуальных машин на экземпляр vCenter Server. Выполните миграцию в пакетах по 10 виртуальным машинам.

## <a name="can-i-migrate-vms-that-are-already-being-replicated-to-azure"></a>Можно ли перенести виртуальные машины, которые уже реплицируются в Azure? 

Если виртуальные машины уже реплицируются в Azure другими способами, вы не сможете перенести эти компьютеры в качестве виртуальных машин с миграцией Azure Migration Server. В качестве обходного решения виртуальные машины можно рассматривать как физические серверы и переносить их в соответствии с [поддерживаемой миграцией физического сервера](migrate-support-matrix-physical-migration.md).

## <a name="when-do-i-migrate-machines-as-physical-servers"></a>Когда выполняется перенос компьютеров в качестве физических серверов?

Перенос компьютеров в качестве физических серверов полезен в ряде сценариев:

- При переносе локальных физических серверов.
- Если выполняется миграция виртуальных машин, виртуализованных такими платформами, как Xen, KVM.
- Чтобы выполнить миграцию виртуальных машин Hyper-V или VMware, по каким-либо причинам вы не можете использовать стандартный процесс миграции для [Hyper-v](tutorial-migrate-hyper-v.md)или миграции [VMware](server-migrate-overview.md) . Например, если вы не используете VMware vCenter и используете только узлы ESXi.
- Миграция виртуальных машин, которые в настоящее время выполняются в частных облаках, в Azure
- Если вы хотите перенести виртуальные машины, работающие в общедоступных облаках, например Amazon Web Services (AWS) или Google Cloud Platform (обеспечить), в Azure.

## <a name="i-deployed-two-or-more-appliances-to-discover-vms-in-my-vcenter-server-however-when-i-try-to-migrate-the-vms-i-only-see-vms-corresponding-to-one-of-the-appliance"></a>Я развернул два (или более) устройства для обнаружения виртуальных машин в моем vCenter Server. Однако при попытке перенести виртуальные машины отображаются только виртуальные машины, соответствующие одному из устройств.

Хотя это может быть хорошим вариантом использования, мы сейчас не поддерживаем его. Развертывание двух (или более) устройств для обнаружения одного и того же набора виртуальных машин вызывает ошибку службы, при которой владение виртуальной машиной сохраняет переключение между двумя устройствами. Это объясняется тем, что виртуальные машины отображаются и отменяются. В таких случаях для устранения проблемы необходимо удалить одно устройство и выполнить жесткое обновление.

## <a name="do-i-need-vmware-vcenter-to-migrate-vmware-vms"></a>Требуется ли VMware vCenter для переноса виртуальных машин VMware?
Чтобы [перенести виртуальные машины VMware](server-migrate-overview.md) с помощью миграции с агента VMware или без агента, ESXi узлы, на которых расположены виртуальные машины, должны управляться с помощью vCenter Server. Если у вас нет vCenter Server, можно выполнить перенос виртуальных машин VMware, переполнив их как физические серверы. [Подробнее.](migrate-support-matrix-physical-migration.md)
