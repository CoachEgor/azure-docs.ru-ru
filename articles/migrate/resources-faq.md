### <a name="what-data-is-collected-by-azure-migrate"></a>Какие данные собираются службой "Миграция Azure"?

Модуль "Миграция Azure" собирает метаданные для локальных виртуальных машин, включая:

**Данные конфигурации виртуальной машины:**
- отображаемое имя виртуальной машины (в vCenter);
- путь инвентаризации виртуальной машины (узел, кластер или папка в vCenter);
- IP-адрес
- MAC-адрес.
- Операционная система
- Число ядер, дисков и сетевых адаптеров.
- Объем памяти, размеры диска.

**Сведения о производительности виртуальной машины:**
- Загрузка ЦП.
- Использование памяти
- Для каждого диска, подключенного к виртуальной машине:
  - пропускная способность чтения с диска;
  - скорость записи на диск;
  - число операций чтения с диска в секунду;
  - число операций записи на диск в секунду.
- Для каждого сетевого адаптера, подключенного к виртуальной машине:
  - Входящая скорость сети
  - Исходящая скорость сети

TN сложение, при развертывании сопоставление зависимостей, агенты сопоставление зависимостей собирают сведения, включая полное доменное имя компьютера, операционной системы, IP-адреса, MAC-адрес, процессы, запущенные на виртуальной Машине, входящих и исходящих подключений TCP для виртуальной Машины. Это обнаружение является необязательным и используется, только если вы включите сопоставление зависимостей для обнаружения.

### <a name="is-there-any-performance-impact-on-the-analyzed-esxi-host-environment"></a>Есть ли оказания влияния на производительность проанализированного среды узла ESXi?

С помощью непрерывной профилирования данных о производительности, "Миграция Azure" Профили устройство локальных компьютеров для оценки данных производительности виртуальной Машины. This has almost zero performance impact on the ESXi hosts, as well as on the vCenter Server.

### <a name="where-is-the-collected-data-stored-and-for-how-long"></a>Где хранятся собранные данные и как долго?

Данные, собранные устройством "Миграция Azure" хранится в расположении Azure, которая указывается при создании проекта миграции. Данные безопасно хранятся в подписке Microsoft и удаляется при удалении проекта "Миграция Azure".

Для визуализации зависимостей Если установить агенты на виртуальных машинах, данные, собранные агенты зависимостей хранится в США, в рабочей области Log Analytics, созданные в подписке Azure. При удалении рабочей области Log Analytics из подписки эти данные удаляются. [Узнайте больше](concepts-dependency-visualization.md).

### <a name="what-is-the-volume-of-data-uploaded-by-azure-migrate-during-continuous-profiling"></a>Что такое тома данных, отправленных службой "Миграция Azure" во время непрерывного профилирования?

Объем данных, отправляемых "Миграция Azure" отличается в зависимости от нескольких параметров. Чтобы предоставить номером признаком, проект "Миграция Azure" с 10 компьютеров (каждый с одного диска и один сетевой Адаптер), отправляет около 50 МБ в день. Это приблизительное значение, которое изменяется в зависимости от числа точек данных для сетевых адаптеров и дисков (данные, отправляемые нелинейных Если увеличить число компьютеров, сетевых адаптеров и дисков).

### <a name="is-the-data-encrypted-at-rest-and-in-transit"></a>— Данные, зашифрованные при хранении и во время передачи?

Да, для обоих. Метаданные безопасно отправляется в службу "Миграция Azure" через Интернет, по протоколу https. Метаданные хранятся в [Cosmos DB](https://docs.microsoft.com/azure/cosmos-db/database-encryption-at-rest)и в [хранилище больших двоичных объектов](https://docs.microsoft.com/azure/storage/common/storage-service-encryption) в подписке Microsoft, и шифрование неактивных.

Данные, собранные агенты зависимостей также шифруются в процессе передачи (безопасное HTTPS) и хранится в рабочей области Log Analytics в подписке пользователя. Это также шифрование неактивных.

### <a name="how-does-the-azure-migrate-appliance-communicate-with-the-vcenter-server-and-the-azure-migrate-service"></a>Как "Миграция Azure" устройства взаимодействуют с vCenter Server и служба "Миграция Azure"?

Устройство подключается к vCenter Server (порт 443) с помощью учетных данных, указанных при настройке устройства. Он запрашивает vCenter Server с помощью VMware PowerCLI для сбора метаданных о виртуальных машинах под управлением vCenter Server. Он собирает и данные конфигурации о виртуальных машинах (ядра, память, диски, сетевой Адаптер и т.д.), а также журнал производительности каждой виртуальной машины за последний месяц. Собранные метаданные затем отправляется для переноса сервера оценки Azure (через Интернет по протоколу HTTPS) для оценки. 

### <a name="can-i-connect-the-same-appliance-to-multiple-vcenter-servers"></a>Можно подключиться одним устройством на несколько серверов vCenter?

Да, одно устройство "Миграция Azure" можно использовать для обнаружения vCenter на нескольких серверах, но не одновременно. Необходимо запустить обнаружения друг за другом.

### <a name="is-the-ova-template-used-by-site-recovery-integrated-with-the-ova-used-by-azure-migrate"></a>Можно ли интегрировать OVA-файл шаблона, используемый службой Site Recovery, с OVA-файлом, используемым службой "Миграция Azure"?

Сейчас такая интеграция невозможна. В службе Site Recovery OVA-файл используется для настройки репликации виртуальных машин VMware или физических серверов на сервере конфигурации Site Recovery. . Ova-файла, используемый службой "Миграция Azure" используется для обнаружения виртуальных машин VMware, управляемых сервером vCenter, для целей оценки и миграции.

### <a name="i-changed-my-machine-size-can-i-rerun-an-assessment"></a>Я изменил размер виртуальной машины. Повторно выполнять оценку?
Модуль "Миграция Azure" непрерывно собирает сведения о локальной среде. Тем не менее оценка — это моментальный снимок в определенный момент времени каждой локальной виртуальной машине. При изменении параметров на виртуальной Машине, вы хотите оценить, используйте параметр «Обновить», чтобы обновить оценку с последними изменениями.

### <a name="how-can-i-discover-a-multi-tenant-environment-in-azure-migrate"></a>Как обнаружить мультитенантную среду в службе "Миграция Azure"?

Для VMware Если в среде, общей для клиентов, и вы не хотите обнаружение виртуальных машин на одном клиенте в другой клиент подписки, создайте vCenter учетные данные сервера с доступом только для этих виртуальных машин, которые нужно обнаружить. Используйте учетные данные при плановых обнаружения на устройстве "Миграция Azure".

Для Hyper-V, обнаружение использует учетные данные узла Hyper-V, если виртуальные машины используют том же узле Hyper-V, в настоящее время нет способа для разделения обнаружения.  

### <a name="how-many-vms-can-be-discovered-using-a-single-migration-appliance"></a>Сколько виртуальных машин могут быть обнаружены с помощью миграций модуля?

Вы можете обработать до 10 000 виртуальных машин VMware и до 5 000 виртуальных машин Hyper-V, с помощью модуля миграций.  Если у вас несколько компьютеров в вашей локальной среде, сведения о масштабировании [Hyper-V](scale-hyper-v-assessment.md) и [VMware](scale-vmware-assessment.md) оценки.


## <a name="azure-migrate-server-assessment"></a>"Миграция Azure": Оценка Server

### <a name="does-azure-migrate-server-assessment-support-assessment-of-physical-servers"></a>Выполняет миграцию Azure: Оценка Server поддерживает оценку физических серверов?

Нет, "Миграция Azure" в настоящее время не поддерживает оценку физических серверов. 

### <a name="does-azure-migrate-need-vcenter-server-to-discover-a-vmware-environment"></a>Требуется ли службе "Миграция Azure" vCenter Server для обнаружения среды VMware?

Да, "Миграция Azure" должен vCenter Server для обнаружения в среде VMware. Он не поддерживает обнаружение узлов ESXi, которые не находятся под управлением vCenter Server.

### <a name="whats-the-difference-between-using-azure-migrate-server-assessment-and-the-map-toolkit"></a>Какова разница между использованием "Миграция Azure": Оценка Server и Map Toolkit?

"Миграция Azure": Оценка Server предоставляет оценку миграции для миграции готовности и оценку рабочих нагрузок для миграции в Azure. [Microsoft Assessment and Planning (MAP) Toolkit](https://www.microsoft.com/download/details.aspx?id=7826) имеет другие функции, такие как планирование для более новых версиях Windows клиентских и серверных операционных систем и отслеживание использования программного обеспечения миграции. Для всех этих сценариев нужно и дальше использовать MAP Toolkit.

### <a name="how-is-azure-migrate-server-assessment-different-from-azure-site-recovery-deployment-planner"></a>Как является "Миграция Azure": Оценка Server отличается от планировщика развертывания Azure Site Recovery?

"Миграция Azure": Оценка Server — это средство планирования миграции. Планировщик развертывания для Azure Site Recovery — это средство планирования аварийного восстановления.

- **Перенос из VMware или Hyper-V в Azure**: Если планируется перенос локальных серверов в Azure, используйте "Миграция Azure": Средство оценки Server для планирования миграции. Средство оценивает локальные рабочие нагрузки и предоставляет рекомендации, советы и механизмы для переноса в Azure. Когда вы будете готовы с планом миграции, можно использовать средства, такие как "Миграция Azure": Миграция сервера, чтобы перенести компьютеры в Azure.
- **Аварийное восстановление из VMware/Hyper-V в Azure**: Для аварийного восстановления в Azure с помощью Site Recovery используйте планировщик развертывания Site Recovery для аварийного восстановления. Планировщик развертывания Site Recovery выполняет глубокое, оценки сайта, относящиеся к восстановлению из вашей локальной среде. Он предоставляет рекомендации, с помощью Site Recovery, необходимые для успешного аварийного операций, таких как репликации и отработки отказа виртуальных машин.

### <a name="whats-the-difference-between-as-on-premises-sizing-and-performance-based-sizing"></a>Какова разница между как on-premises изменения размера и изменения размеров на основе производительности?

- В локальные изменения размера, "Миграция Azure" не учитывает данных производительности виртуальной Машины. Размер виртуальных машин, на основе локальной конфигурации. -В-выводе, изменения размера зависит от данных об использовании.
- Например если локальная виртуальная машина имеет 4 ядра и память объемом 8 ГБ с 50% загрузки ЦП и использование 50% памяти, как для локальной изменения размера рекомендует номер SKU виртуальной Машины Azure с 4 ядрами и 8 ГБ памяти. Определения размера на основе производительности, однако рекомендует номера SKU виртуальной Машины двух ядер и 4 ГБ, так как считается процент использования.
- Аналогичным образом изменения размера диска зависит от двух свойств оценки - изменения размера критерий и типом хранилища.
= Если условие определения размера на основе производительности и задан автоматический тип хранилища, операций ввода-ВЫВОДА и пропускной способности диска, они рассматриваются как при идентификации тип целевого диска ("стандартный" или "премиум").
- Если условие определения размера на основе производительности и тип хранилища уровня "премиум", рекомендуется использовать диск уровня "премиум". Диск уровня "премиум", выбранный номер SKU на основе размера локального диска. Такая же логика используется на диск изменения размера, когда условие определения размера — как локальные изменения размера и типа хранилища класса standard или premium.

### <a name="what-impact-does-performance-history-and-percentile-utilization-have-on-the-size-recommendations"></a>Как данные журнала производительности и процент использования влияют на рекомендации по размеру?

Эти свойства применимы только при выборе размера на основе производительности.

- Служба "Миграция Azure" получает историю производительности локальных компьютеров и использует его для рекомендует тип размера и диск виртуальной Машины в Azure.
- Модуль постоянно профили в локальной среде для сбора данных об использовании в режиме реального времени каждые 20 секунд. Модуль загружает 20-секундные образцы и создает одну точку данных каждые 15 минут. Чтобы создать одну точку данных, устройство выбирает пиковое значение из всех 20-секундных образцов и отправляет его в Azure.
- При создании оценки в Azure (на основе продолжительность производительности и значение процентиля журнал производительности), "Миграция Azure" вычисляет значение эффективного использования и использует его для изменения размера.
- Например, если задать длительность производительности быть один день, а значение процентиля 95-го процентиля в "Миграция Azure" использует точек 15 минут измерения, отправленные сборщик за последний день, отсортированных в порядке возрастания, а выбирает 95-й процентиль, как эффективного использования.
- Значение процентиля 95 гарантирует, что вы пропускаете выделяющиеся, которые могут возникать при использовании 99-го процентиля. Если вам нужно выбрать пиковое использование для периода и вы не хотите пропустить все выбросы, следует выбрать 99-й процентиль.

### <a name="what-is-dependency-visualization"></a>Что такое визуализация зависимостей?

Визуализация зависимостей позволяет оценить группы виртуальных машин для миграции с большей уверенностью. Он перекрестной проверки зависимостей компьютера перед выполнением оценки. Визуализация зависимостей позволяет убедиться, что гарантируя и избежать непредвиденных сбоев, при миграции в Azure. Служба "Миграция Azure" использует решение схемы услуги в журналах Azure Monitor, чтобы включить визуализацию зависимостей.

> [!NOTE]
> Функция визуализации зависимостей недоступна в Azure для государственных организаций.

### <a name="do-i-need-to-pay-to-use-the-dependency-visualization-feature"></a>Нужно ли платить за использование функции визуализации зависимостей?

Нет. [Дополнительные сведения](https://azure.microsoft.com/pricing/details/azure-migrate/) о ценах на службу "Миграция Azure".

### <a name="do-i-need-to-install-anything-for-dependency-visualization"></a>Нужно ли устанавливать дополнительные программы для визуализации зависимостей?

Чтобы использовать визуализацию зависимостей, необходимо скачать и установить агенты на каждом локальном компьютере, который нужно оценить.

- Агент [Microsoft Monitoring Agent (MMA)](https://docs.microsoft.com/azure/log-analytics/log-analytics-agent-windows) должен быть установлен на каждом компьютере.
- Программа [Dependency Agent](https://docs.microsoft.com/azure/monitoring/monitoring-service-map-configure) должна быть установлена на каждом компьютере.
- Кроме того, если у вас есть компьютеры без подключения к Интернету, необходимо скачать и установить на них шлюз Log Analytics.

Эти агенты не требуется, если вы используете визуализацию зависимостей.

### <a name="can-i-use-an-existing-workspace-for-dependency-visualization"></a>Можно ли использовать существующую рабочую область для визуализации зависимостей?

Да, можно присоединить существующую рабочую область в проект миграции и использовать его для визуализации зависимостей. [Узнайте больше](concepts-dependency-visualization.md#how-does-it-work).

### <a name="can-i-export-the-dependency-visualization-report"></a>Можно ли экспортировать отчет о визуализации зависимостей?

Нет, не может быть экспортирован визуализации зависимостей. Но служба "Миграция Azure" применяет для визуализации зависимостей сопоставление служб, и вы можете использовать [интерфейсы REST API сопоставления служб](https://docs.microsoft.com/rest/api/servicemap/machines/listconnections), чтобы получить зависимости в формате JSON.

### <a name="how-can-i-automate-the-installation-of-microsoft-monitoring-agent-mma-and-dependency-agent"></a>Как автоматизировать установку агентов Microsoft Monitoring Agent (MMA) и Dependency Agent?

[Используйте этот скрипт](https://docs.microsoft.com/azure/monitoring/monitoring-service-map-configure#installation-script-examples) для установки агентов. [Следуйте указаниям, приведенным](https://docs.microsoft.com/azure/azure-monitor/platform/log-analytics-agent#install-and-configure-agent) для установки MMA, с помощью командной строки или службы автоматизации. MMA, использовать [этот сценарий](https://gallery.technet.microsoft.com/scriptcenter/Install-OMS-Agent-with-2c9c99ab).

Помимо сценариев, можно использовать средства развертывания как System Center Configuration Manager, [Intigua](https://www.intigua.com/getting-started-intigua-for-azure-migration) и т.д., для развертывания агентов.

### <a name="can-i-visualize-dependencies-in-azure-migrate-for-more-than-an-hour"></a>Можно визуализировать зависимости в "Миграция Azure" более одного часа?
Нет, вы можете визуализировать зависимости для до одного часа. Вы можете вернуться к определенной даты в журнале, вплоть до последнего месяца, но Максимальная длительность для визуализации равен одному часу. Например промежуток времени в сопоставлении зависимостей можно использовать для просмотра зависимостей для вчера, но могут только просматривать его для окна один час. Тем не менее, вы можете использовать журналы Azure Monitor для [запроса данных зависимостей](https://docs.microsoft.com/azure/migrate/how-to-create-group-machine-dependencies) за более длительный срок.

### <a name="is-dependency-visualization-supported-for-groups-with-more-than-ten-vms"></a>Поддерживается ли визуализация зависимостей для групп с более чем десяти виртуальных машин?
Вы можете [визуализация зависимостей для групп](https://docs.microsoft.com/azure/migrate/how-to-create-group-dependencies) до десяти виртуальных машин. Если у вас есть группа с более чем десяти виртуальных машин, мы рекомендуем разбить группу в для небольших групп и последующей визуализации зависимостей.
