### <a name="how-is-azure-migrate-different-from-azure-site-recovery"></a>Чем служба "Миграция Azure" отличается от Azure Site Recovery?

Служба "миграция Azure" предоставляет централизованный центр для запуска, выполнения и мониторинга обнаружения, оценки и миграции компьютеров и рабочих нагрузок в Azure. [Azure Site Recovery](https://docs.microsoft.com/azure/site-recovery/migrate-tutorial-on-premises-azure) является решением для аварийного восстановления. Миграция сервера в Azure миграция использует Azure Site Recovery в серверной части, чтобы включить сценарии миграции для миграции локальных компьютеров с помощью переноса с помощью перетаскивания.

Дополнительные компоненты (например, антивирусные программы) можно добавить в шаблон, если правила взаимодействия и брандмауэра, необходимые для устройства "миграция Azure", остались недоступными.   

### <a name="what-data-is-collected-by-azure-migrate-appliance"></a>Какие данные собираются устройством "миграция Azure"?

Сведения о данных, собираемых устройством "миграция Azure", можно получить [здесь](https://docs.microsoft.com/azure/migrate/migrate-appliance#collected-performance-data-vmware).

### <a name="is-there-any-performance-impact-on-the-analyzed-vmware-or-hyper-v-environment"></a>Влияет ли производительность на проанализированную среду VMware или Hyper-V?

С непрерывным профилированием данных о производительности Служба "миграция Azure" проводит локальные компьютеры для измерения данных о производительности виртуальной машины. Это почти не влияет на производительность узлов Hyper-V/ESXi, а также на vCenter Server.

### <a name="where-is-the-collected-data-stored-and-for-how-long"></a>Где хранятся собранные данные и как долго?

Данные, собранные устройством "миграция Azure", хранятся в расположении Azure, которое вы указали при создании проекта миграции. Данные безопасно хранятся в подписке Майкрософт и удаляются при удалении проекта службы "миграция Azure".

Для визуализации зависимостей при установке агентов на виртуальных машинах данные, собранные агентами зависимостей, сохраняются в США в рабочей области Log Analytics, созданной в подписке Azure. При удалении рабочей области Log Analytics из подписки эти данные удаляются. [Узнайте больше](concepts-dependency-visualization.md).

### <a name="what-is-the-volume-of-data-uploaded-by-azure-migrate-appliance-during-continuous-profiling"></a>Каков объем данных, передаваемых устройством "миграция Azure" во время непрерывного профилирования?

Объем данных, отправляемых в службу "миграция Azure", зависит от нескольких параметров. Чтобы предоставить индикативное число, проект службы "миграция Azure" с 10 компьютерами (каждый с одним диском и одной сетевой картой) отправляет около 50 МБ в день. Это приблизительное значение, которое изменяется в зависимости от количества точек данных для сетевых карт и дисков (отправляемые данные не являются линейными, если количество компьютеров, сетевых карт или дисков увеличится).

### <a name="is-the-data-encrypted-at-rest-and-in-transit"></a>Зашифрованы ли данные в незащищенном и транзитном месте?

Да для обоих. Метаданные безопасно отправляются в службу "миграция Azure" через Интернет по протоколу HTTPS. Метаданные хранятся в [Cosmos DB](https://docs.microsoft.com/azure/cosmos-db/database-encryption-at-rest), в [хранилище BLOB-объектов Azure](https://docs.microsoft.com/azure/storage/common/storage-service-encryption) в подписке Майкрософт и шифруются неактивных данных.

Данные, собранные агентами зависимостей, также шифруются при передаче (Secure HTTPS) и хранятся в Log Analytics рабочей области в подписке пользователя. Он также шифруется неактивных.

### <a name="how-does-the-azure-migrate-appliance-communicate-with-the-vcenter-server-and-the-azure-migrate-service"></a>Как устройство миграции Azure взаимодействует с vCenter Server и службой "миграция Azure"?

Устройство подключается к vCenter Server (порт 443), используя учетные данные, указанные при настройке устройства. Он запрашивает vCenter Server с помощью VMware PowerCLI, чтобы получить метаданные о виртуальных машинах, управляемых vCenter Server. Он собирает данные конфигурации о виртуальных машинах (ядрах, памяти, дисках, сетевой карте и т. д.), а также историю производительности каждой виртуальной машины за прошлый месяц. Затем собранные метаданные отправляются в оценку сервера Azure Migrate (через Интернет через HTTPS) для оценки.

### <a name="can-i-connect-the-same-appliance-to-multiple-vcenter-servers"></a>Можно ли подключить одно устройство к нескольким серверам vCenter?

Да, можно использовать одно устройство миграции Azure для обнаружения нескольких серверов vCenter, но не параллельно. Необходимо выполнить обнаружение один за другим.

### <a name="i-changed-my-machine-size-can-i-rerun-an-assessment"></a>Я изменил размер виртуальной машины. Можно ли повторно запустить оценку?

Устройство "миграция Azure" непрерывно собирает сведения о локальной среде. Однако оценка — это моментальный снимок локальных виртуальных машин на момент времени. При изменении параметров на виртуальной машине, которую необходимо оценить, используйте параметр "повторное вычисление", чтобы обновить оценку с учетом последних изменений.

### <a name="how-can-i-discover-a-multi-tenant-environment-in-azure-migrate-server-assessment"></a>Как найти среду с несколькими клиентами в разделе "Миграция серверов Azure"?

Для VMware, если у вас есть среда, которая является общей для клиентов и вы не хотите обнаруживать виртуальные машины одного клиента в подписке другого клиента, создайте vCenter Server учетные данные с доступом только к тем виртуальным машинам, которые необходимо обнаружить. Затем используйте учетные данные при запуске обнаружения в устройстве "миграция Azure".

Для Hyper-V обнаружение использует учетные данные узла Hyper-V. Если виртуальные машины совместно используют один и тот же узел Hyper-V, в настоящее время невозможно разделить обнаружение.  

### <a name="how-many-vms-can-be-discovered-using-a-single-migration-appliance"></a>Сколько виртуальных машин можно обнаружить с помощью одного устройства миграции?

Вы можете обнаружить до 10 000 виртуальных машин VMware и до 5 000 виртуальных машин Hyper-V с помощью одного устройства миграции.  Если в локальной среде имеется больше компьютеров, Узнайте, как масштабировать оценку [Hyper-V](scale-hyper-v-assessment.md) и [VMware](scale-vmware-assessment.md) .

## <a name="azure-migrate-server-assessment"></a>Миграция Azure Оценка сервера

### <a name="does-azure-migrate-server-assessment-support-assessment-of-physical-servers"></a>Выполняет миграцию Azure: Поддерживает ли оценка серверов оценку физических серверов?

Нет, сейчас служба "миграция Azure" не поддерживает оценку физических серверов.

### <a name="does-azure-migrate-need-vcenter-server-to-discover-a-vmware-environment"></a>Требуется ли службе "Миграция Azure" vCenter Server для обнаружения среды VMware?

Да, службе "миграция Azure" требуются vCenter Server для обнаружения среды VMware. Обнаружение узлов ESXi, не управляемых с помощью vCenter Server, не поддерживается.

### <a name="whats-the-difference-between-using-azure-migrate-server-assessment-and-the-map-toolkit"></a>В чем разница между использованием службы "миграция Azure": Оценка серверов и набор средств MAP Toolkit?

Миграция Azure Оценка сервера обеспечивает оценку миграции для обеспечения готовности к миграции и оценки рабочих нагрузок для миграции в Azure. [Набор средств оценки и планирования Microsoft (Map)](https://www.microsoft.com/download/details.aspx?id=7826) включает другие функции, такие как планирование миграции для новых версий клиентских и серверных операционных систем Windows, а также отслеживание использования программного обеспечения. Для всех этих сценариев нужно и дальше использовать MAP Toolkit.

### <a name="how-is-azure-migrate-server-assessment-different-from-azure-site-recovery-deployment-planner"></a>Как работает служба "миграция Azure": Оценка сервера отличается от Планировщик развертывания Azure Site Recovery?

Миграция Azure Оценка сервера — это средство планирования миграции. Планировщик развертывания Azure Site Recovery — это средство планирования аварийного восстановления.

- **Миграция из VMware или Hyper-V в Azure**: Если вы планируете перенести локальные серверы в Azure, используйте службу "миграция Azure": Средство оценки серверов для планирования миграции. Средство оценивает локальные рабочие нагрузки и предоставляет рекомендации, аналитические сведения и механизмы, помогающие выполнить миграцию в Azure. Когда вы будете готовы к работе с планом миграции, вы можете использовать такие средства, как миграция Azure: Миграция сервера для миграции компьютеров в Azure.
- **Аварийное восстановление из VMware/Hyper-V в Azure**: Для аварийного восстановления в Azure с помощью Site Recovery Используйте Планировщик развертывания Site Recovery для планирования аварийного восстановления. Site Recovery Планировщик развертывания выполняет глубокую Site Recoveryную оценку локальной среды. Он предоставляет рекомендации, необходимые Site Recovery для успешных аварийных операций, таких как репликация и отработка отказа виртуальных машин.

### <a name="whats-the-difference-between-as-on-premises-sizing-and-performance-based-sizing"></a>В чем разница между локальным изменением размера и размером на основе производительности?

- При изменении размера в локальной среде служба "миграция Azure" не учитывает данные о производительности виртуальной машины. Он изменяет размер виртуальных машин на основе локальной конфигурации. — При определении размера на основе производительности размер зависит от данных об использовании.
- Например, если локальная виртуальная машина содержит 4 ядра и 8 ГБ памяти с 50% использования ЦП и 50% использования памяти, то как локальное изменение размера рекомендует использовать номер SKU виртуальной машины Azure с четырьмя ядрами и 8 ГБ памяти. Однако при определении размера на основе производительности рекомендуется использовать номер SKU виртуальной машины двух ядер и 4 ГБ, поскольку считается процент использования.
- Точно так же размер диска зависит от двух свойств оценки — критерия изменения размера и типа хранилища.
= Если критерий изменения размера имеет значение "автоматически", значения операций ввода-вывода и пропускной способности диска учитываются при определении типа целевого диска ("Стандартный" или "Премиум").
- Если критерии изменения размера основаны на производительности, а тип хранилища — Premium, рекомендуется использовать диск класса Premium. Номер SKU диска Premium выбирается в зависимости от размера локального диска. Одна и та же логика используется для изменения размера диска, если критерий изменения размера — локальное, а тип хранилища — Standard или Premium.

### <a name="what-impact-does-performance-history-and-percentile-utilization-have-on-the-size-recommendations"></a>Как данные журнала производительности и процент использования влияют на рекомендации по размеру?

Эти свойства применимы только при выборе размера на основе производительности.

- Служба "миграция Azure" собирает данные журнала производительности локальных компьютеров и использует их для рекомендуемого размера виртуальной машины и типа диска в Azure.
- Устройство непрерывно выполняет профилирование локальной среды для сбора данных об использовании в режиме реального времени каждые 20 секунд. Модуль загружает 20-секундные образцы и создает одну точку данных каждые 15 минут. Чтобы создать одну точку данных, устройство выбирает пиковое значение из всех 20-секундных образцов и отправляет его в Azure.
- При создании оценки в Azure (на основе значения длительности производительности и производительности процентиля) служба "миграция Azure" вычисляет значение эффективного использования и использует его для изменения размера.
- Например, если задать для длительности производительности один день, а значение процентиля — 95 процентиль, то служба "миграция Azure" использует точки выборки 15 минут, отправленные сборщиком за последний день, сортирует их в возрастающем порядке и выбирает значение 95 процентиль процентиль в качестве значения свойства эффективное использование.
- Значение 95 процентиль процентиль позволяет пропускать выбросы, которые могут возникнуть при использовании 99-м процентиль. Если вам нужно выбрать пиковое использование для периода и вы не хотите пропустить все выбросы, следует выбрать 99-й процентиль.

### <a name="what-is-dependency-visualization"></a>Что такое визуализация зависимостей?

Визуализация зависимостей позволяет оценивать группы виртуальных машин для миграции с большей достоверностью. Перекрестная проверка зависимостей компьютера перед выполнением оценки. Визуализация зависимостей позволяет убедиться, что ничего не осталось, и избежать непредвиденных простоев при миграции в Azure. Служба "миграция Azure" использует Сопоставление служб решение в журналах Azure Monitor, чтобы включить визуализацию зависимостей.

> [!NOTE]
> Функция визуализации зависимостей недоступна в Azure для государственных организаций.

### <a name="do-i-need-to-pay-to-use-the-dependency-visualization-feature"></a>Нужно ли платить за использование функции визуализации зависимостей?

Нет. [Дополнительные сведения](https://azure.microsoft.com/pricing/details/azure-migrate/) о ценах на службу "Миграция Azure".

### <a name="do-i-need-to-install-anything-for-dependency-visualization"></a>Нужно ли устанавливать дополнительные программы для визуализации зависимостей?

Чтобы использовать визуализацию зависимостей, необходимо скачать и установить агенты на каждом локальном компьютере, который нужно оценить.

- Агент [Microsoft Monitoring Agent (MMA)](https://docs.microsoft.com/azure/log-analytics/log-analytics-agent-windows) должен быть установлен на каждом компьютере.
- Программа [Dependency Agent](https://docs.microsoft.com/azure/monitoring/monitoring-service-map-configure) должна быть установлена на каждом компьютере.
- Кроме того, если у вас есть компьютеры без подключения к Интернету, необходимо скачать и установить на них шлюз Log Analytics.

Эти агенты не нужны, если не используется Визуализация зависимостей.

### <a name="can-i-use-an-existing-workspace-for-dependency-visualization"></a>Можно ли использовать существующую рабочую область для визуализации зависимостей?

Да, можно подключить существующую рабочую область к проекту миграции и использовать ее для визуализации зависимостей. [Узнайте больше](concepts-dependency-visualization.md#how-does-it-work).

### <a name="can-i-export-the-dependency-visualization-report"></a>Можно ли экспортировать отчет о визуализации зависимостей?

Нет, не удается экспортировать визуализацию зависимостей. Но служба "Миграция Azure" применяет для визуализации зависимостей сопоставление служб, и вы можете использовать [интерфейсы REST API сопоставления служб](https://docs.microsoft.com/rest/api/servicemap/machines/listconnections), чтобы получить зависимости в формате JSON.

### <a name="how-can-i-automate-the-installation-of-microsoft-monitoring-agent-mma-and-dependency-agent"></a>Как автоматизировать установку агентов Microsoft Monitoring Agent (MMA) и Dependency Agent?

[Используйте этот скрипт](https://docs.microsoft.com/azure/monitoring/monitoring-service-map-configure#installation-script-examples) для установки агентов. [Выполните эти инструкции](https://docs.microsoft.com/azure/azure-monitor/platform/log-analytics-agent#install-and-configure-agent) , чтобы установить MMA с помощью командной строки или службы автоматизации. Для MMA используйте [этот сценарий](https://gallery.technet.microsoft.com/scriptcenter/Install-OMS-Agent-with-2c9c99ab).

В дополнение к скриптам можно использовать средства развертывания, такие как System Center Configuration Manager, [Intigua](https://www.intigua.com/getting-started-intigua-for-azure-migration) и т. д., чтобы развернуть агенты.

### <a name="can-i-visualize-dependencies-in-azure-migrate-for-more-than-an-hour"></a>Можно ли визуализировать зависимости в Azure в течение нескольких часов?
Нет, можно визуализировать зависимости до часа. Можно вернуться к определенной дате в журнале, вплоть до последнего месяца, но максимальная продолжительность визуализации — это час. Например, период времени на карте зависимостей можно использовать для просмотра зависимостей за вчерашний период, но только для просмотра в течение одного часового окна. Однако можно использовать журналы Azure Monitor для [запроса данных зависимости](https://docs.microsoft.com/azure/migrate/how-to-create-group-machine-dependencies) в течение более длительного времени.

### <a name="is-dependency-visualization-supported-for-groups-with-more-than-10-vms"></a>Поддерживается ли визуализация зависимостей для групп, в которых более 10 виртуальных машин?
Можно [визуализировать зависимости для групп](https://docs.microsoft.com/azure/migrate/how-to-create-group-dependencies) , имеющих до 10 виртуальных машин. Если у вас есть группа с более чем 10 виртуальными машинами, рекомендуется разделить группу на группы меньшего размера, а затем визуализировать зависимости.

## <a name="azure-migrate-server-migration"></a>Миграция Azure Миграция сервера

### <a name="how-is-azure-migrate-server-migration-different-from-azure-site-recovery"></a>Как работает служба "миграция Azure": Серверная миграция отличается от Azure Site Recovery?

Миграция Azure Миграция сервера использует модуль репликации Site Recovery для миграции виртуальных машин VMware на основе агентов, переноса виртуальных машин Hyper-V и миграции физических серверов в Azure. Функция без агента для переноса виртуальных машин VMware изначально встроена в серверную миграцию.
