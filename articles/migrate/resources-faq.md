# <a name="azure-migrate-frequently-asked-questions-faq"></a>Миграция Azure Часто задаваемые вопросы

В этой статье содержатся ответы на часто задаваемые вопросы о службе "миграция Azure". При наличии дальнейших запросов после прочтения этой статьи опубликуйте их на [форуме по миграции Azure](https://aka.ms/AzureMigrateForum).

## <a name="general"></a>Общее

### <a name="which-azure-geographies-does-azure-migrate-support"></a>Какие географические диаграммы Azure поддерживают миграцию Azure?

См. [список для VMware](https://docs.microsoft.com/azure/migrate/migrate-support-matrix-vmware#azure-migrate-projects) и [список для Hyper-V](https://docs.microsoft.com/azure/migrate/migrate-support-matrix-hyper-v#azure-migrate-projects).

### <a name="whats-the-difference-between-azure-migrate-and-azure-site-recovery"></a>В чем разница между Azure Migrate и Azure Site Recovery?

Служба "миграция Azure" предоставляет централизованный центр для начала миграции, выполнения и мониторинга обнаружения и оценки компьютеров и рабочих нагрузок, а также выполнения и мониторинга миграции компьютеров и рабочих нагрузок в Azure. [Azure Site Recovery](https://docs.microsoft.com/azure/site-recovery/migrate-tutorial-on-premises-azure) является решением для аварийного восстановления. Миграция сервера в Azure миграция использует Azure Site Recovery на серверной части, чтобы включить сценарии миграции для миграции на локальные компьютеры с помощью операций переноса с миграцией и сдвигом.

## <a name="azure-migrate-appliance-vmwarephysical-servers"></a>Устройство для миграции Azure (VMware или физические серверы)

### <a name="how-does-the-azure-migrate-appliance-connect-to-azure"></a>Как устройство, выполняющее миграцию Azure, подключается к Azure?

Подключение может быть через Интернет, или Azure ExpressRoute можно использовать с общедоступным или пирингом Майкрософт.

### <a name="what-are-the-network-connectivity-requirements-for-azure-migrate-server-assessment-and-migration"></a>Каковы требования к сетевому подключению для оценки и миграции сервера в Azure?

Сведения о URL-адресах и портах, необходимых для обеспечения взаимодействия Azure с Azure, см. в статье матрицы поддержки [VMware](migrate-support-matrix-vmware.md) и [Hyper-V](migrate-support-matrix-hyper-v.md) .

### <a name="can-i-harden-the-appliance-vm-that-i-set-up-with-the-template"></a>Можно ли защитить виртуальную машину устройства, настроенную с помощью шаблона?

Вы можете добавить компоненты (например, антивирусную программу) в шаблон, если вы оставляете правила взаимодействия и брандмауэра, необходимые для устройства "миграция Azure", как есть.

### <a name="what-data-is-collected-by-the-azure-migrate-appliance"></a>Какие данные собираются с помощью устройства "миграция Azure"?

Сведения о данных, собираемых устройством "миграция Azure", можно просмотреть [в документации по службе "миграция Azure"](https://docs.microsoft.com/azure/migrate/migrate-appliance#collected-performance-data-vmware).

### <a name="is-there-any-performance-impact-on-the-analyzed-vmware-or-hyper-v-environment"></a>Влияет ли производительность на проанализированную среду VMware или Hyper-V?

Служба "миграция Azure" позволяет непрерывно выполнять измерение данных о производительности виртуальной машины для локальных компьютеров. Это профилирование почти не влияет на производительность узлов Hyper-V/ESXi или vCenter Server.

### <a name="where-is-the-collected-data-stored-and-for-how-long"></a>Где хранятся собранные данные и сколько времени?

Данные, собранные устройством "миграция Azure", хранятся в расположении Azure, выбранном при создании проекта миграции. Данные безопасно хранятся в подписке Майкрософт и удаляются при удалении проекта службы "миграция Azure".

Для визуализации зависимостей при установке агентов на виртуальных машинах данные, собранные агентами зависимостей, сохраняются в США в рабочей области Log Analytics, созданной в подписке Azure. При удалении рабочей области Log Analytics из подписки эти данные удаляются. Дополнительные сведения см. в разделе [Визуализация зависимостей](concepts-dependency-visualization.md).

### <a name="what-volume-of-data-is-uploaded-by-the-azure-migrate-appliance-during-continuous-profiling"></a>Какой объем данных отправляется устройством миграции Azure во время непрерывного профилирования?

Объем данных, отправляемых в службу "миграция Azure", зависит от нескольких параметров. Чтобы придать смысл тому: проект службы "миграция Azure" с 10 компьютерами (каждый с одним диском и одной сетевой картой) отправляет около 50 МБ в день. Это значение является приближением. Он изменяется в зависимости от количества точек данных для сетевых карт и дисков. (Увеличение отправляемых данных происходит нелинейно, если число компьютеров, сетевых карт или дисков увеличивается.)

### <a name="is-the-data-encrypted-at-rest-and-in-transit"></a>Зашифрованы ли данные при хранении и передаче?

Да, и то и другое. Метаданные безопасно отправляются в службу "миграция Azure" через Интернет по протоколу HTTPS. Метаданные хранятся в базе данных [Azure Cosmos DB](https://docs.microsoft.com/azure/cosmos-db/database-encryption-at-rest) и в [хранилище BLOB-объектов Azure](https://docs.microsoft.com/azure/storage/common/storage-service-encryption) в подписке Майкрософт. Метаданные шифруются неактивных данными.

Данные, собранные агентами зависимостей, также шифруются при передаче (Secure HTTPS). Он хранится в Log Analytics рабочей области в подписке. Он также шифруется при хранении.

### <a name="how-does-the-azure-migrate-appliance-communicate-with-vcenter-server-and-the-azure-migrate-service"></a>Как устройство миграции Azure взаимодействует с vCenter Server и службой "миграция Azure"?

Устройство подключается к vCenter Server (порт 443), используя учетные данные, указанные при настройке устройства. Он использует VMware PowerCLI для запроса vCenter Server, чтобы получить метаданные о виртуальных машинах, управляемых vCenter Server. Он собирает данные конфигурации о виртуальных машинах (ядрах, памяти, дисках, сетевых адаптерах и т. д.), а также историю производительности каждой виртуальной машины за прошлый месяц. Собранные метаданные затем отправляются в Azure для оценки серверов (через Интернет через HTTPS).

### <a name="can-i-connect-the-same-appliance-to-multiple-vcenter-server-instances"></a>Можно ли подключить одно устройство к нескольким экземплярам vCenter Server?

Нет. Между устройством и vCenter Serverом существует однозначное сопоставление. Если необходимо обнаружить виртуальные машины на нескольких экземплярах vCenter Server, необходимо развернуть несколько устройств.


### <a name="i-changed-my-machine-size-can-i-run-the-assessment-again"></a>Я изменил размер виртуальной машины. Можно ли запустить оценку еще раз?

Устройство "миграция Azure" непрерывно собирает сведения о локальной среде. Но оценка — это моментальный снимок локальных виртуальных машин на момент времени. При изменении параметров на виртуальной машине, которую необходимо оценить, используйте параметр пересчитать, чтобы обновить оценку с учетом последних изменений.

### <a name="how-can-i-perform-discovery-in-a-multitenant-environment-in-azure-migrate-server-assessment"></a>Как выполнить обнаружение в многоклиентской среде при оценке сервера "миграция Azure"?

Для VMware, если у вас есть среда, совместно используемая несколькими клиентами, и вы не хотите обнаруживать виртуальные машины одного клиента в подписке другого клиента, создайте vCenter Server учетные данные, которые могут получить доступ только к виртуальным машинам, которые нужно обнаружить. Затем используйте эти учетные данные при запуске обнаружения в устройстве "миграция Azure".

Для Hyper-V при обнаружении используются учетные данные узла Hyper-V. Если виртуальные машины используют один и тот же узел Hyper-V, в настоящее время невозможно разделить обнаружение.  

### <a name="how-many-vms-can-i-discover-with-a-single-migration-appliance"></a>Сколько виртуальных машин можно обнаружить с помощью одного устройства миграции?

Вы можете обнаружить до 10 000 виртуальных машин VMware и до 5 000 виртуальных машин Hyper-V с одним устройством миграции. Если в локальной среде имеется больше компьютеров, Узнайте, как масштабировать оценку [Hyper-V](scale-hyper-v-assessment.md) и [VMware](scale-vmware-assessment.md) .

## <a name="azure-migrate-server-assessment"></a>Оценка сервера службы "Миграция Azure"

### <a name="does-azure-migrate-server-assessment-support-assessment-of-physical-servers"></a>Поддерживает ли служба оценки серверов Azure Server оценку физических серверов?

Нет, служба "миграция Azure" сейчас не поддерживает оценку физических серверов.

### <a name="does-azure-migrate-need-vcenter-server-to-perform-discovery-in-a-vmware-environment"></a>Требуется ли службе "миграция Azure" vCenter Server выполнить обнаружение в среде VMware?

Да, служба "миграция Azure" требует vCenter Server для выполнения обнаружения в среде VMware. Обнаружение узлов ESXi, не управляемых с помощью vCenter Server, не поддерживается.

### <a name="whats-the-difference-between-azure-migrate-server-assessment-and-the-map-toolkit"></a>В чем разница между оценкой сервера "миграция Azure" и набором средств MAP Toolkit?

Оценка сервера "миграция Azure" обеспечивает оценку миграции для обеспечения готовности к миграции и оценки рабочих нагрузок для миграции в Azure. [Набор средств оценки и планирования Майкрософт (Map)](https://www.microsoft.com/download/details.aspx?id=7826) помогает с другими задачами, такими как планирование миграции для новых версий клиентских и серверных операционных систем Windows и отслеживание использования программного обеспечения. Для всех этих сценариев нужно и дальше использовать MAP Toolkit.

### <a name="whats-the-difference-between-azure-migrate-server-assessment-and-the-azure-site-recovery-deployment-planner"></a>В чем разница между оценкой сервера "миграция Azure" и Планировщик развертывания Azure Site Recovery?

Оценка серверов в службе "миграция Azure" — это средство планирования миграции. Планировщик развертывания Azure Site Recovery — это средство планирования аварийного восстановления.

**Миграция из VMware или Hyper-V в Azure**: Если вы планируете перенести локальные серверы в Azure, используйте оценку сервера Azure Migration для планирования миграции. Он оценивает локальные рабочие нагрузки и предоставляет рекомендации, аналитические сведения и средства, помогающие перенести серверы в Azure. После того как вы будете готовы к работе с планом миграции, вы можете перенести компьютеры в Azure с помощью таких средств, как миграция миграции сервера Azure.

**Аварийное восстановление из VMware или Hyper-V в Azure**: Для аварийного восстановления в Azure с помощью Site Recovery Используйте Планировщик развертывания Site Recovery для планирования. Site Recovery Планировщик развертывания выполняет глубокую Site Recoveryную оценку локальной среды. Он предоставляет рекомендации, необходимые Site Recovery для успешных аварийных операций, таких как репликация и отработка отказа виртуальных машин.

### <a name="does-azure-migrate-support-cost-estimation-for-the-enterprise-agreement-ea-program"></a>Переносит ли Azure оценку стоимости поддержки для программы Соглашение Enterprise (EA)?

### <a name="whats-the-difference-between-as-on-premises-sizing-and-performance-based-sizing"></a>В чем разница между локальным изменением размера и размером на основе производительности?

При изменении размера в локальной среде служба "миграция Azure" не учитывает данные о производительности виртуальной машины. Он изменяет размер виртуальных машин на основе локальной конфигурации. Изменение размера на основе производительности зависит от данных об использовании.

Например, рассмотрим локальную виртуальную машину с 4 ядрами и 8 ГБ памяти на 50% использования ЦП и 50% использования памяти. Для этой виртуальной машины в качестве локального размера рекомендуется использовать номер SKU виртуальной машины Azure с четырьмя ядрами и 8 ГБ памяти. При определении размера на основе производительности рекомендуется использовать номер SKU виртуальной машины с двумя ядрами и 4 ГБ памяти, поскольку считается процент использования.

Точно так же размер диска зависит от двух свойств оценки: критериев изменения размера и типа хранилища. Если критерии изменения размера основаны на производительности и тип хранения — "автоматически", служба "миграция Azure" учитывает значения операций ввода-вывода в секунду и пропускную способность диска, если он определяет тип целевого диска (Standard или Premium).

Если критерии изменения размера основаны на производительности, а тип хранилища — Premium, Azure Migrate рекомендует диск уровня "Премиум". Номер SKU диска Premium выбирается в зависимости от размера локального диска. Эта же логика используется для изменения размера диска, если критерий изменения размера имеет значение как локальное, так и тип хранилища "Стандартный" или "Премиум".

### <a name="what-impact-does-performance-history-and-utilization-percentile-have-on-size-recommendations"></a>Какое влияние на размер журнала производительности и степени использования зависит от рекомендаций по размеру?

Эти свойства применимы только при выборе размера на основе производительности.

Служба "миграция Azure" собирает данные журнала производительности локальных компьютеров и использует их для рекомендуемого размера виртуальной машины и типа диска в Azure.

Устройство непрерывно выполняет профилирование локальной среды для сбора данных об использовании в режиме реального времени каждые 20 секунд. Устройство объединяет 20-секундные выборки и создает одну точку данных каждые 15 минут. Чтобы создать точку данных, устройство выбирает пиковое значение из всех 20-секундных выборок. Устройство отправляет эту точку данных в Azure.

При создании оценки в Azure (в зависимости от значения длительности производительности и производительности процентиля) служба "миграция Azure" вычисляет эффективное значение использования и использует его для определения размера.

Например, предположим, что длительность производительности равна одному дню, а значение процентиля — 95 процентиль. Служба "миграция Azure" сортирует 15-минутные точки выборки, отправленные сборщиком за последний день в возрастающем порядке, и выбирает значение 95 процентиль процентиль в качестве эффективного использования.

Значение 95 процентиль процентиль позволяет пропускать выбросы. При использовании 99-м процентиль можно включать выбросы. Если вы хотите выбрать пиковое использование для периода и не хотите пропускать выбросы, выберите 99-м процентиль.

### <a name="what-is-dependency-visualization"></a>Что такое визуализация зависимостей?

Визуализация зависимостей позволяет оценивать группы виртуальных машин для миграции с большей достоверностью. Перекрестная проверка зависимостей компьютера перед выполнением оценки. Визуализация зависимостей помогает убедиться, что при миграции в Azure ничего не осталось, чтобы избежать непредвиденных простоев. Служба "миграция Azure" использует решение Сопоставление служб в журналах Azure Monitor, чтобы включить визуализацию зависимостей.

> [!NOTE]
> Визуализация зависимостей недоступна в Azure для государственных организаций.

### <a name="do-i-need-to-pay-to-use-dependency-visualization"></a>Нужно ли платить за использование визуализации зависимостей?

Нет. Дополнительные сведения см. в статье [цены на миграцию Azure](https://azure.microsoft.com/pricing/details/azure-migrate/).

### <a name="do-i-need-to-install-anything-for-dependency-visualization"></a>Нужно ли устанавливать дополнительные программы для визуализации зависимостей?

Чтобы использовать визуализацию зависимостей, необходимо скачать и установить агенты на каждом локальном компьютере, который нужно оценить.

На каждом компьютере необходимо установить следующие агенты:
- [Microsoft Monitoring Agent (MMA)](https://docs.microsoft.com/azure/log-analytics/log-analytics-agent-windows).
- [Агент зависимостей](https://docs.microsoft.com/azure/monitoring/monitoring-service-map-configure).
- Если у вас есть компьютеры без подключения к Интернету, необходимо скачать и установить шлюз Log Analytics.

Эти агенты не нужны, если не используется Визуализация зависимостей.

### <a name="can-i-use-an-existing-workspace-for-dependency-visualization"></a>Можно ли использовать существующую рабочую область для визуализации зависимостей?

Да, можно подключить существующую рабочую область к проекту миграции и использовать ее для визуализации зависимостей. Дополнительные сведения см. в разделе «как работает» статьи визуализация [зависимостей](concepts-dependency-visualization.md#how-does-it-work) .

### <a name="can-i-export-the-dependency-visualization-report"></a>Можно ли экспортировать отчет о визуализации зависимостей?

Нет, не удается экспортировать визуализацию зависимостей. Но поскольку служба "миграция Azure" использует Сопоставление служб для визуализации зависимостей, можно использовать [REST API сопоставление служб](https://docs.microsoft.com/rest/api/servicemap/machines/listconnections) , чтобы получить зависимости в формате JSON.

### <a name="how-can-i-automate-the-installation-of-microsoft-monitoring-agent-mma-and-the-dependency-agent"></a>Как автоматизировать установку Microsoft Monitoring Agent (MMA) и агента зависимостей?

Используйте этот [скрипт для установки агентов](https://docs.microsoft.com/azure/monitoring/monitoring-service-map-configure#installation-script-examples). ### <a name="can-i-visualize-dependencies-in-azure-migrate-for-more-than-an-hour"></a>Можно ли визуализировать зависимости в Azure в течение нескольких часов?
Нет. Можно визуализировать зависимости до часа. Вы можете вернуться к определенной дате в журнале, как и до конца месяца, но максимальная продолжительность визуализации — это час. Например, период времени на карте зависимостей можно использовать для просмотра зависимостей за вчерашний период, но его можно просмотреть только в течение одного часового окна. Однако можно использовать журналы Azure Monitor для [запроса данных зависимости](https://docs.microsoft.com/azure/migrate/how-to-create-group-machine-dependencies) в течение более длительного времени.

### <a name="can-i-use-dependency-visualization-for-groups-that-contain-more-than-10-vms"></a>Можно ли использовать визуализацию зависимостей для групп, содержащих более 10 виртуальных машин?
Можно [визуализировать зависимости для групп](https://docs.microsoft.com/azure/migrate/how-to-create-group-dependencies) , содержащих до 10 виртуальных машин. Если у вас есть группа с более чем 10 виртуальными машинами, рекомендуется разбить группу на группы меньшего размера, а затем визуализировать зависимости.

## <a name="azure-migrate-server-migration"></a>Миграция сервера службы "Миграция Azure"

### <a name="whats-the-difference-between-azure-migrate-server-migration-and-azure-site-recovery"></a>В чем разница между миграцией Azure Migration Server и Azure Site Recovery?

Миграция сервера в Azure миграция использует модуль репликации Site Recovery для миграции виртуальных машин VMware на основе агентов, миграции виртуальных машин Hyper-V и миграции физических серверов в Azure. Функция без агента для переноса виртуальных машин VMware встроена в серверную миграцию.
