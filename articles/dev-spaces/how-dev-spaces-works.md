# <a name="how-azure-dev-spaces-works"></a>Принцип работы Azure Dev Spaces

Разработка приложения Kubernetes может быть сложной задачей. Вам нужны файлы конфигурации Docker и Kubernetes. Вам нужно выяснить, как протестировать приложение локально и взаимодействовать с другими зависимыми службами. Возможно, вам придется обрабатывать разработку и тестирование нескольких служб одновременно и с командой разработчиков.

Azure Dev Spaces предоставляет вам множество способов быстрой итерации и отладки приложений Kubernetes и совместной работы с вашей командой. В этой статье описывается, что могут делать пространства Azure Dev spaces и как они работают.

## <a name="rapidly-iterate-and-debug-your-kubernetes-application"></a>Быстро итерировать и отладить приложение Kubernetes

Пространства Azure Dev снижают усилия по разработке, тестированию и итерации приложения Kubernetes в контексте кластера AKS. Это сокращение усилий позволяет разработчикам сосредоточиться на бизнес-логике своих приложений, а не настраивать свои службы для работы в Kubernetes.

### <a name="connect-your-development-machine-to-aks"></a>Подключите машину разработки к AKS

С помощью Azure Dev Spaces можно подключить компьютер разработки к кластеру AKS, что позволяет запускать и отлажовать код на компьютере разработки, как если бы он работал на кластере. Пространства Azure Dev перенаправляют трафик между подключенным кластером AKS, запустив стручок на кластере, который действует как удаленный агент для перенаправления трафика между машиной разработки и кластером. Это перенаправление трафика позволяет коду на компьютере разработки и службам, работающим в кластере AKS, общаться так, как если бы они находились в одном кластере AKS. Для получения дополнительной информации о подключении машины разработки к AKS см., [как работает подключение компьютера разработки к кластеру AKS.][how-it-works-connect]

### <a name="run-your-code-in-aks"></a>Запустите код в AKS

Помимо перенаправления трафика между компьютером разработки и кластером AKS, с Azure Dev Spaces вы можете настроить и быстро запустить свой код непосредственно в AKS. С помощью Visual Studio, Visual Studio Code или Azure Dev Spaces CLI пробелы Azure Dev будут загружать ваш код в кластер, а затем создавать и запускать его. Пространства Azure Dev также могут разумно синхронизировать изменения кода и перезапустить службу, чтобы отразить изменения по мере необходимости. Во время выполнения кода журналы сборки и http-traces передаются клиенту, чтобы вы могли отслеживать прогресс и диагностировать любые проблемы. Вы также можете использовать Azure Dev Spaces, чтобы прикрепить отладчик в Visual Studio и Visual Studio Code к службам Java, Node.js и .NET Core. Для получения дополнительной информации см. [Как работает подготовка проекта для Azure Dev Spaces,][how-it-works-prep]как работает код с [Azure Dev Spaces][how-it-works-up]и как работает [удаленная отладка кода с Azure Dev Spaces.][how-it-works-remote-debugging]

## <a name="team-development"></a>Командная разработка

Пространства Azure Dev помогают группам продуктивно работать над своим приложением в том же кластере AKS, не нарушая их.

### <a name="intelligent-routing-between-dev-spaces"></a>Интеллектуальная размывание между пространствами разработчиков

С Azure Dev Spaces команда может совместно использовать один кластер AKS, работающий с облачным приложением, и создавать изолированные пространства разработчиков, где команда может разрабатывать, тестировать и отладить, не мешая другим пространствам разработчиков. Базовая версия приложения работает в пространстве корневой наклонной. Затем члены группы создают независимые пробелы для разработчиков на основе корневого пространства для разработки, тестирования и отладки изменений в приложении. С помощью возможностей маршрутизаний в Dev Spaces пространства для разработчиков могут направлять запросы между службами, работающими в пространстве девятого ребенка, и пространством родительского разработчика. Эта routing позволяет разработчикам запускать свою собственную версию службы при повторном использовании зависимых служб из родительского пространства. Каждое детское пространство имеет свой уникальный URL-адрес, который может быть общим и доступным для совместной работы другими пользователями. Для получения дополнительной информации о том, как работает routing в пространствах Azure Dev, см. [Как работает routing с пространствами Azure Dev Spaces.][how-it-works-routing]

### <a name="live-testing-an-open-pull-request"></a>Live тестирование открытого запроса притяжения

Вы также можете использовать действия GitHub с пробелами Azure Dev для тестирования изменений в приложении в запросе на вытягивание непосредственно в кластере перед слиянием. Пространства Azure Dev могут автоматически развертывать версию обзора приложения в кластер, позволяя автору, а также другим членам группы просматривать изменения в контексте всего приложения. Используя возможности реуктора пространства Azure Dev, эта версия обзора приложения также развертывается в кластере без влияния на другие пробелы разработчиков. Все эти возможности позволяют уверенно утверждать и объединять запросы на вытягивание. Ознакомиться с примерами действий GitHub и пробелов Azure Dev можно с [&м.][pr-flow]
