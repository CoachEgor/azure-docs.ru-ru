# <a name="azure-cosmos-db-as-a-key-value-store--cost-overview"></a>Azure Cosmos DB как ключевой магазин ценностей - обзор затрат

Azure Cosmos DB — это глобально распределенная многомодельная служба базы данных, предназначенная для быстрого создания высокодоступных крупномасштабных приложений. По умолчанию Azure Cosmos DB автоматически и эффективно индексирует все данные, которые он глотает. Это позволяет быстро и последовательно запрашивать данные в [быстрых](how-to-sql-query.md) [и](stored-procedures-triggers-udfs.md)последовательных запросах. 

В этой статье рассматриваются затраты на использование среды Azure Cosmos DB для простых операций записи и чтения, если она используется как хранилище пар "ключ —значение". Операции записи включают вставки, замены, удаления и upserts элементов данных. Помимо гарантии доступности SLA на 99,999% для всех многорайонных счетов, Azure Cosmos DB предлагает гарантированную <10-мс задержку для чтения и для (индексированного) пишет, на 99-м процентили. 

## <a name="why-we-use-request-units-rus"></a>Для чего используются единицы запроса (ЕЗ)

Производительность Azure Cosmos DB основана на количестве прокладки, выраженной в [единицах запроса](request-units.md) (RU/s). Подготовка находится на второй детализации и приобретается в RU/s[(не следует путать с почасовой выставлением счетов).](https://azure.microsoft.com/pricing/details/cosmos-db/) RUs следует рассматривать как логическую абстракцию (валюту), которая упрощает подготовку требуемой пропускной выливки приложения. Пользователям не нужно думать о дифференциации между пропускной стоимостью чтения и записи. Одновалютная модель ЕЗ повышает эффективность, позволяя разделять подготовленную емкость между операциями чтения и записи. Эта модель производительности позволяет сервису обеспечить **предсказуемую и последовательную пропускную способность, гарантированную низкую задержку и высокую доступность.** Наконец, в то время как модель RU используется для изображения пропускной записи, каждая подготовленная RU также имеет определенное количество ресурсов (например, память, ядра/процессор и IOPS).

Как глобально распределенная система баз данных, Cosmos DB является единственной службой Azure, которая обеспечивает комплексные SLA, охватывающие задержку, пропускную готовность, согласованность и высокую доступность. Проликция, которая предоставляется в каждом из регионов, связанных с вашей учетной записью «Космос». Для операций чтения Cosmos DB предлагает на ваш выбор несколько четко определенных [уровней согласованности](consistency-levels.md). 

В следующей таблице показано количество RUs, необходимых для выполнения операций чтения и записи на основе элемента данных размером 1 КБ и 100 КБ с отключенной автоматической индексацией по умолчанию. 

|Размер элемента|1 операция чтения|1 операция записи|
|-------------|------|-------|
|1 КБ|1 ЕЗ|5 ЕЗ|
|100 КБ|10 ЕЗ|50 ЕЗ|

## <a name="cost-of-reads-and-writes"></a>Стоимость операций чтения и записи

Если вы предоставляете 1000 руб./с, то это составляет 3,6 млн руб./час и будет стоить $0,08 за час (в США и Европе). Для элемента данных размера 1 КБ это означает, что вы можете потреблять 3,6 миллиона считывает или 0,72 миллиона пишет (3,6 миллиона RU / 5) с помощью вашей прописной пропускной записи. Нормализованная до миллиона читает и пишет, стоимость будет $ 0,022 / млн читает ($ 0,08 / 3,6) и $ 0,111 / млн пишет ($ 0,08 / 0,72). Стоимость 1 млн операций становится минимальной, как показано в следующей таблице.

|Размер элемента|Стоимость 1 миллиона считываемых|Стоимость 1 миллиона пишет|
|-------------|-------|--------|
|1 КБ|0,022 долл. США|0,111 долл. США|
|100 КБ|0,222 долл. США|1,111 долл. США|


Большинство базовых хранилищ BLOB-объектов или объектов, обойдутся в 0,40 долл. США за 1 миллион транзакций чтения и 5 долл. США за 1 миллион транзакций записи. При оптимальном использовании Cosmos DB может быть на 98% дешевле, чем эти другие решения (для 1 КБ транзакций).
