# <a name="known-issuesmigration-limitations-with-online-migrations-to-azure-sql-database-managed-instance"></a>Известные проблемы/ограничения миграции с помощью миграций в режиме онлайн-миграции в управляемый экземпляр базы данных Azure S'L

Ниже описаны известные проблемы и ограничения, связанные с миграцией в Интернет е-е-на-Ай-Сервере в управляемый экземпляр базы данных Azure S'L.

> [!IMPORTANT]
> При миграции онлайн-сервера в базу данных Azure S'L не поддерживается миграция SQL_variant типов данных.

## <a name="backup-requirements"></a>Требования к резервному копированию

- **Резервное копирование с чеком**

    Миграционная служба базы данных Azure использует метод резервного копирования и восстановления для переноса данных на непреднамеренную базу данных в управляемый экземпляр базы данных S'L. Миграционная служба базы данных Azure поддерживает только резервные данные, созданные с помощью checksum.

    [Включить или отключить проверки резервного копирования во время резервного копирования или восстановления (сервер S'L)](https://docs.microsoft.com/sql/relational-databases/backup-restore/enable-or-disable-backup-checksums-during-backup-or-restore-sql-server?view=sql-server-2017)

    > [!NOTE]
    > Если взять резервные данные с помощью сжатия, чек- и будет поведением по умолчанию, если только не будет явно отключен.

    При переносных миграциях, если вы выберете, **я позволю службе миграции базы данных Azure...**, затем служба миграции базы данных Azure возьмет резервную копию базы данных с включенной опцией checksum.

- **Резервное копирование мультимедиа**

    Убедитесь в том, чтобы взять все резервные резервные на отдельном резервном носителе (файлы резервного копирования). Миграционная служба базы данных Azure не поддерживает резервные отработки, которые прикладываются к одному файлу резервного копирования. Возьмите полную резервную копию и резервные копии журнала для разделения резервных файлов.

## <a name="data-and-log-file-layout"></a>Макет файлов данных и журналов

- **Количество файлов журналов**

    Миграционная служба базы данных Azure не поддерживает базы данных с несколькими файлами журнала. Если у вас есть несколько файлов журнала, сократите и реорганизуйте их в единый файл журнала транзакций. Поскольку вы не можете удаленно гонять для входа файлов, которые не являются пустыми, вам нужно сначала создать резервную копию файла журнала.

## <a name="sql-server-features"></a>функции SQL Server

- **FileStream/FileTables**

    В настоящее время управляемый экземпляр базы данных S'L не поддерживает FileStream и FileTables. Для рабочих нагрузок, зависящих от этих функций, мы рекомендуем выбрать серверы S'L, работающие на Вазовых вс-во, в качестве целевой цели Azure.

- **Таблицы в памяти**

    OLTP в памяти доступен в премиум-и бизнес-критических уровнях для управляемого экземпляра базы данных S'L; уровень Общего назначения не поддерживает OLTP in-memory.

## <a name="migration-resets"></a>Миграция сбрасывает

- **Развертывания**

    Управляемый экземпляр базы данных является сервисом PaaS с автоматическим исправлением и обновлениями версий. Во время миграции управляемого экземпляра базы данных S'L некритические обновления помогают до 36 часов. После этого (и для критических обновлений), если миграция нарушена, процесс сбрасывается до полного состояния восстановления.

    Миграция может быть вызвана только после восстановления полного резервного копирования и догоняет все резервные копии журнала. Если это касается миграционных переносов производства, обратитесь [в псевдоним Azure DMS Feedback.](mailto:dmsfeedback@microsoft.com)
