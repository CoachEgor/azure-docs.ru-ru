# <a name="connect-windows-security-events"></a>Подключение событий безопасности Windows 

Разъем событий безопасности позволяет передавать все события безопасности из систем Windows (серверы и рабочие станции, физические и виртуальные) в рабочее пространство Azure Sentinel. Это позволяет просматривать события безопасности Windows в панели мониторинга, использовать их при создании пользовательских оповещений и полагаться на них для улучшения исследований, давая вам более глубокое представление о сети вашей организации и расширяя операции по обеспечению безопасности Возможности. Вы можете выбрать, какие события поток из следующих наборов:<a name="event-sets"></a>

- **Все события** - все события безопасности Windows и AppLocker.
- **Общие** - стандартный набор событий для целей аудита. Полный аудиторский след пользователя включен в этот набор. Например, он содержит как события регистрации пользователя, так и события регистрации пользователя (идентионы событий 4624, 4634). Существуют также действия аудита, такие как изменения группы безопасности, ключевые операции контроллера домена Kerberos и другие типы событий в соответствии с общепринятыми передовыми практиками.

    **Общий** набор событий может содержать некоторые типы событий, которые не так распространены.  Это объясняется тем, что основная цель **общего** набора заключается в снижении объема событий до более управляемого уровня, сохраняя при этом полную возможность аудита.

- **Минимальный** - небольшой набор событий, которые могут указывать на потенциальные угрозы. Этот набор не содержит полного аудита. Он охватывает только события, которые могут указывать на успешное нарушение, и другие важные события, которые имеют очень низкие показатели возникновения. Например, он содержит успешные и неудавшиеся пользовательские логии (идентиматологи еде 4624, 4625), но не содержит информации о вывески (4634), которая, хотя и важна для аудита, не имеет смысла для обнаружения нарушений и имеет относительно большой объем. Большая часть объема данных этого набора состоит из событий ввходом и событий создания процессов (event ID 4688).

- **Нет** - Нет безопасности или AppLocker событий. (Этот параметр используется для отключения разъема.)

> [!NOTE]
> Сбор событий безопасности в контексте единого рабочего пространства может быть настроен либо из Центра безопасности Azure или Azure Sentinel, но не из обоих. Если вы работаете на Azure Sentinel в рабочей области, которая уже работает в Центре безопасности Azure, и настроена на сбор событий безопасности, у вас есть два варианта:
> - Оставьте коллекцию событий безопасности в Центре безопасности Azure как есть. Вы сможете задать запрос и анализ этих событий в Azure Sentinel, а также в Центре безопасности Azure. Однако вы не сможете контролировать состояние соединения или изменять его конфигурацию в Azure Sentinel. Если это важно для вас, рассмотрим второй вариант.
>
> - [Отключите коллекцию событий безопасности](../security-center/security-center-enable-data-collection.md) в Центре безопасности Azure и только после этого добавьте разъем событий безопасности в Azure Sentinel. Как и в случае с первым вариантом, вы сможете заготавлить и анализировать события как в Azure Sentinel, так и в Центре безопасности Azure, но теперь вы сможете отслеживать состояние соединения или изменять его конфигурацию в - и только в - Azure Sentinel.

## <a name="set-up-the-windows-security-events-connector"></a>Настройка разъема событий безопасности Windows

Чтобы собрать события безопасности Windows в Azure Sentinel:

1. Из меню навигации Azure Sentinel выберите **разъемы данных.** Из списка разъемов нажмите на **события безопасности,** а затем на кнопку **"Открытая разъемная страница"** в правом нижнем правом. Затем следуйте инструкциям на экране под вкладкой **Инструкции,** как описано в остальной части этого раздела.

1. Убедитесь, что у вас есть соответствующие разрешения, описанные в **Prerequisites.**

1. Загрузите и установите [агент Log Analytics](../azure-monitor/platform/log-analytics-agent.md) (также известный как агент мониторинга Майкрософт или MMA) на машинах, для которых вы хотите передавать события безопасности в Azure Sentinel.

    Для виртуальных машин Azure:
    
    1. Нажмите на **установку агента на Azure Windows Виртуальная машина**, а затем по ссылке, которая появляется ниже.
    1. Для каждой виртуальной машины, которую вы хотите подключить, нажмите на его имя в списке, который появляется справа, а затем нажмите **Connect.**

    Для не-Azure Windows машины (физические, виртуальные на прем или виртуальные в другом облаке):

    1. Нажмите на **ссылку «Установка» на неazure Windows Machine,** а затем на ссылку, которая отображается ниже.
    1. Нажмите на соответствующие ссылки загрузки, которые отображаются справа, под **Windows Computers**.
    1. Используя загруженный исполняемый файл, установите агент в системы Windows по вашему выбору и назначьте его с помощью **идентификатора рабочего пространства и ключей,** которые отображаются ниже упомянутых выше ссылок на загрузку.

    > [!NOTE]
    >
    > Чтобы позволить системам Windows без необходимого подключения к Интернету по-прежнему передавать события в Azure Sentinel, загрузите и установите **OMS Gateway** на отдельной машине, используя ссылку в правом нижнем, чтобы выступать в качестве прокси-сервера.  Вам все равно нужно будет установить агент Log Analytics на каждой системе Windows, события которых вы хотите собрать.
    >
    > Для получения дополнительной информации [ **Log Analytics gateway** ](../azure-monitor/platform/gateway.md)по этому сценарию см.

    Дополнительные варианты установки и более подробную информацию можно узнать из [документации **агента Log Analytics.** ](../azure-monitor/platform/agent-windows.md)

1. Выберите, какой набор событий[(Все, Общие или Минимальные](#event-sets)) вы хотите поток.

1. Нажмите кнопку **Обновить**.

1. Чтобы использовать соответствующую схему в журнале Analytics `SecurityEvent` для событий безопасности Windows, введите окно запроса.
