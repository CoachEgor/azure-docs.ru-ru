# <a name="tutorial-detect-threats-with-azure-sentinel-preview"></a>Руководство по Обнаружение угроз с предварительной версией Sentinel Azure

> Эта предварительная версия предоставляется без соглашения об уровне обслуживания и не рекомендована для использования рабочей среде. Некоторые функции могут не поддерживаться или их возможности могут быть ограничены. Дополнительные сведения см. в статье [Дополнительные условия использования предварительных выпусков Microsoft Azure](https://azure.microsoft.com/support/legal/preview-supplemental-terms/).

После того как вы [подключенных источников данных](quickstart-onboard.md) Azure Sentinel, чтобы получать уведомления, когда происходит что-то подозрительные. Чтобы позволяют это сделать, Azure Sentinel возможность создать дополнительные правила генерации оповещений, которые создают случаи, которые можно назначить и глубоко изучить аномалии и угроз в вашей среде с помощью. 

Это руководство поможет вам обнаруживать угрозы с помощью Azure Sentinel.

> * Реагирование на угрозы

## <a name="create-detection-rules"></a>Создание правила обнаружения

Для изучения вариантов, сначала необходимо создать правила обнаружения. 

> [!NOTE]
> Предупреждения, созданные в Azure Sentinel доступны через [Microsoft Graph Security](https://aka.ms/securitygraphdocs). Ссылаться на [документации по оповещениям системы безопасности Microsoft Graph](https://aka.ms/graphsecurityreferencebetadocs) для получения дополнительных сведений и партнеры по интеграции.

Правила обнаружения основаны на типы угроз и аномалий, которые могут быть подозрительных в вашей среде, которые вы хотите узнать о прямо сейчас, обеспечивая отображена, изучить и исправить. 

1. На портале Azure в разделе Azure Sentinel, выберите **Analytics**.

   ![Аналитика](./media/tutorial-detect-threats/alert-rules.png)

2. В верхней строке меню, щелкните **+ добавить**.  

   ![Создать правило генерации оповещений](./media/tutorial-detect-threats/create-alert-rule.png)

3. В разделе **создать правило генерации оповещений**, укажите описательное имя и задайте **серьезность** при необходимости. 

4. Создать запрос в Log Analytics, а затем вставьте его в **правила генерации оповещений набора** поля. Ниже приведен пример запроса, будет оповестить вас о подозрительной активности количество ресурсов создается действия в Azure.

        AzureActivity
        | where OperationName == "Create or Update Virtual Machine" or OperationName == "Create Deployment"
        | where ActivityStatus == "Succeeded"
        | make-series dcount(ResourceId)  default=0 on EventSubmissionTimestamp in range(ago(7d), now(), 1d) by Caller

   > [!NOTE]
   > Длина запроса должна быть от 1 до 10000 символов и не может содержать «поиска *» и «union *».


5. В **сопоставления сущностей** используйте поля в разделе **тип сущности** сопоставление столбцов в запросе к полям сущности, распознаваемый Azure Sentinel. Для каждого поля сопоставьте соответствующий столбец в запрос, который вы создали в Log Analytics, чтобы поле сущности, соответствующие. Выберите имя соответствующего столбца в **свойство**. Каждая сущность включает несколько полей, например идентификатор безопасности, GUID и т. д. Вы можете сопоставить сущности в соответствии с любым из полей, а не только сущность верхнего уровня.

6. Определение условий триггера оповещения в разделе **триггера оповещения**. Определяет условия, которые вызывают оповещения. 

7. Задайте **частота** для частоту выполнения запроса — как часто каждые 5 минут или указываемому один раз в день. 

8. Задайте **период** для управления временного окна для объема данных, запрос выполняется в - например, он выполняется каждый час между данные за 60 минут.

9. Можно также задать **подавления**. Подавление полезно в тех случаях, когда вы хотите остановить повторяющиеся предупреждения активации для тот же инцидент. Таким образом вы можете остановить оповещения активации в течение определенного периода. Это может помочь избежать повторяющихся оповещений для тот же инцидент и позволяют подавлять последовательных оповещений для определенного периода времени. К примеру Если **оповещений планирования** **частота** равно 60 минутам и **оповещений планирования период** присваивается два часа, и результаты запроса превышение определенного Пороговое значение, он создает оповещение дважды, когда сначала при его обнаружении за последние 60 минут, и снова ее в первые 60 минут 2-часов используется в выборке данных. Мы рекомендуем использовать, если оповещение активируется, подавление следует период времени, в период оповещения. В нашем примере может потребоваться задать подавление 60 минут, чтобы только оповещений для событий, произошедших в течение последнего часа.

8. После вставки запроса в **правила генерации оповещений набора** , можно сразу увидеть работающую модель предупреждения в группе **оповещения моделирование логики** так, что позволяет получить общее представление о какой объем данных будет созданные за определенный период времени для оповещения, который вы создали. Это будет зависеть от того, задано для **частота** и **пороговое значение**. Если вы видите, что в среднем, оповещения будут создаваться слишком часто, необходимо задать число результаты более поздней версии, так как это выше среднего базовых показателей.

9. Нажмите кнопку **создать** инициализировать правило оповещения. После создания оповещение создается случай, содержащий предупреждение. Вы увидите правила обнаружения, определенные в виде строк **аналитика безопасности** вкладки. Также вы увидите число совпадений для каждого правила - оповещений, активируемых. Из этого списка можно включить, отключить или удалить каждое правило. Вы можете также справа выберите кнопку с многоточием (...) в конце строки для каждого оповещения изменить, клонирование, поиска совпадений, отключение или удаление правила. **Analytics** страницы — это набор всех действующих правил оповещений, включая шаблоны включен и правила генерации оповещений, создаваемых в зависимости от шаблонов.

1. Результаты правил генерации оповещений могут быть видны в **случаях** страницы, где можно рассмотреть, [исследовать случаях](tutorial-investigate-cases.md), и устранить угрозы.



## <a name="respond-to-threats"></a>Реагирование на угрозы

Azure Sentinel предоставляет два основных варианта за реагирование на угрозы с помощью наборов инструкций playbook. Можно задать playbook для автоматического запуска при активации оповещения, или вы можете вручную запустить playbook в ответ на оповещение.

- Задайте playbook для автоматического запуска при активации оповещения при настройке сборника. 

- Вручную запустить сборник тренировочных заданий из внутри предупреждения, щелкнув **просмотреть модули playbook** и выбрав playbook для запуска.
