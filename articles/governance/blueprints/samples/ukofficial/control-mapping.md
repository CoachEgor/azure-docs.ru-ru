# <a name="control-mapping-of-the-uk-official-and-uk-nhs-blueprint-samples"></a>Управление сопоставлением образцов для официальных и ОТДЕЛЕНИЯных чертежей Великобритании

В следующей статье подробно описано, как учебные примеры "Официальный отделения" и "Великобритания" в Великобритании и Великобритании отделения. Дополнительные сведения об элементах управления см. в разделе " [официальная Великобритания](https://www.gov.uk/government/publications/government-security-classifications)".

Следующие сопоставления относятся к элементам управления " **официальный** отделения" в Великобритании и **Великобритании** . Используйте панель навигации справа для перехода непосредственно к сопоставлению конкретных элементов управления. Многие сопоставленные элементы управления реализуются с помощью инициативы [Политика Azure](../../../policy/overview.md). Чтобы просмотреть полную инициативу, откройте раздел **Политика** на портале Azure и перейдите на страницу **Определения**. Затем найдите и выберите **\[Предварительная версия\] аудит UK официальные и Великобритания отделения Controls и развертывание конкретных расширений виртуальной машины для поддержки** встроенной в политику требований аудита.

> [!IMPORTANT]
> Каждый элемент управления ниже связан с одним или несколькими определениями [Политики Azure](../../../policy/overview.md). Такие политики помогут вам в [оценке соответствия](../../../policy/how-to/get-compliance-data.md) с помощью элементов управления, но часто полное или точное соответствие между элементом управления и одной или несколькими политиками отсутствует. Поэтому состояние **Соответствует** в Политике Azure применимо только к самим политикам и не означает полное соответствие всем требованиям элемента управления. Кроме того, стандарт соответствия включает элементы управления, на которые сейчас не распространяются определения Политики Azure. Следовательно, сведения о соответствии в Политике Azure — это только частичное представление общего состояния соответствия. Связи между элементами управления и определениями Политики Azure для этого примера схемы соответствия могут измениться в будущем. Историю изменений можно просмотреть на странице [журнала фиксаций в GitHub](https://github.com/MicrosoftDocs/azure-docs/commits/master/articles/governance/blueprints/samples/ukofficial/control-mapping.md).

## <a name="52-vulnerability-management"></a>5,2 Управление уязвимостью

Эта схема помогает управлять уязвимостями информационной системы, назначая определения [политик Azure](../../../policy/overview.md) , которые отслеживают отсутствующую защиту конечных точек, недостающие обновления системы, уязвимости операционной системы, уязвимости SQL и уязвимости виртуальных машин. Эти полезные сведения предоставляют информацию в реальном времени о состоянии безопасности развернутых ресурсов и помогают вам приоритезировать действия по исправлению.

- Уязвимости в настройках безопасности на ваших компьютерах должны быть устранены
- Уязвимости в базах данных SQL должны быть устранены.
- Уязвимости должны быть устранены с помощью решения для оценки уязвимостей.

- Адаптивные элементы управления приложениями должны быть включены на виртуальных машинах.

## <a name="9-secure-user-management--10-identity-and-authentication"></a>9 безопасное управление пользователями/удостоверение и аутентификация в 10

Azure реализует управление доступом на основе ролей (RBAC), позволяющее управлять доступом к ресурсам в Azure. С помощью портала Azure вы можете просмотреть, у кого есть доступ к ресурсам Azure и какие им назначены разрешения. Эта схема позволяет ограничивать и контролировать права доступа путем назначения определений [политик Azure](../../../policy/overview.md) для аудита внешних учетных записей с владельцами и/или правами на чтение и запись, а также с правами доступа для чтения и записи с владельцами, чтение и/или запись, для которых не включена многофакторная проверка подлинности.

- MFA должна быть включена для учетных записей с разрешениями владельца в вашей подписке.
- Для MFA должны быть включены учетные записи с разрешениями на запись в вашей подписке.
- MFA должна быть включена для учетных записей с разрешениями на чтение в вашей подписке.
- Внешние учетные записи с разрешениями владельца следует удалять из подписки.
- Внешние учетные записи с разрешениями на запись следует удалять из подписки.
- Внешние учетные записи с разрешениями на чтение следует удалять из подписки.

Эта схема назначает определения политик Azure для аудита использования Azure Active Directory проверки подлинности для серверов SQL Server и Service Fabric. Использование проверки подлинности Azure Active Directory упрощает контроль разрешений и обеспечивает централизованное управление удостоверениями для пользователей баз данных и других служб Майкрософт.

- Администратор Azure Active Directory должен быть подготовлен для серверов SQL Server.
- Кластеры Service Fabric должны использовать для проверки подлинности клиента только Azure Active Directory.

Эта схема также назначает определения политик Azure для аудита учетных записей, которые должны быть упорядочены для проверки, включая счета с амортизацией и внешние учетные записи. При необходимости можно запретить вход для учетных записей (или удалить их), что немедленно отменит все права доступа к ресурсам Azure. Этот проект назначает два определения политики Azure для аудита учетной записи с амортизацией, которую следует считать на удаление.

- Устаревшие учетные записи следует удалять из подписки.
- Устаревшие учетные записи с разрешениями владельца следует удалять из подписки.
- Внешние учетные записи с разрешениями владельца следует удалять из подписки.
- Внешние учетные записи с разрешениями на запись следует удалять из подписки.

В этой схеме также назначается определение политики Azure, которое проверяет разрешения файла паролей ВМ Linux на предупреждения, если они заданы неправильно. Такая схема позволяет предпринять действия по исправлению, гарантирующие, что проверка подлинности не скомпрометирована.

- \[Предварительная версия\]: разрешения на аудит виртуальных машин Linux/etc/passwd имеют значение 0644.

Эта схема позволяет применять надежные пароли, назначая определения политик Azure, которые подлежат аудиту виртуальных машин Windows, которые не применяют минимальную стойкость и другие требования к паролю. Определение виртуальных машин, которые нарушают политику надежности паролей, помогает выполнить корректирующие действия и обеспечить соблюдение требований политики для паролей всех учетных записей пользователей на виртуальных машинах.

- \[предварительной версии\]: развертывание требований для аудита виртуальных машин Windows, для которых не включена сложность пароля.
- \[предварительной версии\]: развертывание требований для аудита виртуальных машин Windows, не имеющих максимального срока действия пароля 70 дней.
- \[предварительной версии\]: развертывание требований для аудита виртуальных машин Windows без минимального срока действия пароля в 1 день
- \[предварительной версии\]: развертывание требований для аудита виртуальных машин Windows, не ограничивающих минимальную длину пароля до 14 символов.
- \[предварительной версии\]: развертывание требований для аудита виртуальных машин Windows, которые позволяют повторно использовать предыдущие 24 пароля
- \[предварительной версии\]: аудит виртуальных машин Windows, для которых не установлен параметр сложности пароля.
- \[предварительный просмотр\]: аудит виртуальных машин Windows, не имеющих максимального срока действия пароля 70 дней.
- \[Предварительная версия\]: аудит виртуальных машин Windows без минимального срока действия пароля, равного одному дню
- \[предварительный просмотр\]: аудит виртуальных машин Windows, не ограничивающих минимальную длину пароля до 14 символов.
- \[Предварительная версия\]: аудит виртуальных машин Windows, которые позволяют повторно использовать предыдущие 24 пароля.

Эта схема также помогает управлять доступом к ресурсам Azure, назначая определения политик Azure. для аудита использования типов и конфигураций ресурсов, которые предоставляют повышенные права доступа к ресурсам. Определение ресурсов, которые нарушают эти политики, помогает применить корректирующие действия и ограничить круг доступа к ресурсам Azure только авторизованными пользователями.

- \[предварительной версии\]: развертывание требований для аудита виртуальных машин Linux с учетными записями без паролей
- \[предварительной версии\]: развертывание требований для аудита виртуальных машин Linux, разрешающих удаленные подключения из учетных записей без паролей
- \[Предварительная версия\]: аудит виртуальных машин Linux с учетными записями без паролей
- \[предварительной версии\]: аудит виртуальных машин Linux, разрешающих удаленные подключения от учетных записей без паролей
- Учетные записи хранения должны быть перенесены в новые ресурсы Azure Resource Manager
- Виртуальные машины следует перенести на новые ресурсы Azure Resource Manager
- Аудит виртуальных машин, которые не используют управляемые диски

## <a name="11-external-interface-protection"></a>11. Защита внешнего интерфейса
