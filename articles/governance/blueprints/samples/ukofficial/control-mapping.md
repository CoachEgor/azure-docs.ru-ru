# <a name="control-mapping-of-the-uk-official-and-uk-nhs-blueprint-samples"></a>Управление сопоставлением образцов для официальных и ОТДЕЛЕНИЯных чертежей Великобритании

В следующей статье подробно описано, как учебные примеры "Официальный отделения" и "Великобритания" в Великобритании и Великобритании отделения. Дополнительные сведения об элементах управления см. в разделе " [официальная Великобритания](https://www.gov.uk/government/publications/government-security-classifications)".

Следующие сопоставления относятся к элементам управления " **официальный** отделения" в Великобритании и **Великобритании** . Используйте панель навигации справа для перехода непосредственно к сопоставлению конкретных элементов управления. Многие сопоставленные элементы управления реализуются с помощью инициативы [Политика Azure](../../../policy/overview.md). Чтобы просмотреть полную инициативу, откройте раздел **Политика** на портале Azure и перейдите на страницу **Определения**. Затем найдите и выберите  **\[элементы управления "\] просмотр официальных событий" и "Великобритания отделения" и разверните определенные расширения виртуальных машин для поддержки** встроенной в политику требований аудита.

## <a name="1-data-in-transit-protection"></a>1 данные в транзитной защите

Схема помогает обеспечить безопасность передаваемых данных со службами Azure, назначая определения [политик Azure](../../../policy/overview.md) , которые регистрируют небезопасные подключения к учетным записям хранения и кэшу Redis.

- Должны быть включены только безопасные подключения к кэшу Redis
- Необходимо включить безопасную пересылку в учетные записи хранения

## <a name="23-data-at-rest-protection"></a>2,3. Защита неактивных данных

Эта схема позволяет принудительно применить политику для использования элементов управления криптограф, назначив определения [политик Azure](../../../policy/overview.md) , которые применяют определенные элементы управления криптограф и аудит использования слабых криптографических параметров.
Определение областей, в которых ресурсы Azure могут иметь неоптимальные криптографические конфигурации, поможет вам выполнить корректирующие действия, чтобы убедиться, что ресурсы настроены в соответствии с вашей информационной политикой безопасности. В частности, политикам, назначенным этим планом, требуется шифрование для учетных записей хранения data Lake. требовать прозрачного шифрования данных в базах данных SQL; Аудит отсутствия шифрования для учетных записей хранения, баз данных SQL, дисков виртуальных машин и переменных учетной записи службы автоматизации; Аудит небезопасных подключений к учетным записям хранения и кэшу Redis; Аудит слабого шифрования пароля виртуальной машины; и аудит незашифрованных Service Fabric связи.

- Мониторинг незашифрованных баз данных SQL в центре безопасности Azure
- Шифрование дисков должно применяться к виртуальным машинам
- Переменные учетной записи службы автоматизации должны быть зашифрованы
- Необходимо включить безопасную пересылку в учетные записи хранения
- В Service Fabric кластерах свойство Клустерпротектионлевел должно иметь значение EncryptAndSign
- Необходимо включить прозрачное шифрование данных в базах данных SQL
- Развертывание прозрачного шифрования данных для SQL DB
- Требовать шифрование учетных записей Data Lake Store
- Разрешенные расположения (жестко закодированы как «Южная Великобритания» и «Западная часть Великобритании»)
- Разрешенные расположения для групп ресурсов (жестко закодированы как "Южная часть Великобритании" и "Великобритания")

## <a name="52-vulnerability-management"></a>5,2 Управление уязвимостью

Эта схема помогает управлять уязвимостями информационной системы, назначая определения [политик Azure](../../../policy/overview.md) , которые отслеживают отсутствующую защиту конечных точек, отсутствующие обновления системы, уязвимости операционной системы, уязвимости SQL и виртуальные уязвимости компьютера. Эти полезные сведения предоставляют информацию в реальном времени о состоянии безопасности развернутых ресурсов и помогают вам приоритезировать действия по исправлению.

- Мониторинг отсутствия Endpoint Protection в Центре безопасности Azure
- На компьютерах должны быть установлены обновления системы
- Следует исправить уязвимости в конфигурации безопасности на компьютерах
- Следует исправить уязвимости в базах данных SQL
- Уязвимости следует устранять с помощью решения для оценки уязвимостей

## <a name="53-protective-monitoring"></a>5,3. Мониторинг защиты

Эта схема помогает защитить ресурсы информационной системы путем назначения определений [политик Azure](../../../policy/overview.md) , обеспечивающих защиту при неограниченном доступе, список разрешений действиях и угрозах.

- Аудит неограниченного сетевого доступа к учетным записям хранения
- Адаптивные элементы управления приложениями должны быть включены на виртуальных машинах
- Развертывание обнаружения угроз на серверах SQL Server
- Развертывание расширения Microsoft IaaS для защиты от вредоносных программ по умолчанию для Windows Server

## <a name="9-secure-user-management--10-identity-and-authentication"></a>9 безопасное управление пользователями/удостоверение и аутентификация в 10

Azure реализует управление доступом на основе ролей (RBAC), позволяющее управлять доступом к ресурсам в Azure. С помощью портала Azure вы можете просмотреть, у кого есть доступ к ресурсам Azure и какие им назначены разрешения. Эта схема позволяет ограничивать и контролировать права доступа путем назначения определений [политики Azure](../../../policy/overview.md) для аудита внешних учетных записей с владельцами и/или правами на чтение и запись и учетными записями с владельцами, правами на чтение и запись, у которых нет многофакторных Проверка подлинности включена.

- Необходимо включить MFA для учетных записей с разрешениями владельца в подписке.
- Для MFA должны быть включены учетные записи с разрешениями на запись в вашей подписке.
- MFA необходимо включить для учетных записей с разрешениями на чтение в вашей подписке.
- Внешние учетные записи с разрешениями владельца должны быть удалены из подписки.
- External accounts with write permissions should be removed from your subscription (Внешние учетные записи с разрешениями на запись следует удалять из подписки)
- Внешние учетные записи с разрешениями на чтение должны быть удалены из подписки.

Эта схема назначает определения политик Azure для аудита использования Azure Active Directory проверки подлинности для серверов SQL Server и Service Fabric. Использование проверки подлинности Azure Active Directory упрощает контроль разрешений и обеспечивает централизованное управление удостоверениями для пользователей баз данных и других служб Майкрософт.

- Для серверов SQL Server необходимо подготовить администратора Azure Active Directory
- Service Fabric кластеры должны использовать только Azure Active Directory для проверки подлинности клиента.

Эта схема также назначает определения политик Azure для аудита учетных записей, которые должны быть упорядочены для проверки, включая счета с амортизацией и внешние учетные записи. При необходимости можно запретить вход для учетных записей (или удалить их), что немедленно отменит все права доступа к ресурсам Azure. Этот проект назначает два определения политики Azure для аудита учетной записи с амортизацией, которую следует считать на удаление.

- Устаревшие учетные записи следует удалить из подписки.
- Устаревшие учетные записи с разрешениями владельца должны быть удалены из подписки.
- Внешние учетные записи с разрешениями владельца должны быть удалены из подписки.
- External accounts with write permissions should be removed from your subscription (Внешние учетные записи с разрешениями на запись следует удалять из подписки)

В этой схеме также назначается определение политики Azure, которое проверяет разрешения файла паролей ВМ Linux на предупреждения, если они заданы неправильно. Такая схема позволяет предпринять действия по исправлению, гарантирующие, что проверка подлинности не скомпрометирована.

- \[Предварительная версия.\] Параметр Audit Linux/etc/passwd File Permissions имеет значение 0644

Эта схема позволяет применять надежные пароли, назначая определения политик Azure, которые подлежат аудиту виртуальных машин Windows, которые не применяют минимальную стойкость и другие требования к паролю. Определение виртуальных машин, которые нарушают политику надежности паролей, помогает выполнить корректирующие действия и обеспечить соблюдение требований политики для паролей всех учетных записей пользователей на виртуальных машинах.

- \[Предварительная версия.\] Развертывание требований для аудита виртуальных машин Windows, для которых включен параметр сложности пароля
- \[Предварительная версия.\] Развертывание требований для аудита виртуальных машин Windows, не имеющих максимального срока действия пароля 70 дней
- \[Предварительная версия.\] Развертывание требований для аудита виртуальных машин Windows без минимального срока действия пароля в 1 день
- \[Предварительная версия.\] Развертывание требований для аудита виртуальных машин Windows, не ограничивающих минимальную длину пароля до 14 символов
- \[Предварительная версия.\] Развертывание требований для аудита виртуальных машин Windows, которые позволяют повторно использовать предыдущие 24 пароля
- \[Предварительная версия.\] Аудит виртуальных машин Windows, для которых не установлен параметр сложности пароля
- \[Предварительная версия.\] Аудит виртуальных машин Windows, не имеющих максимального срока действия пароля 70 дней
- \[Предварительная версия.\] Аудит виртуальных машин Windows без минимального срока действия пароля, равного одному дню
- \[Предварительная версия.\] Аудит виртуальных машин Windows, не ограничивающих минимальную длину пароля до 14 символов
- \[Предварительная версия.\] Аудит виртуальных машин Windows, которые позволяют повторно использовать предыдущие 24 пароля

Эта схема также помогает управлять доступом к ресурсам Azure, назначая определения политик Azure. для аудита использования типов и конфигураций ресурсов, которые предоставляют повышенные права доступа к ресурсам. Определение ресурсов, которые нарушают эти политики, помогает применить корректирующие действия и ограничить круг доступа к ресурсам Azure только авторизованными пользователями.

- \[Предварительная версия.\] Развертывание требований для аудита виртуальных машин Linux с учетными записями без паролей
- \[Предварительная версия.\] Развертывание требований для аудита виртуальных машин Linux, которые разрешают удаленные подключения из учетных записей без паролей
- \[Предварительная версия.\] Аудит виртуальных машин Linux с учетными записями без паролей
- \[Предварительная версия.\] Аудит виртуальных машин Linux, которые разрешают удаленные подключения от учетных записей без паролей
- Учетные записи хранения должны быть перенесены в новые ресурсы Azure Resource Manager
- Виртуальные машины следует перенести на новые ресурсы Azure Resource Manager
- Аудит виртуальных машин, которые не используют управляемые диски

## <a name="11-external-interface-protection"></a>11. Защита внешнего интерфейса

Кроме использования более 25 политик для обеспечения надежного управления пользователями, эта схема помогает защитить интерфейсы служб от несанкционированного доступа путем назначения определения [политики Azure](../../../policy/overview.md) , которое наблюдает за неограниченными учетными записями хранения. Учетные записи хранения с неограниченным доступом могут предоставлять непредусмотренный доступ к информации, содержащейся в информационной системе. Эта схема также назначает политику, которая включает адаптивные элементы управления приложениями на виртуальных машинах.

- Аудит неограниченного сетевого доступа к учетным записям хранения
- Адаптивные элементы управления приложениями должны быть включены на виртуальных машинах

## <a name="12-secure-service-administration"></a>12. безопасное Администрирование службы

Azure реализует управление доступом на основе ролей (RBAC), позволяющее управлять доступом к ресурсам в Azure. С помощью портала Azure вы можете просмотреть, у кого есть доступ к ресурсам Azure и какие им назначены разрешения. Эта схема помогает ограничить права привилегированного доступа и контролировать их, назначая пять определений [политик Azure](../../../policy/overview.md) для аудита внешних учетных записей с владельцами и/или записью и учетными записями с владельцами или разрешениями на запись, которые не имеют многофакторная проверка подлинности включена.

Системы, используемые для администрирования облачной службы, будут иметь строго привилегированный доступ к этой службе. Их компрометирование окажет значительное влияние, в том числе попытки обхода контроля безопасности и кражи или манипулирования большими объемами данных. Методы, используемые администраторами поставщика услуг для управления рабочей службой, должны быть спроектированы таким образом, чтобы снизить риск использования, который может наугрожать безопасности службы. Если этот принцип не реализован, злоумышленник может иметь возможность обойти элементы управления безопасностью и украсть или манипулировать большими объемами данных.

- Необходимо включить MFA для учетных записей с разрешениями владельца в подписке.
- Для MFA должны быть включены учетные записи с разрешениями на запись в вашей подписке.
- Внешние учетные записи с разрешениями владельца должны быть удалены из подписки.
- External accounts with write permissions should be removed from your subscription (Внешние учетные записи с разрешениями на запись следует удалять из подписки)

Эта схема назначает определения политик Azure для аудита использования Azure Active Directory проверки подлинности для серверов SQL Server и Service Fabric. Использование проверки подлинности Azure Active Directory упрощает контроль разрешений и обеспечивает централизованное управление удостоверениями для пользователей баз данных и других служб Майкрософт.

- Для серверов SQL Server необходимо подготовить администратора Azure Active Directory
- Service Fabric кластеры должны использовать только Azure Active Directory для проверки подлинности клиента.

В этой схеме также назначаются определения политик Azure для аудита учетных записей, которые должны быть упорядочены для проверки, включая счета с амортизацией и внешние учетные записи с повышенными разрешениями. При необходимости можно запретить вход для учетных записей (или удалить их), что немедленно отменит все права доступа к ресурсам Azure. Этот проект назначает два определения политики Azure для аудита учетной записи с амортизацией, которую следует считать на удаление.

- Устаревшие учетные записи следует удалить из подписки.
- Устаревшие учетные записи с разрешениями владельца должны быть удалены из подписки.
- Внешние учетные записи с разрешениями владельца должны быть удалены из подписки.
- External accounts with write permissions should be removed from your subscription (Внешние учетные записи с разрешениями на запись следует удалять из подписки)

В этой схеме также назначается определение политики Azure, которое проверяет разрешения файла паролей ВМ Linux на предупреждения, если они заданы неправильно. Такая схема позволяет предпринять действия по исправлению, гарантирующие, что проверка подлинности не скомпрометирована.

- \[Предварительная версия.\] Параметр Audit Linux/etc/passwd File Permissions имеет значение 0644
