# <a name="control-mapping-of-the-uk-official-blueprint-sample"></a>Сопоставление элемента управления Великобритании ОФИЦИАЛЬНЫЙ образца схемы

Ниже статье подробно описано, как часть соединенного Королевства ОФИЦИАЛЬНЫЕ образцы схемы сопоставляются с ОФИЦИАЛЬНОЙ Великобритании элементы управления.
Дополнительные сведения об элементах управления см. в разделе [Великобритании ОФИЦИАЛЬНЫЙ](https://www.gov.uk/government/publications/government-security-classifications).

Следующие сопоставления должны **Великобритании ОФИЦИАЛЬНЫЙ** элементов управления. Используйте панель навигации справа для перехода непосредственно к сопоставлению конкретных элементов управления. Многие сопоставленные элементы управления реализуются с помощью инициативы [Политика Azure](../../../policy/overview.md). Чтобы просмотреть полную инициативу, откройте раздел **Политика** на портале Azure и перейдите на страницу **Определения**. Затем найдите и выберите  **[Предварительный просмотр] аудита Великобритании ОФИЦИАЛЬНЫЙ управляет и развернуть специальные расширения виртуальной Машины для поддержки требований аудита** инициатива встроенные политики.

## <a name="1-data-in-transit-protection"></a>1 — данные в передаче защиты

Эта схема помогает обеспечить безопасность передачи информации с помощью служб Azure, назначая два определения [Политики Azure](../../../policy/overview.md) для аудита небезопасных подключений к учетным записям хранения и Кэшу Redis.

- Должен быть включен только безопасных подключений к кэшу Redis
- Должен быть включен безопасную передачу учетных записей хранения

## <a name="23-data-at-rest-protection"></a>2.3 Защита неактивных данных

В рамках этой схемы позволяет принудительно применить политику на использование элементов управления криптографии, назначив 11 ["Политика Azure"](../../../policy/overview.md) определений, которые применяют элементы управления конкретного криптографии и аудит использования параметров слабого шифрования.
Определение областей, в которых ресурсы Azure могут иметь неоптимальные криптографические конфигурации, поможет вам выполнить корректирующие действия, чтобы убедиться, что ресурсы настроены в соответствии с вашей информационной политикой безопасности. В частности политики, назначенный в рамках этой схемы обязательное использование шифрования для учетных записей хранения Озера данных; требуется прозрачное шифрование данных для баз данных SQL; Проверка отсутствия шифрования в учетных записях хранения, баз данных SQL, диски виртуальной машины и переменные для учетной записи службы автоматизации; Аудит небезопасные подключения к учетным записям хранения и кэша Redis; Проверка шифрования пароля слабые виртуальной машины; и аудита незашифрованный обмен данными с Service Fabric.

- Мониторинг незашифрованных баз данных SQL в центре безопасности Azure
- Шифрование дисков, которые должны быть выполнены с виртуальных машин
- Переменные для учетной записи службы автоматизации должны быть зашифрованы
- Должен быть включен безопасную передачу учетных записей хранения
- Должен быть включен безопасную передачу учетных записей хранения
- Кластеры Service Fabric должно быть задано свойство ClusterProtectionLevel для EncryptAndSign
- Прозрачное шифрование данных для баз данных SQL должна быть включена
- Развертывание базы данных SQL прозрачное шифрование данных
- Требовать шифрование на учетные записи Data Lake Store
- Допустимые расположения (была жестких «ЮЖНАЯ часть соединенного Королевства» и «ЗАПАДНАЯ часть соединенного Королевства» в коде)
- Разрешенные расположения для группы ресурсов (была жестких «ЮЖНАЯ часть соединенного Королевства» и «ЗАПАДНАЯ часть соединенного Королевства» в коде)

## <a name="52-vulnerability-management"></a>5.2 управление уязвимостями

Эта схема помогает управлять уязвимостях информационной системы путем назначения пять ["Политика Azure"](../../../policy/overview.md) определения, отслеживание отсутствия endpoint protection, обновлений системы, эксплуатации уязвимостей системы, SQL уязвимости и уязвимости виртуальной машины. Эти полезные сведения предоставляют информацию в реальном времени о состоянии безопасности развернутых ресурсов и помогают вам приоритезировать действия по исправлению.

- Мониторинг отсутствия Endpoint Protection в Центре безопасности Azure
- Обновления системы, которые должны быть установлены на компьютерах
- Уязвимости в конфигурации безопасности на компьютерах должен быть восстановлен
- Уязвимости в базах данных SQL должен быть восстановлен
- Уязвимостей следует исправить с помощью решения для оценки уязвимостей

## <a name="53-protective-monitoring"></a>5.3 защитный мониторинг

В рамках этой схемы позволяет защитить ресурсы информационной системы путем назначения ["Политика Azure"](../../../policy/overview.md) определение, которое отслеживает неограниченный учетные записи хранения. В рамках этой схемы также назначает определение "Политика Azure", который отслеживает действие в список разрешений.

- Аудит неограниченного сетевого доступа к учетным записям хранения
- Адаптивные элементы управления приложениями, которые должны быть включены на виртуальных машинах
- Развертывание обнаружения угроз на серверах SQL Server
- Развертывание расширения Microsoft IaaS защиты от вредоносных программ по умолчанию для Windows Server

## <a name="9-secure-user-management--10-identity-and-authentication"></a>9 безопасное управление пользователями / 10 идентификации и проверки подлинности

Azure также реализована на основе ролей управления доступом (RBAC) для помогает, которыми вы управляете у кого есть доступ к ресурсам в Azure. С помощью портала Azure вы можете просмотреть, у кого есть доступ к ресурсам Azure и какие им назначены разрешения. В рамках этой схемы, которая помогает ограничить и контролировать права привилегированного доступа, назначив шесть ["Политика Azure"](../../../policy/overview.md) определений аудита внешние учетные записи с владельцем и/или чтения и записи, разрешения и учетные записи с владельцем, считывают или записывают разрешения, которые не имеют включена многофакторная проверка подлинности.

- Многофакторной проверки Подлинности должен быть включен на учетные записи с правами владельца в подписке
- Многофакторной проверки Подлинности должно быть включенных учетных записей с разрешениями на запись для одной подписки
- Многофакторной проверки Подлинности должен быть включен на учетные записи с разрешениями на чтение в подписке
- Аудит внешних учетных записей с разрешениями владельца для подписки
- Аудит внешних учетных записей с разрешениями на запись для подписки
- Аудит внешних учетных записей с разрешениями на чтение для подписки

В рамках этой схемы назначает два определения политики Azure для аудита использования проверки подлинности Azure Active Directory для серверов SQL и Service Fabric. Использование проверки подлинности Azure Active Directory упрощает контроль разрешений и обеспечивает централизованное управление удостоверениями для пользователей баз данных и других служб Майкрософт.

- Администратор Azure Active Directory должен быть подготовлен для серверов SQL Server
- Кластеры Service Fabric следует использовать только Azure Active Directory для проверки подлинности клиента

В рамках этой схемы также назначает пять определения политики Azure для учетных записей, которые должны иметь приоритет для проверки, включая амортизированной учетные записи и внешние учетные записи аудита. При необходимости можно запретить вход для учетных записей (или удалить их), что немедленно отменит все права доступа к ресурсам Azure. В рамках этой схемы назначение два определения политик Azure следует учитывать для удаления учетной записи аудита амортизированы.

- Устаревшие учетные записи должны быть удалены из подписки
- Устаревшие учетные записи с правами владельца, которые должны быть удалены из подписки
- Внешние учетные записи с правами владельца, которые должны быть удалены из подписки
- External accounts with write permissions should be removed from your subscription (Внешние учетные записи с разрешениями на запись следует удалять из подписки)

В рамках этой схемы также назначает определение "Политика Azure", который проверяет виртуальной Машины Linux пароль разрешения для создания предупреждения, если их можно установить, неправильно. Такой подход позволяет предпринять действия по исправлению, чтобы убедиться, что не скомпрометированы структур проверки подлинности.

- [Предварительный просмотр]: Audit Linux VM /etc/passwd file permissions are set to 0644

В рамках этой схемы служит для применения надежных паролей, назначив 10 определения политики Azure, аудит виртуальных машин Windows, не применяют минимальное надежность и другие требования к паролю. Определение виртуальных машин, которые нарушают политику надежности паролей, помогает выполнить корректирующие действия и обеспечить соблюдение требований политики для паролей всех учетных записей пользователей на виртуальных машинах.

- [Предварительный просмотр]: Deploy requirements to audit Windows VMs that do not have the password complexity setting enabled
- [Предварительный просмотр]: Deploy requirements to audit Windows VMs that do not have a maximum password age of 70 days
- [Предварительный просмотр]: Deploy requirements to audit Windows VMs that do not have a minimum password age of 1 day
- [Предварительный просмотр]: Deploy requirements to audit Windows VMs that do not restrict the minimum password length to 14 characters
- [Предварительный просмотр]: Deploy requirements to audit Windows VMs that allow re-use of the previous 24 passwords
- [Предварительный просмотр]: Audit Windows VMs that do not have the password complexity setting enabled
- [Предварительный просмотр]: Audit Windows VMs that do not have a maximum password age of 70 days
- [Предварительный просмотр]: Audit Windows VMs that do not have a minimum password age of 1 day
- [Предварительный просмотр]: Audit Windows VMs that do not restrict the minimum password length to 14 characters
- [Предварительный просмотр]: Audit Windows VMs that allow re-use of the previous 24 passwords

Это схемы, в которой также помогает управлять доступом к ресурсам Azure, назначив семь определения политики Azure. для аудита использования типов и конфигураций ресурсов, которые предоставляют повышенные права доступа к ресурсам. Определение ресурсов, которые нарушают эти политики, помогает применить корректирующие действия и ограничить круг доступа к ресурсам Azure только авторизованными пользователями.

- [Предварительный просмотр]: Deploy requirements to audit Linux VMs that have accounts without passwords
- [Предварительный просмотр]: Deploy requirements to audit Linux VMs that allow remote connections from accounts without passwords
- [Предварительный просмотр]: Audit Linux VMs that have accounts without passwords
- [Предварительный просмотр]: Audit Linux VMs that allow remote connections from accounts without passwords
- Учетные записи хранения следует перенести в новые ресурсы Azure Resource Manager
- Виртуальные машины следует перенести в новые ресурсы Azure Resource Manager
- Аудит виртуальных машин, которые не используют управляемые диски

## <a name="11-external-interface-protection"></a>Защита 11 внешнего интерфейса

Помимо использования более чем 25 политики для соответствующего пользователя безопасного управления, в рамках этой схемы помогает защитить интерфейсов служб от несанкционированного доступа путем назначения ["Политика Azure"](../../../policy/overview.md) определение, которое неограниченный мониторов учетные записи хранения. Учетные записи хранения с неограниченным доступом можно разрешить непредусмотренного доступа к информации, содержащейся в информационной системе. В рамках этой схемы также назначает политики, разрешающей адаптивного управления приложениями на виртуальных машинах.

- Аудит неограниченного сетевого доступа к учетным записям хранения
- Адаптивные элементы управления приложениями, которые должны быть включены на виртуальных машинах

## <a name="12-secure-service-administration"></a>12 безопасной службы администрирования

Системы, используемые для администрирования облачной службы, будут иметь строго привилегированный доступ к этой службе. Их компрометирование окажет значительное влияние, в том числе попытки обхода контроля безопасности и кражи или манипулирования большими объемами данных. Методы, используемые администраторами службы поставщика для управления службой оперативной должны разрабатываться для снижения риска использования, могут подорвать безопасность службы. Если этот принцип не реализован, злоумышленник может иметь средства для обхода контроля безопасности и кражи или манипулирования большими объемами данных.

- Многофакторной проверки Подлинности должен быть включен на учетные записи с правами владельца в подписке

- External accounts with write permissions should be removed from your subscription (Внешние учетные записи с разрешениями на запись следует удалять из подписки)

Использование проверки подлинности Azure Active Directory упрощает контроль разрешений и обеспечивает централизованное управление удостоверениями для пользователей баз данных и других служб Майкрософт.

- Администратор Azure Active Directory должен быть подготовлен для серверов SQL Server
- Кластеры Service Fabric следует использовать только Azure Active Directory для проверки подлинности клиента

В рамках этой схемы также назначает четыре определения политик Azure аудит учетных записей, которые должны иметь приоритет для проверки, включая амортизированной учетные записи и внешние учетные записи с повышенными разрешениями. При необходимости можно запретить вход для учетных записей (или удалить их), что немедленно отменит все права доступа к ресурсам Azure. Эта схема помогает обеспечить системы событий, назначив 6 ["Политика Azure"](../../../policy/overview.md) определений, которые журнала параметров аудита на ресурсы Azure. Также эта политика ведет аудит отправки виртуальными машинами журналов в указанную рабочую область Log Analytics.

- [Предварительный просмотр]: Deploy Log Analytics Agent for Linux VMs
- [Предварительный просмотр]: Deploy Log Analytics Agent for Windows VMs
