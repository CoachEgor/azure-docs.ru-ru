# <a name="control-mapping-of-the-pci-dss-v321-blueprint-sample"></a>Сопоставление элементов управления с примером чертежа PCI-DSS v 3.2.1

В следующей статье подробно описано, как пример схемы Azure-DSS v версии 3.2.1 соответствует схеме с элементами управления PCI-DSS v 3.2.1. Дополнительные сведения об элементах управления см. в разделе [PCI-DSS v 3.2.1](https://www.pcisecuritystandards.org/documents/PCI_DSS_v3-2-1.pdf).

Следующие сопоставления относятся к элементам управления **PCI-DSS v 3.2.1:2018** . Используйте панель навигации справа для перехода непосредственно к сопоставлению конкретных элементов управления. Многие сопоставленные элементы управления реализуются с помощью инициативы [Политика Azure](../../../policy/overview.md). Чтобы просмотреть полную инициативу, откройте раздел **Политика** на портале Azure и перейдите на страницу **Определения**. Затем найдите и выберите  **\[элементы управления предварительный\] аудит PCI v 3.2.1:2018 и разверните определенные расширения виртуальных машин для поддержки** встроенной в политику требований аудита.

## <a name="132-and-134-boundary-protection"></a>Защита границ 1.3.2 и 1.3.4

Эта схема помогает управлять сетями и контролировать их, назначая определения [политик Azure](../../../policy/overview.md) , которые отслеживают группы безопасности сети с разрешающими правилами. Использование недостаточно строгих правил может привести к несанкционированному доступу по сети. Такие правила следует пересмотреть. Эта схема назначает одно определение политики Azure, которое отслеживает незащищенные конечные точки, приложения и учетные записи хранения. Использование конечных точек и приложений, которые не защищены брандмауэром, а также учетных записей хранения с неограниченным доступом может привести к нежелательному доступу к данным, содержащимся в информационной системе.

- Аудит неограниченного сетевого доступа к учетным записям хранения
- Доступ через конечную точку с выходом в Интернет должен быть ограничен

## <a name="34a-41-41g-41h-and-653-cryptographic-protection"></a>3.4. a, 4,1, 4.1. g, 4.1. h и 6.5.3 криптографическую защиту

Эта схема позволяет применить политику с помощью элементов управления криптограф, назначив определения [политик Azure](../../../policy/overview.md) , которые применяют определенные элементы управления криптограф и аудит использования слабых криптографических параметров. Определение областей, в которых ресурсы Azure могут иметь неоптимальные криптографические конфигурации, поможет вам выполнить корректирующие действия, чтобы убедиться, что ресурсы настроены в соответствии с вашей информационной политикой безопасности. В частности, политикам, назначенным этим планом, требуется прозрачное шифрование данных в базах данных SQL. Аудит отсутствия шифрования для учетных записей хранения и переменных учетной записи службы автоматизации. Кроме того, существуют политики, которые устраняют небезопасные подключения к учетным записям хранения, приложениям-функциям, WebApp, приложениям API и кэшу Redis, а также аудиту незашифрованных Service Fabric связи.

- Приложение-функция должно быть доступно только по HTTPS
- Веб-приложение должно быть доступно только по HTTPS
- Приложение API должно быть доступно только по протоколу HTTPS
- Мониторинг баз данных SQL без шифрования в Центре безопасности Azure
- Шифрование дисков должно применяться к виртуальным машинам
- Переменные учетной записи службы автоматизации должны быть зашифрованы
- Должны быть включены только безопасные подключения к кэшу Redis
- Необходимо включить безопасную пересылку в учетные записи хранения
- В Service Fabric кластерах свойство Клустерпротектионлевел должно иметь значение EncryptAndSign
- Необходимо включить прозрачное шифрование данных в базах данных SQL
- Развертывание прозрачного шифрования данных для SQL DB

## <a name="51-62-66-and-1121-vulnerability-scanning-and-system-updates"></a>Сканирование уязвимостей 5,1, 6,2, 6,6 и 11.2.1 и обновления системы

Эта схема помогает управлять уязвимостями информационной системы, назначая определения [политик Azure](../../../policy/overview.md) , которые отслеживают недостающие обновления системы, уязвимости операционной системы, уязвимости SQL и виртуальные машины в Azure. Центр безопасности. Центр безопасности Azure предоставляет возможности подготовки отчетов, которые позволяют получать полезные сведения в реальном времени о состоянии безопасности развернутых ресурсов Azure.

- Мониторинг отсутствия Endpoint Protection в Центре безопасности Azure
- Развертывание стандартного расширения IaaSAntimalware корпорации Майкрософт для Windows
- Развертывание обнаружения угроз на серверах SQL Server
- На компьютерах должны быть установлены обновления системы
- Следует исправить уязвимости в конфигурации безопасности на компьютерах
- Следует исправить уязвимости в базах данных SQL
- Уязвимости следует устранять с помощью решения для оценки уязвимостей

## <a name="711-712-and-713-separation-of-duties"></a>7.1.1. 7.1.2 и 7.1.3 разделение обязанностей

Наличие только одного владельца подписки Azure не позволит организовать административную избыточность. И наоборот, слишком большое число владельцев подписки Azure может увеличить вероятность бреши в системе безопасности через скомпрометированную учетную запись владельца. Эта схема помогает поддерживать соответствующее количество владельцев подписок Azure, назначая определения [политик Azure](../../../policy/overview.md) , которые подлежат аудиту числа владельцев для подписок Azure. Управление разрешениями владельца подписки помогает реализовать соответствующее разделение обязанностей.

- К подписке должен быть назначен более одного владельца.
- Для подписки должно быть назначено не более 3 владельцев. 

## <a name="32-721-831a-and-831b-management-of-privileged-access-rights"></a>3,2, 7.2.1, 8.3.1. a и 8.3.1. b. Управление привилегированными правами доступа

Эта схема помогает ограничить и контролировать привилегированные права доступа путем назначения определений [политики Azure](../../../policy/overview.md) для аудита внешних учетных записей с правами владельца, записи и/или чтения, а также учетных записей сотрудников с правами владельца и/или записи, у которых нет многофакторная проверка подлинности включена. В Azure реализовано управление доступом на основе ролей (RBAC) для управления пользователями, которые обращаются к ресурсам Azure. Определение областей применения настраиваемых правил RBAC поможет вам убедиться в их необходимости и правильной реализации, так как в настраиваемых правилах RBAC легко ошибиться. Эта схема также назначает определения [политик Azure](../../../policy/overview.md) для аудита использования проверки подлинности Azure Active Directory для серверов SQL Server. Использование Azure Active Directory проверки подлинности упрощает управление полномочиями и централизует управление удостоверениями пользователей базы данных и других служб Майкрософт  
Обслуживание.
 
- Внешние учетные записи с разрешениями владельца должны быть удалены из подписки.
- External accounts with write permissions should be removed from your subscription (Внешние учетные записи с разрешениями на запись следует удалять из подписки)
- Внешние учетные записи с разрешениями на чтение должны быть удалены из подписки.
- Необходимо включить MFA для учетных записей с разрешениями владельца в подписке.
- Для MFA должны быть включены учетные записи с разрешениями на запись в вашей подписке.
- MFA необходимо включить для учетных записей с разрешениями на чтение в вашей подписке.
- Для серверов SQL Server необходимо подготовить администратора Azure Active Directory
- Аудит использования настраиваемых правил RBAC

## <a name="812-and-815-least-privilege-and-review-of-user-access-rights"></a>BIND 8.1.2 и 8.1.5 минимальные привилегии и проверка прав доступа пользователя

Azure реализует управление доступом на основе ролей (RBAC), позволяющее управлять доступом к ресурсам в Azure. С помощью портала Azure вы можете просмотреть, у кого есть доступ к ресурсам Azure и какие им назначены разрешения. Эта схема назначает определения [политик Azure](../../../policy/overview.md) для аудита учетных записей, которые должны быть упорядочены для проверки, включая счета с амортизацией и внешние учетные записи с повышенными разрешениями.

- Устаревшие учетные записи следует удалить из подписки.
- Устаревшие учетные записи с разрешениями владельца должны быть удалены из подписки.
- Внешние учетные записи с разрешениями владельца должны быть удалены из подписки.
- External accounts with write permissions should be removed from your subscription (Внешние учетные записи с разрешениями на запись следует удалять из подписки)
- Внешние учетные записи с разрешениями на чтение должны быть удалены из подписки.

## <a name="813-removal-or-adjustment-of-access-rights"></a>8.1.3 Удаление или корректировка прав доступа

Azure реализует управление доступом на основе ролей (RBAC), помогающее управлять доступом к ресурсам в Azure. С помощью Azure Active Directory и RBAC можно обновить роли пользователей, чтобы отразить организационные изменения. При необходимости можно запретить вход для учетных записей (или удалить их), что немедленно отменит все права доступа к ресурсам Azure. Эта схема назначает определения [политик Azure](../../../policy/overview.md) для аудита учетной записи, которую следует учитывать при удалении.

- Устаревшие учетные записи следует удалить из подписки.
- Устаревшие учетные записи с разрешениями владельца должны быть удалены из подписки.

## <a name="823ab-824ab-and-825-password-based-authentication"></a>8.2.3. a, b, 8.2.4. a, b и 8.2.5 Проверка подлинности на основе пароля

Эта схема позволяет применять надежные пароли, назначая определения [политик Azure](../../../policy/overview.md) , которые подлежат аудиту виртуальных машин Windows, которые не применяют минимальную стойкость и другие требования к паролю. Определение виртуальных машин, которые нарушают политику надежности паролей, помогает выполнить корректирующие действия и обеспечить соблюдение требований политики для паролей всех учетных записей пользователей на виртуальных машинах.

- \[Предварительная версия.\] Аудит виртуальных машин Windows, не имеющих максимального срока действия пароля 70 дней
- \[Предварительная версия.\] Развертывание требований для аудита виртуальных машин Windows, не имеющих максимального срока действия пароля 70 дней
- \[Предварительная версия.\] Аудит виртуальных машин Windows, не ограничивающих минимальную длину пароля до 14 символов
- \[Предварительная версия.\] Развертывание требований для аудита виртуальных машин Windows, не ограничивающих минимальную длину пароля до 14 символов
- \[Предварительная версия.\] Аудит виртуальных машин Windows, которые позволяют повторно использовать предыдущие 24 пароля
- \[Предварительная версия.\] Развертывание требований для аудита виртуальных машин Windows, которые позволяют повторно использовать предыдущие 24 пароля

## <a name="103-and-1054-audit-generation"></a>Создание аудита 10,3 и 10.5.4

Эта схема позволяет убедиться в регистрации системных событий путем назначения определений [политики Azure](../../../policy/overview.md) , которые используются для настройки журнала аудита ресурсов Azure.
Журналы диагностики позволяют подробно проанализировать операции, выполненные с ресурсами Azure. Журналы Azure используют синхронизированные внутренние часы для создания записей о событиях с ресурсами с метками времени.

- Мониторинг непроверенных серверов SQL Server в центре безопасности Azure
- Аудит параметров диагностики
- Аудит параметров аудита на уровне сервера SQL
- Развертывание аудита для серверов SQL Server
- Учетные записи хранения должны быть перенесены в новые ресурсы Azure Resource Manager
- Виртуальные машины следует перенести на новые ресурсы Azure Resource Manager
