# <a name="control-mapping-of-the-pci-dss-v321-blueprint-sample"></a>Сопоставление элемента управления образца схемы v3.2.1 PCI-DSS

Ниже статье подробно описано, как образец схемы v3.2.1 чертежей Azure PCI-DSS сопоставляются с элементами управления v3.2.1 PCI-DSS. Дополнительные сведения об элементах управления см. в разделе [v3.2.1 PCI-DSS](https://www.pcisecuritystandards.org/documents/PCI_DSS_v3-2-1.pdf).

Следующие сопоставления должны **v3.2.1:2018 PCI-DSS** элементов управления. Используйте панель навигации справа для перехода непосредственно к сопоставлению конкретных элементов управления. Многие сопоставленные элементы управления реализуются с помощью инициативы [Политика Azure](../../../policy/overview.md). Чтобы просмотреть полную инициативу, откройте раздел **Политика** на портале Azure и перейдите на страницу **Определения**. Затем найдите и выберите  **[Предварительный просмотр] v3.2.1:2018 аудита PCI управляет и развертывать специальные расширения виртуальной Машины для поддержки требований аудита** инициатива встроенные политики.

## <a name="132-and-134-boundary-protection"></a>1.3.2 и 1.3.4 защита границ

В рамках этой схемы, которая помогает управлять и контролировать сетей, назначив ["Политика Azure"](../../../policy/overview.md) определений, которые отслеживает групп безопасности сети с помощью разрешительными правилами. Использование недостаточно строгих правил может привести к несанкционированному доступу по сети. Такие правила следует пересмотреть. В рамках этой схемы назначает один определения политики Azure, которые отслеживают незащищенные конечные точки, приложений и учетные записи хранения. Использование конечных точек и приложений, которые не защищены брандмауэром, а также учетных записей хранения с неограниченным доступом может привести к нежелательному доступу к данным, содержащимся в информационной системе.

- Аудит неограниченного сетевого доступа к учетным записям хранения
- Доступ через конечную веб-точку следует ограничивать

## <a name="34a-41-41g-41h-and-653-cryptographic-protection"></a>3.4.a 4.1, 4.1.g, 4.1.h и 6.5.3 Криптографическая защита

В рамках этой схемы позволяет принудительно применить политику с использованием элементов криптографии, назначив ["Политика Azure"](../../../policy/overview.md) определений, которые определяют элементы управления конкретного криптографии и аудит использования параметров слабого шифрования. Определение областей, в которых ресурсы Azure могут иметь неоптимальные криптографические конфигурации, поможет вам выполнить корректирующие действия, чтобы убедиться, что ресурсы настроены в соответствии с вашей информационной политикой безопасности. В частности политики, назначенный в рамках этой схемы требуют прозрачное шифрование данных для баз данных SQL; Проверка отсутствия шифрования в учетных записях хранения и переменных учетной записи службы автоматизации. Кроме того, существуют также политики, какой адрес аудит небезопасные подключения для учетных записей хранения, приложения-функции, веб-приложения, приложения API и кэш Redis и аудит незашифрованный обмен данными с Service Fabric.

- Приложение-функция должно быть доступно только по HTTPS
- Веб-приложение должно быть доступно только по HTTPS
- Приложения API должны быть доступны только по протоколу HTTPS
- Мониторинг баз данных SQL без шифрования в Центре безопасности Azure
- Шифрование дисков, которые должны быть выполнены с виртуальных машин
- Переменные для учетной записи службы автоматизации должны быть зашифрованы
- Должен быть включен только безопасных подключений к кэшу Redis
- Должен быть включен безопасную передачу учетных записей хранения
- Кластеры Service Fabric должно быть задано свойство ClusterProtectionLevel для EncryptAndSign
- Прозрачное шифрование данных для баз данных SQL должна быть включена
- Развертывание базы данных SQL прозрачное шифрование данных

## <a name="51-62-66-and-1121-vulnerability-scanning-and-system-updates"></a>5.1, 6.2, 6.6 и 11.2.1 сканирование уязвимостей и обновлений системы

Эта схема помогает управлять уязвимостях информационной системы путем назначения ["Политика Azure"](../../../policy/overview.md) определений, которые отслеживают отсутствующих обновлений системы, уязвимости операционной системы, уязвимости SQL и виртуальной машины уязвимости в центре безопасности Azure. Центр безопасности Azure предоставляет возможности подготовки отчетов, которые позволяют получать полезные сведения в реальном времени о состоянии безопасности развернутых ресурсов Azure.

- Мониторинг отсутствия Endpoint Protection в Центре безопасности Azure
- Развертывание расширения Майкрософт IaaSAntimalware по умолчанию для Windows Server
- Развертывание обнаружения угроз на серверах SQL Server
- Обновления системы, которые должны быть установлены на компьютерах
- Уязвимости в конфигурации безопасности на компьютерах должен быть восстановлен
- Уязвимости в базах данных SQL должен быть восстановлен
- Уязвимостей следует исправить с помощью решения для оценки уязвимостей

## <a name="711-712-and-713-separation-of-duties"></a>7.1.1. 7.1.2 и 7.1.3 разделение обязанностей

Наличие только одного владельца подписки Azure не позволит организовать административную избыточность. И наоборот, слишком большое число владельцев подписки Azure может увеличить вероятность бреши в системе безопасности через скомпрометированную учетную запись владельца. Эта схема помогает поддерживать на соответствующее число владельцев подписки Azure, назначив ["Политика Azure"](../../../policy/overview.md) определений, которые аудита количество владельцев для подписок Azure. Управление разрешениями владельца подписки помогает реализовать соответствующее разделение обязанностей.

- Должно быть больше одного владельца, назначенные вашей подписке
- Более 3 владельцы должны быть обозначены для вашей подписки 

## <a name="32-721-831a-and-831b-management-of-privileged-access-rights"></a>3.2, 7.2.1, 8.3.1.a и 8.3.1.b управления прав привилегированного доступа

В рамках этой схемы, которая помогает ограничить и контролировать права привилегированного доступа, назначив ["Политика Azure"](../../../policy/overview.md) определения для аудита внешние учетные записи с владельцем, записи и/или разрешения на чтение и учетных записей сотрудников с владельцем и/или записи разрешения, у которых нет включена многофакторная проверка подлинности. В Azure реализовано управление доступом на основе ролей (RBAC) для управления пользователями, которые обращаются к ресурсам Azure. Определение областей применения настраиваемых правил RBAC поможет вам убедиться в их необходимости и правильной реализации, так как в настраиваемых правилах RBAC легко ошибиться. Эта схема также назначает ["Политика Azure"](../../../policy/overview.md) определения для аудита использования проверки подлинности Azure Active Directory для серверов SQL Server. Упрощает управление разрешениями с помощью Azure Active Directory проверка подлинности и управления удостоверениями пользователей базы данных и другие Microsoft централизует  
Службы.
 
- Внешние учетные записи с правами владельца, которые должны быть удалены из подписки
- External accounts with write permissions should be removed from your subscription (Внешние учетные записи с разрешениями на запись следует удалять из подписки)
- Внешние учетные записи с разрешениями на чтение должен быть удален из подписки
- Многофакторной проверки Подлинности должен быть включен на учетные записи с правами владельца в подписке
- Многофакторной проверки Подлинности должно быть включенных учетных записей с разрешениями на запись для одной подписки
- Многофакторной проверки Подлинности должен быть включен на учетные записи с разрешениями на чтение в подписке
- Администратор Azure Active Directory должен быть подготовлен для серверов SQL Server
- Аудит использования настраиваемых правил RBAC

## <a name="812-and-815-least-privilege-and-review-of-user-access-rights"></a>8.1.2 и 8.1.5 наименьших прав доступа и его проверки права доступа пользователя

Azure также реализована на основе ролей управления доступом (RBAC) для помогает, которыми вы управляете у кого есть доступ к ресурсам в Azure. С помощью портала Azure вы можете просмотреть, у кого есть доступ к ресурсам Azure и какие им назначены разрешения. В рамках этой схемы назначает ["Политика Azure"](../../../policy/overview.md) определения аудит учетных записей, которые должны иметь приоритет для проверки, включая амортизированной учетные записи и внешние учетные записи с повышенными разрешениями.

- Устаревшие учетные записи должны быть удалены из подписки
- Устаревшие учетные записи с правами владельца, которые должны быть удалены из подписки
- Внешние учетные записи с правами владельца, которые должны быть удалены из подписки
- External accounts with write permissions should be removed from your subscription (Внешние учетные записи с разрешениями на запись следует удалять из подписки)
- Внешние учетные записи с разрешениями на чтение должен быть удален из подписки

## <a name="813-removal-or-adjustment-of-access-rights"></a>8.1.3 удаления или настройки прав доступа

Azure реализует управление доступом на основе ролей (RBAC) для управления, у кого есть доступ к ресурсам в Azure. С помощью Azure Active Directory и RBAC, можно обновить роли пользователей в соответствии с изменениями в организации. При необходимости можно запретить вход для учетных записей (или удалить их), что немедленно отменит все права доступа к ресурсам Azure. В рамках этой схемы назначает ["Политика Azure"](../../../policy/overview.md) определения для аудита амортизированной учетной записи, который должен применяться для удаления.

- Устаревшие учетные записи должны быть удалены из подписки
- Устаревшие учетные записи с правами владельца, которые должны быть удалены из подписки

## <a name="823ab-824ab-and-825-password-based-authentication"></a>8.2.3.a,b, 8.2.4.a,b и 8.2.5 проверки подлинности на основе пароля

В рамках этой схемы позволяет применить надежные пароли, назначив ["Политика Azure"](../../../policy/overview.md) определения, аудит виртуальных машин Windows, не применяют минимальное надежность и другие требования к паролю. Определение виртуальных машин, которые нарушают политику надежности паролей, помогает выполнить корректирующие действия и обеспечить соблюдение требований политики для паролей всех учетных записей пользователей на виртуальных машинах.

- [Предварительный просмотр]: Audit Windows VMs that do not have a maximum password age of 70 days
- [Предварительный просмотр]: Deploy requirements to audit Windows VMs that do not have a maximum password age of 70 days
- [Предварительный просмотр]: Audit Windows VMs that do not restrict the minimum password length to 14 characters
- [Предварительный просмотр]: Deploy requirements to audit Windows VMs that do not restrict the minimum password length to 14 characters
- [Предварительный просмотр]: Audit Windows VMs that allow re-use of the previous 24 passwords
- [Предварительный просмотр]: Deploy requirements to audit Windows VMs that allow re-use of the previous 24 passwords

## <a name="103-and-1054-audit-generation"></a>10.3 и 10.5.4 Создание записей аудита

Эта схема помогает обеспечить системы событий, назначив ["Политика Azure"](../../../policy/overview.md) определений, которые журнала параметров аудита на ресурсы Azure.
Журналы диагностики позволяют подробно проанализировать операции, выполненные с ресурсами Azure. Журналы Azure используют синхронизированные внутренние часы для создания записей о событиях с ресурсами с метками времени.

- Непроверенные серверов SQL в центре безопасности Azure
- Аудит параметров диагностики
- Аудит параметров аудита на уровне сервера SQL
- Развертывание аудита на серверах SQL Server
- Учетные записи хранения следует перенести в новые ресурсы Azure Resource Manager
- Виртуальные машины следует перенести в новые ресурсы Azure Resource Manager

## <a name="1236-and-1237-information-security"></a>12.3.6 и 12.3.7 информационной безопасности

В рамках этой схемы, которая помогает управлять и управления сетью, назначив ["Политика Azure"](../../../policy/overview.md) определений, которые аудита допустимых сетевых расположений и продуктов компании утвержденных, разрешенное для среды. Их можно настраивать по каждой компании с помощью параметров политики в каждом из этих политик.

- Допустимые расположения
- Разрешенные расположения для групп ресурсов
