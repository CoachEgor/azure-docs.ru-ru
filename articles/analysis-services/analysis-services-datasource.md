# <a name="data-sources-supported-in-azure-analysis-services"></a>Источники данных, поддерживаемые в службах Azure Analysis Services

Источники данных и разъемы, отображаемые в Get Data или Table Import Wizard в Visual Studio с проектами аналитических служб, отображаются как для служб анализа Azure, так и для служб анализа серверов S'L. Однако не все источники данных и разъемы, отображаемые, поддерживаются в службах анализа Azure. Типы источников данных, к которые можно подключиться, зависят от многих факторов, таких как уровень совместимости моделей, доступные разъемы данных, тип проверки подлинности и поддержка шлюза данных на местах. В следующих таблицах описывают поддерживаемые источники данных для служб анализа Azure.

## <a name="understanding-providers"></a>Понимание поставщиков

При создании табликовых проектов 1400 и более высоких моделей в Visual Studio по умолчанию не указывается поставщик данных при подключении к источнику данных с помощью **Get Data.** Tabular 1400 и более высокие модели используют разъемы [Power Query](/power-query/power-query-what-is-power-query) для управления соединениями, запросами данных и гибридными приложениями между источником данных и аналитическими службами. Иногда они называются *структурированным* соединением источника данных в настройках свойств соединения соединения, установленные для вас. Однако можно включить устаревшие источники данных. При включении можно использовать **Table Import Wizard** для подключения к определенным источникам данных, традиционно поддерживаемым в таблике 1200 и более низких моделях в качестве *устаревших*или *поставщиков* данных. При указании в качестве источника данных поставщика можно указать конкретного поставщика данных и другие расширенные свойства соединения. Например, можно подключиться к штату хранилища серверных данных S'L или даже к базе данных Azure S'L в качестве устаревшего источника данных. Затем можно выбрать драйвер OLE DB для поставщика данных сервера S'L Server MSOLEDBS'L. В этом случае выбор поставщика данных OLE DB может обеспечить повышение производительности по сравнению с разъемом Power Query. 

При использовании Мастера импорта таблицы в Visual Studio для подключения к любому источнику данных требуется поставщик данных. Для вас выбран поставщик данных по умолчанию. При необходимости можно изменить поставщика данных. Тип поставщика, который вы выбираете, может зависеть от производительности, использует ли модель хранилище в памяти или Direct'ery, и на какой платформе Аналитических служб вы развертываете модель.

### <a name="specify-provider-data-sources-in-tabular-1400-and-higher-model-projects"></a>Укажите источники данных поставщика в табулярных проектах 1400 и более высоких моделей

Для того, чтобы источники данных поставщика, в Visual Studio, нажмите **Инструменты** > **Параметры** > **Анализ Услуги Tabular** > **Data Import**, выберите **Включить устаревшие источники данных.**

![Включить устаревшие источники данных](media/analysis-services-datasource/aas-enable-legacy-datasources.png)

С включенными устаревшими источниками данных, в **Tabular Model Explorer,** правой кнопкой кнопкой мыши **Источники** > данных**импортируют из источника данных (Наследие)**.

![Устаревшие источники данных в Tabular Model Explorer](media/analysis-services-datasource/aas-import-legacy-datasources.png)

Так же, как с табликами 1200 типовых проектов, используйте **Table Import Wizard** для подключения к источнику данных. На странице подключения нажмите **Расширенный**. Укажите поставщика данных и другие параметры подключения в **Set Advanced Properties.**

![Устаревшие источники данных Расширенные свойства](media/analysis-services-datasource/aas-import-legacy-advanced.png)


## <a name="impersonation"></a>Олицетворение
В некоторых случаях может быть необходимо указать другую учетную запись олицетворения. Учетная запись олицетворения может быть указана в Visual Studio или SSMS.

Для локальных источников данных:

* При использовании проверки подлинности SQL олицетворением должна быть учетная запись службы.
* При использовании проверки подлинности Windows задайте пользователя и пароль Windows. Для SQL Server проверка подлинности Windows с использованием конкретной учетной записи олицетворения поддерживается только для моделей данных в памяти.

Для облачных источников данных

* При использовании проверки подлинности SQL олицетворением должна быть учетная запись службы.

## <a name="oauth-credentials"></a>Учетные данные OAuth

Для табличных моделей на уровне 1400 и более высокого уровня совместимости, используя режим памяти, базу данных Azure S'L, Azure Synapse Analytics (хранилище данных S'L), Dynamics 365 и SharePoint List поддерживают учетные данные OAuth. Службы анализа azure управляют обновлением токенов для источников данных OAuth, чтобы избежать тайм-аутов для длительных операций обновления. Для создания действительных токенов установите учетные данные с помощью SSMS.

Режим прямого запроса не поддерживается учетными данными OAuth.
