# <a name="performance-considerations-for-azure-netapp-files"></a>Рекомендации по ускорению Azure NetApp Files

[Предельное значение пропускной способности](azure-netapp-files-service-levels.md) тома определяется сочетание квот, назначаемое тому и выбранный уровень службы. При создании планов работы о службе файлов Azure NetApp, необходимо понять несколько соображений. 

## <a name="quota-and-throughput"></a>Квоты и пропускной способности  

Предельная пропускная способность — только один определитель фактическую производительность, будет обеспечиваться.  

Вопросы производительности обычно хранилища, включая чтения и записи набора, в общая производительность доставки влияет размер передачи, случайный или последовательный шаблоны и множество других факторов.  

Максимальная пропускная способность эмпирический, наблюдалась при тестировании — 4500 Миб/с.  На уровне хранилища уровня "премиум" Квота на количество из 70.31 Тиб подготовит предельное значение пропускной способности, достаточно велик, чтобы достичь этого уровня производительности.  

Если вы рассматриваете назначение квот суммы за пределы 70.31 Тиб тома, дополнительная Квота может быть назначен тома для хранения дополнительных данных. Тем не менее добавлены квоты не приведет к дальнейшей увеличению фактическая пропускная способность.  

См. в разделе [тест производительности для службы файлов Azure NetApp](azure-netapp-files-performance-benchmarks.md) Дополнительные сведения.

## <a name="overprovisioning-the-volume-quota"></a>Избыточная Подготовка Квота на количество

Если производительность рабочей нагрузки привязан ограничения пропускной способности, возможна используют избыточную подготовку Квота на количество задали более высокий уровень пропускной способности и добиться более высокой производительности.  

Например, если в томе хранилища уровня "премиум" имеет только 500 Гиб данных, но требует 128 Миб/с пропускной способности, можно установить квоту 2 тиб таким образом, чтобы соответствующим образом устанавливается уровень пропускной способности (до 64 Миб/с на ТБ * 2 Тиб = 128 Миб/с).  

Если вы постоянно используют избыточную подготовку том для достижения более высокую пропускную способность, рекомендуется использовать более высокий уровень службы.  В приведенном выше примере, можно выполнить такое же ограничение пропускной способности с половину Квота на количество, вместо этого использовать уровень Ultra хранилища (128 Миб/с на Тиб * 1 Тиб = 128 Миб/с).

## <a name="dynamically-increasing-or-decreasing-volume-quota"></a>Динамически увеличить или уменьшить Квота на количество

Если требований к производительности являются временными, или если после повышения требований к производительности за фиксированный период времени, можно динамически увеличить или уменьшить Квота на количество мгновенно настроить ограничения пропускной способности.  Обратите внимание на следующие моменты: 

* Квота на количество можно увеличивать или уменьшать без каких-либо для приостановки операций ввода-ВЫВОДА, и доступ к тому не будет прерван или не влияет.  

    Квоты можно настроить во время активной транзакции операций ввода-вывода для тома.  Обратите внимание, что квота на количество никогда не может быть уменьшен меньше объема логические данные, хранящиеся на томе.

* Когда квота на количество изменяется, соответствующее изменение в предельное значение пропускной способности практически мгновенно. 

    Изменения не прерываний и не влияет на доступ к тому или ввода-вывода.  

* Настройка Квота на количество требует изменения размера пула ресурсов.  

    Размер емкости пула можно настроить динамически и без ущерба для доступности тома или ввода-вывода.
