# <a name="overview-of-offline-backup"></a>Общие сведения о автономной архивации

В этой статье приводятся общие сведения о автономной архивации.

Первоначальное полное резервное копирование в Azure, как правило, переносит большие объемы данных в оперативный режим и требует больше пропускной способности сети по сравнению с последующими резервными копиями В удаленных офисах или центрах обработки данных в определенных географических регионах не всегда достаточно пропускной способности сети. По этой причине эти начальные резервные копии занимают несколько дней. В течение этого времени резервные копии непрерывно используют ту же сеть, которая была подготовлена для приложений, запущенных в локальном центре обработки данных.

Azure Backup поддерживает автономную архивацию, которая передает исходные данные резервных копий в автономном режиме без использования пропускной способности сети. Он обеспечивает механизм копирования резервных копий данных на физические устройства хранения. Затем устройства поставляются в ближайший центр обработки данных Azure и передаются в хранилище служб восстановления. Этот процесс обеспечивает надежную пересылку данных резервных копий без использования пропускной способности сети.

## <a name="offline-backup-options"></a>Параметры автономного резервного копирования

Автономное резервное копирование предлагается в двух режимах в зависимости от принадлежности запоминающих устройств.

- Автономное резервное копирование на основе Azure Data Box (Предварительная версия)
- Автономное резервное копирование на основе службы импорта и экспорта Azure

## <a name="offline-backup-based-on-azure-data-box-preview"></a>Автономное резервное копирование на основе Azure Data Box (Предварительная версия)

В настоящее время этот режим поддерживается агентом Службы восстановления Microsoft Azure (MARS) в режиме предварительной версии. Этот вариант использует преимущества [Azure Data Box](https://azure.microsoft.com/services/databox/) для предоставления защищенных от Майкрософт, безопасных устройств перемещения с помощью соединителей USB в центр обработки данных или удаленный офис. Данные резервной копии записываются непосредственно на эти устройства. Этот параметр сохраняет усилия, необходимые для приобретения собственных дисков и соединителей, совместимых с Azure, или для подготовки временного хранилища в качестве промежуточного расположения. Корпорация Майкрософт также обрабатывает комплексную логистику перемещения, которую можно отнести к портал Azure. 

Здесь показана архитектура, описывающая перемещение данных резервных копий с помощью этого параметра.

![Архитектура Data Box Azure Backup](./media/offline-backup-overview/azure-backup-databox-architecture.png)

Ниже приведена сводка архитектуры.

1. Azure Backup непосредственно копирует данные резервных копий на эти предварительно настроенные устройства.
2. Затем эти устройства можно отправить обратно в центр обработки данных Azure.
3. Azure Data Box копирует данные в учетную запись хранения, принадлежащую заказчику.
4. Azure Backup автоматически копирует данные резервных копий из учетной записи хранения в указанное хранилище служб восстановления. Добавочное оперативное резервное копирование запланировано.

Чтобы использовать автономное резервное копирование на основе Azure Data Box, см. раздел [автономная архивация с помощью Azure Data Box](offline-backup-azure-data-box.md).

## <a name="offline-backup-based-on-the-azure-importexport-service"></a>Автономное резервное копирование на основе службы импорта и экспорта Azure

Этот параметр поддерживается Microsoft Azure Backup Server (MABS), System Center Data Protection Manager (DPM) DPM-а и агентом MARS. В нем используется [Служба импорта и экспорта Azure](https://docs.microsoft.com/azure/storage/common/storage-import-export-service). Вы можете передавать исходные данные резервных копий в Azure с помощью собственных дисков и соединителей, совместимых с Azure. При таком подходе необходимо подготовить временное хранилище, которое называется промежуточным расположением, и использовать готовые служебные программы для форматирования и копирования данных резервных копий на диски, принадлежащие заказчику. 

Здесь показана архитектура, описывающая перемещение данных резервных копий с помощью этого параметра.

![Azure Backup архитектура службы импорта и экспорта](./media/offline-backup-overview/azure-backup-import-export.png)

Ниже приведена сводка архитектуры.

1. Вместо отправки данных резервных копий по сети Azure Backup записывает данные резервных копий в промежуточное расположение.
2. Данные в промежуточном расположении записываются на один или несколько дисков SATA с помощью настраиваемой служебной программы.
3. В рамках подготовительной работы программа создает задание импорта Azure. Диски SATA поставляются в ближайший центр обработки данных Azure и ссылаются на задание импорта для подключения действий.
4. В центре обработки данных Azure данные с дисков копируются в учетную запись хранилища Azure.
5. Azure Backup копирует данные резервных копий из учетной записи хранения в хранилище служб восстановления. Добавочное резервное копирование запланировано.

Чтобы использовать автономное резервное копирование на основе службы импорта и экспорта Azure с агентом MARS, см. раздел [Рабочий процесс автономного резервного копирования в Azure Backup](https://docs.microsoft.com/azure/backup/backup-azure-backup-import-export).

Сведения о том, как использовать эту же программу с MABS или DPM-A, см. в разделе [Рабочий процесс автономного резервного копирования для DPM и Azure Backup Server](https://docs.microsoft.com/azure/backup/backup-azure-backup-server-import-export-).
