# <a name="back-up-system-state-and-restore-to-bare-metal-by-using-azure-backup-server"></a>Резервное копирование состояния системы и восстановление до голого металла с помощью сервера резервного копирования Azure

Azure Backup Server создает резервную копию состояния системы и защищает восстановление исходного состояния системы (BMR).

* **Резервное копирование состояния системы:** Резервное копирование файлов операционной системы. Эта резервная система позволяет восстановить, когда компьютер запускается, но системные файлы и реестр потеряны. Резервное копирование состояния системы включает в себя следующие элементы:
  * член домена (загрузочные файлы, база данных регистрации класса COM+, реестр);
  * контроллер домена (Windows Server Active Directory (NTDS), загрузочные файлы, база данных регистрации класса COM+, реестр, системный том (SYSVOL));
  * компьютер, на котором работают службы кластеров (метаданные сервера кластера);
  * компьютер, на котором работают службы сертификации (данные сертификата).
* **Резервное копирование:** Резервное копирование файлов операционной системы и всех данных о критических объемах, за исключением пользовательских данных. По определению резервная копия исходного состояния системы включает резервную копию состояния системы. Она обеспечивает защиту на случай, если компьютер не запускается и вам нужно все восстановить.

## <a name="how-system-state-backup-works"></a>Как работает резервное копирование состояния системы

Когда выполняется резервное копирование состояния системы, Backup Server обращается к системе архивации данных Windows Server, чтобы запросить резервную копию состояния системы сервера. По умолчанию Backup Server и система архивации данных Windows Server используют диск с наибольшим объемом свободного места. Информация об этом диске сохраняется в файле *PSDataSourceConfig.xml.* 

Можно настроить диск, который используется Backup Server для резервного копирования состояния системы: 

1. На защищенном сервере перейдите на *C:'Программные файлы (S:Program Files):Менеджер по защите данных Microsoft* 
1. Откройте файл *PSDataSourceConfig.xml* для редактирования. 
1. Замените значение \<FilesToProtect\> буквой диска. 
1. Сохраните файл и закройте его. 

Если группа защиты настроена на защиту состояния компьютера системы, запустите проверку согласованности. Если сгенерировано оповещение, выберите **группу защиты в** оповещении, а затем заполните страницы в мастере. После этого запустите проверку согласованности еще раз.

Если сервер защиты находится в кластере, в качестве диска, свободного пространства которого можно выбрать кластерный диск, может быть выбран кластерный диск. Если это владение диском переключено на другой узла и запускается резервное копирование состояния системы, то диск недоступен и резервное копирование выходит из строя. В этом сценарии измените *PSDataSourceConfig.xml,* чтобы указать на локальный диск.

Далее резервное копирование Windows Server создает папку под названием *WindowsImageBackup* в корне папки восстановления. Когда система архивации данных Windows Server создает резервную копию, все данные сохраняются в этой папке. Когда резервное копирование завершается, файл передается на компьютер Backup Server. Обратите внимание на следующие сведения:

* Эта папка и ее содержимое не очищаются при завершении резервного копирования или передачи. Лучший способ думать об этом является то, что пространство зарезервировано для следующего времени резервного копирования заканчивается.
* Папка создается для каждого резервного копирования. Метка времени и даты отражает время последнего резервного копирования состояния системы.

## <a name="how-bmr-backup-works"></a>Как работает резервное копирование BMR

Для восстановления исходного состояния системы (в том числе из резервной копии состояния системы) файл резервной копии сохраняется непосредственно в общую папку на компьютере Backup Server. Он не сохраняется в папке на защищенном сервере.

Backup Server вызывает систему архивации данных Windows Server и предоставляет доступ к тому реплики этой резервной копии для восстановления исходного состояния системы. В этом случае не требуется резервное копирование сервера Windows, чтобы использовать диск, который имеет наибольшее свободное пространство. Вместо этого Backup Server использует общую папку, которая была создана для задания.

Когда резервное копирование завершается, файл передается на компьютер Backup Server. Логи хранятся в *C: »Окна »Журналы »WindowsServerBackup*.

## <a name="prerequisites-and-limitations"></a>Предварительные требования и ограничения

* Восстановление исходного состояния системы не поддерживается для компьютеров под управлением Windows Server 2003 и для компьютеров под управлением клиентской операционной системы.

* Невозможно защитить состояние системы и восстановление исходного состояния системы для одного компьютера в разных группах защиты.

* Backup Server не может защитить себя для восстановления исходного состояния системы.

* Краткосрочная защита ленты (от диска к ленте, или D2T) не поддерживается для BMR. Поддерживается длительное хранение ленты (от диска к диску к ленте, или D2D2T).

* Для защиты восстановления исходного состояния на защищенном компьютере должна быть установлена система архивации данных Windows Server.

* Для защиты BMR, в отличие от защиты состояния системы, резервный сервер не имеет требований к пространству на защищенном компьютере. Система архивации данных Windows Server напрямую передает резервные копии на компьютер Backup Server. Задание передачи резервных копий не отображается в представлении **Задания** Backup Server.

* Backup Server резервирует 30 ГБ дискового пространства на томе реплики для восстановления исходного состояния системы. Вы можете изменить выделение этого пространства на странице **распределения дисков** в «Изумии группы изменений». Или вы можете использовать Get-DatasourceDiskAllocation и Set-DatasourceDiskAllocation PowerShell cmdlets. На томе точки восстановления защита восстановления исходного состояния системы требует зарезервировать 6 ГБ на пять дней.
  * Нельзя уменьшить размер объема реплики до менее 15 ГБ.
  * Backup Server не вычисляет размер источника данных восстановления исходного состояния системы. Предполагается, что это значение равно 30 ГБ для всех серверов. Измените значение с учетом размера резервных копий для восстановления исходного состояния системы, которые планируется создать в вашей среде. Можно приблизительно рассчитать размер резервного копирования BMR как сумму используемого пространства на всех критических томах. Критические тома — это загрузочный том, системный том и том данных состояния системы, например Active Directory.

* Если вы перейдете от защиты состояния системы к защите BMR, то защита BMR требует меньше места на *томе точки восстановления.* Тем не менее, дополнительное пространство на томе не рекультиваемы. Можно вручную уменьшить размер громкости на странице **распределения дисков Modify** Of. Или вы можете использовать Get-DatasourceDiskAllocation и Set-DatasourceDiskAllocation PowerShell cmdlets.

    Если вы перейдете от защиты состояния системы к защите BMR, то защита BMR требует больше места на *объеме реплики.* Этот том увеличивается автоматически. Если вы хотите изменить выделение пространства по умолчанию, используйте смдлет Modify-DiskAllocation PowerShell.

* Если вы перейдете от защиты BMR к защите состояния системы, то вам нужно больше места на томе точки восстановления. Backup Server может попытаться автоматически увеличить том. Если пул хранения не имеет достаточного места, возникает ошибка.

    Если вы перейдете от защиты BMR к защите состояния системы, то вам нужно пространство на защищенном компьютере. Вам нужно пространство, потому что защита состояния системы сначала записывает реплику на локальный компьютер, а затем передает реплику на компьютер Backup Server.

## <a name="before-you-begin"></a>Перед началом

1. **Разверните Azure Backup Server**. Убедитесь, что развертывание Backup Server выполнено правильно. Дополнительные сведения см. в разделе:
    * [Системные требования для Azure Backup Server](https://docs.microsoft.com/system-center/dpm/install-dpm#setup-prerequisites).
    * [Матрица защиты Azure Backup Server](backup-mabs-protection-matrix.md).

1. **Настройка хранилища.** Данные резервных копий можно хранить на диске, на магнитной ленте или в облаке Azure. Дополнительные сведения см. в статье о [подготовке хранилища данных](https://docs.microsoft.com/system-center/dpm/plan-long-and-short-term-data-storage).

1. **Настройте агент защиты**. Установите агент защиты на компьютере, резервную копию которого вы хотите создать. Дополнительные сведения см. в статье о [развертывании агента защиты DPM](https://docs.microsoft.com/system-center/dpm/deploy-dpm-protection-agent).

## <a name="back-up-system-state-and-bare-metal"></a>Резервное копирование состояния системы и восстановление исходного состояния системы

Для резервного копирования системного состояния и голого металла:

1. Чтобы открыть группу «Создать новую группу защиты» в консоли администратора резервного копирования, выберите**группу действий** >  **защиты** > **Create Protection Group.**

1. На странице **Выбор типа группы защиты** выберите **Серверы** и нажмите кнопку **Далее**.

1. На странице **Выбор элементов группы** разверните узел компьютера и выберите **BMR** или **Состояние системы**.

    Помните, что вы не можете защитить восстановление исходного состояния системы и состояние системы для одного компьютера в разных группах защиты. Кроме того, при выборе BMR состояние системы включается автоматически. Дополнительные сведения см. в статье о [развертывании групп защиты](https://docs.microsoft.com/system-center/dpm/create-dpm-protection-groups).

1. На странице **«Выберите метод защиты данных»** выберите, как обрабатывать краткосрочное резервное копирование и долгосрочное резервное копирование. 

    Краткосрочное резервное копирование всегда на диске во-первых, с возможностью резервного копирования с диска на Azure с помощью Azure резервного копирования (краткосрочные или долгосрочные). Альтернативой для долгосрочного резервного копирования в облако является настройка долгосрочного резервного копирования на автономное устройство с магнитными лентами или библиотеку магнитных лент, подключенную к Backup Server.

1. На странице **«Выбрать краткосрочные цели»** выберите, как настроить резервное копирование на диске:
    * Для **диапазона удержания**выберите, как долго хранить данные на диске.
    * Для **частоты синхронизации**выберите частоту выполнения дополнительного резервного копирования на диск. Если вы не хотите устанавливать интервал резервного копирования, вы можете выбрать **перед точкой восстановления.** Сервер резервного копирования запустит полное резервное копирование непосредственно перед тем, как будет назначена каждая точка восстановления.

1. Если вы хотите хранить данные на ленте для долгосрочного хранения, то на странице **«Указать долгосрочные цели»** выберите, как долго хранить данные ленты (от 1 до 99 лет).
    1. Для **частоты резервного копирования**выберите частоту выполнения резервного копирования на ленту. Частота основана на выбранном диапазоне удержания:
        * Когда диапазон удержания составляет от 1 до 99 лет, вы можете резервное копирование ежедневно, еженедельно, раз в две недели, ежемесячно, ежеквартально, от полугода, или ежегодно.
        * Когда диапазон удержания составляет от 1 до 11 месяцев, вы можете резервное копирование ежедневно, еженедельно, раз в две недели или ежемесячно.
        * Когда диапазон удержания составляет от 1 до 4 недель, вы можете резервное копирование ежедневно или еженедельно.

    1. На странице **«Выберите ленту» и «Подробности библиотеки»** выберите ленту и библиотеку для использования. Также выберите, следует ли сжимать и шифровать данные.

1. На странице **распределения дисков обзора** просмотрите дисковое пространство пула хранения, доступное для группы защиты.

    * **Общий размер данных** — это размер данных, для которых необходимо создать резервную копию.
    * **Дисковое пространство, которое нужно подготовить на Azure Backup Server** — пространство, которое Backup Server рекомендует выделить для группы защиты. Резервный сервер использует эти настройки для выбора идеального объема резервного копирования. Можно отсеять выбор громкости резервного копирования в **деталях распределения диска.**
    * Выберите в раскрывающемся меню предпочитаемый объем хранилища для рабочих нагрузок. Вы можете изменить значения для параметров **Общий объем** и **Доступный объем** в области **Доступный объем на диске**. Недооцененное пространство — это объем хранилища, который сервер backup Server предлагает добавить в объем для обеспечения бесперебойного резервного копирования.

1. На странице **«Выбрать метод создания реплики»** выберите, как обрабатывать начальную репликацию полных данных. 

    Если вы выбираете репликацию по сети, рекомендуем выполнять ее в часы с наименьшей загрузкой. Для больших объемов данных или неоптимальных условий сети лучше выбрать автономную репликацию данных с использованием съемных носителей.

1. На странице **«Проверка соответствия выбору»** выберите, как автоматизировать проверки согласованности. 

    Вы можете запускать проверку по расписанию или только в том случае, когда данные реплики становятся несогласованными. Если вы не хотите настраивать автоматическую проверку согласованности, то вы можете запустить ручную проверку в любое время. В области **Защита** консоли администратора Backup Server щелкните правой кнопкой мыши группу защиты и выберите пункт **Выполнить проверку согласованности**.

1. Если вы решили создать резервную связь с помощью резервного копирования Azure на странице **Данных о защите от** уточнения, выберите рабочие нагрузки, которые требуется для резервного копирования в Azure.

1. На странице **«Расписание резервного копирования» —расписание «Указать на интернет»** выберите, как часто происходит постепенное резервное копирование в Azure. 

    Вы можете запланировать резервное копирование для выполнения каждый день, неделю, месяц и год. Вы также можете выбрать время и дату выполнения резервных копирований. Максимальная частота резервного копирования — дважды в день. Каждый раз при запуске резервного копирования в Azure создается точка восстановления данных из копии данных резервного копирования, хранящихся на диске Сервера резервного копирования.

1. На странице **Политики удержания данных «Указать онлайн»** выберите, как в Azure хранятся точки восстановления, создаваемые из ежедневных, еженедельных, ежемесячных и ежегодных резервных откопов.

1. На странице **Выберите репликацию в сети** укажите, как должна выполняться первоначальная полная репликация данных. 

    Вы можете реплицировать по сети или резервное копирование в автономном режиме (оффлайн посев). Резервное копирование в автономном режиме использует функцию Azure Import. Дополнительные сведения см. в статье [Автономное резервное копирование в службе архивации Azure](offline-backup-azure-data-box.md).

1. На странице **Сводка** проверьте выбранные параметры. Когда вы выберете **Создать группу**, будет выполнена начальная репликация данных. Когда репликация данных завершается на странице **"Статус",** статус группы защиты **в порядке.** Затем резервное копирование происходит в соответствии с настройками группы защиты.

## <a name="recover-system-state-or-bmr"></a>Восстановление состояния системы или восстановление исходного состояния системы

Состояние системы или исходное состояние системы можно восстановить в сетевую папку. Если вы резервное копирование BMR, а затем использовать Windows Восстановление среды (WinRE), чтобы запустить систему и подключить его к сети. Затем воспользуйтесь системой архивации данных Windows Server для восстановления из сетевой папки. Если вы резервное копирование состояния системы, то просто используйте резервное копирование Windows Server для восстановления от местоположения сети.

### <a name="restore-bmr"></a>Восстановление исходного состояния системы

Для выполнения восстановления на компьютере Резервного сервера:

1. В панели **восстановления** найдите компьютер, который вы хотите восстановить. Затем выберите **Bare Metal Recovery.**

1. Доступные точки восстановления отмечены на календаре полужирным шрифтом. Выберите дату и время точки восстановления, которую вы хотите использовать.

1. На странице **Select Recovery Type** выберите Copy в **папку сети.**

1. На странице **«Определить пункт назначения»** выберите пункт назначения для скопированных данных. 

    Помните, что пункт назначения должен иметь достаточно места для данных. Мы рекомендуем создать новую папку для назначения.

1. На странице **«Параметры восстановления уточнения»** выберите параметры безопасности. Затем выберите, следует ли использовать аппаратные снимки на основе системхранения хранения данных (SAN) для более быстрого восстановления. Эта опция доступна только в том случае, если:
    * У вас есть SAN, который предоставляет эту функциональность. 
    * Вы можете создать и разделить клон, чтобы сделать его writable.
    * Защищенный компьютер и компьютер Backup Server подключены к одной и той же сети.

1. Настройте параметры уведомлений. 
1. На странице **Подтверждение** нажмите кнопку **Восстановить**.

Для настройки местоположения общего налажива:

1. В расположении восстановления откройте папку с резервной копией.

1. Поделитесь папкой, которая на один уровень выше *WindowsImageBackup,* так что корень общей папки папки *WindowsImageBackup* папку. 

    Если вы не разделите эту папку, восстановление не найдется резервная. Чтобы подключиться с помощью WinRE, вам нужна доля, к которым вы можете получить доступ в WinRE с правильным IP-адресом и учетными данными.

Для восстановления системы:

1. Запустите компьютер, на котором вы хотите восстановить изображение с помощью WINDOWS DVD для системы, которую вы восстанавливаете.

1. На первой странице проверьте настройки языка и языка. На странице **Установка** выберите **Восстановить компьютер**.

1. На странице **Параметры восстановления системы** выберите параметр **Restore your computer using a system image that you created earlier** (Восстановление компьютера с помощью созданного ранее образа системы).

1. На странице **Выбор архивного образа системы** выберите **Выбор архивного образа системы** > **Дополнительно** > **Искать образ системы в сети**. Если появится предупреждение, выберите **Да**. Откройте общую папку, введите учетные данные, а затем выберите точку восстановления. Система сканирует для конкретных резервных ups, которые доступны в этой точке восстановления. Выберите точку восстановления, которую следует использовать.

1. На **выбор, как восстановить страницу резервного копирования,** выберите **Формат и диски перераздела.** На следующей странице проверьте настройки.

1. Чтобы начать восстановление, нажмите кнопку **Готово**. Требуется перезапуск.

### <a name="restore-system-state"></a>Восстановление состояния системы

Для выполнения восстановления в сервере резервного копирования:

1. В области **Восстановление** найдите компьютер, который необходимо восстановить, затем выберите **Восстановление исходного состояния системы**.

1. Доступные точки восстановления отмечены на календаре полужирным шрифтом. Выберите дату и время точки восстановления, которую вы хотите использовать.

1. На странице **Select Recovery Type** выберите Copy в **папку сети.**

1. На странице **«Определить пункт назначения»** выберите, где можно скопировать данные. 

    Помните, что выбранный вами пункт назначения должен иметь достаточно места для данных. Мы рекомендуем создать новую папку для назначения.

1. На странице **«Параметры восстановления уточнения»** выберите параметры безопасности. Затем выберите, следует ли использовать сани-аппаратные снимки для более быстрого восстановления. Эта опция доступна только в том случае, если: 
    * У вас есть SAN, который предоставляет эту функциональность.
    * Вы можете создать и разделить клон, чтобы сделать его writable. 
    * Защищенный компьютер и сервер Резервного копирования подключены к одной сети.

1. Настройте параметры уведомлений. 
1. На странице **Подтверждение** нажмите кнопку **Восстановить**.

Для запуска резервного копирования сервера Windows:

1. Выберите **действия** > **Восстановление** > **этого сервера** > **далее**.

1. Выберите **Другой сервер**, выберите страницу **Выбор типа размещения**, а затем параметр **Удаленная общая папка**. Введите путь к папке, содержащей точку восстановления.

1. На странице **Выбор типа восстановления** выберите параметр **Состояние системы**.

1. На странице **Выбор расположения для восстановления состояния системы** выберите **Исходное расположение**.

1. На странице **Подтверждение** нажмите кнопку **Восстановить**. 

1. После восстановления перезапустите сервер.

Вы также можете запустить восстановление состояния системы в запросе команды: 

1. Запустите резервное копирование Windows Server на компьютере, который вы хотите восстановить. 

1. Для получения идентификатора версии в командной строке введите следующую команду: 

   ```wbadmin get versions -backuptarget \<servername\sharename\>```

1. Используйте идентификатор версии для запуска восстановления состояния системы. В командной строке введите: 

    ```wbadmin start systemstaterecovery -version:<versionidentified> -backuptarget:<servername\sharename>```

1. Подтвердите, что вы хотите начать восстановление. Вы увидите ход процесса в окне командной строки. Журнал восстановления создан. 

1. После восстановления перезапустите сервер.
