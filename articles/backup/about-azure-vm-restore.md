# <a name="about-azure-vm-restore"></a>Сведения о восстановлении виртуальных машин Azure

В этой статье описывается, как [служба Azure Backup](https://docs.microsoft.com/azure/backup/backup-overview) восстанавливает виртуальные машины Azure (ВМ). Существует ряд вариантов восстановления. Мы обсудим различные сценарии, которые они поддерживают.

## <a name="concepts"></a>Основные понятия

- **Точка восстановления** (также известная как **точка восстановления**). точка восстановления — это копия исходных данных, для которых выполняется резервное копирование.

- **Уровень (моментальный снимок и хранилище)**. Резервное копирование виртуальных машин Azure происходит в два этапа:

  - На этапе 1 создаваемый моментальный снимок сохраняется вместе с диском. Это называется **уровнем моментального снимка**. Восстановление уровня моментальных снимков выполняется быстрее (чем Restore из хранилища), так как они исключают время ожидания копирования моментальных снимков в хранилище перед запуском восстановления. Поэтому восстановление из уровня моментального снимка также называется [мгновенным восстановлением](https://docs.microsoft.com/azure/backup/backup-instant-restore-capability).
  - На этапе 2 моментальный снимок передается и сохраняется в хранилище, управляемом службой Azure Backup. Это называется **уровнем хранилища**.

- **Восстановление в исходном расположении (OLR)**. восстановление выполняется с точки восстановления на ИСХОДную виртуальную машину Azure, из которой были созданы резервные копии, и заменяются состоянием, хранящимся в точке восстановления. Это заменяет диск операционной системы и диски данных исходной виртуальной машины.

- **Восстановление альтернативного расположения (ALR)**. восстановление выполняется с точки восстановления на сервере, отличном от исходного сервера, на котором были созданы резервные копии.

- **Восстановление на уровне элементов (ILR):** Восстановление отдельных файлов или папок в виртуальной машине из точки восстановления

- **Доступность (типы репликации)**. Azure Backup предлагает два типа репликации для обеспечения высокой доступности хранилища и данных:
  - [Локально избыточное хранилище (LRS)](../storage/common/storage-redundancy-lrs.md) реплицирует данные три раза (они создают три копии данных) в единице масштабирования хранилища в центре обработки данных. Все копии данных находятся в одном регионе. LRS — это экономный способ защиты данных от сбоев локального оборудования.
  - [Геоизбыточное хранилище (GRS)](../storage/common/storage-redundancy-grs.md) — это рекомендуемый вариант репликации по умолчанию. Геоизбыточное хранилище реплицирует данные в дополнительный регион, который находится в сотнях километров от основного расположения данных. Этот вариант дороже, чем LRS, но GRS обеспечивает высокий уровень устойчивости данных, даже если региональный центр обработки данных выйдет из строя.

- **Восстановление между регионами (КРР)**. в качестве одного из [вариантов восстановления](https://docs.microsoft.com/azure/backup/backup-azure-arm-restore-vms#restore-options). восстановление между регионами (КРР) позволяет восстановить виртуальные машины Azure в дополнительном регионе, который является [парным регионом Azure](https://docs.microsoft.com/azure/best-practices-availability-paired-regions#what-are-paired-regions).

## <a name="restore-scenarios"></a>Сценарии восстановления

![Сценарии восстановления ](./media/about-azure-vm-restore/recovery-scenarios.png)

| **Сценарий**                                                 | **Что делать**                                             | **Назначение**                                              |
| ------------------------------------------------------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |
| [Восстановление для создания новой виртуальной машины](https://docs.microsoft.com/azure/backup/backup-azure-arm-restore-vms) | Восстанавливает всю виртуальную машину до OLR (если исходная виртуальная машина по-прежнему существует) или ALR | <li> Если исходная виртуальная машина потеряна или повреждена, можно восстановить всю виртуальную машину.  <li> Вы можете создать копию виртуальной машины.  <li> Вы можете выполнить детализацию восстановления для аудита или соответствия требованиям.  <li> Этот параметр не будет работать для виртуальных машин Azure, созданных из образов Marketplace (если они недоступны из-за истечения срока действия лицензии). |
| [Восстановление дисков виртуальной машины](https://docs.microsoft.com/azure/backup/backup-azure-arm-restore-vms#restore-disks) | Восстановление дисков, подключенных к виртуальной машине                             |  Все диски: этот параметр создает шаблон и восстанавливает диск. Этот шаблон можно изменить с помощью специальных конфигураций (например, групп доступности) в соответствии с вашими требованиями, а затем использовать шаблон и восстановить диск для повторного создания виртуальной машины. |
| [Восстановление конкретных файлов в виртуальной машине](https://docs.microsoft.com/azure/backup/backup-azure-restore-files-from-vm) | Выберите пункт точка восстановления, обзор, выберите файлы и восстановите их в той же (или совместимой) ОС, что и резервная копия виртуальной машины. |  Если известно, какие именно файлы нужно восстановить, используйте этот параметр вместо восстановления всей виртуальной машины. |
| [Восстановление зашифрованной виртуальной машины](https://docs.microsoft.com/azure/backup/backup-azure-vms-encryption) | На портале восстановите диски, а затем с помощью PowerShell создайте виртуальную машину. | <li> [Зашифрованная виртуальная машина с Azure Active Directory (AAD)](https://docs.microsoft.com/azure/virtual-machines/windows/disk-encryption-windows-aad)  <li> [Зашифрованная виртуальная машина без AAD](https://docs.microsoft.com/azure/virtual-machines/windows/disk-encryption-windows) <li> [Зашифрованная виртуальная машина *с AAD* перенесена в *без AAD*](https://docs.microsoft.com/azure/virtual-machines/windows/disk-encryption-faq#can-i-migrate-vms-that-were-encrypted-with-an-azure-ad-app-to-encryption-without-an-azure-ad-app) |
| [Восстановление между регионами](https://docs.microsoft.com/azure/backup/backup-azure-arm-restore-vms#cross-region-restore) | Создание новой виртуальной машины или восстановление дисков в дополнительный регион (парный регион Azure) | <li> **Полный сбой**. при использовании функции восстановления между регионами нет времени ожидания для восстановления данных в дополнительном регионе. Вы можете инициировать восстановление в дополнительном регионе, даже прежде чем Azure объявляет сбой. <li> **Частичный сбой**. время простоя может происходить в конкретных кластерах хранения, где Azure Backup хранит резервные копии данных или даже в сети, подключаясь к Azure Backup и кластерам хранилища, связанным с резервными копиями данных. При восстановлении между регионами можно выполнить восстановление в дополнительном регионе, используя реплику резервных копий данных в дополнительном регионе. <li> **Без сбоя**. Вы можете выполнять детализацию непрерывности бизнес-процессов и аварийного восстановления (BCDR) для целей аудита или обеспечения соответствия с данными дополнительного региона. Это позволяет выполнять восстановление резервных копий данных в дополнительном регионе, даже если в основном регионе нет полного или частичного сбоя для обеспечения непрерывности бизнес-процессов и аварийного восстановления.  |
