# <a name="support-matrix-for-backup-with-the-microsoft-azure-recovery-services-mars-agent"></a>Матрица поддержки для резервного копирования с помощью агента Служб восстановления Microsoft Azure (MARS)

Службу [резервного копирования Azure](backup-overview.md) можно использовать для резервного копирования на месте машин и приложений и резервного копирования виртуальных машин Azure (VM). В этой статье кратко излагаются параметры и ограничения поддержки при использовании агента Microsoft Azure Recovery Services (MARS) для резервного копирования машин.

## <a name="the-mars-agent"></a>Агент MARS

Резервное копирование Azure использует агент MARS для резервного копирования данных с наемных компьютеров и VM Azure в хранилище резервных служб восстановления в Azure. Агент MARS может:

- Запуск на предприимчезных компьютерах Windows, чтобы они могли резервное копирование непосредственно в хранилище резервных служб восстановления в Azure.
- Выполнить на Windows VMs, чтобы они могли резервное копирование непосредственно в хранилище.
- Выполнить на сервере резервного копирования Microsoft Azure (MABS) или на сервере системного центра защиты данных (DPM). В этом случае машины и рабочие нагрузки обратно до MABS или на сервере DPM. Затем агент MARS резервирует этот сервер в хранилище в Azure.

> [!NOTE]
>Резервное копирование Azure не поддерживает автоматическую регулировку часов для летнего времени (DST). Изменение политики для обеспечения экономии дневного света принимается во внимание, чтобы предотвратить расхождение между фактическим временем и запланированным временем резервного копирования.

Параметры резервного копирования зависят от того, где установлен агент. Для получения дополнительной информации [см. архитектуру резервного копирования Azure с помощью агента MARS.](backup-architecture.md#architecture-direct-backup-of-on-premises-windows-server-machines-or-azure-vm-files-or-folders) Для получения информации об архитектуре резервного копирования MABS и DPM [см.](backup-architecture.md#architecture-back-up-to-dpmmabs) Также [обратитесь к требованиям](backup-support-matrix-mabs-dpm.md) к архитектуре резервного копирования.

**Установки** | **Подробно**
--- | ---
Скачать новейший агент MARS | Последнюю версию агента можно скачать из хранилища или [напрямую](https://aka.ms/azurebackup_agent).
Установить непосредственно на компьютере | Вы можете установить агент MARS непосредственно на предприимство сервер Windows или на Windows VM, который работает любой из [поддерживаемых операционных систем.](https://docs.microsoft.com/azure/backup/backup-support-matrix-mabs-dpm#supported-mabs-and-dpm-operating-systems)
Установить на сервере резервного копирования | При настройке сервера DPM или MABS для резервного копирования в Azure нужно скачать и установить агент MARS на этом сервере. Вы можете установить агент на [поддерживаемые операционные системы](backup-support-matrix-mabs-dpm.md#supported-mabs-and-dpm-operating-systems) в матрице поддержки сервера резервного копирования.

> [!NOTE]
> По умолчанию в установке расширения расширения Azure, включенных для резервного копирования, включена установка расширения резервного копирования Azure. Это расширение создает резервную копию всей виртуальной машины. Вы можете установить и запустить агент MARS на виртуальной машине Azure вместе с расширением, если хотите выполнить резервное копирование определенных папок или файлов, а не всей виртуальной машины.
> При запуске агента MARS на Azure VM он резервное копирование файлов или папок, которые находятся во временном хранении на VM. Резервное копирование не удается, если файлы или папки удалены из временного хранилища или если временное хранилище удалено.

## <a name="cache-folder-support"></a>Поддержка папок кэширования

При использовании агента MARS для резервного копирования данных агент делает снимок данных и хранит его в локальной папке кэша перед отправкой данных в Azure. Папка кэша (царапина) имеет несколько требований:

**Кэша** | **Подробно**
--- | ---
Размер |  Свободное пространство в папке кэша должно составить не менее 5-10 процентов от общего размера данных резервного копирования.
Расположение | Папка кэша должна локально храниться на резервной машине, и она должна быть в режиме онлайн. Папка кэша не должна находиться в сети, на съемных носителях или в автономном объеме.
Папка | Папка кэша не должна быть зашифрована на сжатом томе или в сжатой папке, которая является разреженной или имеет точку репарации.
Изменения местоположения | Местоположение кэша можно изменить, остановив`net stop bengine`резервный движок () и скопируя папку кэша на новый диск. (Убедитесь, что новый диск имеет достаточно места.) Затем обновите две записи в реестре в рамках резервного копирования **HKLM-SOFTWARE-Windows Azure** (**Config/ScratchLocation** и **Config/CloudBackupProvider/ScratchLocation)** в новое место и перезапустите движок.

## <a name="networking-and-access-support"></a>Поддержка сети и доступа

### <a name="url-and-ip-access"></a>URL и IP-доступ

Агенту MARS требуется доступ к таким URL-адресам:

- <http://www.msftncsi.com/ncsi.txt>
- *.Microsoft.com
- *.WindowsAzure.com
- *. MicrosoftOnline.com
- *. Windows.net

И на эти IP-адреса:

- 20.190.128.0/18
- 40.126.0.0/18

Доступ ко всем URL-адресам и IP-адресам, перечисленным выше, использует протокол HTTPS в порту 443.

### <a name="azure-expressroute-support"></a>Поддержка Azure ExpressRoute

Вы можете создать резервную связь с данными azure ExpressRoute с помощью общедоступных пирингов (доступных для старых схем) и спиринга Майкрософт. Резервное копирование над приватным пирингом не поддерживается.

С публичным пирингом: Обеспечить доступ к следующим доменам/адресам:

- `http://www.msftncsi.com/ncsi.txt`
- `microsoft.com`
- `.WindowsAzure.com`
- `.microsoftonline.com`
- `.windows.net`

С помощью пиринга Майкрософт, пожалуйста, выберите следующие службы/регионы и соответствующие значения сообщества:

- Активный каталог Azure (12076:5060)
- Регион Microsoft Azure (в зависимости от местоположения хранилища служб восстановления)
- Хранилище Azure (в зависимости от местоположения хранилища служб восстановления)

Для получения дополнительной [ExpressRoute routing requirements](https://docs.microsoft.com/azure/expressroute/expressroute-routing)информации см.

>[!NOTE]
>Общественное Пиринг утихает для новых схем.

## <a name="backup-limits"></a>Ограничения резервного копирования

### <a name="size-limits"></a>Ограничения размера

Резервное копирование Azure ограничивает размер источника данных файла или папки, который может быть резервироваться. Элементы, которые вы резервируете из одного тома, не могут превышать размеры, суммируемые в этой таблице:

**Операционная система** | **Ограничение размера**
--- | ---
Windows Server 2012 или более поздней версии; |54 400 ГБ
Windows Server 2008 R2 с пакетом обновления 1 (SP1) |1700 ГБ
Windows Server 2008 с пакетом обновления 2 (SP2)| 1700 ГБ
Windows 8 или более поздняя версия| 54 400 ГБ
Windows 7| 1700 ГБ

### <a name="other-limitations"></a>Другие ограничения

- MARS не поддерживает защиту нескольких машин с одинаковым названием в одном хранилище.

## <a name="support-for-initial-offline-backup"></a>Поддержка начального автономного резервного копирования

Azure Backup поддерживает *автономное посевное* средство для передачи исходных данных резервного копирования в Azure с помощью дисков. Эта поддержка полезна, если исходное резервное копирование, скорее всего, будет находиться в диапазоне размеров терабайт (TBs). Автономное резервное копирование поддерживается для:

- Прямая резервная копирование файлов и папок на непредпочном компьютере, работая под управлением агента MARS.
- резервного копирования рабочих нагрузок и файлов с сервера DPM или MABS;

Резервное копирование в автономном режиме не может использоваться для файлов состояния системы.

## <a name="support-for-data-restoration"></a>Поддержка восстановления данных

Используя функцию [мгновенного восстановления](backup-instant-restore-capability.md) резервного копирования Azure, можно восстановить данные до их копирования в хранилище. Машина, которую вы резервируете, должна работать под управлением .NET Framework 4.5.2 или выше.

Резервное копирование не может быть восстановлено в целевой машине, которая работает более ранняя версия операционной системы. Например, резервное копирование, взятое с компьютера под управлением Windows 7, может быть восстановлено на Windows 8 или позже. Но резервная версия, взятая с компьютера, работая под управлением Windows 8, не может быть восстановлена на компьютере, который работает под управлением Windows 7.
