# <a name="auto-enable-backup-on-vm-creation-using-azure-policy"></a>Автоматическое включение резервного копирования при создании виртуальной машины с использованием Политики Azure

Одной из ключевых обязанностей резервного копирования или соответствия admin в организации является обеспечение того, чтобы все критически важные для бизнеса машины были подкреплены соответствующим удержанием.

Сегодня Резервное копирование Azure предоставляет встроенную политику (с использованием политики Azure), которая может быть назначена **всем MMs Azure в определенном месте в группе подписок или ресурсов.** Когда эта политика присваивается заданной области, все новые ВМ, созданные в этой области, автоматически настраиваются для резервного копирования **в существующее хранилище в том же месте и подписке.** Пользователь может указать хранилище и политику хранения, к которой должны быть связаны резервные ввои.

## <a name="supported-scenarios"></a>Поддерживаемые сценарии

* Встроенная политика в настоящее время поддерживается только для M-м и вывМ Azure. Пользователи должны позаботиться о том, чтобы политика удержания, указанная во время назначения, была политикой удержания VM. Обратитесь к [этому](https://docs.microsoft.com/azure/backup/backup-azure-policy-supported-skus) документу, чтобы увидеть все VM SKUs, поддерживаемые этой политикой.

* Политика может быть назначена на одно место и подписку одновременно. Для обеспечения резервного копирования для вс-а-м в разных местоположениях и подписках необходимо создать несколько экземпляров назначения политики, по одному для каждой комбинации местоположения и подписки.

* Указанное хранилище и vMs, настроенные для резервного копирования, могут находиться под различными группами ресурсов.

* Сфера охвата Группы управления в настоящее время не поддерживается.

* Встроенная политика в настоящее время недоступна в национальных облаках.

## <a name="using-the-built-in-policy"></a>Использование встроенной политики

Чтобы присвоить политику требуемому объему, пожалуйста, выполните ниже приведенные ниже шаги:

1. Войдите на портал Azure и перейдите к панели мониторинга **политики.**
2. Выберите **определения** в левом меню, чтобы получить список встроенных политик в ресурсах Azure.
3. Отфильтруйсписоте список для **резервного копирования категории.** Список будет отфильтрован в одну политику под названием "Настройка резервного копирования на вс-чатах местоположения в существующем центральном Убежище в том же месте".
![Панель мониторинга политики](./media/backup-azure-auto-enable-backup/policy-dashboard.png)
4. Нажмите на название политики. Вы будете перенаправлены на подробное определение этой политики.
![Клинок определения политики](./media/backup-azure-auto-enable-backup/policy-definition-blade.png)
5. Нажмите на кнопку **Присваиваете в** верхней части лезвия. Это перенаправляет вас к лезвию **политики присваиваема.**
6. В соответствии с **основами,** нажмите на три точки рядом с полем **области области.** Это открывает правильный контекстлез, где можно выбрать подписку для политики, на которой будет применена политика. Можно также по желанию выбрать группу ресурсов, чтобы политика применялась только для вс-мавлов в определенной группе ресурсов.
![Основы назначения политики](./media/backup-azure-auto-enable-backup/policy-assignment-basics.png)
7. Во вкладке **Параметры** выберите место из выпадения и выберите политику хранилища и резервного копирования, к которой должны быть связаны ввода-носом в области.
![Параметры назначения политики](./media/backup-azure-auto-enable-backup/policy-assignment-parameters.png)
8. Убедитесь, что **эффект** настроен на развертываниеIfNotExists.
9. Перейдите к **просмотре и** нажмите **Создать**.

> [!NOTE]
>
> Политика Azure также может использоваться на существующих внезависимостях вымотка с использованием [исправлений.](https://docs.microsoft.com/azure/governance/policy/how-to/remediate-resources)

> [!NOTE]
>
> Рекомендуется, чтобы эта политика не была назначена более чем 200 ВМ одновременно. Если политика назначена более чем 200 ВМ, это может привести к тому, что резервное копирование сработает на несколько часов позже, чем это предусмотрено графиком.
