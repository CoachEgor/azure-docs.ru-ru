# <a name="overview-of-security-features-in-azure-backup"></a>Обзор функций безопасности в резервном копировании Azure

Одним из наиболее важных шагов, которые можно предпринять для защиты данных, является наличие надежной инфраструктуры резервного копирования. Но не менее важно обеспечить надежное резервное копирование ваших данных и защиту резервных копий в любое время. Резервное копирование Azure обеспечивает безопасность среды резервного копирования как при переходе данных, так и в состоянии покоя. В этой статье перечислены возможности безопасности в резервном копировании Azure, которые помогут защитить данные резервного копирования и удовлетворить потребности в безопасности вашего бизнеса.

## <a name="management-and-control-of-identity-and-user-access"></a>Управление и контроль идентификации и доступа пользователей

Резервное копирование Azure позволяет управлять мелкозернистым доступом с помощью [управления доступом на основе ролей Azure (RBAC).](https://docs.microsoft.com/azure/role-based-access-control/built-in-roles) RBAC позволяет сегрегировать обязанности внутри вашей команды и предоставлять только объем доступа к пользователям, необходимым для их работы.

Azure Backup предоставляет три встроенные роли для управления операциями управления резервным копированием:

* Резервный вкладчик - для создания и управления резервными копиями, за исключением удаляния хранилища служб восстановления и предоставления доступа к другим
* Оператор резервного копирования - все, что делает участник, кроме удаления резервных копий и управления политиками резервного копирования
* Резервное копирование Reader - разрешения для просмотра всех операций управления резервного копирования

Подробнее о [элементе управления доступом на основе ролей для управления резервным копированием Azure.](https://docs.microsoft.com/azure/backup/backup-rbac-rs-vault)

Резервное копирование Azure имеет несколько элементов управления безопасностью, встроенных в службу для предотвращения, обнаружения и реагирования на уязвимости безопасности. Подробнее о [средствах управления безопасностью для резервного копирования Azure.](https://docs.microsoft.com/azure/backup/backup-security-controls)

## <a name="separation-between-guest-and-azure-storage"></a>Разделение между гостем и хранилищем Azure

С помощью резервного копирования Azure, которое включает в себя резервное копирование виртуальной машины и S'L и SAP HANA в резервном копировании VM, данные резервного копирования хранятся в хранилище Azure, и гость не имеет прямого доступа к резервному хранению или его содержимому.  При виртуальном резервном копировании резервного копирования создание и хранение резервного копирования осуществляется тканью Azure, где гость не имеет никакого участия, кроме как таяние рабочей нагрузки для резервного копирования приложений.  С помощью S'L и SAP HANA расширение резервного копирования получает временный доступ для записи к определенным каплям.  Таким образом, даже в скомпрометированной среде существующие резервные копирования не могут быть подделаны или удалены гостем.

## <a name="internet-connectivity-not-required-for-azure-vm-backup"></a>Подключение к Интернету не требуется для резервного копирования Azure VM

Резервное копирование виртуальных машин Azure требует перемещения данных с диска виртуальной машины в хранилище служб восстановления. Однако все необходимые коммуникации и передача данных происходят только в магистральной сети Azure без доступа к виртуальной сети. Поэтому резервное копирование VMs Azure, размещенных внутри защищенных сетей, не требует доступа к каким-либо ИП или VPN.

## <a name="private-endpoints-for-azure-backup"></a>Частные конечные точки для резервного копирования Azure

Теперь вы можете использовать [Private Endpoints](https://docs.microsoft.com/azure/private-link/private-endpoint-overview) для резервного копирования данных с серверов внутри виртуальной сети в хранилище служб восстановления. Частная конечная точка использует IP-адресное пространство VNET для вашего хранилища, поэтому вам не нужно подвергать ваши виртуальные сети любым общедоступным IP-адресам. Частные конечные точки могут использоваться для резервного копирования и восстановления баз данных S'L и SAP HANA, которые работают внутри ваших VMs Azure. Он также может быть использован для ваших серверов с использованием агента MARS.

## <a name="encryption-of-data-in-transit-and-at-rest"></a>Шифрование данных в пути и в состоянии покоя

Шифрование защищает ваши данные и помогает вам выполнять свои организационные обязательства по обеспечению безопасности и соответствию требованиям. В Azure данные, проходящие транзит между хранилищем Azure и хранилищем, защищены HTTPS. Эти данные остаются в магистральной сети Azure.

* Данные резервного копирования автоматически шифруются с помощью ключей, управляемых Корпорацией Майкрософт. Вы также можете шифровать резервные управляемые дисковые ввода-чатах в Хранилище служб восстановления, используя [управляемые клиентом ключи,](https://docs.microsoft.com/azure/backup/backup-azure-security-feature-cloud#encryption-of-backup-data-using-customer-managed-keys) хранящиеся в Хранилище ключей Azure. Вам не нужно предпринимать никаких явных действий, чтобы включить это шифрование. Это относится ко всем рабочим нагрузкам, резервным в хранилище служб восстановления.

* Резервное копирование Azure поддерживает резервное копирование и восстановление VMs Azure, которые имеют свои диски ОС/данных, зашифрованные с помощью шифрования дисков Azure (ADE). [Подробнее о зашифрованных ВМ Azure и резервном копировании Azure.](https://docs.microsoft.com/azure/backup/backup-azure-vms-encryption)

## <a name="protection-of-backup-data-from-unintentional-deletes"></a>Защита данных резервного копирования от непреднамеренных удалений

Резервное копирование Azure предоставляет функции безопасности, чтобы защитить данные резервного копирования даже после удаления. При мягком удалении, если пользователь удаляет резервную кавер-базу VM, данные резервного копирования сохраняются в течение 14 дополнительных дней, что позволяет восстановить этот элемент резервного копирования без потери данных. Дополнительное 14-дневное хранение данных резервного копирования в состоянии «мягкого удаления» не несет никаких затрат для клиента. [Подробнее о мягком удалении](https://docs.microsoft.com/azure/backup/backup-azure-security-feature-cloud#soft-delete).

## <a name="monitoring-and-alerts-of-suspicious-activity"></a>Мониторинг и оповещения о подозрительной деятельности

Резервное копирование Azure обеспечивает [встроенные возможности мониторинга и оповещения](https://docs.microsoft.com/azure/backup/backup-azure-monitoring-built-in-monitor) для просмотра и настройки действий для событий, связанных с резервным копированием Azure. [Отчеты о резервном копировании](https://docs.microsoft.com/azure/backup/configure-reports) служат единым пунктом назначения для отслеживания использования, аудита резервных upups и восстановления, а также выявления ключевых тенденций на различных уровнях детализации. Использование инструментов мониторинга и отчетности Резервного копирования Azure может предупредить вас о любых несанкционированных, подозрительных или вредоносных действиях, как только они происходят.

## <a name="security-features-to-help-protect-hybrid-backups"></a>Функции безопасности для защиты гибридных резервных копирований

Служба резервного копирования Azure использует агент службы восстановления Microsoft Azure (MARS) для резервного копирования и восстановления файлов, папок и состояния громкости или системы с предприимченского компьютера в Azure. MARS теперь предоставляет функции безопасности, чтобы защитить гибридные резервные копирования. Эти функции включают в себя следующие:

* При выполнении критически важных операций, таких как изменение парольной фразы, добавляется дополнительный уровень аутентификации. Это гарантирует, что такие операции могут выполнять только пользователи с действительными учетными данными Azure. [Подробнее об особенностях, предотвращаевих атаки.](https://docs.microsoft.com/azure/backup/backup-azure-security-feature#prevent-attacks)

* Удаленные данные резервного копирования сохраняются в течение еще 14 дней с даты удаления. Это обеспечивает восстановление данных в течение определенного периода времени, поэтому потеря данных не происходит даже в случае атаки. Кроме того, повышение минимального числа точек восстановления позволяет защититься от повреждения данных. [Подробнее о восстановлении удаленных данных резервного копирования.](https://docs.microsoft.com/azure/backup/backup-azure-security-feature#recover-deleted-backup-data)
