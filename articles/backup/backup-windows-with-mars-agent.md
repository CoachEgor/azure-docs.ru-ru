# <a name="back-up-windows-machines-by-using-the-azure-backup-mars-agent"></a>Резервное копирование машин Windows с помощью агента Azure Backup MARS

В этой статье объясняется, как создать резервную связь с windows-машинами с помощью службы [резервного копирования Azure](backup-overview.md) и агента Microsoft Azure Recovery Services (MARS). MARS также известен как агент резервного копирования Azure.

В этой статье раскрываются следующие темы:

> [!div class="checklist"]
>
> * Проверка предварительных требований
> * Создайте политику резервного копирования и расписание.
> * Выполните резервную по требованию резервную.

## <a name="before-you-start"></a>Перед началом работы

* Узнайте, как [резервное копирование Azure использует агент MARS для резервного копирования машин Windows.](backup-architecture.md#architecture-direct-backup-of-on-premises-windows-server-machines-or-azure-vm-files-or-folders)
* Узнайте об [архитектуре резервного копирования,](backup-architecture.md#architecture-back-up-to-dpmmabs) которая запускает агент MARS на вторичном сервере MABS или manager Data Manager.
* [Просмотрите, что поддерживается и что вы можете создать резервную поддержку](backup-support-matrix-mars-agent.md) агентом MARS.
* [Проверьте доступ в Интернет](install-mars-agent.md#verify-internet-access) на машинах, которые вы хотите создать резервную.
* Если агент MARS не установлен, узнайте, как его установить [здесь.](install-mars-agent.md)

## <a name="create-a-backup-policy"></a>создание политики архивации;

Политика резервного копирования определяет, когда следует делать снимки данных для создания точек восстановления. В нем также указывается, как долго держать точки восстановления. Вы используете агент MARS для настройки политики резервного копирования.

Резервное копирование Azure автоматически не учитывает летнее время (DST). Это значение по умолчанию может привести к некоторому расхождению между фактическим временем и запланированным временем резервного копирования.

Чтобы создать политику резервного копирования, выполните следующее.

1. После загрузки и регистрации агента MARS откройте консоль агента. Его можно найти, выполнив поиск строки **Microsoft Azure Backup** на компьютере.  

1. В соответствии с **действиями,** выберите **Резервное копирование расписания.**

    ![Планирование архивации Windows Server](./media/backup-configure-vault/schedule-first-backup.png)
1. В Мастере резервного копирования расписания выберите Начало**следующего.** **Getting started** > 
1. При **выборе элементов для резервного копирования**выберите **Добавить элементы.**

    ![Добавление элементов для резервного копирования](./media/backup-azure-manage-mars/select-item-to-backup.png)

1. В поле **Select Items** выберите элементы для резервного копирования, а затем выберите **OK.**

    ![Выбор элементов для резервного копирования](./media/backup-azure-manage-mars/selected-items-to-backup.png)

1. На странице **«Выберите элементы для резервного копирования»** выберите **следующий**.
1. На странице **Расписание расписания уточнения** уточнить, когда принимать ежедневные или еженедельные резервные отработки. Затем выберите **Следующий**.

    * Точка восстановления создается при взятии резервного копирования.
    * Количество точек восстановления, созданных в среде, зависит от графика резервного копирования.
    * Вы можете запланировать до трех ежедневных резервных копирования в день. В следующем примере происходит два ежедневных резервных копирования: одно в полночь и одно в 18:00.

        ![Настройка ежедневного расписания резервного копирования](./media/backup-configure-vault/day-schedule.png)

    * Вы можете запустить еженедельные резервные копирования тоже. В следующем примере резервные копирования принимаются каждое альтернативное воскресенье и среду в 9:30 и 1:00 утра.

        ![Настройка еженедельного графика резервного копирования](./media/backup-configure-vault/week-schedule.png)

1. На странице **Политики удержания выбора** укажите, как хранить исторические копии данных. Затем выберите **Следующий**.

    * Параметры удержания определяют, какие точки восстановления хранить и как долго их хранить.
    * Для дневного параметра удержания указывается, что в указанное для ежедневного удержания время последняя точка восстановления будет сохранена в течение определенного количества дней. Или можно указать ежемесячную политику удержания, чтобы указать, что точка восстановления, созданная 30 числа каждого месяца, должна храниться в течение 12 месяцев.
    * Сохранение ежедневных и еженедельных точек восстановления обычно совпадает с графиком резервного копирования. Поэтому, когда график запускает резервное копирование, точка восстановления, которую создает резервное копирование, сохраняется на время, очем является ежедневная или еженедельная политика удержания.
    * Рассмотрим следующий пример:

        * Ежедневные резервные копирования в полночь и 6:00 вечера хранятся в течение семи дней.
        * Резервное копирование, принятое в субботу в полночь и 6:00 вечера хранятся в течение четырех недель.
        * Резервное копирование, сделанное в последнюю субботу месяца в полночь и 18:00, сохраняется в течение 12 месяцев.
        * Резервное копирование, сделанное в последнюю субботу марта, сохраняется в течение 10 лет.

        ![Пример политики удержания](./media/backup-configure-vault/retention-example.png)

1. На странице **«Выбрать первоначальный тип резервного копирования»** решите, хотите ли вы использовать начальную резервную прикладную резервную часть по сети или использовать резервное копирование в автономном режиме. Чтобы провести начальную резервную передачу по сети, выберите **Автоматически по сети** > **Next.**

    Для получения дополнительной информации о резервном копировании в автономном режиме [см.](offline-backup-azure-data-box.md)

    ![Выберите исходный тип резервного копирования](./media/backup-azure-manage-mars/choose-initial-backup-type.png)

1. На странице **Подтверждения** просмотрите информацию, а затем выберите **Finish**.

    ![Подтверждение типа резервного копирования](./media/backup-azure-manage-mars/confirm-backup-type.png)

1. После завершения мастером создания графика резервного копирования выберите **Close**.

    ![Просмотр прогресса графика резервного копирования](./media/backup-azure-manage-mars/confirm-modify-backup-process.png)

Создайте политику на каждой машине, где установлен агент.

### <a name="do-the-initial-backup-offline"></a>Уисподнейное резервное копирование в автономном режиме

Вы можете запустить начальную резервную прикладную резервную прикладавтоматически по сети, или вы можете создать резервную настройку в автономном режиме. Оффлайн посев для первоначального резервного копирования полезен, если у вас есть большие объемы данных, которые потребуют много пропускной способности сети для передачи.

Для перевода в автономном режиме:

1. Напишите данные резервного копирования в место постановки.
1. Используйте инструмент AzureOfflineBackupDiskPrep для копирования данных с места постановки на один или несколько дисков SATA.

    Инструмент создает задание Azure Import. Для получения дополнительной [What is the Azure Import/Export service](https://docs.microsoft.com/azure/storage/common/storage-import-export-service)информации см.
1. Отправьте диски SATA в центр обработки данных Azure.

    В центре обработки данных о диске копируется учетная запись хранения Azure. Резервное копирование Azure копирует данные из учетной записи хранилища в хранилище, и запланированы дополнительные резервные копии.

Для получения дополнительной информации о автономном посеве [см.](offline-backup-azure-data-box.md)

### <a name="enable-network-throttling"></a>Включение регулирования сети

Вы можете контролировать, как агент MARS использует пропускную способность сети, позволяя регулировать сеть. Регулирование полезно, если вам нужно резервное копирование данных в рабочее время, но вы хотите контролировать, сколько пропускной способности резервного копирования и восстановления использования активности.

Сетевое регулирование в резервном копировании Azure использует [качество обслуживания (ЗОС)](https://docs.microsoft.com/windows-server/networking/technologies/qos/qos-policy-top) в локальной операционной системе.

Сетевое регулирование резервных отсеивов доступно на Windows Server 2012 и позже, а также на Windows 8 и позже. Операционные системы должны работать с новейшими пакетами услуг.

Для обеспечения регулирования сети:

1. В агенте MARS выберите **Свойства изменений.**
1. На вкладке **Throttling** выберите **Использование пропускной способности интернета для операций резервного копирования.**

    ![Настройка регулирования сети для операций резервного копирования](./media/backup-configure-vault/throttling-dialog.png)
1. Укажите допустимую пропускную способность в рабочее и нерабочее и нерабочее время. Значения пропускной способности начинаются с 512 Кбит/с и до1023 Мбит/с. Затем выберите **OK**.

## <a name="run-an-on-demand-backup"></a>Выполнение резервного копирования по запросу

1. В агенте MARS выберите **Back Up Now.**

    ![Резервное копирование в Windows Server](./media/backup-configure-vault/backup-now.png)

1. Если версия агента MARS 2.0.9169.0 или новее, то вы можете установить пользовательскую дату удержания. В разделе **Retain Backup Till** выберите дату из календаря.

   ![Используйте календарь для настройки даты удержания](./media/backup-configure-vault/mars-ondemand.png)

1. На странице **Подтверждения** просмотрите настройки и выберите **Back Up.**
1. Выберите **Близко,** чтобы закрыть мастера. Если вы закрываете мастера до завершения резервного копирования, мастер продолжает работать в фоновом режиме.

После завершения первоначальной резервной работы состояние **завершенного вакансии** отображается в консоли резервного копирования.

## <a name="set-up-on-demand-backup-policy-retention-behavior"></a>Настройка поведения удержания политики резервного копирования по требованию

> [!NOTE]
> Эта информация распространяется только на версии агента MARS, которые старше 2.0.9169.0.
>

| Опция резервного копирования-графика | Длительность хранения данных
| -- | --
| День | **Удержание по умолчанию**: эквивалентно "удержанию дней для ежедневных резервных копирований". <br/><br/> **Исключение**: Если ежедневное запланированное резервное копирование, установленное для долгосрочного удержания (недели, месяцы или годы), не удается, резервное копирование по требованию, срабатывающее сразу после сбоя, считается долгосрочным удержанием. В противном случае следующее запланированное резервное копирование рассматривается для долгосрочного удержания.<br/><br/> **Пример сценария**: Запланированная резервная часть в четверг в 8:00 не удалось. Эта резервная задержка должна была рассматриваться для еженедельного, ежемесячного или ежегодного удержания. Таким образом, первая резервная поддержка по требованию, срабатывающая до следующего запланированного резервного копирования в пятницу в 8:00 утра, автоматически помечается для еженедельного, ежемесячного или годового удержания. Это резервное копирование заменяет четверг 8:00 AM резервного копирования.
| Неделя | **Удержание по умолчанию**: Один день. На следующий день удаляются резервные ожидания по требованию, которые используются для источника данных с еженедельной политикой резервного копирования. Они удаляются, даже если они являются последними резервными копированиями для источника данных. <br/><br/> **Исключение**: Если еженедельное запланированное резервное копирование, установленное для долгосрочного удержания (недели, месяцы или годы), не удается, резервное копирование по требованию, которое срабатывает сразу после сбоя, считается долгосрочным удержанием. В противном случае следующее запланированное резервное копирование рассматривается для долгосрочного удержания. <br/><br/> **Пример сценария**: Запланированная резервная часть в четверг в 8:00 не удалось. Эта резервная часть должна была рассматриваться для ежемесячного или ежегодного удержания. Таким образом, первое резервное копирование по требованию, срабатывающее до следующего запланированного резервного копирования в четверг в 8:00 утра, автоматически помечается для ежемесячного или ежегодного удержания. Это резервное копирование заменяет четверг 8:00 AM резервного копирования.
