# <a name="back-up-hyper-v-virtual-machines-with-azure-backup-server"></a>Резервное копирование виртуальных машин Hyper-V с помощью сервера резервного копирования Azure

В этой статье объясняется, как создать резервную связь с помощью виртуального сервера Microsoft Azure Backup Server (MABS).

## <a name="supported-scenarios"></a>Поддерживаемые сценарии

MABS может создать резервную работу по виртуальным машинам, работающим на серверах Hyper-V в следующих сценариях:

- **Виртуальные машины с локальным или прямо подключенным хранилищем** — резервное копирование виртуальных машин, размещенных на отдельных серверах Hyper-V с локальным или непосредственно подключенным хранилищем. Например: жесткий диск, устройство сети хранения данных (SAN) или устройство, подключенное к сети хранения (NAS). Агент защиты MABS должен быть установлен на всех узлах.

- **Виртуальные машины в кластере с хранилищем CSV** — резервное копирование виртуальных машин, размещенных в кластере Hyper-V, с хранилищем общих томов кластера (CSV). На каждом кластерном узлах устанавливается защитный агент MABS.

## <a name="host-versus-guest-backup"></a>Хост против резервного копирования гостей

MABS может сделать резервную приготовительку Hyper-V M на уровне хоста или гостей. На уровне хоста агент защиты MABS устанавливается на сервер или кластер Hyper-V и защищает все vMs и файлы данных, работающие на этом хостеле.   На гостевом уровне агент устанавливается на каждую виртуальную машину и защищает рабочую нагрузку, присутствуюую на этой машине.

Оба способа имеют свои преимущества и недостатки.

- Резервное копирование на уровне хоста является гибким, поскольку они работают независимо от типа ОС, работающую на гостевых машинах, и не требуют установки агента защиты MABS на каждый VM. При развертывании резервного копирования уровня хоста можно восстановить всю виртуальную машину или файлы и папки (восстановление уровня элементов).

- Резервное копирование на уровне гостей полезно, если вы хотите защитить определенные рабочие нагрузки, работающие на виртуальной машине. На уровне хоста можно восстановить весь VM или определенные файлы, но он не обеспечит восстановление в контексте конкретного приложения. Например, чтобы восстановить определенные элементы SharePoint из резервного VM, необходимо сделать резервную копию этого VM уровня гостя. Используйте резервную защиту на уровне гостей, если вы хотите защитить данные, хранящиеся на дисках. Passthrough позволяет виртуальной машине напрямую получить доступ к устройству хранения данных и не хранить данные виртуального объема в файле VHD.

## <a name="how-the-backup-process-works"></a>Как работает процесс резервного копирования

MABS выполняет резервное копирование с VSS следующим образом. Шаги в этом описании нумеруются для ясности.

1. Двигатель синхронизации на основе блоков MABS является первоначальной копией защищенной виртуальной машины и гарантирует, что копия виртуальной машины является полной и последовательной.

2. После того, как первоначальная копия сделана и проверена, MABS использует hyper-V VSS writer для захвата резервных копий. Автор VSS предоставляет согласованный с данными набор дисковых блоков, синхронизированных с сервером MABS. Этот подход обеспечивает преимущество «полного резервного копирования» с сервером MABS при минимизации данных резервного копирования, которые должны быть переданы по сети.

3. Агент защиты MABS на сервере, который работает Hyper-V использует существующие Hyper-V AIS, чтобы определить, является ли защищенная виртуальная машина также поддерживает VSS.

   - Если виртуальная машина соответствует требованиям к оперативной архивации и на ней установлены службы интеграции Hyper-V, модуль записи VSS Hyper-V рекурсивно перенаправляет запрос VSS ко всем процессам, поддерживающим VSS, в виртуальной машине. Эта операция происходит без установки защитного агента MABS на виртуальную машину. Благодаря рекурсивному запросу VSS модуль записи VSS Hyper-V обеспечивает синхронизацию операций записи на диск, поэтому запись моментального снимка VSS происходит без потери данных.

     Компонент служб интеграции Hyper-V вызывает модуль записи VSS в службах VSS на виртуальных машинах, чтобы убедиться, что их данные приложений находятся в согласованном состоянии.

   - Если виртуальная машина не соответствует требованиям резервного копирования в Режиме онлайн, MABS автоматически использует Hyper-V AA для приостановки использования виртуальной машины до захвата файлов данных.

4. После того, как начальная базовая копия виртуальной машины синхронизируется с сервером MABS, все изменения, внесенные в ресурсы виртуальной машины, фиксируются в новой точке восстановления. Точка восстановления представляет согласованное состояние виртуальной машины в конкретный момент времени. Записи точки восстановления могут происходить как минимум один раз в день. При создании новой точки восстановления MABS использует репликацию на уровне блоков в сочетании с автором Hyper-V VSS, чтобы определить, какие блоки были изменены на сервере, который работает Hyper-V после создания последней точки восстановления. Эти блоки данных затем передаются на сервер MABS и применяются к реплике защищенных данных.

5. Сервер MABS использует VSS на томах, которые умещают данные восстановления, так что несколько теневых копий доступны. Каждая из этих теневых копий предоставляет отдельный процесс восстановления. Точки восстановления VSS хранятся на сервере MABS. Временная копия, сделанная на сервере под управлением Hyper-V, хранится только на время синхронизации MABS.

>[!NOTE]
>
>Начиная с Windows Server 2016, виртуальные жесткие диски Hyper-V имеют встроенные отслеживания изменений, известные как устойчивое отслеживание изменений (RCT). MABS использует RCT (отслеживание коренных изменений в Hyper-V), что уменьшает потребность в трудоемких проверках согласованности в таких сценариях, как сбои VM. RCT обеспечивает лучшую устойчивость, чем отслеживание изменений, предоставляемое резервными копиями на основе моментальных снимков VSS. MABS V3 оптимизирует использование ресурсов сети и хранилища, передавая только измененные данные во время каких-либо проверок согласованности.

## <a name="back-up-virtual-machines"></a>Резервное копирование виртуальных машин

1. Настройка [сервера MABS](backup-azure-microsoft-azure-backup.md) и [хранилища.](backup-mabs-add-storage.md) При настройке хранилища используйте следующие рекомендации по выделению емкости хранилища.
   - Средний размер виртуальной машины — 100 ГБ
   - Количество виртуальных машин на сервер MABS - 800
   - Общий размер 800 виртуальных машин — 80 ТБ
   - Объем необходимого пространства для хранения резервных копий — 80 ТБ

2. Настройка средства защиты MABS на сервере Hyper-V или кластерных узлах Hyper-V. Если вы выполняете резервную проверку на уровне гостей, вы установите агент на vMs-сообщениях, которые вы хотите создать на уровне гостей.

3. В консоли администратора MABS нажмите кнопку **«Защита** > **Создайте группу защиты»,** чтобы открыть мастера **Группы создания.**

4. На странице **Выбор элементов группы** выберите виртуальные машины, для которых необходимо обеспечить защиту, на серверах узлов Hyper-V, где они размещаются. Рекомендуется поместить все виртуальные машины, для которых задана одна политика защиты, в одну группу защиты. Для эффективного использования пространства можно включить совместное размещение. Совместное размещение позволяет размещать данные из разных групп защиты на одном диске или ленточном накопителе, с тем чтобы нескольким источникам данных был назначен один том реплики и точки восстановления.

5. На странице **Выбор метода защиты данных** укажите имя группы защиты. Выберите параметры **Краткосрочная защита с использованием диска** и **Мне нужна оперативная защита** , чтобы выполнить резервное копирование данных в Azure с помощью службы архивации Azure.

6. В**определении краткосрочного диапазона удержания** **целей** > укажите, как долго вы хотите сохранить данные диска. В **частоте синхронизации**укажите, как часто должны запускаться дополнительные резервные копирования данных. Вместо выбора интервала для добавочных резервных копий можно выбрать параметр **Непосредственно перед точкой восстановления**. С помощью этой настройки MABS запустит полное резервное копирование непосредственно перед каждой запланированной точкой восстановления.

    > [!NOTE]
    >
    >Если вы защищаете рабочие нагрузки приложения, точки восстановления создаются в соответствии с частотой синхронизации (если приложение поддерживает добавочное резервное копирование). Если это не так, то MABS запускает полное резервное копирование, а не дополнительное резервное копирование, и создает точки восстановления в соответствии с графиком резервного копирования.

7. На странице **распределения диска Review** просмотрите дисковое пространство пула хранения, выделенное для группы защиты.

   **Общий размер данных** — это размер данных, которые вы хотите создать для резервного копирования, а **пространство диска, которое будет подготовлено на MABS,** — это пространство, которое MABS рекомендует для группы защиты. MABS выбирает идеальный объем резервного копирования, основываясь на настройках. Тем не менее, можно отсеить выбор громкости резервного копирования в **деталях распределения диска.** Для рабочих нагрузок выберите в раскрывающемся меню предпочитаемый объем хранилища. Вы можете изменить значения для параметров **Общий объем** и **Доступный объем** в области **Доступный объем на диске**. Недооцененное пространство — это объем хранилища, который MABS предлагает добавить к объему, чтобы продолжить резервное копирование плавно в будущем.

8. На странице **Выбор метода создания реплики** определите, каким образом будет выполняться начальная репликация данных в группе защиты. Если вы выберете **для автоматического воспроизведения по сети,** мы рекомендуем вам выбрать непиковое время. Для больших объемов данных или меньше, чем оптимальные условия сети, рассмотреть возможность выбора **Вручную**, которая требует репликации данных в автономном режиме с помощью съемных носителей.

9. На странице **Параметры проверки согласованности** выберите способ автоматизации проверки согласованности. Проверку можно выполнять, когда данные реплики становятся несогласованными или по расписанию. Если вы не хотите настраивать автоматическую проверку согласованности, ее можно выполнить вручную в любое время, щелкнув правой кнопкой мыши группу защиты и выбрав команду **Выполнить проверку согласованности**.

    После создания группы защиты осуществляется начальная репликация данных в соответствии с выбранным вами способом. После начальной репликации каждая операция резервного копирования выполняется в соответствии с параметрами группы защиты. ## <a name="back-up-virtual-machines-configured-for-live-migration"></a>Резервное копирование виртуальных машин, настроенных для динамической миграции

Когда виртуальные машины участвуют в живой миграции, MABS продолжает защищать виртуальные машины до тех пор, пока агент защиты MABS установлен на хосте Hyper-V. Способ защиты виртуальных машин ЗАВИСИТ от типа живой миграции.

**Миграция в кластере** - Когда виртуальная машина мигрирует внутри кластера, MABS обнаруживает миграцию и резервное копирование виртуальной машины из нового кластерного узла без каких-либо требований для вмешательства пользователя. Поскольку место хранения не изменилось, MABS продолжает с экспресс-полных резервных копий.

**Миграция в реальном времени за пределами кластера** - Когда виртуальная машина мигрирует между автономными серверами, различными кластерами или между автономным сервером и кластером, MABS обнаруживает миграцию и может создать резервную копию виртуальной машины без вмешательства пользователя.

### <a name="requirements-for-maintaining-protection"></a>Требования для поддержания защиты

Ниже приведены требования для поддержания защиты во время динамической миграции.

- Хосты Hyper-V для виртуальных машин должны располагаться в облаке System Center VMM, на vMM-сервере, работая по крайней мере System Center 2012 с SP1.

- Средства защиты MABS должны быть установлены на всех хостах Hyper-V.

- Серверы MABS должны быть подключены к серверу VMM. Все серверы Hyper-V в облаке VMM также должны быть подключены к серверам MABS. Это позволяет MABS общаться с vMM-сервером, чтобы MABS мог узнать, на каком сервере Hyper-V находится виртуальная машина в настоящее время работает, и создать новую резервную систему с этого Hyper-V сервера. Если соединение не может быть установлено на сервере Hyper-V, резервное копирование завершается с убоем с сообщением о недосягаемом агенте защиты MABS.

- Все серверы MABS, VMM-серверы и серверы Hyper-V должны находиться в одном домене.

### <a name="details-about-live-migration"></a>Сведения о динамической миграции

Учтите следующие аспекты резервного копирования во время динамической миграции.

- Если живая миграция передает хранилище, MABS выполняет полную проверку согласованности виртуальной машины, а затем продолжает экспресс полные резервные копии. Когда происходит живая миграция хранилища, Hyper-V реорганизует виртуальный жесткий диск (VHD) или VHDX, что вызывает одноразовый всплеск размера данных резервного копирования MABS.

- Включите автоподключение на узле виртуальной машины для обеспечения защиты виртуальной машины и отключите разгрузку TCP Chimney.

- MABS использует порт 6070 в качестве порта по умолчанию для размещения службы помощи DPM-VMM. Изменение реестра

    1. Перейдите в раздел **HKLM\Software\Microsoft\Microsoft Data Protection Manager\Configuration**.
    2. Создайте 32-разрядное значение DWORD DpmVmmHelperServicePort и запишите обновленный номер порта в раздел реестра.
    3. Откройте ```<Install directory>\Azure Backup Server\DPM\DPM\VmmHelperService\VmmHelperServiceHost.exe.config``` и измените номер порта с 6070 на новый порт. Например: ```<add baseAddress="net.tcp://localhost:6080/VmmHelperService/" />```
    4. Перезапустите вспомогательную службу DPM-VMM и перезапустите службу DPM.

### <a name="set-up-protection-for-live-migration"></a>Настройка защиты для динамической миграции

Чтобы настроить защиту для динамической миграции, выполните следующие действия.

1. Навлажь сервер MABS и его хранилище и установите агент защиты MABS на каждый сервер-хост Hyper-V или кластерный узло в облаке VMM. Если вы используете хранение sMB в кластере, установите агент защиты MABS на все кластерные узлы.

2. Установите консоль VMM в качестве клиентского компонента на сервере MABS, чтобы MABS мог общаться с сервером VMM. Версия консоли должна совпадать с версией, запущенной на сервере VMM.

3. Назначайте учетную запись MABSMachineName$ в качестве учетной записи администратора только для чтения на сервере управления VMM.

4. Подключите все серверы-хосты `Set-DPMGlobalProperty` Hyper-V ко всем серверам MABS с помощью cmdlet PowerShell. Cmdlet принимает несколько названий серверов MABS. Используйте формат `Set-DPMGlobalProperty -dpmservername <MABSservername> -knownvmmservers <vmmservername>`. Для получения дополнительной информации [см.](https://docs.microsoft.com/powershell/module/dataprotectionmanager/set-dpmglobalproperty?view=systemcenter-ps-2019)

5. После размещения всех виртуальных машин, работающих на узлах Hyper-V, в облаках VMM и обнаружения их в VMM настройте группу защиты и добавьте виртуальные машины, которые необходимо защитить. Автоматическая проверка согласованности должна быть включена на уровне группы защиты для защиты в сценариях виртуальной мобильности машин.

6. После настройки параметров, когда виртуальная машина мигрирует из одного кластера в другой, все резервные копии продолжаются, как ожидалось. Проверить надлежащую активацию динамической миграции можно, как показано ниже.

   1. Проверьте, что запущена вспомогательная служба DPM и диспетчера виртуальных машин. Если это не так, начните.

   2. Откройте студию управления серверами Microsoft S'L и подключитесь к экземпляру, в котором размещена база данных MABS (DPMDB). В базе данных DPM выполните следующий запрос: `SELECT TOP 1000 [PropertyName] ,[PropertyValue] FROM[DPMDB].[dbo].[tbl_DLS_GlobalSetting]`.

      Этот запрос содержит свойство под названием `KnownVMMServer`. Его значение должно совпадать со значением, которое вы указали в командлете `Set-DPMGlobalProperty`.

   3. Выполните следующий запрос для проверки параметра*VMMIdentifier* в данных `PhysicalPathXML` для конкретной виртуальной машины. Замените `VMName` на имя виртуальной машины.

      ```sql
      select cast(PhysicalPath as XML) from tbl_IM_ProtectedObject where DataSourceId in (select datasourceid from tbl_IM_DataSource   where DataSourceName like '%<VMName>%')
      ```

   4. Откройте XML-файл, возвращенный запросом и убедитесь, что в поле *VMMIdentifier* есть значение.

### <a name="run-manual-migration"></a>Выполнение миграции вручную

После завершения этапов в предыдущих разделах и завершения задания сводным менеджером MABS включена миграция. По умолчанию задание запускается в полночь и выполняется каждое утро. Чтобы запустить ручную миграцию для проверки надлежащей работы всех компонентов и процессов, выполните следующие действия.

1. Откройте студию управления серверами и подключитесь к экземпляру, в котором размещена база данных MABS.

2. Выполните следующий запрос: `SELECT SCH.ScheduleId FROM tbl_JM_JobDefinition JD JOIN tbl_SCH_ScheduleDefinition SCH ON JD.JobDefinitionId = SCH.JobDefinitionId WHERE JD.Type = '282faac6-e3cb-4015-8c6d-4276fcca11d4' AND JD.IsDeleted = 0 AND SCH.IsDeleted = 0`. Этот запрос возвращает значение **ScheduleID**. Запишите этот идентификатор, так как он потребуется на следующем шаге.

3. В SQL Server Management Studio последовательно разверните узлы **Агент SQL Server** и **Задания**. Правой кнопкой щелкните значение **ScheduleID**, а затем выберите пункт **Запустить задание на шаге**.

Производительность резервного копирования зависит при выполнении задания. Время выполнения этого задания зависит от размера и масштаба развертывания.

## <a name="back-up-replica-virtual-machines"></a>Резервное копирование реплик виртуальных машин

Если MABS работает на Windows Server 2012 R2 или больше, то вы можете создать резервную копию виртуальных машин. Это может быть удобно по ряду причин.

**Уменьшение влияния резервного копирования на выполняющуюся рабочую нагрузку**. Создание резервной копии виртуальной машины влечет за собой дополнительные издержки, поскольку создается моментальный снимок. При разгрузке процесса резервного копирования на вторичный удаленный участок работа с работой больше не зависит от операции резервного копирования. Это применимо только к развертываниям, в которых резервная копия хранится в удаленном расположении. Например, можно выполнять ежедневное резервное копирование и хранить данные локально, чтобы затем быстро восстанавливать нужные объекты, и ежемесячно или ежеквартально создавать резервные копии из хранящихся удаленно виртуальных машин реплик, чтобы обеспечить долгосрочную защиту.

**Экономия пропускной способности**. В стандартном удаленном развертывании офиса филиала или головного офиса вам потребуется выделить соответствующий объем пропускной способности, необходимый для передачи резервных копий данных между расположениями. При разработке стратегии репликации и отработки отказа в дополнение к стратегии резервного копирования данных можно сократить объем избыточных данных, передаваемых по сети. Резервное копирование данных виртуальной машины реплики, а не первичных, вы сохраняете накладные расходы на отправку резервных данных по сети.

**Резервное копирование в среде поставщика услуг размещения**. Размещенный центр обработки данных можно использовать как расположение реплики: в этом случае нет необходимости в дополнительном центре обработки данных. В этом случае хостер SLA требует последовательного резервного копирования реплики виртуальных машин.

До запуска отработки отказа реплика виртуальной машины находится в отключенном состоянии, и модуль VSS не может гарантировать выполнение резервного копирования с учетом используемых приложений. Таким образом, отказоустойчивой будет только резервная копия реплики виртуальной машины. Если отказоустойчивость не может быть гарантирована, резервное копирование завершится ошибкой. Это может случаться по ряду причин.

- Реплика виртуальной машины неисправна и находится в критическом состоянии.

- Выполняется повторная синхронизация реплики виртуальной машины (в состоянии "Выполняется повторная синхронизация" или "Требуется повторная синхронизация").

- Начальная репликация между основным и дополнительным сайтом находится в процессе выполнения или ожидает виртуальную машину.

- Журналы .hrl применяются к виртуальной машине реплики, или предыдущее действие, чтобы применить журналы .hrl на виртуальном диске не удалось, или был отменен или прерван.

- Выполняется миграция или отработка отказа реплики виртуальной машины.

## <a name="recover-backed-up-virtual-machines"></a>Восстановление из резервной копии виртуальных машин

При восстановлении резервной копии виртуальной машины для выбора виртуальной машины и конкретной точки восстановления используется мастер восстановления. Чтобы открыть мастер восстановления и восстановить виртуальную машину, выполните следующие действия.

1. В консоли MABS Administrator введите имя VM или расширьте список защищенных элементов и выберите выбранный VM.

2. В области **Точки восстановления для** в календаре щелкните любую дату, чтобы просмотреть доступные точки восстановления. Затем в области **Путь** выберите точку восстановления, которая будет использоваться в мастере восстановления.

3. В меню **Действия** щелкните **Восстановить**, чтобы открыть мастер восстановления.

    Выбранная виртуальная машина и точка восстановления будут выведены в разделе **Просмотр выбора для восстановления**. Нажмите кнопку **Далее**.

4. На странице **Выбор типа восстановления** выберите расположение для восстановления данных и нажмите кнопку **Далее**.

    - **Восстановить в исходном экземпляре**. При восстановлении в исходном экземпляре исходный VHD-файл удаляется. MABS восстанавливает VHD и другие файлы конфигурации в исходное местоположение с помощью писателя Hyper-V VSS. По окончании процесса восстановления виртуальные машины по-прежнему имеют высокую доступность.
        Для восстановления группа ресурсов должна существовать. Если группа ресурсов недоступна, восстановите виртуальную машину в другом расположении, а затем сделайте ее высокодоступной.

    - **Восстановление в качестве виртуальной машины для любого узла:** MABS поддерживает альтернативное восстановление местоположения (ALR), которое обеспечивает бесшовное восстановление защищенной виртуальной машины Hyper-V для другого хоста Hyper-V, независимо от архитектуры процессора. Виртуальные машины Hyper-V, которые восстанавливаются на узел кластера, не будут обладать высокой доступностью. Если выбран этот параметр, мастер восстановления выводит дополнительную страницу для определения места назначения и пути к нему.

    - **Копирование в сетевую папку:** MABS поддерживает восстановление уровня элемента (ILR), что позволяет выполнять восстановление файлов, папок, объемов и виртуальных жестких дисков (VHD) от резервного копирования виртуальных машин Hyper-V на уровне хоста до сетевого долевого участия или объема на защищенном MABS сервере. Агент защиты MABS не должен быть установлен внутри гостя для выполнения восстановления уровня элемента. Если выбран этот параметр, мастер восстановления выводит дополнительную страницу для определения места назначения и пути к нему.

5. В разделе **Задание параметров восстановления** настройте параметры восстановления и нажмите кнопку **Далее**.

    - При восстановлении виртуальной машины по сети с низкой пропускной способностью щелкните **Изменить**, чтобы включить **регулирование использования полосы пропускания сети**. После включения функции регулирования полосы пропускания можно указать необходимый объем пропускной способности и время, когда будет доступна такая пропускная способность.
    - Установите флажок **Разрешить восстановление на основе SAN с использованием моментальных снимков оборудования**, если сеть настроена соответствующим образом.
    - Выберите **Отправить по электронной почте сообщение о завершении этого восстановления**, а затем укажите адреса электронной почты, если требуется задать отправку уведомлений после завершения процесса восстановления.

6. На странице "Сводка" убедитесь, что все сведения верны. Если сведения неверны или вы хотите внести изменения, нажмите кнопку **Назад**. Если все параметры вас устраивают, нажмите кнопку **Восстановить** для запуска процесса восстановления.

7. На странице **Состояния восстановления** содержатся сведения о задании восстановления.
