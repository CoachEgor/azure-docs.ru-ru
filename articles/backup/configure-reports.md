# <a name="configure-azure-backup-reports"></a>Настройка отчетов службы Azure Backup

Распространенным требованием для администраторов резервного копирования является получение ценных сведений о резервных копиях на основе данных, охватывающих длительный период времени. Существует несколько вариантов использования таких решений, как распределение и прогнозирование использования облачного хранилища, аудит резервного копирования и восстановления, а также определение ключевых тенденций на разных уровнях гранулярности.

Сегодня Azure Backup предоставляет решение для создания отчетов, которое использует [журналы Azure Monitor](https://docs.microsoft.com/azure/azure-monitor/log-query/get-started-portal) и [книги Azure](https://docs.microsoft.com/azure/azure-monitor/app/usage-workbooks), помогая получать подробные сведения о резервных копиях во всей резервной копии. В этой статье описывается настройка и просмотр отчетов резервного копирования.

## <a name="supported-scenarios"></a>Поддерживаемые сценарии

* Отчеты о резервном копировании поддерживаются для виртуальных машин Azure, SQL на виртуальных машинах Azure, SAP HANA/ASE на виртуальных машинах Azure, Azure Backup Agent (MARS), Azure Backup Server (MABS) и System Center DPM.
* Для рабочих нагрузок DPM отчеты о резервном копировании поддерживаются для DPM версии 5.1.363.0 и выше и агента версии 2.0.9127.0 и выше.
* Для рабочих нагрузок MABS отчеты о резервном копировании поддерживаются для MABS версии 13.0.415.0 и более поздних версий и агента версии 2.0.9170.0 и выше.
* Отчеты по резервному копированию можно просматривать во всех элементах, хранилищах, подписках и регионах, пока их данные отправляются в рабочую область Log Analytics (LA), к которой у пользователя есть доступ. 
* Если вы являетесь пользователем [Azure лигхсаусе](https://docs.microsoft.com/azure/lighthouse/) с делегированным доступом к подпискам ваших клиентов, вы можете использовать эти отчеты с Azure лигхсаусе для просмотра отчетов по всем клиентам.
* Данные для заданий резервного копирования журналов в настоящее время не отображаются в отчетах.

## <a name="getting-started"></a>Начало работы

Чтобы приступить к работе с отчетами, выполните три шага, описанные ниже.

1. **Создайте Log Analytics (LA) рабочую область (или используйте существующую):**

Необходимо настроить одну или несколько LA рабочих областей для хранения данных отчетов о резервном копировании. Расположение и подписка, в которых может быть создана эта LA Рабочая область, не зависит от расположения и подписки, где существуют хранилища. 

См. следующую статью: [создание log Analytics рабочей области на портал Azure](https://docs.microsoft.com/azure/azure-monitor/learn/quick-create-workspace) для настройки La Рабочая область.

По умолчанию данные в LA рабочей области сохранены в течение 30 дней. Чтобы просмотреть данные на более длительный период времени, измените срок хранения в LA Workspace. Чтобы изменить срок хранения, ознакомьтесь со следующей статьей: [Управление использованием и затратами с помощью журналов Azure Monitor](https://docs.microsoft.com/azure/azure-monitor/platform/manage-cost-storage).

2. **Настройте параметры диагностики для хранилищ:**

Azure Resource Manager ресурсы, такие как хранилища служб восстановления, записывают сведения о запланированных операциях и активируемых пользователем операциях в виде диагностических данных. 

В разделе мониторинг хранилища служб восстановления выберите **параметры диагностики** и укажите целевой объект для диагностических данных хранилища служб восстановления. Дополнительные [сведения об использовании диагностических событий](https://docs.microsoft.com/azure/backup/backup-azure-diagnostic-events).

![Колонка "параметры диагностики"](./media/backup-azure-configure-backup-reports/resource-specific-blade.png)

Azure Backup также предоставляет встроенную политику Azure, которая автоматизирует настройку параметров диагностики для всех хранилищ в заданной области. Сведения об использовании этой политики см. в следующей статье: [Настройка параметров диагностики хранилища в масштабе](https://docs.microsoft.com/azure/backup/azure-policy-configure-diagnostics)

3. **Просмотр отчетов на портал Azure:**

После настройки хранилищ для отправки данных в LA просмотрите отчеты о резервном копировании, перейдя в колонку любого хранилища и щелкнув пункт меню **отчеты резервного копирования** . 

![Панель мониторинга хранилища](./media/backup-azure-configure-backup-reports/vault-dashboard.png)

Щелкнув эту ссылку, вы откроете книгу отчет резервное копирование.

> [!NOTE]
> В настоящее время начальная загрузка отчета может занять до 1 минуты.

Ниже приведено описание различных вкладок, содержащихся в отчете.

* **Сводка** . на вкладке Сводка представлен общий обзор области резервного копирования. На вкладке Сводка можно получить краткий обзор общего числа элементов резервного копирования, общего объема использованного облачного хранилища, количества защищенных экземпляров и частоты успешных заданий для каждого типа рабочей нагрузки. Чтобы получить более подробные сведения о конкретном типе артефакта резервного копирования, перейдите к соответствующим вкладкам.

![Вкладка "Сводка"](./media/backup-azure-configure-backup-reports/summary.png)

* **Элементы архивации** . на вкладке "элементы архивации" можно просмотреть сведения и тенденции облачного хранилища, используемые на уровне элементов резервного копирования. Например, если вы используете SQL в резервной копии виртуальной машины Azure, вы можете увидеть, какие облачные хранилища используются для каждой резервной копии базы данных SQL. Вы также можете просмотреть данные для элементов резервного копирования с определенным состоянием защиты. Например, щелкнув плитку **защита остановлена** в верхней части вкладки, вы фильтруете все перечисленные ниже мини-приложения, чтобы отображать данные только для элементов резервного копирования в состоянии остановлено.

![Вкладка "архивные элементы"](./media/backup-azure-configure-backup-reports/backup-items.png)

* **Использование** . Вкладка "использование" позволяет просматривать ключевые параметры выставления счетов для резервных копий. Сведения, показанные на этой вкладке, находятся на уровне сущности выставления счетов (защищенном контейнере). Например, в случае резервного копирования сервера DPM в Azure можно просмотреть тенденцию защищенных экземпляров и облачного хранилища, используемого для сервера DPM. Аналогичным образом, если вы используете SQL в Azure Backup или SAP HANA в Azure Backup, на этой вкладке отображаются сведения об использовании на уровне виртуальной машины, в которой содержатся эти базы данных.

![Вкладка "использование"](./media/backup-azure-configure-backup-reports/usage.png)

* **Задания** . Вкладка задания позволяет просматривать длительные тенденции в заданиях, например число заданий, завершившихся сбоем, в день, а также основные причины сбоя задания. Эти сведения можно просмотреть как на уровне агрегирования, так и на уровне элементов резервного копирования. Щелкнув определенный элемент резервного копирования в сетке, вы сможете просмотреть подробные сведения о каждом задании, которое было запущено для этого элемента архивации в выбранном диапазоне времени.

![Вкладка "Задания"](./media/backup-azure-configure-backup-reports/jobs.png)

* **Политики** . на вкладке политики можно просмотреть сведения обо всех активных политиках, таких как число связанных элементов, и Общее облачное хранилище, используемое элементами, резервные копии которых были заданной политикой. Если щелкнуть определенную политику, вы сможете просмотреть сведения о каждом из связанных с ней резервных элементов.

![Вкладка "политики"](./media/backup-azure-configure-backup-reports/policies.png)

## <a name="exporting-to-excel"></a>Экспорт в Excel

При нажатии кнопки со стрелкой вниз в правом верхнем углу любого мини-приложения (таблицы или диаграммы) содержимое этого мини-приложения экспортируется как лист Excel с применением существующих фильтров. Чтобы экспортировать в Excel больше строк таблицы, можно увеличить число строк, отображаемых на странице, используя раскрывающийся список **строк на страницу** в верхней части каждой сетки.

## <a name="pinning-to-dashboard"></a>Закрепление на панели мониторинга

Чтобы закрепить мини-приложение на панели мониторинга портал Azure, щелкните значок закрепления в верхней части мини-приложения. Это позволяет создавать настраиваемые панели мониторинга, предназначенные для просмотра наиболее важных сведений.

## <a name="cross-tenant-reports"></a>Отчеты между клиентами

Если вы являетесь пользователем [Azure лигхсаусе](https://docs.microsoft.com/azure/lighthouse/) с делегированным доступом к подпискам в нескольких средах клиента, можно использовать фильтр подписок по умолчанию (щелкнув значок фильтра в правом верхнем углу портал Azure), чтобы выбрать все подписки, для которых вы хотите просмотреть данные. Это позволит выбрать LA рабочие области в клиентах для просмотра отчетов с несколькими клиентами.

## <a name="conventions-used-in-backup-reports"></a>Соглашения, используемые в отчетах резервного копирования

* Фильтры работают слева направо и сверху вниз на каждой вкладке, то есть любой фильтр применяется ко всем мини-приложениям, расположенным либо справа от этого фильтра, либо ниже этого фильтра. 
* Щелкнув цветную плитку, можно фильтровать мини-приложения под плиткой для записей, относящихся к значению этой плитки. Например, щелкнув плитку *защита остановлена* на вкладке Элементы архивации, вы фильтруете сетки и диаграммы ниже, чтобы отобразить данные для элементов резервного копирования в состоянии "Защита остановлено".
* Плитки, которые не являются цветными, нельзя щелкать.
* Данные по части текущего дня в отчетах не отображаются. Таким словами, если выбранное значение **диапазона времени** равно, за ***последние 7 дней***, в отчете отображаются записи за последние 7 завершенных дней (не включая текущий день).
* В отчете отображаются сведения о заданиях (помимо заданий журнала), которые были **активированы** в выбранном диапазоне времени. 
* Значения, показанные для облачного хранилища и защищенных экземпляров, находятся в **конце** выбранного диапазона времени.
* Элементы резервного копирования, отображаемые в отчетах, — это элементы, которые существуют в **конце** выбранного диапазона времени. Элементы резервного копирования, которые были удалены в середине выбранного диапазона времени, не отображаются. Для политик резервного копирования также применяется одно и то же соглашение.

## <a name="query-load-times"></a>Время загрузки запроса

Мини-приложения в отчете резервного копирования работают с Kusto запросами, которые выполняются в LA рабочей области пользователя. Так как эти запросы обычно включают обработку больших объемов данных с несколькими объединениями для получения более подробных сведений, мини-приложения могут загружаться мгновенно, когда пользователь просматривает отчеты в большой области резервного копирования. В таблице ниже приведена приблизительная оценка времени, которое может потребоваться для загрузки различных мини-приложений в зависимости от количества элементов резервного копирования и диапазона времени, за который просматриваются отчеты.

| **Число источников данных**                         | **Горизонт времени** | **Время загрузки (примерно)**                                              |
| --------------------------------- | ------------- | ------------------------------------------------------------ |
| ~ 5 КБ                       | 1 месяц          | Плитки: 5-10 с <br> Сетки: 5-10 с <br> Диаграммы: 5-10 с <br> Фильтры уровня отчета: 5-10 с|
| ~ 5 КБ                       | 3 месяца          | Плитки: 5-10 с <br> Сетки: 5-10 с <br> Диаграммы: 5-10 с <br> Фильтры уровня отчета: 5-10 с|
| ~ 10 КБ                       | 3 месяца          | Плитки: 15-20 с <br> Сетки: 15-20 с <br> Диаграммы: 1-2 мин <br> Фильтры уровня отчета: 25-30 с|
| ~ 15 КБ                       | 1 месяц          | Плитки: 15-20 с <br> Сетки: 15-20 с <br> Диаграммы: 50-60 с <br> Фильтры уровня отчета: 20-25 с|
| ~ 15 КБ                       | 3 месяца          | Плитки: 20-30 с <br> Сетки: 20-30 с <br> Диаграммы: 2-3 мин <br> Фильтры уровня отчета: 50-60 с |

## <a name="what-happened-to-the-power-bi-reports"></a>Что случилось с Power BI отчетами?
* Предыдущее приложение шаблона Power BI для создания отчетов (исходные данные из учетной записи хранения Azure) находится на пути устаревания. Рекомендуется начать отправку диагностических данных хранилища в Log Analytics, как описано выше, для просмотра отчетов.

* Кроме того, схема версии 1 для отправки диагностических данных в учетную запись хранения или Ла-рабочую область также находится на пути устаревания. Это означает, что если вы написали пользовательские запросы или службы автоматизации на основе схемы версии 1, рекомендуется обновить эти запросы, чтобы использовать текущую поддерживаемую схему v2.
