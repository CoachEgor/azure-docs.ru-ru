# <a name="configure-azure-backup-reports"></a>Настройка отчетов службы Azure Backup

Общим требованием для резервных админинов является получение информации о резервных копированиях на основе данных, которые охватывают длительный период времени. Примеры использования для такого решения включают:

 - Распределение и прогнозирование потребляемого облачного хранилища.
 - Аудит резервных и реставрационных работ.
 - Определение ключевых тенденций на различных уровнях детализации.

Сегодня резервное копирование Azure предоставляет решение для отчетов, в которое используются [журналы Azure Monitor](https://docs.microsoft.com/azure/azure-monitor/log-query/get-started-portal) и [рабочие книги Azure.](https://docs.microsoft.com/azure/azure-monitor/app/usage-workbooks) Эти ресурсы помогут вам получить богатую информацию о резервных системах по всему резервному поместью. В этой статье объясняется, как настроить и просматривать отчеты резервного копирования Azure.

## <a name="supported-scenarios"></a>Поддерживаемые сценарии

* Отчеты о резервном копировании поддерживаются для Azure VMs, S'L в Azure VMs, SAP HANA/ASE в Azure VMs, агента Microsoft Azure Recovery Services (MARS), сервера резервного копирования Microsoft Azure (MABS) и менеджера по защите данных Системного центра (DPM).
* Для рабочих нагрузок DPM отчеты резервного копирования поддерживаются для DPM Version 5.1.363.0 и выше и агентская версия 2.0.9127.0 и выше.
* Для рабочих нагрузок MABS отчеты резервного копирования поддерживаются для MABS Version 13.0.415.0 и выше и агентская версия 2.0.9170.0 и выше.
* Отчеты о резервном копировании можно просматривать во всех элементах резервного копирования, хранилищах, подписках и регионах, если их данные отправляются в рабочее пространство Log Analytics, к которому пользователь имеет доступ. Для просмотра отчетов для набора хранилищ достаточно иметь доступ к рабочему пространству Log Analytics, в которое хранилища отправляют свои данные. Вам не нужно иметь доступ к отдельным хранилищам.
* Если вы являетесь пользователем [Azure Lighthouse](https://docs.microsoft.com/azure/lighthouse/) с делегированным доступом к подпискам ваших клиентов, вы можете использовать эти отчеты с Azure Lighthouse для просмотра отчетов среди всех ваших клиентов.
* Данные для заданий резервного копирования журнала в настоящее время не отображаются в отчетах.

## <a name="get-started"></a>Начало работы

Выполните следующие действия, чтобы начать использовать отчеты.

#### <a name="1-create-a-log-analytics-workspace-or-use-an-existing-one"></a>1. Создание рабочего пространства анализа журналов или использование существующего

Настроите одно или несколько рабочих областей для хранения данных отчетов о резервном копировании. Местоположение и подписка, в которой может быть создано рабочее пространство Log Analytics, не зависит от местоположения и подписки, в которых существуют хранилища. 

Чтобы настроить рабочее пространство analytics log, [см.](https://docs.microsoft.com/azure/azure-monitor/learn/quick-create-workspace)

По умолчанию данные в рабочей области Log Analytics сохраняются в течение 30 дней. Чтобы просмотреть данные в течение более длительного периода времени, измените период хранения рабочего пространства Log Analytics. Чтобы изменить период хранения, [см. Управление использованием и затратами с помощью журналов Azure Monitor.](https://docs.microsoft.com/azure/azure-monitor/platform/manage-cost-storage)

#### <a name="2-configure-diagnostics-settings-for-your-vaults"></a>2. Настройка параметров диагностики для хранилищ

Ресурсы Менеджера ресурсов Azure, такие как хранилища служб восстановления, записывая информацию о запланированных операциях и пользовательских операциях в качестве диагностических данных. 

В разделе мониторинга хранилища служб восстановления выберите **параметры диагностики** и укажите цель диагностических данных хранилища восстановления. Чтобы узнать больше об использовании диагностических событий, [см.](https://docs.microsoft.com/azure/backup/backup-azure-diagnostic-events)

![Панель параметров диагностики](./media/backup-azure-configure-backup-reports/resource-specific-blade.png)

Azure Backup также предоставляет встроенную политику Azure, которая автоматизирует настройки диагностики для всех хранилищ в данной области. Чтобы узнать, как использовать эту политику, [см.](https://docs.microsoft.com/azure/backup/azure-policy-configure-diagnostics)

> [!NOTE]
> После настройки диагностики для завершения первоначального нажатия данных может потребоваться до 24 часов. После того, как данные начнут поступать в рабочее пространство Log Analytics, данные могут не отображаться сразу же, поскольку данные за текущий частичный день не отображаются в отчетах. Для получения дополнительной [информации см.](https://docs.microsoft.com/azure/backup/configure-reports#conventions-used-in-backup-reports) Мы рекомендуем вам начать просмотр отчетов через два дня после настройки хранилищ для отправки данных в log Analytics.

#### <a name="3-view-reports-in-the-azure-portal"></a>3. Просмотр отчетов на портале Azure

После настройки хранилищ для отправки данных в журнал Analytics просмотрите отчеты резервного копирования, перейдя в панель любого хранилища и выбрав **отчеты о резервном копировании.**

![Панель мониторинга хранилища](./media/backup-azure-configure-backup-reports/vault-dashboard.png)

Выберите эту ссылку, чтобы открыть справочную книгу отчета о резервном копировании.

> [!NOTE]
> * В настоящее время начальная загрузка отчета может занять до 1 минуты.
> * Хранилище служб восстановления является лишь отправной точкой для отчетов резервного копирования. После того, как рабочая книга отчета о резервном копировании откроется из панели хранилища, выберите подходящий набор рабочих областей Log Analytics, чтобы увидеть данные, агрегированные во всех хранилищах.

Отчет содержит различные вкладки:

* **Резюме**: Используйте эту вкладку, чтобы получить обзор на высоком уровне вашего резервного поместья. Можно быстро взглянуть на общее количество элементов резервного копирования, общее количество потребляемых облачных хранилищ, количество защищенных экземпляров и частоту успешности выполнения заданий на тип рабочей нагрузки. Для получения более подробной информации о конкретном типе артефакта резервного копирования перейдите на соответствующие вкладки.

   ![Вкладка Summary (Сводка)](./media/backup-azure-configure-backup-reports/summary.png)

* **Элементы резервного копирования**: Используйте эту вкладку, чтобы увидеть информацию и тенденции в облачном хранилище, потребляемом на уровне резервного копирования. Например, если вы используете S'L в резервном копировании Azure VM, можно увидеть облачное хранилище, потребленное для каждой базы данных S'L, которое резервируется. Вы также можете просмотреть данные для элементов резервного копирования определенного статуса защиты. Например, выбор плитки **Protection Stopped** в верхней части вкладки фильтрует все виджеты внизу, чтобы отображать данные только для элементов резервного копирования в состоянии остановленной защиты.

   ![Вкладка элементов резервного копирования](./media/backup-azure-configure-backup-reports/backup-items.png)

* **Использование**: Используйте эту вкладку для просмотра ключевых параметров выставления счетов для резервных копирования. Информация, отображаемая на этой вкладке, находится на уровне биллинговой сущности (защищенный контейнер). Например, в случае резервного копирования сервера DPM в Azure можно просмотреть тенденцию защищенных экземпляров и облачного хранилища, потребляемого для сервера DPM. Аналогичным образом, если вы используете S'L в резервном копировании Azure или SAP HANA в Azure Backup, эта вкладка предоставляет информацию, связанную с использованием, на уровне виртуальной машины, в которой содержатся эти базы данных.

   ![Вкладка использования](./media/backup-azure-configure-backup-reports/usage.png)

* **Вакансии**: Используйте эту вкладку для просмотра долгосрочных тенденций на заданиях, таких как количество неудачных заданий в день и основные причины сбоя в работе. Эту информацию можно просматривать как на агрегированном уровне, так и на уровне элемента резервного копирования. Выберите определенный элемент резервного копирования в сетке для просмотра подробной информации о каждом заданиях, которые были срабатываны на этом элементе резервного копирования в выбранном временном диапазоне.

   ![Вкладка "Задания"](./media/backup-azure-configure-backup-reports/jobs.png)

* **Политики:** Используйте эту вкладку для просмотра информации обо всех активных политиках, таких как количество связанных элементов и общее хранилище облачных технологий, потребляемых элементами, резервными в рамках данной политики. Выберите конкретную политику для просмотра информации по каждому из связанных с ней элементов резервного копирования.

   ![Вкладка политики](./media/backup-azure-configure-backup-reports/policies.png)

## <a name="export-to-excel"></a>Экспорт в Excel

Выберите кнопку вниз стрелка в правом верхнем углу любого виджета, как таблица или диаграмма, экспортировать содержимое этого виджета, как лист Excel, как есть с существующими фильтрами применяется. Для экспорта большего количества строк таблицы в Excel можно увеличить количество строк, отображаемых на странице, используя стрелку **строк на странице** выпадения вниз в верхней части каждой сетки.

## <a name="pin-to-dashboard"></a>Закрепить на панели мониторинга

Выберите кнопку контакта в верхней части каждого виджета, чтобы прикрепить виджет к панели мониторинга портала Azure. Эта функция помогает создавать индивидуальные панели мониторинга, адаптированные для отображения наиболее важной информации, которая вам нужна.

## <a name="cross-tenant-reports"></a>Отчеты о кросс-арендаторах

Если вы используете [Azure Lighthouse](https://docs.microsoft.com/azure/lighthouse/) с делегированным доступом к подпискам в нескольких средах для арендаторов, можно использовать фильтр подписки по умолчанию. Выберите кнопку фильтра в правом верхнем углу портала Azure, чтобы выбрать все подписки, для которых вы хотите видеть данные. Это позволяет выбрать рабочие места Log Analytics в разностороннем доступе к многотенантным отчетам.

## <a name="conventions-used-in-backup-reports"></a>Конвенции, используемые в отчетах о резервном копировании

* Фильтры работают слева направо и сверху вниз на каждой вкладке. То есть, любой фильтр применяется только ко всем тем виджетам, которые расположены либо справа от этого фильтра, либо ниже этого фильтра. 
* Выбор цветной плитки фильтрует виджеты под плиткой для записей, относящихся к значению этой плитки. Например, выбор плитки **Protection Stopped** на вкладке **Элементы резервного копирования** фильтрует сетки и диаграммы ниже, чтобы показать данные для элементов резервного копирования в состоянии остановленной защиты.
* Плитки, которые не окрашены, не являются интерактивными.
* Данные за текущий частичный день не отображаться в отчетах. Таким образом, когда выбранное значение **диапазона времени** составляет **последние 7 дней,** отчет показывает записи за последние семь завершенных дней. Текущий день не включен.
* Отчет показывает детали заданий (кроме заданий журнала), которые были *срабатываны* в выбранном временном диапазоне. 
* Значения, отображаемые для **облачных хранилищ** и **защищенных экземпляров,** находятся в *конце* выбранного диапазона времени.
* Элементы резервного копирования, отображаемые в отчетах, являются теми элементами, которые существуют в *конце* выбранного диапазона времени. Элементы резервного копирования, удаленные в середине выбранного диапазона времени, не отображаются. Та же конвенция применяется и к политикам резервного копирования.

## <a name="query-load-times"></a>Время загрузки запроса

Виджеты в отчете о резервном копировании питаются от запросов Kusto, которые работают на рабочих пространствах анализа журнала пользователя. Эти запросы обычно включают обработку больших объемов данных, с несколькими соединениями, чтобы позволить более богатые идеи. В результате виджеты могут не загружаться мгновенно, когда пользователь просматривает отчеты в большом резервном поместья. В этой таблице приведена приблизительная оценка времени загрузки различных виджетов в зависимости от количества элементов резервного копирования и диапазона времени, для которого просматривается отчет.

| **Источники данных**                         | **Горизонт времени** | **Приблизительное время загрузки**                                              |
| --------------------------------- | ------------- | ------------------------------------------------------------ |
| No 5 K                       | 1 месяц          | Плитки: 5-10 сек <br> Сетки: 5-10 сек <br> Диаграммы: 5-10 сек <br> Фильтры уровня отчета: 5-10 сек|
| No 5 K                       | 3 месяца          | Плитки: 5-10 сек <br> Сетки: 5-10 сек <br> Диаграммы: 5-10 сек <br> Фильтры уровня отчета: 5-10 сек|
| 10 K                       | 3 месяца          | Плитки: 15-20 сек <br> Сетки: 15-20 сек <br> Графики: 1-2 мин. <br> Фильтры уровня отчета: 25-30 сек|
| No 15 K                       | 1 месяц          | Плитки: 15-20 сек <br> Сетки: 15-20 сек <br> Диаграммы: 50-60 сек <br> Фильтры уровня отчета: 20-25 сек|
| No 15 K                       | 3 месяца          | Плитки: 20-30 сек <br> Сетки: 20-30 сек <br> Графики: 2-3 мин. <br> Фильтры уровня отчета: 50-60 сек |

## <a name="what-happened-to-the-power-bi-reports"></a> Что произошло с отчетами Power BI? 
* Предыдущее приложение шаблона Power BI для отчетности, которое исходило от учетной записи хранения Azure, находится на пути амортизации. Мы рекомендуем начать отправку диагностических данных хранилища в журнал Analytics для просмотра отчетов.

* Кроме того, схема V1 по отправке диагностических данных на учетную запись хранения или рабочее пространство Log Analytics также находится на пути амортизации. Если вы написали какие-либо пользовательские запросы или автоматизацию на основе схемы V1, рекомендуется обновить эти запросы, чтобы использовать поддерживаемую в настоящее время схему V2.
