# <a name="configure-azure-backup-reports"></a>Настройка отчетов службы Azure Backup

Распространенным требованием для администраторов резервного копирования является получение ценных сведений о резервных копиях на основе данных, охватывающих длительный период времени. Ниже перечислены варианты использования такого решения.

- Выделение и прогнозирование использования облачного хранилища.
- Аудит резервного копирования и восстановления.
- Определение ключевых тенденций на разных уровнях гранулярности.

Сегодня Azure Backup предоставляет решение для создания отчетов, которое использует [журналы Azure Monitor](https://docs.microsoft.com/azure/azure-monitor/log-query/get-started-portal) и [книги Azure](https://docs.microsoft.com/azure/azure-monitor/app/usage-workbooks). Эти ресурсы помогут вам получить подробные сведения о резервных копиях во всей резервной копии. В этой статье объясняется, как настроить и просмотреть отчеты Azure Backup.

## <a name="supported-scenarios"></a>Поддерживаемые сценарии

- Отчеты о резервном копировании поддерживаются для виртуальных машин Azure, SQL на виртуальных машинах Azure, SAP HANA/ASE на виртуальных машинах Azure, агентах Службы восстановления Microsoft Azure (MARS), Microsoft Azure Backup Server (MABS) и System Center Data Protection Manager (DPM). Данные для резервного копирования файловых ресурсов Azure в настоящее время не отображаются в отчетах по резервному копированию.
- Для рабочих нагрузок DPM отчеты о резервном копировании поддерживаются для DPM версии 5.1.363.0 и выше и агента версии 2.0.9127.0 и выше.
- Для рабочих нагрузок MABS отчеты о резервном копировании поддерживаются для MABS версии 13.0.415.0 и выше, а агент — 2.0.9170.0 и более поздних версий.
- Отчеты о резервном копировании можно просматривать во всех элементах, хранилищах, подписках и регионах, если их данные отправляются в Log Analytics рабочую область, к которой у пользователя есть доступ. Для просмотра отчетов по набору хранилищ необходимо иметь доступ только для чтения к рабочей области Log Analytics, в которую хранилища отправляют свои данные. Вам не нужно иметь доступ к отдельным хранилищам.
- Если вы являетесь пользователем [Azure лигхсаусе](https://docs.microsoft.com/azure/lighthouse/) с делегированным доступом к подпискам ваших клиентов, вы можете использовать эти отчеты с Azure лигхсаусе для просмотра отчетов по всем клиентам.
- Данные для заданий резервного копирования журналов в настоящее время не отображаются в отчетах.

## <a name="get-started"></a>Начало работы

Чтобы начать работу с отчетами, выполните следующие действия.

### <a name="1-create-a-log-analytics-workspace-or-use-an-existing-one"></a>1. Создайте Log Analytics рабочую область или используйте существующую.

Настройте одну или несколько рабочих областей Log Analytics для хранения данных отчетов о резервном копировании. Расположение и подписка, в которых может быть создана эта Log Analyticsная Рабочая область, не зависит от расположения и подписки, где существуют хранилища.

Сведения о настройке рабочей области Log Analytics см. в разделе [создание log Analytics рабочей области на портал Azure](https://docs.microsoft.com/azure/azure-monitor/learn/quick-create-workspace).

По умолчанию данные в рабочей области Log Analytics храниться в течение 30 дней. Чтобы просмотреть данные на более длительный период времени, измените срок хранения Log Analytics рабочей области. Сведения об изменении срока хранения см. в статье [Управление использованием и затратами с помощью журналов Azure Monitor](https://docs.microsoft.com/azure/azure-monitor/platform/manage-cost-storage).

### <a name="2-configure-diagnostics-settings-for-your-vaults"></a>2. Настройка параметров диагностики для хранилищ

Azure Resource Manager ресурсы, такие как хранилища служб восстановления, записывают сведения о запланированных операциях и активируемых пользователем операциях в виде диагностических данных.

В разделе мониторинг хранилища служб восстановления выберите **параметры диагностики** и укажите целевой объект для диагностических данных хранилища служб восстановления. Дополнительные сведения об использовании диагностических событий см. в статье [Использование параметров диагностики для хранилищ служб восстановления](https://docs.microsoft.com/azure/backup/backup-azure-diagnostic-events).

![Панель параметров диагностики](./media/backup-azure-configure-backup-reports/resource-specific-blade.png)

Azure Backup также предоставляет встроенное определение политики Azure, которое автоматизирует настройку параметров диагностики для всех хранилищ в заданной области. Сведения об использовании этой политики см. в статье [Настройка параметров диагностики хранилища в масштабе](https://docs.microsoft.com/azure/backup/azure-policy-configure-diagnostics).

> [!NOTE]
> После настройки диагностики для завершения отправки начальных данных может потребоваться до 24 часов. После того как данные переводятся в рабочую область Log Analytics, данные в отчетах могут не отображаться немедленно, поскольку данные за текущий день не отображаются в отчетах. Дополнительные сведения см. [в разделе соглашения, используемые в отчетах резервного копирования](https://docs.microsoft.com/azure/backup/configure-reports#conventions-used-in-backup-reports). Рекомендуется начать просмотр отчетов через два дня после настройки хранилищ для отправки данных в Log Analytics.

#### <a name="3-view-reports-in-the-azure-portal"></a>3. Просмотр отчетов в портал Azure

После настройки хранилищ для отправки данных в Log Analytics просмотрите отчеты о резервном копировании, перейдя на панель любого хранилища и выбрав пункт **отчеты резервного копирования**.

![Панель мониторинга хранилища](./media/backup-azure-configure-backup-reports/vault-dashboard.png)

Выберите эту ссылку, чтобы открыть книгу отчет резервное копирование.

> [!NOTE]
>
> - В настоящее время начальная загрузка отчета может занять до 1 минуты.
> - Хранилище служб восстановления — это просто точка входа для резервных отчетов. После открытия книги "отчет о резервном копировании" из области хранилища выберите соответствующий набор Log Analytics рабочих областей, чтобы просмотреть данные, агрегированные по всем хранилищам.

Отчет содержит различные вкладки:

- **Сводка**. Используйте эту вкладку, чтобы получить общий обзор пространства резервного копирования. Вы можете получить краткий обзор общего числа элементов резервного копирования, общего объема использованного облачного хранилища, количества защищенных экземпляров и частоты успешных заданий для каждого типа рабочей нагрузки. Чтобы получить более подробные сведения о конкретном типе артефакта резервного копирования, перейдите на соответствующие вкладки.

   ![Вкладка Summary (Сводка)](./media/backup-azure-configure-backup-reports/summary.png)

- **Архивные элементы**. Используйте эту вкладку, чтобы просмотреть сведения и тенденции в облачном хранилище, потребленном на уровне элементов резервного копирования. Например, если вы используете SQL в резервной копии виртуальной машины Azure, вы можете увидеть, как используется облачное хранилище для каждой базы данных SQL, для которой выполняется резервное копирование. Вы также можете просмотреть данные для элементов резервного копирования с определенным состоянием защиты. Например, если щелкнуть плитку **защита остановлена** в верхней части вкладки, все мини-приложения отобразятся под, чтобы отобразить данные только для элементов резервного копирования в состоянии остановлена защита.

   ![Вкладка "архивные элементы"](./media/backup-azure-configure-backup-reports/backup-items.png)

- **Использование**. Используйте эту вкладку, чтобы просмотреть ключевые параметры выставления счетов для резервных копий. Сведения, отображаемые на этой вкладке, находятся на уровне сущности выставления счетов (защищенном контейнере). Например, в случае резервного копирования сервера DPM в Azure можно просмотреть тенденцию защищенных экземпляров и облачного хранилища, используемого для сервера DPM. Аналогичным образом, если вы используете SQL в Azure Backup или SAP HANA в Azure Backup, на этой вкладке отображаются сведения об использовании на уровне виртуальной машины, в которой содержатся эти базы данных.

   ![Вкладка "использование"](./media/backup-azure-configure-backup-reports/usage.png)

- **Задания**. Используйте эту вкладку, чтобы просматривать длительные тенденции в заданиях, например количество невыполненных заданий в день и основные причины сбоя задания. Эти сведения можно просмотреть на уровне статистической обработки и на уровне элементов резервного копирования. Выберите определенный элемент резервного копирования в сетке, чтобы просмотреть подробные сведения о каждом задании, которое было запущено для этого элемента архивации в выбранном диапазоне времени.

   ![Вкладка "Задания"](./media/backup-azure-configure-backup-reports/jobs.png)

- **Политики**. Используйте эту вкладку для просмотра сведений обо всех активных политиках, таких как число связанных элементов и Общее облачное хранилище, использованное элементами, резервные копии которых были заданной политикой. Выберите конкретную политику, чтобы просмотреть сведения о каждом из связанных элементов архивации.

   ![Вкладка "политики"](./media/backup-azure-configure-backup-reports/policies.png)

## <a name="export-to-excel"></a>Экспорт в Excel

Нажмите кнопку со стрелкой вниз в правом верхнем углу любого мини-приложения, например таблицы или диаграммы, чтобы экспортировать содержимое этого мини-приложения в виде таблицы Excel "как есть" с применением существующих фильтров. Чтобы экспортировать больше строк таблицы в Excel, можно увеличить число строк, отображаемых на странице, с помощью раскрывающегося списка **строки на страницу** в верхней части каждой сетки.

## <a name="pin-to-dashboard"></a>Закрепить на панели мониторинга

Чтобы закрепить мини-приложение на панели мониторинга портал Azure, нажмите кнопку закрепить в верхней части мини-приложения. Эта функция позволяет создавать настраиваемые панели мониторинга, предназначенные для просмотра наиболее важных сведений.

## <a name="cross-tenant-reports"></a>Отчеты между клиентами

Если вы используете [Azure лигхсаусе](https://docs.microsoft.com/azure/lighthouse/) с делегированным доступом к подпискам в нескольких средах клиента, можно использовать фильтр подписок по умолчанию. Нажмите кнопку Фильтр в правом верхнем углу портал Azure, чтобы выбрать все подписки, для которых требуется просмотреть данные. Это позволит выбрать Log Analytics рабочих областей в клиентах для просмотра отчетов с несколькими клиентами.

## <a name="conventions-used-in-backup-reports"></a>Соглашения, используемые в отчетах резервного копирования

- Фильтры работают слева направо и сверху вниз на каждой вкладке. То есть любой фильтр применяется ко всем мини-приложениям, расположенным либо справа от этого фильтра, либо ниже этого фильтра.
- Выбор цветовой плитки фильтрует мини-приложения под плиткой для записей, относящихся к значению этой плитки. Например, если щелкнуть плитку **защита остановлена** на вкладке **элементы архивации** , в сетке и диаграммах будет отфильтрованы сетки и диаграммы, чтобы отобразить данные для элементов резервного копирования в состоянии остановлена защита.
- Плитки, которые не являются цветными, не щелкают.
- Данные за текущий неполный день не отображаются в отчетах. Таким образом, если выбранное значение **диапазона времени** составляет **последние 7 дней**, в отчете отображаются записи за последние семь завершенных дней. Текущий день не включается.
- В отчете отображаются сведения о заданиях (помимо заданий журнала), которые были *активированы* в выбранном диапазоне времени.
- Значения, показанные для **облачного хранилища** и **защищенных экземпляров** , находятся в *конце* выбранного диапазона времени.
- Элементы резервного копирования, отображаемые в отчетах, — это элементы, которые существуют в *конце* выбранного диапазона времени. Элементы архивации, удаленные в середине выбранного диапазона времени, не отображаются. Для политик резервного копирования также применяется одно и то же соглашение.

## <a name="query-load-times"></a>Время загрузки запроса

Мини-приложения в отчете резервного копирования работают с Kusto запросами, которые выполняются в рабочих областях пользователя Log Analytics. Эти запросы обычно включают обработку больших объемов данных с несколькими объединениями для обеспечения расширенной аналитики. В результате мини-приложения могут не загружаться мгновенно, когда пользователь просматривает отчеты в большой области резервного копирования. Эта таблица представляет собой приблизительную оценку времени, которое могут предпринять различные мини-приложения в зависимости от количества элементов резервного копирования и диапазона времени, за который просматриваются отчеты.

| **Число источников данных**                         | **Горизонт времени** | **Приблизительное время загрузки**                                              |
| --------------------------------- | ------------- | ------------------------------------------------------------ |
| ~ 5 КБ                       | 1 месяц          | Плитки: 5-10 с <br> Сетки: 5-10 с <br> Диаграммы: 5-10 с <br> Фильтры уровня отчета: 5-10 с|
| ~ 5 КБ                       | 3 месяца          | Плитки: 5-10 с <br> Сетки: 5-10 с <br> Диаграммы: 5-10 с <br> Фильтры уровня отчета: 5-10 с|
| ~ 10 КБ                       | 3 месяца          | Плитки: 15-20 с <br> Сетки: 15-20 с <br> Диаграммы: 1-2 мин <br> Фильтры уровня отчета: 25-30 с|
| ~ 15 КБ                       | 1 месяц          | Плитки: 15-20 с <br> Сетки: 15-20 с <br> Диаграммы: 50-60 с <br> Фильтры уровня отчета: 20-25 с|
| ~ 15 КБ                       | 3 месяца          | Плитки: 20-30 с <br> Сетки: 20-30 с <br> Диаграммы: 2-3 мин <br> Фильтры уровня отчета: 50-60 с |

## <a name="what-happened-to-the-power-bi-reports"></a> Что произошло с отчетами Power BI? 

- Предыдущее приложение шаблона Power BI для создания отчетов, источник данных из учетной записи хранения Azure, находится на пути устаревания. Рекомендуется начать отправку диагностических данных хранилища в Log Analytics для просмотра отчетов.

- * Кроме того, [схема](https://docs.microsoft.com/azure/backup/backup-azure-diagnostics-mode-data-model#v1-schema-vs-v2-schema) версии 1 для отправки диагностических данных в учетную запись хранения или Ла-рабочую область также находится на пути устаревания. Это означает, что если вы написали пользовательские запросы или службы автоматизации на основе схемы версии 1, рекомендуется обновить эти запросы, чтобы использовать текущую поддерживаемую схему v2.
