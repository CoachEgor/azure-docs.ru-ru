# <a name="understand-the-automatic-migration-process-for-your-classic-alert-rules"></a>Понимание процесса автоматической миграции для классических правил оповещения

Как [было объявлено ранее,](monitoring-classic-retirement.md)классические оповещения в Azure Monitor удаляются в сентябре 2019 года (изначально был июль 2019 года). В рамках процесса выхода на пенсию на портале Azure доступен [инструмент миграции,](alerts-using-migration-tool.md) который клиенты сами могут инициировать миграцию. Если вы не использовали инструмент миграции до 31 августа 2019 года, Azure Monitor начнет процесс автоматической миграции классических оповещений с 1 сентября 2019 года.
Эта статья поможет вам решить любые проблемы, с которыми вы можете столкнуться.

  > [!NOTE]
  > Эта статья относится только к общедоступному облаку Azure. Процесс выхода на пенсию для классических оповещений Azure Monitor в облаке Azure Government и Azure China 21Vianet будет объявлен в будущем.

## <a name="what-will-happen-during-the-automatic-migration-process"></a>Что произойдет в процессе автоматической миграции?

- С **1 сентября 2019**года клиенты не смогут создавать новые классические правила оповещения, за исключением [определенных метрик.](alerts-understand-migration.md#classic-alert-rules-that-will-not-be-migrated)
  - Для исключений клиент может продолжать создавать новые классические правила оповещения и использовать свои классические оповещения до июня 2020 года.
- С **1 сентября 2019**года миграция классических оповещений будет срабатывать партиями для любых клиентов с классическими оповещениями.
- Как и в случае с инструментом добровольной миграции, некоторые классические правила оповещения, которые не являются миграционными, будут оставлены такими, как есть. Эти классические правила оповещения будут поддерживаться до июня 2020 года. Однако любые недействительные правила классического оповещения будут удалены, поскольку они нефункционирую.
Любые классические правила оповещения, которые отслеживают удаленные целевые ресурсы или [метрики, которые больше не поддерживаются,](alerts-understand-migration.md#classic-alert-rules-on-deprecated-metrics) считаются недействительными.
- Как только начинается миграция для подписки, если нет проблем, миграция должна быть завершена в течение часа. Клиенты могут следить за состоянием миграции на [лезвии миграции в Azure Monitor.](https://portal.azure.com/#blade/Microsoft_Azure_Monitoring/MigrationBladeViewModel)
- Владельцы подписок получат электронное письмо об успешном завершении миграции.
- Если есть какие-либо проблемы во время миграции, владельцы подписки также получат электронное письмо, информирующее их о том же. Клиенты могут использовать лезвие миграции, чтобы увидеть полную информацию о проблеме.
- В случае необходимости действия от клиента, например временного отключения блокировки ресурсов или изменения назначения политики, клиенты должны будут решить любые проблемы до 31 октября 2019 года. Если проблемы не будут решены к тому времени, успешная миграция классических оповещений не может быть гарантирована.

    > [!NOTE]
    > Если вы не хотите ждать начала автоматического процесса миграции, вы все равно можете запустить миграцию добровольно с помощью инструмента миграции.

## <a name="important-things-to-note"></a>Важные вещи, чтобы отметить

Процесс миграции преобразует классические правила оповещения в новые, эквивалентные правила оповещения и создает группы действий. В процессе подготовки, будьте в курсе следующих моментов:

- Форматы полезной нагрузки уведомлений для новых правил оповещения отличаются от форматов классических правил оповещения, поскольку они поддерживают больше функций. Если у вас есть логические приложения, runbooks или webhooks, которые вызваны классическим правилом оповещения, они могут перестать функционировать, как ожидалось, как только миграция будет завершена из-за различий в полезной нагрузке уведомлений. [Узнайте, как подготовиться к миграции.](alerts-prepare-migration.md)

- Некоторые классические правила оповещения не могут быть перенесены с помощью инструмента. [Узнайте, какие правила не могут быть перенесены и что с ними делать.](alerts-understand-migration.md#classic-alert-rules-that-will-not-be-migrated)

    > [!NOTE]
    > Процесс миграции не повлияет на оценку классических правил оповещения. Они будут продолжать выполнять и отправлять оповещения до тех пор, пока они не будут перенесены и новые правила оповещения не всменяются.

## <a name="what-if-the-automatic-migration-fails"></a>Что делать, если автоматическая миграция выходит из строя?

При сбое автоматического процесса миграции владельцы подписок получат электронное письмо с уведомлением о проблеме. Для получения полной информации о проблеме можно использовать лезвие миграции в Azure Monitor.

Ознакомьтесь с руководством по [устранению неполадок](alerts-understand-migration.md#common-problems-and-remedies) для решения проблем, с которыми вы можете столкнуться во время миграции.

  > [!NOTE]
  > В случае необходимости действия от клиента, например временного отключения блокировки ресурсов или изменения назначения политики, клиенты должны будут решить любые проблемы до 31 октября 2019 года. Если проблемы не будут решены к тому времени, успешная миграция классических оповещений не может быть гарантирована.
