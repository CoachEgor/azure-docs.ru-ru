# <a name="monitor-your-kubernetes-cluster-performance-with-azure-monitor-for-containers"></a>Мониторинг производительности кластера Kubernetes с помощью Azure Monitor для контейнеров

С помощью Azure Monitor для контейнеров можно использовать диаграммы производительности и состояние работоспособности для мониторинга рабочей нагрузки кластеров Kubernetes, размещенных в службе Azure Kubernetes (AKS), Azure Stack или другой среде с двух точек зрения. Вы можете контролировать непосредственно из кластера, или вы можете просмотреть все кластеры в подписке от Azure Monitor. Просмотр экземпляров контейнеров Azure также возможен при мониторинге конкретного кластера AKS.

Эта статья поможет вам понять две точки зрения, а также то, как Azure Monitor помогает быстро оценивать, исследовать и устранять обнаруженные проблемы.

Для получения информации о том, как включить Azure Monitor для контейнеров, можно ознакомиться на [борту Azure Monitor для контейнеров.](container-insights-onboard.md)

Azure Monitor предоставляет многокластерное представление, которое показывает состояние работоспособности всех контролируемых кластеров Kubernetes под управлением Linux и Windows Server 2019, развернутых в группах ресурсов в ваших подписях. Он показывает кластеры, обнаруженные во всех средах, которые не контролируются решением. Вы можете сразу понять работоспособность кластера, а отсюда можно просверлить страницу производительности узлов и контроллеров или сориентироваться, чтобы увидеть диаграммы производительности для кластера. Для кластеров AKS, которые были обнаружены и идентифицированы как неконтролируемые, можно включить мониторинг для них в любое время. 

Основные различия в мониторинге кластера Windows Server с Azure Monitor для контейнеров по сравнению с кластером Linux описаны [здесь](container-insights-overview.md#what-does-azure-monitor-for-containers-provide) в обзорной статье.

## <a name="multi-cluster-view-from-azure-monitor"></a>Мультикластерное представление из Azure Monitor

Для просмотра состояния работоспособности всех развернутых кластеров Kubernetes выберите **Монитор** из левой панели на портале Azure. В разделе **Аналитика** выберите **Контейнеры**. 

![Пример мультикластерной панели мониторинга Azure Monitor](./media/container-insights-analyze/azmon-containers-multiview.png)

Можно определить результаты, представленные в сетке, чтобы показать кластеры, которые:

* **Кластеры Azure** - AKS и AKS-Engine, размещенные в службе Azure Kubernetes
* **Azure Stack (Предварительный просмотр)** - кластеры AKS-Engine, размещенные на Azure Stack
* **Non-Azure (Предварительный просмотр)** - Кластеры Kubernetes, размещенные на территории
* **Все** - Просмотр всех кластеров Kubernetes, размещенных в Azure, Azure Stack, и на бортовые среды, которые находятся на борту Azure Monitor для контейнеров

Для просмотра кластеров из определенной среды выберите их из **таблетки Среды** в левом верхнем углу страницы.

![Пример селектора пилюльки окружающей среды](./media/container-insights-analyze/clusters-multiview-environment-pill.png)

На вкладке **«Мониторинг кластеров»** вы узнаете следующее:

- Сколько кластеров находятся в критическом или неработоспособном состоянии, по сравнению с тем, сколько из них являются здоровыми или не сообщают (именуемые неизвестным состоянием).
- Здоровы ли все развертывания [Azure Kubernetes Engine (AKS-engine).](https://github.com/Azure/aks-engine)
- Сколько узлов и пользовательских и системных стручков развернуто на кластере.
- Сколько дискового пространства доступно и есть ли проблема с емкостью.

Здесь используются следующие состояния работоспособности: 

* **Здоровый**: Никаких проблем не обнаружено для VM, и он функционирует по мере необходимости. 
* **Критические:** Обнаружены одна или несколько критических проблем, которые необходимо устранить для восстановления нормального операционного состояния, как ожидалось.
* **Предупреждение**: Обнаружены одна или несколько проблем, которые необходимо решить или состояние здоровья может стать критическим.
* **Неизвестно:** Если служба не смогла установить связь с узлами или стручком, статус изменяется в неизвестное состояние.
* **Не найдено:** либо рабочее пространство, группа ресурсов или подписка, содержащая рабочее пространство для этого решения, были удалены.
* **Несанкционированное**: Пользователь не имеет необходимых разрешений для чтения данных в рабочей области.
* **Ошибка**: Ошибка произошла при попытке прочитать данные из рабочего пространства.
* **Неправильно настроенный:** Azure Monitor для контейнеров не был настроен правильно в указанном рабочем пространстве.
* **Нет данных**: Данные не отчитались в рабочей области за последние 30 минут.

Состояние работоспособности вычисляет общий статус кластера как *худший из* трех состояний за одним исключением. Если какое-либо из трех состояний неизвестно, общее состояние кластера отображает **сяочку Неизвестного**.

## <a name="view-performance-directly-from-a-cluster"></a>Просмотр производительности непосредственно из кластера

Доступ к Azure Monitor для контейнеров доступен непосредственно из кластера AKS, выбирая**кластер** **Insights** > из левой панели или при выборе кластера из многокластерного представления. Информация о вашем кластере разделена на четыре точки зрения:

- Кластер
- Узлы 
- Controllers; 
- Контейнеры

>[!NOTE]
>Опыт, описанный в оставшейся части этой статьи, также применим для просмотра производительности и состояния работоспособности кластеров Kubernetes, размещенных в Azure Stack или другой среде при выборе из многокластерного представления. 

Страница по умолчанию открывается и отображает четыре диаграммы производительности строки, отображающие ключевые показатели производительности кластера. 

![Пример диаграммы производительности на вкладке "Кластер"](./media/container-insights-analyze/containers-cluster-perfview.png)

На диаграммах производительности отображаются четыре метрики производительности:

- **Использование&nbsp;процессора узла**: агрегированная перспектива использования процессора для всего кластера. Чтобы отфильтровать результаты для временной диапазон, выберите **Avg,** **Min,** **50th,** **90th,** **95th,** или **Max** в селекторе процентилей выше графика. Фильтры можно использовать как индивидуально, так и комбинировать. 
- **Использование&nbsp;памяти узлов**: агрегированная перспектива использования памяти для всего кластера. Чтобы отфильтровать результаты для временной диапазон, выберите **Avg,** **Min,** **50th,** **90th,** **95th,** или **Max** в селекторе процентилей выше графика. Фильтры можно использовать как индивидуально, так и комбинировать. 
- **Число узлов**: число узлов и сведения о состоянии, полученные из Kubernetes. Статусы представленных кластерных узлов — Это всего, готово и не готово. Они могут быть отфильтрованы по отдельности или объединены в селектора над графиком. 
- **Активное количество стручков**: количество стручков и состояние от Kubernetes. Статусы представленных стручков: Total, Pending, Running, Unknown, Succeeded или Failed. Они могут быть отфильтрованы по отдельности или объединены в селектора над графиком. 

Используйте клавиши левой и правой стрелки для цикла через каждую точку данных на диаграмме. Используйте клавиши стрелки Вверх и вниз, чтобы проехать через линии процентиля. Выберите значок штифта в правом верхнем углу любой из диаграмм, чтобы прикрепить выбранную диаграмму к последней панели мониторинга Azure, которая была просмотрена. С панели мониторинга можно изменить размер и изменить положение диаграммы. Выбор диаграммы с панели мониторинга перенаправляет вас в Azure Monitor для контейнеров и загружает правильный объем и представление.

Azure Monitor для контейнеров также поддерживает [explorer метрик](../platform/metrics-getting-started.md)Azure Monitor, где можно создавать собственные графики сюжетов, соотносить и исследовать тенденции, а также прикреплять к панелям мониторинга. От исследователя метрик можно также использовать критерии, которые вы установили для визуализации метрик в качестве основы [правила оповещения на основе метрики.](../platform/alerts-metric.md)

## <a name="analyze-nodes-controllers-and-container-health"></a>Анализ узлов, контроллеров и работоспособности контейнеров

При переключении на **вкладки узлов,** **контроллеров**и **контейнеров** панель свойства автоматически отображается на правой стороне страницы. Он показывает свойства выбранного элемента, который включает в себя определенные метки для организации объектов Kubernetes. При выборе узла Linux в разделе **«Локальный дисковый потенциал»** также отображается доступное дисковое пространство и процент, используемый для каждого диска, представленного узлам. Выберите **>>** ссылку в панели для просмотра или скрыть панель.

При развертывании объектов в иерархии панель свойств обновляется на основе выбранного объекта. С панели также можно просматривать журналы контейнеров Kubernetes (stdout/stderror), события и метрики стручка, выбрав ссылку **View Live data (предварительный просмотр)** в верхней части панели. Для получения дополнительной информации о конфигурации, необходимой для предоставления и контроля доступа к просмотру этих данных, [см.](container-insights-livedata-setup.md) При просмотре кластерных ресурсов эти данные можно увидеть из контейнера в режиме реального времени. Для получения дополнительной информации об этой функции смотрите [Как просматривать журналы Kubernetes, события и метрики стручка в режиме реального времени.](container-insights-livedata-overview.md) Для просмотра данных журнала Kubernetes, хранящихся в рабочей области на основе заранее определенных поисков журналов, выберите **журналы контейнеров View** из списка выпадающих **аналитических** данных View. Для получения дополнительной информации по этой теме смотрите [журналы поиска для анализа данных.](container-insights-log-search.md#search-logs-to-analyze-data)

Используйте опцию **«Добавить фильтр»** в верхней части страницы, чтобы отфильтровать результаты для представления **службой,** **узелом,** **namespace**или **узелом.** После выбора области фильтра выберите одно из значений, отображаемых в поле **значения Select (ы).** После настройки фильтра он применяется глобально при просмотре любой точки зрения кластера AKS. Формула поддерживает только знак равенства. Поверх первого фильтра можно добавить дополнительные фильтры, чтобы сузить диапазон результатов. Например, если вы указываете фильтр по **узелу,** можно выбрать только **службу** или **namespace** для второго фильтра.

Указание фильтра в одной вкладке продолжает применяться при выборе другой. Он удаляется после выбора символа **x** рядом с указанным фильтром. 

Переключитесь на вкладку **узла** и иерархию строк, следует модели объекта Kubernetes, которая начинается с узла в кластере. Расширьте узла для просмотра одного или нескольких стручков, работающих на уде. Если несколько контейнеров сгруппированы в стручок, они отображаются как последняя строка в иерархии. Вы также можете просмотреть, сколько рабочих нагрузок, не связанных с стручками, работает на хосте, если у хоста есть процессор или давление памяти.

![Пример иерархии узла Kubernetes в представлении производительности](./media/container-insights-analyze/containers-nodes-view.png)

Контейнеры Windows Server, запускаемые ОС Windows Server 2019, отображаются после того, как все узлы на базе Linux в списке. При расширении узла Windows Server можно просмотреть один или несколько стручков и контейнеров, которые работают на уде. После выбора узла панель свойств показывает информацию о версии. Информация об агенте исключается из-за того, что узлы Windows Server не устанавливаются агентом. 

![Пример иерархии узлов с перечисленными узлами Windows Server](./media/container-insights-analyze/nodes-view-windows.png) 

Виртуальные узлы Azure Container Instances, запускаемые ОС Linux, отображаются после последнего кластерного узла AKS в списке. При расширении виртуального узла Контейнерных инстанций можно просмотреть один или несколько стручков контейнеров и контейнеров, которые работают на уде. Метрика не собирается и не регистрируется для узлов, только для стручков.

![Пример иерархии узлов с перечисленными Экземплярами контейнера](./media/container-insights-analyze/nodes-view-aci.png)

От расширенного узла можно просверлить от стручка или контейнера, который работает на узеле, до контроллера для просмотра данных о производительности, отфильтрованных для этого контроллера. Выберите значение под столбцей **контроллера** для конкретного узла.
 
![Пример перехода из узла к контроллеру в представление производительности](./media/container-insights-analyze/drill-down-node-controller.png)

Выберите контроллеры или контейнеры в верхней части страницы для просмотра состояния и использования ресурсов для этих объектов. Для просмотра использования памяти в списке **метрических** выпадающих данных выберите рабочий набор **Memory RSS** или **Memory.** Параметр **Memory RSS** (Память (RSS)) поддерживается только для Kubernetes версии 1.8 и более поздних. В противном случае значения для **Min&nbsp; ** — *как NaN,&nbsp;* которое является числовым значением типа данных, которое представляет собой неопределенное или непредвиденное значение.

![Представление производительности узлов контейнера](./media/container-insights-analyze/containers-node-metric-dropdown.png)

**Рабочий набор памяти** показывает как резидентную память, так и виртуальную память (кэш), включенную, и представляет собой общее число приложений. **Память RSS** показывает только основную память (которая является не чем иным, резидентпамяти памяти другими словами). Эта метрика показывает фактическую емкость доступной памяти. В чем разница между резидентной памятью и виртуальной памятью?

- Память резидента или основная память— это фактическое количество памяти машины, доступной для узлов кластера.

- Виртуальная память зарезервирована пространство жесткого диска (кэш), используемое операционной системой для обмена данными из памяти на диск, когда под давлением памяти, а затем получить его обратно в память, когда это необходимо.

По умолчанию данные о производительности основаны на последних шести часах, но можно изменить окно, используя опцию **TimeRange** в левом верхнем углу. Вы также можете отфильтровать результаты в пределах временной дистанции, выбрав **Мин,** **Авг,** **50-й,** **90-й,** **95-й**и **Макс** в селекторе процентиля.

После того, как вы выберете диаграмму тренда через клавиатуру, используйте клавишу Alt-Page up key или Alt-Page down key to cycle through each bar individually. Вы получаете те же детали, что вы бы, если вы зависли над баром.

## <a name="workbooks"></a>Workbooks

В богатых интерактивных отчетах рабочие книги объединяют текст, [запросы журналов,](../log-query/query-language.md) [метрики](../platform/data-platform-metrics.md)и параметры. Их могут редактировать другие участники команды с доступом к тому же ресурсу Azure.

Azure Monitor для контейнеров включает в себя четыре трудовых книжек для начала работы:

- **Емкость диска**: Представляет интерактивные диаграммы использования диска для каждого диска, представленного узлам в контейнере следующими перспективами:

    - Использование дисков процентов для всех дисков.
    - Свободное дисковое пространство для всех дисков.
    - Сетка, отображая диск каждого узла, его процент используемого пространства, тренд процента используемого пространства, свободное дисковое пространство (GiB) и тренд свободного дискового пространства (GiB). При выборе строки в таблице под строкой отображается процент используемого пространства и свободного дискового пространства (GiB). 

- **Disk IO**: Представляет интерактивные диаграммы использования диска для каждого диска, представленного узлам в контейнере следующими перспективами:

    - Диск ВВ/O обобщен на всех дисках, читая байты/сек, пишет байты/сек, а также читает и пишет байты/сек тенденции.
    - Восемь диаграмм производительности показывают ключевые показатели производительности, помогая измерять и выявлять узкие места ввода/с диска.

- **Кубелет**: Включает в себя две сетки, которые показывают статистику работы ключевых узлов:

    - Обзор сетки узлов суммирует общую работу, общие ошибки и успешные операции по процентам и тренд для каждого узла.
    - Обзор по типу операции суммирует для каждой операции общую операцию, общие ошибки и успешные операции по процентам и тенденциям.

- **Сеть**: Представляет интерактивные диаграммы использования сети для сетевого адаптера каждого узла, а сетка представляет ключевые показатели производительности, чтобы помочь измерить производительность адаптеров сети.

Вы получаете доступ к этим трудовым книжки, выбирая каждый из списка выпадающих книг **View Workbooks.**

![Просмотр списка выпадающих книг по учебникам](./media/container-insights-analyze/view-workbooks-dropdown-list.png)
