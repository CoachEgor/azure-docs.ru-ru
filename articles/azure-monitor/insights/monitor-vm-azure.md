# <a name="monitoring-azure-virtual-machines-with-azure-monitor"></a>Мониторинг виртуальных машин Azure с помощью Azure Monitor
В этой статье описывается, как использовать Azure Monitor для сбора и анализа данных мониторинга с виртуальных машин Azure для поддержания их работоспособности. Виртуальные машины можно контролировать на наличие и производительность с помощью Azure Monitor, как и любой [другой ресурс Azure,](monitor-azure-resource.md)но они уникальны из других ресурсов, так как вам также необходимо контролировать работу и систему гостей, а также рабочие нагрузки, которые работают в нем. 

> [!NOTE]
> В этой статье представлен полный обзор концепций и опций мониторинга виртуальных машин в Azure Monitor. Чтобы быстро начать мониторинг виртуальных машин, не фокусируясь на базовых концепциях, [см.](../learn/quick-monitor-azure-vm.md)


## <a name="differences-from-other-azure-resources"></a>Отличия от других ресурсов Azure
[Мониторинг ресурсов Azure с помощью Azure Monitor](monitor-azure-resource.md) описывает данные мониторинга, генерируемые ресурсами Azure, и то, как можно использовать функции Azure Monitor для анализа и оповещения об этих данных. Вы можете собирать и действовать на тех же данных мониторинга с виртуальных машин Azure со следующими различиями:

- [Метрики платформы](../platform/data-platform-metrics.md) собираются автоматически для виртуальных машин, но только для [виртуального множесца.](#monitoring-data) Вам нужен агент для сбора данных о производительности из гостевой операционной системы. 
- Виртуальные машины не генерируют [журналы ресурсов,](../platform/platform-logs-overview.md) которые дают представление об операциях, выполняемых в ресурсе Azure. Для сбора данных журнала из гостевой операционной системы используется агент.
- Можно создать [диагностические настройки](../platform/diagnostic-settings.md) для виртуальной машины для отправки метрик платформы в другие пункты назначения, такие как концентраторы хранения и события, но вы не можете настроить эти диагностические настройки на портале Azure. 

## <a name="monitoring-data"></a>Данные мониторинга
Виртуальные машины в Azure в Azure генерируют [журналы](../platform/data-platform-logs.md) и [метрики, показанные](../platform/data-platform-metrics.md) на следующей диаграмме.

![Обзор](media/monitor-vm-azure/logs-metrics.png)


### <a name="virtual-machine-host"></a>Виртуальный хост машины
Виртуальные машины в Azure генерируют следующие данные для виртуальной машины, такие же, как и другие ресурсы Azure, описанные в [данных мониторинга.](monitor-azure-resource.md#monitoring-data)

- [Метрики платформы](../platform/data-platform-metrics.md) - численные значения, которые автоматически собираются через регулярные промежутки времени и описывают некоторые аспекты ресурса в определенное время. Метрики платформы собираются для виртуального моста, но для сбора метрик для гостевой операционной системы требуется расширение диагностики.
- [Журнал действий](../platform/platform-logs-overview.md) - Обеспечивает понимание операций на каждом ресурсе Azure в подписке извне (плоскость управления). Для виртуальной машины это включает в себя такую информацию, как, когда она была запущена и любые изменения конфигурации.


### <a name="guest-operating-system"></a>Операционная система на виртуальной машине
Для сбора данных из гостевой операционной системы виртуальной машины требуется агент, который локально работает на каждой виртуальной машине и отправляет данные в Azure Monitor. Для Azure Monitor доступно несколько агентов, каждый из которых собирает различные данные и записывает данные в разных местах. Получите подробное сравнение различных агентов в [обзоре агентов Azure Monitor.](../platform/agents-overview.md) 

- [Агент log Analytics](../platform/agents-overview.md#log-analytics-agent) — доступен для виртуальных машин в Azure, других облачных средах и на местах. Сбор данных в журналы мониторинга Azure. Поддерживает Azure Monitor для вс-м и решений мониторинга. Это тот же агент, который используется для менеджера по операциям Системного центра.
- [Агент зависимости](../platform/agents-overview.md#dependency-agent) - собирает данные о процессах, работающих на виртуальной машине, и их зависимостях. Опирается на агент Log Analytics для передачи данных в Azure и поддерживает Azure Monitor для решений Azure Monitor для вс-карт, сервисных карт и проводных данных 2.0.
- [Расширение Azure Diagnostic](../platform/agents-overview.md#azure-diagnostics-extension) - доступно только для виртуальных машин Azure Monitor. Можно собирать данные в нескольких местах, но в основном используется для сбора данных о производительности гостей в Azure Monitor Metrics для виртуальных машин Windows.
- [Агент Телеграфа](../platform/collect-custom-metrics-linux-telegraf.md) - Сбор данных о производительности с Linux VMs в Azure Monitor Metrics.

### <a name="enable-azure-monitor-for-vms"></a>Включение Azure Monitor для виртуальных машин
[Azure Monitor для виртуальных технологий](vminsights-overview.md) — это [информация](insights-overview.md) в Azure Monitor, который является основным инструментом для мониторинга виртуальных машин в Azure Monitor. Он обеспечивает следующее дополнительное значение по сравнению со стандартными функциями Azure Monitor.

- Упрощенная посадка агента Log Analytics и агента зависимости для мониторинга операционной системы и рабочих нагрузок виртуального гостя машины. 
- Заранее определенные трендовые диаграммы производительности и трудовые книжки, которые позволяют анализировать основные показатели производительности из гостевой операционной системы виртуальной машины.
- Карта зависимости, отображая процессы, работающие на каждой виртуальной машине, и взаимосвязанные компоненты с другими машинами и внешними источниками.

![Azure Monitor для виртуальных машин](media/monitor-vm-azure/vminsights-01.png)

![Azure Monitor для виртуальных машин](media/monitor-vm-azure/vminsights-02.png)


Включите Azure Monitor для виртуальных вывизм из опции **Insights** в меню виртуальной машины портала Azure. Подробнее о других сведениях и других методах конфигурации можно ознакомиться с обзором [Вок-монитора для вьювов.](vminsights-enable-overview.md)

![Включение Azure Monitor для виртуальных машин](media/monitor-vm-azure/enable-vminsights.png)

### <a name="configure-log-analytics-workspace"></a>Настройка рабочего пространства анализа журналов
Агент log Analytics, используемый Azure Monitor для вс-мидов, отправляет данные в [рабочее пространство Log Analytics.](../platform/data-platform-logs.md#how-is-data-in-azure-monitor-logs-structured) Можно включить сбор дополнительных данных о производительности, событий и других данных мониторинга от агента путем настройки рабочего пространства Log Analytics. Он должен быть настроен только один раз, так как любой агент, подключающийся к рабочей области, автоматически загрузит конфигурацию и сразу же начнет собирать определенные данные. 

Вы можете получить доступ к конфигурации рабочего пространства непосредственно из Azure Monitor для VMs, выбрав **конфигурацию Рабочего пространства** из **Get Started.** Нажмите на имя рабочего пространства, чтобы открыть свое меню.

![Конфигурация рабочей области](media/monitor-vm-azure/workspace-configuration.png)

Выберите **Расширенные настройки** из меню рабочей области, а затем **данные** для настройки источников данных. Для агентов Windows выберите **журналы событий Windows** и добавьте общие журналы событий, такие как *System* and *Application.* Для агентов Linux выберите **Syslog** и добавьте общие средства, такие как *керн* и *деймон.* Ознакомиться [с источниками данных агента в Azure Monitor](../platform/agent-data-sources.md) можно найти список доступных источников данных и подробную информацию о настройке. 

![Настройка событий](media/monitor-vm-azure/configure-events.png)


> [!NOTE]
> Можно настроить счетчики производительности для сбора из конфигурации рабочего пространства, но это может быть излишним с счетчиками производительности, собранными Azure Monitor для VMs. Azure Monitor для VMs собирает наиболее распространенный набор счетчиков с частотой один раз в минуту. Только настроить счетчики производительности, которые будут собираться в рабочем пространстве, если вы хотите собрать счетчики, которые еще не собраны Azure Monitor для vMs или если у вас есть существующие запросы с использованием данных производительности.


### <a name="enable-diagnostics-extension-and-telegraf-agent"></a>Включить расширение диагностики и агент Телеграф
Azure Monitor для VMs основан на агенте Log Analytics, который собирает данные в рабочее пространство log Analytics. Это поддерживает [несколько функций Azure Monitor,](../platform/data-platform-logs.md#what-can-you-do-with-azure-monitor-logs) таких как [запросы журналов,](../log-query/log-query-overview.md) [оповещения о журналах](../platform/alerts-log.md)и [трудовые книжки.](../platform/workbooks-overview.md) [Расширение диагностики](../platform/diagnostics-extension-overview.md) собирает данные о производительности от гостевой операционной системы виртуальных компьютеров Windows в Azure Storage и по желанию отправляет данные о производительности в [Azure Monitor Metrics.](../platform/data-platform-metrics.md) Для виртуальных машин Linux [агент Telegraf](../platform/collect-custom-metrics-linux-telegraf.md) обязан отправлять данные в Azure Metrics.  Это позволяет использовать другие функции Azure Monitor, такие как [исследователь метрик](../platform/metrics-getting-started.md) и [оповещения метрик.](../platform/alerts-metric.md) Можно также настроить расширение диагностики для отправки событий и данных о производительности за пределами Azure Monitor с помощью концентраторов Azure Event.

Установите расширение диагностики для одной виртуальной машины Windows на портале Azure из **параметра настройки диагностики** в меню VM. Выберите опцию включения **Azure Monitor** во вкладке **Sinks.** Для включения расширения из шаблона или командной строки для нескольких виртуальных машин [см. Installation и configuration.](../platform/diagnostics-extension-overview.md#installation-and-configuration) В отличие от агента Log Analytics, данные для сбора определяются в конфигурации для расширения на каждой виртуальной машине.

![Параметр диагностики](media/monitor-vm-azure/diagnostic-setting.png)

Подробнее о настройке агентов Telegraf на виртуальных машинах Linux можно найти [установить и настроить Telegraf.](../platform/collect-custom-metrics-linux-telegraf.md#install-and-configure-telegraf) Параметр меню **диагностических параметров** доступен для Linux, но он позволит отправлять данные только в хранилище Azure.

### <a name="collect-platform-metrics-and-activity-log"></a>Сбор метрик платформы и журнал активности
Вы можете просмотреть метрики платформы и журнал активности, собранные для каждого виртуального моста на портале Azure. Соберите эти данные в то же рабочее пространство log Analytics, что и Azure Monitor для виртуальных компьютеров, чтобы проанализировать их с помощью других данных мониторинга, собранных для виртуальной машины. Эта коллекция настроена с [диагностической настройкой.](../platform/diagnostic-settings.md) Соберите журнал активности с [диагностической настройкой для подписки.](../platform/diagnostic-settings.md#create-diagnostic-settings-in-azure-portal)

Соберите метрики платформы с диагностической настройкой для виртуальной машины. В отличие от других ресурсов Azure, вы не можете создать диагностическую настройку для виртуальной машины на портале Azure, но необходимо использовать [другой метод.](../platform/diagnostic-settings.md#create-diagnostic-settings-using-powershell) Следующие примеры показывают, как собирать метрики для виртуальной машины с помощью PowerShell и CLI.

```powershell
Set-AzDiagnosticSetting -Name vm-diagnostics -ResourceId "/subscriptions/monitor diagnostic-settings create \
xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx/resourceGroups/my-resource-group/providers/Microsoft.Compute/virtualMachines/my-vm" -Enabled $true -MetricCategory AllMetrics -workspaceId "/subscriptions/monitor diagnostic-settings create \
xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx/resourcegroups/my-resource-group/providers/microsoft.operationalinsights/workspaces/my-workspace"
```

```CLI
az monitor diagnostic-settings create \
--name VM-Diagnostics 
--resource /subscriptions/xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx/resourceGroups/my-resource-group/providers/Microsoft.Compute/virtualMachines/my-vm \
--metrics '[{"category": "AllMetrics","enabled": true}]' \
--workspace /subscriptions/xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx/resourcegroups/my-resource-group/providers/microsoft.operationalinsights/workspaces/my-workspace

```

- Используйте Azure Monitor для виртуальных технологий для мониторинга наборов виртуальных машин в масштабе.
- Проанализируйте данные для одной виртуальной машины из ее меню на портале Azure. В таблице ниже перечислены различные варианты мониторинга меню виртуальных машин.

| Журнал действий | Записи [журналов активности,](../platform/activity-log-view.md) отфильтрованные для текущей виртуальной машины. |

Подробную информацию об использовании этого инструмента можно узнать из-за использования этого инструмента с [помощью Azure Metrics Explorer.](../platform/metrics-getting-started.md)

| Виртуальный гость машины | Показатели гостевой операционной системы, собранные из виртуальных машин с расширением диагностики, установлены и настроены для отправки в раковину Azure Monitor. |

## <a name="analyzing-log-data"></a>Анализ данных журнала
Виртуальные машины Azure будут собирать следующие данные в журналы Azure Monitor. 

Azure Monitor для VMs позволяет собирать заранее определенный набор счетчиков производительности, которые записываются в таблицу *InsightsMetrics.* Это та же таблица, которая используется [Azure Monitor для контейнеров.](container-insights-overview.md)

## <a name="alerts"></a>видны узлы
[Оповещения](../platform/alerts-overview.md) в Azure Monitor упреждающе уведомляют вас о том, что в данных мониторинга находятся важные условия, и потенциально запускать такие действия, как запуск Logic App или вызов веб-крючка. Правила оповещения определяют логику, используемую для определения того, когда должно быть создано оповещение. Azure Monitor собирает данные, используемые правилами оповещения, но необходимо создать правила для определения условий оповещения в подписке Azure.

В следующих разделах описаны типы правил оповещения и рекомендации о том, когда вы должны использовать каждый из них. Эта рекомендация основана на функциональности и стоимости типа правил оповещения. Для получения подробной информации [Azure Monitor pricing](https://azure.microsoft.com/pricing/details/monitor/)о ценах на оповещения см.


### <a name="activity-log-alert-rules"></a>Правила генерации оповещений журнала действий
[Оповещение журнала активности](../platform/alerts-activity-log.md) сравняется, когда запись, соответствующая определенным критериям, создается в журнале действий. Они не имеют никакой стоимости, поэтому они должны быть вашим первым выбором, если логика, которая требуется, находится в журнале действий. 

Целевой ресурс для оповещений о журнале активности может быть конкретной виртуальной машиной, всеми виртуальными машинами в группе ресурсов или всеми виртуальными машинами в подписке.

Например, создайте оповещение, если критическая виртуальная машина остановлена, выбрав *виртуальную машину Power Off* для имени сигнала.

![Оповещение журнала действий](media/monitor-vm-azure/activity-log-alert.png)


### <a name="metric-alert-rules"></a>Правила генерации оповещений метрик
[Правила метеосигнала](../platform/alerts-metric.md) сравниваются, когда значение метрики превышает пороговое значение. Можно определить определенное значение порога или позволить Azure Monitor динамически определить порог на основе исторических данных.  Используйте метрические оповещения, когда это возможно, с метрическими данными, так как они стоят меньше и более отзывчивы, чем правила оповещения журнала. Они также stateful смысл они будут решать сами, когда метрика падает ниже порога.

Целевой ресурс для оповещений о журнале активности может быть конкретной виртуальной машиной или всеми виртуальными машинами в группе ресурсов.

Например, чтобы создать оповещение, когда процессор виртуальной машины превышает определенное значение, создайте правило метрикаэвых оповещений, используя в качестве типа сигнала *процессор.* Установите либо определенное значение порога, либо позвольте Azure Monitor установить динамический порог. 

![Метическое оповещение](media/monitor-vm-azure/metric-alert.png)

### <a name="log-alerts"></a>Оповещения журналов
[Правила оповещения о журнале](../platform/alerts-log.md) работают, когда результаты запланированного запроса журнала соответствуют определенным критериям. Оповещения о запросах журналов являются наиболее дорогими и наименее отзывчивыми из правил оповещения, но они имеют доступ к самым разнообразным данным и могут выполнять сложную логику, которая не может быть выполнена другими правилами оповещения. 

Целевой ресурс для запроса журнала — рабочее пространство анализа журналов. Фильтр для определенных компьютеров в запросе.

Например, чтобы создать оповещение, проверяя, находятся ли виртуальные машины в определенной группе ресурсов в автономном режиме, используйте следующий запрос, который возвращает запись для каждого компьютера, пропустившего сердцебиение за последние десять минут. Используйте порог 1, который сравнивает, если хотя бы один компьютер имеет пропущенное сердцебиение.

```kusto
Heartbeat
| where TimeGenerated < ago(10m)
| where ResourceGroup == "my-resource-group"
| summarize max(TimeGenerated) by Computer
```

![Предупреждение о журнале](media/monitor-vm-azure/log-alert-01.png)

Чтобы создать оповещение, если на любых виртуальных компьютерах Windows в подписке произошло чрезмерное количество неисправных логонов, используйте следующий запрос, который возвращает запись для каждого неудавшегося события входа в за последний час. Используйте пороговое значение для числа неисправных логонов, которые вы позволите. 

```kusto
Event
| where TimeGenerated < ago(1hr)
| where EventID == 4625
```

![Предупреждение о журнале](media/monitor-vm-azure/log-alert-02.png)


## <a name="system-center-operations-manager"></a>System Center Operations Manager
Диспетчер операций Системного центра (SCOM) обеспечивает детальный мониторинг рабочих нагрузок на виртуальных машинах. Ознакомиться с платформами мониторинга и различными стратегиями реализации можно в [Руководстве по мониторингу обучки.](https://docs.microsoft.com/azure/cloud-adoption-framework/manage/monitor/)

Если у вас есть существующая среда SCOM, которую вы собираетесь продолжать использовать, вы можете интегрировать ее с Azure Monitor, чтобы обеспечить дополнительную функциональность. Агент Log Analytics, используемый Azure Monitor, является тем же самым, что используется для SCOM, так что вы отслеживаете виртуальные машины, отправляющие данные обоим. Все еще нужно добавить агент в Azure Monitor для виртуальных технологий и настроить рабочее пространство для сбора дополнительных данных, как указано выше, но виртуальные машины могут продолжать запускать существующие пакеты управления в среде SCOM без изменений.

Особенности Azure Monitor, которые дополнят существующие функции SCOM, включают в себя следующее:

- Используйте Log Analytics для интерактивного анализа данных журнала и производительности.
- Используйте оповещения о журнале для определения условий оповещения в нескольких виртуальных машинах и использования долгосрочных тенденций, которые невозможны с помощью предупреждений в SCOM.   

Подробнее о подключении существующей группы управления SCOM к рабочей области Log Analytics читайте в материале [«Менеджер по операциям подключения к Azure Monitor».](../platform/om-agents.md)
