# <a name="monitoring-azure-virtual-machines-with-azure-monitor"></a>Наблюдение за виртуальными машинами Azure с помощью Azure Monitor
В этой статье описывается, как использовать Azure Monitor для получения и анализа данных мониторинга из виртуальных машин Azure для поддержания работоспособности. Виртуальные машины можно отслеживать для обеспечения доступности и производительности с Azure Monitor, как и любой [другой ресурс Azure](monitor-azure-resource.md), но они уникальны из других ресурсов, так как вам также потребуется отслеживать операционную систему и системную рабочую нагрузку и рабочие нагрузки, которые выполняются в ней. 

> [!NOTE]
> В этой статье содержится полный обзор концепций и параметров мониторинга виртуальных машин в Azure Monitor. Чтобы быстро начать мониторинг виртуальных машин, не заботясь об основных понятиях, см. раздел [Краткое руководство. мониторинг виртуальной машины Azure с помощью Azure Monitor](../learn/quick-monitor-azure-vm.md).


## <a name="differences-from-other-azure-resources"></a>Отличия других ресурсов Azure
[Мониторинг ресурсов Azure с помощью Azure Monitor](monitor-azure-resource.md) описывает данные мониторинга, созданные ресурсами Azure, и способы использования функций Azure Monitor для анализа и оповещения об этих данных. Вы можете получать данные мониторинга и работать с ними на виртуальных машинах Azure со следующими отличиями.

- [Метрики платформы](../platform/data-platform-metrics.md) собираются автоматически для виртуальных машин, но только для [узла виртуальной машины](#monitoring-data). Для получения данных о производительности из операционной системы на виртуальной машине необходим агент. 
- Виртуальные машины не создают [журналы ресурсов](../platform/platform-logs-overview.md) , которые позволяют получить представление об операциях, которые выполнялись в ресурсе Azure. Агент используется для получения данных журнала из операционной системы на виртуальной машине.
- Вы можете создать [параметры диагностики](../platform/diagnostic-settings.md) для виртуальной машины, чтобы передавать метрики платформы другим местам назначения, таким как хранилище и концентраторы событий, но эти параметры диагностики нельзя настроить в портал Azure. 

## <a name="monitoring-data"></a>Данные мониторинга
Виртуальные машины в Azure в Azure создают [журналы](../platform/data-platform-logs.md) и [метрики](../platform/data-platform-metrics.md) , показанные на следующей схеме.

![Обзор](media/monitor-vm-azure/logs-metrics.png)


### <a name="virtual-machine-host"></a>Узел виртуальной машины
Виртуальные машины в Azure создают следующие данные для узла виртуальной машины, такие же, как и другие ресурсы Azure, как описано в разделе [мониторинг данных](monitor-azure-resource.md#monitoring-data).

- [Метрики платформы](../platform/data-platform-metrics.md) — числовые значения, которые автоматически собираются через регулярные интервалы и описывают некоторые аспекты ресурса в определенное время. Метрики платформы собираются для узла виртуальной машины, но для сбора метрик для гостевой операционной системы требуется расширение диагностики.
- [Журнал действий](../platform/platform-logs-overview.md) . предоставляет подробные сведения об операциях с каждым ресурсом Azure в подписке извне (плоскость управления). Для виртуальной машины это включает такие сведения, как при запуске и изменении конфигурации.


### <a name="guest-operating-system"></a>Операционная система на виртуальной машине
Чтобы получить данные из гостевой операционной системы виртуальной машины, требуется агент, который выполняется локально на каждой виртуальной машине и отправляет данные в Azure Monitor. Для Azure Monitor доступны несколько агентов, каждый из которых собирает различные данные и записывает данные в разные расположения. Получите подробное сравнение различных агентов в [обзоре агентов Azure Monitor](../platform/agents-overview.md). 

- [Агент log Analytics](../platform/agents-overview.md#log-analytics-agent) — доступен для виртуальных машин в Azure, в других облачных средах и в локальной среде. Собирает данные в журналы Azure Monitor. Поддерживает Azure Monitor для виртуальных машин и мониторинг решений. Это тот же агент, который используется для System Center Operations Manager.
- [Агент зависимостей](../platform/agents-overview.md#dependency-agent) — собирает данные о процессах, выполняемых на виртуальной машине, и их зависимостях. Использует агент Log Analytics для передачи данных в Azure и поддерживает Azure Monitor для виртуальных машин, Сопоставление служб и Wire Data 2.0 решения.
- [Расширение диагностики Azure](../platform/agents-overview.md#azure-diagnostics-extension) — доступно только для Azure Monitor виртуальных машин. Может получать данные в несколько расположений, но в основном используется для сбора данных о производительности гостя в метрики Azure Monitor для виртуальных машин Windows.
- [Агент Telegraf](../platform/collect-custom-metrics-linux-telegraf.md) . Собирайте данные производительности с виртуальных машин Linux на метрики Azure Monitor.

### <a name="enable-azure-monitor-for-vms"></a>Включение Azure Monitor для виртуальных машин
[Azure Monitor для виртуальных машин](vminsights-overview.md) представляет собой [аналитическое представление](insights-overview.md) о Azure Monitor, которое является основным средством мониторинга виртуальных машин в Azure Monitor. Он предоставляет следующее дополнительное значение для стандартных функций Azure Monitor.

- Упрощенное подключение агента Log Analytics и агента зависимостей для обеспечения мониторинга гостевой операционной системы и рабочих нагрузок виртуальной машины. 
- Предварительно определенные диаграммы и книги производительности тенденций, позволяющие анализировать основные метрики производительности из гостевой операционной системы виртуальной машины.
- Схема зависимостей, которая отображает процессы, выполняемые на каждой виртуальной машине, и взаимосвязанных компонентах с другими компьютерами и внешними источниками.

![Azure Monitor для виртуальных машин](media/monitor-vm-azure/vminsights-01.png)

![Azure Monitor для виртуальных машин](media/monitor-vm-azure/vminsights-02.png)


Включите Azure Monitor для виртуальных машин из параметра **Insights** в меню виртуальной машины портал Azure. Дополнительные сведения и другие методы настройки см. в разделе [включение Azure Monitor для виртуальных машин обзор](vminsights-enable-overview.md) .

![Включение Azure Monitor для виртуальных машин](media/monitor-vm-azure/enable-vminsights.png)

### <a name="configure-log-analytics-workspace"></a>Настройка рабочей области Log Analytics
Агент Log Analytics, используемый Azure Monitor для виртуальных машин, отправляет данные в [рабочую область log Analytics](../platform/data-platform-logs.md#how-is-data-in-azure-monitor-logs-structured). Вы можете включить сбор дополнительных данных о производительности, событий и других данных мониторинга из агента, настроив рабочую область Log Analytics. Его необходимо настроить только один раз, так как любой агент, подключающийся к рабочей области, автоматически загрузит конфигурацию и немедленно начнет сбор определенных данных. 

Вы можете получить доступ к конфигурации рабочей области непосредственно из Azure Monitor для виртуальных машин, выбрав пункт **Конфигурация рабочей области** в меню **Приступая к работе**. Щелкните имя рабочей области, чтобы открыть его меню.

![Конфигурация рабочей области](media/monitor-vm-azure/workspace-configuration.png)

Выберите **Дополнительные параметры** в меню Рабочая область, а затем — **данные** для настройки источников данных. Для агентов Windows выберите **журналы событий Windows** и добавьте общие журналы событий, такие как *система* и *приложение*. Для агентов Linux выберите **syslog** и добавьте общие средства, такие как *Кернинг* и *управляющая*программа. Список доступных источников данных и сведения о его настройке см. [в разделе источники данных агента в Azure Monitor](../platform/agent-data-sources.md) . 

![Настройка событий](media/monitor-vm-azure/configure-events.png)


> [!NOTE]
> Можно настроить сбор счетчиков производительности из конфигурации рабочей области, но это может быть избыточным счетчиками производительности, собранными Azure Monitor для виртуальных машин. Azure Monitor для виртуальных машин собирает наиболее распространенный набор счетчиков с частотой один раз в минуту. Настраивайте счетчики производительности, которые будут собраны рабочей областью, только если вы хотите собирать счетчики, которые еще не были собраны Azure Monitor для виртуальных машин или если у вас есть запросы, использующие данные о производительности.


### <a name="enable-diagnostics-extension-and-telegraf-agent"></a>Включение расширения диагностики и агента Telegraf
Azure Monitor для виртуальных машин основан на агенте Log Analytics, который собирает данные в рабочую область Log Analytics. Это поддерживает [несколько функций Azure Monitor](../platform/data-platform-logs.md#what-can-you-do-with-azure-monitor-logs) , таких как [запросы журналов](../log-query/log-query-overview.md), [оповещения журналов](../platform/alerts-log.md)и [книги](../platform/workbooks-overview.md). [Расширение диагностики](../platform/diagnostics-extension-overview.md) собирает данные о производительности из операционной системы виртуальной машины Windows в службу хранилища Azure и при необходимости отправляет данные о производительности в [метрики Azure Monitor](../platform/data-platform-metrics.md). Для виртуальных машин Linux [Агент Telegraf](../platform/collect-custom-metrics-linux-telegraf.md) требуется для отправки данных в метрики Azure.  Это позволяет выполнять другие функции Azure Monitor, такие как [Обозреватель метрик](../platform/metrics-getting-started.md) и [оповещения метрик](../platform/alerts-metric.md). Можно также настроить расширение системы диагностики для отправки событий и данных о производительности вне Azure Monitor с помощью концентраторов событий Azure.

Установите расширение системы диагностики для отдельной виртуальной машины Windows в портал Azure из параметра **Диагностика** в меню Виртуальная машина. Выберите параметр, чтобы включить **Azure Monitor** на вкладке **приемники** . Сведения о включении расширения из шаблона или командной строки для нескольких виртуальных машин см. в разделе [Установка и настройка](../platform/diagnostics-extension-overview.md#installation-and-configuration). В отличие от агента Log Analytics, данные для собраний определяются в конфигурации для расширения на каждой виртуальной машине.

![Параметр диагностики](media/monitor-vm-azure/diagnostic-setting.png)

Дополнительные сведения о настройке агентов Telegraf на виртуальных машинах Linux см. в [статье Установка и настройка Telegraf](../platform/collect-custom-metrics-linux-telegraf.md#install-and-configure-telegraf) . Пункт меню **параметров диагностики** доступен для Linux, но он позволяет только отсылать данные в службу хранилища Azure.

### <a name="collect-platform-metrics-and-activity-log"></a>Получение метрик платформы и журнала действий
Метрики платформы и журнал действий, собранные для каждого узла виртуальной машины, можно просмотреть в портал Azure. Собирайте эти данные в той же рабочей области Log Analytics, которая Azure Monitor для виртуальных машин для анализа с другими данными мониторинга, собранными для виртуальной машины. Для этой коллекции настроен [параметр диагностики](../platform/diagnostic-settings.md). Сбор сведений о журнале действий с [параметром диагностики для подписки](../platform/diagnostic-settings.md#create-diagnostic-settings-in-azure-portal).

Сбор метрик платформы с помощью параметра диагностики для виртуальной машины. В отличие от других ресурсов Azure вы не можете создать параметр диагностики для виртуальной машины в портал Azure, но должен использовать [другой метод](../platform/diagnostic-settings.md#create-diagnostic-settings-using-powershell). В следующих примерах показано, как получить метрики для виртуальной машины с помощью PowerShell и CLI.

```powershell
Set-AzDiagnosticSetting -Name vm-diagnostics -ResourceId "/subscriptions/monitor diagnostic-settings create \
xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx/resourceGroups/my-resource-group/providers/Microsoft.Compute/virtualMachines/my-vm" -Enabled $true -MetricCategory AllMetrics -workspaceId "/subscriptions/monitor diagnostic-settings create \
xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx/resourcegroups/my-resource-group/providers/microsoft.operationalinsights/workspaces/my-workspace"
```

```CLI
az monitor diagnostic-settings create \
--name VM-Diagnostics 
--resource /subscriptions/xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx/resourceGroups/my-resource-group/providers/Microsoft.Compute/virtualMachines/my-vm \
--metrics '[{"category": "AllMetrics","enabled": true}]' \
--workspace /subscriptions/xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx/resourcegroups/my-resource-group/providers/microsoft.operationalinsights/workspaces/my-workspace

```

## <a name="monitoring-in-the-azure-portal"></a>Мониторинг на портале Azure

- Используйте меню **Azure Monitor** для доступа к данным из всех отслеживаемых ресурсов. 
- Используйте Azure Monitor для виртуальных машин для мониторинга наборов виртуальных машин в масштабе.
- Анализ данных для одной виртуальной машины из его меню в портал Azure. В следующей таблице перечислены различные параметры для наблюдения за меню "виртуальные машины".

| Журнал действий | Записи [журнала действий](../platform/activity-log-view.md) отфильтрованы для текущей виртуальной машины. |
| Подробные сведения | Открывает [Azure Monitor для виртуальных машин](../insights/vminsights-overview.md) с картой для выбранной текущей виртуальной машины. |

## <a name="analyzing-metric-data"></a>Анализ данных метрик
Вы можете анализировать метрики для виртуальных машин с помощью обозревателя метрик, открыв **метрики** из меню виртуальной машины. Дополнительные сведения об использовании этого средства см. в статье [Приступая к работе с Azure обозреватель метрик](../platform/metrics-getting-started.md) . 

Виртуальные машины используют два пространства имен для метрик:

| Пространство имен | Описание |
|:---|:---|
| Узел виртуальной машины | Метрики узла автоматически собираются для всех виртуальных машин Azure. Подробный список метрик в [Microsoft. COMPUTE/virtualMachines](../platform/metrics-supported.md#microsoftcomputevirtualmachines). |
| Гостевая виртуальная машина | Метрики операционной системы на виртуальной машине, собранные с виртуальных машин с расширением системы диагностики, настроенными для отправки в приемник Azure Monitor. |

![Метрики](media/monitor-vm-azure/metrics.png)

Виртуальные машины Azure будут собираются следующие данные в журналы Azure Monitor. 

Azure Monitor для виртуальных машин включает сбор предварительно определенного набора счетчиков производительности, записываемых в таблицу *инсигхтсметрикс* . Это та же самая таблица, которая используется [Azure Monitor для контейнеров](container-insights-overview.md).

> Данные о производительности, собираемые агентом Log Analytics, записываются в таблицу *производительности* , пока Azure Monitor для виртуальных машин собирают их в таблицу *инсигхтсметрикс* . Это те же данные, но таблицы имеют различную структуру. Если у вас есть запросы, основанные на *производительности*, потребуется перезапись, чтобы использовать *инсигхтсметрикс*.


## <a name="alerts"></a>Предупреждения
[Оповещения](../platform/alerts-overview.md) в Azure Monitor заранее уведомляют Вас при обнаружении важных условий в данных мониторинга и потенциально запускают такие действия, как запуск приложения логики или вызов веб-перехватчика. Правила генерации оповещений определяют логику, используемую для определения момента создания предупреждения. Azure Monitor собирает данные, используемые правилами генерации оповещений, но необходимо создать правила для определения условий предупреждений в подписке Azure.

В следующих разделах описываются типы правил генерации оповещений и рекомендации по их использованию. Эта рекомендация основана на функциональности и стоимости типа правила генерации оповещений. Дополнительные сведения о ценах на оповещения см. в разделе [цены на Azure Monitor](https://azure.microsoft.com/pricing/details/monitor/).


### <a name="activity-log-alert-rules"></a>Правила генерации оповещений журнала действий
[Правила генерации оповещений журнала действий](../platform/alerts-activity-log.md) срабатывают, когда в журнале действий создается запись, соответствующая определенным условиям. Они не изменяют затраты, поэтому они должны быть вашим первым выбором, если нужная логика находится в журнале действий. 

Целевым ресурсом для оповещений журнала действий может быть определенная виртуальная машина, все виртуальные машины в группе ресурсов или все виртуальные машины в подписке.

Например, создайте оповещение, если критическая виртуальная машина остановлена, выбрав в качестве имени сигнала *виртуальную машину выключения* .

![Оповещение журнала действий](media/monitor-vm-azure/activity-log-alert.png)


### <a name="metric-alert-rules"></a>Правила генерации оповещений метрик
[Правила генерации оповещений метрик](../platform/alerts-metric.md) срабатывают, если значение метрики превышает пороговое. Можно определить конкретное пороговое значение или разрешить Azure Monitor динамического определения порогового значения на основе исторических данных.  Используйте оповещения метрик, когда это возможно, с данными метрик, так как они снижают затраты и более быстро отвечают, чем правила генерации оповещений журнала. Они также определяют, что они будут разрешать себя, когда метрика падает ниже порогового значения.

Целевым ресурсом для оповещений журнала действий может быть определенная виртуальная машина или все виртуальные машины в группе ресурсов.

Например, чтобы создать оповещение, когда процессор виртуальной машины превышает определенное значение, создайте правило оповещения метрики, используя *процент загрузки ЦП* в качестве типа сигнала. Задайте либо конкретное пороговое значение, либо разрешите Azure Monitor задать динамическое пороговое значения. 

![Оповещение метрики](media/monitor-vm-azure/metric-alert.png)

### <a name="log-alerts"></a>Оповещения журналов
[Правила генерации оповещений журнала](../platform/alerts-log.md) срабатывают, когда результаты запланированного запроса журнала соответствуют определенным критериям. Оповещения о запросах журнала — это наиболее ресурсоемкие и наименее время реагирования правил генерации оповещений, но они имеют доступ к наиболее разнообразным данным и могут выполнять сложную логику, которая не может быть выполнена другими правилами генерации оповещений. 

Целевым ресурсом для запроса журнала является Рабочая область Log Analytics. Фильтр для конкретных компьютеров в запросе.

Например, чтобы создать оповещение, которое проверяет, находятся ли какие-либо виртуальные машины в определенной группе ресурсов вне сети, используйте следующий запрос, возвращающий запись для каждого компьютера, который пропустил пульс за последние десять минут. Используйте пороговое значение 1, которое срабатывает, если по крайней мере один компьютер имеет пропущенный пульс.

```kusto
Heartbeat
| where TimeGenerated < ago(10m)
| where ResourceGroup == "my-resource-group"
| summarize max(TimeGenerated) by Computer
```

![Оповещение журнала](media/monitor-vm-azure/log-alert-01.png)

Чтобы создать оповещение, если на всех виртуальных машинах Windows в подписке произошло слишком большое число неудачных попыток входа, используйте следующий запрос, возвращающий запись для каждого события неудачного входа за последний час. Используйте пороговое значение, равное количеству неудачных попыток входа. 

```kusto
Event
| where TimeGenerated < ago(1hr)
| where EventID == 4625
```

![Оповещение журнала](media/monitor-vm-azure/log-alert-02.png)


## <a name="system-center-operations-manager"></a>System Center Operations Manager
System Center Operations Manager (SCOM) обеспечивает детальный мониторинг рабочих нагрузок на виртуальных машинах. Сравнение платформ мониторинга и различных стратегий реализации см. в разделе [рекомендации по мониторингу облака](https://docs.microsoft.com/azure/cloud-adoption-framework/manage/monitor/) .

Если у вас есть среда SCOM, которую вы собираетесь использовать, ее можно интегрировать с Azure Monitor для предоставления дополнительных функциональных возможностей. Агент Log Analytics, используемый Azure Monitor, аналогичен тому, который используется для SCOM, поэтому отслеживаемые виртуальные машины будут отсылать данные в оба. Вам по-прежнему необходимо добавить агент в Azure Monitor для виртуальных машин и настроить рабочую область для получения дополнительных данных, как указано выше, но виртуальные машины могут продолжать запускать существующие пакеты управления в среде SCOM без изменения.

К функциям Azure Monitor, дополняющим существующие функции SCOM, относятся следующие.

- Используйте Log Analytics для интерактивного анализа данных журнала и производительности.
- Используйте оповещения журнала, чтобы определить условия предупреждений для нескольких виртуальных машин и использовать долгосрочные тенденции, которые невозможно использовать при использовании оповещений в SCOM.   

Дополнительные сведения о подключении существующей группы управления SCOM к рабочей области Log Analytics см. в статье [подключение Operations Manager к Azure Monitor](../platform/om-agents.md) .
