# <a name="azure-storage-explorer-troubleshooting-guide"></a>Руководство по устранению неполадок в обозревателе хранилищ Azure

Microsoft Azure Storage Explorer — это автономное приложение, которое упрощает работу с данными хранения данных Azure Storage на Windows, macOS и Linux. Приложение может подключаться к учетным записям хранения, размещенным в Azure, национальных облаках и Azure Stack.

В этом руководстве кратко излагаются решения проблем, которые обычно встречаются в Storage Explorer.

## <a name="rbac-permissions-issues"></a>Вопросы разрешений RBAC

Управление доступом на основе ролей [RBAC](https://docs.microsoft.com/azure/role-based-access-control/overview) обеспечивает высокое детальное управление доступом ресурсов Azure путем объединения наборов разрешений в _роли._ Вот несколько стратегий, чтобы получить RBAC работает оптимально в хранилище Explorer.

### <a name="how-do-i-access-my-resources-in-storage-explorer"></a>Как получить доступ к ресурсам в Хранилище Explorer?

Если у вас возникли проблемы с доступом к ресурсам хранения через RBAC, возможно, вам не были назначены соответствующие роли. В следующих разделах описаны разрешения, необходимые для доступа к ресурсам хранения данных. Если вы не уверены, что у вас есть соответствующие роли или разрешения, обратитесь к администратору учетной записи Azure.

#### <a name="read-listget-storage-accounts-permissions-issue"></a>"Читать: Список / Получить учетную запись хранения (ы)" разрешения вопрос

У вас должно быть разрешение на список учетных записей хранения. Чтобы получить это разрешение, вам необходимо назначить роль _читателя._

#### <a name="list-storage-account-keys"></a>Вывод списка ключей учетной записи хранения

Исследователь хранения также может использовать ключи учетной записи для проверки подлинности запросов. Вы можете получить доступ к ключам учетной записи с помощью более мощных ролей, таких как роль _вкладчика._

> [!NOTE]
> Ключи доступа предоставляют неограниченные разрешения любому, кто их держит. Поэтому мы не рекомендуем раздавать эти ключи пользователям учетных записей. Если вам необходимо отозвать ключи доступа, вы можете регенерировать их с [портала Azure](https://portal.azure.com/).

#### <a name="data-roles"></a>Роли данных

Вам должна быть назначена по крайней мере одна роль, которая предоставляет доступ к читаемым данным из ресурсов. Например, если вы хотите перечислить или скачать капли, вам понадобится, по крайней мере, роль _читателя данных Хранилища._

### <a name="why-do-i-need-a-management-layer-role-to-see-my-resources-in-storage-explorer"></a>Зачем нужна роль управленческого уровня для просмотра ресурсов в Хранилище Explorer?

Azure Storage имеет два уровня доступа: _управление_ и _данные._ Доступ к учетным записям подписок и хранилищам осуществляется через уровень управления. Контейнеры, капли и другие ресурсы данных доступны через слой данных. Например, если требуется получить список учетных записей хранилища из Azure, вы отправляете запрос в конечную точку управления. Если вам нужен список контейнеров с каплями в учетной записи, вы отправляете запрос в соответствующую конечную точку обслуживания.

Роли RBAC могут содержать разрешения на доступ к управлению или уровень данных. Роль читателя, например, предоставляет доступ только для чтения ресурсам уровень управления.

Строго говоря, роль Reader не дает разрешений на уровень данных и не требует доступа к слою данных.

Storage Explorer упрощает доступ к вашим ресурсам, собирая необходимую информацию для подключения к ресурсам Azure. Например, для отображения контейнеров с каплями, Storage Explorer отправляет запрос на "список контейнеров" в конечную точку службы blob. Чтобы получить эту конечную точку, Storage Explorer ищет список подписок и учетных записей хранения, к которым у вас есть доступ. Чтобы найти подписи и учетные записи хранения данных, Storage Explorer также необходим доступ к уровеньу управления.

Если у вас нет роли, которая предоставляет разрешения на уровень управления, Storage Explorer не может получить информацию, необходимую для подключения к устью данных.

### <a name="what-if-i-cant-get-the-management-layer-permissions-i-need-from-my-administrator"></a>Что делать, если я не могу получить необходимые разрешения на уровень управления от администратора?

В настоящее время у нас нет решения, связанного с RBAC, для решения этой проблемы. В качестве обходного пути вы можете запросить SAS URI [прикрепить ваш ресурс.](https://docs.microsoft.com/azure/vs-azure-tools-storage-manage-with-storage-explorer?tabs=linux#use-a-shared-access-signature-uri)

### <a name="recommended-built-in-rbac-roles"></a>Рекомендуемые встроенные роли RBAC

Существует несколько встроенных ролей RBAC, которые могут предоставить разрешения, необходимые для использования Storage Explorer. Некоторые из этих ролей:
- [Владелец](/azure/role-based-access-control/built-in-roles#owner): Управляйте всем, включая доступ к ресурсам. **Примечание:** эта роль даст вам доступ к ключу.
- Автор : [Управляйте](/azure/role-based-access-control/built-in-roles#contributor)всем, исключая доступ к ресурсам. **Примечание:** эта роль даст вам доступ к ключу.
- [Читатель](/azure/role-based-access-control/built-in-roles#reader): Прочитайте и перечислите ресурсы.
- [Вкладчик учетной записи хранилища](/azure/role-based-access-control/built-in-roles#storage-account-contributor): Полное управление учетными записями хранения. **Примечание:** эта роль даст вам доступ к ключу.
- [Хранение владельца данных Blob](/azure/role-based-access-control/built-in-roles#storage-blob-data-owner): Полный доступ к контейнерам и данным azure Storage.
- [Хранилище данных Blob:](/azure/role-based-access-control/built-in-roles#storage-blob-data-contributor)Прочитайте, напишите и удалите контейнеры и капли хранения Azure.
- [Хранение blob Data Reader](/azure/role-based-access-control/built-in-roles#storage-blob-data-reader): Прочитайте и перечислите контейнеры и капли для хранения azure.

## <a name="error-self-signed-certificate-in-certificate-chain-and-similar-errors"></a>Ошибка: Самоподписанный сертификат в цепочке сертификатов (и аналогичные ошибки)

Ошибки сертификата обычно возникают в одной из следующих ситуаций:

- Приложение подключено через _прозрачный прокси,_ что означает, что сервер (например, сервер вашей компании) перехватывает трафик HTTPS, расшифровывает его, а затем шифрует его с помощью сертификата с автоподписью.
- Вы работаете с приложением, которое вводит самоподписанный сертификат TLS/SSL в получаемые сообщения HTTPS. Примеры приложений, которые вводят сертификаты включают антивирус и программное обеспечение для проверки сетевого трафика.

Когда Storage Explorer видит самоподписанный или ненадежный сертификат, он больше не знает, было ли изменено полученное сообщение HTTPS. Если у вас есть копия сертификата, подписанного самостоятельно, вы можете поручить Storage Explorer доверять ему, выдыхая следующие действия:

1. Получить закодированную копию сертификата Base-64 X.509 (.cer).
2. Перейдите к **правоке** > **SSL Сертификаты** > **Импорт**, а затем использовать файл сборщика, чтобы найти, выбрать и открыть файл .cer.

Эта проблема может также возникнуть при наличии нескольких сертификатов (корневых и промежуточных). Чтобы исправить эту ошибку, необходимо добавить оба сертификата.

Если вы не уверены в том, откуда идет сертификат, выполните следующие действия, чтобы найти его:

1. Установите OpenSSL.
    * [Windows](https://slproweb.com/products/Win32OpenSSL.html): Любой из световых версий должно быть достаточно.
    * Mac и Linux: должны быть включены в вашу операционную систему.
2. Выполнить OpenSSL.
    * Windows: Откройте каталог установки, выберите **/bin/**, а затем дважды щелкните **openssl.exe**.
    * Mac и Linux: Выполнить `openssl` из терминала.
3. Выполните `s_client -showcerts -connect microsoft.com:443`.
4. Найдите самозаверяющие сертификаты. Если вы не уверены, какие сертификаты подписаны самостоятельно, `("s:")` обратите внимание `("i:")` на то, где-нибудь предмет и эмитент одинаковы.
5. Когда вы найдете самостоятельно подписанные сертификаты, для каждого из них, скопировать и вставить все от (и в том числе) `-----BEGIN CERTIFICATE-----` через `-----END CERTIFICATE-----` в новый файл .cer.
6. Открыть исследователь хранения и перейти к **отожобья** > **SSL Сертификаты** > **Импортные сертификаты**. Затем используйте сборщик файлов, чтобы найти, выбрать и открыть созданные файлы .cer.

Если вы не можете найти сертификаты, подписанные самостоятельно, выяснив эти действия, свяжитесь с нами через инструмент обратной связи. Вы также можете открыть Исследователь хранения от `--ignore-certificate-errors` командной строки с помощью флага. При открытии с этим флагом, Storage Explorer игнорирует ошибки сертификата.

## <a name="sign-in-issues"></a>Проблемы при входе

### <a name="blank-sign-in-dialog-box"></a>Пустой диалоговый ящик

Пустые диалоговые ящики чаще всего возникают, когда Службы Федерации активных каталогов (AD FS) побуждают Storage Explorer выполнять перенаправление, которое не поддерживается Electron. Чтобы обойти эту проблему, вы можете попробовать использовать Device Code Flow для входиный в. Для этого выполните следующие действия.

1. На левой вертикальной панели инструментов, **открытые настройки**. В панели настроек перейдите на**регистрацию** **приложения** > в . Включить **использование устройства код потока потока в в.**
2. Откройте окно диалога **Connect** (либо через значок штепсельной вилки на левой вертикальной панели, либо выбрав учетную **запись Add на** панели счетов).
3. Выберите среду, в которой вы хотите войти.
4. Выберите **Войти**.
5. Следуйте инструкциям на следующей панели.

Если вы не можете войти в учетную запись, которая вы хотите использовать, потому что ваш браузер по умолчанию уже подписан на другую учетную запись, сделайте одну из следующих:

- Вручную скопировать ссылку и код в закрытом режиме вашего браузера.
- Вручную скопировать ссылку и код в другой браузер.

### <a name="reauthentication-loop-or-upn-change"></a>Цикл повторной аутентификации или изменение имени участника-пользователя

Если вы находитесь в цикле реавентации или изменили UPN одной из ваших учетных записей, выполните следующие действия:

1. Удалите все учетные записи, а затем закройте исследователь хранения.
2. Удалите папку .IdentityService с компьютера. В Windows папка находится в расположении `C:\users\<username>\AppData\Local`. Для Mac и Linux вы можете найти папку в корневой папке каталога пользователя.
3. Если вы работаете под управлением Mac или Linux, вам также необходимо удалить запись Microsoft.Developer.IdentityService из клавиатуры вашей операционной системы. На Mac, ключевым магазином является *приложение Gnome Keychain.* В Linux приложение обычно называется _Keyring,_ но имя может отличаться в зависимости от дистрибутива.

### <a name="conditional-access"></a>Условный доступ

Из-за ограничения в библиотеке Azure AD, используемой Системой хранения, условный доступ не поддерживается при использовании Storage Explorer на Windows 10, Linux или macOS.

## <a name="mac-keychain-errors"></a>Ошибки цепочки ключей Mac

MacOS Keychain иногда может ввести состояние, вызывая проблемы для библиотеки аутентификации Storage Explorer. Чтобы вывести Keychain из этого состояния, выполните следующие действия:

1. Закройте Обозреватель службы хранилища.
2. Открытый Keychain (нажмите Команда-Spacebar, введите **брелок**и нажмите Enter).
3. Выберите "логин" Keychain.
4. Выберите значок замка, чтобы заблокировать Keychain. (Замок будет отображаться заблокированным, когда процесс будет завершен. Это может занять несколько секунд, в зависимости от того, какие приложения у вас есть открытые).

    ![Значок Padlock](./media/storage-explorer-troubleshooting/unlockingkeychain.png)

5. Откройте обозреватель хранилищ.
6. Вам предлагается сообщение, как "Сервис концентратор хочет получить доступ к Keychain". Введите пароль учетной записи Mac-аккаунта и выберите **Always Allow** (или **разрешить,** если **всегда разрешить** не доступен).
7. Попробуйте выполнить вход.

### <a name="general-sign-in-troubleshooting-steps"></a>Общие действия по устранению неполадок входа

* Если вы находитесь на macOS, и окно входа никогда не появляется над окном диалога **проверки подлинности,** попробуйте [эти шаги.](#mac-keychain-errors)
* Перезапустите исследователь хранения данных.
* Если диалоговое окно проверки подлинности пустое, подождите не менее одной минуты, прежде чем его закрыть.
* Убедитесь, что настройки прокси и сертификата должным образом настроены как для машины, так и для Storage Explorer.
* Если вы работаете под управлением Windows и имеете доступ к Visual Studio 2019 на той же машине и к учетным данным, попробуйте войти в Visual Studio 2019. После успешного вхвнутри visual Studio 2019, вы можете открыть Storage Explorer и увидеть свой аккаунт в панели счетов.

Если ни один из этих методов не работает, [откройте проблему в GitHub](https://github.com/Microsoft/AzureStorageExplorer/issues).

### <a name="missing-subscriptions-and-broken-tenants"></a>Отсутствие подписки и неисправные клиенты

Если вы не можете получить подписку после успешного вхлечения, попробуйте следующие методы устранения неполадок:

* Убедитесь, что учетная запись имеет доступ к необходимым подпискам. Вы можете проверить свой доступ, воспроблагодарявходя на портал для среды Azure, которую вы пытаетесь использовать.
* Убедитесь, что вы зарегистрировались через правильную среду Azure (Azure, Azure China 21Vianet, Azure Germany, правительство США Azure или пользовательская среда).
* Если вы находитесь за прокси-сервером, убедитесь, что вы правильно настроили прокси-сервер Storage Explorer.
* Попробуйте удалить и повторно добавить учетную запись.
* Если есть ссылка "Больше информации", проверьте, какие сообщения об ошибках сообщаются для арендаторов, которые не удается. Если вы не знаете, как реагировать на сообщения об ошибках, не [стесняйтесь открыть проблему в GitHub.](https://github.com/Microsoft/AzureStorageExplorer/issues)

## <a name="cant-remove-an-attached-account-or-storage-resource"></a>Не удалить прилагаемую учетную запись или ресурс хранения

Если вы не можете удалить прилагаемую учетную запись или ресурс хранения через uI, вы можете вручную удалить все прилагаемые ресурсы, удалив следующие папки:

* Windows: `%AppData%/StorageExplorer`
* macOS: `/Users/<your_name>/Library/Application Support/StorageExplorer`
* Linux: `~/.config/StorageExplorer`

> [!NOTE]
> Закройте исследователь хранения перед удалением этих папок.

> [!NOTE]
> Если вы когда-либо импортировали какие-либо Сертификаты SSL, подспонйте содержимое каталога. `certs` Позже вы сможете использовать резервную копию, чтобы повторно импортировать сертификаты SSL.

## <a name="proxy-issues"></a>Проблемы с прокси-сервером

Во-первых, убедитесь, что следующая информация, которую вы ввели правильно:

* URL-адрес прокси-сервера и номер порта;
* Имя пользователя и пароль, если прокси требует их

> [!NOTE]
> Storage Explorer не поддерживает прокси-файлы автоматической конфигурации для настройки настроек прокси.

### <a name="common-solutions"></a>Распространенные решения

Если у вас все еще возникают проблемы, попробуйте следующие методы устранения неполадок:

* Если вы можете подключиться к Интернету без использования прокси-сервера, убедитесь, что Storage Explorer работает без включенных настроек прокси. Если это так, возможно, проблема связана с параметрами прокси-сервера. Работайте с администратором, чтобы определить проблемы.
* Проверить, что другие приложения, которые используют прокси-сервер работать, как ожидалось.
* Убедитесь, что вы можете подключиться к порталу для среды Azure, которую вы пытаетесь использовать.
* Убедитесь, что вы можете получать ответы от конечных точек службы. Введите один из URL-адресов конечной точки в браузере. Если вы можете подключиться, вы должны получить Invalid'sParameterValue или аналогичный ответ XML.
* Если другие пользователи также используют обозреватель хранилищ с вашим прокси-сервером, проверьте, могут ли они подключаться. Если это возможно, возможно, вам придется связаться с вашим администратором прокси-сервера.

### <a name="tools-for-diagnosing-issues"></a>Инструменты для выявления проблем

Если у вас есть сетевые инструменты, такие как Fiddler для Windows, вы можете диагностировать проблемы следующим образом:

* Если вам необходимо подключаться через прокси-сервер, может потребоваться настроить сетевые средства для подключения через прокси-сервер.
* Проверьте номер порта, используемый средством для работы с сетью.
* Введите URL-адрес локального узла и номер порта сетевого средства как параметры прокси-сервера в обозревателе хранилищ. При правильном мы делаем это, сетевой инструмент начинает регистрировать сетевые запросы, сделанные Storage Explorer для управления и конечных точек обслуживания. Например, `https://cawablobgrs.blob.core.windows.net/` введите конечную точку blob в браузере, и вы получите ответ, который напоминает следующее:

  ![Пример кода](./media/storage-explorer-troubleshooting/4022502_en_2.png)

  Этот ответ позволяет предположить, что ресурс существует, даже если вы не можете получить к нему доступ.

### <a name="contact-proxy-server-admin"></a>Обратитесь к администратору прокси-сервера

Если настройки прокси-сервера верны, возможно, вам придется связаться с администратором прокси-сервера, чтобы:

* Убедитесь, что ваш прокси-сервер не блокирует трафик в управлении Azure или конечных точках ресурса.
* Проверьте протокол аутентификации, используемый прокси-сервером. В настоящее время Storage Explorer не поддерживает прокси-подобие NTLM.

## <a name="unable-to-retrieve-children-error-message"></a>Сообщение об ошибке "Не удается получить дочерние элементы"

Если вы подключены к Azure через прокси-сервер, убедитесь, что настройки прокси верны. Если вы получили доступ к ресурсу от владельца подписки или учетной записи, убедитесь, что вы прочитали или перечислите разрешения на этот ресурс.

## <a name="connection-string-doesnt-have-complete-configuration-settings"></a>Строка подключения не имеет полных настроек конфигурации

Если вы получили это сообщение об ошибке, возможно, у вас нет необходимых разрешений на получение ключей для вашей учетной записи хранения. Чтобы подтвердить, что это так, перейдите на портал и найти свой счет хранения. Вы можете сделать это, нажав на узла справа и выбрав **Open in Portal.** Затем перейдите к лезвию **ключей доступа.** Если у вас нет разрешений на просмотр ключей, вы увидите сообщение "У вас нет доступа". Чтобы обойти эту проблему, вы можете получить ключ от учетной записи от кого-то другого и прикрепить через имя и ключ, или вы можете попросить кого-то для SAS на счет хранения и использовать его для прикрепления учетной записи хранения.

Если вы видите ключи учетной записи, подайте жалобу в GitHub, чтобы мы могли помочь вам решить проблему.

## <a name="error-occurred-while-adding-new-connection-typeerror-cannot-read-property-version-of-undefined"></a>Ошибка произошла при добавлении нового соединения: TypeError: Не удается прочитать свойство 'версия' неопределенных

Если вы получаете это сообщение об ошибке при попытке добавить пользовательское соединение, данные соединения, хранящиеся в локальном диспетчере учетных данных, могут быть повреждены. Чтобы обойти эту проблему, попробуйте удалить поврежденные локальные соединения, а затем повторно добавьте их:

1. Запустите Обозреватель хранилища. Из меню, перейдите, чтобы **помочь** > **переключать разработчик инструменты**.
2. В открытом окне, на вкладке **Приложения,** перейдите на **локальное хранилище** (слева) > **file://**.
3. В зависимости от типа соединения, с помощью чего возникла проблема, ищите его ключ, а затем копируйте его значение в текстовый редактор. Значение представляет собой массив пользовательских имен соединения, как и следующее:
    * Учетные записи хранения
        * `StorageExplorer_CustomConnections_Accounts_v1`
    * Контейнеры больших двоичных объектов
        * `StorageExplorer_CustomConnections_Blobs_v1`
        * `StorageExplorer_CustomConnections_Blobs_v2`
    * Общие папки
        * `StorageExplorer_CustomConnections_Files_v1`
    * Очереди
        * `StorageExplorer_CustomConnections_Queues_v1`
    * Таблицы
        * `StorageExplorer_CustomConnections_Tables_v1`
4. После сохранения текущих имен соединений установите значение в Developer Tools. `[]`

Если вы хотите сохранить соединения, которые не повреждены, вы можете использовать следующие шаги, чтобы найти поврежденные соединения. Если вы не возражаете потерять все существующие соединения, вы можете пропустить эти шаги и следовать инструкциям, предъследующим конкретные платформы, чтобы очистить данные о подключении.

1. Из текстового редактора повторно добавьте каждое имя соединения в Developer Tools, а затем проверьте, работает ли соединение.
2. Если соединение работает правильно, оно не повреждено, и вы можете безопасно оставить его там. Если соединение не работает, удалите его значение из инструментов разработчика и запишите его так, чтобы можно было добавить его позже.
3. Повторяйте до тех пор, пока вы не изучите все ваши соединения.

Пройдя через все ваши соединения, для всех имен соединений, которые не добавлены назад, вы должны очистить их поврежденные данные (если таковые имеются) и добавить их обратно с помощью стандартных шагов в Storage Explorer:

# <a name="windows"></a>[Windows](#tab/Windows)

1. В меню **«Пуск»** ищите **менеджера по учетным данным** и откройте его.
2. Перейдите на **учетные данные Windows**.
3. В соответствии с **общими полномочиями,** ищите записи, которые имеют `<connection_type_key>/<corrupted_connection_name>` ключ (например, `StorageExplorer_CustomConnections_Accounts_v1/account1`).
4. Удалите эти записи и повторно добавьте соединения.

# <a name="macos"></a>[macOS](#tab/macOS)

1. Открытый spotlight (Command-Spacebar) и поиск **доступа Keychain**.
2. Ищите записи, `<connection_type_key>/<corrupted_connection_name>` которые имеют `StorageExplorer_CustomConnections_Accounts_v1/account1`ключ (например, ).
3. Удалите эти записи и повторно добавьте соединения.

# <a name="linux"></a>[Linux](#tab/Linux)

Локальное управление учетными данными зависит от дистрибутива Linux. Если дистрибутив Linux не предоставляет встроенный инструмент ГРАФИЧЕСКОго интерфейса для управления учетными данными, можно установить сторонний инструмент для управления локальными учетными данными. Например, можно использовать [Seahorse](https://wiki.gnome.org/Apps/Seahorse/), инструмент gui с открытым исходным кодом для управления локальными учетными данными Linux.

1. Откройте свой локальный инструмент управления учетными данными и найдите сохраненные учетные данные.
2. Ищите записи, `<connection_type_key>/<corrupted_connection_name>` которые имеют `StorageExplorer_CustomConnections_Accounts_v1/account1`ключ (например, ).

## <a name="issues-with-sas-url"></a>Проблемы с URL-адресом SAS

Если вы подключаетесь к службе через URL SAS и испытываете ошибку:

* убедитесь, что URL-адрес предоставляет необходимые разрешения на чтение или перечисление ресурсов.
* Убедитесь, что срок действия URL-адреса не истек.
* Если URL-адрес основан на политике доступа, убедитесь, что политика доступа не была отозвана.

Если вы случайно прикрепили, используя недействительный URL SAS и теперь не можете отсоединиться, выполните следующие действия:

1. При запуске Системы хранения нажмите F12, чтобы открыть окно Developer Tools.
2. На вкладке **Приложения** выберите **file:// локального хранения** > **на** дереве слева.
3. Найти ключ, связанный с типом службы для проблемного URI SAS. Например, если неверный URI SAS относится к контейнеру больших двоичных объектов, найдите ключ с именем `StorageExplorer_AddStorageServiceSAS_v1_blob`.
4. Значение ключа должно быть массивом JSON. Найдите объект, связанный с плохим URI, а затем удалите его.
5. Нажмите клавиши CTRL+R, чтобы перезагрузить обозреватель хранилища.

## <a name="linux-dependencies"></a>Зависимости Linux

Storage Explorer 1.10.0 и позже доступен в виде оснастки из Snap Store. Снимок Storage Explorer автоматически устанавливает все свои зависимости и обновляется при доступной новой версии привязки. Установка оснастки Storage Explorer является рекомендуемым методом установки.

Исследователь хранения требует использования менеджера паролей, который может потребоваться подключить вручную, прежде чем Storage Explorer будет работать правильно. Вы можете подключить Storage Explorer к менеджеру паролей вашей системы, запустив следующую команду:

```bash
snap connect storage-explorer:password-manager-service :password-manager-service
```

Вы также можете скачать приложение в виде файла .tar.gz, но вам придется устанавливать зависимости вручную.

> [!IMPORTANT]
> Исследователь хранения, как это предусмотрено в загрузке .tar.gz, поддерживается только для дистрибутивов Ubuntu. Другие дистрибутивы не были проверены и могут потребовать альтернативных или дополнительных пакетов.

Эти пакеты являются наиболее распространенными требованиями для storage Explorer на Linux:

* [Среда выполнения .NET Core 2.2](/dotnet/core/install/dependencies?tabs=netcore22&pivots=os-linux)
* `libgconf-2-4`
* `libgnome-keyring0` либо `libgnome-keyring-dev`
* `libgnome-keyring-common`

> [!NOTE]
> Версия 1.7.0 и более ранняя версия Для хранения explorer а также ранее требуют .NET Core 2.0. Если установлена новая версия ядра .NET, вам придется [патч Storage Explorer.](#patching-storage-explorer-for-newer-versions-of-net-core) Если вы работаете в Хранилище Explorer 1.8.0 или позже, вы должны иметь возможность использовать до .NET Core 2.2. Версии за пределами 2.2 не были проверены на работу в настоящее время.

### <a name="patching-storage-explorer-for-newer-versions-of-net-core"></a>Патчинг исследователь хранения для новых версий .NET Core

Для storage Explorer 1.7.0 или раньше, возможно, придется исправления версии .NET Ядро, используемое Storage Explorer:

1. Скачать версию 1.5.43 StreamJsonRpc [от NuGet](https://www.nuget.org/packages/StreamJsonRpc/1.5.43). Ищите ссылку "Загрузить пакет" на правой стороне страницы.
2. После загрузки пакета измените его `.nupkg` `.zip`расширение файла с .
3. Расстегните пакет.
4. Откройте папку `streamjsonrpc.1.5.43/lib/netstandard1.1/`.
5. Копируйте `StreamJsonRpc.dll` следующие места в папке Storage Explorer:
   * `StorageExplorer/resources/app/ServiceHub/Services/Microsoft.Developer.IdentityService/`
   * `StorageExplorer/resources/app/ServiceHub/Hosts/ServiceHub.Host.Core.CLR.x64/`
