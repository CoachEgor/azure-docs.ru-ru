# <a name="multi-protocol-access-on-azure-data-lake-storage"></a>Multi-protocol access on Azure Data Lake Storage (preview) (Доступ с использованием нескольких протоколов на Azure Data Lake Storage (предварительная версия))

APIs Blob теперь работают с учетными записями, которые имеют иерархическое пространство имен. Это открывает экосистему инструментов, приложений и служб, а также несколько функций хранения Blob для учетных записей, которые имеют иерархическое пространство имен.

До недавнего времени, возможно, приходилось поддерживать отдельные решения для хранения объектов и аналитики. Это связано с тем, что Azure Data Lake Storage Gen2 имела ограниченную поддержку экосистем. Она также имела ограниченный доступ к функциям службы Blob, таким как диагностическая регистрация. Раздробленное решение для хранения трудно поддерживать, поскольку для выполнения различных сценариев необходимо перемещать данные между учетными записями. Тебе больше не нужно этого делать.

С помощью многопротокольного доступа в Data Lake Storage вы можете работать с данными, используя экосистему инструментов, приложений и служб. Это также включает в себя сторонние инструменты и приложения. Вы можете указать их на учетные записи, которые имеют иерархическое пространство имен без необходимости их изменять. Эти приложения работают *так же, как и* в случае, если они называют Blob AIS, поскольку APIs Blob теперь могут работать на данных в учетных записях, которые имеют иерархическое пространство имен.

Функции хранения Blob, такие как [диагностические журналы,](../common/storage-analytics-logging.md) [уровни доступа](storage-blob-storage-tiers.md)и [политики управления жизненным циклом хранения Blob,](storage-lifecycle-management-concepts.md) теперь работают с учетными записями, которые имеют иерархическое пространство имен. Таким образом, можно включить иерархические пространства имен в учетных записях хранилища каблов, не теряя доступа к этим важным функциям. 

> [!NOTE]
> Доступ к multi-protocol на хранении озера данных вообще доступен и имеющийся в всех зонах. Некоторые службы Azure или функции хранения Blob, включенные с помощью многопротокольного доступа, остаются в предварительном просмотре. ## <a name="how-multi-protocol-access-on-data-lake-storage-works"></a>Как работает мультипротоколный доступ к хранению озер данных

APIs Blob ИАПы хранения данных и Data Lake Storage Gen2 могут работать на тех же данных в учетных записях хранения, которые имеют иерархическое пространство имен. Data Lake Storage Gen2 направляет ApIs Blob через иерархическое пространство имен, чтобы можно было получить преимущества первоклассных операций каталога и списков контроля доступа, соответствующих POSIX (ACL). 

![Многопротоколный доступ на концептуальном хранении озер данных](./media/data-lake-storage-interop/interop-concept.png) 

Существующие инструменты и приложения, которые используют Blob API, автоматически получают эти преимущества. Разработчикам не придется их изменять. Data Lake Storage Gen2 последовательно применяет акфолы каталогов и аКЛ на уровне файлов независимо от протокола, который используют инструменты и приложения для доступа к данным.
