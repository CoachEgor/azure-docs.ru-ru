# <a name="data-collection-in-azure-security-center"></a>Сбор данных в центре безопасности Azure
Центр безопасности собирает данные из виртуальных машин Azure, масштабируемых наборов виртуальных машин, контейнеров IaaS и не Azure (включая локальные компьютеры) для отслеживания уязвимостей и угроз безопасности. Сбор данных выполняется с помощью агента Log Analytics, который считывает различные конфигурации, связанные с безопасностью, и журналы событий с компьютера и копирует данные в рабочую область для анализа. К примерам таких данных относятся тип и версия операционной системы, журналы операционной системы (журналы событий Windows), выполняющиеся процессы, имя компьютера, IP-адреса и имя вошедшего пользователя. Агент Log Analytics также копирует файлы аварийного дампа в рабочую область.

Сбор данных необходим, чтобы обнаруживать недостающие обновления и неправильно настроенные параметры безопасности ОС, узнавать, включена ли защита конечных точек, а также определять работоспособность и обнаруживать угрозы. 

В этой статье описывается, как установить агент Log Analytics и задать Log Analytics рабочей области, в которой должны храниться собранные данные. Чтобы включить сбор данных, необходимо выполнить обе операции. 

> [!NOTE]
> - Сбор данных требуется только для ресурсов вычислений (виртуальных машин, масштабируемых наборов виртуальных машин, контейнеров IaaS и компьютеров, не относящихся к Azure). Вы можете пользоваться центром безопасности Azure даже без подготовки агентов. Однако защита будет ограничена, а перечисленные выше возможности не будут поддерживаться.  
> - Список поддерживаемых платформ представлен в статье [Поддерживаемые платформы в центре безопасности Azure](security-center-os-coverage.md).
> - Хранение данных в Log Analytics, будь то новая или существующая Рабочая область, может взимать дополнительную плату за хранение данных. Дополнительные сведения см. на [странице с расценками](https://azure.microsoft.com/pricing/details/security-center/).

## Включение автоматической подготовки агента Log Analytics<a name="auto-provision-mma"></a>

Чтобы получить данные с компьютеров, необходимо установить агент Log Analytics. Установку агента можно выполнить автоматически (рекомендуется) или установить агент вручную.  

>[!NOTE]
> Автоматическая подготовка по умолчанию отключена. Чтобы центр безопасности использовал автоматическую подготовку по умолчанию, выберите вариант **Вкл**.
>

Если включена автоматическая подготовка, центр безопасности подготавливает агент Log Analytics на всех поддерживаемых виртуальных машинах Azure и новых созданных. Мы настоятельно рекомендуем применять автоматическую подготовку, но вы также можете установить агент вручную. [Узнайте, как установить расширение агента log Analytics](#manual-agent).



Чтобы включить автоматическую подготовку агента Log Analytics, выполните следующие действия.
1. В главном меню Центра безопасности выберите **Pricing & settings** (Цены и параметры).
2. Щелкните подходящую подписку.

   ![Выберите подписку.][7]

3. Выберите **сбор данных**.
4. В разделе **Автоматическая подготовка** выберите **Включить**, чтобы включить автоматическую подготовку.
5. Щелкните **Сохранить**.

   ![Включение автоматической подготовки][1]

>[!NOTE]
> - Инструкции по подготовке уже существующей установки см. в разделе [Автоматическая подготовка уже существующей установки агента](#preexisting).
> - Инструкции по подготовке вручную см. в [статье Установка расширения log Analytics Agent вручную](#manual-agent).
> - Инструкции по отключению автоматической подготовки см. в разделе [Отключение автоматической подготовки](#offprovisioning).
> - Инструкции о том, как подключиться к центру безопасности с помощью PowerShell, см. в разделе [Automate onboarding of Azure Security Center using PowerShell](security-center-powershell-onboarding.md) (Автоматизация подключения центра безопасности Azure с помощью PowerShell).
>

## <a name="workspace-configuration"></a>Конфигурация рабочей области
Данные, собранные центром безопасности, хранятся в рабочих областях Log Analytics. Вы можете настроить хранение данных, собранных с виртуальных машин Azure, в рабочих областях, созданных центром безопасности, или в созданной вами рабочей области. 

Конфигурация рабочей области задается для каждой подписки отдельно, а несколько подписок могут использовать одну и ту же рабочую область.

### <a name="using-a-workspace-created-by-security-center"></a>Использование рабочей области, созданной центром безопасности

Центр безопасности может автоматически создать стандартную рабочую область для хранения данных. 

Чтобы выбрать рабочую область, созданную центром безопасности:

1. В разделе **Конфигурация рабочей области по умолчанию** выберите параметр "Использовать рабочие области, созданные центром безопасности".
   ![Выбор ценовой категории][10] 

1. В нижней части страницы нажмите кнопку **Save**.<br>
    Центр безопасности создает группу ресурсов и рабочую область по умолчанию в этом географическом расположении, а затем подключает агент к этой рабочей области. Соглашение об именовании для рабочей области и группы ресурсов:<br>
   **Рабочая область: DefaultWorkspace-[ИД подписки]-[георасположение]<br> Группа ресурсов: DefaultResouceGroup-[георасположение]**

   Если подписка содержит виртуальные машины из нескольких географических расположений, центр безопасности создает несколько рабочих областей. Они создаются для соблюдения правил конфиденциальности данных.
1. Центр безопасности автоматически включит решение центра безопасности в рабочей области согласно ценовой категории, заданной для подписки. 

> [!NOTE]
> Ценовая категория Log Analytics для рабочих областей, созданных центром безопасности, не влияет на выставление счетов за его использование. Счета за использование центра безопасности всегда выставляются в зависимости от его политики безопасности и решений, установленных в рабочей области. Для уровня "Бесплатный" центр безопасности включает решение *SecurityCenterFree* в рабочей области по умолчанию. Для уровня "Стандартный" центр безопасности включает решение *Security* в рабочей области по умолчанию.
> Хранение данных в Log Analytics может повлечь за собой дополнительную плату за хранение данных. Дополнительные сведения см. на [странице с расценками](https://azure.microsoft.com/pricing/details/security-center/).

Дополнительные сведения о существующих учетных записях log Analytics см. в разделе [существующие клиенты log Analytics](security-center-faq.md#existingloganalyticscust).

### <a name="using-an-existing-workspace"></a>Использование имеющейся рабочей области

Если у вас уже есть рабочая область Log Analytics, может потребоваться использовать ту же рабочую область.

Чтобы использовать имеющуюся рабочую область Log Analytics, вам потребуются разрешение на чтение и запись в этой рабочей области.

> [!NOTE]
> Решения, включенные в имеющейся рабочей области, будут применяться к подключенным к ним виртуальным машинам Azure. В случае платных решений за это может взиматься дополнительная плата. Из соображений конфиденциальности данных убедитесь, что выбранная рабочая область находится в нужном географическом регионе.
> Хранение данных в log Analytics может повлечь за собой дополнительную плату за хранение данных. Дополнительные сведения см. на [странице с расценками](https://azure.microsoft.com/pricing/details/security-center/).

Чтобы выбрать имеющуюся рабочую область Log Analytics, сделайте следующее:

1. В разделе **Конфигурация рабочей области по умолчанию** выберите **Использовать другую рабочую область**.

   ![Выбор имеющейся рабочей области][2]

2. В раскрывающемся меню выберите рабочую область для хранения собранных данных.

   > [!NOTE]

Ознакомьтесь с разделом [Выбор рабочих областей между подписками](security-center-enable-data-collection.md#cross-subscription-workspace-selection), чтобы получить дополнительную информацию. Вам необходимо разрешение на доступ к рабочей области.
   >
   >

3. Щелкните **Сохранить**.
4. Если нажать **Сохранить**, появится запрос на изменение настроек отслеживаемых виртуальных машин, которые ранее были подключены к рабочей области по умолчанию.

   - Выберите **Нет**, если нужно, чтобы параметры новой рабочей области применялись только к новым виртуальным машинам. Новые параметры рабочей области применяются только к новым установкам агента. недавно обнаруженные виртуальные машины, на которых не установлен агент Log Analytics.
   - Выберите **Да**, если нужно, чтобы параметры новой рабочей области применялись ко всем виртуальным машинам. Кроме того, каждая виртуальная машина, подключенная к рабочей области, созданной центром безопасности, переподключается к новой целевой рабочей области.

   > [!NOTE]
   > Если вы выбрали "Да", не удаляйте рабочие области, созданные центром безопасности, пока все виртуальные машины не подключатся повторно к новой целевой рабочей области. Операция завершится сбоем, если удалить рабочую область слишком рано.
   >
   >

   - Выберите **Отменить**, чтобы отменить операцию.

     ![Выбор имеющейся рабочей области][3]

5. Выберите ценовую категорию для нужной рабочей области, чтобы задать агент Log Analytics. <br>Чтобы использовать имеющуюся рабочую область, задайте для нее ценовую категорию. При этом в рабочей области будет установлено решение центра безопасности, если его еще нет.

    а)  В главном меню центра безопасности выберите пункт **Параметры цен &** .
     
    б)  Выберите нужную рабочую область, в которой требуется подключить агент.
        ![Выбор рабочей области][7] c. Выберите ценовую категорию.
        ![Выбор ценовой категории][9]
   
   >[!NOTE]
   >Если в рабочей области уже включено решение **Security** или **SecurityCenterFree**, ценовая категория будет задана автоматически. 

## <a name="cross-subscription-workspace-selection"></a>Выбор рабочих областей между подписками
При выборе рабочей области для хранения данных доступны все рабочие области во всех ваших подписках. Выбор рабочих областей между подписками позволяет собирать данные из виртуальных машин, работающих в разных подписках, и сохранять их в выбранной рабочей области. Это полезно, если в организации используется централизованная рабочая область, которую требуется использовать для сбора данных о безопасности. Дополнительные сведения о том, как управлять рабочими областями, см. в статье [Управление рабочими областями](https://docs.microsoft.com/azure/log-analytics/log-analytics-manage-access).


## <a name="data-collection-tier"></a>Уровень сбора данных
Выбор уровня сбора данных в центре безопасности Azure повлияет только на хранение событий безопасности в рабочей области Log Analytics. Агент Log Analytics по-прежнему соберет и проанализирует события безопасности, необходимые для обнаружения угроз в центре безопасности Azure, независимо от уровня событий безопасности, которые вы выбираете для хранения в рабочей области Log Analytics (если они есть). Выбор хранения событий безопасности в рабочей области позволит проводить расследование, поиск и аудит этих событий в вашей рабочей области. 
> [!NOTE]
> Хранение данных в log Analytics может повлечь за собой дополнительную плату за хранение данных. Дополнительные сведения см. на [странице с расценками](https://azure.microsoft.com/pricing/details/security-center/).
> 
> Вы можете выбрать необходимую политику фильтрации для подписок и рабочих областей из четырех наборов событий, которые хранятся в рабочей области. 

- **Отсутствует.** Отключение сбора событий безопасности. Это значение по умолчанию.
- **Минимальный.** Меньший набор событий для клиентов, которые хотят сократить количество событий.
- **Общий.** Этот набор событий удовлетворяет потребности большинства клиентов и позволяет просмотреть весь журнал аудита.
- **Все события.** Для клиентов, которые хотят убедиться, что все события сохранены.


> [!NOTE]
> Эти наборы событий безопасности доступны только на уровне "Стандартный" центра безопасности. Дополнительные сведения о ценовых категориях центра безопасности см. на странице [Цены](security-center-pricing.md).
Эти наборы предназначены для стандартных сценариев. Выберите набор, который соответствует вашим требованиям, прежде чем внедрить его.
>
>

Для определения событий, которые будут принадлежать к наборам событий **Общий** и **Минимальный**, мы ознакомились с отзывами клиентов и отраслевыми стандартами, чтобы изучить нефильтрованную частоту каждого события и их использования. В процессе мы использовали приведенные ниже рекомендации.

- **Минимальный.** Убедитесь, что этот набор содержит только события, которые могут указывать на брешь в системе безопасности и важные события небольших объемов. Например, этот набор содержит успешные и неудачные попытки входа (идентификаторы событий 4624, 4625), но не содержит сведений о том, что важно для аудита, но не имеет смысла для обнаружения и имеет относительно большой объем. Большую часть данных этого набора занимают события входа и создания процессов (идентификатор событий 4688).
- **Общий.** В этом наборе предоставляется полный журнал аудита пользователя. Например, этот набор содержит как имена входа пользователей, так и пользовательские входы (идентификатор события 4634). Мы добавили действия аудита, такие как изменение группы безопасности, операции основного контроллера домена Kerberos и другие события, рекомендованные отраслевыми организациями.

События небольших объемов были включены в набор "Общий". Основной причиной для его выбора среди всех событий является возможность уменьшить объем и оставить определенные события нефильтрованными.

### Автоматическая подготовка в случае установки ранее созданного агента<a name="preexisting"></a> 

В описанных ниже вариантах использования указано, как автоматическая подготовка работает в тех случаях, когда агент или расширение уже установлены. 

- Агент Log Analytics установлен на компьютере, но не является расширением (Direct Agent)<br>
Если агент Log Analytics установлен непосредственно на виртуальной машине (а не в качестве расширения Azure), центр безопасности установит расширение агента Log Analytics и может обновить агент Log Analytics до последней версии.
Установленный агент будет продолжать сообщать о уже настроенных рабочих областях, а также будет сообщать о рабочей области, настроенной в центре безопасности (поддержка нескольких сайтов поддерживается на компьютерах под управлением Windows).
Если настроенная рабочая область является рабочей областью пользователя (не является рабочей областью по умолчанию центра безопасности), необходимо установить решение "Security/Секуритифри" в ней для центра безопасности, чтобы начать обработку событий с виртуальных машин и компьютеров, передающих данные в эту рабочую область.<br>
<br>
Для компьютеров Linux поддержка нескольких сетевых интерфейсов еще не поддерживается, поэтому при обнаружении существующей установки агента Автоматическая подготовка не выполняется, а конфигурация компьютера не изменяется.
<br>
Для существующих компьютеров в подписках, подключенных к центру безопасности до 2019-03-17, при обнаружении существующего агента расширение Log Analytics Agent не будет установлено и компьютер не будет затронуты. Сведения об этих компьютерах см. в рекомендации по устранению проблем с установкой агента на этих компьютерах.

  
- На компьютере установлен Агент System Center Operations Manager<br>
Центр безопасности установит расширение агента Log Analytics параллельно с существующим Operations Manager. Существующий агент Operations Manager будет продолжать отчитываться о сервере Operations Manager обычно. Обратите внимание, что агент Operations Manager и агент Log Analytics используют общие библиотеки времени выполнения, которые будут обновлены до последней версии во время этого процесса.
Примечание. Если установлен агент Operations Manager версии **2012, не** включайте автоматическую подготовку.<br>

- Расширение виртуальной машины уже установлено<br>
    - Если агент мониторинга установлен как расширение, конфигурация расширения позволяет создавать отчеты только для одной рабочей области. Центр безопасности не переопределяет установленные подключения к рабочим областям пользователя. Центр безопасности будет хранить данные безопасности с виртуальной машины в уже подключенной рабочей области при условии, что на нем установлено решение "безопасность" или "Секуритифри". Центр безопасности может обновить версию расширения до последней версии в этом процессе.  
    - Чтобы узнать, в какую рабочую область отправляет данные имеющееся расширение, выполните тест для [проверки соединения с центром безопасности Azure](https://blogs.technet.microsoft.com/yuridiogenes/2017/10/13/validating-connectivity-with-azure-security-center/). Кроме того, можно открыть Log Analytics рабочие области, выбрать рабочую область, выбрать виртуальную машину и просмотреть соединение агента Log Analytics. 
    - Если у вас есть среда, в которой агент Log Analytics установлен на клиентских рабочих станциях и модуль отчетов в существующей Log Analytics рабочей области, ознакомьтесь со списком [операционных систем, поддерживаемых центром безопасности Azure](security-center-os-coverage.md) , чтобы убедиться, что ваша операционная система поддерживается. Дополнительные сведения см. в разделе [существующие клиенты log Analytics](security-center-faq.md#existingloganalyticscust).
 
### Отключение автоматической подготовки <a name="offprovisioning"></a>
Отключить автоматическую подготовку ресурсов можно в любое время, отключив этот параметр в политике безопасности. 


1. Вернитесь к основному меню центра безопасности и выберите "Политика безопасности".
2. Щелкните **изменить параметры** в строке подписки, для которой требуется отключить автоматическую подготовку.
3. В колонке **Security policy – Data Collection** (Политика безопасности — сбор данных) в разделе **Автоматическая подготовка** выберите **Отключить**.
4. Щелкните **Сохранить**.

   ![Отключение автоматической подготовки][6]

Если автоматическая подготовка отключена, раздел конфигурации рабочей области по умолчанию не отображается.

Если отключить автоматическую подготовку:
-   агенты не будут подготавливаться на новых виртуальных машинах;
-   центр безопасности перестанет собирать данные из рабочей области по умолчанию.
 
> [!NOTE]
>  Отключение автоматической подготовки не приводит к удалению агента Log Analytics из виртуальных машин Azure, на которых был подготовлен агент. Сведения о том, как удалить расширение OMS, см. в разделе [Как удалить расширения OMS, установленные центром безопасности](security-center-faq.md#remove-oms).
>
    
## Подготовка агента вручную <a name="manual-agent"></a>
 
Существует несколько способов установить агент Log Analytics вручную. При установке вручную обязательно отключите автоматическую подготовку.

### <a name="operations-management-suite-vm-extension-deployment"></a>Развертывание расширения виртуальной машины Operations Management Suite 

Вы можете вручную установить агент Log Analytics, чтобы центр безопасности мог получать данные безопасности от виртуальных машин и предоставлять рекомендации и оповещения.
1. Выберите для параметра "Автоматическая подготовка" значение "ВЫКЛ".
2. Создайте рабочую область и задайте ценовую категорию для рабочей области, которую вы хотите задать для агента Log Analytics.

   а)  В главном меню центра безопасности выберите пункт **Политика безопасности**.
     
   б)  Выберите рабочую область, в которой требуется подключить агент. Убедитесь, что рабочая область находится в той же подписке, которая используется в центре безопасности, а у вас есть разрешения на чтение и запись в этой рабочей области.
       ![Выбор рабочей области][8]
3. Выберите ценовую категорию.
   ![Выбор ценовой категории][9] 
   >[!NOTE]
   >Если в рабочей области уже включено решение **Security** или **SecurityCenterFree**, ценовая категория будет задана автоматически. 
   > 

4. Если вы хотите развернуть агенты на новых виртуальных машинах с помощью шаблона Resource Manager, установите расширение виртуальной машины OMS:

   а)  [Установка расширения виртуальной машины OMS для Windows](../virtual-machines/extensions/oms-windows.md)
    
   б)  [Установка расширения виртуальной машины OMS для Linux](../virtual-machines/extensions/oms-linux.md)
5. Чтобы развернуть расширения на имеющихся виртуальных машинах, следуйте инструкциям из статьи [Сбор данных о виртуальных машинах Azure](../azure-monitor/learn/quick-collect-azurevm.md).

   > [!NOTE]
   > Действия из раздела **Сбор данных о событиях и производительности** выполнять необязательно.
   >
6. Чтобы развернуть расширение с помощью PowerShell, используйте следующий пример PowerShell.
   
   [!INCLUDE [updated-for-az](../../includes/updated-for-az.md)]
   
   1. Перейдите к **Log Analytics** и выберите **Дополнительные параметры**.
    
      ![Настройка Log Analytics][11]

   2. Скопируйте значения из полей **WorkspaceID** и **Первичный ключ**.
  
      ![Копирование значений][12]

   3. Заполните общедоступную и частную конфигурации следующими значениями:
     
           $PublicConf = @{
               "workspaceId"= "<WorkspaceID value>"
           }
 
           $PrivateConf = @{
               "workspaceKey"= "<Primary key value>"
           }

      - При установке на виртуальной машине Windows:
        
            Set-AzVMExtension -ResourceGroupName $vm.ResourceGroupName -VMName $vm.Name -Name "MicrosoftMonitoringAgent" -Publisher "Microsoft.EnterpriseCloud.Monitoring" -ExtensionType "MicrosoftMonitoringAgent" -TypeHandlerVersion '1.0' -Location $vm.Location -Settingstring $PublicConf -ProtectedSettingString $PrivateConf -ForceRerun True 
    
      - При установке на виртуальной машине Linux:
        
            Set-AzVMExtension -ResourceGroupName $vm1.ResourceGroupName -VMName $vm1.Name -Name "OmsAgentForLinux" -Publisher "Microsoft.EnterpriseCloud.Monitoring" -ExtensionType "OmsAgentForLinux" -TypeHandlerVersion '1.0' -Location $vm.Location -Settingstring $PublicConf -ProtectedSettingString $PrivateConf -ForceRerun True`

> [!NOTE]
> Инструкции о том, как подключиться к центру безопасности с помощью PowerShell, см. в разделе [Automate onboarding of Azure Security Center using PowerShell](security-center-powershell-onboarding.md) (Автоматизация подключения центра безопасности Azure с помощью PowerShell).
