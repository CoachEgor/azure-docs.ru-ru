# <a name="data-collection-in-azure-security-center"></a>Сбор данных в центре безопасности Azure
Центр безопасности собирает данные с виртуальных машин Azure (VM), виртуальных наборов машин, контейнеров IaaS и компьютеров, не связанных с Azure (включая на местах), для мониторинга уязвимостей и угроз безопасности. Данные собираются с помощью агента по анализу журналов, который считывает различные конфигурации, связанные с безопасностью, и журналы событий из машины и копирует данные в рабочее пространство для анализа. К примерам таких данных относятся тип и версия операционной системы, журналы операционной системы (журналы событий Windows), выполняющиеся процессы, имя компьютера, IP-адреса и имя вошедшего пользователя. Агент аналитики журнала также копирует файлы аварийного дампа в рабочее пространство.

Сбор данных необходим для обеспечения видимости недостающих обновлений, неправильно настроенных настроек безопасности ОС, состояния защиты конечных точек, а также защиты здоровья и угроз. 

В этой статье описывается, как установить агента аналитики журнала и установить рабочее пространство для анализа журналов, в котором можно хранить собранные данные. Чтобы включить сбор данных, необходимо выполнить обе операции. 

> [!NOTE]
> - Сбор данных необходим только для ресурсов Compute (VMs, виртуальные наборы машин, контейнеры IaaS и компьютеры, не связанные с Azure). Вы можете пользоваться центром безопасности Azure даже без подготовки агентов. Однако защита будет ограничена, а перечисленные выше возможности не будут поддерживаться.  
> - Список поддерживаемых платформ представлен в статье [Поддерживаемые платформы в центре безопасности Azure](security-center-os-coverage.md).
> - Хранение данных в журнале Analytics, независимо от того, используете ли вы новое или существующее рабочее пространство, может повлечь за собой дополнительные расходы на хранение данных. Для получения дополнительной информации смотрите [страницу ценообразования](https://azure.microsoft.com/pricing/details/security-center/).

## <a name="enable-automatic-provisioning-of-the-log-analytics-agent"></a>Включить автоматическую подготовку агента по аналитике журналов<a name="auto-provision-mma"></a>

Для сбора данных с машин необходимо установить агента по анализу журналов. Установка агента может быть выполнена автоматически (рекомендуется) или вы можете установить агента вручную.  

>[!NOTE]
> Автоматическая подготовка по умолчанию отключена. Чтобы центр безопасности использовал автоматическую подготовку по умолчанию, выберите вариант **Вкл**.
>

Когда автоматическая подготовка выполняется, Центр безопасности предоставляет агенту аналитики журналов все поддерживаемые VMs Azure и любые новые, которые создаются. Мы настоятельно рекомендуем применять автоматическую подготовку, но вы также можете установить агент вручную. [Узнайте, как установить расширение агента аналитики журнала.](#manual-agent)



Для автоматического подготовки агента по анализу журналов:
1. В главном меню Центра безопасности выберите **Pricing & settings** (Цены и параметры).
2. Нажмите на соответствующую подписку

   ![Выбор подписки][7]

3. Выберите **сбор данных**.
4. В разделе **Автоматическая подготовка** выберите **Включить**, чтобы включить автоматическую подготовку.
5. Нажмите кнопку **Сохранить**.

   ![Включение автоматической подготовки][1]

>[!NOTE]
> - Инструкции по подготовке уже существующей установки см. в разделе [Автоматическая подготовка уже существующей установки агента](#preexisting).
> - Для получения инструкций по ручной подготовке [см.](#manual-agent)
> - Инструкции по отключению автоматической подготовки см. в разделе [Отключение автоматической подготовки](#offprovisioning).
> - Инструкции о том, как подключиться к центру безопасности с помощью PowerShell, см. в разделе [Automate onboarding of Azure Security Center using PowerShell](security-center-powershell-onboarding.md) (Автоматизация подключения центра безопасности Azure с помощью PowerShell).

## <a name="workspace-configuration"></a>Конфигурация рабочей области
Данные, собранные центром безопасности, хранятся в рабочих областях Log Analytics. Вы можете настроить хранение данных, собранных с виртуальных машин Azure, в рабочих областях, созданных центром безопасности, или в созданной вами рабочей области. 

Конфигурация рабочей области задается для каждой подписки отдельно, а несколько подписок могут использовать одну и ту же рабочую область.

### <a name="using-a-workspace-created-by-security-center"></a>Использование рабочей области, созданной центром безопасности

Центр безопасности может автоматически создать стандартную рабочую область для хранения данных. 

Чтобы выбрать рабочую область, созданную центром безопасности:

1. В разделе **Конфигурация рабочей области по умолчанию** выберите параметр "Использовать рабочие области, созданные центром безопасности".
   ![Выбор ценовой категории][10] 

1. Нажмите **Сохранить**.<br>
    Центр безопасности создает группу ресурсов и рабочую область по умолчанию в этом географическом расположении, а затем подключает агент к этой рабочей области. Соглашение об именовании для рабочей области и группы ресурсов:<br>
   **Рабочая область: DefaultWorkspace-[ИД подписки]-[георасположение]<br> Группа ресурсов: DefaultResouceGroup-[георасположение]**

   Если подписка содержит виртуальные машины из нескольких географических расположений, центр безопасности создает несколько рабочих областей. Они создаются для соблюдения правил конфиденциальности данных.
1. Центр безопасности автоматически включит решение центра безопасности в рабочей области согласно ценовой категории, заданной для подписки. 

> [!NOTE]
> Ценовая категория Log Analytics для рабочих областей, созданных центром безопасности, не влияет на выставление счетов за его использование. Счета за использование центра безопасности всегда выставляются в зависимости от его политики безопасности и решений, установленных в рабочей области. Для уровня "Бесплатный" центр безопасности включает решение *SecurityCenterFree* в рабочей области по умолчанию. Для уровня "Стандартный" центр безопасности включает решение *Security* в рабочей области по умолчанию.
> Хранение данных в журнале Analytics может повлечь за себя дополнительные расходы на хранение данных. Для получения дополнительной информации смотрите [страницу ценообразования](https://azure.microsoft.com/pricing/details/security-center/).

Для получения дополнительной информации о существующих учетных записях аналитики журналов [см.](./faq-azure-monitor-logs.md)

### <a name="using-an-existing-workspace"></a>Использование имеющейся рабочей области

Если у вас уже есть существующее рабочее пространство Analytics журнала, можно использовать то же рабочее пространство.

Из соображений конфиденциальности данных убедитесь, что выбранная рабочая область находится в нужном географическом регионе.
> Хранение данных в журнале аналитики может повлечь за себя дополнительные расходы на хранение данных. Для получения дополнительной информации смотрите [страницу ценообразования](https://azure.microsoft.com/pricing/details/security-center/).

## <a name="cross-subscription-workspace-selection"></a>Выбор рабочих областей между подписками
При выборе рабочей области для хранения данных доступны все рабочие области во всех ваших подписках. Выбор рабочих областей между подписками позволяет собирать данные из виртуальных машин, работающих в разных подписках, и сохранять их в выбранной рабочей области. Это полезно, если в организации используется централизованная рабочая область, которую требуется использовать для сбора данных о безопасности. Дополнительные сведения о том, как управлять рабочими областями, см. в статье [Управление рабочими областями](https://docs.microsoft.com/azure/log-analytics/log-analytics-manage-access).


## <a name="data-collection-tier"></a>Уровень сбора данных
Выбор уровня сбора данных в центре безопасности Azure повлияет только на хранение событий безопасности в рабочей области Log Analytics. Агент Log Analytics по-прежнему будет собирать и анализировать события безопасности, необходимые для защиты угроз Центра безопасности Azure, независимо от того, какой уровень событий безопасности вы выберете для хранения в рабочем пространстве Analytics журнала (если таковые имеется). Выбор хранения событий безопасности в рабочей области позволит проводить расследование, поиск и аудит этих событий в вашей рабочей области. 
> [!NOTE]
> Хранение данных в журнале аналитики может повлечь за себя дополнительные расходы на хранение данных. Для получения дополнительной информации смотрите [страницу ценообразования](https://azure.microsoft.com/pricing/details/security-center/).
> 
> Вы можете выбрать необходимую политику фильтрации для подписок и рабочих областей из четырех наборов событий, которые хранятся в рабочей области. 

- **Отсутствует.** Отключение сбора событий безопасности. Это параметр по умолчанию.
- **Минимальный.** Меньший набор событий для клиентов, которые хотят сократить количество событий.
- **Общий.** Этот набор событий удовлетворяет потребности большинства клиентов и позволяет просмотреть весь журнал аудита.
- **Все события.** Для клиентов, которые хотят убедиться, что все события сохранены.


> [!NOTE]
> Эти наборы событий безопасности доступны только на уровне "Стандартный" центра безопасности. Дополнительные сведения о ценовых категориях центра безопасности см. на странице [Цены](security-center-pricing.md).
Эти наборы предназначены для стандартных сценариев. Выберите набор, который соответствует вашим требованиям, прежде чем внедрить его.
>
>

Для определения событий, которые будут принадлежать к наборам событий **Общий** и **Минимальный**, мы ознакомились с отзывами клиентов и отраслевыми стандартами, чтобы изучить нефильтрованную частоту каждого события и их использования. В процессе мы использовали приведенные ниже рекомендации.

- **Минимальный.** Убедитесь, что этот набор содержит только события, которые могут указывать на брешь в системе безопасности и важные события небольших объемов. Например, этот набор содержит успешный и неудачный логин пользователя (идентитарные данные событий 4624, 4625), но не содержит вывески, которая важна для аудита, но не имеет смысла для обнаружения и имеет относительно большой объем. Большую часть данных этого набора занимают события входа и создания процессов (идентификатор событий 4688).
- **Общий.** В этом наборе предоставляется полный журнал аудита пользователя. Например, этот набор содержит как пользовательские логины, так и выходы на вывески для вывески (event ID 4634). Мы добавили действия аудита, такие как изменение группы безопасности, операции основного контроллера домена Kerberos и другие события, рекомендованные отраслевыми организациями.

События небольших объемов были включены в набор "Общий". Основной причиной для его выбора среди всех событий является возможность уменьшить объем и оставить определенные события нефильтрованными.

> - Чтобы собирать данные о [событиях с кодом 5156](https://www.ultimatewindowssecurity.com/securitylog/encyclopedia/event.aspx?eventID=5156) на платформе фильтрации Windows, необходимо включить [Аудит подключения платформы фильтрации](https://docs.microsoft.com/windows/security/threat-protection/auditing/audit-filtering-platform-connection) (Auditpol /set /subcategory:"Filtering Platform Connection" /Success:Enable)
>

Чтобы выбрать политику фильтрации, сделайте следующее:
1. На странице **сбора данных** выберите политику фильтрации в соответствии с **событиями безопасности.**
2. Нажмите кнопку **Сохранить**.

   ![Выбор политики фильтрации][5]

### <a name="automatic-provisioning-in-cases-of-a-pre-existing-agent-installation"></a>Автоматическое обеспечение в случае уже существующей установки агента<a name="preexisting"></a> 

В описанных ниже вариантах использования указано, как автоматическая подготовка работает в тех случаях, когда агент или расширение уже установлены. 

- Агент аналитики журнала устанавливается на машине, но не в качестве расширения (Прямой агент)<br>
Если агент аналитики журнала установлен непосредственно на VM (не в качестве расширения Azure), Центр безопасности установит расширение агента аналитики журнала и может обновить агент аналитики журнала до последней версии.
Установленный агент будет продолжать отчитываться перед уже настроенным рабочим пространством (ы) и дополнительно отчитываться перед рабочей областью, настроенной в Центре безопасности (Multi-homing поддерживается на компьютерах Windows).
Если настроенное рабочее пространство является рабочим пространством пользователя (не рабочим пространством Центра безопасности по умолчанию), то вам нужно будет установить на него решение "Security/"securityFree" для начала обработки событий с вс-миноносов и компьютеров, сообщающих об этом рабочем пространстве.<br>
<br>
Для машин Linux мультинахонирование агента пока не поддерживается - следовательно, если будет обнаружена существующая установка агента, автоматическая подготовка не произойдет и конфигурация машины не будет изменена.
<br>
Для существующих машин по подписке на борт Центра безопасности до 2019-03-17, когда будет обнаружен существующий агент, расширение Log Analytics Agent не будет установлено и машина не будет затронута. Для этих машин, просмотреть "Разрешить мониторинг аментация агента здоровья на ваших машинах" рекомендации для решения вопросов установки агента на этих машинах.

  
- Агент диспетчера операций Системного центра установлен на машине<br>
Центр безопасности установит расширение log Analytics Agent рядом с существующим менеджером операций. Существующий агент менеджера операций будет продолжать отчитываться перед сервером менеджера операций в обычном режиме. Обратите внимание, что агент менеджера по операциям и агент по аналитике журналов имеют общие библиотеки времени выполнения, которые будут обновлены до последней версии в ходе этого процесса.
Примечание - Если версия агента операционного менеджера 2012 установлен, **не** включайте автоматическое обеспечение.<br>

- Расширение виртуальной машины уже установлено<br>
    - Когда агент мониторинга устанавливается в качестве расширения, конфигурация расширения позволяет отчитываться только об одном рабочем пространстве. Центр безопасности не переопределяет установленные подключения к рабочим областям пользователя. Центр безопасности будет хранить данные безопасности от VM в уже подключенном рабочем пространстве, при условии, что на нем установлено решение «Security» или «securityFree». Центр безопасности может обновить версию расширения до последней версии в этом процессе.  
    - Чтобы узнать, в какую рабочую область отправляет данные имеющееся расширение, выполните тест для [проверки соединения с центром безопасности Azure](https://blogs.technet.microsoft.com/yuridiogenes/2017/10/13/validating-connectivity-with-azure-security-center/). Кроме того, можно открыть рабочие пространства Log Analytics, выбрать рабочее пространство, выбрать VM и посмотреть на подключение агента Log Analytics. 
    - Если у вас есть среда, в которой агент Log Analytics устанавливается на клиентские рабочие станции и отчитывается перед существующим рабочим пространством Log Analytics, просмотрите список [операционных систем, поддерживаемых Центром безопасности Azure,](security-center-os-coverage.md) чтобы убедиться, что ваша операционная система поддерживается. Для получения дополнительной информации [см.](./faq-azure-monitor-logs.md)
 
### <a name="turn-off-automatic-provisioning"></a>Отключение автоматической подготовки <a name="offprovisioning"></a>
Отключить автоматическую подготовку ресурсов можно в любое время, отключив этот параметр в политике безопасности. 


1. Вернитесь к основному меню центра безопасности и выберите "Политика безопасности".
2. Нажмите **настройки edit** в строке подписки, для которой вы хотите отключить автоматическое обеспечение.
3. В колонке **Security policy – Data Collection** (Политика безопасности — сбор данных) в разделе **Автоматическая подготовка** выберите **Отключить**.
4. Нажмите кнопку **Сохранить**.

   ![Отключение автоматической подготовки][6]

Если автоматическая подготовка отключена, раздел конфигурации рабочей области по умолчанию не отображается.

Если отключить автоматическую подготовку:
-   агенты не будут подготавливаться на новых виртуальных машинах;
-   центр безопасности перестанет собирать данные из рабочей области по умолчанию.
 
> [!NOTE]
>  Отключение автоматической подготовки не удаляет агент анализа журналов из VMs-м внедряемых vMs-накопительов Azure, где был подготовлен агент. Сведения о том, как удалить расширение OMS, см. в разделе [Как удалить расширения OMS, установленные центром безопасности](faq-data-collection-agents.md#remove-oms).
>
    
## <a name="manual-agent-provisioning"></a>Подготовка агента вручную <a name="manual-agent"></a>
 
Существует несколько способов установки агента по анализу журналов вручную. При установке вручную обязательно отключите автоматическую подготовку.

### <a name="operations-management-suite-vm-extension-deployment"></a>Развертывание расширения виртуальной машины Operations Management Suite 

Вы можете вручную установить агент по аналитике журналов, чтобы Центр безопасности мог собирать данные о безопасности с ваших вс-чатов и предоставлять рекомендации и оповещения.
1. Выберите для параметра "Автоматическая подготовка" значение "ВЫКЛ".
2. Создайте рабочее пространство и установите уровень ценообразования для рабочей области, которую вы собираетесь установить агентом аналитики журнала:

   а.  В главном меню центра безопасности выберите пункт **Политика безопасности**.
     
   b.  Выберите рабочую область, в которой требуется подключить агент. Убедитесь, что рабочая область находится в той же подписке, которая используется в центре безопасности, а у вас есть разрешения на чтение и запись в этой рабочей области.
       ![Выбор рабочей области][8]
3. Выберите ценовую категорию.
   ![Выбор ценовой категории][9] 
   >[!NOTE]
   >Если в рабочей области уже включено решение **Security** или **SecurityCenterFree**, ценовая категория будет задана автоматически. 
   > 

4. Если вы хотите развернуть агенты на новых виртуальных машинах с помощью шаблона Resource Manager, установите расширение виртуальной машины OMS:

   а.  [Установка расширения виртуальной машины OMS для Windows](../virtual-machines/extensions/oms-windows.md)
    
   b.  [Установка расширения виртуальной машины OMS для Linux](../virtual-machines/extensions/oms-linux.md)
5. Чтобы развернуть расширения на имеющихся виртуальных машинах, следуйте инструкциям из статьи [Сбор данных о виртуальных машинах Azure](../azure-monitor/learn/quick-collect-azurevm.md).

   > [!NOTE]
   > Действия из раздела **Сбор данных о событиях и производительности** выполнять необязательно.
   >
6. Чтобы развернуть расширение с помощью PowerShell, используйте следующий пример PowerShell.
   
   [!INCLUDE [updated-for-az](../../includes/updated-for-az.md)]
   
   1. Перейдите к **Log Analytics** и выберите **Дополнительные параметры**.
    
      ![Настройка Log Analytics][11]

   2. Скопируйте значения из полей **WorkspaceID** и **Первичный ключ**.
  
      ![Копирование значений][12]

   3. Заполните общедоступную и частную конфигурации следующими значениями:
     
           $PublicConf = @{
               "workspaceId"= "<WorkspaceID value>"
           }
 
           $PrivateConf = @{
               "workspaceKey"= "<Primary key value>"
           }

      - При установке на виртуальной машине Windows:
        
            Set-AzVMExtension -ResourceGroupName $vm.ResourceGroupName -VMName $vm.Name -Name "MicrosoftMonitoringAgent" -Publisher "Microsoft.EnterpriseCloud.Monitoring" -ExtensionType "MicrosoftMonitoringAgent" -TypeHandlerVersion '1.0' -Location $vm.Location -settings $PublicConf -ProtectedSettingString $PrivateConf -ForceRerun True 
    
      - При установке на виртуальной машине Linux:
        
            Set-AzVMExtension -ResourceGroupName $vm1.ResourceGroupName -VMName $vm1.Name -Name "OmsAgentForLinux" -Publisher "Microsoft.EnterpriseCloud.Monitoring" -ExtensionType "OmsAgentForLinux" -TypeHandlerVersion '1.0' -Location $vm.Location -Settingstring $PublicConf -ProtectedSettingString $PrivateConf -ForceRerun True`

> [!NOTE]
> Инструкции о том, как подключиться к центру безопасности с помощью PowerShell, см. в разделе [Automate onboarding of Azure Security Center using PowerShell](security-center-powershell-onboarding.md) (Автоматизация подключения центра безопасности Azure с помощью PowerShell).

## <a name="troubleshooting"></a>Устранение неполадок

-   Руководство по устранению проблем с автоматической подготовкой см. в разделе [Проблемы с работоспособностью агента мониторинга](security-center-troubleshooting-guide.md#mon-agent).

-  Требования к сети см. в разделе [Устранение неполадок, связанных с требованиями к сети для агента мониторинга](security-center-troubleshooting-guide.md#mon-network-req).
-   Чтобы определить проблемы с бортами вручную, [см.](https://support.microsoft.com/help/3126513/how-to-troubleshoot-operations-management-suite-onboarding-issues)

- Для выявления неконтролируемых проблем с ввозами и компьютерами:

    Виртуальная машина или компьютер не отслеживается центром безопасности, если на ней не запущено расширение Microsoft Monitoring Agent. В машине может быть установлен местный агент, например, прямой агент OMS или агент диспетчера операций System Center. Виртуальные машины или компьютеры с этими агентами определены как неотслеживаемые, так как эти агенты не полностью поддерживаются в центре безопасности. Чтобы использовать все преимущества центра безопасности, требуется расширение Microsoft Monitoring Agent.

    Для получения дополнительной информации о причинах, по которым Центр безопасности не в состоянии успешно контролировать вс-мина и компьютеры инициализированы для автоматической подготовки, [см.](security-center-troubleshooting-guide.md#mon-agent)
