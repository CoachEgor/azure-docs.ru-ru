# <a name="data-collection-in-azure-security-center"></a>Сбор данных в центре безопасности Azure
Центр безопасности собирает данные c ваших виртуальных машин Azure, масштабируемых наборов виртуальных машин, контейнеров IaaS и компьютеров, не относящихся к Azure (в том числе локальных), чтобы отслеживать уязвимости и угрозы безопасности. Для сбора данных используется агент Log Analytics, который считывает различные конфигурации, связанные с безопасностью, и журналы событий с компьютера, а также копирует данные в рабочую область для анализа. К примерам таких данных относятся тип и версия операционной системы, журналы операционной системы (журналы событий Windows), выполняющиеся процессы, имя компьютера, IP-адреса и имя вошедшего пользователя.

Сбор данных необходим, чтобы обнаруживать недостающие обновления и неправильно настроенные параметры безопасности ОС, узнавать, включена ли защита конечных точек, а также определять работоспособность и обнаруживать угрозы. Сбор данных необходим только для ресурсов вычислительной среды (виртуальных машин, масштабируемых наборов виртуальных машин, контейнеров IaaS и компьютеров, не относящихся к Azure). Вы можете пользоваться центром безопасности Azure даже без подготовки агентов. Однако защита будет ограничена, а перечисленные выше возможности не будут поддерживаться.  

В этой статье описывается, как установить агент Log Analytics и настроить рабочую область Log Analytics для хранения собранных данных. Чтобы включить сбор данных, необходимо выполнить обе операции. За хранение данных в Log Analytics, будь то новая или существующая рабочая область, может взиматься дополнительная плата. Дополнительные сведения см. на [странице с расценками](https://azure.microsoft.com/pricing/details/security-center/).

> [!TIP]
> Список поддерживаемых платформ представлен в статье [Поддерживаемые платформы в центре безопасности Azure](security-center-os-coverage.md).

## <a name="enable-automatic-provisioning-of-the-log-analytics-agent"></a>Включение автоматической подготовки агента Log Analytics <a name="auto-provision-mma"></a>

> [!NOTE]
> Пользователям Azure Sentinel нужно обратить внимание, что сбор данных о событиях безопасности в контексте одной рабочей области следует настраивать только в одном месте: либо в Центре безопасности Azure, либо в Azure Sentinel. Если вы планируете добавить Azure Sentinel в рабочую область, которая уже получает оповещения Azure Defender из Центра безопасности Azure и настроена на сбор данных о событиях безопасности, у вас есть два варианта:
> - Не меняйте способ сбора данных о событиях безопасности в Центре безопасности Azure. Так вы сможете запрашивать и анализировать эти события как в Azure Sentinel, так и в Azure Defender. Однако вы не сможете отслеживать состояние подключения соединителя или изменять его конфигурацию в Azure Sentinel. Если это важно для вас, рассмотрите второй вариант.
>
> - [Отключите сбор данных о событиях безопасности](#data-collection-tier) в Центре безопасности Azure и только после этого добавьте соединитель событий безопасности в Azure Sentinel. Как и в первом варианте, вы сможете запрашивать и анализировать события как в Azure Sentinel, так и в Azure Defender или Центре безопасности Azure, а также отслеживать состояние подключения соединителя или изменять его конфигурацию в Azure Sentinel.


Для сбора данных с компьютеров должен быть установлен агент Log Analytics. Установить агент можно автоматически (рекомендуется) или вручную. Автоматическая подготовка по умолчанию отключена.

Если автоматическая подготовка включена, Центр безопасности развертывает агент Log Analytics на всех поддерживаемых существующих и новых виртуальных машинах Azure. Рекомендуется автоматическая подготовка, но при необходимости можно установить агент вручную (см. раздел [Установка агента Log Analytics вручную](#manual-agent)).

Благодаря развернутому на компьютерах агенту Центр безопасности может предоставлять дополнительные рекомендации, связанные с состоянием обновления системы, конфигурациями безопасности ОС и защитой конечных точек, а также создавать дополнительные оповещения системы безопасности.

Чтобы включить автоматическую подготовку агента Log Analytics, сделайте следующее:

1. В главном меню Центра безопасности выберите **Цены и параметры**.
1. Выберите соответствующую подписку.
1. На странице **Сбор данных** выберите для параметра **Автоматическая подготовка** значение **Вкл.**
1. Нажмите **Сохранить**.

    :::image type="content" source="./media/security-center-enable-data-collection/enable-automatic-provisioning.png" alt-text="Включение автоматической подготовки агента Log Analytics":::

    >[!TIP]
    > Если необходимо подготовить рабочую область, установка агента может занять до 25 минут.


## <a name="workspace-configuration"></a>Конфигурация рабочей области
Данные, собранные центром безопасности, хранятся в рабочих областях Log Analytics. Вы можете настроить сбор данных с виртуальных машин Azure в рабочих областях, созданных центром безопасности, или в имеющихся созданных рабочих областях. 

Конфигурация рабочей области задается для каждой подписки отдельно, а несколько подписок могут использовать одну и ту же рабочую область.

### <a name="using-a-workspace-created-by-security-center"></a>Использование рабочей области, созданной центром безопасности

Центр безопасности может автоматически создать стандартную рабочую область для хранения данных. 

Чтобы выбрать рабочую область, созданную центром безопасности:

1. В разделе **Конфигурация рабочей области по умолчанию** выберите параметр "Использовать рабочие области, созданные центром безопасности".
    :::image type="content" source="./media/security-center-enable-data-collection/workspace-selection.png" alt-text="Включение автоматической подготовки агента Log Analytics"::: 

1. Выберите команду **Сохранить**.<br>
    Центр безопасности создает группу ресурсов и рабочую область по умолчанию в этом географическом расположении, а затем подключает агент к этой рабочей области. Соглашение об именовании для рабочей области и группы ресурсов:<br>
   **Рабочая область: DefaultWorkspace-[ИД_подписки]-[географическое_расположение]<br>. Группа ресурсов: DefaultResourceGroup-[георасположение]** .

   Если подписка содержит виртуальные машины из нескольких географических расположений, центр безопасности создает несколько рабочих областей. Они создаются для соблюдения правил конфиденциальности данных.
1. Центр безопасности автоматически включит решение центра безопасности в рабочей области согласно ценовой категории, заданной для подписки. 

> [!NOTE]
> Ценовая категория Log Analytics для рабочих областей, созданных центром безопасности, не влияет на выставление счетов за его использование. Счета за использование центра безопасности всегда выставляются в зависимости от его политики безопасности и решений, установленных в рабочей области. Для подписок без доступа к Azure Defender Центр безопасности включает решение *SecurityCenterFree* в рабочей области по умолчанию. Для подписок с доступом к Azure Defender Центр безопасности включает решение *Security* в рабочей области по умолчанию.
> Сохранение данных в Log Analytics может привести к дополнительным расходам на хранение данных. Дополнительные сведения см. на [странице с расценками](https://azure.microsoft.com/pricing/details/security-center/).

Дополнительные сведения об имеющихся учетных записях Log Analytics см. в разделе [Существующие клиенты Log Analytics](./faq-azure-monitor-logs.md).

### <a name="using-an-existing-workspace"></a>Использование имеющейся рабочей области

Если у вас уже есть рабочая область Log Analytics, вы можете продолжать использовать ее.

Чтобы использовать имеющуюся рабочую область Log Analytics, вам потребуются разрешение на чтение и запись в этой рабочей области.

> [!NOTE]
> Решения, включенные в имеющейся рабочей области, будут применяться к подключенным к ним виртуальным машинам Azure. В случае платных решений за это может взиматься дополнительная плата. Из соображений конфиденциальности данных убедитесь, что выбранная рабочая область находится в нужном географическом регионе.
> Сохранение данных в Log Analytics может привести к дополнительным расходам на хранение данных. Дополнительные сведения см. на [странице с расценками](https://azure.microsoft.com/pricing/details/security-center/).

Чтобы выбрать имеющуюся рабочую область Log Analytics, сделайте следующее:

1. В разделе **Конфигурация рабочей области по умолчанию** выберите **Использовать другую рабочую область**.
    :::image type="content" source="./media/security-center-enable-data-collection/use-another-workspace.png" alt-text="Включение автоматической подготовки агента Log Analytics"::: 

2. В раскрывающемся меню выберите рабочую область для хранения собранных данных.

   > [!NOTE]
   > В этом раскрывающемся меню доступны все рабочие области во всех ваших подписках. Ознакомьтесь с разделом [Выбор рабочих областей между подписками](security-center-enable-data-collection.md#cross-subscription-workspace-selection), чтобы получить дополнительную информацию. Вам необходимо разрешение на доступ к рабочей области.
   >
   >

3. Щелкните **Сохранить**.
4. Если нажать **Сохранить**, появится запрос на изменение настроек отслеживаемых виртуальных машин, которые ранее были подключены к рабочей области по умолчанию.

   - Выберите **Нет**, если нужно, чтобы параметры новой рабочей области применялись только к новым виртуальным машинам. Параметры новой рабочей области применяются только к новым установленным агентам (на недавно обнаруженных виртуальных машинах, на которых не был установлен агент Log Analytics).
   - Выберите **Да**, если нужно, чтобы параметры новой рабочей области применялись ко всем виртуальным машинам. Кроме того, каждая виртуальная машина, подключенная к рабочей области, созданной центром безопасности, переподключается к новой целевой рабочей области.

   > [!NOTE]
   > Если вы выбрали "Да", не удаляйте рабочие области, созданные центром безопасности, пока все виртуальные машины не подключатся повторно к новой целевой рабочей области. Операция завершится сбоем, если удалить рабочую область слишком рано.
   >
   >

   - Чтобы отменить операцию, нажмите **Отмена**.

     ![Запрос на изменение настроек отслеживаемых виртуальных машин][3]

5. Укажите, будет ли в рабочей области включена служба Azure Defender.

    Чтобы использовать имеющуюся рабочую область, задайте для нее ценовую категорию. При этом в рабочей области будет установлено решение центра безопасности, если его еще нет.

    1. В главном меню Центра безопасности выберите **Цены и параметры**.
     
    1. Выберите рабочую область, к которой будет подключен агент.

    1. Выберите **Azure Defender on** (Включить Azure Defender) или **Azure Defender off** (Отключить Azure Defender).

   
   >[!NOTE]
   >Если в рабочей области уже включено решение **Security** или **SecurityCenterFree**, ценовая категория будет задана автоматически. 


## <a name="cross-subscription-workspace-selection"></a>Выбор рабочих областей между подписками
При выборе рабочей области для хранения данных доступны все рабочие области во всех ваших подписках. Выбор рабочих областей между подписками позволяет собирать данные из виртуальных машин, работающих в разных подписках, и сохранять их в выбранной рабочей области. Это полезно, если в организации используется централизованная рабочая область, которую требуется использовать для сбора данных о безопасности. Дополнительные сведения о том, как управлять рабочими областями, см. в статье [Управление рабочими областями](https://docs.microsoft.com/azure/log-analytics/log-analytics-manage-access).



## <a name="data-collection-tier"></a>Уровень сбора данных
Выбор уровня сбора данных в центре безопасности Azure повлияет только на хранение событий безопасности в рабочей области Log Analytics. Агент Log Analytics по-прежнему будет собирать и анализировать события безопасности, необходимые для защиты от угроз в Центре безопасности Azure, независимо от того, какой уровень событий безопасности вы хотите сохранить в рабочей области Log Analytics (при необходимости). Выбор хранения событий безопасности в рабочей области позволит проводить расследование, поиск и аудит этих событий в вашей рабочей области. 
> [!NOTE]
> Сохранение данных в Log Analytics может привести к дополнительным расходам на хранение данных. Дополнительные сведения см. на [странице с расценками](https://azure.microsoft.com/pricing/details/security-center/).

Вы можете выбрать необходимую политику фильтрации для подписок и рабочих областей из четырех наборов событий, которые хранятся в рабочей области. 
- **Отсутствует.** Отключение сбора событий безопасности. Это параметр по умолчанию.
- **Минимальный.** Меньший набор событий для клиентов, которые хотят сократить количество событий.
- **Общий.** Этот набор событий удовлетворяет потребности большинства клиентов и позволяет просмотреть весь журнал аудита.
- **Все события.** Для клиентов, которые хотят убедиться, что все события сохранены.

Эти наборы событий безопасности доступны только при работе с Azure Defender. Дополнительные сведения о ценовых категориях центра безопасности см. на странице [Цены](security-center-pricing.md).

Эти наборы предназначены для стандартных сценариев. Выберите набор, который соответствует вашим требованиям, прежде чем внедрить его.

Для определения событий, которые будут принадлежать к наборам событий **Общий** и **Минимальный**, мы ознакомились с отзывами клиентов и отраслевыми стандартами, чтобы изучить нефильтрованную частоту каждого события и их использования. В процессе мы использовали приведенные ниже рекомендации.

- **Минимальный.** Убедитесь, что этот набор содержит только события, которые могут указывать на брешь в системе безопасности и важные события небольших объемов. Например, этот набор содержит относительно большой объем сведений об успешных и неудачных попытках входа пользователя (идентификаторы события 4624 и 4625), однако не содержит сведений о выходе, что очень важно для аудита и не важно для обнаружения. Большую часть данных этого набора занимают события входа и создания процессов (идентификатор событий 4688).
- **Общий.** В этом наборе предоставляется полный журнал аудита пользователя. Например, этот набор содержит данные о входе и выходе пользователя (идентификатор события 4634). Мы добавили действия аудита, такие как изменение группы безопасности, операции основного контроллера домена Kerberos и другие события, рекомендованные отраслевыми организациями.

События небольших объемов были включены в набор "Общий". Основной причиной для его выбора среди всех событий является возможность уменьшить объем и оставить определенные события нефильтрованными.

В результате AppLocker будет генерировать события, которые затем собираются и используются центром безопасности. Важно отметить, что эта политика не настраивается на компьютерах, где уже настроена политика AppLocker. 
> - Чтобы собирать данные о [событиях с кодом 5156](https://www.ultimatewindowssecurity.com/securitylog/encyclopedia/event.aspx?eventID=5156) на платформе фильтрации Windows, необходимо включить [Аудит подключения платформы фильтрации](https://docs.microsoft.com/windows/security/threat-protection/auditing/audit-filtering-platform-connection) (Auditpol /set /subcategory:"Filtering Platform Connection" /Success:Enable)
>

Чтобы выбрать политику фильтрации, сделайте следующее:
1. На странице **Сбор данных** выберите политику фильтрации в разделе **Хранение дополнительных необработанных данных — События безопасности Windows**.
 
1. Щелкните **Сохранить**.
    :::image type="content" source="./media/security-center-enable-data-collection/data-collection-tiers.png" alt-text="Включение автоматической подготовки агента Log Analytics":::

### <a name="automatic-provisioning-in-cases-of-a-pre-existing-agent-installation"></a>Автоматическая подготовка уже существующей установки агента <a name="preexisting"></a> 

В описанных ниже вариантах использования указано, как автоматическая подготовка работает в тех случаях, когда агент или расширение уже установлены. 

- Агент Log Analytics непосредственно установлен на компьютере, но не в качестве расширения (прямой агент)<br>
Если агент Log Analytics установлен непосредственно на виртуальной машине (не в качестве расширения Azure), Центр безопасности установит расширение агента Log Analytics и может обновить агент Log Analytics до последней версии.
Установленный агент продолжит передавать данные в уже настроенные рабочие области и в дополнение будет передавать их в рабочую область, настроенную в Центре безопасности (на компьютерах Windows поддерживается одновременная работа с несколькими поставщиками услуг).
Если настроенная рабочая область является пользовательской (не рабочей областью Центра безопасности по умолчанию), вам потребуется установить в ней решение "security" или "securityFree" для Центра безопасности, чтобы начать обработку событий с виртуальных машин и компьютеров, передающих данные в эту рабочую область.<br>
<br>
Для компьютеров Linux одновременная работа с несколькими поставщиками услуг еще не поддерживается, поэтому при обнаружении существующей установки агента автоматическая подготовка не выполняется, и конфигурация компьютера не будет изменена.
<br>
Для существующих компьютеров в подписках, подключенных к Центру безопасности до 17 марта 2019 года, при обнаружении существующего агента расширение агента Log Analytics не устанавливается и конфигурация компьютера не изменяется. Рекомендации по устранению проблем установки агента на таких компьютерах см. в разделе "Устранение проблем работоспособности агента мониторинга на ваших компьютерах".

  
- Агент System Center Operations Manager уже установлен на компьютере<br>
Центр безопасности установит расширение агента Log Analytics параллельно с существующим агентом Operations Manager. Существующий агент Operations Manager продолжит отправлять отчеты на сервер Operations Manager, как обычно. Агент Operations Manager и агент Log Analytics используют общие библиотеки времени выполнения, которые во время этого процесса будут обновлены до последней версии. Если установлен агент Operations Manager версии 2012, **не** включайте автоматическую подготовку.<br>

- Расширение виртуальной машины уже установлено<br>
    - Если агент мониторинга установлен в качестве расширения, конфигурация расширения позволяет создавать отчеты только для одной рабочей области. Центр безопасности не переопределяет установленные подключения к рабочим областям пользователя. Центр безопасности будет хранить данные безопасности с виртуальной машины в уже подключенной рабочей области при условии, что в ней установлено решение "security" или "securityFree". В рамках этого процесса Центр безопасности может обновлять расширение до последней версии.  
    - Чтобы узнать, в какую рабочую область отправляет данные имеющееся расширение, выполните тест для [проверки соединения с центром безопасности Azure](https://blogs.technet.microsoft.com/yuridiogenes/2017/10/13/validating-connectivity-with-azure-security-center/). Вы также можете открыть рабочие области Log Analytics, выбрать рабочую область и виртуальную машину, а затем проверить подключение агента Log Analytics. 
    - Если в вашей среде на клиентских рабочих станциях установлен агент Log Analytics, отправляющий отчеты в имеющуюся рабочую область Log Analytics, просмотрите список [операционных систем, поддерживаемых Центром безопасности Azure](security-center-os-coverage.md), чтобы убедиться, что ваша операционная система поддерживается. Дополнительные сведения см. в статье [Существующие клиенты Log Analytics](./faq-azure-monitor-logs.md).
 
### <a name="turn-off-automatic-provisioning"></a>Отключение автоматической подготовки <a name="offprovisioning"></a>
Чтобы отключить автоматическую подготовку агента Log Analytics, сделайте следующее:

1. В меню центра безопасности на портале выберите **Цены и параметры**.
2. Выберите соответствующую подписку.

    :::image type="content" source="./media/security-center-enable-data-collection/select-subscription.png" alt-text="Включение автоматической подготовки агента Log Analytics":::

3. Выберите **Сбор данных**.
4. В разделе **Автоматическая подготовка** выберите элемент **Отключить**, чтобы отключить автоматическую подготовку.
5. Щелкните **Сохранить**. 


Если автоматическая подготовка отключена, раздел конфигурации рабочей области по умолчанию не отображается.

Если вы отключите автоматическую подготовку после того, как она была включена, на новых виртуальных машинах агенты не будут подготавливаться.

 
> [!NOTE]
>  При отключении автоматической подготовки агент Log Analytics не будет удален с виртуальных машин Azure, на которых он был подготовлен. Сведения о том, как удалить расширение OMS, см. в разделе [Как удалить расширения OMS, установленные центром безопасности](faq-data-collection-agents.md#remove-oms).
>
    
## <a name="manual-agent-provisioning"></a>Подготовка агента вручную <a name="manual-agent"></a>
 
Установить агент Log Analytics вручную можно несколькими способами. При установке вручную обязательно отключите автоматическую подготовку.

### <a name="operations-management-suite-vm-extension-deployment"></a>Развертывание расширения виртуальной машины Operations Management Suite 

Вручную установите агент Log Analytics, чтобы Центр безопасности мог собирать данные безопасности с ваших виртуальных машин, а также предоставлять рекомендации и предупреждения.

1. Отключите автоматическую подготовку.

1. При необходимости создайте рабочую область.

1. Включите Azure Defender в рабочей области, в которой устанавливаете агент Log Analytics:

    1. В главном меню Центра безопасности выберите **Цены и параметры**.

    1. Укажите рабочую область, в которой устанавливаете агент. Убедитесь, что рабочая область находится в той же подписке, которая используется в центре безопасности, а у вас есть разрешения на чтение и запись в этой рабочей области.

    1. Включите Azure Defender и нажмите кнопку **Сохранить**.

       >[!NOTE]
       >Если в рабочей области уже включено решение **Security** или **SecurityCenterFree**, ценовая категория будет задана автоматически. 

1. Если вы хотите развернуть агенты на новых виртуальных машинах с помощью шаблона Resource Manager, установите агент Log Analytics:

   - см. сведения об [установке агента Log Analytics для Windows](../virtual-machines/extensions/oms-windows.md);
   - см. сведения об [установке агента Log Analytics для Linux](../virtual-machines/extensions/oms-linux.md).

1. Чтобы развернуть расширения на имеющихся виртуальных машинах, следуйте инструкциям из статьи [Сбор данных о виртуальных машинах Azure](../azure-monitor/learn/quick-collect-azurevm.md).

   > [!NOTE]
   > Действия из раздела **Сбор данных о событиях и производительности** выполнять необязательно.
   >

1. Чтобы развернуть расширение с помощью PowerShell, следуйте инструкциям из документации по виртуальным машинам:

    - [Для компьютеров Windows](https://docs.microsoft.com/azure/virtual-machines/extensions/oms-windows?toc=%2Fazure%2Fazure-monitor%2Ftoc.json#powershell-deployment)
    - [Для компьютеров Linux](https://docs.microsoft.com/azure/virtual-machines/extensions/oms-linux?toc=%2Fazure%2Fazure-monitor%2Ftoc.json#azure-cli-deployment)



> [!NOTE]
> Инструкции о том, как подключиться к центру безопасности с помощью PowerShell, см. в разделе [Automate onboarding of Azure Security Center using PowerShell](security-center-powershell-onboarding.md) (Автоматизация подключения центра безопасности Azure с помощью PowerShell).

## <a name="troubleshooting"></a>Устранение неполадок

-   Руководство по устранению проблем с автоматической подготовкой см. в разделе [Проблемы с работоспособностью агента мониторинга](security-center-troubleshooting-guide.md#mon-agent).

-  Требования к сети см. в разделе [Устранение неполадок, связанных с требованиями к сети для агента мониторинга](security-center-troubleshooting-guide.md#mon-network-req).
-   Руководство по устранению проблем с подключением вручную см. в статье [Устранение неполадок подключения к Operations Management Suite](https://support.microsoft.com/help/3126513/how-to-troubleshoot-operations-management-suite-onboarding-issues).

- Поиск проблем с неотслеживаемыми виртуальными машинами:

    Виртуальная машина или компьютер не отслеживается Центром безопасности, если на них не запущено расширение агента Log Analytics. На виртуальной машине или компьютере может быть установлен локальный агент, например прямой агент OMS или агент System Center Operations Manager. Виртуальные машины или компьютеры с этими агентами определены как неотслеживаемые, так как эти агенты не полностью поддерживаются в центре безопасности. Чтобы использовать все преимущества центра безопасности, требуется расширение агента Log Analytics.

    Дополнительные сведения о том, почему Центру безопасности не удалось выполнить мониторинг виртуальных машин и компьютеров, инициализированных для автоматической подготовки, см. в разделе [Проблемы с работоспособностью агента мониторинга](security-center-troubleshooting-guide.md#mon-agent).
