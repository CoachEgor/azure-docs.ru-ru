# <a name="introduction-to-azure-defender-for-kubernetes"></a>Общие сведения о защитнике Azure для Kubernetes

Служба Azure Kubernetes Service (AKS) — это управляемая служба Майкрософт для разработки и развертывания контейнерных приложений и управления ими.

Центр безопасности Azure и AKS формируют лучшее облачное решение по обеспечению безопасности Kubernetes, а также обеспечивают усиление защиты, защиту рабочих нагрузок и защиту во время выполнения, как описано ниже.

Для обнаружения угроз для кластеров Kubernetes включите **защитник Azure для Kubernetes**.

Обнаружение угроз на уровне узла для узлов Linux AKS доступно при включении [защитника Azure для серверов](defender-for-servers-introduction.md).

## <a name="what-are-the-benefits-of-azure-defender-for-kubernetes"></a>Каковы преимущества защитника Azure для Kubernetes?

### <a name="run-time-protection"></a>Защита во время выполнения

С помощью непрерывного анализа следующих источников AKS центр безопасности обеспечивает защиту от угроз в режиме реального времени для контейнерных сред и создает предупреждения для угроз и вредоносных действий, обнаруженных на уровне узла *и* кластера AKS. Эти сведения можно использовать для быстрого устранения проблем безопасности и повышения безопасности контейнеров.

Центр безопасности обеспечивает защиту от угроз на разных уровнях. 

- **Уровень узла (предоставляется защитником Azure для серверов)** . с помощью того же агента log Analytics, который используется центром безопасности на других виртуальных машинах, защитник Azure отслеживает узлы Linux AKS для подозрительных действий, таких как обнаружение веб-оболочки и подключение с известными подозрительными IP-адресами. Агент также отслеживает аналитику, относящуюся к контейнеру, например создание привилегированного контейнера, подозрительный доступ к серверам API, а также серверы Secure Shell (SSH), работающие в контейнере DOCKER.

    Список предупреждений на уровне узла AKS см. в [справочной таблице предупреждений](https://docs.microsoft.com/azure/security-center/alerts-reference#alerts-containerhost).

    >[!IMPORTANT]
    > Если вы решили не устанавливать агенты на узлах, то получите только часть преимуществ защиты от угроз и лишь некоторые из оповещений системы безопасности. Вы будете по-прежнему получать оповещения, связанные с анализом сети и обменом данными с вредоносными серверами.

- **Уровень кластера AKS (предоставленный защитником Azure для Kubernetes)** . на уровне кластера защита от угроз основана на анализе журналов аудита Kubernetes. Чтобы включить этот **безагентный** мониторинг, включите Защитник Azure. Для создания предупреждений на этом уровне Центр безопасности отслеживает службы, которыми управляет служба AKS, с помощью полученных ею журналов. Примерами событий на этом уровне могут служить представление панелей мониторинга Kubernetes, создание ролей с высоким уровнем привилегий и создание конфиденциальных подключений.

    Список оповещений на уровне кластера AKS см. в [справочной таблице оповещений](https://docs.microsoft.com/azure/security-center/alerts-reference#alerts-akscluster).

    >[!NOTE]
    > Центр безопасности создает оповещения системы безопасности для действий и развертываний Службы Azure Kubernetes, происходящих после включения параметра Kubernetes в параметрах подписки. 

Следует также отметить, что наша международная команда специалистов по безопасности постоянно отслеживает ландшафт угроз. Они добавляют оповещение и уязвимости для конкретных контейнеров по мере их обнаружения.

> [!TIP]
> Вы можете имитировать оповещения для контейнеров, выполнив инструкции в [этой записи блога](https://techcommunity.microsoft.com/t5/azure-security-center/how-to-demonstrate-the-new-containers-features-in-azure-security/ba-p/1011270).



## <a name="how-does-security-centers-kubernetes-protection-work"></a>Как работает защита Kubernetes в центре безопасности?

Ниже приведена высокоуровневая схема взаимодействия между центром безопасности Azure, службой Kubernetes Azure и политикой Azure.

Вы видите, что следующие элементы, полученные и проанализированные центром безопасности, включают:

- журналы аудита с сервера API
- необработанные события безопасности от агента Log Analytics
- сведения о конфигурации кластера из кластера AKS
- Конфигурация рабочей нагрузки из политики Azure (с помощью **надстройки Azure Policy для Kubernetes**). [Дополнительные сведения о рекомендациях по защите рабочих нагрузок с помощью управления допуском Kubernetes](container-security.md#workload-protection-best-practices-using-kubernetes-admission-control)

## <a name="azure-defender-for-kubernetes---faq"></a>"Защитник Azure для Kubernetes" — часто задаваемые вопросы

### <a name="can-i-still-get-aks-protections-without-the-log-analytics-agent"></a>Можно ли по-прежнему получать защиту AKS без агента Log Analytics?

Как упоминалось выше, необязательный план **Azure Defender для Kubernetes** обеспечивает защиту на уровне кластера, агент log Analytics **защитника Azure для серверов** защищает узлы. 

Рекомендуется развернуть оба варианта, чтобы обеспечить максимально полную защиту.

Если вы решили не устанавливать агент на узлах, вы получите только подмножество преимуществ защиты от угроз и оповещений системы безопасности. Вы будете по-прежнему получать оповещения, связанные с анализом сети и обменом данными с вредоносными серверами.
