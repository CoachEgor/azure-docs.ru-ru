# <a name="whats-new-in-azure-security-center"></a>Новые возможности центра безопасности Azure

Безопасность Azure — это активная разработка, которая получает усовершенствования на постоянной основе. Чтобы оставаться в курсе последних разработок, на этой странице предоставляются следующие сведения:

- Новые функции
- Исправления ошибок
- Нерекомендуемые функции.

Эта страница регулярно обновляется, поэтому часто посещайте ее. Если вы ищете элементы старше шести месяцев, вы найдете их в [архиве для новых возможностей в центре безопасности Azure](release-notes-archive.md).


## <a name="may-2020"></a>Май 2020 г.

### <a name="changes-to-just-in-time-jit-virtual-machine-vm-access"></a>Изменения в доступе к виртуальной машине по требованию (JIT)

Центр безопасности включает дополнительный компонент для защиты портов управления виртуальных машин. Это обеспечивает защиту от наиболее распространенной формы атак методом подбора.

Это обновление предоставляет следующие изменения для этой функции:

- Рекомендация по включению JIT-компилятора на виртуальной машине была переименована. Ранее в виртуальных машинах следует применять управление JIT-доступом к сети. Теперь это "порты управления виртуальных машин должны быть защищены с помощью JIT-управления доступом к сети".

- Рекомендация была настроена для активации только при наличии открытых портов управления.

Дополнительные [сведения о функции JIT-доступа](security-center-just-in-time.md).


### <a name="custom-recommendations-have-been-moved-to-a-separate-security-control"></a>Настраиваемые рекомендации перемещены в отдельный элемент управления безопасностью

Одним из элементов управления безопасностью, появившихся с повышенным уровнем безопасности, был «реализация рекомендаций по обеспечению безопасности». Все пользовательские рекомендации, созданные для ваших подписок, автоматически помещаются в этот элемент управления. 

Чтобы упростить поиск пользовательских рекомендаций, мы перенесли их в выделенный элемент управления безопасностью "настраиваемые рекомендации". Этот элемент управления не влияет на оценку безопасности.

Дополнительные сведения об элементах управления безопасностью см. [в статье повышенная безопасность (Предварительная версия) в центре безопасности Azure](secure-score-security-controls.md).


### <a name="toggle-added-to-view-recommendations-in-controls-or-as-a-flat-list"></a>Переключатель добавлен для просмотра рекомендаций в элементах управления или в виде плоского списка

Элементы управления безопасностью — это логические группы связанных рекомендаций по безопасности. Они соответствуют уязвимым областям атак. Элемент управления — это набор рекомендаций по обеспечению безопасности с инструкциями, помогающими реализовать эти рекомендации.

Чтобы немедленно увидеть, насколько хорошо ваша организация обеспечивает безопасность каждой отдельной уязвимой зоны, просмотрите баллы за каждый контроль безопасности.

По умолчанию рекомендации отображаются в элементах управления безопасностью. В этом обновлении можно также отобразить их в виде списка. Чтобы просмотреть их в виде простого списка, отсортированного по состоянию работоспособности затронутых ресурсов, используйте новый переключатель "Группировать по элементам управления". Переключатель находится над списком на портале.

Элементы управления безопасностью и этот переключатель являются частью нового процесса оценки безопасности. Не забудьте отправить нам свои отзывы с помощью портала.

Дополнительные сведения об элементах управления безопасностью см. [в статье повышенная безопасность (Предварительная версия) в центре безопасности Azure](secure-score-security-controls.md).


### <a name="account-security-recommendations-moved-to-security-best-practices-security-control"></a>Рекомендации по безопасности учетной записи, перемещенные в "контроль передовых рекомендаций по безопасности"

Одним из элементов управления безопасностью, появившихся с повышенным уровнем безопасности, является "рекомендации по обеспечению безопасности". Если рекомендация находится в этом элементе управления, она не влияет на оценку безопасности. 

В этом обновлении три рекомендации были перемещены из элементов управления, в которых они были изначально размещены, и в этот элемент управления практическими рекомендациями. Мы выделили этот шаг, так как мы определили, что риск этих трех рекомендаций ниже первоначально считался.

Ниже приведены рекомендации.

- MFA необходимо включить в учетных записях с разрешениями на чтение для вашей подписки (первоначально в элементе управления "включить MFA").
- Внешние учетные записи с разрешениями на чтение должны быть удалены из подписки (изначально в элементе управления "Управление доступом и разрешениями").
- Для подписки необходимо назначить не более 3 владельцев (первоначально в элементе управления "Управление доступом и разрешениями").

Дополнительные сведения об элементах управления безопасностью см. [в статье повышенная безопасность (Предварительная версия) в центре безопасности Azure](secure-score-security-controls.md).


### <a name="custom-policies-with-custom-metadata-generally-available"></a>Настраиваемые политики с настраиваемыми метаданными общедоступны

Пользовательские политики теперь являются частью рекомендаций центра безопасности, оценки безопасности и панели мониторинга стандартов соответствия нормативным требованиям. Эта функция теперь общедоступна и позволяет расширить покрытие оценки безопасности Организации в центре безопасности. 

Создайте пользовательскую инициативу в политике Azure, добавьте в нее политики и подключите ее к центру безопасности Azure и визуализируйте ее как рекомендации.

Теперь мы также добавили параметр для изменения метаданных пользовательской рекомендации. Параметры метаданных включают серьезность, шаги исправления, сведения об угрозах и многое другое.  

Дополнительные [сведения об улучшении пользовательских рекомендаций см. в этой статье](custom-security-policies.md#enhancing-your-custom-recommendations-with-detailed-information).


## <a name="april-2020"></a>Апрель 2020 г.

### <a name="dynamic-compliance-packages-now-generally-available"></a>Пакеты динамического соответствия теперь общедоступны

Панель мониторинга соответствия требованиям центра безопасности Azure теперь включает **пакеты динамического соответствия** (теперь общедоступные) для контроля дополнительных отраслевых и нормативных стандартов.

Пакеты динамического соответствия можно добавить в подписку или группу управления на странице Политика безопасности центра безопасности. При выставлении стандартного или тестового уровня Стандартный отображается на панели мониторинга соответствия нормативным требованиям со всеми связанными данными о соответствии, сопоставленными как оценки. Сводный отчет для всех встроенных стандартов будет доступен для загрузки.

Теперь можно добавить следующие стандарты:

- **NIST SP 800-53, ред. 4**
- **SWIFT CSP КСКФ-v2020**
- **Официальная и отделения UK**
- **Canada Federal PBMM**
- **Azure CIS 1.1.0 (новое)** (это более полное представление об Azure CIS 1.1.0)

Кроме того, мы недавно добавили **показатели безопасности Azure**, соответствующие требованиям Майкрософт рекомендации по обеспечению безопасности и соответствия требованиям, основанные на общих платформах соответствия. Дополнительные стандарты будут поддерживаться на панели мониторинга по мере их появления.  
 
[Узнайте больше о настройке набора стандартов на панели мониторинга соответствия нормативным требованиям](update-regulatory-compliance-packages.md).


### <a name="identity-recommendations-now-included-in-azure-security-center-free-tier"></a>Рекомендации по удостоверениям теперь входят в уровень "бесплатный" центра безопасности Azure

Рекомендации по безопасности для удостоверений и доступа на уровне "бесплатный" центра безопасности Azure теперь общедоступны. Это является частью усилий по обеспечению доступности функций управления безопасностью в облаке (КСПМ). До настоящего момента эти рекомендации были доступны только в ценовой категории "Стандартный".

Примеры рекомендаций по удостоверениям и доступу:

- "Многофакторная проверка подлинности должна быть включена для учетных записей с правами владельца в вашей подписке".
- "Для вашей подписки должно быть назначено не более трех владельцев".
- "Устаревшие учетные записи должны быть удалены из подписки".

Если у вас есть подписки на ценовую категорию "бесплатный", это изменение повлияет на свои оценки безопасности, так как они никогда не были оценены для обеспечения безопасности идентификации и доступа.

[Узнайте больше о рекомендациях по идентификации и доступу](recommendations-reference.md#recs-identity).

### <a name="workflow-automation-is-now-generally-available"></a>Теперь Автоматизация рабочих процессов доступна

Кроме того, для оповещений доступны ручные триггеры и все рекомендации, для которых доступен параметр быстрое исправление.

Каждая программа безопасности включает несколько рабочих процессов для реагирования на инциденты. Эти процессы могут включать уведомление соответствующих заинтересованных лиц, запуск процесса управления изменениями и применение конкретных действий по исправлению. Автоматизация сокращает издержки и может повысить безопасность, обеспечивая быстрое и согласованное выполнение шагов процесса и в соответствии с предопределенными требованиями.

После перемещения сервера из центра администрирования Windows в центр безопасности Azure вы сможете:

- Просмотрите уровень безопасности и получите дополнительные подробные сведения об управляемых серверах центра администрирования Windows в центре безопасности в портал Azure (или через API-интерфейс).

### <a name="protection-for-azure-kubernetes-service"></a>Защита для службы Kubernetes Azure

Центр безопасности Azure размещает свои функции безопасности контейнеров для защиты службы Azure Kubernetes (AKS).

Популярная платформа с открытым кодом Kubernetes была принята так, что теперь она является отраслевым стандартом для оркестрации контейнеров. Несмотря на эту обширную реализацию, по-прежнему остается знать о том, как защитить среду Kubernetes. Для защиты областей атаки контейнерного приложения требуется опыт обеспечения безопасной настройки инфраструктуры и постоянного отслеживания потенциальных угроз.

Защита центра безопасности включает в себя:

- **Обнаружение и видимость** — непрерывное обнаружение УПРАВЛЯЕМЫХ экземпляров AKS в подписках, зарегистрированных в центре безопасности.
- **Рекомендации по безопасности** — практические рекомендации, которые помогут обеспечить соответствие рекомендациям по обеспечению безопасности для AKS. Эти рекомендации включены в вашу оценку безопасности, чтобы убедиться, что они просматриваются в рамках вашей организации. Пример AKSной рекомендации, которую можно увидеть, — это "Управление доступом на основе ролей" следует использовать для ограничения доступа к кластеру службы Kubernetes.
- **Защита от угроз** . Благодаря непрерывному анализу развертывания AKS центр безопасности предупреждает об угрозах и вредоносных действиях, обнаруженных на уровне узла и кластера AKS.

Дополнительные [сведения об интеграции Azure Kubernetes Services с центром безопасности](azure-kubernetes-service-integration.md).
Дополнительные [сведения о функциях безопасности контейнеров в центре безопасности](container-security.md).


### <a name="improved-just-in-time-experience"></a>Улучшенная JIT-работа

Функции, операции и пользовательский интерфейс для JIT-средств центра безопасности Azure, которые защищают порты управления, были усовершенствованы следующим образом: 

- **Поле обоснования** — при запросе доступа к виртуальной машине с помощью JIT-страницы портал Azure для ввода обоснования запроса доступно новое необязательное поле. Информация, вводимых в это поле, может быть записана в журнал действий. 
- **Автоматическая очистка избыточных правил** JIT. при каждом обновлении политики JIT автоматически запускается средство очистки для проверки допустимости всего набора правил. Средство ищет несоответствия между правилами в политике и правилами в NSG. Если средство очистки обнаруживает несовпадение, оно определяет причину и, когда это можно сделать, удаляет встроенные правила, которые больше не нужны. Средство удаления никогда не удаляет созданные вами правила. 

Дополнительные [сведения о функции JIT-доступа](security-center-just-in-time.md).


### <a name="two-security-recommendations-for-web-applications-deprecated"></a>Две рекомендации по безопасности для веб-приложений устарели

Две рекомендации по безопасности, связанные с веб-приложениями, являются устаревшими: 

- Правила для веб-приложений в IaaS группы безопасности сети должны быть зафиксированы.
    (Связанная политика. Правила группы безопасности сети для веб-приложений в IaaS должны быть зафиксированы)

- Доступ к службам приложений должен быть ограничен.
    (Связанная политика: доступ к службам приложений должен быть ограничен [Предварительная версия])

Эти рекомендации больше не будут отображаться в списке рекомендаций центра безопасности. Связанные политики больше не будут включаться в инициативу с именем "Центр безопасности по умолчанию".

## <a name="february-2020"></a>Февраль 2020 г.

### <a name="fileless-attack-detection-for-linux-is-now-in-preview"></a>Обнаружение атак с несвязанными файлами для Linux теперь находится на этапе предварительной версии

Поскольку злоумышленники увеличивают использование методов стеалсиер во избежание обнаружения, центр безопасности Azure расширяет возможности обнаружения атак без файлов для Linux в дополнение к Windows. Атаки, не имеющие файлов, используют уязвимости программного обеспечения, вставляют вредоносные полезные данные в опасные системные процессы и скрывают память. Эти методы:

- свести или исключить трассировку вредоносных программ на диске
- значительно сокращается вероятность обнаружения с помощью решений для поиска вредоносных программ на диске.

Чтобы просчитать эту угрозу, центр безопасности Azure выпустил Обнаружение атак с незащищенным файлом для Windows в октябре 2018 и теперь также имеет расширенное Обнаружение атак с незащищенными файлами в Linux.
