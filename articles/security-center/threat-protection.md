# <a name="threat-protection-in-azure-security-center"></a>Защита от угроз в Центре безопасности Azure

Когда Центр безопасности обнаруживает угрозу в любой области среды, он генерирует оповещение. Эти предупреждения описывают детали затронутых ресурсов, предлагаемые шаги исправления, а в некоторых случаях возможность запуска приложения логики в ответ.

Защита от угроз Центра безопасности Azure обеспечивает комплексную защиту среды:

* **Защита от угроз для вычислительных ресурсов Azure:** компьютеры Windows, Linux-машины, служба приложений Azure и контейнеры Azure

* **Защита от угроз для ресурсов данных Azure:** База данных S'L и Хранилище данных, Хранилище данных Azure и DB Azure Cosmos

* **Защита от угроз для уровней службы Azure:** уровень сети Azure, уровень управления Azure (менеджер ресурсов Azure) (Preview) и Хранилище ключей Azure (Preview)

Независимо от того, генерируется ли оповещение Центром безопасности или получено Центром безопасности из другого продукта безопасности, его можно экспортировать. Для экспорта оповещений в Azure Sentinel (или сторонний SIEM) или любой другой внешний инструмент следуйте инструкциям в [Exporting alerts к SIEM.](continuous-export.md) 




## <a name="threat-protection-for-windows-machines"></a>Защита от угроз для компьютеров Windows<a name="windows-machines"></a>

Центр безопасности Azure интегрируется с службами Azure для мониторинга и защиты компьютеров на базе Windows. Центр безопасности представляет оповещения и предложения по исправлению всех этих служб в простом в использовании формате.

* **Microsoft Defender ATP** <a name="windows-atp"></a> - Центр безопасности расширяет свои облачные платформы защиты рабочей нагрузки, интегрируясь с Microsoft Defender Advanced Threat Protection (ATP). Вместе они обеспечивают комплексное обнаружение конечных точек и реагирование (EDR).

    > [!IMPORTANT]
    > Датчик Microsoft Defender ATP автоматически включен на серверах Windows, которые используют Центр безопасности.

    Когда Microsoft Defender ATP обнаруживает угрозу, он запускает оповещение. Предупреждение отображается на панели мониторинга Центра безопасности. С панели мониторинга можно перейти на консоль Microsoft Defender ATP и провести детальное исследование, чтобы раскрыть масштабы атаки. Для получения дополнительной информации о Microsoft Defender ATP, см [бортовых серверов в Microsoft Defender ATP службы](https://docs.microsoft.com/windows/security/threat-protection/microsoft-defender-atp/configure-server-endpoints).

* **Анализ** <a name="windows-dump"></a> аварийной свалки - При сбое программного обеспечения аварийный свал захватывает часть памяти во время сбоя.

    Сбой мог быть вызван вредоносными программами или содержать вредоносные программы. Чтобы избежать обнаружения продуктами безопасности, различные формы вредоносных программ используют безфайлую атаку, которая позволяет избежать записи на диск или шифрование компонентов программного обеспечения, записанных на диск. Этот тип атаки трудно обнаружить с помощью традиционных подходов на основе диска.

    Однако, используя анализ памяти, вы можете обнаружить такого рода атаки. Анализируя память в аварийной свалке, Центр безопасности может обнаружить методы атаки. Например, атака может быть попытка использовать уязвимости в программном обеспечении, доступ к конфиденциальным данным, и тайно сохраняются в скомпрометированной машине. Центр безопасности выполняет эту работу с минимальным воздействием на производительность узлов.

    Подробную информацию об аналитических сигналах анализа аварийных дампов можно узнать в [справочной таблице предупреждений.](alerts-reference.md#alerts-windows)

* **Fileless обнаружения** <a name="windows-fileless"></a> атак - Fileless атак, направленных на конечные точки являются общими. Чтобы избежать обнаружения, атаки без файлов вводят в память вредоносные полезные нагрузки. Полезные нагрузки злоумышленников сохраняются в памяти скомпрометированных процессов и выполняют широкий спектр вредоносных действий.

    При обнаружении безфайловых атак автоматизированные методы судебно-медицинской экспертизы памяти определяют наборы инструментов, методы и поведение без файла. Это решение периодически сканирует вашу машину во время выполнения и извлекает информацию непосредственно из памяти критически важных процессов безопасности.

    Он находит доказательства эксплуатации, впрыска кода и выполнения вредоносных полезных нагрузок. Обнаружение fileless атаки генерирует подробные оповещения безопасности для ускорения сортировки предупреждений, корреляции и времени отклика вниз по течению. Этот подход дополняет решения EDR на основе событий, обеспечивая более широкий охват обнаружения.

    Для получения подробной информации о предупреждениях об обнаружении файлов без файла смотрите [справочную таблицу предупреждений.](alerts-reference.md#alerts-windows)

> Вы можете имитировать оповещения Windows, загрузив [Playbook Центра безопасности Azure: Security Alerts.](https://gallery.technet.microsoft.com/Azure-Security-Center-f621a046)

Он фильтрует их определенным набором правил и записывает сообщения для них в гнездо. Центр безопасности интегрирует функции из проверенного пакета в агент по мониторингу Майкрософт (MMA). Эта интеграция позволяет собирать проверенные события во всех поддерживаемых дистрибутивах Linux без каких-либо предпосылок.  

    проверенные записи собираются, обогащаются и агрегируются в события с помощью агента Linux MMA. ## <a name="threat-protection-for-azure-app-service"></a>Защита от угроз для службы приложений Azure<a name="app-services"></a>

> [!NOTE]
> Эта услуга в настоящее время недоступна в правительственных и суверенных облачных регионах Azure.

Центр безопасности использует масштаб облака для выявления атак, нацеленных на приложения, работающие на Службу безопасности. Злоумышленники исследуют веб-приложения, чтобы найти и использовать слабые стороны. Перед тем, как быть направлены в определенные среды, запросы в приложения, работающие в Azure, проходят через несколько шлюзов, где они проверяются и регистрируются. Затем эти данные используются для идентификации эксплойтов и злоумышленников, а также для изучения новых шаблонов, которые будут использованы позже.

Используя видимость, которую Azure имеет в качестве поставщика облачных услуг, Центр безопасности анализирует внутренние журналы службы безопасности для определения методологии атак на нескольких целевых объектах. Например, методология включает широкое сканирование и распределенные атаки. Этот тип атаки обычно происходит от небольшого подмножества ИП и показывает шаблоны сканирования в похожие конечные точки на нескольких узлах. Атаки ищут уязвимую страницу или плагин, и не могут быть идентифицированы с точки зрения одного узла.

Если вы работаете с планом службы приложений на базе Windows, Центр безопасности также имеет доступ к основным песочницам и вдовым выемке. Вместе с данными журнала, упомянутыми выше, инфраструктура может рассказать историю, от новой атаки, циркулирующей в дикой природе, до компромиссов в машинах клиентов. Таким образом, даже если Центр безопасности развернут после использования веб-приложения, он может обнаружить текущие атаки.

Для получения дополнительной информации [App Service plans](https://azure.microsoft.com/pricing/details/app-service/plans/)о планах Службы Приложений см.





## <a name="threat-protection-for-azure-containers"></a>Защита от угроз для контейнеров Azure<a name="azure-containers"></a>

> [!NOTE]
> Эта услуга в настоящее время недоступна в правительственных и суверенных облачных регионах Azure.

Центр безопасности обеспечивает защиту угроз в режиме реального времени для контейнерных сред и генерирует оповещения о подозрительных действиях. Эти сведения можно использовать для быстрого устранения проблем безопасности и повышения безопасности контейнеров.

Центр безопасности обеспечивает защиту от угроз на различных уровнях: 

* **Уровень хоста** - агент Центра безопасности (доступен на стандартном уровне, см. [цены](security-center-pricing.md) на детали) отслеживает Linux для подозрительных действий. Агент вызывает оповещения о подозрительных действиях, происходящих из узла или контейнера, работающего на нем. Примерами таких действий являются обнаружение веб-оболочки и связь с известными подозрительными IP-адресами.

    Для более глубокого понимания безопасности вашей контейнерной среды агент отслеживает аналитику, специфичную для контейнеров. Он будет вызывать оповещения о таких событиях, как создание привилегированных контейнеров, подозрительный доступ к серверам API и серверы Secure Shell (SSH), работающие внутри контейнера Docker.

    >[!IMPORTANT]
    > Если вы решите не устанавливать агенты на узлы, вы получите только подмножество преимуществ защиты от угроз и оповещений о безопасности. Вы по-прежнему будете получать оповещения, связанные с анализом сети и коммуникациями с вредоносными серверами.

    Для списка предупреждений уровня хоста [Reference table of alerts](alerts-reference.md#alerts-containerhost)см.


* На **уровне кластера AKS**защита угроз основана на анализе журналов аудита Kubernetes. Чтобы включить этот мониторинг **без агента,** добавьте опцию Kubernetes в подписку со **страницы настройки & ценообразования** (см. [цены).](security-center-pricing.md) Для создания оповещений на этом уровне Центр безопасности отслеживает службы, управляемые AKS, используя журналы, извлеченные с помощью AKS. Примеры событий на этом уровне включают открытые панели мониторинга Kubernetes, создание высоких привилегированных ролей и создание чувствительных креплений.

    >[!NOTE]
    > Центр безопасности генерирует оповещения о безопасности для действий и развертываний службы Azure Kubernetes после включения опции Kubernetes в настройках подписки. 

    Для списка предупреждений уровня кластера AKS [см.](alerts-reference.md#alerts-akscluster)

Кроме того, наша глобальная команда исследователей безопасности постоянно следит за ландшафтом угроз. Они добавляют контейнерные оповещения и уязвимости по мере их обнаружения.

## <a name="threat-protection-for-azure-network-layer"></a>Защита от угроз для сетевого слоя Azure<a name="network-layer"></a>

Аналитика сетевого уровня Центра безопасности основана на выборочных [данных IPFIX,](https://en.wikipedia.org/wiki/IP_Flow_Information_Export)которые представляют собой заголовки пакетов, собранные основными маршрутизаторами Azure. Основываясь на этой ленте данных, Центр безопасности использует модели машинного обучения для выявления и выявления вредоносных действий трафика. Центр безопасности также использует базу данных Microsoft Threat Intelligence для обогащения IP-адресов.

Некоторые сетевые конфигурации могут ограничивать Центр безопасности в генерации оповещений о подозрительной сетевой активности. Для создания сетевых оповещений ВЦентр безопасности убедитесь, что:

- Ваша виртуальная машина имеет общедоступный IP-адрес (или находится на балансе нагрузки с общедоступным IP-адресом).

- Трафик сети виртуальной машины не блокируется внешним решением IDS.

- Виртуальной машине был присвоен один и тот же IP-адрес в течение всего часа, в течение которого произошло подозрительное сообщение. Это также относится к вм: ВМ, созданные в рамках управляемой службы (например, AKS, Databricks).

Для списка предупреждений сетевого слоя Azure см. [таблицу оповещений с множественной информацией.](alerts-reference.md#alerts-azurenetlayer)

Подробную информацию о том, как Центр безопасности может использовать сетевые сигналы для защиты от угроз, смотрите [в Центре безопасности Heuristic DNS.](https://azure.microsoft.com/blog/heuristic-dns-detections-in-azure-security-center/)



## <a name="threat-protection-for-azure-management-layer-azure-resource-manager-preview"></a>Защита от угроз для слоя управления Azure (менеджер ресурсов Azure) (Preview)<a name ="management-layer"></a>

В настоящее время предварительный просмотр находится в защитном слое Центра безопасности, основанный на менеджере ресурсов Azure.

Центр безопасности предлагает дополнительный уровень защиты с помощью событий Azure Resource Manager, который считается плоскостью управления для Azure. Анализируя записи диспетчера ресурсов Azure, Центр безопасности обнаруживает необычные или потенциально опасные операции в среде подписки Azure.

Для списка предупреждений менеджера ресурсов Azure (Preview) см. [таблицу оповещений с множеством](alerts-reference.md#alerts-azureresourceman).



>[!NOTE]
> Некоторые из предыдущих аналитиков питаются от Microsoft Cloud App Security. Чтобы извлечь выгоду из этой аналитики, необходимо активировать лицензию Cloud App Security. Если у вас есть лицензия Cloud App Security, то эти оповещения включены по умолчанию. Чтобы отключить оповещения:
>
> 1. В лезвии **Центра безопасности** выберите **политику безопасности.** Для подписки, которая требуется изменить, выберите **настройки edit.**
> 2. Выберите **обнаружение угроз.**
> 3. При **интеграции Enable,** очистить **Разрешить Microsoft Cloud App Security для доступа к моим данным,** и выберите **Сохранить**.

>[!NOTE]
>Центр безопасности хранит данные клиентов, связанные с безопасностью, в той же гео, что и его ресурс. Если корпорация Майкрософт еще не развернула Центр безопасности в георесурсе ресурса, то она хранит данные в США. При включении Cloud App Security эта информация хранится в соответствии с правилами геолокации Cloud App Security. Для получения дополнительной [Data storage for non-regional services](https://azuredatacentermap.azurewebsites.net/)информации см.








## <a name="threat-protection-for-azure-key-vault-preview"></a>Защита от угроз для убежища ключей Azure (Предварительный просмотр)<a name="azure-keyvault"></a>

> [!NOTE]
> Эта услуга в настоящее время недоступна в правительственных и суверенных облачных регионах Azure.

Azure Key Vault — это облачная служба, которая обеспечивает защиту ключей шифрования и секретов (например, сертификатов, строк подключения и паролей). 

Центр безопасности Azure включает в себя наиболее расширенную защиту угроз Azure Key Vault, обеспечивающую дополнительный уровень интеллекта безопасности. Центр безопасности обнаруживает необычные и потенциально опасные попытки получить доступ к учетным записям Key Vault или использовать их. Этот уровень защиты позволяет устракиваться в угрозах, не будучи экспертом по безопасности и не управляя сторонними системами мониторинга безопасности.  

При возникновении аномальных действий Центр безопасности показывает оповещения и по желанию отправляет их по электронной почте администраторам подписки. Эти предупреждения включают подробную информацию о подозрительной деятельности и рекомендации о том, как расследовать и исправления угроз. 

Список предупреждений о убежище ключей Azure смотрите [справочную таблицу предупреждений.](alerts-reference.md#alerts-azurekv)





## <a name="threat-protection-for-other-microsoft-services"></a>Защита от угроз для других служб Майкрософт<a name="alerts-other"></a>

### <a name="threat-protection-for-azure-waf"></a>Защита от угроз для Azure WAF<a name="azure-waf"></a>

Шлюз приложений Azure предоставляет брандмауэр веб-приложения (WAF), обеспечивающий централизованную защиту веб-приложений от распространенных эксплойтов и уязвимостей.

Веб-приложения все чаще подвергаются вредоносным атакам, использующим общеизвестные уязвимости. Портал приложений WAF основан на наборе основных правил 3.0 или 2.2.9 проекта open Web Application Security Project. WAF обновляется автоматически для защиты от новых уязвимостей. 

Если у вас есть лицензия на Azure WAF, ваши оповещения WAF передаются в Центр безопасности без необходимости дополнительной конфигурации. Для получения дополнительной информации о предупреждениях, генерируемых WAF, [см.](../web-application-firewall/ag/application-gateway-crs-rulegroups-rules.md?tabs=owasp31#crs911-31)


### <a name="threat-protection-for-azure-ddos-protection"></a>Защита от угроз для Защиты от DDoS Azure<a name="azure-ddos"></a>

Распределенные атаки типа «отказ в обслуживании» (DDoS), как известно, легко выполняются. Они стали большой проблемой безопасности, особенно если вы перемещаете приложения в облако. 

Целью DDoS-атаки является исчерпание ресурсов приложения, чтобы оно стало недоступным для обычных пользователей. DDoS-атаки могут быть нацелены на любую конечную точку, которая может быть достигнута через Интернет.

Чтобы защититься от DDoS-атак, купите лицензию на DDoS-защиту Azure и убедитесь, что вы следите за рекомендациями по разработке приложений. DDoS Protection предоставляет различные уровни обслуживания. Для получения дополнительной информации смотрите [обзор защиты DDoS Azure](https://docs.microsoft.com/azure/virtual-network/ddos-protection-overview).

Для списка предупреждений о DDoS-защите [Reference table of alerts](alerts-reference.md#alerts-azureddos)Azure см.
