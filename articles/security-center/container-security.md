# <a name="container-security-in-security-center"></a>Безопасность контейнеров в центре безопасности

## <a name="vulnerability-management---scanning-container-images"></a>Управление уязвимостью — сканирование образов контейнеров

Чтобы отслеживать образы в реестрах контейнеров Azure на основе Azure Resource Manager, включите [защитник Azure для реестров контейнеров](defender-for-container-registries-introduction.md). Центр безопасности сканирует все образы, полученные за последние 30 дней, отправленные в реестр или импортированные. Встроенный сканер обеспечивается ведущим поставщиком сканирования уязвимостей, Qualys.

При обнаружении проблем — по Qualys или центру безопасности — вы получите уведомления на [панели мониторинга защитника Azure](azure-defender-dashboard.md). Для каждой уязвимости центр безопасности предоставляет практические рекомендации, а также классификацию серьезности и рекомендации по исправлению проблемы. Сведения о рекомендациях центра безопасности для контейнеров см. в [справочном списке рекомендаций](recommendations-reference.md#recs-containers).

Центр безопасности фильтрует и классифицирует результаты проверки. Если образ работоспособен, центр безопасности помечает его как таковой. Центр безопасности создает рекомендации по безопасности только для образов с проблемами, которые могут быть разрешены. Только уведомляя о наличии проблем, центр безопасности сокращает вероятность нежелательных информационных оповещений.

## <a name="environment-hardening"></a>Усиление защиты окружения

### <a name="continuous-monitoring-of-your-docker-configuration"></a>Непрерывный мониторинг конфигурации DOCKER

Центр безопасности Azure определяет неуправляемые контейнеры, размещенные на виртуальных машинах IaaS Linux или на других компьютерах Linux с контейнерами DOCKER. Центр безопасности постоянно оценивает конфигурации этих контейнеров. Затем он сравнивает их с [центром тестирования в центре безопасности Интернета (CIS) DOCKER](https://www.cisecurity.org/benchmark/docker/).

Центр безопасности включает весь набор правил тестового тестирования DOCKER и предупреждает вас, если контейнеры не соответствуют какому бы то ни было элементу управления. При обнаружении несоответствующих конфигураций центр безопасности создает рекомендации по безопасности. Используйте **страницу рекомендаций** центра безопасности для просмотра рекомендаций и устранения проблем. Тесты производительности CI не выполняются на управляемых виртуальных машинах AKS управляемых экземпляров или модулях.

Сведения о соответствующих рекомендациях центра безопасности, которые могут появиться для этой функции, см. в [разделе "контейнер](recommendations-reference.md#recs-containers) " справочной таблицы рекомендаций.

При изучении проблем безопасности виртуальной машины центр безопасности предоставляет дополнительные сведения о контейнерах на компьютере. Такие сведения включают версию DOCKER и количество образов, выполняемых на узле. 

Чтобы отслеживать неуправляемые контейнеры, размещенные на виртуальных машинах IaaS Linux, включите дополнительный [защитник Azure для серверов](defender-for-servers-introduction.md).


### <a name="continuous-monitoring-of-your-kubernetes-clusters"></a>Непрерывный мониторинг кластеров Kubernetes
Центр безопасности работает вместе со службой Azure Kubernetes Service (AKS), управляемой службой оркестрации контейнеров Майкрософт для разработки и развертывания контейнерных приложений, а также управления ими.

AKS предоставляет средства управления безопасностью и прозрачность для обеспечения безопасности кластеров. Центр безопасности использует эти функции для:
* Постоянное отслеживание конфигурации кластеров AKS
* Создание рекомендаций по безопасности, соответствующих отраслевым стандартам

Сведения о соответствующих рекомендациях центра безопасности, которые могут появиться для этой функции, см. в [разделе "контейнер](recommendations-reference.md#recs-containers) " справочной таблицы рекомендаций.

###  <a name="workload-protection-best-practices-using-kubernetes-admission-control"></a>Рекомендации по защите рабочих нагрузок с помощью управления допуском Kubernetes

Установите  **надстройку политики Azure для Kubernetes** , чтобы получить набор рекомендаций по защите рабочих нагрузок контейнеров Kubernetes.

Как описано на [странице Политика Azure для Kubernetes](../governance/policy/concepts/policy-for-kubernetes.md), надстройка расширяет веб-перехватчик с открытым исходным кодом версии [3](https://github.com/open-policy-agent/gatekeeper)   для устройства [Open Policy Agent](https://www.openpolicyagent.org/). Контроллеры Kubernetes допусков — это подключаемые модули, обеспечивающие использование кластеров. Надстройка регистрируется в качестве веб-перехватчика для Kubernetesного управления допуском и позволяет централизованно и согласованно применять принудительные применения и меры безопасности в кластерах. 

После установки надстройки в кластере AKS каждый запрос к серверу API Kubernetes будет отслеживаться по предопределенному набору рекомендаций перед сохранением в кластере. Затем можно настроить для применения рекомендаций и **задействовать** их для будущих рабочих нагрузок. 

Например, можно сделать так, чтобы привилегированные контейнеры не создавались, и все последующие запросы будут заблокированы.

Дополнительные сведения см. в статье [Защита рабочих нагрузок Kubernetes](kubernetes-workload-protections.md).
