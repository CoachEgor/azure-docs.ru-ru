# <a name="azure-security-policies-monitored-by-security-center"></a>Политики безопасности Azure, отслеживаемые центром безопасности
В этой статье содержится список определений политик Azure, которые можно отслеживать в центре безопасности Azure. Дополнительные сведения об использовании политик безопасности см. в статье [Использование политик безопасности](tutorial-security-policy.md).

## <a name="available-security-policies"></a>Доступные политики безопасности

Дополнительные сведения о встроенных политиках, отслеживаемых центром безопасности, см. в следующей таблице:

| Политика | Предназначение политики |
| --- | --- |
|Журналы диагностики в масштабируемых наборах виртуальных машин должны быть включены|Рекомендуется включить журналы, чтобы получить доступ к журналу действий после инцидента или компрометации.|
|Все правила авторизации, кроме RootManageSharedAccessKey, должны быть удалены из пространства имен концентратора событий.|Клиенты концентраторов событий Azure не должны использовать политику доступа на уровне пространства имен, которая предоставляет доступ ко всем очередям и разделам в пространстве имен. Чтобы обеспечить соответствие модели безопасности с наименьшими привилегиями, необходимо создать политики доступа на уровне сущности для очередей и разделов, чтобы предоставить доступ только определенной сущности.|
|Необходимо определить правила авторизации для сущности концентратора событий|Аудит существования правил авторизации в сущностях концентраторов событий для предоставления минимальных прав доступа.|
|Доступ к учетным записям хранения с конфигурациями брандмауэра и виртуальной сети должен быть ограничен.|Проведите аудит неограниченного сетевого доступа с применением параметров брандмауэра учетной записи хранения. Настройте сетевые правила, чтобы доступ к учетной записи хранения могли получать только приложения из разрешенных сетей. Чтобы разрешить подключения от конкретных клиентов в Интернете или локальных клиентах, предоставьте доступ к трафику из конкретных виртуальных сетей Azure или через общедоступные диапазоны IP-адресов.|
|Аудит использования настраиваемых правил RBAC|Аудит встроенных ролей, таких как "владелец", "участник", "читатель", вместо пользовательских ролей "Управление доступом на основе ролей" (RBAC), которые могут быть подвержены ошибкам. Использование пользовательских ролей рассматривается как исключение и требует тщательного рассмотрения и моделирования угроз.|
|Журналы диагностики в Azure Stream Analytics должны быть включены|Аудит включения журналов и их поддержка в течение года. Это создает журнал действий для изучения при возникновении инцидента безопасности или компрометации сети.|
|Необходимо включить безопасную пересылку в учетные записи хранения|Требования аудита безопасной пересылки в учетной записи хранения. Безопасная передача данных — это параметр, при включении которого ваша учетная запись хранения принимает запросы только с безопасных подключений (HTTPS). Использование протокола HTTPS обеспечивает проверку подлинности между сервером и службой. Он также защищает данные при передаче от атак сетевого уровня, таких как "злоумышленник в середине", "перехваты" и "перехват сеансов".|
|Необходимо подготовить администратора Azure AD для SQL Server|Аудит подготовки администратора Azure Active Directory (Azure AD) для SQL Server для включения проверки подлинности Azure AD. Аутентификация Azure AD поддерживает упрощенное управление разрешениями и централизованное управление удостоверениями пользователей базы данных и других служб Майкрософт.|
|Все правила авторизации, кроме RootManageSharedAccessKey, должны быть удалены из пространства имен служебной шины.|Клиенты служебной шины Azure не должны использовать политику доступа на уровне пространства имен, которая предоставляет доступ ко всем очередям и разделам в пространстве имен. Чтобы согласовать модель безопасности с минимальными правами доступа, создайте политики доступа на уровне сущности для очередей и разделов, чтобы предоставить доступ только определенной сущности.|
|Журналы диагностики в служебной шине должны быть включены|Аудит включения журналов и их поддержка в течение года. Это создает журнал действий для изучения при возникновении инцидента безопасности или компрометации сети.|
|Необходимо задать свойство Клустерпротектионлевел для EncryptAndSign в Service Fabric|Service Fabric предоставляет три уровня защиты для обмена данными между узлами, в которых используется первичный сертификат кластера: None, Sign и EncryptAndSign. Задайте уровень защиты, чтобы обеспечить шифрование и цифровое подписывание всех сообщений, передаваемых между узлами.|
|Проверка подлинности клиента должна использовать Azure Active Directory|Аудит использования проверки подлинности клиента только через Azure AD в Service Fabric.|
|Журналы диагностики в службах поиска должны быть включены|Аудит включения журналов и их поддержка в течение года. Это создает журнал действий для изучения при возникновении инцидента безопасности или компрометации сети.|
|Должны быть включены только безопасные подключения к кэшу Redis|Аудит включает только подключения через SSL к кэшу Azure для Redis. Использование безопасных соединений обеспечивает проверку подлинности между сервером и службой. Он также защищает данные при передаче от атак сетевого уровня, таких как "злоумышленник в середине", "перехваты" и "перехват сеансов".|
|Журналы диагностики в Logic Apps должны быть включены|Аудит включения журналов и их поддержка в течение года. Это создает журнал действий для изучения при возникновении инцидента безопасности или компрометации сети.|
|Журналы диагностики в Key Vault должны быть включены|Аудит включения журналов и их поддержка в течение года. Это создает журнал действий для изучения при возникновении инцидента безопасности или компрометации сети.|
|Журналы диагностики в концентраторе событий должны быть включены|Аудит включения журналов и их поддержка в течение года. Это создает журнал действий для изучения при возникновении инцидента безопасности или компрометации сети.|
|Журналы диагностики в Azure Data Lake Store должны быть включены|Аудит включения журналов и их последить до года. Это создает журнал действий для изучения при возникновении инцидента безопасности или компрометации сети.|
|Журналы диагностики в Data Lake Analytics должны быть включены|Аудит включения журналов и их поддержка в течение года. Это создает журнал действий для изучения при возникновении инцидента безопасности или компрометации сети.|
|Учетные записи хранения должны быть перенесены в новые ресурсы AzureRM|Используйте Azure Resource Manager для учетных записей хранения, чтобы обеспечить улучшение безопасности. в частности такие: <br>-Более надежное управление доступом (RBAC)<br>— Улучшенный аудит<br>— Развертывание и управление на основе Azure Resource Manager<br>— Доступ к управляемым удостоверениям<br>— Доступ к Azure Key Vault для секретов<br>— Проверка подлинности на основе Azure AD<br>— Поддержка тегов и групп ресурсов для упрощения управления безопасностью.|
