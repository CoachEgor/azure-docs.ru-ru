В этой статье список определений "Политика Azure", которые можно отслеживать в центре безопасности Azure. Дополнительные сведения об использовании политик безопасности см. в статье [Использование политик безопасности](tutorial-security-policy.md).

## <a name="available-security-policies"></a>Доступные политики безопасности

| Политика | Предназначение политики |
| --- | --- |

|Правила авторизации для пространства имен концентраторов событий аудита|Клиенты концентраторов событий Azure, не следует использовать политику доступа на уровне пространства имен, которая предоставляет доступ для всех очередей и разделов в пространстве имен. В соответствии с моделью безопасности наименьших привилегий, следует создать политики доступа на уровне сущности для очередей и разделов для предоставления доступа только определенную сущность.|
|Проверка наличия правил авторизации на сущности концентраторов событий|Аудит наличия правил авторизации на сущности концентраторов событий для предоставления минимальных привилегий доступа.|
|Аудит неограниченного сетевого доступа к учетным записям хранения|Проведите аудит неограниченного сетевого доступа с применением параметров брандмауэра учетной записи хранения. Настройте сетевые правила, позволяя только приложения из разрешенных сетей получать доступ к учетной записи хранения. Чтобы разрешить подключения из конкретных Интернет или локальные клиенты, предоставьте доступ к трафику с определенным виртуальным сетям Azure или общедоступны через Интернет IP-адрес диапазонов адресов.|
|Аудит использования настраиваемых правил RBAC|Аудит встроенных ролей, таких как «Владелец, участник или Читатель» вместо настраиваемого доступа на основе ролей (RBAC) элемента управления ролей, являющихся подвержено ошибкам. Использование настраиваемых ролей рассматривается как исключение и требует строгой проверки и моделирования угроз.|
|Аудит активации журналов диагностики в Azure Stream Analytics|Аудит Включение журналов и сохранять их для до одного года. Это создает журналы действий для исследования, при возникновении инцидента безопасности или несанкционированного доступа к сети.|
|Аудит безопасного перемещения в учетных записях хранения|Аудит требования безопасной передачи в учетной записи хранения. Безопасная передача данных — это параметр, при включении которого ваша учетная запись хранения принимает запросы только с безопасных подключений (HTTPS). Использование HTTPS обеспечивает проверку подлинности между сервером и службой. Он также защищает передаваемые данные от атак сетевого уровня, такие как man-in--middle, перехвата и перехвата сеанса.|
|Аудит Подготовка администратора Azure Active Directory для SQL Server|Аудит, Подготовка администратора Azure Active Directory (Azure AD) для SQL Server включить проверку подлинности Azure AD. Проверка подлинности Azure AD поддерживает управление разрешениями упрощенное и централизованное управление идентификаторами пользователей базы данных и другими службами Майкрософт.|

None, входа и EncryptAndSign. Задайте уровень защиты, чтобы обеспечить шифрование и цифровое подписывание всех сообщений, передаваемых между узлами.|
