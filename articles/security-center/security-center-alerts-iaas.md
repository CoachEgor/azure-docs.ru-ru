# <a name="threat-detection-for-vms--servers-in-azure-security-center"></a>Обнаружение угроз для виртуальных машин и серверов в центре безопасности Azure

В этом разделе представлены различные типы методов обнаружения и оповещениях, доступных для виртуальных машин и серверов в следующих системах операции. Список поддерживаемых версий см. в разделе [платформ и функции, поддерживаемые центром безопасности Azure](https://docs.microsoft.com/azure/security-center/security-center-os-coverage).

* [Windows](#windows-machines)
* [Linux](#linux-machines)

## Windows <a name="windows-machines"></a>

Центр безопасности интегрируется со службами Azure для мониторинга и защиты на компьютерах под управлением Windows.  Центр безопасности предоставит оповещения и рекомендации по исправлению из всех этих служб в формате простой в использовании.

### ATP в Защитнике Microsoft Server <a nanme="windows-atp"></a>

Центр безопасности Azure расширяет его облачные платформы защиты рабочей нагрузки, интеграция с Windows Defender Advanced Threat Protection (ATP). Это обеспечивает широкие возможности обнаружения конечной точки и ответ (EDR).

> [!NOTE]
> Датчик ATP в Защитнике Windows Server включается автоматически на серверах Windows, будут подключены к центру безопасности Azure.

Если ATP в Защитнике Windows Server обнаруживает угрозу, он создает оповещение. Оповещения отображаются на панели мониторинга центра безопасности. На панели мониторинга сводных консоль ATP в Защитнике Windows, чтобы выполнить детальное исследование для анализа данных в области атаки. Дополнительные сведения о ATP в Защитнике Windows Server, см. в разделе [подключение серверов к службе ATP в Защитнике Windows](https://docs.microsoft.com/windows/security/threat-protection/microsoft-defender-atp/configure-server-endpoints).

### Анализ аварийного дампа <a nanme="windows-dump"></a>

Когда программа дает сбой, аварийный дамп записывает часть памяти в момент сбоя.

Сбой может быть вызвана вредоносными программами или сдерживать обнаруженные вредоносные программы. Чтобы избежать обнаружения с помощью продуктов безопасности, различные виды вредоносных программ используйте fileless атак, что позволяет избежать записи на диск или шифрования программные компоненты, записанные на диск. Этот тип атаки трудно обнаружить с помощью традиционных подходов на диске.

Тем не менее атаки такого рода можно обнаружить с помощью метода анализа памяти. Анализируя память в аварийном дампе, Центр безопасности может выявить методы, которые использует атаки для использования уязвимостей в программном обеспечении, доступ к конфиденциальным данным и тайно пользуется скомпрометированным компьютером. Это делается в центре безопасности серверной части с минимальное влияние на узлы.

> [!div class="mx-tableFixed"]

|Предупреждение|Описание|
|---|---|
|**Обнаружение внедрения кода**|Внедрение кода — это вставка исполняемых модулей в выполняемые процессы или потоки. Этот метод используется во вредоносных программ, для доступа к данным, пока успешно прячась во избежание найден и удален. <br/>Это оповещение означает, что внедренный модуль присутствует в аварийном дампе. Чтобы различать вредоносных и других модулей внедренного, Центр безопасности проверяет, соответствует ли внедренный модуль профилю подозрительного поведения.|
|**Подозрительный сегмент кода обнаружен**|Указывает на то, что сегмент кода выделены нестандартными способами, например с отражением и скрытии процесса. Оповещение содержит дополнительные характеристики сегмента кода, которые были обработаны для предоставления контекста для возможности и поведение сегмента.|
|**Обнаружение кода оболочки**|Код оболочки — это атакующий код, который выполняется, когда вредоносная программа воспользовалась уязвимостью программного обеспечения.<br/>Это предупреждение означает, что анализа аварийного дампа обнаружил исполняемый код, который выполняет обработку, обычно выполняемые вредоносных программ. Несмотря на то, что не вредоносные программы также могут выполнять это поведение, он не является типичным для обычного программного обеспечения.|

### Обнаружение атак fileless <a nanme="windows-fileless"></a>

В Azure мы регулярно см. в разделе fileless атак, направленных конечные точки для наших клиентов.

Чтобы избежать обнаружения, fileless атак внедрения вредоносных программ в память. Злоумышленник полезные данные сохраняются в памяти скомпрометированных процессов и выполнять широкий спектр вредоносных действий.

С системой обнаружения fileless атаки методики анализа работы автоматического памяти определите наборы средств fileless атаки, методик и поведения. Это решение периодически проверяет машину во время выполнения и извлекает insights непосредственно из памяти с точки зрения безопасности процессов.

Он находит свидетельства эксплуатация, внедрение кода и выполнение вредоносных программ. Fileless обнаружение атак создает предупреждения подробные сведения о безопасности для ускорения оповещения "Рассмотрение", корреляции и время ответа подчиненных. Такой подход дополняет событий EDR решений на основе предоставляя больше охвата обнаружения.

## Linux <a name="linux-machines"></a>

Центр безопасности собирает записи из Linux с помощью **auditd**, используя один из наиболее распространенных Linux аудит платформы. auditd имеет преимущество уже много времени и жизнь в магистраль ядра. 

### Оповещения Linux auditd и интеграции Microsoft Monitoring Agent (MMA) <a name="linux-auditd"></a>

Система auditd состоит из подсистема уровня ядра, который отвечает за мониторинг системных вызовов, фильтрации по заданного набора правил и сообщения записываются для них сокета. Центр безопасности интегрируется функциональные возможности из пакета auditd в Microsoft Monitoring Agent (MMA). Эта интеграция включает сбор событий auditd на всех поддерживаемых дистрибутивах Linux без все необходимые компоненты.  

записи auditd собираются, насыщенные и объединению событий с помощью агента MMA для Linux. Центр безопасности постоянно работу по добавлению новой аналитики, что использование Linux сигналы для обнаружения вредоносного поведения на облачных и локальных компьютеров Linux. Как и возможности Windows, эти аналитические охватывают подозрительные процессы, сомнительные попытки входа, загрузка модулей ядра и другие действия, которые указывают компьютер находится под атакой или возникло нарушение.  

Ниже приведено несколько примеров для службы анализа, которые демонстрируют, как мы распространяться на разных этапах жизненного цикла атаки.

> [!div class="mx-tableFixed"]

|Предупреждение|Описание|
|---|---|
|**Увидеть, доступ к файлу авторизованных ключей SSH необычным способом процесса**|Файл авторизованных ключей SSH уже был осуществлен в метод, схожий с кампаний известных вредоносных программ. Этот доступ может означать, что злоумышленник пытается получить постоянный доступ к машине|
|**Попытка обнаруженных сохраняемости**|Анализ данных узла обнаружила, что установлено сценарий запуска в однопользовательском режиме. <br/>Так как маловероятно, что любой процесс законным потребовался бы для выполнения в этом режиме, это может означать, что злоумышленник добавил злонамеренный процесс для каждого уровня выполнения для обеспечения сохраняемости.|
|**Манипуляции обнаружил запланированных задач**|Анализ данных узла обнаружила возможных манипуляции запланированных заданий. Злоумышленники часто добавляет запланированные задачи для компьютеров, которые они скомпрометированы для получения постоянного хранения.|
|**Изменение отметки подозрительный файл**|Анализ данных узла обнаружил изменение подозрительных отметки. Злоумышленники часто скопируйте отметки времени по существующим файлам с законным новые средства, чтобы избежать обнаружения этих недавно перенесенных файлов|
|**Новый пользователь был добавлен в группу sudoers**|Анализ данных узла обнаружил, что пользователь был добавлен в группу sudoers, позволяющий его члены для выполнения команд с высоким уровнем привилегий.|
|**Скорее всего воспользоваться DynoRoot уязвимость в DHCP-клиента**|Анализ данных узла обнаружил выполнение необычные команды с родительский процесс скрипта dhclient.|
|**Обнаружен модуль подозрительных ядра**|Анализ данных узла обнаружен файл в общий объект, загружаемый как модуль ядра. Это может быть допустимые действия, или значение, указывающее, один из ваших компьютеров был скомпрометирован.|
|**Процесс, связанное с интеллектуальным анализом цифровой валюты обнаружено**|Анализ данных узла обнаружил выполнения процесса, обычно связанных с интеллектуальным анализом цифровой валюты|
|**Потенциальные перенаправление портов внешний IP-адрес**|Анализ данных узла обнаружил инициирования внешний IP-адрес перенаправления портов.|
