# <a name="move-mainframe-storage-to-azure"></a>Перемещение хранилища мэйнфрейма в Azure

Для выполнения рабочих нагрузок мейнфреймов в Microsoft Azure необходимо знать, как возможности мейнфрейма соотносятся с Azure. Массово масштабируемые ресурсы хранения могут помочь организациям начать модернизацию, не отказываясь от приложений, на которые они полагаются.

Azure предоставляет мэйнфрейм-подобные функции и емкость хранилища, сопоставимые с системами на основе IBM z14 (самая актуальная модель на момент написания этой статьи). В этой статье рассказывается, как получить сопоставимые результаты на Azure.

## <a name="mainframe-storage-at-a-glance"></a>Хранение мейнфреймов с первого взгляда

Мэймфрейм IBM характеризует хранение двумя способами. Во-первых, это устройство хранения прямого доступа (DASD). Во-вторых, последовательное хранение. Для управления хранилищем данных мэйнфрейм обеспечивает подсистему управления хранением данных (DFSMS). Он управляет доступом к данным к различным устройствам хранения данных.

[DASD](https://en.wikipedia.org/wiki/Direct-access_storage_device) относится к отдельному устройству для вторичного (не в памяти) хранилища, которое позволяет использовать уникальный адрес для прямого доступа к данным. Первоначально термин DASD применяется к вращающимся дискам, магнитным барабанам или ячейкам данных. Однако теперь этот термин может применяться и к твердотельным устройствам хранения данных (SSD), сетям складских помещений (SANs), сетевым хранилищам (NAS) и оптическим дискам. Для целей этого документа DASD относится к вращающимся дискам, SANs и SSD.

В отличие от хранилища DASD, последовательное хранение на мэйнфрейме относится к устройствам, таким как ленточные диски, где доступ к данным осуществляется с отправной точки, а затем читается или записывается строкой.

Устройства хранения данных, как правило, крепятся с помощью волоконного соединения (FICON) или доступны непосредственно на io-автобусе мейнфрейма с использованием [HiperSockets](https://www.ibm.com/support/knowledgecenter/zosbasics/com.ibm.zos.znetwork/znetwork_85.htm), технологии IBM для высокоскоростной связи между перегородками на сервере с гипервизором.

Большинство мэйнфреймных систем разделяют хранилище на два типа:

- *Онлайн-хранилище* (также известное как горячее хранилище) необходимо для ежедневных операций. Хранение DASD обычно используется для этой цели. Однако для этой цели также может использоваться последовательное хранение, такое как ежедневные резервные копии ленты (логические или физические).

- *Архивное хранилище* (также известное как холодное хранилище) не гарантируется к установке в данный момент времени. Вместо этого он устанавливается и к нему при необходимости. Архивное хранилище часто реализуется с использованием последовательных резервных копий ленты (логических или физических) для хранения.

## <a name="mainframe-versus-io-latency-and-iops"></a>Мейнфрейм против задержки ентентия и IOPS

Мэйнфреймы часто используются для приложений, требующих высокой производительности и низкой задержки истого. Они могут сделать это с помощью подключения FICON к устройствам и HiperSockets. Когда HiperSockets используются для подключения приложений и устройств непосредственно к каналу обработки данных мэйнфрейма, задержка в микросекундах может быть достигнута.

## <a name="azure-storage-at-a-glance"></a>Хранилище Azure с первого взгляда

Варианты инфраструктуры Azure как службы[(IaaS)](https://azure.microsoft.com/overview/what-is-iaas/)для хранения обеспечивают сопоставимую емкость мэйнфрейма.

Корпорация Майкрософт предлагает хранение петабайтов для приложений, размещенных в Azure, и у вас есть несколько вариантов хранения. Они варьируются от хранения SSD для высокой производительности до недорогого хранения кабл для массового хранения и архивов. Кроме того, Azure предоставляет опцию избыточности данных для хранения— что-то, что требует больше усилий для настройки в среде мэйнфрейма.

## <a name="azure-hot-online-and-cold-archive-storage"></a>Лазурное горячее (онлайн) и холодное (архивное) хранение

Тип хранилища для данной системы зависит от требований системы, включая размер хранилища, пропускную стоимость и IOPS. Для хранения dASD-типа на мэйнфрейме приложения Azure обычно используют хранилище дисков Azure Disks. Для хранения архива мейнфреймов в Azure используется хранилище кабр.

SSD обеспечивают самую высокую производительность хранилища в Azure. Доступны следующие варианты (на момент написания настоящего документа):

| Тип         | Размер           | ОПЕРАЦИЙ ВВОДА-ВЫВОДА                  |
|--------------|----------------|-----------------------|
| SSD ценовой категории "Ультра"    | От 4 ГБ до 64 ТБ  | от 1 200 до 160 000 IOPS |
| SSD (цен. категория "Премиум")  | От 32 ГБ до 32 ТБ | от 12 до 15 000 IOPS     |
| SSD (цен. категория "Стандартный") | От 32 ГБ до 32 ТБ | От 12 до 2000 IOPS      |

Хранилище Blob обеспечивает наибольший объем хранилища в Azure. Помимо размера хранилища, Azure предлагает как управляемое, так и неуправляемое хранилище. С управляемым хранилищем Azure заботится об управлении базовыми учетными записями хранения. При неуправляемом хранилище пользователь берет на себя ответственность за настройку учетных записей хранения Данных хранилища Azure соответствующего размера для удовлетворения требований к хранению.
