# <a name="sap-hana-on-azure-large-instance-migration-to-azure-virtual-machines"></a>SAP HANA на Azure Большой экземпляр миграции в Azure виртуальных машин
В этой статье описаны возможные сценарии развертывания Azure Large Instance и предлагается подход планирования и миграции с минимизированным временем перехода

## <a name="overview"></a>Обзор
С момента объявления в сентябре 2016 года о проведении Azure Large Instances for SAP HANA (HLI) многие клиенты приняли это оборудование в качестве сервиса для своей вычислительной платформы в памяти.  В последние годы расширение размера Azure VM в сочетании с поддержкой развертывания масштабирования HANA превысило потребность большинства корпоративных клиентов в емкости ERP-базы данных.  Мы начинаем видеть, как клиенты выражают заинтересованность в переносе рабочей нагрузки SAP HANA с физических серверов на MMs-носители Azure.
Это руководство не является поэтапным документом конфигурации. В нем описаны общие модели развертывания и предлагаются рекомендации по планированию и миграции.  Цель состоит в том, чтобы вызвать необходимые соображения для подготовки к минимизации времени простоя перехода.

## <a name="assumptions"></a>Предположения
Для работы с этой статьей предполагается следующее:
- Единственным интересом считается однородная миграция службы данных HANA из Hana Large Instance (HLI) в Azure VM без существенного обновления программного обеспечения или исправления. Эти незначительные обновления включают использование более свежей версии ОС или версии HANA, явно указанной в поддержку соответствующих заметок SAP. 
- Все действия по обновлению/обновлению должны быть выполнены до или после миграции.  Например, преобразование SAP HANA MCOS в развертывание MDC. 
- Миграционный подход, который предлагает наименьшее время простоя, — это репликация системы SAP HANA. Другие методы миграции не входят в сферу действия этого документа.
- Это руководство применимо как для Rev3, так и для Rev4 SKUs HLI.
- Архитектура развертывания HANA остается в основном неизменной во время миграции.  То есть система с одним экземпляром DR останется такой же в пункте назначения.
- Клиенты рассмотрели и поняли Соглашение об уровне обслуживания (SLA) целевой (к быть) архитектуры. 
- Коммерческие условия между ИП и ВМ различны. Клиенты должны контролировать использование своих ВМ для управления затратами.
- Клиенты понимают, что HLI — это специальная вычислительная платформа, в то время как VM работают на общей, но изолированной инфраструктуре.
- Клиенты проверили, что целевые ВМ поддерживают предполагаемую архитектуру. Чтобы увидеть все поддерживаемые VM SKUs, сертифицированные для развертывания SAP HANA, [см.](https://www.sap.com/dmc/exp/2014-09-02-hana-hardware/enEN/iaas.html#categories=Microsoft%20Azure)
- Клиенты подтвердили план проектирования и миграции.
- План аварийного восстановления VM вместе с основным сайтом.  Клиенты не могут использовать HLI в качестве узла DR для основного сайта, работаемого на ВМ после миграции.
- Клиенты скопировали требуемые файлы резервного копирования для таргетинга на вс-ми, основываясь на возможности восстановления бизнеса и соответствии требованиям. С помощью доступных резервных upups VM он позволяет восстановить момент времени в переходный период.
- Для HSR HA, клиенты должны настроить и настроить устройство STONITH на SAP HANA HA руководства для [SLES](https://docs.microsoft.com/azure/virtual-machines/workloads/sap/high-availability-guide-suse-pacemaker) и [RHEL](https://docs.microsoft.com/azure/virtual-machines/workloads/sap/high-availability-guide-rhel-pacemaker).  Это не предварительно настроено, как случай HLI.
- Этот миграционный подход не охватывает HLI SKUs с конфигурацией Optane.

## <a name="source-hli-planning"></a>Планирование источника (HLI)
При посадке на HLI-сервер и управление службами майкрософт и клиенты прошли планирование параметров вычислений, сетей, хранения и ОС для запуска базы данных SAP HANA.  Аналогичное планирование должно иметь место для миграции в Azure VM.

### <a name="sap-hana-housekeeping"></a>Домашнее хозяйство SAP HANA 
Это хорошая оперативная практика, чтобы привести в порядок содержимое базы данных, чтобы нежелательные, устаревшие данные или устаревшие журналы не мигрировали в новую базу данных.  Ведение домашнего хозяйства обычно включает в себя удаляние или архивирование старых, просроченных или неактивных данных.  Эти действия по гигиене данных должны быть проверены в непроизводственных системах для проверки их обрезки данных перед использованием продукции.

### <a name="allow-network-connectivity-for-new-vms-and-or-virtual-network"></a>Разрешить подключение к сети для новых виртуальных и виртуальных сетей 
При развертывании HLI клиента сеть была настроена на основе информации, описанной в [сетевой архитектуре SAP HANA (Large Instances).](https://docs.microsoft.com/azure/virtual-machines/workloads/sap/hana-network-architecture) Кроме того, трассировка сетевого трафика осуществляется в порядке, изложенном в разделе «Трассирование в Azure».
- При настройке нового VM в качестве цели миграции, если он помещен в существующую виртуальную сеть с уже разрешенными диапазонами IP-адресов для подключения к HLI, дальнейшее обновление подключения не требуется.
- Если новый Azure VM размещен в новой виртуальной сети Microsoft Azure, может находиться в другом регионе и заглянул в существующую виртуальную сеть, ключ службы ExpressRoute и идентификатор ресурсов из оригинального комплекта HLI можно найти для доступа к этой новой виртуальной сети диапазон Ip сети.  Координация с microsoft Service Management, чтобы позволить виртуальной сети подключение HLI.  Примечание: Чтобы свести к минимуму задержку сети между слоями приложения и базы данных, слои приложения и базы данных должны находиться в одной виртуальной сети.  

### <a name="existing-app-layer-availability-set-availability-zones-and-proximity-placement-group-ppg"></a>Существующий набор доступности слоя приложений, зоны доступности и группа размещения близости (PPG)
Текущая модель развертывания выполняется для удовлетворения определенных целей уровня обслуживания.  В этом шаге убедитесь, что целевая инфраструктура будет соответствовать или превышать поставленные цели.  
Скорее всего, серверы приложений SAP помещаются в набор доступности.  Если текущий уровень службы развертывания является удовлетворительным и 
- Если целевой VM принимает имя хоста логического имени HLI, обновление службы доменных имен (DNS) разрешение адреса, указавщее на IP VM, будет работать без обновления каких-либо профилей SAP
- Если вы не используете PPG, не забудьте разместить все приложения и DB-серверы в одной зоне, чтобы свести к минимуму задержку сети.
- Если вы используете PPG, обратитесь к разделу этого документа: "Планирование назначения, набор доступности, зоны доступности и группа размещения близости (PPG)".

### <a name="storage-replication-discontinuance-process-if-used"></a>Процесс прерывания репликации хранилища (при использовании)
Если репликация хранилища используется в качестве решения DR, она должна быть прекращена (незапланированная) после выключения приложения SAP.  Кроме того, последний каталог SAP HANA, файл журнала и резервные копии данных были воспроизведены на удаленных объемах хранения ДАННЫх DR HLI.  Это в качестве меры предосторожности в случае сбоя во время перехода на vM-сервера.

### <a name="data-backups-preservation-consideration"></a>Рассмотрение сохранения резервных данных
После перекройки sAP HANA на Azure VM все данные на основе моментального снимка или резервные копии журналов на HLI не являются легкодоступными или восболеемыми для VM, если это необходимо. В начале переходного периода, прежде чем резервное копирование на основе Azure накапливает достаточно истории для удовлетворения требований восстановления Point-in-time, мы рекомендуем использовать резервные копии файлов в дополнение к снимкам на HLI, за несколько дней или недель до сокращения.  Скопировать эти резервные копии на учетную запись хранения Azure доступна новому SAP HANA VM.
В дополнение к резервному копированию контента HLI, разумно иметь полные резервные копирования ландшафта SAP легко доступны в случае необходимости отката.

### <a name="adjusting-system-monitoring"></a>Корректировка системы мониторинга 
Клиенты используют множество различных инструментов для мониторинга и отправки уведомлений для систем в рамках sAP-ландшафта.  Этот элемент является всего лишь призывом к соответствующим действиям по включению изменений для мониторинга и обновления получателей уведомлений о помятии, если это необходимо.

### <a name="microsoft-operations-team-involvement"></a>Участие команды Microsoft Operations 
Откройте билет с портала Azure на основе существующего экземпляра HLI.  После создания билета поддержки инженер службы поддержки свяжется с вами по электронной почте.  

### <a name="engage-microsoft-account-team"></a>Привлечение команды учетных записей Майкрософт
Планируйте миграцию ближе к времени продления контракта HLI, чтобы свести к минимуму ненужные расходы на вычислительный ресурс. Для вывода из эксплуатации лезвия HLI необходимо координировать расторжение контракта и фактическое отключение устройства.

## <a name="destination-planning"></a>Планирование назначения
Создание новой инфраструктуры, чтобы занять место существующей заслуживает некоторого мышления, чтобы обеспечить новое дополнение будет вписываться в большую схему вещей.  Ниже приведены некоторые ключевые моменты для созерцания.

### <a name="resource-availability-in-the-target-region"></a>Наличие ресурсов в целевом регионе 
Текущая область развертывания серверов приложений SAP обычно находится в непосредственной близости от связанных ими ими ихлопов.  Однако игоры предлагаются в меньшем количестве мест, чем в доступных регионах Azure.  При переходе физического HLI в Azure VM также хорошее время для «тонкой настройки» расстояния близости всех связанных служб для оптимизации производительности.  При этом одним из ключевых соображений является обеспечение того, чтобы выбранный регион распоряжался всеми необходимыми ресурсами.  Например, наличие определенной семьи VM или предложение зон Azure для настройки высокой доступности.

### <a name="virtual-network"></a>Виртуальная сеть 
Клиентам необходимо выбрать, запускать ли новую базу данных HANA в существующей виртуальной сети или создавать новую.  Основным решающим фактором является текущая компоновка сетей для ландшафта SAP.  Кроме того, когда инфраструктура переходит из одной зоны в двухзонное развертывание и использует PPG, она накладывает архитектурные изменения. Для получения дополнительной информации смотрите статью [Azure PPG для оптимальной задержки сети с приложением SAP](https://docs.microsoft.com/azure/virtual-machines/workloads/sap/sap-proximity-placement-scenarios).   

### <a name="security"></a>Безопасность
Независимо от того, является ли новая посадка SAP HANA VM на новую или существующую vnet/subnet, она представляет собой новую важную для бизнеса услугу, требующую защиты.  Контроль доступа, соответствующий политике безопасности информации компании, должен быть оценен и развернут для этого нового класса обслуживания.

### <a name="vm-sizing-recommendation"></a>Рекомендация vM по размеру размеров
Эта миграция также дает возможность правильного размера вычислительного движка HANA.  Можно использовать [представления системы](https://help.sap.com/viewer/7c78579ce9b14a669c1f3295b0d8ca16/Cloud/3859e48180bb4cf8a207e15cf25a7e57.html) HANA совместно с HANA Studio, чтобы понять потребление ресурсов системы, что позволяет правильно использовать размеры для повышения эффективности расходов.

### <a name="storage"></a>Хранилище 
Производительность хранилища является одним из факторов, влияющих на пользовательский интерфейс приложения SAP.  База на данном VM SKU, Есть минимальная компоновка хранения опубликованных [SAP HANA Azure виртуальные конфигурации машин хранения.](https://docs.microsoft.com/azure/virtual-machines/workloads/sap/hana-vm-operations-storage) Мы рекомендуем пересмотреть эти минимальные характеристики и сравнить с существующей статистикой системы HLI, чтобы обеспечить адекватную емкость и производительность IO для нового HANA VM.

Если вы наверстируете PPG для нового HANA VM и связанных с ним отделов, отправьте билет поддержки для проверки и обеспечения совместного расположения хранилища и VM. Поскольку решение для резервного копирования может потребоваться изменить, стоимость хранилища также должна быть пересмотрена, чтобы избежать непредвиденных расходов на эксплуатацию.

### <a name="storage-replication-for-disaster-recovery"></a>Репликация хранилища для аварийного восстановления
С ПОМОЩЬю HLI репликация хранилища была предложена в качестве опции по умолчанию для аварийного восстановления. Эта функция не является опцией по умолчанию для SAP HANA на Azure VM. Рассмотрим HSR, резервное копирование/восстановление или другие поддерживаемые решения, удовлетворяющие ваши бизнес-потребности.

### <a name="availability-sets-availability-zones-and-proximity-placement-groups"></a>Наборы доступности, зоны доступности и группы размещения близости 
Чтобы сократить расстояние между слоем приложений и SAP HANA, чтобы свести к минимуму задержку сети, новая база данных VM и текущие серверы приложений SAP должны быть помещены в PPG. Обратитесь в [группу размещения proximity,](https://docs.microsoft.com/azure/virtual-machines/workloads/sap/sap-proximity-placement-scenarios) чтобы узнать, как набор и зоны доступности Azure работают с PPG для развертывания SAP.
Если члены целевой системы HANA развернуты в более чем одной зоне Azure, клиенты должны иметь четкое представление о профиле задержки выбранных зон. Размещение компонентов системы SAP является оптимальным в отношении проксимального расстояния между приложением SAP и базой данных.  Инструмент [проверки на задержку зоны доступности](https://github.com/Azure/SAP-on-Azure-Scripts-and-Utilities/tree/master/AvZone-Latency-Test) общественного достояния помогает упростить измерение.  


### <a name="backup-strategy"></a>Стратегия резервного копирования
Многие клиенты уже используют сторонние решения резервного копирования для SAP HANA на HLI.  В этом случае необходимо настроить только дополнительные защищенные базы данных VM и HANA.  Текущие задания резервного копирования HLI теперь могут быть незапланированными, если машина вывозится из эксплуатации после миграции.
Резервное копирование Azure для SAP HANA на VM теперь общедоступно.  Смотрите эти ссылки для получения подробной информации о: [Резервное копирование](https://docs.microsoft.com/azure/backup/backup-azure-sap-hana-database), [Восстановление](https://docs.microsoft.com/azure/backup/sap-hana-db-restore), [Управление](https://docs.microsoft.com/azure/backup/sap-hana-db-manage) SAP HANA резервного копирования в Azure VMs.

### <a name="dr-strategy"></a>Стратегия DR
Если цели вашего уровня обслуживания вмещают более длительное время восстановления, простая резервная копию для хранения кабы и восстановления на месте или восстановления нового VM является самой простой и наименее дорогостоящей стратегией DR.  
Как и большая платформа экземпляра, где HANA DR обычно делается с HSR; На Azure VM HSR также является самым естественным и естественным решением SAP HANA DR.  Независимо от того, является ли развертывание источника одноэкземпляром или кластерным, в регионе DR требуется копия исходной инфраструктуры.
Эта реплика DR будет настроена после завершения первичной миграции HLI в VM.  DR HANA DB зарегистрируется в основной SAP HANA на экземпляре VM в качестве вторичного сайта репликации.  

### <a name="sap-application-server-connectivity-destination-change"></a>Изменение назначения сервера подключения сервера SAP
Миграция HSR приводит к попатению нового хоста HANA DB и, следовательно, нового h-name DB для слоя приложения, профили SAP должны быть изменены, чтобы отразить новое имя хоста.  Если переключение осуществляется по разрешению имени, сохраняя имя, изменение профиля не требуется.

### <a name="operating-system"></a>Операционная система
Изображения операционной системы для HLI и VM, несмотря на то, что они на одном уровне выпуска, SLES 12 SP4, например, не идентичны. Клиенты должны проверить необходимые пакеты, горячие исправления, патчи, ядро и исправления безопасности на HLI, чтобы установить те же пакеты на цель.  Он поддерживается для использования HSR для репликации из старой ОС на VM с новой версией ОС.  Проверьте конкретные поддерживаемые версии, просмотрев [Примечание SAP 2763388](https://launchpad.support.sap.com/#/notes/2763388).

### <a name="new-sap-license-request"></a>Новый запрос лицензии SAP
Простой вызов для запроса новой лицензии SAP для новой системы HANA теперь, когда она была перенесена на VMs.

### <a name="service-level-agreement-sla-differences"></a>Различия в соглашении об уровне обслуживания (SLA)
Авторы любят называть разницу в доступности SLA между HLI и Azure VM.  Например, кластерные пары HLIs HA предлагают доступность 99,99%. Чтобы достичь той же SLA, необходимо развернуть VMs в зонах доступности. В [этой статье описывается](https://azure.microsoft.com/support/legal/sla/virtual-machines/v1_9/) доступность с связанными архитектурами развертывания, чтобы клиенты могли соответствующим образом планировать свою целевую инфраструктуру.


## <a name="migration-strategy"></a>Стратегия миграции
В этом документе мы рассмотрим только подход репликации системЫ HANA для миграции из HLI в Azure VM.  В зависимости от целевого решения для хранения данных процесс немного отличается. Ниже описаны шаги высокого уровня.

### <a name="vm-with-premiumultra-disks-for-data"></a>VM с премиальными/ультра-дисками для данных
Для VMs, которые развернуты с премиум или ультра-дисков, стандартная конфигурация репликации системы SAP HANA применима для настройки HSR.  [В статье справки SAP](https://help.sap.com/viewer/6b94445c94ae495c83a19646e7c3fd56/2.0.04/099caa1959ce4b3fa1144562fa09e163.html) приводится обзор шагов, связанных с настройкой репликации системы, захватом вторичной системы, отказом назад к первичной репликации системы.  Для миграции нам потребуется только настройка, захват и отключение шагов репликации.  

### <a name="vm-with-anf-for-data-and-log-volumes"></a>VM с ANF для данных и объемов журналов
На высоком уровне последние снимки хранения HLI полных объемов данных и журналов должны быть скопированы в Хранилище Azure, где они доступны и восстанавливаются целевой HANA VM.  Процесс копирования может быть сделан с любыми родными инструментами копирования Linux.  

> [!IMPORTANT]
> Копирование и передача данных может занять несколько часов, зависит от размера базы данных HANA и пропускной способности сети.  Основная часть процесса копирования должна быть выполнена до первичного простоя HANA DB.

### <a name="mcos-to-mdc-conversion"></a>MCOS к преобразованию MDC
Модель развертывания multiple Components in One System (MCOS) была использована некоторыми нашими клиентами HLI.  Мотивация состояла в том, чтобы обойти ограничение хранилища нескольких баз данных (MDC) более ранних версий SAP HANA.  В модели MCOS несколько независимых экземпляров SAP HANA укладываются в одно лезвие HLI.  Использование HSR для миграции будет работать нормально, но в результате чего несколько HANA VMs с одним арендатором DB каждый.  Это состояние делает для занят пейзаж, чем то, что клиент, возможно, привыкли.  Поскольку развертывание sAP HANA 2.0 по умолчанию является MDC, жизнеспособной альтернативой является выполнение [перемещения клиента HANA](https://launchpad.support.sap.com/#/notes/2472478) после миграции HSR.  Этот процесс «объединяет» эти независимые базы данных HANA в cotenants в одном контейнере HANA.  

### <a name="application-layer-consideration"></a>Рассмотрение уровня приложения
Сервер DB рассматривается как центр системы SAP.  Все серверы приложений должны быть расположены рядом с DB SAP HANA.  В некоторых случаях, когда требуется новое использование PPG, может потребоваться перемещение существующих серверов приложений на PPG, где hanA VM.  Создание новых серверов приложений может быть признано проще, если у вас уже есть шаблоны развертывания под рукой.  
Если существующие серверы приложений и новый HANA VM оптимально расположены, не требуется создание новых серверов приложений, если не требуется дополнительная емкость.  
Если новая инфраструктура построена для повышения доступности службы, существующие серверы приложений могут стать ненужными и должны быть закрыты и удалены.
Если целевое имя VM-хоста изменилось и отличается от хост-множец HLI, профили сервера приложений SAP должны быть скорректированы, чтобы указать на новый хост.  Если изменился только IP-адрес HANA DB, необходимо обновление записи DNS, чтобы привести входящие соединения к новому HANA VM.

### <a name="acceptance-test"></a>Приемочный тест
Хотя переход от HLI к VM не делает никаких существенных изменений в содержании базы данных по сравнению с неоднородной миграцией, мы по-прежнему рекомендуем проверку ключевых функциональных возможностей и аспекта производительности новой установки.  

### <a name="cutover-plan"></a>План сокращения
Хотя эта миграция происходит прямо вперед, она, однако, предполагает вывод из эксплуатации существующего DB.  Тщательное планирование для сохранения исходной системы с ее соответствующим содержанием и резервными изображениями имеют решающее значение в случае необходимости резервного копирования.  Хорошее планирование предлагает более быстрый разворот.   


## <a name="post-migration"></a>Действия после перемещения
Задание по миграции не выполняется до тех пор, пока мы не разъясняем любые услуги или подключения, зависящие от HLI, чтобы обеспечить целостность данных.  Кроме того, отключите ненужные услуги.  В этом разделе высылают несколько первоклассных элементов.

### <a name="decommissioning-the-hli"></a>Вывод из эксплуатации HLI
После успешной миграции HANA DB в Azure VM, обеспечить отсутствие продуктивных бизнес-транзакций, запускаемых на HLI DB.  Однако сохранение работы HLI в течение определенного периода времени равно локальному окну удержания резервного копирования является безопасной практикой, обеспечивающей более быстрое восстановление, если это необходимо.  Только после этого лезвие HLI должно быть выведено из эксплуатации.  Клиенты должны заключить контрактные обязательства с корпорацией Майкрософт, связавшись со своими представителями корпорации Майкрософт.

### <a name="remove-any-proxy-ex-iptables-bigip-configured-for-hli"></a>Удалите любой прокси (например: Iptables, BIGIP), настроенный для HLI 
Если прокси-сервис, такой как IPTables, используется для маршрутизации трафика в и из HLI, он больше не нужен после успешной миграции в VM.  Тем не менее, эта служба подключения должна храниться до тех пор, пока лезвие HLI все еще находится рядом.  Выключите службу только после полного вывоза лезвия HLI.

### <a name="remove-global-reach-for-hli"></a>Удалить глобальный охват для HLI 
Global Reach используется для подключения шлюза ExpressRoute клиентов с шлюзом HLI ExpressRoute.  Это позволяет клиентам на месте трафика для достижения HLI арендатора непосредственно без использования прокси-сервиса.  Это соединение больше не требуется при отсутствии единицы HLI после миграции.  Как и в случае прокси-сервиса IPTables, GlobalReach также должен храниться до полного вывода из эксплуатации лезвия HLI.

### <a name="operating-system-subscription--movereuse"></a>Подписка на операционную систему - перемещение/повторное использование
Поскольку VM-серверы выстраиваются в растяжение и лезвия HLI списаны, подписки на ОС можно заменить или повторно использовать, чтобы избежать двойной оплаты лицензий ОС.
