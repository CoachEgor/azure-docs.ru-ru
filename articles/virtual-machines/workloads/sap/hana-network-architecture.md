# <a name="sap-hana-large-instances-network-architecture"></a>Сетевая архитектура SAP HANA в Azure (крупные экземпляры)

Архитектура сетевых служб Azure является ключевым компонентом для успешного развертывания приложений SAP в крупных экземплярах HANA. Как правило, развернутое решение "SAP HANA в Azure (крупные экземпляры)" отличается большим ландшафтом SAP, который включает в себя несколько разных решений SAP с различными размерами баз данных, а также показателями потребления ресурсов ЦП и памяти. Вполне вероятно, что не все ИТ-системы уже находятся в Azure. Ваш ландшафт SAP часто также является гибридным с точки dBMS и точки зрения применения SAP, используя смесь NetWeaver, и S/4HANA и SAP HANA и других DBMS. Azure предлагает различные службы, которые позволяют запускать различные системы DBMS, NetWeaver и S/4HANA в Azure. Azure также предлагает вам сетевую технологию, чтобы сделать Azure похожим на виртуальный центр обработки данных для развертывания программного обеспечения

Если все ИТ-системы не размещаются в Azure. Функциональность сети Azure используется для подключения к миру помещений с вашими активами Azure, чтобы Сделать Azure похожим на ваш виртуальный центр обработки данных. Используемая функциональность сети Azure: 

- Виртуальные сети Azure подключены к схеме [ExpressRoute,](https://azure.microsoft.com/services/expressroute/) которая подключается к вашим сетевым ресурсам.
- Схема ExpressRoute, которая соединяет помещение с Azure, должна иметь минимальную пропускную способность [1 Гбит/с или выше.](https://azure.microsoft.com/pricing/details/expressroute/) Это позволяет обеспечить достаточную пропускную способность для передачи данных между локальными системами и системами, запущенными на виртуальных машинах, а также подключение к системам Azure из локальной среды конечных пользователей.
- Все системы SAP в Azure настроены в виртуальных сетях для связи друг с другом.
- Active Directory и DNS, размещенные на базе, расширяются в Azure через ExpressRoute из находящихся на месте или работают в Azure.

Для конкретного случая интеграции HANA Large Instances в структуру сети центров обработки данных Azure также используется технология Azure ExpressRoute


> [!NOTE] 
> Только одна подписка Azure может быть связана только с одним арендатором в марке HANA Large Instance в определенном регионе Azure. И наоборот, один арендатор штампа HANA Large Instance может быть связан только с одной подпиской Azure. Этот факт не отличается от любых других объектов в Azure, которые подлежат оплате.

Если SAP HANA на Azure (Большие инстанции) развернут в нескольких различных регионах Azure, в штампе HANA Large Instance разворачивается отдельный арендатор. Оба клиента могут выполняться в одной подписке Azure, если эти экземпляры являются частью одного ландшафта SAP. 

> [!IMPORTANT] 
> Только метод развертывания диспетчера ресурсов Azure поддерживается SAP HANA в Azure (Большие инстанции).

## <a name="networking-architecture-for-hana-large-instance"></a>Архитектура сети для крупных экземпляров HANA
Архитектуру сети для крупных экземпляров HANA можно разделить на четыре части:

- Локальная сеть и подключение ExpressRoute к Azure. Эта часть является доменом пользователей и подключается к Azure через ExpressRoute. Эта схема Expressroute полностью оплачивается вами как клиентом. Пропускная способность должна быть достаточно большой, чтобы обрабатывать сетевой трафик между вашими активами и областью Azure, против чего вы подключены. См. правый нижний угол на схеме ниже.
- Сетевые службы Azure, как уже говорилось ранее, с виртуальными сетями, которые снова нуждаются в добавленных шлюзах ExpressRoute. В этой области необходимо определить подходящую структуру в соответствии с требованиями приложений, безопасности и нормативных требований. При использовании крупных экземпляров HANA также необходимо выбрать число виртуальных сетей и SKU шлюзов Azure. См. правый верхний угол на схеме.
- Подключение крупных экземпляров HANA к Azure с помощью технологии ExpressRoute. Эту часть развертывает и обслуживает корпорация Майкрософт. Вам достаточно предоставить диапазоны IP-адресов и после развертывания ресурсов в крупных экземплярах HANA подключить канал ExpressRoute к виртуальным сетям. Для получения дополнительной информации см. [SAP HANA (Большие инстанции) инфраструктуры и подключения на Azure](hana-overview-infrastructure-connectivity.md?toc=%2fazure%2fvirtual-machines%2flinux%2ftoc.json). Для вас как клиента не взимается дополнительная плата за подключение между сетью центров обработки данных Azure и подразделениями HANA Large Instance.
- Сеть в HANA Большой штамп, который в основном прозрачный для вас.

![Виртуальная сеть подключена к SAP HANA в Azure (крупные экземпляры) и к локальной сети](./media/hana-overview-architecture/image1-architecture.png)

Использование крупных экземпляров HANA не отменяет требование подключать локальные ресурсы к Azure через ExpressRoute. Также требуется иметь одну или несколько виртуальных сетей с виртуальными машинами, на которых размещен уровень приложений, который подключается к экземплярам HANA, размещенным в единицах крупных экземпляров HANA. 

Отличия развертываний SAP только в Azure заключаются в следующем:

- Единицы крупных экземпляров HANA пользовательского клиента соединены через другой канал ExpressRoute с виртуальными сетями. Чтобы отделить условия загрузки, на территории виртуальной сети Azure ExpressRoute и схемы между виртуальными сетями Azure и HANA Large Instances не имеют одинаковых маршрутизаторов.
- Профиль рабочей нагрузки между уровнем приложений SAP и крупным экземпляром HANA является другой причиной возникновения множества небольших запросов и пиков передачи данных (результирующие наборы) между SAP HANA и уровнем приложений.
- Архитектура приложения SAP более чувствительна к задержкам в сети, чем в типичных сценариях, где обмен данными происходит между локальной средой и Azure.
- Шлюз Azure ExpressRoute имеет по крайней мере два подключения ExpressRoute. Одна схема, которая соединена с наемными помещениями, и одна, которая соединена с HANA Large Instances. Это оставляет только место для еще двух дополнительных схем из различных MSEEs для подключения к на ExpressRoute шлюз. Это ограничение не зависит от использования Скоростного маршрута ExpressRoute. Все связанные схемы имеют максимальную пропускную способность для входящих данных шлюза ExpressRoute.

С revision 3 марок HANA Large Instance задержка сети между VMs и ПОДРАЗДЕЛЕНИЯми HANA Large Instance может быть выше, чем типичная задержка сети VM-to-VM. В зависимости от региона Azure значения задержки приема-передачи могут превышать 0,7 мс. Такие значения классифицируются как значения ниже среднего в [примечании к SAP № 1100926 с часто задаваемыми вопросами к производительности сети](https://launchpad.support.sap.com/#/notes/1100926/E). Измеренная задержка может достигать до 2 миллисекунд. Этот показатель зависит от региона Azure и средства измерения задержки приема-передачи в сети между виртуальной машиной Azure и единицей крупного экземпляра HANA. Тем не менее клиенты успешно развертывают производственные приложения SAP на основе SAP HANA в крупных экземплярах SAP HANA. Следует тщательно протестировать бизнес-процессы в крупных экземплярах HANA в Azure. Новая функциональность, называемая ExpressRoute Fast Path, позволяет существенно снизить задержку сети между КРУПНЫМи экземплярами HANA и VMs-технологиями для приложений в Azure (см. ниже). 

С revision 4 марок HANA Large Instance задержка сети между VMs Azure, которые развернуты в непосредственной близости от штампа HANA Large Instance, имеет опыт для удовлетворения средней или лучшей, чем средняя классификация, как описано в [SAP Note #1100926 - часто задаваемые вопросы: Производительность сети,](https://launchpad.support.sap.com/#/notes/1100926/E) если Azure ExpressRoute Fast Path настроен (см. ниже). Для развертывания VMs Azure в непосредственной близости от подразделений HANA Large Instance от Revision 4 необходимо использовать [группы размещения proximity Azure.](https://docs.microsoft.com/azure/virtual-machines/linux/co-location) Способ использования групп размещения близости для определения местообнаружения слоя приложения SAP в том же центре обработки данных Azure, что и блоки «Редакция 4», размещенные в группах по размещению частой связи [Azure, для оптимальной задержки сети с помощью приложений SAP.](sap-proximity-placement-scenarios.md)

Для обеспечения детерминированной задержки сети между VMs и HANA Large Instance, выбор шлюза ExpressRoute SKU имеет важное значение. В отличие от шаблонов трафика между локальными и виртуальными машинами, шаблон трафика между виртуальными машинами и крупными экземплярами HANA может показывать короткий, но высокий пик запросов и объемов передаваемых данных. Для нормальной обработки таких пиков рекомендуется использовать шлюз с номером SKU UltraPerformance. Для класса HANA Large Instance SKUs класса HANA Large Instance использование шлюза UltraPerformance в качестве шлюза ExpressRoute является обязательным.

> [!IMPORTANT] 
> Учитывая общий сетевой трафик между уровнями базы данных и приложений SAP, для подключения виртуальных сетей к SAP HANA в Azure (крупные экземпляры) поддерживается только шлюз с номером SKU HighPerformance или UltraPerformance. Для SKUs типа II HANA, только шлюз UltraPerformance SKU поддерживается как шлюз ExpressRoute. Исключения применяются при использовании Скоростного маршрута ExpressRoute (см. ниже)

### <a name="expressroute-fast-path"></a>ЭкспрессРут Быстрый путь
Чтобы снизить задержку, ExpressRoute Fast Path был представлен и выпущен в мае 2019 года для конкретного подключения HANA Large Instances к виртуальным сетям Azure, в которых размещено приложение SAP VMs. Основное отличие решения, развернутого до сих пор, заключается в том, что потоки данных между ВМ и HANA Large Instances больше не направляются через шлюз ExpressRoute. Вместо этого виртуальные вывеши, назначенные в подсети (ы) виртуальной сети Azure, напрямую взаимодействуют со выделенным маршрутизатором края предприятия. 

> [!IMPORTANT] 
> Функциональность ExpressRoute Fast Path требует, чтобы подсети под управлением M-приложений SAP были в той же виртуальной сети Azure, которая подключилась к HANA Large Instances. Виртуальные виртуальные сети Azure, которые всматриваются в виртуальную сеть Azure, подключенную непосредственно к подразделениям HANA Large Instance, не получают выгоды от ExpressRoute Fast Path. В результате типичный концентратор и говорил виртуальные сетевые проекты, где схемы ExpressRoute соединяются с концентратором виртуальной сети и виртуальные сети, содержащие слой приложения SAP (спицы) получают заглянул, оптимизация ExpressRoute Быстро Путь не будет работать. В добавлении ExpressRoute Fast Path не поддерживает правила маршрутирования (UDR) пользователя сегодня. Для получения дополнительной [ExpressRoute virtual network gateway and FastPath](https://docs.microsoft.com/azure/expressroute/expressroute-about-virtual-network-gateways)информации см. 


Для получения более подробной информации о том, как настроить ExpressRoute Fast Path, прочитайте документ [Подключите виртуальную сеть к крупным экземплярам HANA](https://docs.microsoft.com/azure/virtual-machines/workloads/sap/hana-connect-vnet-express-route).    

> [!NOTE]
> Шлюз UltraPerformance ExpressRoute необходим для работы с ExpressRoute Fast Path


## <a name="single-sap-system"></a>Одна система SAP

Локальная инфраструктура, показанная выше, подключена к Azure через канал ExpressRoute. Схема ExpressRoute подключается к маршрутизатору корпоративного края корпорации Майкрософт (MSEE). Дополнительные сведения см. в статье [Обзор ExpressRoute](../../../expressroute/expressroute-introduction.md?toc=%2fazure%2fvirtual-machines%2flinux%2ftoc.json). После того, как маршрут установлен, он соединяется с позвоночником Azure.

> [!NOTE] 
> Для работы с ландшафтом SAP в Azure выполните подключение к маршрутизатору MSEE, который расположен ближе других к региону Azure в пределах ландшафта SAP. Марки HANA Large Instance подключаются к специализированным устройствам маршрутизатора enterprise edge, чтобы свести к минимуму задержку сети между ВМ в Марках Azure IaaS и HANA Large Instance.

Шлюз ExpressRoute для VMs, в которых размещаются экземпляры приложений SAP, подключен к одной схеме ExpressRoute, которая подключается к наместным. Та же виртуальная сеть подключена к отдельному маршрутизатору MSEE, предназначенному для подключения к стекам крупных экземпляров. Используя ExpressRoute Fast Path, поток данных от HANA Large Instances к VMs-налажено по слою приложений SAP больше не переправляется через шлюз ExpressRoute, а с этим уменьшает задержку в сети туда и обратно.

Это простой пример отдельной системы SAP. В ней уровень приложений SAP размещен в Azure. База данных SAP HANA выполняется в SAP HANA в Azure (крупные экземпляры). Предположение состоит в том, что пропускная способность шлюза ExpressRoute 2-Gbps или 10-Gbps пропускной способности не представляет собой узкое место.

## <a name="multiple-sap-systems-or-large-sap-systems"></a>Несколько обычных или крупных систем SAP

Если для подключения к SAP HANA (большие экземпляры) развернуты несколько систем SAP или большие системы SAP, пропускная связь шлюза ExpressRoute может стать узким местом. Или вы хотите изолировать производственные и непроизводственные системы в различных виртуальных сетях Azure. В этом случае необходимо разбить уровни приложений на несколько виртуальных сетей. Также рекомендуется создать специальные виртуальные сети, которые подключаются к крупным экземплярам HANA в таких случаях:

- Выполнение архивации непосредственно из экземпляров HANA в крупных экземплярах HANA на виртуальную машину в Azure, где размещаются общие ресурсы NFS.
- Копирование больших резервных копий и других файлов из крупных экземпляров HANA на диск, управляемый в Azure.

Используйте отдельную виртуальную сеть для размещения виртуальных вымов, которые управляют хранением данных для массовой передачи данных между HANA Large Instances и Azure. Такое расположение позволяет избежать переноса большого файла или передачи данных из HANA Large Instance в Azure на шлюзе ExpressRoute, который обслуживает vMs-запуск слаженного приложения SAP. 

Для более масштабируемой сетевой архитектуры:

- Используйте несколько виртуальных сетей для одного большего уровня приложений SAP.
- Вместо объединения систем SAP из разных подсетей в одну виртуальную сеть разверните отдельную виртуальную сеть для каждой развернутой системы SAP.

  Более масштабируемая сетевая архитектура для SAP HANA в Azure (крупные экземпляры):

## <a name="routing-in-azure"></a>Маршрутизация в Azure

При развертывании по умолчанию для SAP HANA в Azure (большие инстанции) важны три соображения сетевой развязывания.

* Доступ к SAP HANA на Azure (Большие инстанции) можно получить только через Azure VMs и выделенное соединение ExpressRoute, а не непосредственно из предварительных. Прямой доступ из локальной среды к единицам крупных экземпляров HANA, поставляемым корпорацией Майкрософт, сразу невозможен. Это ограничения переходной маршрутизации текущей архитектуры сети Azure, используемой для крупных экземпляров SAP HANA. Таким образом, клиенты администрирования и любые приложения, требующие прямого доступа, например запущенный локально SAP Solution Manager, не могут подключиться к базе данных SAP HANA. Для исключений проверьте раздел "Прямая реауктора на крупные инстанции HANA".

* Если у вас есть подразделения HANA Large Instance, развернутые в двух разных регионах Azure для аварийного восстановления, те же временные ограничения на скорость, применявшиеся в прошлом. Другими словами, IP-адреса подразделения HANA Large Instance в одном регионе (например, West) не были направлены в подразделение HANA Large Instance, развернутое в другом регионе (например, на Востоке США). Это ограничение было независимо от использования сети Azure, вглядывающейся в регионы или кросс-соединения схем ExpressRoute, которые соединяют подразделения HANA Large Instance с виртуальными сетями. См. рисунок в разделе "Использование единиц крупных экземпляров HANA в нескольких регионах". Это ограничение, которое пришло с развернутой архитектурой, запрещало немедленное использование репликации системы HANA в качестве функции аварийного восстановления. Для недавних изменений ищите раздел «Использование единиц крупной инстанции HANA в нескольких регионах». 

* Подразделения SAP HANA на Azure (Большие инстанции) имеют назначенный IP-адрес из диапазона адресов IP-адреса сервера, который вы отправили при запросе развертывания HANA Large Instance. Для получения дополнительной информации см. [SAP HANA (Большие инстанции) инфраструктуры и подключения на Azure](hana-overview-infrastructure-connectivity.md?toc=%2fazure%2fvirtual-machines%2flinux%2ftoc.json). Этот IP-адрес доступен через подписки и схемы Azure, которые соединяют виртуальные сети Azure с HANA Large Instances. IP-адрес, назначенный из этого диапазона IP-адресов пула серверов, напрямую присваивается аппаратной единице. NAT больше *не* используется, как это имело место в первых развертываниях этого решения. 

### <a name="direct-routing-to-hana-large-instances"></a>Прямая перенаправление в крупные инстанции HANA

По умолчанию переходная маршрутизирующая маршрутизм не работает в следующих сценариях:

* Между подразделениями HANA Large Instance и развертыванием на территории.

* Между routing HANA Large Instance, которые развернуты в двух разных регионах.

Существует три способа обеспечения транзитной маршрутизании в этих сценариях:

- Обратный прокси-сервер для маршрутизации входных и выходных данных. Например, F5 BIG-IP, NGINX с диспетчером трафика, развернутые в виртуальной сети Azure, которая подключается к HANA Large Instances и в качестве виртуального решения для разгрома брандмауэра/трафика.
- Используйте [правила IPTables](http://www.linuxhomenetworking.com/wiki/index.php/Quick_HOWTO_%3a_Ch14_%3a_Linux_Firewalls_Using_iptables#.Wkv6tI3rtaQ) на виртуальной машине Linux для маршрутизации между локальным расположением и единицами HANA (крупные экземпляры) или между единицами HANA (крупные экземпляры) в разных регионах. VM под управлением IPTables необходимо развертывать в виртуальной сети Azure, которая подключается к HANA Large Instances и к наместным. VM должен быть размер соответствующим образом, так что пропускная информация сети VM достаточно для ожидаемого сетевого трафика. Для получения подробной информации о пропускной способности сети VM, проверьте статью [Размеры Linux виртуальных машин в Azure](https://docs.microsoft.com/azure/virtual-machines/linux/sizes?toc=%2fazure%2fvirtual-network%2ftoc.json).
- Еще одним решением для обеспечения прямого трафика между подразделениями на территории и HANA Large instance будет [брандмауэр Azure Firewall.](https://azure.microsoft.com/services/azure-firewall/) 

Весь трафик этих решений будет направляться через виртуальную сеть Azure и как таковой трафик может быть дополнительно ограничен мягкими приборами, используемыми или группами безопасности сети Azure, так что некоторые IP-адреса или IP-адреса варьируются от на-закрытых может быть заблокирован или явно разрешен доступ к HANA крупных инстанций. 

> [!NOTE]  
> Имейте в виду, что внедрение и поддержка пользовательских решений с использованием сетевых устройств сторонних производителей или утилиты IPTables не предоставляется корпорацией Майкрософт. Поддержка должна предоставляться поставщиком используемого компонента или интегратором. 

#### <a name="express-route-global-reach"></a>Экспресс-маршрут Глобальный охват
Корпорация Майкрософт представила новую функциональность под названием [ExpressRoute Global Reach.](https://docs.microsoft.com/azure/expressroute/expressroute-global-reach) Глобальный охват может быть использован для крупных инстанций HANA в двух сценариях:

- Включить прямой доступ из помещений к вашим подразделениям HANA Large Instance, развернутым в различных регионах
- Включить прямую связь между подразделениями HANA Large Instance, развернутыми в различных регионах


##### <a name="direct-access-from-on-premises"></a>Прямой доступ из помещений
В регионах Azure, где предлагается Global Reach, можно запросить включение функциональности Global Reach для вашей схемы ExpressRoute, которая соединяет вашу локта с виртуальной сетью Azure, которая также подключается к вашим подразделениям HANA Large Instance. Есть некоторые последствия для предварительной стороне вашей схемы ExpressRoute. Для цен, проверить цены на [Global Reach Add-On](https://azure.microsoft.com/pricing/details/expressroute/). Дополнительные расходы для вас, связанные с схемой, которая соединяет блок HANA Large Instance (ы) с Azure, не связаны с ней. 

> [!IMPORTANT]  
> В случае использования Global Reach для обеспечения прямого доступа между подразделениями HANA Large Instance и корпоративными активами сетевой поток данных и управления **не передается через виртуальные сети Azure,** а непосредственно между маршрутизаторами обмена корпоративных устройств Майкрософт. В результате любые правила NSG или ASG, или любой тип брандмауэра, NVA или прокси, развернутые в виртуальной сети Azure, не затрагиваются. **Если вы используете ExpressRoute Global Reach для обеспечения прямого доступа из находиный HANA Большой экземпляр единиц ограничения и разрешения на доступ к HANA больших единиц экземпляра должны быть определены в брандмауэрах на стороне на месте** 

##### <a name="connecting-hana-large-instances-in-different-azure-regions"></a>Подключение крупных экземпляров HANA в различных регионах Azure
Точно так же, как ExpressRoute Global Reach может быть использован для подключения на территории к подразделениям HANA Large Instance, он может быть использован для подключения двух арендаторов HANA Large Instance, которые развернуты для вас в двух разных регионах. Изоляция — это схемы ExpressRoute, которые ваши арендаторы HANA Large Instance используют для подключения к Azure в обоих регионах. Дополнительные сборы за подключение двух арендаторов HANA Large Instance, развернутых в двух разных регионах, не взимается. 

> [!IMPORTANT]  
> Поток данных и контрольный поток сетевого трафика между различными арендаторами HANA Large не будут направляться через лазурные сети. В результате вы не можете использовать функциональность Azure или NVA для обеспечения соблюдения ограничений на связь между двумя арендаторами HANA Large Instances. 

Для получения более подробной информации о том, как получить ExpressRoute Global Reach включен, прочитайте документ [Подключите виртуальную сеть к HANA больших экземпляров](https://docs.microsoft.com/azure/virtual-machines/workloads/sap/hana-connect-vnet-express-route).


## <a name="internet-connectivity-of-hana-large-instance"></a>Подключение крупных экземпляров HANA к Интернету
Крупные экземпляры HANA *не* подключены к Интернету напрямую. Это ограничивает ваши возможности, например зарегистрировать образ ОС напрямую с использованием поставщика ОС. Вам может потребоваться поработать с локальным сервером средства для управления подписками SUSE Linux Enterprise Server или Red Hat Enterprise Linux Subscription Manager.

## <a name="data-encryption-between-vms-and-hana-large-instance"></a>Шифрование данных между виртуальными машинами и крупными экземплярами HANA
Данные, передаваемые между виртуальными машинами и крупными экземплярами HANA, не шифруются. Однако исключительно для обмена между СУБД HANA и приложениями на основе JDBC/ODBC вы можете включить шифрование трафика. Дополнительные сведения см. в этой [документации по SAP](http://help-legacy.sap.com/saphelp_hanaplatform/helpdata/en/db/d3d887bb571014bf05ca887f897b99/content.htm?frameset=/en/dd/a2ae94bb571014a48fc3b22f8e919e/frameset.htm&current_toc=/en/de/ec02ebbb57101483bdf3194c301d2e/plain.htm&node_id=20&show_children=false).

## <a name="use-hana-large-instance-units-in-multiple-regions"></a>Использование единиц крупных экземпляров HANA в нескольких регионах

Для реализации моментов аварийного восстановления необходимо иметь подразделения SHANA Large Instance в нескольких регионах Azure. Даже при использовании Azure (Global Vnet Peering) переходная маршрутизм по умолчанию не работает между арендаторами HANA Large Instance в двух разных регионах. Тем не менее, Global Reach открывает путь связи между подразделениями HANA Large Instance, которые вы создали в двух разных регионах. Этот сценарий использования ExpressRoute Global Reach позволяет:

 - Репликация системы HANA без каких-либо дополнительных прокси или брандмауэров
 - Копирование резервных копий между подразделениями HANA Large Instance в двух разных регионах для выполнения системных копий или обновления системы


![Виртуальные сети, подключенные к стекам крупных экземпляров Azure в разных регионах Azure](./media/hana-overview-architecture/image8-multiple-regions.png)

На рисунке показано, как различные виртуальные сети в обоих регионах соединены с двумя различными схемами ExpressRoute, которые используются для подключения к SAP HANA на Azure (Большие инстанции) в обоих регионах Azure (серые линии). Причина этого двух перекрестных соединений заключается в защите от простоя МСП с обеих сторон. Поток связи между двумя виртуальными сетями в двух регионах Azure предполагается обрабатывать через [глобальное пиринг](https://blogs.msdn.microsoft.com/azureedu/2018/04/24/how-to-setup-global-vnet-peering-in-azure/) двух виртуальных сетей в двух разных регионах (голубая пунктирная линия). Толстая красная линия описывает соединение ExpressRoute Global Reach, которое позволяет подразделениям HANA Large Instance ваших арендаторов в двух разных регионах общаться друг с другом. 

> [!IMPORTANT] 
> Если используется несколько каналов ExpressRoute, чтобы обеспечить правильную маршрутизацию трафика, следует использовать атрибут AS PATH и настройки локального предпочтения BGP.
