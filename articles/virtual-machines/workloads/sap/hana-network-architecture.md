# <a name="sap-hana-large-instances-network-architecture"></a>Сетевая архитектура SAP HANA в Azure (крупные экземпляры)

Архитектура сетевых служб Azure является ключевым компонентом для успешного развертывания приложений SAP в крупных экземплярах HANA. Как правило, развернутое решение "SAP HANA в Azure (крупные экземпляры)" отличается большим ландшафтом SAP, который включает в себя несколько разных решений SAP с различными размерами баз данных, а также показателями потребления ресурсов ЦП и памяти. Вполне вероятно, что не все ИТ-систем, находятся в Azure уже. Ландшафт SAP чаще гибридного также из точки СУБД и SAP точки зрения приложения как NetWeaver и S/4HANA и SAP HANA и других СУБД. Azure предлагает разные службы, которые позволяют запускать разных систем СУБД, NetWeaver и S/4HANA в Azure. Azure также предлагает сетевые технологии, чтобы сделать поиск Azure, такие как виртуального центра обработки данных к развертываниям локального программного обеспечения

Если полный ИТ-систем в среде Azure. Функциональность Azure сети используется для подключения локальную среду с ресурсами Azure, чтобы сделать вид виртуального центра обработки данных ваше Azure. Функциональные возможности сети Azure, используемые является: 

- Виртуальные сети Azure подключены к [ExpressRoute](https://azure.microsoft.com/services/expressroute/) канала, который подключается к ресурсам в локальной сети.
- Канал ExpressRoute между локальной Azure должны иметь минимальной пропускной способности сети [1 Гбит/с или более поздней версии](https://azure.microsoft.com/pricing/details/expressroute/). Это позволяет обеспечить достаточную пропускную способность для передачи данных между локальными системами и системами, запущенными на виртуальных машинах, а также подключение к системам Azure из локальной среды конечных пользователей.
- Все системы SAP в Azure настроены в виртуальных сетях смогут взаимодействовать друг с другом.
- Active Directory и DNS, размещенным на предприятиях распространяются в Azure с помощью ExpressRoute между локальной или при запуске выполнения в Azure.

Для определенного случая интеграции крупных экземпляров HANA в структуре сети центра обработки данных Azure также используется технология Azure ExpressRoute


> [!NOTE] 
> Только одна подписка Azure можно связать только один клиент в стеке крупных экземпляров HANA в определенном регионе Azure. И наоборот один клиент в стеке крупных экземпляров HANA могут быть связаны только одной подписке Azure. Этот факт не отличается от любых других объектов в Azure, которые подлежат оплате.

Если SAP HANA в Azure (крупные экземпляры) развертывается в нескольких разных регионах Azure, другой клиент будет развернут в стеке крупных экземпляров HANA. Оба клиента могут выполняться в одной подписке Azure, если эти экземпляры являются частью одного ландшафта SAP. 

> [!IMPORTANT] 
> SAP HANA в Azure (крупные экземпляры) поддерживается только метод развертывания Azure Resource Manager.

## <a name="networking-architecture-for-hana-large-instance"></a>Архитектура сети для крупных экземпляров HANA
Архитектуру сети для крупных экземпляров HANA можно разделить на четыре части:

- Локальная сеть и подключение ExpressRoute к Azure. Эта часть является доменом пользователей и подключается к Azure через ExpressRoute. Полностью, этот канал Expressroute оплачивается по вы как клиент. Пропускная способность должна быть достаточной для обработки сетевого трафика между ресурсами в локальной и регион Azure, которому вы подключаетесь к. См. правый нижний угол на схеме ниже.
- Сетевые службы Azure, как описывалось ранее, с помощью виртуальных сетей, в которых снова требуется добавить шлюзов ExpressRoute. В этой области необходимо определить подходящую структуру в соответствии с требованиями приложений, безопасности и нормативных требований. При использовании крупных экземпляров HANA также необходимо выбрать число виртуальных сетей и SKU шлюзов Azure. См. правый верхний угол на схеме.
- Подключение крупных экземпляров HANA к Azure с помощью технологии ExpressRoute. Эту часть развертывает и обслуживает корпорация Майкрософт. Вам достаточно предоставить диапазоны IP-адресов и после развертывания ресурсов в крупных экземплярах HANA подключить канал ExpressRoute к виртуальным сетям. Дополнительные сведения см. в статье [Инфраструктура и возможности подключения SAP HANA в Azure (крупные экземпляры)](hana-overview-infrastructure-connectivity.md?toc=%2fazure%2fvirtual-machines%2flinux%2ftoc.json). Нет без дополнительной платы для вас как клиента для подключений между структуры сети центра данных Azure и единиц крупных экземпляров HANA.
- Работа с сетью в стеке крупных экземпляров HANA, которая по большей части прозрачна для вас.

![Виртуальная сеть подключена к SAP HANA в Azure (крупные экземпляры) и к локальной сети](./media/hana-overview-architecture/image1-architecture.png)

Использование крупных экземпляров HANA не отменяет требование подключать локальные ресурсы к Azure через ExpressRoute. Также требуется иметь одну или несколько виртуальных сетей с виртуальными машинами, на которых размещен уровень приложений, который подключается к экземплярам HANA, размещенным в единицах крупных экземпляров HANA. 

Отличия развертываний SAP только в Azure заключаются в следующем:

- Единицы крупных экземпляров HANA пользовательского клиента соединены через другой канал ExpressRoute с виртуальными сетями. Для разделения условий нагрузки, локальной для канала ExpressRoute виртуальной сети Azure, а также каналов между виртуальными сетями Azure и крупными экземплярами HANA не совместно используют те же маршрутизаторы.
- Профиль рабочей нагрузки между уровнем приложений SAP и крупным экземпляром HANA является другой причиной возникновения множества небольших запросов и пиков передачи данных (результирующие наборы) между SAP HANA и уровнем приложений.
- Архитектура приложения SAP более чувствительна к задержкам в сети, чем в типичных сценариях, где обмен данными происходит между локальной средой и Azure.
- Шлюз Azure ExpressRoute имеет по крайней мере два подключения ExpressRoute. Один канал, подключенный для локальных и который подключен из крупных экземпляров HANA. При этом остается только место для другого два дополнительных каналов из разных маршрутизаторов Msee для подключения к на шлюз ExpressRoute. Это ограничение зависит от использования ExpressRoute быстрого пути. Все подключенные каналы совместно использовать максимальную пропускную способность для входящих данных шлюза ExpressRoute.

Сетевая задержка между виртуальными машинами и единицами крупных экземпляров HANA может быть выше, чем обычная задержка приема-передачи между виртуальными машинами. Зависимости от региона Azure, значения могут превышать 0,7 мс задержка приема-передачи, классифицируются как значения ниже среднего в [1100926 # Примечание SAP - часто задаваемые вопросы: Производительность сети](https://launchpad.support.sap.com/#/notes/1100926/E). Измеренная задержка может достигать до 2 миллисекунд. Этот показатель зависит от региона Azure и средства измерения задержки приема-передачи в сети между виртуальной машиной Azure и единицей крупного экземпляра HANA. Тем не менее клиенты успешно развертывают производственные приложения SAP на основе SAP HANA в крупных экземплярах SAP HANA. Следует тщательно протестировать бизнес-процессы в крупных экземплярах HANA в Azure. Новые функциональные возможности, называется ExpressRoute быстрый путь может значительно уменьшить задержки в сети между крупных экземпляров HANA и уровнем приложений виртуальных машин в Azure (см. ниже). 

Чтобы обеспечить детерминированную сетевую задержку между виртуальными машинами и крупными экземплярами HANA, выбор шлюза ExpressRoute важно SKU. В отличие от шаблонов трафика между локальными и виртуальными машинами, шаблон трафика между виртуальными машинами и крупными экземплярами HANA может показывать короткий, но высокий пик запросов и объемов передаваемых данных. Для нормальной обработки таких пиков рекомендуется использовать шлюз с номером SKU UltraPerformance. Для номеров SKU крупных экземпляров HANA класса типа II использование номера SKU шлюза UltraPerformance в качестве ExpressRotue шлюз является обязательным.

> [!IMPORTANT] 
> Учитывая общий сетевой трафик между уровнями базы данных и приложений SAP, для подключения виртуальных сетей к SAP HANA в Azure (крупные экземпляры) поддерживается только шлюз с номером SKU HighPerformance или UltraPerformance. Для SKU крупных экземпляров HANA типа II поддерживается только номер SKU шлюза UltraPerformance в качестве шлюза ExpressRoute. Исключения при использовании ExpressRoute быстрый путь (см. ниже)

### <a name="expressroute-fast-path"></a>Путь ExpressRoute Fast
Чтобы уменьшить задержку, быстрый путь ExpressRoute появилась и выпущена в мая 2019 для определенного подключения крупных экземпляров HANA к виртуальным сетям Azure, на которых размещены виртуальные машины приложений SAP. — Основное отличие в решение, на данный момент распространен, что потоки данных между виртуальными машинами и крупными экземплярами HANA, не маршрутизируются через шлюз ExpressRoute больше. Вместо этого виртуальных машин, назначенных в подсетей виртуальной сети Azure напрямую взаимодействуют с выделенной маршрутизатору msee. 

> [!IMPORTANT] 
> Функциональность ExpressRoute быстрый путь необходимы подсетей, работающих виртуальных машин приложений SAP в Azure виртуальной сети есть подключения к крупным экземплярам HANA. Виртуальными машинами, размещенными в виртуальных сетях Azure, которые являются одноранговыми с помощью виртуальной сети Azure, подключенный непосредственно к единицам крупных экземпляров HANA не являются выгоду от быстрого пути ExpressRoute. В результате типичных звездообразной схем виртуальных сетей, где каналов ExpressRoute при подключении к центральной виртуальной сети, а получение установлена пиринговая связь виртуальной сети, в которых уровень приложений SAP (периферийные зоны), оптимизацию путем ExpressRoute Fast Путь не будет работать. Кроме того быстрый путь ExpressRoute не поддерживает правила маршрутизации определяемые пользователем (UDR) уже сегодня. Дополнительные сведения см. в разделе [ExpressRoute виртуальной сети, шлюза и FastPath](https://docs.microsoft.com/azure/expressroute/expressroute-about-virtual-network-gateways). 


Дополнительные сведения о настройке ExpressRoute быстрого пути, см. в документе [подключение виртуальной сети к крупным экземплярам HANA](https://docs.microsoft.com/azure/virtual-machines/workloads/sap/hana-connect-vnet-express-route).    

> [!NOTE]
> Шлюз UltraPerformance ExpressRoute может иметь рабочий ExpressRoute быстрый путь


## <a name="single-sap-system"></a>Одна система SAP

Локальная инфраструктура, показанная выше, подключена к Azure через канал ExpressRoute. Канал ExpressRoute подключается в маршрутизатором Microsoft enterprise edge router (MSEE). Дополнительные сведения см. в статье [Обзор ExpressRoute](../../../expressroute/expressroute-introduction.md?toc=%2fazure%2fvirtual-machines%2flinux%2ftoc.json). После установки этот маршрут, он подключается к магистральной сети Azure.

> [!NOTE] 
> Для работы с ландшафтом SAP в Azure выполните подключение к маршрутизатору MSEE, который расположен ближе других к региону Azure в пределах ландшафта SAP. Стеки крупных экземпляров HANA подключены через выделенный устройств msee для минимизации сетевой задержки между виртуальными машинами в Azure IaaS и стеками крупных экземпляров HANA.

Для одного канала ExpressRoute, который подключается к локальным подключен шлюз ExpressRoute для виртуальных машин, на которых размещены экземпляры приложений SAP. Та же виртуальная сеть подключена к отдельному маршрутизатору MSEE, предназначенному для подключения к стекам крупных экземпляров. С помощью ExpressRoute быстрого пути, поток данных из крупных экземпляров HANA на уровень приложения SAP виртуальные машины, не маршрутизируются через шлюз ExpressRoute больше и, снизить сети задержка приема-передачи.

Это простой пример отдельной системы SAP. В ней уровень приложений SAP размещен в Azure. База данных SAP HANA выполняется в SAP HANA в Azure (крупные экземпляры). Предполагается, что пропускная способность шлюза ExpressRoute, 2 Гбит/с или 10 Гбит/с пропускной способности не подразумевает узкое.

## <a name="multiple-sap-systems-or-large-sap-systems"></a>Несколько обычных или крупных систем SAP

Если несколько обычных или крупных систем SAP развернуты для подключения к SAP HANA в Azure (крупные экземпляры), пропускная способность шлюза ExpressRoute может стать узким местом. Или вы хотите изолировать производственных и непроизводственных систем, в разных виртуальных сетях Azure. В этом случае необходимо разбить уровни приложений на несколько виртуальных сетей. Также рекомендуется создать специальные виртуальные сети, которые подключаются к крупным экземплярам HANA в таких случаях:

- Выполнение архивации непосредственно из экземпляров HANA в крупных экземплярах HANA на виртуальную машину в Azure, где размещаются общие ресурсы NFS.
- Копирование больших резервных копий и других файлов из крупных экземпляров HANA на диск, управляемый в Azure.

Используйте отдельную виртуальную сеть для размещения виртуальных машин, управление хранилищем для массовой передачи данных между крупных экземпляров HANA и Azure. Такой подход не повлияет на работу больших файлов или передачу данных из крупных экземпляров HANA в Azure на шлюзе ExpressRoute, обслуживающем виртуальные машины под управлением уровень приложений SAP. 

Для более масштабируемой сетевой архитектуры:

- Используйте несколько виртуальных сетей для одного большего уровня приложений SAP.
- Вместо объединения систем SAP из разных подсетей в одну виртуальную сеть разверните отдельную виртуальную сеть для каждой развернутой системы SAP.

  Более масштабируемая сетевая архитектура для SAP HANA в Azure (крупные экземпляры):

![Развертывание уровня приложений SAP в нескольких виртуальных сетях](./media/hana-overview-architecture/image4-networking-architecture.png)

Зависит от правила и ограничения, необходимо применить между разными виртуальными сетях, размещение виртуальных машин из разных систем SAP, следует создать пиринг между этими виртуальными сетями. Дополнительные сведения о пиринге виртуальных сетей см. в [этой статье](https://docs.microsoft.com/azure/virtual-network/virtual-network-peering-overview).


## <a name="routing-in-azure"></a>Маршрутизация в Azure

При стандартном развертывании три рекомендации относительно сетевой маршрутизации, очень важны для SAP HANA в Azure (крупные экземпляры):

* SAP HANA в Azure (крупные экземпляры) может осуществляться только через виртуальные машины Azure и выделенное подключение ExpressRoute, не напрямую из локальной. Прямой доступ из локальной среды к единицам крупных экземпляров HANA, поставляемым корпорацией Майкрософт, сразу невозможен. Это ограничения переходной маршрутизации текущей архитектуры сети Azure, используемой для крупных экземпляров SAP HANA. Таким образом, клиенты администрирования и любые приложения, требующие прямого доступа, например запущенный локально SAP Solution Manager, не могут подключиться к базе данных SAP HANA. Исключения см. в разделе «Прямая Маршрутизация к крупным экземплярам HANA».

* Если у вас есть единиц крупных экземпляров HANA, развернутых в двух разных регионах Azure для аварийного восстановления, те же маршрутизации ограничения на переходные применены ранее. Другими словами IP-адреса единицы крупного экземпляра HANA в одном регионе (например, Западная часть США), не были направляться в крупных экземпляров HANA, развернутый в другом регионе (например, восточная часть США). Это ограничение было зависит от использования сети Azure, пиринг по регионам или перекрестных соединений каналов ExpressRoute, которые подключаются к виртуальным сетям единицах крупных экземпляров HANA. См. рисунок в разделе "Использование единиц крупных экземпляров HANA в нескольких регионах". Это ограничение, который поставляется с развернутой архитектуры, запрещено немедленно использовать функцию репликации системы HANA как функцию аварийного восстановления. Последние изменения см. раздел «Единиц крупных экземпляров HANA используется в нескольких регионах». 

* SAP HANA в Azure (крупные экземпляры) иметь назначенный IP-адрес из диапазона адресов пула IP-адрес сервера, отправленного при запросе развертывания крупных экземпляров HANA. Дополнительные сведения см. в статье [Инфраструктура и возможности подключения SAP HANA в Azure (крупные экземпляры)](hana-overview-infrastructure-connectivity.md?toc=%2fazure%2fvirtual-machines%2flinux%2ftoc.json). Этот IP-адрес доступен через подписки Azure и канала, который подключает виртуальные сети Azure к крупным экземплярам HANA. IP-адрес, назначенный из этого диапазона IP-адресов пула серверов, напрямую присваивается аппаратной единице. NAT больше *не* используется, как это имело место в первых развертываниях этого решения. 

### <a name="direct-routing-to-hana-large-instances"></a>Прямая Маршрутизация к крупным экземплярам HANA
По умолчанию транзитивной маршрутизацией между единиц крупных экземпляров HANA и на локальном или крупных экземпляров HANA маршрутизации, которые развертываются в двух разных регионах не работает. Возможны несколько причин для включения таких транзитивной маршрутизацией.

- Обратный прокси-сервер для маршрутизации входных и выходных данных. Например F5 BIG-IP, NGINX с диспетчером трафика, развернутый в виртуальной сети Azure, подключается к крупным экземплярам HANA и к локальным как решение для маршрутизации трафика/виртуального брандмауэра.
- Используйте [правила IPTables](http://www.linuxhomenetworking.com/wiki/index.php/Quick_HOWTO_%3a_Ch14_%3a_Linux_Firewalls_Using_iptables#.Wkv6tI3rtaQ) на виртуальной машине Linux для маршрутизации между локальным расположением и единицами HANA (крупные экземпляры) или между единицами HANA (крупные экземпляры) в разных регионах. Виртуальная машина работала IPTables должен быть развернуты в виртуальной сети Azure, подключается к крупным экземплярам HANA, так и к локальным. Виртуальная машина должна быть размера соответствующим образом, так, что пропускная способность сети виртуальной машины достаточно для ожидаемой сетевого трафика. Сведения о виртуальной Машине пропускной способности сети, см. в статье [размеров для виртуальных машин в Azure](https://docs.microsoft.com/azure/virtual-machines/linux/sizes?toc=%2fazure%2fvirtual-network%2ftoc.json).
- [Брандмауэр Azure](https://azure.microsoft.com/services/azure-firewall/) было бы другое решение, чтобы включить передачу трафика между локальными и единицами крупных экземпляров HANA. 

Весь трафик из этих решений, будет направляться через виртуальную сеть Azure, и таким образом трафик может быть дополнительно ограничить, обратимо устройств, которые используются или групп безопасности сети Azure, таким образом, что определенные IP-адреса или IP-адрес в диапазоне от локально удалось заблокирован или разрешен доступ к крупным экземплярам HANA. 

> [!NOTE]  
> Имейте в виду, что внедрение и поддержка пользовательских решений с использованием сетевых устройств сторонних производителей или утилиты IPTables не предоставляется корпорацией Майкрософт. Поддержка должна предоставляться поставщиком используемого компонента или интегратором. 

#### <a name="express-route-global-reach"></a>Express Route в мире
Корпорация Майкрософт представила новый функциональность, которая называется [ExpressRoute глобальным доступом](https://docs.microsoft.com/azure/expressroute/expressroute-global-reach). В мире может использоваться для крупных экземпляров HANA в двух сценариях:

- Включить прямой доступ из локальной к к единицам крупных экземпляров HANA, развернутых в разных регионах
- Включить прямой обмен данными между к единицам крупных экземпляров HANA, развернутых в разных регионах


##### <a name="direct-access-from-on-premise"></a>Прямой доступ из локальной
В регионах Azure, где предлагается глобальным доступом вы можете запросить, включив функцию глобальным доступом для своего канала ExpressRoute, который подключается к виртуальной сети Azure, а также к единицам крупных экземпляров HANA, к локальной сети. Существуют некоторые ограничения затрат на локальной стороне канала ExpressRoute. См. цены, цены на [глобальных надстроек достичь](https://azure.microsoft.com/pricing/details/expressroute/). Существуют без дополнительных затрат для вас относящиеся к каналу, который подключается единицами крупных экземпляров HANA в Azure. 

> [!IMPORTANT]  
> В случае использования глобальным доступом для включения прямого доступа между локальными ресурсами и единиц крупных экземпляров HANA, является сетевой поток данных и управления **не маршрутизируются через виртуальные сети Azure**, но напрямую между Microsoft Корпоративные маршрутизаторы exchange. В результате все правила NSG и ASG или брандмауэра, виртуальный сетевой модуль или прокси-сервера, развернутый в виртуальной сети Azure, любого типа, не начало менялись. **Если вы используете ExpressRoute глобальным доступом включить прямой доступ из локальной для ограничения единиц HANA крупных экземпляра и должны быть определены в брандмауэрах на локальной стороне разрешения на доступ к единицам крупных экземпляров HANA** 

##### <a name="connecting-hana-large-instances-in-different-azure-regions"></a>Подключение крупных экземпляров HANA в разных регионах Azure
Так же как глобальным доступом ExpressRoute можно использовать для подключения локальной к единицам крупных экземпляров HANA, его можно использовать для подключения переход которые клиентов крупных экземпляров HANA, развертываются в двух разных регионах. Уровень изоляции — каналов ExpressRoute, которые клиенты крупных экземпляров HANA при использовании для подключения к Azure в обоих регионах. Отсутствуют дополнительные накладные расходы по подключению двух клиентов крупных экземпляров HANA, развернутых в двух разных регионах. 

> [!IMPORTANT]  
> Поток данных и поток управления сетевым трафиком между разных HANA крупных экземпляра клиентов не будут направляться через сети azure. В результате нельзя использовать функциональные возможности Azure или виртуальные сетевые устройства для принудительного применения ограничения связи между двумя клиентов крупных экземпляров HANA. 

Дополнительные сведения о том, как получить ExpressRoute глобальным доступом включен прочитать документ [подключение виртуальной сети к крупным экземплярам HANA](https://docs.microsoft.com/azure/virtual-machines/workloads/sap/hana-connect-vnet-express-route).


## <a name="internet-connectivity-of-hana-large-instance"></a>Подключение крупных экземпляров HANA к Интернету
Крупные экземпляры HANA *не* подключены к Интернету напрямую. Это ограничивает ваши возможности, например зарегистрировать образ ОС напрямую с использованием поставщика ОС. Вам может потребоваться поработать с локальным сервером средства для управления подписками SUSE Linux Enterprise Server или Red Hat Enterprise Linux Subscription Manager.

## <a name="data-encryption-between-vms-and-hana-large-instance"></a>Шифрование данных между виртуальными машинами и крупными экземплярами HANA
Данные, передаваемые между виртуальными машинами и крупными экземплярами HANA, не шифруются. Однако исключительно для обмена между СУБД HANA и приложениями на основе JDBC/ODBC вы можете включить шифрование трафика. Дополнительные сведения см. в этой [документации по SAP](http://help-legacy.sap.com/saphelp_hanaplatform/helpdata/en/db/d3d887bb571014bf05ca887f897b99/content.htm?frameset=/en/dd/a2ae94bb571014a48fc3b22f8e919e/frameset.htm&current_toc=/en/de/ec02ebbb57101483bdf3194c301d2e/plain.htm&node_id=20&show_children=false).

## <a name="use-hana-large-instance-units-in-multiple-regions"></a>Использование единиц крупных экземпляров HANA в нескольких регионах

Чтобы реализовать ИБП набор аварийного восстановления, необходимо иметь единиц крупных экземпляров SHANA в нескольких регионах Azure. Даже при использовании Azure [глобальный Пиринг] транзитивной маршрутизацией по умолчанию не работает между клиентами крупных экземпляров HANA в двух разных регионах. Тем не менее глобальным доступом открывает путь обмена данными между единиц крупных экземпляров HANA, который вы подготовили в двух разных регионах. Этот сценарий использования ExpressRoute глобальным доступом позволяет:

 - Репликация системы HANA без дополнительные прокси-серверы или брандмауэры
 - Копирование резервных копий между единицами крупных экземпляров HANA в двух разных регионах для выполнения копии системы или обновления системы


![Виртуальные сети, подключенные к стекам крупных экземпляров Azure в разных регионах Azure](./media/hana-overview-architecture/image8-multiple-regions.png)

Показаны как разные виртуальные сети в обоих регионах подключены к двум разным каналам ExpressRoute, которые используются для подключения к SAP HANA в Azure (крупные экземпляры) в обоих регионах Azure (серые линии). Причина этого два кросс-подключений является защита от сбоя для маршрутизаторов Msee на обеих сторонах. Поток связи между двумя виртуальными сетями в двух регионах Azure должен обрабатываться через [пиринга глобальной](https://blogs.msdn.microsoft.com/azureedu/2018/04/24/how-to-setup-global-vnet-peering-in-azure/) двух виртуальных сетей в двух разных регионах (синяя линия пунктирная). Толстая красная линия описывает подключение ExpressRoute глобальным доступом, которое позволяет клиентам возможность взаимодействовать друг с другом в двух разных регионах единицы крупных экземпляров HANA. 

> [!IMPORTANT] 
> Если используется несколько каналов ExpressRoute, чтобы обеспечить правильную маршрутизацию трафика, следует использовать атрибут AS PATH и настройки локального предпочтения BGP.
