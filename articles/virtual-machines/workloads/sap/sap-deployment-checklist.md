# <a name="sap-workloads-on-azure-planning-and-deployment-checklist"></a>Рабочие нагрузки SAP в Azure: контрольный список планирования и развертывания

Этот контрольный список предназначен для клиентов, перемещающих приложения SAP NetWeaver, S/4HANA и Hybris в инфраструктуру Azure в качестве службы. На протяжении всего проекта клиент и/или партнер SAP должны ознакомиться с контрольным списком. Важно отметить, что многие проверки завершены в начале проекта и на этапе планирования. После развертывания простые изменения в развернутой инфраструктуре Azure или выпусках программного обеспечения SAP могут стать сложными.

Просмотрите контрольный список на ключевых этапах проекта. Это позволит вам обнаружить небольшие проблемы, прежде чем они станут большими проблемами. У вас также будет достаточно времени, чтобы перестроиться и протестировать все необходимые изменения. Не считайте этот контрольный список полным. В зависимости от ситуации, возможно, потребуется выполнить еще много проверок.

Контрольный список не включает задачи, не зависят от Azure. Например, интерфейсы приложений SAP изменяются при переходе на платформу Azure или хостинг-провайдера.

Этот контрольный список также может быть использован для уже развернутых систем. Новые функции, такие как зоны записи accelerator и Availability, и новые типы VM, возможно, были добавлены после развертывания. Поэтому полезно периодически просматривать контрольный список, чтобы убедиться, что вы знаете о новых функциях в платформе Azure.

## <a name="project-preparation-and-planning-phase"></a>Этап подготовки проекта и планирования
На этом этапе планируется перенос рабочей нагрузки SAP на платформу Azure. Как минимум, на этом этапе необходимо создать следующие документы и определить и обсудить следующие элементы миграции:

1. Проектно-изыскваемый документ высокого уровня. Этот документ должен содержать:
    - Текущий перечень компонентов и приложений SAP и целевой инвентаризации приложений для Azure.
    - Матрица назначения ответственности (RACI), определяющая обязанности и назначения вовлеченных сторон. Начните с высокого уровня и работайте на более детальном уровне в рамках планирования и первых развертываний.
    - Архитектура решений высокого уровня.
    - Решение о том, в какие регионы Azure развертывать. Смотрите [список регионов Azure](https://azure.microsoft.com/global-infrastructure/regions/). Чтобы узнать, какие услуги доступны в каждом регионе, [см.](https://azure.microsoft.com/global-infrastructure/services/)
    - Сетевая архитектура для подключения от нахлок до Azure. Начните знакомиться с [планом виртуального центра обработки данных для Azure.](https://docs.microsoft.com/azure/architecture/vdc/)
    - Принципы безопасности для запуска высокоэффективных бизнес-данных в Azure. Чтобы узнать о безопасности данных, начните с [документации безопасности Azure.](https://docs.microsoft.com/azure/security/)
2.  Технический проектно-смесят документ. Этот документ должен содержать:
    - Диаграмма блока для разрешения.
    - Размер компонентов вычислений, хранения и сетей в Azure. Для sAP размеров Azure VMs [#1928533](https://launchpad.support.sap.com/#/notes/1928533)см.
    - Архитектура непрерывности бизнеса и аварийного восстановления.
    - Подробная информация о версиях пакетов пакетов поддержки Os, DB, ядра и SAP. Не обязательно, что каждый выпуск ОС, поддерживаемый SAP NetWeaver или S/4HANA, поддерживается на MMs Azure. То же самое относится и к релизам DBMS. Проверьте следующие источники для выравнивания и при необходимости обновления SAP-релизов, dBMS-релизов и выпусков ОС, чтобы обеспечить поддержку SAP и Azure. Для получения полной поддержки от SAP и Microsoft необходимо иметь комбинации выпусков, поддерживаемые SAP и Azure. При необходимости необходимо спланировать обновление некоторых компонентов программного обеспечения. Более подробная информация о поддерживаемом программном обеспечении SAP, OS и DBMS описана здесь:
        - [Примечание поддержки SAP #1928533](https://launchpad.support.sap.com/#/notes/1928533). Эта заметка определяет минимальные выпуски ОС, поддерживаемые на Azure VMs. Он также определяет минимальные релизы баз данных, необходимые для большинства баз данных, не связанных с HANA. Наконец, он предоставляет размер SAP для типов Azure VM, поддерживаемых SAP.
        - [Примечание поддержки SAP #2015553](https://launchpad.support.sap.com/#/notes/2015553). В этой заметке определены политики поддержки вокруг систем хранения данных Azure и отношения поддержки, необходимые корпорации Майкрософт.
        - [Заметка поддержки SAP #2039619](https://launchpad.support.sap.com/#/notes/2039619). Эта заметка определяет матрицу поддержки Oracle для Azure. Oracle поддерживает только Windows и Oracle Linux в качестве гостевых операционных систем на Azure для рабочих нагрузок SAP. Это заявление поддержки также применяется к слою приложения SAP, который выполняет экземпляры SAP. Тем не менее, Oracle не поддерживает высокую доступность SAP Central Services в Oracle Linux через Pacemaker. Если вам нужна высокая доступность ASCS на Oracle Linux, вам необходимо использовать SIOS Protection Suite для Linux. Для получения подробных данных о сертификации SAP [#1662610](https://launchpad.support.sap.com/#/notes/1662610)см. Для Windows поддерживаемое SAP решение для кластеризации Windows Server Failover для SAP Central Services поддерживается совместно с Oracle в качестве слоя DBMS.
        - [Примечание поддержки SAP #2235581](https://launchpad.support.sap.com/#/notes/2235581). Эта заметка предоставляет матрицу поддержки SAP HANA на различных версиях ОС.
        - На [веб-сайте SAP](https://www.sap.com/dmc/exp/2014-09-02-hana-hardware/enEN/iaas.html#categories=Microsoft%20Azure)перечислены поддерживаемые SAP Компании Azure VMs и [HANA Large Instances.](https://docs.microsoft.com/azure/virtual-machines/workloads/sap/hana-overview-architecture)
        - [Матрица доступности продуктов SAP](https://support.sap.com/en/).
        - [Заметка поддержки SAP #2555629 - SAP HANA 2.0 Динамическое многоуровнее - Гипервизор и поддержка облака](https://launchpad.support.sap.com/#/notes/2555629)
        - [Примечание поддержки SAP #1662610 - Подробная информация о поддержке sIOS Protection Suite для Linux](https://launchpad.support.sap.com/#/notes/1662610)
        - Примечания SAP для других продуктов SAP.     
    - Мы рекомендуем строгие трехуровневые конструкции для производственных систем SAP. Мы не рекомендуем объединять ASCS и/или DBMS и/или серверы приложений на одном VM. Использование конфигураций кластеров с несколькими СИД для SAP Central Services поддерживается на гостевых операционных системах Windows в Azure. Но эта конфигурация не поддерживается для SAP Central Services на операционных системах Linux в Azure. В следующих статьях можно найти документацию по сценарию гостевой ОС Windows:
        - [Обеспечение высокого уровня доступности с несколькими идентификаторами безопасности для экземпляра SAP ASCS/SCS с помощью отказоустойчивой кластеризации Windows Server и общего диска в Azure](https://docs.microsoft.com/azure/virtual-machines/workloads/sap/sap-ascs-ha-multi-sid-wsfc-shared-disk)
        - [Обеспечение высокого уровня доступности экземпляра ASCS/SCS с несколькими ИД безопасности с помощью отказоустойчивой кластеризации Windows Server и файлового ресурса в Azure](https://docs.microsoft.com/azure/virtual-machines/workloads/sap/sap-ascs-ha-multi-sid-wsfc-file-share)
    - Высокая доступность и архитектура аварийного восстановления.
        - На основе RTO и RPO определите, как должна выглядеть архитектура высокой доступности и аварийного восстановления.
        - Для высокой доступности в пределах зоны проверьте, что может предложить желаемый DBMS в Azure. Большинство пакетов DBMS предлагают синхронные методы синхронного горячего ожидания, которые мы рекомендуем для производственных систем. Также проверьте документацию, связанную с SAP, для различных баз данных, начиная с [рассмотрения развертывания DBMS виртуальных машин Azure для рабочих нагрузок SAP](https://docs.microsoft.com/azure/virtual-machines/workloads/sap/dbms_guide_general) и связанных с ними документов.
           Использование кластеризации Windows Server Failover с общей конфигурацией диска для слоя DBMS, как, например, [описано для S'L Server,](https://docs.microsoft.com/sql/sql-server/failover-clusters/windows/always-on-failover-cluster-instances-sql-server?view=sql-server-2017)не поддерживается. Вместо этого используйте такие решения, как:
           - [AlwaysOn в SQL Server](https://docs.microsoft.com/azure/virtual-machines/windows/sqlclassic/virtual-machines-windows-classic-ps-sql-alwayson-availability-groups)
           - [Oracle Data Guard](https://docs.microsoft.com/azure/virtual-machines/workloads/oracle/configure-oracle-dataguard)
           - [Репликация системы HANA](https://help.sap.com/viewer/6b94445c94ae495c83a19646e7c3fd56/2.0.01/en-US/b74e16a9e09541749a745f41246a065e.html)
        - Для аварийного восстановления в регионах Azure просмотрите решения, предлагаемые различными поставщиками DBMS. Большинство из них поддерживают асинхронную репликацию или доставку журналов.
        - На уровне приложений SAP определите, будете ли вы запускать системы тестирования бизнес-регрессии, которые в идеале являются копиями развертываний вашего производства, в том же регионе Azure или в регионе DR. Во втором случае можно настроить таргетинг на эту бизнес-регрессионную систему в качестве цели DR для развертывания производственных задач.
        - Если вы решили не размещать непроизводственные системы на сайте DR, загляните в восстановление сайта Azure как метод репликации слоя приложения SAP в область Azure DR. Для получения дополнительной [Set up disaster recovery for a multi-tier SAP NetWeaver app deployment](https://docs.microsoft.com/azure/site-recovery/site-recovery-sap)информации см.
        - Если вы решили использовать комбинированную конфигурацию HADR с помощью [зон доступности Azure,](https://docs.microsoft.com/azure/availability-zones/az-overview)ознакомьтесь с регионами Azure, где доступны зоны доступности. Также примите во внимание ограничения, которые могут быть введены в связи с увеличением просечения сети между двумя зонами доступности.  
3.  Перечень всех интерфейсов SAP (SAP и non-SAP).
4.  Проектирование базовых услуг. Эта конструкция должна включать в себя следующие элементы:
    - Активный дизайн каталога и DNS.
    - Топология сети в Azure и назначение различных систем SAP.
    - Структура [доступа на основе ролей](https://docs.microsoft.com/azure/role-based-access-control/overview) для групп, управляющих инфраструктурой и приложениями SAP в Azure.
    - Топология группы ресурсов.
    - [Стратегия метки](https://docs.microsoft.com/azure/azure-resource-manager/resource-group-using-tags#tags-and-billing).
    - Конвенции именования для ВМ и других компонентов инфраструктуры и/или логических названий.
5.  Контракт на поддержку Microsoft Premier. Определите свой технический менеджер учетной записи Майкрософт (TAM). Для требований поддержки SAP см [#2015553.](https://launchpad.support.sap.com/#/notes/2015553)
6.  Количество подписок Azure и основная квота для подписок. [Открытые запросы на поддержку для увеличения квот подписок Azure](https://docs.microsoft.com/azure/azure-portal/supportability/resource-manager-core-quotas-request) по мере необходимости.
7.  План сокращения данных и миграции данных для переноса данных SAP в Azure. Для систем SAP NetWeaver SAP имеет рекомендации по ограничению объема больших объемов данных. Ознакомьтесь с [этим руководством SAP](https://wiki.scn.sap.com/wiki/download/attachments/247399467/DVM_%20Guide_7.2.pdf?version=1&modificationDate=1549365516000&api=v2) по управлению данными в системах SAP ERP. Некоторые из содержимого также относится к NetWeaver и S/4HANA систем в целом.
8.  Автоматизированный подход к развертыванию. Цель автоматизации развертывания инфраструктуры на Azure заключается в развертывании детерминированным способом и получения детерминированных результатов. Многие клиенты используют скрипты powerShell или CLI. Но существуют различные технологии с открытым исходным кодом, которые можно использовать для развертывания инфраструктуры Azure для SAP и даже установки программного обеспечения SAP. Вы можете найти примеры на GitHub:
    - [Автоматическое развертывание SAP в облаке Azure](https://github.com/Azure/sap-hana)
    - [Установка SAP HANA](https://github.com/AzureCAT-GSI/SAP-HANA-ARM)
9.  Определите регулярную каденцию обзора проектирования и развертывания между вами как заказчиком, системным интегратором, корпорацией Майкрософт и другими заинтересованными сторонами.

 
## <a name="pilot-phase-strongly-recommended"></a>Пилотная фаза (настоятельно рекомендуется)
 
Вы можете запустить пилотный проект до или во время планирования и подготовки проекта. Вы также можете использовать пилотный этап для тестирования подходов и конструкций, сделанных на этапе планирования и подготовки. И вы можете расширить пилотный этап, чтобы сделать его реальным доказательством концепции.

Мы рекомендуем настроить и проверить полное решение HADR и дизайн безопасности во время пилотного развертывания. Некоторые клиенты выполняют тесты масштабируемости на этом этапе. Другие клиенты используют развертывание систем песочницы SAP в качестве пилотного этапа. Мы предполагаем, что вы уже определили систему, которую хотите перенести в Azure для пилота.

1. Оптимизация передачи данных в Azure. Оптимальный выбор сильно зависит от конкретного сценария. Переход из помещения через [Azure ExpressRoute](https://azure.microsoft.com/services/expressroute/) является самым быстрым, если схема ExpressRoute имеет достаточную пропускную способность. В других сценариях передача через Интернет происходит быстрее.
2. Для неоднородной миграции платформы SAP, которая включает в себя экспорт и импорт данных, тестирование и оптимизация этапов экспорта и импорта. Для больших миграций, в которых s'L Server является платформой назначения, вы можете найти [рекомендации.](https://techcommunity.microsoft.com/t5/Running-SAP-Applications-on-the/SAP-OS-DB-Migration-to-SQL-Server-8211-FAQ-v6-2-April-2017/ba-p/368070) Вы можете использовать Migration Monitor/SWPM, если вам не нужно комбинированное обновление релиза. При объединении миграции с обновлением выпуска SAP можно использовать процесс [SAP DMO.](https://blogs.sap.com/2013/11/29/database-migration-option-dmo-of-sum-introduction/) Для этого необходимо соответствовать определенным требованиям к комбинации платформы DBMS и целевой цели. Этот процесс задокументирован в [варианте миграции базы данных (DMO) SUM 2.0 SP03](https://launchpad.support.sap.com/#/notes/2631152).
   1.  Экспорт на источник, загрузка экспортного файла в Azure и производительность импорта. Максимальное перекрытие между экспортом и импортом.
   2.  Оцените объем базы данных по целевым и назначениянным платформам для целей размеров инфраструктуры.
   3.  Проверка и оптимизация времени.
1. Техническая проверка.
   1. Типы VM.
        - Просмотрите ресурсы в примечаниях поддержки SAP, в аппаратном каталоге SAP HANA и в SAP PAM снова. Убедитесь, что нет никаких изменений в поддерживаемых вс-минах для Azure, поддерживаемых оС-релизов для этих типов VM и поддерживаемых sAP и DBMS-релизов.
        - Повторно проверьте размер приложения и инфраструктуры, развертываемой в Azure. Если вы перемещаете существующие приложения, вы часто можете получить необходимые SAPS из инфраструктуры, которая используется, и [веб-страницу с бенчмарком SAP](https://www.sap.com/dmc/exp/2018-benchmark-directory/#/sd) и сравнить ее с номерами SAPS, перечисленными в [примечании поддержки SAP #1928533.](https://launchpad.support.sap.com/#/notes/1928533) Также держите [эту статью на рейтингах SAPS](https://techcommunity.microsoft.com/t5/Running-SAP-Applications-on-the/SAPS-ratings-on-Azure-VMs-8211-where-to-look-and-where-you-can/ba-p/368208) в разуме.
        - Оцените и проверьте размеры визов ваших VMs Azure с точки зрения максимальной пропускной памяти и пропускной мощности сети типов VM, которые вы выбрали на этапе планирования. Вы можете найти данные здесь:
           -  [Размеры виртуальных машин Windows в Azure](https://docs.microsoft.com/azure/virtual-machines/windows/sizes?toc=%2fazure%2fvirtual-network%2ftoc.json). Важно учитывать *максимальную некэшированную пропускную выливку диска* для калибровки.
           -  [Размеры для виртуальных машин Linux в Azure](https://docs.microsoft.com/azure/virtual-machines/linux/sizes?toc=%2fazure%2fvirtual-network%2ftoc.json). Важно учитывать *максимальную некэшированную пропускную выливку диска* для калибровки.
   2. "Память".
        - Как минимум, используйте [хранилище SSD Azure Standard Для](https://docs.microsoft.com/azure/virtual-machines/windows/disks-types#standard-ssd) vMs, представляющих слои приложений SAP, и для развертывания DBMS, которые не чувствительны к производительности.
        - Как правило, мы не рекомендуем использовать [HDD диски Azure Standard.](https://docs.microsoft.com/azure/virtual-machines/windows/disks-types#standard-hdd)
        - Используйте [Azure Premium Storage](https://docs.microsoft.com/azure/virtual-machines/windows/disks-types#premium-ssd) для любых MMS DBMS, которые удаленно чувствительны к производительности.
        - Используйте [управляемые диски Azure.](https://azure.microsoft.com/services/managed-disks/)

## <a name="non-production-phase"></a>Непроизводственной фазы 
На этом этапе мы предполагаем, что после успешного пилотного проекта или доказательства концепции (POC) вы начинаете развертывать непроизводственные системы SAP в Azure. Включите все, что вы узнали и испытали во время POC к этому развертыванию. Все критерии и шаги, перечисленные для POCs, также применяются к этому развертыванию.

На этом этапе обычно развертываются системы разработки, модульные системы тестирования и бизнес-системы регрессионных испытаний в Azure. Мы рекомендуем, чтобы по крайней мере одна непроизводственная система в одной линейке приложений SAP имеет полную конфигурацию высокой доступности, которую будет иметь будущая производственная система. Вот некоторые дополнительные шаги, которые необходимо выполнить на этом этапе:  

1.  Перед перемещением систем со старой платформы в Azure соберите данные о потреблении ресурсов, такие как использование процессора, пропускная память и данные IOPS. Особенно собирать эти данные из блоков слоя DBMS, но и собирать его из блоках уровня приложения. Кроме того, измерьте задержки сети и хранилища.
2.  Запишите модели использования систем доступности. Цель состоит в том, чтобы выяснить, должны ли непроизводственные системы быть доступными в течение всего дня, каждый день или существуют ли непроизводственные системы, которые могут быть закрыты в определенные фазы недели или месяца.
3.  Проверьте и определите, хотите ли вы создать собственные изображения ОС для виможно в Azure или хотите использовать изображение из общей галереи изображений Azure. Если вы используете изображение из общей галереи изображений, обязательно используйте изображение, отражающее контракт на поддержку с поставщиком ОС. Для некоторых поставщиков ОС, Shared Image Gallery позволяет принести свои собственные изображения лицензии. Для других изображений ОС поддержка включена в цену, указанную Azure. Если вы решили создать свои собственные образы ОС, то соответствующую документацию можно найти в следующих статьях.
    -   [Создайте обобщенное изображение Windows VM, развернутого в Azure](https://docs.microsoft.com/azure/virtual-machines/windows/capture-image-resource)
    -   [Создайте обобщенное изображение Linux VM, развернутого в Azure](https://docs.microsoft.com/azure/virtual-machines/linux/capture-image)
3.  Если вы используете изображения SUSE и Red Hat Linux из общей галереи изображений, вам необходимо использовать изображения для SAP, предоставленные поставщиками Linux в общей галерее изображений.
4.  Убедитесь в том, что выполнить требования поддержки SAP для соглашений о поддержке Майкрософт. Смотрите [примечание поддержки SAP #2015553.](https://launchpad.support.sap.com/#/notes/2015553) Для крупных инстанций HANA [см.](https://docs.microsoft.com/azure/virtual-machines/workloads/sap/hana-onboarding-requirements)
4.  Убедитесь, что нужные люди получают [запланированные уведомления о техническом обслуживании,](https://azure.microsoft.com/blog/a-new-planned-maintenance-experience-for-your-virtual-machines/) чтобы вы могли выбрать лучшие простои.
5.  Часто проверяйте презентации Azure на каналах, таких как [Channel 9,](https://channel9.msdn.com/) на наличие новых функциональных возможностей, которые могут применяться к развертыванию.
6.  Проверьте sAP-заметки, связанные с Azure, [например, примечания к поддержке #1928533,](https://launchpad.support.sap.com/#/notes/1928533)для новых VM SKUs и недавно поддерживаемых релизов ОС и DBMS. Сравните цены новых типов VM с ценами старых типов VM, так что вы можете развернуть VMs с лучшим соотношением цена/производительность.
7.  Перепроверка заметок поддержки SAP, аппаратного каталога SAP HANA и SAP PAM. Убедитесь, что не было никаких изменений в поддерживаемых VMs для Azure, поддерживаемых оС-релизов на этих VMs и поддерживаемых sAP и DBMS-релизов.
8.  Проверьте [веб-сайт SAP](https://www.sap.com/dmc/exp/2014-09-02-hana-hardware/enEN/iaas.html#categories=Microsoft%20Azure) для новых сертифицированных HANA SkUs в Azure. Сравните цены на новые СКУ с теми, которые вы планировали использовать. В конце концов, внести необходимые изменения, чтобы использовать те, которые имеют лучшее соотношение цена / производительность.
9.  Адаптируйте сценарии развертывания для использования новых типов VM и включите новые функции Azure, которые вы хотите использовать.
10. После развертывания инфраструктуры, тест и оценка задержки сети между SAP приложения уровня VMs и DBMS VMs, в соответствии с sAP отмечает поддержку [#500235](https://launchpad.support.sap.com/#/notes/500235) и [#1100926](https://launchpad.support.sap.com/#/notes/1100926/E). Оцените результаты по отношению к рекомендациям по задержке сети в [примечании поддержки SAP #1100926.](https://launchpad.support.sap.com/#/notes/1100926/E) Задержка сети должна быть в умеренном или хорошем диапазоне. Исключения применяются к трафику между VMs и ПОДРАЗДЕЛЕНИЯми HANA Large Instance, как описано в [этой статье.](https://docs.microsoft.com/azure/virtual-machines/workloads/sap/hana-network-architecture#networking-architecture-for-hana-large-instance) Убедитесь, что ни одно из ограничений, упомянутых в рассмотрении развертывания [DBMS виртуальных машин Azure для рабочих нагрузок SAP](https://docs.microsoft.com/azure/virtual-machines/workloads/sap/dbms_guide_general#azure-network-considerations) и [конфигураций инфраструктуры SAP HANA в Azure,](https://docs.microsoft.com/azure/virtual-machines/workloads/sap/hana-vm-operations) не применяется к развертыванию.
11. Убедитесь, что ваши чаты в [группе размещения непосредственной близости Azure,](https://docs.microsoft.com/azure/virtual-machines/linux/co-location)как описано в [группах размещения близости Azure, для оптимальной задержки сети с приложениями SAP.](sap-proximity-placement-scenarios.md)
11. Выполните все другие проверки, перечисленные для доказательства фазы концепции, прежде чем применить рабочую нагрузку.
12. По мере загрузки записываете потребление ресурсов систем в Azure. Сравните это потребление с записями со старой платформы. Отрегулируйте размер ы ВМ будущих развертываний, если вы видите, что у вас есть большие различия. Имейте в виду, что при сокращении, хранении и пропускной способности сети, а также будет сокращена.
    - [Размеры для виртуальных машин Windows в Azure](https://docs.microsoft.com/azure/virtual-machines/windows/sizes?toc=%2fazure%2fvirtual-network%2ftoc.json)
    - [Размеры виртуальных машин Linux в Azure](https://docs.microsoft.com/azure/virtual-machines/linux/sizes?toc=%2fazure%2fvirtual-network%2ftoc.json) 
13. Экспериментируйте с функциональностью и процессами копирования системы. Цель состоит в том, чтобы сделать его легким для вас, чтобы скопировать систему разработки или тестовую систему, так что проектные группы могут получить новые системы быстро. Рассмотрите возможность использования [SAP LaMa](https://wiki.scn.sap.com/wiki/display/ATopics/SAP+Landscape+Management+%28SAP+LaMa%29+at+a+Glance) для этих задач.
14. Оптимизируйте и оттачивайте доступ, разрешения и процессы, основанные на роли вашей команды Azure, чтобы убедиться, что у вас есть разделение обязанностей. В то же время убедитесь, что все команды могут выполнять свои задачи в инфраструктуре Azure.
15. Упражнение, тестирование и документирование высокой доступности и процедуры аварийного восстановления, чтобы позволить сотрудникам выполнять эти задачи. Определите недостатки и адаптируйте новую функциональность Azure, которую вы интегрируете в развертывание.

 
## <a name="production-preparation-phase"></a>Этап подготовки к производству 
На этом этапе соберите опыт и изученное во время непроизводственных развертываний и примените его к будущим развертываниям. Также необходимо подготовить работу передачи данных между текущим местоположением хостинга и Azure.

1.  Перед переходом в Azure необходимо ею обновление выпуска SAP.
1.  Согласен с владельцами бизнеса по функциональным и бизнес-тестам, которые необходимо проводить после миграции производственной системы.
1.  Убедитесь, что эти тесты завершены с исходными системами в текущем месте хостинга. Избегайте проведения тестов в первый раз после того, как система была перемещена в Azure.
1.  Проверьте процесс переноса производственных систем в Azure. Если вы не перемещаете все производственные системы в Azure в течение одного и того же периода времени, создайте группы производственных систем, которые должны находиться в одном месте хостинга. Тест-миграция данных. Вот некоторые распространенные методы:
    - Используйте методы DBMS, такие как резервное копирование/восстановление в сочетании с s'L Server Always On, репликацией системы HANA или доставкой журналов для семени и синхронизации содержимого базы данных в Azure.
    - Используйте резервное копирование/восстановление для небольших баз данных.
    - Используйте SAP Migration Monitor, который интегрирован в SAP SWPM, для выполнения неоднородных миграций.
    - Используйте процесс [SAP DMO,](https://blogs.sap.com/2013/11/29/database-migration-option-dmo-of-sum-introduction/) если вам нужно объединить миграцию с обновлением выпуска SAP. Имейте в виду, что не все комбинации исходного DBMS и целевого DBMS поддерживаются. Более подробную информацию можно найти в конкретных примечаниях поддержки SAP для различных выпусков DMO. Например, [вариант миграции баз данных (DMO) SUM 2.0 SP04](https://launchpad.support.sap.com/#/notes/2644872).
    - Проверьте, лучше ли передача данных через Интернет или через ExpressRoute, в случае необходимости перемещения резервных данных или экспортных файлов SAP. Если вы перемещаете данные через Интернет, возможно, потребуется изменить некоторые из ваших сетевых правил группы безопасности/группы безопасности приложений, которые необходимо иметь для будущих производственных систем.
1.  Прежде чем перейти системы со старой платформы в Azure, соберите данные о потреблении ресурсов. Полезные данные включают использование процессора, пропускную стоимость хранилища и данные IOPS. Особенно собирать эти данные из блоков слоя DBMS, но и собирать его из блоках уровня приложения. Кроме того, измерьте задержки сети и хранилища.
1.  Перепроверка заметок поддержки SAP и необходимых настроек ОС, аппаратного каталога SAP HANA и SAP PAM. Убедитесь, что не было никаких изменений в поддерживаемых VMs для Azure, поддерживаемых оС-релизов в этих VMs и поддерживаемых sAP и DBMS-релизов.
1.  Обновляйте сценарии развертывания с учетом последних решений, принятых по типам VM и функциональности Azure.
1.  После развертывания инфраструктуры и приложений, проверить, что:
    - Были развернуты правильные типы VM с правильными атрибутами и размерами хранилища.
    - VMs находятся на правильно и пожелано отпусках OS и заплаты и равномерно.
    - ВМ затвердевают по мере необходимости и единообразно.
    - Правильные релизы приложений и патчи были установлены и развернуты.
    - VMs были развернуты в наборах доступности Azure, как и планировалось.
    - Azure Premium Storage используется для дисков с задержкой или там, где требуется [одно-VM SLA в 99,9%.](https://azure.microsoft.com/support/legal/sla/virtual-machines/v1_8/)
    - Ускоритель записи Azure записи развертывается правильно.
        - Убедитесь, что в пределах VMs, места хранения, или полосы наборы были построены правильно на дисках, которые нуждаются в записи ускоритель.
        - Проверьте [конфигурацию программного обеспечения RAID на Linux](https://docs.microsoft.com/azure/virtual-machines/linux/configure-raid).
        - Проверьте [конфигурацию LVM на Linux VMs в Azure.](https://docs.microsoft.com/azure/virtual-machines/linux/configure-lvm)
    - [Управляемые Azure диски](https://azure.microsoft.com/services/managed-disks/) используются исключительно.
    - ВМ были развернуты в правильных наборах доступности и зонах доступности.
    - [Ускоренная сеть Azure](https://azure.microsoft.com/blog/maximize-your-vm-s-performance-with-accelerated-networking-now-generally-available-for-both-windows-and-linux/) включена на вс-мис, используемых в слое приложения SAP и слое SAP DBMS.
    - Нет [виртуальных устройств сети Azure](https://azure.microsoft.com/solutions/network-appliances/) находятся в пути связи между приложением SAP и слоем DBMS систем SAP на основе SAP NetWeaver, Hybris или S/4HANA.
    - Правила группы безопасности приложений и группы сетевой безопасности позволяют общаться по желанию и планировать и блокировать связь, когда это необходимо.
    - Настройки тайм-аута настроены правильно, как описано ранее.
    - VMs развертываются в правильной [группе размещения близости Azure,](https://docs.microsoft.com/azure/virtual-machines/linux/co-location)как описано в [группах размещения близости Azure для оптимальной задержки сети с приложениями SAP.](sap-proximity-placement-scenarios.md)
    - Задержка сети между VMs-ивгами sAP и DBMS VMs проверяется и проверяется, как описано в примечаниях поддержки SAP [#500235](https://launchpad.support.sap.com/#/notes/500235) и [#1100926.](https://launchpad.support.sap.com/#/notes/1100926/E) Оцените результаты по отношению к рекомендациям по задержке сети в [примечании поддержки SAP #1100926.](https://launchpad.support.sap.com/#/notes/1100926/E) Задержка сети должна быть в умеренном или хорошем диапазоне. Исключения применяются к трафику между VMs и ПОДРАЗДЕЛЕНИЯми HANA Large Instance, как описано в [этой статье.](https://docs.microsoft.com/azure/virtual-machines/workloads/sap/hana-network-architecture#networking-architecture-for-hana-large-instance)
    - Шифрование осуществлялось там, где это необходимо, и с соответствующим методом шифрования.
    - Интерфейсы и другие приложения могут соединять недавно развернутую инфраструктуру.
1.  Создайте игровую книгу для реагирования на запланированное техническое обслуживание Azure. Определите порядок перезагрузки систем и ВМ для планового технического обслуживания.
    

## <a name="go-live-phase"></a>Перейти жить фазы
Во время фазы go-live, не забудьте следовать playbooks вы разработали на более ранних этапах. Выполните шаги, которые вы тестировали и практиковали. Не принимайте изменения в конфигурациях и процессах в последнюю минуту. Также выполнить следующие шаги:

1. Убедитесь, что мониторинг на портале Azure и другие инструменты мониторинга работают. Мы рекомендуем Windows Performance Monitor (perfmon) для Windows и SAR для Linux.
    - Счетчики процессора.
        - Среднее время процессора, всего (все процессоры)
        - Среднее время процессора, каждый отдельный процессор (128 процессоров на M128 VMs)
        - Время ядра процессора, каждый отдельный процессор
        - Время пользователя процессора, каждый отдельный процессор
    - Память.
        - объем свободной памяти;
        - Страница памяти в/секунде
        - Страница памяти вне/секунда
    - (Краткосрочная защита с использованием:) и вариант "Диск".
        - Диск читать в KBps, на отдельный диск
        - Диск читает/секунду, на отдельный диск
        - Диск, считываемый в микросекундах/чтении, на отдельный диск
        - Диск писать в KBps, на отдельный диск
        - Запись диска в секунду, на отдельный диск
        - Диск писать в микросекундах / читать, на отдельный диск
    - Network.
        - Сетевые пакеты в/секунде
        - Пакеты сети вне в секунду
        - Сеть КБ в/секунду
        - Сеть КБ вне/секунде
1.  После миграции данных выполните все тесты проверки, согласованные с владельцами бизнеса. Принимать результаты теста проверки только тогда, когда у вас есть результаты для исходных систем.
1.  Проверьте, функционируют ли интерфейсы и могут ли другие приложения взаимодействовать с недавно развернутыми производственными системами.
1.  Проверьте систему транспортировки и коррекции через транзакцию SAP STMS.
1.  Выполните резервное копирование базы данных после выпуска системы для производства.
1.  Выполните резервное копирование VM для VMs-приложений SAP после выпуска системы для производства.
1.  Для систем SAP, которые не были частью текущей фазы go-live, но которые взаимодействуют с системами SAP, которые вы перешли в Azure на этом этапе go-live, необходимо сбросить буфер имен хоста в SM51. Это позволит удалить старые кэшированные IP-адреса, связанные с именами экземпляров приложения, которые вы перенесли в Azure.  


## <a name="post-production"></a>Этап после запуска рабочих систем
Этот этап посвящен мониторингу, эксплуатации и администрированию системы. С точки зрения SAP применяются обычные задачи, которые необходимо выполнить в старом местоположении хостинга. Выполните эти задачи, связанные с Azure:

1. Просмотрите счета-фактуры Azure для систем высокой зарядки.
2. Оптимизация эффективности цены/производительности на стороне VM и на складе.
3. Оптимизация времени, когда вы можете выключить системы.
