# <a name="azure-hana-large-instances-control-through-azure-portal"></a>Управление крупными экземплярами HANA в Azure с помощью портала Azure
В этом документе рассказывается о том, как [hana Large Instances](https://docs.microsoft.com/azure/virtual-machines/workloads/sap/hana-overview-architecture) представлены на [портале Azure](https://portal.azure.com) и какие действия могут осуществляться через портал Azure с развернутыми для вас подразделениями HANA Large Instance. Видимость крупных экземпляров HANA на портале Azure обеспечивается через поставщика ресурсов Azure для HANA Large Instances, который в настоящее время находится в открытом доступе

## <a name="register-hana-large-instance-resource-provider"></a>Регистрация поставщика ресурсов крупного экземпляра HANA
Обычно ваша подписка Azure, используемая для развертывания HANA Large Instance, регистрируется для поставщика ресурсов HANA Large Instance. Однако, если вы не видите развернутых подразделений HANA Large Instance, необходимо зарегистрировать поставщика ресурсов в подписке Azure. Существует два способа регистрации поставщика ресурсов HANA Large Instance

### <a name="register-through-cli-interface"></a>Зарегистрируйтесь через интерфейс CLI
Необходимо войти в подписку Azure, используемую для развертывания HANA Large Instance через интерфейс Azure CLI. Вы можете (перерегистрировать провайдера крупной инстанции HANA с этой командой:
    
    az provider register --namespace Microsoft.HanaOnAzure

Для получения дополнительной информации смотрите статью [поставщиков ресурсов Azure и типов](https://docs.microsoft.com/azure/azure-resource-manager/resource-manager-supported-services#azure-cli)


### <a name="register-through-azure-portal"></a>Зарегистрируйтесь через портал Azure
Вы можете (перерегистрировать поставщика ресурсов крупной инстанции HANA) через портал Azure. Вам нужно перечислить подписку на портале Azure и дважды нажать на подписку, которая использовалась для развертывания подразделения HANA Large Instance(s). Один вы находитесь в обзоре страницы вашей подписки, выберите "Ресурс провайдеров", как показано ниже, и введите "HANA" в окне поиска. 

![Регистрация HLI RP через портал Azure](./media/hana-li-portal/portal-register-hli-rp.png)

На скриншоте показан, поставщик ресурсов уже зарегистрирован. В случае, если поставщик ресурсов еще не зарегистрирован, нажмите "перерегистр" или "регистрация".

Для получения дополнительной информации смотрите статью [поставщиков ресурсов Azure и типов](https://docs.microsoft.com/azure/azure-resource-manager/resource-manager-supported-services#azure-powershell)


## <a name="display-of-hana-large-instance-units-in-the-azure-portal"></a>Отображение единиц крупной инстанции HANA на портале Azure
При отправке запроса на развертывание HANA Large Instance требуется указать подписку Azure, которую вы также подключаете к an HANA Large Instances. Рекомендуется использовать ту же подписку, что и для развертывания слоя приложения SAP, который работает против подразделений HANA Large Instance.
По мере развертывания первых крупных инстанций HANA создается новая [группа ресурсов Azure](https://docs.microsoft.com/azure/azure-resource-manager/manage-resources-portal) в подписке Azure, отправленной в запросе развертывания для вашего HANA Large Instance(s).  Новая группа ресурсов будет перечислять все ваши подразделения HANA Large Instance, развернутые в конкретной подписке.

Чтобы найти новую группу ресурсов Azure, вы перечислите группу ресурсов в подписке, пройдя через левое навигационное стекло портала Azure

![Навигационное панно на портале Azure](./media/hana-li-portal/portal-resource-group.png)

В списке групп ресурсов, которые вы получаете в списке, возможно, потребуется отфильтровать подписку, в какой вы использовали, чтобы развертываться в крупных инстанциях HANA

![Группы ресурсов фильтров на портале Azure](./media/hana-li-portal/portal-filtering-subscription.png)

После фильтрации правильной подписки у вас все еще может быть длинный список групп ресурсов. Ищите один с пост-фикс **-Txxx,** где "xxx" являются три цифры, как **-T050**. 

Как вы нашли группу ресурсов, перечислите ее детали. Полученный вами список может выглядеть следующим:

![Список HLI на портале Azure](./media/hana-li-portal/portal-hli-units-list.png)

Все перечисленные подразделения представляют собой одно подразделение HANA Large Instance, развернутое в вашей подписке. В этом случае вы посмотрите на восемь различных подразделений HANA Large Instance, которые были развернуты в вашей подписке.

Если вы развернули несколько клиентов HANA Large Instance в рамках той же подписки Azure, вы найдете несколько групп ресурсов Azure 


## <a name="look-at-attributes-of-single-hli-unit"></a>Посмотрите на атрибуты единого HLI-подразделения
В списке единиц HANA Large Instance вы можете нажать на одну единицу и добраться до деталей единого подразделения HANA Large Instance. 

На экране обзора, после нажатия "Показать больше", вы получаете представление единицы, которая выглядит следующим образом:

![Показать обзор блока HLI](./media/hana-li-portal/portal-show-overview.png)

Если посмотреть на различные атрибуты, показанные, эти атрибуты практически не отличаются от атрибутов Azure VM. На замзаголовке левой стороны отображается группа ресурсов, область Azure, имя подписки и идентификатор, а также некоторые теги, которые вы добавили. По умолчанию единицы крупной инстанции HANA не имеют тега. На правой стороне заголовка, имя подразделения указано как назначенное, когда развертывание было сделано. Отображается операционная система, а также IP-адрес. Как и в случае с VMs, отображается также тип единицы экземпляра HANA Large с числом потоков и памяти процессора. Более подробная информация о различных единицах HANA Large Instance, приведены здесь:

- [Доступные номера SKU для HLI](https://docs.microsoft.com/azure/virtual-machines/workloads/sap/hana-available-skus)
- [Архитектура хранилища SAP HANA (крупные экземпляры)](https://docs.microsoft.com/azure/virtual-machines/workloads/sap/hana-storage-architecture) 

Дополнительными данными на правой нижней стороне является пересмотр штампа HANA Large Instance. Возможны следующие значения:

- Редакция 3
- Редакция 4

Revision 4 — это последняя архитектура, выпущенная из HANA Large Instances с существенными улучшениями в задержке сети между VMs-ми чками Azure и HANA Large instance, развернутыми в марках или строках Revision 4.
Еще одна очень важная информация находится в правом нижнем углу обзора с названием группы размещения близости Azure, которая автоматически создается для каждого развернутого подразделения HANA Large Instance. Эта группа размещения близости должна быть отображана при развертывании VMs-ми, вмещающих слой приложения SAP. Используя [группу размещения близости Azure,](https://docs.microsoft.com/azure/virtual-machines/linux/co-location) связанную с подразделением HANA Large Instance, вы убедитесь, что VMs Azure развернуты в непосредственной близости от подразделения HANA Large Instance. Способ использования групп размещения близости для определения местообнаружения слоя приложения SAP в том же центре обработки данных Azure, что и блоки «Редакция 4», размещенные в группах по размещению частой связи [Azure, для оптимальной задержки сети с помощью приложений SAP.](sap-proximity-placement-scenarios.md)

Дополнительное поле в правой колонке заголовка сообщает о состоянии питания блока HANA Large instance.

> [!NOTE]
> Состояние питания описывает, включен или выключен аппаратный блок. Он не дает информацию о том, что операционная система работает. При перезапуске подразделения HANA Large Instance вы будете испытывать небольшое время, когда состояние единицы **изменяется** для перехода в состояние **Запущенного.** Находясь в состоянии **запущенных** означает, что ОС запускается или что ОС была запущена полностью. В результате, после перезагрузки устройства, вы не можете ожидать, чтобы сразу войти в устройство, как только состояние переключается на **Запущенный**.
> 

Если вы нажмете "Смотрите больше", отображается дополнительная информация. Еще одна информация показывает пересмотр штампа HANA Large Instance, в который было развернуто подразделение. Смотрите статью [Что такое SAP HANA на Azure (Большие инстанции)](https://docs.microsoft.com/azure/virtual-machines/workloads/sap/hana-overview-architecture) для различных изменений hanA Большой инстанции марки

## <a name="check-activities-of-a-single-hana-large-instance-unit"></a>Проверка деятельности одного подразделения крупной инстанции HANA 
Помимо представления обзора подразделений HANA Large Instance, вы можете проверить действия конкретного подразделения. Лог активности может выглядеть следующим:

![Навигационное панно на портале Azure](./media/hana-li-portal/portal-activity-list.png)

Одним из основных зарегистрированных видов деятельности являются перезапуск единицы. Перечисленные данные включают состояние действия, срабатывание действия, идентификатор подписки, из которого сработало действие, и пользователь Azure, который инициировал действие. 

Еще одно зарегистрированное действие — это изменения в блоке метаданных Azure. Помимо начала перезагрузки, вы можете увидеть активность **Написать HANAInstances**. Этот тип действия не выполняет никаких изменений в самом блоке HANA Large Instance, но документирует изменения в метаданных блока в Azure. В указанном случае мы добавили и удалили тег (см. следующий раздел).

## <a name="add-and-delete-an-azure-tag-to-a-hana-large-instance-unit"></a>Добавление и удаление тега Azure в подразделение HANA Large Instance
Еще одна возможность, у вас есть, чтобы добавить [тег](https://docs.microsoft.com/azure/azure-resource-manager/resource-group-using-tags) в подразделение HANA Большой инстанции. Способ присвоения тегов не отличается от присвоения тегов для VMs. Как и в случае с VMs, метки существуют в мета-данных Azure, а для HANA Large Instances имеют те же ограничения, что и метки для VMs.

Удаляющие теги работают так же, как и с VMs. Оба действия, применяющие и удаляющие теги, будут перечислены в журнале действий конкретного подразделения HANA Large Instance.

## <a name="check-properties-of-a-hana-large-instance-unit"></a>Проверка свойств подразделения HANA Large Instance
Раздел **Свойства** включает в себя важную информацию, которую вы получаете, когда экземпляры передаются вам. Это раздел, где вы получаете всю информацию, которая может потребоваться в случаях поддержки или которая вам нужна при настройке конфигурации моментального снимка хранилища. Таким образом, этот раздел представляет собой набор данных вокруг экземпляра, подключение экземпляра к Azure и бэкэнда хранилища. В верхней части раздела выглядит следующим:


![верхняя часть свойств HLI на портале Azure](./media/hana-li-portal/portal-properties-top.png)

Первые несколько элементов данных, которые вы видели на экране обзора уже. Но важной частью данных является идентификатор трассы ExpressRoute, который вы получили при передаче первых развернутых юнитов. В некоторых случаях поддержки эти данные могут быть запрошены. Важная запись данных отображается в нижней части скриншота. Отображаемые данные — это IP-адрес главы хранилища NFS, который изолирует ваше хранилище **для вашего клиента** в стеке HANA Large Instance. Этот IP-адрес также необходим при необходимости отсваивателя [файла конфигурации для резервного копирования моментального хранения.](https://docs.microsoft.com/azure/virtual-machines/workloads/sap/hana-backup-restore#set-up-storage-snapshots) 

При прокрутке вниз в панели свойств вы получаете дополнительные данные, такие как уникальный идентификатор ресурсов для подразделения HANA Large Instance, или идентификатор подписки, назначенный развертыванию.

## <a name="restart-a-hana-large-instance-unit-through-azure-portal"></a>Перезапуск подразделения HANA Large Instance через портал Azure
Ивнотируя перезапуск операционной системы Linux, были различные ситуации, когда ОС не могла успешно завершить перезагрузку. Для того, чтобы принудить к перезагрузке, необходимо открыть запрос на обслуживание, чтобы операции Майкрософт выполнили перезапуск блока HANA Large Instance. Функциональность перезапуска блока HANA Large Instance теперь интегрирована в портал Azure. Как вы находитесь в обзорной части блока HANA Large Instance, вы видите кнопку для перезагрузки в верхней части раздела данных

![Перезапуск шага #1 на портале Azure](./media/hana-li-portal/portal-restart-first-step.png)

При нажатии кнопки перезагрузки вас спрашивают, действительно ли вы хотите перезапустить устройство. Как вы подтверждаете, нажав кнопку "Да", устройство будет перезапустить.

> [!NOTE]
> В процессе перезагрузки вы будете испытывать небольшое время, когда **Starting** состояние блока изменяется, чтобы начать двигаться в состояние **Запущенного.** Находясь в состоянии **запущенных** означает, что ОС запускается или что ОС была запущена полностью. В результате, после перезагрузки устройства, вы не можете ожидать, чтобы сразу войти в устройство, как только состояние переключается на **Запущенный**.

> [!IMPORTANT]
> В зависимости от объема памяти в блоке HANA Large Instance перезагрузка и перезагрузка оборудования и операционной системы может занять до одного часа


## <a name="open-a-support-request-for-hana-large-instances"></a>Откройте запрос на поддержку для крупных инстанций HANA
Из отображения портала Azure подразделений HANA Large Instance можно создавать запросы на поддержку специально для подразделения крупной инстанции HANA. По мере того как вы следуете за соединением **Новый запрос поддержки** 

![инициировать шаг #1 запроса на обслуживание на портале Azure](./media/hana-li-portal/portal-initiate-support-request.png)

Для того, чтобы получить услугу SAP HANA Large Instances, перечисленных на следующем экране, может потребоваться выбрать "Все услуги", как показано ниже

![Выберите все службы на портале Azure](./media/hana-li-portal/portal-create-service-request.png)

В списке услуг можно найти сервис **SAP HANA Large Instance.** При выборе этой службы можно выбрать определенные типы проблем в показано:


![Выберите класс проблем на портале Azure](./media/hana-li-portal/portal-select-problem-class.png)

В рамках каждого из различных типов проблем вам предлагается выбор проблемных подтипов, которые необходимо выбрать, чтобы охарактеризовать вашу проблему дальше. После выбора подтипа можно назвать объект съемки. Как только вы закончите с процессом отбора, вы можете перейти к следующему этапу создания. В разделе **Решения** вы указали на документацию вокруг HANA Large Instances, которая может дать указатель на решение вашей проблемы. Если вы не можете найти решение проблемы в предлагаемой документации, вы перейдите к следующему шагу. На следующем этапе вас спросят, связана ли проблема с ВМ или с подразделениями HANA Large Instance. Эта информация помогает направить запрос на поддержку к правильным специалистам. 

![Подробная информация о случае поддержки на портале Azure](./media/hana-li-portal/portal-support-request-details.png)

Как вы ответили на вопросы и представил дополнительные сведения, вы можете сделать следующий шаг для того, чтобы рассмотреть запрос на поддержку и представить его.
