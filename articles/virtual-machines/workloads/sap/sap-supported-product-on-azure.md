# <a name="what-sap-software-is-supported-for-azure-deployments"></a>Какое программное обеспечение SAP поддерживается для развертывания Azure
В этой статье описывается, как можно узнать, какое программное обеспечение SAP поддерживается для развертывания Azure и каковы необходимые релизы операционной системы или DBMS-релизы.

Оценивая, поддерживается ли текущее программное обеспечение SAP и какие релизы OS и DBMS поддерживаются вашим программным обеспечением SAP в Azure, вам понадобится доступ к:

- Примечания поддержки SAP
- Матрица доступности продуктов SAP



## <a name="general-restrictions-for-sap-workload"></a>Общие ограничения для рабочей нагрузки SAP
Службы Azure IaaS, которые могут использоваться для рабочей нагрузки SAP, ограничены оборудованием x86-64 или x64. Нет предложений на основе sparc или Power CPU, которые применяются к рабочей нагрузке SAP. Клиентам, работающим на своих приложениях на операционных системах, несвободных для аппаратных архитектур, таких как IBM mainframe или AS400, или когда используются операционные системы HP-UX, Solaris или AIX, необходимо изменить свои приложения SAP, включая DBMS, на одну из следующих операционных систем:

- Сервер Windows 64bit для платформы x86-64
- SUSE linux 64bit для платформы x86-64
- Красная шляпа Linux 64Bit для платформы x86-64
- Oracle Linux 64bit для платформы x86-64

В сочетании с программным обеспечением SAP никакие другие релизы ОС или дистрибутивы Linux не поддерживаются. Точные сведения о конкретных версиях и случаях задокументированы позднее в документе.


## <a name="you-start-here"></a>Вы начинаете здесь
Отправной точкой для вас является [примечание поддержки SAP #1928533.](https://launchpad.support.sap.com/#/notes/1928533) По мере того как вы идете через эту примечание SAP от верхней части к низшвам, несколько зон поддержанного произносного обеспечения и VMs показаны

В первом разделе перечислены минимальные требования к операционным выпускам, которые поддерживаются программным обеспечением SAP в Azure VMs в целом. Если вы не достигаете этих минимальных требований и запускаете старые выпуски этих операционных систем, вам необходимо обновить выпуск ОС до такого минимального выпуска или даже более поздних релизов. Это правильно, что Azure в целом будет поддерживать старые релизы некоторых из этих операционных систем. Но ограничения или минимальные релизы, как указано в перечне, основаны на тестах и выполненных квалификационных требованиях и не будут продлены дальше. 


> [!NOTE]
>Есть некоторые конкретные типы VM, HANA Большие instances или SAP рабочих нагрузок, которые будут требовать более поздних релизов ОС. Подобные случаи будут упоминаться в документе. Подобные случаи четко задокументированы либо в примечаниях SAP, либо в других публикациях SAP.

В следующем разделе перечислены общие платформы SAP, которые поддерживаются поддерживаемыми релизами и что более важные поддерживаемые ядра SAP. В нем перечислены netWeaver/ABAP или Java-стеки, которые поддерживаются и которым требуются минимальные выбросы ядра. Более поздние стеки ABAP поддерживаются на Azure, но не нуждаются в минимальных выпусках ядра, так как изменения для Azure были реализованы с самого начала разработки более поздних стеков

Вам нужно проверить:

- О том, используются ли приложения SAP, которые вы используете, предусмотрены минимальные заявленные релизы. Если нет, необходимо определить новый целевой релиз, проверить в Матрице доступности продуктов SAP, какие сборки операционной системы и комбинации DBMS поддерживаются с новым целевым выпуском. Таким образом, что вы можете выбрать правильный релиз операционной системы и DBMS релиз
- Нужно ли обновлять ядра SAP при переходе в Azure
- Нужно ли обновлять пакеты поддержки SAP. Особенно базовые пакеты поддержки, которые могут потребоваться в тех случаях, когда требуется перейти к более недавнему выпуску DBMS


В следующем разделе более подробно рассказывается о других продуктах SAP и dBMS-релизах, которые поддерживаются SAP на Azure для Windows и Linux. 

> [!NOTE]
> Минимальные выпуски различных DBMS тщательно подобраны и не всегда могут отражать весь спектр DBMS релизы различных поставщиков DBMS поддержки на Azure в целом. Для определения этих минимальных выбросов были учтены многие соображения, связанные с рабочей нагрузкой SAP. Нет никаких усилий для тестирования и квалификации старых релизов DBMS. 

> [!NOTE]
> Минимальные перечисленные релизы представляют собой старую версию операционных систем и релизов баз данных. Мы настоятельно рекомендуем использовать самые последние выпуски операционных систем и релизы баз данных. Во многих случаях более поздние выпуски операционной системы и баз данных учитывали случай использования в общедоступном облаке и адаптировали код для оптимизации для работы в общедоступном облаке или, более конкретно, в Azure

## <a name="oracle-dbms-support"></a>Поддержка Oracle DBMS
Операционная система, релизы Oracle DBMS и функциональность Oracle, поддерживаемая Azure, конкретно указаны в [примечании поддержки SAP #2039619.](https://launchpad.support.sap.com/#/notes/2039619) Сущность из этой записки можно резюмировать как:

- Минимальный релиз Oracle, поддерживаемый на VMs Azure, сертифицированных для NetWeaver, является Oracle 11g Release 2 Patchset 3 (11.2.0.4)
- В качестве гостевых операционных систем к валидируют только Windows и Oracle Linux. Точные релизы ОС и связанные с ними минимальные релизы DBMS перечислены в примечании
- Поддержка Oracle Linux распространяется и на клиента Oracle DBMS. Это означает, что все компоненты SAP, такие как диалоговые экземпляры ABAP или Java Stack, также должны работать на Oracle Linux. Только компоненты SAP в такой системе SAP, которые не будут подключаться к Oracle DBMS, будут иметь возможность запускать другую операционную систему Linux
- Oracle RAC не поддерживается 
- Oracle ASM поддерживается в некоторых случаях. Подробности перечислены в заметке
- Не-Unicode SAP системы поддерживаются только с серверами приложений работает с Гостевой ОС Windows. Гостевой операционной системой DBMS может быть Oracle Linux или Windows. Причина этого ограничения очевидна при проверке матрицы доступности продуктов SAP (PAM). Для Oracle Linux SAP никогда не выпускала ядра, не относятсяе от Unicode SAP

Зная релизы DBMS, которые поддерживаются целевой инфраструктурой Azure, необходимо проверить матрицу доступности продуктов SAP, поддерживается ли релизы ОС и DBMS с помощью выпусков продуктов SAP, которые вы намеревались запустить. 


## <a name="sap-hana-support"></a>Поддержка SAP HANA
В Azure есть две службы, которые можно использовать для запуска базы данных HANA:

- Виртуальные машины Azure
- [HANA Большие инстанции](https://docs.microsoft.com/azure/virtual-machines/workloads/sap/hana-overview-architecture)

Для запуска SAP HANA SAP имеет больше и более сильные условия, необходимые для инфраструктуры, чем для запуска NetWeaver или других приложений SAP и DBMS. В результате меньшее число VMs Azure имеет право на запуск SAP HANA DBMS. Список поддерживаемой инфраструктуры Azure, поддерживаемой для SAP HANA, можно найти в так называемом [аппаратном каталоге SAP HANA.](https://www.sap.com/dmc/exp/2014-09-02-hana-hardware/enEN/iaas.html#categories=Microsoft%20Azure) 

> [!NOTE]
> Единицы, начиная с буквы "S" являются [HANA больших инстанций](https://docs.microsoft.com/azure/virtual-machines/workloads/sap/hana-overview-architecture) единиц. 

> [!NOTE]
> SAP не имеет специальной сертификации, зависящей от основных релизов SAP HANA. Вопреки распространенному мнению, **сценарий сертификации** столбца в [сертифицированных платформах IaaS,](https://www.sap.com/dmc/exp/2014-09-02-hana-hardware/enEN/iaas.html#categories=Microsoft%20Azure)колонка не делает **никаких заявлений о мажорном или незначительном выпуске, сертифицированном.** Вы должны предположить, что все перечисленные единицы, которые могут быть использованы для HANA 1.0 и HANA 2.0 до тех пор, пока сертифицированные релизы операционной системы для конкретных единиц поддерживаются релизами HANA 1.0. 

Для использования SAP HANA могут применяться различные минимальные релизы ОС, чем в общих случаях NetWeaver. Вы должны проверить поддерживаемые операционные системы для каждого подразделения индивидуально, так как они могут варьироваться. Вы делаете это, нажав на каждую единицу. Более подробная информация появится. Одной из перечисленных деталей является различные операционные системы, поддерживаемые для данного конкретного подразделения.

> [!NOTE]
> Подразделения Azure HANA Large Instance более ограничены в поддержке операционных систем по сравнению с MMs Azure. С другой стороны, Azure VMs могут применять более поздние операционные релизы в качестве минимальных релизов. Это особенно верно для некоторых крупных VM единиц, которые требуют изменений в ядрах Linux

Зная поддерживаемую ОС для инфраструктуры Azure, необходимо проверить [примечание поддержки SAP #2235581](https://launchpad.support.sap.com/#/notes/2235581) для точных выпусков SAP HANA и уровней исправлений, которые поддерживаются на единицах Azure, на которые вы ориентируетесь. 

> [!IMPORTANT]
> Шаг проверки точных релизов SAP HANA и поддерживаемых уровней патча очень важен. Во многих случаях поддержка определенного выпуска ОС зависит от определенного уровня патча исполнителей SAP HANA.

Как вы знаете, конкретные релизы HANA, которые можно запускать на целевой инфраструктуре Azure, необходимо проверить матрицу доступности продуктов SAP, чтобы выяснить, существуют ли ограничения с выпусками продуктов SAP, которые поддерживают отфильтрованные релизы HANA


## <a name="certified-azure-vms-and-hana-large-instance-units-and-business-transaction-throughput"></a>Сертифицированные Пазур VMs и HANA Large Instance единиц и пропускной записи бизнес-транзакций
Помимо оценки поддерживаемых релизов операционных систем, релизов DBMS и зависимых программных релизов SAP для подразделений инфраструктуры Azure, необходимо квалифицировать эти единицы по пропускной способности бизнес-транзакций, что выражается в блоке SAP SAP. Все размеры SAP зависят от расчетов SAPS. Оценивая существующие системы SAP, вы обычно можете с помощью поставщика инфраструктуры рассчитать SAPS единиц. Для слоя DBMS, а также для слоя приложения. В других случаях, когда создается новая функциональность, упражнение размеров с SAP может выявить требуемые номера SAPS для слоя приложения и слоя DBMS. Как поставщик инфраструктуры, корпорация Майкрософт обязана предоставить характеристику пропускной способности SAP различных единиц, которые либо NetWeaver и/или HANA сертифицированы.

Для M-м для пазла Azure эти номера пропускной способности SAPS задокументированы в [примечании поддержки SAP #1928533.](https://launchpad.support.sap.com/#/notes/1928533) Для подразделений Azure HANA Large Instance номера пропускной способности SAPS задокументированы в [примечаниях поддержки SAP #2316233](https://launchpad.support.sap.com/#/notes/2316233)

Глядя в [SAP поддержки отметить #1928533](https://launchpad.support.sap.com/#/notes/1928533), следующие замечания применяются:

- **Для M-Series Azure VMs и Mv2-Series Azure VMs применяются различные минимальные релизы ОС, чем для других типов Azure VM.** Требование для более поздних релизов ОС основано на изменениях, которые различные поставщики операционных систем должны были предоставить в своих выпусках операционной системы, чтобы либо включить их операционные системы, работающие на конкретных типах Azure VM, либо оптимизировать производительность и пропускную способность рабочей нагрузки SAP на этих типах VM
- Есть две таблицы, в которых указаны различные типы VM. Вторая таблица определяет пропускную способности SAPS для типов Azure VM, которые поддерживают только стандартное хранилище Azure. Развертывание DBMS на единицах, указанных во второй таблице заметки, не поддерживается


## <a name="other-sap-products-supported-on-azure"></a>Другие продукты SAP, поддерживаемые Azure

Тем не менее, и напротив частной визуализации облаков, SAP по-прежнему выражает поддержку различных продуктов SAP явно для различных поставщиков облаков hyerpscale.
