# <a name="scenario-reducer-is-slow-in-azure-hdinsight"></a>Сценарий: Редуктор медленно работает в Azure HDInsight

В этой статье описаны шаги устранения неполадок и возможные разрешения проблем при использовании компонентов интерактивного запроса в кластерах Azure HDInsight.

## <a name="issue"></a>Проблема

При запуске запроса, такого как `insert into table1 partition(a,b) select a,b,c from table2` план запроса, запускается куча редукторов, но данные из каждого раздела переходят в один редуктор. Это приводит к тому, что запрос будет таким же медленным, как и время, задеваемые редуктором крупнейшего раздела.

## <a name="cause"></a>Причина

Откройте [Билайн](../hadoop/apache-hadoop-use-hive-beeline.md) и проверьте `hive.optimize.sort.dynamic.partition`значение набора.

Значение этой переменной должно быть установлено на истинное/ложное в зависимости от характера данных.

Если разделы в таблице ввода меньше (скажем, меньше 10), и это число выходных разделов, `true`и переменная установлена, это приводит к тому, что данные будут глобально отсортированы и написаны с помощью одного редуктора на раздел. Даже если количество доступных редукторов больше, несколько редукторов могут отставать из-за перекоса данных и макс параллелизм не может быть достигнута. При изменении на, `false`более чем один редуктор может обрабатывать одну перегородку и несколько небольших файлов будут выписаны, в результате чего быстрее вставки. Это может повлиять на дальнейшие запросы, хотя из-за присутствия небольших файлов.

Значение `true` имеет смысл, когда количество разделов больше и данные не перекос. В таких случаях результат фазы карты будет выписан таким образом, что каждый раздел будет обрабатываться одним редуктором, что приведет к улучшению последующей производительности запроса.

## <a name="resolution"></a>Решение

1. Попробуйте переразделить данные, чтобы нормализовать сява в несколько разделов.

1. Если #1 не представляется возможным, установите значение конфигурации на ложное в сеансе Билайн и повторите попытку запроса. `set hive.optimize.sort.dynamic.partition=false`. Установка значения на ложное на уровне кластера не рекомендуется. Значение `true` является оптимальным и установить параметр по мере необходимости на основе характера данных и запроса.
