# <a name="release-notes"></a>Заметки о выпуске

Эта статья содержит сведения о **последних** обновлениях выпуска Azure HDInsight. Дополнительные сведения о предыдущих выпусках см. в статье [Заметки о выпуске для компонентов Hadoop в Azure HDInsight (архив)](hdinsight-release-notes-archive.md).

## <a name="summary"></a>Сводка

Azure HDInsight является одним из самых популярных сервисов среди корпоративных клиентов для аналитики с открытым исходным кодом в Azure.

## <a name="release-date-01092020"></a>Дата выпуска: 01/09/2020

Этот релиз применяется как для HDInsight 3.6 и 4.0. Релиз HDInsight доступен во всех регионах в течение нескольких дней. Дата релиза указана на первой дате выпуска региона. Если вы не видите ниже изменения, пожалуйста, подождите, пока релиз будет жить в вашем регионе в течение нескольких дней.

## <a name="new-features"></a>новые функции;
### <a name="tls-12-enforcement"></a>Обязательное использование TLS 1.2
TLS и SSL являются протоколами шифрования, которые обеспечивает безопасность передачи данных по сети. Узнайте больше о [TLS](https://en.wikipedia.org/wiki/Transport_Layer_Security#SSL_1.0.2C_2.0_and_3.0). HDInsight использует TLS 1.2 на общедоступных конечных точках HTTPs, но TLS 1.1 по-прежнему поддерживается для обратной совместимости. 

С помощью этого выпуска клиенты могут выбрать TLS 1.2 только для всех соединений через конечную точку общедоступного кластера. Для поддержки этого вводится новое **свойство minSupportedTlsVersion,** которые могут быть указаны при создании кластера. Если свойство не установлено, кластер по-прежнему поддерживает TLS 1.0, 1.1 и 1.2, что является таким же, как поведение сегодня. Клиенты могут установить значение для этого свойства до "1.2", что означает, что кластер поддерживает только TLS 1.2 и выше. Для получения дополнительной информации [см.](./transport-layer-security.md)

### <a name="bring-your-own-key-for-disk-encryption"></a>Принесите свой собственный ключ для шифрования диска
Все управляемые диски в HDInsight защищены с помощью шифрования службы хранилища Azure (SSE). Данные на этих дисках шифруются ключами, управляемыми корпорацией Майкрософт, по умолчанию. Начиная с этого выпуска, вы можете принести свой собственный ключ (BYOK) для шифрования диска и управлять им с помощью Azure Key Vault. Шифрование BYOK — это одноступенчатая конфигурация при создании кластера без дополнительных затрат. Просто зарегистрируйте HDInsight в качестве управляемой идентификации с Помощью Azure Key Vault и добавьте ключ шифрования при создании кластера. Для получения дополнительной [Customer-managed key disk encryption](https://docs.microsoft.com/azure/hdinsight/disk-encryption)информации см.

## <a name="upcoming-changes"></a>Предстоящие изменения
Следующие изменения произойдут в предстоящих релизах. 

### <a name="a-minimum-4-core-vm-is-required-for-head-node"></a>Для головного узла требуется минимальный 4-ядерный VM 
Для обеспечения высокой доступности и надежности кластеров HDInsight требуется минимальный 4-ядерный VM. Начиная с 6 апреля 2020 года, клиенты могут выбрать только 4-ядерный или выше VM в качестве головного узла для новых кластеров HDInsight. Существующие кластеры будут продолжать работать, как ожидалось. 

### <a name="esp-spark-cluster-node-size-change"></a>Изменение размера кластерного узла esP Spark 
В предстоящем выпуске минимальный допустимый размер узла для кластера ESP Spark будет изменен на Standard_D13_V2. VMs-серии могут вызвать проблемы кластера ESP из-за относительно низкой емкости процессора и памяти. Им., серия a, будет унитесадля для создания новых кластеров ESP.

### <a name="moving-to-azure-virtual-machine-scale-sets"></a>Переход на наборы виртуальных машин Azure
Теперь HDInsight использует виртуальные машины Azure для обеспечения кластера. В предстоящем выпуске HDInsight вместо этого будет использовать наборы виртуальных машин Azure. Узнайте больше о наборах виртуальных машин Azure.

### <a name="hbase-20-to-21"></a>HBase от 2,0 до 2,1
В предстоящем выпуске HDInsight 4.0 версия HBase будет обновлена с версии 2.0 до 2.1.
