# <a name="collaborative-coding-with-git"></a>Совместное программирование с помощью Git

В этой статье описывается, как использовать Git в качестве совместной платформы разработки кода для проектов в области науки о данных. В статье рассказывается о том, как связать код в Azure Repos с гибкими рабочими элементами [разработки](agile-development.md) в досках Azure, как выполнять обзоры кода, а также как создавать и объединять запросы на получение запросов на изменения.

## <a name="link-a-work-item-to-an-azure-repos-branch"></a><a name='Linkaworkitemwithagitbranch-1'></a>Связать рабочий элемент с вепой репоaz Azure 

Azure DevOps предоставляет удобный способ подключения элемента «История пользователя» или задачи «Лазурные доски» с ветвью репозитория Репозитория Репоса Azure. Вы можете связать свою историю пользователей или задачу непосредственно с кодом, связанным с ним. 

Чтобы соединить рабочий элемент с новой ветвью, выберите **«Действия** эллипсис»**(...)** рядом с рабочим элементом, а в контекстном меню прокрутите и выберите **новую ветку.**  

![1](./media/collaborative-coding-with-git/1-sprint-board-view.png)

В диалоге **«Создать ветвь»** укажите новое название ветви и базовое репозиторий и ветвь Azure Repos Git. Базовый репозиторий должен быть в том же проекте Azure DevOps, что и рабочий элемент. Базовая ветвь может быть основной ветвью или другой существующей ветвью. Выберите **Создать ветку**. 

![2](./media/collaborative-coding-with-git/2-create-a-branch.png)

Вы также можете создать новую ветку, используя следующую команду Git Bash в Windows или Linux:

```bash
git checkout -b <new branch name> <base branch name>

```
Если вы не укажете имя базовой \<ветви>, новая ветвь основана на `master`. 

Чтобы переключиться на рабочую ветку, запустите следующую команду: 

```bash
git checkout <working branch name>
```

После переключения на рабочую ветвь можно приступить к разработке артефактов кода или документации для завершения элемента работы. Запуск `git checkout master` переключает вас `master` обратно в ветку.

Создание ветки Git для каждого элемента работы «История пользователя» является хорошей практикой. Затем для каждого элемента задачи можно создать ветку на основе ветви «История пользователя». Организуйте ветви в иерархии, которая соответствует отношениям «История пользователя и задачи», когда у вас есть несколько человек, работающих над разными историями пользователей для одного и того же проекта или над различными задачами для одной и той же истории пользователя. Можно свести к минимуму конфликты, заставив каждого члена группы работать в другой ветке, или над другим кодом или другими артефактами при совместном использовании ветки. 

На следующей диаграмме показана рекомендуемая стратегия ветвления для TDSP. Возможно, вам не понадобится столько ветвей, как показано здесь, особенно когда над проектом работает только один или два человека или только один человек работает над всеми задачами истории пользователя. Но отделение ветви разработки от основной ветви всегда является хорошей практикой и может помочь предотвратить прерывание ветки выпуска деятельностью разработки. Для полного описания модели ветви Git [см.](https://nvie.com/posts/a-successful-git-branching-model/)

![3](./media/collaborative-coding-with-git/3-git-branches.png)

Также вы можете связать рабочий элемент с существующей ветвью. На странице **Подробной** части рабочего элемента выберите **ссылку Добавить**. Затем выберите существующую ветку, с вспять, и выберите **OK.** 

![4](./media/collaborative-coding-with-git/4-link-to-an-existing-branch.png)

## <a name="work-on-the-branch-and-commit-changes"></a><a name='WorkonaBranchandCommittheChanges-2'></a>Работа на ветке и фиксация изменений 

После вашего изменения для рабочего элемента, например, добавления `script` файла R-скрипта в ветку локальной машины, можно совершить изменение из локальной ветви в рабочую ветвь вверх по течению, используя следующие команды Git Bash:

```bash
git status
git add .
git commit -m "added an R script file"
git push origin script
```

![5](./media/collaborative-coding-with-git/5-sprint-push-to-branch.png)

## <a name="create-a-pull-request"></a><a name='CreateapullrequestonVSTS-3'></a>Создание запроса на вытягивание

После одного или нескольких коммитов и толчков, когда вы будете готовы объединить текущую рабочую ветвь в базовую ветвь, можно создать и отправить *запрос на вытягивание* в Azure Repos. 

С главной страницы проекта Azure DevOps укажите**запросы** **Repos** > Pull в левой навигации. Затем выберите одну из новых кнопок **запроса на вытягивание** или **ссылку «Создать ссылку запроса на вытягивание».**

![6](./media/collaborative-coding-with-git/6-spring-create-pull-request.png)

На **экране Нового запроса Pull** при необходимости перейдите к репозиторию Git и ветке, в который вы хотите объединить изменения. Добавить или изменить любую другую информацию, которая вам нужна. Под **рецензентами**добавляйте имена рецензентов, а затем выберите **Create.** 

![7](./media/collaborative-coding-with-git/7-spring-send-pull-request.png)

## <a name="review-and-merge"></a><a name='ReviewandMerge-4'></a>Проверка и слияние

Как только вы создаете запрос на вытягивание, рецензенты получают уведомление по электронной почте для рассмотрения запроса на вытягивание. Рецензенты проверяют, работают ли изменения, и проверяют изменения с проситерем, если это возможно. Рецензенты могут делать комментарии, запрашивать изменения, а также утверждать или отклонять запрос на вытягивание на основе их оценки. 

![8](./media/collaborative-coding-with-git/8-add_comments.png)

После утверждения рецензентами изменений вы или кто-то другой с разрешениями слияния может объединить рабочую ветвь в базовую ветвь. Выберите **Полное,** а затем выберите **Полное слияние** в диалоге **запроса «Полный тяну».** Вы можете удалить рабочую ветвь после ее слияния. 

![10](./media/collaborative-coding-with-git/10-spring-complete-pullrequest.png)

Подтвердите, что запрос помечен как **COMPLETED**. 

![11](./media/collaborative-coding-with-git/11-spring-merge-pullrequest.png)

Когда вы возвращаетесь к **Репосу** в левой навигации, вы можете `script` увидеть, что вы были переключены на ветку master, так как ветка была удалена.

![12](./media/collaborative-coding-with-git/12-spring-branch-deleted.png)

Можно также использовать следующие команды Git `script` Bash, чтобы объединить рабочую ветвь с базовой ветвью и удалить рабочую ветвь после слияния:

```bash
git checkout master
git merge script
git branch -d script
```

![13](./media/collaborative-coding-with-git/13-spring-branch-deleted-commandline.png)
