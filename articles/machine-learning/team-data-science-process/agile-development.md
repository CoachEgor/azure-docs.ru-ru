# <a name="agile-development-of-data-science-projects"></a>Гибкая разработка проектов обработки и анализа данных

Из этого документа вы узнаете, как с помощью [командного процесса обработки и анализа данных](overview.md) (TDSP) разработчики могут выполнять проекты обработки и анализа данных с соблюдением системного подхода, контролем версий и поддержкой совместной работы. TDSP — это платформа, разработанная корпорацией Майкрософт, которая обеспечивает структурированную последовательность действий для эффективного выполнения облачных решений для прогнозной аналитики. Для набросков ролей и задач, которые обрабатываются группой по анализу данных, стандартизирующей TDSP, [см.](roles-tasks.md) 

Из этой статьи вы узнаете, как: 

- Планируя *спринт* для рабочих элементов, участвующих в проекте.
- Добавление *рабочих элементов* в спринты.
- Создайте и используйте *шаблон рабочего элемента гибкого получения,* который конкретно соответствует этапам жизненного цикла TDSP.

В следующих инструкциях описаны шаги, необходимые для создания среды группы TDSP с использованием советов Azure и Репо Azure в Azure DevOps. Инструкции используют Azure DevOps, потому что именно так можно реализовать TDSP в корпорации Майкрософт. Если группа использует другую платформу хостинга кода, задачи руководства группы обычно не меняются, но способ выполнения задач другой. Например, связывание рабочего элемента с ветки Git может быть не одинаковым с GitHub, как с Azure Repos.

Следующая цифра иллюстрирует типичный процесс планирования спринта, кодирования и рабочего процесса управления исходным кодом для проекта по науке о данных:

![Командный процесс обработки и анализа данных](./media/agile-development/1-project-execute.png)

##  <a name="work-item-types"></a><a name='Terminology-1'></a>Типы рабочих элементов

В рамках планирования спринта TDSP есть четыре часто используемых типа *рабочих элементов:* *функции,* *истории пользователей,* *Задачи*и *ошибки.* Отставание по всем рабочим элементам находится на уровне проекта, а не на уровне репозитория Git. 

Ниже приведены определения типов элементов работы:

- **Характеристика**: Функция соответствует вовлеченности проекта. Различные взаимодействия с клиентом являются различными функциями, и лучше всего рассматривать различные этапы проекта как различные функции. Если вы выберете схему, такую как * \<ClientName>-\<EngagementName>* назвать свои функции, вы можете легко распознать контекст проекта и вовлеченность по самим именам.
  
- **История пользователя:** Истории пользователей — это рабочие элементы, необходимые для завершения сквозного выполнения функции. Примеры историй пользователей:
  - Получение данных 
  - Анализ данных 
  - Создание функций
  - Создание моделей
  - Ввод модели в эксплуатацию 
  - Повторное обучение моделей
  
- **Задача**: Задачи — это присваиваемые рабочие элементы, которые необходимо сделать для завершения конкретной истории пользователей. Например, задачи в истории пользователя *Получить данные* могут быть:
  - Получите учетные данные сервера S'L
  - Загрузка данных в Хранилище данных S'L
  
- **Ошибки**: Ошибки — это проблемы в существующем коде или документах, которые должны быть исправлены для выполнения задачи. Если ошибки вызваны отсутствием рабочих элементов, они могут перерасти в истории пользователей или задачи. 

Специалисты по обработке данных могут чувствовать себя более комфортно, используя гибкий шаблон, который заменяет функции, истории пользователей и задачи на этапах жизненного цикла TDSP и подступений. Для создания шаблона гибкого получения, который конкретно соответствует этапам жизненного цикла TDSP, [см.](#set-up-agile-dsp-6)

> [!NOTE]
> TDSP заимствует концепции функций, историй пользователей, задач и ошибок из управления программным кодом (SCM). Концепции TDSP могут несколько отличаться от их обычных определений СКМ.

## <a name="plan-sprints"></a><a name='SprintPlanning-2'></a>Планирование спринтов

Многие специалисты по обработке данных занимаются несколькими проектами, которые могут занять несколько месяцев, чтобы завершить и продолжить в разных темпах. Планирование спринта полезно для определения приоритетов, планирования и выделения ресурсов по проекту. В Azure Boards вы можете легко создавать, управлять и отслеживать рабочие элементы для своих проектов, а также проводить планирование спринта, чтобы гарантировать, что проекты продвигаются вперед, как ожидалось.

Для получения дополнительной информации о [планировании спринта](https://en.wikipedia.org/wiki/Scrum_(software_development)#Sprint)см. 

Для получения дополнительной информации о планировании спринта в досках Azure [см.](/azure/devops/boards/sprints/assign-work-sprint) 

## <a name="add-a-feature-to-the-backlog"></a><a name='AddFeature-3'></a>Добавление функции в невыполненную книжку 

После создания репозитория проекта и кода проекта можно добавить функцию в невыполненную работу, чтобы представить работу для вашего проекта.

1. На странице проекта выберите невыполненные **работы в** > левой**навигации.** 
   
1. На вкладке **Невыполненная работа,** если тип элемента работы в верхней панели **— Это Истории,** упадите и выберите **функции.** Затем выберите **новый элемент работы.**
   
   ![Выберите новый пункт работы](./media/agile-development/2-sprint-team-overview.png)
   
1. Введите заголовок для функции, обычно название проекта, а затем выберите **Добавить в верхней .** 
   
   ![Введите заголовок и выберите Добавить к началу](./media/agile-development/3-sprint-team-add-work.png)
   
1. Из списка **невыполненных работ** выберите и откройте новую функцию. Заполните описание, назначайте члена команды и установите параметры планирования. 
   
   Вы также можете связать функцию с репозиторием кода Azure Repos проекта, выбрав **ссылку Add** в разделе **Разработка.** 
   
   После того, как функция будет отсечена, выберите **Сохранить & закрыться.**
   
   ![Отсечение функции и выберите Сохранить & Закрыть](./media/agile-development/3a-add-link-repo.png)

## <a name="add-a-user-story-to-the-feature"></a><a name='AddStoryunderfeature-4'></a>Добавление функциональной функции пользователя 

В соответствии с функцией можно добавить «Истории пользователей» для описания основных шагов, необходимых для завершения проекта. 

Чтобы добавить новую историю пользователя в функцию:

1. На вкладке **"Невыполненная группа"** выберите **+** слева от функции. 
   
   ![Добавить новую историю пользователя под функцией](./media/agile-development/4-sprint-add-story.png)
   
1. Дайте истории пользователя заголовок и отодевите такие сведения, как назначение, статус, описание, комментарии, планирование и приоритет. 
   
   Вы также можете связать «Историю пользователя» с ветвью репозитория репозитория репозитория репосу с помощью azure Repos проекта, выбрав **ссылку Add** в разделе **«Разработка».** Выберите репозиторий и ветвь, с которые необходимо связать элемент работы, а затем выберите **OK.**
   
   ![Добавление связи](./media/agile-development/5-sprint-edit-story.png)
   
1. Когда вы закончите редактирование истории пользователя, выберите **Сохранить & закрыть**. 

## <a name="add-a-task-to-a-user-story"></a><a name='AddTaskunderstory-5'></a>Добавление задачи в историю пользователя 

Задачи представляют собой конкретные подробные шаги, необходимые для завершения каждой истории пользователя. После завершения всех задач истории пользователей также должна быть завершена и история пользователя. 

Чтобы добавить задачу в историю пользователя, выберите **+** следующий элемент «История пользователя» и выберите **задачу.** Заполните название и другую информацию в задаче.

![Добавление задачи в историю пользователя](./media/agile-development/7-sprint-add-task.png)

После создания функций, историй пользователей и задач можно просмотреть их в **представлениях невыполненных** работ или **в таблицах советов** для отслеживания их статуса.

## <a name="use-an-agile-tdsp-work-template"></a><a name='set-up-agile-dsp-6'></a>Используйте гибкий шаблон работы TDSP

Специалисты по обработке данных могут чувствовать себя более комфортно, используя гибкий шаблон, который заменяет функции, истории пользователей и задачи на этапах жизненного цикла TDSP и подступений. В досках Azure можно создать шаблон гибкого получения, который использует этапы жизненного цикла TDSP для создания и отслеживания элементов работы. Следующие шаги проходят через создание шаблона гибкого процесса обработки данных и создание элементов работы по науке данных на основе шаблона.

### <a name="set-up-an-agile-data-science-process-template"></a>Настройка шаблона agile Data Science Process

1. На главной странице организации Azure DevOps выберите **настройки организации** из левой навигации. 
   
1. В **настройках организации** левая навигация, под **досками,** выберите **процесс**. 
   
1. В **All processes** панели, выберите **Agile** **...** **Create inherited process**
   
   ![Создание унаследованного процесса из Agile](./media/agile-development/10-settings.png) 
   
1. В **процессе создания, унаследованный от Agile** диалог, введите имя *AgileDataScienceProcess*и выберите **процесс создания.**
   
   ![Создание процесса AgileDataScienceProcess](./media/agile-development/11-agileds.png)
   
1. Во **всех процессах**выберите новый **AgileDataScienceProcess.** 
   
1. На вкладке **"Рабочий элемент"** отображайте **Epic,** **Feature,** **User Story**и **Task,** выбрав **...** **Disable** 
   
   ![Отключить типы элементов работы](./media/agile-development/12-disable.png)
   
1. Во **всех процессах**выберите вкладку **уровней невыполненной работы.** Под **невыполненными работами по портфелям**выберите **...** **Epic (disabled)** **Edit/Rename** 
   
1. В диалоговом окне **уровня Edit:**
   1. Под **именем,** заменить **Epic** с *TDSP проектов*. 
   1. Под **рабочими типами элементов на этом уровне невыполненной работы**выберите **новый тип элемента работы,** введите *tDSP Project*и **выберите Добавить.** 
   1. При **типе элемента работы по умолчанию,** падение вниз и выберите **TDSP проекта.** 
   1. Нажмите кнопку **Сохранить**.
   
   ![Установить уровень отставания портфеля](./media/agile-development/13-rename.png)  
   
1. Выполните те же шаги, чтобы переименовать **функции** в *TDSP Этапы*, и добавить следующие новые типы элементов работы:
   
   - *Понимание бизнеса*
   - *Сбор данных.*
   - *Моделирование*
   - *Развертывание*
   
1. В соответствии с **невыполненным требованием,** переименовайте **Истории** в *подставы TDSP,* добавьте новый тип рабочего элемента *TDSP Substage*и установите тип элемента работы по умолчанию в **Подставы TDSP.**
   
1. В **невыполненной работе итерации**добавьте новый тип *элемента работы TDSP Task*и установите его в тип рабочего элемента по умолчанию. 
   
После завершения выполнения шагов уровни невыполненной работы должны выглядеть следующим образом:
   
 ![Уровни невыполненной работы шаблонов TDSP](./media/agile-development/14-template.png)  

### <a name="create-agile-data-science-process-work-items"></a>Создание рабочих элементов гибкого процесса обработки данных

Шаблон процесса обработки данных можно использовать для создания проектов TDSP и отслеживания рабочих элементов, соответствующих этапам жизненного цикла TDSP.

1. На главной странице организации Azure DevOps выберите **новый проект.** 
   
1. В диалоге **Нового проекта Создайте новый проект,** дайте проекту имя, а затем выберите **Advanced**. 
   
1. В **процессе работы элемента,** падение вниз и выбрать **AgileDataScienceProcess**, а затем выбрать **Создать.**
   
   ![Создание проекта TDSP](./media/agile-development/15-newproject.png)
   
1. В недавно созданном проекте выберите **невыполненные работы в** > левой**навигации.**
   
1. Чтобы сделать проекты TDSP видимыми, выберите значок **настроек настройки группы.** На экране **настройки** выберите флажок **TDSP Projects,** а затем выберите **Сохранить и закрыть.**
   
   ![Выберите флажок TDSP Projects](./media/agile-development/16-enabledsprojects1.png)
   
1. Для создания проекта TDSP, специфичного для данных, выберите **проекты TDSP** в верхней панели, а затем выберите **новый элемент работы.** 
   
1. Во всплывающем окне привить элементу работы проекта TDSP имя и **выберите Добавить сверху.**
   
   ![Создание элемента проекта по науке о данных](./media/agile-development/17-dsworkitems0.png)
   
1. Чтобы добавить рабочий элемент в проект **+** TDSP, выберите следующий к проекту, а затем выберите тип элемента работы для создания. 
   
   ![Выберите тип элемента работы науки о данных](./media/agile-development/17-dsworkitems1.png)
   
1. Заполните сведения в новом элементе работы и выберите **Сохранить & Закрыть**.
   
1. Продолжайте выбирать **+** символы рядом с рабочими элементами, чтобы добавить новые этапы TDSP, подэтапы и задачи.
