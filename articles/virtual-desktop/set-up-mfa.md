# <a name="set-up-azure-multi-factor-authentication"></a>Настройка Многофакторной идентификации Azure

Клиент Windows для виртуальных рабочих столов для Windows — это отличный вариант для интеграции виртуальных рабочих столов Windows с локальным компьютером. Однако при настройке учетной записи виртуального рабочего стола Windows в клиенте Windows существуют определенные меры, которые необходимо предпринять, чтобы обеспечить безопасность своих пользователей.

При первом входе клиент запрашивает имя пользователя, пароль и Azure MFA. После этого при следующем входе клиент запомнит ваш токен из приложения Azure Active Directory (AD) Enterprise. При выборе **Запомнить меня**пользователи смогут войти в систему после перезапуска клиента без необходимости повторного ввода учетных данных.

Хотя запоминание учетных данных удобно, это также может сделать развертывание на корпоративных сценариях или персональных устройствах менее безопасным. Чтобы защитить пользователей, необходимо убедиться, что клиент запрашивает учетные данные многофакторной идентификации Azure (MFA). В этой статье показано, как настроить политику условного доступа для виртуальных рабочих столов Windows, чтобы включить этот параметр.

## <a name="prerequisites"></a>Предварительные условия

Чтобы приступить к работе, необходимо выполнить следующие действия.

- Назначьте всем пользователям одну из следующих лицензий:
  - Microsoft 365 E3 или в.
  - Azure Active Directory Premium P1 или P2
  - Enterprise Mobility + Security E3 или в.
- Группа Azure Active Directory с пользователями, назначенными членами группы.
- Включите Azure MFA для всех пользователей. Дополнительные сведения о том, как это сделать, см. [в разделе как требовать двухфакторную проверку подлинности для пользователя](../active-directory/authentication/howto-mfa-userstates.md#view-the-status-for-a-user).

>[!NOTE]
>Следующий параметр также применяется к [веб-клиенту виртуальных рабочих столов Windows](https://rdweb.wvd.microsoft.com/webclient/index.html).

## <a name="opt-in-to-the-conditional-access-policy"></a>Согласие на политику условного доступа

1. Откройте **Azure Active Directory**.

2. Перейдите на вкладку **все приложения** . В раскрывающемся меню "тип приложения" выберите **корпоративные приложения**, а затем найдите **клиент виртуальных рабочих столов Windows**.

    ![Снимок экрана вкладки "все приложения". Пользователь указал "клиент виртуальных рабочих столов Windows" в строке поиска, и приложение отобразится в результатах поиска.](media/all-applications-search.png)

3. Выберите **Условный доступ**.

    ![Снимок экрана, показывающий, что пользователь наводит указатель мыши на вкладку условный доступ.](media/conditional-access-location.png)

4. Выберите **+ создать политику**.

   ![Снимок экрана со страницей условного доступа. Пользователь наводит указатель мыши на кнопку Создать политику.](media/new-policy-button.png)

5. Введите **имя** **правила**, а затем **выберите** имя **группы** , созданной в списке необходимые компоненты.

6. Выберите **выбрать**, а затем нажмите кнопку **Готово**.

7. Затем откройте **облачные приложения или действия**.

8. На панели **Выбор** выберите приложение **Windows виртуальный рабочий стол** корпоративное.

    ![Снимок экрана со страницей "облачные приложения" или "действия". Пользователь выбрал приложение Windows Virtual Desktop, установив флажок рядом с ним. Выбранное приложение выделено красным цветом.](media/cloud-apps-select.png)
    
    >[!NOTE]
    >Кроме того, в левой части экрана должно отобразиться клиентское приложение виртуальных рабочих столов Windows, как показано на следующем рисунке. Для работы политики необходимы как виртуальные рабочие столы Windows, так и клиентские корпоративные приложения Windows.
    >
    > ![Снимок экрана со страницей "облачные приложения" или "действия". Клиентские приложения Windows для виртуальных рабочих столов и виртуальных рабочих столов Windows выделены красным цветом.](media/cloud-apps-enterprise-selected.png)

9. Нажмите **кнопку Выбрать** .

10. Затем откройте **предоставление** 

11. Выберите **требовать многофакторную проверку подлинности**, а затем выберите **требовать один из выбранных элементов управления**.
   
    ![Снимок экрана страницы предоставления разрешений. Выбрано значение "требовать многофакторную проверку подлинности".](media/grant-page.png)

    >[!NOTE]
    >Если в вашей организации есть зарегистрированные в MDM устройства, которые не должны отображать запрос MFA, можно также выбрать параметр **требовать, чтобы устройство было помечено как соответствующее**.

12. Выберите **сеанс**.

13. Установите для параметра **Частота входа** значение **активно**, а затем задайте для значения **1 час**.

    ![Снимок экрана страницы сеанса. В меню сеанса отображаются раскрывающиеся меню частота входа, измененные на "1" и "Часы".](media/sign-in-frequency.png)
   
    >[!NOTE]
    >Активные сеансы в среде виртуальных рабочих столов Windows продолжат работать при изменении политики. Однако если вы отключится или выйдете из системы, вам потребуется ввести учетные данные еще раз через 60 минут. При изменении параметров можно продлить время ожидания (при условии, что оно соответствует политике безопасности Организации).
    >
    >Значение по умолчанию — это период в 90 дней. Это означает, что клиент попросит пользователя войти в систему снова при попытке доступа к ресурсу после того, как он будет неактивен на своем компьютере в течение 90 дней или дольше.

14. Включите политику.

15. Выберите **создать** , чтобы подтвердить политику.

Готово! Вы можете протестировать политику, чтобы убедиться, что список разрешений работает должным образом.
