## <a name="for-users-in-your-directory"></a>Для пользователей в каталоге

Выполните следующие действия, если вы хотите, чтобы пользователи в вашем каталоге, чтобы иметь возможность запросить этот пакет доступа. При определении политики запроса можно указать отдельных пользователей или чаще всего группы пользователей. Например, в вашей организации уже может быть группа, например **все сотрудники.**  Если эта группа добавлена в политику для пользователей, которые могут запросить доступ, то любой член этой группы может запросить доступ.

1. В разделе **Пользователи, которые могут запрашивать доступ** щелкните **Для пользователей в каталоге**.

    При выборе этой опции появляются новые опции для дальнейшего уточнения того, кто в вашем каталоге может запросить этот пакет доступа.

    ![Пакет доступа - Запросы - Для пользователей в каталоге](./media/active-directory-entitlement-management-request-policy/for-users-in-your-directory.png)

1. Выберите один из следующих вариантов.

    |  |  |
    | --- | --- |
    | **Конкретные пользователи и группы** | Выберите эту опцию, если в каталоге нужны только пользователи и группы, указанные вами, чтобы иметь возможность запросить этот пакет доступа. |
    | **Все участники (за исключением гостей)** | Выберите эту опцию, если вы хотите, чтобы все пользователи-члены в вашем каталоге могли запросить этот пакет доступа. Эта опция не включает в свой каталог приглашенных пользователей, которых вы могли бы пригласить. |
    | **Все пользователи (включая гостей)** | Выберите эту опцию, если вы хотите, чтобы все пользователи-члены и гостевые пользователи в вашем каталоге могли запросить этот пакет доступа. |

    Гость пользователи ссылаются на внешних пользователей, которые были приглашены в ваш каталог с [Azure AD B2B.](../articles/active-directory/b2b/what-is-b2b.md) Для получения дополнительной информации о различиях между пользователями-членами и гостевыми пользователями [смотрите, что такое пользовательские разрешения по умолчанию в Active Directory Azure?.](../articles/active-directory/fundamentals/users-default-permissions.md)

1. Если вы выбрали **конкретных пользователей и группы,** нажмите **Добавить пользователей и групп.**

1. В панели выбора пользователей и групп выберите пользователей и групп, которые вы хотите добавить.

    ![Пакет доступа - Запросы - Выберите пользователей и группы](./media/active-directory-entitlement-management-request-policy/select-users-groups.png)

1. Нажмите **Выберите,** чтобы добавить пользователей и группы.

1. Перейдите в раздел [Утверждение.](#approval)

## <a name="for-users-not-in-your-directory"></a>Для пользователей, не всданных в каталоге

 **Пользователи, не входящий в ваш каталог,** относятся к пользователям, которые находятся в другом каталоге или домене Azure AD. Возможно, эти пользователи еще не были приглашены в ваш каталог. Каталоги Azure AD должны быть настроены таким образом, чтобы можно было разрешить приглашения в **ограничениях совместной работы.** Для получения дополнительной информации [см.](../articles/active-directory/b2b/delegate-invitations.md)

> [!NOTE]
> Учетная запись гостевого пользователя будет создана для пользователя, еще не вашего каталога, запрос которого одобрен или автоматически одобрен. Гость будет приглашен, но не получит приглашение по электронной почте. Вместо этого они получат электронное письмо, когда будет доставлено их назначение пакета доступа. По умолчанию, позже, когда у этого гостевого пользователя больше нет назначений пакета доступа, поскольку их последнее назначение истекло или отменено, эта учетная запись пользователя гостя будет заблокирована и впоследствии удалена. Если вы хотите, чтобы гостевые пользователи оставались в каталоге на неопределенный срок, даже если у них нет назначений пакета доступа, можно изменить настройки для конфигурации управления правами. Для получения дополнительной информации об объекте гостевого пользователя смотрите [свойства пользователя совместной работы Azure Active Directory B2B.](../articles/active-directory/b2b/user-properties.md)

Выполните следующие действия, если вы хотите разрешить пользователям, не всданным в вашем каталоге, запрашивать этот пакет доступа:

1. В **разделе Пользователей, которые могут запросить раздел доступа,** нажмите **Для пользователей, не всданный в вашем каталоге.**

    При выборе этой опции появляются новые параметры.

    ![Пакет доступа - Запросы - Для пользователей, не всданных в каталоге](./media/active-directory-entitlement-management-request-policy/for-users-not-in-your-directory.png)

1. Выберите один из следующих вариантов.

    |  |  |
    | --- | --- |
    | **Конкретные связанные организации** | Выберите эту опцию, если вы хотите выбрать из списка организаций, которые администратор ранее добавил. Все пользователи из выбранных организаций могут запросить этот пакет доступа. |
    | **Все подключенные организации** | Выберите эту опцию, если все пользователи из всех подключенных организаций могут запросить этот пакет доступа. |
    | **Все пользователи (все подключенные организации и любые новые внешние пользователи)** | Выберите эту опцию, если все пользователи из всех подключенных организаций могут запросить этот пакет доступа и что B2B позволяет или отрицает настройки списка, должны иметь приоритет для любого нового внешнего пользователя. |

    Связанная организация — это внешний каталог Azure AD или домен, с которым вы имеете отношения.

1. Если вы выбрали **организации, подключенные к конкретным,** нажмите **«Добавить каталоги»,** чтобы выбрать из списка подключенных организаций, которые администратор ранее добавил.

1. Введите имя или доменное имя для поиска ранее подключенной организации.

    ![Пакет доступа - Запросы - Выберите каталоги](./media/active-directory-entitlement-management-request-policy/select-directories.png)

    Если организации, с помощью которого вы хотите сотрудничать, нет в списке, вы можете попросить администратора добавить ее как связанную организацию. Для получения дополнительной информации [см.](../articles/active-directory/governance/entitlement-management-organization.md)

1. Как только вы выберете все подключенные организации, нажмите **Выберите**.

    > [!NOTE]
    > Все пользователи из выбранных подключенных организаций смогут запросить этот пакет доступа. Это включает пользователей azure AD из всех поддоменов, связанных с организацией, если эти домены не заблокированы Azure B2B, разрешающих или опровергающих список. Дополнительные сведения см. в статье [Предоставление или отзыв приглашений пользователям B2B из отдельных организаций](../articles/active-directory/b2b/allow-deny-list.md).

1. Перейдите в раздел [Утверждение.](#approval)

## <a name="none-administrator-direct-assignments-only"></a>Ни одно (только прямые назначения администратора)

Выполните следующие действия, если вы хотите обойти запросы доступа и позволить администраторам непосредственно назначать конкретных пользователей в этот пакет доступа. Пользователям не придется запрашивать пакет доступа. Можно настроить настройки жизненного цикла, но нет настроек запроса.

1. В **пользователях, которые могут запросить раздел доступа,** нажмите **Нет (администратор прямых назначений только**.

    ![Пакет доступа - Запросы - Ни один администратор прямых назначений только](./media/active-directory-entitlement-management-request-policy/none-admin-direct-assignments-only.png)

    После создания пакета доступа можно напрямую назначить определенных внутренних и внешних пользователей пакету доступа. Если вы укажете внешнего пользователя, в вашем каталоге будет создана учетная запись гостевого пользователя. Для получения информации о непосредственном назначении пользователя, [см. Посмотреть, добавить и удалить назначения для пакета доступа.](../articles/active-directory/governance/entitlement-management-access-package-assignments.md)

1. Перейдите в раздел [Запросы включить.](#enable-requests)

## <a name="approval"></a>Утверждение

В разделе Утверждение указывается, требуется ли утверждение, когда пользователи запрашивают этот пакет доступа. Настройки утверждения работают следующим образом:

- Только один из выбранных одобрителей или резервных одобрителей должен утвердить запрос на одноэтапное утверждение. 
- Только один из выбранных утверждений на каждом этапе должен утвердить запрос на 2-этапное утверждение.
- Утверждение может быть менеджером, внутренним спонсором или внешним спонсором в зависимости от того, кто регулирует доступ политики.
- Утверждение от каждого выбранного утверждателя не требуется для одно- или 2-этапного утверждения.
- Решение об утверждении основано на том, какой бы утверждаетель не пересматривал запрос в первую очередь.

Для демонстрации того, как добавить утверждение в политику запроса, смотрите следующее видео:

>[!VIDEO https://www.microsoft.com/videoplayer/embed/RE4cZfg]

Для демонстрации того, как добавить многоступенчатое утверждение в политику запроса, смотрите следующее видео:

>[!VIDEO https://www.microsoft.com/videoplayer/embed/RE4d1Jw]

Выполните следующие действия, чтобы указать параметры утверждения запросов на пакет доступа:

1. Чтобы получить одобрение запросов от выбранных пользователей, установите переключатель **утверждения Требование** **на Yes.** Или, чтобы запросы были автоматически одобрены, установите переключатель в **No.**

1. Чтобы потребовать от пользователей предоставить обоснование запроса пакета доступа, установите **обоснование запроса "Требуемое"** переключиться на **Yes.**
    
1. Теперь определите, потребуют ли запросы одно- или 2-этапного утверждения. Установите **сколько этапов** переключения до **1** для утверждения одного этапа или установить переключатель до **2** для 2-ступенчатого утверждения.

    ![Пакет доступа - Запросы - Настройки утверждения](./media/active-directory-entitlement-management-request-policy/approval.png)

Используйте следующие шаги, чтобы добавить утверждающих после выбора, сколько этапов требуется: 

### <a name="single-stage-approval"></a>Одноэтапное утверждение

1. Добавить **первого утвердитель:**
    
    Если политика установлена для управления доступом для пользователей в каталоге, вы можете выбрать **менеджера в качестве утверждающего.** Или добавьте определенного пользователя, нажав **на утверждение Добавить** после выбора Выберите конкретных удобрятелей из меню выпадения.
    
    ![Пакет доступа - Запросы - Для пользователей в каталоге - Первый Правообладатель](./media/active-directory-entitlement-management-request-policy/approval-single-stage-first-approver-manager.png)

    Если эта политика установлена для регулирования доступа для пользователей, не ввашем каталоге, вы можете выбрать **внешнего спонсора** или **внутреннего спонсора.** Или добавьте конкретного пользователя, нажав **на утверждение Добавить** или группы в рамках «Выбрать конкретных одобрителей».
    
    ![Пакет доступа - Запросы - Для пользователей из каталога - Первый Утверждение](./media/active-directory-entitlement-management-request-policy/out-directory-first-approver.png)
    
1. Если вы выбрали **менеджера** в качестве первого одобрителя, нажмите **Добавить запасной вариант,** чтобы выбрать одного или нескольких пользователей или групп в каталоге, чтобы быть резервным атвердителем. Правообладатели получают запрос, если руководство по праву не может найти менеджера для пользователя, запрашивающего доступ.

    Менеджер обнаруживается по управлению правом с помощью атрибута **Manager.** Атрибут находится в профиле пользователя в Azure AD. Для получения дополнительной информации [см. Добавить или обновить информацию о профиле пользователя с помощью Active Directory Azure.](../articles/active-directory/fundamentals/active-directory-users-profile-azure-portal.md)

1. Если вы выбрали Выбранных лиц, набрав **определенных одобрителей,** нажмите **Add approvers,** чтобы выбрать одного или нескольких пользователей или групп в вашем каталоге для утверждения.

1. В поле в соответствии с **решением должно быть сделано в течение нескольких дней?**, указать количество дней, что одобритель должен рассмотреть запрос на этот пакет доступа.

    Если запрос не будет одобрен в течение этого периода времени, он будет автоматически отклонен. Пользователь должен будет отправить еще один запрос на пакет доступа.

1. Чтобы требовать, чтобы утверждая предоставить обоснование для их решения, установить Требование утверждения обоснование **Да**.

    Обоснование отображается другим и запрашивает.

1. Добавить **второго утвердитель:** 
    
    Если пользователи находятся в вашем каталоге, добавьте конкретного пользователя в качестве второго одобрителя, нажав **на утверждение Добавить** в соответствии с Select specific approvers.

    ![Пакет доступа - Запросы - Для пользователей в каталоге - Второй Удобрятель](./media/active-directory-entitlement-management-request-policy/in-directory-second-approver.png)

    Если пользователи не в вашем каталоге, выберите **Внутреннего спонсора** или **внешнего спонсора** в качестве второго одобрителя. После выбора одобрителя добавьте резервные одобрители.

    ![Пакет доступа - Запросы - Для пользователей из каталога - Второй Approver](./media/active-directory-entitlement-management-request-policy/out-directory-second-approver.png) 

1. Укажите количество дней, в течение которых второй утверждатель должен утвердить запрос в поле в соответствии с **решением, в течение нескольких дней?.** 

1. Установите обоснование утверждения требовать переключения **на Да** или **Нет.**

### <a name="alternate-approvers"></a>Альтернативные одобрители

Можно указать альтернативные утверждающие, аналогичные указаниям первого и второго утверждений, которые могут утверждать запросы. Наличие альтернативных утверждений поможет гарантировать, что запросы одобрены или отклонены до истечения срока их действия (тайм-аут). Можно перечислить альтернативных утверждение, первого и второго утверждение для 2-ступенчатого утверждения. 

Указывая альтернативные утверждающие, в случае, если первый или второй утверждающие не смогли утвердить или отклонить запрос, ожидающий запрос направляется альтернативным утверждением в график переадресов, указанный во время настройки политики. Они получают электронное письмо, чтобы утвердить или отклонить ожидающий запрос.

После того, как запрос будет направлен альтернативным утверждением, первый или второй утверждающих все еще может утвердить или отклонить запрос. Альтернативные правообладатели используют тот же сайт My Access для утверждения или отказа в запросе.

Мы можем перечислить людей или группы людей, которые будут одобрить и альтернативных одобрителей. Пожалуйста, убедитесь, что вы перечислите различные наборы людей, чтобы быть первым, вторым и альтернативным утверждением.
Например, если вы указали Алису и Боба в качестве первого утвердителя(ы), перечислите Кэрол и Дейв в качестве альтернативных одобрителей. Используйте следующие шаги, чтобы добавить альтернативные утвердители в пакет доступа:

1. В соответствии с первым утверждением, Вторым Утверждением, или оба, нажмите **Показать расширенные настройки запроса.**

    ![Пакет доступа - Политика - Отображите расширенные настройки запроса](./media/active-directory-entitlement-management-request-policy/alternate-approvers-click-advanced-request.png)

1. Установить **Если никаких действий, вперед, чтобы альтернативные утверждающих?** переключаться **на Да**.

1. Нажмите **Добавить альтернативные утвердители** и выберите альтернативный утверждение (ы) из списка.

    ![Пакет доступа - Политика - Добавить альтернативных одобрителей](./media/active-directory-entitlement-management-request-policy/alternate-approvers-add.png)

1. В **Форварде, чтобы заставить утвердить (ы) после того, как много дней** поле, положить в число дней, утверждающих должны утвердить или отклонить запрос. Если ни один из одобрителей не одобрил или не отклонил запрос до истечения срока запроса, срок действия запроса (тайм-аут), и пользователю придется отправить другой запрос на пакет доступа. 

    Запросы могут быть пересланы альтернативным утверждением только на следующий день после того, как продолжительность запроса достигнет периода полураспада. В этом примере продолжительность запроса составляет 14 дней. Таким образом, продолжительность запроса достигает полураспада в 7-й день. Таким образом, запрос не может быть переадресован раньше 8-го дня. Кроме того, запросы не могут быть пересылаться в последний день действия запроса. Таким образом, в примере, последний запрос может быть направлен день 13.

## <a name="enable-requests"></a>Включить запросы

1. Если вы хотите, чтобы пакет доступа был немедленно доступен пользователям в политике запроса, переместите переключатель Enable в **Yes.**

    Вы всегда можете включить его в будущем после того, как вы закончили создание пакета доступа.

    Если вы выбрали **None (только администратор прямые назначения)** и установили включить **No,** то администраторы не могут напрямую назначить этот пакет доступа.

    ![Пакет доступа - Настройка политики- Включить](./media/active-directory-entitlement-management-request-policy/enable-requests.png)

1. Щелкните **Далее**.
