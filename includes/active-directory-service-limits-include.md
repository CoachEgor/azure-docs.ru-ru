Ниже перечислены ограничения на использование и другие ограничения для службы Azure Active Directory (Azure AD).

| Категория | Ограничение |
| --- | --- |
| Каталоги | Один пользователь может принадлежать максимум к 500 каталогам Azure AD как член или гость.<br/>Один пользователь может создать до 200 каталогов. |
| Домены | Можно добавить не более 900 управляемых доменных имен. Если вы настраиваете все домены для федерации с помощью локальной службы Active Directory, в каждый каталог можно добавить до 450 доменных имен. |
|Ресурсы |<ul><li>Пользователи бесплатного выпуска службы Azure Active Directory могут по умолчанию создать в одном каталоге до 50 000 ресурсов Azure AD. Если у вас есть хотя бы один проверенный домен, заданная по умолчанию квота Azure AD для вашей организации увеличивается до 300 000 ресурсов Azure AD. Это ограничение службы не связано с ограничением ценовой категории в 500 000 ресурсов на странице цен на Azure AD. Чтобы не выйти за пределы квоты по умолчанию, необходимо обратиться в службу поддержки Майкрософт.</li><li>Пользователь без прав администратора может создать до 250 ресурсов Azure AD. В это число входят как активные ресурсы, так и удаленные ресурсы, доступные для восстановления. Восстановить можно только те ресурсы Azure AD, которые были удалены менее 30 дней назад. При расчете этой квоты для каждого удаленного ресурса Azure AD, который уже не доступен для восстановления, принимается значение одной четверти в течение 30 дней. Если у вас есть разработчики, которые предположительно будут постоянно превышать эту квоту в рамках своих обычных обязанностей, можно [создать и назначить настраиваемую роль](../articles/active-directory/users-groups-roles/roles-quickstart-app-registration-limits.md) с разрешением на создание бесконечного числа регистраций приложений.</li></ul> |
| Расширения схемы |<ul><li>Расширения типа String не должны превышать 256 символов. </li><li>Расширения типа Binary не должны превышать 256 байт.</li><li>В один ресурс Azure AD можно записать не более 100 значений расширений для *всех* типов и *всех* приложений.</li><li>Дополнить однозначными атрибутами типов String или Binary можно только сущности User, Group, TenantDetail, Device, Application и ServicePrincipal.</li><li>Расширения схемы доступны только в предварительной версии API Graph 1.21. Приложению должен быть предоставлен доступ на запись для регистрации расширения.</li></ul> |
| Приложения |Владельцами одного приложения могут быть не более 100 пользователей. |
|Манифест приложения |В манифест приложения можно добавить до 1200 записей. |
| Группы |<ul><li>Пользователь может создать до 250 групп в организации Azure AD.</li><li>В организации Azure AD может быть до 5000 динамических групп.<li>Владельцами одной группы могут быть не более 100 пользователей.</li><li>Участниками одной группы может быть любое число ресурсов Azure AD.</li><li>Пользователь может участником любого количества групп.</li><li>По умолчанию число участников в группе, которых можно синхронизировать из локальной службы Active Directory в Azure Active Directory с помощью Azure AD Connect, ограничено 50 000. Если вам нужно синхронизировать членство в группе сверх этого ограничения, подключите [API конечной точки версии 2 для синхронизации Azure AD Connect](../articles/active-directory/hybrid/how-to-connect-sync-endpoint-api-v2.md).</li><li>Вложенные группы в Azure AD поддерживаются не во всех сценариях.</li></ul><br/> Сейчас поддерживаются следующие сценарии с вложенными группами:<ul><li> Одну группу можно добавить в качестве члена другой группы, чтобы создать вложенность.</li><li> Утверждения членства в группах (когда приложение настроено для получения утверждений о членстве в группах в маркере; включены вложенные группы, в которые входит вошедший пользователь).</li><li>Условный доступ (при области применения политики условного доступа к группе).</li><li>Ограниченный доступ для самостоятельного сброса пароля.</li><li>Ограниченный круг пользователей, которые могут выполнять присоединение к Azure AD и регистрацию устройств.</li></ul><br/>Следующие сценарии НЕ поддерживают вложенные группы:<ul><li> Назначение ролей приложения (назначение групп приложению поддерживается, но группы, вложенные в непосредственно назначенную группу, не будут иметь доступа) для получения доступа и подготовки.</li><li>Лицензирование на основе групп (автоматическое назначение лицензии всем членам группы).</li><li>Группы Office 365.</li></ul> |
| Прокси приложения | <ul><li>До 500 транзакций в секунду на приложение прокси приложения.</li><li>До 750 транзакций в секунду на организацию Azure AD.</li></ul><br/>Транзакция определяется как один HTTP-запрос и ответ для уникального ресурса. При регулировании клиенты получат ответ 429 (слишком много запросов). |
| Панель доступа |Число приложений, которые могут отображаться на Панели доступа для каждого пользователя, не ограничено независимо от назначенных лицензий.  |
| Отчеты | В любом отчете можно просматривать и загружать не более 1000 строк. Любые дополнительные данные будут усечены. |
| Административные единицы | Ресурс Azure AD может входить не более чем в 30 административных единиц. |
| Роли и разрешения администратора | <ul><li>Группу нельзя добавить в качестве [владельца](https://docs.microsoft.com/azure/active-directory/fundamentals/users-default-permissions?context=azure/active-directory/users-groups-roles/context/ugr-context#object-ownership).</li><li>Возможность чтения сведений о каталоге других пользователей не может быть ограничена за пределами переключателя уровня организации Azure AD, чтобы отключить доступ всех пользователей без прав администратора ко всем данным каталога (не рекомендуется). См. сведения о [разрешениях по умолчанию](https://docs.microsoft.com/azure/active-directory/fundamentals/users-default-permissions?context=azure/active-directory/users-groups-roles/context/ugr-context#to-restrict-the-default-permissions-for-member-users).</li><li>Добавление и отмена членства в роли администратора может занимать до 15 минут и требовать выполнения выхода и входа.</li></ul> |
