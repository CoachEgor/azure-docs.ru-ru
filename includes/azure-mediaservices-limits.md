<sup>1</sup> Если изменить тип, например, с S2 на S1, максимальные зарезервированные лимиты единицы сбросить.

<sup>2</sup> Это число включает в себя очереди, законченные, активные и отмененные задания. Он не включает удаленные задания. Вы можете удалить старые задания с помощью **IJob.Delete** или запроса **DELETE** HTTP.

С 1 апреля 2017 года любая запись о вакансиях в вашем аккаунте старше 90 дней автоматически удаляется вместе с сопутствующими записями задач. Автоматическое удаление происходит даже в том случае, если общее количество записей ниже максимальной квоты. Для архивирования информации о заданиях и задачах используйте код, описанный в [активах Управления с медиа-сервисами .NET SDK.](../articles/media-services/previous/media-services-dotnet-manage-entities.md)

<sup>3</sup> При запросе на список объектов заданий возвращается не более 1000 заданий за запрос. Чтобы отслеживать все представленные задания, используйте верхние или пропустить запросы, описанные в [вариантах запроса системы OData.](/previous-versions/dynamicscrm-2015/developers-guide/gg309461(v=crm.7))

<sup>4</sup> Локаторы не предназначены для управления доступом к пользователю. Чтобы предоставить различные права доступа отдельным пользователям, используйте решения для управления цифровыми правами (DRM). Для получения дополнительной информации смотрите [В защиту вашего контента с Azure Media Services](../articles/media-services/previous/media-services-content-protection-overview.md).

<sup>5</sup> Учетные записи хранилища должны быть взяты из той же подписки Azure.

<sup>6</sup> Для различных политик Media Services установлен лимит в 1 000 000 полисов. Примером может быть политика Locator или ContentKeyAuthorizationPolicy. 

>[!NOTE]
> Если вы всегда используете одни и те же дни и разрешения доступа, используйте тот же идентификатор политики. Для получения информации и, например, [см. Управление активами с медиа-сервисами .NET SDK](../articles/media-services/previous/media-services-dotnet-manage-entities.md#limit-access-policies).

<sup>7 (г.</sup> Максимальный размер, поддерживаемый для одной капли, в настоящее время составляет до 5 ТБ в Azure Blob Storage. Дополнительные ограничения применяются в службах мультимедиа в зависимости от размеров VM, которые используются службой. Ограничение размера применяется к файлам, которые вы загружаете, а также файлам, которые генерируются в результате обработки медиа-услуг (кодирования или анализа). Если размер исходного файла превышает 260 ГБ, скорее всего, задание завершится ошибкой. 

В следующей таблице показаны ограничения на зарезервированные для мультимедиа единицы S1, S2 и S3. Если исходный файл больше, чем ограничения, определенные в таблице, задание кодирования завершается сбой. Если вы кодируете источники разрешения 4K длительной продолжительности, вам необходимо использовать зарезервированные устройства S3 для достижения требуемой производительности. Если у вас есть 4K-контент, превышающей лимит в 260 ГБ на зарезервированных устройствах S3, откройте билет поддержки.

|Тип единицы, зарезервированный для мультимедиа    |Максимальный размер ввода (GB)|
|---|---|
|S1 |    26|
|S2    | 60|
|S3    |260|
