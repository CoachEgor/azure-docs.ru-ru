Shared Image Gallery — это служба, которая помогает строить структуру и организацию вокруг управляемых изображений. Общие изображения Галереи обеспечивают:

- Управляемая глобальная репликация изображений.
- Версия и группировка изображений для облегчения управления.
- Высокодоступные изображения с учетными записями зон избыточного хранения данных в регионах, поддерживающих зоны доступности. ЗРС обеспечивает лучшую устойчивость к зональным сбоям.
- Обмен между подписками и даже между арендаторами Active Directory (AD) с помощью RBAC.
- Масштабирование развертываний с помощью реплик изображений в каждом регионе.

Используя коллекцию общих образов, вы можете делиться своими образами с другими пользователями, участниками службы или группами конструктора приложений в вашей организации. Общие образы можно реплицировать в несколько регионов для более быстрого масштабирования развертываний.

Управляемый образ представляет собой либо полную копию виртуальной машины (вместе со всеми подключенными дисками данных), либо просто копию диска операционной системы, в зависимости от того, как создается образ. При создании виртуальной машины из образа копии виртуальных жестких дисков в образе используются для создания дисков для новой виртуальной машины. Управляемый образ остается в хранилище и может использоваться для создания виртуальных машин снова и снова.

Если у вас есть большое количество управляемых изображений, которые необходимо поддерживать, и вы хотели бы сделать их доступными на всей территории вашей компании, вы можете использовать Общую галерею изображений в качестве репозитория, что позволяет легко обмениваться изображениями. 

Функция коллекции общих образов имеет несколько типов ресурсов.

| Ресурс | Описание|
|----------|------------|
| **Управляемое изображение** | Базовое изображение, которое может быть использовано в одиночку или использовано для создания **версии изображения** в картинной галерее. Управляемые изображения создаются из [обобщенных](#generalized-and-specialized-images) VM. Управляемый образ — это специальный тип виртуального жесткого диска, который может применяться для создания нескольких виртуальных машин. Теперь его можно использовать для создания версий общих образов. |
| **Снимок** | Копия VHD, которая может быть использована для использования **версии изображения.** Снимки могут быть взяты из [специализированных](#generalized-and-specialized-images) VM (тот, который не был обобщен), а затем использовать сярвидев дисков данных, для создания специализированной версии изображения.
| **Фотогалерея** | Как и Azure Marketplace, **коллекция образов** — это репозиторий для управления и совместного использования образов, но в отличие от Azure Marketplace доступ к коллекции образов контролируете вы. |
| **Определение образа** | Изображения определяются в галерее и содержат информацию об изображении и требованиях к его использованию в организации. Вы можете включить информацию, например, является ли изображение обобщенным или специализированным, операционную систему, минимальные и максимальные требования к памяти, а также примечания к выпуску. Это определение типа образа. |

Так же как и управляемый образ при использовании **версии образа** для создания виртуальной машины, версия образа используется для создания новых дисков для виртуальной машины. Версии образов можно использовать несколько раз. |

## <a name="image-definitions"></a>Определения изображений

Определения изображений являются логической группировкой для версий изображения. Определение изображения содержит информацию о том, почему изображение было создано, для чего оно предназначено для ОС, и информацию об использовании изображения. Определение изображения подобно плану для всех деталей вокруг создавать специфически изображение. Вы развертываете VM не из определения изображения, а из версии изображения, созданной из определения.

Есть три параметра для каждого определения изображения, которые используются в комбинации - **Издатель,** **Предложение** и **SKU**. Они используются для поиска определенного определения изображения. У вас могут быть версии образов, которые совместно используют одно или два, но не все три значения.  В качестве примера ниже приведены три определения образа и их значения.

|Определение образа|Издатель|ПРЕДЛОЖЕНИЕ|Sku|
|---|---|---|---|
|myImage1|Contoso|Finance.|Серверная часть|
|myImage2|Contoso|Finance.|Внешний интерфейс|
|myImage3|Тестирование|Finance.|Внешний интерфейс|

Все три определения образа имеют уникальные наборы значений. Формат аналогичен тому, как в настоящее время можно указать издателя, предложение и SKU для [изображений Azure Marketplace](../articles/virtual-machines/windows/cli-ps-findimage.md) в Azure PowerShell, чтобы получить последнюю версию изображения Marketplace. Каждое определение изображения должно иметь уникальный набор этих значений.

Ниже приведены другие параметры, которые могут быть установлены на определение изображения, так что вы можете легко отслеживать ваши ресурсы:

* Состояние операционной системы - Вы можете настроить состояние ОС на [обобщенный или специализированный.](#generalized-and-specialized-images)
* Операционная система - может быть либо Windows, либо Linux.
* Описание - используйте описание, чтобы дать более подробную информацию о том, почему существует определение изображения. Например, у вас может быть определение изображения для сервера переднего конца с предустановленным приложением.
* Eula - может быть использован атак на лицензионное соглашение с конечным пользователем, специфичным для определения изображения.
* Заметки о конфиденциальности и выписки - заметки о выпуске и выписки о конфиденциальности в хранилище Azure и предоставляют URI для доступа к ним в рамках определения изображения.
* Дата окончания срока службы - приложите дату окончания срока службы к определению изображения, чтобы иметь возможность использовать автоматизацию для удаления старых определений изображений.
* Тег - вы можете добавить теги при создании определения изображения. Для получения дополнительной информации о тегах [см.](../articles/azure-resource-manager/management/tag-resources.md)
* Минимальные и максимальные рекомендации vCPU и памяти - если ваше изображение имеет рекомендации vCPU и памяти, вы можете прикрепить эту информацию к определению изображения.
* Запрещенные типы дисков - вы можете предоставить информацию о потребностях в хранении для вашего VM. Например, если изображение не подходит для стандартных дисков С НС, вы добавляете их в список запретители.

## <a name="generalized-and-specialized-images"></a>Обобщенные и специализированные изображения

Есть два состояния операционной системы, поддерживаемых Общей галереей изображений. Обычно изображения требуют, чтобы VM, используемый для создания изображения, был обобщен перед съемкой изображения. Обобщение — это процесс, который удаляет информацию о машине и пользователе из VM. Для Windows используется и Sysprep. Для Linux можно использовать [waagent](https://github.com/Azure/WALinuxAgent) `-deprovision` или `-deprovision+user` параметры.

Специализированные ВМ не прошли через процесс удаления конкретной информации и учетных записей машины. Кроме того, VMs, созданные `osProfile` из специализированных изображений, не связаны с ними. Это означает, что специализированные изображения будут иметь некоторые ограничения.

- Учетные записи, которые могут быть использованы для входа в VM, также могут быть использованы на любом VM, созданном с помощью специализированного изображения, созданного из этого VM.
- VMs будет иметь **компьютерное имя** VM изображение было взято из. Вы должны изменить имя компьютера, чтобы избежать столкновений.
- Это `osProfile` как некоторые конфиденциальные сведения передаются `secrets`в ВМ, используя . Это может вызвать проблемы с использованием KeyVault, `osProfile`WinRM и других функций, которые используются `secrets` в . В некоторых случаях для решения этих ограничений можно использовать идентификаторы управляемых служб (MSI).

> [!IMPORTANT]
> Специализированные изображения в настоящее время в публичном предварительном просмотре.
> Эта предварительная версия предоставляется без соглашения об уровне обслуживания и не рекомендована для использования рабочей среде. Некоторые функции могут не поддерживаться или их возможности могут быть ограничены. Дополнительные сведения см. в статье [Дополнительные условия использования предварительных выпусков Microsoft Azure](https://azure.microsoft.com/support/legal/preview-supplemental-terms/).
>
> **Известные ограничения предварительного просмотра** ВМ могут быть созданы только из специализированных изображений с помощью портала или API. Поддержка предварительного просмотра не является поддержкой CLI или PowerShell.

## <a name="limits"></a>Ограничения 

Существуют ограничения на развертывание ресурсов с использованием общих изображений:
- 100 общих галерей изображений, за подписку, в регионе
- 1000 определений изображений, за подписку, в регионе
- 10 000 версий изображений, за подписку, на регион
- Любой диск, прикрепленный к изображению, должен быть меньше или равен размеру 1 ТБ

Для получения дополнительной информации [см. Проверьте использование ресурсов на примерах,](https://docs.microsoft.com/azure/networking/check-usage-against-limits) как проверить текущее использование.
 
## <a name="scaling"></a>Масштабирование
Коллекция общих образов позволяет указать число реплик, которые вы хотите, чтобы Azure сохранил для образов. Это помогает в сценариях развертывания нескольких виртуальных машин, поскольку их развертывания могут быть распространены на разные реплики, уменьшая вероятность того, что процесс создания экземпляра будет регулироваться из-за перегрузки отдельной реплики.

С помощью Общей image Gallery теперь можно развернуть до 1000 экземпляров VM в виртуальном наборе машин (до 600 с управляемыми изображениями). Реплики изображений обеспечивают более высокую производительность развертывания, надежность и согласованность. Можно установить различные колыбели реплик в каждом целевом регионе, основываясь на потребностях масштаба для региона. Поскольку каждая реплика является глубокой копией изображения, это помогает линейно масштабировать развертывание с каждой дополнительной репликой. Хотя мы понимаем, что нет одинаковых двух изображений или регионов, вот наше общее руководство по использованию реплик в регионе:

- Для развертывания невиртуального набора масштабов машин (VMSS) - Для каждых 20 виртуальных компьютеров, которые вы создаете одновременно, мы рекомендуем вам сохранить одну реплику. Например, если вы создаете 120 вм: одновременно с использованием одного и того же изображения в регионе, мы предлагаем вам сохранить по крайней мере 6 копий изображения. 
- Для развертывания виртуального набора масштабов машин (VMSS) - Для каждого набора масштабов развертывания до 600 экземпляров мы рекомендуем вам сохранить по крайней мере одну реплику. Например, если одновременно создается 5 наборов масштабов, каждый из которых имеет 600 экземпляров VM с использованием одного и того же изображения в одном регионе, мы предлагаем вам сохранить не менее 5 копий изображения. 

Мы всегда рекомендуем вам перезадезировать количество реплик из-за таких факторов, как размер изображения, содержание и тип ОС.

## <a name="replication"></a>Репликация
Коллекция общих образов также позволяет автоматически реплицировать свои образы в других регионах Azure. Каждую версию общего образа можно реплицировать в разных регионах в зависимости от того, что лучше всего подходит для вашей организации. Примером может служить репликация последних образов в нескольких регионах, хотя все старые версии доступны только в одном регионе. Это позволяет экономить на стоимости хранения для версий общих образов. 

Области, в которых реплицирована версия общего образа, можно обновить после создания. Время, необходимое для репликации в разных регионах, зависит от объема копируемых данных и количества регионов, в которые реплицируется версия. В некоторых случаях это может занять несколько часов. В процессе репликации можно просмотреть ее состояние для каждого региона. После завершения репликации изображения в регионе можно развернуть VM или масштабную версию, используя эту версию изображения в регионе.

## <a name="access"></a>Доступ

Поскольку общая галерея изображений, определение изображений и версия изображения являются ресурсами, они могут быть общими с помощью встроенных элементов управления Azure RBAC. С помощью RBAC вы можете поделиться этими ресурсами с другими пользователями, директорами служб и группами. Вы даже можете поделиться доступом к отдельным лицам за пределами арендатора они были созданы внутри. После того, как пользователь имеет доступ к версии Shared Image, он может развернуть VM или набор виртуальной шкалы машин.  Ниже приведена матрица общего доступа, которая помогает понять, к чему пользователь получает доступ.

| Доступ предоставлен пользователю     | Общая коллекция образов | Определение образа | Версия образа |
|----------------------|----------------------|--------------|----------------------|
| Общая коллекция образов | Да                  | Да          | Да                  |
| Определение образа     | нет                   | Да          | Да                  |

Мы рекомендуем делиться на уровне галереи для лучшего опыта. Мы не рекомендуем делиться отдельными версиями изображений. Для получения дополнительной информации о RBAC [см. Управление доступом к ресурсам Azure с помощью RBAC.](../articles/role-based-access-control/role-assignments-portal.md)

Изображения также могут быть общими, в масштабе, даже между арендаторами с помощью нескольких арендаторов регистрации приложений. Для получения дополнительной информации об обмене изображениями между арендаторами смотрите изображения [VM-галереи Share через арендаторов Azure.](../articles/virtual-machines/linux/share-images-across-tenants.md)

## <a name="billing"></a>Выставление счетов
За использование службы "Коллекция общих образов" дополнительная оплата не взимается. Плата взимается за следующие ресурсы.
- Расходы хранилища для хранения версий общих образов. Стоимость зависит от количества копий версии изображения и количества регионов, в которые реплицируется версия. Например, если у вас есть 2 изображения и оба реплицируются в 3 региона, то с вас будет взиматься плата за 6 управляемых дисков в зависимости от их размера. Дополнительные сведения см. в статье [Цены на управляемые диски](https://azure.microsoft.com/pricing/details/managed-disks/).
- Сборы за репликацию первой версии изображения из исходного региона в реплицированные регионы. Последующие реплики обрабатываются в пределах региона, поэтому нет никаких дополнительных сборов.

### <a name="how-can-i-list-all-the-shared-image-gallery-resources-across-subscriptions"></a>Как создать список всех ресурсов коллекции общих образов в разных подписках?

Чтобы перечислить все ресурсы общей галереи изображений в подписках, к которым у вас есть доступ на портале Azure, выполните ниже:

1. Откройте [портал Azure](https://portal.azure.com).
1. Перейдите в раздел **Все ресурсы**.
1. Выберите все подписки, из которых вы хотите включить в список все ресурсы.
1. Найдите ресурсы типа **Закрытая коллекция**.
 
   Для просмотра определений и версий образов следует также выбрать **Показать скрытые типы**.
 
   Чтобы получить список всех ресурсов коллекции общих образов в разных подписках, к которым у вас есть доступ, выполните следующую команду в Azure CLI.

   ```azurecli
   az account list -otsv --query "[].id" | xargs -n 1 az sig list --subscription
   ```

### <a name="can-i-move-my-existing-image-to-the-shared-image-gallery"></a>Можно ли перенести существующий образ в коллекцию общих образов?
 
Да. Существует 3 сценария, в зависимости от типов образов, которые у вас могут быть.

 Сценарий 1: Если у вас есть управляемое изображение в той же подписке, что и Ваш SIG, то вы можете создать определение изображения и версию изображения из него.

 Сценарий 2: Если у вас есть неуправляемое изображение в той же подписке, что и Ваш SIG, вы можете создать управляемое изображение из него, а затем создать определение изображения и версию изображения из него. 

 Сценарий 3: Если у вас есть VHD в локальной файловой системе, то вам нужно загрузить VHD на управляемое изображение, то вы можете создать определение изображения и версию изображения из него.

- Если VHD имеет Windows VM, [см.](https://docs.microsoft.com/azure/virtual-machines/windows/upload-generalized-managed)
- Если это виртуальный жесткий диск виртуальной машины Linux, см. раздел [Создание виртуальной машины Linux на основе пользовательского диска с помощью Azure CLI](https://docs.microsoft.com/azure/virtual-machines/linux/upload-vhd#option-1-upload-a-vhd).

### <a name="can-i-create-an-image-version-from-a-specialized-disk"></a>Можно ли создать версию образа из специализированного диска?

Да, поддержка специализированных дисков, как изображения в предварительном просмотре. Создать VM можно только на специализированном изображении с помощью портала[(Windows](../articles/virtual-machines/linux/shared-images-portal.md) или [Linux)](../articles/virtual-machines/linux/shared-images-portal.md)и API. Для предварительного просмотра поддержки PowerShell не существует.

### <a name="can-i-move-the-shared-image-gallery-resource-to-a-different-subscription-after-it-has-been-created"></a>Могу ли я переместить ресурс Общей галереи изображений в другую подписку после его создания?

Нет, ресурс коллекции общих образов невозможно переместить в другую подписку. Тем не менее при необходимости можно реплицировать версии образов в коллекции в другие регионы.

### <a name="can-i-replicate-my-image-versions-across-clouds-such-as-azure-china-21vianet-or-azure-germany-or-azure-government-cloud"></a>Могу ли я воспроизвести свои версии изображений в облаках, таких как Azure China 21Vianet или Azure Germany или Azure Government Cloud?

Нет, реплицировать версии образов между облаками невозможно.

### <a name="can-i-replicate-my-image-versions-across-subscriptions"></a>Можно ли реплицировать версии образов между подписками?

Нет, можно реплицировать версии образов между регионами в рамках подписки и использовать их в других подписках с помощью RBAC.

### <a name="can-i-share-image-versions-across-azure-ad-tenants"></a>Возможно ли предоставить общий доступ к версиям образов между клиентами Azure Active Directory? 

Да, вы можете использовать RBAC для обмена с отдельными лицами между арендаторами. Но, чтобы поделиться в масштабе, см. [PowerShell](../articles/virtual-machines/windows/share-images-across-tenants.md) [CLI](../articles/virtual-machines/linux/share-images-across-tenants.md)

### <a name="how-long-does-it-take-to-replicate-image-versions-across-the-target-regions"></a>Сколько времени занимает репликация версии образа между целевыми регионами?

Время репликации версии образа полностью зависит от размера образа и числа регионов, в которые он реплицируется. Однако для достижения наилучших результатов рекомендуется, чтобы размер образа был маленьким, а исходный и целевые регионы находились близко друг к другу. Можно проверить состояние репликации, используя флажок ReplicationStatus.

### <a name="what-is-the-difference-between-source-region-and-target-region"></a>В чем разница между исходным и целевым регионом?

Исходный регион — это регион, в котором будет создана версия образа. А целевые регионы — это регионы, в которых будет храниться копия вашей версии образа. Для каждой версии образа может быть только один исходный регион. Кроме того, убедитесь, что при создании версии образа вы передали расположение исходного региона как одно из целевых регионов.

### <a name="how-do-i-specify-the-source-region-while-creating-the-image-version"></a>Как указать исходный регион при создании версии образа?

При создании версии образа, чтобы указать исходный регион, можно использовать теги **--location** в интерфейсе командной строки и **-Location** в PowerShell. Убедитесь, что управляемый образ, который используется в качестве базового для создания версии образа, находится в том же расположении, в котором вы хотите создать версию образа. Кроме того, убедитесь, что при создании версии образа вы передали расположение исходного региона как одно из целевых регионов.  

### <a name="how-do-i-specify-the-number-of-image-version-replicas-to-be-created-in-each-region"></a>Как задать число реплик версии образа, создаваемых в каждом регионе?

Задать число реплик версии образа, создаваемых в каждом регионе, можно двумя способами:
 
1. Количество региональных реплик, которое указывает на число реплик, создаваемых на регион. 
2. Общее количество реплик, которое является значением по умолчанию для регионов в случае, если не указано количество региональных реплик. 

Чтобы указать количество региональных копий, передайте местоположение вместе с числом реплик, которые вы хотите создать в этом регионе: "South Central US-2". 

Если количество региональных реплик указано не для всех расположений, то количеством по умолчанию будет общее количество реплик, которое вы указали. 

Чтобы указать общее количество реплик в интерфейсе командной строки, используйте аргумент **--replica-count** в команде `az sig image-version create`.

### <a name="can-i-create-the-shared-image-gallery-in-a-different-location-than-the-one-for-the-image-definition-and-image-version"></a>Могу ли я создать общую галерею изображений в другом месте, чем для определения изображения и версии изображения?

Да, это возможно. Но, как наилучшая практика, мы рекомендуем вам сохранить группу ресурсов, общую галерею изображений, определение изображений и версию изображения в одном месте.

### <a name="what-are-the-charges-for-using-the-shared-image-gallery"></a>Какова стоимость использования коллекции общих образов?

За использование службы "Коллекция общих образов" плата не взимается, за исключением расходов на хранение версий образов и на исходящий трафик для репликации версий образов из исходного региона в целевые регионы.

### <a name="what-api-version-should-i-use-to-create-shared-image-gallery-and-image-definition-and-image-version"></a>Какую версию API следует использовать для создания общей галереи изображений и версии об определении изображений и изображения?

Для работы с коллекцией общих образов, определениями образов и версиями образов рекомендуется использовать API версии 2018-06-01. Зона избыточного хранения (ЗРС) требует версии 2019-03-01 или позже.

### <a name="what-api-version-should-i-use-to-create-shared-vm-or-virtual-machine-scale-set-out-of-the-image-version"></a>Какую версию API следует использовать для создания общей шкалы VM или виртуальной машины, установленной из версии изображения?

Для развертываний виртуальной машины и масштабируемого набора виртуальных машин с помощью версии образа рекомендуется использовать API версии 2018-04-01 или более поздней.
