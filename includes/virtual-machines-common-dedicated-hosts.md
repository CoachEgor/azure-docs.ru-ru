## <a name="limitations"></a>Ограничения

- Виртуальные наборы масштабов машин в настоящее время не поддерживаются на выделенных хостах.

## <a name="benefits"></a>Преимущества 

Резервирование всего хоста обеспечивает следующие преимущества:

-   Изоляция оборудования на физическом уровне сервера. Никакие другие VMs не будут размещены на ваших хостовах. Выделенные хосты развертываются в тех же центрах обработки данных и имеют ту же сеть и базовую инфраструктуру хранения данных, что и другие, неизолированные узлы.
-   Контроль за событиями обслуживания, инициированный платформой Azure. Хотя большинство событий обслуживания практически не влияют на ваши виртуальные машины, есть некоторые чувствительные рабочие нагрузки, где каждая секунда паузы может оказать влияние. С выделенными хостами вы можете зарегистрироваться в окне обслуживания, чтобы уменьшить влияние на вашу службу.
-   Благодаря преимуществам гибридного преимущества Azure вы можете принести свои собственные лицензии для Windows и S'L в Azure. Использование гибридных преимуществ дает вам дополнительные преимущества. Для получения дополнительной [Azure Hybrid Benefit](https://azure.microsoft.com/pricing/hybrid-benefit/)информации см.


## <a name="groups-hosts-and-vms"></a>Группы, хосты и ВМ  

![Просмотр новых ресурсов для выделенных узлов.](./media/virtual-machines-common-dedicated-hosts/dedicated-hosts2.png)

**Группа узлов** — это ресурс, представляющий коллекцию выделенных узлов. Вы создаете группу узлов в регионе и зоне доступности и добавляете к ней хосты.

**Хост** — это ресурс, отображенный на физическом сервере в центре обработки данных Azure. Физический сервер выделяется при создании узла. Хост создается в группе хоста. У хоста есть SKU, описывающий, какие размеры VM могут быть созданы. Каждый хост может принимать у себя несколько VMs разных размеров, если они из одной серии размеров.

При создании VM в Azure можно выбрать выделенный хост для использования в вашем VM. Вы имеете полный контроль над тем, какие вмха размещены на хостов.


## <a name="high-availability-considerations"></a>Соображения высокой доступности 

Для высокой доступности следует развернуть несколько вс-случаев, разбросанных по нескольким узлам (минимум 2). С помощью выделенных хостов Azure у вас есть несколько вариантов предоставления инфраструктуры для формирования границ изоляции неисправностей.

### <a name="use-availability-zones-for-fault-isolation"></a>Использование зон доступности для изоляции неисправностей

Зоны доступности являются уникальными физическими местоположениями в регионе Azure. Каждая зона состоит из одного или нескольких центров обработки данных, оснащенных независимыми системами электроснабжения, охлаждения и сетевого взаимодействия. Группа хоста создается в одной зоне доступности. После создания все хосты будут размещены в этой зоне. Для достижения высокой доступности в разных зонах необходимо создать несколько групп узлов (по одной в зоне) и соответствующим образом распределить хосты.

При присвоении группы узла зоне доступности все ввожжа, созданные на этом узлах, должны быть созданы в одной и той же зоне.

### <a name="use-fault-domains-for-fault-isolation"></a>Используйте домены неисправностей для изоляции неисправностей

Узла можно создать в определенном домене неисправности. Так же, как VM в наборе масштабов или наборе доступности, хосты в разных доменах неисправностей будут размещены на разных физических стойках в центре обработки данных. При создании группы узла необходимо указать количество доменов неисправностей. При создании узлов в группе узлов вы назначаете домена неисправностей для каждого узла. VMs не требуют каких-либо назначения домена неисправности.

Домены неисправностей не совпадают с коллокацией. Наличие одного и того же домена неисправности для двух узлов не означает, что они находятся в непосредственной близости друг от друга.

Домены неисправностей относятся к группе хоста. Не следует делать никаких предположений о противородении между двумя группами узлов (за исключением тех случаев, когда они находятся в разных зонах доступности).

VMs, развернутые в хостах с различными доменами неисправностей, будут иметь свои базовые управляемые диски на нескольких марках хранения, чтобы увеличить защиту изоляции неисправностей.

### <a name="using-availability-zones-and-fault-domains"></a>Использование зон доступности и доменов неисправностей

Вы можете использовать обе возможности вместе, чтобы достичь еще большей изоляции неисправностей. В этом случае вы укажете зону доступности и количество доменов неисправностей для каждой группы узлов, назначите домена неисправности каждому из узлов в группе и присвоите зону доступности каждому из ваших ВМ

В найденном [здесь](https://github.com/Azure/azure-quickstart-templates/blob/master/201-vm-dedicated-hosts/README.md) шаблоне «менеджер ресурсов» используются зоны и домены неисправностей для распространения узлов для максимальной устойчивости в регионе.

## <a name="maintenance-control"></a>Управление обслуживанием

Инфраструктура, поддерживающая виртуальные машины, иногда может обновляться для повышения надежности, производительности, безопасности и запуска новых функций. Платформа Azure пытается свести к минимуму влияние обслуживания платформы, когда это возможно, но клиенты с *чувствительными* рабочими нагрузками обслуживания не могут терпеть даже несколько секунд, что VM должен быть заморожен или отключен для обслуживания.

**Maintenance Control** предоставляет клиентам возможность пропускать регулярные обновления платформы, запланированные на их выделенных хостах, а затем применять их в момент их выбора в течение 35-дневного прокатного окна.

> [!NOTE]
>  Управление обслуживанием в настоящее время находится в общедоступном предварительном просмотре. Для получения дополнительной информации смотрите **обновления управления с помощью управления обслуживанием с помощью [CLI](https://docs.microsoft.com/azure/virtual-machines/maintenance-control-cli?toc=/azure/virtual-machines/linux/toc.json&bc=/azure/virtual-machines/linux/breadcrumb/toc.json) или [PowerShell](https://docs.microsoft.com/azure/virtual-machines/maintenance-control-powershell?toc=/azure/virtual-machines/linux/toc.json&bc=/azure/virtual-machines/linux/breadcrumb/toc.json)**.

## <a name="capacity-considerations"></a>Рекомендации по емкости

После того, как выделенный узел подготовлен, Azure назначает его физическому серверу. Это гарантирует наличие емкости, когда вам нужно предоставить ваш VM. Azure использует всю емкость в регионе (или зоне) для выбора физического сервера для узла. Это также означает, что клиенты могут рассчитывать на то, что они смогут увеличить свой выделенный след хоста без заботы о том, что в кластере не будет места.

## <a name="quotas"></a>Квоты

Существует ограничение квоты по умолчанию 3000 vCPUs для выделенных хостов, в регионе. Но количество узлов, которые можно развернуть, также ограничено квотой для семейства размеров VM, используемой для хоста. Например, подписка **с оплатой по мере использования** может иметь только квоту в 10 vCPUs, доступную для серии размеров Dsv3 в восточном регионе США. В этом случае необходимо запросить увеличение квоты по крайней мере до 64 vCPUs, прежде чем вы сможете развернуть выделенный хост. Выберите кнопку **увеличения запроса** в правом верхнем углу, чтобы при необходимости подать запрос.

![Скриншот страницы использования и квот на портале](./media/virtual-machines-common-dedicated-hosts/quotas.png)

Для получения дополнительной информации, см [Виртуальная машина vCPU квоты](/azure/virtual-machines/windows/quotas).

Бесплатная пробная версия и подписка MSDN не имеют квоты для выделенных хостов Azure.

## <a name="pricing"></a>Цены

Пользователи взимаются за выделенный хост, независимо от количества развертываемых VMs. В ежемесячном отчете вы увидите новый оплачиваемый тип ресурсов хостов. ВМ на выделенном хосте по-прежнему будут отображаться в вашем заявлении, но будут иметь цену 0.

Цена хоста устанавливается на основе семейства VM, типа (размер оборудования) и региона. Цена хоста относительно самого большого размера VM, поддерживаемого на хосте.

Лицензирование программного обеспечения, хранение и использование сети выставляются отдельно от хоста и VMs. Эти оплачиваемые элементы не изменяются.

Для получения дополнительной [Azure Dedicated Host pricing](https://aka.ms/ADHPricing)информации см.

Вы также можете сэкономить на расходах с [помощью зарезервированных экземпляров выделенных хостов Azure.](../articles/virtual-machines/prepay-dedicated-hosts-reserved-instances.md)
 
## <a name="sizes-and-hardware-generations"></a>Поколения размеров и оборудования

SKU определяется для хоста и представляет серию и тип размера VM. Вы можете смешивать несколько VMs разных размеров в пределах одного хоста до тех пор, пока они имеют один и тот же размер серии. 

*Тип* аппаратного поколения. Различные типы оборудования для одной и той же серии VM будут от разных поставщиков процессоров и имеют различные поколения процессоров и количество ядер. 

Размеры и типы оборудования варьируются в зависимости от региона. Чтобы узнать больше, обратитесь к [странице ценообразования](https://aka.ms/ADHPricing) хоста.


## <a name="host-life-cycle"></a>Цикл жизни хозяина


Azure отслеживает и управляет состоянием работоспособности узлов. Следующие состояния будут возвращены при запросе хоста:

| Состояние работоспособности   | Описание       |
|----------|----------------|
| Хост доступен     | С хостом не существует никаких проблем.   |
| Хозяин под следствием  | У нас возникли некоторые проблемы с хозяином, который мы изучаем. Это переходное состояние, необходимое Azure, чтобы попытаться определить область и основную причину выявленной проблемы. Виртуальные машины, работающие на хосте, могут быть затронуты. |
| Хост в ожидании дека,   | Azure не может восстановить узел в работоспособном состоянии и попросить вас передислоцировать виртуальные машины из этого узла. Если `autoReplaceOnFailure` включено, ваши виртуальные машины *служат исцелил* на здоровое оборудование. В противном случае, ваша виртуальная машина может работать на хосте, который вот-вот сойдется.|
| Сделка с принимающей стороной  | Все виртуальные машины были удалены из хостов. Вы больше не взимается плата за этот хост, так как оборудование было выведено из ротации.   |
